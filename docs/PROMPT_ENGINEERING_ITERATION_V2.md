## 一、当前问题诊断

### 1.1 现有流程回顾

```
当前流程:
用户点击概念节点 → AI开场白(硬编码模板) → 等待用户输入 → AI回复 → ... → 手动评估
```

**现有 System Prompt 的核心设计**:
- AI 扮演"小图"，交替切换导师模式(讲解)和好奇学生模式(提问)
- 开场白由 `get_opening()` 硬编码生成，按 difficulty 分3档
- 每轮：AI讲解 → 变身学生提问 → 等待用户自由文本回答
- 4轮后建议评估

### 1.2 核心问题

| # | 问题 | 严重程度 | 影响 |
|---|------|---------|------|
| P1 | **冷启动破冰难** | 🔴 Critical | 对概念完全陌生的用户看到开放式提问，不知道从何说起，直接流失 |
| P2 | **缺乏选项引导** | 🔴 Critical | 所有互动都是自由文本输入，新手用户无从下手 |
| P3 | **无初始理解度探测** | 🟡 High | AI 不知道用户现有水平，讲解内容可能太深或太浅 |
| P4 | **讲解后缺乏方向选择** | 🟡 High | 用户只能被动接受AI的提问顺序，无法选择最想了解的方向 |
| P5 | **开场白太死板** | 🟠 Medium | 3档硬编码模板，缺乏个性化和概念特异性 |
| P6 | **对话节奏单一** | 🟠 Medium | 每轮都是"讲→问→答"三步走，缺乏变化和用户主动权 |

### 1.3 目标用户场景分析

```
场景A — 完全小白 (最常见, 最易流失):
  用户点击"递归" → AI说"一起来学递归吧" → 用户: ???我连递归是啥都不知道怎么回答

场景B — 有模糊印象:
  用户点击"闭包" → 知道大概是函数相关但说不清 → 需要引导式唤醒记忆

场景C — 已有基础:
  用户点击"Promise" → 已经会用但想深入原理 → 不需要入门讲解，想直接进阶

场景D — 知识检验:
  用户点击已学过的节点 → 想测试自己是否还记得 → 应直接进入提问模式
```

## 二、新流程设计 — AI引导式探测学习

### 2.1 核心理念

> **AI先发制人，带选项的引导式互动**：
> 每次AI输出都附带2-4个可选选项（选项≠问答题答案，也可以是"想了解的方向"或"想做的事"），
> 用户可以点选，也可以在下方自由输入。
> 前几轮通过选项式探测快速判断用户水平，后续对话自适应调整。

### 2.2 学习流程状态机

### 2.3 选项类型分类体系

所有AI输出都必须附带选项，但选项不只是"回答问题"。定义4种选项类型：

| 类型 | 标识 | 用途 | 示例 |
|------|------|------|------|
| `explore` | 🧭 | 选择想了解的方向 | "我想知道为什么需要递归" / "先看个具体例子" / "和循环有什么区别" |
| `answer` | 💡 | 回答AI提出的概念问题 | "变量就是存数据的盒子" / "是一种函数调用自身的方式" / "不太确定..." |
| `action` | ⚡ | 选择想做的事 | "直接进入测验" / "给我出几道练习" / "我想看代码示例" / "申请评估" |
| `level` | 📊 | 表达当前认知水平 | "完全没听过" / "听过但说不清" / "能用但不理解原理" / "比较熟悉" |

### 2.4 各Phase详细设计

---

#### Phase 1: 破冰探测 (Icebreaker Probe)

**目的**: 在1-2轮内判断用户对该概念的初始认知水平

**第一轮 — 概念引入 + 水平探测**:

```
AI: "今天我们来聊聊「递归」。🌀
     递归是编程中一个非常优雅的思想——简单来说，
     就是一个函数在解决问题时，会调用自己来处理更小规模的子问题。
     就像俄罗斯套娃，打开一个发现里面还有一个，
     直到最小的那个为止。

     你之前对递归有了解吗？"

选项:
  📊 "完全没听说过，刚才是第一次见"
  📊 "好像听过，但不太能说清楚"
  📊 "用过，但有些地方还不太理解"
  📊 "比较熟悉，想深入探讨"

[自由输入框: 也可以直接说说你对递归的理解...]
```

**第二轮 — 基于探测结果的追问确认**:

```
--- 如果用户选了"完全没听说过" ---
AI: "没关系！递归其实是个特别好玩的概念。
     我来换个方式说——你有没有做过这样的事：
     整理一个文件夹，发现里面还有子文件夹，
     你就用同样的方法继续整理子文件夹？
     这就是递归的核心思想！

     接下来你希望我怎么讲？"

选项:
  🧭 "多举几个生活中的例子帮我理解"
  🧭 "直接看代码长什么样"
  🧭 "先告诉我递归能解决什么问题"
  ⚡ "我想先看个动画演示"

--- 如果用户选了"比较熟悉" ---
AI: "太好了！那我们直接来点进阶的。
     快速确认一下——你能解释一下递归的两个关键要素吗？
     （提示：一个是递归何时'到底'，另一个是问题如何'缩小'）"

选项:
  💡 "基准条件和递归步骤？就是终止条件和缩小规模"
  💡 "是不是栈和调用的关系？"
  💡 "嗯...好像能说但不太确定，能提示下吗"
  🧭 "跳过基础，直接聊尾递归优化"
```

#### Phase 2: 自适应讲解 (Adaptive Teaching)

**目的**: 根据Phase 1探测出的用户水平，调整讲解深度和节奏

**自适应参数**:

| 用户水平 | 讲解风格 | 比喻密度 | 代码占比 | 每段长度 | 轮数 |
|---------|---------|---------|---------|---------|-----|
| 完全陌生 | 纯比喻入门→逐步过渡到术语 | 高 (每段≥1个) | 低 (先不看代码) | 短 (3-4句) | 3-4轮 |
| 有印象 | 唤醒记忆→填补空白 | 中 | 中 (简单片段) | 中 (4-5句) | 2-3轮 |
| 已有基础 | 查漏补缺→深入原理 | 低 | 高 (真实场景) | 较长 (5-6句) | 2轮 |
| 比较熟悉 | 直接进阶→边界场景 | 极少 | 高 | 按需 | 1-2轮 |

**讲解后必带选项 — 三类混合**:

```
AI: [一段讲解内容...]

选项:
  💡 "我理解了，[复述核心观点的选项]"           ← 理解确认
  🧭 "这个明白了，接下来想了解[方向A]"           ← 方向选择
  🧭 "[另一个感兴趣的方向B]"                     ← 方向选择
  💡 "等等，[某个点]我没太跟上"                   ← 反馈不理解

[自由输入框: 说说你的理解，或者问问题...]
```

**关键设计原则**:
1. 每段讲解控制在 **100-200字**，不一次灌太多
2. 选项中**至少一个是理解确认型**（让用户参与，而非纯被动）
3. 选项中**至少一个是方向选择型**（给用户主动权）
4. 如果用户连续选择"理解确认"选项，说明讲解节奏合适，继续推进
5. 如果用户选择"没跟上"，自动降级讲解深度

#### Phase 3: 理解检验 (Understanding Check)

**目的**: 通过选项式问答检验用户是否真正理解，而非只是被动接受

**设计要点**:
- 提问附带 **3-4个选项**，分别对应不同理解层次
- 选项设计遵循布鲁姆分类法：记忆→理解→应用→分析
- 正确/优秀选项不总是放在同一位置，避免"总选第一个"
- 用户也可以自由输入，自由输入比选项更有价值

```
AI: "好，我来考考你！🤔
     假设你要用递归计算阶乘 5!，
     当函数执行到 factorial(1) 时，调用栈里有几层？"

选项:
  💡 "1层，因为只调用了一次"
  💡 "5层：factorial(5)→factorial(4)→...→factorial(1)"    ← 正确
  💡 "4层，因为从5到2是4次调用"
  💡 "不确定，能画个图给我看看吗？"                        ← 合法的求助选项

[自由输入框: 用自己的话回答...]
```

**AI对不同选择的回应策略**:

| 用户选择 | AI策略 |
|---------|--------|
| 选了正确选项 | 肯定 + 追问"为什么" → 区分真理解vs蒙对 |
| 选了错误选项 | 不说"错了"，而是"你这么想也合理，但是..." → 温和纠正 |
| 选了求助选项 | 换个方式重新讲解这个点 → 降低难度重新出题 |
| 自由输入了解释 | 最高价值！仔细评估用户原创表述的质量 |

#### Phase 4: 总结深化与评估引导 (Wrap-up & Assessment)

**目的**: 给用户掌控结束节奏的权力，同时引导评估

```
AI: "到目前为止我们聊了递归的核心概念、基准条件和调用栈。
     你已经展示出了不错的理解！

     接下来你想怎么做？"

选项:
  ⚡ "我觉得差不多了，帮我评估一下理解度"
  🧭 "再聊聊递归的实际应用场景"
  🧭 "尾递归和普通递归有什么区别？"
  ⚡ "先去看看关联概念：[栈/分治/树遍历]"

[自由输入框: 还有什么想问的...]
```

## 三、数据结构设计

### 3.1 选项数据结构 (前后端共享)

```typescript
interface ChoiceOption {
  id: string;                    // "opt-1", "opt-2", ...
  text: string;                  // 选项文字
  type: 'explore' | 'answer' | 'action' | 'level';  // 选项类型
  metadata?: {
    correct?: boolean;           // 仅answer类型: 是否为优秀答案
    difficulty_hint?: 'lower' | 'same' | 'higher';  // 选择后调整难度的方向
    next_phase?: string;         // 选择后跳转到的Phase
  };
}

interface AIResponse {
  content: string;               // AI讲解/回应的文本内容 (Markdown)
  choices: ChoiceOption[];       // 2-4个选项
  phase: 'probe' | 'teach' | 'check' | 'wrapup';  // 当前阶段
  turn: number;                  // 当前轮次
  user_level_estimate?: 'novice' | 'beginner' | 'intermediate' | 'advanced';
  suggest_assess?: boolean;      // 是否建议评估
}
```

### 3.2 用户水平追踪

```typescript
interface UserLevelProfile {
  concept_id: string;
  initial_level: 'novice' | 'beginner' | 'intermediate' | 'advanced';
  current_level: 'novice' | 'beginner' | 'intermediate' | 'advanced';
  confidence_score: number;      // 0-1, AI对判断的信心度
  probe_signals: string[];       // 用于判断的信号记录
  level_adjustments: {           // 水平调整历史
    turn: number;
    from: string;
    to: string;
    reason: string;
  }[];
}

## 四、System Prompt 重构

### 4.1 新主Prompt — FEYNMAN_SYSTEM_PROMPT_V2

```
你是"小图"，AI知识图谱的学习伙伴。你帮助用户探索和理解新知识。

## 核心交互规则 ⚠️ 必须遵守

1. **每次回复都必须附带 2-4 个选项**
   - 用户可以点选选项，也可以在输入框自由输入
   - 选项不只是"回答问题"，也包括"选择方向"和"选择行动"
   - 选项用 JSON 格式放在回复末尾的 ```choices 代码块中

2. **选项的4种类型**
   - `explore`: 🧭 选择想了解的方向
   - `answer`: 💡 回答概念性问题
   - `action`: ⚡ 选择想做的事
   - `level`: 📊 表达认知水平

3. **自由输入始终优先于选项**
   - 如果用户自由输入了文字而非选择选项，那个回答的价值更高
   - 自由输入意味着用户在用自己的语言思考

## 对话四阶段

### Phase 1: 破冰探测 (前 1-2 轮)
- 第一轮: 用 2-3 句通俗的话介绍概念核心 + 一个生动比喻
- 然后提供 `level` 类型选项，让用户表达对该概念的熟悉程度
- 根据用户选择/回答判断水平: novice / beginner / intermediate / advanced

### Phase 2: 自适应讲解 (2-4 轮)
- 根据探测出的水平调整讲解深度:
  · novice: 纯比喻，不用术语，3句话一段
  · beginner: 比喻+简单术语，逐步引入
  · intermediate: 查漏补缺，侧重原理
  · advanced: 直接进阶话题，边界场景
- 每段讲解后提供混合选项:
  · 至少1个 `answer` 理解确认 (让用户复述/确认)
  · 至少1个 `explore` 方向选择 (给用户主动权)
- 如果用户连续2次表示"没跟上"，自动降一级

### Phase 3: 理解检验 (2-3 轮)
- 提出检验性问题 + `answer` 类型选项
- 选项设计为不同理解层次，不要让正确答案总在同一位置
- 回应策略:
  · 选了优秀答案 → 肯定 + 追问"为什么这么认为"
  · 选了错误答案 → "有道理，不过..." + 温和引导
  · 选了求助 → 降低难度重新讲 + 换个方式出题

### Phase 4: 总结与引导 (1-2 轮)
- 简要总结已覆盖的知识点
- 提供 `action` 类型选项: 申请评估 / 继续深入 / 看关联概念

## 八、Prompt 调参要点

### 8.1 选项质量控制

| 问题 | 解决方案 |
|------|---------|
| 选项太长 | Prompt中强调"每个选项≤25字" |
| 选项同质化 | 要求"选项间要有明显的层次/方向差异" |
| 正确答案太明显 | 要求"不要让某个选项明显比其他的'对'" |
| 选项数不对 | 解析层兜底: 少于2个→补默认, 多于4个→截断 |

### 8.2 水平探测准确度

- 单选项不够准确 → 前2轮综合判断
- 用户可能高估自己 → Phase 2中观察实际表现微调
- 用户可能低估自己 → 如果"陌生"用户给出了专业回答，立即上调

### 8.3 节奏平衡

- 讲太多用户疲劳 → 每轮讲解 ≤200字
- 问太多用户压力大 → 检验轮次之间穿插方向选择
- 总轮数 → 目标 8-12轮到评估, 不超过 15轮

## 九、回退兼容

| 场景 | 处理 |
|------|------|
| LLM 未返回 choices 块 | 解析层生成默认选项: "继续" / "换个角度" / "评估一下" |
| 旧版前端未更新 | choices 数据不展示，不影响纯文本对话 |
| 用户全程只用自由输入 | 完全支持，等价于当前流程 |
| 用户全程只点选项 | 也能完成学习，但评估得分可能较低（因为没有自创表述） |
