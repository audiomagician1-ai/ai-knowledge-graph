# 思维链（Chain-of-Thought, CoT）

## 概述

思维链（Chain-of-Thought，简称CoT）是一种提示技术，通过在提示词中插入或引导模型生成**逐步推理过程**，使大语言模型在给出最终答案之前先输出中间推理步骤。这种方法由Google Brain团队的Jason Wei等人于2022年1月在论文《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》中正式提出并系统验证。

CoT的提出直接针对大语言模型在算术、常识和符号推理任务上的"直接答题"缺陷。在Wei等人的原始实验中，标准少样本提示在GSM8K数学基准上的准确率约为17.9%，而加入思维链提示后，同等模型（PaLM 540B）的准确率跃升至56.9%，提升幅度超过3倍。

CoT的重要性不仅在于提升准确率，还在于它将模型的推理过程变为**可检查的文本序列**，使人类或下游系统能够验证每一步推理的合理性，为可解释AI提供了一条可操作路径。

## 核心原理

### 示例触发式CoT（Few-Shot CoT）

原始CoT方法属于少样本触发式：在提示词中提供若干包含"问题→推理步骤→答案"格式的示例，模型通过上下文学习（in-context learning）模仿该格式，在回答新问题时自动展开推理链。

标准示例格式如下：

```
问题：Roger有5个网球，他又买了2罐，每罐3个球，共几个球？
推理：Roger原有5个球。2罐 × 每罐3个 = 6个新球。5 + 6 = 11。
答案：11个。
```

关键约束是：示例中的推理步骤必须**语言自然且步步衔接**，不能跳步，否则模型会学习到不完整的推理模式。

### 零样本CoT（Zero-Shot CoT）

Kojima等人于2022年5月发现，仅在问题末尾追加短语 **"Let's think step by step"**（中文常译为"让我们一步步思考"），即可在无任何示例的情况下激活模型的推理链行为，将MultiArith数据集上的准确率从17.7%提升至78.7%。这一发现说明足够大的模型已经内化了链式推理能力，CoT提示只需起到"解锁"作用，而非从头教授。

这一机制背后的假设是：预训练语料中大量存在"step 1…step 2…therefore…"格式的解题文本（如教材、解题博客），使模型学会了将该格式与正确推理关联。

### 模型规模的临界效应

CoT并非对所有模型都有效。Wei等人的实验数据显示，CoT提示对参数量**小于100B**的模型通常无效甚至有害——小模型生成的推理链往往是语义连贯但逻辑错误的"幻觉链"。只有在约100B参数规模以上，CoT才呈现出涌现性能提升。这个临界值被称为CoT的**"涌现阈值"（emergent threshold）**，是设计AI系统时选择基础模型的重要参考依据。

### 推理链的数学表达

设问题为 $x$，推理链为 $z = (z_1, z_2, \ldots, z_n)$，最终答案为 $y$，则CoT可以表达为：

$$P(y \mid x) = \sum_{z} P(y \mid x, z) \cdot P(z \mid x)$$

即最终答案的概率通过对所有可能推理路径的边缘化得到。自一致性采样（Self-Consistency）正是利用这一框架，通过采样多条 $z$ 并投票来提高 $P(y \mid x)$ 的估计质量。

## 实际应用

**数学应用题（Grade School Math）**：在GSM8K基准上，GPT-4使用CoT提示的准确率可达92%以上，而不使用CoT时约为75%。典型应用场景是让模型列出"已知量→中间计算→结果"的每一步，避免直接跨步计算导致的错误。

**多步代码调试**：在工程实践中，可以要求模型先输出"错误原因分析→受影响代码路径→修复方案→验证逻辑"，而不是直接输出修复后代码。这使代码审查者能逐步核查模型的诊断逻辑是否合理。

**结构化决策支持**：在医疗或法律类应用中，CoT提示格式通常为："请先列出相关事实，再逐条分析，最后给出结论。" 这种格式可减少模型跳过关键约束条件直接给出结论的风险，并为人工复审提供审计轨迹。

**提示词示例（中文工程实践）**：

```
你是一个财务分析助手。对于以下计算问题，请先列出每一步推导过程，再给出最终答案。

问题：某产品售价120元，成本80元，本月销售1500件，退货率5%，
请计算实际利润。
```

## 常见误区

**误区一：推理步骤越长越准确**
增加推理链长度并不线性提升准确率。当推理步骤超过必要数量时，模型可能引入无关中间变量或在某一步骤发生"推理漂移"，导致后续步骤均基于错误前提展开。Google的实验发现，在部分逻辑谜题上，过度详细的CoT反而比简洁版本准确率低约4-7个百分点。

**误区二：CoT等同于提供答题模板**
CoT提示中的示例应展示**推理模式**，而非仅提供答案格式。如果示例的推理步骤与问题领域不匹配（如用算术推理示例引导法律分析），模型会套用错误的推理结构。这与少样本提示中仅展示输入输出格式的做法有本质区别——CoT的关键信息承载在推理过程本身，而非答案格式。

**误区三：Zero-Shot CoT可以替代领域示例**
"Let's think step by step"对通用推理任务效果显著，但在高度专业化领域（如量子电路分析或IUPAC化学命名）中，Zero-Shot CoT生成的推理链常包含领域错误。此时必须回归Few-Shot CoT，提供包含正确领域推理的示例，Zero-Shot的便捷性以牺牲专业精度为代价。

## 知识关联

**与少样本提示的关系**：少样本提示（Few-Shot Prompting）为CoT提供了示例注入机制——CoT的Few-Shot变体本质上是将少样本提示中的"答案示例"替换为"推理链+答案示例"。理解少样本提示的上下文窗口限制有助于合理控制CoT示例数量（通常以4-8个为宜，避免挤占问题本身的上下文空间）。

**向自一致性采样的延伸**：CoT生成的推理链具有随机性——对同一问题采样多次会产生不同推理路径。自一致性采样（Self-Consistency）在此基础上引入多数投票机制：对同一问题并行生成若干条CoT推理链，选取最高频的最终答案，这可将GSM8K准确率再提升约10个百分点。

**思维树（ToT）的结构扩展**：CoT是线性的推理序列，而思维树将单条推理链扩展为树状搜索结构，允许在每个推理节点进行分支、评估和回溯。ToT解决了CoT无法处理需要"试错-撤销"的组合搜索问题（如24点游戏或创意写作规划）。

**在ReAct框架中的嵌入**：ReAct（Reasoning + Acting）将CoT推理步骤与外部工具调用（搜索、API）交织排列，格式为"思考→行动→观察→思考→…"。CoT提供了ReAct中"思考"节点的推理生成能力，是ReAct框架的推理基础。

**DSPy的自动化替代**：DSPy框架通过程序化方式自动优化CoT示例的选取和格式，替代人工设计推理链示例的工作。理解手动CoT的构造逻辑是使用DSPy的前提，因为DSPy的优化目标正是最大化CoT推理链的任务准确性。
