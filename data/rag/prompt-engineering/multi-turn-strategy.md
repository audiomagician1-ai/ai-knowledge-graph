# 多轮对话策略

## 概述

多轮对话策略是指在与大语言模型（LLM）交互时，通过设计结构化的Prompt模板和上下文管理机制，使对话能够跨越多个轮次保持语义连贯、任务目标一致的工程方法。与单轮Prompt不同，多轮对话需要显式管理"对话历史"（conversation history）的累积、压缩和遗忘，因为GPT-4等模型的上下文窗口存在硬性限制（如128K tokens），超出后早期信息会被截断丢失。

多轮对话策略的工程实践在2022年前后随ChatGPT的API开放而快速发展。OpenAI的Chat Completions API引入了`messages`数组结构，将对话历史以`[{"role": "user/assistant/system", "content": "..."}]`的格式传递，这一设计直接推动了多轮对话管理工程化的标准化。在此之前，GPT-3的Completion API只支持单文本拼接，开发者需要手动将对话历史拼接成连续字符串，极易引发格式混乱。

多轮对话策略的重要性体现在：任务型对话（如代码调试助手、多步骤表单填写）必须在第N轮中引用第1轮确立的用户意图；客服、教育等场景要求模型记住用户画像（如技术水平、偏好）并在后续轮次中调整回答风格。缺乏有效策略会导致模型在第5轮时"忘记"第1轮用户说"我是Python新手"，从而给出不适合的高级代码。

## 核心原理

### 对话历史的结构化表示

多轮对话的基础数据结构是消息列表（message list），每条消息包含`role`（角色）和`content`（内容）两个必填字段。OpenAI规范中`role`取值为`system`、`user`、`assistant`三种，其中`system`消息仅出现一次且置于列表首位，用于锚定整个对话的行为约束。多轮对话策略的第一个工程决策是**如何在每轮API调用时重新构建这个消息列表**——模型本身无状态，每次调用都需要将完整历史随请求传入。

典型的消息列表结构如下：

```
[
  {"role": "system", "content": "你是一名Python编程助手，用简洁语言回答初学者问题"},
  {"role": "user",   "content": "什么是列表推导式？"},
  {"role": "assistant", "content": "列表推导式是..."},
  {"role": "user",   "content": "能给我一个过滤偶数的例子吗？"}
]
```

第4条消息的回答质量高度依赖第2、3条提供的上下文——这是多轮对话区别于单轮的根本机制。

### 上下文窗口管理策略

当对话轮次增加，消息列表的token总量会线性增长，最终触及模型上下文限制。常用的三种管理策略各有适用场景：

**滑动窗口截断**：只保留最近N轮对话（如最近10轮），丢弃更早的消息。实现简单，但会导致早期确立的关键约束（如用户语言偏好）丢失。适合闲聊类场景。

**摘要压缩（Summarization）**：当历史超过阈值（通常设为目标窗口的70%）时，调用一次LLM将前K轮对话压缩为一段摘要，用单条`system`或`assistant`消息替换原始轮次。例如将前8轮压缩为"用户正在开发一个Flask Web应用，已完成路由设计，正在处理数据库连接问题"。这一方法可将8轮2000 tokens的历史压缩到约200 tokens。

**实体记忆提取**：从对话中抽取结构化的用户画像信息（姓名、偏好、当前任务状态），存入独立的`user_profile`字段，在每轮的`system` prompt中注入。例如：`用户背景：Python初学者，使用Windows 11，项目类型：数据分析`。这种方法token效率最高，但需要额外的信息抽取Prompt。

### 多轮对话Prompt模板设计

有效的多轮对话Prompt模板必须在`system`消息中解决三个问题：**任务边界**（模型在本次对话中能做什么）、**对话状态追踪指令**（如何处理指代消歧）、**拒绝策略**（当用户突然切换话题时如何响应）。

一个面向任务型对话的`system` prompt范例：

```
你是一名代码审查助手。在整个对话中：
1. 记住用户在第一轮提交的原始代码作为审查基准
2. 每次修改建议后，主动询问"是否需要解释这处修改？"
3. 若用户询问与代码审查无关的问题，回复"这超出了本次代码审查的范围"
4. 在第3轮后，如果用户说"完成"，输出最终的完整修改代码
```

这类模板将对话的**状态机逻辑**嵌入到system prompt中，使模型在每一轮都能根据对话阶段调整行为——这是多轮对话策略与单轮Prompt设计的核心差异。

## 实际应用

**代码调试助手**：用户在第1轮粘贴出错代码，模型识别并保存错误类型；第2轮用户提供错误日志，模型结合第1轮的代码上下文定位问题；第3轮用户确认修复方案。实现时需在`system` prompt中指定"始终引用用户最近一次提交的完整代码版本进行回答"，避免模型基于隐式假设生成与用户实际代码不匹配的解答。

**渐进式信息收集**：用于表单填写、需求分析等场景。Prompt模板设计为每轮只询问1-2个关键字段，并在system中维护已收集字段的列表。当所有必填字段收集完毕后，模型自动切换到"确认阶段"输出汇总信息。这种设计避免了单轮大量询问导致的用户流失率高问题（实验数据显示，将10个问题拆分为5轮2问的多轮对话，表单完成率可提升约35%）。

**教学辅导场景**：在`system` prompt中注入学生的已掌握知识列表，每轮回答后调用实体提取子Prompt更新该列表（如"已掌握：变量、循环；当前学习：函数"），下一轮的解释深度基于该列表动态调整。

## 常见误区

**误区一：在每轮对话中重复粘贴完整的背景信息**。部分开发者担心模型遗忘，在每条`user`消息中重新写入"我是Python初学者，我的项目是..."。这会造成token浪费（可能增加40%-60%的不必要消耗），且由于user消息中的重复信息权重高于system消息，可能干扰模型对真实对话意图的判断。正确做法是将固定背景信息置于`system`消息，仅在第1轮user消息中补充动态上下文。

**误区二：认为更长的上下文历史总是更好**。保留全部历史并不能保证对话质量，因为LLM存在"中间位置遗忘"现象（Lost in the Middle，2023年斯坦福研究）：当上下文超过一定长度后，模型对位于中间位置消息的记忆显著下降，而对首尾消息保持较好记忆。在实验中，16K token上下文中段的信息检索准确率可比首段低约20%。因此，关键约束信息应始终保持在system消息（首位）或最近的用户消息（末位），而非依赖模型从长历史中间自行检索。

**误区三：忽视对话状态的显式重置**。当用户在对话中途切换任务（如从"调试代码"切换到"解释概念"），若不在system prompt中定义状态重置规则，模型会将前一任务的上下文错误地带入新任务，例如在解释概念时仍试图"修复"根本不存在的代码错误。有效的多轮策略需要定义明确的**转场触发词**（如"换个话题"、"重新开始"）及对应的上下文清除或保留规则。

## 知识关联

多轮对话策略直接依赖**提示链（Prompt Chaining）**的概念：每一轮对话本质上是以上一轮的`assistant`输出作为隐式输入构成的链式调用，区别在于提示链通常由开发者编排固定流程，而多轮对话允许用户在每轮动态决定下一步方向。掌握提示链的条件分支设计（如"若上一步输出包含错误则走纠错分支"）是实现健壮多轮策略的前置能力。

**System Prompt设计**是多轮对话策略的直接上游技能：单轮场景的system prompt仅需定义角色和输出格式，而多轮场景的system prompt还需编码对话状态机逻辑、轮次感知指令（"在第2轮时..."）和跨轮引用规则。System prompt的长度与多轮管理的精细程度呈正相关，但过长的system prompt（超过2000 tokens）本身会占用宝贵的历史上下文空间，需要在设计阶段做出工程权衡。
