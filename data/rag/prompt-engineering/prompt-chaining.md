# 提示链

## 概述

提示链（Prompt Chaining）是一种将单个复杂任务拆解为多个顺序执行的提示请求，并将前一步的输出作为下一步输入的设计模式。与单一提示尝试一次性完成所有任务不同，提示链通过明确定义的"交接点"（handoff point）在步骤间传递结构化数据，使每个子提示只负责一个清晰界定的子任务。

提示链的概念在2022年前后随着LLM能力的快速提升而系统化。最早的正式描述出现在Anthropic内部文档及后来的Claude使用指南中，将其定义为"让模型在有限上下文窗口内处理超长或超复杂任务"的标准工程方案。相比思维链（CoT）是在单次请求内引导模型自己推理，提示链是工程师在系统层面显式地切割工作流。

提示链的工程价值在于：它将"模型能力边界"与"任务复杂度"解耦。一个需要提取信息、验证逻辑、生成草稿、审查草稿四个阶段的写作任务，若塞入单一提示，模型容易在某个阶段出错后将错误传播到最终输出；而提示链允许在每个节点插入校验逻辑，从根本上降低错误累积概率。

## 核心原理

### 链式数据流与状态传递

提示链的本质是一个**有向无环图（DAG）的线性或分支化子集**，其中每个节点是一次LLM调用，边是数据传递关系。标准线性链的结构可表示为：

```
输出ᵢ = LLM(提示模板ᵢ ∪ 输出ᵢ₋₁ ∪ 固定上下文)
```

其中 `输出ᵢ₋₁` 是上一步的结果，`固定上下文` 是链全局共享的背景信息（如用户原始需求）。关键工程决策在于**输出格式规范化**：步骤1必须输出结构化数据（通常为JSON或Markdown列表），使步骤2的提示模板能以可预测方式引用它。若步骤1输出自由文本，步骤2的提示将变得脆弱，极易因格式变化而失效。

### 门控条件与分支链

提示链不必是纯线性的。在实践中，**门控提示（Gate Prompt）**是关键模式：在链的某个节点，让模型输出一个布尔判断（如 `{"should_continue": true, "reason": "..."}`），由外部代码决定走哪条后续分支。例如，在法律文档审查链中：
- 步骤1：提取合同关键条款 → 输出结构化条款列表
- 步骤2（门控）：判断是否存在违规条款 → 输出 `{"violation_found": true/false}`
- 若 `true`：进入详细违规分析链；若 `false`：直接进入摘要生成链

这种模式将提示链从线性流程升级为条件工作流，是提示链与ReAct推理的主要区别——ReAct中模型自己决定下一步动作，而提示链中分支逻辑由外部代码控制，模型只负责信息提取与判断输出。

### 上下文窗口管理与压缩策略

提示链必须应对**上下文累积问题**：若每一步都将所有历史输出追加到下一步提示中，到第5步时上下文可能已超过模型的32k token限制。工程解决方案有三种：

1. **滚动摘要（Rolling Summary）**：在每步结束后追加一个"压缩提示"，将历史输出压缩为500字以内的结构化摘要，替代原始输出传入下一步；
2. **关键字段提取**：只从前步输出中提取当前步骤所需的特定字段，忽略其余内容；
3. **外部状态存储**：将中间结果存入外部数据库（如向量库或Redis），按需检索，使每个提示的token占用保持恒定。

LangChain框架中的 `SequentialChain` 类实现了方案2的自动化，通过 `input_variables` 和 `output_variables` 显式声明每步的输入输出字段，框架自动完成字段路由。

## 实际应用

**学术论文辅助写作**：一个典型的5步提示链包括：①解析用户研究主题，提取3-5个核心论点；②对每个论点分别搜索相关证据（可调用工具）；③合并证据，生成论文大纲JSON；④按大纲章节逐一生成草稿段落（每章一次LLM调用）；⑤整合所有章节，执行风格统一检查。整个链的设计中，步骤③的JSON大纲是核心"交接点"，其schema需要在工程设计阶段就明确定义，如 `{"sections": [{"title": str, "key_points": list[str], "evidence_ids": list[int]}]}`。

**客服工单自动处理**：阿里云客服系统采用的三步提示链：步骤1用意图分类提示将工单分入12个预定义类别；步骤2根据类别调用不同的解决方案提示，输出结构化处理方案；步骤3将处理方案翻译为自然语言客服回复。这种设计使步骤2可以针对不同类别使用完全不同的提示模板和模型（如退款类用GPT-4o，查询类用GPT-3.5-turbo），在精度与成本间灵活权衡。

**代码安全审查**：安全审查提示链通常将代码按函数拆分（避免单次超过8k token），对每个函数执行漏洞扫描提示，再用聚合提示将所有扫描结果合并为按严重程度排序的安全报告，最后生成修复建议。这比单一提示处理整个代码文件的误报率降低约40%（根据GitHub Copilot内部测试数据）。

## 常见误区

**误区一：步骤越多越好**。部分工程师将提示链与任务质量正相关，导致将可以在2步完成的任务拆成7步。每增加一次LLM调用，延迟增加200-800ms，成本线性增加，且每步都有引入格式错误的风险。正确原则是：仅在**单步提示的可靠性低于80%**，或**输出需要条件分支**，或**上下文超过窗口限制**时才增加新步骤。

**误区二：将提示链等同于ReAct或Agent**。提示链是**静态工作流**——链的拓扑结构由工程师在设计时确定，模型不能在运行时修改链的结构。ReAct中模型可以动态决定调用哪个工具、循环几次；Agent可以完全改变执行路径。混淆这两者会导致误用：对于需要动态规划的任务（如自主调研），用提示链会因固定步骤而遗漏关键信息；对于流程固定的任务（如文档格式转换），用Agent会引入不必要的不确定性。

**误区三：忽略步骤间的错误传播**。若步骤1的输出包含错误，步骤2不仅会继承错误，还可能因"模型幻觉修正"而掩盖错误，使最终输出看起来正确但实质错误。工程上的对策是在每个步骤后添加**验证子提示**（Validation Sub-prompt），要求模型对自己的输出进行自检，输出 `{"valid": bool, "issues": list[str]}`，并在 `valid=false` 时触发重试或人工审核，而非盲目传入下一步。

## 知识关联

**从思维链（CoT）到提示链**：CoT是在单次请求内让模型自己展示中间推理步骤，工程师无法在中间步骤插入外部逻辑或数据；提示链则将CoT的"中间步骤"外化为独立的LLM调用，允许在步骤间执行Python代码、数据库查询或API调用。可以将提示链理解为"工程师可编程控制的显式CoT"。

**从ReAct到提示链**：ReAct通过"思考-行动-观察"循环让模型动态规划，适合开放式任务；提示链的控制流完全由代码决定，适合步骤确定性高的任务。在实践中，提示链的单个步骤内部可以使用ReAct模式——即链的某一节点本身是一个ReAct循环。

**向Agent工作流编排延伸**：掌握提示链后，Agent工作流编排的核心新概念是**动态图生成**——Agent不仅执行预定义的提示链，还能在运行时根据观察结果生成并修改链的拓扑结构。LangGraph框架中的 `StateGraph` 是提示链 `SequentialChain` 的动态化版本，两者共享数据流传递的设计思想，但前者支持节点动态添加与循环边。多轮对话策略则需要解决提示链如何跨越多个用户对话回合保持状态，涉及链状态的持久化序列化设计。
