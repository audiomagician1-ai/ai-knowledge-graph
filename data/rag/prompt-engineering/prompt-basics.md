# 提示词基础

## 概述

提示词（Prompt）是用户或系统传递给大型语言模型的输入文本，模型依据该输入生成相应的补全（Completion）。从技术层面看，提示词并非"指令"的抽象表述，而是直接进入模型 Token 序列的具体字符串——模型在推理时将提示词 Token 化后拼接到上下文窗口的起始位置，随后以自回归方式逐 Token 预测后续内容。

提示词工程作为独立领域的兴起可追溯至 2020 年 GPT-3 的发布。GPT-3 论文（Brown et al., 2020）首次系统性地展示了通过在提示词中嵌入少量示例（few-shot examples）即可激活模型潜在能力的现象，研究者由此意识到"如何输入"与"模型参数"同等重要。2022 年 ChatGPT 普及后，提示词工程从学术实验演变为工业界的日常开发实践。

理解提示词基础的实用价值在于：同一个 GPT-4 模型，措辞不同的提示词可以使任务准确率相差 30% 以上（根据 OpenAI 内部基准测试数据）。掌握提示词的结构和工作机制，是在不修改模型权重的前提下获得可靠输出的最低成本路径。

## 核心原理

### 提示词的组成结构

一条完整的提示词通常包含四个可选层次：**指令（Instruction）**、**上下文（Context）**、**输入数据（Input Data）**、**输出格式要求（Output Indicator）**。例如：

```
[指令] 将以下客服工单分类为"退款"、"投诉"或"咨询"之一。
[上下文] 分类结果将用于自动路由系统，需保持一致性。
[输入] 工单内容：我的订单三天了还没发货，请问什么时候能收到？
[输出格式] 只输出分类标签，不要解释。
```

四层并非必须同时出现，但缺少"输出格式要求"时模型倾向于生成冗余解释，缺少"上下文"时模型只能依赖参数化知识填补语义空白，容易产生歧义。

### Token 化与上下文窗口的关系

提示词在进入模型前经过 BPE（Byte Pair Encoding）或 SentencePiece 算法转化为 Token 序列。以 GPT-4 为例，1 个英文单词平均对应约 1.3 个 Token，1 个汉字对应约 0.7 个 Token（即汉字 Token 效率更高）。模型的上下文窗口（如 GPT-4 Turbo 的 128,000 Token）同时容纳提示词与生成内容，因此提示词越长，可用于生成的空间越小，且超出窗口的内容会被截断而非报错，这是提示词设计中必须提前规划的硬性约束。

关键公式：
> **可用生成 Token 数 = 上下文窗口上限 − 提示词 Token 数 − 系统消息 Token 数**

当该值接近 0 时，模型会输出不完整内容，因此长文档处理时必须估算提示词 Token 数。

### 提示词的语义锚定机制

模型对提示词的响应受到**位置偏置（Position Bias）**影响：实验研究（Liu et al., 2023，"Lost in the Middle"论文）表明，GPT 类模型对提示词开头和结尾的信息赋予更高注意力权重，中间部分的关键信息容易被"遗漏"。这一现象源于 Transformer 的自注意力机制在长序列中对位置编码的衰减效应。

实践含义：如果提示词中包含关键约束（如"不得输出任何个人信息"），应将其放在提示词**最前方或最后方**，而非夹在大段背景描述的中间。

### 温度参数与提示词的协同作用

提示词本身不控制模型的随机性，但提示词内容与采样温度（Temperature）存在协同效应。Temperature=0 时模型每次输出相同结果（贪心解码），适合分类、提取等确定性任务；Temperature=0.7～1.0 适合创意写作。如果提示词要求"给出三种不同方案"，但 Temperature=0，模型可能输出三条高度雷同的内容，此时需要同时调整采样参数而非仅修改提示词措辞。

## 实际应用

**任务：从非结构化文本中提取日期**

低效提示词：
```
提取文中的日期。
```

高效提示词：
```
从以下文本中提取所有日期，以 YYYY-MM-DD 格式输出，用逗号分隔。
如果没有日期，输出"无"。文本如下：
###
张总于三月十五日签署合同，预计2024年9月交付。
###
```

两版提示词的差异在于：第一版缺少格式约束，模型可能输出"三月十五日"和"2024年9月"两种混合格式，下游代码解析时会出错；第二版通过 `###` 分隔符明确划定数据边界，同时规定输出格式，使提取结果可直接被程序消费。

在客服机器人场景中，提示词需要将业务约束（只回答与产品相关的问题）、角色设定（友好的客服人员）和输出要求（不超过 100 字）同时编码进提示词，这三类信息缺一不可，且通常由 System Prompt 承载，User Prompt 只传递用户的实时输入。

## 常见误区

**误区一：提示词越长越好**
增加提示词长度确实能提供更多约束信息，但存在两个反效果：一是位置偏置导致中间信息被忽略（见上文"Lost in the Middle"研究）；二是消耗上下文窗口空间，压缩模型可输出的内容长度。有效的提示词追求"精确"而非"冗长"，每句话都应对模型行为产生可量化的影响。

**误区二：提示词指令等同于硬性代码逻辑**
提示词对模型行为的影响是概率性的，而非确定性的。"只输出 JSON，不输出任何其他内容"这条指令在 99% 的情况下有效，但仍有小概率输出前缀文字（如"以下是JSON："）。这意味着依赖提示词做格式控制时，下游代码必须配合容错解析逻辑，不能假设模型 100% 遵守格式要求。解决方案是使用结构化输出 API（如 OpenAI 的 `response_format: json_object`）在采样层面强制约束，而非仅靠提示词。

**误区三：同一提示词在所有模型上效果相同**
针对 GPT-4 优化的提示词迁移到 Claude 3 或 Llama 3 时，可能因训练数据差异、指令微调方式不同而表现迥异。例如，GPT-4 对 Markdown 格式（如 `**加粗**`、`###标题`）响应良好，而部分开源模型在 Markdown 指令下会输出符号本身而非格式化文本。提示词需要针对目标模型进行独立测试，不能跨模型复用而不验证。

## 知识关联

**前置知识衔接**：理解提示词为何有效，需要 GPT 与解码器模型的机制作为基础——解码器模型以自回归方式生成 Token，意味着提示词中的每个词都参与了后续所有 Token 的注意力计算，这解释了为什么提示词措辞的细微变化会影响整个输出分布。

**延伸方向**：掌握提示词基础后，**零样本提示**（Zero-shot Prompting）探讨在不提供任何示例的情况下如何通过角色设定和任务描述激发模型能力；**少样本提示**（Few-shot Prompting）研究如何在提示词中嵌入 2～8 个示例来引导输出格式和风格；**System Prompt 设计**则专注于 Chat 模型中系统消息的权限层级和越权防护；**提示注入攻防**处理恶意用户通过 User Prompt 覆盖 System Prompt 约束的安全问题——这些都建立在本文所述的结构、Token 机制和概率特性之上。
