# 提示词基础

## 概述

提示词（Prompt）是用户或系统传递给大型语言模型的输入文本，它直接决定了模型从其参数空间中激活哪一部分知识并生成何种输出。本质上，提示词充当了一种"软性程序指令"：与传统编程中的硬编码逻辑不同，提示词通过自然语言引导模型在高维概率空间中进行条件采样。GPT系列模型的解码过程是基于"给定前文Token序列，预测下一个Token"的自回归机制，而提示词正是这个条件分布的起始条件——提示词的每一个词语选择都会在模型的注意力层中留下不同的激活权重。

提示工程（Prompt Engineering）作为一个正式研究领域，大约在2020年GPT-3发布后开始系统化。OpenAI研究人员在2020年的论文《Language Models are Few-Shot Learners》中首次展示了通过精心设计的文本示例来"引导"175B参数模型完成多种任务的可行性，这标志着提示词从临时技巧演变为工程学科。理解提示词基础不仅对调用ChatGPT这类产品有用，更是构建任何基于LLM的应用系统——无论是RAG管道、智能体还是代码生成工具——的前提性技能。

## 核心原理

### 提示词的结构组成

一个完整的提示词通常由以下几个功能性部分构成：**指令（Instruction）**、**上下文（Context）**、**输入数据（Input Data）**和**输出格式说明（Output Indicator）**。例如，要让模型对商品评论进行情感分析，一个结构完整的提示词可以写作：

```
你是一名情感分析专家。【指令】
以下是一条用户评论：【上下文标记】
"这款耳机音质很差，退货！"【输入数据】
请输出：正面/负面/中性，以及置信度（0-1）。【输出格式说明】
```

缺少输出格式说明时，模型可能生成一段自由叙述；加入该部分后，模型输出会向结构化方向收敛。这四个组成部分并非每个提示词都必须全部具备，但意识到每个部分的功能有助于系统性地诊断和改进提示词质量。

### Token化与长度限制的实际影响

模型处理的并非原始字符，而是经过Tokenizer分词后的Token序列。以GPT-4为例，其上下文窗口为128,000 Tokens，而中文字符平均约1.5-2个字符对应1个Token（英文单词平均约1.3个Token）。这意味着，同样的语义内容用中文表达会消耗更多Token预算。提示词设计者需要在以下两者之间做权衡：**提示词的详细程度**（越详细通常效果越好）与**留给模型生成回复的Token空间**（超出上下文窗口会导致截断）。实践中，将系统指令控制在500 Tokens以内、为输出预留至少1000 Tokens，是常见的工程经验值。

### 温度参数与提示词的协同关系

提示词本身定义了"希望模型走哪条路"，而Temperature参数（范围0到2）则控制了"模型走路时的随机程度"。当Temperature=0时，模型每次都选择概率最高的下一个Token，输出具有确定性——此时提示词的措辞哪怕有细微差异也会产生不同结果，因为模型没有随机性来"弥补"模糊的指令。当Temperature=1时，模型按照训练得到的概率分布采样，有一定创造性但可控；Temperature>1.5时输出变得高度随机，即使写得再精准的提示词也难以稳定输出。因此，精确的任务性提示词（如信息抽取）应配合低Temperature（0~0.3），而创意类提示词可以使用0.7~1.0的范围。

### 提示词的语义密度原则

模型对提示词中的"信息密度"非常敏感。表述"帮我写一篇文章"与"用800字，以第一人称，面向高中生，解释量子纠缠的基本原理，避免使用数学公式"所激活的生成空间截然不同。后者通过**字数约束**、**叙述视角**、**目标读者**、**主题范围**、**内容限制**五个维度显著压缩了模型的输出分布，使其更可预测且符合需求。每增加一个有效约束维度，通常能将输出符合预期的概率提高10%~30%（这是来自实践的经验估计，非严格实验数据）。

## 实际应用

**代码调试场景**：相比"帮我修复这段Python代码"，更有效的提示词写法是"以下Python 3.10代码在调用pandas 1.5.3时报TypeError: cannot concatenate object of type '<class 'str'>。请定位错误行，解释原因，并提供修复后的完整代码"。明确指定语言版本和库版本是提示词基础的实际运用，能让模型规避版本差异导致的错误建议。

**内容生成场景**：某电商公司需要批量生成商品描述，直接使用"写一段商品描述"往往需要大量人工修改。加入角色定义（"你是一名擅长转化率优化的文案专家"）、格式约束（"不超过80字，包含至少两个动词"）和品牌语气（"活泼、年轻化，避免使用'品质'这类老套词汇"）之后，输出的可用率从约30%提升至70%以上。

**多轮对话场景**：在ChatGPT等对话式系统中，每一轮用户输入都是一个新的提示词片段。理解这一点意味着：在第1轮确立清晰的任务框架（如"接下来我们将进行一个角色扮演，你扮演一名苏格拉底式的哲学导师……"），比在第5轮才纠正模型行为效率更高，因为早期设立的上下文在整个对话中持续作为"条件"影响后续生成。

## 常见误区

**误区一：提示词越长越好**。许多初学者认为给模型提供大量背景信息总是有益的。实际上，无关信息会引入"干扰Token"，使模型的注意力分散到不重要的内容上。OpenAI的研究表明，在某些任务中，冗余的上下文会使准确率下降5%~15%。有效的提示词应遵循"每个句子都在做贡献"的原则，删除任何不影响输出质量的内容。

**误区二：对模型使用礼貌用语能提升输出质量**。"请你帮我"、"谢谢你"这类社交性语言对模型的输出质量没有系统性影响。模型没有情感偏好，它只识别语义信息和统计规律。真正影响输出质量的是指令的**具体性**（specificity）和**约束的完整性**（constraint completeness），而非语气的礼貌程度。

**误区三：同一个提示词对所有模型通用**。为GPT-4o设计的提示词往往在Claude 3.5或Gemini 1.5上表现下降，因为不同模型的预训练数据分布、指令微调（Instruction Fine-tuning）的数据风格和RLHF偏好各不相同。例如，GPT-4系列对"Step by step"这类CoT触发词响应良好，而某些开源模型（如LLaMA-3）可能需要更明确的格式标记（如`###`分隔符）才能稳定触发结构化输出。提示词需要针对目标模型进行调试，而非视为通用资产。

## 知识关联

理解提示词基础需要以**GPT与解码器模型**的工作原理为前提：正是因为Transformer解码器通过自回归方式处理Token序列，我们才能理解为什么提示词中词语的顺序和措辞会影响模型注意力权重的分配，进而影响输出结果。不掌握Token化机制，就无法做出合理的Token预算决策。

在提示词基础之上，**零样本提示**（Zero-shot Prompting）和**少样本提示**（Few-shot Prompting）进一步探索了如何通过不提供示例或提供少量示例来引导模型完成新任务。**结构化输出（JSON Mode）**研究如何通过提示词与模型API参数的配合确保输出可被程序解析。**System Prompt设计**则专注于在多轮对话系统中如何通过系统级提示词建立持久的角色约束和行为边界。**提示注入攻防**则研究恶意输入如何绕过提示词中设定的安全指令——这是提示词基础知识在安全工程维度的延伸。
