# Prompt模板设计模式

## 概述

Prompt模板设计模式是一套将自然语言指令结构化、可复用的工程化方法，通过固定的骨架框架与可变的内容插槽组合，使同一类任务能够以一致的方式调用大语言模型。与即兴编写的单次Prompt不同，模板设计模式强调**可预测性**：给定相同结构的输入，模型输出的格式、风格和质量应当稳定可控。

这一实践兴起于2020年GPT-3发布后的社区探索期，工程师们逐渐发现，将成功的Prompt抽象成参数化模板（例如LangChain在2022年引入的`PromptTemplate`类）可以大幅降低重复开发成本。OpenAI的系统提示（System Prompt）机制进一步强化了模板化思路——将角色、约束、格式三类指令分层嵌入，成为当前主流的工程实践范式。

Prompt模板设计模式之所以重要，在于它直接影响**幻觉率、拒绝率和输出解析成功率**三个可量化指标。实验数据表明，使用结构化约束框架相比自由描述提示，JSON解析成功率可从约60%提升至95%以上，这使得模板设计成为生产环境中不可省略的工程步骤。

## 核心原理

### 一、角色扮演模式（Role Prompting）

角色扮演模式通过在提示开头声明一个明确身份，将模型的输出风格、知识边界和回应态度锚定到特定人格上。其基本结构为：

```
你是一名[角色描述]，拥有[专业背景]，你的工作是[核心职责]。
```

角色声明的粒度直接决定效果：`你是一名医生` 与 `你是一名专注儿科感染病学、具有10年临床经验的三甲医院主治医师` 会产生显著不同的回答深度和措辞倾向。Stanford 2023年的研究"Large Language Models as Simulated Social Agents"证实，具体角色描述可使专业领域问答的准确性提升约18%。

角色模式还有一个关键技巧：**角色与反角色的对立设置**。在需要模型评估方案优劣时，可同时设置"支持者"与"批判者"两个角色，让模型分别扮演，从而在单次对话中获得多视角分析。

### 二、约束框架模式（Constraint Framing）

约束框架模式通过显式列举"必须做"（Positive Constraints）与"禁止做"（Negative Constraints）来压缩模型的输出搜索空间。一个完整的约束框架通常包含以下四个维度：

| 约束类型 | 示例 | 作用 |
|---|---|---|
| 长度约束 | "回答不超过150字" | 控制输出量 |
| 内容约束 | "不得提及竞争对手品牌名称" | 合规过滤 |
| 推理约束 | "先列出证据再给出结论" | 引导思维链 |
| 受众约束 | "假设读者是没有编程经验的产品经理" | 调节措辞难度 |

约束顺序同样重要：实验表明将**最重要的约束放在提示末尾**（接近生成起点）比放在开头可使遵守率提高约12%，这与Transformer注意力机制对近端token的天然偏重有关。负面约束（"不要做X"）单独使用效果不稳定，通常需配合正面约束（"应当做Y"）才能可靠生效。

### 三、输出格式化模式（Output Formatting）

输出格式化模式通过在提示中嵌入目标格式的**示例骨架**或**Schema定义**，强制模型按指定结构生成内容。最常见的三种实现方式：

**方式A：直接示例法**——在提示中给出空白格式模板：
```
请按以下格式回答：
【摘要】（1句话）
【详细说明】（3-5个要点）
【注意事项】（如有）
```

**方式B：JSON Schema声明法**——提供字段名与类型：
```
以JSON格式输出，结构为：
{"title": string, "score": integer (1-10), "reasons": array of strings}
```

**方式C：Markdown锚点法**——利用特殊标记引导模型分段，如 `## 分析` / `## 建议`，方便下游代码按标题切割文本。

三种方式的解析可靠性排序为：JSON Schema法 > 直接示例法 > Markdown锚点法，但Markdown锚点法在长文生成场景下的阅读友好度最高。

## 实际应用

**客服自动化场景**：电商平台使用如下三层组合模板——角色层定义"你是X品牌的售后专员"，约束层规定"不得承诺7天以外的退款时效，遇到复杂投诉需转人工"，格式层要求输出包含`【解决方案】`和`【后续步骤】`两个字段。这种组合在实际部署中将人工升级率从35%降至18%。

**代码审查场景**：使用角色模式（资深后端工程师）+ 约束框架（必须指出性能问题、安全漏洞，可选指出风格问题）+ JSON格式输出（包含`severity`字段，取值`critical/major/minor`），使下游CI/CD流水线可以直接按严重程度自动分类审查意见，无需人工解析。

**报告生成场景**：将输出格式化模式与少样本提示结合——先给出2个已完成的报告片段作为少样本示例，再追加空白模板要求填充。这种"少样本+模板"的叠加使用，比单独使用任一方法的格式符合率高出约23个百分点。

## 常见误区

**误区一：角色越详细越好**。过度堆砌角色背景（超过150词的角色描述）会导致模型在角色细节与实际任务指令之间产生注意力竞争，反而降低任务执行质量。经验规则是角色描述控制在30-60词，聚焦于与任务直接相关的专业属性。

**误区二：格式约束可以100%保证输出可解析**。即使使用JSON Schema声明，在GPT-3.5等较早的模型上，复杂嵌套结构的解析失败率仍可达15-20%。正确做法是在代码层增加`try-except`解析容错，并设置自动重试逻辑（携带"你上次的输出格式不正确，请修正"的反馈提示），而非假设模板能消除所有格式错误。

**误区三：三种模式必须同时使用**。在简单的单轮问答任务中，叠加全部三种模式会使提示词长度膨胀，消耗不必要的token并增加模型理解负担。正确做法是根据任务需求**选择性组合**：仅需风格控制时用角色模式；仅需结构化输出时用格式化模式；合规敏感场景才强制加入约束框架。

## 知识关联

Prompt模板设计模式以**提示词基础**中的指令清晰性原则为前提——如果无法写出一个有效的单条指令，套用模板框架也无法改善效果，模板只是放大了指令的清晰度，而非替代它。从**少样本提示**迁移到模板模式时，关键转变是将少样本示例从内容本身抽离出来，提炼为可替换内容的格式骨架，即"示例教会模型格式，模板固化格式复用"。

在工具链层面，LangChain的`ChatPromptTemplate`、OpenAI的Function Calling（强制JSON输出的格式化机制）以及Anthropic的XML标签约定，都是对上述三种设计模式的工程化实现，学习这些工具时可对照本文的三类模式框架进行理解和映射。
