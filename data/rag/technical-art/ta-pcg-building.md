# 程序化建筑

## 概述

程序化建筑（Procedural Architecture）是指通过预定义的规则、语法或约束系统，自动生成建筑几何结构与细节的技术方法。与手动建模不同，程序化建筑将建筑设计知识编码为可执行的算法逻辑，使得一栋建筑的外墙开窗、楼层叠加、屋顶形状等细节均可在运行时或离线工具链中按需生成。其核心产物不是单一建筑模型，而是能够产出大量风格一致、细节各异建筑的"建筑生成器"。

该领域的里程碑工作是 2001 年 Pascal Müller 等人提出的 **CGA Shape Grammar**（计算机生成建筑形状语法），该语法后来成为 Esri CityEngine 的核心引擎。CGA 首次系统性地将建筑学中的"形式语言"转化为可由计算机解析的生产式规则，使程序化建筑从学术概念走向了工业工具。在游戏领域，*Watch Dogs 2* 和 *Assassin's Creed: Origins* 的城市区块均大量依赖程序化建筑流程生成底层建筑存量，再由美术手工精修地标建筑。

程序化建筑的重要性体现在它打破了"建筑数量与美术人力成正比"的硬性约束。一座开放世界城市可能需要数千栋外观各异的楼宇，若全部手建则成本不可承受；而通过程序化方法，少量模块素材加上一套规则系统，便可覆盖城市密度的绝大部分需求，美术精力得以集中在视觉焦点建筑上。

## 核心原理

### 形状语法（Shape Grammar）

形状语法是程序化建筑最主要的理论基础。其本质是一套**产生式规则系统**，格式为：

```
前驱形状  -->  后继形状序列 + 操作符
```

以 CGA 语法为例，一条典型规则写作：

```
Facade --> splitY { ~1 : Floor }* { ~0.5 : Roof }
```

这条规则将名为 `Facade`（立面）的形状，沿 Y 轴按相对比例切分为若干 `Floor`（楼层）和顶部 `Roof`（屋顶）子形状。规则递归展开，直到达到终止符或最小尺寸阈值（通常设为 0.3–0.5 米）。整个建筑的生成过程是一棵**派生树（Derivation Tree）**的展开过程，叶节点对应最终可渲染的几何体。

形状语法的关键优势在于**参数化**：同一套规则在不同随机种子（seed）下可输出外观差异显著的建筑，但所有输出都服从相同的风格约束，视觉一致性得以保证。

### 约束求解（Constraint Solving）

当建筑生成需要满足多个相互竞争的约束时——例如"窗户间距不小于 0.8 米"、"每层必须有偶数扇窗"、"立面总宽度为任意值"——纯语法方法难以优雅处理，此时引入**约束求解器**。

常见做法是将建筑划分问题建模为**约束满足问题（CSP）**，变量为各模块的位置与类型，约束为建筑规范与美学规则。求解器（如基于弧一致性 AC-3 算法）搜索满足所有约束的合法配置。Houdini 中的 PDG 流程和 SpeedTree 类似的建筑插件均包含此类约束传播机制。

此外，**波函数坍缩（Wave Function Collapse, WFC）** 近年被广泛用于建筑立面的模块拼接：将手绘立面样例分解为 N×M 瓦片，提取相邻约束，再用 WFC 在任意大小的立面网格上无缝扩展，保证局部模式符合人工样例的风格特征。

### 模块化分层结构

工业级程序化建筑系统通常采用三层分解结构：

1. **宏观体量层（Massing）**：建筑的整体轮廓，由地块边界、容积率、建筑退线等城市规划参数驱动，输出长方体或 L 形等基本体块。
2. **立面细分层（Facade Subdivision）**：在体量上应用形状语法，生成楼层、开间、阳台、窗洞等中尺度元素，此层决定建筑风格（如新古典、现代简约）。
3. **构件装饰层（Component Decoration）**：在开窗位、墙角、檐口等位置插入预制几何模块（窗框、空调机位、排水管等），提供贴近地面观察时的细节丰富度。

三层独立参数化，可分别替换以快速切换建筑风格，而无需重构整个流程。

## 实际应用

**CityEngine 城市街区生成**：导入 GIS 地块数据后，为每类地块（住宅、商业、工业）编写对应的 CGA 规则文件（`.cga`），CityEngine 自动在数千个地块上并行执行规则，生成覆盖整个城市的建筑群。输出可导出为 FBX 或 USD 格式，直接进入游戏引擎。《荒野大镖客：救赎2》的部分小镇背景建筑即采用类似的 GIS 驱动流程。

**Houdini 程序化建筑流程**：在 Houdini 中，建筑生成典型节点链为：`Scatter`（散点）→ `Divide`（地块切分）→ 自定义 SOP 网络（体量生成）→ `Copy to Points`（模块替换）→ `Labs Procedural Facade`（立面细分）。利用 Houdini 的属性（Attribute）系统，可以将地块面积、朝向等数据直接驱动规则参数，实现地理位置感知的建筑变化。

**实时生成与 LOD 配合**：在 UE5 的 PCG（Procedural Content Generation）框架中，程序化建筑可与 Nanite 和 HLOD 系统联动：近景建筑展示完整模块细节（构件装饰层），中景自动降级为仅显示立面面片，远景合并为单个静态网格体，全程无需手动制作多级 LOD。

## 常见误区

**误区一：规则越复杂，生成结果越真实**

初学者往往倾向于在形状语法中堆砌大量嵌套规则，以为规则数量等同于细节质量。实际上，CGA 语法的规则集超过 200 条后，维护成本急剧上升，且调试难度呈指数增长。工业案例表明，控制在 40–80 条精心设计的规则内，配合高质量模块资产，往往比 300 条规则的"全自动"系统产出更稳定的视觉效果。真正决定质量的是模块素材的精度和规则对风格的提炼深度。

**误区二：程序化建筑可以完全替代手工建模**

程序化建筑擅长生成"城市背景噪声"——大量中等质量、视觉可信的普通建筑。但对于具有叙事意义的地标建筑（如游戏主线据点、BOSS 战场景），程序化规则很难编码出"刻意设计感"，仍需手工建模。正确的工作流是：程序化工具负责填充 80% 的城市体量，美术团队聚焦于剩余 20% 的视觉焦点资产。

**误区三：WFC 拼接结果总是风格一致的**

波函数坍缩算法仅保证**相邻瓦片兼容性**，但不保证全局语义连贯性。在立面应用中，若样例瓦片集设计不当（例如同时包含哥特式尖窗和现代铝合金窗框的瓦片），WFC 可能生成局部兼容但整体风格混乱的立面。解决方案是为不同建筑风格维护独立的瓦片集，或在 WFC 之前增加一个风格选择的约束预处理步骤。

## 知识关联

**前置概念——程序化生成概述**：程序化建筑依赖程序化生成的基本思想，即用算法替代手工，但建筑领域对"风格一致性"和"建筑可读性"有比地形或植被更高的语义要求，因此引入了形状语法和约束求解这两种在通用程序化生成中不那么突出的机制。

**后续概念——程序化关卡**：程序化建筑通常生成单体建筑的外壳几何，而程序化关卡进一步关注建筑**内部空间**的房间划分、走廊连通性和功能分区逻辑。建筑外壳的门窗位置数据（通常以属性点云形式输出）可直接作为程序化关卡系统的入口/出口约束输入，两者在数据层面存在明确的上下游关系。
