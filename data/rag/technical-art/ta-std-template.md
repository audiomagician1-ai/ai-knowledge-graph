# 标准模板资产

## 概述

标准模板资产（Standard Template Asset）是技术美术部门制作并维护的一套预设资产文件，用于为美术团队提供符合项目规范的参考范本。这类资产通常以工程文件形式存在，例如 Maya 的 `.ma` 场景文件、Substance Painter 的 `.spp` 模板、Photoshop 的 `.psd` 分层模板，或引擎内的蓝图/材质球实例，其内部已预先配置好命名规则、层级结构、UV 通道分配和材质槽位。

标准模板资产的实践起源于工业级游戏制作流程的规模化需求。当一个项目美术团队超过 10 人时，若没有可直接打开并在其基础上创作的模板文件，每位美术人员都需要手动搭建符合规范的初始环境，不仅耗时且错误率极高。早期 AAA 工作室（如 Naughty Dog、CDPR）在其制作后期复盘中均指出，缺乏早期模板资产导致大量返工。

模板资产的核心价值在于将《审查检查清单》中的抽象条文转化为具体可操作的文件状态。美术师不需要背诵"模型顶点数不超过 65,536"或"Diffuse 贴图必须为 2 的幂次方且不超过 4096×4096"等规定，打开模板文件时这些约束已经内嵌在资产结构中，大幅降低新成员的认知负担。

## 核心原理

### 模板资产的分类体系

标准模板资产按资产类型通常分为四类：**几何体模板**（Mesh Template）、**材质模板**（Material Template）、**贴图模板**（Texture Template）和**绑定模板**（Rig Template）。每类模板解决各自特定的规范传递问题。几何体模板内嵌正确的 LOD 层级（通常预设 LOD0 至 LOD3，面数衰减比例约为每级 50%）；材质模板预连接好正确的着色器节点并锁定不允许美术修改的参数；贴图模板以 Photoshop 图层组的形式约定好通道打包规则，例如将金属度（R 通道）、粗糙度（G 通道）、AO（B 通道）合并为一张 ORM 贴图。

### 命名与目录结构的内嵌方式

模板资产不仅是内容范本，也是**命名规范的物理载体**。一个标准几何体模板文件本身的文件名应遵循 `SM_[AssetType]_Template_v01.ma` 格式，打开后内部对象的命名应包含正确的前缀占位符，如 `SM_RENAME_geo`、`SM_RENAME_LOD0`。文件存储在项目的 `/Art/Templates/` 目录下，子目录按资产类别划分，路径本身也是规范示范。通过这种方式，美术师在重命名资产时会自动参照已有的命名格式，而非凭记忆填写。

### 版本控制与模板的有效期管理

标准模板资产必须纳入版本控制系统（如 Perforce 或 Git LFS），并在文件名中标注主版本号。当项目技术指标变化时——例如目标平台从 PS4 升级至 PS5，导致显存预算从 2GB 提升至 8GB，对应的最大贴图分辨率上限可能随之调整——模板资产需要同步更新，旧版本以 `_deprecated` 标记但不删除，以便对照历史资产的规格差异。技术美术应在每次模板更新后于内部 Wiki 或 Confluence 页面记录变更日志（Changelog），注明变更原因和生效日期。

### 验证钩子与自动化检查

高成熟度的模板资产流程会在模板文件中内嵌**自动化验证脚本**。例如在 Maya 中，模板场景的 `scriptNode` 节点会在文件打开时自动执行一段 Python 脚本，检测当前场景的多边形数量是否超过预设阈值，并在状态栏打印绿色或红色的提示信息。这种"打开即验证"的机制将《审查检查清单》的执行时机从提交前移至创作初期，形成实时反馈闭环。

## 实际应用

**场景一：角色道具模板**
一个写实向 RPG 项目的手持武器模板文件 `SM_Weapon_Template_v02.ma` 中，预设骨架挂点名称为 `weapon_attach_jnt`，LOD0 面数上限标注在 `notes` 节点注释中（"Max: 3000 tris for one-handed weapon"），UV 已展开至第二 UV 通道（Lightmap UV），UV 壳间距为 2 像素（基于 2048 光照贴图分辨率）。美术师用此模板创建新武器时，只需保留结构，替换几何体，所有规范要素已预置到位。

**场景二：UI 贴图模板**
UI 部门的 Photoshop 模板 `UI_Icon_Template_64x64.psd` 内包含四个图层组：`Artwork`（实际绘制区域，限制为 56×56 像素安全区）、`Bleed`（出血区标注）、`Grid`（对齐辅助线，不可见）和 `Export_Settings`（标注导出格式为 `PNG-24` 且禁用透明度抖动）。新 UI 美术人员接到任务后直接从此模板开始创作，不需要询问任何贴图规格。

**场景三：虚幻引擎材质模板**
在 UE5 项目中，技术美术创建一个名为 `M_Template_PBR_Opaque` 的父材质，并将所有可调参数（BaseColor、Roughness、Metallic、Normal 强度）以 `MaterialParameterCollection` 的方式暴露给子材质实例，同时在材质编辑器备注中用红色框标出不允许子材质覆盖的逻辑节点（如像素深度偏移计算）。美术师只能创建子材质实例，无法修改底层着色逻辑。

## 常见误区

**误区一：将模板资产与参考图混淆**
参考图（Reference Sheet）展示的是资产的目标外观，而模板资产是可直接在 DCC 软件或引擎中打开的功能性文件，内部结构本身就是规范的实例。美术师不能仅凭参考图"还原"规范，因为参考图无法传达命名、节点连接顺序、UV 通道分配等不可视化的技术细节。标准模板资产必须是可操作的工程文件，而非截图或 PDF。

**误区二：模板资产一旦发布就不再更新**
部分团队将模板视为"一次性文档"，在项目初期发布后便不再维护。这会导致模板与实际引擎渲染管线脱轨。例如，若项目中途将光照方案从 Baked Lightmap 切换至 Lumen 动态全局光照，原模板中的第二 UV 通道（Lightmap UV）配置便成为误导性信息。模板资产的维护周期应与项目技术里程碑对齐，每个技术方案变更节点都应触发模板复审流程。

**误区三：所有资产类型共用一份模板**
仅创建一个通用模板并要求所有美术类型参照，会导致模板无法有效约束任何特定资产的关键指标。建筑场景模型、角色皮肤、特效粒子贴图、UI 图集对多边形数量、贴图通道打包、UV 精度的要求截然不同，必须各自维护独立的专用模板。一个完整的标准模板资产库在中等规模项目中通常包含 8 至 15 个独立模板文件，分类覆盖主要资产类型。

## 知识关联

标准模板资产直接依赖《**审查检查清单**》作为内容输入：检查清单中列出的每一条量化指标（顶点数上限、贴图分辨率、命名前缀、LOD 数量）都需要对应地内嵌至模板文件的某个结构位置。没有清单的标准化先行，模板资产的内容将缺乏权威依据，变成个人经验的随意汇总而非制度化规范的物理载体。

在学习路径上，完成标准模板资产的制作与发布后，下一步是**规范培训**。模板资产本身是培训的核心教具——技术美术在培训课程中会引导团队成员逐层拆解模板文件的结构，解释每个预设配置对应检查清单中的哪条规则，以及违反该规则会导致哪类具体的引擎报错或渲染异常。模板文件因此兼具"工具"和"教材"的双重功能，使规范从文字描述转化为团队的操作肌肉记忆。
