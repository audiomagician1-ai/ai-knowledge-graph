# 自动导入

## 概述

自动导入（Auto Import）是一种监控指定文件目录，当新资产文件被放入该目录时，自动触发一系列导入、处理和验证流程的技术机制。其核心是操作系统提供的文件系统事件监听接口——例如 Windows 的 `FileSystemWatcher` 或 macOS/Linux 的 `inotify`——通过订阅 `Created`、`Modified` 等文件事件，无需人工干预即可启动资产管线。

自动导入的工程实践最早在游戏行业大规模落地，约在 2010 年代初期随着 Unity 引擎的普及而被开发者广泛认知。Unity 的 Asset Database 在用户将文件拖入 `Assets/` 目录后，会自动调用对应的 `AssetImporter`（如 `TextureImporter`、`ModelImporter`）并写入 `.meta` 文件记录导入设置，这一设计直接影响了后续众多游戏引擎和 DCC 工具的资产管理方案。

自动导入对技术美术的价值在于将"保存文件"和"资产可用"两个动作之间的手动步骤彻底清零。一个中型项目每天可能产生数十到数百个新资产，若每个资产都需要手动执行导入、设置参数、确认结果，累计耗时极为可观；自动导入将这一成本压缩为接近零的监控开销。

## 核心原理

### 文件系统事件监听

自动导入的第一步是捕获文件变化。以 Python 为例，`watchdog` 库封装了跨平台的文件监听能力，核心类为 `FileSystemEventHandler`，其 `on_created(event)` 回调会在新文件写入完成后触发。关键细节是必须等待文件写入完全结束再启动导入，常见做法是轮询文件句柄是否被释放，或等待文件大小在连续两次采样（间隔约 500ms）中保持不变，避免对正在写入的文件执行截断式读取。

```python
# 简化示例：判断文件写入是否完成
import os, time

def is_file_ready(path, wait=0.5):
    size1 = os.path.getsize(path)
    time.sleep(wait)
    return os.path.getsize(path) == size1
```

### 规则匹配与处理器分派

监听到文件事件后，系统根据文件扩展名或路径规则将任务分派给对应的处理器。典型规则表如下：

| 扩展名 | 处理器 | 核心操作 |
|--------|--------|----------|
| `.fbx` / `.obj` | 模型导入器 | 单位换算、坐标轴修正、LOD标记 |
| `.png` / `.tga` / `.exr` | 纹理导入器 | 压缩格式判定、mipmap生成、色彩空间验证 |
| `.wav` / `.ogg` | 音频导入器 | 采样率标准化、响度归一化 |

路径本身也可携带元信息：将文件放入 `Textures/UI/` 时，导入器可自动将纹理类型设为 `Sprite`；放入 `Textures/World/` 时则设为 `Default` 并启用硬件压缩（如 BC7/ASTC）。

### 验证管线（Validation Pipeline）

导入处理完成后，自动导入系统应执行一组预定义的验证规则，确保资产符合项目规范，并在不合规时生成报告或阻断后续流程。常见验证指标包括：

- **纹理分辨率**：必须为 2 的幂次（64、128、256…4096），否则触发警告
- **多边形面数**：角色模型不超过 65,000 个三角面，场景道具不超过 10,000 个
- **命名规范**：通过正则表达式 `^[A-Z][a-zA-Z0-9_]+$` 验证资产名称
- **UV 范围**：UV 坐标必须在 0-1 范围内，不允许超出（用于光照贴图的网格除外）

验证结果通常写入本地日志文件，同时可通过 Webhook 推送至团队 Slack 频道或 JIRA 工单系统，让美术人员即时获知资产是否通过。

## 实际应用

**Unreal Engine 的 Auto Reimport 功能**是自动导入在商业引擎中最成熟的实现。在 `Editor Preferences > Loading & Saving > Auto Reimport` 中配置监控目录后，引擎会持续轮询（默认间隔 3 秒）。当 Photoshop 保存一张贴图的修改版本时，Unreal 会在下次轮询时检测到文件变化并自动重新导入，美术人员无需切回编辑器手动操作。

**Houdini 数字资产（HDA）的热重载**是另一个典型场景。TA 可以编写一个 Python 脚本，监控 `hda/` 目录，每当新的 `.hda` 文件复制进来，脚本自动调用 `hou.hda.installFile()` 安装资产并刷新节点库，供美术人员在 10 秒内看到最新版本的工具。

**CI/CD 资产检查流水线**中，美术人员提交资产到 Git LFS 仓库后，服务器端的 `post-receive` 钩子触发自动导入脚本，对新增资产执行全套验证并将结果附加到对应的 Pull Request 评论，阻止不合规资产合入主干。

## 常见误区

**误区一：认为自动导入等同于"放进去就能直接用"**

自动导入只是触发处理流程的入口，并不保证导入结果一定正确。如果项目从未为 `.fbx` 文件配置模型导入器规则，系统会用默认参数（如 Unit Scale = 1.0）导入，可能导致模型在引擎中比实际大 100 倍（因 Maya 默认导出单位为厘米而引擎使用米）。必须为每类资产明确定义导入预设，自动导入才有意义。

**误区二：将监控目录设置为整个项目根目录**

将监控范围设置过大会导致两个问题：一是引擎自身生成的中间文件（如 `.meta`、`.uasset`）也会触发事件，造成无限导入循环；二是大型项目中每次全盘扫描的 I/O 开销显著上升。正确做法是仅监控专门的"资产入库"目录（通常命名为 `_incoming/` 或 `_drop/`），处理完成后将合规资产移动至正式目录。

**误区三：忽略并发写入导致的竞态条件**

多名美术人员同时向监控目录上传文件时，导入脚本可能并发处理多个文件。若导入器是单例且内部状态不是线程安全的，会出现数据错乱。应为每个导入任务分配独立的工作队列（如 Python `queue.Queue`），或使用文件锁（`fcntl.flock`）确保同一时刻只有一个导入进程访问共享资源。

## 知识关联

自动导入以**自动化工作流概述**中介绍的"事件驱动"和"管线化处理"思想为基础，但将其具体落地为文件系统级别的触发机制，这是区别于定时任务（Cron Job）的关键特征。

掌握自动导入之后，可以自然延伸到三个方向：**自动化测试**关注的是如何在导入触发的验证环节设计更全面的测试用例，例如对导入结果进行渲染输出对比；**自动LOD生成**是在自动导入模型后立即追加一个减面处理步骤，构成"导入→验证→优化"的完整链路；**自动纹理处理**则在导入 PSD 或高分辨率原始图后，自动执行色彩空间转换、压缩格式烘焙等操作，三者共同构成资产自动化管线的主体。
