# 命名规范

## 概述

命名规范（Naming Convention）是技术美术在管线搭建初期制定的一套关于资产、材质、纹理、蓝图等文件命名的强制性书写规则。其核心目标是让项目中所有文件的名称本身携带足够的类型、用途和归属信息，使任何团队成员在不打开文件的前提下，仅凭文件名即可判断该资产的性质和归属。

命名规范的正式化发展与大型项目协作流程密切相关。虚幻引擎官方在2015年前后发布的 [Gamemakin Style Guide](https://github.com/Allar/ue5-style-guide) 将前缀系统（Prefix System）推广为行业共识，该指南规定静态网格体使用 `SM_`、骨骼网格体使用 `SK_`、材质使用 `M_`、材质实例使用 `MI_`、纹理使用 `T_`、蓝图使用 `BP_` 等前缀。这一套基于类型前缀的命名体系已被绝大多数使用UE的商业项目所采纳。

命名规范的实际价值体现在两个工程层面：第一，自动化脚本（如Python批处理资产检查工具）可以通过前缀直接过滤特定类型的资产，无需依赖引擎元数据查询；第二，版本控制系统（如Perforce或Git LFS）的提交日志中，文件名本身即构成可读的变更描述，大幅降低代码审查时的沟通成本。

## 核心原理

### 前缀-主名-后缀的三段式结构

标准命名规范采用 `[前缀]_[主名]_[后缀]` 的三段式结构，各段之间使用下划线 `_` 分隔，严禁使用空格或连字符 `-`（空格在某些引擎路径解析中会引发引用断裂，连字符在部分脚本语言中会被误解析为减号运算符）。

前缀由资产类型决定，主名描述资产的具体内容或所属物体（如 `Character_Warrior`），后缀则描述资产的变体或具体用途（如 `_01`、`_Diffuse`、`_Night`）。以一张角色漫反射贴图为例，完整规范名称应写作 `T_Character_Warrior_Diffuse_01`，其中 `T_` 标识类型为纹理，`Character_Warrior` 标识归属，`Diffuse` 标识贴图通道用途，`01` 标识编号版本。

### 纹理通道后缀的标准化

纹理命名中通道后缀的规范化是防止美术人员误用贴图的关键机制。常见的纹理通道后缀包括：`_D` 或 `_Diffuse`（漫反射/基础色）、`_N`（法线贴图）、`_R`（粗糙度）、`_M`（金属度）、`_AO`（环境光遮蔽）、`_E`（自发光）。部分项目会将粗糙度、金属度、AO三张单通道贴图合并打包进一张RGB贴图，此时后缀规范为 `_ORM`（R通道=遮蔽，G通道=粗糙度，B通道=金属度），这一打包顺序在UE的 `ORM` 标准中有明确规定，命名后缀直接反映通道排列顺序以防止混淆。

### 大小写规范与分词方式

项目中必须强制统一大小写策略，主流选择有两种：**帕斯卡命名法（PascalCase）** 和**全小写加下划线（snake_case）**。UE项目普遍采用帕斯卡命名法处理主名部分（如 `WarriorArmor`），而Unity项目倾向于全小写加下划线（如 `warrior_armor`）。两种方式的核心差异在于跨平台文件系统的兼容性——Windows文件系统不区分大小写，而Linux/Mac区分大小写，因此当资产被打包部署到Android/iOS平台时，大小写不一致可能导致运行时资源加载失败，这是命名规范必须锁定大小写策略的工程原因。

## 实际应用

**UE5项目中的完整前缀表**是命名规范落地的基础文档，技术美术应在项目启动的第一周将其写入项目的 `README.md` 并同步至Confluence或Notion。以下是最常用的15种类型前缀示例：

| 前缀 | 资产类型 |
|------|----------|
| `SM_` | 静态网格体（Static Mesh） |
| `SK_` | 骨骼网格体（Skeletal Mesh） |
| `ABP_` | 动画蓝图（Animation Blueprint） |
| `AM_` | 动画蒙太奇（Anim Montage） |
| `M_` | 材质（Material） |
| `MI_` | 材质实例（Material Instance） |
| `MF_` | 材质函数（Material Function） |
| `T_` | 纹理（Texture） |
| `BP_` | 蓝图类（Blueprint Class） |
| `WBP_` | UI蓝图（Widget Blueprint） |
| `PS_` | 粒子系统（Particle System，Niagara使用 `NS_`） |
| `SB_` | 声音基础（Sound Base） |
| `DA_` | 数据资产（Data Asset） |

**自动化检查脚本的对接**是命名规范产生实际价值的关键场景。技术美术可以用UE的Python编辑器工具编写资产命名校验脚本，扫描内容目录下所有资产，将不符合前缀规则的资产名称输出至日志文件，整个扫描过程通过 `unreal.EditorAssetLibrary.list_assets()` 接口调用，结合正则表达式 `^(SM|SK|M|MI|T|BP)_[A-Z][a-zA-Z0-9_]+$` 进行格式匹配，可在每次提交前自动拦截命名错误。

## 常见误区

**误区一：主名使用中文或含有特殊字符。** 部分美术人员为了直观理解会将资产命名为 `T_角色_主贴图`，这在Windows本地环境中短期内可以正常工作，但在Perforce同步至其他协作者机器或打包至移动端时，中文路径在UE的资产引用系统中极易产生 `ObjectPath` 解析错误，导致贴图丢失或打包失败。规范做法是全程使用英文字母、数字和下划线，禁止中文、空格及除下划线外的所有特殊字符。

**误区二：前缀与资产实际类型不符（幽灵前缀问题）。** 常见情况是将一个材质实例（Material Instance）命名为 `M_Rock_Mossy`（使用了材质而非材质实例的前缀），导致自动化脚本将其识别为基础材质并错误处理。这类问题在资产从外部来源导入后进行类型转换（如将Material转为Material Instance）时最为高发，命名不会自动同步更新，必须在资产审查流程中通过脚本校验强制拦截。

**误区三：将版本号或日期嵌入主名中间。** 如 `T_Rock_20240315_Diffuse` 这类命名方式将日期插入主名，导致按字母排序时同一资产的不同版本无法相邻排列，且日期信息本应由版本控制系统（VCS）的提交历史承载而非文件名。正确做法是用末尾编号 `_01`、`_02` 区分迭代版本，历史版本管理交由Perforce或Git负责。

## 知识关联

命名规范的直接前置知识是**资产管线概述**——了解整个管线中资产从创建到导出的流动路径后，才能理解为何每个环节的资产类型都需要独立的命名标识符。如果不清楚静态网格体和骨骼网格体在管线中承担的不同角色，便无法理解 `SM_` 与 `SK_` 区分命名的必要性。

命名规范确立后，将直接驱动**目录结构**的设计——合理的目录结构往往以资产类型为一级目录，而类型的定义正是通过命名前缀体系来统一认知的（如 `/Content/Textures/` 目录下所有资产应以 `T_` 开头）。同时，**资产审查流程**中的自动化校验脚本直接以命名规范为检验标准，没有明确的命名规则，审查脚本便缺乏可执行的判断依据，两者是规则制定与规则执行的上下游关系。
