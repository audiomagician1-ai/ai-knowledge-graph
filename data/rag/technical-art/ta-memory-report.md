# 内存报告

## 概述

内存报告（Memory Report）是技术美术工作流中用于量化和记录游戏运行时内存占用状况的标准化文档，其核心目的是将引擎内存分配情况按资产类别、场景单元和目标平台三个维度分别汇总，从而为优化决策提供数据依据。与静态的资产大小审计不同，内存报告捕捉的是资产在运行时加载入显存（VRAM）或系统内存（RAM）后的实际占用，同一张 2048×2048 的 BC7 压缩纹理，在磁盘上约占 4 MB，但上传至 GPU 后解压为 RGBA8 则可能膨胀至 16 MB，这一差异正是内存报告需要记录的核心数据。

内存报告作为独立的技术美术规范文件，最早在主机游戏开发周期中被系统化使用，约在 2005 年前后随着 PS3/Xbox 360 世代对内存极度受限（总 RAM 均为 512 MB）的硬件环境而普及。现代引擎如 Unity 和 Unreal Engine 均提供原生工具来辅助生成此类报告：Unity 的 Memory Profiler 包（com.unity.memoryprofiler）可导出 `.snap` 快照文件，Unreal 则通过控制台命令 `memreport -full` 生成详细的 `.memreport` 文本文件。

内存报告的价值在于它将抽象的"内存压力"具象化为可比较的数字表格，使美术总监、技术美术和程序员能在同一数据基础上讨论削减方案，而不是依赖直觉。尤其在同一项目需要同时发布 PC、PS5 和移动端时，跨平台的内存报告对比是制定分级资产策略的前提条件。

## 核心原理

### 按类别分类的内存统计

内存报告的第一个维度是资产类别拆分，通常分为以下几类并单独统计：纹理（Textures）、网格（Meshes）、音频（Audio）、动画（Animation Clips）、着色器变体（Shader Variants）以及托管堆（Managed Heap）。以一个典型的移动端场景为例，纹理往往占据总运行时内存的 50%–70%，网格约占 10%–15%，其余类别合计不超过 30%。

在 Unreal Engine 中，`memreport -full` 输出的文件会专门列出 `Texture2D` 总占用、`StaticMesh` 总顶点缓冲大小等分类数据，每一行格式形如：
```
Texture2D /Game/Characters/T_Hero_Diffuse  2048x2048  PF_DXT1  4.00 MB  (Streaming)
```
这种格式让技术美术能够立即定位超出预算的单项资产。

### 按场景生成报告

按场景生成报告意味着需要对每个独立关卡或场景区域单独采集一份内存快照，而不是只看全局峰值。一个关键指标是**场景驻留内存（Resident Memory）**，即该场景完全加载后，常驻内存池中所有资产的总占用；另一个是**峰值内存（Peak Memory）**，发生在场景切换时新旧两个场景的资产同时存在于内存中的瞬间，该峰值通常比稳态驻留内存高出 20%–40%。报告应同时记录这两个数字，因为内存溢出（OOM Crash）往往发生在峰值时刻而非稳态。

Unity Memory Profiler 提供的快照对比功能（Diff 模式）可以将场景 A 和场景 B 的 `.snap` 文件并排对比，直观显示哪些纹理或网格在场景 B 中新增，哪些因未被卸载而造成内存泄漏（常见于 `Resources.Load` 加载但未调用 `Resources.UnloadUnusedAssets()` 的情况）。

### 按平台分列数据

同一个项目在不同平台上的内存报告差异极大，原因包括：压缩格式不同（iOS 使用 ASTC，Android 使用 ETC2，PC 使用 BC7，每种格式的解压后内存占用各异）、流送策略不同、以及平台本身的 OS 内存开销不同（例如 Android 系统后台进程本身可消耗 300–500 MB 系统内存）。因此，专业的内存报告模板会设置独立的列来并排比较各平台数值：

| 资产名称 | PC (BC7) | iOS (ASTC 6x6) | Android (ETC2) |
|---|---|---|---|
| T_Environment_Rock | 5.3 MB | 2.8 MB | 4.1 MB |

这种横向对比帮助技术美术判断是否需要为特定平台单独制作低精度资产变体，而不是对所有平台一刀切降分辨率。

## 实际应用

**生成 Unreal 内存报告的工作流**：在目标场景中打开 Play-In-Editor，等待流送稳定后打开控制台，输入 `memreport -full`，报告文件将保存至项目的 `Saved/Profiling/` 目录下。随后，技术美术将该文本文件导入 Excel 或 Google Sheets，按 `Texture2D`、`StaticMesh`、`SkeletalMesh` 等关键字筛选并求和，得到各类别的占用总量，与预算表对比后标红超标项。

**移动端 OOM 定位案例**：某移动 RPG 项目在进入第三关卡时 Android 低端机崩溃，内存报告显示该关卡驻留纹理内存为 312 MB，峰值达 487 MB（场景切换时），而目标机型（3 GB RAM 机型）的游戏可用内存上限约为 900 MB，但操作系统已占用约 450 MB，实际可用仅约 450 MB，因此峰值超出可用限额导致 OOM。通过报告定位后，技术美术将 8 张环境纹理从 2048 降至 1024，峰值降低至 391 MB，问题解决。

## 常见误区

**误区一：将磁盘大小等同于内存占用**。资产大小审计（上一步骤）测量的是文件的磁盘压缩后体积，而内存报告测量的是运行时解压后的 GPU/CPU 内存占用。一张 ETC2 格式的 1024×1024 纹理磁盘约 512 KB，但在 GPU 内存中仍占用 512 KB（ETC2 保持硬件压缩状态），而同一张以 RGBA32 存储的纹理则会占用 4 MB。不同格式在硬件解压支持上的差异，导致磁盘到内存的膨胀比率从 1:1 到 1:8 不等，必须通过内存报告才能得到真实数字。

**误区二：只生成一份全局报告而忽略场景级报告**。全局内存报告显示项目总体资产库的大小，但它无法告诉你哪个具体关卡在运行时发生了内存超标。一个项目可能总资产体量为 8 GB，但某个单独的关卡在加载时仅激活 600 MB 的内存，另一个关卡却激活 1.4 GB，超出平台预算。只有按场景生成并对比报告，才能准确定位问题所在关卡。

**误区三：内存报告只需在项目末期生成一次**。内存报告应在每个 Sprint 或里程碑节点定期生成（建议至少每两周一次），因为美术资产的持续增加会导致内存占用的缓慢爬升（称为"内存蠕变"，Memory Creep），等到发布前才发现超标时，修复成本远高于过程中的持续监控。

## 知识关联

内存报告的数据输入来源于**资产大小审计**——审计提供的是各资产的磁盘体积基线，而内存报告在此基础上追加了运行时内存维度，两份文档结合才能完整描述一个资产的"体积轮廓"。例如，审计发现某音频文件磁盘占用过大，而内存报告则可进一步揭示该音频是否以解压缩形式（PCM）驻留内存，还是以压缩流式（Vorbis Streaming）方式按需加载，两种方式的内存占用可相差数十倍。

内存报告的输出直接驱动两个下游工作：**包体大小优化**会参考报告中的纹理和网格类别数据，决定哪些资产适合降级压缩；**内存预算文档**则以内存报告的历史数据为依据，为各资产类别设定具体的 MB 上限值（如"移动端单场景纹理内存不超过 256 MB"），并将此约束写入项目规范，形成对后续美术工作的量化约束框架。
