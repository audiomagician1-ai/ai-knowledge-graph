# 三角面预算

## 概述

三角面预算（Triangle Budget，也称 Poly Budget）是指在游戏或实时渲染项目中，为场景、角色、道具等各类资产预先规定的三角面数上限。这个上限不是随意拍脑袋定出来的，而是基于目标硬件的 GPU 光栅化吞吐量、场景中同屏对象数量以及期望帧率三者共同推算得出的。超出预算的资产会占用过多的顶点着色器处理时间，导致 draw call 前置的几何阶段成为帧时间瓶颈。

三角面预算的概念随着 3D 游戏的普及在 1990 年代末期逐渐形成规范。早期 PlayStation 1 时代，一个玩家角色的全身预算仅约 500–1500 个三角面；到了 PS4/Xbox One 世代，次世代主角角色的近景 LOD0 可达 80,000–150,000 个三角面，而场景背景道具则普遍控制在 200–2000 个三角面之间。这种跨越两个数量级的差异，正是不同平台预算标准随硬件演进而变化的直接体现。

在实际项目中制定并严格执行三角面预算，能够让美术团队在资产生产阶段就对性能风险进行量化管控，而不是等到集成测试时才发现帧率崩溃。这使得技术美术（TA）可以将各类资产的多边形复杂度保持在可预测的范围内，并与程序员协作确定 LOD 切换距离和 Culling 策略。

## 核心原理

### 帧预算的逆向推导

三角面预算的计算起点是目标帧率。假设目标为 60 FPS，则每帧总时间为 16.67 ms。在这 16.67 ms 中，GPU 需要完成几何处理、光栅化、像素着色、后处理等多个阶段。通常将几何阶段（顶点处理 + 三角面遍历）的时间上限控制在总帧时间的 15%–25%，即约 2.5–4 ms。

现代桌面 GPU（如 NVIDIA RTX 3080）的三角面处理峰值可达每秒数十亿个，但实际有效吞吐远低于理论值，因为 **小三角面**（屏幕投影面积小于一个像素的三角面）会严重降低光栅化效率。这是预算制定时必须考虑的关键因素：一个 10 万面的角色在近景无问题，但在远景时会产生大量亚像素三角面，此时即便面数不多也会造成显著的性能损耗，这也是 LOD 系统存在的根本原因。

### 分层预算体系

实际项目中，三角面预算按资产类型分层制定，以下是一套常见的 Mobile AAA 项目参考值：

| 资产类型 | LOD0（近景） | LOD1（中景） | LOD2（远景） |
|---|---|---|---|
| 主角/主要 NPC | 15,000–25,000 | 6,000–8,000 | 1,500–2,500 |
| 次要 NPC | 8,000–12,000 | 3,000–4,000 | 800–1,200 |
| 大型场景道具 | 3,000–6,000 | 1,000–2,000 | 200–500 |
| 小型环境道具 | 300–800 | 100–300 | — |

同屏预算（Screen Budget）是另一个维度：规定任意一帧内所有可见网格的三角面总量上限。移动端（如 iOS/Android 中高端机型）的同屏三角面总量通常控制在 **100,000–300,000** 个三角面以内；PC 主流配置则可放宽至 **1,000,000–3,000,000**。

### 预算监控工具与工作流

在 Unity 中，可通过 **Stats 面板**实时查看当前帧的三角面数（Tris），也可使用 Frame Debugger 和 GPU Profiler 逐 draw call 追踪几何开销。Unreal Engine 则提供 `stat RHI` 命令，输出 `DrawPrimitive calls` 和三角面总量；`Nanite` 虚拟几何体系统在 UE5 中从根本上改变了静态网格的预算逻辑，但骨骼动画网格至今仍需传统预算管控。

技术美术在资产验收流程中通常使用 **Python 脚本或 DCC 插件**（如 Maya 的 polyCount 脚本）对导入的 FBX 文件自动检测面数，并在超出预设阈值时触发警告或拒绝提交，将预算检查嵌入 CI/CD 管线。

## 实际应用

**开放世界场景的预算分配**：在一个如《原神》类型的开放世界手游中，技术美术会规定：玩家角色 LOD0 不超过 30,000 面（含武器装备），每个场景区块的静态背景总面数不超过 500,000 面，且单个建筑物不超过 8,000 面。当场景编辑器中加载全部资产后，TA 使用定制的场景分析工具对每个 100×100 米格子输出三角面热力图，红色区域（超过 80,000 面/格）需要美术删减远景植被或合并小道具。

**角色换装系统的预算管控**：当角色支持换装时，每套装备的面数叠加可能使角色总面数翻倍。解决方案是为基础身体设置"遮挡消除"规则：穿戴全身盔甲时，将被盔甲覆盖的躯干基础网格删除，这样换装后总面数仍维持在预算内。这要求技术美术在预算文档中明确规定每个部位槽（Head/Torso/Legs/Feet/Hands）各自的面数上限，例如头部槽不超过 3,000 面。

**移动端性能压力测试**：在目标机型（如 Snapdragon 888 设备）上，技术美术会构造"最坏情况场景"——将同屏预算填满的极端情形——然后用 Snapdragon Profiler 录制 GPU 时间线，验证几何阶段是否在 4 ms 以内。若超标，则按比例缩减各资产类型的预算上限并更新文档，形成该项目的正式预算规范。

## 常见误区

**误区一：面数越少性能越好，应无限压缩**

面数压缩到极端时，反而会带来新的性能问题。过低面数的网格在近景会出现明显的棱角，迫使美术使用更复杂的法线贴图来弥补，而法线贴图的采样本身有像素着色开销。此外，极少面数的网格在 Skinning（蒙皮计算）时因骨骼影响范围内三角面过大，会产生严重的蒙皮变形错误（皮肤穿插），最终需要增回面数修复。因此，三角面预算是一个**下限与上限都需要管控的区间**，而非单纯越少越好。

**误区二：三角面数等于顶点数**

三角面数与顶点数是不同的概念。在 GPU 处理管线中，真正被顶点着色器处理的单元是**顶点（Vertex）**而非三角面。一个共享边的网格中，顶点数通常约为三角面数的 0.5–0.75 倍；但在有 UV 缝合线、硬边法线、多 UV 通道的角色上，顶点数可能是三角面数的 **1.5–2 倍**，因为 UV 边界处顶点必须拆分。因此，监控顶点数（Vertex Count）比监控面数更能反映真实的顶点着色器开销。

**误区三：面数预算与渲染性能直接线性相关**

三角面数与帧时间并非严格线性关系。在面数未超出 GPU 的几何缓冲区阈值时，增加 50% 的三角面可能仅带来 5% 的帧时间增加；但当单帧三角面数超过某临界点（移动端通常约 500,000 面），GPU 会出现几何阶段停顿，帧时间可能突然跳升 30%–50%。这种非线性特征意味着预算必须留有安全余量（通常建议实际用量不超过上限的 80%），而不能压到极限运行。

## 知识关联

**前置概念衔接**：学习三角面预算之前需要掌握性能优化概述中的帧预算分配思想——明白 CPU 帧时间与 GPU 帧时间是并行但分别瓶颈的。三角面预算专门针对 GPU 的几何处理阶段，不涉及 CPU 侧的 draw call 排序或逻辑运算开销，这是它与"渲染批次优化"等概念的边界所在。

**后续概念延伸**：三角面预算的执行依赖**网格简化（Mesh Simplification）**技术来生成 LOD 各级别的低面版本。网格简化算法（如 Garland-Heckbert 的二次误差度量 QEM 算法）的参数调整需以三角面预算数值为目标——即"将 LOD1 简化至 LOD0 面数的 35%"这类量化目标，正是从预算文档直接转化为简化器的输入参数。理解三角面预算体系是合理设置 LOD 生成参数的前提。
