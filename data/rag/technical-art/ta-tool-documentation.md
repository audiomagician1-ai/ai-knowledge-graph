# 工具文档

## 概述

工具文档是技术美术在完成自制工具开发后，为该工具编写的一套书面说明材料，通常包含三种形态：用户手册（User Manual）、操作教程（Tutorial）和版本变更日志（Changelog）。这三种形态各自服务于不同的读者需求——手册解答"这个按钮是什么"，教程解答"我如何用它完成一个任务"，Changelog则记录"上次更新改变了什么"。

工具文档的编写传统在游戏工业中伴随管线工具的普及而形成。Valve在2004年随《半条命2》发布的Hammer Editor附带了系统化的在线文档，成为游戏行业工具文档编写的早期范本。对技术美术团队来说，一款没有文档的自制工具在编写者离职或换组后往往迅速变成"黑盒"，其他成员无法正确调用，造成管线断裂。

工具文档之所以被单独列为一项技能，是因为它要求作者同时扮演工具开发者和新手用户两个角色——开发者必须打破"知识诅咒"（Curse of Knowledge），强迫自己以零经验视角重新审视工具的每一个操作步骤。这项能力并不随着代码能力的提升而自然获得，需要刻意练习。

## 核心原理

### 用户手册的结构规范

用户手册的标准结构遵循"概览 → 安装/导入 → 界面说明 → 功能参数 → 已知限制"五段式布局。对于Maya插件类工具，界面说明部分必须对每一个UI控件（按钮、滑块、下拉菜单）给出独立的功能描述，格式通常为表格：

| 控件名称 | 类型 | 默认值 | 功能说明 |
|--------|------|--------|---------|
| Threshold | Float滑块 | 0.5 | 控制顶点合并的距离阈值，单位为场景单位 |

参数的单位、范围和默认值是手册中最常被遗漏、却是用户最频繁查阅的三项信息。

### 教程的任务导向写法

教程不等同于手册的重新排列。有效的教程以"完成一个具体任务"为组织单元，例如"用批量LOD工具为一组20个静态网格体生成三级LOD"。每个教程步骤的描述需遵循"操作 + 预期结果"的双重句式：不能只写"点击Generate按钮"，而应写"点击Generate按钮，工具将在Assets/LOD/目录下生成以原文件名加_LOD0/1/2为后缀的三份网格体文件"。用户需要知道操作后屏幕上会发生什么，才能判断自己是否操作正确。

截图是教程的核心辅助元素。Maya或Unreal Engine的工具截图应使用工具本身的UI窗口而非全屏截图，分辨率建议不低于1280×720，并在图中用红色矩形框或编号箭头标注操作位置。

### Changelog的版本记录格式

Changelog的工业标准参考[Keep a Changelog](https://keepachangelog.com)规范，该规范规定每个版本号下的条目必须归类为以下六种标签之一：**Added**（新增功能）、**Changed**（行为变化）、**Deprecated**（即将废弃）、**Removed**（已移除）、**Fixed**（Bug修复）、**Security**（安全修复）。

版本号命名推荐采用语义化版本控制（Semantic Versioning，SemVer）：格式为`主版本号.次版本号.修订号`，例如`v2.1.3`。主版本号变更代表不向下兼容的接口改动，次版本号代表向下兼容的新功能，修订号代表Bug修复。对于技美工具，每次修改了工具的输出文件结构或参数命名时，必须升级主版本号，否则依赖旧版本的管线脚本会静默失败。

一条规范的Changelog条目示例：
```
## [2.0.0] - 2024-03-15
### Changed
- 输出目录结构从平铺式改为按资产类型分级，旧版本批处理脚本需同步更新路径变量
### Fixed
- 修复了当输入网格体面数超过65535时工具崩溃的问题
```

## 实际应用

**场景一：Maya蒙皮权重导出工具的文档套件**
一位技术美术为角色组编写了一个Python插件，可以将蒙皮权重以JSON格式导出并在不同拓扑的网格上重新应用。其文档套件包含：一份500字的手册（描述`Influence Threshold`参数的含义：低于该值的权重影响将被自动清零，默认值0.01意味着影响权重低于1%的骨骼绑定会被忽略）；一篇教程（以"为换装系统的上衣替换体重新映射权重"为题，包含7步操作+5张截图）；以及一份自工具v1.0起维护的Changelog。

**场景二：Houdini程序化建筑生成器的在线文档**
使用Confluence或Notion搭建的内部Wiki适合存放Houdini HDA工具的文档。将手册的参数表格与教程页面分开存放，通过超链接互相引用，方便用户在查阅参数定义时不必从头浏览完整教程。对于有暴露参数的HDA，文档中还应注明哪些参数组合会触发已知的性能瓶颈。

## 常见误区

**误区一：把文档写成代码注释的扩充版**
代码注释面向的读者是会读代码的开发者，而工具文档的首要读者是不看代码的美术同事。将`# clamp vertex count to max_poly`翻译成"该参数用于限制顶点数量"并不等于完成了文档——文档需要进一步说明限制的上限值是多少、超出上限后工具的行为是报错还是自动裁切、以及为什么存在这个上限（例如：受制于DCC软件的16位索引缓冲区限制，最大面数为65535）。

**误区二：Changelog写成Git提交记录的复制粘贴**
Git commit message的受众是代码审查者，其颗粒度过细（如"fix typo in variable name"）且语境缺失。Changelog的条目需要站在工具用户的视角描述变化对他们工作流的影响，而非描述代码层面发生了什么。一条"修改了路径拼接逻辑"的commit在Changelog中应写为"修复了路径包含中文字符时工具无法读取配置文件的问题"。

**误区三：只在工具发布时写文档，之后不再更新**
工具文档和工具代码必须同步迭代。一份描述v1.0行为却挂在v2.3工具旁边的过时文档，比没有文档更危险——用户按照文档操作却得到与描述不符的结果，会浪费大量排查时间。建议在工具的任务追踪系统（如Jira或GitHub Issues）中将"更新文档"作为每个功能变更工单的强制完成条件（Definition of Done）。

## 知识关联

工具文档是技美工具开发流程中的最终交付物，其内容质量直接依赖于对工具本身的深度掌握——包括工具的输入输出规格、参数边界值和已知限制，这些信息在开发阶段的技术设计文档（TDD）中已有记录，编写用户文档时可以以TDD为一手信息来源进行转化改写。

在工具版本管理方面，Changelog的撰写与Git版本控制工作流高度耦合：每次创建新的Git Tag或Release时，应同步生成对应版本的Changelog条目。如果团队使用自动化发布工具（如`git-cliff`或`conventional-changelog`），规范化的commit message格式可以辅助自动生成Changelog草稿，但最终仍需人工以用户视角校订措辞，确保每条记录对非开发者读者可读。
