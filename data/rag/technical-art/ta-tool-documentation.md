# 工具文档

## 概述

工具文档是技术美术在开发自制工具后，为使用者提供的完整书面材料集合，包括用户手册、操作教程、API参考和版本变更记录（Changelog）。与代码本身不同，工具文档的受众是使用工具的美术师或其他技术人员，其核心目标是让用户在不接触源代码的情况下，能够独立完成安装、配置和日常使用。

工具文档的形式在软件行业有相当成熟的规范。1975年，IBM内部推行"Programmer's Workbench"时就强制要求每个内部工具附带独立文档，这一实践逐渐演变为整个行业标准。在游戏技术美术领域，由于工具往往由一两个人开发却服务于数十名美术师，文档的价值甚至超过工具本身的代码质量——一个有完整文档的普通工具，远比一个无文档的优秀工具更受团队欢迎。

对于技美工具开发者而言，编写文档还有一个常被忽视的好处：文档撰写过程本身会暴露工具设计中的逻辑漏洞。当你试图用文字解释某个参数的作用时，若发现无法用一句话说清楚，通常意味着该参数的设计需要重构。

## 核心原理

### 用户手册的三层结构

一份完整的技美工具用户手册通常分为三层：**安装配置层**、**功能参考层**和**工作流集成层**。安装配置层描述工具的依赖环境（例如"需要 Maya 2023+ 及 Python 3.9"）、安装路径和初始化步骤；功能参考层逐一说明每个按钮、参数和菜单项的作用，类似字典；工作流集成层则用具体的美术任务串联功能，例如"如何用本工具完成一套角色LOD的批量生成"。缺少任何一层的文档都是不完整的——只有功能参考而无工作流集成，新用户往往不知从何入手。

### Changelog 的标准写法

Changelog（版本变更记录）应遵循语义化版本号规范（Semantic Versioning，即 MAJOR.MINOR.PATCH 格式），配合分类标签记录每次变更。常用分类包括：`Added`（新增功能）、`Changed`（行为变更）、`Fixed`（Bug修复）、`Deprecated`（即将废弃）和 `Removed`（已删除功能）。一条规范的 Changelog 条目示例如下：

```
## [1.2.0] - 2024-03-15
### Added
- 新增批量导出功能，支持同时处理最多500个FBX文件
### Fixed
- 修复在Maya中选中超过256个物体时工具崩溃的问题
```

对于技美工具，`Changed` 和 `Deprecated` 分类尤其重要，因为美术师通常不会主动更新工具，一旦某个旧参数被悄然移除，可能导致已有的美术流水线中断。

### 操作教程的"任务导向"写法

教程（Tutorial）与手册（Manual）的本质区别在于：手册描述工具是什么，教程描述用户应该做什么。编写教程时应采用任务导向结构：每篇教程只解决一个具体问题，以动词开头命名（如"批量重命名贴图文件"而非"重命名功能介绍"），步骤编号从1开始且每步只包含一个操作动作，并在关键步骤附上截图或预期输出结果。对于涉及参数设置的步骤，需同时给出推荐值和可接受范围，例如"将Smoothing Angle设为60°（范围30°–90°，默认60°）"。

## 实际应用

**Maya 批量处理工具文档示例**：假设你开发了一个Maya插件，用于批量检查场景中的UV错误。用户手册应在功能参考部分明确列出每项检查规则的具体标准（如"UV Island面积小于4×4像素视为过小"），而不是模糊地写"检测异常UV"。这一数值对美术师做出判断至关重要。

**Confluence + Markdown 双轨文档**：许多游戏公司的技美团队同时维护两套文档：一份托管在公司内网Wiki（如Confluence）供美术师查阅，另一份以 `README.md` 形式放在工具的Git仓库根目录供开发者参考。两套文档的侧重点不同——Confluence版本侧重操作图文，README版本侧重安装和依赖说明。在 `README.md` 中应包含徽章（Badge）显示当前版本号和兼容性状态，例如 `![Version](https://img.shields.io/badge/version-1.2.0-blue)`。

**Houdini HDA 文档嵌入**：Houdini的HDA（Houdini Digital Asset）格式支持将文档直接嵌入资产文件内，通过 `Help` 标签页以纯文本或HTML格式呈现。这意味着HDA工具的文档应当与HDA文件同步更新，避免文档与工具版本脱节。

## 常见误区

**误区一：以开发者视角而非用户视角写文档**

技美开发者最常犯的错误是用开发术语描述功能，例如"该模块通过遍历场景DAG节点实现检测"。美术师并不需要了解DAG节点是什么，他们需要的是"工具会扫描场景中所有物体，包括隐藏和锁定的层级"。文档的读者是使用者，不是接手维护代码的程序员。用户手册中的每一个技术名词，要么附带解释，要么替换为美术师的日常用语。

**误区二：Changelog 与发布脱节**

很多开发者在工具上线后才补写Changelog，导致早期版本的变更记录丢失或不准确。正确做法是在每次提交代码时同步更新 `CHANGELOG.md` 草稿区，发布时将草稿内容整理为正式条目。若使用Git，可以在commit message中以`[changelog]`标签标记需要进入Changelog的变更，之后用脚本自动提取，节省手动整理时间。

**误区三：截图不标注版本信息**

工具UI会随版本更新而变化，文档中的截图若不标注对应版本号，当UI改变后截图与实际界面不符，反而会给用户造成困惑。每张截图的alt文本或图注应标明版本，如"v1.2.0 主界面截图"，并在工具进行UI改动后（通常是 MINOR 版本递增时）优先更新相关截图。

## 知识关联

工具文档是技美工具开发概述中"工具生命周期"环节的最后一步，但它的内容反向影响工具的迭代设计。撰写功能参考时发现难以解释的参数，往往指向工具API设计中的命名问题；撰写安装指南时梳理出的依赖列表，直接成为工具打包脚本的输入数据。

从文档类型拓展的维度来看，用户手册是最基础的文档形式，教程在其基础上增加了叙事性和任务完整性，而API文档（针对可二次开发的工具）则要求对每个公开函数标注参数类型、返回值和异常情况，通常配合Python的`docstring`规范使用，并可通过Sphinx等工具自动生成HTML文档站点。掌握这三种文档形式后，技美开发者能够针对不同受众（普通美术师、技术美术、外部开发者）提供适当层次的文档支持。
