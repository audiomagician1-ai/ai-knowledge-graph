# 性能预算分配

## 概述

性能预算分配是指将每帧可用的总时间（帧时间预算）按照功能模块切分为若干子预算，并约束各模块的耗时不得超出分配额度的工程规范。与直觉相反，性能优化并不总是"让每个模块都越快越好"，而是先确定优先级再分配资源——60 FPS 对应 16.67 ms 的总帧时间，30 FPS 对应 33.33 ms，这两个数字是一切分配策略的起点。

这一概念源于嵌入式与实时系统领域的"时间分区"思想，约在 2000 年代初期随着游戏主机性能分析工具（如 PlayStation 3 的 SN Tuner）的普及而被游戏技术美术和引擎程序员广泛采用。在移动端上，由于 GPU 和 CPU 的功耗热量限制更为严苛，帧时间预算分配的重要性进一步放大——超出预算不仅掉帧，还会触发设备降频（Thermal Throttling），导致实际可用预算进一步缩水。

性能预算分配的意义在于它将"性能"从一个模糊的质量形容词转变为可量化、可追踪、可分工的工程指标。技术美术团队可以据此与程序员、动画师就各自模块的开销上限进行协商，并在 CI/CD 流程中自动化地检测回归。

## 核心原理

### 帧时间预算的计算与分层

目标帧率确定后，总帧时间预算随之确定。以 60 FPS 为例，16.67 ms 必须容纳 CPU 主线程逻辑、渲染提交、GPU 执行、音频、物理、动画等所有工作。实践中常见的一级分配比例如下：

| 模块 | 典型占比 | 60 FPS 时的毫秒数 |
|------|---------|-----------------|
| 渲染（CPU 侧提交） | 30–40% | 5–7 ms |
| GPU 执行 | 与 CPU 并行，需 ≤ 帧时间 | ≤ 16.67 ms |
| 游戏逻辑 / AI | 20–25% | 3–4 ms |
| 物理模拟 | 10–15% | 1.5–2.5 ms |
| 动画更新 | 10–15% | 1.5–2.5 ms |
| 缓冲余量（Headroom） | 10–15% | 1.5–2.5 ms |

保留 10–15% 的 Headroom 是行业惯例，用于吸收平台差异、GC 抖动和突发逻辑峰值。

### GPU 与 CPU 的双线预算

GPU 帧时间和 CPU 帧时间是**并行**的两条预算线，均不能超过总帧时间。如果 CPU 提交命令只需 6 ms 而 GPU 执行需要 14 ms，则瓶颈在 GPU，继续压缩 CPU 侧对帧率毫无帮助。判断瓶颈侧的工具有 RenderDoc 的 Pipeline Statistics、Unreal 的 GPU Visualizer（`stat gpu`命令）以及 Unity 的 Frame Debugger。技术美术必须先定位瓶颈所在，才能决定将哪个子预算作为优化目标。

### 动态预算调整策略

静态分配表只是基准线，实际游戏场景变化巨大。动态预算调整通过**优先级队列**在运行时重新分配剩余预算：当物理场景简单（室内无破坏物理）时，可将节省下的 1 ms 分配给渲染（开启额外阴影级联）；当大型战斗场景中角色数量骤增时，则压缩粒子和植被的渲染预算。Unreal Engine 的 `r.DynamicRes`（动态分辨率）正是这一思想的自动化实现——它以 GPU 帧时间为输入信号，实时缩放渲染分辨率，使 GPU 预算始终贴近但不超过 16.67 ms。

### 各模块预算的计算方法

- **渲染 DrawCall 预算**：在目标平台上，每个 DrawCall 的 CPU 开销约 0.01–0.05 ms，由此可反推最大 DrawCall 数。移动端通常限制在 100–200 个以内。
- **物理步长预算**：PhysX / Chaos 等物理引擎以固定步长（如 16.67 ms 或 33.33 ms）推进模拟，若物理计算超时则需减少碰撞体数量或降低求解迭代次数（Solver Iterations，默认值 8，可调至 4）。
- **动画预算**：骨骼数量与混合状态机节点数共同决定动画更新开销。Unity 中可通过 `AnimatorCullingMode.CullUpdateTransforms` 对不可见角色跳过骨骼更新，从而释放预算。

## 实际应用

**移动端 RPG 的预算分配案例**：目标为 30 FPS（33.33 ms），主战斗场景有 10 个角色同屏。初始评估发现动画模块占用 8 ms（占比 24%），超出预算。排查后发现每个角色使用了 60 骨骼的高精度骨架，而屏幕内 4 个远景角色实际像素高度不足 100 px。解决方案是为远景角色切换至 20 骨骼的 LOD 骨架，并将 `Solver Iterations` 从 8 降至 4，动画模块降至 4.5 ms，释放出 3.5 ms 用于渲染更高质量的后处理效果。

**PC 开放世界的 GPU 预算分配**：目标 60 FPS，室外场景 GPU 帧时间达 19 ms，超出 16.67 ms 限制。通过 GPU Visualizer 定位到 Shadow Map 渲染占用 7 ms。将级联阴影（CSM）的第四级距离从 5000 cm 压缩至 2000 cm，并将第四级分辨率从 2048 降至 1024，Shadow Map 降至 4.2 ms，GPU 帧时间回落至 15.8 ms，完成预算修复。

## 常见误区

**误区一：认为预算分配是一次性工作**。许多团队在项目初期制定了分配表后便束之高阁。实际上，每次新增系统（如增加布料模拟）或资产规模升级（角色面数增加 50%），都必须重新执行预算核查。建议将性能预算检查纳入 Sprint Review，并在版本提交时自动运行性能回归测试。

**误区二：CPU 和 GPU 预算相互独立，压缩任意一侧都有效**。如前述，只有瓶颈侧的优化才能提升帧率。若 GPU 已经空闲等待 CPU 数据，在 GPU 端削减着色器复杂度只会增加 GPU 空闲时间，而非提升帧率。正确做法是先用性能分析工具（Profiler）确认瓶颈，再对症分配优化资源。

**误区三：帧时间预算等于平均帧时间达标**。30 FPS 不等于"平均每帧 33 ms"，而是"每一帧都不能超过 33 ms"。帧时间的**方差**（即帧时间抖动，Frame Time Variance）同样是预算指标的一部分。0.1% 的帧超时 50 ms 即会被玩家感知为"卡顿"，因此预算分配时必须为峰值场景而非均值场景预留 Headroom。

## 知识关联

**前置知识——性能优化概述**为本模块提供了 GPU/CPU 流水线的基础模型，以及 Draw Call、顶点数、像素填充率等基础指标的含义；理解这些指标是制定渲染子预算的前提。

**后续概念——画质可伸缩性（Scalability）**是性能预算分配的动态延伸：当目标平台从单一设备扩展为高中低三档硬件时，需要为每一档分别制定帧时间预算表，并通过质量等级参数（如 Unreal 的 `sg.ShadowQuality 0/1/2/3`）在运行时切换各模块的开销级别。性能预算分配是画质可伸缩性的定量基础——没有明确的各档预算数字，可伸缩性调参就会变成凭感觉的试错过程。
