# 动画标准

## 概述

动画标准是技术美术工作中针对骨骼绑定、帧率设定、文件导出参数与命名规则所制定的一套量化规范。它直接决定动画能否被游戏引擎正确读取、动作是否流畅以及资产能否在多人协作流程中准确被引用。不符合动画标准的文件往往会导致骨骼错位、动作抖动或引擎导入报错，返工成本极高。

动画标准的系统化始于20世纪90年代末实时3D游戏的兴起。早期《雷神之锤》（Quake，1996年）时代的开发者必须手动将 MD3 格式骨骼数量限制在最多40根以内，否则引擎无法解析。随着 Unity、Unreal Engine 等现代引擎普及，动画标准逐渐演变为包含 Humanoid Rig、动画状态机绑定要求以及 FBX 导出选项在内的综合规范文档。

在多人协作项目中，动画标准的价值在于消除歧义。一个完整的动画标准文档能让动画师、程序员和技术美术在不需要反复沟通的情况下交付可直接集成的资产，显著缩短迭代周期。

## 核心原理

### 骨骼数量规范

骨骼数量直接影响蒙皮计算的GPU开销。移动端项目通常要求单角色骨骼数不超过**75根**，主机/PC项目可放宽至**150～200根**。Unity的 Humanoid Avatar 系统要求必须包含15根必要骨骼（包括 Hips、Spine、Chest、UpperArm.L/R 等），否则自动映射失败。

骨骼层级同样受到约束。根骨骼（Root Bone）必须位于世界坐标原点，且其旋转和缩放值在绑定姿势（Bind Pose）下必须归零归一。辅助骨骼（Helper Bone）和控制器骨骼不能导出进 FBX，否则引擎会将其识别为额外关节并破坏动画重定向（Retargeting）结果。

### 帧率标准

帧率（FPS）的选择需与项目引擎目标帧率和动作类型匹配。常见规范如下：

- **30 FPS**：移动端游戏标准，适用于大多数循环动作（Idle、Run、Jump）
- **60 FPS**：主机/PC动作游戏，适用于帧精度要求高的格斗或快速攻击动作
- **24 FPS**：过场动画（Cutscene）专用，与影视行业惯例一致

动画片段长度必须以帧为单位标注，而非以秒标注，以避免不同软件中小数精度导致的帧偏移。例如一个1秒的30FPS动作应标注为"第0帧至第29帧（含）"，而非"0秒至1秒"。

### 导出设置规范

FBX 导出是当前主流引擎接受的标准格式。关键导出参数包括：

- **FBX 版本**：推荐使用 FBX 2020 或 FBX 2018，避免使用 FBX 6.0（已被 Maya 2022+ 弃用）
- **单位**：必须统一为**厘米（cm）**，与 Unreal Engine 默认单位一致；若使用 Unity，需在导出时将 Scale Factor 设为 **0.01**
- **烘焙动画（Bake Animation）**：必须启用，确保所有约束（Constraint）和表达式（Expression）被转换为关键帧数据
- **仅导出选中物体**：避免场景中的摄像机、灯光被误打包进 FBX

切线类型（Tangent Type）建议统一设为 **Auto** 或项目规范指定的 **Spline**，禁止混用，否则不同DCC软件读取同一文件时曲线形状会产生差异。

### 命名规范

动画文件命名采用结构化字符串，典型格式为：

```
[角色代号]_[动作类型]_[变体编号]_[版本号]
```

示例：`CH_Knight_Attack_01_v02.fbx`

骨骼命名必须区分左右侧，统一使用后缀`_L` / `_R`（Unreal Engine 约定）或前缀 `Left_` / `Right_`（Unity Humanoid 约定），项目内不可混用两种风格。动画片段（Clip）在引擎内的命名需去掉版本号后缀，保持精简，如 `Knight_Attack_01`。

## 实际应用

在 Unity 项目中，技术美术会在 FBX Import Settings 里将 **Rig Type** 设置为 **Humanoid**，并在 Animation 标签下逐条检查每个片段的 Loop Time、Loop Pose 及 Root Transform 设置是否符合标准。一个典型的错误案例是：动画师导出时未勾选"Bake Animation"，导致约束驱动的骨骼在引擎中静止不动，排查时间往往超过2小时。

在 Unreal Engine 5 项目中，项目组通常维护一份 **Skeleton Asset**（骨骼资产），所有角色动画必须重定向到同一套骨骼，因此骨骼数量和层级的标准化尤为重要。若某个角色骨骼多出一根手指辅助骨，整条重定向链路就会断裂。

手游项目中，为了控制包体大小，动画标准还会规定**关键帧精简策略**：位移曲线精度保留至小数点后3位，旋转曲线压缩至误差不超过0.5度，使用 Unity 的 **Animation Compression** 设置中的 **Optimal** 模式自动处理。

## 常见误区

**误区1：帧率越高动画越流畅**
很多初学者认为导出60 FPS的动画一定比30 FPS更流畅。实际上，如果引擎以30 FPS采样动画，60 FPS的文件只会增加50%的内存占用而没有视觉收益。移动端项目中将所有动画导出为60 FPS是常见的优化反模式，会直接导致动画内存超标。

**误区2：骨骼命名无所谓，引擎会自动识别**
Unity 的 Humanoid 自动映射确实会尝试猜测骨骼，但猜测失败率在非规范命名下高达30%以上，尤其是手指骨骼和脊柱层级。依赖自动识别而不制定命名标准，会在项目后期导致大批动画重定向错误，修复成本远高于前期规范制定的投入。

**误区3：导出时包含控制器骨骼不影响使用**
控制器骨骼（如 IK 目标骨骼、极向骨骼）若被导入引擎，会占用骨骼槽位并在某些引擎的骨骼数上限检查中触发警告，甚至导致 LOD 骨骼剔除逻辑出错。正确做法是在导出前通过脚本或手动方式将控制器骨骼从导出列表中排除。

## 知识关联

动画标准建立在**美术标准概述**所确立的规范化思维之上：命名规则、单位统一和版本控制等通用原则在动画领域被具体化为针对 FBX 参数和骨骼层级的量化要求。理解美术标准概述中"标准即约束边界"的概念，有助于快速理解为何骨骼数上限和帧率选择不是建议而是硬性约束。

动画标准与**模型标准**高度协同：模型标准中规定的面数、LOD层级和蒙皮权重规范直接影响动画骨骼的分配策略。例如，LOD2模型通常需要减少到30根以下的骨骼，这要求动画师在制作时就规划好哪些骨骼可以在低模层级被合并或删除。在项目制作初期同时制定模型标准与动画标准，可以避免后期因标准冲突导致的资产返工。
