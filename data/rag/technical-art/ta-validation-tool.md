# 资产验证工具

## 概述

资产验证工具（Asset Validation Tool）是技术美术工作流中用于自动检测3D资产是否符合项目规范的程序系统。它通过预设的规则集，批量扫描模型的面数上限、UV展开质量、材质槽数量、贴图分辨率以及命名约定等属性，并输出结构化的错误报告，取代了依赖人工逐一审查的低效流程。

该类工具的系统化应用最早在2000年代中期的AAA游戏工作室中出现，当时育碧、EA等大型团队开始将Python脚本集成到Maya和Max的资产提交管线中，以应对单个项目数千个资产的管理压力。早期实现依托DCC软件的内置脚本接口，现代版本则更多地基于独立运行的CLI程序或USD Pipeline中的Validator框架。

资产验证工具的核心价值在于将隐性的"规范理解"转化为可执行的代码逻辑。一个未被验证的模型进入引擎后可能导致LOD生成失败、光照烘焙错误或移动端性能超标，而这类问题在后期发现的修复成本往往是入库前的8到15倍。将验证前置到美术提交阶段，可以将此类返工率降低至10%以下。

## 核心原理

### 规则定义与阈值配置

资产验证工具的基础是一套可配置的规则集，每条规则包含三个要素：**检测项**、**比较逻辑**和**阈值**。例如，一条面数规则可定义为：`mesh.polycount <= project_config["hero_lod0_max_tris"]`，其中 `hero_lod0_max_tris` 可能被设置为 80,000 三角面。规则文件通常以 JSON 或 YAML 格式存储，使非程序员的技术美术也能在不修改源代码的情况下调整标准。

常见检测项的覆盖范围包括：

- **几何规范**：三角面数、四边面/n-gon存在性、孤立顶点、重叠面
- **UV规范**：UV孤岛是否超出0-1空间、UV重叠率（overlap ratio，理想值应<2%）、未展开UV的面比例
- **材质规范**：每个模型的材质槽数量上限（移动端常见限制为3个）、材质命名是否匹配 `M_[AssetName]_[Type]` 格式
- **命名规范**：正则表达式匹配，如 `^SM_[A-Z][a-zA-Z0-9_]+$` 用于静态网格体

### 检测逻辑的实现机制

以 Maya Python API 为例，面数检测通过 `cmds.polyEvaluate(f=True)` 获取三角化后的面数。UV重叠检测则需要遍历每个UV孤岛，构建包围盒并计算交叉面积。命名规范检测使用 Python 的 `re.match()` 函数对节点名称执行正则匹配。对于 Houdini 资产管线，验证逻辑通常被封装为 SOP 节点，利用 `hou.Geometry.primCount()` 等接口读取几何数据。

错误严重级别（Severity）的划分是检测逻辑的重要组成部分，通常分为三级：
- **ERROR（阻断级）**：面数超标、UV超出0-1空间——阻止提交
- **WARNING（警告级）**：命名不符合推荐格式——需确认方可提交
- **INFO（信息级）**：贴图尺寸未达到最优——仅记录，不阻断

### 报告输出与集成

验证结果应输出为机器可读格式，JSON 报告结构示例如下：

```json
{
  "asset": "SM_TreeBirch_01",
  "timestamp": "2024-03-15T14:22:00",
  "errors": [{"rule": "polycount", "value": 95420, "limit": 80000}],
  "warnings": [{"rule": "naming", "node": "pMesh1"}]
}
```

该报告可被 Perforce、Shotgrid 或自研提交系统读取，实现 **提交门禁（Submit Gate）** 机制——仅当 errors 列表为空时，资产才允许进入主库。

## 实际应用

**角色资产入库流程**：在动作类游戏项目中，角色 LOD0 模型需满足：三角面数 ≤ 100,000、材质数 ≤ 5、所有骨骼命名符合 `BIND_[BoneName]` 格式。验证工具在美术完成绑定后自动运行，对100个角色资产的全量检测通常在40秒内完成，输出统一的HTML报告供技术美术审阅。

**场景道具批量验证**：场景组每日产出数十个静态网格体，验证工具作为 Nightly Build 的一部分在服务器端运行，扫描当天新增资产的UV重叠情况。当UV重叠率超过5%时，工具会自动在 Shotgrid 上创建对应的 Task 并指派给原作者，无需技术美术手动跟进。

**移动端资产合规检查**：针对移动平台，验证工具额外检测贴图尺寸是否为2的幂次（128、256、512...），以及是否存在超过 1024×1024 的漫反射贴图。该规则在PC版本的规则集中被设为 INFO 级，在移动版规则集中被提升为 ERROR 级，通过同一套代码框架、不同配置文件实现平台差异化管理。

## 常见误区

**误区一：验证工具能替代人工审查**
资产验证工具只能检测可量化的规则，无法判断UV布局的美观性、贴图内容的艺术质量或骨骼权重的合理性。有些团队在接入验证工具后减少了技术美术的审查频次，导致"通过验证但视觉质量低劣"的资产进入产品。验证工具的定位是**过滤明显违规**，不是替代审美判断。

**误区二：阈值应当越严越好**
将三角面数上限设置为过于保守的值（如将LOD0限制在20,000面）会迫使美术在造型完整性上做出不必要的妥协。合理的阈值应基于目标平台的实测性能数据（如该平台每帧可负担的总三角面预算）和美术需求反复协商后确定，而非凭经验拍脑袋。阈值过严会导致美术频繁申请豁免，反而破坏规范的权威性。

**误区三：验证只需在提交前运行一次**
在资产生命周期中，引用关系变化（例如材质库更新导致材质槽超标）或平台规格调整可能使曾经合规的资产变得不合规。仅在提交时运行验证无法覆盖这类情况，应当将验证工具集成到 **Nightly Build** 流程中，对全量已入库资产定期执行合规扫描。

## 知识关联

**前置概念：资产处理工具**
资产验证工具通常在资产处理工具（如自动减面、UV重排、LOD生成）的下游运行。处理工具负责将资产转换到目标状态，验证工具负责确认该状态是否达标。两者共同构成完整的资产入库管线，处理工具的输出结果直接作为验证工具的输入对象。

**后续概念：工具测试**
资产验证工具本身需要通过工具测试来确认其规则逻辑的正确性——例如构造一个已知面数为80,001的测试模型，验证工具是否必然输出 ERROR。编写针对验证规则的单元测试是保证工具可靠性的标准做法，防止规则代码在迭代中出现回归错误。

**后续概念：数据验证管线**
单个验证工具处理孤立资产，而数据验证管线将多个验证工具串联，形成针对完整资产包（Asset Bundle）的系统化检查流程。管线层面还引入了跨资产的一致性检验，例如检测场景中所有角色的材质命名是否遵循同一套体系，这是单资产验证工具无法独立实现的能力。
