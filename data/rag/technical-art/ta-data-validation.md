# 数据验证管线

## 概述

数据验证管线（Data Validation Pipeline）是技术美术工作流中的自动化检查机制，负责在资产从创作到引擎集成的每个阶段，系统性地验证资产是否满足预设的技术规范。与单次手动检查不同，数据验证管线以串行或并行的节点链形式运行，每个节点执行特定类别的检查规则，前一节点的通过结果是后一节点运行的前提。

这种管线化验证思想在2010年代游戏行业大规模多人协作项目（如AAA级开放世界游戏）中普及，核心动因是"脏资产"流入引擎导致的构建失败代价极高。一个未经验证的高模进入Cook流程可能导致整个构建任务失败，影响整个团队数小时的工作。数据验证管线将错误拦截点从"提交后发现"前移至"提交前预检"，大幅降低集成风险。

数据验证管线之所以不同于简单脚本批处理，在于它维护了**阶段状态**（Stage State）和**错误上报机制**（Error Reporting）。每个验证阶段可以输出三类结果：`PASS`、`WARN`和`FAIL`，后续处理逻辑根据结果类型做出拦截、警告或记录的差异化响应。

## 核心原理

### 管线阶段划分与检查点设计

数据验证管线通常划分为三个主要检查点阶段：**DCC阶段检查**（在Maya/Blender等软件内运行）、**导出阶段检查**（在FBX/USD导出后立即运行）、**引擎入库阶段检查**（在资产导入引擎前运行）。每个阶段的检查规则集合不同，DCC阶段侧重拓扑规范（如多边形面数、UV层级数量），导出阶段侧重文件格式完整性（如FBX版本兼容性、骨骼命名约定），引擎入库阶段侧重渲染相关规范（如贴图尺寸是否为2的幂次方、LOD层级完整性）。

三阶段结构的好处是将验证成本分散：越早发现的问题越容易修复，因为资产尚未经历格式转换，错误定位直接。一个典型的工业标准是，DCC阶段拦截60%以上的基础错误，导出阶段拦截30%，引擎阶段仅处理5%~10%的引擎特定规范。

### 规则引擎与配置化设计

数据验证管线的规则集通常以JSON或YAML格式定义，而非硬编码进验证脚本。一条典型规则的结构如下：

```json
{
  "rule_id": "UV_NO_OVERLAP",
  "stage": "DCC",
  "severity": "FAIL",
  "target": "mesh",
  "threshold": 0.0,
  "description": "UV Channel 1不允许有重叠面积超过0%的区域"
}
```

`rule_id`唯一标识规则，`severity`决定验证失败时的处理动作，`threshold`为可调参数，允许不同项目针对低重要度资产放宽标准（例如背景道具允许UV重叠率低于5%）。配置化设计使同一套管线代码可以服务于不同严格程度的项目需求，而无需修改核心逻辑。

### 结果聚合与阻塞控制

数据验证管线的执行器（Runner）负责收集所有节点的输出结果并执行聚合逻辑。常见的聚合策略是**最严级别优先**原则：若管线中存在任何一个`FAIL`结果，整个管线输出`FAIL`并阻塞后续提交流程；若仅存在`WARN`，管线输出`WARN`并允许提交但记录日志。执行器还维护一个结构化的错误报告，格式通常包含资产路径、失败规则ID、具体数值与允许范围，例如：

```
[FAIL] assets/props/barrel_01.fbx
  Rule: POLY_COUNT_LIMIT
  Value: 12450 tris
  Allowed: <= 8000 tris
  Stage: DCC_EXPORT
```

这种精确到具体数值的报告格式，使美术师无需猜测修改方向，直接获得可操作的修复信息。

## 实际应用

**角色资产提交流程**中，数据验证管线会在美术师点击"提交"按钮时自动触发。以Unreal Engine项目为例，角色骨骼网格体的验证管线通常包含17~25条规则，涵盖骨骼命名前缀（`SK_`）验证、Root骨骼是否在原点、蒙皮权重是否存在未绑定顶点（权重之和不等于1.0的顶点视为非法）等。整个管线在本地执行时间通常控制在30秒以内，超过60秒则视为管线设计问题，需要重新拆分或并行化检查节点。

**贴图资产批量入库**场景中，验证管线作为Perforce或Git LFS提交钩子（Pre-commit Hook）运行，自动检查贴图宽高是否为2的幂次（允许值：64、128、256、512、1024、2048、4096），并验证贴图的色彩空间设置是否与资产类型一致（颜色贴图应为sRGB，法线贴图和遮罩图应为Linear）。这种在版本控制层面的拦截，彻底防止了不合规贴图进入仓库。

**Level Streaming关卡资产**的验证场景更复杂，验证管线需要在资产层面之外追加场景级规则，例如静态网格体碰撞体是否存在、LOD切换距离是否按项目标准配置（LOD0-LOD1切换距离不得大于1500cm）。

## 常见误区

**误区一：将所有规则放在单一阶段集中验证**

部分团队将所有检查集中在引擎导入阶段，认为"只需检查一次"。这会导致问题被发现时资产已经历多次格式转换，错误溯源困难，且美术师需要重新回到DCC工具修改后再次走完导出流程。正确做法是将规则按照**最早可检查阶段**原则分配，在问题产生阶段立即拦截。

**误区二：验证失败时强制阻塞而不区分严重级别**

将所有验证规则的`severity`设为`FAIL`会导致管线过于严苛，例如一个次要道具UV有微小重叠就阻塞整个提交，严重降低团队效率并引发美术师对管线的抵触情绪。实际生产中应将规则分为必须修复（影响渲染正确性或引擎崩溃的问题）和建议修复（仅影响性能优化的问题），并在报告中明确区分。

**误区三：验证管线只负责检查，不需要提供修复建议**

数据验证管线不应只输出"FAIL"而不提供上下文。规则设计时应为每条规则附加`fix_hint`字段，告知美术师修复方法或提供自动修复脚本的入口。例如贴图尺寸不合规时，可在报告中直接附带"在Photoshop中执行Image > Image Size，宽高调整为最近的2的幂次方"的具体步骤。

## 知识关联

数据验证管线以**资产验证工具**为直接基础：资产验证工具提供了单一检查函数（如`check_poly_count(mesh, limit)`），而数据验证管线将多个这样的工具函数组织为有序执行链，添加了阶段概念、结果聚合逻辑和错误上报格式。没有成熟的单体验证工具，就无法有效组装验证管线，因为管线的健壮性依赖每个节点函数本身的检查精度。

数据验证管线完成后，其输出结果可对接**自动化构建系统**（如Jenkins CI/CD流程），形成更大范围的资产质量控制体系。验证管线产生的结构化错误日志也可以输入到**资产质量看板**中，为技术美术主管提供团队整体资产合规率的可视化数据，例如追踪"本周角色资产首次提交通过率"这类效率指标。
