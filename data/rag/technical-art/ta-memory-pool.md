# 内存池管理

## 概述

内存池（Memory Pool）是一种预先向操作系统申请一大块连续内存、再由程序自行分配和回收的技术。与直接调用 `malloc`/`free` 或 C++ `new`/`delete` 不同，内存池绕过了系统通用分配器，将"申请大块"和"小块分发"两个动作分离，从而把单次分配的时间复杂度从 O(n)（堆扫描）降至接近 O(1)（偏移指针移动）。

在游戏引擎与技术美术工作流中，内存池最早在 1990 年代的主机游戏开发中被系统化采用。彼时 PlayStation 主机的物理 RAM 仅有 2 MB，每一帧多余的 `malloc` 调用都可能引起帧率抖动，因此开发者强制将粒子、特效对象、动画骨骼等生命周期相似的对象全部放入同类型池中管理。现代引擎如 Unreal Engine 5 仍保留 `FMemStack`、`TMemStackAllocator` 等池化接口，Unity 则通过 `UnsafeUtility.Malloc` 和 NativeArray 向技术美术暴露池化控制权。

内存池对技术美术的直接价值体现在两点：第一，消除粒子系统、UI 控件等高频创建/销毁场景中的内存碎片，使帧时间标准差缩小；第二，让内存预算可预测——池的上限即为该系统的硬性内存上限，超出时拒绝分配而非静默增长，有利于跨平台内存预算管控。

## 核心原理

### 固定块池（Fixed-Size Block Pool）

最简单也最常用的形态。池在初始化时将整块内存切割为 N 个等大小的块（Block），并用一条**空闲链表（Free List）**串联起来。分配时取链表头节点，回收时将块重新插回链表头，两个操作均为 O(1)。

关键公式：
```
总池大小 = 块大小 × 块数量 + 对齐填充
对齐填充 = (Alignment - (块大小 % Alignment)) % Alignment
```

以技术美术常见的粒子对象为例，若单个粒子结构体占 64 字节，希望支持最多 4096 个并发粒子，则池大小为 64 × 4096 = **256 KB**，完全可在帧开始时一次性锁定，避免运行时向系统堆申请。

### 变长块池与 Slab 分配

当对象尺寸不固定时，固定块池会造成大量内部碎片（Internal Fragmentation）。Slab 分配器的做法是为不同尺寸区间（如 16 B、32 B、64 B、128 B……2048 B）分别维护一个固定块池，每个尺寸档称为一个 **Slab**。分配时按请求大小向上取整到最近的 Slab 档，最大内部碎片不超过 50%（请求 33 B → 分配 64 B）。Linux 内核自 2.6 版本起内置了基于 Slab 的 `kmalloc`，Unreal 的 `TBinnedMalloc2` 也采用了类似的分档策略，档位最大覆盖到 64 KB。

### 线性/帧内存池（Linear / Frame Allocator）

帧内存池不维护空闲链表，只保存一个**偏移指针（Offset Pointer）**。分配时将指针向后移动所需字节数，回收则完全不操作——等到帧结束时，将偏移指针重置为 0，一次性释放全部内容。这种方式分配速度极快（仅一次加法与边界检查），适合每帧临时产生的渲染命令缓冲、动态顶点数据、UI 布局中间结果等生命周期严格限于单帧的数据。Unreal 的 `FMemStack` 正是此类实现，每帧压栈、帧尾弹栈，开销接近零。

### 碎片化控制策略

外部碎片（External Fragmentation）在可变块池中不可避免。技术美术常用以下三种控制手段：

1. **对象分层**：将生命周期相同的对象放入同一个池，避免长生命周期对象散布在短生命周期对象之间造成空洞。
2. **池压缩（Compaction）**：在非关键帧或加载阶段移动对象、更新引用，重新整理池的连续性。Unity 的托管堆从 2021.2 版本引入增量 GC 压缩正是此原理。
3. **双端分配（Double-Ended Stack）**：从缓冲区两端分别向中间分配，一端放长期对象，另一端放临时对象，中间的剩余空间是真正的可用量，有效减少单向分配引起的碎片空洞。

## 实际应用

**粒子系统池化**：在 Unreal Niagara 中，`FNiagaraDataBuffer` 将粒子属性数组以 SoA（Structure of Arrays）形式存储在预分配缓冲区内，最大粒子数由 `MaxCount` 参数在 Emitter 初始化时锁定，超出容量的粒子发射请求会被静默忽略而非触发动态扩容，这正是固定块池思想的直接体现。

**UI 控件对象池**：Unity UI Toolkit 和 UGUI 均建议技术美术对频繁显示/隐藏的 UI 元素（如伤害数字、技能提示）使用对象池，而非每次 Instantiate/Destroy。典型实现使用 `Stack<T>` 作为空闲链表，Get() 对应出池、Release() 对应入池，避免每秒数十次 GC Alloc 导致托管堆扩容。

**加载阶段的线性池**：在关卡流式加载期间，临时需要大量解析中间数据（JSON 节点、贴图 Mip 解码缓冲等）。用线性池分配这些中间结构，加载完成后整体重置，可将加载阶段的内存峰值降低 20%–40%，因为线性池的开销仅有一个指针，而 `malloc` 每次分配有 8–16 字节的元数据开销。

## 常见误区

**误区一：池越大越好**
技术美术有时将粒子池的 `MaxCount` 设置得极大（如 100,000）以求"保险"，但固定块池在初始化时即占用全部内存，`MaxCount = 100000`、每粒子 64 B 意味着常驻 6.1 MB，即使场景中只有 200 个粒子存活。正确做法是根据实际压测中的 P95 峰值数量设定上限，而非凭感觉放大。

**误区二：对象池解决了所有内存问题**
对象池消除的是分配/释放的时间开销和堆碎片，但池本身长期占据内存。若一个特效在过场动画中使用完毕后，其对象池仍驻留内存直到场景卸载，则等于将短暂的内存峰值变成了永久内存占用。针对此问题，应对池实现**最大闲置时间（Idle Timeout）**机制：若池连续若干帧（如 300 帧）未被使用，则归还内存给系统堆。

**误区三：线性池可以随意用于任何数据**
线性池的"重置"操作是集体释放，不支持单独释放某一对象。若将生命周期长短不一的对象（如既有单帧临时数据又有跨帧持久数据）混入同一个线性池，帧尾重置会错误释放仍在使用的持久对象，导致悬空指针（Dangling Pointer）或数据损坏。线性池只适合**生命周期完全一致且边界明确**的对象集合。

## 知识关联

内存池管理建立在**内存管理概述**中虚拟地址空间、堆与栈的区分概念之上——理解为何系统 `malloc` 存在碎片化和时间不确定性，才能理解内存池绕过它的必要性。固定块池的空闲链表结构也直接复用了堆内存中"元数据头部"的概念，只是将其内化进块本身的前几个字节。

向前延伸，内存池管理为**内存泄漏检测**提供了天然的检测边界：池的已用块数与空闲块数之和应始终等于初始块总数，若出现偏差，则意味着某次 `Release()` 调用丢失，即典型的池泄漏。在池层面检测泄漏比在系统堆层面更精准，因为池的元数据完全由开发者控制，可在每个块头部写入分配时的调用栈 Hash，以 O(N) 遍历整个池即可定位泄漏来源。
