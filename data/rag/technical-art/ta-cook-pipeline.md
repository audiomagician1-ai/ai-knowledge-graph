# 资产烘焙管线

## 概述

资产烘焙管线（Asset Baking Pipeline）是将编辑器中的源资产（Source Assets）转换为运行时可直接消费的目标格式的自动化流程。"烘焙"这一术语源于光照烘焙（Lightmap Baking）的历史命名习惯，后被泛化为所有离线预计算转换操作的统称。其核心价值在于：将高成本的数据转换、压缩和优化工作从运行时移至构建时，使游戏在目标平台上以零转换开销直接加载最终格式资产。

资产烘焙管线的概念在商业引擎中约于2000年代中期逐步成型，Unreal Engine 3的 Cooking 系统和 Unity 的 AssetBundle 构建管线是早期代表性实现。与导入管线不同，导入管线负责将外部格式（如 .fbx、.psd）转换为引擎内部编辑器格式，而烘焙管线的起点是已经存在于编辑器中的资产，终点是针对特定目标平台优化的运行时二进制格式。这一区别决定了烘焙管线必须感知目标平台的硬件能力差异。

烘焙管线的重要性体现在两个量化维度上：一是包体体积，未经烘焙优化的纹理资产在 PC 平台上体积可能达到烘焙后 BC7 压缩格式的 4–8 倍；二是加载时间，预先串行化的二进制资产相比运行时即时解析的 XML/JSON 配置文件，首次加载耗时可缩短 60% 以上。

## 核心原理

### 烘焙阶段划分与依赖图

一条完整的资产烘焙管线通常分为三个串行阶段：**收集（Gather）→ 处理（Process）→ 打包（Package）**。收集阶段通过资产引用扫描构建有向无环图（DAG），节点为资产，边为依赖关系（例如材质球引用纹理，Mesh 引用材质）。处理阶段按拓扑排序依次对每个节点执行平台特定转换。打包阶段将处理后的二进制数据按访问模式分组，写入最终的 .pak / .bundle 文件。

依赖图的正确性是烘焙管线的关键约束：若纹理 A 被材质 B 和材质 C 同时引用，烘焙系统必须保证纹理 A 只被处理一次，并在输出包中只存储一份（去重），否则会产生资产冗余膨胀。实际项目中资产重复打包是导致包体超标最常见的原因之一，通常由循环引用或孤立资产分析缺失引起。

### 平台特定的数据转换规则

烘焙管线必须维护一张**平台–格式映射表**。以纹理为例，典型的映射关系如下：

| 目标平台 | 颜色纹理格式 | 法线纹理格式 |
|----------|-------------|-------------|
| PC (DX11+) | BC7 | BC5 |
| Android (高端) | ASTC 6×6 | ASTC 4×4 |
| iOS (A8+) | ASTC 4×4 | ASTC 4×4 |
| Switch | ASTC 4×4 | BC5 |

除格式转换外，烘焙阶段还需执行 Mipmap 生成、贴图尺寸向下采样（LOD Bias）、Mesh 的 LOD 网格生成、动画曲线的关键帧压缩（如将 Float32 曲线量化为 Float16）、以及 Shader Variant 的离线编译。Shader 离线编译尤为耗时，一个中等规模项目的 Shader 变体库可达 10,000 个以上，完整重新编译可能占用烘焙总时长的 40%–60%。

### 增量烘焙与缓存机制

全量烘焙在大型项目中耗时动辄数小时，因此增量烘焙是工程实践的必要手段。增量烘焙依赖**内容哈希（Content Hash）**而非文件修改时间戳，因为时间戳在多人协作或 CI 环境中不可靠。每个资产的烘焙缓存键由以下部分组成：

```
CacheKey = Hash(源资产内容 + 烘焙配置版本 + 目标平台标识符 + 转换器版本号)
```

其中转换器版本号至关重要——当纹理压缩算法库升级时，即使源文件未变，也必须令所有相关缓存失效，否则会出现使用旧算法产生的资产混入新版本构建的问题。Unity 的 Cache Server 和 Unreal 的 Derived Data Cache（DDC）均基于此原理实现分布式烘焙缓存共享，在团队规模较大时可将单个开发者的增量烘焙时间从数十分钟压缩至数分钟。

## 实际应用

**手游项目的纹理压缩管线**：在针对 Android + iOS 双端发布的项目中，同一张 2048×2048 的 RGBA 源纹理（约 16MB 未压缩）需要分别生成 ASTC 4×4（约 2MB）和 ASTC 6×6（约 0.9MB）两个变体，并在对应平台的 .pak 文件中各自打入一份。烘焙管线需要识别纹理的用途标签（颜色图、法线图、遮罩图）来选择不同的压缩质量预设，而非对所有纹理使用统一参数。

**光照贴图烘焙的离线计算**：场景中静态物体的光照信息通过离线光线追踪（如 Enlighten 或 GPU Lightmass）计算并写入 Lightmap 纹理，这是烘焙管线中计算量最大的单一步骤。一个包含 200 个静态光源的中型场景，在 GPU Lightmass 下完整烘焙耗时约 20–40 分钟，烘焙结果直接影响运行时光照质量，无法在不重新烘焙的情况下修改光源参数。

**配置数据的序列化烘焙**：游戏中的 Excel/CSV 策划数据表在烘焙阶段被转换为 FlatBuffers 或 Protocol Buffers 的二进制格式，访问特定字段无需全表反序列化，内存访问延迟相比 JSON 可降低一个数量级。

## 常见误区

**误区一：认为烘焙管线与导入管线可以合并为一个阶段**。导入管线在资产进入引擎时执行，结果存储在编辑器可编辑的中间格式中；烘焙管线在构建发布版本时执行，结果是只读的运行时二进制。将两者合并会导致每次修改资产时都触发平台专项编译，编辑器响应速度会大幅降低。正确的做法是保持两个独立阶段，编辑器始终操作中间格式。

**误区二：增量烘焙只要检查文件时间戳就足够**。在使用版本控制系统（如 Perforce 或 Git LFS）的团队中，Sync 操作会将所有文件时间戳更新为当前时间，导致全量重新烘焙。而基于内容哈希的缓存系统中，如果源文件内容未变，即便时间戳更新也能命中缓存，跳过不必要的重新烘焙。

**误区三：烘焙管线只需处理美术资产**。实际上代码以外的所有运行时数据都应经过烘焙管线处理，包括导航网格（NavMesh）的预计算、物理碰撞体的 BVH 加速结构序列化、以及音频资产的采样率重采样和有损压缩（如 Ogg Vorbis 编码）。将这些步骤遗漏在烘焙管线之外会导致运行时首帧出现可见的延迟卡顿。

## 知识关联

资产烘焙管线的起点依赖**导入管线**（Prerequisites）所生成的编辑器中间格式——没有合规的编辑器内部表示，烘焙管线无法获得正确的源数据。导入阶段产生的元数据（如纹理的 sRGB 标记、法线图标记）会作为烘焙参数的选择依据传递到烘焙阶段。

向上拓展至**多平台管线**时，烘焙管线的平台–格式映射表需要扩展为多维配置矩阵，并引入平台特定的后处理步骤（如 Nintendo Switch 的 NCA 加密打包）。在**迭代速度优化**方向上，增量烘焙缓存的精细化分层设计（本地缓存 → 局域网缓存服务器 → 云端 CDN 缓存）是减少开发者等待时间的核心工程投入点，两者都以本概念所建立的烘焙管线基础架构为前提。
