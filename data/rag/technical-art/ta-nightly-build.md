# 夜间构建

## 概述

夜间构建（Nightly Build）是指在每天凌晨（通常在北京时间 00:00–06:00 之间的非工作时段）自动触发的完整资产编译、验证与性能基准测试流程。与开发时段的增量构建不同，夜间构建执行的是**全量构建**——即对代码仓库中所有美术资产重新导入、重新打包，并与当日提交的最新版本对齐，确保次日早晨工程师到岗时已有一份完整的可测试构建物等待使用。

这一模式最早在游戏行业由大型工作室在 2000 年代初标准化。微软开发 Windows XP 期间将夜间构建制度化，后被 EA、育碧等游戏公司引入美术管线。对于技术美术团队而言，夜间构建的意义在于将"资产质量劣化"的发现时间从数周压缩到最长 24 小时：任何在白天提交的有问题材质、破坏光照的几何体或超出多边形预算的模型，都会在次日晨会前以报告形式呈现。

夜间构建与实时 CI（持续集成）不同，它的运行成本允许执行那些白天无法承受延迟的重型任务：全平台 Shader 编译、TextureArray 合批、NavMesh 烘焙以及 GPU 帧时间回归测试。这些任务单次可能耗时 2–8 小时，只适合在无人操作的时间窗口中执行。

## 核心原理

### 触发机制与时间调度

夜间构建通常由 **Cron 表达式**驱动，标准写法为 `0 1 * * *`（每天凌晨 1:00 触发）。构建服务器（Jenkins、TeamCity 或 GitHub Actions 的 `schedule` 触发器）在该时刻拉取主分支的 HEAD 提交，锁定版本哈希，此后整个流程均基于该快照运行，与凌晨后的任何新提交隔离。版本锁定保证了报告的可追溯性：每份夜间报告都对应唯一一个 Git commit SHA。

### 三阶段流水线结构

夜间构建的完整流程分为三个串行阶段：

**阶段一：资产构建（Asset Build）**  
在无头（headless）引擎实例中批量导入并编译所有资产。以 Unity 为例，构建脚本调用 `BuildPipeline.BuildAssetBundles()` 或 Addressables 的命令行接口，输出各平台（PC/iOS/Android/主机）的 AssetBundle 或 PAK 文件。Unreal 则通过 `UAT BuildCookRun` 命令完成 Cook 流程。此阶段产生的二进制输出存储于构建缓存服务器（通常为 Artifactory 或 S3），供后续阶段消费。

**阶段二：资产验证（Asset Validation）**  
调用预设的验证规则集对构建物进行自动审查。典型规则包括：
- 纹理分辨率不超过 2048×2048（移动平台），否则标记警告
- 网格 LOD0 面数不超过项目约定上限（例如角色 15,000 三角面）
- 材质引用不存在悬空指针（Broken Reference）
- 所有音频资产采样率统一为 44,100 Hz 或 22,050 Hz

验证脚本输出结构化的 JSON 报告，每条规则违反记录资产路径、违规值与阈值，便于自动生成按责任人分类的 Jira 工单或 Slack 通知。

**阶段三：性能回归测试（Performance Regression Test）**  
在专用硬件（锁定 GPU 驱动版本的测试机）上运行自动化场景，采集帧时间、Draw Call 数、显存占用等指标，并与上一次夜间构建的**基准快照（Baseline Snapshot）**进行 delta 比较。若某指标的相对退化超过阈值（例如帧时间增长 >5%），则该阶段标记为失败（FAIL），并附带 RenderDoc 或 PIX 捕获的帧文件供次日排查。

### 失败处理与报告分发

夜间构建的失败不会阻塞任何人的本地开发（这与阻塞式 CI 不同），但会触发**责任人归因**机制：构建系统通过 `git blame` 与提交时间戳，将当日导致失败的提交列表发送给对应提交者。报告通常在次日 08:30 前通过邮件或即时通讯工具推送，确保团队在晨会前已掌握当前构建状态。

## 实际应用

**开放世界游戏的地形验证**  
某 AA 工作室在其开放世界项目中，夜间构建负责对 400+ 个地形分块（Terrain Chunk）执行 NavMesh 烘焙并比较烘焙结果的可通行区域面积。若某分块因美术修改导致可通行面积减少超过 10%，则自动标记为潜在阻挡性修改，通知关卡美术与技术美术负责人。这一机制在项目中期发现了 3 次因 Mesh Collider 设置错误导致的大范围寻路失效，而这些问题在手动测试中被遗漏了数天。

**多平台 Shader 编译监控**  
夜间构建在 PC、PS5、Switch 三个平台上分别编译全量 Shader 变体（Shader Variant），统计编译耗时与变体总数。当某次材质图（Material Graph）改动意外触发变体爆炸（Variant Explosion）——例如变体数从 1,200 增长至 14,000——夜间报告的编译时间异常会在次日晨会前暴露该问题，避免将性能炸弹带入里程碑构建。

## 常见误区

**误区一：夜间构建等同于每日构建**  
夜间构建强调在非工作时段执行**全量**重型任务，而"每日构建"在某些团队语境中可能指代增量构建或仅包含代码编译。混淆二者会导致团队误以为已有完整的性能回归保障，实际上仅有代码层面的编译验证。判断标准是：构建是否包含资产导入（Asset Import）与 GPU 指标采集，这两者缺一不可才能称为完整的夜间构建。

**误区二：夜间构建报告次日再看无妨**  
夜间报告的最大价值在于**24 小时内响应**。若团队习惯在报告累积后每周集中处理，则失败的修复与最初的提交之间会间隔数次后续提交，大幅增加调试成本。有效的团队规范是：责任人在收到报告后的当天工作时段内确认或修复，否则该问题进入缺陷跟踪系统并设置优先级。

**误区三：性能基准机器可以与开发机共用**  
回归测试的核心价值来自于指标的**可重复性**。若测试机同时用于其他任务（如白天的手动测试或开发者的临时调试），GPU 驱动更新、后台进程或显存碎片化会污染帧时间数据，导致误报（False Positive）或漏报（False Negative）。正确做法是为夜间性能测试保留专用硬件，锁定驱动版本，并禁止人工干预。

## 知识关联

夜间构建建立在**美术 CI/CD** 的基础上：美术 CI/CD 提供了资产提交时的基础校验钩子（如文件大小检查、命名规范验证），而夜间构建则在此之上增加了单次无法完成的全量验证与性能基准层。没有 CI/CD 建立的版本控制规范和自动化触发基础设施，夜间构建的脚本将缺少可靠的触发点和版本快照机制。

向前看，夜间构建是**端到端自动化管线**的核心组成节点。端到端管线将夜间构建的产物（验证通过的资产包、性能报告）进一步接入自动化 QA 测试、发布候选版本生成和玩家数据分析闭环，形成从美术提交到运营反馈的完整数据链路。夜间构建在这一链路中扮演的是"质量关卡"角色——只有通过夜间构建三阶段验证的版本，才有资格进入端到端管线的后续环节。
