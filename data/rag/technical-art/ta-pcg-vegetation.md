# 程序化植被

## 概述

程序化植被（Procedural Vegetation）是指通过算法规则自动生成树木、灌木、草地、苔藓等植物资产及其在场景中分布的技术体系。与手工摆放每一棵树的传统工作流相比，程序化植被能够在数分钟内生成覆盖数平方公里的森林，并保证生态合理性——例如针叶树倾向于出现在海拔较高、坡度较陡的区域，而芦苇则自动聚集在水体边缘。

这一技术的理论根基之一是植物学家阿里斯蒂德·林登迈耶（Aristid Lindenmayer）于1968年提出的 **L-System**（Lindenmayer System），最初用于模拟藻类细胞分裂的数学模型，后被计算机图形学引入用于生成树枝、叶脉等分形结构。商业工具方面，**SpeedTree** 自2002年发布以来成为游戏与影视行业的标准植被建模软件，《刺客信条：起源》《荒野大镖客：救赎2》等大型游戏均使用它生成了数十种各异的树种资产。

对技术美术师而言，掌握程序化植被意味着能够在保持美术可控性的前提下，将植被系统与地形高度图、坡度图、湿度图等数据驱动联动，从而实现"地形决定生态"的自动化管线，极大减少手工重复劳动。

## 核心原理

### L-System 的文法规则与树形生成

L-System 本质上是一套**字符串重写文法**。定义一个初始公理（Axiom）和若干产生式规则（Production Rules），每次迭代将字符串中的符号替换展开，最终用"海龟图形学"（Turtle Graphics）将字符串解释为三维几何体。

一个经典的分支规则示例：
- **公理**：`F`
- **规则**：`F → F[+F]F[-F]F`
- **符号含义**：`F` = 向前绘制一段枝干，`+` = 向左旋转25°，`-` = 向右旋转25°，`[` = 压栈（保存当前位置），`]` = 弹栈（返回父节点位置）

经过3次迭代后，该规则可产生外形酷似灌木的分支结构。通过引入**随机参数**（如旋转角度在22°~28°之间随机浮动），可以打破完全对称带来的人工感。L-System 的迭代深度（Recursion Depth）通常控制在4~7级，超过7级后几何体面数会指数级爆炸。

### SpeedTree 的节点化建模流程

SpeedTree 将植被结构分解为**主干（Trunk）→ 主枝（Primary Branch）→ 次枝（Secondary Branch）→ 末梢（Twig）→ 叶片（Leaf/Frond）** 五层节点图，每层节点包含独立的生长参数，例如：
- **弯曲度（Bend）**：主干在重力方向上的弯曲幅度，用于模拟年龄感
- **碰撞球（Collision Spheres）**：防止枝条穿入树冠内部
- **LOD 衰减设置**：可为每层节点单独指定从3D网格切换为Billboard的距离阈值

SpeedTree 生成的树木默认输出包含顶点色通道，其中**红色通道（R）存储枝条末端到根部的归一化距离**，供引擎中的风力着色器采样，实现末梢颤动幅度大、主干几乎不动的真实风效。

### 散布规则与密度控制

植被散布（Scattering）的核心是将"哪里长什么"的生态逻辑转化为可量化的权重图叠加计算。常见的驱动参数包括：

| 驱动图层 | 对应生态逻辑 |
|----------|--------------|
| 地形高度图 | 高山树线以上禁止出现乔木 |
| 坡度图（0°~90°） | 坡度 > 45° 的峭壁无法保留土壤，植被密度趋近0 |
| 湿度/流水图 | 河道两侧5米范围内优先生成柳树或芦苇 |
| 噪声遮罩（Perlin/Worley） | 模拟植物群落的自然斑块分布 |

**密度控制**通常以"每公顷株数"为单位进行调参，温带落叶林的合理密度约为200~400株/公顷，过密会导致 GPU Instance Draw Call 超出预算。在虚幻引擎的 **Procedural Foliage Spawner** 中，可为每种植被类型设置 `Collision Radius`（互斥半径，单位：厘米），以防止两棵大树紧贴生长，同时通过 `Shade Spawns In Shadow` 控制耐阴植物是否只在树荫下出现。

## 实际应用

**《地平线：零之曙光》的生物群系系统**：Guerrilla Games 在该游戏中构建了基于地形分析的植被生成管线，系统读取地形的坡向（Aspect，即山坡朝向北/南）数据，南向坡（北半球受阳光更多）自动生成干旱草本植被，北向坡生成苔藓与蕨类，实现了与实际生态学吻合的视觉分布。

**UE5 实时植被刷新**：在虚幻引擎5中结合 PCG（Procedural Content Generation）框架，可以实现"玩家挖掘地形后，被破坏区域的植被在运行时实时更新"——PCG Graph 在检测到地形修改事件后重新采样高度图并重新散布植被，整个流程无需重新烘焙关卡。

**SpeedTree 在《阿凡达》特效中的应用**：Weta Digital 将 SpeedTree 的 L-System 参数与气候模拟数据结合，生成潘多拉星球上具有顶端发光效果的荧光植被，其分支角度参数被调整到80°以上以获得独特的外星植物形态。

## 常见误区

**误区一：L-System 可以直接用于游戏引擎实时生成树木**
L-System 是离线建模工具而非实时算法。其迭代展开过程计算密集，且结果为非结构化几何体，不适合实时LOD管理。游戏引擎中看到的"程序化树木"通常是 SpeedTree 等工具**离线预生成后导入**的静态/骨骼网格体，运行时不再重新计算L-System规则。

**误区二：提高散布密度只是视觉问题，不影响性能**
植被密度直接影响 GPU 的 Instance Rendering 批次数量和 CPU 的遮挡剔除（Occlusion Culling）消耗。以虚幻引擎为例，当单个 `FoliageType` 的实例数量超过约 **100万个**时，剔除线程的遍历开销会明显拖慢帧率，即便每棵草的面数极低。密度控制应与性能预算同步制定，而非纯粹由美术主观决定。

**误区三：SpeedTree 生成的风效果来自骨骼动画**
SpeedTree 的风效是通过**顶点着色器中的数学函数**实现的，不依赖任何骨骼或蒙皮，这正是它能支持大规模实例化渲染（GPU Instancing）的关键——每个实例无需独立的动画状态机，仅需全局风力参数（风向、风速、阵风频率）即可驱动整片森林统一运动，同时通过相位偏移（Phase Offset，存储于顶点位置哈希值中）避免所有树同步摆动的机器感。

## 知识关联

**前置概念——程序化生成概述**：理解程序化植被需要先掌握噪声函数（尤其是Perlin Noise和Worley Noise）如何生成连续随机分布，以及种子（Seed）值如何保证生成结果的可复现性，这两者是植被散布权重图计算的基础工具。

**后续概念——散布系统**：程序化植被中的密度图与互斥半径规则，直接演进为更通用的散布系统（Scatter System），后者将同样的逻辑扩展到岩石、建筑废墟、道具摆放等非植被资产，并引入碰撞检测与物理模拟约束。

**后续概念——生态群落系统**：当散布规则从单种植物扩展到多物种之间的竞争、共生、阴蔽等关系建模时，即进入生态群落系统的范畴。该系统在植被密度图的基础上叠加物种间影响矩阵，例如高大乔木的阴影范围会压制其正下方同种幼苗的生长权重，从而模拟真实森林中的光竞争现象。
