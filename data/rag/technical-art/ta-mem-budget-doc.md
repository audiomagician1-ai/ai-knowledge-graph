# 内存预算文档

## 概述

内存预算文档（Memory Budget Document）是技术美术工作流中用于记录和管理游戏项目各类资产内存占用上限的结构化表格或电子表格。它规定了每个资产类别——如角色模型、场景贴图、音频文件、特效粒子系统——在目标平台上可合法占用的内存字节数上限，并追踪当前实际占用与预算上限之间的差值（余量）。

内存预算文档的概念随着主机游戏开发在2000年代的规模化而系统化。早期PlayStation 2项目中，开发团队常依赖口头约定来分配32MB总RAM，导致后期冲突频繁。后来，Naughty Dog、DICE等工作室将内存预算正式化为共享电子表格，逐渐演变为今天被行业广泛采用的标准实践。在移动平台，iOS设备的内存警告（Memory Warning）机制会在占用超过约150–200MB时触发崩溃，使预算文档成为规避上架风险的必要工具。

内存预算文档的价值在于它将抽象的"平台内存限制"转化为各部门可执行的具体配额。美术师在制作贴图时只需对照文档中规定的"UI贴图≤4MB"或"场景主贴图集≤64MB"等具体数字，便可独立做出符合项目约束的决策，无需每次都向技术美术人员咨询。

## 核心原理

### 总预算的拆分结构

内存预算文档从目标平台的物理内存容量出发，逐层向下分配。以PS5为例，其系统可用RAM约为12–13.5GB（16GB总量扣除系统保留），内存预算文档需首先在以下一级分类中分配这笔总量：

| 一级分类 | 典型占比示例 |
|---------|------------|
| 游戏代码与引擎 | ~1.5 GB |
| 渲染资产（贴图+Mesh） | ~4 GB |
| 音频 | ~1 GB |
| 流送缓冲（Streaming Buffer） | ~2 GB |
| 动画数据 | ~800 MB |
| 其他（UI、脚本、物理） | 剩余 |

各一级分类再向下拆分为二级预算行，例如"渲染资产"下分为"角色贴图池"、"场景贴图池"、"特效贴图池"等，每行均记录**已分配预算**、**当前实际占用**、**余量**三列数值。

### 预算行的关键字段

一份合格的内存预算文档中，每条预算行至少包含以下字段：

- **资产类别名称**：唯一标识符，如 `ENV_Textures_Streaming`
- **目标平台**：同一文档通常并排列出 PC / PS5 / Xbox Series X / Switch 四列
- **预算上限（Budget）**：单位通常为 MB，由技术美术与项目总监商定
- **当前占用（Actual）**：从内存报告工具（如 Unreal 的 `memreport` 命令或 Unity 的 Memory Profiler）导出的实测数值
- **余量（Headroom）**：Budget − Actual，负数时以红色高亮标注
- **负责人**：该类别的美术 Lead 或技术美术姓名
- **最后更新日期**：确保数据时效性

### 颜色编码与警戒阈值

内存预算文档通常采用三色警戒体系来直观反映健康状态：绿色表示余量大于预算的15%，黄色表示余量在5%–15%之间（需关注），红色表示余量低于5%或为负数（必须立即优化）。15%这一安全边界并非随意设定——它预留了项目后期 bug 修复、DLC 内容或平台认证补丁可能引入的额外内存开销。

## 实际应用

**建立文档的时机**：内存预算文档应在Pre-Production（预生产）阶段与GDD（游戏设计文档）同步创建，而非等到Alpha版本出现内存超标问题时才补救。此时虽然实际占用数据为零，但平台约束已知，可先填入预算上限列，建立分配框架。

**与内存报告的协同**：每当技术美术人员通过 `stat memory`（Unreal）或 `Profiler.BeginSample()`（Unity）等工具生成新的内存报告后，报告中的实测数值应立即更新到预算文档的"当前占用"列。建议将这一更新动作纳入每周的里程碑检查会议，使预算文档始终反映最新状态。

**任天堂Switch的特殊处理**：Switch的可用RAM约为4GB（游戏可用约3.2GB），在同一份跨平台预算文档中，Switch列的预算数字通常比PS5列低60%–70%。美术团队需维护专用的Switch贴图降级规则，并在预算文档旁附注"Switch使用ETC2压缩，贴图尺寸上限减半"等具体说明，防止美术师套用PC参数错误制作资产。

**里程碑门禁**：部分工作室将内存预算文档的"所有行均为绿色"设为Alpha/Beta里程碑的硬性通过条件（Gate），在版本签批流程中与帧率指标并列检查，使内存健康成为可量化的发布标准。

## 常见误区

**误区一：把预算上限设置为平台总内存的100%**
初学者常将角色贴图预算设为"剩余所有内存"，忽略了引擎自身的元数据、资产加载队列、GPU驱动保留区等隐性开销。Unreal Engine 5的Nanite和Lumen在极端场景下自身可消耗400–600MB，若未在预算文档中单独为引擎特性留出专项行，后期往往出现莫名的内存超标。

**误区二：一份文档覆盖所有场景**
内存占用因游戏场景的复杂度差异可达2–3倍。正确做法是在预算文档中为"最坏情况场景"（Worst Case Level，通常是主城或Boss战场景）单独设立一套预算行，与"平均场景"并列，以最坏情况行作为实际约束基准。用平均占用来通过里程碑检查、却在最终关卡崩溃是一个典型的错误模式。

**误区三：预算文档建立后不再更新**
内存预算文档的时效性至关重要。若文档中记录的"角色贴图占用：850MB"来自三个月前的数据，而此后美术团队新增了12个NPC角色，该数据已完全失效。建议在文档顶部添加"数据有效期：7天"的醒目标注，超期后以灰色显示所有数值，强制提醒团队更新。

## 知识关联

**前置概念——内存报告**：内存预算文档依赖内存报告提供"当前占用"列的原始数据。如果不能正确读取和解析内存报告输出（例如区分 `RHIMemory` 与 `AudioAllocated` 等分类），就无法将数据准确填入预算文档的对应行。内存报告是预算文档的数据来源，而预算文档是对这些数据施加约束和决策的管理工具。

**横向关联——贴图压缩与LOD策略**：当预算文档的某一行出现红色预警时，技术美术人员的常规优化手段是调整贴图压缩格式（如从BC3改为BC7节省内存，或从未压缩改为ASTC 4x4）以及降低LOD层级的多边形数量。预算文档本身不规定如何优化，但它精确指出了哪个类别需要优化，将方向与幅度量化呈现，使优化工作具有明确目标而非盲目压缩。
