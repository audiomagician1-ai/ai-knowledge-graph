# 版本兼容性

## 概述

版本兼容性（Version Compatibility）是指游戏客户端与服务端在不同版本号共存时，双方能够正确解析通信协议、读取存档数据并维持正常游戏逻辑的能力。在手游直播运营场景中，由于玩家更新行为存在时间差——部分玩家在新版本上线后72小时内仍未更新——服务端必须同时服务新旧两种协议格式的客户端。

版本兼容性问题在2012年前后随着智能手机游戏的大规模普及而变得突出。早期PC端游可通过强制补丁更新绕过这一问题，但App Store与Google Play的审核周期（通常为1至7天）使得客户端版本与服务端版本之间出现结构性时间差，迫使开发团队必须设计专门的兼容层。

在版本更新节奏管理中，版本兼容性直接决定了强制更新（强更）的触发时机与弱更新（弱更）的容忍窗口。设计得当的兼容策略可以让99%的玩家无感知地过渡到新版本，而设计失误则会导致大规模闪退或数据错乱，引发玩家流失和退款投诉。

## 核心原理

### 协议版本号与向前/向后兼容

协议兼容性分两个方向：**向后兼容（Backward Compatibility）**指新服务端能读懂旧客户端发来的旧格式协议；**向前兼容（Forward Compatibility）**指旧客户端能忽略新服务端返回的新字段而不崩溃。在Protobuf等序列化方案中，字段编号（Field Number）一旦分配便不可复用，正是为了保证向后兼容性。例如，若服务端在v2.3.0新增了一个`battle_pass_level`字段（编号42），v2.2.x的旧客户端在反序列化时会自动跳过该未知字段，实现向前兼容。

通用做法是在每个网络请求头中携带`client_version`字段，格式通常为`Major.Minor.Patch`三段式语义化版本（SemVer）。服务端路由层根据Major版本号决定走哪套协议处理链，Minor版本号差异通常通过字段默认值兼容，Patch版本号差异则不做协议区分。

### 强更与弱更策略

**强制更新（强更）**：服务端检测到客户端版本低于`min_supported_version`时，返回错误码（通常为HTTP 426或自定义错误码如`ERR_VERSION_TOO_OLD`），客户端收到后弹出强制更新弹窗，玩家无法继续游戏。强更适用于以下场景：协议发生不兼容性破坏（Breaking Change）、安全漏洞修复、核心数值体系重构。

**弱更新（弱更）**：客户端启动时检测到新版本可用，弹出可跳过的更新提示，玩家可选择稍后更新并继续游戏。弱更的前提是新旧版本协议完全兼容，主要用于UI优化、新增非核心内容、文案修正等场景。弱更窗口期通常设置为7至14天，到期后自动升级为强更。

判断触发强更的核心公式为：

```
current_version < min_supported_version → 强更
min_supported_version ≤ current_version < latest_version → 弱更
current_version = latest_version → 无操作
```

其中`min_supported_version`由研发团队在每次发版前评估协议破坏程度后手动配置于后台运营系统。

### 数据迁移与存档兼容

当新版本引入存档结构变更时（如数据库表新增列、JSON存档新增字段），需要执行**数据迁移（Data Migration）**。迁移策略分为：

- **懒迁移（Lazy Migration）**：玩家首次登录新版本时触发单用户数据升级，适用于改动较小且玩家基数大的情况，可分散服务器压力。
- **批量迁移（Batch Migration）**：在服务维护期间对全量数据执行升级脚本，适用于结构性重构，要求脚本执行时间在停服窗口（通常2至4小时）内完成。

数据迁移脚本必须满足**幂等性**（Idempotency）：重复执行同一脚本，结果与执行一次相同。这一特性在服务中断后重启场景下至关重要，可防止数据被双重处理。同时需要保留版本号字段（如`save_data_version: 3`），使服务端在读取存档时知道该用哪套反序列化逻辑。

## 实际应用

**案例一：《原神》的资源包分离更新**
《原神》采用"小包体客户端+远端资源热更新"架构，在不触发应用商店审核的前提下更新游戏内容。新版本角色、地图资源通过CDN下发，客户端在进入游戏前完成资源校验（MD5哈希对比）。这种方式本质上是一种弱更模式，仅当核心逻辑代码（Unity运行时层）发生变更时才触发应用商店强更。

**案例二：多服务器版本共存（灰度发布）**
在灰度发布场景中，v2.5.0服务端与v2.4.0服务端同时运行，流量网关根据玩家UID尾号将5%的用户路由至新版本。此时v2.5.0客户端与v2.4.0客户端必须都能连接各自对应的后端，要求双版本协议在灰度窗口期（通常3至5天）内保持完全兼容，数据库表结构不得出现破坏性变更。

**案例三：节日活动的向前兼容陷阱**
某卡牌游戏在春节活动中新增了`festival_bonus`字段，v3.1.0服务端向v3.0.x旧客户端返回含该字段的奖励数据。由于旧客户端未做未知字段容错，导致解析异常，造成约8%的未更新用户无法领取奖励。根本原因是JSON解析器使用了严格模式（strict mode），这一教训推动行业广泛转向Protobuf的宽松字段解析策略。

## 常见误区

**误区一："弱更不需要考虑兼容性"**
许多初级研发认为弱更只是"可选的"，因此随意修改协议。实际上弱更期间（7至14天），线上同时存在新旧两个版本的客户端，任何导致旧客户端崩溃的协议改动都会放大为大规模事故。弱更的前提恰恰是最严格的兼容性保证。

**误区二："数据迁移可以在不停服的情况下随时执行"**
在线数据迁移（Online Migration）需要特殊的双写策略：同时写入旧结构和新结构，待读流量完全切换至新结构后再删除旧结构，整个过程至少需要两个版本的迭代周期（通常4至8周）。若直接在高峰期执行全量迁移SQL，行级锁会导致数据库性能骤降，触发服务雪崩。

**误区三："Major版本号升级就必须强更"**
Major版本号仅代表功能上的重大更新，是否触发强更取决于`min_supported_version`的配置，而非版本号大小。v3.0.0完全可以对v2.9.x客户端保持兼容并允许弱更，只要协议层没有Breaking Change即可。将版本号升级与强更机制混淆，会导致不必要的玩家流失。

## 知识关联

**与版本回滚的关系**：版本回滚是版本兼容性设计失败后的应急手段。若新版本协议与旧版本完全不兼容，回滚时还需同步回滚数据库结构，这正是数据迁移必须支持反向迁移脚本（Rollback Script）的原因。幂等性设计在回滚场景中同样适用。

**通向公测/先行服的桥梁**：先行服（PBE/PTR服）是一个刻意制造版本分裂的环境：先行服运行v_next版本，正式服运行v_current版本。两服之间的账号数据同步涉及跨版本存档读取，正是版本兼容性在多服架构下的直接应用。公测阶段玩家规模大幅扩大，任何兼容性缺陷的影响范围都会成倍放大，因此公测前必须基于先行服的兼容性测试结果冻结`min_supported_version`配置。
