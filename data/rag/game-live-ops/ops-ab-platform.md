# 实验平台

## 概述

实验平台（Experiment Platform）是游戏Live Ops中专门用于管理A/B测试全生命周期的基础设施，将**分流（Traffic Splitting）、曝光记录（Exposure Logging）、指标计算（Metric Computation）和统计分析（Statistical Analysis）**四个模块集成为一个统一服务。与散落在各系统中的零散实验逻辑不同，实验平台通过中心化配置让游戏策划或产品经理无需编写代码就能发起一个完整的A/B实验。

实验平台的概念最早由大型互联网公司的工程实践演化而来。微软在2009年发表的论文《Online Controlled Experiments at Large Scale》首次系统描述了工业级实验平台的架构要求，随后Google、Netflix等公司分别建立了自己的实验基础设施。游戏行业的实验平台相比通用互联网平台有一个独特挑战：玩家行为数据（击杀数、在线时长、付费金额）的方差远高于网页点击率，导致实验需要更长的运行周期或更大的样本量才能达到统计显著性。

游戏实验平台之所以不能简单套用一个随机数函数来替代，根本原因在于实验的四个组件必须严格联动：分流决策必须在用户首次接触实验时就固定下来（称为"粘性分流"），曝光日志必须精确记录哪些用户真正触发了实验逻辑（而非只是被分配了实验组），指标系统需要在实验维度而非全局维度上聚合数据，最终分析报告需要对多重比较（Multiple Comparisons）问题进行校正。

## 核心原理

### 分流模块：哈希一致性分配

分流模块负责将玩家稳定地分配到对照组（Control）或若干实验组（Variant）。最常见的实现方式是对 `hash(user_id + experiment_id) % 10000` 取模，将结果映射到百分比桶（Bucket）上。例如，一个50%流量的实验会将桶号0–4999分配给对照组，5000–9999分配给实验组。

**关键设计点是实验盐值（Salt）**：不同实验必须使用不同的 salt（通常就是 experiment_id），否则两个同时运行的实验会产生用户重叠模式，导致相关性污染。正确的多实验并行架构允许同一玩家同时参与多个正交实验（Orthogonal Experiments），每个实验独立决定该玩家所属的组，互不干扰。

### 曝光记录：区分分配与触发

实验平台中"被分配到实验组"和"真正被曝光到实验变量"是两件不同的事。假设一个实验修改了游戏内商店的第三个位置的道具，若某个玩家本次会话从未打开商店，他虽然被分配到了实验组，但从未真正接触实验变量，将他计入分析会**稀释（Dilution）**真实效果。

成熟的实验平台要求在代码层面明确调用曝光埋点（Exposure Event），例如：

```
experiment_sdk.log_exposure(user_id="u123", experiment_id="shop_item_layout_v2")
```

这条日志记录的时间戳成为该玩家在本次实验中数据统计的**起点**，只有曝光后发生的行为才被纳入指标计算，消除了先验行为差异对结果的干扰。

### 指标模块：实验粒度聚合与方差估算

实验平台的指标层不是简单地对数据仓库跑 GROUP BY。标准实验分析要求以**用户为单位（User-level）**计算指标，而非以事件为单位。例如，计算实验组的ARPU时，正确做法是先算每个用户在实验期间的总付费，再对用户求均值，而非直接用实验组总收入除以总事件数。

方差估算方面，实验平台通常内置 Delta Method 或 Bootstrap 来处理比率型指标（如付费转化率 = 付费用户数 / 曝光用户数）的标准误计算。对于游戏中高度偏斜的付费数据（少数鲸鱼用户贡献90%收入），平台还需支持 **CUPED（Controlled-experiment Using Pre-Experiment Data）**，利用实验前的历史数据作为协变量来降低方差，理论上可将所需样本量减少30%–70%。

### 分析模块：统计检验与多重比较校正

分析模块对每个指标执行双样本 t 检验（连续型指标）或 Z 检验（比率型指标），计算 p 值与置信区间。当一个实验同时监控20个指标时，在 α=0.05 的显著性水平下，纯随机情况下期望出现 20×0.05=1 个假阳性（False Positive）。实验平台通过内置 **Bonferroni 校正**或 **Benjamini-Hochberg FDR 控制**来减少此类问题，但代价是降低了对单个真实效果的检测灵敏度，因此平台通常要求在实验设计阶段就声明**主指标（Primary Metric）**与**护栏指标（Guardrail Metric）**，分别适用不同的显著性阈值。

## 实际应用

**战斗通行证定价实验**：某手游发布了新赛季战斗通行证，通过实验平台将服务器玩家随机分为三组：对照组保持原价30元，实验组A定价25元，实验组B定价35元。平台自动完成分流（使用 user_id + "bp_price_s12" 作为哈希键）、曝光记录（玩家进入商店页面时触发）以及指标聚合（7天内通行证购买率、总ARPU、其他付费行为是否受影响）。实验运行14天后收集到各组约8万名曝光用户，分析模块报告25元组购买率提升22%，但ARPU仅提升4%，护栏指标（次月留存率）无显著差异，平台据此自动计算建议的最优价格区间。

**新手引导流程优化**：实验平台可以在玩家首次注册后立即触发分流，对比引导流程A（强制教程，约8分钟）与流程B（可跳过教程，约3分钟）。曝光事件在教程开始时记录，主指标为第7日留存率（D7 Retention）。由于新注册用户每天数量有限，平台的样本量计算器会提示该实验需要约21天才能在 α=0.05、统计功效（Power）80%的条件下检测到留存率2个百分点的差异。

## 常见误区

**误区一：以"被分配"代替"被曝光"来计算实验指标**

许多团队在没有成熟实验平台时，直接以分流分配结果作为分析人群，把从未接触到实验变量的玩家也算进去。这会系统性地低估实验效果——假设只有40%的分配用户实际触发了实验功能，真实效果10%的提升在分析中只会显现为约4%，可能导致有价值的功能被错误放弃。实验平台强制区分这两个事件正是为了避免此问题。

**误区二：看到p值<0.05就立刻停止实验（偷窥问题）**

实验平台的看板每天都在更新指标，团队很容易在中途发现某天p值恰好低于0.05就匆忙结案。这种**偷窥（Peeking）**行为会将实际假阳性率从5%膨胀到超过30%（在连续监控的情况下）。成熟的实验平台通过实现**序贯检验（Sequential Testing）**或 **Always-Valid Inference** 方法（如 mSPRT）来允许在任意时刻查看结果而不膨胀错误率，是区分生产级平台与简单统计工具的关键功能之一。

**误区三：认为实验平台只需要一套哈希函数**

部分工程师认为只要有稳定的哈希分流逻辑就搭建好了实验平台。实际上，缺失曝光日志系统会导致稀释问题，缺失用户级指标聚合会导致独立性假设破坏，缺失多重比较校正会导致决策错误率失控。这三个缺失各自独立地会导致实验结论不可信，而补全这三个模块恰好是实验平台相对于临时脚本的核心价值所在。

## 知识关联

**与UI/UX实验的关系**：UI/UX实验是实验平台承载的典型实验类型，其特点是曝光事件简单（页面渲染即曝光）、样本量充足、实验周期短（通常3–7天）。UI/UX实验的实践让团队熟悉了分流配置和指标查看的操作流程，但当游戏实验扩展到定价、平衡性、核心玩法改动时，就需要实验平台提供CUPED降方差、序贯检验等高级能力。

**通往分群分析（Segmented Analysis）**：实验平台产出的曝光日志与用户属性（设备类型、付费等级、游龄）天然结合，可以对实验结果进行异质性分析（Heterogeneous Treatment Effect），即检验实验效果在不同玩家群体中是否存在显著差异。这种分群分析能力依赖实验平台提供的干净曝光人群作为基础，否则跨维度切割数据时会引入选择偏差。
