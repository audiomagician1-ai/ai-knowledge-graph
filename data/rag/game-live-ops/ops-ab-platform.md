# 实验平台

## 概述

实验平台（Experiment Platform）是游戏运营中专门用于管理和执行A/B测试全流程的技术基础设施，将用户分流（Traffic Splitting）、曝光记录（Exposure Logging）、指标计算（Metric Computation）和统计分析（Statistical Analysis）四个环节整合在同一套系统内，消除各环节数据孤岛问题。

该类平台的早期代表是Google在2010年发布的内部系统Overlapping Experiment Infrastructure（OEI），它首次提出了"正交分层"架构，允许同一用户在不同实验维度上同时参与多个测试。游戏行业基于此演进出针对高并发、低延迟场景的专用版本——因为玩家进入游戏大厅时，分流决策必须在50毫秒以内完成，否则会造成可感知的加载延迟。

在游戏Live-Ops场景下，实验平台的价值在于将"改动一个数值是否值得上线"这一决策从经验判断转变为数据驱动：一次典型的装备掉落率实验，可以在72小时内用10%的流量测出该改动对次日留存率（D1 Retention）和付费转化率（Day-7 IAP Conversion）的影响，从而决定是否全量推送。

## 核心原理

### 分流引擎：哈希取模与正交分层

实验平台的分流引擎通过对用户唯一标识符（如Device ID或User ID）进行一致性哈希（Consistent Hashing）来分配流量。最常见的公式为：

```
bucket = Hash(user_id + experiment_salt) mod N
```

其中 `N` 为桶总数（通常为1000或10000），`experiment_salt` 是每个实验独有的盐值，确保不同实验之间的用户分配相互独立，即"正交性"。正交分层机制允许同一游戏同时运行50+个实验而不发生流量干扰，这是游戏运营高频迭代的基础能力。

### 曝光记录：Assignment与Exposure的区分

实验平台严格区分"分配（Assignment）"和"曝光（Exposure）"两个事件。Assignment是系统决定某用户属于实验组的时刻，而Exposure是用户实际触发实验变体的时刻——例如玩家打开了新版商城界面。只有产生曝光的用户才计入实验分析，这一设计将"非触达污染"从数据中剔除。若将所有Assignment用户计入分析，会稀释实验效应，导致检测功效（Statistical Power）下降，通常需要多投入30%~40%的实验时间才能达到同等置信度。

### 指标层：护栏指标与核心指标的双轨制

实验平台内置指标体系分为两类：核心实验指标（Primary Metrics）和护栏指标（Guardrail Metrics）。核心指标是每次实验主动优化的目标，例如"7日付费转化率提升5%"；护栏指标是全局健康指标，例如"崩溃率不超过0.5%""次日留存率不下降超过2个百分点"。平台自动对护栏指标执行双尾t检验（Two-Tailed T-Test），一旦某实验组在护栏指标上触发告警，系统自动暂停实验并推送通知——这一机制在2021年Riot Games公开的实验基础设施分享中被明确提及为游戏平台的必备安全网。

### 统计分析模块：序贯测试与固定时域测试

平台提供两种统计检验模式。**固定时域测试（Fixed-Horizon Testing）**要求在实验开始前确定样本量（通常使用公式 `n = (Z_α/2 + Z_β)² × 2σ² / δ²` 计算），实验运行至预定时长后一次性读取结果，适合版本更新前的计划性实验。**序贯测试（Sequential Testing / Always-Valid Inference）**允许运营人员每天查看数据而不引入α膨胀（Alpha Inflation）问题，适合游戏活动期间需要快速决策的紧急实验，代价是所需样本量增加约10%~20%。

## 实际应用

**场景1：新手引导流程改版实验**
某手游将新手引导从线性流程改为分支式流程，在实验平台上设置3个变体（Control, Branch-Light, Branch-Deep），各分配20%流量，剩余40%为保留缓冲。平台同时监控D1留存、第3关通关率和首次付费时间三项指标。实验运行5天后，Branch-Light组D1留存提升3.2个百分点（p=0.03），平台自动标记为统计显著，运营团队据此决定全量。

**场景2：限时活动价格弹性测试**
在为期7天的节日活动中，实验平台配合功能开关（Feature Flag）对礼包价格设置$4.99、$6.99、$9.99三档，仅在活动开启后激活实验分组。平台的曝光系统只记录进入活动商城的玩家，有效过滤了未参与活动的休眠用户，最终分析样本纯净度显著提升。

**场景3：跨平台设备ID合并**
当同一玩家在iOS和Android双端登录时，实验平台通过User ID覆盖Device ID进行分桶，避免同一用户被分入不同实验组导致的SRM（Sample Ratio Mismatch）问题。平台内置SRM检测，当实验组与对照组实际用户比例偏差超过1%时自动触发警告。

## 常见误区

**误区1：实验开始后可以随时调整流量比例**
部分运营人员在发现实验组数据不理想时，会在平台上临时将对照组流量从50%调整至30%，试图加速实验组数据累积。这一操作会破坏随机化假设（Randomization Assumption），导致分流变更前后的用户群体构成不一致，使得后续所有统计结论无效。正确做法是在实验开始前用样本量计算器锁定流量比例，整个实验周期不得修改。

**误区2：曝光量等于实验平台的Assignment量**
不少团队直接用后台日志中的实验分配记录（Assignment Log）替代曝光日志做分析，导致将从未看到变体界面的用户计入实验组。以游戏商城实验为例，若某服务器只有60%的玩家在测试期内访问过商城，用Assignment量分析会将另外40%的无关用户稀释进来，相当于人为降低了实验的检测功效。

**误区3：单一指标显著即可全量**
游戏实验平台的双轨指标体系（核心指标+护栏指标）设计初衷正是防止"局部优化、全局损伤"。曾有案例显示某实验将商城首页图改为高强度刺激设计，7日付费转化率提升8%（核心指标达标），但护栏指标显示30日留存率下降4.5%，平台因此拦截了该实验的全量发布——说明护栏指标的监控与核心指标同等重要，不可省略。

## 知识关联

**与前置概念UI/UX实验的关系**：UI/UX实验聚焦于单个界面元素变体的设计与评估，实验平台则为其提供从分流到分析的完整执行环境。在UI/UX实验阶段学会定义"曝光触发条件"的能力，在实验平台层面直接映射为Exposure Event的埋点规范——错误的曝光定义会导致平台记录的实验数据从根本上失真。

**与后续概念分群分析的关系**：实验平台的分析模块产出的是全量实验用户的聚合指标，而分群分析（Cohort Analysis）则在此基础上对实验结果做细粒度拆解——例如区分付费用户与免费用户在同一实验中的响应差异。实验平台通常以用户属性标签（User Attributes）为接口，将实验数据输出至分群分析模块进行二次切片。

**与功能开关的关系**：实验平台的分流引擎与功能开关（Feature Flag）系统高度耦合——功能开关控制"某功能是否对特定用户可见"，实验平台控制"哪些用户进入哪个实验桶"。在工程实现上，许多游戏公司将Feature Flag服务作为实验平台的底层分发机制，实验平台在其上增加统计分析和指标监控能力，形成完整的实验闭环。
