# 服务端更新

## 概述

服务端更新是指在不要求玩家下载新客户端包体的前提下，通过修改游戏后端服务器逻辑、配置数据或数据库结构来改变游戏行为的技术手段。与客户端包体更新不同，服务端更新的变更内容完全存储和运行在服务器侧，玩家设备上的可执行文件不发生任何变化。这使得服务端更新成为Live Ops团队在活动运营、数值调整和BUG热修方面最高频使用的更新手段——Riot Games内部统计显示，《英雄联盟》每周发布的变更中，约60%属于纯服务端配置推送，无需触发客户端版本号递增。

服务端更新的工程化体系随"游戏即服务（Games as a Service，GaaS）"模式在2010年代的大规模普及而系统化。早期页游和MMO已具备"后台下发配置"的雏形，但蓝绿部署（Blue-Green Deployment）、金丝雀发布（Canary Release）和零停机数据库迁移等现代标准，在2015年前后随着《英雄联盟》《Clash of Clans》等头部产品的SLA（服务等级协议）压力下逐步成型。《Clash of Clans》的开发商Supercell公开披露，其后端服务自2016年起实现了全年99.95%可用性目标，对应每年最多允许约4.38小时的停机时间，这一目标的达成直接依赖不停服发布体系的落地。

对于Live Ops团队而言，服务端更新的核心价值在于：任何由服务器逻辑控制的内容都可以在分钟级别完成上线，彻底绕开App Store和Google Play的审核周期（通常为1至7个工作日）。这一特性让运营团队能够在赛季活动结束前几小时紧急调整掉落率，或在节假日热度爆发时实时扩容限时商品池，而无需在商店审核队列中等待。

## 核心原理

### 不停服更新（Zero-Downtime Deployment）

不停服更新要求新旧版本服务在切换过程中短暂共存，保证玩家请求不被中断。最常见的实现方式是**蓝绿部署（Blue-Green Deployment）**：维护两套完全镜像的生产环境，蓝环境运行旧版本，绿环境部署并预热新版本，通过负载均衡器的单次流量切换完成版本迁移，切换动作本身通常在1秒内完成。若绿环境上线后出现异常，只需将负载均衡器流量切回蓝环境，整个回滚操作同样在秒级完成。蓝绿部署的显著代价是资源成本翻倍——在切换期间必须同时维持两套完整的生产规格实例，对于日活数百万、需要数百台服务器支撑的游戏，这意味着在部署窗口期内云计算账单实际上翻倍。

另一种更节约资源的方式是**滚动更新（Rolling Update）**：在Kubernetes等容器编排平台中，新版本Pod逐批替换旧版本Pod。每批次替换比例通常配置为总实例数的25%，即旧版本缩减到0之前，整个服务的Pod池中新旧版本始终共存。滚动更新期间，处理同一玩家相邻两次请求的服务器可能分属不同版本，因此强制要求新旧版本代码必须同时兼容同一套数据库Schema——这是滚动更新最核心的架构约束，也是"Expand-Contract"数据库迁移模式存在的根本原因（见"数据库迁移"章节）。

### 灰度发布（Canary Release）

"金丝雀发布"一词来源于19世纪英国矿工将金丝雀带入矿井侦测一氧化碳的做法——金丝雀比人类更敏感，一旦中毒倒下，矿工立刻撤离。在软件部署中，小比例真实用户流量扮演"金丝雀"的角色：先用生产环境的真实负载验证新版本，再逐步扩大覆盖面。Google SRE工程师Niall Murphy等人在《Site Reliability Engineering》（O'Reilly, 2016）中将金丝雀发布列为降低变更风险最有效的单项措施之一。

标准灰度流程按如下比例递进：

$$P_{canary}(t) = \{5\%, 20\%, 50\%, 100\%\}$$

其中每个阶段的监测窗口通常为30分钟，触发回滚的阈值指标包括：新版本服务器崩溃率超过基准值0.1%、API响应P99延迟超过500ms、支付接口成功率低于99.5%。任意指标触警，自动回滚流程立即将该批次流量切回旧版本。

灰度发布的用户分桶策略对游戏运营至关重要。工业实践中最常见的做法是**按UID尾号取模**：

```python
def is_in_canary(user_id: int, canary_percent: int) -> bool:
    """
    按UID取模决定用户是否进入灰度桶。
    canary_percent: 灰度比例，如 5 表示 5%
    """
    return (user_id % 100) < canary_percent

# 示例：UID=10234的玩家，5%灰度期间不在灰度桶（10234 % 100 = 34 >= 5）
# UID=10203的玩家，5%灰度期间在灰度桶（10203 % 100 = 3 < 5）
```

取模分桶优于随机分桶的关键原因：同一玩家在整个灰度周期内始终命中同一版本服务器，避免因服务版本切换导致玩家游戏状态（如背包数据、进行中的副本状态）出现不一致。此外，付费用户、公会会长、大R玩家等高价值账号通常被单独配置为排除在前两个灰度批次之外，以保护核心用户体验。

### 配置热更新与Feature Flag

配置热更新（Hot Config Reload）是服务端更新中响应最快的一类：服务进程无需重启，运行时从远端配置中心（如Consul、Etcd或自研的配置管理平台）拉取最新配置并即时生效。典型场景包括活动掉落率的实时调整、限时商品池的开关控制以及服务器性能参数（如匹配队列等待上限）的动态调节。

Feature Flag（功能开关）是配置热更新的高级形式，允许新功能代码随包体部署到生产环境但默认关闭，由运营人员在后台手动开启。《英雄联盟》的新英雄上线流程就大量依赖Feature Flag：新英雄的完整代码通常随上一个版本的服务端部署上线，但对玩家不可见，到正式公告日期再通过Feature Flag开关激活，实现"代码提前部署、内容准时公开"的运营节奏。

## 关键公式与技术规范

### 数据库迁移（Database Migration）

数据库迁移是服务端更新中技术难度最高的环节。游戏后端通常使用MySQL或PostgreSQL等关系型数据库，当新版本需要新增字段、修改字段类型或重构表结构时，若直接在行数超过千万级的表上执行`ALTER TABLE`语句，在MySQL 5.7及以下版本中会触发全表拷贝锁，可能造成数十分钟的写操作阻塞，等同于主动停服。

规避表锁的标准方案是**展开-收缩模式（Expand-Contract Pattern）**，分三个阶段执行：

**阶段一：Expand（展开）**
新增目标字段，但不删除旧字段，新旧字段同时存在。双写逻辑在服务端代码中上线：写操作同时写入旧字段和新字段；读操作优先读新字段，降级读旧字段。此阶段新旧版本服务代码均可正常运行。

```sql
-- Expand阶段：新增 exp_new 字段，保留旧字段 exp
ALTER TABLE player_profile ADD COLUMN exp_new BIGINT DEFAULT 0;

-- 存量数据回填（使用 pt-online-schema-change 或分批UPDATE，避免锁表）
UPDATE player_profile SET exp_new = exp WHERE exp_new = 0 LIMIT 5000;
```

**阶段二：Migrate（迁移）**
存量数据回填完成，确认新字段数据完整性后，将读操作切换为只读新字段，旧字段停止写入（但保留结构不删除）。此阶段为过渡期，持续时间通常为1至2个版本周期（约1至2周）。

**阶段三：Contract（收缩）**
确认旧字段在所有服务节点中均已停止读写后，执行`ALTER TABLE DROP COLUMN`删除旧字段，完成迁移。此时表结构已完全收敛到新Schema。

这三个阶段通常跨越2至3次独立的服务端部署完成，每次部署均可零停机执行，代价是迁移周期拉长至数周。

## 实际应用

### 案例一：《原神》赛季活动掉落率紧急调整

2021年某版本更新后，《原神》玩家社区反馈某限时副本的稀有材料掉落率显著低于官方公告数值，BUG根因为配置文件中掉落权重字段单位错误（应填千分比却填入了万分比）。由于掉落率属于纯服务端配置参数，运营团队在确认BUG后约15分钟内完成了配置修正并推送至全区服务器，同步发布补偿公告，全程无需重启服务器、无需客户端更新，受影响时间窗口被压缩在2小时以内。若该参数硬编码在客户端包体中，等待App Store审核的最短时间为24小时，事故影响面将扩大10倍以上。

### 案例二：游戏节点扩容的蓝绿切换

某头部手游在春节档期间预计同时在线人数将从日常的80万峰值冲至280万。运维团队提前48小时在绿环境中预热扩容后的服务器集群（约新增600台实例），完成压测后在活动开始前2小时通过负载均衡器将100%流量切换至绿环境，旧蓝环境保留待命12小时后释放。整个切换过程中，玩家侧最大感知延迟增加不超过50ms，切换完成后P99延迟恢复至基准值。

## 常见误区

**误区一："服务端更新不需要测试"**
由于服务端更新不触发App Store审核流程，部分团队误以为可以跳过QA直接推送生产环境。实际上，服务端逻辑错误（如掉落权重溢出、付费接口字段缺失）同样会导致严重的玩家体验问题乃至资损，且服务端变更影响全量用户的速度比客户端更新更快——配置热更新可在60秒内覆盖全部在线玩家，留给发现和回滚的时间窗口极短。正确做法是为每次服务端变更配套自动化回归测试，并强制经过灰度发布流程。

**误区二："蓝绿部署可以完全替代灰度发布"**
蓝绿部署解决的是"部署动作本身是否会中断服务"的问题，切换是一次性的全量操作。灰度发布解决的是"新版本逻辑在真实生产流量下是否有隐藏BUG"的问题，通过控制暴露比例来限制潜在故障的影响面。两者目标不同，在成熟的服务端发布体系中应组合使用：蓝绿部署保证切换动作零停机，金丝雀策略保证变更内容低风险。

**误区三："Expand-Contract模式仅适用于关系型数据库"**
展开-收缩模式同样适用于Redis等KV存储的数据结构变更、MongoDB等文档数据库的字段重命名，以及消息队列（如Kafka）的消息格式升级（旧消费者需同时兼容新旧格式直到全量迁移完成）。任何涉及"多版本服务并发访问同一存储"的场景，都应考虑该模式。
