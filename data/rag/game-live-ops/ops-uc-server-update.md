# 服务端更新

## 概述

服务端更新是指在不要求玩家下载新客户端包体的前提下，通过修改后端逻辑、配置数据或数据库结构来改变游戏行为的技术实践。与需要玩家主动安装新 APK/IPA 的包体更新不同，服务端更新的变更内容存储在游戏服务器端，客户端启动时通过接口拉取最新配置或行为规则，因此可以在数分钟内对全球数百万在线玩家同时生效。

这一技术在 2010 年代随着"游戏即服务"（Games as a Service，GaaS）模式的普及而成为标准实践。Supercell 旗下《部落冲突》最早将"服务端热配置"作为日常运营工具，使策划可以在不走 App Store 审核流程的情况下调整单位攻击力、资源掉落率等数值。此后《原神》《王者荣耀》等现代手游均建立了完整的服务端更新管道，每次 Live Ops 活动几乎都依赖这套机制。

服务端更新的核心价值在于它将"发布风险"与"玩家中断体验"解耦——运营侧可以每天推送多次配置变更，而玩家不会看到任何下载提示或版本弹窗。但这也意味着错误配置可以以同样的速度大规模传播，因此灰度发布与回滚机制是不可或缺的安全网。

## 核心原理

### 不停服更新（Hot Update / Zero-Downtime Deployment）

不停服更新的技术基础是**蓝绿部署（Blue-Green Deployment）**或**滚动更新（Rolling Update）**。以 Kubernetes 滚动更新为例，假设服务集群共有 10 个 Pod，系统会先启动 1 个携带新版本代码的 Pod，待其健康检查通过后再下线 1 个旧版本 Pod，如此循环直到全量替换完成。整个过程中始终有至少 9 个 Pod 在线处理玩家请求，停机时间理论上为 0。

蓝绿部署则维护两套完全独立的生产环境（蓝环境和绿环境），新版本部署在绿环境完成验证后，通过修改负载均衡器的路由规则将 100% 流量切换过去，旧蓝环境保留 15–30 分钟作为快速回滚的后备。这种方式切换速度快（通常 < 30 秒完成流量切换），但资源成本是滚动更新的约 2 倍。

游戏服务器还需要处理**会话连续性**问题：玩家在副本战斗中途不能被强制踢下线。通用解法是在旧版本 Pod 设置"优雅退出（Graceful Shutdown）"倒计时（通常 120–300 秒），只拒绝新连接，存量会话完成后才真正关闭进程。

### 灰度发布（Canary Release / Feature Flag）

灰度发布将新版本服务端逻辑先推送给一小部分玩家，用真实流量验证稳定性后再逐步扩量。常见的分流维度包括：

- **按 UID 哈希取模**：`uid % 100 < 5` 即 5% 灰度，实现简单且同一玩家每次进入相同分组；
- **按地区分流**：先在新加坡/台湾服测试，再推向大陆/全球；
- **按玩家等级分层**：优先向高等级核心玩家灰度，因其更容易主动反馈 Bug。

功能开关（Feature Flag）是灰度发布的配套工具。服务端代码中预埋 `if (featureFlag.isEnabled("new_guild_war_logic", userId))` 这样的条件分支，开关状态存储在 Redis 或专用配置服务（如 LaunchDarkly、自研的 Flag Service）中，无需重新部署代码即可实时变更受众范围。《炉石传说》团队曾公开表示，其后端同时维护超过 200 个活跃功能开关。

### 数据库迁移（Database Migration）

服务端更新中最高风险的操作是数据库结构变更（Schema Migration），因为它涉及已有数据的物理存储格式修改。生产环境数据库迁移必须遵循**扩展-收缩模式（Expand-Contract Pattern）**：

1. **扩展阶段**：新增字段/表，旧代码和新代码同时兼容，不删除任何旧结构；
2. **迁移阶段**：运行后台脚本（如 Python 的 Alembic 或 Liquibase）批量将旧字段数据回填到新字段，批次大小通常设为 1000–5000 行/批，避免锁表；
3. **收缩阶段**：确认新旧数据一致性并等待所有服务端实例升级完成后，再删除废弃的旧字段。

对于 MySQL 的 `ALTER TABLE ADD COLUMN` 操作，数据量超过 1000 万行时必须使用 `pt-online-schema-change` 或 `gh-ost` 工具进行在线 DDL，否则会产生全表锁导致服务中断数分钟甚至数十分钟。

## 实际应用

**活动数值热修复**：某手游节日活动上线后，运营发现限定角色的爆率参数配置错误（0.3% 写成了 3%），服务端更新可以在 5 分钟内通过修改配置服务的 JSON 文件并触发缓存刷新来修正错误，而无需走任何审核或重新发包流程。修正前已产生的抽卡记录需要通过运营补偿方案处理，技术侧不回滚已完成的数据库写入。

**赛季天梯系统上线**：新赛季逻辑涉及新增 `rank_season_v3` 数据表、修改排行榜计分公式。团队先执行扩展阶段（新建表），灰度 10% 玩家启用新赛季逻辑并观察 P99 接口延迟（目标 < 200ms），确认无异常后在 24 小时内分 4 个批次（10% → 30% → 70% → 100%）完成全量。

**紧急安全补丁**：发现服务端存在可被恶意利用的刷金币接口漏洞，蓝绿部署可将携带修复代码的新环境在 5 分钟内接管全部流量，同时保留旧环境供事后追查日志。

## 常见误区

**误区一：服务端更新可以随意修改已上线的游戏规则**。服务端更新在技术上可以修改任意后端逻辑，但游戏内已公示的概率（如抽卡概率声明）、已售出道具的功能描述，在中国大陆受《网络游戏管理办法》约束，未经公告擅自修改将面临监管处罚。技术能力边界与法律合规边界是两套不同的约束。

**误区二：灰度发布期间灰度用户与全量用户之间的数据可以完全隔离**。在 MMO 或社交游戏中，灰度用户与全量用户往往在同一游戏世界中交互。若新版本引入了新的物品或货币类型，灰度用户创建的数据可能写入全量用户也会读取的共享数据库，导致非灰度用户看到无法解析的数据字段，引发客户端报错。解决方案是对灰度范围的新字段做向下兼容处理，旧版本客户端收到未知字段时应静默忽略而非崩溃。

**误区三：蓝绿部署的"零停机"等于玩家无感知**。流量切换瞬间，部分玩家的 TCP 长连接（如 WebSocket 游戏通信连接）会被强制断开，客户端通常会触发自动重连逻辑（1–3 秒延迟）。对于回合制或卡牌游戏影响较小，但对实时对战（RTS/FPS）类游戏，切换时机必须选在服务器负载最低的时间窗（通常是凌晨 4–6 点），以将影响玩家数降至最低。

## 知识关联

**与包体管理的关系**：包体管理决定了哪些资源和逻辑必须打包在客户端本地（如核心引擎代码、3D 模型），而服务端更新负责那些可以动态下发的部分（如活动配置、数值表、部分脚本逻辑）。两者的技术边界通常体现在"热更新白名单"文档中——哪些文件走包体审核，哪些走服务端下发，是 Live Ops 工程架构设计的基础决策。

**与变更日志文化的关系**：每次服务端更新必须生成可追溯的变更记录，包括：变更内容摘要、灰度范围、发布时间戳、执行人、回滚步骤。这些记录构成了变更日志文化的原始素材。没有结构化服务端更新记录的团队，往往在线上事故复盘时无法确定"哪个配置变更在几点导致了指标异常"，使 MTTR（平均故障恢复时间）显著延长。
