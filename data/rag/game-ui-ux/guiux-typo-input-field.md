# 输入框设计

## 概述

输入框（Input Field）是游戏UI中允许玩家键入文字的交互控件，常见于角色命名、聊天系统、好友搜索、公会申请表单等场景。与按钮或下拉菜单不同，输入框需要同时处理视觉样式、实时字符验证和输入法拦截（IME）三个并发问题，其复杂度远超同类控件。

游戏输入框的设计规范从早期街机时代的字符选择轮盘演变而来。1980年代的高分榜仅支持3个大写英文字母，到2000年代网游兴起后，必须支持中文、日文等复杂输入法。如今一款面向全球的MMO游戏，其角色命名输入框至少需要处理Unicode 14.0标准下的数万个字符，同时过滤违禁词库，延迟必须控制在16ms帧时间之内。

游戏输入框与网页或App输入框的核心差异在于：渲染管线不同（GPU实时渲染而非HTML DOM）、输入法集成方式不同（需要手动调用平台IME API），以及字体栅格化精度要求更高（需要在动态分辨率缩放下保持可读性）。忽视这些差异会导致中文输入时出现文字重叠、光标错位等严重问题。

## 核心原理

### 视觉样式与状态层

输入框必须明确区分至少四种视觉状态：**默认（Default）、聚焦（Focused）、禁用（Disabled）、错误（Error）**。聚焦状态的边框高亮应使用游戏主色调，但亮度增量建议不低于40%，以确保在HDR显示器和SDR显示器上均可感知。输入框的最小点击热区（Hit Area）应为44×44像素（遵循Apple HIG标准），但视觉边框可以更小，以维持游戏UI的精致感。

输入框内部的文字应使用等宽或接近等宽的字体，避免在角色命名时因字符宽度不一导致预览名字长度判断失准。例如"IllIll"与"mmmmm"在比例字体下视觉宽度差异可达2倍，而游戏界面通常有严格的名字显示宽度限制（如最宽120像素）。建议在输入框右侧实时显示剩余字符数，格式为"当前/最大"，如"8/16"。

### 占位符文字设计

占位符（Placeholder）文字不是装饰，是功能性引导文本。它的颜色透明度应设置在40%–60%之间，过低（<30%）在深色主题下不可见，过高（>70%）会干扰玩家判断输入框是否已有内容。占位符文字应使用动词引导句式，如"输入角色名称"而非"角色名称"，A/B测试数据显示前者能减少约18%的空提交错误。

占位符文字的字号应与输入内容字号完全一致，禁止出现占位符字号比输入字号小2px以上的情况，否则输入时会有视觉跳动感。在角色命名场景中，占位符可展示示例名字（如"勇者·赤焰"），同时配合淡灰色以区分真实输入内容。

### 字符验证与实时反馈

输入框验证分为**前端即时验证**和**后端异步验证**两层。前端验证在每次`onValueChanged`事件触发时执行，应在200ms内完成，职责是过滤非法字符（如特殊符号、超长字符串）和违禁词初步检测。后端验证负责名字唯一性校验，应使用防抖（Debounce）技术，延迟设置为500–800ms，避免在玩家快速输入时频繁发送网络请求。

验证错误的反馈必须同时使用颜色和图标两种信号，不能仅依赖红色边框——约8%的男性玩家存在红绿色盲。错误提示文字应出现在输入框正下方，字号建议比输入框正文小2pt（例如正文14pt则提示12pt），内容必须明确说明原因，如"名称包含违禁词"而非泛泛的"输入无效"。

### IME支持与光标处理

IME（输入法编辑器）支持是游戏输入框最易出错的技术点。在Unity引擎中，需要通过`Input.imeCompositionMode = IMECompositionMode.On`开启合成模式，并在`OnGUI()`中使用`Input.compositionString`获取正在组合的拼音串，将其以虚线下划线样式临时渲染在光标位置。若直接使用`Input.inputString`，将只能获取到最终上屏的汉字，导致拼音输入过程中组合字符串丢失、玩家看不到正在输入什么。

光标（Caret）位置计算必须基于字形（Glyph）宽度而非字符索引乘固定值。一个中文字符宽度约为两个英文字符，若用`charIndex × fixedWidth`计算，输入"你好abc"后光标将偏移约20像素。主流游戏引擎的TextMeshPro组件提供`TMP_TextInfo.characterInfo[i].origin`字段，可精确获取每个字形的起始X坐标，光标动画（通常以1Hz频率闪烁）应绑定到该值。

## 实际应用

**角色命名输入框**是最典型的场景。通常限制长度为2–16个全角字符（或4–32个半角字符），禁止纯数字、纯空格或以特殊字符开头。输入框旁常放置"随机名字"按钮，点击后填充预设名字并触发一次后端唯一性验证，该流程可将命名完成率提升约25%。

**聊天输入框**需要支持快捷表情插入（通过`[emoji:001]`语法），插入后光标应自动跳至标签末尾而非标签内部，防止玩家误编辑。发送按钮应在输入框非空时才变为可点击状态，禁用状态的不透明度设为40%。

**搜索输入框**（如好友搜索）需要在右侧增加清空（×）按钮，该按钮仅在有内容时显示，点击后清空内容并保持聚焦状态，方便玩家立即重新输入。搜索结果列表与输入框的间距不应超过8px，视觉上保持为一个整体组件。

## 常见误区

**误区一：用占位符替代标签（Label）**。在复杂表单（如公会申请表）中，多个输入框若仅靠占位符区分，玩家填写途中一旦聚焦就无法再看到字段名称，导致填错率上升。正确做法是在输入框外部或顶部保留永久性标签，占位符仅作为补充提示。

**误区二：IME输入时触发字符限制逻辑**。若在组合状态（composing）下就执行字数截断，会导致玩家输入"huan"（还未选字）时，系统误以为已输入4个英文字母而截断，最终无法完成汉字输入。正确做法是仅对`compositionString`为空（即已上屏状态）的文本执行长度限制校验。

**误区三：移动端未适配软键盘遮挡**。iOS和Android的软键盘弹出后会覆盖屏幕下半约40%区域，若输入框位于屏幕下方而游戏未监听`onScreenKeyboardResize`事件并上移UI面板，玩家将完全看不到正在输入的内容。Unity中可通过`TouchScreenKeyboard.area`获取键盘占用区域并动态调整Canvas偏移量。

## 知识关联

**前置知识：命名显示规范**为输入框设计提供了字符集约束（允许哪些Unicode字符）和最大长度标准，输入框的前端验证规则直接来源于命名显示规范中定义的合法字符白名单。若命名规范规定名字不得超过16个全角字符，则输入框的`characterLimit`参数和剩余字符计数器均以此为基准。

**后续知识：无障碍字体**在输入框场景中重点涉及屏幕朗读器（Screen Reader）对输入框状态的播报规则——输入框的`aria-label`或游戏引擎中的Accessibility API需要正确上报当前值、错误状态和字符剩余数，这是无障碍字体章节中"动态文本无障碍"的核心实践场景。输入框的错误提示文字也必须满足无障碍字体章节规定的最小对比度要求（WCAG 2.1 AA级，即4.5:1）。
