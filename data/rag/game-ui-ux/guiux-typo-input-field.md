# 输入框设计

## 概述

输入框（Input Field）是游戏UI中允许玩家输入文字信息的交互元素，常见于角色命名、聊天系统、好友搜索、密码输入等场景。与纯展示性的文本标签不同，输入框必须处理用户的主动输入行为，因此其设计需要同时覆盖**视觉样式、输入状态反馈、内容验证逻辑**和**输入法兼容**四个维度。

输入框在游戏UI中的出现可追溯至早期PC端RPG游戏的角色命名界面。随着移动端游戏兴起，输入框设计面临了前所未有的挑战——移动端软键盘会遮盖屏幕下方约40%至60%的区域，这要求设计师必须专门处理键盘弹出时的界面布局偏移逻辑，而这一问题在主机或PC平台上几乎不存在。

在实际项目中，输入框设计的质量直接影响玩家的首次游戏体验。以角色命名环节为例，一个无法正确支持中文IME输入或拼音候选词的输入框，会导致中文玩家在游戏开始的第一分钟内产生明显的挫败感。因此，输入框设计不是单纯的视觉工作，而是涉及前端逻辑与UI视觉的协作设计任务。

## 核心原理

### 输入框的视觉状态系统

一个完整的游戏输入框至少需要设计**5种视觉状态**：默认（Default）、聚焦（Focused）、输入中（Active Typing）、验证错误（Error）和禁用（Disabled）。每种状态应有明确可区分的视觉差异，通常通过边框颜色、发光效果（Glow）或背景色变化来实现。

以一个典型的幻想风格RPG为例，聚焦状态下边框可从灰白色（`#AAAAAA`）切换为金色高亮（`#FFD700`），并附加4px的外发光效果，明确告知玩家当前光标已锁定至该输入框。错误状态则通常使用红色描边（如`#FF4444`）配合抖动动画，持续时长建议为300ms至500ms，避免过长的动画干扰玩家再次输入。

输入框的最小可点击/可触摸区域在移动端不应低于**44×44 dp**（参考Apple HIG规范及Android Material Design规范），但在游戏UI中，考虑到沉浸式美术风格，视觉尺寸可以小于触摸热区，即通过透明扩展区域增大实际交互范围。

### 占位符文字（Placeholder）的设计规范

占位符是输入框在内容为空时显示的提示文字，其核心作用是告知玩家该框的预期输入内容。占位符文字颜色的透明度通常设置为正文文字的**40%至50%**，确保其"弱于"实际输入内容但仍可辨识。

在游戏场景中，占位符文字应使用具体的引导性语言，而非笼统描述。例如角色命名框的占位符应写"输入你的角色名（最多12个字符）"，而不是"请输入内容"。对于聊天搜索框，可使用"搜索玩家名或公会名"，明确告知搜索的对象范围。占位符文字的字号应与输入文字保持一致，防止玩家输入后出现文字跳动的视觉突变。

占位符在玩家开始输入**第一个字符时立即消失**，而不是等输入框失去焦点后才消失，这是普遍的交互规范。反之，若占位符在聚焦时立即消失（无论是否输入），则会使玩家在未完成输入前忘记提示内容，尤其在复杂表单中会造成混乱。

### 输入验证与字符限制

游戏输入框的验证规则直接与游戏业务逻辑绑定。角色名输入框通常需要同时执行以下验证：**字符数量限制**（如2至12个字符）、**非法字符过滤**（屏蔽特殊符号`@#$%`等）、**敏感词检测**（通常为后端异步验证）和**重名检测**（服务器端实时验证）。

验证反馈的时机分为两种模式：**实时验证（On Change）** 在每次键入后立即触发，适用于字符数量显示等轻量级规则；**提交验证（On Submit）** 在玩家点击确认后触发，适用于需要请求服务器的重名或敏感词检测。字符计数器应以"当前字数/最大字数"的格式（如`5/12`）实时显示在输入框右下角，字体字号建议比输入文字小2至4pt，使用与占位符相同的弱化色调。

当输入达到最大字符数时，界面应通过计数器变色（如红色）提示玩家，而不是静默地阻止继续输入——后者会让玩家误以为键盘或输入法出现了故障。

### IME（输入法编辑器）支持

IME支持是游戏输入框在中日韩（CJK）市场的核心技术挑战。用户在使用拼音输入法时，存在一个**拼音候选词合成阶段（Composition Phase）**，此时输入框中会出现带下划线的临时拼音字母（如`zhōngwén`），尚未转化为最终汉字。

字符数量验证**必须区分合成阶段和最终提交阶段**。若在合成阶段就计算字符数，拼音字母会占用字符配额，导致玩家在实际输入汉字之前便触发字符上限，引发严重的输入体验问题。在Web和Unity等主流游戏引擎环境中，应监听`compositionstart`和`compositionend`事件（或对应引擎的IME回调），仅在`compositionend`之后执行字符计数验证。

此外，当IME候选词列表弹出时，其显示位置不应与游戏UI的关键信息层（如确认按钮）发生遮挡，部分引擎需要手动处理IME候选框的绘制层级（Z-order）。

## 实际应用

**角色命名场景**：玩家创建角色时，输入框应在获得焦点后自动弹出键盘（移动端），同时界面整体上移，确保输入框保持在键盘上方20px以上的可见区间。命名确认按钮应在字符数满足最小要求（如≥2个字符）后才从灰色禁用状态切换为可点击状态，以实时视觉反馈降低玩家提交空名的概率。

**聊天搜索框**：实时搜索框应在玩家停止输入**300ms后**（防抖延迟）才触发搜索请求，避免每次按键都发送请求导致服务器压力过高。搜索结果列表的文字中，匹配关键词部分应以高亮色（如黄色）单独标注，帮助玩家快速确认匹配位置。

**密码/兑换码输入框**：密码框应默认将输入字符替换为圆点（•），并提供"显示/隐藏"切换按钮，图标尺寸不应低于24×24dp。兑换码输入框通常包含连字符格式（如`XXXX-XXXX-XXXX`），可通过自动插入连字符的逻辑提升输入流畅度。

## 常见误区

**误区一：将占位符作为标签（Label）使用**
部分游戏UI设计中，输入框没有独立的标签文字，仅依赖占位符传达输入目的。当玩家开始输入后，占位符消失，玩家若中途忘记该框的用途，无法回溯提示信息。正确做法是在输入框外部配置固定标签，占位符仅作为补充说明，而非唯一说明。

**误区二：忽略合成阶段导致CJK输入截断**
如前文所述，若字符上限验证在`compositionstart`期间生效，玩家输入"你好"（拼音：`nihao`，5个拼音字母）时，5字符的限制会在汉字形成之前被触发，最终输入框内可能只剩下"你"或完全无法输入第二个汉字。这是CJK市场游戏中最高频的输入框Bug之一。

**误区三：错误状态仅改变边框颜色**
单纯改变边框颜色作为错误提示，对于色觉障碍（红绿色盲影响约8%的男性玩家）的玩家几乎不可见。正确的错误状态设计应同时包含：颜色变化、图标（如⚠️错误图标）和文字说明三种信号中的至少两种，以确保不同用户都能感知到验证错误。

## 知识关联

**前置概念：命名显示规范**
命名显示规范定义了游戏内角色名、公会名等文字的字符集范围、长度上限和显示截断规则，这些规则直接决定了输入框的验证逻辑参数。例如，若命名规范规定角色名最多支持12个全角字符，则输入框的字符计数器和硬限制必须与之对齐，两者共用同一套字符集白名单（如Unicode CJK统一汉字区块`U+4E00`至`U+9FFF`）。
