# 富文本系统

## 概述

富文本系统（Rich Text System）是游戏UI中一种能够在单个文本渲染管线内同时处理多种格式化指令的排版技术，允许开发者将纯文字、内嵌图标精灵、颜色渐变标签、超链接热区等异构元素混合排列于同一文本行内。与普通文本渲染不同，富文本系统需要维护一个**混合元素序列**，每个元素可以是字符串片段、纹理四边形或可交互区域，渲染时按基线（baseline）统一对齐。

这一技术在20世纪90年代桌面端HTML/CSS标准普及后逐渐被游戏领域借鉴。Unity引擎在2013年的4.x版本中正式内置了基于XML标签语法的富文本解析器，而Unreal Engine的UMG系统则在4.8版本（2015年）引入了`<RichTextBlock>`组件，支持通过数据表驱动图标和样式映射。这两套系统至今仍是业界最主流的游戏富文本实现参考。

富文本系统在游戏UI中的价值集中体现在技能描述、聊天频道和物品提示框三类场景：一条"造成`<color=#FF4444>150点火焰伤害</color>`并使目标`<icon=debuff_burn>`灼烧"这样的技能文本，若拆分为多个独立文本组件来实现，将产生严重的动态布局对齐问题，而富文本系统能将其视为一个整体段落进行流式排版。

## 核心原理

### 标签解析与元素序列构建

富文本系统的第一步是将原始字符串解析为**标记化元素列表（Token List）**。解析器从左至右扫描字符串，遇到开闭标签（如`<color=#FF0000>`…`</color>`）时将其间的文本标记为带属性的文本Token；遇到自闭合标签（如`<sprite index=3>`）时生成一个图标Token。每个Token记录以下字段：类型（文本/图标/链接开始/链接结束）、内容或资源引用、以及从父标签继承或覆盖的样式属性（字体大小、颜色、粗体标志等）。

Unity TextMeshPro的标签解析器在内部使用**递归下降解析**，支持标签嵌套，例如`<color=red><b>重要</b></color>`是合法结构。需注意嵌套深度过大会导致解析性能下降，建议游戏内富文本标签嵌套层级不超过4层。

### 混合元素的基线对齐

当图标精灵（Sprite）与文字混排时，必须解决高度不一致的基线对齐问题。标准做法是将图标精灵视为一个**宽度等于其像素宽度、高度等于当前行高（line height）的虚拟字形**，并通过垂直偏移（vertical offset）参数将图标中心或底部对齐到文字基线。

设当前字体的cap height为 `H_cap`，图标纹理的原始高度为 `H_sprite`，则图标的缩放比例为：

```
scale = H_cap / H_sprite
```

使用cap height而非line height作为基准，能确保图标视觉上与大写字母等高，避免图标过高压缩行间距。Unreal Engine的`URichTextBlock`默认采用此策略，并额外暴露`ImageBasePadding`属性供微调。

### 颜色标签的渐变实现

简单颜色标签（`<color=#RRGGBB>`）仅对每个顶点着色，属于逐顶点颜色插值。要实现文字从红到金的**水平渐变**，需要对同一文本片段内每个字形的左侧两顶点和右侧两顶点分别赋予不同颜色，然后依靠GPU在三角形内部进行线性插值。TextMeshPro通过`<gradient>`标签配合`TMP_ColorGradient`资产实现此效果，支持四角独立着色（top-left、top-right、bottom-left、bottom-right各一个Color32值），理论上可表达任意双线性颜色映射。

### 超链接热区计算

超链接需要在文本布局完成后，收集`<link>`标签所覆盖的所有字形的包围矩形，合并成一组矩形热区列表，供输入系统逐帧检测鼠标/触摸位置。若链接文本跨越多行（例如聊天消息换行），则需生成**多个独立矩形**而非一个大矩形，否则会出现行间空白区域误触发的问题。TextMeshPro提供`TMP_Text.GetLinkInfo()`接口返回`TMP_LinkInfo`结构体，其中`linkTextfirstCharacterIndex`和`linkTextLength`字段用于定位热区范围。

## 实际应用

**物品提示框（Tooltip）的品质颜色标签**：绝大多数RPG游戏使用约定俗成的品质颜色体系，如《暗黑破坏神》系的白/蓝/黄/橙，开发者只需维护一张颜色常量表（例如`LEGENDARY_COLOR = #FF8000`），在文本数据库中统一使用`<quality=legendary>`这样的语义标签，由富文本系统的预处理器将其展开为具体颜色值，后续修改品质颜色只需改一处。

**聊天系统的表情包内嵌**：MMO游戏的聊天频道中，玩家输入`:smile:`后客户端将其替换为`<sprite="emoji_atlas" name="smile">`，服务端传输的是原始文本代码，客户端负责将图标精灵映射到本地图集（Atlas）的具体UV坐标。这种设计使客户端可以在不重启的情况下更新表情图集，而不影响历史消息的重新渲染。

**技能描述的动态数值嵌色**：在数值驱动的游戏中，技能描述文本中的伤害数字往往来自运行时计算。实现方式是将模板字符串存储为`"造成<color=#FF4444>{damage}</color>点伤害"`，运行时用`string.Format`或`StringBuilder`填入实际数值后再传入富文本渲染器，避免在热路径上直接拼接带标签的字符串（GC压力）。

## 常见误区

**误区一：认为富文本标签在所有平台解析规则相同**
Unity的旧版`Text`组件与TextMeshPro对标签的解析存在显著差异。旧版`Text`使用简单的字符串匹配，不区分大小写且不支持嵌套；TextMeshPro的解析器则区分大小写并支持参数化标签（如`<size=120%>`中的百分比单位）。若项目在二者之间迁移，`<color>` `<b>` `<i>`以外的标签均可能失效，必须逐一核查。

**误区二：用大量富文本标签代替专用动画组件来实现文字特效**
开发者有时会每帧动态更新`<color>`标签的十六进制值来实现文字闪烁效果。这种做法每帧都触发完整的字符串解析→顶点重建→网格上传流程，当文本量超过200字符时，在移动端（如骁龙870）上单帧CPU耗时可达0.8ms以上。正确做法是解析完成后直接修改`TMP_Text.textInfo`中对应字形的顶点颜色数组，然后调用`UpdateVertexData(TMP_VertexDataUpdateFlags.Colors32)`，跳过解析阶段。

**误区三：假设图标Token不占用字符索引**
在使用光标定位、文本选择或链接热区检测时，开发者常误以为`<sprite>`标签不消耗字符索引。实际上TextMeshPro将每个图标精灵作为Unicode私用区（PUA，U+E000起）的一个字符插入字符序列，占据一个字符索引位置。若代码中用字符索引反推原始字符串位置时未考虑此偏移，会导致链接热区错位或选择范围计算错误。

## 知识关联

富文本系统建立在**文本溢出处理**的基础上：混合元素序列的折行算法需要在计算断点时同时考虑文字宽度和图标精灵宽度，图标不可在内部断行，这要求折行器将图标Token视为不可分割的原子单元（类似CJK全角字符）。若项目的文本溢出处理模块未开放逐Token宽度查询接口，富文本系统将难以正确实现图文混排的自动换行。

向前延伸，富文本系统直接支撑了**伤害数字设计**的实现需求：暴击伤害的金色放大字、格挡后的蓝色减伤数字，本质上都是带颜色标签和尺寸标签的富文本片段，只是它们需要在世界空间而非屏幕空间渲染，对基线对齐和渲染层级有额外要求。理解富文本系统的顶点着色机制，是进一步实现伤害数字描边、阴影与缩放动画组合效果的前提。
