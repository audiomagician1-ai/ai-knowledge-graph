# 页面转场动画

## 概述

页面转场动画（Page Transition Animation）是指游戏界面从一个页面切换到另一个页面时，用于连接两个状态的视觉过渡效果，常见形式包括淡入淡出（Fade In/Out）、方向性滑动（Directional Slide）、缩放（Scale Zoom）、翻转（Flip）以及遮罩擦除（Mask Wipe）。转场动画的首要职责不是装饰，而是让玩家感知到空间层级的变化——例如从主菜单"进入"战斗准备界面，需要让玩家感受到这是一次"向下钻取"而非平级跳转。

转场动画在数字游戏界面中的系统性运用，最早在1980年代街机游戏的吸引画面（Attract Mode）切换中出现，但现代意义上的转场设计规范源自2000年代移动平台兴起后的触屏交互体系。苹果iOS 7（2013年）引入了以物理层级为语义基础的推入（Push）和模态（Modal）切换动效，这套思路被大量游戏UI设计沿用至今。

游戏中转场动画之所以不同于普通应用界面，在于游戏存在多套并行的时间压力——玩家在匹配等待时可以接受600ms以上的缓慢溶解淡入，但在战斗中调出技能树界面则必须控制在200ms以内完成入场，否则打断感知流。这种"情境感知时长"是游戏转场设计最核心的差异化需求。

## 核心原理

### 时间曲线（Easing）与时长配置

转场动画的物理可信度依赖缓动函数（Easing Function）的选择。Ease-In-Out（先慢后快再慢）模拟真实物体的惯性运动，适合层级跳转；Linear（匀速）常用于淡入淡出，因为人眼对亮度变化的感知本身就接近匀速；Ease-Out（快进慢出）则适合页面滑入，让新内容像"自然停靠"而非突然中止。

在时长上，谷歌Material Design规范将界面转场分为三档：简单元素变换100–200ms、复杂页面切换200–500ms、强调性展开动效最长600ms。游戏UI通常压缩这些数值，将复杂页面切换控制在150–300ms，以减少操作响应延迟感。

### 空间层级与方向性语义

转场方向必须与信息架构一致，这是防止玩家迷失的关键机制。水平滑动（左右）通常表达同级页签切换，如角色属性页的各分类；垂直向上滑动表达"向前深入"，如从世界地图下钻到城镇界面；垂直向下或右滑关闭，表达"返回上一层"。缩放式转场（Scale-Up）传达"从小物件展开成全屏"的物理隐喻，例如点击背包中某个道具图标后该图标"放大"变为详情页。

当同一游戏内两种方向的转场互相冲突时（例如弹窗既从下方出现又从右侧消失），玩家会产生"空间感混乱"，这是转场设计中最常见的认知负担来源。

### 遮罩与上下文连续性

遮罩转场（Mask Transition）通过圆形、放射状或自定义形状的Alpha遮罩实现渐进式页面揭露，典型案例是《英雄联盟》客户端进入游戏大厅时的六边形扩散揭幕效果。遮罩转场相比直接淡入淡出，优势在于能保留前一页面的视觉锚点（Anchor Point）——玩家点击的位置成为遮罩展开的原点，从而维持"你触发了这个变化"的因果感知。

从技术实现角度，遮罩转场在GPU端通过Stencil Buffer或Alpha Shader完成，性能开销约为普通淡入淡出的1.5–2倍，在移动设备上需要评估目标机型的片元着色器（Fragment Shader）带宽限制。

## 实际应用

**RPG主菜单→角色选择界面**：常用从中心向四周的Scale-Up转场，因为主菜单通常视觉密度低而角色选择页密度高，放大效果让玩家感觉"进入了一个更大的空间"，而非被替换掉内容。

**卡牌游戏手牌→详情放大**：采用"共享元素转场"（Shared Element Transition），卡牌图片本身作为共享元素从手牌位置飞行放大至屏幕中央，避免玩家丢失对"这是我点击的那张卡"的追踪。Unity的DoTween或Animator State Machine均可实现此类共享元素转场，关键是前后界面复用同一RectTransform路径而非销毁重建。

**战斗内技能界面弹出**：通常使用从屏幕底部边缘向上滑入（Bottom Sheet模式）配合背景半透明遮罩，时长控制在180ms，Ease-Out曲线。这种方案保留战场背景可见，强化了"这是临时覆盖层而非页面替换"的语义。

**游戏过关结算页**：常使用延迟序列淡入（Staggered Fade），各数值元素按0–80ms的错位延迟依次出现，让玩家的视线被引导从总评分→细分得分→奖励依次流动，而非所有信息同时弹出造成视觉噪声。

## 常见误区

**误区一：所有转场都加Ease-In-Out曲线显得"专业"**
Ease-In-Out在页面滑入时会产生初始的"缓慢启动感"，导致200ms的动效主观感受像300ms以上。对于游戏中频繁触发的功能页面（背包、技能树），应优先使用Ease-Out（快入慢停），让响应感更即时，仅在全屏重量级转场（如进入战斗loading）中使用Ease-In-Out的仪式感。

**误区二：转场时长统一使用固定值**
将全游戏转场统一设置为300ms是常见的懒惰实现。小范围弹窗（屏幕面积＜25%）300ms会显得迟缓，而全屏替换如果只有150ms则会造成视觉跳跃（Visual Jump）。正确做法是按界面覆盖面积比例设定时长，参考公式：`T = 100ms + 覆盖面积比 × 250ms`，小弹窗约125ms，全屏转场约350ms。

**误区三：转场动画可以掩盖加载延迟**
用长时间转场动画掩盖资源加载是反人体工程学的设计。玩家能感知到"动画结束后仍然在等待"与"动画结束立即可交互"的差别，前者会显著降低界面的流畅感评分。正确方案是异步预加载目标页面资源，转场动画仅作视觉过渡而非加载等待的掩护。

## 知识关联

页面转场动画以**微交互设计**为前置知识——微交互中对反馈时机（Trigger→Feedback延迟）和状态变化可感知性的理解，直接决定转场何时启动、如何传达"切换正在发生"的即时反馈。微交互中的单元素状态动效（如按钮按压形变）与转场动画的区别在于：前者是同页内元素响应，后者涉及两个完整页面状态之间的过渡，复杂度和性能成本不在同一量级。

在此基础上，**元素入场动画**是转场动画的自然延伸：转场负责"两页之间的通道"，而元素入场动画负责"新页面内部各元素如何依次呈现"。两者的设计必须在时间轴上协调——转场结束时机与首批入场元素开始时机之间的重叠比例（通常建议重叠50–80ms）直接影响整体流畅感。如果转场和入场之间留有超过50ms的空白帧，玩家会感知到两段割裂的动效而非一个连续的"进入新界面"体验。
