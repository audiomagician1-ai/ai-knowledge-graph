# 文本溢出处理

## 概述

文本溢出处理是指当字符串内容超出其容器边界时，游戏UI系统所采用的视觉呈现与交互策略。在游戏界面中，角色名称框、物品描述栏、对话气泡、任务标题等大量元素都存在固定尺寸约束，而玩家输入或数据库内容的字符数量往往无法预测，因此溢出处理成为字体排版中必须主动设计的环节，而非事后补救的措施。

这一问题在游戏UI中比通用软件界面更为突出，原因在于游戏内容高度多语言化。同一段英文UI文字在德语本地化后长度平均增加30%到35%，而中日韩文字在等宽字体下每个字符占据更大的像素宽度。Unity的TextMeshPro组件和Unreal Engine的UMG文本控件均内置了溢出模式枚举（Overflow Mode Enum），说明这一需求已被引擎层面标准化。

正确处理文本溢出直接影响游戏的可读性和专业感。未处理的溢出会导致文字渲染超出UI边框、与相邻元素重叠，甚至触发裁切层级错误，在移动端游戏中尤其会引发用户投诉。

## 核心原理

### 截断（Truncation）

截断是最简单的溢出策略：当文本宽度超出容器时，超出部分直接被裁掉，不显示任何提示符号。截断点通常以像素为单位计算，渲染器逐字符测量累计宽度，一旦第N个字符的右边界超过容器宽度，第N个及之后的字符全部丢弃。

截断的技术实现依赖**字形宽度测量（Glyph Advance Width）**，每个字符的前进宽度在字体文件的`hmtx`表中存储。截断适用于玩家ID标签、排行榜名次列等对内容完整性要求低、空间极为紧张的场景。其缺点是用户无法感知内容被省略，因此使用场景受限。

### 省略号（Ellipsis）

省略号策略在截断的基础上，将最后可见区域的若干字符替换为"…"（U+2026，单字符省略号）或三个英文句点"..."。计算逻辑如下：设容器宽度为 `W`，省略号字形宽度为 `E`，则实际可用于正文字符的宽度为 `W - E`，渲染器在此宽度内尽量排列原始字符，剩余空间填入省略号。

CSS标准中对应属性为 `text-overflow: ellipsis`，需配合 `overflow: hidden` 和 `white-space: nowrap` 才能生效，这三行声明构成省略号显示的最小完整配置。在Unity TextMeshPro中，对应模式为 `TextOverflowModes.Ellipsis`。省略号广泛用于物品名称、任务标题（如《魔兽世界》任务日志标题超过约28个英文字符时自动省略），是兼顾空间节省与内容提示的平衡方案。

### 自动换行（Word Wrap / Auto Wrap）

自动换行允许文本在遇到容器右边界时折到下一行，容器高度随之增长或保持固定（固定高度时超出部分被截断）。换行算法有两种主流策略：**贪心换行（Greedy Wrap）**逐行填充尽可能多的单词；**最优换行（Knuth-Plass Algorithm）**由Donald Knuth和Michael Plass于1981年在TeX排版系统中提出，通过最小化全段落的"坏度值（Badness）"来决定断行位置，避免出现最后一行只有单个孤立单词的情况。

游戏引擎通常使用贪心换行而非Knuth-Plass，因为游戏文本段落较短，且Knuth-Plass的计算开销在逐帧刷新的动态UI中代价较高。中文文本由于无空格分词，换行点可在任意字符间插入，但应避免标点符号（句号、逗号、顿号）出现在行首，这一规则称为**行首禁则（Forbidden Start Characters）**，Unicode标准在LineBreak属性中对其有明确定义。

### 滚动显示（Scrolling / Marquee）

滚动显示通过动画位移让超长文本在固定窗口内水平或垂直滚动，完整内容依次呈现给玩家。水平滚动（跑马灯效果）常见于游戏内广播频道消息、Loading提示文本；垂直滚动则用于角色属性面板说明、NPC对话历史记录框。

实现上，文本渲染到一张离屏纹理（Off-screen Texture），其宽度等于完整文字的像素宽度，然后以固定速度（通常为每秒60至120像素）更新纹理的UV偏移量或Transform位移。循环滚动时需在文本末尾与开头之间插入一段空白间距（Gap），防止两次循环的内容视觉上粘连在一起，间距通常取容器宽度的50%至100%。

## 实际应用

**角色命名框**：《英雄联盟》的玩家ID最长允许16个字符，超出时在战绩页面使用省略号截断，但在游戏内血条上方则强制使用较小字号加截断，优先保证空间一致性。

**物品描述框**：RPG游戏中装备词条描述采用自动换行，描述框高度固定为约4行行高，超出时出现竖向滚动条（PC端）或滑动手势区域（移动端）。《原神》圣遗物描述框在超长词条下采用自动缩减字号（Auto Size）配合换行，将字号从16px逐步收缩至最小10px，再触发滚动。

**成就与任务标题**：单行标题区域几乎统一采用省略号策略，因为玩家可通过悬停Tooltip查看完整文本，溢出处理只需做到"可知有更多内容"即可。

**聊天频道滚动公告**：跑马灯文本的速度设计需考虑阅读舒适度，过快（超过200像素/秒）导致无法阅读，过慢（低于40像素/秒）让玩家等待焦躁，80至120像素/秒是多数游戏采用的经验值区间。

## 常见误区

**误区一：省略号可以随意截断中文字符**。部分开发者在截断中文时仅以字节数而非字符数（或字形宽度）计算截断点，导致多字节编码（如UTF-8下中文占3字节）被截断在字符中间，输出乱码或问号。正确做法是始终以**字符（Codepoint）**或**字形宽度（px）**为单位操作，而非字节数。

**误区二：自动换行不需要考虑连字符（Hyphenation）**。在英文长单词（如"internationalization"）或德语合成词中，贪心换行会将无法在单词间断行的长词整体推到下一行，导致上一行留有大片空白。游戏文本框宽度通常偏小（200至400px），此问题尤为明显。应为英文内容配置连字符词典，或设置最小字符数阈值（如单词超过8个字符才允许连字符断行）。

**误区三：滚动显示可以替代换行**。跑马灯对于需要快速扫读信息的玩家极不友好，特别是在时间敏感的战斗UI（如Buff描述）中，玩家往往在看完一遍之前已经需要作出决策。滚动显示仅适用于装饰性或非紧急信息，不应作为正文内容溢出的通用解决方案。

## 知识关联

字距与行距的设置直接决定了溢出触发的临界点：行距（Line Height）的增大会加快垂直溢出的发生，而字距（Letter Spacing）的正值会让同等字符数的文本更早触发水平溢出。在调试溢出策略时，必须在字距行距参数已锁定的状态下进行，避免两组参数的交叉影响。

文本溢出处理为后续的**富文本系统**奠定了基础。富文本中嵌入的内联图标（Inline Sprite）、颜色标签（`<color=#FF0000>`）、加粗斜体标签会改变单个"字符"的实际渲染宽度，使得简单的字符计数截断失效，必须对富文本节点树进行宽度计算后再决定溢出切断点。TextMeshPro的富文本解析器在计算省略号位置时，会先剥离标签获得可见字符序列，再测量其字形宽度，这一机制正是从基础溢出处理演进而来的。
