# 可用性测试概述

## 概述

可用性测试（Usability Testing）是通过观察真实用户执行具体任务来发现界面设计缺陷的系统性方法。在游戏UI/UX领域，可用性测试专门评估玩家能否高效、准确、满意地完成游戏内操作——例如打开背包、装备武器、导航至任务地图或完成首次付费流程。与问卷调查不同，可用性测试记录的是玩家的**实际行为**，而非其主观表述。

该方法论由人因工程学（Human Factors Engineering）在20世纪70年代发展而来，Jakob Nielsen与Rolf Molich于1990年将其正式化并引入软件界面评估领域。游戏行业在2000年代初期随着主机游戏复杂度提升（如《魔兽世界》2004年上线时的系统界面）才开始大规模应用可用性测试，任天堂、EA、育碧等大型游戏公司逐渐建立了专职的用户研究团队。

游戏UI的可用性问题会直接造成玩家流失。研究数据显示，移动游戏在首次启动后的前10分钟内流失率高达60%—80%，其中相当大比例源于界面引导不清、按钮位置不直觉、信息层级混乱等可用性缺陷。早期发现这些问题的成本，比产品上线后修复低约100倍（Nielsen的"百倍规则"）。

## 核心原理

### 可用性的五维度模型

Jakob Nielsen定义了可用性的五个可测量维度，游戏UI测试必须逐一评估：

1. **可学习性（Learnability）**：新玩家第一次接触UI时完成任务所需的时间。例如，一款MOBA游戏的新玩家应能在5分钟内理解小地图的标记含义。
2. **效率（Efficiency）**：熟练玩家完成特定操作的速度。例如，顶级玩家在RPG中打开并关闭角色属性面板的平均操作时长不应超过2秒。
3. **可记忆性（Memorability）**：玩家停玩一段时间后重返游戏，恢复操作熟练度的速度。
4. **错误率（Errors）**：玩家在执行任务时发生误操作的频率与严重程度。
5. **满意度（Satisfaction）**：通过SUS量表（System Usability Scale，共10题，满分100分）或NPS净推荐值量化玩家的主观感受。SUS得分68分以上被认为达到行业平均水平。

### 测试类型与游戏阶段对应

游戏开发的不同阶段适用不同的可用性测试方式：

- **纸质原型测试（Paper Prototyping）**：适用于预生产阶段，测试菜单结构和信息架构，成本极低。测试者手动模拟界面响应，玩家用手指"点击"纸质按钮。
- **思出声测试（Think-Aloud Protocol）**：适用于Alpha阶段，要求玩家在完成任务时实时说出所有想法，研究员记录困惑点。这是发现"玩家以为自己理解但实际不理解"问题的最有效方法。
- **远程无主持测试（Remote Unmoderated Testing）**：适用于Beta阶段，使用UserTesting.com或Lookback等平台向数十至数百名玩家分发任务，通过录屏和眼动热图分析数据。
- **A/B测试**：适用于上线后的迭代优化，同时向不同玩家群体展示两种UI方案，通过点击率、任务完成率等量化指标决定最终版本。

### 关键量化指标

可用性测试必须收集以下可量化数据，否则无法进行版本间横向对比：

- **任务完成率（Task Completion Rate）**：成功完成指定任务的用户百分比。游戏核心流程（如首次购买道具）的完成率目标通常设定为≥85%。
- **任务完成时长（Time on Task）**：从任务开始到完成所经过的秒数，越短说明界面越直觉。
- **错误次数（Error Count）**：玩家在完成任务前触发的错误操作次数，包括点击了错误按钮、进入了错误页面等。
- **首次点击准确率（First Click Accuracy）**：研究表明，用户第一次点击正确位置的概率与最终任务完成率高度相关（相关系数约0.87）。

## 实际应用

**案例一：手机RPG的新手引导优化**  
某手机RPG游戏在封闭测试阶段发现，"强化装备"功能的任务完成率仅为43%——远低于85%的目标值。通过思出声测试，研究员发现玩家普遍无法区分UI中"强化"与"合成"两个相邻按钮（两者图标相似度过高，尺寸均为44×44dp，文字标签字号仅为10pt）。设计团队将两个按钮的色彩区分度从原来的对比度1.8:1提升至4.5:1（符合WCAG 2.1 AA标准），并将字号增大至14pt。下一轮测试中，该任务完成率回升至89%。

**案例二：PC射击游戏的HUD（平视显示器）密度评估**  
一款战术射击游戏通过眼动追踪测试发现，玩家在战斗中有72%的注视时间集中在屏幕中央30%的区域，而生命值和弹药数量显示在屏幕四角，导致关键信息读取延迟平均达到1.2秒。研究团队据此建议将核心状态信息迁移至屏幕下方中央区域，并在后续测试中验证该改动将信息读取时间缩短至0.3秒。

**案例三：主机游戏的控制器按键映射**  
在游戏机版本的测试中，可用性测试专门评估"按键映射直觉性"——即玩家首次接触游戏时猜测正确按键功能的比例。若"跳跃"功能绑定在非行业惯例键位（例如违反Xbox的A键=确认惯例），玩家错误操作率将显著上升，这类问题在纸质原型阶段即可通过5人小组测试发现。

## 常见误区

**误区一：测试用户越多越好**  
许多团队认为需要招募数十名测试者才能获得有效结论。但Nielsen的研究证明，**5名用户可以发现约85%的可用性问题**，每增加一个用户发现新问题的边际收益递减。对于游戏UI测试，建议按用户细分群体（如核心玩家、休闲玩家、首次玩家）各招募5人，总计15人，而非盲目扩大样本量。

**误区二：可用性测试等同于用户喜好调研**  
可用性测试的核心是测量"玩家能否完成任务"，而非"玩家喜不喜欢这个界面"。玩家可能会表示非常喜欢某个华丽的UI动效，但实际完成任务的时间却是简洁版本的3倍。这类"喜欢但无法高效使用"的界面正是可用性测试要揭露的典型问题，单纯的满意度问卷无法识别此类缺陷。

**误区三：游戏可用性测试只需在上线前进行一次**  
可用性问题会随每次版本更新引入新的UI改动而出现。以《堡垒之夜》为例，每个赛季更新后开发商Epic Games都会重新评估新功能（如新武器的购买界面、新赛季通行证的导航流程）的可用性，而非依赖旧版本的测试结论。可用性测试应嵌入游戏的整个生命周期，特别是每次涉及UI改版的大版本更新前后都应执行。

## 知识关联

**前置知识**：可用性测试不要求任何正式前置知识，但了解游戏设计基础概念（如HUD、菜单层级、交互反馈）有助于设计更有针对性的测试任务。

**衔接启发式评估**：完成本章学习后，下一步将学习**启发式评估（Heuristic Evaluation）**。可用性测试依赖真实用户参与，而启发式评估由3—5名专家根据Nielsen的10条可用性原则（如"系统状态可见性""错误预防"）自行检查界面。两种方法发现的问题类型存在差异——启发式评估更擅长发现系统性设计违规，可用性测试更擅长发现真实用户的具体困惑点，两者结合使用覆盖率最高。

**衔接无障碍设计**：可用性测试中采集的对比度数据、字号数据和操作错误率数据，与**无障碍设计（Accessibility Design）**直接相关。可用性测试发现的"普通用户难以识别"的UI元素，对于色觉障碍或运动障碍玩家而言问题更为严重。因此，可用性测试结果常被用作制定无障碍设计改进优先级的依据。
