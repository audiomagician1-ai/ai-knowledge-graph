# 缓动曲线

## 概述

缓动曲线（Easing Curve）是描述动画属性值随时间变化速率的数学函数，它决定了一个动画从起始值到结束值的"加速/减速节奏"。最简单的缓动是线性函数 `f(t) = t`，即速率恒定；而现实中几乎所有自然运动都不是匀速的——球落地时加速，推开门时先慢后快再慢停。缓动曲线正是用来模拟这种非线性运动的工具。

缓动函数最早由 Robert Penner 在2002年出版的《Flash ActionScript 动画》中系统整理并发布，他定义了一套以 `easeIn`、`easeOut`、`easeInOut` 为分类的函数族，涵盖 Quad、Cubic、Quart、Quint、Sine、Expo、Circ、Elastic、Back、Bounce 共10种形态，这套体系至今仍是 CSS `animation-timing-function` 和各类游戏引擎动画系统的参考基础。

在游戏UI设计中，缓动曲线直接影响玩家对界面"重量感"和"响应性"的感知。一个技能冷却图标用线性缓动填充，显得呆板机械；换成 `ease-out` 则开头快速响应、结尾平滑收尾，符合玩家期待"立刻有反应"的心理预期。错误的缓动选择会让原本精心设计的UI动效显得廉价或卡顿。

## 核心原理

### 三次贝塞尔曲线与 cubic-bezier 参数

现代UI系统（包括Unity的Animation Curve和CSS）普遍用**三次贝塞尔曲线**来定义缓动，其数学形式为：

```
B(t) = (1-t)³·P0 + 3(1-t)²t·P1 + 3(1-t)t²·P2 + t³·P3
```

其中 `t ∈ [0,1]`，`P0=(0,0)` 和 `P3=(1,1)` 固定为起止点，用户只需控制两个控制点 `P1(x1,y1)` 和 `P2(x2,y2)`。CSS 的 `cubic-bezier(x1, y1, x2, y2)` 就是这四个参数的简写。

以 `ease-out` 为例，其等价于 `cubic-bezier(0, 0, 0.58, 1)`——P1 靠近起点意味着初速快，P2 的 y 值接近1而 x 值偏后意味着末段减速舒缓。了解这四个数字的物理含义，才能在引擎中精准调整而不是随机拖拽手柄。

### 五种基础缓动类型及适用场景

**Linear（线性）**：`f(t) = t`，速率恒定为1。仅适用于颜色渐变、不透明度淡出等感知上不需要"动感"的属性变化，以及进度条这类需要精确对应实际进度的场景。在位移动画中几乎不单独使用。

**Ease-in（渐入加速）**：开始慢、结束快，等价于 `cubic-bezier(0.42, 0, 1, 1)`。适合表示"被推出屏幕"或"消失"的元素——界面元素退出时从缓慢启动到高速离场，符合物体被甩出的物理感。如游戏中击败怪物后奖励面板滑出屏幕时使用。

**Ease-out（渐出减速）**：开始快、结束慢，等价于 `cubic-bezier(0, 0, 0.58, 1)`。适合"进入"动作，元素从屏幕外快速冲入然后平稳停止，像一个有重量的物体滑入摩擦停止。游戏UI中最常用的缓动，弹出面板、提示卡片几乎默认使用此类型。

**Ease-in-out（两端慢中间快）**：`cubic-bezier(0.42, 0, 0.58, 1)`，模拟物体从静止加速再到静止的完整运动。适用于同屏内两点间移动的元素，例如角色选择界面中角色卡片左右轮播。

**Elastic（弹性）**：超出目标值后反弹，Robert Penner 的弹性公式包含振荡分量，产生"弹簧感"。常见于游戏内成就徽章弹出、血量条回弹、金币数字跳动等强调"活泼"的反馈动效。振荡频率由参数 `amplitude` 和 `period` 控制，`period` 越小弹动越快越密集。

### 缓动与动画时长的联动关系

缓动曲线必须与动画时长（Duration）配合才能产生预期感知。Google Material Design 3 明确规定：退出动画使用 `emphasizedAccelerate`（自定义 ease-in），时长推荐 **200ms**；进入动画使用 `emphasizedDecelerate`（自定义 ease-out），时长推荐 **300ms**。同样的 ease-out 曲线在100ms内完成显得"跳出"，在600ms内完成则显得"沉重拖拉"。游戏UI通常比移动APP动效更短促（100~250ms），以避免打断玩家节奏。

## 实际应用

**技能图标冷却遮罩**：冷却覆盖层从满到空的扫除动画，如果用 ease-in-out，玩家会误以为技能"快好了"又"变慢了"，产生焦虑感。正确做法是使用 **linear**，因为冷却时间与游戏逻辑时间严格对应，线性才符合玩家的时间预期。

**背包/仓库面板滑入**：背包从右侧滑入使用 `ease-out`（约250ms），让玩家感觉面板"稳稳停下"。但若面板关闭（滑出）用同样的 `ease-out`，初始速度慢会让玩家觉得"反应迟钝"。因此关闭应改用 `ease-in`（约180ms），开头快速响应点击，符合"立刻响应"的交互预期。进出使用不同缓动是游戏UI的常见模式。

**伤害数字飘字**：伤害数字从怪物头顶向上飘出并淡出，上升阶段用 `ease-out`（从快到慢，像物体上抛后减速），透明度淡出用 `linear`。两个属性可以同时运动但使用不同的缓动曲线，Unity 的 Animator 和 DOTween 均支持对单个 Tween 独立设置 `Ease` 枚举值。

**弹窗出现弹性效果**：充值成功弹窗出现时，缩放从 `0.8` 到 `1.0` 使用 `Back ease-out`（overshoot 参数约1.7），弹窗会短暂放大到约 `1.05` 后回弹到1.0，产生"砰"地出现的活泼感。但严肃的错误提示或警告弹窗应禁止使用弹性缓动，否则会破坏严肃感。

## 常见误区

**误区一：所有动画统一使用 ease-in-out**
许多初学者认为"两端慢中间快"最自然就全部套用。但 ease-in-out 假设动画发生在同一坐标系内的往返运动，当元素从屏幕外进入时起点在视口外，没有"缓慢启动"的视觉空间，开头的慢速段会让玩家感觉界面"卡了一下"才开始响应。进入动画应用 ease-out，退出动画应用 ease-in，是基于感知心理学的规律而非偏好。

**误区二：弹性缓动 = 更生动，越多越好**
弹性缓动（Elastic/Back）在同一界面中若超过3处同时使用，会造成视觉噪音，让玩家觉得界面"嘈杂"而非"活泼"。弹性缓动的超调量（overshoot）若超过目标值的10%（例如缩放超过1.1），在小屏设备上还会触发元素超出容器边界的裁剪 bug。应将弹性效果保留给最核心的奖励/强调反馈场景。

**误区三：以为修改缓动曲线等同于改变动画速度**
缓动曲线只改变速率分布，不改变总时长。把 `linear` 改为 `ease-out` 后，动画开头更快，但整体完成时间不变。若玩家反馈"动画太慢"，正确解法是缩短 Duration，而非把缓动改成"更陡"的曲线——改变曲线陡峭度只会让中间某段更快，但开头/结尾的感知时间可能反而变长。
