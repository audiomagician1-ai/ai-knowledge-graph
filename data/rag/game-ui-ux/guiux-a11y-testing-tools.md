# 无障碍测试工具

## 概述

无障碍测试工具是专门用于检测游戏UI/UX是否满足残障玩家使用需求的软件和方法集合，涵盖色盲模拟器、屏幕阅读器兼容性测试以及自动化合规检查工具三大类别。与通用游戏测试（Playtest）不同，无障碍测试工具针对视觉、听觉、运动和认知障碍场景进行专项验证，而非检测游戏逻辑或性能问题。

无障碍测试工具的系统化应用始于2010年代初，随着WCAG（网页内容无障碍指南）2.0标准在2008年发布后逐步扩展至游戏领域，开发者开始将原本用于网页的测试方法移植并改造为适配游戏引擎的专用工具。2019年Xbox推出《无障碍功能指南》后，主机游戏开发商开始将这类工具纳入正式QA（质量保证）流程，形成标准化测试规范。

使用无障碍测试工具的核心价值在于：仅凭开发者的主观判断无法可靠评估色盲玩家或屏幕阅读器用户的实际体验，工具能够客观量化对比度比值、标签覆盖率等可测量指标，将"无障碍设计是否充分"这一抽象问题转化为可验证的具体数据。

## 核心原理

### 色盲模拟器的工作机制

色盲模拟器通过色彩矩阵变换算法，将屏幕像素颜色转换为特定类型色盲患者所感知的等效色彩。常见工具如 **Coblis** 和 **Color Oracle** 支持模拟七种视觉异常类型，包括红绿色盲（Deuteranopia，影响约8%男性）、蓝黄色盲（Tritanopia）和全色盲（Achromatopsia）。模拟器使用的核心变换公式为线性RGB空间中的3×3矩阵乘法，例如模拟Deuteranopia的变换矩阵会将绿色通道的感知权重大幅削减并部分转移至蓝色通道。使用色盲模拟器时，测试人员应重点检查游戏地图、状态指示器和敌我识别标记——这些元素若仅靠颜色区分，在色盲模拟下通常会完全失去可读性。

### 屏幕阅读器兼容性测试

屏幕阅读器测试验证游戏UI元素是否具备正确的可访问性标签（Accessibility Label）和语义树结构，以便 **NVDA**（Windows）、**VoiceOver**（iOS/macOS）或 **TalkBack**（Android）等工具能够将界面内容朗读给视障玩家。测试者需要在关闭视觉输出的条件下，仅凭屏幕阅读器朗读的内容完成核心游戏流程，例如角色创建、物品栏管理和主菜单导航。Unity引擎通过 `AccessibilityNode` 组件暴露无障碍树，Unreal Engine则通过 `SlatePlatformAccessible` 接口实现；测试时需逐一验证每个可交互控件是否拥有描述性标签（如"确认按钮，当前不可用"而非仅播报"按钮"）。

### 自动化合规检查工具

自动化检查工具通过扫描UI截图或运行时DOM/控件树，依据 **WCAG 2.1 AA级** 标准自动计算对比度比值、检测缺失的Alt文本标签和键盘焦点顺序异常。专为游戏设计的工具如 **AXE for Games**（基于Deque Systems的axe-core引擎）可在CI/CD管道中集成，每次构建后自动生成无障碍问题报告。WCAG 2.1要求普通文本与背景的对比度不低于 **4.5:1**，大号文本（18pt以上）不低于 **3:1**；自动检查工具会对游戏中所有UI文字逐一计算此比值并标记不达标项。需要注意的是，自动化工具约能检测出全部无障碍问题的 **30%～40%**，其余问题仍需人工测试发现。

## 实际应用

**《地平线 零之曙光》（2022年PC版）** 在发布前使用色盲模拟器系统检查了全部HUD元素，发现资源标记图标的红绿配色组合在Deuteranopia模拟下无法区分，随后将图标形状差异化并增加白色轮廓，修复了该问题。

**《最后生还者 第二部》** 将屏幕阅读器测试作为核心QA流程之一，测试团队完整使用PlayStation自带的屏幕阅读器（PS Screen Reader）通关验证了所有菜单操作，确保全部60余项无障碍设置选项均可在不看屏幕的情况下独立调整。

在移动游戏领域，开发团队通常使用 **Google Accessibility Scanner**（安卓）在设备上实时扫描游戏界面，该工具会将检测结果以叠加层形式显示在屏幕上，直接标注出对比度不足或触控目标过小（低于48×48dp推荐尺寸）的UI元素。

## 常见误区

**误区一：色盲模拟器覆盖一次即视为通过色盲测试。** 许多开发者仅对最终截图运行色盲模拟器，却忽略了动态UI状态——如受伤时屏幕变红、选中状态高亮、倒计时颜色渐变等动态效果。色盲测试必须覆盖所有UI状态变化，而非静态截图。

**误区二：自动化工具通过即代表无障碍合规。** 自动化工具无法检测逻辑层面的无障碍问题，例如屏幕阅读器朗读顺序与视觉阅读顺序是否一致、游戏教程是否依赖颜色传达关键操作指令，这些问题必须通过具有残障经验的真实用户测试或专业测试员手动验证。

**误区三：放大镜功能正常等同于无障碍缩放测试通过。** 游戏内置放大镜功能（UI Scale）与操作系统级放大镜是两套独立机制，使用系统放大镜时游戏画面可能因全屏独占模式而无法正常放大，或放大后UI元素发生裁切。需使用操作系统放大镜工具（如Windows放大镜设置为400%）专项测试游戏界面的可读性。

## 知识关联

理解游戏内**放大镜功能**的实现方式（UI缩放与像素级放大的区别）是使用无障碍测试工具的前提：测试时需区分哪些界面问题源于放大镜机制本身，哪些是独立的颜色或标签缺陷。同样，具备**游戏测试（Playtest）**的标准流程经验有助于将无障碍测试整合进现有QA节点，而非将其视为独立的附加环节。

掌握无障碍测试工具的使用方法后，可直接对接**Xbox无障碍指南**（Xbox Accessibility Guidelines，XAG）的具体合规要求——XAG的47项功能标准中超过20项需要通过本文所述的色盲模拟、屏幕阅读器测试和自动对比度检查来验证，形成从工具使用到平台规范合规的完整能力链条。
