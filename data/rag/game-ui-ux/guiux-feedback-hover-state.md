# 悬停状态设计

## 概述

悬停状态设计（Hover State Design）专指鼠标光标移入UI元素热区时触发的视觉与信息变化机制。这种状态存在于鼠标"空闲"与"点击"之间的中间地带，持续时间通常为用户决策阶段，可能仅持续100毫秒，也可能长达数秒。悬停状态的核心职责是向玩家传递"此处可交互"的信号，并在不打断游戏流程的前提下提供额外信息。

悬停反馈在桌面GUI游戏中出现于1980年代末，随着《魔兽争霸》（1994）等策略游戏将单位属性显示在悬停工具提示中而被系统化。移动端因缺乏悬停输入设备，长期按压（Long Press）作为其功能替代方案。时至今日，PC游戏的悬停状态设计仍是桌面平台区别于主机和移动端的核心交互优势之一。

悬停设计直接影响玩家的决策效率。在RPG物品栏中，如果悬停工具提示在300毫秒内未出现，玩家会感到明显迟滞；若超过500毫秒，会主动移开光标重新触发。正确设计悬停反馈能让玩家在不打开任何子菜单的情况下完成超过60%的信息核查行为，显著降低界面层级跳转频率。

## 核心原理

### 悬停视觉变化的三层结构

悬停时的视觉变化通常分为三个层次叠加呈现。**第一层**是元素自身的形态变化，包括亮度提升（通常为原始亮度的115%～130%）、边框高亮、轻微缩放（scale 1.0 → 1.05）或颜色偏移。**第二层**是光标形态变化，例如从箭头切换为手型指针，明确告知玩家当前元素可被操作。**第三层**是辅助信息层的出现，即工具提示（Tooltip）或预览面板浮现。三层变化叠加时需注意视觉权重分配，避免过度动效抢占玩家注意力。

### 工具提示的出现时机与消失时机

工具提示的显示延迟（Hover Delay）是设计中最关键的参数之一。行业通行标准是**200～400毫秒**的显示延迟，这一区间源于Fitts定律的延伸研究：低于200ms会导致玩家无意中触发大量提示，形成视觉干扰；高于400ms则使玩家感到系统迟钝。《暗黑破坏神3》采用了约250ms的延迟，而《英雄联盟》的物品提示延迟设定在约300ms。消失延迟同样重要：光标移出后，工具提示应在**100～150毫秒**内淡出，保留短暂残影避免画面跳变。

### 预览信息的层级设计

悬停状态下的预览信息需遵循"渐进式披露"原则：初级悬停（0.3秒内）只显示名称与简短描述；持续悬停（超过1秒）可展开完整属性面板或比对信息。在装备类RPG中，悬停预览往往还需触发"属性对比"逻辑——将悬停物品与玩家当前装备数值进行差值计算，用绿色（正增益）和红色（负增益）数字直观呈现，例如「攻击力 +24」。这种对比预览是悬停状态独有的信息密度优势，点击操作无法在同一步骤完成相同功能。

### 悬停热区与视觉边界的一致性

悬停触发区域（Hit Area）必须与视觉显示区域保持一致或略大，不可小于视觉边界。常见的错误设计是图标视觉尺寸为32×32像素，而实际热区仅有24×24像素，导致玩家在图标边缘移动时无法触发悬停反馈，产生"点不准"的挫败感。推荐的热区扩展量为在视觉边界外扩4～8像素，尤其对小尺寸图标（小于40×40px）这一扩展尤为必要。

## 实际应用

**物品栏系统**：在《路径流放者》（Path of Exile）中，装备栏每个格子的悬停工具提示包含词缀类型、数值范围及与同类物品的评分对比，工具提示宽度固定为320像素以保证排版稳定，不随词缀数量动态扩展宽度。

**技能与天赋树**：《暗黑破坏神4》的天赋树节点悬停时，右侧会实时渲染一个独立的技能预览卡，包含动态伤害算式和当前属性代入的计算结果。这要求悬停触发事件同时向数据层发起一次轻量计算请求，设计与工程需共同约定这次请求的响应时间上限不超过100ms。

**策略游戏单位**：《星际争霸2》中悬停在单位上时会在状态栏显示移速、护甲、伤害类型等参数，同时在小地图上以浅色光晕标示该单位的视野范围——这是将悬停预览从文字扩展至地图层级可视化的典型案例。

**菜单按钮**：主菜单按钮悬停时通常伴随音效反馈（频率在800Hz～1200Hz的短促音效效果最佳），视觉变化以亮度偏移为主，持续时长约80～120毫秒的过渡动画，避免使用超过200毫秒的缓动动画以防菜单显得迟缓。

## 常见误区

**误区一：将悬停状态与选中状态混用**。悬停（Hover）代表"我正在看这个"，选中（Selected/Active）代表"我选择了这个"，二者必须使用视觉上可区分的样式。若均采用相同的高亮颜色，玩家在重新打开界面后会误认为之前已选中的元素仍处于悬停聚焦中，导致误操作。标准做法是悬停使用较浅的色调叠加，选中态使用填充色或更强的边框对比。

**误区二：在移动端强行模拟悬停行为**。部分开发者在移动端游戏中使用"手指触碰但不抬起"来模拟悬停，并弹出工具提示。这与移动端玩家的操作惯例相冲突——触碰不抬起在大多数移动游戏中意味着拖拽操作的开始，工具提示的突然出现会打断拖拽判断逻辑，造成交互歧义。移动端应以专用的"信息按钮（ℹ）"代替悬停提示触发。

**误区三：工具提示遮挡悬停目标元素本身**。当工具提示弹出方向未经动态判断时，极易覆盖在被悬停的元素上方，使玩家无法同时查看提示内容与元素状态。正确做法是工具提示生成时检测四个方向的剩余空间，优先向光标移动方向的反方向弹出，并确保提示框与触发元素之间保留至少8像素间距。

## 知识关联

**前置概念——视觉确认反馈**：悬停状态是视觉确认反馈体系中专属于"预交互阶段"的子集。视觉确认反馈涵盖了从悬停到点击完成的整个操作周期，而悬停状态设计专注于光标进入热区到用户决定是否点击之间的那段时间窗口内的反馈策略。

**后续概念——按压响应设计**：悬停状态结束后，如果玩家决定点击，则进入按压响应（Press/Active State）阶段。按压状态的视觉变化需在悬停态基础上进一步强化，通常采用更深的颜色偏移或元素内缩效果（scale 1.05 → 0.97），形成"悬停亮起 → 按压下沉"的连贯动态叙事。两者的设计需在同一组件规范文档中统一定义，确保状态切换的视觉权重关系合理递进。
