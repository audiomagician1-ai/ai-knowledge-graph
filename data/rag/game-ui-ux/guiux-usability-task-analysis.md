# 任务分析

## 概述

任务分析（Task Analysis）是可用性测试中用于系统分解用户操作路径的方法论，其核心是将玩家完成某一目标（如"购买游戏内道具"或"组建五人队伍"）所经历的每一步操作转化为可测量的数据序列。与出声思维法记录玩家主观感受不同，任务分析专注于**客观行为路径**——记录按钮点击顺序、菜单层级跳转次数以及操作耗时，形成可重复比较的量化指标。

任务分析方法起源于1960年代工业工程领域的作业分析（Job Task Analysis），由 Thomas Gilbert 在行为工程模型中正式化。1990年代，随着GUI软件的普及，Jakob Nielsen 和 Stuart Card 将其引入人机交互领域，确立了以"任务完成率（Task Completion Rate）"和"平均操作步骤数"作为核心度量指标的分析框架。游戏行业则在2005年前后大规模采用该方法，尤其在大型MMO和移动游戏的新手引导设计优化中形成了专门的应用规范。

在游戏UI/UX中，任务分析的价值在于其**精确定位失败节点**的能力。一款手游的内购转化率下降3%，可能源于支付流程中某个隐藏的确认弹窗，而非整体界面设计问题。没有任务分析将操作路径拆解到单步层级，这种细粒度的问题定位无法实现。

## 核心原理

### 层级任务分析（HTA）结构

层级任务分析（Hierarchical Task Analysis）是任务分析最主流的结构化方式。其基本逻辑是将一个顶层目标递归拆解为子任务，直到每个末端操作不可再分。以游戏内"修改角色装备"为例：

```
目标：更换武器
  └─ 1. 打开背包界面（点击背包图标）
  └─ 2. 定位武器栏位（滚动列表或切换标签）
  └─ 3. 选择目标武器（点击物品）
  └─ 4. 确认装备（点击"装备"按钮或拖拽）
```

每个末端操作对应一个**操作节点（Operation Node）**，分析员记录每个节点的完成率、平均耗时和错误率。如果玩家在节点2的平均耗时超过节点1的3倍，则该步骤被标记为"摩擦点（Friction Point）"，需优先排查UI设计问题。

### 三大量化指标

任务分析依赖以下三项核心指标，构成评估游戏UI可用性的最小数据集：

**① 任务完成率（Task Completion Rate，TCR）**
$$TCR = \frac{\text{成功完成任务的测试者人数}}{\text{参与该任务测试的总人数}} \times 100\%$$
行业通用标准：新手引导核心任务的TCR应不低于85%，否则需触发重新设计流程。

**② 操作步骤数偏差（Step Count Deviation，SCD）**
测量实际操作步骤数与设计预期最短路径的差值。若某任务的最优路径为4步，但测试者平均需要7.2步，SCD = 3.2，表明UI引导存在明显歧义。

**③ 任务完成时间（Time on Task，ToT）**
记录从任务开始到确认完成的总秒数。游戏行业经验值：移动端核心交互任务ToT超过45秒时，玩家放弃率显著上升。

### 任务脚本设计规范

任务分析的执行质量高度依赖任务脚本的措辞。脚本必须使用**目标导向语言**而非操作导向语言。

- ❌ 错误写法："请点击右上角的背包图标，然后选择武器栏位。"
- ✅ 正确写法："请把你的角色换上库存里那把新获得的火焰剑。"

错误写法会提前暴露操作路径，导致测量数据失效，无法真实反映玩家面对界面时的自然导航行为。每个测试任务脚本应限制在2-3句话，避免引入多个并行子目标干扰数据纯洁性。

## 实际应用

### 移动端新手引导优化

某款三消手游在版本迭代后，第一关引导完成率从92%跌至71%。测试团队对"完成第一关并领取奖励"这一任务进行HTA分析，发现玩家在"点击奖励邮件"这一步骤的SCD高达4.1（预期1步，实际平均5.1步），原因是新版本将邮箱图标从主界面底部导航栏移至个人中心二级菜单。该问题通过任务分析在2小时内被精确定位，而非依赖模糊的玩家投诉反馈。

### MOBA游戏英雄选择界面

针对PC端MOBA的英雄选择流程，设计团队设定了以下四个分析任务：锁定英雄、绑定召唤师技能、设置符文页、查看对位英雄胜率。通过对比新手玩家（游戏时长<10小时）与熟练玩家（>100小时）的ToT数据，发现新手在"设置符文页"的平均耗时为熟练玩家的6.3倍（新手127秒 vs 熟练玩家20秒），直接推动了"推荐符文"一键应用功能的开发。

### 与出声思维法的协同使用

在实际游戏UX项目中，任务分析通常与出声思维法组合运行：先用出声思维法捕捉玩家在困惑节点的主观表述，再通过任务分析的SCD和ToT数据确认该节点是否在统计意义上构成系统性障碍。单独的出声思维法无法区分"玩家个人习惯差异"与"界面设计缺陷"，任务分析的量化数据提供了这一区分所需的统计基础。

## 常见误区

### 误区一：将"任务完成"定义为"最终目标达成"

许多初学者将任务完成率简单定义为"玩家是否最终完成了任务"，忽略**完成方式**的差异。若玩家绕过预设UI路径（如通过搜索功能代替导航菜单找到目标）完成了任务，应记录为"替代路径完成（Alternative Path Completion）"而非标准完成，否则会掩盖主流程的真实可用性问题。

### 误区二：最短路径即最优路径

任务分析中，操作步骤数并非越少越好。游戏内某些高风险操作（如删除角色、确认大额消费）故意增加确认步骤以降低误操作率，此时SCD为正值反而是设计意图的体现。分析员需在任务脚本设计阶段明确区分"效率型任务"（如背包整理）和"安全型任务"（如道具出售），采用不同的步骤数基准线。

### 误区三：样本量5人即足够

尼尔森（Nielsen）提出的"5人测试发现85%问题"结论适用于**发现性测试**，而任务完成率的统计可靠性需要更大样本。要使TCR的95%置信区间误差控制在±10%以内，至少需要约96名测试参与者。在游戏UX实践中，核心付费流程的任务分析至少应覆盖30名目标用户，才能使SCD均值具备基本统计意义。

## 知识关联

**前置概念：出声思维法**
出声思维法（Think Aloud Protocol）为任务分析提供了节点级别的质性解释材料。当任务分析的数据显示某步骤ToT异常偏高，对应的出声思维录音能直接揭示玩家的认知卡点，两者形成"量化定位→质性解释"的互补关系。

**后续概念：问卷与访谈**
任务分析结束后，测试者通常会填写任务后问卷（如系统可用性量表SUS或单题易用性量表SEQ）。问卷数据与任务分析的客观行为数据构成交叉验证——玩家主观评分与客观TCR的背离（如玩家打分较高但TCR偏低）本身就是重要的设计洞察，提示界面可能存在"玩家未意识到自己走了弯路"的隐性可用性问题。
