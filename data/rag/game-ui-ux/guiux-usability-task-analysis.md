# 任务分析

## 概述

任务分析（Task Analysis）是可用性测试中用于系统性拆解用户操作路径的研究方法，通过将玩家在游戏界面上的一次完整操作分解为若干可量化的子步骤，记录每个步骤的完成状态、耗时与错误类型。与出声思维法侧重捕捉玩家内心独白不同，任务分析更关注行为序列本身的结构：玩家点击了什么、按照什么顺序点击、在哪一步卡住并放弃。

该方法最初来源于1950年代的人因工程学（Human Factors Engineering），由工业心理学家用于分析工厂操作员的作业流程。进入数字产品领域后，尼尔森·诺曼集团在1990年代将其引入软件可用性评估，形成了以"层级任务分析"（Hierarchical Task Analysis, HTA）为代表的标准化流程。游戏UI/UX领域在此基础上加入了"容错步骤"和"误触率"两项专属指标，以应对触屏与手柄操作的特殊挑战。

在游戏开发流程中，任务分析的价值体现在两个具体节点：一是Alpha版本的界面原型验证，用于确认核心操作路径是否符合设计意图；二是上线后的版本迭代，通过比较两次测试的任务完成率差值来量化改版效果。一款手游的新手引导流程通常包含15～30个可拆解任务节点，任务分析能精确定位哪个节点的流失率超过可接受阈值（通常设定为20%）。

## 核心原理

### 层级任务分析（HTA）结构

层级任务分析将一个高层目标分解为多层嵌套的子任务，用"目标—操作—计划"三元组表示。以游戏内购买装备为例，顶层目标为"完成首次装备购买"，第一层子任务包括"打开商店界面""选择目标装备""确认支付"三个节点，每个节点再向下拆解为具体的界面点击步骤。HTA的正式符号记法将每条子任务标注为 `Task n.m`（n为父任务编号，m为子任务序号），并用"计划（Plan）"描述子任务之间的执行逻辑（顺序、条件分支或循环）。

### 三大核心量化指标

任务分析依赖三项可直接测量的指标：

**任务完成率（Task Completion Rate）**：在N名测试参与者中成功完成该任务的人数比例，公式为 `TCR = 成功人数 / 总参与人数 × 100%`。行业基准值为：新手引导关键任务TCR应≥85%，可选功能（如公会系统）TCR可接受下限约为65%。

**操作步骤数偏差（Step Deviation）**：测试者实际执行的操作步骤数与设计预期最优路径步骤数之差。若设计路径为4步，而90%测试者平均走了7步，步骤偏差为+3，说明界面存在明显的路径混淆。偏差值为负数时（测试者找到了比设计更短的路径）同样需要记录，这往往揭示了意外的捷径设计价值。

**首次点击准确率（First Click Accuracy）**：测试者面对新界面时，第一次点击落在正确区域的比例。研究表明，首次点击位置正确的用户最终完成任务的概率是首次点击错误用户的3倍，因此该指标对游戏UI布局优化具有高预测效力。

### 任务脚本的编写规范

任务分析的质量高度依赖任务脚本的表述方式。任务描述必须使用"场景化语言"而非"操作指令"：不能写"请点击右上角的商店按钮"，而应写"你刚刚获得了100金币，希望用它购买一把短剑"。前者已经透露了界面位置，后者才能真实测量玩家是否能自主发现入口。脚本中还需明确定义"成功标准"（Success Criterion），例如"装备出现在角色装备栏中即为任务完成"，避免测试员主观判断产生数据偏差。

## 实际应用

**案例一：手游新手引导漏斗分析**
某MMORPG手游在上线前测试中，将新手引导拆解为22个HTA节点，发现第7个节点"合成装备"的TCR仅为58%，低于85%基准线。通过逐步回放操作录像，研究者发现有43%的测试者在"材料选择"子步骤上步骤偏差高达+5，因为合成材料列表没有按照稀有度排序，玩家需要滚动很长的列表才能定位所需材料。修改UI排序逻辑后，复测TCR提升至89%。

**案例二：主机游戏菜单导航测试**
一款动作RPG的测试中，设计者预设"切换角色技能树"操作路径为3步。任务分析数据显示，16名测试者的平均实际步骤数为6.8步，步骤偏差+3.8。首次点击准确率仅为31%——超过2/3的玩家第一次点击了"装备"按钮而非"技能"按钮，说明这两个入口在视觉层级上缺乏足够区分度。设计团队据此将技能树入口图标替换为带有醒目光效的专属图标，并调整了其在菜单中的物理位置。

## 常见误区

**误区一：将步骤数多等同于设计失败**
操作步骤数偏差必须结合任务性质解读。对于高风险操作（如删除存档、确认付款），设计者有时会故意增加1～2个"确认步骤"作为防误触机制，此时测试者步骤数超出最短路径并不是负面发现，而是安全设计起效的证据。判断标准是：额外步骤是否出现在设计者预期的防误触节点，而非随机分布于流程中。

**误区二：TCR越高说明界面设计越好**
任务完成率的有效性依赖于测试任务本身的代表性。若任务脚本过于简单（例如测试一个全屏高亮的按钮能否被点击），TCR自然接近100%，但这对界面优化毫无指导意义。反之，若测试任务超出了目标用户的游戏经验范围，则TCR的下降反映的是玩家背景知识不足，而非UI设计缺陷。任务分析报告必须附上测试参与者的玩家档案（如游戏类型偏好、日均游戏时长），才能使TCR数据具有可解释性。

**误区三：任务分析可以替代出声思维法**
任务分析记录的是可见的行为序列，无法捕捉行为背后的认知原因。当步骤偏差为+4时，任务分析只能告知"玩家走了错误路径"，但无法回答"为什么走错"——是图标含义不清、文字提示被忽略、还是玩家主动探索所致。这正是为什么任务分析通常与出声思维法配合使用：前者提供精确的行为数据，后者提供解释这些数据的语言证据。

## 知识关联

**与出声思维法的关系**：出声思维法（Think-Aloud Protocol）是任务分析的前置研究，两者在同一场测试中协同运行。出声思维提供玩家在每个HTA节点上的认知负荷信息（"我不知道这个图标是什么意思"），任务分析则将这些主观陈述对应到具体的步骤偏差和首次点击准确率数据上，使定性观察获得定量支撑。没有出声思维法的任务分析数据是"知其然不知其所以然"的行为记录。

**向问卷与访谈的延伸**：任务分析结束后，研究者通常会针对TCR低于阈值的问题节点设计后测问卷或深度访谈问题。例如，若"技能树"界面的首次点击准确率仅有31%，研究者可在访谈中追问玩家对该界面各区域功能的预期认知，收集的定性数据将直接指导下一轮界面方案的修改方向。问卷与访谈的问题设计因此高度依赖任务分析暴露出来的具体痛点，而非泛化的满意度调查。
