# 启发式评估

## 概述

启发式评估（Heuristic Evaluation）是由Jakob Nielsen和Rolf Molich于1990年提出的一种低成本可用性检查方法，核心做法是让3至5名评估者依据一套固定的"启发式原则"列表，独立审查界面并记录违反原则的问题。Nielsen在后续研究中将原则精炼为著名的"尼尔森十原则"（10 Usability Heuristics），并通过实验证明，使用5名评估者平均可发现约75%的可用性问题，而单人评估者的发现率仅约35%。

这套方法被引入游戏UI/UX领域后，针对游戏特有的信息层级（HUD、菜单、背包系统等）进行了适配。与需要真实玩家参与的可用性测试不同，启发式评估不依赖测试样本，仅需专家或有经验的设计师即可执行，因此在项目早期原型阶段就能快速发现结构性问题，节省后期大量修改成本。

在游戏行业的实践中，启发式评估通常产出一份"问题清单"，每条问题都会被标注违反的具体原则编号（如H3表示"用户控制与自由度"），并按严重程度从0分（非问题）到4分（可用性灾难）打分，这一严重性评级量表同样由Nielsen在1994年标准化。

## 核心原理

### 尼尔森十原则在游戏UI中的对应关系

十条原则各有明确定义，在游戏界面上的映射如下：

- **H1 系统状态可见性**：游戏必须实时告知玩家当前状态。典型例子是血条、弹药数、技能冷却倒计时——如果冷却结束时没有任何视觉或音效反馈，就违反了H1。
- **H2 系统与真实世界的匹配**：UI语言应符合目标玩家的认知模型。RPG游戏若将"装备栏"标注为"物品序列存储矩阵"，即违反H2。
- **H3 用户控制与自由度**：玩家应能随时撤销或退出操作，例如误触"分解全部装备"后必须提供取消或确认弹窗。
- **H4 一致性与标准**：同一游戏内"确认"按钮在不同界面应始终位于相同位置；主机平台上PlayStation的"×"与任天堂的"A"键历史上曾造成跨平台一致性违规问题。
- **H5 错误预防**：在不可逆操作（如删除存档）前主动提示，优于事后显示错误信息。
- **H6 识别而非回忆**：玩家不应被迫记忆跨界面的信息。背包界面比较装备时，应直接显示属性差值而非要求玩家记住当前装备的数值。
- **H7 灵活性与效率**：为新手提供引导图标，为老手提供快捷键或宏功能。《暗黑破坏神》系列的"shift+点击"快速装备机制即体现H7。
- **H8 美观且简约的设计**：HUD中每增加一个元素都会分散战斗注意力，不相关信息应在非战斗状态下隐藏或淡出。
- **H9 帮助用户识别、诊断并从错误中恢复**：错误信息必须用玩家能理解的语言描述。"Error Code 0x8AC70013"比"无法连接到服务器，请检查网络"对玩家毫无意义。
- **H10 帮助文档与说明**：最优设计无需文档，但必要时工具提示（Tooltip）应能在3秒内让玩家理解功能。

### 评估流程的标准步骤

启发式评估遵循以下四步结构：

1. **准备阶段**：确定评估范围（例如只评估角色创建流程或仅评估主菜单），并向所有评估者提供相同版本的界面截图或可交互原型。
2. **独立评估阶段**：每位评估者单独审查界面，不与他人交流，通常需要两轮遍历——第一轮建立整体认知，第二轮对照原则逐条检查并记录问题。
3. **汇总阶段**：召开汇报会议，将所有评估者的问题列表合并，去除重复条目，并协商每条问题的严重性评分（0–4分）。
4. **报告阶段**：输出包含问题编号、违反的启发式原则、严重性评分和改进建议的结构化报告，交由设计团队按优先级排序修复。

### 严重性评分量表

Nielsen的严重性量表将游戏UI问题分为五级：
- **0分**：不认为是可用性问题
- **1分**：仅是表面问题，优先级最低
- **2分**：轻微可用性问题，修复优先级较低
- **3分**：重要问题，应给予高优先级修复
- **4分**：可用性灾难，必须在发布前修复

在游戏中，背包界面的拖拽丢失物品通常被评为3–4分，因为它涉及不可逆的数据损失。

## 实际应用

**案例：MOBA游戏商店界面评估**

以《英雄联盟》早期客户端商店为例，启发式评估发现了多个H6（识别而非回忆）违规：玩家购买装备时需要记住英雄的属性需求，界面本身不提供推荐逻辑说明。该问题在2.x版本后通过引入"推荐出装"标签修复。

**案例：手机RPG新手引导评估**

对某国产手机RPG的新手引导进行评估时，评估者记录到以下典型问题：
- H1违规：技能升级完成时无任何提示，玩家不知道可以升级（严重性3分）
- H3违规：新手引导中的强制点击无法跳过，老玩家重新开号时体验极差（严重性3分）
- H5违规：首次消费钻石无确认弹窗（严重性4分）

5名评估者合计记录了31条独立问题，汇总后合并为22条唯一问题，其中4分问题3条，3分问题9条。

## 常见误区

**误区一：评估者数量越多越好**

Nielsen的研究数据表明，使用15名评估者的问题发现率仅比5名高约10%，但成本增加3倍。多数游戏团队将最优评估者数量设定为3至5人。当评估者超过10人时，新发现的问题数量呈边际递减，因为后来的评估者主要在重复记录前人已发现的问题。

**误区二：启发式评估可以替代真实玩家测试**

启发式评估只能发现界面结构性问题，无法预测真实玩家的情绪反应、学习曲线和沉浸感破坏点。例如，一个技术上完全符合十原则的游戏教程仍可能因为叙事节奏过慢而导致玩家流失——这类问题只有通过实际的Playtest才能被发现。

**误区三：所有违规都必须修复**

严重性评分为0分或1分的问题在迭代资源有限时可以合理忽略。游戏UI设计中存在大量"故意违规"——黑暗魂系列刻意省略说明文字（违反H10）是设计风格而非缺陷，这类情况应在评估报告中标注为"已知且有意为之的设计决策"。

## 知识关联

**与可用性测试概述的关系**：可用性测试概述建立了"为什么要测试"和"测试评估的几种基本类型"的框架认知；启发式评估是其中"专家检查法"这一类型的具体实施方案，它不依赖用户样本而依赖原则列表，与需要真实用户参与的思考出声法（Think-Aloud Protocol）形成方法论上的互补。

**通向游戏测试（Playtest）的路径**：启发式评估通常在Playtest之前执行，目的是在让真实玩家参与之前先过滤掉明显的结构性问题。如果一个界面存在H1或H3的严重违规，这些问题会在Playtest中产生大量噪音，干扰对游戏核心乐趣的测量。因此，行业惯例是先完成启发式评估并修复3–4分问题，再安排Playtest收集玩家行为数据。
