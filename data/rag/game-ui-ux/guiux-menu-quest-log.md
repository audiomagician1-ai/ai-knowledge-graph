# 任务日志

## 概述

任务日志（Quest Journal / Task Log）是游戏UI系统中专门用于显示、追踪和管理玩家任务状态的界面模块。它将当前进行中的任务、已完成任务和失败任务分层归档，并实时同步任务目标的完成进度，让玩家随时掌握自身在游戏叙事与功能进程中的位置。区别于普通信息展示界面，任务日志需要同时承担"档案库"和"导航仪"两种角色——前者负责存储任务描述与背景文本，后者负责以进度条、计数器或地图标记的形式引导玩家行动。

任务日志的设计规范在2000年代早期随着开放世界RPG的兴起而逐渐成熟。《上古卷轴III：晨风》（2002年）首次将任务日志拆分为"活跃任务"与"完成任务"两个独立分类，这一结构被此后十余年的RPG设计广泛沿用。《巫师3：狂猎》（2015年）进一步引入任务优先级标记和地图联动高亮系统，将任务日志从单纯的文本列表演化为一个动态的游戏内导航工具。

任务日志在游戏体验中的价值在于降低玩家的"目标迷失感"。研究显示，当玩家同时持有超过7个活跃任务时，若缺乏有效的分类与追踪机制，任务放弃率会显著上升。因此，任务日志的UI设计直接影响游戏的用户留存和内容消费深度，是游戏UX评估中的核心测量指标之一。

## 核心原理

### 任务分类层级结构

任务日志通常采用三级或四级分类体系。第一级为状态分类：**进行中（Active）、已完成（Completed）、失败（Failed）**；第二级为任务类型分类：主线任务（Main Quest）、支线任务（Side Quest）、日常任务（Daily）、世界任务（World Quest）；第三级为任务内部的子目标（Objective）列表，每条子目标配有独立的完成状态标记。

在视觉层级上，主线任务通常以金色图标或专属字体权重与支线任务区分，防止玩家在大量支线内容中遗漏核心叙事进程。《最终幻想XIV》的任务日志采用彩色感叹号标记系统：金色感叹号为主线，蓝色为支线，紫色为重复性任务，这一颜色编码系统已成为MMO类游戏的行业惯例。

### 进度追踪与数据显示

任务日志的进度显示需要将后端的数值型变量转化为可读的UI元素。常见的映射方式包括：
- **计数型进度**：`已完成 X / 目标 N`，适用于击杀、收集类任务（例："收集狼皮 3/10"）
- **阶段型进度**：以节点时间线（Step Indicator）展示多步骤任务的当前阶段，每个节点对应一个可勾选的子目标
- **百分比进度条**：适用于连续性任务（如探索区域覆盖率），公式为 `Progress = (completed_objectives / total_objectives) × 100%`

任务进度数据必须做到**实时刷新**，即玩家在游戏世界中完成某个击杀或交互动作后，日志内对应计数器应在500毫秒内完成更新，否则玩家会产生系统延迟感，降低任务完成的即时满足感。

### 追踪模式与地图联动

任务日志通常允许玩家将某个任务设为"追踪中（Tracked）"状态，被追踪的任务会在HUD上显示最小化的目标提示条（Objective Tracker），并在地图上生成对应的路径标记或区域高亮。这种"日志→HUD→地图"三层联动是现代任务系统的标准设计范式。

追踪槽位数量设计是关键决策点：过多会造成HUD信息过载，过少则限制玩家的多线程游戏体验。多数AAA游戏将同时追踪上限设置为**3至5个任务**，《赛博朋克2077》固定只允许同时追踪1个任务，以此强化玩家的单线沉浸感。

## 实际应用

**分页标签设计**：在任务数量较多的RPG中，任务日志通常以Tab标签形式呈现各分类，标签上附带数字徽章（Badge）显示该分类下的未读或新增任务数量。《原神》的任务日志在主界面Tab上实时更新未完成任务总数，引导玩家主动打开界面查看，形成行为闭环。

**任务描述文本排版**：任务描述区通常分为两部分——**叙事文本**（Lore Text，通常以斜体呈现，字号约14-16px）和**目标文本**（Objective Text，正体，加粗，字号16-18px）。两者的视觉权重差异确保玩家能快速扫描到行动指令，而不必逐字阅读背景故事。

**完成动效设计**：任务完成时，日志界面通常触发专属动效——如条目从"进行中"列表滑动归档至"已完成"列表，配合完成音效（通常为上升音调的叮声）。这一设计直接利用了行为心理学中的"完成奖励循环"原理，《空洞骑士》的任务/日记完成动画时长约为0.4秒，被玩家社区高度评价为"刚好的节奏感"。

## 常见误区

**误区一：将所有任务展示在同一平级列表中**
部分设计者为简化开发，将主线、支线、日常任务混列在无分类的单一列表中。这会导致玩家花费大量时间筛选信息，且主线任务极易被海量支线淹没。正确做法是至少实现状态分类和类型分类两个维度的过滤功能，并为主线任务提供视觉置顶机制。

**误区二：任务描述过于依赖文字，忽略图标化进度**
冗长的任务描述文本对于计数型目标而言是信息冗余。"击败10只哥布林，目前已击败3只"与在UI中直接显示`3/10`配合一个剑形图标相比，后者的信息获取速度快约40%（根据眼动追踪研究数据）。任务日志应在叙事文本之外，为每个子目标提供独立的图标化进度元素。

**误区三：忽视任务失败状态的UI反馈**
许多游戏对任务失败的界面处理过于简单，仅在列表中将条目变灰或标注"失败"字样，缺乏失败原因说明。玩家在无法理解失败原因时，往往产生困惑甚至负面情绪。正确设计应在失败条目中附注失败触发条件（如"超时未完成"或"关键NPC阵亡"），并提供是否可重启的明确说明。

## 知识关联

任务日志的**地图联动功能**直接依赖于地图界面的技术架构——任务追踪标记、目标区域高亮和路径引导线均需要通过地图系统的坐标标记层（Marker Layer）来渲染。设计任务日志前，必须先确认地图系统是否支持动态标记注入接口，否则追踪功能将无法实现与地图的双向同步。

向上扩展时，任务日志系统为**商城/商店界面**的设计提供了重要的上下文信息链接——现代游戏常在任务日志的奖励预览区直接嵌入商店物品的快捷购买入口，或在完成任务后通过日志界面跳转至商城解锁限定商品。这要求任务日志的奖励展示模块必须预留与商店系统的数据接口，以支持物品跳转和解锁状态同步。
