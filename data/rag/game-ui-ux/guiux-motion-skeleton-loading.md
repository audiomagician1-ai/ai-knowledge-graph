# 骨架屏加载

## 概述

骨架屏加载（Skeleton Screen Loading）是一种在内容尚未就绪时，使用与真实内容形状相似的灰色占位区块填充界面的动效技术。这些占位块通常呈现为矩形、圆形或线条，精确模拟最终内容的布局结构——例如用圆形占位头像位置、用宽度不等的横条占位文字行。用户看到的不是空白或转圈的加载图标，而是一个"轮廓清晰但内容待填充"的界面草稿。

这一技术由设计师 Luke Wroblewski 于 2013 年首次系统性提出，并在 Facebook 的移动端应用中得到早期大规模使用。他在文章中指出，相比菊花转圈式加载动画，骨架屏让用户感知到"内容即将出现"，而非"系统正在卡顿"，从而将用户的主观等待感降低约 20%。在游戏 UI 场景中，骨架屏广泛用于角色卡牌列表、装备背包格子、好友排行榜等需要从服务器异步拉取数据的界面模块。

骨架屏之所以适用于游戏 UI 的动效设计，是因为游戏玩家对"界面响应速度"极为敏感。当背包界面打开时，若使用骨架屏填充 64 个装备格子并附加从左到右的光泽扫描动画，玩家的停留时间显著高于直接显示空格或旋转图标的方案。

## 核心原理

### 占位结构的形状映射

骨架屏的关键设计规则是**形状必须与真实内容的边界框高度吻合**。游戏角色卡牌通常包含：顶部 80×80px 的头像圆形占位、两条宽度为 100% 和 60% 的文字横条、底部一个按钮矩形占位。这种精确的形状映射让大脑在看到骨架屏的第一帧时就能预测出真实内容的排布，从而提前建立空间期望，减少内容加载完成时的"跳变感"。

### 扫光动画（Shimmer Effect）

骨架屏最常见的动态表现是扫光效果（Shimmer），即一条半透明的高亮光带从左到右持续扫过所有占位块。其 CSS 实现核心是将渐变背景（`background: linear-gradient(90deg, #e0e0e0 25%, #f5f5f5 50%, #e0e0e0 75%)`）配合 `@keyframes` 动画将 `background-position` 从 `-200%` 移动至 `200%`，循环周期通常设定为 **1.5 秒**。在游戏引擎（如 Unity UI 或 Cocos Creator）中，等效实现是对 Mask 组件内的高亮 Sprite 使用 tween 控制 X 坐标平移。光带宽度一般为整个占位块宽度的 30%，过宽会显得廉价，过窄则不够醒目。

### 渐进显示与内容替换

骨架屏并非在所有数据到齐后才一次性替换，而是采用**逐块渐进显示**策略：每当一条数据记录从网络层返回，就立即将对应的占位块以 `opacity: 0 → 1` 的淡入动画（持续约 **200ms**）替换为真实内容。游戏排行榜场景中，第 1 名的数据先到，则第 1 条目先完成替换并淡入，其余位置继续显示扫光骨架。这种方式使界面具有"数据流入"的动态质感，同时将单次视觉冲击分散为多次小型动画，降低了认知负荷。

### 颜色与层级规范

游戏骨架屏通常使用**主色调的低饱和度、低明度变体**作为占位色，而非直接使用通用的 `#E0E0E0`。例如，暗色科幻题材游戏中背景为 `#1A1A2E` 时，骨架占位色应选用 `#2E2E4A`，扫光高亮色选用 `#3D3D5C`，保持整体视觉风格一致。使用与主题完全脱节的浅灰色骨架屏，会破坏游戏的沉浸感。

## 实际应用

**装备背包界面**：当玩家打开 64 格背包时，所有格子立即显示正方形骨架占位块并启动扫光动画，服务端数据按批次返回后，每格装备图标以 150ms 的弹入动画（scale 从 0.8 → 1.0）替换占位块。这比等待全部 64 件数据加载完毕后一次性显示，能将玩家首次看到内容的时间从平均 800ms 缩短至约 120ms。

**好友排行榜**：头像位置使用圆形骨架占位，玩家名称位置使用宽 70% 的矩形横条，积分数字位置使用宽 30% 的短横条。当玩家自身排名数据优先返回时，该条目优先完成替换，其余条目逐行从上到下完成淡入，形成自然的"数据刷入"视觉节奏。

**商店物品列表**：横向滑动的卡牌商店中，屏幕内可见的 3 张卡位同步显示骨架动画，当用户手指向左滑动时，新出现的卡位骨架应在进入视口的同时立即启动扫光动画，而非从骨架动画的随机帧开始——这要求每个骨架组件独立维护动画起始时间戳。

## 常见误区

**误区一：用骨架屏替代所有加载状态**。骨架屏仅适合"内容结构已知、数据待填充"的场景。对于首次打开游戏时加载 300MB 资源包的过程，骨架屏毫无意义——此时需要进度条加载界面，因为用户需要知晓进度百分比，而骨架屏无法传达这一信息。

**误区二：所有骨架占位块使用相同的扫光动画相位**。若 64 个背包格子全部同步闪烁，光带会在同一时刻扫过所有格子，形成强烈的频闪感，可能引发光敏性不适。正确做法是为每个格子的动画设置 `animation-delay`，使相邻格子之间错开约 **50ms**，产生波浪式扫光效果。

**误区三：骨架屏的占位块尺寸与实际内容不匹配**。若骨架阶段的头像占位圆形直径为 60px，但真实头像渲染后为 80px，则替换瞬间会发生明显的位移跳变（Layout Shift），CUMULATIVE LAYOUT SHIFT（CLS）评分恶化。在游戏 UI 中，这会表现为整个卡牌在数据到位时发生抖动，严重影响视觉稳定性。

## 知识关联

骨架屏加载的视觉动态基础来自**视差效果**的层次感知原理——视差效果中不同深度层以不同速率运动，训练了设计师对"层级独立运动"的理解，这与骨架屏中各占位块独立维护动画状态的设计思路一脉相承。同样，视差效果中使用 `transform: translateX()` 而非 `left` 属性进行偏移的性能优化准则，也完全适用于骨架屏扫光动画的实现——始终应使用 `transform` 和 `opacity` 触发合成器层动画，而非修改 `width` 或 `background-color` 触发重排。

骨架屏加载是迈向**状态变化动画**的直接前置技术。状态变化动画处理的是"组件从一个完整状态切换至另一个完整状态"的过渡，而骨架屏本质上是"空状态 → 加载中状态 → 完成状态"这一三阶段状态链中的中间环节。掌握了骨架屏的占位替换与淡入时序设计后，可以自然延伸至处理更复杂的多状态组件动效，如角色卡牌的"普通/选中/禁用/升级中"四态切换动画。
