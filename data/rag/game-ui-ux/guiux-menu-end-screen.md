# 结算/结束画面

## 概述

结算/结束画面（End Screen / Results Screen）是玩家完成一局游戏、关卡或对局后自动弹出的总结性UI界面，其核心职责是将本局的关键数据——得分、时间、击杀数、伤害量、经验值获取等——以可视化方式呈现给玩家，并触发奖励发放流程。它不是一个被动的信息展板，而是一个主动的反馈机制，通过动画、音效和高亮元素将"表现→奖励"的因果链路清晰传递。

从设计史来看，最早的结算画面出现在1980年代街机游戏中，仅显示一行最终分数。1994年前后，随着《街头霸王II》引入对战统计（连段次数、最高伤害），结算画面开始承载多维度数据。2009年《英雄联盟》将MVP评选、每项数据的全队对比、以及段位LP增减全部集成进同一个结算界面，确立了现代多人竞技游戏结算页的基本范式。

结算画面在玩家心理中扮演"一局游戏的句号"角色。研究显示，玩家在结算画面停留的平均时间为15～45秒，这段时间是游戏向玩家传递"你做得如何"和"你获得了什么"的黄金窗口。设计不当的结算画面——如信息过载、奖励动画缺失或数据排版混乱——会直接降低玩家对本局游戏体验的整体评分，即使游戏过程本身令人满意。

## 核心原理

### 数据展示层级与信息优先级

结算画面必须将数据按"最想让玩家看到的"排列优先级。通常分三层：第一层是**核心结果**（胜/负、总分、排名），字体最大，占据画面顶部或中央，响应时间要求在进入结算画面后0.5秒内完成渲染；第二层是**个人统计**（KDA、伤害、治疗量等），以表格或卡片形式呈现；第三层是**奖励明细**（经验值、货币、道具掉落），往往在动画序列末尾逐条显示。《APEX英雄》的结算页将"击倒数"和"伤害总量"以特大号数字置于个人卡片顶部，正是对这一优先级逻辑的典型应用。

### MVP评选与高光标记机制

MVP（Most Valuable Player）标记是结算画面中社交激励的关键元素。常见实现方式是根据一个加权公式计算玩家贡献分，例如：

**贡献分 = 击杀×3 + 助攻×1 + 伤害总量/1000 + 目标参与率×5**

权重系数因游戏类型而异，MOBA类游戏倾向于拔高目标参与率权重，射击游戏则倾向于伤害和击杀。MVP玩家的头像通常附带金色边框、专属粒子动效或"Best Player"徽章，与其他队友形成视觉区分度。设计时需避免MVP每局都由同一位置/角色垄断，否则非MVP玩家会产生"我的数据不被认可"的挫败感。

### 经验值与奖励动画序列

奖励动画是结算画面中技术实现复杂度最高的部分，其设计目标是通过**渐进式揭示**（Progressive Disclosure）最大化玩家满足感。标准动画序列为：

1. 经验条从当前值开始填充，以约0.8秒匀速动画到新值；
2. 若触发升级，经验条填满后播放溢出粒子效果并重置，同时弹出等级提升面板（约1.2秒）；
3. 随机奖励箱/宝箱以翻转或展开动画逐一亮相（每个0.3～0.5秒），过快会让玩家感觉廉价，过慢则造成等待焦虑；
4. 最终展示全部奖励的汇总列表，保留5秒供玩家查看。

《英雄联盟》2023年赛季的结算页将奖励动画时长压缩到总计约8秒，玩家反馈认为过于仓促；随后版本将宝箱开启动画恢复到每个0.6秒，满意度指标随之回升。

### 操作按钮设计

结算画面通常包含三到四个操作出口：**再来一局（快速重新匹配）**、**返回主菜单**、**查看详细战报**、**分享战绩**。其中"再来一局"按钮需要使用最显眼的视觉权重（高饱和度主色调、最大尺寸），因为玩家留存的核心目标就是让玩家点击这个按钮。《使命召唤：战区》的结算界面将"再来一局"按钮设计为宽度占屏幕70%的底部通栏，点击率比同类设计高出约23%。

## 实际应用

**竞技射击游戏（如VALORANT）**：结算页分为"胜负与轮次比分"→"全队KDA与ACS（平均战斗得分）排名"→"MVP徽章"三段，按Tab键可切换到对方队伍数据视图，让玩家横向比较己方与敌方的发挥。ACS的具体算法被Riot公开：ACS = 击杀×200 + 首杀奖励×100 + 爆头率加成 - 死亡惩罚，这使结算数据具有可解读性而非黑箱。

**手机卡牌/休闲游戏（如Clash Royale）**：结算画面采用三星评级制，星星逐一蹦出并伴随金属撞击音效；随后宝箱以3D翻转动画弹出，玩家需点击宝箱触发开箱——这将被动接受奖励变为主动交互，增加了仪式感和心理所有权。

**角色扮演类关卡结算（如《原神》秘境）**：结算界面展示本次通关时间与最优时间的差值、使用角色的输出/受伤统计，并在首次通关时触发"成就解锁"弹窗叠加显示，利用结算时机同时完成多条信息的呈现。

## 常见误区

**误区一：把所有数据都放在第一屏**
设计者常认为"展示更多数据=更有深度"，实际上当结算页一次性呈现超过12个数据字段时，眼动研究显示玩家的注视点会分散到无序状态，导致核心数据（如胜负和奖励）反而被忽视。正确做法是默认展示5～7个最重要字段，通过"详细战报"入口提供完整数据。

**误区二：奖励动画可跳过就无所谓设计质量**
部分开发者认为既然玩家可以点击Skip，奖励动画的质量就无关紧要。但数据显示，约60%的玩家不会主动跳过奖励动画，而即使是会跳过的玩家，奖励动画的视觉品质仍然影响其对"奖励丰厚度"的主观感知。一个粗糙的升级动画会让玩家感觉"这个等级没什么价值"。

**误区三：结算画面不需要加载优化**
结算画面往往在战斗结束后立即调用大量贴图和粒子资源，开发者有时将其排在性能优化的低优先级。实际上，结算画面出现前超过2秒的黑屏或加载转圈，会严重打断"游戏结束→获得反馈"的心理连贯性，玩家会误以为游戏崩溃，流失率在这个节点可高达8～15%。

## 知识关联

结算/结束画面与**排行榜界面**存在直接的数据流向关系：排行榜界面展示的是跨局次的历史积累数据，而结算画面是每一局将新数据写入排行榜的"提交节点"。设计时需确保两套界面对同一指标（如段位分、总胜场）使用完全一致的数字格式和颜色语义——例如排行榜用绿色表示上升，结算画面也必须用绿色表示本局的得分增加，否则会产生认知矛盾。

从UI系统的数据架构角度看，结算画面需要对接至少四个后端模块：战斗数据统计系统（提供KDA等原始数据）、奖励计算系统（输出经验和道具结果）、社交系统（推送MVP徽章和战绩分享）、以及存档系统（触发段位和等级的持久化写入）。这四个模块的响应时序若设计不当，会导致结算页数据出现"先显示旧等级再跳变到新等级"的闪烁bug，这是结算画面开发中最高频的技术问题之一。
