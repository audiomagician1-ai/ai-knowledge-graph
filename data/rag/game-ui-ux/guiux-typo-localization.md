# 多语言排版

## 概述

多语言排版（Multilingual Typography）是指在游戏UI中同时支持多种书写系统的文本显示技术，核心挑战来自三类语言特性的根本性差异：CJK（中日韩）字符集的字形密度、RTL（Right-to-Left）语言的阅读方向，以及拉丁语系与亚洲语系之间高达40%的字符串长度变化率。游戏本地化到德语时文本通常比英语原文长30-35%，而日语版本可能短20-30%，这意味着同一个UI按钮在不同语言版本下必须经过专门的排版处理才能正常显示。

Unicode标准自1991年发布以来逐步建立了多语言文本处理的基础，其中Unicode双向算法（Unicode Bidirectional Algorithm，简称UBA）规定了混合LTR/RTL文本的渲染逻辑，而OpenType字体格式的GSUB（字形替换）和GPOS（字形定位）表则为阿拉伯语的连笔字形和天城文的元音符号提供了技术支撑。对于游戏开发团队而言，忽视多语言排版往往导致阿拉伯语版本的对话框出现字母断开无法连写的严重显示错误。

多语言排版在游戏行业的重要性体现在直接影响玩家对游戏质量的感知。《最终幻想14》在同时维护多达九种语言服务器时，其字体排版系统必须处理日语纵排、英语横排和阿拉伯语RTL布局在同一UI框架下的共存问题，这要求开发者在架构阶段就引入语言感知型布局系统，而非在上线前临时修补。

## 核心原理

### CJK字符排版规则

CJK字符在排版中有别于拉丁字符的核心特征是字符宽度的固定性：每个汉字、假名或谚文音节块均占用一个全角（Full-width）单位，相当于拉丁字符平均宽度的两倍。这使得CJK文本的行长度计算不能沿用拉丁系"字符数×平均字宽"的近似公式，而必须采用**全角宽度累加**的方法。

换行规则（Line Breaking Rules）方面，中文和日语不允许在以下字符前换行：句尾标点「。」「、」「」」；不允许在以下字符后换行：「「」「（」「【」。这套规则被称为**禁则处理（Kinsoku Shori）**，Unicode标准中用`lb=NS`（Non-Starter）和`lb=CL`（Close Punctuation）属性来标记。游戏引擎如果未实现禁则处理，中文对话框将出现标点挤在行首的排版错误。

日语还支持纵向书写模式（vertical-rl），此时标点符号需要旋转90度显示，部分字符（如括号）需要替换为竖排专用字形。Unity引擎的TextMeshPro在4.x版本之前不原生支持竖排，开发者通常需要通过`VertexModifier`手动实现字形旋转。

### RTL语言的双向文本处理

阿拉伯语和希伯来语的书写方向为从右到左，但数字和嵌入的英文单词仍保持从左到右。Unicode双向算法（UBA，完整规范见Unicode标准第9章）通过为每个字符分配**双向类别（Bidi Category）**来解决这一问题：阿拉伯字母类别为`AL`，拉丁字母为`L`，数字为`EN`。渲染引擎根据这些类别对字符重新排序后再输出到屏幕。

对游戏UI而言，RTL适配不仅限于文字方向。整个界面布局通常需要镜像翻转：主菜单中原来在左侧的返回按钮移至右侧，血条从右向左减少，任务列表的图标从右侧对齐。《塞尔达传说：旷野之息》的阿拉伯语本地化即采用了部分UI镜像策略，但并非所有元素都适合镜像（如游戏内时钟的旋转方向保持不变）。

阿拉伯语字母的**连写形态**（Contextual Forms）进一步增加了复杂性：同一个字母在词首、词中、词尾和独立出现时有四种不同字形，OpenType的`init`、`medi`、`fina`、`isol`特性标签负责自动切换这些字形。若游戏UI渲染管线不支持OpenType整形（Shaping），阿拉伯语文本会显示为一串不相连的孤立字母，完全失去可读性。

### 变长文本的溢出处理策略

变长文本溢出是多语言排版中最高频的实际问题。应对策略分为三个层级：

**一级：弹性容器设计** — UI容器预留足够的扩展空间，通常建议为英文原文宽度的130%（应对德语、芬兰语等长词语言）。技能名称、按钮标签等短文本的扩展系数应设为1.4倍，因为单词越短，翻译后的相对增幅越大。

**二级：文本自适应缩放** — 当文本超出容器后自动缩小字号，但必须设定最小可读字号下限，通常为原始字号的75%（即如果原始字号为16px，最小不得低于12px）。低于该阈值应触发三级处理。

**三级：截断与省略** — 对超长文本进行截断并显示省略号（`…`），但CJK文本的省略号为全角三点（`……`），与拉丁省略号字符（U+2026）是不同的字符，混用会造成对齐错误。此外，阿拉伯语和希伯来语的省略号位置应出现在文本的视觉末尾（即文本的左侧），这与LTR语言的位置相反。

## 实际应用

**角色对话框的多语言适配**：RPG游戏对话框通常固定宽度，日语原版的一行对话在英语本地化后往往需要1.5至2倍行数，直接导致对话框高度不足。解决方案是将对话框设计为垂直方向可滚动，并在日语环境下启用假名缩写（Small Kana）优化，例如将「ていねい」的「い」缩写处理，每行可多容纳约8%的字符。

**HUD血条与状态数值**：阿拉伯语HUD中，玩家HP数字虽然使用印度-阿拉伯数字（٣٧٥）还是西方阿拉伯数字（375）取决于目标市场，沙特市场倾向于西方数字，而埃及市场两者皆可接受。血条的减少方向需随UI镜像而改变，从右向左减少。

**字体选择与回退机制**：单一字体无法覆盖所有Unicode码位，因此游戏必须配置字体回退链（Font Fallback Chain）。典型配置为：主字体（覆盖拉丁字符）→ CJK字体（如思源黑体，覆盖U+4E00至U+9FFF）→ 阿拉伯字体（如Noto Naskh Arabic）→ 系统回退字体。若链中断，未覆盖的字符将显示为方块（豆腐块，Tofu），这正是Google Noto字体族名称的由来——"No more Tofu"。

## 常见误区

**误区一：认为CJK语言不需要换行处理**
许多开发者认为中文没有空格、自然按字符换行，因此无需特殊处理。实际上禁则处理、文字间距（字距调整Kerning的CJK等效处理称为字间距Tracking）以及行间距设置对中文排版质量有决定性影响。中文正文推荐行高为字号的1.5至1.8倍，远高于英文的1.2至1.4倍，若使用英文行高设置中文正文，将导致汉字视觉上过于拥挤。

**误区二：RTL适配只需翻转文字方向**
部分团队以为在CSS或引擎中设置`direction: rtl`或等效属性即完成阿拉伯语适配。实际上，这仅处理了文字流方向，而OpenType整形（用于阿拉伯连写字形）、UI镜像布局、双向嵌入数字的正确定位、以及标点符号的镜像（如问号在阿拉伯语中为`؟`，Unicode U+061F）均需要独立处理。遗漏OpenType整形会导致阿拉伯语文本呈现完全不可读的孤立字母序列。

**误区三：字体文件体积不是多语言排版的问题**
开发者常在本地化末期才发现，支持完整CJK字符集的字体文件体积高达15-20MB（思源黑体的单字重文件约为15MB），而移动平台对安装包体积有严格限制（Google Play推荐APK不超过150MB）。解决方案是在构建时生成**字体子集（Font Subsetting）**，仅保留游戏文本实际使用的字形，可将CJK字体文件压缩至1-3MB，但这要求本地化文本在子集生成前全部确定，不能在运行时动态加载新字符。
