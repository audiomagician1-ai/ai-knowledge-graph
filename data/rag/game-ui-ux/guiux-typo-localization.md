# 多语言排版

## 概述

多语言排版是指在游戏UI中同时支持多种书写系统——包括中日韩（CJK）表意文字、阿拉伯语和希伯来语等从右向左（RTL）书写系统、以及拉丁语系的变长文本——并对各类溢出、换行、对齐问题进行适配处理的技术体系。与单一语言排版不同，多语言排版需要一套字符集之间动态切换、布局方向实时反转、文本长度自动伸缩的综合方案。

多语言排版问题在游戏行业的重要性随着2000年代主机游戏全球同步发售（Day-One Worldwide Release）策略兴起而急剧上升。早期本地化往往是事后补丁，导致日文版《最终幻想》系列在SNES上使用压缩字体以应对16×16像素点阵限制。如今AAA游戏在设计阶段便必须将10至20种语言的排版需求纳入UI框架，否则在波斯语、泰语等长词语言环境下，按钮文字溢出率可高达40%。

这一领域的核心挑战在于：同一个语义内容在不同语言中字符数可相差3至5倍（例如德语单词平均比英语长35%），而游戏UI控件的物理尺寸往往是固定像素或固定比例的。任何排版方案的失效都会直接破坏玩家界面的可读性与操作准确性，严重时导致点击目标错位或信息截断。

## 核心原理

### CJK排版：字符宽度与换行规则

CJK文字中每个汉字、平假名、片假名均为全角字符，宽度等于行高（即1em），与拉丁字母的可变字宽形成根本差异。游戏引擎处理CJK文本时需要实现**禁则处理（Kinsoku Shori）**：句末标点（。、！？）不能出现在行首，特定括号（「」【】）不能出现在行尾，违反此规则的字幕或对话框在日语市场会被视为严重本地化缺陷。

标准Unicode规范为CJK换行定义了行首禁止字符（Line Start Prohibition）和行末禁止字符（Line End Prohibition）两套列表，游戏引擎的文本渲染模块必须在分行算法中逐字检索这两套列表。竖排文字（Vertical Text Layout，CSS writing-mode: vertical-rl）在部分日系RPG的UI中仍然存在，此时全角标点需要旋转90°显示，而拉丁字母和数字则保持横排嵌入（Tate-Chu-Yoko）。

### RTL排版：双向文本算法（BiDi）

阿拉伯语、希伯来语、波斯语（Farsi）、乌尔都语均为从右向左书写，但数字和嵌入的拉丁词汇仍从左向右排列。这种混合方向文本由**Unicode双向算法（Unicode Bidirectional Algorithm，UBA，定义于Unicode标准附件TR#9）**规范处理。UBA将每个字符分配一个方向级别（Embedding Level），偶数级别为LTR，奇数级别为RTL，然后按视觉顺序重排字符序列。

在游戏UI实践中，RTL语言不仅要求文字方向翻转，还要求整个界面布局镜像翻转：血条从右向左减少，背包格子从右上角开始排列，对话框的说话人头像移至右侧。《赛博朋克2077》的阿拉伯语版本即因未完全镜像UI而在发布初期遭到玩家投诉，后续补丁修正了约60处UI元件的位置。实现上通常通过设置`RightToLeft`布局标志位并对所有锚点坐标执行`x' = containerWidth - x - elementWidth`的镜像变换来完成。

### 变长文本的溢出处理策略

拉丁语系中，德语复合词（如`Eingabeaufforderung`）和芬兰语的格变化形式可产生极长的单词，导致标准换行点缺失。游戏常用以下三种策略应对变长文本：

1. **自动字号缩放（Auto Font Scaling）**：在`[minFontSize, maxFontSize]`区间内按0.5pt步长逐步缩小字号直至文本适配控件，Unity的`TextMeshPro`组件通过`enableAutoSizing`属性支持此特性，最小字号建议不低于10pt以保证可读性。
2. **文本截断加省略号（Ellipsis Truncation）**：对固定单行控件（如角色名称标签）以`…`结尾截断，但必须确保截断后文本在所有语言中语义仍然可辨，CJK文字因信息密度高通常截断损失较小。
3. **动态控件扩展（Dynamic Layout Expansion）**：允许按钮、工具提示等控件在水平或垂直方向上按文本内容自动扩张，同时设置最大宽度阈值（通常为屏幕宽度的70%）防止布局崩溃。

泰语、藏语等语言缺乏显式词间空格，必须依赖**字典分词算法（Dictionary-based Word Segmentation）**确定合法换行位置，无法仅凭Unicode字符属性判断。

## 实际应用

《暗黑破坏神IV》在多语言排版上面临的典型挑战是技能描述文本：英语版技能名称平均12个字符，德语版可达28个字符，而日语版汉字组合通常仅需6至8个全角字符。团队最终采用以下方案：英语/德语/俄语使用自动字号缩放（18pt降至13pt），日语/中文固定字号18pt不缩减，对话系统框架则设置独立的语言档案（Language Profile）存储每种语言的行距系数（Line Height Multiplier）——阿拉伯语设为1.4以容纳字符上下的附标（Diacritics/Harakat），泰语设为1.5以容纳声调符号。

在独立游戏实践中，使用Unity开发的2D RPG若要支持阿拉伯语，最低成本方案是集成**RTLTMPro**插件，该插件在`TextMeshPro`基础上封装了BiDi算法和阿拉伯字形整形（Arabic Glyph Shaping），因为阿拉伯字母根据在单词中的位置（词首/词中/词尾/独立）呈现4种不同字形，标准TrueType字体通过OpenType的`init`、`medi`、`fina`、`isol`特性表存储这4套字形，引擎必须在渲染前完成字形选择（Glyph Selection）。

## 常见误区

**误区一：用字号缩放代替布局设计**  
许多开发者将自动字号缩放视为万能解决方案，却忽视了当德语文本从18pt缩至9pt时，同屏的中文文本仍维持18pt，造成视觉比例严重失调。正确做法是在UI设计阶段为每种语言组预留25%至40%的额外空间，而非依赖运行时缩放修补。《星露谷物语》早期版本即因德语描述文字过小而在Steam评测中遭到批评。

**误区二：RTL仅需镜像文字方向**  
许多团队只处理文本的显示方向，却保持UI控件的LTR排列。然而RTL语言的用户期待整体阅读流从右上角开始，这意味着对话选项列表、物品栏格子、主菜单项的视觉重心都应在右侧。仅翻转文字而不翻转布局会导致RTL用户的视线路径与操作路径产生认知冲突，增加操作失误率。

**误区三：CJK字符不需要字距和字型调整**  
开发者有时认为CJK文字等宽且不需要字距微调（Kerning），但实际上标点挤压（Punctuation Squeezing）是日文排版的基本要求：连续的全角标点（例如`。」`）之间按JIS X 4051标准需压缩至0.5em间距，否则视觉上会出现过大空白，被日本玩家认定为未经打磨的本地化质量。

## 知识关联
