# 采样率选择

## 概述

采样率（Sample Rate）是指每秒钟对音频信号进行数字化采集的次数，单位为赫兹（Hz）或千赫兹（kHz）。游戏音效制作中常见的采样率包括 44.1kHz、48kHz 和 96kHz，三者对应的每秒采样次数分别为 44,100 次、48,000 次和 96,000 次。根据奈奎斯特定理，采样率必须至少是被录制信号最高频率的两倍，才能无失真地重建原始波形——因此 44.1kHz 的理论最高可还原频率为 22,050Hz，略高于人耳上限约 20,000Hz。

44.1kHz 这一数值起源于 20 世纪 70 年代末的 CD 音频标准（由索尼与飞利浦于 1980 年联合制定的红皮书标准），而 48kHz 则是专业视频和广播行业长期沿用的标准，被 SMPTE 等专业机构确立为影视后期的基准采样率。96kHz 属于高解析度（Hi-Res）录音规格，主要用于录音棚母带处理阶段，为后期音调调整和时间拉伸留出充足的频率余量。

在游戏音效资源管理中，采样率的选择直接影响音频文件体积、内存占用和运行时 CPU 解码开销。以一段 10 秒的单声道未压缩 WAV 文件为例，96kHz/24bit 版本的体积约为 2.88MB，而同等时长的 48kHz/24bit 版本仅约 1.44MB——两者相差整整一倍，这在同时加载数百个音效的移动端游戏中会造成显著的内存压力。

## 核心原理

### 奈奎斯特-香农采样定理与截止频率

采样率选择的物理依据来自奈奎斯特-香农定理：$f_s \geq 2 \cdot f_{max}$，其中 $f_s$ 为采样率，$f_{max}$ 为信号中包含的最高频率成分。若采样率低于该门限，高于 $f_s/2$（即奈奎斯特频率）的成分将发生混叠（Aliasing），产生难以滤除的谐波失真。游戏引擎在运行时对音频进行变调（Pitch Shift）或多普勒效应处理时，信号实际频率会被向上推移，此时使用 48kHz 而非 44.1kHz 可多保留约 3,900Hz 的安全余量，降低混叠风险。

### 三种主流采样率的适用场景对比

**44.1kHz** 适合最终交付给玩家的纯音乐类资产（BGM、主题曲），因为其 CD 标准来源使大量软件和硬件均对该格式有原生优化。将此类素材以 44.1kHz 存储后，Wwise 或 FMOD 在导入时无需执行采样率转换（SRC），可节省构建时间。

**48kHz** 是游戏音效素材的行业主流推荐标准。Wwise 2022.1 版本的默认工程采样率即设置为 48kHz，Unity 的 Audio Mixer 输出端同样默认为 48kHz。这一选择确保了音效与游戏内过场动画（通常遵循视频行业 48kHz 标准）的采样率一致，避免实时重采样带来的轻微音质损耗与 CPU 消耗。

**96kHz** 仅推荐用于原始录音素材的存档与编辑阶段。当一个爆炸音效需要被拉伸至原始时长的 200% 或做大幅度音调下移（如 -12 半音）时，从 96kHz 源文件开始处理可以保留更多谐波细节，完成后再降采样至 48kHz 交付。直接在游戏引擎中使用 96kHz 素材将使内存带宽需求翻倍，且主流手机 SoC 的音频 DSP 通常不对该采样率进行硬件加速。

### 采样率转换（SRC）的性能代价

当游戏引擎的输出采样率与音频资产采样率不一致时，混音器需要实时执行 SRC 运算。高质量 SRC（如 Sinc 插值）会在每帧混音回调中额外消耗 15%～25% 的音频线程 CPU 时间（数据来源于 FMOD 官方性能测试文档）。因此，资源管理规范应要求所有音效素材在导入资产管线时统一转换为工程采样率，而非交给引擎在运行时处理。

## 实际应用

**移动端射击游戏**：枪声、脚步、UI 音效等高频触发资产统一采用 48kHz 存储，在 Wwise 的 SoundBank 配置中设置平台专属的降质策略——iOS 平台保持 48kHz，低端安卓设备可将非关键音效自动转换为 22.05kHz，节省约 50% 的内存占用。

**主机 RPG 游戏的过场动画对白**：对白录音以 96kHz/24bit 完成，经降噪与 EQ 处理后降采样输出 48kHz/16bit 版本用于游戏内播放，既保留了编辑阶段的处理精度，又符合游戏资产体积约束。

**程序化音效（Procedural Audio）**：使用 Pure Data 或 Max/MSP 生成的实时合成音效，其输出采样率需与引擎音频系统采样率精确匹配（通常为 48kHz），否则将产生可听见的音调漂移（Pitch Drift），在循环音效中尤为明显。

## 常见误区

**误区一：采样率越高，最终游戏音质越好**。玩家在游戏中通过消费级音频设备（耳机、电视扬声器）收听的最高频率通常不超过 18kHz，而 48kHz 的奈奎斯特频率为 24kHz，已完全覆盖人耳可听范围。将最终音效资产保存为 96kHz 并不会提升玩家的主观听感，却会使每个音效文件体积和内存占用加倍。

**误区二：44.1kHz 与 48kHz 可以随意混用**。两种采样率之间的比率为 44100/48000 = 147/160，不是整数比，因此从 44.1kHz 转换到 48kHz 时必须进行高计算量的分数阶 SRC，而非简单的整数倍插值。项目内部混用这两种采样率会在资产构建流水线中引入隐性的质量损耗节点，且这种损耗在波形层面可被频谱分析工具（如 iZotope RX）检出。

**误区三：高采样率录音可以直接投入游戏以"备用"**。部分音效师将 96kHz 原始录音未经处理地导入 FMOD 项目，误以为引擎会自动优化。实际上，FMOD Studio 在构建 SoundBank 时若未设置降采样规则，会直接打包原始 96kHz 数据，导致最终发行包体积异常膨胀，且在运行时造成不必要的内存带宽占用。

## 知识关联

**前置概念——版本控制**：采样率选择一旦在项目初期确定，就应通过版本控制系统（如 Git LFS 或 Perforce）将工程采样率配置文件（Wwise 的 `.wproj` 或 FMOD 的 `.fspro`）纳入追踪。若中途更改工程采样率，所有已入库的音频资产需要重新转换并提交新版本，版本历史中两套采样率的资产共存会引发平台构建错误，必须通过分支策略加以隔离。

**后续概念——位深度**：采样率决定了频率轴的分辨率（每秒采集多少个时间切片），而位深度（Bit Depth）决定了幅度轴的分辨率（每个切片能表示多少个电平等级）。两者共同构成数字音频质量的基础参数，且在存储体积计算公式中相互关联：$\text{文件大小(bytes)} = \frac{f_s \times b \times c \times t}{8}$，其中 $f_s$ 为采样率，$b$ 为位深度，$c$ 为声道数，$t$ 为时长（秒）。确定采样率策略后，下一步需要用同样的场景分析方法为不同类型的游戏音效制定位深度规范。
