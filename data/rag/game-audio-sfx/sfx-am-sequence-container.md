# 序列容器

## 概述

序列容器（Sequence Container）是音频中间件中一种按照固定顺序逐个播放音效素材的容器类型。与随机容器每次随机抽取素材不同，序列容器维护一个内部播放指针，每次触发时依次播放列表中的下一个素材，形成有规律的循环或单次播放序列。

序列容器最早在 Audiokinetic Wwise 的早期版本（2006年前后）中作为核心容器类型被引入，用于解决游戏音效中"避免重复"（Anti-Repetition）问题的另一种思路——不是通过随机化，而是通过可预期的轮转（Round-Robin）机制让玩家感知到素材在有规律地变化。这种行为在脚步声、武器连击声等场景中尤为实用，因为玩家对这类音效有周期性预期。

序列容器之所以在游戏音频工作流中受到重视，是因为它提供了对播放顺序的精确控制能力。设计师可以精心编排素材顺序，例如将音色最厚重的版本放在第三位，使第三击产生"重击感"，这种细粒度的叙事节奏控制是随机容器无法实现的。

## 核心原理

### 播放指针与轮转逻辑

序列容器内部维护一个整数型播放指针（Play Index），初始值为 0，每次容器被触发时取出指针对应位置的素材播放，随后将指针加 1。当指针值等于列表总长度 N 时，触发重置逻辑：若容器设置为"循环模式（Loop）"，指针归零；若设置为"单次模式（One-Shot）"，后续触发将不再播放任何内容。这一机制保证了在 N 次连续触发中，每个素材恰好被播放一次。

### 随机化起始位置（Shuffle Start）

为防止玩家在每次游戏会话开始时总是听到序列第一个素材，许多实现提供了"随机起点"选项。启用该选项后，游戏会话初始化时播放指针会被随机设置到 0 到 N-1 之间的某个位置，但后续仍按顺序递增。Wwise 中对应的参数名为 **Shuffle**，勾选后序列的起始点在每次加载时随机，但整体轮转顺序不变。这与随机容器的完全随机有本质区别——整体周期结构被保留，只是起始相位不同。

### 全局与局部播放状态

序列容器的播放指针可以设置为**全局共享（Global）**或**每个游戏对象独立（Per Object）**。在全局模式下，场景中所有使用同一容器的声源共享同一个播放指针，第一个士兵的脚步声触发索引1，第二个士兵紧接触发索引2；在 Per Object 模式下，每个角色拥有独立指针，各自从头开始轮转。游戏中通常对角色动作音效选择 Per Object，避免多个角色相互"抢占"序列进度导致音色偏差。

### 素材排列与心理声学设计

序列容器中素材的物理排列顺序直接决定了玩家的听感节奏。经典的"三击重拳"设计为：[轻击_A, 轻击_B, 重击_C]，每三次攻击循环一次重击音效，与动画系统的连击节奏对齐。设计师可利用此特性在不修改代码的情况下，单纯通过调整容器内素材顺序来改变战斗的音频节奏感，这是序列容器区别于其他容器类型的核心设计价值。

## 实际应用

**脚步声系统**：角色在硬地面行走时，将4个录制自不同踩踏力度和位置的脚步声片段依序排列进序列容器。玩家会感知到脚步声"活"了起来，但又不像随机容器那样偶尔出现连续两次相同音效的尴尬。在 Wwise 的实际工程中，通常为每种地面材质（金属、木板、混凝土）各建立一个序列容器，再通过 Switch Container 根据地面类型切换。

**武器装弹动作**：半自动手枪每扣一次扳机对应一次独特的枪机循环声，使用序列容器排列 [击发音, 抛壳音起始, 抛壳音落地] 三段式音效，确保声音与动画帧同步，不会因随机容器的跳序而导致抛壳声提前或滞后出现。

**环境叙事音效**：在线性叙事游戏中，玩家每次经过某个区域时，背景对话或广播内容按序推进剧情。序列容器配合"单次模式"可实现播放到最后一条后静默，无需在蓝图或脚本中维护额外的状态变量，将叙事状态管理内聚在音频中间件层。

## 常见误区

**误区一：认为序列容器等同于播放列表**
序列容器并不像音乐播放器那样"排队播放"所有素材——它的每次触发只播放当前指针位置的单个素材，后续素材需要等待下一次事件触发。如果在 Wwise 中错误地期望一次 Post Event 就播放全部4个素材，结果只会听到第一个，其余三个在后续触发中才逐次出现。若需要在一次事件中顺序播放多段音频，应使用 Blend Container 中的播放列表功能或 Music Sequence 结构。

**误区二：忽略多声源共享指针导致的音色失衡**
在全局指针模式下，若场景中同时有10个敌人使用同一序列容器，指针会被10个声源迅速"消耗"推进，某些设计为关键节拍的素材（如上述第三个重击音效）会与预期触发时机完全错位。因此凡涉及多实例并发的音效，默认应选择 Per Object 模式，全局模式仅适用于在整个游戏范围内需要严格保持"全局进度"的叙事型音效。

**误区三：将序列容器与随机容器混用来"增加随机性"**
部分设计师会在序列容器外再套一层随机容器，试图兼顾两者优点。这会导致随机容器从多个序列容器中随机选一个，而每个序列容器内部维护独立指针，最终行为既非纯随机也非纯顺序，调试时极难预判下一次触发的音色。此需求在 Wwise 中应通过混合容器（Blend Container）的更精细权重配置实现。

## 知识关联

序列容器是在掌握随机容器之后自然延伸的容器类型：随机容器解决的是"每次听起来不一样"的问题，序列容器解决的是"每次以可控方式不一样"的问题。两者共同使用了"多素材池"的设计思想，但播放选择逻辑从概率抽样变成了确定性轮转。理解序列容器的 Play Index 机制是后续学习混合容器（Blend Container）的前提，因为混合容器将多个序列容器或随机容器作为子轨进行权重叠加，需要设计师清楚地知道每个子容器各自维护独立的播放状态。在 FMOD Studio 中，对应的概念是 Multi Instrument 中将 Playlist Mode 设置为 Sequential，功能与 Wwise 序列容器高度一致，参数命名略有不同但核心行为相同。
