# 平台预算

## 概述

平台预算（Platform Audio Budget）是指在音效优化工作中，根据目标运行平台的硬件限制，为音频系统分配的计算资源、内存空间与同时播放音效数量的总量限制。不同平台的处理器架构、内存容量和音频硬件差异巨大，因此 PC、主机（PlayStation/Xbox）和移动设备（iOS/Android）各自拥有截然不同的音频预算上限。

平台预算的概念随着多平台游戏开发的普及而正式化。2000年代初期，Xbox 和 PlayStation 2 的硬件差异迫使音频工程师开始系统性地记录每个平台的音频资源消耗。到了 2010 年代，移动游戏的兴起使移动端音频预算管理成为独立的专项工作，因为 iOS 和 Android 设备的可用内存往往只有主机的 1/10 甚至更少。

了解平台预算的意义在于：超出预算上限会导致音效丢帧（Audio Dropout）、内存溢出崩溃或同时播放数量超限时的强制静音，这些问题直接影响玩家体验，且往往在开发后期才被发现，修复成本极高。

## 核心原理

### 三大平台的典型预算参数

**PC平台**的音频预算相对宽松，现代游戏通常分配 256MB 至 512MB 的音频内存，同时播放音效数量（Voice Count）可达 512 个通道，CPU 占用率目标控制在 3%–5%。

**主机平台**（以 PlayStation 5 为例）专门配备了 Tempest 3D Audio 音频处理单元，将音频计算从主 CPU 卸载，允许同时处理最多数百个空间化音源。但开发者仍需遵循索尼规定的内存预算指南，通常为游戏整体 RAM 的 5%–8%。Xbox Series X 则通过 Project Acoustics 算法将音频内存预算设定在 64MB–128MB 之间的参考区间。

**移动平台**是限制最严格的环境。iOS 建议音频内存占用不超过 32MB，Android 因碎片化严重，保守目标为 16MB–24MB。同时播放通道数建议控制在 32–64 个以内，否则会触发系统级音频节流（Audio Throttling）。

### 预算的四个维度

平台预算并非单一数值，而是由以下四个维度共同构成：

1. **内存预算（Memory Budget）**：存储音频资产（PCM数据、压缩流）所占用的 RAM，单位为 MB。
2. **CPU预算（CPU Budget）**：每帧用于音频解码、DSP效果运算和混音的 CPU 时间，通常以毫秒（ms）为单位，移动端目标为每帧不超过 1–2ms。
3. **通道预算（Voice Budget）**：同时处于活跃播放状态的音效实例数量上限，超出后触发优先级裁剪（Voice Stealing）。
4. **流媒体预算（Streaming Budget）**：从磁盘实时读取音频流的 I/O 带宽，主机 SSD 可支持高带宽流式传输，而移动设备受限于闪存读取速度，单次允许流式加载的音频轨道数更少。

### 预算分配公式

估算静态音频内存占用时，使用如下公式：

**内存占用（MB）= 采样率（Hz）× 位深度（bit）× 声道数 × 时长（s）÷ 8 ÷ 1,048,576**

例如，一个 44100Hz、16bit、立体声、10秒的未压缩音效占用约 1.68MB。在移动端预算仅 24MB 的前提下，必须通过 Vorbis 或 ADPCM 压缩将其缩减至原始体积的 1/5 至 1/10。

## 实际应用

**跨平台项目的预算分层策略**：在使用 Wwise 或 FMOD 开发时，音频设计师会为同一个音效资产配置多个平台特定的变体。例如，PC 版本使用 24bit/48kHz PCM 格式，主机版本使用 ATRAC9（PS5）或 XMA2（Xbox），移动版本则降级为 16bit/22050Hz Vorbis 压缩，三个版本指向同一逻辑事件但消耗不同预算量。

**通道预算超限的处理案例**：《怪物猎人：世界》在开发过程中曾面临 PS4 版本音效通道超过 128 个的问题，解决方案是为近处敌人分配高优先级（Priority 100），远处环境声自动降为低优先级（Priority 10），当总通道数逼近上限时，系统自动裁剪最低优先级的声音实例。

**移动端预算监控**：在 Unity Audio Profiler 中，开发者可实时查看 AudioManager 的 DSP CPU 时间和内存分配。目标是将 Android 中低端设备（骁龙 665 级别）的音频 CPU 帧时间维持在 2ms 以内，超出则需削减活跃音效数量或降低 Reverb 等 DSP 效果的质量。

## 常见误区

**误区一：PC 预算无限，无需规划**
部分开发者认为 PC 平台内存充裕，可以忽略音频预算管理。实际上，即使是高配 PC，当同时播放通道数超过 1024 时，Windows Audio Session API（WASAPI）的混音延迟会显著上升，导致音频与画面不同步。此外，面向低配 PC 用户（4GB RAM、集成声卡）开发时，仍需遵循接近移动端的预算限制。

**误区二：压缩格式只影响内存，不影响通道预算**
通道预算与内存预算是两个独立维度。一个压缩为 Vorbis 格式的音效虽然内存占用减小了，但播放时每个活跃实例仍然占据一个通道槽位（Voice Slot）。将 100 个小音效全部压缩并不能让系统同时播放 200 个，通道上限由平台硬件和中间件配置共同决定，与文件格式无关。

**误区三：主机平台预算比移动端宽松，所以优先优化移动端**
这种思路会导致主机版本在认证（Certification）阶段被驳回。索尼和微软的平台认证要求中明确包含音频性能指标，若游戏在 PS5 上的音频 CPU 占用持续超过核心预算的 10%，会被标记为性能问题。主机和移动端的预算优化应并行进行，而非先后顺序。

## 知识关联

**前置概念——内存分析**：内存分析是量化平台预算现状的基础手段。通过对音频资产进行内存 profiling，可以得到当前实际消耗的 MB 数值，与平台预算上限对比后，才能确定优化优先级。没有准确的内存分析数据，平台预算只是抽象的限制数字，无法指导具体的资产删减或压缩决策。

**后续概念——解码成本**：确定了平台的 CPU 预算上限后，下一步需要量化每种音频格式的解码 CPU 开销。解码成本（Decode Cost）研究的是 Vorbis、ADPCM、Opus 等不同编解码器在目标平台上每帧消耗的计算量，这直接决定了在既定 CPU 预算内能同时解码多少个压缩音效实例，是平台预算向具体格式选择落地的关键环节。
