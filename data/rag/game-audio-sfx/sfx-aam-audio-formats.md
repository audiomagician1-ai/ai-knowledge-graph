# 音频格式

## 概述

音频格式是指数字音频数据的编码与存储规范，决定了声音文件的压缩方式、音质表现、解码开销和内存占用。在游戏音效管理领域，格式选择直接影响加载延迟、CPU解码消耗和最终包体大小。常见的游戏音频格式包括 WAV、OGG Vorbis、ADPCM、MP3 和 Opus，每种格式在比特率、延迟和平台兼容性方面各有取舍。

WAV（Waveform Audio File Format）由微软与IBM于1991年联合制定，采用线性脉冲编码调制（LPCM）存储未压缩的PCM音频数据。OGG Vorbis 则是 Xiph.Org 基金会于1998年开始开发的开源有损压缩格式，专为替代 MP3 设计。Opus 是2012年由 IETF 标准化的新一代编解码器，结合了 SILK 和 CELT 算法，在低比特率下仍能维持优异音质，逐渐在手游和网络语音场景中获得采用。

选择正确的音频格式对游戏项目至关重要：一款典型的主机游戏可能包含数千个音效文件，总计数GB的原始 PCM 数据；通过 OGG 压缩可将体积缩减至原来的 1/5 至 1/10，而 ADPCM 则以约 4:1 的压缩比在解码速度和质量之间提供平衡。

## 核心原理

### WAV（无压缩 PCM）

WAV 文件使用线性 PCM 编码，采样精度通常为 16-bit 或 24-bit，采样率常见值为 44100 Hz 或 48000 Hz。一个 16-bit、44100 Hz 的立体声 WAV 文件，每秒数据量计算公式为：

**数据量（字节/秒）= 采样率 × 位深度（bits）× 声道数 ÷ 8**

代入数值：44100 × 16 × 2 ÷ 8 = **176,400 字节/秒**（约 172 KB/s）

WAV 格式无需解码，CPU 读取后可直接送入音频硬件，因此解码延迟极低，适合需要**零延迟触发**的短促音效（如枪声、UI点击音）。但其文件体积大，不适合长音乐或大量资产的打包场景。

### OGG Vorbis（有损感知编码）

OGG Vorbis 采用改进的离散余弦变换（MDCT）结合心理声学模型，丢弃人耳不易感知的频率信息。典型游戏中使用 128 kbps 码率即可获得接近 CD 质量的音效，而 WAV 等效码率为 1411 kbps，压缩比约为 **11:1**。Vorbis 格式**无专利授权费用**，这使得 Unity、Godot 等引擎将其设为 PC/主机平台的默认压缩格式。其缺点是需要软件解码器，解码时占用一定 CPU 资源，且存在约 20~100 毫秒的解码初始化延迟。

### ADPCM（自适应差分脉冲编码调制）

ADPCM（Adaptive Differential Pulse-Code Modulation）是一种轻量级有损压缩算法，通过仅存储相邻采样点之间的**差值**而非绝对值来实现压缩。标准 IMA ADPCM 将 16-bit PCM 压缩为 4-bit，固定实现约 **4:1 的压缩比**，且解码运算量极小，可在嵌入式硬件或移动芯片上高效执行。ADPCM 特别适合**高频触发的短音效**（脚步声、环境循环音），因为其固定压缩比使随机访问变得简单，可以从文件任意位置开始解码，无需从头读取整个文件流。

### Opus（混合编解码器）

Opus 支持 6 kbps 至 510 kbps 的宽泛比特率范围，在 32 kbps 下音质即可超越同码率的 MP3 和 AAC。Opus 内部在低码率时使用基于 LPC 的 SILK 层处理语音，高码率时切换至 CELT 层处理音乐内容，两者可动态混合。帧长可选 2.5 ms 至 60 ms，帧长越短则延迟越低，适合实时语音通信；游戏环境音或背景音乐可选较长帧以提升压缩效率。

## 实际应用

**UI音效与武器音效（WAV 或 ADPCM）**：按钮点击、弹壳落地等需要在玩家操作后**立即响应**的音效，通常以 WAV 或 ADPCM 存储并**完整预加载至内存**。Unity 中对应 Load Type 为 "Decompress On Load"，读取后保留解压后的 PCM 数据在内存中，交换时间几乎为零。

**背景音乐（OGG Vorbis 流式播放）**：游戏背景音乐文件通常超过 2 分钟，以 OGG 格式存储后可从磁盘**流式读取**（Streaming），仅解压当前播放缓冲区，避免将整首曲目加载进内存。Unity 的 "Streaming" 模式即针对此场景，配合 OGG 使用时可将内存占用控制在数十 KB 级别而非数 MB。

**手游大量音效（ADPCM 批量压缩）**：在内存预算严格的移动平台（如目标设备内存仅 2 GB），对 500 个以上的短音效统一采用 ADPCM 编码，相比原始 WAV 节省约 75% 的存储空间，且解码不依赖复杂软件解码器，可减少低端设备上的卡顿风险。

**实时语音（Opus）**：多人在线游戏的玩家语音通信使用 Opus 编码，以 16 kbps、20 ms 帧长传输，在保证清晰语音的同时将网络带宽消耗压缩至极低水平。

## 常见误区

**误区一：WAV 质量最高，所有音效都应使用 WAV**

WAV 提供无损 PCM 数据，但这并不意味着所有场景都应使用。对于人耳难以分辨压缩痕迹的游戏背景音乐，以 192 kbps OGG 编码的感知质量与 WAV 几乎相同，却只需要约 1/7 的存储空间。在大型项目中盲目使用 WAV 会导致安装包超标，并使内存预算快速耗尽。

**误区二：OGG 可以替代所有格式**

OGG Vorbis 虽然通用性强，但它不支持从文件中间位置随机解码，这意味着触发频率极高的短音效（如每秒可能触发数十次的脚步声）若使用 OGG，每次触发都需从头解压，累积 CPU 开销显著。这类音效应优先选择 ADPCM 或直接使用预解压的 PCM。

**误区三：Opus 格式在所有游戏引擎中均获完整支持**

Opus 作为相对年轻的格式（2012年），在旧版本引擎中支持不完整。Unity 在 2021 年之前的版本对 Opus 的原生支持有限，部分平台（如较旧的 iOS 版本）需要通过插件实现解码。选用 Opus 前需确认目标平台和引擎版本的兼容矩阵。

## 知识关联

音频格式的理解是游戏声音资源管理的入门起点。掌握各格式的压缩比与解码特性之后，自然引出**压缩策略**的问题：如何根据音效类型、触发频率、目标平台的内存预算和 CPU 性能，系统性地制定整个项目的格式选用规则。例如，"短音效用 ADPCM、长音乐用 OGG 流式、高频环境音预加载 PCM" 这样的规则体系，正是在理解各格式内部编码机制后才能合理制定的压缩策略框架。音频格式的选择还直接影响**音频内存管理**：不同解压时机（加载时解压 vs. 播放时解压）对应着不同的内存与 CPU 的权衡，是后续优化工作的基础数据。
