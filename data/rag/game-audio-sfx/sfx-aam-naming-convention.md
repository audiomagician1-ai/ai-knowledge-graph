# 命名规范

## 概述

命名规范是指对游戏音效文件采用统一的前缀、分隔符、编号和描述词组合，使每个文件名本身就能传递资产类型、归属系统和版本信息的标准化实践。一个典型的游戏音效命名格式如 `SFX_ENV_Forest_Wind_Loop_01.wav`，其中每个字段段落承担独立的语义职责，与路径结构配合，使任何团队成员无需打开文件即可判断其内容和用途。

命名规范在游戏行业中随着项目规模增大而被标准化。早期90年代的小型游戏只有数十个音效文件，开发者可以随意命名。但到了AAA级项目时代，单部游戏的音效资产数量常超过10,000个文件，无规则命名导致的重复导入、错误引用和版本混乱问题使得流水线崩溃成为常态。Wwise、FMOD等音频中间件的普及进一步推动了行业对结构化命名的需求，因为这些工具的事件系统和资产浏览器完全依赖文件名进行批量操作和自动化导入。

音效文件命名规范直接影响流式加载系统的运行效率：当引擎按通配符模式批量预加载某类音效时（如加载所有匹配 `SFX_WPN_*` 的文件到武器声音池），规范的命名是这种自动化预加载得以实现的前提条件。

## 核心原理

### 字段结构与分隔符选择

标准的音效文件名由4至6个字段组成，字段间使用下划线 `_` 分隔（而非空格或连字符），原因是空格在命令行脚本和URL中会产生转义问题，而连字符容易与版本标记中的负号混淆。推荐字段顺序为：

```
[类型前缀]_[系统分类]_[对象名称]_[动作/描述]_[变体标签]_[编号].[扩展名]
```

例如 `SFX_UI_Button_Click_Soft_03.wav`，其中：
- `SFX` 表明这是音效资产（区别于 `MUS` 音乐、`AMB` 环境声、`VO` 语音）
- `UI` 是所属游戏系统
- `Button` 是触发对象
- `Click` 是动作描述
- `Soft` 是变体风格标签
- `03` 是同组第三个变体，编号从 `01` 开始而非 `0`，以确保双位数排序正确

### 目录结构层级设计

目录结构应与文件名的前两个字段保持镜像关系，这样当文件被移动或路径发生变化时，名称仍能提供足够的上下文信息。推荐的三级目录结构如下：

```
Audio/
  SFX/
    UI/
    Weapons/
      Melee/
      Ranged/
    Environment/
      Forest/
      Urban/
    Characters/
      Player/
      Enemy/
```

目录层级不应超过五层，过深的目录树会导致完整路径长度在Windows系统下超过260字符的MAX_PATH限制，使部分构建工具报错失败。

### 变体编号与循环标记规则

同一音效的多个变体（randomization pool）必须共享相同的前缀并以两位数字结尾，如 `SFX_WPN_Sword_Swing_01.wav` 到 `SFX_WPN_Sword_Swing_05.wav`。循环文件（Loop）必须在名称中显式标注 `_Loop` 而非依赖元数据标记，因为许多构建管线在导入时会根据文件名自动设置循环标志位，避免手动配置的人为失误。

单次触发（one-shot）文件则不添加任何循环标记，与Loop文件在名称上形成明确区分，方便音频程序员通过正则表达式 `.*_Loop_\d{2}\.wav$` 批量筛选所有循环文件进行专项处理。

### 平台后缀与格式标记

当同一音效需要针对不同平台输出不同格式时（PC使用Vorbis `.ogg`，主机平台使用AT9或XMA），可在扩展名前加入平台标记，如 `SFX_ENV_Rain_Loop_01_PS5.at9`。但更推荐的做法是将平台文件放入同名子目录，保持主文件名不变，由构建脚本按目标平台选择对应目录，这样可以避免平台名称污染主命名字段。

## 实际应用

**Wwise批量导入场景**：在Wwise的批量导入配置文件（`.wsources`）中，可以配置规则 `SFX_{@WorkUnit}_{@AudioSource}` 直接将文件名的第二和第三字段映射为Work Unit路径和Sound SFX对象名称。若命名规范不统一，这一自动映射完全失效，音频设计师需要手动配置数千个条目。

**版本控制冲突规避**：在Git LFS或Perforce管理的项目中，两名设计师同时新增武器音效时，若都命名为 `sword_new.wav`，必然产生冲突。规范命名下，第一位设计师提交 `SFX_WPN_Sword_Swing_06.wav`，第二位提交 `SFX_WPN_Sword_Hit_03.wav`，因为路径和名称完全不同，冲突不会发生。

**CI/CD自动化验证**：许多工作室的持续集成流水线中包含命名校验脚本，使用正则表达式 `^(SFX|MUS|AMB|VO)_[A-Z]{2,6}_[A-Za-z]+_[A-Za-z]+(_[A-Za-z]+)?_\d{2}\.(wav|ogg|mp3)$` 扫描所有新提交的音频文件，不符合规范的文件会阻断提交并触发报警通知。

## 常见误区

**误区一：认为中文或特殊字符命名更易读**
部分国内开发团队为了方便沟通，使用中文字符或括号命名音效文件，如 `剑击音效（重击）版本2.wav`。这在Windows资源管理器中可以正常显示，但当文件路径传入Python脚本、Unity资源加载API或命令行工具时，非ASCII字符会导致编码错误（常见错误为 `UnicodeDecodeError` 或 `FileNotFoundError`），且在Mac与Windows之间传输时因文件系统编码差异（HFS+ vs NTFS）可能产生乱码。

**误区二：用日期作为版本标记字段**
将文件命名为 `SFX_WPN_Gun_Shot_20240315.wav` 的做法在初期看似直观，但一旦音效在同一天被修改两次，日期标记失去区分意义；更严重的是，日期字段破坏了字母排序顺序，使同一对象的所有变体无法在文件浏览器中连续排列，导致设计师在数百个文件中难以快速定位整组变体。

**误区三：目录结构与文件名分类系统不一致**
将 `SFX_WPN_Sword_Swing_01.wav` 放入 `Audio/Characters/` 目录（因为剑是角色使用的）是一种常见的分类逻辑混乱。文件名中的系统分类字段（`WPN`）必须与其所在目录路径中的对应层级完全一致，否则两套分类系统会在音频中间件导入时产生路径-名称不匹配警告，且使基于路径的批量操作脚本失效。

## 知识关联

命名规范直接建立在**流式加载**机制的需求之上：流式加载系统依赖文件名通配符模式来决定哪些资产进入预加载队列、哪些按需加载，因此格式混乱的文件名会使流式分组逻辑完全失效。理解流式加载帮助设计师明白为何 `_Loop` 标记必须是名称的一部分而非元数据属性。

掌握命名规范后，学习**音频资产管线**时会发现，从录音→剪辑→格式转换→引擎导入→中间件事件绑定的完整流水线中，每个工具的交接接口（Handoff Interface）都以文件名作为唯一标识符。音频资产管线中的自动化脚本、批量处理工具和版本验证钩子，全部以本文所述的命名规范格式为基础进行设计，两者形成紧密的工程依存关系。
