# 声音虚拟化

## 概述

声音虚拟化（Voice Virtualization）是游戏音频引擎在超出硬件或软件发声通道（Voice）上限时，对低优先级声音实例进行"暂停追踪"处理的技术机制。当活跃声音数量超过引擎设定的最大发声限制——例如 Unreal Engine 的默认上限为 32 个活跃 Voice、Wwise 的默认 Virtual Voice 阈值约为 64 个——引擎不会直接丢弃这些超限声音，而是将其转入"虚拟"状态，保留播放位置与参数，等待重新激活的时机。

这一技术最早在 2000 年代中期随着开放世界游戏的兴起而变得不可或缺。早期游戏中声音数量有限，直接丢弃超限声音即可；但《上古卷轴》《侠盗猎车手》等场景复杂的作品中，同帧可能产生数百个声音请求，简单丢弃会导致重要音效的缺失或明显的音频跳变。虚拟化机制正是为解决这一矛盾而生。

声音虚拟化的意义在于它让引擎能够"假装"跟踪一个超出实际渲染能力的声音世界，当玩家移动靠近某个之前被虚拟化的音源时，该声音可以在正确的播放时间点重新加入混音，而不是重头播放——这对循环环境音和长时间持续音效尤为关键。

## 核心原理

### 虚拟化的触发条件

虚拟化的触发依赖于两个前置判断：**优先级排序**与**距离-音量阈值**。以 Wwise 为例，当一个声音的计算音量低于 Virtual Voice 设定的 Volume Threshold（默认 -80 dBFS）并且活跃 Voice 数量达到上限时，该声音进入虚拟状态。Unreal Engine 的 MetaSound 和 Sound Cue 系统则使用 `Max Concurrent Sounds` 配合优先级枚举值（0–100）进行竞争排序，被挤出的声音根据 `VirtualizationMode` 属性决定处理方式。

### 虚拟声音的三种处理模式

被虚拟化的声音通常有三种处理策略：

1. **Continue（持续追踪）**：引擎继续在内部推进该声音的播放游标（Playback Cursor），不消耗 DSP 资源，但保留时间信息。适用于音乐、环境循环音等需要时间同步的素材。
2. **Restart（重新开始）**：声音被标记为虚拟后停止追踪，当条件满足重新激活时从头播放。适用于短促的一次性音效，如脚步声或枪声，避免从中间位置突然出现怪异声响。
3. **Discard（直接丢弃）**：声音实例被彻底销毁，不再参与任何竞争。适用于转瞬即逝的低优先级反馈音效。

在 Wwise 中，这三种模式对应 Virtual Voice 设置面板中的 `Continue to play`、`Kill` 和 `Kill if longest`，可在 Sound Property 的 Advanced Settings 标签页逐资产配置。

### 重激活与时间补偿

当虚拟声音重新进入活跃范围，引擎需要决定从哪个时间点恢复播放。对于采用 Continue 模式的声音，引擎通过以下公式计算理论播放位置：

```
恢复位置 = (虚拟化进入时的播放游标) + (虚拟化持续时长 × 播放速率)
```

以一段 8 小节、120 BPM 的循环环境音（总时长 16 秒）为例：若声音在 5.2 秒处被虚拟化，虚拟状态持续了 3 秒，则重激活时应从 8.2 秒处恢复，保持与游戏世界时间线的同步。部分引擎（如 Unreal 5.x 的 Quartz 系统）还支持以量化网格（Quantization Grid）为单位进行时间对齐，避免环境音在错误的节拍点恢复时产生节奏错位。

## 实际应用

**开放世界城市环境音**：在《赛博朋克 2077》这类高密度城市场景中，每条街道同时存在人声、交通、机械、广播等数十个音源。距离玩家 80 米以上的行人对话声会被虚拟化为 Continue 模式，当玩家接近至 20 米范围内时，对话从正确的语句位置恢复，避免每次靠近都触发"新对话开头"的重复感。

**大型 RTS 战场音效**：在《星际争霸 II》类型的多单位战场中，可能同时存在超过 200 个攻击声音请求，但 Voice 上限通常仅为 64。距离摄像机较远的单位攻击音效被虚拟化为 Discard 模式，因为这类短促的射击声从中间重激活毫无意义，直接丢弃后等待下次攻击事件自然触发新实例即可。

**第一人称射击游戏的武器音效保障**：玩家自身武器音效会被赋予最高优先级（通常为 100/100），确保在任何虚拟化竞争中都不会进入虚拟状态。音效设计师会专门在 Wwise 或 FMOD 中将玩家武器的 Priority 设置为最高档，并将 `Limit Behavior` 设为 `Kill Lowest Priority Voice`，保证即使场面混乱，玩家自己的枪声永远清晰。

## 常见误区

**误区一：虚拟化等同于静音或丢弃声音**。实际上，进入虚拟状态的声音（在 Continue 模式下）仍在引擎内部"运行"，只是不消耗音频线程的 DSP 计算资源。它的播放游标仍在推进，这与直接 Kill 掉声音实例有本质区别。混淆这两者会导致设计师错误地依赖 Discard 模式处理需要时间连续性的环境音，产生玩家靠近时环境音始终从头播放的违和感。

**误区二：提高所有声音的优先级可以解决虚拟化问题**。这是典型的"优先级膨胀"（Priority Inflation）——当所有声音都被标为高优先级时，虚拟化的竞争判断失去参考基准，引擎退化为随机丢弃，反而比合理配置优先级时的表现更差。正确做法是建立清晰的优先级层级：玩家直接操控音效（100）→ 剧情关键音效（80–99）→ 近距离环境音（40–79）→ 远距离背景音（0–39）。

**误区三：虚拟化设置只需在引擎全局层面统一配置**。实际上，不同类型的音效需要逐资产设置不同的虚拟化行为。一个循环的工厂机械声需要 Continue 模式，而一个碰撞音效则应使用 Restart 或 Discard 模式。在 Wwise 中忽略 Sound 级别的 Virtual Voice 覆写设置，完全依赖默认全局参数，是导致音效行为异常的常见工程失误。

## 知识关联

声音虚拟化建立在**音效优先级**系统的判断结果之上——没有经过优先级排序的输入，虚拟化引擎无法判断哪些声音应当进入虚拟状态、哪些应当保持活跃。每次发声请求触发时，优先级计算必须先于虚拟化竞争完成。

掌握虚拟化机制后，自然延伸到**音频 LOD（Level of Detail）**系统。虚拟化是一种二值判断（活跃/虚拟），而音频 LOD 则是多级精细化控制——根据距离动态降低声音的采样率、减少混响计算复杂度、切换高低精度音频资产。可以将虚拟化理解为音频 LOD 中"完全不渲染"这一极端档位，而 LOD 系统在虚拟化之前提供了更细粒度的性能调节手段，共同构成游戏音频性能优化的完整工具链。
