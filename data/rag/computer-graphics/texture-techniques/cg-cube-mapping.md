# 立方体贴图

## 概述

立方体贴图（Cubemap）是一种由六张独立的二维纹理拼合而成的特殊纹理类型，这六张纹理分别对应一个虚拟立方体的六个面：正X（+X）、负X（−X）、正Y（+Y）、负Y（−Y）、正Z（+Z）、负Z（−Z）。采样时，程序不使用普通的 (u, v) 二维坐标，而是使用一个三维方向向量，该向量从立方体中心出发，射向六个面中的某一面，然后在该面上确定具体采样点。这种"以方向为索引"的机制使立方体贴图天然适合表示以某一点为中心、向四周延伸的全向环境信息。

立方体贴图最早在1986年由Greene在论文《Environment Mapping and Other Applications of World Projections》中正式提出，用于实时近似地反映周围场景。OpenGL在1.3版本（2001年）将其纳入核心规范，DirectX则在DirectX 7（1999年）中将其作为标准功能引入。此后，立方体贴图几乎成为实时图形渲染的标配特性，被所有主流图形API及着色器语言（GLSL、HLSL）支持。

立方体贴图的价值在于它以极低的运行时成本解决了三大常见渲染需求：天空盒（Skybox）渲染、动态或静态环境反射，以及镜面高光的方向性采样。相比全景等距柱状投影（Equirectangular），立方体贴图在极点处不存在像素密度失真，六个面的纹素密度均匀分布，从而在相同分辨率下提供更一致的视觉质量。

## 核心原理

### 方向向量到面坐标的转换

给定一个采样方向向量 **r** = (rx, ry, rz)，GPU 首先找出绝对值最大的分量，确定命中哪一面，再将其余两个分量除以最大分量的绝对值，把结果从 [−1, 1] 映射到 [0, 1] 得到该面上的 (s, t) 坐标。例如，当 |rx| 最大且 rx > 0 时，命中 +X 面，此时：

> s = (−rz / |rx| + 1) / 2  
> t = (−ry / |rx| + 1) / 2

这个投影本质上是中心投影（透视投影的特例，视角恰好为90°），因此立方体贴图的每张子纹理都对应一台视场角为90°×90°的虚拟摄像机所拍摄的画面，六台摄像机合起来恰好覆盖360°全向空间，无重叠、无遗漏。

### 天空盒的渲染原理

天空盒利用立方体贴图存储远景环境。渲染时，通常将天空盒的深度写入值强制设为最大值（深度缓冲中的1.0），使其始终位于所有场景物体之后。顶点着色器中，相机的平移分量被从视图矩阵中剔除（只保留旋转部分），以确保观察者无论移动到何处，天空盒始终表现为"无限远"。由于只传入方向向量而非位置，立方体贴图的采样结果仅与观察方向有关，完美吻合"远处不可达"的视觉感知。

### 环境反射与采样方向的计算

用立方体贴图模拟物体表面反射时，片段着色器需要计算入射视线方向 **v** 关于表面法线 **n** 的反射向量 **r**：

> **r** = **v** − 2(**v** · **n**)**n**

随后以 **r** 为索引采样立方体贴图，得到的颜色即近似为该表面在当前帧所能"看到"的环境颜色。这一方法被称为**静态环境映射**（Static Environment Mapping），它假设环境在无限远处不随物体移动而改变，因此适合反射天空、远处建筑等静态场景，对于近处动态物体的反射则需要改用动态立方体贴图（每帧从物体位置渲染六个方向）。

### 动态立方体贴图的代价

生成动态立方体贴图需要对场景进行**6次独立渲染Pass**，每次使用90°视场角的摄像机，分别面向六个轴向。这意味着绘制调用数量和几何处理量均扩大为6倍，是实时渲染中成本较高的技术。因此，现代游戏引擎（如Unreal Engine）通常将动态立方体贴图的更新频率降低（例如每秒更新一次，而非每帧），或仅对关键反射体启用，其余物体使用预烘焙的静态立方体贴图。

## 实际应用

**天空盒与天空球**：几乎所有3D游戏和可视化应用都使用立方体贴图绘制天空。Unity和Unreal Engine均内置了Cubemap类型的天空材质，开发者只需提供六张分辨率为512×512至4096×4096的HDR图片，引擎自动处理无缝拼接与采样。

**IBL（基于图像的光照）**：在基于物理渲染（PBR）管线中，立方体贴图被预滤波处理成两张：一张表示漫反射辐照度的"辐照度贴图"（Irradiance Map），另一张是按粗糙度分Mipmap级别存储的"预滤波环境贴图"（Pre-filtered Environment Map）。前者每个方向存储半球积分后的漫射光，后者结合Split-Sum近似（Brian Karis，2013年）实现无需光线追踪的高质量镜面反射。

**阴影立方体贴图（Omnidirectional Shadow Map）**：点光源（如灯泡）的阴影计算中，深度信息被写入立方体贴图的六个面，片段着色器通过从片段到光源的方向向量采样深度值，与当前距离比较，从而判断是否在阴影中，支持全向360°软硬阴影。

## 常见误区

**误区一：认为六张子纹理的坐标方向是统一的**。实际上，OpenGL和Direct3D在立方体贴图各面的Y轴朝向定义上存在差异，直接将为OpenGL制作的立方体贴图素材用于DirectX（或反之）会导致上下翻转或左右镜像的错误。具体而言，OpenGL的+Y面与DirectX的+Y面在V轴方向上相反，跨平台迁移时必须逐面检查翻转设置。

**误区二：用普通2D纹理的 (u, v) 坐标采样立方体贴图**。立方体贴图的GLSL采样函数是 `textureCube(samplerCube, vec3)` 或 `texture(samplerCube, vec3)`，传入的是三维方向向量，而非二维UV坐标。若错误地将顶点位置或屏幕坐标当作方向传入，会因向量未归一化或方向错误而产生严重的采样失真。

**误区三：认为立方体贴图的六个面之间一定无缝**。当立方体贴图的各面由不同图片拼合时，面与面交界处可能出现明显接缝。OpenGL通过 `GL_TEXTURE_CUBE_MAP_SEAMLESS`（OpenGL 3.2引入）启用硬件级的跨面双线性过滤，消除接缝；如果未开启此选项，在光滑反射表面的边缘会看到明显的颜色跳变。

## 知识关联

立方体贴图以普通2D纹理映射为基础——理解Mipmap层级、纹理过滤（双线性、三线性）、纹理环绕模式等概念，有助于解释为什么立方体贴图同样支持 `GL_LINEAR_MIPMAP_LINEAR` 过滤以及为何需要专门的 `SEAMLESS` 过滤模式。Mipmap在立方体贴图中依然按面独立存储，但LOD计算逻辑与2D纹理相同，均基于屏幕空间导数。

在PBR渲染流程中，立方体贴图是IBL实现的存储载体：辐照度贴图和预滤波环境贴图均以Cubemap格式储存，并通过球谐函数（Spherical Harmonics，最常用到L2阶，共9个系数）进行低频近似，以极小的显存（仅27个浮点数）替代完整的低分辨率辐照度Cubemap。此外，理解立方体贴图也是学习阴影技术中全向阴影贴图（Omnidirectional Shadow Map）的直接前提，后者将深度写入Cubemap六面，实现点光源的全向阴影投射。
