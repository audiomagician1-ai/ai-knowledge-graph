# 虚拟纹理

## 概述

虚拟纹理（Virtual Texturing），又称稀疏纹理（Sparse Texture），是一种将超大尺寸纹理按需分块加载到显存的技术。其核心思想类似于操作系统的虚拟内存分页机制：游戏中可能存在一张 16384×16384 像素的地形纹理，但摄像机每帧实际可见并需要高分辨率的区域只占其中一小部分，虚拟纹理只将这些"页面"载入物理显存，其余部分保留在低分辨率的后备图或磁盘上。

这项技术由 John Carmack 在 2012 年 QuakeCon 大会上进行了具体阐述，但学术界更早的奠基工作来自 Sean Barrett 于 2008 年发表的"Sparse Virtual Textures"演讲及相关论文。虚拟纹理技术解决的核心痛点是：GPU 显存容量（通常为 8GB 至 24GB）远小于游戏艺术资产的总纹理数据量（可能超过 100GB），常规预加载策略完全无法满足开放世界游戏的需求。

虚拟纹理的重要性体现在它打破了显存对纹理规模的硬性限制，使开发者能够在不牺牲近处细节的前提下渲染任意尺寸的连续地形或建筑表面。Unreal Engine 4.23 起将此技术作为内置功能（Runtime Virtual Texture）正式提供，Id Software 的 id Tech 6/7 引擎也将虚拟纹理作为其"Megatexture"系统的演进版本大量使用。

## 核心原理

### 页表与物理纹理页

虚拟纹理将逻辑上的大纹理划分为固定大小的"页"（Page），常见页尺寸为 128×128 或 256×256 texels。一张 16384×16384 的虚拟纹理在页尺寸 128 下产生 (16384/128)² = 16384 个页面。系统维护两张关键资源：

- **页表纹理（Page Table Texture）**：一张小尺寸的 R16G16 格式纹理，每个 texel 记录对应虚拟页在物理纹理中的坐标偏移，以及该页当前缓存的 mip 级别。
- **物理纹理（Physical Texture）**：一张真正驻留显存的纹理图集（Texture Atlas），其尺寸与显存预算匹配，例如 4096×4096，内部被切分为若干物理槽位，每个槽位恰好存放一个页面。

着色器采样时，先用原始 UV 查询页表纹理，得到物理纹理中的偏移，再用修正后的 UV 采样物理纹理，完成从"虚拟地址"到"物理地址"的两次查找。

### 反馈分析（Feedback Analysis）

每帧 GPU 需要主动告知 CPU"当前帧哪些页面被访问了"，这通过渲染一张低分辨率的**反馈缓冲（Feedback Buffer）**实现。反馈缓冲通常以屏幕分辨率的 1/8 或 1/16 大小渲染，每个像素写入当前所需的虚拟页 ID 及 mip 级别（通过 `ddx`/`ddy` 内置函数计算所需 mip）。CPU 读回该缓冲后对请求进行去重与优先级排序，将缺失的页面加入流式加载队列。这一反馈-加载-上传的往返延迟通常需要 1~3 帧，因此系统必须用低分辨率的"降级页"（Fallback Page）临时填充尚未加载完成的槽位，避免出现黑色区域。

### 页面的压缩格式与边界填充

物理纹理中的每个页面以 GPU 支持的块压缩格式存储，最常见的是 BC7（用于颜色/法线）或 BC4/BC5（用于单/双通道数据），以节省物理纹理图集的空间并提升带宽效率。由于 mip 过滤在页面边界处需要相邻页的数据，每个页面在实际存储时会包含若干 texel 的**边界填充（Border/Guard Band）**，典型值为 4 texels，因此一个"128 核心 + 4 边界"的页面实际占用 136×136 texels 的物理空间。边界填充的缺失会导致页面拼接处出现明显的滤波缝隙（Seam Artifact）。

## 实际应用

**开放世界地形渲染**：《地平线：零之曙光》使用虚拟纹理技术将整个游戏世界的地形颜色、高度、材质混合权重打包进单一的虚拟纹理空间，避免了传统分块地形在不同 Tile 材质之间产生的拼缝问题，并使材质混合计算可以完全在 GPU 侧完成。

**Unreal Engine Runtime Virtual Texture（RVT）**：UE4/5 中 RVT 的典型工作流是将地形材质的 Base Color、Normal、Roughness 预烘焙进虚拟纹理，静态网格（如岩石、建筑）可以在采样地形 RVT 后将自身颜色混入，实现贴地融合（Blending）效果。RVT 的物理纹理尺寸默认为 4096×4096，每页 128 texels，支持最多 14 个 mip 级别。

**DirectX 12 与 Vulkan 原生支持**：DirectX 11.2 引入了"Tiled Resources"，Vulkan 1.0 以扩展形式提供了"Sparse Binding"，允许开发者以 GPU 硬件内存页（通常 64KB）为粒度动态绑定或解绑显存，从而以更低开销实现稀疏纹理，无需手动维护物理纹理图集，着色器可以直接稀疏访问虚拟地址空间中已提交的区域。

## 常见误区

**误区一：虚拟纹理可以完全替代 Mipmap**

虚拟纹理的页表本身是按 mip 层级组织的，每个 mip 级别都有独立的页面集合，因此虚拟纹理内部仍然依赖 mip 层次结构。反馈缓冲通过 `ddx`/`ddy` 计算所需 mip 后，流式加载系统加载对应 mip 级别的页面。若误认为"虚拟纹理无需 mip"而只存储最高分辨率页面，远距离渲染时将出现严重的摩尔纹（Aliasing），且因为频繁加载无效的高分辨率页面导致带宽爆炸。

**误区二：反馈缓冲读回不存在性能问题**

GPU→CPU 的反馈数据回读（Readback）需要等待 GPU 完成渲染后 CPU 才能访问，若在同一帧内同步等待会导致 GPU 流水线停顿（Pipeline Stall）。正确做法是使用"延迟回读"策略：将第 N 帧的反馈结果在第 N+2 帧才读取，利用双重或三重缓冲的反馈缓冲避免阻塞，这也是系统能接受 1~3 帧加载延迟的根本原因。

**误区三：页面尺寸越小越好**

缩小页面尺寸（如从 128 降至 64）会使总页面数量增加 4 倍，导致页表纹理尺寸膨胀、反馈缓冲写入次数增加、页面管理 CPU 开销上升，同时边界填充的相对比例也从 (128+8)/128 ≈ 6.25% 上升至 (64+8)/64 = 12.5%，物理纹理空间浪费加剧。页面尺寸需要在流式加载粒度与管理开销之间权衡，128 或 256 是大多数实现的工程选择。

## 知识关联

**与纹理压缩的关系**：虚拟纹理的物理页面必须以压缩格式存储，这要求磁盘/内存中预存的页面数据已经是 BC1~BC7 等格式，而不是 RGBA8 原始数据，否则物理纹理图集的容量会减少 4~8 倍。纹理压缩的"块对齐"约束（BC 格式以 4×4 texel 为最小编码单位）也直接影响了虚拟纹理的边界填充必须为 4 的倍数这一设计决策。

**与流式加载（Streaming）系统的关系**：虚拟纹理的 I/O 子系统需要高效的异步磁盘读取能力，现代实现依赖 DirectStorage（Windows）或平台专有的直通 DMA 接口，将压缩页面数据直接传输至 GPU 显存而无需经过 CPU 内存拷贝，显著降低每次页面上传的延迟，使 1~2 帧内完成页面补全成为可能。
