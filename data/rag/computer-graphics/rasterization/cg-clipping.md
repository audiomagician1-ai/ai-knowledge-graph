# 裁剪

## 概述

裁剪（Clipping）是图形光栅化流水线中的一道工序，其目的是将几何图元限定在可见区域（裁剪窗口或视锥体）之内，丢弃完全在区域外的部分，并对跨越边界的图元进行切割，仅保留可见片段。没有裁剪步骤，光栅化器会试图处理坐标超出帧缓冲范围的像素，导致内存越界或错误渲染。

裁剪问题的系统性研究始于1960至1970年代。Cohen-Sutherland 算法由 Danny Cohen 与 Ivan Sutherland 于1967年前后提出，专门面向线段裁剪；Sutherland-Hodgman 多边形裁剪算法则由 Ivan Sutherland 与 Gary Hodgman 于1974年发表在《Communications of the ACM》上。这两篇里程碑文章奠定了现代实时裁剪的理论框架，至今仍被主流图形API（如OpenGL、Vulkan）的硬件实现所借鉴。

裁剪发生在图形管线的**齐次裁剪空间（Clip Space）**阶段，紧接顶点着色器输出之后、透视除法之前。在该空间中裁剪而非在屏幕空间裁剪，可以避免除以接近零的 w 值带来的数值不稳定问题，这是现代硬件将裁剪置于此阶段的根本原因。

## 核心原理

### Cohen-Sutherland 线段裁剪

Cohen-Sutherland 算法针对**矩形窗口与线段**的裁剪，核心思想是用一个4位区域编码（Outcode）描述端点相对于裁剪矩形的位置。矩形由四条边界 $x_{min}, x_{max}, y_{min}, y_{max}$ 定义，4位分别代表：

| 位 | 含义 | 条件 |
|---|---|---|
| bit3（左） | $x < x_{min}$ | LEFT |
| bit2（右） | $x > x_{max}$ | RIGHT |
| bit1（下） | $y < y_{min}$ | BOTTOM |
| bit0（上） | $y > y_{max}$ | TOP |

对线段两端点 $P_1, P_2$ 分别计算 Outcode $C_1, C_2$：

- 若 $C_1 \mathbin{|} C_2 = 0000$：两端点均在窗口内，**完全接受**；
- 若 $C_1 \mathbin{\&} C_2 \neq 0000$：两端点在同一侧窗口外，**完全拒绝**；
- 否则：线段跨越边界，求交点后缩短线段，重复判断。

这种"先快速判断再求交"的策略使大量完全在内或完全在外的线段无需执行昂贵的求交运算，显著减少计算量。

### Sutherland-Hodgman 多边形裁剪

Sutherland-Hodgman 算法将多边形裁剪分解为**依次针对裁剪窗口每条边界**的连续裁剪操作。对于凸多边形裁剪窗口，算法对多边形的每条边逐一遍历顶点，按如下四种情况输出新顶点序列：

1. 前一顶点在内、当前顶点在内 → 输出当前顶点；
2. 前一顶点在内、当前顶点在外 → 输出与裁剪边的交点；
3. 前一顶点在外、当前顶点在外 → 不输出；
4. 前一顶点在外、当前顶点在内 → 输出交点和当前顶点。

一个 $n$ 条边的多边形经过 $k$ 条裁剪边处理后，最多产生 $n + k$ 个顶点。Sutherland-Hodgman 适用于**凸多边形裁剪区域**；若裁剪区域为凹多边形，需使用更复杂的 Weiler-Atherton 算法（1977年）。

### 齐次空间裁剪与视锥体六平面

在3D图形管线中，裁剪针对视锥体的六个平面（近平面、远平面、左、右、上、下）。齐次裁剪空间中点 $(x, y, z, w)$ 满足可见条件：

$$-w \leq x \leq w,\quad -w \leq y \leq w,\quad -w \leq z \leq w$$

（OpenGL NDC约定，DirectX 中 $z$ 范围为 $0 \leq z \leq w$）

对三角形图元，当某顶点超出某平面时，图形硬件或驱动执行**三角形细分裁剪**：一个跨越边界的三角形可能被分裂为1到2个新三角形，以保证输出图元完整落在视锥体内。

## 实际应用

**UI渲染中的矩形裁剪**：在游戏引擎或操作系统GUI中，滚动容器（Scroll View）需要将子控件的绘制限定在容器矩形内。OpenGL 的 `glScissor(x, y, width, height)` 命令直接利用矩形裁剪原理，在光栅化阶段丢弃裁剪区域外的片元，避免子控件"溢出"到父容器之外。

**近平面裁剪防止除零**：当一个三角形的某顶点位于摄像机后方（$w < 0$ 或 $z < 0$），直接做透视除法 $x/w$ 会产生翻转或无穷大坐标，导致严重渲染错误。近平面裁剪在透视除法前将该三角形裁切，是驱动程序必须正确处理的关键边界情况。

**地图渲染中的瓦片裁剪**：矢量地图引擎（如 Mapbox GL）在将地理矢量线段投影到屏幕坐标后，使用 Cohen-Sutherland 的变体对线段进行瓦片边界裁剪，确保每块 256×256 像素瓦片只处理落在其范围内的线段，从而支持并行渲染。

## 常见误区

**误区一：认为裁剪和背面剔除（Back-face Culling）是同一操作**。背面剔除是根据法向量与视线方向的点积判断三角形整体是否朝向摄像机，属于图元整体丢弃；裁剪则是对跨越可见区域**边界**的图元进行几何切割，会产生新顶点。两者发生在管线的不同阶段，解决的是不同的可见性问题。

**误区二：认为 Sutherland-Hodgman 可直接处理凹多边形裁剪区域**。该算法要求裁剪多边形必须是凸多边形，若用其对凹形窗口裁剪，会产生多余的连接边，破坏输出多边形的正确性。对凹多边形裁剪窗口必须先将其分解为凸多边形集合，或改用 Weiler-Atherton 算法。

**误区三：认为裁剪越完整越好，应尽量细分**。硬件实现中，裁剪产生的额外顶点和三角形会增加后续光栅化的负担。因此 GPU 驱动通常先用**保守裁剪（Conservative Clipping）**快速剔除明显不可见的图元，仅对真正跨越边界的三角形执行精确裁剪，以在正确性与性能之间取得平衡。

## 知识关联

裁剪直接依赖**图形管线各阶段**的顺序约定：顶点着色器将顶点变换到齐次裁剪空间后，裁剪在该空间执行，随后才进行透视除法（除以 $w$）得到 NDC 坐标，最后映射到屏幕空间交由光栅化器处理。理解 MVP 矩阵变换（Model-View-Projection）是读懂裁剪坐标系的前提——特别是投影矩阵将视锥体压缩为标准立方体的过程，决定了裁剪的六个平面方程形式。

在裁剪之后，光栅化（Rasterization）接收已裁剪的凸多边形图元，对其进行扫描线填充与片元生成。裁剪保证了送入光栅化器的所有图元顶点均落在 NDC 的 $[-1,1]^3$ 范围内，从而使屏幕坐标映射和深度缓冲写入在数值上始终合法。
