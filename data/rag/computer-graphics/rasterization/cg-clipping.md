# 裁剪

## 概述

裁剪（Clipping）是图形光栅化流程中的一个几何处理步骤，其目标是剔除或截断位于可视区域（视口或裁剪窗口）之外的几何图元，确保只有落在屏幕或视锥体内的部分才会进入后续光栅化计算。如果不执行裁剪而直接将超出范围的图元送入光栅化，会导致错误的像素填充、内存越界写入，乃至硬件渲染错误。

裁剪算法的研究可以追溯到1960至1970年代。Cohen-Sutherland算法由丹·科恩（Dan Cohen）和伊万·萨瑟兰（Ivan Sutherland）在1967年前后提出，专门处理二维线段与矩形窗口的裁剪；Sutherland-Hodgman多边形裁剪算法则于1974年由萨瑟兰与加里·霍德曼（Gary Hodgman）联合发表，将裁剪扩展到任意凸多边形边界。这两种算法至今仍是图形学教学和硬件实现的基础参考。

在现代GPU管线中，裁剪发生在顶点着色器之后、光栅化之前的图元装配阶段。OpenGL和Vulkan规范均明确要求对标准化设备坐标（NDC）立方体 `[-1, 1]³` 以外的图元执行裁剪，硬件固定功能单元负责完成这一操作。

## 核心原理

### Cohen-Sutherland 线段裁剪

Cohen-Sutherland 算法使用一个4位**区域编码（Outcode）**对线段两端点进行分类。矩形裁剪窗口将平面划分为9个区域，每个端点的编码位定义为：

| 位 | 含义 | 条件 |
|----|------|------|
| Bit 3 (左) | 点在窗口左侧 | x < x_min |
| Bit 2 (右) | 点在窗口右侧 | x > x_max |
| Bit 1 (下) | 点在窗口下方 | y < y_min |
| Bit 0 (上) | 点在窗口上方 | y > y_max |

对两端点编码 `C1` 和 `C2` 进行如下判断：
- 若 `C1 | C2 == 0`（两端点均在窗口内），**完全接受（trivially accept）**；
- 若 `C1 & C2 != 0`（两端点在同侧窗口外），**完全拒绝（trivially reject）**；
- 否则，选取窗口外的端点，求其与窗口某条边界的交点，用交点替换该端点，并迭代。

交点计算利用线段参数方程，以与左边界（`x = x_min`）相交为例：

```
t = (x_min - x1) / (x2 - x1)
y_intersect = y1 + t * (y2 - y1)
```

此算法对大量线段均在窗口外的场景极为高效，因为多数线段可通过位运算快速拒绝，无需计算浮点交点。

### Sutherland-Hodgman 多边形裁剪

Sutherland-Hodgman 算法将凸裁剪区域的每条边界视为一个**半平面**，逐一对输入多边形进行裁剪，每次输出新的顶点列表再送入下一条边界的裁剪。对于矩形窗口，需依次经过4条裁剪边（左、右、上、下）。

针对输入多边形的每条边（顶点 `S` 到顶点 `P`），算法分四种情况处理：

1. **S 在内，P 在内**：输出 `P`；
2. **S 在内，P 在外**：输出 `S-P` 与裁剪边的交点；
3. **S 在外，P 在外**：不输出任何点；
4. **S 在外，P 在内**：输出交点和 `P`。

该算法要求裁剪区域必须是**凸多边形**，否则会产生多余的连接边。对凹多边形（如自交多边形），需改用Weiler-Atherton算法（1977年）。

### 三维视锥体裁剪（齐次坐标裁剪）

在三维图形管线中，裁剪在**裁剪坐标（Clip Space）**中执行，此时顶点坐标仍保留齐次分量 `w`。标准裁剪条件为：

```
-w ≤ x ≤ w
-w ≤ y ≤ w
-w ≤ z ≤ w   （OpenGL约定）
  0 ≤ z ≤ w   （DirectX/Vulkan约定）
```

在齐次空间中进行裁剪可以避免透视除法引入的数值不稳定性，特别是当顶点位于观察平面（`w ≈ 0`）附近时，直接在NDC空间裁剪会发生除以零的问题。因此现代GPU总是在透视除法之前完成裁剪，这是图形API规范的强制要求。

## 实际应用

**场景1：2D UI渲染中的滚动列表**  
在游戏引擎或浏览器渲染引擎绘制超出控件边界的列表项时，需要对每条文本行或图标矩形执行2D裁剪。利用Cohen-Sutherland的快速拒绝特性，可以在CPU端快速剔除完全不可见的UI元素，避免将其提交给GPU。

**场景2：OpenGL近裁剪面**  
当三角形跨越近裁剪面（near plane，`z = -w` 或 `z = 0`）时，硬件裁剪单元将生成新顶点并拆分图元。若近裁剪面设置过于接近0（如 `zNear = 0.001`），则深度精度丢失和裁剪频率都会显著增加，实践中推荐 `zFar / zNear ≤ 10000` 以保持深度缓冲精度。

**场景3：软光栅化实现**  
在CPU软渲染器（如游戏《毁灭战士》使用的BSP+列渲染）中，直接实现Sutherland-Hodgman多边形裁剪对所有三角形进行窗口裁剪，随后才进行扫描线转换，确保不会访问帧缓冲越界地址。

## 常见误区

**误区1：裁剪等同于视口变换后的边界检测**  
许多初学者认为裁剪只是判断像素是否在屏幕范围内，但实际上裁剪发生在光栅化之前的图元级别，处理的是连续几何体（线段、三角形），而非离散像素。像素级别的范围检查叫做**裁剪测试（scissor test）**，是光栅化之后的片段操作，与几何裁剪是两个不同阶段。

**误区2：Sutherland-Hodgman 可以裁剪任意多边形**  
Sutherland-Hodgman 只对**凸裁剪边界**正确。若将其用于凹多边形裁剪区域，输出多边形会出现虚假的连接边，视觉上表现为错误的填充区域。此场景必须使用支持凹多边形的Weiler-Atherton或Greiner-Hormann（1998年）算法。

**误区3：硬件总会将超出NDC的顶点自动丢弃**  
实际上GPU在`w < 0`（顶点在相机后方）时行为是未定义的，不同驱动和硬件的处理方式不一致。正确做法是依赖硬件裁剪单元在齐次空间内裁剪三角形，而不是假设超出范围的顶点会被简单丢弃。这一细节在实现自定义软件渲染器时尤为重要。

## 知识关联

**前置概念：图形管线阶段**  
裁剪在管线中的位置是：顶点着色 → 图元装配 → **裁剪** → 透视除法 → 视口变换 → 光栅化。理解裁剪需要掌握裁剪坐标系（Clip Space）和齐次坐标 `(x, y, z, w)` 的含义，特别是 `w` 分量在近平面裁剪中的作用。

**延伸方向：背面剔除与视锥体剔除**  
裁剪是精确的图元级操作，但代价较高（需生成新顶点）。实际渲染中常在裁剪之前先执行**背面剔除（Back-face Culling）**和**视锥体剔除（Frustum Culling）**，在更高层次（网格级别）快速拒绝不可见物体，从而减少到达裁剪阶段的图元数量，这两者是裁剪的效率补充手段。
