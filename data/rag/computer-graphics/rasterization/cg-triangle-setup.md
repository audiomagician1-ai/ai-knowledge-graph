# 三角形装配

## 概述

三角形装配（Triangle Assembly）是光栅化管线中将顶点数据转化为可填充几何图元的关键步骤，发生在顶点着色器与片段着色器之间。具体而言，该阶段接收经过视口变换后的屏幕空间顶点，按照索引缓冲区（Index Buffer）中存储的整数索引，每次取出三个顶点组合成一个三角形图元，供后续扫描线转换阶段使用。

三角形装配的概念随着1970至1980年代固定功能图形硬件的兴起而被系统化。早期的VLSI图形芯片（如SGI于1982年推出的相关芯片）便内置了图元组装逻辑，将其与几何变换电路分离，形成流水线结构。这一分离设计使得硬件可以在上一组顶点仍在变换时，同步对已完成变换的顶点进行图元组装，显著提升了吞吐量。

三角形装配的核心意义在于：它是光栅化管线中唯一负责"语义重建"的阶段——在此之前管线只知道孤立的顶点，在此之后管线才拥有具有面积、法向量和绕序（Winding Order）的封闭几何单元，后续的面剔除、扫描线转换和属性插值都依赖于此阶段输出的完整三角形结构。

## 核心原理

### 图元组装（Primitive Assembly）

图元组装的输入是一个有序的顶点流和一张索引表。以最常见的 `GL_TRIANGLES` 模式为例，每3个索引值 $(i_0, i_1, i_2)$ 被取出，从顶点缓冲区中读取对应的三顶点 $V_0, V_1, V_2$，构成一个独立三角形。OpenGL/Vulkan 同时支持 `GL_TRIANGLE_STRIP` 和 `GL_TRIANGLE_FAN` 两种条带/扇形模式，可分别用 $n+2$ 个顶点表达 $n$ 个共边三角形，将顶点冗余度从 $3n$ 降至 $n+2$，节省约 67% 的顶点带宽（对大型网格而言）。

索引缓冲区中的数据类型可以是 `uint16` 或 `uint32`，前者最多寻址 65,536 个不同顶点，后者可寻址约 42 亿个，选择时需与顶点缓冲区实际大小匹配，以避免越界读取产生的未定义行为。

### 面剔除（Face Culling）

面剔除是三角形装配阶段的重要优化步骤，它通过判断三角形在屏幕空间的绕序（Winding Order）来决定是否丢弃该三角形，从而避免对不可见背面进行后续的光栅化计算。

绕序通过计算屏幕空间中三角形的**有符号面积**来判定：

$$S = \frac{1}{2} \left[ (x_1 - x_0)(y_2 - y_0) - (x_2 - x_0)(y_1 - y_0) \right]$$

其中 $(x_i, y_i)$ 为第 $i$ 个顶点的屏幕坐标。若 $S > 0$，三顶点呈逆时针（CCW）排列；若 $S < 0$，则为顺时针（CW）排列。OpenGL 默认以 CCW 为正面，CW 为背面；开发者可通过 `glFrontFace(GL_CW)` 反转此约定。面剔除通过 `glEnable(GL_CULL_FACE)` 开启，并用 `glCullFace(GL_BACK)` 指定剔除背面。

对于封闭实体网格，面剔除可以去除约 50% 的三角形，使后续扫描线转换和片段着色的工作量减少一半。

### 三角形遍历准备（Triangle Setup for Traversal）

装配完成并通过面剔除的三角形，在进入扫描线转换之前还需进行"三角形设置"（Triangle Setup），即预计算三条边的斜率和增量信息。具体包括：计算三顶点的屏幕坐标包围盒（Bounding Box）以确定需扫描的像素行范围，以及计算每条边的 $\Delta x / \Delta y$（即每行扫描时左右边界的水平位移量）。这些预计算值以固定点数（通常为 4 位或 8 位子像素精度）存储，直接供扫描线转换硬件使用，避免在逐行遍历时重复除法运算，GPU 硬件通常在单个时钟周期内完成这一计算。

## 实际应用

**游戏引擎中的背面剔除配置**：虚幻引擎（Unreal Engine）在渲染不透明几何体时默认开启 `GL_BACK` 面剔除。对于树叶、栅栏等双面可见的网格，美术人员需在材质设置中勾选"Two Sided"，引擎会在渲染时临时禁用 `GL_CULL_FACE`，或使用 `gl_FrontFacing` 内置变量在片段着色器中翻转法线，而非生成双份几何体。

**地形渲染中的条带索引**：大型地形通常以规则网格存储，使用 `GL_TRIANGLE_STRIP` 加重启索引（Restart Index，值为 0xFFFF 或 0xFFFFFFFF）可将整行地形顶点打包为单次绘制调用，相比 `GL_TRIANGLES` 减少约 65% 的索引数据量，降低了 CPU-GPU 数据传输压力。

**绕序一致性错误排查**：当模型导入引擎后出现"黑洞"（某些面不可见）时，通常是因为建模软件（如 Blender）导出时保留了 CW 绕序，而渲染 API 期望 CCW。此时可通过在 RenderDoc 等帧调试工具中开启"线框模式 + 背面高亮"，快速定位绕序错误的三角形面片。

## 常见误区

**误区一：认为面剔除发生在顶点着色器之后、图元装配之前**
实际顺序恰好相反：必须先将三个顶点组装成三角形，才能计算有符号面积来判断绕序。孤立顶点没有"正面"或"背面"的概念，面剔除在逻辑上不可能先于图元组装执行。部分教材将"剔除"与"裁剪"混淆——裁剪（Clipping）发生在齐次除法前，针对的是图元相对于视锥体的位置，而面剔除针对的是图元面向摄像机的朝向，两者在管线中的位置和算法均不同。

**误区二：认为有符号面积为零的三角形会通过面剔除**
当三角形三点共线时，有符号面积 $S = 0$，该三角形为退化三角形（Degenerate Triangle）。此类三角形不应进入扫描线阶段——它在屏幕上没有可填充的像素，但若不过滤，扫描线算法中的除法运算可能出现除以零错误。正确的实现应在面剔除阶段同时丢弃 $S = 0$ 的退化三角形，而非仅处理 $S < 0$ 的情况。

**误区三：条带模式（TRIANGLE_STRIP）不需要关心绕序**
`GL_TRIANGLE_STRIP` 模式中，GPU 会对偶数序号三角形自动交换两个顶点的顺序，以确保整条条带的绕序保持一致（全部 CCW 或全部 CW）。若开发者手动构造条带索引时未遵从此规律，将导致相邻三角形绕序交替翻转，开启背面剔除后会出现"斑马纹"——每隔一个三角形被剔除的视觉异常。

## 知识关联

**前置依赖——视口变换**：三角形装配的输入顶点必须已完成从裁剪空间到屏幕空间的转换，屏幕坐标的数值范围（如 $[0, 1920] \times [0, 1080]$）直接决定了有符号面积公式中 $(x_i, y_i)$ 的合法取值范围。若在视口变换前执行装配，绕序判断将基于错误的坐标系，导致面剔除结果不可预期。

**后续阶段——扫描线转换**：三角形装配输出的三角形包围盒和边斜率预计算值，正是扫描线转换的直接输入。扫描线算法逐行扫描包围盒内的像素行，利用装配阶段已计算好的 $\Delta x / \Delta y$ 值增量式地推进左右边界，避免从头重新计算交点。

**后续阶段——属性插值**：装配阶段除坐标外还保留了三顶点的全部属性（纹理坐标、颜色、法线等），这些属性将在片段着色阶段通过重心坐标插值分配给三角形内的每个像素。装配阶段对绕序和退化三角形的处理，确保了重心坐标计算中分母（即有符号面积的两倍）不为零，从而保证插值的数值稳定性。
