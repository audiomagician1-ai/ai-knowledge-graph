# 视口变换

## 概述

视口变换（Viewport Transform）是光栅化管线中将归一化设备坐标（NDC，Normalized Device Coordinates）映射到屏幕像素坐标的最后一步几何变换。经过投影变换和透视除法后，所有可见几何体的坐标均落在 NDC 空间的 \([-1, 1]^3\) 立方体内，而视口变换的任务就是把这个标准立方体的 XY 平面"拉伸"到实际屏幕分辨率所定义的矩形窗口上。

视口变换的概念随着可编程光栅化硬件的普及而标准化。OpenGL 在 1992 年发布的规范中即明确定义了 `glViewport(x, y, width, height)` 接口，Direct3D 则通过 `D3D11_VIEWPORT` 结构体描述相同的概念。无论 API 如何变化，其数学本质始终是一个线性映射，将 NDC 的 \([-1,1]\) 区间缩放平移至屏幕像素范围。

视口变换直接决定渲染结果在窗口中的位置和尺寸，理解它的像素中心约定对于正确处理亚像素精度、纹理采样和抗锯齿至关重要。错误地理解像素中心的对齐方式会导致纹理偏移半个像素（half-pixel offset），这是从 Direct3D 9 迁移到 Direct3D 10 以上版本时最常见的历史遗留问题之一。

## 核心原理

### 数学公式：从 NDC 到屏幕坐标

设视口左下角像素坐标为 \((x_0, y_0)\)，视口宽度为 \(W\)，高度为 \(H\)，则视口变换公式为：

$$
x_{screen} = \frac{W}{2} \cdot x_{NDC} + \left(x_0 + \frac{W}{2}\right)
$$

$$
y_{screen} = \frac{H}{2} \cdot y_{NDC} + \left(y_0 + \frac{H}{2}\right)
$$

深度值的映射公式单独定义：

$$
z_{screen} = \frac{f - n}{2} \cdot z_{NDC} + \frac{f + n}{2}
$$

其中 \(n\) 和 \(f\) 是近、远深度范围（在 OpenGL 中由 `glDepthRange(n, f)` 设置，默认为 \([0.0, 1.0]\)）。注意这里的 \(n, f\) 不是投影矩阵中的近远裁剪平面距离，而是深度缓冲区的值域范围，两者含义不同。

### 像素中心约定（Pixel Center Convention）

像素中心约定规定每个像素的采样点位于其中心，而非左上角或左下角。OpenGL 从规范诞生之日起就采用像素中心位于半整数坐标的约定，即像素 \((i, j)\) 的中心位于屏幕坐标 \((i + 0.5,\; j + 0.5)\)。

Direct3D 9 及更早版本曾采用像素中心位于整数坐标 \((i, j)\) 的约定，这导致全屏四边形绘制时必须施加 \((-0.5, -0.5)\) 的偏移来对齐纹素和像素。Direct3D 10 起统一改为与 OpenGL 相同的半整数约定，消除了这一历史包袱。Vulkan 和 Metal 也均采用半整数像素中心约定。

### Y 轴方向差异

NDC 空间中 Y 轴向上为正，而许多屏幕坐标系（如 Vulkan 的裁剪空间、HTML Canvas、大多数图像格式）以左上角为原点、Y 轴向下为正。OpenGL 屏幕坐标以左下角为原点，Y 轴向上，因此其视口变换不涉及 Y 轴翻转。Vulkan 则在 NDC 层面就定义了 Y 轴向下，或者通过 `VK_KHR_maintenance1` 扩展允许负值高度来翻转视口，从而在着色器层面与 OpenGL 保持兼容。开发者需要根据所用 API 明确判断是否需要在视口变换阶段执行 Y 轴翻转。

### 视口裁剪与视口变换的关系

视口变换本身**不执行裁剪**。裁剪（Clipping）在 NDC 之前于裁剪空间完成，视口变换仅负责线性映射。因此，如果上游裁剪步骤没有正确剔除 NDC 范围外的图元，视口变换后的坐标可能超出屏幕分辨率范围。光栅化器通常还有一道"保护带（guard band）"裁剪机制作为兜底，但这属于实现细节而非规范要求。

## 实际应用

**分屏渲染（Split-screen）**：多人游戏中将屏幕分为若干区域，每个玩家视角对应一个不同的视口矩形。通过为每个相机调用不同的 `glViewport` 参数，同一套 NDC \([-1,1]^2\) 坐标可以分别映射到屏幕左半部分和右半部分，而无需修改投影矩阵或几何数据。

**小地图覆盖层（Minimap Overlay）**：将视口设为屏幕右下角一块 200×200 像素的区域，用俯视正交投影渲染场景轮廓，叠加在主视角之上。两次绘制调用使用不同的视口参数，但共享同一个帧缓冲区。

**纹理渲染与离屏渲染**：渲染到纹理时，视口必须设置为与纹理分辨率匹配，例如渲染到 512×512 的阴影贴图时需设 `glViewport(0, 0, 512, 512)`，否则 NDC 坐标只会映射到纹理的一个子区域，产生内容缩小或偏移的错误。

**像素精确 2D UI 渲染**：对于像素精确的 UI 绘制，顶点坐标通常直接以屏幕像素为单位给出，然后在顶点着色器中除以分辨率转为 NDC，再经视口变换还原。这一过程与像素中心约定密切相关：若顶点坐标为整数，视口变换后恰好落在像素边界而非像素中心，光栅化可能产生半像素的抖动，因此顶点坐标通常应偏移 0.5 以对准像素中心。

## 常见误区

**误区一：视口变换会改变深度值的裁剪行为**
部分初学者认为调整 `glDepthRange` 的值域可以扩展深度测试的精度范围。实际上，`glDepthRange(n, f)` 仅影响 NDC 深度 \(z_{NDC} \in [-1, 1]\) 到深度缓冲区值 \([n, f]\) 的线性映射，不改变深度裁剪发生在 NDC 层的规则。深度精度由投影矩阵的近远平面比值决定，而非由视口变换的深度范围参数决定。

**误区二：视口大小与帧缓冲大小相同时无需关注视口设置**
窗口大小改变（如用户拖动调整分辨率）时，帧缓冲区往往需要重建，而视口参数不会自动跟随更新。在 OpenGL 中，视口初始值等于窗口创建时的大小，但窗口 resize 事件不会自动触发视口更新，必须手动在回调中重新调用 `glViewport`。忽略这一点会导致渲染内容只占据新窗口的一个角落。

**误区三：Direct3D 9 的半像素偏移问题已经自动解决**
在使用将旧版 D3D9 着色器移植到现代 API 的代码时，开发者可能认为 API 层面的像素中心约定统一后问题自动消失。但如果原始顶点数据中已经内嵌了 \(-0.5\) 偏移（这在旧代码库中极为常见），移植到 D3D10+/Vulkan/OpenGL 后反而会引入 0.5 像素的错位，必须在视口变换或顶点坐标处理阶段显式移除该偏移。

## 知识关联

**前置概念——投影变换**：投影变换（透视或正交）将观察空间坐标变换到裁剪空间，随后透视除法（除以 \(w\) 分量）产生 NDC 坐标，视口变换的输入正是透视除法的输出。理解投影矩阵如何压缩 Z 值有助于正确配置视口变换的 `glDepthRange` 参数。

**后续概念——三角形装配**：视口变换完成后，顶点坐标已经是屏幕像素空间中的浮点坐标，三角形装配阶段将这些坐标按图元类型（三角形、线段、点）组合，随后光栅化器基于像素中心约定判断哪些像素被三角形覆盖。视口变换输出坐标的精度和像素中心对齐方式直接影响光栅化的覆盖判断结果，两者之间存在紧密的数学衔接关系。
