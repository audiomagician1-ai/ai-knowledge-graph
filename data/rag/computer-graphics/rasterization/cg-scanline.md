# 扫描线转换

## 概述

扫描线转换（Scan-line Conversion）是光栅化阶段将几何图元（通常是三角形）转换为像素集合的核心算法。其基本思想是：用一系列水平扫描线（y = 常数）从上到下逐行扫过多边形，找出每条扫描线与多边形边界的交点，然后填充两个交点之间的所有像素。这个过程将连续的几何描述转化为离散的像素网格，是 GPU 光栅化单元执行的最基础操作之一。

该算法起源于 1960 至 70 年代早期计算机图形学研究。1967 年，Jack Bresenham 提出了高效的直线光栅化算法，而针对多边形的扫描线填充则在 Ivan Sutherland 等人的工作中逐步成型。早期实现完全由 CPU 软件处理，直到 1980 年代专用图形加速硬件出现后，扫描线转换才被固化进硬件流水线。现代 GPU 的光栅化单元本质上仍然执行扫描线转换的变体逻辑，只是高度并行化。

扫描线转换之所以重要，在于它直接决定了哪些像素片段（Fragment）会被送入片段着色器处理。理解扫描线转换有助于解释为什么小三角形在亚像素尺度下可能产生零片段，以及为什么规范化设备坐标系中三角形的顶点顺序（CW vs CCW）与最终像素覆盖范围密切相关。

## 核心原理

### 活动边表（Active Edge Table, AET）

扫描线转换的经典数据结构是**边表（Edge Table, ET）与活动边表（Active Edge Table, AET）**的组合。边表按多边形每条边的最小 y 坐标（ymin）分桶存储，每条边记录四个字段：

- `ymax`：边的最大 y 坐标（扫描线超过此值时从 AET 删除该边）
- `x_at_ymin`：边在 ymin 处的 x 坐标（初始交点）
- `1/m`：x 的增量，即斜率的倒数（`Δx = Δy / 斜率`）
- `next`：指向同一桶中下一条边的指针

当扫描线从 y = ymin_global 逐行向上移动时，将新进入的边加入 AET，将超出 ymax 的边移出 AET，并对 AET 内所有边的 x 坐标执行 `x += 1/m` 的增量更新。AET 始终按 x 坐标排序，使得相邻交点对可以直接确定填充区间 `[x_left, x_right]`。

### 交点的奇偶规则与非零绕数规则

当扫描线与多边形有多个交点时，如何判断哪些区间属于多边形内部，依赖两种规则：

**奇偶规则（Even-Odd Rule）**：从任意点向外射一条射线，射线与边界的交叉次数为奇数则在内部，偶数则在外部。对于简单凸多边形，每条扫描线恰好产生 2 个交点，填充两点之间即可。对于自相交多边形，可能有 4、6 个交点，奇偶规则决定哪些区间填充。

**非零绕数规则（Non-Zero Winding Rule）**：计算射线与各边相交时边的方向贡献（向上穿越 +1，向下穿越 −1），绕数不为零则在内部。SVG 和 OpenGL 的多边形填充默认可配置使用这两种规则之一。

### x 坐标增量计算与顶点特殊处理

扫描线与三角形边的交点 x 坐标通过线性插值计算：

$$x = x_0 + (y - y_0) \cdot \frac{x_1 - x_0}{y_1 - y_0}$$

其中 $(x_0, y_0)$、$(x_1, y_1)$ 为边的两端顶点坐标。

顶点处理是扫描线转换的难点。当扫描线恰好经过顶点时，必须避免重复计数该顶点导致奇偶性错误。通用约定是：**水平边完全忽略**；对于非水平边，若顶点是该边的局部最小点（ymin端），计为一个交点；若是局部最大点（ymax端），则不计入当前扫描线的交点集合。这一"上闭下开"约定确保每条扫描线的交点数目始终保持偶数。

### 重心坐标插值与属性填充

现代扫描线转换不仅填充像素颜色，还需对顶点属性（UV 坐标、法线、深度值）进行插值。对于扫描线上位置为 $(x, y)$ 的像素，利用三角形三顶点 $V_0, V_1, V_2$ 的重心坐标 $(\lambda_0, \lambda_1, \lambda_2)$（满足 $\lambda_0 + \lambda_1 + \lambda_2 = 1$）进行线性插值：

$$\text{attr}(x,y) = \lambda_0 \cdot \text{attr}_0 + \lambda_1 \cdot \text{attr}_1 + \lambda_2 \cdot \text{attr}_2$$

需要注意的是，在透视投影下直接对屏幕空间坐标做线性插值会产生扭曲，必须使用**透视正确插值**：先除以各顶点的 $w$ 值，插值后再乘以插值后的 $w$。

## 实际应用

**软件光栅化器（如 Mesa llvmpipe）**：Mesa 的软件光栅化实现中，三角形被分解为"平顶三角形"和"平底三角形"两种标准形式，每种形式各有固定的两条斜边，扫描线可以直接从底部到顶部线性步进，不需要维护完整 AET，提升了缓存局部性。

**游戏引擎的 CPU 端遮挡剔除**：Unreal Engine 和 Unity 的 CPU 端遮挡剔除系统（如 Software Occlusion Culling）会在低分辨率缓冲区上执行扫描线转换，将简化的遮挡体写入深度缓冲，再查询待渲染对象是否被遮挡，从而在向 GPU 提交 DrawCall 之前就剔除不可见对象。

**字体渲染（FreeType）**：FreeType 库在将 TrueType 字体轮廓光栅化为位图时，对每个字形轮廓执行扫描线转换，使用非零绕数规则处理复杂字符（如汉字的内外轮廓），并在亚像素级别（水平方向 3 倍分辨率）执行扫描线以实现 ClearType 式抗锯齿。

**OpenGL 光栅化规范**：OpenGL 规范定义了"Diamond Exit Rule"来处理扫描线恰好经过像素中心的边界情况，确保相邻三角形共享的边不会被重复光栅化（即不出现重复填充）也不会遗漏（即不产生空隙），这是扫描线转换在 GPU 硬件层面的具体规范体现。

## 常见误区

**误区一：认为扫描线转换只处理"填充"而不处理边界**
实际上，扫描线转换同时负责确定哪些像素属于边界（边缘像素）与内部像素。现代 GPU 使用覆盖率（Coverage）测试：判断像素中心（或多采样点）是否在三角形内部，这本质上就是扫描线的"点在多边形内"判断，只不过对每个采样点独立执行。许多学生误以为边界像素由单独的"描边"步骤处理，但在三角形光栅化中，边界和内部统一由扫描线的交点计算决定。

**误区二：认为 1/m 增量更新在所有情况下都精确**
用整数或有限精度浮点数累加 `1/m` 时，每一步都会引入舍入误差，多步累积后 x 坐标的偏移可能达到 0.5 像素以上。Bresenham 直线算法通过只使用整数加法和误差项避免了浮点累积误差，这正是经典扫描线算法在实现时常用 Bresenham 的增量误差方法替代直接浮点除法的原因。用浮点 `x += 1/m` 的简单实现在高分辨率或长边情况下会产生可见的锯齿偏移。

**误区三：将屏幕空间线性插值等同于透视正确插值**
直接对三角形顶点的纹理坐标 (u, v) 在屏幕空间做线性插值，在透视投影下会产生明显的"纹理扭曲"（纹理沿对角线方向弯曲）。透视正确插值要求插值 `u/w` 和 `1/w`，最后用 `(u/w) / (1/w)` 恢复真实的 `u` 值。这一错误在早期 PlayStation 1 游戏中普遍可见，因为 PS1 的 GPU 使用了仿射纹理映射（即屏幕空间线性插值）而非透视正确插值。
