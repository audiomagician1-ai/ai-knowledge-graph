# 投影变换

## 概述

投影变换是将三维空间中的几何体映射到二维平面的数学操作，在图形渲染管线中负责将观察空间（View Space）中的顶点坐标转换为裁剪空间（Clip Space）坐标。经过投影变换后，顶点坐标会被存储为齐次坐标形式 `(x, y, z, w)`，其中 w 分量不再为 1，后续通过透视除法（将 xyz 各分量除以 w）才能得到归一化设备坐标（NDC）。

投影变换的概念最早随着计算机图形学在 1960–1970 年代的形成期确立，Ivan Sutherland 在 1965 年提出的"终极显示"概念和他主导的 Sketchpad 系统奠定了空间变换流水线的基础。两种主流的投影类型——**透视投影**与**正交投影**——分别对应人眼感知世界的方式与工程制图的需求，适用于完全不同的应用场景。

投影变换的意义在于它将三维场景的深度信息以特定方式编码进 z 分量，同时定义了可见区域（视锥体 Frustum 或正交盒），为后续的视锥体裁剪提供了标准化的坐标系。选择错误的投影类型或设置不当的参数（如近平面 near 过小）会直接导致 z-fighting、深度精度不足等严重渲染问题。

## 核心原理

### 透视投影矩阵

透视投影模拟"近大远小"的视觉效果，其核心思想是：距离相机越远的物体，投影后在屏幕上占据的像素越少。OpenGL 风格的透视投影矩阵由四个参数定义：垂直视野角 `fovy`（单位：弧度）、宽高比 `aspect`、近裁剪面 `near`（记为 n）、远裁剪面 `far`（记为 f）。

令 `t = n × tan(fovy/2)`，`r = t × aspect`，透视投影矩阵为：

```
M_persp = 
| n/r   0      0           0        |
|  0   n/t     0           0        |
|  0    0   -(f+n)/(f-n)  -2fn/(f-n)|
|  0    0      -1          0        |
```

第四行中的 `-1` 使得变换后顶点的 w 分量等于原始的 `-z`（在 OpenGL 右手坐标系中，相机看向 -z 轴）。透视除法后，z 值被压缩到 `[-1, 1]` 区间，但这一压缩是**非线性的**——靠近 near 平面的区域精度高，靠近 far 平面的区域精度急剧下降。这正是为什么 n 和 f 之间的比值 `f/n` 不应超过 1000 的实践准则的数学根源。

### 正交投影矩阵

正交投影不产生近大远小的效果，平行线在投影后仍保持平行，常用于 CAD 软件、2D 游戏 UI 层和等距视角游戏。正交投影将一个轴对齐的盒子 `[l, r] × [b, t] × [n, f]` 映射到 NDC 的 `[-1,1]³` 立方体，矩阵为：

```
M_ortho = 
| 2/(r-l)    0        0       -(r+l)/(r-l) |
|   0     2/(t-b)     0       -(t+b)/(t-b) |
|   0        0    -2/(f-n)   -(f+n)/(f-n)  |
|   0        0        0            1       |
```

正交投影矩阵的 w 行保持 `(0,0,0,1)`，因此变换后 w 分量始终为 1，透视除法不改变 xyz 的值。这也意味着正交投影的 z 缓冲精度是**线性分布**的，深度精度问题远少于透视投影。

### 视锥体与裁剪空间的关系

透视投影定义的视锥体是一个四棱锥台（Frustum），由 6 个平面围成：近平面、远平面、左右上下四个侧平面。正交投影定义的则是一个矩形盒子（Orthographic Box）。投影矩阵的作用是将这两种形状分别"压扁"为标准 NDC 立方体，使得裁剪算法只需对 `[-1,1]³` 这一统一形状进行判断。在 DirectX 的约定中，NDC 的 z 范围是 `[0,1]` 而非 `[-1,1]`，对应的矩阵第三行系数有所不同，这是跨平台开发时的常见错误来源。

## 实际应用

**第一人称 3D 游戏**：使用透视投影，典型 fovy 设置为 60°–90°（约 1.047–1.571 弧度）。fovy 过小（如 30°）会产生"长焦压缩"感，fovy 过大（如 120°）则产生鱼眼畸变，开发者依据游戏体验选择合适值。Unity 引擎中 Camera 组件的 `Field of View` 属性直接对应此参数。

**等距策略游戏（如《文明》系列）**：使用正交投影，确保地图上不同位置的单位图标大小一致，避免透视导致远处单位缩小而影响信息读取。

**深度精度优化——Reversed-Z 技术**：现代渲染器（如 Unreal Engine 4.22 之后默认开启）将深度缓冲反转，让 near 平面对应 z=1、far 平面对应 z=0。配合浮点深度缓冲，这一技术显著改善了透视投影远处的 z-fighting 问题，因为浮点数在 `[0,1]` 区间的精度集中分布在接近 0 的一侧，正好补偿了透视投影 z 值的非线性压缩。

**UI 渲染**：游戏中的 HUD（抬头显示）元素通常使用正交投影，near=-1、far=1，将屏幕坐标直接映射为 NDC，绕过透视计算。

## 常见误区

**误区一：透视除法是投影矩阵完成的**

投影矩阵本身**不执行**透视除法，它只是将顶点从观察空间变换到裁剪空间，输出的是齐次坐标 `(xw, yw, zw, w)`。真正的除法 `(x/w, y/w, z/w)` 由 GPU 固定管线在裁剪之后自动完成，称为"透视除法"阶段。初学者经常混淆这两个步骤，误以为乘以投影矩阵后就直接得到了 NDC 坐标。

**误区二：正交投影没有 near/far 裁剪面**

正交投影同样有 near 和 far 参数，它们决定了正交盒在深度方向的范围。如果将 near 设为负值（如 Unity 2D 默认的 near=-10），超出此范围的物体同样会被裁剪。只是正交投影的 near 可以设置为负数（物体在相机"后方"也能显示），而透视投影的 near 必须为正值，否则矩阵中的 `2fn/(f-n)` 项会产生 z 值符号错误。

**误区三：增大 far 平面可以无限延伸可见范围**

将 far 设置为极大值（如 1,000,000）在透视投影中会导致严重的深度精度损失。以 16 位深度缓冲为例，当 `f/n = 10000` 时，50% 的深度值被分配给距相机 near 到 `2×near` 的极小范围内，远处物体的深度几乎无法区分，产生 z-fighting。正确做法是将 far 设置为场景实际所需的最小值，或改用 Reversed-Z 配合 32 位浮点深度缓冲。

## 知识关联

投影变换的输入是**顶点变换**（模型→世界→观察空间）流程的终点——顶点在观察空间中已经以相机为原点、-z 轴为朝向完成了定位，投影矩阵正是作用于这一坐标系下的顶点。理解观察矩阵（View Matrix）中相机的坐标系定义（右手系 vs 左手系）直接影响透视矩阵第四行符号的选取。

投影变换的输出是**视口变换**的前驱步骤。裁剪空间坐标经透视除法得到 NDC 后，视口变换将 NDC 的 `[-1,1]²` 平面线性映射到屏幕的像素坐标范围 `[0, width] × [0, height]`。视口变换本身是简单的线性缩放平移，而投影变换才是决定场景"看起来像什么"的关键数学步骤。两者共同构成从 3D 空间到 2D 屏幕的完整坐标变换链的最后两环。
