# 状态排序

## 概述

状态排序（State Sorting）是渲染优化中的一种技术，通过在提交绘制命令前对渲染对象按照材质、着色器、纹理或深度等属性进行分组排序，从而将相同渲染状态的物体集中在一起连续绘制，最小化GPU管线中昂贵的状态切换次数。每一次状态切换（State Change）都会打断GPU的批处理流水线，迫使驱动程序重新配置管线状态对象（PSO），这一操作在OpenGL和Direct3D 11中可能消耗数十微秒的CPU时间。

这一技术的系统性应用可追溯到2000年代初期的游戏引擎设计。Doom 3（2004年）的渲染架构明确将材质排序作为渲染路径的核心步骤，而后续的虚幻引擎3/4中引入了更精细的多维键值排序体系。随着Draw Call数量在现代3A游戏中动辄超过数千次，状态排序的收益愈加显著。

状态排序的必要性源于GPU驱动的工作方式：在未排序的情况下，100个物体若交错使用10种材质，可能产生多达99次材质切换；而经过材质优先排序后，相同的100个物体只需9次材质切换。这种减少幅度在实际场景中能够将CPU渲染线程的状态提交开销降低30%到70%。

## 核心原理

### 排序键（Sort Key）的设计

状态排序的实现依赖于为每个渲染对象分配一个64位或128位的排序键（Sort Key），将所有影响状态切换代价的属性编码进这个整数值中。典型的Sort Key布局如下：

```
[渲染层级 4位][着色器ID 16位][材质ID 16位][纹理ID 16位][深度 12位]
```

高位比特对应切换代价最高的状态（如着色器程序切换）。对所有渲染对象按此64位整数做一次基数排序（Radix Sort），时间复杂度为O(n)，远优于基于比较的O(n log n)排序，且对CPU缓存友好。

### 切换代价的层次结构

不同的GPU状态切换代价差异悬殊，状态排序的优先级设计必须反映这一层次：

- **着色器/PSO切换**：代价最高，在D3D12/Vulkan中切换Pipeline State Object可能引发数毫秒的停顿
- **纹理绑定切换**：代价中等，尤其是跨纹理集（Texture Atlas之外）的切换会引起缓存失效
- **常量缓冲区（Constant Buffer）更新**：代价较低，通过动态偏移可进一步摊薄
- **顶点/索引缓冲区切换**：代价最低，现代API中接近零开销

基于此，排序键的高位应优先编码着色器ID，其次是纹理集ID，最后才考虑深度顺序。

### 深度排序与材质排序的权衡

深度排序（Depth Sorting）和材质排序（Material Sorting）之间存在根本矛盾：

- **不透明物体**应优先按材质排序，再在相同材质内按从前到后（Front-to-Back）的深度顺序排列。从前到后排列能触发Early-Z剔除，减少过度绘制（Overdraw），两者叠加效果最佳。
- **半透明物体**必须严格按从后到前（Back-to-Front）的深度顺序排列（Painter's Algorithm），否则混合结果错误，材质批处理在此类型下几乎无法实施。

这意味着实际引擎通常维护**两条独立的排序队列**：一条给不透明物体（材质优先），一条给半透明物体（深度优先），分阶段提交渲染。

## 实际应用

**场景中的批量角色渲染**：在一个有200个NPC的场景中，若每个NPC使用3种材质（皮肤、衣物、装备），未排序时可能产生约600次材质切换。经过材质排序后，相同材质的所有NPC皮肤部件连续绘制，切换次数降至约2次（3种材质之间的2次切换），在此基础上配合GPU Instancing可进一步合并为3次Draw Call。

**虚幻引擎的排序实现**：UE4的`FMeshDrawCommand`系统使用一个`uint64`类型的`SortKey`，将静态绘制命令预排序并缓存，避免每帧重新排序。对于静态场景几何体，这个预排序结果可以跨帧复用，进一步节省每帧的CPU排序时间。

**移动端渲染的特别收益**：在iOS/Android的基于瓦片的延迟渲染架构（TBDR）中，状态切换会打断瓦片的合并流程，导致中间渲染目标被迫刷新到主存（Framebuffer Store），代价极高。按材质排序后减少瓦片内状态切换，在Mali和Apple GPU上实测帧时间可下降10%-20%。

## 常见误区

**误区一：深度排序对不透明物体始终是最优策略。** 许多初学者认为从前到后的深度排序是通用最优解。然而对不透明物体，若按深度排序导致材质频繁切换，其CPU状态切换开销可能远超Early-Z节省的GPU填充时间。在材质种类多、几何体轻量的场景中，材质优先排序的收益显著高于单纯的深度排序。正确做法是在材质分组内再进行深度子排序。

**误区二：状态排序与Instancing等价，可以互相替代。** 状态排序将相同材质的物体聚集在一起，是实施GPU Instancing的**前提条件**而非替代品。Instancing要求物体共享同一着色器、同一网格、同一材质参数；而状态排序仅要求共享材质类型，排序后的物体仍可能因网格不同而无法Instancing。两者解决的是不同层级的问题。

**误区三：排序开销可以忽略不计。** 当场景有10,000个渲染对象时，对10,000个64位整数进行排序本身也有成本。若每帧所有对象的Sort Key都是动态生成的，且未使用Radix Sort而是使用`std::sort`（O(n log n)），在CPU主线程上可能消耗0.5ms以上。解决方案是将静态物体的排序结果持久化缓存，仅对移动物体（通常占总数的5%-20%）执行增量重排。

## 知识关联

状态排序建立在**Draw Call优化**的理论基础之上：Draw Call优化揭示了CPU向GPU提交命令的瓶颈所在，而状态排序正是通过减少每次提交之间的状态差异来降低这一瓶颈的具体手段。掌握Draw Call的CPU端开销模型（尤其是驱动程序验证状态的代价）是理解为什么排序优先级要与切换代价对齐的前提。

在掌握状态排序之后，可自然延伸到**Render Queue设计**（引擎级别的多优先级渲染队列实现）、**Bindless Rendering**（D3D12/Vulkan中通过将所有纹理绑定为一个大数组来从根本上消除纹理切换，使纹理维度的状态排序变得不再必要）以及**Indirect Draw**（通过GPU驱动的绘制命令完全绕过CPU排序瓶颈）。这三个方向都是在状态排序概念已熟练掌握后，针对其剩余局限性提出的进阶解决方案。
