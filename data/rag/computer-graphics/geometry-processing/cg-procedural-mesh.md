# 程序化网格

## 概述

程序化网格（Procedural Mesh）是指通过算法在运行时动态生成或修改三角形/多边形网格数据的技术方案，而非依赖预先由美术人员手工制作并存储在磁盘上的静态资产。其核心数据结构是顶点缓冲区（Vertex Buffer）和索引缓冲区（Index Buffer），程序在运行时直接向这两个缓冲区写入坐标、法线、UV等属性，随后提交给GPU渲染。

程序化网格的概念可追溯至1980年代的分形地形算法。Ken Musgrave 在1989年的博士论文中系统描述了用噪声函数生成地形网格的方法，此后 Minecraft（2011年发布）将基于区块的程序化网格生成带入主流大众视野。Unity 引擎在2010年前后开放了 `Mesh` 类的运行时写入接口，使程序化网格在游戏开发中成为常规工具。

程序化网格的核心价值在于它能以极低的存储成本表达海量几何变体。一个存储容量仅数KB的噪声参数组合，可在运行时展开为包含数百万顶点的地形网格；而同样质量的手工网格资产可能占用数百MB磁盘空间。这使得无限地形、可破坏环境和实时几何变形成为可能。

## 核心原理

### 顶点与索引的运行时构建

程序化网格的最小实现单位是一个函数，该函数输出两张表：顶点表（每行包含位置 `(x, y, z)`、法线 `(nx, ny, nz)`、纹理坐标 `(u, v)`）和索引表（每三个整数描述一个三角面）。对于一个 N×N 的平面网格，顶点数为 N²，三角面数为 2×(N−1)²，索引数组长度为 6×(N−1)²。例如一块 256×256 的地形块包含 65536 个顶点和 390150 个三角形。

在 Unity 中，运行时赋值的关键调用顺序为：先写入 `mesh.vertices`，再写入 `mesh.triangles`，最后调用 `mesh.RecalculateNormals()` 或手动计算法线。若顺序颠倒（先写索引），引擎会因越界索引抛出异常，这是初学者最常见的崩溃点。

### 噪声函数与高度图映射

地形类程序化网格最常用的高度函数是分形布朗运动（fBm），其公式为：

$$H(x,z) = \sum_{i=0}^{n-1} \text{amplitude}^i \cdot \text{noise}(\text{frequency}^i \cdot x,\ \text{frequency}^i \cdot z)$$

其中 `amplitude`（持续度）通常取 0.5，`frequency`（频率倍增）通常取 2.0，叠加层数 n 一般为 6~8 层。每个顶点的 y 坐标由 H(x,z) 直接决定，x 和 z 坐标则由网格索引线性推导。法线向量通过对相邻顶点做叉积计算得到，而非解析梯度，以确保顶点共享时法线方向一致。

### 网格的动态修改与脏标记

程序化网格不仅能生成，还能在每帧或特定事件触发时修改已有网格。Unity 的 `Mesh` 类维护一个内部脏标记（dirty flag），仅当顶点数据被修改后才将新数据上传至 GPU，避免每帧全量拷贝。对于顶点数量固定但坐标变化的场景（如布料模拟、波浪网格），只需更新 `mesh.vertices` 而无需重建索引表，GPU 端的 VAO/VBO 可保持复用，减少驱动层开销。

Unreal Engine 的 `ProceduralMeshComponent` 提供了 `CreateMeshSection` 与 `UpdateMeshSection` 两套接口，前者分配新的 Section（含索引表），后者仅更新顶点位置和法线，二者的 CPU→GPU 带宽消耗相差约一个数量级。

## 实际应用

**游戏中的无缝地形**：《No Man's Sky》在星球表面使用行进立方体（Marching Cubes）算法，以玩家位置为中心按 LOD 层级动态生成若干网格区块。距离玩家最近的区块顶点密度最高，远处区块顶点减少，所有区块均在玩家移动时实时重算和替换，实现表观无限的地表网格。

**可破坏物体**：实时切割算法（如 BSP 切分）在玩家击碎墙壁时将原始网格沿切面分裂为两个子网格，并对新暴露的截面填充封口三角形。整个过程需在单帧内（约16ms）完成顶点重分配和索引重建。

**UI 与图表渲染**：Unity UI Toolkit 和 UGUI 均允许开发者实现 `OnPopulateMesh` 接口，向 VertexHelper 写入自定义四边形或多边形，绘制折线图、雷达图等不规则形状，而无需额外贴图资产。

## 常见误区

**误区一：修改顶点后法线自动正确**。很多初学者修改顶点坐标后仅调用 `RecalculateNormals()`，却忽略该函数对硬边的处理存在阈值参数（Unity 默认为60°，低于此角度的相邻面会共享平滑法线）。对于程序生成的悬崖等尖锐地形，若不手动拆分顶点或调整阈值，法线会出现错误的插值，导致光照在边缘处产生漏光现象。

**误区二：每帧重建网格等同于每帧重新 new Mesh()**。反复 `new Mesh()` 会在堆上产生大量需要 GC 回收的托管对象，在 Unity 中会触发 GC Spike，造成帧率抖动。正确做法是在初始化阶段创建一个 Mesh 实例并缓存，后续仅替换其 `vertices` 数组，重用同一个对象。

**误区三：程序化网格可以完全替代手工建模**。程序化网格擅长处理规律性几何（地形、建筑外立面、管道网络），但对于需要艺术精确控制的角色面部或道具，算法难以复现美术设计意图。实际项目中常见的做法是将手工低模与程序化细节层叠加：骨骼动画驱动手工网格，程序化网格负责实时生成草丛、碎石等环境细节。

## 知识关联

程序化网格直接建立在**几何处理概述**中三角网格表示法（半边结构、索引列表）的概念之上，理解顶点共享和索引寻址是编写任何程序化网格代码的前提。顶点坐标与法线的计算依赖基础线性代数中的叉积运算：法线 $\mathbf{n} = (\mathbf{v_1} - \mathbf{v_0}) \times (\mathbf{v_2} - \mathbf{v_0})$，这是几何处理课程的早期内容。

在图形渲染管线方向，程序化网格生成的顶点缓冲区直接送入顶点着色器，因此理解 VAO/VBO 的 GPU 内存模型有助于优化上传频率。在几何算法方向，行进立方体（Marching Cubes）、Delaunay 三角剖分等更复杂的网格生成算法均以本文介绍的运行时顶点/索引填充机制为实现基础；掌握程序化网格的基本接口之后，可自然过渡到学习这些高级空间重建算法。
