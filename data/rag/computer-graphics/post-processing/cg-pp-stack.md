# 后处理栈设计

## 概述

后处理栈（Post-Processing Stack）是指将多个屏幕空间效果按照特定顺序串联执行的管线架构。每一帧渲染完成后，GPU 依次对全屏纹理应用色调映射、景深、运动模糊、抗锯齿等效果，最终输出到显示缓冲区。栈的"顺序"不是可以随意调换的——错误的排列会导致视觉错误或性能浪费。

该概念随着现代游戏引擎的成熟而系统化。Unity 在 2017 年发布的 Post Processing Stack v2 首次将效果顺序、资源预算、体积混合整合为统一框架，迫使开发者显式声明每个效果在栈中的位置。虚幻引擎 4 则通过材质域（Material Domain）中的"Post Process"类型允许自定义效果插入标准栈的任意阶段。这一设计思想确立了后处理不再是"叠加滤镜"，而是需要工程化管理的渲染子系统。

后处理栈设计的核心挑战有两个：**依赖排序**（某些效果必须在另一效果之后运行才能产生正确结果）和**性能预算管理**（多个全屏 Pass 叠加时显卡带宽和着色器时间成本急剧上升）。忽视这两点会导致在 1080p 分辨率下仅后处理阶段就消耗超过 4ms 帧时间，直接压垮 60fps 目标。

## 核心原理

### 效果依赖排序规则

后处理栈中存在若干**硬性顺序约束**，违反会产生不可接受的视觉错误：

1. **色调映射（Tonemapping）必须最后执行**。色调映射将 HDR 线性光照值压缩到 \[0, 1\] 的 LDR 范围，公式如 ACES 映射 $f(x) = \frac{x(ax+b)}{x(cx+d)+e}$（其中 $a=2.51, b=0.03, c=2.43, d=0.59, e=0.14$）。若在色调映射之后再叠加 Bloom，Bloom 的高光扩散会基于已被压缩的 LDR 值计算，导致高光区域发光强度严重不足。

2. **景深（Depth of Field）必须在运动模糊之前**。景深模糊依赖深度缓冲区中每个像素的场景深度值；运动模糊会将多个像素的颜色混合，破坏深度边界的清晰判断，若顺序颠倒，运动中的物体边缘会出现景深与运动模糊的错误叠加伪影。

3. **屏幕空间环境光遮蔽（SSAO）必须在光照合并之后、色调映射之前**。SSAO 在线性 HDR 空间中将遮蔽因子乘以已合并光照，若在 LDR 空间计算，乘法会导致暗部细节丢失并产生带状伪影。

4. **时域抗锯齿（TAA）通常紧接在几何渲染完成后、Bloom 之前执行**。TAA 的历史帧混合需要基于原始几何锯齿，若先经过 Bloom 的高斯模糊，历史帧权重计算会因模糊边界而产生鬼影（ghosting）。

### 性能预算模型

后处理栈的帧时间开销遵循以下模型。设分辨率为 $W \times H$，每个效果消耗 $t_i$ 毫秒，则总后处理开销为：

$$T_{total} = \sum_{i=1}^{n} t_i \cdot k_i$$

其中 $k_i$ 是该效果的 Pass 数量（例如双通道高斯模糊 $k=2$，景深需要 CoC 计算 + 散景模糊 + 合成共 $k=3$）。在 1920×1080 分辨率下，典型效果的参考开销：Bloom 约 1.2ms（6次降采样+上采样），SSAO 约 0.8ms，景深约 1.5ms，色调映射约 0.3ms。一旦总和超过目标帧预算（60fps 对应 16.67ms）的 25%（约 4ms），需要启动效果剔除或降分辨率策略。

### 半分辨率渲染与重建

为控制性能，后处理栈设计中广泛使用**降分辨率执行**策略：将 Bloom、景深、运动模糊等带宽密集型效果在半分辨率（1/2 宽高，即 1/4 像素数）下运算，再双线性上采样回全分辨率。该策略将这类效果的像素着色器调用量降低至原来的 25%。然而景深的半分辨率方案会在前景遮挡边缘产生"泄露"（bleeding）伪影，因此边缘重建 Pass 必须在全分辨率下进行，这是栈中必须单独保留一个全分辨率 Pass 的典型案例。

## 实际应用

**游戏主机场景：动态效果分级**

在 PS5/Xbox Series X 上面对 4K60 目标时，后处理栈通常按优先级分三档管理：
- **必要效果**（始终开启）：色调映射、TAA，合计约 0.6ms
- **质量效果**（性能充裕时开启）：Bloom、景深，合计约 2.5ms  
- **奢侈效果**（仅画质模式）：SSAO、屏幕空间反射后处理合成，合计约 1.8ms

当 GPU 帧时间超过 14ms 时，引擎自动剔除第三档效果，确保总预算不破。

**移动端场景：Pass 合并优化**

移动端 GPU（如 Adreno 740）对 Framebuffer Fetch 特性支持良好，可将色调映射与颜色分级（Color Grading）合并为单一 Pass，将两次全屏读写降为一次。Unity URP 在 2021.2 版本起将这一合并设为移动端默认行为，消除了一次完整的帧缓冲区往返（round-trip），节省约 0.4ms。

**编辑器工具：依赖图可视化**

Unreal Engine 5 的后处理调试视图（`r.PostProcessAAQuality` 命令配合 `VisualizePostProcessing`）可以将整个后处理栈渲染为有向无环图（DAG），每个节点显示该效果的实际 GPU 时间，开发者可以直观看到哪些效果是瓶颈，哪些依赖关系被系统自动强制执行。

## 常见误区

**误区1：以为后处理效果可以任意叠加而不影响结果**

许多初学者认为后处理只是"滤镜叠加"，调整顺序不影响最终画面。实际上，仅调换 Bloom 与色调映射的顺序，同一场景中太阳附近区域的发光扩散强度会相差 3 到 5 倍，因为 Bloom 在 HDR 线性空间的高光采样值可达 10 以上，而在 LDR 空间被压缩到 1 以下，扩散能量损失巨大。

**误区2：认为关闭某个效果一定能线性回收其帧时间**

后处理栈中，关闭景深并不意味着节省整整 1.5ms。许多效果共用同一个降采样纹理（Half-Resolution Buffer），景深和 Bloom 可能并行写入该纹理的不同通道。关闭其中一个效果后，若引擎不重新调度 Pass 依赖，共用纹理的读取 Pass 仍会存在，实际节省时间可能只有 0.6ms。这要求后处理栈设计必须支持**动态 Pass 图重建**，而不是简单的开关标志。

**误区3：半分辨率效果不需要考虑原生分辨率的 TAA 历史缓冲**

若在 TAA 执行之前将场景缩小到半分辨率进行 Bloom 计算，再上采样叠加到 TAA 输出上，TAA 的历史帧混合将看到每帧Bloom轮廓都有微小偏移（因为半分辨率上采样引入的亚像素位移），导致高亮边缘出现一圈 1~2 像素宽的闪烁。正确做法是 Bloom 必须在 TAA 完成并稳定之后执行。

## 知识关联

后处理栈设计直接建立在**后处理体积（Post-Processing Volume）**概念之上：体积系统负责定义"何时何地"激活某个效果及其参数混合权重，而栈设计解决的是"以何种顺序、以何种资源消耗"来执行这些被激活的效果。体积的混合结果（各效果参数的插值值）是栈的输入数据，栈的 Pass 调度是体积参数的执行机制，两者分别负责逻辑层和执行层。

在技术实现层面，后处理栈设计与**渲染管线（Render Pipeline）架构**紧密耦合：延迟渲染（Deferred Rendering）中深度缓冲和 G-Buffer 始终在后处理阶段可用，而前向渲染（Forward Rendering）中 SSAO 的输入数据需要额外 Pre-Pass 生成，这直接改变了栈中 SSAO 的成本和可插入位置。理解后处理栈设计是优化完整帧渲染预算的前置步骤，也是评估不同
