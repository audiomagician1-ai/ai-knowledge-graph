# 屏幕空间反射

## 概述

屏幕空间反射（Screen-Space Reflection，SSR）是一种利用已渲染帧的深度缓冲和颜色缓冲来计算反射效果的实时图形后处理技术。与传统的平面反射或立方体贴图反射不同，SSR 能够反映出场景中动态物体、角色和粒子效果，因为其采样来源是当前帧实际渲染的像素内容，而非预烘焙的静态数据。

SSR 最早在 2011 年由 Morgan McGuire 和 Michael Mara 在其论文 *Efficient GPU Screen-Space Ray Tracing* 中系统化地提出，随后在 Ubisoft、Epic Games 等公司的商业引擎中得到广泛应用。虚幻引擎4 在 2014 年将 SSR 作为标准后处理选项正式引入，推动了该技术在次世代游戏开发中的普及。

SSR 之所以重要，在于它以极低的额外渲染Pass代价补全了延迟渲染管线中金属和高光材质的视觉完整性。地板反射、水面倒影、湿润石材等效果均依赖 SSR 才能在不使用光线追踪硬件的条件下实现可信的实时结果。

## 核心原理

### 屏幕空间光线步进（Ray Marching in Screen Space）

SSR 的基础算法是从着色点沿反射向量在屏幕空间内逐步推进。反射向量 `R = reflect(-V, N)`，其中 `V` 为视线方向，`N` 为法线方向。将反射光线从观察空间变换到裁剪空间后，沿像素坐标方向迭代步进，每步采样当前深度缓冲值与步进点深度进行比较，若步进点深度大于缓冲深度（即射线穿入几何体），则认为在该处发生相交，并读取对应颜色缓冲作为反射颜色。

线性步进的致命缺陷是步长均匀，每帧需要 32～64 次采样才能保证精度，在 1080p 分辨率下性能开销极大。

### Hi-Z 加速光线步进（Hierarchical-Z Ray March）

Hi-Z Ray March 是 SSR 的核心优化手段，由 Yasin Uludag 在 GPU Pro 5（2014年）中详细描述。其核心思路是为深度缓冲构建一个多层级的深度最小/最大金字塔（Hierarchical Depth Buffer，Hi-Z），结构类似 MIP Map，但每一层存储的是对应区域像素的**最大深度值**（最远深度）。

光线步进时采用"二分/降级"策略：
1. **从高层级（粗粒度）开始**，快速跳过大片无交叉区域；
2. 若当前层级的最大深度小于射线深度，说明该区域无遮挡，直接跳跃 `2^level` 个像素；
3. 若存在潜在交叉，则降回更低层级（更细粒度）精确检测。

这使采样次数从线性步进的 64 次降低到通常只需 **16～24 次**，性能提升约 3 倍，特别是在反射光线接近水平方向（掠射角）时效果最为显著。

Hi-Z 层级数量通常取 `log2(max(width, height))`，对于 1920×1080 的 G-Buffer，大约需要 **11 层**金字塔。

### 反射颜色采样与消隐

确定交叉点后，SSR 需要处理两个关键问题：

**边界消隐**：当反射光线超出屏幕边界或指向摄像机背面时，无法获得有效颜色，需要根据光线终点与屏幕边缘的距离进行 Fade 衰减处理，公式为：
```
fade = 1 - saturate((|uv - 0.5| * 2 - threshold) / (1 - threshold))
```
其中 `threshold` 通常取 0.8，即在屏幕边缘 20% 的区域开始淡出。

**粗糙度模糊**：SSR 反射结果需与材质粗糙度（Roughness）结合，常见做法是根据 roughness 值在 SSR 颜色与环境球谐/IBL 之间进行混合。当 roughness > 0.5 时通常完全回退到 IBL，因为 Hi-Z 步进无法高效模拟散焦模糊反射。

## 实际应用

**游戏中的地板反射**：《赛博朋克 2077》和《地平线：禁忌之西》均使用 SSR 渲染城市湿润地面的动态反射。与光线追踪反射相比，SSR 在中等配置显卡上（如 GTX 1080）可将帧时保持在 2ms 以内，而光追反射通常消耗 8～15ms。

**延迟渲染的集成方式**：SSR 作为后处理 Pass 在延迟渲染的 Lighting Pass 之后执行。它直接读取 G-Buffer 中的世界法线（通常为 RG16F 格式存储）和线性深度缓冲，以及 HDR 颜色缓冲（通常为 R11G11B10F 格式）。这种设计使 SSR 对几何复杂度完全不敏感，1000 个三角形的场景与 1000 万个三角形的场景 SSR 开销完全相同。

**水面实时倒影**：虚幻引擎的水体插件默认在水面材质中启用 SSR，水面法线由 Normal Map 驱动，Hi-Z 步进可以正确捕获水面上方动态角色和植被的倒影，这是平面反射贴图无法做到的。

## 常见误区

**误区一：SSR 能反射屏幕外的物体**

SSR 只能反射出现在当前帧颜色缓冲中的内容。若一个高反射率金属球旁边有一个离屏光源，SSR 无法重建该光源的反射，只会在反射区域出现黑色或回退到 IBL，这是 SSR 的本质局限而非实现缺陷。

**误区二：Hi-Z 层级越多越好**

Hi-Z 金字塔超过 `log2(分辨率)` 层后，额外层级对步进加速毫无帮助，反而增加 GPU 显存带宽和构建金字塔的计算开销。对于渲染分辨率为 960×540 的半分辨率 SSR（常见优化做法），金字塔只需 **10 层**。

**误区三：SSR 可以替代光线追踪反射**

SSR 无法处理掠射角超过约 75° 的反射光线（此时射线几乎平行于屏幕，步进距离极长），也无法反映被遮挡物体、自反射和透明物体后的内容。Nvidia 在 RTX 白皮书中指出，SSR 在高光面积超过屏幕 15% 时误差率显著上升，而混合光追方案（SSR + RTRT）才是当前的工程最优解。

## 知识关联

SSR 直接依赖**延迟渲染**提供的 G-Buffer 数据——世界法线缓冲用于计算反射向量，线性深度缓冲用于 Hi-Z 金字塔构建，HDR 颜色缓冲是反射颜色的来源。没有预先渲染完整的 G-Buffer，SSR 无法执行任何步进操作。正向渲染管线若要使用 SSR，需要额外渲染一个仅包含深度和法线的 Pre-Pass，成本比延迟渲染高约 20%。

在技术演进方向上，SSR 是理解**时序累积抗锯齿（TAA）辅助降噪**的重要场景——现代引擎（如 Unity HDRP）使用 TAA 将 SSR 的采样数降至每像素 **1 次射线**，依赖时域混合重建高质量反射，这使 Hi-Z SSR 的性能开销降至约 0.8ms（1080p，RTX 3060 级别硬件）。此外，SSR 的 Hi-Z 遍历思想直接影响了软件光线追踪中 DDGI 探针和 Lumen 系统的屏幕空间加速结构设计。
