# 幂等性设计

## 概述

幂等性（Idempotency）源自数学概念：若函数 f 满足 f(f(x)) = f(x)，则称 f 为幂等函数。在HTTP和API设计领域，幂等性意味着**同一请求无论执行一次还是多次，服务器端的最终状态完全相同**。例如，`DELETE /orders/123` 执行一次与执行十次的结果应当一致——订单123不存在于数据库中。

HTTP 1.1规范（RFC 7231，2014年发布）明确规定了各方法的幂等性：GET、HEAD、PUT、DELETE 是幂等方法，而 POST 不是。这一规范成为现代 RESTful API 设计的基础参考标准。在 AI 工程系统中，由于模型推理服务、向量数据库写入和特征存储更新都依赖网络调用，网络抖动引发的重试会导致重复操作，幂等性设计因此成为保障数据一致性的关键工程手段。

AI 推理系统每天处理数百万级别的请求，在 P99 延迟超时场景下，客户端会自动重试。若写入操作缺乏幂等性保障，一次用户点击"提交反馈"可能触发 3 次实际数据库写入，直接污染模型训练数据集，导致反馈标签分布偏移。

## 核心原理

### 幂等键（Idempotency Key）机制

幂等键是实现 POST 请求幂等性的核心手段。客户端在 HTTP Header 中携带一个唯一标识符，格式通常为 `Idempotency-Key: <UUID-v4>`，服务器将该键与请求结果存储在 Redis 或数据库的幂等记录表中。

处理流程分为三步：
1. **首次请求**：键不存在，执行业务逻辑，将键与响应结果以原子操作写入存储，设置过期时间（Stripe 采用 24 小时，OpenAI API 采用类似策略）。
2. **重复请求**：键已存在且业务已完成，直接返回缓存的原始响应，不再执行任何业务操作。
3. **并发请求**：键已存在但业务仍在处理中，返回 HTTP 409 Conflict，提示客户端稍后查询。

幂等键的存储可以采用 Redis `SET key value EX 86400 NX` 命令，其中 `NX` 保证只有在键不存在时才写入，天然防止并发重入。

### HTTP 方法的幂等性分类

| HTTP 方法 | 幂等性 | 安全性 | 典型场景 |
|-----------|--------|--------|----------|
| GET       | ✅     | ✅     | 查询模型推理结果 |
| PUT       | ✅     | ❌     | 更新特征向量（全量替换） |
| DELETE    | ✅     | ❌     | 删除训练样本 |
| POST      | ❌     | ❌     | 创建新任务、触发异步推理 |
| PATCH     | ❌     | ❌     | 增量更新模型配置 |

PUT 与 POST 的核心区别在于语义：PUT `/models/v2/config` 表示"将配置**设置为**此值"（幂等），而 POST `/models/config/update` 表示"**执行一次**配置更新操作"（非幂等）。在设计 AI 服务接口时，优先选择 PUT 来替代 PATCH 可以天然获得幂等保障。

### 数据库层的幂等性保障

API 层的幂等键只能防止重复执行业务逻辑，数据库层还需要独立的幂等设计。常见方案包括：

**唯一约束防重**：在训练数据入库表中，为 `(sample_id, version)` 联合字段添加 UNIQUE 索引。重复写入时捕获 `DuplicateKeyException`，直接返回成功而非报错，实现"写入至少一次，效果恰好一次"。

**乐观锁版本控制**：在向量数据库更新操作中使用条件写入。DynamoDB 的条件表达式 `ConditionExpression="version = :expected_version"` 确保只有当前版本匹配时才允许更新，多次提交相同版本号的请求，只有第一次成功。

**消息去重**：Kafka 消费者在处理模型训练任务消息时，将 `message_id` 存入 Redis Set，消费前检查是否已处理，去重窗口通常设置为消息最大保留时间的 1.5 倍。

## 实际应用

### AI 推理服务的重试安全设计

在 LLM API 调用中，客户端向 OpenAI 兼容接口发送请求时，若遇到 503 超时后重试，服务端必须区分"新请求"与"重试请求"。标准做法是在请求头加入 `X-Request-Id`，服务端以此作为幂等键，在 Redis 中缓存模型推理结果（考虑到 LLM 推理成本，缓存 TTL 通常设置为 5-30 分钟）。这样重试请求直接命中缓存，既保证了结果一致性，又节省了 GPU 算力。

### 向量数据库的 Upsert 操作

Pinecone、Milvus 等向量数据库原生支持 `upsert` 操作而非分开的 `insert` 和 `update`，这正是幂等性设计的体现。`upsert(id="doc_001", vector=[...])` 无论执行多少次，doc_001 对应的向量值始终是最后一次提交的值，完全符合幂等定义。在构建 RAG 系统的文档索引更新 Pipeline 时，全部使用 upsert 而非 insert，可以安全地支持整个 Pipeline 的任意步骤重跑。

### 支付回调的幂等处理

AI 应用接入第三方支付时，支付平台会在网络不稳定时多次发送 Webhook 通知。正确做法是以支付平台提供的 `payment_id` 作为幂等键，在数据库中创建状态机：`PENDING → PROCESSING → COMPLETED`，并用数据库事务保证"检查键是否存在"和"写入键"是原子操作，否则并发回调会导致余额被多次充值。

## 常见误区

### 误区一：GET 请求无需考虑幂等性问题

GET 请求本身是幂等且安全的，但错误地将**有副作用的操作实现为 GET 接口**会破坏这一特性。例如 `GET /inference?prompt=xxx&save=true` 在返回结果的同时将推理记录写入数据库，此时 GET 接口已不再是纯幂等的。爬虫、CDN 缓存和负载均衡器会随意重放 GET 请求，这种设计会导致数据库中出现大量重复训练记录。

### 误区二：幂等性等同于请求去重

幂等性要求**相同操作的最终状态一致**，而去重要求**只执行一次**。二者目标不同：对于"扣减用户配额"操作，去重确保只扣一次；而对于"将配额设置为100"操作，幂等性保证无论执行几次，最终配额都是100，不需要去重。混淆两者会导致错误地对天然幂等的 PUT 操作增加不必要的去重逻辑，浪费 Redis 存储资源。

### 误区三：幂等键存储失败时应继续执行业务

正确逻辑应当是：**幂等键写入成功才执行业务**，而非业务执行成功后再写键。若先执行业务再写键，当写键操作失败时，下一次重试仍会重复执行业务，完全丧失幂等保障。这要求幂等键的存储与业务执行的起点绑定，Redis `SET NX` 失败时直接返回已有响应，成功时才进入业务处理流程。

## 知识关联

**依赖 RESTful API 设计**：HTTP 方法语义（PUT vs POST、安全性 vs 幂等性）是幂等性设计的前置基础。在理解 RFC 7231 对各方法的规范描述后，才能正确判断哪些接口天然幂等、哪些需要主动添加幂等键机制。

**依赖分布式事务**：幂等键的"检查-写入"操作必须是原子的，这与分布式事务中的原子性要求直接相关。在跨服务的幂等场景（如订单服务调用库存服务），需要借助 Saga 模式或 TCC 事务将幂等性保障延伸到整条调用链路，而非仅在入口 API 层处理。单独在 API 网关层实现幂等键，而下游服务没有对应的幂等保障，会导致"API 层已去重，但数据库仍然双写"的问题。
