# API网关

## 概述

API网关（API Gateway）是部署在客户端与后端服务群之间的反向代理层，负责统一接收所有外部请求并将其路由至对应的微服务。与普通反向代理不同，API网关不仅转发流量，还在请求链路上执行身份认证、限流、协议转换、请求聚合等横切关注点逻辑，使各个微服务可以专注于业务逻辑而无需重复实现这些功能。

API网关的概念随微服务架构的普及而成熟。2013年Netflix开源其内部使用的Zuul网关，2016年AWS发布Amazon API Gateway托管服务，同年Kong（基于Nginx/OpenResty）发布开源版本，这些事件标志着API网关从架构模式变为可直接落地的成熟工具。在AI工程场景下，API网关承担着将多个模型推理服务（如不同版本的LLM端点）统一对外暴露的职责，同时提供Token级别的限流和成本追踪能力。

选择API网关而非直接暴露微服务端点，根本原因在于"单一入口"原则带来的可控性。所有流量必须经过网关，意味着安全策略、监控埋点、流量切分等横切逻辑只需在一处维护。如果有10个微服务各自实现JWT验证，任何一处漏洞或版本不一致都会造成安全风险；将验证逻辑集中到网关，则只需维护一份代码。

## 核心原理

### 请求路由与路径匹配

网关通过路由规则将入站请求映射到目标服务。路由规则通常基于**URL前缀**、**HTTP方法**、**请求头**的组合进行匹配。例如：`POST /v1/completions` 路由到推理服务A，`GET /v1/models` 路由到模型注册服务B。路由匹配采用优先级队列，规则越具体优先级越高（精确路径 > 前缀匹配 > 通配符）。Kong网关使用基于优先级整数的`priority`字段控制规则顺序，默认值为0，数字越大越先匹配。

路由还支持**金丝雀发布**：通过设置权重将5%的流量路由至新版模型服务，95%保持旧版，从而在生产环境中低风险验证新版本的推理质量和稳定性。

### 身份认证与授权

API网关在请求到达微服务之前完成身份校验，常见机制包括：

- **API Key认证**：客户端在请求头（通常为`X-API-Key`或`Authorization`）携带密钥，网关校验其有效性后附加用户身份信息到请求头再转发。
- **JWT（JSON Web Token）校验**：网关验证Token签名（RS256或HS256算法）及`exp`字段，无需每次请求查询数据库，实现无状态认证。JWT的标准载荷结构包含`iss`（签发方）、`sub`（主体）、`aud`（受众）三个关键字段，网关通常对这三个字段进行声明校验。
- **OAuth 2.0 Token内省**：网关调用授权服务器的`/introspect`端点验证访问令牌的有效性。

认证通过后，网关通过**注入请求头**（如`X-User-ID: 12345`）的方式将身份信息传递给下游服务，下游服务无需重新做认证，只需信任来自网关的头部信息。

### 限流与熔断

限流（Rate Limiting）防止单个客户端耗尽服务资源。API网关通常实现以下算法：

- **令牌桶（Token Bucket）**：桶容量为`B`，以速率`r`（令牌/秒）补充，每个请求消耗1个令牌。允许短时突发（最多`B`个并发请求），适合LLM推理这类响应时间不均匀的场景。
- **固定窗口**：每分钟/每小时重置计数器，实现简单但存在窗口边界处的流量突刺问题。
- **滑动窗口**：用Redis的Sorted Set存储请求时间戳，精确统计任意60秒内的请求数，比固定窗口更平滑但内存占用更高。

熔断（Circuit Breaker）是保护下游服务的补充机制：当目标服务连续失败次数超过阈值（如5秒内错误率 > 50%），网关将熔断器切换至**开路**状态，直接返回503而不再转发请求，等待冷却时间（如30秒）后进入**半开**状态探测恢复情况。

### 协议转换与请求聚合

网关可以将外部的HTTP/REST请求转换为内部服务使用的gRPC或消息队列消息。例如客户端发送`POST /analyze`，网关将其转换为gRPC的`AnalyzeService.Analyze()`调用，客户端无需安装Protobuf工具链。

**请求聚合（Request Aggregation）**是网关另一项重要能力：客户端一次HTTP请求，网关并行调用多个微服务并将结果合并后返回。这减少了客户端的网络往返次数（RTT），对移动端和低带宽场景尤为重要。

## 实际应用

**AI推理服务的统一入口**：在部署多个LLM服务（如GPT-4o、Claude 3.5、本地部署的Llama 3）时，通过API网关统一对外暴露`/v1/chat/completions`接口，不同模型映射到不同后端，客户端通过请求头`X-Model-Preference: cost-optimized`触发路由策略，网关根据策略选择最低成本的可用模型。

**Token消耗追踪与成本控制**：网关拦截推理服务的响应体，解析`usage.total_tokens`字段，累加至Redis中的用户配额计数器。当用户当月Token消耗超过100万时自动降级为更便宜的模型，或直接返回402状态码要求升级套餐。

**AB测试与模型版本灰度**：在不中断服务的情况下，对特定`user_id`哈希尾号为0-4的用户路由至新版Fine-tuned模型，收集效果指标后再决定是否全量切换，避免一次性替换带来的不可预测风险。

## 常见误区

**误区一：网关应该包含业务逻辑**

部分开发者将"用户是否有权访问某篇具体文档"这类业务层权限判断放在网关中。这是错误的。网关只应处理**传输层和基础设施层**的横切关注点（认证≠授权的粗粒度部分），业务级授权（如RBAC中的资源权限）应由各微服务自己判断。将业务逻辑放入网关会导致网关成为新的单体，任何业务变更都需重新部署网关。

**误区二：API网关和服务网格（Service Mesh）是同一件事**

两者解决不同层面的问题。API网关处理**南北向流量**（外部客户端到内部服务），关注的是外部访问控制和聚合。Service Mesh（如Istio、Linkerd）处理**东西向流量**（微服务之间的内部调用），通过Sidecar代理实现服务间的mTLS、重试、可观测性。生产级AI平台通常同时部署两者：网关负责对外，Mesh负责内部服务间通信。

**误区三：网关是高可用的天然保障**

网关本身也会成为单点故障。如果只部署一个网关实例，它的宕机意味着所有服务对外不可用。生产环境需要部署至少2个网关实例，并在前面配置负载均衡器（L4/L7 LB），同时将限流等有状态功能的存储（如Redis）也做高可用部署。

## 知识关联

学习API网关需要具备**微服务入门**的基础，理解为什么服务拆分后需要一个统一入口来管理通信复杂度——单体应用中函数调用变成了网络请求，横切关注点不能再靠面向切面编程（AOP）在同一进程内解决，这正是网关存在的动机。

从API网关出发可以延伸到**服务网格**的学习：网关解决了进入集群的流量管理，而服务网格解决集群内部服务间调用的可靠性和安全性，两者构成完整的流量治理体系。此外，深入研究网关的限流算法会引出**分布式计数器**与**Redis数据结构**的设计问题；研究网关的高可用部署会引出**Kubernetes Ingress Controller**（如Nginx Ingress、Traefik）与网关的层级关系——Ingress Controller是K8s生态中实现API网关功能的具体技术手段之一。
