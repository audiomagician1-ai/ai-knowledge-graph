# 设计限流器

## 概述

限流器（Rate Limiter）是一种控制请求速率的系统组件，用于限制客户端在特定时间窗口内能够发送的请求数量。例如，一个典型的 AI API 服务可能规定：每个用户每分钟最多调用 60 次，或每秒最多处理 100 个 token。当请求超出阈值时，限流器返回 HTTP 429（Too Many Requests）状态码，拒绝多余的请求。

限流器的设计起源于电信行业的流量整形（Traffic Shaping）技术，后来随着 Web API 的大规模商业化（约 2010 年代）被广泛引入分布式系统。在 AI 工程领域，OpenAI、Anthropic 等公司的 API 都采用了多维度限流策略，同时控制 RPM（每分钟请求数）和 TPM（每分钟 token 数），以保护后端 GPU 资源不被单一用户耗尽。

设计一个限流器需要权衡精度、内存消耗、分布式一致性和实现复杂度四个维度。不同的算法在这四个维度上表现各异，选型错误会直接导致要么过度限制合法用户，要么无法有效阻止突发流量冲击系统。

## 核心原理

### 主流限流算法对比

**固定窗口计数器（Fixed Window Counter）** 是最简单的实现，将时间轴切成等长的窗口（如每 60 秒一个窗口），在每个窗口内维护一个计数器。缺点明显：假设限制为每分钟 100 次，用户可以在第 0:59 发送 100 次、在第 1:01 再发送 100 次，实际在 2 秒内产生 200 次请求，形成边界突刺（Boundary Burst）。

**滑动窗口日志（Sliding Window Log）** 通过记录每次请求的精确时间戳解决边界突刺问题。每次请求到来时，删除所有早于 `当前时间 - 窗口大小` 的日志条目，然后统计剩余条目数。若数量小于阈值则放行并记录新时间戳。精度最高，但内存消耗与请求量成正比，在高并发场景下存储开销可达 GB 级别。

**令牌桶算法（Token Bucket）** 维护一个容量为 `capacity` 的桶，以固定速率 `rate`（如每秒 10 个 token）向桶中添加令牌，每次请求消耗一个或多个令牌。公式为：`available_tokens = min(capacity, last_tokens + (current_time - last_refill_time) × rate)`。令牌桶允许一定程度的突发流量（最多消耗桶内全部令牌），适合 AI 推理场景中偶发的批量请求。

**漏桶算法（Leaky Bucket）** 以恒定速率处理请求，超出队列容量的请求直接丢弃。与令牌桶的关键区别：漏桶输出速率绝对平滑，令牌桶允许瞬时突发。AI 推理服务通常选择令牌桶而非漏桶，因为用户的会话型请求天然具有突发性。

**滑动窗口计数器（Sliding Window Counter）** 是固定窗口与滑动日志的折中：使用两个相邻固定窗口的计数器加权计算估算值。公式：`rate ≈ prev_count × (1 - overlap_ratio) + curr_count`，其中 `overlap_ratio` 为当前时刻在当前窗口中所处的比例。内存仅需存储两个整数，误差通常在 1% 以内。

### 分布式限流的实现挑战

单机限流使用进程内内存即可，但分布式场景下多台服务器各自计数会导致实际放行量是限制值的 N 倍（N 为服务器数）。解决方案有三种：

1. **集中式存储**：使用 Redis 的原子操作（`INCR` + `EXPIRE`）或 Lua 脚本实现跨节点一致的计数。Redis 单节点每秒可处理约 10 万次 `INCR` 操作，但引入了网络往返延迟（通常 1-5ms）。

2. **一致性哈希路由**：将同一用户的所有请求路由到同一台服务器，避免分布式计数，但影响负载均衡效果。

3. **本地缓存 + 定期同步**：每台服务器维护本地计数，每秒向中心节点同步一次，允许短暂的过量请求（软限流）。

### 限流器的存储层设计

在 Redis 中实现令牌桶时，需存储三个字段：`last_tokens`（上次剩余令牌数）、`last_refill_time`（上次填充时间戳，精度需达到毫秒级）、`capacity`（桶容量）。使用 Redis Hash 结构存储，典型的 Key 格式为 `ratelimit:{user_id}:{resource_type}`，TTL 设置为窗口大小的 2 倍以自动清理过期数据。

## 实际应用

**AI API 网关的多维度限流**：GPT-4 API 同时执行 RPM 和 TPM 两个维度的限制，且两者使用不同的桶容量。工程实现上，每次请求先检查 RPM 令牌桶（消耗 1 个令牌），通过后再检查 TPM 令牌桶（消耗 `estimated_tokens` 个令牌）。由于请求处理前无法精确知道 token 数量，通常采用预估值 + 事后校正的方式：响应返回后，若实际 token 数与预估偏差超过 20%，则在下次请求时补扣差额。

**用户分级限流**：商业 AI 平台通常为不同付费等级设置不同阈值，如免费用户 10 RPM、付费用户 60 RPM、企业用户 600 RPM。实现时，限流器从用户认证 Token 中解析出 `tier` 字段，动态加载对应的 `capacity` 和 `rate` 配置，无需为每个等级维护独立的限流实例。

**IP 级别的防爬虫限流**：对未经认证的请求按 IP 地址进行限流，通常比用户级限制严格 10 倍。需注意 IPv6 和 NAT 场景：同一公司网络可能共享同一 IPv4 地址，此时应以 `/24` 子网为粒度而非单个 IP。

## 常见误区

**误区一：认为令牌桶和漏桶可以互换使用**。两种算法在突发处理上有根本性差异：令牌桶允许积累令牌并在短时间内消耗（如积累 10 秒的令牌后一次性发送 100 个请求），而漏桶的输出速率是绝对恒定的。在 AI 推理场景中错误使用漏桶，会导致用户的正常对话型请求（如连续发送几条消息）被频繁拒绝。

**误区二：认为 Redis 的原子自增操作能完整实现令牌桶**。`INCR` 操作只能实现固定窗口计数器，无法原生支持令牌桶的持续补充逻辑。令牌桶需要读取上次状态、计算经过时间、更新令牌数三步原子操作，必须使用 Redis Lua 脚本或 `WATCH`/`MULTI`/`EXEC` 事务来避免竞态条件（Race Condition）。若使用普通 `GET`+`SET` 实现，在并发场景下会导致限流失效，实测在 1000 QPS 下漏过的请求可达应限流量的 30%。

**误区三：忽略限流器自身的性能开销**。每次 API 请求都需要额外的一次 Redis 网络调用（约 1-5ms），在高频场景下这会成为瓶颈。若 API 本身处理时间只有 2ms，加入远程限流检查后总延迟可能增加 150%。解决方案包括：将限流检查与业务逻辑并行执行、使用本地内存缓存热点用户的令牌状态（TTL 设为 100ms）、或在 Nginx/Envoy 等网关层而非应用层实施限流。

## 知识关联

**与限流策略的关系**：限流策略定义了"限制什么、限制多少"的业务决策（如每个用户 60 RPM），而限流器是执行这些策略的技术实现。同一套限流策略可以用固定窗口、滑动窗口或令牌桶等不同算法实现，选择哪种算法取决于系统对突发流量的容忍程度和内存预算。

**与系统设计入门的关联**：限流器的分布式实现依赖 CAP 定理的权衡。强一致性限流（如 Redis 主从同步）会增加延迟并在主节点故障时影响可用性；最终一致性限流（本地计数 + 异步同步）牺牲短暂精度换取更高可用性。设计时需明确业务对"偶尔超量"的容忍程度。
