# 系统设计入门

## 概述

系统设计（System Design）是将一个模糊的需求转化为由多个协作组件构成的技术架构的过程，核心产出是一张能够承载特定规模流量、满足可用性与一致性要求的架构图及其配套决策文档。与算法题考察固定答案不同，系统设计没有唯一正确解，评估标准是候选人是否能在给定约束下做出有理有据的取舍（trade-off）。

系统设计作为独立面试环节出现，最早由 Google、Amazon 等公司在 2000 年代初引入，用于筛选能够设计互联网规模分布式系统的工程师。2018 年前后，随着《Designing Data-Intensive Applications》（Martin Kleppmann 著）和 Grokking System Design 课程的普及，系统设计面试准备逐渐体系化，形成了"估算 → 组件选型 → 深挖瓶颈"的标准解题框架。

对于 AI 工程师而言，系统设计入门尤为关键：一个机器学习模型本身可能只占整个 AI 产品工程量的 10%~20%，其余部分是特征存储、模型服务、数据管道和监控系统——这些全部属于系统设计范畴。不掌握系统设计基础，ML 模型就无法可靠地运行在生产环境中。

## 核心原理

### 1. 容量估算（Back-of-Envelope Estimation）

容量估算是系统设计的第一步，目的是将"1 亿用户"这类模糊描述转化为具体的 QPS（每秒查询数）、存储量和带宽需求，从而驱动后续的架构选择。

常用数量级记忆：1 次内存访问约 100 ns，1 次 SSD 顺序读约 1 μs，1 次磁盘寻道约 10 ms，跨大陆网络往返约 150 ms。这些数字的相差倍数决定了缓存层是否必要。

典型估算公式：**每日活跃用户（DAU）× 每用户平均请求数 ÷ 86400 = 平均 QPS**，峰值 QPS 通常取平均值的 2~5 倍。以 Twitter 为例，若 DAU = 1.5 亿，每用户每天读 100 条推文，则读 QPS ≈ 1.5×10⁸ × 100 ÷ 86400 ≈ 173,600，选型时需要能支撑约 20 万 QPS 读请求的缓存架构。

### 2. CAP 定理与一致性取舍

CAP 定理由 Eric Brewer 于 2000 年正式提出，2002 年 Gilbert 和 Lynch 给出数学证明：在网络分区（Partition）发生时，分布式系统只能在一致性（Consistency）和可用性（Availability）中择其一，不可兼得。

**实际含义**：选择 CP 的系统（如 HBase、Zookeeper）在分区期间会拒绝写入或返回错误，保证数据不出现冲突；选择 AP 的系统（如 Cassandra、DynamoDB 默认配置）在分区期间继续服务但可能返回旧数据。AI 推荐系统的特征存储通常选 AP，因为读到稍旧的用户特征比服务不可用造成的业务损失小得多。

### 3. 水平扩展 vs 垂直扩展

垂直扩展（Scale Up）是给单台服务器增加 CPU、内存或更快的磁盘；水平扩展（Scale Out）是增加服务器数量并通过负载均衡分发请求。单台服务器的硬件上限约为 128 核 CPU + 数 TB 内存（2024 年主流云主机规格），超过此限必须水平扩展。

水平扩展要求服务做到**无状态（Stateless）**：每个请求可以由集群中任意实例处理，Session 状态外移到 Redis 等中心化存储。这是系统设计入门中最重要的架构原则之一，也是后续讨论可扩展性的出发点。

### 4. 分层架构与单点故障消除

经典三层架构（展示层 → 逻辑层 → 数据层）中，每层都必须排查单点故障（SPOF，Single Point of Failure）。消除 SPOF 的标准手段是在每层引入冗余：负载均衡器部署主备两台，数据库配置主从复制（Master-Replica），CDN 多 PoP 节点就近提供静态资源。

可用性计算遵循串联乘法原则：若系统有三个串联组件，可用性分别为 99.9%、99.9%、99.9%，则整体可用性 = 0.999³ ≈ 99.7%，即每年约 26 小时宕机时间。互联网产品通常要求 SLA 达到 99.99%（每年停机 ≤ 52 分钟），这要求并联冗余组件而非串联堆叠。

## 实际应用

**设计 URL 短链接服务（入门级练习场景）**：这是系统设计入门中最经典的练习题。核心设计决策包括：① 使用 Base62 编码（字符集 0-9+a-z+A-Z）将长 URL 映射为 7 位短码，理论容量为 62⁷ ≈ 3.5 万亿条，可支撑百年级别的 URL 生成量；② 读写比约为 100:1（短链接被点击次数远多于创建次数），因此读路径需加 Redis 缓存层；③ 重定向使用 HTTP 301（永久重定向，浏览器缓存）还是 302（临时重定向，每次请求都回源）需根据是否需要统计点击量来决定。

**AI 模型推理服务的系统设计**：在 AI 工程场景中，模型推理服务需要解决的系统设计问题包括：模型权重（通常数 GB 到数百 GB）的加载与热更新、批推理（Batching）以提高 GPU 利用率、特征计算的在线/离线一致性。例如，将实时特征存入 Redis（延迟 < 1 ms）、批量特征存入 Cassandra（吞吐高），推理服务同时查询两个来源并合并，是业界常见的特征服务架构。

## 常见误区

**误区一：过早优化，上来就设计微服务**。系统设计入门要求先从单体架构（Monolith）出发，仅在识别出具体瓶颈后才拆分服务。Netflix 最初也是单体应用，2009 年才开始微服务化迁移，历时约 7 年完成。面试中直接画出 20 个微服务的候选人往往无法解释为什么要拆分以及拆分边界如何划定。

**误区二：混淆数据库的"复制"与"分片"**。复制（Replication）是同一份数据存多份副本，解决的是高可用和读扩展问题；分片（Sharding）是将数据按某个键拆分到不同节点，解决的是写扩展和存储上限问题。两者可以同时使用，但混淆后会导致方案无法解决实际瓶颈——例如用复制解决写热点问题是无效的，必须使用分片。

**误区三：忽略"信封背面估算"直接跳到组件选型**。没有容量数字作支撑的选型是不可信的。例如直接说"用 Kafka 做消息队列"，如果消息量只有每天 1000 条，用 Kafka 是严重过度工程；而如果是每秒 100 万条，Kafka 又几乎是唯一合理选项。容量估算决定了哪些组件是必须引入的，而不是可选的。

## 知识关联

**与前置概念的衔接**：RESTful API 设计提供了系统各组件之间接口契约的基础语言，系统设计中服务间通信的第一选择通常是 REST over HTTP；数据库基本概念（索引、事务、范式）是系统设计中选择关系型 vs. NoSQL 数据库、决定是否需要分片的直接依据。

**通向后续概念**：本文建立的"估算 → 分层 → 消除 SPOF → 扩展"框架是学习**可扩展性**的起点；URL 短链接系统的完整设计案例直接引出**设计短链接系统**专题；CAP 定理与状态管理的讨论为**设计聊天系统**中的消息一致性方案奠定理论基础；容量估算中的 QPS 分析是**设计限流器**的直接动因；无状态服务的原则与 **12-Factor App** 的第六条原则（进程）完全对应，系统设计实践与云原生应用规范在此汇合。
