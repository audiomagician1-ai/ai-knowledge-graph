# 熔断器模式

## 概述

熔断器模式（Circuit Breaker Pattern）由 Michael Nygard 在 2007 年出版的《Release It!》一书中系统阐述，是分布式系统中防止级联失败的关键防御机制。其名称来源于电气工程中的保险丝（断路器）——当电路过载时自动断开，保护整个电路系统不被烧毁。在软件架构中，熔断器同样充当"自动开关"，当检测到某个下游服务持续失败时，主动切断请求链路，避免故障在微服务网络中扩散蔓延。

熔断器模式在 AI 工程场景中尤为重要，因为 AI 推理服务（如大模型 API 调用）往往延迟高且不稳定。当 GPU 资源耗尽或模型服务超载时，每次请求可能耗时 30 秒以上才超时，若没有熔断保护，上游服务的线程池将被这些"僵尸请求"完全占满，最终导致整个调用链雪崩。Netflix 的 Hystrix 库是该模式最著名的工业级实现，其在 2012 年开源后推动了熔断器模式的广泛工程实践。

## 核心原理

### 三状态自动机

熔断器的本质是一个拥有三种状态的有限状态机：**关闭（Closed）**、**打开（Open）**、**半开（Half-Open）**。

- **关闭状态（Closed）**：正常工作状态，所有请求正常通过，熔断器在后台统计失败率。当失败率超过预设阈值（例如 50% 的失败率，或 10 秒窗口内 20 次失败）时，熔断器跳转至打开状态。
- **打开状态（Open）**：所有请求被立即拒绝，不再调用下游服务，直接返回预设的 fallback 响应。打开状态持续一段固定时间（通常配置为 5~60 秒，称为"休眠窗口"），让下游服务有时间恢复。
- **半开状态（Half-Open）**：休眠窗口结束后，熔断器允许少量"探针请求"（例如 5 个请求）通过，若这些请求成功率达标，熔断器回到关闭状态；若仍然失败，重新跳回打开状态。

### 统计窗口与阈值计算

熔断器必须在**统计精度**与**响应速度**之间取得平衡。常见实现采用滑动时间窗口（Sliding Window）或滑动计数窗口：

- **时间窗口**：过去 N 秒内的请求统计（如最近 10 秒）
- **计数窗口**：最近 N 次请求的统计（如最近 100 次）

触发熔断的条件通常是复合判断，例如 Resilience4j（Java 生态主流熔断库）默认配置为：最近 100 次请求中，失败率 ≥ 50% **且**请求次数达到最小触发量（minimumNumberOfCalls，默认 100），才会触发熔断。这一"最小触发量"设计防止在流量极低时（如仅 2 次请求中有 1 次失败就触发熔断）产生误判。

失败率计算公式：

```
failure_rate = (failed_calls / total_calls) × 100%
```

其中 `failed_calls` 包括异常返回和超时请求，`total_calls` 为统计窗口内总请求数。

### 慢调用率熔断

除了失败率，现代熔断器实现还支持**慢调用率（Slow Call Rate）熔断**。Resilience4j 允许配置 `slowCallDurationThreshold`（默认 60 秒）和 `slowCallRateThreshold`（默认 100%），当响应时间超过阈值的请求占比过高时同样触发熔断。这对 AI 推理场景尤为适用——大模型调用本身不一定"失败"，但响应延迟 45 秒的成功调用对上游服务来说同样是灾难。

## 实际应用

### AI 大模型 API 调用保护

在调用 OpenAI GPT-4 或内部 LLM 推理服务时，可配置如下熔断策略：设置超时阈值为 15 秒（slowCallDurationThreshold），慢调用率阈值为 30%，失败率阈值为 40%。当模型服务因 GPU 资源竞争导致响应变慢时，熔断器自动打开，fallback 逻辑切换至较轻量的 GPT-3.5 或本地缓存结果，保证用户侧体验不中断。

### 多模型路由中的熔断链

在 AI 系统中常见多级模型调用链：主模型 → 辅助模型 → 规则引擎。每一层都应配置独立的熔断器，而非共享同一个实例。若主模型熔断器打开，请求路由至辅助模型；辅助模型熔断器也打开时，降级至规则引擎。这种分层熔断架构需要配合服务注册中心（如 Consul 或 Nacos）动态更新熔断状态，避免全链路同时熔断导致服务完全不可用。

### Python 实现示例

使用 `pybreaker` 库可在 Python AI 服务中快速接入熔断器：

```python
import pybreaker

# 配置：连续5次失败后熔断，休眠窗口60秒
breaker = pybreaker.CircuitBreaker(fail_max=5, reset_timeout=60)

@breaker
def call_llm_api(prompt: str) -> str:
    return openai_client.chat(prompt)
```

当 `call_llm_api` 连续抛出 5 次异常后，后续调用直接抛出 `CircuitBreakerError`，无需等待超时。

## 常见误区

### 误区一：熔断器可以替代超时设置

熔断器依赖统计窗口的失败/慢调用数据来判断是否熔断，而**超时控制决定单次请求的最长等待时间**。如果不设置超时，单次慢请求会阻塞线程直到服务端主动关闭连接（可能长达数分钟），此时即便熔断器打开了，之前已经在等待的线程依然被占用。正确做法是**超时 + 熔断器同时配置**：超时保护单次请求，熔断器保护整体流量。

### 误区二：熔断器应该配置极低阈值以"快速响应"

将失败率阈值设为 5% 或最小触发量设为 10，会导致熔断器在短暂的网络抖动或极低流量波动时频繁误触发，进而引发不必要的服务降级，反而降低系统可用性。Netflix 在生产环境的 Hystrix 配置中，通常将统计窗口设置为 10 秒、最小请求量设为 20，失败率阈值不低于 50%，以过滤正常的随机失败噪声。

### 误区三：熔断器状态应该全局共享

在多实例部署的微服务中，若每个实例维护独立的本地熔断器状态，可能出现实例 A 已熔断而实例 B 仍在发送请求的不一致情况。但将熔断状态存入 Redis 等共享存储并非总是更好——集中式状态本身成为单点瓶颈，且网络延迟会导致状态同步滞后。业界通常的解法是：本地熔断器处理绝大多数场景，辅以服务网格（如 Istio）在网关层实现集群级熔断，两者分工协作。

## 知识关联

### 与微服务的关系

微服务架构使服务间网络调用从进程内方法调用变为跨网络 RPC，网络的不可靠性是熔断器存在的根本原因。在微服务入门阶段理解了服务拆分和服务注册后，熔断器是保障拆分后服务间调用稳定性的第一道防线，它处理的是"服务存在但不可用"的场景，与服务发现处理的"服务不存在"场景互补。

### 通往可观测性

熔断器的状态变化（Closed → Open → Half-Open）是分布式系统中极有价值的可观测性信号。每次熔断触发都意味着某个依赖出现了系统性问题，应当将熔断器的状态转换事件、当前失败率、拒绝请求数等指标通过 Prometheus 暴露，并在 Grafana 仪表盘中建立熔断事件告警。没有这些监控数据，熔断器在"默默保护"系统的同时，工程师会对下游服务的真实健康状态一无所知。

### 与优雅降级的协同

熔断器定义了"何时停止调用下游"，而优雅降级定义了"停止调用后返回什么"。两者必须配合设计：熔断器打开后触发的 fallback 逻辑，就是优雅降级策略的执行入口。若 fallback 逻辑本身也调用了另一个可能失败的服务，则该调用同样需要配置独立的熔断器，形成多层降级防护体系。
