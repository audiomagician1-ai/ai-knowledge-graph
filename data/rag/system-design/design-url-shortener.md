# 设计短链接系统

## 概述

短链接系统（URL Shortener）是将长URL映射为短字符串的Web服务，例如将 `https://www.example.com/very/long/path?param=value` 压缩为 `https://short.ly/aB3xK`。系统的核心挑战在于用6-8位字符（Base62编码下，6位可表示约568亿个唯一短链，即62⁶ ≈ 568亿）唯一标识一个长URL，同时在高并发读写下保证重定向延迟低于50毫秒。

该类系统最早由TinyURL在2002年商业化，2009年Twitter普及短链需求后，Bitly等服务的QPS峰值达到每秒数十万次重定向请求。短链接系统在现实工程中是高读低写比（读写比约100:1）、需要全球化部署的典型场景，其设计决策直接影响用户的点击跳转体验及广告追踪的准确性。

理解短链接系统的设计价值在于：它集中体现了哈希冲突处理、数据库选型、缓存策略、全球CDN分发等多个系统设计核心问题，是面试和实际工程中的经典参考案例。

## 核心原理

### 短码生成策略

**方案一：哈希截断法**
对长URL计算MD5或MurmurHash，取前6位字节并转换为Base62字符。Base62字符集为 `[0-9, a-z, A-Z]`，共62个字符。6位Base62可表示 62⁶ = 56,800,235,584 个不同短码。该方案无需数据库自增序列，但需要处理哈希冲突：当两个不同的长URL映射到相同短码时，可对原URL追加盐值（如拼接 `+1`）后重新哈希，直到无冲突。

**方案二：自增ID + Base62转换**
使用全局唯一自增ID（数据库AUTO_INCREMENT或分布式ID如Snowflake），将整数ID转换为Base62字符串。例如ID=12345 → Base62结果为 `dnh`（12345 = 3×62² + 32×62 + 35）。该方案无冲突且可预测，但暴露了业务量信息（短码可被遍历），且依赖全局计数器成为单点。

**方案三：随机字符串 + 去重校验**
生成6位随机Base62字符串，写入前查询数据库是否已存在该短码。在存储量达到容量30%前冲突概率低，但随着数据增长（超过约1亿条记录后）碰撞率上升，需要更长的短码。

### 数据存储设计

短链接系统的核心数据模型极简：

| 字段 | 类型 | 说明 |
|------|------|------|
| `short_code` | VARCHAR(8), PK | 短码，主键 |
| `long_url` | TEXT | 原始长URL |
| `user_id` | BIGINT | 创建用户 |
| `created_at` | TIMESTAMP | 创建时间 |
| `expire_at` | TIMESTAMP | 过期时间（可选） |

**数据库选型**：写入量低（每天约1亿次创建），读取量极高（每天约100亿次重定向查询）。可选方案：
- **关系型数据库（MySQL）**：通过 `short_code` 字段的B+树索引支持O(log n)查找，单表支撑数十亿条记录需要做水平分片（按short_code取模分16或32个分片）。
- **KV存储（Redis/Cassandra）**：short_code → long_url 天然契合键值模型，Redis单节点QPS可达10万+，适合用作读缓存层；Cassandra适合持久化存储，写放大系数约1.5。

### 重定向机制与缓存

用户访问短链时，服务器返回 **HTTP 301** 或 **HTTP 302** 状态码：
- **301（永久重定向）**：浏览器缓存该映射，后续请求直接跳转，不经过短链服务器，**降低服务器负载但无法统计点击数**。
- **302（临时重定向）**：每次点击都经过服务器，可实时统计并支持动态修改目标URL，**适合需要追踪分析的业务场景**（如广告投放）。

缓存层设计：在应用服务器与数据库之间加入Redis缓存，采用 **Cache-Aside** 模式，命中率通常超过90%（因为热门短链的访问遵循Zipf分布，前20%的短链占约80%的访问量）。设置TTL为24小时，过期后回源数据库。

### 系统容量估算

以中等规模系统为例进行估算（Back-of-Envelope Calculation）：
- 写入：每天1亿次新建 → QPS ≈ 100,000,000 / 86,400 ≈ 1,160 写QPS
- 读取（100:1读写比）：116,000 读QPS
- 存储：每条记录约500字节，10年数据量 = 1亿 × 365 × 10 × 500B ≈ **182.5 TB**
- 缓存：缓存当天20%的热门请求所需内存 = 1亿 × 0.2 × 500B ≈ **10 GB**

## 实际应用

**Bitly的架构实践**：Bitly使用Cassandra作为主存储，将short_code与long_url分别存储在两个列族中，支持双向查询（短码→长URL，以及长URL→短码，后者用于检测同一长URL是否已被创建过）。其前端使用Nginx处理重定向，将热门短链直接写入Nginx的共享内存字典，绕过应用层，实现微秒级响应。

**Twitter/t.co的安全需求**：t.co在重定向前会对目标URL进行恶意网站扫描，因此强制使用302临时重定向。系统为每个URL生成确定性短码（同一长URL始终生成同一个t.co链接），通过对长URL计算HMAC-SHA256后截取前40位并Base62编码实现。

**自定义短码功能**：允许用户指定短码（如 `short.ly/myproduct`）时，需在写入时做唯一性校验，并为自定义短码设置最小长度（通常≥4位）防止与系统生成的短码命名空间冲突。

## 常见误区

**误区一：使用MD5全长作为短码**
MD5输出32位十六进制字符，直接用作短码虽然冲突概率极低，但用户体验差且URL过长失去意义。正确做法是截取前6-8位Base62字符，并在写入数据库时做冲突检测，冲突时追加随机后缀重新生成。

**误区二：选用301重定向以减轻服务器压力**
在需要统计点击量、A/B测试链接效果或修改目标URL的场景中，301重定向会导致后续点击绕过服务器，使统计数据永久缺失，且目标URL修改后用户浏览器仍跳转到旧地址。商业短链服务几乎全部使用302。

**误区三：忽略短码冲突与数据库唯一索引配合**
仅在应用层检查冲突存在并发安全漏洞——两个请求同时检查同一短码均发现"不存在"，然后同时写入，导致主键冲突。正确方案是依赖数据库的唯一索引约束，在INSERT失败时（捕获 `Duplicate Key` 异常）由应用层重新生成短码重试，而非先SELECT再INSERT（违反"先检查后行动"的原子性原则）。

## 知识关联

**前置知识**：系统设计入门中的QPS估算方法、CAP定理和BASE理论是进行上述容量估算和数据库选型的基础。理解HTTP状态码（301 vs 302）和DNS解析流程有助于准确判断重定向行为对缓存和追踪的影响。

**延伸方向**：短链接系统是设计**粘贴板服务（Pastebin）**的近似问题——二者同样涉及随机短ID生成与KV存储，差异在于Pastebin存储文本内容（对象存储如S3更合适）而非URL映射。此外，短链接系统的分布式ID生成部分可深入延展至**Snowflake算法**（Twitter 2010年提出，使用41位时间戳+10位机器ID+12位序列号，每毫秒每节点生成4096个唯一ID），是分布式系统中无协调全局唯一ID的标准方案。
