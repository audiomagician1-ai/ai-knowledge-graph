# 数据隐私合规

## 概述

数据隐私合规是指游戏开发商在收集、存储和处理玩家个人数据时，必须遵守特定法律法规的要求。对于网络多人游戏而言，这意味着数据库设计从第一行表结构开始，就必须内嵌隐私保护逻辑，而非事后补丁。最典型的两部法规是欧盟的《通用数据保护条例》（GDPR，2018年5月25日正式生效）和美国的《儿童在线隐私保护法》（COPPA，1998年颁布，保护13岁以下儿童数据）。

游戏公司违规的代价极为具体：Google旗下YouTube因违反COPPA被罚款1.7亿美元（2019年），Epic Games因《堡垒之夜》违反COPPA被罚款2.75亿美元（2022年）。这些案例直接说明，游戏数据库若未按法规设计，法律风险会转化为真实的财务损失。

对游戏数据库设计师而言，隐私合规不是法务部门的工作，而是表设计、索引策略、数据保留期限和访问权限的技术实现问题。一个设计合理的玩家数据库，应当能够在收到用户请求后的30天内（GDPR要求），完整提取或彻底删除该用户的所有关联数据。

## 核心原理

### GDPR对游戏数据库的具体要求

GDPR第17条"被遗忘权"要求游戏数据库必须支持级联删除或匿名化处理。这意味着`player_id`不能作为跨表的硬编码外键而不配置删除策略。在数据库设计上，玩家账户表（`accounts`）删除一条记录时，必须通过`ON DELETE CASCADE`或`ON DELETE SET NULL`处理游戏日志、聊天记录、支付历史等关联表。

GDPR还要求"数据最小化"原则：数据库只存储实现游戏功能所必需的字段。例如，多人游戏的段位排行榜功能，只需存储`player_id`和`rating_score`，而非连同IP地址、设备型号一起存入同一张宽表。违反最小化原则的常见表现是把所有遥测数据塞入一张包含50+列的`player_events`表，其中大量列对核心业务从无查询。

### COPPA对未成年人数据的隔离设计

COPPA要求在收集13岁以下用户数据前必须获得家长可验证同意（Verifiable Parental Consent，VPC）。在数据库层面，这要求玩家表中必须存在`birth_date`或`age_verified`字段，并在插入记录时触发年龄检查逻辑。许多游戏采用独立的`child_accounts`表，该表禁止存储行为画像类字段（如`playtime_pattern`、`purchase_tendency`），并对广告ID字段强制置NULL。

技术实现上，常见方案是在`accounts`表中加入`is_minor TINYINT(1)`列，并在应用层和数据库层双重设置写入限制：当`is_minor = 1`时，相关的行为追踪表对该`player_id`的INSERT操作被存储过程或触发器拒绝。

### 数据保留期限与自动清理机制

GDPR第5条第1款(e)要求数据存储期限不得超过目的所必需的时间。游戏数据库需要为每类数据定义具体的TTL（Time To Live）：聊天日志通常保留90天，支付交易记录因财务法规需保留7年，会话日志（session logs）一般30天后自动清除。

实现方式是在相关表中添加`expires_at DATETIME`列，并设置定期运行的清理作业（如每日凌晨执行的MySQL EVENT或PostgreSQL的pg_cron任务），删除所有`expires_at < NOW()`的记录。这一机制必须在数据库设计阶段规划，否则后期改造涉及的外键约束和数据迁移成本极高。

## 实际应用

以一款全球发行的多人在线RPG为例，其玩家数据库通常包含以下合规设计决策：

**账户注册表**的`email`字段需要加密存储（如AES-256），而非明文VARCHAR，因为邮件地址属于GDPR定义的个人可识别信息（PII）。查询时通过应用层解密，数据库文件泄露时攻击者无法直接读取。

**游戏内聊天记录表**须在表定义中加入`retention_days INT DEFAULT 90`的元数据列，并配套自动清理脚本。如果游戏支持欧洲玩家，聊天数据不能无限期保留用于"改善游戏体验"，除非有明确的用户同意记录存档（`consent_log`表中有对应条目）。

**数据导出接口**（对应GDPR第20条数据可携带权）要求数据库设计时为每位玩家的数据建立清晰的归属关系。如果`player_id`分散在20张表中且没有统一外键设计，响应30天导出请求时工程师需要手写20条独立查询，极易遗漏，因此规范的外键设计本身就是合规的技术保障。

## 常见误区

**误区一：匿名化等同于删除`username`字段。** 真正的匿名化要求删除所有可以重新关联到特定个人的字段组合。仅删除用户名但保留`ip_address + device_id + birth_date`的组合，依然可以通过"准标识符"重新识别用户身份，GDPR仍然适用。正确做法是将所有PII字段替换为随机生成的哈希值，且哈希盐值不得保留在同一数据库中。

**误区二：COPPA只在美国服务器适用。** 只要游戏对美国儿童用户提供服务（即使服务器在欧洲），COPPA均适用。Epic Games案例中，Epic正是因为在全球服务器上未对识别为美国儿童的账户实施COPPA保护而被罚款。数据库设计时，`region`字段的写入逻辑必须与年龄验证逻辑联动，而非两套独立流程。

**误区三：加密存储PII后就不需要访问控制。** 加密是数据静态保护手段，但GDPR第25条"隐私设计"还要求限制数据库账户权限。DBA账户不应拥有读取加密PII的密钥权限，游戏逻辑服务账户只需SELECT特定列，而非全表SELECT权限。单靠加密而不做列级别权限隔离，仍属于不合规设计。

## 知识关联

学习数据隐私合规无需编程或数据库基础，但需要理解"个人可识别信息（PII）"这一基本概念：姓名、邮箱、IP地址、设备ID、生日均属于PII，而纯粹的游戏内数值（如角色等级、击杀数）通常不属于PII，除非与账户ID直接关联。

掌握数据隐私合规后，自然引出**数据归档**这一后续主题。归档策略必须在合规约束下设计：哪些数据在保留期结束后需要物理删除（DELETE），哪些可以脱敏后迁移到冷存储（anonymized archive），这一决策直接决定归档表的结构和清理脚本的逻辑。合规要求划定了归档的法律边界，归档技术则是执行这些边界的工程手段。
