# 邀请系统

## 概述

邀请系统（Invite System）是网络多人游戏大厅中用于在玩家之间发起组队或加入特定游戏房间请求的功能模块。具体来说，它负责处理"发送方向接收方传递一条可操作的邀请消息，接收方确认后双方进入同一游戏大厅或房间"这一完整流程。与聊天消息不同，邀请消息携带房间ID、游戏模式、剩余席位等结构化元数据，接收方点击"接受"后系统会自动执行加入逻辑，而不需要玩家手动查找并输入房间号。

邀请系统的雏形出现在2000年代初期的PC对战平台，如Battle.net（1997年上线）中的好友邀请频道，当时邀请本质上只是一条含有房间名的聊天指令 `/invite`。随着主机联机时代的到来，PlayStation Network（2006年）和Xbox Live（2002年）将邀请升级为平台级系统调用，使其脱离了具体游戏逻辑而成为操作系统层面的服务。如今邀请系统分为**游戏内邀请**和**平台级邀请**两种架构层次，两者在权限范围、触达能力和延迟处理方式上存在本质差异。

邀请系统对游戏留存率有直接影响。数据显示，通过好友邀请进入游戏的玩家首周回访率比自然流入玩家高出约40%，因此正确实现邀请流程不仅是功能需求，也是商业目标的重要支撑点。

## 核心原理

### 邀请消息的数据结构

一条标准邀请消息至少包含以下字段：

- `invite_id`：全局唯一邀请标识符（UUID v4格式）
- `sender_id`：发送方玩家账号ID
- `receiver_id`：接收方玩家账号ID
- `session_id`：目标游戏房间或大厅的会话ID
- `game_mode`：游戏模式标识字符串，如 `"ranked_5v5"`
- `expire_at`：邀请过期时间戳（Unix时间，通常设置为发送后60秒内有效）
- `status`：枚举值，取值为 `PENDING / ACCEPTED / DECLINED / EXPIRED`

平台级邀请（如Steam的 `ISteamFriends::InviteUserToGame`）在此基础上还附加 `connect_string`，这是一个游戏自定义的连接参数字符串，平台将其透传给游戏客户端，游戏再解析出服务器地址和加入令牌。

### 游戏内邀请与平台级邀请的区别

**游戏内邀请**由游戏服务器直接通过长连接推送给目标玩家客户端，仅当接收方已在游戏运行状态时才能收到。其优点是完全可控，房间容量校验、权限检查（如是否禁止加入进行中的对局）可在推送前完成。

**平台级邀请**（如Epic Online Services的 `EOS_Lobby_SendInvite`、Steam Overlay邀请）由平台服务接管，即使接收方尚未启动游戏，平台也会在操作系统层面弹出通知，玩家接受后平台自动启动游戏并传入 `+connect` 参数。这意味着游戏客户端必须在启动早期阶段（通常在主菜单加载完成前）检查命令行参数或平台回调，并据此执行延迟加入逻辑（Deferred Join）。

### 邀请的状态机与超时处理

邀请状态机的合法转换路径为：

```
PENDING → ACCEPTED
PENDING → DECLINED
PENDING → EXPIRED（服务器定时任务触发）
```

超时时间的设置直接影响体验：设置过短（如10秒）会导致接收方来不及查看通知就失效；设置过长（如300秒）则可能出现"房间已满但邀请仍有效"的竞态条件。主流做法是将超时设为60秒，并在接收方接受邀请后立即进行服务端的二次席位校验：若房间已满则返回错误码 `ERR_ROOM_FULL`，客户端据此提示"邀请已失效，房间已满"而非直接加入失败。

### 防滥用机制

邀请系统是垃圾信息的高风险入口。常见防护措施包括：限制每位发送方每分钟向同一接收方最多发送1条邀请（重复邀请需等待冷却）；阻止被屏蔽用户发送邀请；对非好友用户发起的邀请可根据隐私设置选择性屏蔽或需二次确认。在实现上，服务端通常使用Redis的滑动窗口计数器来执行发送频率限制，键名格式为 `invite_rate:{sender_id}:{receiver_id}`，TTL设为60秒。

## 实际应用

**《英雄联盟》大厅邀请**是游戏内邀请的典型实现。发送方在英雄选择前的大厅界面点击好友头像选择"邀请进入房间"，Riot的聊天服务（基于XMPP协议）将邀请以特殊Message Type推送给接收方客户端，接收方在游戏界面右下角收到弹窗，有效期约60秒。接受后客户端向大厅服务发送 `JOIN_LOBBY` 请求并携带 `invite_id` 以供服务端鉴权。

**《Fortnite》跨平台邀请**利用Epic Online Services同时支持游戏内和平台外两条路径。若接收方在PS5上未启动游戏，平台会向其发送系统通知，接受后自动启动游戏并通过 `EOS_Lobby_OnJoinLobbyAccepted` 回调传入大厅句柄，游戏客户端在启动序列的第3阶段（账号登录完成后）检查此回调并执行加入操作。

**《Among Us》房间码模式**是邀请系统的简化变体——并非推送邀请消息，而是由房主将6位字母房间码（如 `ABCDEF`）通过任意渠道（如语音、截图）分享，接收方手动输入。这种方式省去了服务端邀请状态管理，代价是无法追踪接受/拒绝状态，也无法在房间满员时给出实时反馈。

## 常见误区

**误区一：认为接受邀请等于成功加入房间**。实际上，"接受邀请"只是客户端向服务端表达意向，服务端仍需在此刻执行席位预留或抢占操作。若在邀请发出到接受的60秒内房间已被其他玩家填满，接受动作会得到服务端拒绝。因此客户端UI必须处理 `ACCEPTED` 状态后仍收到加入失败响应的情况，而不能假设接受即成功。

**误区二：游戏内邀请和平台级邀请可以完全替代彼此**。有的开发者认为只要接入了Steam邀请就不需要游戏内邀请，实际上两者互补。平台级邀请能触达离线/未启动游戏的玩家，但它无法访问游戏服务器的实时状态（如当前对局进度、房间密码验证）；游戏内邀请可以在推送前做精细校验，但对离线玩家无效。同时支持两种路径是生产环境的标准做法。

**误区三：邀请冷却限制只需在客户端实现**。若仅在客户端屏蔽重复邀请按钮，攻击者可通过直接构造HTTP请求绕过限制，向目标玩家每秒发送大量邀请消息，造成通知骚扰。防滥用逻辑必须在服务端执行，客户端的UI限制仅用于优化用户体验。

## 知识关联

邀请系统以**好友系统**为前提，好友列表提供了"可邀请对象"的数据来源，好友关系也是判断"是否允许发送邀请"的默认权限依据——未互相添加好友的玩家通常需要隐私设置放行才能收到陌生人邀请。好友在线状态（Online Presence）字段还为邀请系统提供了"该玩家当前是否在游戏中/是否可被邀请"的实时信息，避免向正在对局中的好友发送大厅邀请。

在大厅系统的功能链条中，邀请系统是玩家从"好友列表"进入"游戏大厅"的入口通道。成功接受邀请后，玩家进入的是**房间/大厅管理**模块所维护的会话状态，后续的队长权限、准备状态同步、游戏开始倒计时等逻辑均由大厅管理模块负责，邀请系统在完成加入动作后即退出生命周期。
