# 网络可见性过滤

## 概述

网络可见性过滤（Network Visibility Filtering）是多人游戏状态同步系统中的一项技术，用于决定哪些游戏实体的状态更新应当发送给特定客户端。其核心逻辑是：并非所有客户端都需要接收所有实体的状态变更包，服务器根据距离、队伍归属、视线遮挡、游戏规则等条件，对每个连接的客户端构建一份"当前可见实体集合"，只将该集合内实体的同步数据下发。

这一概念在2000年代初大型多人在线游戏（MMORPG）服务端架构成熟期逐步被系统化。早期游戏如《网络创世纪》（Ultima Online，1997年）已采用基于坐标格的简单剔除，而《魔兽世界》（2004年）的服务端则将可见性过滤拆分为独立的"Interest Management"子系统，与Ghosting机制结合，使单台区域服务器能同时承载数百名玩家密集交互而不产生广播风暴。

可见性过滤对带宽消耗和服务器CPU负载具有直接的量化影响。假设地图中有500个实体，若不进行任何过滤，每帧需要向每个客户端发送499份状态更新；引入半径为100单位的距离过滤后，在玩家密度均匀的场景中，平均可见实体数可降低至20～40个，带宽节省比例可超过90%。

## 核心原理

### 距离条件过滤

距离过滤是最基础且计算开销最低的过滤手段。服务器为每类实体（玩家角色、NPC、投射物、环境交互对象）分别定义可见半径阈值 `R_vis`。判断条件为：

```
dist(pos_client, pos_entity) ≤ R_vis
```

其中 `pos_client` 是接收方客户端控制角色的世界坐标，`pos_entity` 是目标实体的世界坐标。实际工程实现中通常省略开方运算，改用平方距离比较以减少浮点开销：

```
(Δx)² + (Δy)² + (Δz)² ≤ R_vis²
```

不同实体类型的 `R_vis` 取值差异显著。在典型的第三人称射击游戏中，子弹或技能特效的可见半径设为50～80单位，而玩家角色通常为200～500单位。对于战略游戏中的"战争迷雾"设计，`R_vis` 本身会随玩家侦察状态动态调整。

### 团队与阵营过滤

团队过滤针对的是"玩家不应看见特定信息"的游戏设计语义，而非物理距离。最典型的应用场景是隐身机制：处于隐身状态的敌方单位，其实体包即便在距离阈值内也应从敌对客户端的可见集合中剔除，但在己方队友的可见集合中仍保留。

实现上，服务器为每个实体维护一个 `visibility_rule` 枚举，常见取值包括：`ALL`（所有人可见）、`TEAM_ONLY`（仅同队玩家可见）、`OWNER_ONLY`（仅拥有者可见，如个人背包道具）、`CUSTOM`（由脚本逻辑自定义）。在计算可见集合时，距离过滤先于团队过滤执行，以最小化需要检查阵营规则的实体数量。

### 状态差异过滤（Delta Filtering）

可见性过滤不仅控制"哪些实体可见"，还可控制"可见实体的哪些属性字段需要同步"。当一个实体已在客户端的可见集合中，但其某帧的状态与上一帧完全相同时，服务器可以跳过该实体的同步包生成，这称为 Delta Filtering 或变更检测过滤。

具体实现方式是为每个（客户端, 实体）对维护一份"上次发送状态快照"。每帧同步时，服务器比对当前状态与快照，仅当存在字段差异时才生成并发送更新包，同时刷新快照。在玩家密集但大多数NPC静止的场景中（如城镇区域），此机制可将实际发包量再降低40%～60%。

### 进入与离开事件的处理

当实体从客户端可见集合外进入集合内时，服务器必须发送该实体的**完整初始化快照**（Full State Snapshot），因为客户端本地没有该实体的任何数据。反之，当实体离开可见集合时，服务器向该客户端发送"实体销毁"通知，客户端据此从本地模拟中移除该实体。这两类边界事件的处理逻辑必须保证幂等性，避免网络重传导致重复创建或销毁的BUG。

## 实际应用

**Battle Royale 游戏的分层过滤**：在《绝地求生》类游戏中，可见性过滤通常分为三层：第一层基于距离（800米内才同步玩家位置）；第二层基于地形遮挡（服务端利用简化碰撞网格做射线检测，剔除被建筑完全遮挡的实体）；第三层针对载具和掉落物使用更大的可见半径（约1200米），因为这类实体的移动频率低，带宽成本可接受。

**MOBA中的战争迷雾实现**：《英雄联盟》服务端对每个客户端维护一张动态可见格（Fog of War Grid），分辨率约为每格128游戏单位。敌方英雄只有当其所在格被己方至少一个视野源覆盖时，才进入该客户端的可见集合。离开视野后，服务端会发送最后一次确认位置的快照并附带"进入迷雾"标记，客户端据此停止渲染该英雄但保留位置记忆用于本地预测显示。

**MMO副本中的强制全局可见**：在Boss战副本场景中，由于副本内玩家数量被限制在10～25人，且所有玩家都需要看到Boss的全部状态（血量、技能指示器、仇恨列表），此场景下通常临时禁用距离过滤，改为"副本成员全局可见"规则，简化逻辑并确保战斗信息完整性。

## 常见误区

**误区一：可见性过滤等同于反作弊**。部分开发者认为，只要客户端收不到隐身敌人的同步包，就能防止透视外挂。这一理解在理论上正确，但实际上，大量游戏因过滤时机（如提前50ms发送进入视野通知以补偿网络延迟）或过滤半径设置过大，导致客户端实际上仍能收到本应被过滤的数据。精确的可见性过滤确实能大幅提升反作弊效果，但必须与服务端权威验证配合使用，而非作为唯一的安全手段。

**误区二：过滤半径越小越好**。将距离阈值设置得极小（如20单位）虽然最大程度节省带宽，但会导致实体在客户端视野边缘频繁出现和消失，形成"闪烁"体验。更严重的是，每次实体重新进入可见集合都需要发送完整初始化快照，在玩家密集频繁移动的场景中，这些突发的初始化包流量反而可能超过维持一个较大稳定可见集合的增量更新流量。工程上通常使用"进入半径"（Enter Radius）与"离开半径"（Leave Radius）的滞后双阈值设计（Hysteresis），如进入阈值为200、离开阈值为240，避免边界抖动。

**误区三：Delta Filtering可以完全替代可见性过滤**。即使某帧实体没有状态变化，服务器仍需对其执行"是否需要发送"的检查运算。若不事先用可见性过滤缩小候选集合，Delta Filtering每帧仍需遍历场景中所有实体来判断是否有变化，CPU开销不会随实体数量增加而减少。两者解决的是不同维度的问题，必须组合使用。

## 知识关联

网络可见性过滤直接建立在**兴趣管理（AOI，Area of Interest）**的基础上。AOI系统（通常采用九宫格划分或四叉树空间索引）提供了高效查询"哪些实体在某坐标附近"的数据结构，可见性过滤则在AOI查询结果之上叠加团队规则、状态差异检测、视线遮挡等附加条件，最终生成精确的"每客户端可见集合"。可以理解为：AOI是空间层面的粗粒度过滤，可见性过滤是语义与状态层面的细粒度过滤，两者共同构成完整的同步裁剪流水线。

在状态同步架构的更宏观视角中，可见性过滤的输出结果（每客户端可见实体集合及其变更列表）直接驱动**网络序列化**与**发包调度**模块的工作输入。理解可见性过滤的边界事件处理（进入/离开集合时的完整快照 vs. 增量更新）对于设计可靠的客户端实体生命周期管理系统至关重要。
