# 网关服务器

## 概述

网关服务器（Gateway Server）是多人游戏服务端架构中客户端的统一接入点，所有玩家连接请求必须先经过网关服务器，再由其转发至后端的逻辑服务器、战斗服务器或聊天服务器。与负载均衡器单纯分配流量不同，网关服务器还承担身份预验证、会话管理和协议转换等职责，是客户端与后端集群之间的"守门人"。

网关服务器的设计模式源于企业级 API Gateway 概念，约在 2010 年前后随着 MOBA 和 MMORPG 游戏规模扩大而被引入游戏后端架构。早期的网游（如 2000 年代的端游）通常让客户端直接连接区服服务器，这导致一旦某个区服 IP 泄露，服务器就会面临直接的 DDoS 攻击。网关服务器的出现将后端真实 IP 完全隐藏，客户端只知道网关地址。

在实际运营中，网关服务器的重要性体现在两个可量化的维度：一是安全隔离，所有非法封包在网关层被丢弃，后端服务器不接受任何来自网关之外的直接连接；二是路由效率，一台高性能网关服务器可维持 10 万以上的并发 TCP 长连接，通过单点接入避免客户端因服务器地址变更而断线。

## 核心原理

### 连接接入与协议解析

客户端通过 TCP 或 UDP 与网关建立连接后，网关首先解析客户端发来的握手包（Handshake Packet）。该握手包通常包含客户端版本号、设备指纹和临时 Token 三个字段。网关解析后判断版本是否在白名单内（例如只允许版本号 ≥ 2.3.1 的客户端接入），不合法的连接在此阶段直接被 RST 断开，不会产生任何后端开销。协议层面，网关常做二进制私有协议到内部 Protobuf 格式的转换，使后端服务无需感知不同平台客户端的封包差异。

### 身份预验证与 Token 路由

网关服务器的路由决策依赖一个轻量级的身份预验证流程。客户端携带登录服务器颁发的短期 JWT（有效期通常为 30 秒至 5 分钟）向网关发起认证请求，网关使用共享密钥（HMAC-SHA256）在本地验证 JWT 签名，无需每次都请求认证服务，验证耗时控制在 1ms 以内。验证通过后，网关从 JWT Payload 中提取 `server_id` 字段，据此将该连接的后续所有数据包路由到对应的逻辑服务器节点，这一过程称为"基于身份的有状态路由"（Stateful Identity-based Routing）。

路由表结构如下：

```
RouteEntry {
    client_token:  string   // 客户端会话标识
    backend_addr:  string   // 目标逻辑服务器 IP:Port
    last_active:   timestamp
    byte_in/out:   counter
}
```

网关在内存中维护此路由表，每条记录占用约 200 字节，10 万并发连接约消耗 20MB 内存，开销极低。

### 流量控制与反作弊过滤

网关服务器实现了两层流量限制：连接级限速（每个连接每秒最多发送 N 个封包，典型值为 60 pps，对应 60Hz 的客户端帧率上限）和 IP 级限速（同一 IP 每秒建立新连接数上限，防止 SYN Flood）。超过连接级限速的封包直接在网关丢弃并计数，当某个 Token 的超速次数在 10 秒内超过 100 次时，网关将该连接标记为"可疑"并同步通知反作弊服务。此外，网关还做封包长度校验：如果客户端发来的封包长度超过协议规定的最大值（如 4096 字节），直接断开该连接，防止缓冲区溢出攻击。

## 实际应用

**《英雄联盟》区域网关架构**：Riot Games 公开披露其使用多个地区性网关集群，每个集群前置一层 Anycast IP，玩家 DNS 解析到最近的 PoP 节点，该节点上的网关服务器再通过内网专线将连接转发至位于数据中心的逻辑服务器。这使得跨区延迟降低约 30ms，同时后端逻辑服务器的 IP 对外完全不可见。

**移动游戏的 HTTP/WebSocket 混合网关**：手游中客户端常在弱网环境下频繁重连。网关服务器对此专门维护"会话恢复"机制：当玩家 TCP 连接断开后，网关保留其路由表条目 30 秒，玩家在此窗口内携带原 Token 重连时，网关直接复用原有路由关系，后端逻辑服务器无感知，玩家游戏状态不丢失。若超过 30 秒仍未重连，路由条目才被释放，逻辑服务器收到下线通知。

**灰度发布路由**：网关服务器还被用于版本灰度控制。通过在路由规则中配置账号 ID 的哈希分段（如 `uid % 100 < 5` 的玩家路由到新版本逻辑服务器），网关可将 5% 的流量引导至新服务节点，实现不停服的灰度发布，同时大多数玩家无感知。

## 常见误区

**误区一：网关服务器等同于负载均衡器**。负载均衡器（如 Nginx upstream、LVS）的核心功能是按算法（轮询、最少连接等）分配请求，它对请求内容无感知，也不维护会话状态。网关服务器则在此之上增加了身份验证、有状态路由和封包过滤逻辑，它知道"这个连接属于哪个玩家、该去哪台逻辑服务器"。两者通常串联部署：负载均衡器在最外层将新连接均匀分配给多台网关服务器，再由网关服务器做身份路由。

**误区二：网关服务器负责完整的身份认证**。网关层只做 JWT 签名校验（验证 Token 是否被篡改），不负责颁发 Token，也不查询用户数据库验证密码。完整的账号认证（用户名/密码核对、二次验证、封号检查）由专门的认证服务完成，认证通过后才向客户端颁发 JWT，客户端再持此 JWT 向网关接入。网关信任已通过认证服务颁发的 Token，但不重新执行认证逻辑。

**误区三：单台网关是性能瓶颈**。实际部署中，网关服务器是无状态水平扩展的（路由表通常存于共享 Redis，而非网关本地内存），多台网关实例共享同一份路由数据。因此增加网关实例数量可线性提升接入能力，不存在单点瓶颈问题。

## 知识关联

**与负载均衡的关系**：负载均衡是网关服务器的前置流量入口，解决的是"多台网关服务器之间如何分配新连接"的问题。掌握负载均衡的会话保持（Session Persistence）原理有助于理解为什么同一玩家的重连必须落到同一台或能访问共享路由表的网关节点上。

**通向身份认证的桥梁**：网关服务器中的 JWT 校验逻辑直接引出下一个主题——身份认证服务的设计。理解网关只做"验签"而不做"颁证"，能帮助学习者明确认证服务需要解决哪些问题：Token 的生命周期管理、刷新机制（Refresh Token）、多设备并发登录检测，以及封号状态的实时同步。这些都是认证服务独立于网关存在的核心原因。
