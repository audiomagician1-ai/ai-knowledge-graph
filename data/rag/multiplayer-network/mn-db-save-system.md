# 存档系统

## 概述

存档系统是网络多人游戏中负责将玩家游戏进度持久化到远端服务器的数据库子系统，其核心职责包括：定时将本地游戏状态写入云端、在玩家跨设备登录时恢复进度，以及在多设备同时在线时解决写入冲突。与单机游戏将存档写入本地文件不同，云存档必须应对网络延迟、断线重连和并发写入这三类额外挑战。

云存档的概念随网络游戏普及在2000年代初期逐渐成型。早期的网游（如2004年的《魔兽世界》）直接在服务器端维护全量角色状态，客户端不保留任何持久化数据；而移动游戏兴起后（约2010年后），玩家频繁在手机与平板之间切换，迫使开发者设计出能合并多份本地存档的同步机制。

对于大型多人游戏而言，存档系统的可靠性直接影响玩家留存率。行业数据显示，若玩家因存档丢失而损失超过30分钟的游戏进度，该玩家的7日流失概率将上升约40%。因此，存档系统不仅是技术问题，也是核心商业指标的保障手段。

## 核心原理

### 存档数据结构

存档记录在数据库中通常以如下字段组织：

```
save_record {
  player_id       : UUID        // 对应玩家数据模型中的主键
  save_slot       : INT         // 存档槽位编号，从0开始
  payload         : JSONB       // 游戏进度的序列化内容
  version         : BIGINT      // 单调递增的版本号
  client_timestamp: TIMESTAMP   // 客户端本地时间
  server_timestamp: TIMESTAMP   // 服务器写入时间
  checksum        : VARCHAR(64) // payload的SHA-256哈希前64位
}
```

其中 `version` 字段是冲突检测的关键。每次客户端发起写入请求时，必须携带它上次读取到的 `version` 值；服务器只有在数据库中当前 `version` 与请求携带的值相等时才执行更新，否则拒绝并返回冲突错误码（通常定义为 HTTP 409 Conflict）。

### 版本向量与冲突检测

当玩家在设备A和设备B均处于离线状态并分别修改了存档时，两份数据都基于同一个旧版本产生了新的 `version`。这种情形称为**分叉冲突（Fork Conflict）**。仅靠单一整数版本号无法区分哪份更新"更新"，需引入**版本向量（Version Vector）**：

```
version_vector = { device_id_1: seq_1, device_id_2: seq_2, ... }
```

服务器对两份版本向量做分量比较：若向量A在每个分量上均 ≥ 向量B，则A支配B，直接丢弃B；若存在某分量A大而另一分量B大，则判定为真正冲突，需进入合并流程。

### 冲突解决策略

存档系统常用的三种冲突解决策略各有适用场景：

1. **Last-Write-Wins（LWW）**：以 `server_timestamp` 较大的存档为准，直接覆盖旧版本。实现简单，但可能丢失玩家在另一设备上的进度，适用于进度完全线性的游戏（如单线剧情游戏）。

2. **字段级合并（Field-Level Merge）**：将存档的 `payload` 拆分为互相独立的字段（如 `gold`、`level`、`inventory`），对每个字段分别取最大值或最新值。例如金币数取两份存档中的较大值，已解锁关卡取两份解锁集合的并集。这是手机RPG游戏最常用的策略。

3. **提示用户选择**：将两份存档的关键差异（如等级差、游戏时长差）呈现给玩家，由玩家手动选择保留哪一份。此策略零数据丢失，但会中断游戏体验，通常仅在冲突差异超过阈值（如等级差 ≥ 5）时触发。

### 定时自动存档

客户端应以固定间隔（常见值为60秒或120秒）向服务器发送增量存档请求，而非每次状态变化都触发全量写入。增量存档只上传自上次成功存档以来发生变化的字段，可将单次写入的网络流量从数十KB降低到数百字节。服务器收到增量数据后，先在内存中与最新全量存档合并，再批量持久化到磁盘，通常每10分钟执行一次批量落盘以降低磁盘I/O压力。

## 实际应用

**《原神》的跨平台存档同步**：米哈游为《原神》设计了基于账号绑定的全量云存档方案。玩家在PS5、PC、手机之间切换时，游戏启动后自动从服务器拉取最新存档，本地缓存会在成功拉取后失效。由于《原神》不支持离线游玩，不会产生分叉冲突，因此使用简单的LWW策略即可满足需求。

**《我的世界》基岩版的存档冲突案例**：微软的Minecraft基岩版允许玩家在离线模式下修改世界存档。当玩家在无网络的飞机上游玩后重新联网，客户端检测到本地 `version` 与服务器 `version` 不一致，会弹窗提示玩家选择保留"本地存档"还是"云端存档"，这是典型的"提示用户选择"策略的实际部署。

**手机卡牌游戏的字段级合并**：某款手机卡牌游戏（如《皇室战争》类）在服务器端为每个玩家维护一张存档差异表（`save_diff_log`），记录每个字段的最后修改时间戳。当检测到冲突时，对每个字段独立比较时间戳，取较新的值覆盖较旧的值，整个合并过程在100毫秒内完成，玩家完全无感知。

## 常见误区

**误区一：客户端时间戳可以作为冲突仲裁依据**。客户端时钟可被玩家篡改，或因设备时区设置错误而产生偏差。曾有玩家通过将手机时间调至未来，强制使本地存档在LWW策略下覆盖服务器存档，从而"撤销"付费道具消耗。正确做法是以服务器时间戳作为最终仲裁依据，客户端时间戳仅作参考日志使用。

**误区二：版本号递增就足以防止所有冲突**。单调递增的整数版本号只能检测到"有人在我读取之后写入了数据"，无法识别两个客户端基于同一版本号各自产生新版本的分叉场景。只有引入版本向量或操作日志（Operation Log）才能准确识别分叉冲突并执行语义正确的合并。

**误区三：存档频率越高越安全**。将自动存档间隔从120秒改为5秒，并不能成比例地提升数据安全性，反而会使数据库写入QPS增加约24倍，在百万DAU的游戏中可能直接压垮数据库。正确策略是结合增量存档、内存合并和批量落盘，在写入频率与服务器负载之间取得平衡。

## 知识关联

存档系统直接依赖**玩家数据模型**中定义的 `player_id` 作为存档记录的外键，存档表的分片（Sharding）策略也通常与玩家表保持一致，以避免跨分片JOIN查询。玩家数据模型中对字段类型的定义（如金币是 `INT` 还是 `BIGINT`）决定了存档 `payload` 的序列化格式，两者必须保持版本同步，否则反序列化时会出现字段溢出或类型错误。

在数据库设计层面，存档系统的冲突检测依赖关系型数据库的**乐观锁（Optimistic Locking）**机制，其SQL实现形如：
```sql
UPDATE save_records
SET payload = $new_payload, version = version + 1
WHERE player_id = $pid AND save_slot = $slot AND version = $expected_version;
```
若该UPDATE影响行数为0，则说明发生冲突，需触发冲突解决流程。这一机制将并发控制的复杂度从数据库锁层面转移到了应用层，是存档系统高并发写入的基础保障。
