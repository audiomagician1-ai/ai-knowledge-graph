# 连接生命周期

## 概述

连接生命周期（Connection Lifecycle）描述一条网络连接从诞生到终止的完整过程，具体包含四个顺序阶段：握手建立（Handshake）、心跳维持（Heartbeat）、超时检测（Timeout Detection）和断线处理（Disconnection Handling）。在多人游戏服务器中，每一条客户端连接都会经历这四个阶段，服务器需要对所有并发连接的当前阶段进行追踪和状态管理。

这一概念在1990年代多人网络游戏兴起时随实践逐步形成体系。早期MUD游戏的Telnet连接管理已经包含超时踢出机制，而现代游戏引擎（如Unity的Mirror框架、Unreal的OnlineSubsystem）均将连接生命周期抽象为显式的状态机，开发者可以注册每个阶段的回调函数。

理解连接生命周期对服务器稳定性至关重要，因为"僵尸连接"（Zombie Connection）——即客户端已经断开但服务器未能检测到的连接——会持续占用内存槽位和玩家编号，导致房间无法正常填满或逻辑状态错乱。正确实现生命周期管理可以将僵尸连接存活时间从数分钟压缩到数秒。

## 核心原理

### 阶段一：连接建立（握手）

TCP协议本身的三次握手（SYN → SYN-ACK → ACK）完成后，游戏层通常还需要执行**应用层握手**，步骤典型如下：

1. 客户端发送 `ClientHello` 包，携带协议版本号（例如 `Protocol Version: 3`）和玩家令牌（Token）。
2. 服务器验证令牌有效性，回复 `ServerHello`，分配唯一的 `ConnectionID`（通常为32位或64位整数）。
3. 客户端确认收到 `ConnectionID`，连接正式进入**已建立（Established）**状态。

若在握手阶段超过规定时限（一般为5秒）未完成，服务器直接关闭底层Socket，避免半开连接（Half-Open Connection）占用文件描述符。

### 阶段二：心跳维持（Heartbeat）

心跳机制的本质是双方约定一个固定频率的"存活证明"包。游戏中常见配置是**每隔2秒客户端发送一个 `PING` 包**，服务器回复 `PONG` 包。心跳包的有效载荷极小，通常只有4~8字节（包含序列号和时间戳），对带宽影响可以忽略不计。

心跳并非对称的义务——在大多数游戏架构中，**由客户端发起心跳、服务器被动响应**。这样设计的原因是：服务器需要同时管理数百甚至数千条连接，让客户端承担主动发包的职责可以避免服务器广播风暴。Mirror框架的 `NetworkManager` 默认心跳间隔（`HeartbeatInterval`）为1秒，且该值可在 Inspector 面板直接调整。

### 阶段三：超时检测

服务器为每个连接维护一个 `LastHeartbeatTime` 时间戳，每收到一个心跳包就刷新该时间戳。超时检测逻辑在服务器端以固定周期（例如每500ms）运行：

```
若 (CurrentTime - LastHeartbeatTime) > TimeoutThreshold
    则触发超时事件
```

`TimeoutThreshold`（超时阈值）的设置需要在**响应速度**和**误判率**之间取得平衡。典型阈值为**心跳间隔的3到5倍**：若心跳间隔为2秒，则超时阈值设为6~10秒。将阈值设得过短（如等于心跳间隔）会在正常网络抖动时产生大量误判断线。

### 阶段四：断线处理

超时检测触发后，服务器执行以下清理序列：

1. 将连接状态标记为 `Disconnecting`，停止向该 `ConnectionID` 发送游戏逻辑包。
2. 触发 `OnPlayerDisconnected(connectionId)` 回调，游戏逻辑层可在此保存玩家状态到数据库。
3. 从活跃连接池（Active Connection Pool）中移除该连接对象，释放 `ConnectionID`。
4. 若断线重连窗口开启（见后文知识关联），将玩家状态保留在内存中一段时间（通常30~60秒）而非立即销毁。

## 实际应用

**射击游戏房间管理**：在一场10人对战的FPS房间中，服务器维护一张 `ConnectionID → PlayerState` 的哈希表。若某玩家路由器突然断电，TCP的 `FIN` 包不会被发出，服务器只能依靠超时检测在约8秒后发现该玩家消失，随后将其从队伍中移除并向其他9名玩家广播 `PlayerLeft` 事件。

**移动端弱网环境心跳调优**：在4G/5G网络下，信号切换可能导致2~3秒的短暂断包。如果心跳阈值设为6秒，则这类正常的信号切换不会触发断线；若阈值设为3秒，玩家每次过隧道都会被踢出游戏。因此移动端游戏通常将超时阈值设得比PC端更宽松，常见值为15~20秒。

**MMO服务器的连接池预热**：大型MMO在服务器启动时预先分配固定数量的连接对象（例如预分配5000个 `Connection` 实例），而非在每次握手时动态分配。这种对象池化（Object Pooling）模式减少了GC压力，而连接生命周期管理负责在断线处理阶段将 `Connection` 对象重置并归还到池中，而非销毁。

## 常见误区

**误区一：TCP自身能检测到所有断线情况**
TCP的 `keepalive` 机制默认探测间隔为**75秒**（Linux默认 `tcp_keepalive_intvl`），且需要操作系统级别的配置才能生效，完全不适合游戏场景的快速断线检测需求。游戏必须在应用层实现独立的心跳机制，而非依赖TCP keepalive。

**误区二：心跳包越频繁越好**
将心跳间隔从2秒缩短到100ms会使心跳流量增加20倍，在一个500人在线的服务器上，仅心跳流量就可能达到每秒数百KB。更致命的是，100ms心跳意味着300ms的抖动就会触发超时，这在任何真实网络环境中都过于激进。心跳频率必须与超时阈值协同设计，不能孤立调整。

**误区三：超时即断线，断线即销毁**
超时检测触发后，连接进入 `Disconnecting` 状态，而非立即销毁玩家数据。将超时与玩家状态销毁耦合在一起，会导致断线重连功能无法实现——因为玩家数据在检测到超时的瞬间就已被清除，重新上线的玩家只能从零开始。正确做法是将"连接销毁"和"玩家状态销毁"解耦为两个独立的事件。

## 知识关联

**前置概念——断线重连**：断线重连机制是连接生命周期第四阶段（断线处理）的直接扩展。断线重连要求服务器在超时后不立即销毁玩家状态，而是将其保持在"等待重连"的暂存状态；连接生命周期定义了这个暂存状态的起点（超时检测触发时刻）和终点（重连成功或等待窗口过期）。

**后续概念——带宽管理**：心跳包虽然单包体积极小，但在高并发场景下心跳流量的总量不可忽视；带宽管理需要将心跳流量纳入整体带宽预算计算。此外，连接生命周期的"已建立"阶段是所有游戏数据包传输的前提条件，带宽管理的全部策略（如包合并、优先级队列）均在该阶段生效。

**后续概念——断线重连（深入）**：在掌握连接生命周期的四个阶段后，断线重连的实现细节（例如如何在新的TCP连接上恢复旧的 `ConnectionID` 绑定关系）将变得更加清晰，因为此时开发者已经理解"连接"和"玩家会话"是两个可以独立存在的对象。
