# 断线重连

## 概述

断线重连（Reconnect）是网络多人游戏大厅系统中处理玩家意外掉线后重新加入进行中游戏的技术机制。当玩家因网络抖动、客户端崩溃或设备切换等原因与服务器失去连接时，断线重连机制允许该玩家在一定时间窗口内重新建立连接，并恢复其在游戏中的状态，而不必从大厅重新匹配。

这一机制最早在 MMORPG 时代就已存在，但在 MOBA 类游戏（如2013年前后的《英雄联盟》版本更新中明确强化了重连流程）和实时对战游戏流行后变得尤为关键。由于一局对战游戏通常持续20至60分钟，任何一名玩家的意外离线都会严重破坏其他参与者的游戏体验，因此断线重连成为保障对局完整性的必要设计。

断线重连的核心价值在于它需要服务器在客户端缺席期间继续托管该玩家的游戏状态，并在重连成功后将积累的状态差异同步给客户端。这与普通连接建立不同——普通连接从零开始加载状态，而断线重连必须从某个中间时间点恢复，涉及状态快照、指令回放等复杂技术问题。

## 核心原理

### 会话令牌与身份识别

断线重连依赖一个在初次连接时由服务器颁发的**会话令牌（Session Token）**。这个令牌通常是一个64位或128位的随机字符串，在玩家断线期间保留在服务器侧的会话表中。当玩家尝试重连时，客户端携带此令牌发送重连请求，服务器通过令牌查找对应的游戏房间和玩家槽位，而非重新走完整的匹配流程。令牌有严格的有效期，常见设置为**60至300秒**，超出该窗口服务器将销毁会话，玩家只能认负或重新排队。

### 断线检测与状态保持

服务器通过心跳包（Heartbeat）机制检测玩家是否真正断线。典型配置是客户端每隔1至5秒发送一次心跳，若服务器在连续3至5次心跳周期内未收到响应，则将该连接标记为`DISCONNECTED`状态。在此状态下，服务器不会立即销毁玩家对象，而是将其切换为**AI托管模式**或**冻结模式**：前者由服务器用简单AI控制该角色继续参与游戏，后者则让角色原地停止行动等待重连。这段时间内，服务器仍然持续记录游戏状态的完整变更日志（Change Log）。

### 状态追赶（State Catch-up）

重连成功后，服务器面临的核心技术挑战是将玩家离线期间错过的所有状态变更推送给客户端。常见的两种方案是：

- **完整状态快照（Full State Snapshot）**：服务器将当前游戏的完整状态序列化后一次性发送，客户端丢弃旧状态直接加载新状态。适合状态数据量较小的游戏（如棋牌类），重连时间通常在1至3秒以内。
- **增量指令回放（Delta Command Replay）**：服务器保存玩家断线后所有游戏指令的有序队列，重连时向客户端重放这些指令。适合状态复杂的实时对战游戏，但对服务器的日志存储提出更高要求，回放过程中客户端需要以加速模式处理指令，通常表现为进度条加载画面。

### 重连流程的状态机

断线重连的完整流程可以用如下状态转移描述：
```
CONNECTED → DISCONNECTED（心跳超时）
DISCONNECTED → RECONNECTING（客户端携令牌请求重连）
RECONNECTING → CONNECTED（令牌验证通过 + 状态追赶完成）
DISCONNECTED → ABANDONED（令牌过期或玩家主动放弃）
```
服务器需要在`DISCONNECTED`状态下维持该玩家的房间席位，这意味着这段时间其他玩家无法顶替其位置，这也是为什么重连窗口时间不能设置得过长的原因。

## 实际应用

**《英雄联盟》的重连机制**采用了完整快照与增量补丁混合的方式：重连时首先加载最近一次服务器保存的游戏快照（约每30秒保存一次），再回放快照之后的增量指令。玩家在重连界面会看到进度条，本质上是客户端在高速模拟追赶服务器当前帧。

**大厅系统侧的配合设计**体现在：当某个房间内有玩家处于`DISCONNECTED`状态时，大厅服务器需要将该房间标记为"进行中且有断线玩家"，防止其被匹配系统误判为可加入的候补房间。同时，断线玩家的客户端会被引导至专用的重连入口（通常是游戏主界面显示"检测到未完成对局，点击重连"的提示），而不是普通的创建/加入房间流程。

**移动端游戏**（如《王者荣耀》）由于用户频繁切换网络（Wi-Fi与4G之间），断线重连的触发频率远高于PC端，因此其重连窗口设计得更长（约120秒），并在断线期间由AI接管角色操作，以减少对队友的负面影响。

## 常见误区

**误区一：断线重连与"重新加入"是同一件事。**
实际上两者有本质区别。断线重连依赖服务器侧保留的会话令牌，整个重连过程中玩家的游戏对象从未被销毁，只是连接通道中断；而"重新加入"（Rejoin）是玩家被踢出后重新走加入流程，服务器需要重建玩家状态。许多游戏引擎文档（如Photon SDK）将这两个接口`Reconnect()`和`ReJoinRoom()`分开实现，混用会导致令牌验证失败或状态错乱。

**误区二：只要重连成功，客户端状态自动与服务器同步。**
状态追赶不是自动完成的，开发者必须显式实现状态同步逻辑。若客户端在重连后仅恢复了TCP/UDP连接而未处理离线期间的增量状态，玩家看到的游戏画面将停留在断线瞬间，而实际游戏已推进了数十秒甚至数分钟。这种"连接成功但画面卡住"的bug在新手开发者中极为常见。

**误区三：延长重连窗口时间对体验只有正面效果。**
过长的重连窗口（如超过5分钟）会使对局中的其他玩家长时间面对一个AI托管的队友或敌人，体验远不如正常对战。同时，恶意玩家可能故意断线以触发AI托管状态来规避不利局面，再通过重连回归。因此，重连窗口的时长设置需要结合具体游戏类型和对局时长进行权衡，并配合防滥用检测逻辑。

## 知识关联

断线重连建立在**连接生命周期**的基础之上。理解连接生命周期中`CONNECTED`、`DISCONNECTED`、`CLOSED`各状态的转换条件，是正确实现断线检测和令牌保留逻辑的前提——若不清楚何时应该销毁会话（`CLOSED`）、何时应该保留会话等待重连（`DISCONNECTED`），就无法正确设计会话令牌的生命周期管理。

反过来，断线重连的实现反过来深化了对连接生命周期的理解：它揭示了`DISCONNECTED`状态并非一个简单的终止节点，而是一个具有时间窗口的中间状态，服务器在此期间需要维持资源占用。这对设计更完善的连接生命周期状态机（包括超时策略、资源回收时机、心跳参数调优）有直接指导意义。
