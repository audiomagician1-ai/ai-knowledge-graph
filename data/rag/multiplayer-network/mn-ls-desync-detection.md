# 不同步检测

## 概述

不同步检测（Desync Detection）是帧同步网络游戏中用于发现客户端之间游戏状态发生分歧的技术手段。当多个客户端在相同帧号执行相同指令后，若各自维护的游戏世界状态出现差异，即发生"不同步"（desync）。不同步检测的目标是在这种分歧发生后，尽快准确地定位到是哪一帧、哪一个游戏对象、哪一个属性值出现了偏差。

帧同步架构于2010年前后随《星际争霸》《魔兽争霸》等RTS游戏的广泛分析而被深入研究。早期游戏对不同步的处理方式非常粗暴——仅当客户端之间传输的哈希值不匹配时弹出"You have been dropped"错误，玩家被踢出房间但无法知道任何根因信息。现代不同步检测系统则追求在毫秒级内精确定位偏差对象，甚至支持在生产环境中实时诊断。

不同步检测之所以重要，在于帧同步系统的"蝴蝶效应"特性：一帧内哪怕0.0001的浮点误差，经过数百帧迭代后会导致两端单位坐标差距超过数十米，最终游戏完全失控。若缺乏精确的检测手段，开发者只能凭猜测逐一排查代码，修复周期可达数周。

## 核心原理

### 状态哈希校验与粒度分层

不同步检测的基础是**同步校验**机制生成的哈希值比对。最简单的方案是对整个游戏世界做一次全量哈希（Full State Hash），将所有实体的位置、速度、HP等序列化后送入CRC32或MD5运算，并在每帧结束后通过服务器进行对比。若哈希不匹配则确认该帧发生不同步。

然而全量哈希只能回答"是否发生不同步"，无法回答"在哪里发生"。为此，现代方案采用**分层哈希树（Hierarchical Hash Tree）**：

- **Level 0（全局哈希）**：所有实体序列化后的总哈希，每帧必传。
- **Level 1（对象组哈希）**：按对象类型（如"所有单位"、"所有子弹"、"地形状态"）分组计算哈希。
- **Level 2（单实体哈希）**：对每个单独实体计算哈希，包含其每个关键字段。

当Level 0出现不匹配时，系统请求客户端上传Level 1数据；Level 1定位到偏差组后，再请求Level 2，最终精确到单个对象的具体字段。这种按需下钻的方式使带宽消耗降低约90%。

### 快照差异比对（Snapshot Diff）

另一种检测方案是**快照差异比对**：在不同步发生的帧N，所有客户端将该帧的完整世界状态序列化为二进制快照并上传至服务器。服务器对两份快照执行逐字节的Diff操作，直接输出偏差字段的名称、客户端A的值、客户端B的值。

一个典型输出示例如下：

```
Frame 1523 | Entity#42 (Soldier) | field: position.x | ClientA: 312.4417 | ClientB: 312.4419
```

这种方案的精度极高，但快照体积可能达到数百KB，不适合在生产环境中对每帧执行。常见做法是将其作为**诊断模式**（Debug Mode），仅在复现不同步的测试场景中启用，或在检测到Level 0哈希不匹配后自动触发一次性的快照上传。

### 录像回放验证（Replay Verification）

录像回放验证利用帧同步架构的确定性特性：给定相同的初始状态和相同的输入序列，必然产生相同的游戏结果。将一场完整对局的操作指令录制为Replay文件，在不同机器上（或不同版本代码上）分别执行，对比每隔N帧的状态哈希。

具体流程：
1. 以帧为单位记录所有玩家指令（Replay文件，通常仅数KB）。
2. 在至少两个客户端上重放，各自每隔10帧输出状态哈希。
3. 对比两个哈希序列，找到**第一个出现偏差的帧号**，该帧即为不同步的真实起源帧。

这种方法对于复现**概率性不同步**（如浮点数跨平台行为差异、随机数种子问题）尤为有效，因为问题可能每100局才触发一次，而录像可重复执行直到捕获。

## 实际应用

**《王者荣耀》的不同步监控系统**将分层哈希与日志上报结合：每局比赛中，服务器每秒采样约20帧的全局哈希，当连续3帧出现哈希不匹配时触发告警并自动收集Level 1数据。该系统帮助团队将平均不同步定位时间从"数天"缩短至"数小时"。

在**《星际争霸II》**的战网系统中，录像文件本身即是不同步检测的基础设施——所有Replay上传后，战网服务器会在云端异步重放，若云端重放的帧哈希与客户端记录的哈希不符，则标记为潜在不同步案例并推送给开发团队分析。

对于独立开发者，Unity生态下的**Determinism Test Framework**提供了一套实用工具：在Editor中开启双实例模式，两个实例并行执行同一帧同步逻辑，每帧自动比对其内存中所有被标记为`[SyncField]`属性的字段值，一旦发现差异立即暂停并高亮显示偏差变量。

## 常见误区

**误区一：哈希不匹配 = 代码逻辑有Bug**

许多开发者在看到哈希不匹配后立即去审查游戏逻辑代码，而忽视了**浮点数跨平台差异**这一常见根源。x86架构的FPU在80位精度模式下与ARM设备的32位IEEE 754运算结果可能在小数点后第7位产生偏差。不同步检测定位到具体字段后，若该字段的偏差仅在`1e-6`量级，应优先怀疑浮点确定性问题，而非逻辑错误。

**误区二：检测频率越高越好**

将Level 0哈希的计算和传输频率设置为"每帧必检"会显著增加CPU和带宽消耗。对于60帧/秒的游戏，每帧额外的哈希序列化可能消耗0.3-0.5ms，占帧预算的3%-5%。合理做法是区分**生产频率**（每5-10帧检测一次，用于快速发现偏差）与**诊断频率**（每帧检测，仅用于已知问题的精确定位）。

**误区三：找到偏差字段即找到根因**

分层哈希树精确指出"第1523帧，实体42的position.x发生偏差"，但这不代表根因在该实体的移动逻辑。帧同步中数据依赖关系复杂，真正的错误可能发生在第1510帧的一次碰撞检测中，其副作用传播13帧后才体现在position.x上。正确做法是以检测到的偏差帧为终点，**向前回溯**寻找依赖链上最早出现偏差的帧和字段。

## 知识关联

不同步检测以**同步校验**（CRC/哈希生成机制）为直接前提——若校验值本身的计算顺序不确定（如遍历HashMap时的随机顺序），则检测结果会产生误报。在学习不同步检测之前，必须理解状态序列化的字段顺序保证和哈希的选取依据（CRC32速度快但碰撞率约为1/2³²，MD5抗碰撞更强但计算约慢3倍）。

不同步检测是帧同步系统调试链路的最终环节。掌握这一技术后，开发者具备了从"游戏出现异常"到"精确定位到某帧某对象某字段"的完整能力，可进一步深入**确定性物理引擎设计**和**定点数运算替代浮点数**等从根本上消除不同步的预防性技术。
