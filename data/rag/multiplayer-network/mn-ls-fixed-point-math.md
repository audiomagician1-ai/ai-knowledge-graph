# 定点数运算

## 概述

定点数运算（Fixed-Point Arithmetic）是一种用整数模拟小数计算的数学方法，通过人为规定小数点位置，将实数域的计算转化为纯整数操作。在帧同步网络游戏中，定点数运算的核心价值在于：所有平台上相同的整数运算产生**逐位相同（bit-exact）**的结果，从而消除由IEEE 754浮点标准实现差异导致的跨平台数值漂移。

浮点数的跨平台问题由来已久。x86处理器的x87 FPU内部使用80位扩展精度，而SSE2指令集使用严格64位；ARM处理器的VFP单元在flush-to-zero模式下会将非规格化浮点数归零；不同编译器优化级别（如GCC的`-ffast-math`）会重排浮点运算顺序，改变最终结果。这些差异在帧同步游戏中会导致客户端之间的游戏状态在数百帧后完全分叉（desync），而定点数运算从根本上绕过了这一问题。

知名帧同步RTS游戏《星际争霸》系列、《魔兽争霸3》均采用定点数进行物理计算，现代帧同步引擎如Deterministic Lockstep Framework也以定点数数学库作为确定性保证的基础模块。

## 核心原理

### 定点数的表示方法

定点数使用一个整数类型（通常是32位或64位有符号整数）来表示，其中低N位表示小数部分，高位表示整数部分。最常见的格式为 **Q16.16**，即用32位整数存储，其中高16位为整数部分，低16位为小数部分。

真实值与存储值的关系为：

```
真实值 = 存储整数 / 2^N
```

以Q16.16为例，N=16，精度为 `1/65536 ≈ 0.0000153`，可表示的整数范围约为 `-32768` 到 `32767`。游戏物理场景中常用 **Q24.8** 格式（精度1/256≈0.0039）换取更大的整数范围，或使用 **Q32.32**（基于64位整数）兼顾精度与范围。

### 四则运算规则

**加减法**直接对存储整数进行加减，无需额外处理：
```
Q(a + b) = Q(a) + Q(b)
```

**乘法**需要右移N位，否则小数部分被翻倍计数：
```
Q(a × b) = (Q(a) × Q(b)) >> N
```
例如Q16.16中，两个存储值相乘得到一个64位中间结果，再右移16位得到正确的Q16.16结果。为避免精度丢失，必须先提升为64位再做乘法，这是实现中最常见的错误来源。

**除法**需要左移N位后再做整除：
```
Q(a / b) = (Q(a) << N) / Q(b)
```
同样需要先将被除数提升为64位以防止溢出。

### 超越函数的实现

正弦、余弦、平方根等超越函数无法直接用整数运算表达，游戏引擎通常采用两种方案：

1. **查找表（LUT）**：预计算固定步长的函数值存入数组。例如sin函数可预计算360度×精度级别的Q16.16值，运行时直接索引。典型实现使用256或1024个采样点，通过线性插值提高精度。

2. **CORDIC算法**：通过迭代旋转（每次旋转角度为 `arctan(2^-i)`）计算三角函数，全程只使用移位和加减法，无乘法操作。16次迭代可达到约 `5×10^-5` 的精度，满足大多数游戏物理需求。

平方根常用**牛顿迭代法**的定点数版本实现，也可使用"快速反平方根"的定点数变体。

## 实际应用

### 碰撞检测中的定点向量

在帧同步物理引擎中，角色位置、速度向量均以定点数存储。一个典型的2D碰撞点积运算如下：

```cpp
// Q16.16定点数点积
int64_t dot(int32_t ax, int32_t ay, int32_t bx, int32_t by) {
    return ((int64_t)ax * bx + (int64_t)ay * by) >> 16;
}
```

此函数在x86、ARM、WASM平台上输出完全相同的结果，而等价的浮点版本无法保证。

### 子弹轨迹与抛物线

抛体运动中 `y = v0t - 0.5gt²` 需要在定点数下展开。重力加速度 `g` 存储为Q16.16定点数（如 `9.8 × 65536 = 642252`），每帧的速度更新和位置更新均通过定点乘法完成，确保所有客户端的子弹在第1000帧时落点坐标按位相同。

### 定点数数学库选型

开源库**libfixmath**提供完整的Q16.16实现，包含溢出检测；Unity帧同步方案常使用**FixedMath.Net**（Q31.32格式）；自研引擎可参考Godot的`FixedPoint64`实现，其sin/cos误差小于 `2^-14`。

## 常见误区

### 误区一：认为定点数精度低于浮点数

Q16.16格式的最小精度约为 `1.5×10^-5`，单精度float的最小精度约为 `1.2×10^-7`，看似浮点数更精确。但游戏中的精度需求是**确定性一致**而非绝对精度最高——定点数在所有平台上误差相同且可预测，而浮点数的误差因平台而异。两者目标不同，不能直接比较优劣。

### 误区二：乘法时不提升数据类型

初学者常犯的错误是直接用两个`int32_t`相乘后右移，而不先转换为`int64_t`：
```cpp
// 错误：可能溢出
int32_t result = (a * b) >> 16;

// 正确：先扩展为64位
int32_t result = (int32_t)(((int64_t)a * b) >> 16);
```
两个Q16.16值相乘的中间结果最大可达 `32767 × 32767 × 65536²`，远超int32_t的范围（约2.1×10^9），必须用int64_t承接。

### 误区三：直接用浮点数初始化定点数常量

将浮点字面量赋给定点变量会在编译时引入浮点运算：
```cpp
// 危险：pi的定点表示经过浮点中间步骤
int32_t pi = (int32_t)(3.14159265f * 65536); // 编译器行为依赖实现

// 安全：直接指定整数值
int32_t pi = 205887; // 3.14159... × 65536 = 205887.xxx，取整
```
所有物理常量应在设计阶段预先计算好整数值并硬编码。

## 知识关联

定点数运算建立在**确定性物理**的需求之上：确定性物理要求相同输入必须产生相同输出，而定点数是实现这一承诺的数学工具。当确定性物理模块调用位置更新、碰撞响应等计算时，底层全部委托给定点数数学库执行。

定点数运算同时影响帧同步系统中的**状态哈希校验**机制——正因为定点数保证了跨平台一致性，游戏状态的MD5或CRC32哈希值才能用于检测客户端间的状态分叉（desync detection）。若使用浮点数，即使游戏逻辑完全相同，不同硬件上的哈希值也会因最低有效位差异而不同，使校验机制完全失效。掌握定点数的实现细节，是构建可靠帧同步网络游戏不可跳过的工程基础。
