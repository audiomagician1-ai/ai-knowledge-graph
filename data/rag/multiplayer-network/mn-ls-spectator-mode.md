# 旁观者模式

## 概述

旁观者模式（Observer Mode，俗称OB系统）是帧同步网络游戏中允许非参与玩家实时观看对局的技术实现方案。与录像回放不同，旁观者模式要求观战客户端以接近零延迟的状态跟随正在进行的对局，观战者能看到当前帧的游戏状态，而不是历史存档的重播。《星际争霸》、《DOTA 2》和《王者荣耀》等竞技游戏均以OB系统作为电竞赛事直播的核心基础设施。

帧同步OB系统的起源可追溯到RTS游戏时代。早期《魔兽争霸3》的观战功能本质上是将观战客户端作为一个"隐形玩家"加入对局，服务器向其同步所有玩家的操作指令（即关键帧），观战客户端本地执行同样的确定性逻辑并渲染结果。这一设计思路至今仍是帧同步OB系统的主流架构：**观战者只需接收操作流，无需接收游戏状态快照**，因为帧同步的确定性保证了本地推演结果与参战玩家完全一致。

OB系统在电竞场景中的意义不仅限于观看体验，它还直接影响反作弊能力。由于观战视角拥有完整的全局信息（包括迷雾之战下被遮挡的单位），若OB数据被对局参与者获取，则构成严重的信息泄露漏洞。因此，OB系统必须处理延迟转发与视角权限两个关键问题。

## 核心原理

### 延迟缓冲机制

帧同步OB系统通常引入固定的**观战延迟（Spectate Delay）**，以防止观战者将当前信息传递给对局参与者（即防止"间谍观战"作弊）。主流竞技游戏的观战延迟设置在 **2～5分钟**（约7200～18000帧，以60帧/秒计算）。服务器维护一个指令环形缓冲区，对局产生的每一帧操作包均被暂存，观战客户端订阅的是延迟 D 帧之前的操作序列：

```
观战帧号 = 当前对局帧号 - DelayFrames
```

观战客户端从帧0开始执行本地帧同步逻辑，当本地推演帧追上可消费的最新帧时进入等待。这与回放系统的快速追帧逻辑一致，但OB模式下追帧速度受网络实时推送速率约束，而不能无限制加速。

### 观战客户端的初始化同步

当观战者中途加入正在进行的对局时，无法从第0帧开始重演——若对局已进行20分钟（约72000帧），从头推演需要数秒甚至数十秒的追帧时间。解决方案是**状态快照（State Snapshot）加速追帧**：服务器每隔固定间隔（通常每300～600帧）保存一次完整的游戏状态序列化快照，新加入的观战客户端下载最近一次快照后直接恢复状态，再补充执行快照到当前观战帧之间的操作指令即可。这正是旁观者模式对回放系统中"关键帧快照"技术的直接复用。

### 全知视角与雾战处理

参战玩家的帧同步客户端仅渲染己方视野内的单位，但观战者需要同时看到所有玩家的视野（全知视角）或选择跟随特定玩家视角（主观视角）。在帧同步架构中，所有单位的逻辑状态始终在本地完整推演（即所谓"逻辑层与渲染层分离"），雾战仅作用于渲染过滤层。因此观战客户端只需关闭渲染层的雾战遮罩，即可展示全局信息，无需从服务器额外下载数据——这是帧同步相对于状态同步在OB实现上的显著优势。

### 多路观战与服务器带宽

在电竞赛事直播场景下，可能有数千名观战者同时订阅同一场对局的操作流。为避免服务器对每个观战连接单独推送，通常采用**OB中继服务器（Relay Server）**拓扑：对局服务器仅向少量中继节点推送一份操作流，中继节点负责扇出分发给末端观战客户端。单场对局的操作流数据量极低（通常每帧仅几十字节），64路中继扇出可支持数万并发观战者。

## 实际应用

**《DOTA 2》的OB实现**采用了5分钟观战延迟作为默认配置，同时提供"真实时间观战"选项供受信任的直播人员使用（延迟降至约2分钟）。其观战客户端在加入时通过下载约1～2 MB的状态快照文件恢复现场，随后以常规速率接收后续帧操作包。

**《王者荣耀》**的战队赛OB系统中，由于移动端带宽受限，操作包采用差量压缩编码，单帧操作包压缩后通常不超过50字节，使得5G网络下同时支持百万级观战并发成为可能。

在独立游戏开发中，简化的OB实现方案是将观战者视为一个"无操作玩家"接入帧同步房间，服务器对其发送完整操作流但忽略其空操作。这种方案实现成本极低，缺点是观战者数量会影响帧同步的锁步等待逻辑，需要将观战连接排除在帧推进的等待列表之外。

## 常见误区

**误区一：OB系统需要服务器推送游戏状态而非操作指令**。部分开发者认为观战者需要持续接收单位坐标、血量等状态数据。实际上在帧同步架构中，观战客户端接收的仍然是玩家操作指令流，所有状态均在本地推演得出。只有在初始化加速追帧时才需要一次性的状态快照，此后与正常帧同步参与者接收的数据格式完全相同。

**误区二：移除观战延迟对安全性影响可忽略**。部分实现为了降低技术复杂度省去了延迟缓冲。在休闲游戏中这或许可接受，但在竞技对抗场景下，即使2分钟的观战延迟也能有效遏制通过"战队OB成员实时语音指挥"的作弊手段。职业电竞赛事通常要求不低于3分钟的强制延迟，这是行业规范而非技术建议。

**误区三：全知视角需要服务端额外计算视野**。由于帧同步客户端本地已推演所有单位的完整逻辑状态（包括被迷雾遮挡的敌方单位），OB全知视角只需在渲染层取消视野过滤，无需任何服务端改动。若开发者发现自己的OB需要服务端发送额外的"全局状态数据"，通常意味着该游戏的帧同步实现存在逻辑层与渲染层未分离的架构问题。

## 知识关联

旁观者模式直接依赖**回放系统**的两项核心技术：操作指令录制存储格式，以及关键帧快照的序列化与反序列化机制。OB系统本质上是"实时流式回放"——回放系统处理的是离线存档文件，而OB系统处理的是通过网络实时推送的相同数据流。掌握回放系统中快速追帧（以2～4倍速推演历史帧）的实现后，OB系统的追帧初始化逻辑可以直接复用相同代码路径，区别仅在于追帧的目标帧号来源于网络推送而非文件末尾。

在帧同步体系中，OB系统也是验证确定性实现正确性的隐性测试工具：若OB客户端与参战客户端推演结果出现偏差（例如单位位置不一致），则必然存在非确定性逻辑缺陷，这使OB系统兼具功能性与调试价值。
