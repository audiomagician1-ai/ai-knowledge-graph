# 旁观者模式

## 概述

旁观者模式（Observer Mode，通称OB模式）是帧同步多人游戏中允许第三方玩家实时观看正在进行对局的功能系统。与录像回放不同，旁观者模式要求观察者以接近零延迟（实际工程中为固定人工延迟）的方式跟随当前游戏状态，而非播放已存储的历史帧文件。其技术本质是将目标对局的操作帧输入流实时分发给旁观客户端，旁观客户端在本地独立执行相同的确定性逻辑，从而在自身设备上重建与参战玩家完全一致的游戏世界状态。

旁观系统最早在《星际争霸》（StarCraft，Blizzard，1998）的 Battle.net 对战平台中被大规模工程化：Battle.net 对战房间允许玩家以"observer"席位加入，最多支持 8 名旁观者同时在线，这一设计直接奠定了电竞直播的技术雏形。在帧同步架构兴起之后，旁观模式与该架构的天然契合性变得尤为突出——帧同步本身保证了"相同初始状态 + 相同输入序列 = 相同输出状态"，旁观客户端只需接收与参战玩家完全相同的操作帧序列，便可自行在本地推演出完整的游戏世界，无需服务器额外推送全量状态快照。

相比状态同步架构中旁观需要服务器持续广播完整场景状态（每帧数据量通常为 10–50 KB），帧同步旁观系统的每帧带宽开销与增加一名普通对战玩家几乎相当（通常仅为 100–500 B/帧）。当赛事直播场景下同时存在数百名旁观者时，两种架构在服务器出口带宽消耗上可产生 20–100 倍的量级差异。

## 核心原理

### 帧数据分发与追帧机制

帧同步旁观的关键约束是：旁观客户端必须获取**从第 0 帧开始的完整操作帧序列**，才能将本地游戏状态推演至与当前对局一致。当旁观者在第 $F_{join}$ 帧时进入一场正在进行的对局，服务器需要向该客户端补发第 $0$ 帧至第 $F_{join}-1$ 帧的历史帧数据，客户端随后以加速模拟（Fast-Forward）快速追上当前帧。

追帧过程中，逻辑帧推进速率 $R_{ff}$ 与实际物理帧率 $R_{render}$ 的关系为：

$$
R_{ff} = k \times R_{render}, \quad k \in [5, 20]
$$

典型工程实践中 $k$ 取 10，即以 10 倍速纯逻辑模拟追帧，渲染线程在此阶段完全暂停，仅展示进度条 UI。追帧完成后客户端切换为正常帧率继续接收实时帧流。以《王者荣耀》20 fps 的逻辑帧率为例，若旁观者在第 1200 帧（即对局开始后第 60 秒）入场，以 $k=10$ 追帧只需约 6 秒的本地 CPU 时间即可完成追帧。

### 延迟设计与防剧透机制

旁观者画面通常会人为引入固定延迟 $D_{obs}$，典型值为 **180 秒（3 分钟）**，部分竞技场景下为 **300 秒（5 分钟）**。该延迟服务于两个独立目标：

1. **竞技公平性（Anti-Ghosting）**：防止场外人员通过旁观视角实时向参战选手传递敌方情报。在职业赛事规则（如 LCK、LPL 等联赛技术规范）中，此延迟是强制要求，违规可判负。
2. **网络缓冲（Jitter Buffer）**：为旁观端提供足够的帧序列缓冲窗口，确保播放时帧序列已连续完整地到达，消除网络抖动导致的卡顿。

延迟的实现并非在网络层丢弃数据包，而是在客户端维护一个**帧回放缓冲队列**：旁观客户端完成追帧后将收到的实时帧压入队列，但逻辑层仅消费时间戳满足 $t_{frame} \leq t_{now} - D_{obs}$ 的帧，其余帧在队列中等待。

```python
# 旁观端帧缓冲队列消费逻辑（伪代码）
def consume_ob_frame(frame_queue, delay_seconds=180):
    current_time = get_server_time()
    while frame_queue:
        frame = frame_queue.peek()
        # 仅消费时间戳早于 (当前时间 - 旁观延迟) 的帧
        if frame.timestamp <= current_time - delay_seconds:
            execute_logic_frame(frame_queue.pop())
        else:
            break  # 剩余帧尚未到期，停止消费
```

### 旁观视角控制与全图视野

旁观者不参与游戏逻辑，其输入操作仅影响本地摄像机，不产生任何操作帧上传到服务器。在代码层面，旁观客户端运行完整游戏逻辑但持有一个 `isObserver = true` 标志位，该标志位绕过战争迷雾（Fog of War）的视野裁剪逻辑，使旁观者拥有全图视野。

以《王者荣耀》的英雄跟随视角为例：逻辑层以环形缓冲区记录每个单位过去 $D_{obs}$ 秒内每逻辑帧的世界坐标，旁观摄像机在本地读取目标英雄在**当前旁观帧时刻**对应的历史坐标进行平滑跟随，而非直接读取单位的当前坐标（因当前坐标对应的是比旁观画面超前 3 分钟的真实状态）。

### 与回放系统的代码复用

旁观模式与回放系统（Replay System）共享同一套**帧序列消费引擎**。两者的唯一本质差异在于数据来源：

| 特性 | 回放系统 | 旁观模式 |
|---|---|---|
| 数据来源 | 本地磁盘 `.replay` 文件 | 网络实时帧流 |
| 起始帧 | 第 0 帧（文件头） | 第 0 帧（服务器历史帧补发） |
| 播放速度控制 | 支持 0.5×/1×/2×/4× | 固定 1×（追帧阶段除外） |
| 跳帧 | 支持（结合快照） | 不支持（必须顺序消费） |

因此，许多团队将旁观客户端设计为"以网络流为数据源的回放播放器"，代码层仅通过依赖注入切换 `IFrameDataSource` 接口的实现，二者逻辑层代码复用率可达 90% 以上（Glenn Fiedler 在其网络编程系列文章中将这一模式称为"deterministic lockstep replay"，2018）。

## 关键公式与数据指标

旁观系统的服务端需要为每名旁观者维护一条独立的帧流发送管道，服务端总出口带宽 $B_{total}$ 的估算公式为：

$$
B_{total} = N_{players} \times b_{player} + N_{observers} \times b_{obs}
$$

其中 $b_{player}$ 为单名参战玩家的每秒帧数据量（典型值 2–8 KB/s），$b_{obs}$ 为单名旁观者的每秒帧数据量（与 $b_{player}$ 量级相同，约 2–8 KB/s）。在《英雄联盟》的观战系统架构中（Riot Engineering Blog，2012），Riot 将观战流量路由至独立的"Spectator Service"节点，使 $N_{observers}$ 的增长完全不影响对战服务器的 $B_{total}$ 计算。

追帧所需时间 $T_{catchup}$ 可估算为：

$$
T_{catchup} = \frac{F_{join}}{R_{render} \times k}
$$

例如，对局进行到第 3600 帧（30 fps 下即对局开始后 120 秒），旁观者以 $k=10$ 追帧，则 $T_{catchup} = 3600 / (30 \times 10) = 12$ 秒。

## 实际应用案例

**《英雄联盟》观战系统（Riot Games，2012）**：Riot 在 2012 年 S2 赛季引入"Spectator Mode"，采用固定 3 分钟延迟策略，并部署专用的 Spectator Service 集群将观战流量与对战流量物理隔离。观战端接收的帧数据格式与对战客户端完全一致，服务端无需进行任何格式转换，极大降低了维护成本。

**《绝地求生》（PUBG，2018）**：PUBG 的旁观系统需要处理多达 100 名参战玩家的输入帧，每局对局的历史帧数据体积可达数十 MB。服务端采用分片（Chunked）传输方式，将历史帧压缩为 16 帧一组的数据块，旁观客户端并行下载多个数据块后顺序执行，将追帧等待时间压缩至 5 秒以内。

**电竞赛事导播台**：职业赛事导播台接入旁观数据后，可在本地运行一个拥有全图视野的旁观客户端，导播通过自定义摄像机脚本实现自由飞行视角、英雄特写、技能特效回放等演出效果。导播端可选择将旁观延迟降低至 0（赛事现场封闭网络环境下无防剧透需求），但需在赛事规则中明确约定参赛选手无法接触赛场外的观众屏幕。

## 常见误区

**误区一：旁观客户端可以随意从任意帧开始接收数据**
实际上，帧同步旁观客户端必须从第 0 帧开始消费完整的操作帧序列，中间任何一帧的缺失都会导致本地游戏状态与服务端永久性偏离（逻辑分叉）。唯一的例外是引入了**状态快照（Snapshot）机制**的混合方案：服务端周期性（如每 300 帧）存储一份完整的游戏状态快照，旁观者可从最近的快照帧开始恢复状态，再从该帧继续消费后续操作帧，从而避免从第 0 帧开始漫长的追帧过程。

**误区二：旁观模式中旁观者必须与游戏服务器直连**
旁观帧流只是只读的数据广播，不需要旁观者与对战服务器进行双向通信。工程实践中通常在对战服务器前部署一层**CDN 式帧流分发节点**（Frame Relay Node），对战服务器仅向该中继节点推送一路帧流，中继节点再向所有旁观客户端做 1:N 广播，将对战服务器的旁观相关压力降低到常数级。

**误区三：旁观延迟越短越好**
对于非竞技场景（如观看好友的休闲对局），低延迟旁观体验更好；但在职业赛事场景下，过短的延迟会产生严重的竞技公平性风险。《英雄联盟》职业赛联赛规则明确要求观战客户端延迟不低于 180 秒，选手席位与观众席之间的屏幕信号延迟也受物理隔离约束。
