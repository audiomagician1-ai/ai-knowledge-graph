# 版本管理

## 概述

版本管理（Version Management）是网络多人游戏热更新系统的基础机制，负责追踪客户端与服务端的代码、资源和配置所处的迭代状态，并在连接建立阶段执行兼容性检查以决定是否允许玩家进入游戏。与通用软件版本管理不同，游戏领域的版本管理需要同时处理二进制大包（通过平台商店发布）和热更新资源包（通过CDN下发）两条并行的版本轨道，因此版本号体系通常比普通应用复杂。

游戏行业普遍采用三段式版本号 `Major.Minor.Patch`（如 `1.4.23`），其中 Major 代表大版本（通常需过审的商店包变更），Minor 代表功能性热更新，Patch 代表仅修复缺陷的小补丁。服务端维护一个"最低兼容版本"字段（MinSupportedVersion），当客户端版本低于该值时，服务端拒绝建立游戏会话，迫使客户端触发强制更新流程。这一机制直接保障了多人游戏中所有在线玩家共享一致的战斗逻辑和数值表现。

版本管理之所以在多人游戏中至关重要，是因为版本不一致会引发反外挂失效、技能判定错误、存档格式不匹配等严重问题。2019年某款热门手游因热更新版本校验逻辑存在漏洞，导致部分玩家在旧版本下仍能进入战局，最终造成跨版本技能穿模的大规模事故，直接推动了行业对版本握手协议的重新规范。

## 核心原理

### 版本号的结构设计

游戏版本号通常由以下四个字段组成：

```
AppVersion   : 商店包版本，如 "2.1.0"（提交平台审核的版本）
ResVersion   : 热更资源版本，如 "2.1.0-r147"（CDN上资源包的编号）
ProtocolVersion: 网络协议版本，如 "17"（决定消息格式是否兼容）
ConfigVersion: 配置表版本，如 "c20240315"（控制数值与关卡数据）
```

`AppVersion` 受平台商店约束，每次变更需重新走审核流程（iOS 约 1-3 天，Google Play 约数小时至 1 天）；而 `ResVersion` 可通过CDN随时热推，不需过审。两套版本独立演进，服务端需要同时维护对这两个维度的兼容性判断。

### 握手阶段的版本检查流程

客户端在连接服务端时，握手包（Handshake Packet）中必须携带上述所有版本字段。服务端执行如下判断逻辑：

1. **协议版本检查**：若 `ProtocolVersion` 不匹配，立即断开连接并返回错误码 `ERR_PROTOCOL_MISMATCH`，此时客户端无法通过热更解决问题，必须引导用户重新下载安装包。
2. **最低App版本检查**：对比客户端 `AppVersion` 与服务端配置的 `MinSupportedAppVersion`。若低于最低值，返回 `ERR_APP_TOO_OLD`，触发强制更新提示。
3. **资源版本检查**：对比 `ResVersion` 与服务端下发的 `LatestResVersion`。若不匹配，客户端需先从CDN拉取差量补丁后重连。
4. **配置版本检查**：若 `ConfigVersion` 落后，服务端可选择拒绝连接（严格模式）或在线推送配置差量（宽松模式）。

整个握手过程一般在 TCP 三次握手完成后的第一个业务数据包中完成，延迟控制目标为 50ms 以内。

### 版本兼容性矩阵

服务端通常维护一张兼容性矩阵表，记录哪些历史客户端版本仍被支持：

| AppVersion | 是否兼容 | 备注 |
|---|---|---|
| 2.1.x | ✅ 完全兼容 | 当前主流版本 |
| 2.0.x | ✅ 有限兼容 | 禁止进入排位赛 |
| 1.9.x | ❌ 不兼容 | 协议版本差异过大 |

这张表由运营后台动态配置，无需服务端重新部署。当新版本上线后，运营人员根据实际更新率（通常目标是 3 天内覆盖 95% 活跃用户）决定何时将旧版本标记为不兼容，从而关闭灰度兼容窗口。

## 实际应用

**启动时的版本自检流程**：客户端每次启动后，首先向版本检查接口（通常是一个轻量级 HTTP GET 请求，如 `GET /api/version/check?appver=2.1.0&resver=r147`）查询最新版本信息。服务端返回 JSON 包含 `latestResVersion`、`minAppVersion`、`forceUpdateUrl` 等字段。客户端据此判断是静默热更（后台下载资源包）、强制热更（阻塞进入游戏）还是跳转商店（无法热更）。

**灰度发布中的版本管理**：在大版本灰度期间，服务端同时运行 v2.0 和 v2.1 两套逻辑分支。客户端握手包中携带的 `AppVersion` 用于路由到对应的逻辑服务器集群，确保 v2.0 用户不会与 v2.1 用户匹配到同一局游戏中。《王者荣耀》等头部游戏在灰度期间通常按 UID 尾号分批放量，如先对 UID 末位为 0-1 的用户开放新版本，观察稳定性后再逐步扩大比例。

**版本号持久化存储**：客户端需要将 `ResVersion` 持久化写入本地文件（如 `version.json`），每次启动后读取并与服务端对比，而非每次重新计算本地资源哈希（后者耗时可达数十秒）。若本地 `version.json` 损坏或丢失，则触发全量资源校验流程。

## 常见误区

**误区一：认为热更版本号只需要一个数字递增即可**。实际上，热更资源版本（ResVersion）需要与AppVersion绑定，不能跨大版本复用。例如 v2.0 的 `r200` 号资源包与 v2.1 的 `r200` 号资源包内容完全不同，若混用会导致资源加载到错误的文件，表现为贴图错乱或脚本执行异常。正确做法是在大版本号变更时重置热更版本计数器，或采用全局递增方案并在服务端维护版本到AppVersion的映射关系。

**误区二：认为协议版本（ProtocolVersion）与AppVersion同步更新**。事实上两者变更频率完全不同。AppVersion 每次过审都会变化，但 ProtocolVersion 只在网络消息结构（如 Protobuf 定义文件）发生破坏性变更时才递增。一个 AppVersion 跨度内可能多次热更资源但 ProtocolVersion 始终不变，也可能一次版本迭代同时升级了 ProtocolVersion。开发者若将两者混同，会造成不必要的强制重装或错误地允许协议不兼容的客户端连接。

**误区三：把"服务端版本"理解为单一值**。实际上，多人游戏服务端在灰度发布期间同时存在多个活跃版本。服务端自身也有版本管理需求，需要记录每个逻辑服分片（Shard）运行的代码版本，以便在出现问题时快速定位是哪个批次的部署引入了缺陷，这与客户端的版本管理是两个并行但密切关联的体系。

## 知识关联

掌握版本管理的版本号结构和握手检查流程后，可以直接进入**强制更新策略**的学习——后者专注于当版本不兼容时如何设计用户体验流程（如弹窗文案、深度链接跳转商店、下载进度展示等），是版本检查失败后的处理分支。

**补丁系统**在版本管理的基础上解决"如何高效传输两个 ResVersion 之间的差量"，其核心输入正是版本管理中记录的当前版本号与目标版本号，差量计算依赖对版本历史的完整追踪。

**版本回滚**是版本管理的逆向操作，要求版本管理系统除了记录"当前是什么版本"，还必须保留历史版本的完整元数据（文件列表、哈希、CDN地址），以支持在新版本出现严重缺陷时将服务端快速切回旧的 MinSupportedVersion。

**平台商店适配**则涉及 AppVersion 字段在 iOS（使用 `CFBundleShortVersionString`）和 Android（使用 `versionCode` 整数 + `versionName` 字符串）两个平台的不同存储格式，要求客户端在上报握手包前先将平台原生版本号转换为统一格式。
