# 负载均衡

## 概述

负载均衡（Load Balancing）是指将来自客户端的网络请求或玩家连接，按照特定策略分配到多台后端服务器上处理的技术机制。在多人游戏服务端架构中，负载均衡的目标是避免单台游戏服务器因承载过多玩家而出现帧率下降、逻辑延迟积压或连接超时等问题，同时让空闲服务器得到充分利用。

负载均衡的概念最早在1990年代随着互联网商业化而出现，初期主要用于HTTP Web服务。游戏行业大规模应用负载均衡始于2000年代的MMORPG热潮，《魔兽世界》（2004年发布）在服务端架构中引入了分区（Realm）机制，本质上是一种手动负载均衡——将玩家预先分配到不同的服务器集群，以控制每台服务器上的并发人数上限（通常为数千到两万人）。

在现代多人游戏中，尤其是百人同服的大逃杀或MMO副本模式下，单场游戏实例可能需要处理100至数百名玩家的实时状态同步，负载均衡直接决定了玩家能否获得稳定的Tick Rate（游戏服务器每秒处理逻辑的次数）。若某台服务器超载，Tick Rate从64Hz跌至20Hz以下，玩家会明显感受到"橡皮筋"卡顿现象。

## 核心原理

### 常见负载均衡算法及其适用场景

游戏服务端最常用的四种分配策略如下：

**轮询（Round Robin）**：将新连接请求依次分配给每台服务器。实现简单，适合无状态服务（如登录验证服务器），但无法感知各服务器当前的玩家数量差异，容易在服务器性能不均时造成局部过载。

**加权轮询（Weighted Round Robin）**：为不同配置的服务器赋予权重值，例如16核服务器权重为2、8核服务器权重为1，则前者分配到两倍的玩家连接。这在混合配置的服务器集群中比纯轮询更精准。

**最少连接数（Least Connections）**：每次将新玩家分配给当前活跃连接数最少的服务器。对于游戏大厅、匹配等待室等连接时长参差不齐的场景效果显著，因为玩家在大厅中停留时间从几秒到数小时不等，单纯轮询无法反映这种不平衡。

**基于负载的动态分配（Resource-Aware Balancing）**：负载均衡器主动采集各游戏服务器的CPU占用率、内存使用量、当前玩家数等指标，将新玩家路由到综合负载最低的节点。《英雄联盟》的游戏服务器调度系统使用了类似机制，每隔数秒向负载均衡层上报实例状态。

### 四层与七层负载均衡的区别

**四层负载均衡**工作在传输层（TCP/UDP），只根据IP地址和端口号进行转发，不解析游戏数据包内容。延迟极低（微秒级转发），适合对实时性要求极高的游戏战斗数据流。主流方案包括LVS（Linux Virtual Server）和硬件F5设备。

**七层负载均衡**工作在应用层，可以读取HTTP/WebSocket头或自定义协议字段，实现基于玩家ID、地理位置、游戏模式的智能路由。代价是每个数据包需要经过完整协议解析，引入约0.5~2ms的额外延迟。Nginx和HAProxy是典型的七层负载均衡软件。游戏登录、匹配服务等低频请求适合七层；战斗同步数据包应尽量走四层。

### 会话保持（Session Persistence）

游戏负载均衡中必须处理的核心挑战是**会话保持**（又称粘性会话，Sticky Session）：一名玩家一旦连接到某台游戏服务器，其后续所有数据包必须路由到同一台服务器，因为该服务器持有该玩家的实时状态（位置、血量、技能冷却等）。

实现会话保持的常用方法是**IP哈希（IP Hash）**：对玩家IP地址执行哈希运算，将结果映射到固定服务器。公式为：

```
server_index = hash(client_ip) % N
```

其中 `N` 为当前可用服务器总数。此方法的缺陷是当服务器数量变化（新增或宕机）时，大量玩家的映射关系会被打乱，需要配合**一致性哈希（Consistent Hashing）**算法来最小化重新分配的玩家比例——理论上服务器数量变化时，只有 `K/N` 比例的映射需要改变（K为总玩家数，N为服务器数）。

## 实际应用

**匹配阶段的负载均衡**：玩家在游戏大厅点击"开始匹配"时，请求首先到达负载均衡层，被分配至负载最低的匹配服务器节点。《PUBG》在峰值时期同时运行数十个匹配服务实例，每个实例负责处理特定大区和游戏模式组合的排队请求。

**游戏房间创建的分配**：匹配成功后，系统需要为一局游戏分配一台专用游戏服务器（Dedicated Server）。此时负载均衡器根据各节点的CPU余量和当前房间数决定在哪台物理机或虚拟机上启动新的游戏进程。《Fortnite》使用AWS云服务，通过负载均衡触发EC2实例的按需启动，单次分配决策需要在200ms内完成，否则玩家感知到排队延迟。

**跨数据中心的地理负载均衡**：腾讯、网易等国内大型游戏公司通常在华东、华南、华北部署多个数据中心。DNS解析层根据玩家IP所属运营商和地区，将玩家请求定向至延迟最低的数据中心，这属于全局负载均衡（GSLB，Global Server Load Balancing）。

## 常见误区

**误区一：负载均衡等同于简单的轮询分配**
许多初学者认为负载均衡只是"依次给每台服务器分一个玩家"。实际上轮询仅是最基础的一种算法，且在游戏场景中效果较差——因为不同游戏模式（如100人大逃杀 vs 5v5小队）对服务器资源的消耗差距悬殊，轮询无法感知这种差异。生产环境中通常使用基于实时负载指标的动态分配策略。

**误区二：负载均衡可以完全消除服务器过载**
负载均衡能分散压力，但不能凭空创造服务器容量。当全服在线人数超过总承载上限时，即使分配再均衡，每台服务器依然会过载。此时需要结合**自动扩缩容**机制，动态增加服务器实例数量才能从根本上解决问题。将负载均衡误认为是无限扩展能力的手段，是架构设计中的危险认知。

**误区三：游戏战斗数据可以随意切换服务器**
某些开发者认为，当某台服务器过载时，可以实时将玩家"迁移"到另一台服务器来降低负载。实际上游戏战斗中玩家状态极为细碎（每帧的位置、动画状态、buff列表等），在不中断连接的情况下进行热迁移（Live Migration）技术难度极高，通常只在玩家处于非战斗状态（如城镇、大厅）时才可行。战斗中的负载分配必须在**游戏开始前**由负载均衡层一次性决定。

## 知识关联

**前置概念**：理解负载均衡不需要特定前置知识，但熟悉TCP/UDP协议的基本连接模型（如三次握手、UDP无连接特性）有助于理解四层负载均衡的工作机制，以及为什么实时游戏数据倾向于选择UDP而非TCP。

**网关服务器**：负载均衡层通常与网关服务器紧密配合——网关服务器是玩家连接的第一个节点，负责鉴权和协议转换，而负载均衡决定鉴权通过后的玩家应被路由至哪个后端游戏服务器。两者共同构成游戏服务端的流量入口层。

**自动扩缩容**：负载均衡与自动扩缩容是互补关系。当负载均衡器监测到所有服务器的平均CPU占用率持续超过80%时，可以触发扩容信号，自动启动新的服务器实例；新实例就绪后，负载均衡器自动将其纳入分配池，承接新玩家流量。

**延迟优先匹配**：负载均衡在进行玩家分配时，除了考虑服务器负载，还需要与延迟优先匹配策略协同——优先将地理位置相近、到同一服务器延迟相似的玩家分配至同一台游戏服务器，以保证同局玩家的网络条件尽量均等，避免因跨地区混服导致部分玩家出现高延迟劣势。
