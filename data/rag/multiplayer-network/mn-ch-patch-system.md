# 补丁系统

## 概述

补丁系统（Patch System）是网络多人游戏热更新流程中负责生成、分发和应用增量差异文件的机制。其核心思想是通过比较新旧版本之间的二进制或文本差异，生成体积远小于完整版本的补丁文件（diff patch），用户只需下载并应用该补丁即可将本地客户端升级到最新版本，而无需重新下载整个游戏包。

补丁系统的算法基础可追溯至1984年Walter Tichy提出的RCS（Revision Control System）差分方法，以及后来被广泛采用的bsdiff算法（Colin Percival，2003年发布）。bsdiff专门针对二进制可执行文件的差异优化，能将同一程序不同版本之间的补丁体积压缩至旧文件大小的约20%以内，这一特性使其成为游戏客户端补丁生成的主流选择。

对于日活百万级别的网络游戏而言，补丁系统的效率直接影响版本更新期间服务器带宽成本与玩家等待时间。若一次版本迭代涉及5GB的完整客户端，而通过增量补丁仅需分发200MB的差异数据，则在100万用户同时更新的场景下，流量节省幅度高达4.8PB，CDN分发成本随之大幅降低。

## 核心原理

### 差异算法：bsdiff 与 xdelta3

主流游戏补丁系统使用两种差异算法：**bsdiff** 和 **xdelta3**。

bsdiff的工作分为两阶段：首先通过后缀排序（Suffix Sorting）对旧文件建立索引，然后将新旧文件的差异拆分为三部分写入补丁包：
- `ctrl` 块：记录复制指令和偏移量
- `diff` 块：记录字节级差异数据
- `extra` 块：记录新文件中全新增加的内容

三部分均经过bzip2压缩后合并为最终补丁文件。时间复杂度为 O(n log n)，其中 n 为旧文件大小，内存占用约为旧文件的5倍。

xdelta3则基于VCDIFF标准（RFC 3284），更适合处理体积超过1GB的大型资源包，因为它采用流式处理，内存占用固定在约32MB，与文件大小无关。

### 补丁包的分块与版本链

单版本补丁（V1→V2）能解决相邻版本升级，但游戏运营中用户可能停留在任意历史版本。为此，补丁系统需要维护一张**版本补丁链表**：

```
V1 → V2 → V3 → V4（当前版本）
```

为避免每次更新都需要生成从每个历史版本到最新版本的全量补丁（版本数为N时，补丁数量为N×(N-1)/2），实际工程中通常采用**基准版本+增量链**策略：保留距当前版本6个迭代以内的补丁，超过6个版本差距的用户强制下载全量包。这一阈值需要在CDN存储成本与用户体验之间权衡确定。

### 补丁完整性校验

补丁应用后必须通过校验确认本地文件与目标版本完全一致。标准做法是在补丁文件头部嵌入目标版本所有文件的**SHA-256哈希清单**。客户端应用补丁后逐文件比对哈希值，任何文件校验失败都触发该文件的单独重下载流程，而非回滚整个补丁。SHA-256的抗碰撞性确保即使有1字节的修改也会被检测到。

## 实际应用

### 分级补丁分发策略

大型多人游戏通常将资源分为三类，对应不同的补丁处理策略：

- **可执行文件（.exe/.dll）**：使用bsdiff生成补丁，体积通常为原文件的5%~15%
- **资源包文件（.pak/.bundle）**：因包内文件打包方式固定，先解包再对单个资源文件做diff，最后重新打包，效率优于直接对整个包做diff
- **配置/脚本文件（.json/.lua）**：直接使用文本diff（unified diff格式），补丁体积极小且人类可读，便于QA审核

### 《英雄联盟》的 Maestro 补丁系统

Riot Games公开的Maestro系统是游戏行业补丁工程的代表案例。它将整个客户端切分为数千个独立文件块（chunk），每个chunk独立计算版本哈希。更新时客户端只需下载哈希发生变化的chunk，而非对整个版本做全量diff。这一设计将主要服务器节点的更新带宽峰值从单次版本发布时的数百Gbps降低到相对平稳的水平，同时支持断点续传，因为chunk是最小的下载原子单位。

## 常见误区

### 误区一：补丁越小越好，应始终追求最小化差异

bsdiff在比较两个高度相似的二进制文件时确实能生成极小的补丁，但若开发工具链修改了编译优化选项或链接顺序，生成的可执行文件二进制布局会发生大幅变化，导致bsdiff无法找到有效的相似块，最终生成的补丁甚至大于原文件本身。此时应改用xdelta3或直接分发全量替换文件，而非盲目坚持增量补丁策略。判断标准通常是：若补丁大小超过目标文件大小的70%，全量替换更为经济。

### 误区二：补丁应用是可逆操作，可以随时回滚

增量补丁的应用过程是**单向的**：bsdiff生成的`.patch`文件依赖原始旧文件才能还原出新文件，如果本地旧文件在补丁应用过程中损坏或部分修改，则无法完成应用，也无法通过该补丁还原旧文件。游戏补丁系统通常不内置回滚机制，版本回退需要单独维护回退补丁（V_new→V_old）或全量历史包，这是与代码版本控制系统（Git等）的重要区别。

### 误区三：哈希校验通过即代表补丁正确应用

SHA-256校验只能验证文件内容一致性，无法检测文件权限、时间戳或文件路径等元数据问题。在Windows平台，若补丁工具未正确处理文件锁（游戏进程仍在运行时更新.dll文件），文件内容可能已替换成功但下次启动时因权限或缓存问题仍加载旧版本。因此，补丁系统在校验后还需要强制验证关键可执行文件的加载版本号（通常嵌入在文件头的Version Resource字段中）。

## 知识关联

补丁系统的正常运转以**版本管理**为前提：版本号体系（通常采用语义化版本SemVer或游戏业务中常见的`主版本.年份.周次`格式）决定了补丁链的构建逻辑，每个版本对应的文件清单哈希表是生成增量diff的基准数据。没有明确的版本标识，补丁生成工具无法确定新旧文件的对应关系。

在补丁系统的基础上，**热修复系统（Hotfix System）**进一步解决了更紧迫的需求：当线上出现严重BUG需要在数分钟内完成修复时，传统补丁流程（生成→上传CDN→触发客户端更新）的延迟不可接受。热修复系统通常绕过完整的补丁链，直接通过内存补丁或脚本层注入修复代码，这要求游戏逻辑层与引擎层必须有明确的解耦边界——而这一边界的划分策略，正是从补丁系统的资源分类实践中演化而来。
