# 内存扫描防护

## 概述

内存扫描防护（Memory Scan Protection）是反作弊系统中用于阻止外部进程读取或修改游戏运行时内存数据的技术体系。其核心目标是防止作弊工具通过 `ReadProcessMemory` / `WriteProcessMemory` 等 Windows API，或 Linux 下的 `/proc/[pid]/mem` 接口，直接获取血量、弹药数量、玩家坐标等敏感游戏状态。

内存扫描类作弊的历史可追溯至 1990 年代的单机外挂工具（如 Game Genie），但在多人在线游戏普及后演变为严重威胁。2003 年 Valve 发布 VAC（Valve Anti-Cheat）初版时，针对 Counter-Strike 的内存修改挂正是最主要的整治目标之一。VAC 记录了早期通过哈希扫描游戏进程内存页来检测注入模块的工程实践，这一思路至今仍是许多商业反作弊引擎的基础层。

内存扫描防护之所以重要，在于游戏逻辑状态不可避免地驻留在客户端内存中。即便服务端权威架构（authoritative server）将移动合法性验证放在服务器侧，客户端内存中仍需存储本地预测数据、渲染所需的敌方位置等信息，这些数据一旦被 ESP（透视）或 Aimbot 脚本实时读取，即构成完整的作弊信息链。

## 核心原理

### 进程句柄权限控制

Windows 安全模型中，跨进程内存访问须先通过 `OpenProcess` 获取含 `PROCESS_VM_READ` 权限的句柄。内存扫描防护的第一道防线是在内核层拦截此调用：反作弊驱动（通常以 Ring-0 内核驱动形式运行）注册 **OB_CALLBACK**（对象回调），当任何第三方进程尝试以读/写权限打开游戏进程句柄时，回调强制剥离 `PROCESS_VM_READ` 和 `PROCESS_VM_WRITE` 标志位，使句柄降权为无法访问内存的空壳句柄。Riot Games 的 Vanguard 和 EasyAntiCheat 均采用此机制，且必须在驱动层实现——用户态 Hook 可被作弊程序绕过。

### 内存页加密与混淆

敏感数据结构（如玩家结构体中的坐标字段）可通过**运行时加密**减轻内存扫描危害。具体做法是用一个随机生成的 XOR 密钥 `K` 对真实值 `V` 加密后存储为 `V ^ K`，读写时在寄存器内完成解密，内存中永远不出现明文。以弹药计数为例：若 `K = 0xA3F1` 而真实弹药数为 `30`，则内存中存储的是 `30 ^ 0xA3F1 = 0xA3DF`，作弊工具扫描特征值 `0x1E`（30 的十六进制）将无法定位该字段。密钥每隔数秒由游戏逻辑层随机刷新，进一步干扰基于地址稳定性的偏移查找。

### 内存完整性校验（哈希守护）

反作弊模块定期对自身代码段、游戏可执行文件的 `.text` 段以及已加载 DLL 计算 SHA-256 校验值，并与启动时记录的基准哈希对比。若检测到差异，说明有注入代码篡改了内存中的指令流（如 Hook 游戏函数以跳转到作弊模块）。这一过程的采样间隔通常在 **500 ms 至 2000 ms** 之间——过短会对帧率产生可测量的 CPU 开销，过长则给作弊程序留下足够的执行窗口。部分实现会结合硬件性能计数器（Intel PT）追踪异常分支，以毫秒级精度发现短暂的代码注入行为。

### DLL 注入检测

作弊工具常通过 `CreateRemoteThread` + `LoadLibrary` 将恶意 DLL 注入游戏进程，绕过外部内存读写限制，改为在进程内部直接操作数据。内存扫描防护在进程内通过枚举已加载模块列表（`EnumProcessModules` 或遍历 PEB.Ldr 链表）与白名单对比，发现未签名或不在预期列表中的模块即触发报告。更深层的防护会扫描进程所有内存区域的属性，标记那些带有 `PAGE_EXECUTE_READWRITE` 属性且不对应任何已知文件映射的匿名可执行区域——这是 shellcode 驻留的典型特征。

## 实际应用

**《VALORANT》的 Vanguard 保护实例**：Vanguard 在系统启动时加载内核驱动，在游戏进程创建之前就注册对象回调，确保即使先于游戏启动的作弊程序也无法预先缓存有效句柄。这一设计解决了"驱动注册竞态条件"问题——若驱动晚于游戏启动，作弊程序已获取的高权限句柄不受 OB_CALLBACK 保护。

**《堡垒之夜》EasyAntiCheat 的内存特征库**：EAC 维护一份持续更新的作弊工具内存特征库，包含超过 **10,000** 条已知外挂程序的代码特征字节序列。当扫描到游戏进程地址空间中存在匹配特征的内存区域时（无论该区域是否挂钩了任何函数），立即判定为违规。这使得"只读不写"的 ESP 作弊（仅读取内存而不修改游戏行为）同样可被检出。

**服务端辅助验证**：纯客户端内存保护可被物理读取绕过（如使用 DMA 卡直接经 PCIe 总线读取物理内存，完全不触发 OS 层 API）。针对此类硬件级攻击，服务端通过行为异常检测（玩家在 200 ms 内连续完成 15 次精确爆头）与客户端保护形成双层防御。

## 常见误区

**误区一：用户态 Hook 足以防止内存扫描**
部分早期反作弊系统在用户态 Hook `ReadProcessMemory` 来拦截读取请求，但这种保护极易被绕过——作弊程序可通过直接系统调用（`syscall` 指令跳过 ntdll.dll 的 Hook 层）或使用未文档化的 `NtReadVirtualMemory` 绕过所有用户态拦截。有效的内存扫描防护必须依赖内核驱动级别的 OB_CALLBACK 或 SSDT Hook，这是不可绕过的内核路径。

**误区二：加密内存后作弊程序无法定位目标数据**
XOR 加密等混淆手段能显著增加静态扫描的难度，但专业作弊开发者会使用**动态追踪**：在加密密钥刷新的瞬间（真实值出现在寄存器中时）通过硬件断点捕获明文，或追踪解密函数的调用链反推加密结构体的地址。因此内存加密应与访问频率控制、代码虚拟化（VM 保护）结合使用，而非单独依赖加密。

**误区三：DMA 攻击属于理论威胁，无需防护**
直接内存访问攻击曾被认为是高门槛的硬件攻击，但自 2019 年前后，搭载 FPGA 的 PCIe DMA 卡（如 Screamer M2）在作弊社区售价降至 **$200** 以下，已被用于《Escape from Tarkov》等竞技游戏的职业赛场作弊。DMA 攻击不经过任何 OS API，完全规避软件层内存保护，这也是为何服务端行为分析成为必要的补充防线而非可选项。

## 知识关联

内存扫描防护建立在**客户端完整性**验证的基础上：客户端完整性保证游戏可执行文件在启动前未被篡改，而内存扫描防护则接力保护程序运行期间的动态状态不被外部读写。两者共同构成"启动前静态完整性 + 运行时动态保护"的完整链条，缺少后者则前者的保护在进程运行后立即失效。

内存扫描防护的局限性（尤其是硬件 DMA 攻击无法从软件层完全防御）直接推动了**可信计算（Trusted Computing）**技术在反作弊领域的应用：TPM 2.0 芯片与安全启动（Secure Boot）的组合可以为平台完整性提供硬件级证明，Intel TXT 技术则允许远程服务器验证客户端运行环境的可信状态，从而将防护边界从操作系统内核层延伸到硬件固件层。
