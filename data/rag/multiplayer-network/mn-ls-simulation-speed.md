# 模拟速度控制

## 概述

模拟速度控制（Simulation Speed Control）是帧同步回放系统中的一项机制，允许以非实时速率执行逻辑帧的推进。在标准帧同步系统中，逻辑帧以固定间隔（如 66ms，即 15 帧/秒，或 100ms，即 10 帧/秒）推进；而模拟速度控制则通过调整"每次 Update 推进几帧"或"每帧等待多久才推进"来实现快进、慢放与变速回放。

这一机制最早在即时战略游戏（如 StarCraft 的录像系统）中被广泛应用，玩家可将回放速率设置为 1/2x、1x、2x、4x、8x 等档位，用于战术复盘与精彩片段截取。在移动端多人对战游戏兴起后，模拟速度控制被移植到手机帧同步框架中，用于加速补帧追赶（Catch-up）及赛后录像功能。

模拟速度控制的意义在于：帧同步的所有游戏状态都由确定性逻辑帧序列完全描述，因此只需控制帧推进节奏，即可无损地重现任意速率的游戏过程，而无需对游戏逻辑本身做任何修改。

## 核心原理

### 速度倍率与帧推进节奏

模拟速度控制的核心参数是速度倍率（Speed Multiplier），记作 `S`。设逻辑帧间隔为 `T_logic`（单位 ms），实际物理帧时间为 `T_render`，则每次渲染帧应推进的逻辑帧数为：

```
推进帧数 = T_render × S / T_logic
```

例如，当 `T_logic = 100ms`，`T_render = 16.67ms`（60fps渲染），`S = 4.0`（4倍速）时，每渲染帧应推进约 0.667 逻辑帧，累计超过 1 帧时才实际执行一次 Tick。这里通常使用一个浮点型**帧累积器**（Frame Accumulator）跨渲染帧积分，避免每帧整除带来的误差。

### 慢放（S < 1）的实现

当 `S = 0.25`（四分之一速度）时，每推进 1 个逻辑帧需要等待 `100ms / 0.25 = 400ms` 的物理时间。实现上，累积器增长极慢，多个渲染帧后才触发一次逻辑帧 Tick。慢放主要用于对战后的关键帧复盘，例如 MOBA 游戏中的"击杀回放"通常以 0.25x 或 0.5x 播放最后 3 秒的录像内容，使玩家清楚看到技能释放时序。

需注意：慢放并不改变插值渲染的平滑度。渲染层依然以 60fps 运行，只是在相邻两个逻辑帧之间插值停留时间变长，角色动画表现上会呈现明显的"太空步"感，需要动画系统也同步按逻辑帧进度驱动，而非按物理时钟驱动。

### 快进（S > 1）与补帧追赶

快进最常见的应用场景是**断线重连追赶（Reconnect Catch-up）**。假设玩家断线 30 秒，逻辑帧率 15fps，需要追赶 450 帧。若以 `S = 8` 倍速追赶，仅需物理时间 `450 / (15 × 8) = 3.75 秒` 即可完成同步。

快进时的关键约束是：逻辑 Tick 必须连续、完整地按顺序执行，不得跳帧。帧同步的确定性依赖于每一帧输入的完整处理，即使在 8 倍速下也必须执行全部 450 帧的逻辑，而不能仅执行第 450 帧。这与状态同步的"直接设置最终状态"截然不同。

快进期间通常关闭渲染（仅执行无渲染的"无头模式" Headless Tick），以最大化 CPU 吞吐，将追赶时间压至最短。

### 变速回放的帧控制接口

完整的模拟速度控制接口一般暴露以下三个操作：

- **SetSpeed(S)**：动态修改倍率，可在回放过程中实时切换。
- **StepFrame(n)**：单步推进 n 帧，`S` 被暂时设为 0 后执行 n 帧 Tick，常用于逐帧调试。
- **SeekToFrame(f)**：跳转至第 f 帧，实现方式为从帧 0 或最近的**关键帧快照**开始，以最大速率追赶至帧 f。

SeekToFrame 的追赶起点决定了跳转延迟，因此生产实践中通常每隔 300 帧保存一次完整状态快照，使任意跳转的追赶量不超过 300 帧。

## 实际应用

**赛后录像快进**：《王者荣耀》等 MOBA 手游的战绩录像支持 2x 和 4x 回放，其底层即是在帧同步录像文件基础上调整速度倍率，而录像文件本身仅存储每帧的输入序列（约数十字节/帧），10 分钟对局录像文件通常不超过 500KB。

**断线重连**：《守望先锋》早期版本（基于状态同步）在断线重连时直接同步最终状态；而采用帧同步的游戏（如《英雄联盟》手游）则必须以快进方式追赶所有历史帧，其追赶速率上限受设备 CPU 性能约束，低端设备可能只能支持 4x 而高端设备可支持 16x 以上。

**开发调试**：使用 StepFrame(1) 逐帧推进，开发者可精确复现某帧出现的碰撞异常或 AI 决策错误，这是帧同步相对于状态同步调试体验更好的原因之一。

## 常见误区

**误区一：快进时可以跳过中间帧以节省时间**。帧同步的确定性要求每一帧的输入必须被顺序处理，跳过任意一帧将导致后续所有帧的计算结果出现偏差（蝴蝶效应），最终状态与其他客户端不一致。跳帧是帧同步中的严重错误。

**误区二：慢放需要降低物理引擎的 Tick 频率**。慢放仅改变逻辑帧的推进节奏，物理 Tick 仍以固定步长 `T_logic` 执行，只是触发频率在实际时间轴上被拉伸。若错误地将物理步长也乘以 `1/S`，会改变物理计算结果，破坏确定性。

**误区三：速度倍率可以任意设置到无限大**。实际上倍率上限受单帧可用 CPU 时间约束。若一个逻辑 Tick 耗时 2ms，渲染帧时间为 16ms，则理论最大有效倍率约为 `16ms / 2ms × (T_logic / T_render) = 8 × (100ms / 16ms) ≈ 50x`；超过此值会导致渲染帧率下降，产生卡顿。

## 知识关联

模拟速度控制直接依赖**回放系统**提供的帧输入序列存储能力：没有完整的帧输入记录，就无法以任意速率重放。回放系统负责存储"播放什么"，模拟速度控制负责决定"以多快速度播放"，两者分工明确。

帧累积器的实现与帧同步的**固定步长 Tick（Fixed Timestep）**机制紧密相关——正是因为逻辑帧采用固定步长而非可变步长，速度倍率才能被简洁地表达为"单位物理时间内执行多少个固定步长"，避免了可变步长带来的浮点精度问题。在支持关键帧快照的帧同步框架中，SeekToFrame 的效率还进一步受到**快照间隔**参数的影响，是调优回放跳转性能时必须考量的配置项。
