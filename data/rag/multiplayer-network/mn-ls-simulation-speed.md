# 模拟速度控制

## 概述

模拟速度控制（Simulation Speed Control）是帧同步回放系统中调节逻辑帧推进速率的技术机制，允许以高于、低于或等于原始录制速率来重新执行确定性模拟。与视频播放的变速不同，帧同步的模拟速度控制必须每一帧都完整执行游戏逻辑，不能跳帧或插值——加速时每个渲染帧执行多个逻辑帧，减速时一个逻辑帧跨多个渲染帧才执行一次。

该技术最早在即时战略游戏的录像系统中得到系统化应用。StarCraft 的 .rep 回放文件从1998年就支持通过快捷键切换回放速度档位（最快约为正常速度的8倍），其底层正是按固定逻辑帧间隔批量推进模拟。移动端 MOBA 和 RTS 游戏（如《王者荣耀》《皇室战争》）将这一机制引入帧同步回放，提供2倍、4倍快进以节省用户观看时间。

模拟速度控制的实用价值体现在三个层面：赛后回放的快速浏览、游戏内追帧（断线重连时高速模拟追上当前进度）、以及调试时的慢动作分析。三种场景对速度控制的实现要求各有侧重，但底层机制共享同一套逻辑帧调度框架。

## 核心原理

### 逻辑帧与渲染帧的解耦

帧同步的标准设计将游戏逻辑帧（Simulation Tick）与渲染帧（Render Frame）分离。逻辑帧以固定间隔（典型值为 66ms，对应 15 逻辑帧/秒）驱动确定性状态机；渲染帧以显示刷新率（60fps 或 120fps）运行，从当前逻辑状态插值绘制画面。模拟速度控制本质上是修改**逻辑帧推进速率**，而不触动渲染管线。

实现方式是维护一个**模拟时间累加器**（Simulation Accumulator）。每个渲染帧将 `deltaTime × speedMultiplier` 累加进去，当累加量超过 `logicTickInterval` 时消耗一次逻辑帧推进。伪代码如下：

```
accumulator += deltaTime * speedMultiplier
while accumulator >= logicTickInterval:
    simulationTick()
    accumulator -= logicTickInterval
```

`speedMultiplier = 2.0` 时快进两倍，`speedMultiplier = 0.25` 时慢放为四分之一速。

### 快进的执行限制与跳帧策略

快进时每个渲染帧可能需要执行数十个逻辑帧，若倍率过高会导致单帧 CPU 占用突增造成卡顿。常见做法是设置**每渲染帧最大逻辑帧推进数**（Max Steps Per Frame），典型上限为 8～16 帧。超出上限时允许丢弃渲染帧（逻辑继续推进但不绘制），牺牲画面流畅度换取追帧速度。

追帧场景（断线重连）往往需要极高倍率，此时渲染完全关闭，以纯逻辑循环速度推进，一秒内可完成数百个逻辑帧，直到追上服务器的当前帧号后再恢复正常渲染。《王者荣耀》的断线重连追帧即采用类似机制，玩家看到的"Loading 进度条"背后是高速模拟在执行。

### 慢放的精度保障

慢放（`speedMultiplier < 1.0`）使同一个逻辑状态被渲染多帧，期间游戏逻辑状态不变，只有渲染层做时间插值。此时需注意**输入事件不可被插值**——如果慢放期间累积了来自输入文件的操作指令，必须严格在对应逻辑帧边界执行，而非提前或延迟。慢放比例越大（如 0.1 倍速），输入指令在哪个渲染帧实际触发逻辑就越敏感，需要用帧号而非时间戳来索引输入序列。

### 变速档位设计

实际产品通常提供离散档位而非连续变速，常见设计为：`×0.5`、`×1`、`×2`、`×4`，对应 speedMultiplier 为 0.5、1.0、2.0、4.0。离散档位简化了 UI 设计，也避免了极端倍率引发的边界问题。部分调试工具支持连续滑条，范围通常为 `[0.05, 10.0]`，低于 0.05 倍时实质上等同于逐帧步进。

## 实际应用

**赛后回放快速浏览**：移动端 MOBA 游戏在回放界面提供 ×2 和 ×4 按钮，对应将 speedMultiplier 设为 2.0 或 4.0。一局 20 分钟的比赛在 ×4 速下约 5 分钟看完，对应执行了同等数量的逻辑帧（约 18000 帧，按 15fps 逻辑帧率计算）。

**断线重连追帧**：客户端记录断线前已确认的最后帧号，重连后从本地存档的快照帧（通常每 300 帧存一次完整状态）开始，以无渲染全速模拟推进至服务器当前帧。若断线 3 分钟（约 2700 帧），从最近快照（约 2400 帧位置）追起只需模拟约 300 帧，毫秒级完成。

**调试分析**：开发阶段将 speedMultiplier 调至 0.1 可以观察单个技能命中判定、碰撞检测的精确时序，配合逐帧步进（每次按键推进一个逻辑帧，等效于 speedMultiplier 趋近 0）排查确定性 bug。

## 常见误区

**误区一：快进可以跳过逻辑帧**。视频播放器的快进是时间轴跳跃，但帧同步快进必须执行中间每一个逻辑帧，因为第 N 帧的状态依赖第 N-1 帧的完整计算结果。跳帧会破坏确定性，导致回放后期状态与原始游戏不一致，即所谓"回放漂移"。

**误区二：慢放时渲染插值等同于游戏逻辑**。慢放期间角色位置的平滑移动仅是渲染层的线性或曲线插值，并不代表游戏逻辑层的中间状态。如果把慢放画面中间某时刻的角色坐标当作碰撞判定依据，则会产生逻辑错误，碰撞判定只能在真实逻辑帧边界发生。

**误区三：任意倍率都能流畅运行**。speedMultiplier 受 CPU 算力硬上限约束。若单个逻辑帧耗时 4ms，在 60fps 渲染下（渲染帧预算约 16ms），理论最大有效倍率约为 4 倍（16ms ÷ 4ms = 4 帧/渲染帧）。盲目设置 ×16 会导致每帧逻辑执行超时，造成渲染帧率骤降，反而不如稳定的 ×4 快。

## 知识关联

模拟速度控制以**回放系统**为前提：必须先有完整的输入序列存档和确定性模拟基础，变速控制才有意义——回放系统提供了按帧号索引的输入流，模拟速度控制调节消费该输入流的快慢节奏。追帧功能还依赖回放系统的**快照机制**，快照间距（如每 300 帧）直接决定追帧的最坏耗时。

在实现层面，模拟速度控制与**逻辑帧与渲染帧分离**架构深度耦合，正是因为两者已解耦，才能独立调节逻辑推进速率而不破坏渲染管线。对于需要精细调试帧同步逻辑的开发团队，慢放与逐帧步进是排查非确定性问题（如浮点数差异、随机数种子错误）的重要工具，能将原本转瞬即逝的帧间状态放大到可观察的时间尺度。
