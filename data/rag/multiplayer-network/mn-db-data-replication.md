# 数据复制

## 概述

数据复制（Data Replication）是指将同一份数据自动同步到多个数据库节点的技术机制。在网络多人游戏场景中，数据复制解决了单点故障问题——当主节点宕机时，从节点可以立即接管，保证玩家的游戏进度、排行榜、好友关系等数据不会因服务器故障而丢失或不可访问。

数据库复制技术起源于1990年代的关系型数据库系统。MySQL在5.0版本（2005年发布）引入了基于二进制日志（Binary Log）的主从复制机制，成为游戏行业使用最广泛的复制方案之一。此后随着Redis Sentinel（2012年）和MongoDB副本集（Replica Set）的成熟，游戏开发者拥有了更多高可用选择。

在多人游戏中，数据复制的价值在于实现读写分离：主节点负责处理玩家的写操作（购买道具、升级技能），而多个从节点分担读请求（查看排行榜、浏览好友列表）。一个典型的大型MMO（大型多人在线游戏）服务器集群可能配置1主3从的拓扑结构，将读负载分散到从节点，从而将数据库层的读吞吐量提升至原来的3-4倍。

## 核心原理

### 主从复制（Master-Slave Replication）

主从复制是单向数据同步模式：所有写操作只能发往主节点，主节点将变更记录写入二进制日志（Binlog），从节点通过I/O线程持续拉取Binlog，再由SQL线程回放执行，从而保持与主节点的数据一致。

主从复制存在固有的**复制延迟**问题，用公式表达为：

```
延迟时间 = 网络传输时间 + 从节点SQL线程回放时间
```

在游戏场景中，复制延迟通常在1毫秒到数百毫秒之间波动。这意味着玩家刚完成的交易数据写入主节点后，立即从从节点读取可能得到旧数据。游戏后端需要针对这种**最终一致性**特点做出设计决策，例如将玩家自身的背包查询强制路由到主节点，而将全服排行榜查询路由到从节点（允许数秒的延迟）。

主从架构的致命弱点是主节点单点故障。当主节点宕机时，需要通过**故障转移**（Failover）将某个从节点提升为新主节点，MySQL的MHA（Master High Availability）工具可以将这个过程压缩到30秒以内。

### 多主复制（Multi-Master Replication）

多主复制允许多个节点同时接受写操作，各节点之间双向同步数据。这种模式特别适合跨地域部署的游戏服务——例如在北美和欧洲各部署一个主节点，玩家就近写入本地节点，再异步同步到对端。

多主复制面临的核心挑战是**写冲突**（Write Conflict）：当两个节点几乎同时修改同一条玩家记录时，如何决定最终状态？常见的冲突解决策略包括：

- **最后写入胜利（LWW, Last Write Wins）**：比较时间戳，保留时间戳更大的写操作。游戏中用于更新玩家登录时间等对精确性要求不高的字段。
- **自定义合并逻辑**：例如两个节点同时给玩家增加金币，最终结果应为两次增量之和，而非其中一次操作的结果。这需要将操作设计为**CRDT（无冲突复制数据类型）**语义。

MySQL Group Replication和Galera Cluster均实现了多主同步复制，采用**认证机制**在事务提交前检测冲突，冲突事务会被回滚并要求客户端重试，将冲突检测延迟控制在1-2毫秒量级。

### 半同步复制与同步复制

复制模式按数据安全级别分为三类：

| 模式 | 主节点确认写入时机 | 数据丢失风险 | 写入延迟 |
|------|-------------------|-------------|---------|
| 异步复制 | 写入本地即返回 | 主节点宕机可能丢失数据 | 最低 |
| 半同步复制 | 至少1个从节点收到日志后返回 | 极低 | 增加1个网络RTT |
| 同步复制 | 所有节点提交完成后返回 | 无 | 最高 |

游戏中涉及真实货币的充值流水通常采用**半同步复制**，而玩家聊天记录可接受异步复制。

## 实际应用

**排行榜系统的读写分离设计**：王者荣耀等竞技游戏的全服排行榜每分钟刷新，但读取频率远高于写入频率。典型做法是将排名积分的更新写入主节点，由3-5个从节点专门承载排行榜的读请求。允许排行榜数据有最多10秒的延迟展示，完全在玩家感知阈值之内。

**跨区服数据同步**：采用多主复制实现区服合并（服务器合并）时，需要提前处理玩家ID冲突问题。常见方案是为每个区服分配不同的ID段（例如A区玩家ID为1-999999，B区为1000000-1999999），从而避免多主写入时的主键冲突。

**Redis主从用于会话缓存**：游戏的在线玩家会话数据（Token、房间信息）存储在Redis主节点，从节点提供只读副本。当主节点故障时，Redis Sentinel在5-15秒内完成自动故障转移，玩家最多被强制重新登录一次，而非遭遇长时间服务不可用。

## 常见误区

**误区一：从节点可以完全替代主节点处理所有读操作**。从节点由于复制延迟的存在，不适合处理"先写后读"的强一致性场景。例如玩家完成支付后立即查询订单状态，如果读请求落到从节点，可能返回"订单不存在"的错误结果。正确做法是对强一致性读请求设置路由标记，强制走主节点，或在写操作后等待从节点同步完成再查询（MySQL支持`WAIT_FOR_EXECUTED_GTID_SET`函数）。

**误区二：多主复制等同于无限水平扩展写能力**。多主复制的每个节点仍需执行其他节点产生的所有写操作（通过日志回放），因此集群的整体写入能力并不随节点数线性增长。在写密集型场景（如大型战斗服务器的每帧状态持久化）中，多主复制无法解决写瓶颈，需要结合**数据分片**将数据水平切分到不同的复制组。

**误区三：复制延迟为零即表示数据完全一致**。即使复制延迟监控显示为0毫秒，从节点也可能正在回放主节点的最后一批日志，存在极短窗口的不一致。这种情况在游戏服务滚动发布期间尤为突出，需要在部署流程中增加从节点延迟检查步骤。

## 知识关联

学习数据复制需要先掌握**数据分片**的概念：分片解决的是单节点存储容量不足的横向扩展问题，而数据复制解决的是单节点可用性和读负载问题。两者通常组合使用——先按玩家ID分片将数据分布到多个分片，每个分片内再配置主从复制保障高可用，形成"分片×副本"的二维拓扑。

数据复制为后续学习**CQRS模式**（命令查询职责分离）奠定了实践基础。CQRS在架构层面将"写模型"和"读模型"分离，这与主从复制将写操作路由到主节点、读操作路由到从节点的物理拓扑高度契合。在CQRS架构中，从节点的数据库可以进一步构建针对查询优化的非规范化视图（例如预聚合的排行榜表），而主节点保持规范化的事务写模型，两者通过复制日志异步同步，实现读写两端的独立优化与扩展。
