# 客户端预测

## 概述

客户端预测（Client-Side Prediction）是网络多人游戏中的一种延迟补偿技术：客户端不等待服务器确认，而是在本地**立即执行玩家输入**并更新游戏状态，从而消除玩家感知到的操作延迟。其核心假设是"本地计算的结果大概率与服务器最终判定一致"，当两者不一致时再由服务器调和（Reconciliation）进行修正。

这一技术最早由 Valve 工程师 Yahn Bernier 在2001年发布的《Latency Compensating Methods in Client/Server In-game Protocol Design and Optimization》中系统阐述，并在 Half-Life 引擎（GoldSrc）的网络模型中得到工业级应用。此后成为第一人称射击游戏网络编程的标准实践，Quake III、CS:GO、Overwatch 均采用类似机制。

没有客户端预测时，即便是 50ms 的往返延迟（RTT）也会让玩家感到角色"拖着走"。以 60Hz 刷新率为例，50ms 延迟意味着玩家按键后需等待约 3 帧才能看到角色移动，这在竞技游戏中是不可接受的体验。客户端预测将这一感知延迟压缩至接近零，是高竞技性网络游戏可行性的前提条件。

## 核心原理

### 输入缓冲与本地执行

客户端维护一个**输入序列队列**，每条输入记录包含：序列号（sequence number）、输入内容（如 WASD 状态）、时间戳和 delta time。玩家每次按键，客户端立刻在本地运行与服务器**完全相同的物理/移动逻辑**，将预测结果直接渲染给玩家，同时将该输入包发送给服务器。

预测的正确性依赖"确定性"（Determinism）——客户端与服务器必须使用相同的移动函数和相同的 delta time，才能在相同输入下得到相同输出。为此，客户端预测的移动代码通常与服务器共享同一份代码，例如 Unreal Engine 中 `UCharacterMovementComponent` 同时在客户端（预测路径）和服务器（权威路径）执行。

### 服务器确认与状态快照

客户端不会丢弃已执行的输入，而是将其保存在**未确认输入缓冲区**（Unacknowledged Input Buffer）中。服务器处理完某输入后，会在状态快照中附带"已处理的最大序列号（last_processed_input）"返回给客户端。客户端收到此确认后，才能从缓冲区中移除序列号 ≤ last_processed_input 的所有输入。

以 20Hz 服务器 tick 率（每帧 50ms）为例，若客户端 RTT 为 100ms，则缓冲区中通常积压 100ms / 50ms × 2 = 约 4 条待确认输入。这说明输入缓冲区的大小需与 RTT 成正比规划。

### 预测误差的量化

预测误差定义为：

```
error = server_position - predicted_position
```

当 `|error|` 超过预设阈值（如 Valve 的实现中通常为 0.5 个游戏单位）时，才触发位置修正，避免因浮点精度差异导致不必要的抖动。误差修正可以选择**瞬间回滚**（Snap）或**插值平滑**（Smooth），前者适合竞技环境（保证正确性），后者适合休闲游戏（保证视觉舒适度）。

## 实际应用

**第一人称射击游戏的角色移动**是客户端预测最典型的应用场景。CS:GO 中，玩家按下 W 键后角色**同帧**开始移动，网络延迟对操作手感完全透明。服务器以 64Hz（竞技模式）或 128Hz（第三方平台如 FACEIT）进行 tick，客户端每帧生成输入包并立即本地预测执行。

**在线赛车游戏**的本车操控同样使用客户端预测。玩家驾驶的车辆由本地预测驱动，其他车辆则通过插值或外推显示——这解释了为何本车操控总比其他车辆显示更流畅。

**RTS 游戏中的单位命令**则较少使用传统客户端预测，而倾向于采用帧锁定（Lockstep）协议，因为 RTS 中单位状态的预测复杂度远超角色移动，误差修正代价极高。

**Unreal Engine 5 的网络角色移动**提供了开箱即用的客户端预测实现：`SavedMove` 结构体记录每帧移动输入，`ClientUpdatePosition` 在收到服务器状态后触发重放（Replay）——即将服务器权威状态作为起点，将缓冲区中所有未确认输入依次重新执行，得到修正后的当前预测位置。

## 常见误区

**误区一：客户端预测意味着客户端拥有权威状态**

客户端预测的计算结果仅用于本地渲染，**从不向服务器提交位置**。客户端发送的永远是**输入**（Input），而非状态（State）。如果客户端可以直接告知服务器"我在这个位置"，则退化为信任客户端的架构，导致速度外挂（Speed Hack）等作弊方式完全无法防御。权威服务器始终基于自身对输入的计算结果来判定最终状态。

**误区二：预测必须100%准确才有价值**

客户端预测允许偶尔出错。实际上，当玩家撞墙、被击退或遭遇服务器端触发的状态变更（如被 AOE 技能减速）时，预测必然与服务器不符。这些情形下，服务器调和（Reconciliation）负责修正偏差。预测准确率只需"足够高"——在正常移动中误差率接近0%，在边界碰撞时允许出现可被平滑掩盖的小幅修正。

**误区三：高频率服务器 tick 可以替代客户端预测**

即使服务器运行在 128Hz（每帧约 7.8ms），100ms RTT 的玩家仍会感受到 100ms 的输入延迟。服务器 tick 率影响状态更新的精度，但无法消除光速物理延迟带来的往返时间。客户端预测针对的是 RTT 本身导致的感知延迟，两者解决的是不同层次的问题，不能互相替代。

## 知识关联

客户端预测建立在**权威服务器模型**的基础上：正因为服务器是最终裁判，客户端才需要"猜测"服务器的结论而非直接使用本地结果。若采用 P2P 架构，则不存在需要预测的权威源，客户端预测这一概念失去意义。同时，客户端预测是**延迟掩盖技术**家族中专门针对"本地操作响应"的成员，区别于用于掩盖其他玩家位置延迟的插值（Interpolation）技术。

客户端预测直接引出**服务器调和**（Server Reconciliation）问题：预测执行后必须设计一套机制处理预测与服务器权威状态之间的差异，包括回滚（Rollback）、重放（Replay）和误差平滑三种主要策略。在此基础上，**物理同步**进一步挑战了客户端预测的适用边界——刚体物理由于其混沌特性，即便输入相同也可能因浮点顺序差异导致结果发散，需要专门的物理预测策略（如影子物理体、物理状态快照插值）来维持稳定性。
