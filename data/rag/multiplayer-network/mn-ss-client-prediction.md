# 客户端预测

## 概述

客户端预测（Client-Side Prediction）是网络多人游戏中一种让客户端在收到服务器确认之前，**立即在本地执行玩家输入**并更新游戏状态的技术。当玩家按下移动键时，客户端不等待服务器的响应，而是直接将角色移动到预测位置，从而将玩家感知到的输入延迟从网络往返时间（RTT，通常 50–200ms）降低到接近零。

该技术由 id Software 工程师 John Carmack 在 1996 年《Quake》的网络代码中率先工程化实现，后经 Valve 在《Half-Life》（1998）的 GoldSrc 引擎中正式体系化，并通过 2001 年 Yahn Bernier 发表的论文《Latency Compensating Methods in Client/Server In-game Protocol Design and Optimization》向整个行业传播。

客户端预测之所以在竞技游戏中不可或缺，是因为人类对输入延迟超过 **50ms** 就会明显感到"输入不跟手"。在没有预测的纯权威服务器模型中，玩家每次操作都要等待服务器回包，100ms 的 RTT 会使游戏感觉迟钝，严重影响竞技公平性与体验。

## 核心原理

### 输入序列号与本地状态快照

客户端为每一帧的玩家输入分配一个单调递增的**序列号（Sequence Number）**，例如第 237 帧的输入包含方向向量、按键位图和时间戳，格式为 `{seq: 237, move: (1,0), buttons: 0x03, dt: 16ms}`。客户端在发送该输入到服务器的同时，将其存入本地**未确认输入缓冲区（Unacknowledged Input Buffer）**，并立刻用本地物理/移动逻辑模拟出新的角色位置，渲染给玩家。

### 预测执行与状态推进

本地预测的执行逻辑与服务器端的移动逻辑**必须是逐比特确定性一致的（deterministic replicas）**。以第一人称射击游戏为例，角色速度更新公式为：

```
v_new = clamp(v_old + accel * input_dir * dt, -maxSpeed, maxSpeed)
pos_new = pos_old + v_new * dt
```

客户端和服务器使用完全相同的 `accel`、`maxSpeed` 常量与积分步长 `dt`，以保证在无网络误差的理想情况下两者计算结果完全吻合。如果双方逻辑存在哪怕 0.001 的浮点差异，累积多帧后就会导致频繁的位置校正抖动。

### 服务器确认与未确认队列清理

服务器处理输入后，将权威结果连同**最后确认的序列号（Last Acknowledged Sequence）**一起发回客户端，例如 `{ack_seq: 235, auth_pos: (12.4, 0, 8.1)}`。客户端收到后执行三步操作：
1. 从未确认缓冲区中删除序列号 ≤ 235 的所有历史输入；
2. 将本地状态回滚至服务器给出的权威位置；
3. 将缓冲区中剩余的未确认输入（236、237……）按顺序**重放（Replay）**，得到新的预测位置。

这一"发送→预测→回滚→重放"的循环是客户端预测的完整闭环，也是与服务器调和（Server Reconciliation）直接相连的核心机制。

## 实际应用

### 《Overwatch》的输入预测帧率解耦

暴雪在《Overwatch》中将客户端预测的模拟帧率固定为 **64Hz**，与服务器 Tick Rate 一致，避免因客户端帧率（可能达 300fps）与服务器 Tick Rate 不同步而产生预测偏差。玩家的技能冷却、弹药消耗等状态均在本地预测中即时反馈，只有伤害判定结果等待服务器权威确认。

### Unity Netcode 中的预测实现

在 Unity 的 Netcode for GameObjects（版本 1.x）中，开发者通过继承 `NetworkBehaviour` 并在 `Update()` 中检测 `IsOwner` 标志来实现客户端预测：仅本地拥有者执行预测移动，`NetworkTransform` 组件负责将服务器权威位置同步回来并自动做插值校正。典型配置下，`NetworkTransform` 的插值缓冲为 **2 个 Tick**，用于平滑服务器校正带来的突变。

### 射击游戏中的武器后坐力预测

武器后坐力（Recoil）的视觉表现通常**完全在客户端预测**，不等服务器确认，否则 100ms 延迟会使枪口抖动明显滞后于射击动作。服务器只校正最终的弹道命中位置，而不对后坐力动画做回滚，这是一种刻意接受的"预测与权威分离"设计取舍。

## 常见误区

### 误区一：客户端预测会让作弊更容易

客户端预测仅影响**本地渲染的视觉位置**，服务器始终以权威服务器模型为准，对每个客户端发来的输入进行独立校验。若客户端发送非法的位置数据（而非输入数据），服务器会忽略并用权威状态覆盖，因此预测不增加位置类作弊的可行性。真正需要警惕的是客户端发送**伪造输入序列**（如超速移动输入），这需要服务器端的速度上限校验来拦截。

### 误区二：预测逻辑与渲染逻辑可以共用同一套代码

许多开发者误以为将碰撞检测或状态机代码复用于预测即可。然而，渲染层常含有**非确定性**操作（如粒子系统随机数、GPU 浮点精度）。一旦这些逻辑污染预测模拟，就会造成客户端与服务器计算结果持续不一致，引发高频率的位置纠错抖动。正确做法是将"游戏逻辑层"与"表现层"严格分离，只在逻辑层进行预测模拟。

### 误区三：预测帧率越高越好

将预测步长 `dt` 设置为与渲染帧率相同（如每帧 3.3ms @300fps）会导致每帧都重放数十个未确认输入，CPU 开销呈线性增长。《Counter-Strike 2》等游戏的实践表明，将预测 Tick Rate 固定为 **64Hz 或 128Hz**，与服务器 Tick Rate 对齐，在性能与精度之间取得最优平衡，而非无上限地追求高频模拟。

## 知识关联

**前置概念——权威服务器模型**：客户端预测必须在权威服务器的框架下运行。客户端只有输入的发送权，没有状态的最终裁决权。若没有权威服务器兜底，客户端预测的结果将直接成为游戏真实状态，等同于信任客户端，这是所有竞技游戏的安全红线。

**前置概念——延迟掩盖技术**：客户端预测是延迟掩盖技术族群中针对"本地玩家自身操作"的专项手段。其他延迟掩盖手段（如实体插值）负责处理远程玩家的平滑显示，二者协同工作，共同覆盖多人游戏中延迟的不同来源。

**后续概念——服务器调和**：服务器调和（Server Reconciliation）是客户端预测的必然配套机制，专门解决预测结果与服务器权威状态出现偏差时的回滚与重放问题。掌握客户端预测的未确认输入缓冲区模型，是理解服务器调和为何需要保留历史输入快照的直接前提。

**后续概念——物理同步**：当游戏引入刚体物理（如车辆、可推动箱子）时，客户端预测的确定性要求从简单的角色移动公式扩展到完整的物理引擎状态，需要解决浮点非确定性、碰撞事件顺序等更复杂的问题，这正是物理同步专题所处理的核心挑战。
