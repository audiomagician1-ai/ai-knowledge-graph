# 状态保存与加载

## 概述

状态保存与加载（State Save/Load）是帧同步网络模型中将完整游戏逻辑状态序列化为字节流、并能从该字节流精确还原到同一逻辑帧的机制。其核心约束是：从同一个存档点恢复后，继续执行相同的输入序列，必须产生与原始游戏完全相同的后续状态——即状态加载必须保证确定性重现。

这一机制最早在格斗游戏的单机录像回放系统中出现，随后被 GGPO（Good Game Peace Out）等帧同步回滚（Rollback Netcode）框架在2008年前后系统化。GGPO 的核心设计要求游戏每帧都能在约 1 毫秒内完成一次完整状态快照，以支持最多8帧的预测与回滚窗口。

状态保存与加载的意义在于：它使帧同步系统能够脱离"只能从第0帧开始重播"的限制，直接从任意帧号恢复逻辑现场。无论是回滚纠错、断线重连的快速追帧，还是观战系统的任意跳转，都依赖这一机制提供的"时间锚点"。

## 核心原理

### 序列化范围：什么必须被捕获

帧同步的状态快照必须包含所有影响逻辑输出的变量，通常分为三类：

- **逻辑实体状态**：每个游戏对象的位置（以定点数表示，如 `Q16.16` 格式）、速度、朝向、生命值、技能冷却计时器等。
- **随机数生成器状态**：帧同步使用确定性伪随机数生成器（如线性同余生成器或 xorshift），其当前种子值必须被记录，否则恢复后的随机序列将与原始序列不同。
- **全局逻辑变量**：当前逻辑帧号、碰撞检测用的空间分区数据、技能系统的全局计数器等。

一个常见错误是遗漏物理模拟中间状态——如果碰撞求解器在单帧内迭代多次，每次迭代的中间约束力也属于状态的一部分，必须被序列化。

### 序列化格式与定长结构

高性能帧同步状态快照通常采用**定长二进制结构**，而非 JSON 或 Protocol Buffers 等变长格式。原因是定长结构支持直接 `memcpy`，避免动态内存分配，单次快照耗时可控制在 0.1–0.5 毫秒范围内。典型做法是将所有逻辑状态放入一块连续内存区域（称为 State Buffer），快照时直接拷贝该内存块。

以一个100个单位的 RTS 游戏为例，每个单位存储 64 字节的定长状态，加上全局状态 512 字节，单帧快照大小约为 `100 × 64 + 512 = 6912 字节`，约 6.75 KB。对于 60 帧/秒的游戏，若保留8帧回滚窗口，峰值内存占用约 54 KB，完全可接受。

### 快照触发策略与帧号索引

状态快照不一定每帧都存储。常见策略有两种：

1. **固定间隔快照**（Keyframe Snapshot）：每 N 帧（如每30帧）强制保存一次完整快照，其余帧仅记录输入差量。追帧时从最近的快照重播至目标帧，时间复杂度为 O(N)。
2. **回滚窗口快照**（Rolling Snapshot）：仅保存最近 W 帧（如 W=8）的连续快照，用于 GGPO 风格的预测回滚。每帧保存一次，老帧快照被循环覆盖，内存占用固定为 `W × 单帧快照大小`。

两种策略分别针对"大范围追帧"和"小范围纠错"场景，实际项目中常组合使用：每30帧一个关键帧快照，外加滚动的8帧回滚缓冲。

### 加载时的一致性验证

状态加载后，通常需要执行一致性校验。常用方案是对加载后的完整状态计算 32 位或 64 位校验和（如 FNV-1a 哈希），与存档时记录的校验和比对。若不一致，说明序列化逻辑存在遗漏字段，必须报错而非静默继续，否则会产生难以追踪的逻辑分叉。

## 实际应用

### GGPO 框架中的回滚纠错

在 GGPO 模型中，本地客户端预测对方输入并推进逻辑帧。当收到对方真实输入（通常滞后1–3帧）且与预测不符时，系统将当时的帧快照加载，重新执行正确输入序列，再将结果与渲染层同步。整个过程对玩家不可见，延迟体验接近本地游戏。

### 《街头霸王5》与《原神》的实践差异

《街头霸王5》采用 GGPO 回滚方案，状态快照仅需覆盖两名角色加少量场景变量，单帧快照可压缩至1 KB以内。《原神》的多人副本使用帧同步，但场景实体数量更多，通常通过分区域懒加载减少单次快照大小，并仅在副本边界（如Boss战开始时）强制全量快照。

### 录像文件的存储格式

帧同步录像通常只存储初始状态快照（第0帧）加上每帧的完整输入序列，而非每帧快照。回放时从初始状态重播全部输入，文件体积极小。例如一场20分钟60帧/秒的游戏，若每帧输入为4字节×4玩家，录像仅需 `20×60×60×16 = 1,152,000 字节`，约 1.1 MB。

## 常见误区

### 误区一：渲染状态也需要被序列化

帧同步快照只需保存**逻辑层状态**，动画播放进度、粒子效果、音效队列均属于表现层，不影响逻辑帧的推进。将渲染状态混入快照会使快照体积膨胀数倍，且毫无必要。判断标准是：该变量是否在 `update()` 逻辑中被其他逻辑变量读取？若否，则不需要序列化。

### 误区二：状态加载等价于重置再回放

直接从快照加载与从第0帧重播到目标帧，结果应当相同，但耗时完全不同。若游戏运行了3000帧，重播需执行3000次逻辑更新，而快照加载仅需一次 `memcpy`。项目中曾有团队因快照序列化不完整，转而用重播替代加载，导致追帧时玩家体验到明显的画面卡顿。

### 误区三：校验和相同代表状态完全正确

两个不同状态恰好产生相同 32 位 FNV-1a 哈希的概率约为 1/2³²（约 2.3×10⁻¹⁰），在单次验证中几乎不可能误判。然而，如果序列化函数系统性地跳过某一字段（如漏写了随机数种子），则每次保存和加载都遗漏相同字段，校验和对比仍会一致，但状态实际上是错误的。因此，校验和只能验证"加载是否还原了存档内容"，不能验证"存档是否完整"。

## 知识关联

状态保存与加载机制直接支撑**中途加入**（Late Join）功能。当新玩家在游戏进行中连接时，服务器无需让其从第0帧开始重播全部历史帧，而是将最近的关键帧快照发送给新客户端，新客户端加载该快照后仅需追赶快照帧到当前帧的少量输入差量，通常可在数秒内完成同步。没有高效的状态加载机制，中途加入在长局游戏（如持续30分钟的 MOBA）中将不可实现。

此外，状态保存与加载也是帧同步**断线重连**的基础——服务器定期存档关键帧快照，断线玩家重连后下载最近快照，而非被迫重新加入或等待游戏结束。理解序列化边界（哪些字段必须捕获）是后续研究确定性碰撞检测与定点数运算的实践前提，因为只有逻辑层完全使用确定性数据类型，序列化才能保证跨平台的精确还原。
