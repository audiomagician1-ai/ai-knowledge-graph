# 数据分片

## 概述

数据分片（Database Sharding）是一种将单一数据库中的数据按照特定规则拆分，分布存储到多个独立数据库节点（称为"分片"或 Shard）上的水平扩展技术。在网络多人游戏中，数据分片专指按玩家ID、区服编号、地理大区等维度将玩家数据表横向切割，使每个分片只负责全体玩家数据的一个子集。与垂直拆分（将不同字段分放不同库）不同，水平分片的每个节点保有相同的表结构，只是存储的行记录不同。

数据分片概念起源于2000年代中期大型互联网公司对关系型数据库瓶颈的应对实践。2006年，Flickr 工程团队公开发表了一篇题为 *Sharding the Flickr Data* 的技术博客，系统阐述了基于用户ID取模（user_id % N）进行数据库分片的方法，这篇文章奠定了现代水平分片设计模式的基础。MMORPG领域随后广泛采用该方案，典型案例包括《魔兽世界》2008年扩展版的区服数据库架构以及腾讯游戏平台的全区全服系统重构。

对于日活跃玩家（DAU）超过百万量级的网络游戏，单节点关系型数据库的写入吞吐量通常在每秒 3000～5000 次事务（TPS）附近达到瓶颈。数据分片将写压力分散至 N 个节点后，理论峰值吞吐量可线性扩展为原来的 N 倍，这是单纯依靠读写分离（主从复制）无法实现的效果，因为读写分离无法降低主库的写入压力。

## 核心原理

### 分片键（Shard Key）的选取

分片键是决定某条玩家记录归属哪个分片的计算依据，选取不当将导致数据分布严重不均（称为"热点分片"）。在游戏场景中，最常用的分片键有两类：

- **玩家ID取模分片**：`shard_id = player_id % shard_count`。当 `shard_count = 16` 时，player_id 为 10000 的玩家被路由至第 0 号分片（10000 % 16 = 0），player_id 为 10001 的玩家被路由至第 1 号分片。此方法数据分布均匀，但扩容时需要对 N 的变更进行全量数据迁移，成本极高。
- **区服ID范围分片**：将区服编号 1～100 分配到分片A，101～200 分配到分片B，以此类推。此方法扩容只需添加新分片并分配新区服，无需迁移旧数据，但存在新区服活跃度远超旧区服导致的负载倾斜问题。

游戏中绝不能选用角色等级、在线状态等高频变动的字段作为分片键，因为这意味着每次升级或上下线都需要在分片间迁移数据行，产生巨量的跨分片事务开销。

### 一致性哈希（Consistent Hashing）

为解决取模分片在扩容时全量迁移的问题，现代游戏服务器架构普遍采用一致性哈希算法。该算法将哈希值空间（通常为 0 至 2³²−1，即约 43 亿）组织成一个虚拟环，每个物理分片节点负责环上的一段弧区间。当从 8 个分片扩容至 9 个分片时，理论上只需迁移约 1/9（约 11.1%）的数据，远低于取模分片扩容时需迁移的 ~87.5% 的数据量。

为防止节点分布不均，每个物理分片通常在虚拟环上部署 150～200 个虚拟节点（Virtual Node），使得每个物理分片实际负责多段不连续的哈希区间，从统计学上保障数据均匀分布。

### 跨分片查询的处理

玩家的单条记录操作（如读取装备栏、更新经验值）可由分片键直接路由到唯一目标分片，时延等价于单机查询。但公会成员列表、全服排行榜等需要聚合多个分片数据的查询，则必须采用**分散-聚合（Scatter-Gather）**模式：中间件将查询广播至全部 N 个分片，收集各自的局部结果后在应用层进行归并排序。以全服战力排行榜为例，若分布在 16 个分片上，系统需并发查询 16 个节点并合并 16 × top_K 条记录，最终取前 K 名，查询延迟通常比单分片查询高出 3～8 倍。正因如此，实际产品中全服排行榜通常采用异步预计算（每 5 分钟刷新一次），而非实时跨分片查询。

## 实际应用

### 按玩家ID分片的背包系统

以一款 DAU 500万的 MMORPG 为例，玩家背包数据表（`inventory`）包含约 20 亿行记录，单节点 MySQL 已无法在 200ms 内完成玩家登录时的背包加载查询。采用 `player_id % 32` 分片方案后，每个分片平均承载约 6250 万行记录，登录背包加载 P99 延迟从 480ms 降至 62ms。分片代理层（如 MyCat 或 Vitess）在接收到 `WHERE player_id = 12345678` 的查询时，自动计算 `12345678 % 32 = 14` 并将请求路由至第 14 号分片，业务代码无需感知分片逻辑。

### 按区服分片的区服独立玩法

《天涯明月刀》等采用区服制的游戏，以区服ID作为分片键，将同一区服的所有玩家数据（角色、公会、好友关系）集中存储于同一分片。这样公会列表、区服排行榜等常见查询全部命中单一分片，无需跨分片聚合。代价是当某个区服因版本更新涌入大量新玩家时，对应分片会成为热点，通常需要配合该分片的纵向扩容（升级更高配置的硬件）或对大区服进行子分片拆分来缓解压力。

### 分片扩容的平滑迁移实践

当游戏扩容从 16 分片扩至 32 分片时，为避免停服，通常采用"双写迁移"策略：迁移期间新写入同时写向旧分片和新分片（双写），后台异步任务将旧分片中需要迁移的历史数据复制到新分片，完成后切流并停止旧分片的写入。整个迁移过程一般需要 3～7 天，期间存储成本翻倍，是分片运维中最高风险的操作之一。

## 常见误区

**误区一：分片数量越多性能越好。**
每增加一个分片，跨分片查询的并发请求数就线性增加。当分片数从 16 扩展到 64 时，全服排行榜的 Scatter-Gather 查询需要发起 64 次并行 I/O，数据库连接池的连接数压力和网络开销可能反而使聚合查询性能下降。经验值是，单分片数据量控制在 5000 万至 2 亿行之间，在保证单分片查询性能的同时不过度增加跨分片开销。

**误区二：分片键选定后可以任意修改。**
在 `player_id % 16` 方案运行期间，如果将分片键改为 `region_id`，则所有已存在的玩家数据行都需要按新规则重新计算路由目标并完成物理迁移，这是一次全量数据搬迁操作，对于存储了数十亿行的生产库来说往往需要数周时间，且伴随极高的服务可用性风险。因此，分片键的选取在数据库上线前就必须经过充分论证，一旦投产便视为不可轻易更改的架构决策。

**误区三：数据分片能替代读写分离。**
数据分片解决的是写入容量（Capacity）和数据存储量（Storage）的水平扩展问题；读写分离解决的是读取吞吐量（Read Throughput）的放大问题。一个设计完善的游戏数据库通常同时使用这两种技术：N 个分片各自再配置 1 主 2 从的复制结构，写请求走各分片主节点，读请求分散到各分片从节点。两者并非替代关系，而是作用于不同瓶颈维度的互补手段。

## 知识关联

数据分片的实现以**玩家数据模型**为设计前提——只有在明确了玩家主键（player_id）的生成规则（如雪花算法 Snowflake ID 保证全局唯一递增）和核心数据表的关联结构之后，才能确定合理的分片键和分片粒度。例如，若玩家数据模型中公会表与玩家表以 `guild_id` 关联，则应考虑将 `guild_id` 而非 `player_id` 作为公会相关表的分片键，以避免跨分片的 JOIN 操作。
