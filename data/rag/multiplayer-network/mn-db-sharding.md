# 数据分片

## 概述

数据分片（Database Sharding）是一种将单一逻辑数据库的数据行按照特定维度切分、分散存储到多台物理数据库节点上的水平扩展技术。在网络多人游戏中，一台单机MySQL服务器通常在玩家数量超过约500万活跃账号后开始出现写入瓶颈，而数据分片通过将玩家数据分布到N个独立节点，使写吞吐量理论上线性扩展至原来的N倍。

数据分片的概念最早在2000年代中期随着Web 2.0的爆发而成熟，Flickr工程团队于2007年公开的分片架构方案成为行业重要参考。在游戏领域，《魔兽世界》早期使用区服（Realm）隔离天然形成了物理分片形态，而后来的《英雄联盟》则在单大区内对玩家数据库实施了精细的哈希分片，以支撑单区数千万玩家的并发写入。

数据分片在游戏数据库设计中之所以不可回避，是因为玩家背包、战绩、社交关系等核心表的数据量增长速度与玩家数量成正比，且这些写入请求无法通过只读副本的读写分离来缓解——每一次道具获取、每一场对战结算都是强写操作。

## 核心原理

### 分片键的选择

分片键（Shard Key）是决定每行数据落入哪个分片的核心字段。在游戏场景中最常见的分片键有两类：**玩家ID（Player ID）** 和 **区服ID（Server/Zone ID）**。

使用玩家ID作为分片键时，通常采用对分片总数取模的哈希策略：

```
shard_index = hash(player_id) % N
```

其中 `N` 为分片节点总数，`hash()` 通常选用一致性哈希（Consistent Hashing）而非简单取模，原因是当需要扩容增加节点时，一致性哈希仅需迁移约 `1/N` 的数据，而简单取模需要重新分配接近全量数据。

使用区服ID作为分片键时，每个区服的全部玩家数据存储在同一分片节点上，这使区服内的跨表联查（如排行榜、公会数据聚合）无需跨分片，查询效率极高，代价是不同区服间玩家数量差异可能导致热点分片（Hot Shard）问题。

### 范围分片与哈希分片的差异

**范围分片（Range Sharding）** 按玩家ID的数值区间划分，例如 player_id 1–10,000,000 存入Shard-1，10,000,001–20,000,000 存入Shard-2。这种方式支持高效的范围扫描（如"查询注册时间最早的10万名玩家"），但新用户注册会集中写入当前最大分片，造成写入热点，在游戏上线初期尤为突出。

**哈希分片（Hash Sharding）** 将写入流量随机打散到所有节点，消除热点写入，但同时破坏了数据的局部性——对连续玩家ID区间的批量查询必须访问全部分片后再聚合结果（即Scatter-Gather模式），延迟随分片数量线性增长。

### 跨分片查询的处理

跨分片查询是分片架构最棘手的问题。典型场景如"全区服战力排行榜"需要从所有分片读取每位玩家的战力数值后在应用层聚合。常见的工程解法有两种：

1. **全局汇总表**：在独立的汇总节点上维护一张只保存排行榜所需字段的冗余表，各分片在数据变更时异步同步至该表，查询排行榜只读汇总表。
2. **应用层聚合（Fan-out Query）**：应用服务器并发向所有N个分片发出相同查询，收齐响应后在内存中排序合并。当N较小（如8个分片以内）时该方案可行，N较大时延迟难以接受。

## 实际应用

**按玩家ID哈希分片的背包系统**：玩家背包表（inventory）以player_id为分片键哈希到16个分片节点。玩家拾取道具时，应用服务器通过 `shard_index = murmur3(player_id) % 16` 直接定位目标分片，单次写入延迟控制在2ms以内，16个节点的总写入吞吐可达单节点的14–15倍（余量用于节点间管理开销）。

**区服分片的公会系统**：同一区服内的所有公会数据、成员关系表、公会战记录存储于同一分片节点。公会查询（成员列表、贡献排名）完全在单分片内完成，无需跨节点联查。当某个热门区服的数据增长超过单节点容量时，运维团队可将该区服迁移至独立的更大节点，而不影响其他区服。

**大型MMO的二级分片方案**：国内某款头部MMO在区服分片基础上，对每个区服内的玩家数据再按player_id进行二级哈希分片，形成"区服→玩家"的两级分片树，使单区服支持的并发在线玩家从5万扩展至50万。

## 常见误区

**误区一：分片数量越多越好**。增加分片数量确实提升了写吞吐上限，但同时放大了跨分片查询的代价。当分片数N从8增至64时，Scatter-Gather查询的响应时间几乎同比增长，且分布式事务的协调开销也随N的平方级别增加。实践中游戏后端通常选择8、16、32等2的幂次作为分片数，以便后续扩容时通过分裂（Split）操作减少数据迁移量。

**误区二：分片后可以使用外键约束维护跨表一致性**。绝大多数分片中间件（如ShardingSphere、Vitess）在跨分片的表之间不支持数据库层面的FOREIGN KEY约束，因为外键两端的行可能分布在不同物理节点上，数据库引擎无法跨连接校验。游戏中玩家表与背包表的引用完整性必须由应用层逻辑保证。

**误区三：分片键确定后可以随时修改**。一旦数据库投入生产并积累了数亿行数据，变更分片键意味着对全量数据重新计算分片归属并完成迁移，这一过程在不停服的条件下通常需要数周的双写过渡期（同时写入新旧两套分片方案），技术风险极高。因此，分片键的设计必须在玩家数据模型确定阶段就与预期的业务查询模式一同评估。

## 知识关联

数据分片的设计依赖**玩家数据模型**中确定的核心实体结构——只有在明确player_id是唯一稳定标识符、区服ID是独立隔离维度之后，才能评估两类分片键的适用性。玩家数据模型中定义的表关系（背包与玩家的一对多、公会与成员的多对多）直接决定了哪些查询会触发跨分片问题。

分片架构落地后，自然引出两个后续问题：当主分片节点故障时如何保障数据可用性，以及多个分片副本之间如何同步状态——这正是**数据复制**所解决的范畴。而多个分片副本之间的数据同步不可能做到严格实时，副本之间存在短暂的状态不一致窗口，这一现象及其容忍策略构成**最终一致性**的核心议题。三者共同构成网络多人游戏分布式数据库架构的完整技术链条。
