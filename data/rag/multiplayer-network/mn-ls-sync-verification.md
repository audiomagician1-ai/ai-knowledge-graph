# 同步校验

## 概述

同步校验（Sync Verification）是帧同步网络游戏中用于检测各客户端游戏状态是否一致的技术手段，其核心做法是在每一逻辑帧（或每隔若干帧）结束后，对游戏世界的关键状态数据计算一个固定长度的数字指纹（哈希值或CRC校验码），并在所有玩家之间广播比对该指纹。若任意两个客户端的指纹不相符，则判定发生了"不同步"（Desync）。

该技术在 StarCraft（星际争霸，1998年）和 Age of Empires II（世纪帝国II，1999年）时代已有应用。这两款经典RTS游戏均采用帧同步架构，其重放系统（Replay System）本质上也依赖同步校验——重放时重新计算每帧哈希值，与录制时保存的哈希值对比，一旦发现差异即说明重放出现了漂移。这一历史背景说明同步校验是帧同步可靠性的根本保障，没有它就无法区分"游戏本身存在不确定性"还是"网络传输出现了问题"。

同步校验之所以不可或缺，是因为帧同步架构的前提——确定性执行——在工程实践中极难完美保证。浮点数运算跨平台精度差异、第三方物理引擎的内部状态、多线程竞态等因素都可能悄悄引入微小偏差。没有同步校验，这类偏差会随帧数累积，最终造成游戏结果南辕北辙，而开发者却毫不知情。

## 核心原理

### 状态快照与哈希计算

每帧结束时，客户端对"游戏状态"进行序列化，提取所有影响逻辑结果的字段——包括每个单位的位置坐标（通常为定点数）、生命值、行为状态机节点、随机数生成器（RNG）的当前种子值等——将这些数据拼成字节流，然后计算哈希值。

常用算法包括：
- **CRC32**：计算速度极快（现代CPU上约 3–8 GB/s 吞吐量），输出 32 位整数，碰撞概率约 1/2³²，适合对性能敏感的高频校验。
- **MD5 / SHA-1**：输出 128 位或 160 位，碰撞概率更低，但计算开销约是 CRC32 的 3–10 倍，常用于对关键帧的精确校验。
- **FNV-1a（Fowler–Noll–Vo）**：非加密哈希，延迟极低，在《守望先锋》等游戏的帧同步模块中有文档化使用记录。

哈希公式可简化为：

```
H_frame = Hash(serialize(GameState_frame))
```

其中 `GameState_frame` 包含且仅包含纯逻辑层数据，渲染层（粒子效果、相机位置等）不参与计算，以避免因渲染差异触发误报。

### 校验帧的选取策略

并非每帧都需要广播哈希，常见策略有三种：

1. **逐帧校验**：每逻辑帧都广播，延迟检测最低，但带宽与服务器压力大。若逻辑帧率为 20 Hz，每客户端每秒需额外发送 20 条 4–16 字节的校验消息。
2. **固定间隔校验**：每 N 帧（如每 10 帧，即 0.5 秒）校验一次，折中方案，Warcraft III 采用类似间隔。
3. **关键帧触发校验**：仅在关键事件发生时（如玩家输入结算、英雄死亡、建筑建造完成）强制校验，可大幅减少无意义的校验广播。

### 广播与比对机制

在星型拓扑（客户端-服务器）架构中，各客户端将本帧哈希值上报给服务器，服务器收集到所有 N 个客户端的哈希后执行比对：若存在任何哈希值与多数值（Majority Hash）不同，则标记该客户端为"疑似不同步"。在 P2P 架构中，每个客户端向其他所有客户端广播自身哈希，再本地执行 N×N 比对。

服务器端比对的伪逻辑如下：

```
majority = most_frequent(hashes[0..N-1])
for each client_i:
    if hashes[i] != majority:
        flag_desync(client_i, frame_id)
```

## 实际应用

**《王者荣耀》帧同步实践**：腾讯技术团队公开分享过其帧同步系统中每隔 5 逻辑帧（约 166ms）进行一次 CRC32 校验的方案。一旦检测到不同步，服务器会记录该帧的完整状态快照用于离线分析，帮助工程师定位是定点数运算、碰撞检测还是技能公式出现了非确定性行为。

**重放系统验证**：《英雄联盟》早期重放系统曾出现重放与实际对局不一致的问题，根本原因是部分运算依赖了非确定性系统时间戳。引入逐帧 CRC 校验后，QA 团队能在数分钟内定位到具体是哪一帧、哪一个对象的状态开始偏离，将排查时间从数天压缩到数小时。

**调试辅助工具**：开发阶段可构建"哈希日志比较器"——在两台本地机器上运行同一场对局，导出每帧哈希序列到文本文件后做 diff 对比，可精确定位第一次出现差异的帧编号，再结合该帧的完整状态 dump 即可快速找出非确定性代码路径。

## 常见误区

**误区一：校验的状态越多越安全**

很多开发者误以为将所有数据（包括 UI 状态、音效播放队列、渲染粒子数量）都纳入哈希计算能更彻底地发现不同步。实际上这会造成大量误报——不同客户端的帧率不同导致粒子数量差异，UI 动画进度不一致，这些都与逻辑结果无关。正确做法是严格划分"逻辑层"与"表现层"，只对逻辑层状态做校验。《命令与征服》系列在早期版本中就曾因误将渲染对象计入校验导致大量虚假不同步报告。

**误区二：哈希碰撞会导致漏检不同步**

使用 CRC32 时，两个不同状态恰好产生相同 CRC32 值的概率约为 1/4,294,967,296（约 2.3×10⁻¹⁰），在实际游戏中几乎不可能在有意义的时间窗口内发生。因此在性能敏感场景下无需为了降低碰撞概率而升级到 SHA-256，CRC32 或 FNV-1a 的安全裕量完全足够（同步校验不是加密场景，无需防止有意构造的碰撞）。

**误区三：校验通过则状态一定完全一致**

同步校验只能保证被纳入哈希计算的字段一致。若某个影响结果的变量（如某子系统的内部计时器）被遗漏在校验范围之外，即使每帧校验全部通过，游戏仍然可能在数百帧后出现分叉。这类"静默不同步"（Silent Desync）往往在压力测试或特定操作序列下才会暴露，是帧同步系统最难排查的问题之一。

## 知识关联

同步校验建立在**确定性帧同步**的基础上：只有当所有客户端执行相同输入序列时，同一帧的哈希值才理论上应该相同；如果帧同步本身的输入广播不可靠，同步校验检测到的"不同步"将无法区分是确定性破坏还是输入丢失，导致误判。因此同步校验的有效性以确定性帧同步的正确实现为前提。

同步校验检测到异常后，下一步需要的技术是**不同步检测（Desync Detection）**——这是一套更完整的诊断与处理流程，包括：确定哪个客户端是"少数方"（出错方）、是否需要断开该客户端连接、是否触发状态同步修复（State Reconciliation）或直接判负处理。同步校验提供的帧编号和客户端标识信息是不同步检测流程的直接输入，两者形成紧密的检测-处理链条。
