# 服务器调和

## 概述

服务器调和（Server Reconciliation）是网络多人游戏中客户端预测技术的纠错机制：客户端在本地执行预测后，持续将每一条输入指令连同其序列号（sequence number）发送给服务器；当服务器的权威状态回包到达时，客户端将本地状态回滚到该权威帧，然后将所有尚未被服务器确认的待定输入（pending inputs）按顺序重新模拟一遍，最终得到经过纠错的最新本地状态。

这一技术由加布里埃尔·甘博（Gabriel Gambetta）在其著名系列文章《Fast-Paced Multiplayer》（约2015年系统整理）中得到广泛传播，但实践最早可追溯至1996年《Quake》的客户端-服务器架构设计。它之所以不可或缺，是因为纯粹的客户端预测在遭遇服务器校正时会产生视觉抖动甚至状态跳变——若不做调和，玩家会看到角色突然"瞬移"到服务器认为的正确位置。服务器调和将这种跳变转化为平滑的无缝修正。

## 核心原理

### 输入序列号与待定输入缓冲

客户端为每一帧的输入分配单调递增的序列号，例如第42帧的输入为 `{seq: 42, moveX: 1, moveY: 0, timestamp: 16ms}`。这些输入不会在发送后立刻丢弃，而是存入**待定输入队列**（pending input buffer）。服务器处理完某条输入后，会在状态回包中附带它所处理的最后一条输入序列号，称为 `last_processed_input`。客户端收到该回包后，将队列中序列号 ≤ `last_processed_input` 的所有条目全部清除，剩余的便是尚待重放的待定输入。

### 回滚与重放流程

调和的核心公式可表示为：

```
reconciled_state = replay(server_state, pending_inputs[last_processed_input+1 .. now])
```

具体步骤如下：
1. **保存权威状态**：将服务器回包中的位置、速度等字段写入本地状态，覆盖同一时间点的预测值；
2. **重放待定输入**：以与服务器完全相同的物理积分步长（通常为固定16ms或20ms一帧）依次执行每条待定输入；
3. **得到调和结果**：重放完毕后的状态即为既经服务器校正、又包含本地最新输入的综合状态。

这里的关键约束是**确定性（determinism）**：客户端的物理积分逻辑必须与服务器逐位（bit-exact）一致，否则每次重放会累积误差，导致调和结果仍然偏离服务器预期。为此，许多引擎（如Valve的Source引擎）使用定点数或严格锁步的浮点运算来保证跨平台一致性。

### 误差阈值与平滑处理

并非所有的服务器-客户端偏差都需要强制回滚。常见实现会设置一个位置容差，例如偏差小于0.05单位时直接线性插值修正，超过该阈值才执行完整的回滚-重放流程。《Overwatch》的网络架构文档（GDC 2017，来自Blizzard工程师Glenn Fiedler等人的相关分享）提到，对于高速移动角色，该阈值会动态放大，以避免频繁的重放计算拖累客户端帧率。待定输入队列一般控制在200ms内的输入量，超出部分直接丢弃，以防止在极端网络抖动下重放成本无限上升。

## 实际应用

**第一人称射击游戏**：在《反恐精英：全球攻势》（CS:GO）中，客户端以64或128 tick速率向服务器发送输入，服务器返回 `last_processed_input` 和位置校正。客户端维护的待定输入队列通常包含约3至8条输入（对应约50ms的网络往返延迟），调和后角色移动几乎无可察觉的抖动。

**实体区分应用**：服务器调和通常仅应用于**本地玩家**自身的角色，其他玩家的实体由独立的实体插值（Entity Interpolation）系统处理，两套系统并行工作而互不干扰。

**角色能力取消的一致性问题**：若服务器判定某次技能释放无效（如CD未冷却完毕），则调和时必须撤销本地已预测的技能特效和状态变化，这要求游戏状态快照必须包含完整的技能冷却计时器，而非仅位置和速度信息。

## 常见误区

**误区一：服务器调和等同于直接用服务器位置覆盖本地位置**
直接覆盖会丢弃客户端所有未确认的新输入，造成角色在视觉上"退回"到100ms之前的位置。真正的调和是将权威状态作为基础，在其之上重放所有待定输入，因此最终结果仍然接近当前时刻，而非过去时刻。

**误区二：只要网络好就不需要调和**
即便在10ms延迟的局域网内，服务器也有可能因碰撞检测、玩家交互等原因否定客户端预测。服务器调和处理的不仅是延迟，还有**逻辑不一致**：服务器可能认为某堵墙阻挡了移动，而客户端预测未考虑到另一个玩家同时推开了该墙。

**误区三：待定输入无限保留更安全**
无限保留待定输入会在高延迟或断线重连场景下导致重放计算量爆炸。若一次重放需处理500条输入，在60fps的客户端上可能占用超过8ms的单帧预算，引发帧率下降。因此必须设置队列上限，并结合丢包处理策略共同设计。

## 知识关联

服务器调和直接依赖**客户端预测**：没有预测，就没有需要调和的待定输入队列。客户端预测让玩家输入立即响应，而调和保证这种即时响应不会因服务器校正而产生视觉断裂——两者共同构成了一套完整的"预测-纠错"闭环。

学习服务器调和之后，自然引出**延迟补偿（Lag Compensation）**的需求：调和解决了本地玩家自身的状态一致性，但当本地玩家向服务器上报"我在T时刻击中了对方"时，服务器需要将整个世界状态回溯到T时刻进行命中判定，这便是延迟补偿要解决的问题。理解了调和中"权威状态 + 重放"的思维模型，有助于直接迁移理解延迟补偿中"历史状态回溯 + 校验"的机制。
