# RTS网络方案

## 概述

RTS网络方案是专为实时策略游戏设计的帧同步网络解决方案，其核心思想来源于1990年代早期局域网对战游戏的工程实践。Westwood Studios 在1995年开发《命令与征服》时，首次将"锁步协议"（Lockstep Protocol）系统化应用于RTS对战，确立了该方案的基本架构：所有客户端只传输玩家操作指令，而非游戏状态，服务器（或P2P对等节点）收集并广播当前帧的所有指令，各端本地独立推进模拟。

该方案之所以能在RTS游戏中长期占主导地位，根本原因在于RTS单局内的可操作单位数量可达数百乃至数千，若采用状态同步则需每帧传输数千个单位的位置、血量、行为状态，带宽开销极难控制。而帧同步方案中每帧传输的数据量与"玩家发出的指令条数"正相关，而非与"场上单位数量"正相关，这使得即便是复杂的千人混战局面，网络流量也保持在可接受范围内。

从《星际争霸》到《魔兽争霸III》，再到《英雄联盟》早期版本，RTS网络方案已演化出多个分支，但锁步-指令广播的核心范式从未改变。理解该方案是开发任何RTS或MOBA类在线对战游戏的必要前提。

## 核心原理

### 指令帧模型

RTS网络方案将游戏时间切分为固定大小的"逻辑帧"（Logic Turn），《魔兽争霸III》采用的逻辑帧时长为200ms，每帧收集玩家在该帧内的所有指令（移动、攻击、建造等）并打包成"指令包"。客户端在本地缓存自己的指令，同时等待接收其他所有玩家的指令包后，才执行本帧的模拟推进。这种"先收齐再执行"的机制是保证所有客户端状态一致的关键。

具体流程为：  
1. 帧 T 开始，本地玩家输入被记录为 `CommandPacket(playerID, frameT, [cmd1, cmd2, ...])`  
2. 该包被广播到所有对等节点  
3. 所有节点收齐帧 T 的全部玩家指令包后，按相同顺序执行  
4. 推进到帧 T+1

### 输入延迟（Input Delay）机制

为了避免每帧都等待网络传输而造成卡顿，RTS方案引入了固定输入延迟（Fixed Input Delay）：玩家在第 T 帧的操作，实际在第 T+D 帧才被执行，其中 D 为预设的延迟帧数。《星际争霸：母巢之战》在局域网下 D 通常为2帧（约83ms），在广域网对战中自动调升至4~6帧。这一机制将网络传输的不确定性"吸收"进固定延迟内，确保本地模拟始终有充足时间收到远端指令，避免因等待而暂停。

延迟帧数 D 的选取遵循经验公式：`D ≥ ceil(RTT_max / frame_duration)`，其中 `RTT_max` 为网络最大往返延迟，`frame_duration` 为单帧时长。若 RTT=100ms，帧长=50ms，则 D≥2。

### 确定性要求与随机数同步

RTS网络方案强依赖底层确定性帧同步的一个特殊场景是随机数同步。RTS游戏中的命中率、暴击、散布等随机事件必须在所有客户端产生完全相同的结果。标准做法是所有客户端共享同一个随机数种子（在游戏开始时由服务器下发，如《魔兽争霸III》在房间创建时广播一个32位初始种子），之后每个客户端使用相同的伪随机数生成器（通常为线性同余生成器 LCG：`X(n+1) = (a*X(n) + c) mod m`）按相同调用顺序推进，保证随机结果完全一致。

### 断线重连与录像回放

RTS网络方案天然支持录像功能：只需记录初始种子和每帧的指令包序列，即可在任意时刻完整重现整局游戏，文件体积极小（《星际争霸》一盘20分钟的录像文件通常小于100KB）。断线重连则通过"快进模拟"实现——重连客户端从帧0开始，利用保存的指令记录高速重放至当前帧，追上在线玩家进度。

## 实际应用

**《魔兽争霸III》对战系统**：采用P2P架构，每200ms为一个逻辑帧，玩家指令由各端直接互传而非经过中央服务器转发。游戏内置了帧等待检测，若某玩家300ms内未收到指定客户端的指令包，游戏画面显示"等待玩家连接"并暂停推进，超时后踢出该玩家。

**《英雄联盟》早期架构**：直到2012年前仍使用帧同步方案处理技能和单位逻辑，每帧30ms，5v5局在高峰期每帧传输的指令数据约为每玩家50~200字节，10名玩家合计不超过2KB/帧，带宽占用极低。

**《部落冲突：皇室战争》**：移动端RTS/卡牌混合游戏，采用客户端-服务器帧同步，帧长100ms，服务器收集双方出牌指令后广播，单局流量控制在约10KB/s以内，适配移动网络环境。

## 常见误区

**误区一：认为帧同步方案不需要服务器**  
虽然早期RTS采用P2P架构，但P2P方案存在严重的作弊隐患：任何一个客户端都可以修改自己的指令包或拒绝转发其他玩家的指令。现代竞技RTS（如《星际争霸II》）已改用中央服务器收集并转发指令，服务器本身不运行游戏逻辑，仅做"指令中继"，但这足以防止单边作弊和指令篡改。

**误区二：认为降低帧率可以减少延迟感**  
实际上，逻辑帧时长越长（帧率越低），输入延迟反而越高，因为 D 帧的延迟以帧时长为单位计算。将帧长从100ms降至50ms（帧率从10fps提升到20fps），在相同RTT下可将感知延迟缩短近一半。RTS方案中逻辑帧率与渲染帧率是解耦的，通常逻辑帧率10~30fps，渲染帧率保持60fps，中间通过插值平滑画面。

**误区三：认为RTS方案无法处理大量单位的精确同步**  
该方案根本不传输单位状态，每帧传输的是玩家发出的"指令"而非单位数据。场上有1000个单位移动，网络流量不会因此增加——增加的只是本地CPU的计算负担，而非带宽。这正是该方案在RTS领域的核心优势。

## 知识关联

RTS网络方案直接建立在**确定性帧同步**的基础上：确定性帧同步保证相同输入序列产生相同输出，RTS网络方案在此之上增加了具体的指令收集协议、输入延迟补偿机制、P2P或中继服务器拓扑选择，以及随机数种子同步等工程细节。如果底层的浮点数不确定性或物理引擎非确定性问题未解决，RTS网络方案会立即出现"状态分歧"（Desync），表现为不同客户端看到的单位位置逐渐偏离，最终导致游戏结果不一致。因此，整数定点数运算替代浮点数（如使用Q16.16格式）是RTS方案落地的前置工程要求，而非可选优化。
