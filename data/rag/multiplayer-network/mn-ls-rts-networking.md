# RTS网络方案

## 概述

RTS网络方案是专为实时策略游戏设计的帧同步网络解决方案，其核心思想来源于1990年代早期局域网对战游戏的工程实践。Westwood Studios 在1995年开发《命令与征服》（*Command & Conquer*）时，首次将"锁步协议"（Lockstep Protocol）系统化应用于RTS对战，确立了该方案的基本架构：所有客户端只传输玩家操作指令，而非游戏状态，服务器（或P2P对等节点）收集并广播当前帧的所有指令，各端本地独立推进模拟。这一架构奠定了此后二十余年RTS网络同步的基础范式。

该方案能在RTS游戏中长期占主导地位，根本原因在于RTS单局内可操作单位数量可达数百乃至数千。若采用状态同步，则需每帧传输数千个单位的位置（3×float）、血量（int）、行为状态（enum），仅100个单位每帧就需至少约1.6KB的原始数据，以30fps计算带宽开销约达384KB/s，在1990年代56kbps的拨号网络下完全不可行。而帧同步方案中每帧传输的数据量与"玩家发出的指令条数"正相关，而非与"场上单位数量"正相关：一条移动指令仅需约10~20字节（玩家ID 1B + 帧号 2B + 指令类型 1B + 目标坐标 8B），使得即便千人混战，网络流量仍可控制在1~5KB/s以内。

从《星际争霸》（1998）到《魔兽争霸III》（2002），再到《英雄联盟》早期版本（2009），RTS网络方案已演化出多个分支，但"锁步-指令广播"的核心范式从未改变。学术界对该范式的系统性总结可参见 Terrence C. Cronin 的技术报告《Networking for Game Programmers》（GDC 2000）以及 Glenn Fiedler 的系列文章《Deterministic Lockstep》（2014，gafferongames.com），后者给出了大量可量化的工程参数建议。

## 核心原理

### 指令帧模型（Lockstep Turn Model）

RTS网络方案将游戏时间切分为固定大小的"逻辑帧"（Logic Turn）。《魔兽争霸III》采用的逻辑帧时长为200ms（即5帧/秒的模拟频率，渲染帧独立插值），《星际争霸：母巢之战》采用约41.67ms（24逻辑帧/秒）。每帧收集玩家在该帧内的所有指令（移动、攻击、建造、技能释放等）并打包成"指令包"（Command Packet）。客户端本地缓存自己的指令，同时等待接收所有其他玩家的指令包，收齐后按预定的玩家ID排序依次执行，再推进到下一帧。

完整的单帧处理流程如下：

```
// 伪代码：单逻辑帧执行流程
function ProcessLogicFrame(frameIndex T):
    // Step 1: 收集本地输入，打包并广播
    localCmds = CollectLocalInput()
    packet = CommandPacket(localPlayerID, T + D, localCmds)  // D为输入延迟帧数
    Broadcast(packet)
    inputBuffer[T + D][localPlayerID] = packet

    // Step 2: 等待收齐所有玩家在帧T的指令（含本地已缓存）
    while not HasAllPackets(frameIndex = T, playerCount = N):
        ReceiveNetworkPackets()  // 阻塞式或轮询

    // Step 3: 按玩家ID升序执行所有指令，保证顺序一致性
    for playerID in sorted(0 .. N-1):
        ExecuteCommands(inputBuffer[T][playerID].commands)

    // Step 4: 推进物理/AI模拟一个逻辑帧
    SimulateWorld(deltaTime = frameDuration)
    frameIndex += 1
```

"先收齐再执行"的机制是保证所有客户端状态完全一致的关键——任何一个客户端缺少任意玩家的指令包，该帧模拟均不得推进。

### 输入延迟（Fixed Input Delay）机制

为避免每帧阻塞等待网络传输而造成游戏卡顿，RTS方案引入固定输入延迟（Fixed Input Delay）：玩家在第 $T$ 帧的操作，实际在第 $T+D$ 帧才被执行，其中 $D$ 为预设的延迟帧数。这一机制将网络传输的不确定性"预吸收"进固定延迟内，确保本地模拟在执行第 $T+D$ 帧时已有充足时间收到所有远端在第 $T$ 帧发出的指令包。

延迟帧数 $D$ 的选取遵循以下公式：

$$D \geq \left\lceil \frac{RTT_{\max}}{T_{\text{frame}}} \right\rceil$$

其中 $RTT_{\max}$ 为当前网络会话中所有玩家间最大往返延迟（单位ms），$T_{\text{frame}}$ 为单逻辑帧时长（单位ms）。

**例如**：若最差连接的 $RTT_{\max} = 120\text{ms}$，逻辑帧时长 $T_{\text{frame}} = 50\text{ms}$，则 $D \geq \lceil 120/50 \rceil = 3$ 帧。《星际争霸：母巢之战》在局域网下 $D$ 通常为2帧（约83ms感知延迟），广域网战网对战中自动调升至4~6帧（167~250ms感知延迟）。《魔兽争霸III》引入了动态输入延迟机制（Dynamic Input Delay），每隔约5秒检测一次网络RTT并重新协商 $D$ 值，在网络质量改善时能自动缩短延迟，改善游戏体验。

### 确定性要求与随机数同步

RTS网络方案强依赖确定性帧同步的最敏感场景是随机事件同步。RTS游戏中的命中率、暴击判定、单位散布角、AI决策扰动等随机事件必须在所有客户端产生完全相同的结果，否则将导致状态分叉（Desync）。标准做法是所有客户端共享同一个随机数种子：游戏开始时由主机或服务器广播一个32位初始种子（《魔兽争霸III》在房间创建时通过Battle.net服务器下发），之后每个客户端使用完全相同的伪随机数生成器（PRNG）按相同调用顺序推进。

《星际争霸》系列使用的是线性同余生成器（LCG），其递推公式为：

$$X_{n+1} = (1103515245 \cdot X_n + 12345) \bmod 2^{31}$$

其中 $a = 1103515245$，$c = 12345$，$m = 2^{31}$，为标准C库 `rand()` 的Glibc实现参数。各客户端必须以完全相同的调用次序消耗随机数，因此所有涉及随机的逻辑（包括AI的随机路径选取）均需纳入"确定性逻辑帧"中执行，严禁在渲染线程或异步线程中消耗共享随机数序列。

## 断线重连与录像回放

RTS网络方案天然支持录像（Replay）功能，原理极为简洁：只需持久化两项数据——**初始随机种子**与**逐帧指令包序列**——即可在任意时刻以任意速度完整重现整局游戏。录像文件体积极小：一场45分钟的《星际争霸：母巢之战》天梯对局，录像文件（.rep格式）通常仅有100~300KB，远小于同时长视频的几百MB，根本原因正是存储的是"指令"而非"画面"。

断线重连（Reconnect）则是该方案的工程难点。由于缺失期间的指令包无法补全，断线玩家必须从某一关键帧的完整快照（Snapshot）出发，重放该快照之后的所有指令包，才能重新追上当前帧。《魔兽争霸III》每隔约60秒生成一次全局状态快照（序列化所有单位、建筑、资源的完整状态），断线重连时最多需要重放60秒的指令历史，重放速度可以超实时（以10×速度回放约需6秒），实现快速追帧。若两名玩家检测到第三方断线，则进入"托管AI"模式（由主机代为操控断线玩家的单位），避免整局游戏因单人断线而中止。

## 反作弊与状态校验（Checksum验证）

纯P2P架构的RTS网络方案存在作弊风险：任意客户端均可篡改本地游戏逻辑（例如修改命中率计算），而其他节点无法感知。工业界的对策是引入周期性**状态校验和**（State Checksum）机制：每隔固定帧数（《魔兽争霸III》约每10逻辑帧，即2秒），各客户端计算当前游戏状态的哈希值（通常为CRC32或Adler-32）并广播给所有其他节点。若任意两个节点的校验和不一致，则立即判定发生了状态分叉（Desync），游戏终止并记录错误日志。

校验和的计算对象需覆盖所有影响游戏逻辑的状态字段：单位坐标（定点数）、当前血量、行为状态机的状态枚举、当前随机数生成器的 $X_n$ 值等。浮点数坐标因跨平台精度差异（x86与ARM的FPU舍入规则不同）历来是Desync的高发区，这也是RTS引擎普遍采用**32位或64位定点数**（Fixed-Point Arithmetic）替代浮点数进行物理运算的根本原因。

## 实际应用案例

### 《星际争霸II》的混合架构

暴雪在《星际争霸II》（2010）开发时，面对单局最多可同屏出现500+单位的设计目标，对传统RTS锁步方案做出重要改进：引入**服务器中继节点**（Battle.net Game Server）取代纯P2P架构，所有玩家的指令包先上传至中继服务器，服务器统一打时间戳后转发给所有客户端，解决了P2P网络中"谁等谁"的协调问题，并将反作弊责任从客户端转移至服务器侧。逻辑帧频率提升至16帧/秒（62.5ms/帧），同时引入帧预测机制：若某玩家在截止时间前未能上传指令包，服务器将其本帧输入视为"空指令"继续推进，并在下一帧携带补偿包。

### 《英雄联盟》的历史演变

《英雄联盟》在2009~2014年间采用帧同步方案，逻辑帧率为30帧/秒，固定输入延迟为4帧（约133ms）。然而由于其使用C++与Lua混合脚本驱动的技能系统，Lua脚本中大量使用了浮点数运算，加之跨平台版本（Mac客户端与Windows客户端的浮点行为差异），Desync问题长期困扰运营团队。自2015年起，Riot逐步将核心玩法迁移至状态同步架构，这一历史案例被学界（参见 Luca Muscariello,《Game Networking》, GDC Vault 2016）引用为"确定性维护成本过高导致放弃帧同步"的典型案例。

## 常见误区

**误区一：逻辑帧越短，游戏响应越流畅。**
实际上，逻辑帧时长缩短会成比例增加网络同步频率和Checksum计算频率，并压缩输入延迟帧数的容错空间。《魔兽争霸III》刻意选用200ms的较长逻辑帧，正是为了在广域网RTT普遍较高（100~200ms）的2002年网络环境下，用 $D=1$ 帧即可吸收大部分延迟，将玩家感知的操作延迟控制在200ms以内。
