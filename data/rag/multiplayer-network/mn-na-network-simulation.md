# 网络模拟测试

## 概述

网络模拟测试（Network Simulation Testing）是在多人游戏开发过程中，通过人为引入延迟（Latency）、丢包（Packet Loss）、乱序（Packet Reordering）、带宽限制（Bandwidth Throttling）等异常网络条件，对游戏的网络协议实现和同步逻辑进行验证的一种测试方法。其根本目的是在受控的实验室环境中复现玩家在真实网络下会遭遇的各类恶劣状况，从而在上线前提前发现并修复潜在问题。

网络模拟测试的实践可追溯至1990年代中期，随着《Quake》（1996年）等早期联网射击游戏的诞生，开发者发现局域网测试结果与广域网真实体验之间存在巨大落差，这促使行业开始系统性地研究如何在开发阶段模拟广域网环境。现代游戏引擎和网络中间件（如Photon、Mirror）已将网络模拟功能作为标准调试工具内置其中。

对于多人游戏而言，网络模拟测试的意义在于：玩家的真实网络条件极为多样——根据全球宽带统计，移动4G网络的平均延迟约为50~100ms，而国际跨洲连接可达200ms以上，丢包率从0.1%到5%不等。如果游戏代码仅在0ms延迟的本地回环（Loopback）下测试，客户端预测、服务端权威校验、延迟补偿等机制的实际效果根本无法得到验证。

## 核心原理

### 延迟模拟的工作机制

延迟模拟的核心是在数据包发送与实际传输之间插入一个人工等待队列。以Unity的Multiplayer Tools Network Simulator为例，当开发者设置"模拟延迟 = 100ms"时，工具会在每个数据包离开本机socket之前将其暂存100毫秒再放行。更真实的做法是模拟"延迟抖动"（Jitter），即延迟并非固定值而是在某个范围内随机波动，例如设置均值100ms、抖动±30ms的正态分布延迟，以接近真实网络的行为特征。**延迟抖动是导致客户端预测抖动（Prediction Jitter）的直接原因**，这一点在纯延迟模拟中很容易被忽视。

### 丢包模拟与协议压力测试

丢包模拟通过按概率随机丢弃数据包来触发游戏网络协议的重传和纠错路径。常见的测试配置参数为：**丢包率设置为5%时，基于UDP的游戏协议通常可以正常运行；当丢包率达到15%时，未经优化的可靠UDP实现会出现明显卡顿；超过25%则几乎所有实时游戏协议都会出现严重的状态不一致**。通过这组梯度测试，可以量化游戏的网络容错边界。对于使用自定义可靠层（Reliability Layer）的协议，丢包模拟还能验证ACK确认包、重传定时器（通常设为RTT×1.5）是否正确实现。

### 乱序与重复包测试

乱序测试模拟数据包因路由路径不同而以非发送顺序到达的情况。例如，先发送的第10号序列包反而在第11号包之后到达。这是测试序列号（Sequence Number）处理逻辑的关键手段——一个正确的实现应当能够缓存乱序包直到缺失包到达或超时，而错误的实现会直接丢弃乱序包，导致游戏状态回滚或位置跳变。重复包测试则向接收端发送同一个数据包的两份副本，验证接收方是否通过序列号去重机制正确丢弃重复数据，防止玩家受到两次伤害或触发两次事件。

### 常用工具与参数配置

- **Linux tc（Traffic Control）命令**：通过`tc qdisc add dev eth0 root netem delay 100ms 30ms loss 5%`可在操作系统层面模拟延迟和丢包，适合服务端压力测试。
- **Clumsy（Windows）**：专为游戏开发者设计的Windows网络模拟工具，支持实时调整延迟、丢包、乱序参数，无需重启应用。
- **引擎内置模拟器**：Unity Transport Package自带Network Simulator组件，可直接在Editor中对客户端-服务器连接注入模拟参数，适合快速迭代测试。

## 实际应用

**客户端预测验证**：在射击游戏中测试本地玩家移动预测是否在100ms延迟下仍保持流畅，方法是启动Clumsy设置单向延迟100ms，然后操控角色移动，观察角色是否出现橡皮筋（Rubber-banding）回弹现象。如果出现回弹，说明服务端校验逻辑过于严格或客户端预测状态与服务端不一致。

**断线重连测试**：将丢包率临时设为100%持续3秒，模拟短暂断网，验证游戏的断线重连机制能否在网络恢复后正确重建游戏状态，而不是将玩家踢出房间。这对于手游（移动网络切换频繁）尤为重要。

**大厅与房间系统压力测试**：在200ms高延迟环境下测试匹配系统的超时阈值设置是否合理——若超时阈值设置为500ms，在200ms延迟下一次往返（RTT）耗时400ms，加上服务器处理时间极易触发误超时，需要将阈值调整至1500ms以上。

## 常见误区

**误区一：只测试延迟，忽略抖动**。许多开发者在测试时将延迟设置为固定的100ms，认为这已经足够严苛。然而真实网络的延迟抖动（Jitter）才是导致插值（Interpolation）缓冲区设计失败的元凶。固定100ms延迟下运行完美的插值逻辑，在±50ms抖动环境下可能出现明显的角色位置跳变，因为插值缓冲区的深度设计必须基于最大抖动值而非平均延迟。

**误区二：认为本地模拟完全等同于真实网络**。`tc netem`和Clumsy等工具模拟的是单一链路的统计特性，而真实广域网的丢包往往具有突发性（Burst Loss）——连续3~5个包同时丢失，而非均匀分散。使用`tc netem`的`loss gemodel`（Gilbert-Elliott模型）参数可以模拟突发丢包，参数`loss gemodel 1% 10% 90% 0%`表示进入坏状态的概率为1%，在坏状态下连续丢包率为10%，这比均匀丢包更接近真实的无线网络行为。

**误区三：仅在开发末期进行网络模拟测试**。网络同步逻辑一旦成型便难以重构，若在Alpha测试阶段才首次启用丢包模拟，发现的问题可能牵涉整个帧同步或状态同步架构的重新设计，修复成本极高。正确做法是从实现第一个网络同步特性起便将网络模拟测试纳入每日构建（Daily Build）的自动化测试流程。

## 知识关联

**与网络协议设计的关系**：网络模拟测试是对网络协议设计成果的直接验证手段。在协议设计阶段确定的序列号位宽（如16位序列号支持最多65535个未确认包）、ACK策略（累计确认 vs. 选择性确认SACK）、重传超时（RTO）计算公式（`RTO = SRTT + 4×RTTVAR`，其中SRTT为平滑RTT，RTTVAR为RTT方差），都需要通过模拟测试在真实参数范围内验证其有效性。若协议设计阶段假设"最大丢包率为3%"，则模拟测试中必须覆盖3%这一边界条件以及稍高于此值的场景（如4%、5%），才能确认协议在预期条件内的行为是否符合设计规格，并了解其超出边界时的降级（Degradation）方式。
