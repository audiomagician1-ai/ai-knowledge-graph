# 网络协议设计

## 概述

游戏专用应用层协议是在传输层（UDP或TCP）之上，针对游戏通信特征量身定制的消息格式与交互规范。与HTTP或FTP等通用协议不同，游戏协议必须同时满足低延迟（通常要求端到端往返时间低于100ms）、高频更新（主流射击游戏每秒发送20至64个状态包）以及带宽严格受限（移动端每秒可用带宽常低于50KB）三项相互矛盾的目标。

游戏网络协议的专属设计最早可追溯至1996年《Quake》的NetQuake协议。id Software的约翰·卡马克公开了其核心思路：将游戏状态划分为可丢弃的位置更新与不可丢弃的事件触发两类，前者用UDP发送、丢失不补，后者叠加可靠传输机制。这一分类思想至今仍是游戏协议设计的基础骨架，后续的ENet、RakNet、Mirror Networking等库都是这套思路的工程化延伸。

理解游戏协议设计的价值在于：一个错误的协议决策（如将所有消息设为可靠有序传输）会导致即使底层网络质量良好，游戏体验仍然卡顿——因为可靠传输在丢包时会阻塞整个消息队列，使后续位置更新被迫等待重传完成，产生人为的"队头阻塞"（Head-of-Line Blocking）。

## 核心原理

### 消息分类体系

游戏协议的第一设计决策是将消息按**时效性**和**重要性**分为至少三类：

- **不可靠非有序（Unreliable Unordered）**：玩家坐标、朝向、动画状态等高频位置信息。每帧刷新，旧数据无价值，丢包直接丢弃。
- **可靠有序（Reliable Ordered）**：开枪事件、技能触发、聊天消息。必须到达且顺序不能颠倒，否则游戏逻辑错误。
- **可靠非有序（Reliable Unordered）**：玩家登录确认、背包物品变更。必须到达，但与其他消息顺序无关，可独立重传而不阻塞位置流。

将"可靠有序"与"不可靠"合并进同一消息流是最常见的设计失误。正确做法是为每个类别维护独立的序列号空间，彼此互不干扰。

### 消息头部结构设计

一个典型的游戏协议包头结构如下（以字节为单位）：

```
[协议魔数 2B][消息类型 1B][序列号 2B][确认号 2B][确认位域 4B][时间戳 4B][负载长度 2B]
总计：17字节固定头部
```

其中**确认位域（ACK Bitfield）**是游戏协议的关键创新：用32个比特位表示最近32个包的接收情况，接收方每帧连同自身数据一起回传这32位，发送方由此在单个往返时间内得知过去32个包的送达状态，无需单独发送ACK包。这一机制将确认开销压缩到每包4字节，是Glenn Fiedler在其著名系列文章《Networking for Game Programmers》（2008年）中系统化阐述的方案。

序列号通常选择2字节（范围0至65535），配合循环回绕（Wraparound）处理：判断序列号`s1`是否晚于`s2`的公式为 `(s1 - s2 + 65536) % 65536 < 32768`，避免简单大小比较在回绕时出错。

### 增量压缩与优先级调度

游戏协议不应逐帧发送实体的完整状态，而应只发送与上一个已确认帧相比的**差量（Delta）**。例如，玩家坐标每帧变化通常不超过0.5米，用10比特的有符号增量值替代32比特浮点坐标，单个实体位置数据压缩率可达70%。

当带宽不足以发送所有实体更新时，需要优先级调度：

```
优先级得分 = 基础重要性 × 距本地玩家距离衰减因子 × 自上次发送以来的帧数
```

优先级最高的实体优先填入当前数据包，其余延至下一帧。《Overwatch》的团队在GDC 2017公开分享了类似机制，他们将实体按相关性分档并动态调整发送频率，使6人团战时每客户端带宽控制在8Kbps以内。

## 实际应用

**射击游戏位置同步协议**：以第一人称射击游戏为例，服务器以64Hz向每个客户端广播所有可见玩家的位置增量包，单个玩家的位置+朝向数据压缩后约为10字节，10名玩家同时在线时每秒下行流量为 `64 × 10 × 10 = 6400字节 ≈ 6.4KB/s`，远低于带宽上限。命中事件作为可靠消息独立发送，不影响位置流的连续性。

**MMORPG区域同步协议**：大世界游戏按"感知范围"（Interest Area）过滤实体，服务器只向客户端推送感知范围内（通常半径200米）的实体状态，并对感知范围边缘的实体降低更新频率至10Hz，中心范围保持20Hz。消息头部增加"区域ID"字段，便于客户端快速路由到对应的地图区块处理器。

## 常见误区

**误区一：用TCP就能解决所有可靠性问题**
TCP的Nagle算法会将小包合并延迟发送（默认等待200ms），直接导致游戏输入延迟翻倍。即使关闭Nagle（`TCP_NODELAY`），TCP在丢包时的自动重传同样引发队头阻塞——所有后续数据必须等待丢失包重传完成，而该包携带的位置信息早已过时毫无价值。游戏协议必须在UDP之上自行实现选择性可靠传输，而非依赖TCP的全局可靠保证。

**误区二：序列号越大越好，直接用4字节**
将序列号从2字节扩展到4字节虽然消除了回绕问题，但每个数据包头部多出2字节，在64Hz更新频率下每秒额外消耗 `64 × 2 = 128字节`，对移动端积少成多。正确做法是保留2字节序列号并实现上述回绕比较算法，这是所有主流游戏网络库的通行选择。

**误区三：消息类型用字符串标识**
部分开发者仿照HTTP请求方法用字符串（如`"PLAYER_MOVE"`）标识消息类型，导致每个包浪费10至20字节在类型名称上，且解析需要字符串比较。游戏协议应使用1至2字节的整型消息ID，配合编译期生成的枚举表，解析开销降为O(1)的数组下标访问。

## 知识关联

本概念直接建立在**可靠UDP**的实现机制之上——可靠UDP提供了序列号管理、重传队列和ACK处理的底层机制，而游戏协议设计决定了如何在这些机制上组织消息分类和头部格式。没有明确的消息分类，可靠UDP的重传逻辑无法区分哪些消息需要保证送达、哪些丢弃即可。

掌握消息格式与分类设计后，自然过渡到**RPC框架**——RPC框架本质上是在本文定义的可靠消息通道上，将函数调用序列化为消息ID加参数列表的自动化机制。协议设计中定义的消息类型ID体系，正是RPC框架中"远程方法注册表"的底层映射基础。

在进入**网络模拟测试**阶段时，协议设计的质量会被直接检验：通过人工注入丢包率20%、延迟150ms±50ms的网络干扰，可以观察消息分类是否合理——若位置更新的丢失导致游戏卡顿，说明该类消息被错误地放入了可靠有序队列；若命中事件偶发丢失，则说明可靠传输机制未正确覆盖该消息类型。
