# 后台下载

## 概述

后台下载（Background Download）是指游戏客户端在玩家正常游玩的同时，悄无声息地在后台异步获取并缓存更新资源包的技术机制。与强制玩家等待的前台阻塞式下载不同，后台下载允许玩家在资源尚未完全到位的情况下进入游戏，先下载并解锁当前场景所需的最小资源集，其余内容在后台按优先级队列逐步拉取。《原神》和《王者荣耀》均采用此机制，使玩家在版本更新后无需经历漫长等待即可立刻登录游戏。

这一技术最早在主机游戏领域得到成熟应用。PlayStation 4 于 2013 年发布时将"后台下载补丁"列为系统级功能，允许游戏在待机模式下自动完成更新。移动端游戏随后借鉴该理念，结合 CDN 分片传输，在 2016 年前后逐渐成为大型手游的标配。其核心价值在于将"下载等待时间"从玩家感知路径中剥离出去，直接提升留存率——业界数据显示，要求玩家在进入游戏前等待超过 3 分钟的更新流程会导致约 15%～20% 的即时流失。

后台下载在热更新体系中承担资源预置的职能：CDN 将补丁包切分为若干个 2～4 MB 的分块（Chunk），后台下载模块按需请求这些分块，完成校验后写入本地缓存目录，供热更新加载器在需要时直接读取，从根本上消除了"下载→重启→应用补丁"这一传统三段式流程的中断感。

## 核心原理

### 异步下载线程与主游戏线程隔离

后台下载必须运行在独立的工作线程（Worker Thread）上，与渲染线程和逻辑线程完全隔离。如果下载线程与主线程共享 CPU 时间片而不加以限制，密集的网络 I/O 和磁盘写入操作会造成帧率抖动（Frame Stutter）。通常做法是将下载线程的优先级设置为 `THREAD_PRIORITY_BELOW_NORMAL`（Windows）或对应平台的低优先级值，确保在 CPU 负载高峰期下载线程主动让出资源。Unity 引擎中常用 `UnityWebRequest` 配合 C# 的 `async/await` 或协程实现这一隔离，而 Unreal Engine 则提供 `FHttpModule` 的异步请求接口。

### 带宽限速与网络状况自适应

无限速的后台下载会抢占游戏实时数据包（如玩家位置同步、技能指令）的带宽，导致游戏延迟飙升。实践中后台下载模块通常设置一个可配置的带宽上限，常见默认值为 512 KB/s 至 1 MB/s，并实现令牌桶（Token Bucket）限速算法。令牌桶以固定速率 *r*（字节/秒）生成令牌，桶容量上限为 *b* 字节，每次发送数据前必须消耗等量令牌，公式表示为：

> 可发送量 = min(当前令牌数, 请求数据量)

此外，模块需监听网络类型变化：检测到 Wi-Fi 时可提升至最大配置带宽；切换至移动数据网络时自动降速或暂停，避免消耗玩家流量产生投诉。

### 断点续传与分块校验

由于游戏会话可能随时中断（玩家主动退出、系统杀进程、网络断连），后台下载模块必须支持 HTTP Range 请求以实现断点续传。每个 Chunk 下载完毕后立即计算 MD5 或 SHA-256 校验值并与服务端清单（Manifest）中的预期值比对，校验失败则丢弃该块并重新请求，而不是等到所有内容下载完成后才做整体校验。下载进度持久化到本地的一个轻量级 SQLite 数据库或简单的 JSON 进度文件中，记录每个 Chunk 的下载状态（待下载/进行中/完成/校验失败），重启后直接从该状态恢复而无需重头开始。

### 资源优先级队列

并非所有待更新资源都具有相同的紧迫性。后台下载模块维护一个优先级队列，将资源分为多个等级：玩家即将进入的下一个地图场景资源优先级最高（P0），当前版本新增的活动界面资源次之（P1），背景音乐和非核心美术资源优先级最低（P2）。游戏逻辑层可在运行时动态调整优先级，例如玩家点击进入某副本时，立刻将该副本的资源包提升为 P0 并插队到队列头部。

## 实际应用

**分区解锁式新手引导**：《使命召唤：手游》采用后台下载实现"先下载主城资源，后台下载战场资源"的策略。玩家完成约 200 MB 的基础包安装后即可进入新手教程，教程进行期间后台静默下载对战所需的武器和地图资源包（约 1.2 GB），教程结束时资源恰好就绪，玩家几乎感受不到等待。

**版本日当天的无感更新**：《英雄联盟》PC 端客户端在每周三版本日（北美服务器时间）凌晨结束维护后，客户端自动在后台完成差量补丁下载，玩家打开客户端时补丁已基本下载完毕，仅需数秒的本地应用操作。该客户端使用的是自研的 Patcher 下载管理器，将补丁切分为 4 MB 的分块并行从多个 CDN 节点拉取。

**用户流量感知**：国内手游普遍在后台下载模块中加入"仅 Wi-Fi 下载"的默认设置，玩家首次启动时弹窗确认是否允许移动数据下载。这一设计使移动数据环境下的投诉率显著下降，同时保证了 Wi-Fi 用户的流畅更新体验。

## 常见误区

**误区一：后台下载等同于多线程下载加速**。后台下载强调的是"对玩家透明、不打断游戏进行"，而多线程下载是指同时开启多个 TCP 连接下载不同分块以榨取带宽。两者可以组合使用，但概念不同——一个低优先级的单线程下载也属于后台下载，而一个阻塞主界面的 8 线程并发下载不属于后台下载。

**误区二：后台下载完成后资源立即可用，无需额外步骤**。下载完毕仅代表原始压缩包（通常为 .zip 或自定义 .bundle 格式）已落盘，热更新加载器还需完成解压、路径注册和资源清单更新等步骤才能让运行时读取到新资源。省略这一步会导致文件已存在但游戏仍使用旧版本资源的"幽灵更新"问题。

**误区三：后台下载失败不需要通知玩家**。由于过程静默，部分开发者忽视了失败处理。实际上若关键资源（如下一个大版本的必需内容）在数次重试后仍下载失败，客户端应在玩家下次进入相关功能前提示"资源未就绪，请检查网络后重试"，否则会出现玩家点击进入副本后触发临时前台强制下载的糟糕体验，比没有后台下载时更令人反感。

## 知识关联

后台下载直接依赖**下载管理器**提供的基础能力：分块请求调度、并发连接池管理、断点续传状态记录等均由下载管理器封装，后台下载模块在其上追加了线程隔离、带宽限速和优先级队列逻辑。理解下载管理器中 HTTP Range 头的工作方式（`Range: bytes=1048576-2097151`）是实现后台断点续传的前提。

在 CDN 与热更新的整体链路中，后台下载处于"资源从 CDN 节点到本地缓存"这一传输环节。上游的 CDN 版本清单（Manifest）决定了后台下载的任务列表和分块校验值；下游的热更新资源加载器读取后台下载写入的本地缓存目录，按需将新资源注入运行时的资源映射表。掌握后台下载机制后，开发者可进一步研究**增量热更新**和**资源分包策略**，以精细化控制哪些内容值得后台预取、哪些内容可以延迟到玩家真正需要时再触发下载。
