# 事务设计

## 概述

事务设计（Transaction Design）是指在游戏数据库层面，将多个相关数据库操作捆绑为一个不可分割的原子单元，确保这组操作要么全部成功提交，要么全部回滚至操作前状态。在多人游戏中，最典型的事务场景是玩家A向玩家B转账金币：这一动作必须同时扣减A的余额并增加B的余额，若只有一步成功则会产生金币凭空消失或凭空增加的数据灾难。

事务的概念源自1970年代的关系型数据库理论。IBM研究员Jim Gray在1981年正式提出ACID属性框架：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）和持久性（Durability），这四个属性至今仍是游戏事务设计的核心评判标准。在游戏领域，事务设计的难点不在于理解概念，而在于如何在高并发（如万人同时交易同一热门道具）场景下维持ACID保证而不引起性能崩溃。

游戏事务设计的失误代价极高。2012年《暗黑破坏神3》真实货币拍卖行因事务处理漏洞导致玩家资产被重复扣款，引发大规模玩家投诉；《EVE Online》曾出现因数据库事务隔离级别配置错误导致的星际市场价格数据污染。正确的事务设计是保护游戏经济系统完整性的技术底线。

## 核心原理

### ACID四属性在游戏场景中的具体含义

**原子性**要求游戏中"购买道具"这一操作的所有子步骤（扣除金币、减少商店库存、增加玩家背包道具数量）绑定在同一事务中。任何一步失败，数据库会执行`ROLLBACK`，将所有已执行的子操作撤销。数据库层面的实现依赖Undo Log（撤销日志），MySQL InnoDB引擎将事务前的数据旧版本写入`ibdata`文件中的回滚段。

**隔离性**直接决定了多玩家并发操作时的数据可见性。SQL标准定义了四个隔离级别：读未提交（READ UNCOMMITTED）、读已提交（READ COMMITTED）、可重复读（REPEATABLE READ）和串行化（SERIALIZABLE）。游戏服务器通常选择**可重复读**级别，因为它能防止"幻读"场景——即玩家A在检查背包容量和实际写入道具之间，其他事务插入新道具导致容量超限的问题。

**持久性**依赖WAL（Write-Ahead Logging，预写式日志）机制。MySQL的Redo Log在事务提交时先将变更记录写入磁盘日志，即使此后发生宕机，重启时也能根据Redo Log将未落盘的数据恢复。游戏服务器宕机重启后，玩家刚完成的充值记录不会丢失，依赖的正是这一机制。

### 游戏中五种典型的事务场景

**场景一：玩家间交易**。玩家A出售一把武器给玩家B，需要在单一事务中执行：①检查A背包是否有该武器；②从A背包删除该武器；③从B账户扣除金币；④向A账户增加金币；⑤向B背包添加该武器。五步操作必须原子完成。

**场景二：副本奖励发放**。队伍击败Boss后，系统需要向6名玩家各自发放不同奖励，同时更新Boss击杀计数器。若奖励发放到第4名玩家时服务器故障，未发放的2名玩家需在重启后自动补发，这依赖事务回滚与补偿机制的配合设计。

**场景三：竞价/拍卖场景**。多名玩家同时出价竞拍同一物品时，必须使用`SELECT ... FOR UPDATE`加行锁，确保同一物品不被两个竞价事务同时判断为"当前最高出价者"并双重成交。

**场景四：限量商品秒杀**。库存字段的扣减操作必须使用带条件的原子更新：`UPDATE items SET stock = stock - 1 WHERE item_id = 12345 AND stock > 0`，而非先查询库存再减库存的两步操作，否则在高并发下会出现超卖。

**场景五：跨服转移**。玩家角色从服务器A迁移至服务器B时，因两台服务器使用独立数据库，需要引入**两阶段提交（2PC）**协议：协调者先向两台数据库发送Prepare指令，两台数据库均确认后再发送Commit指令，任何一方失败则全部回滚。

### 事务的隔离问题与锁策略

游戏事务设计必须处理三类并发异常：**脏读**（读取未提交事务的数据）、**不可重复读**（同一事务内两次读取同一行结果不同）、**幻读**（同一事务内两次查询返回的行数不同）。针对游戏排行榜更新场景，脏读可能导致玩家看到一个尚未提交的虚假排名；针对经济系统，幻读可能使一次完整的库存盘点结果出现误差。

锁的粒度选择影响游戏吞吐量。InnoDB提供行级锁（Row Lock）和表级锁（Table Lock）。游戏服务器应尽量使用行级锁，仅锁定被操作玩家的数据行，避免锁住整张`player_wallet`表导致所有交易串行排队。行级锁在使用`WHERE`条件命中索引时才会生效，若`WHERE`条件未命中索引，InnoDB会退化为表级锁——这是游戏数据库性能问题的常见根源。

## 实际应用

在《魔兽世界》早期版本的拍卖行系统中，Blizzard使用的是类似如下的事务结构（伪SQL）：

```sql
BEGIN TRANSACTION;
  -- 步骤1：锁定拍卖记录
  SELECT * FROM auctions WHERE auction_id = ? FOR UPDATE;
  -- 步骤2：校验出价合法性
  -- 步骤3：更新最高出价
  UPDATE auctions SET highest_bid = ?, bidder_id = ? WHERE auction_id = ?;
  -- 步骤4：从竞拍者账户冻结金币
  UPDATE wallets SET frozen = frozen + ? WHERE player_id = ?;
  -- 步骤5：退还被超越竞拍者的冻结金币
  UPDATE wallets SET frozen = frozen - ? WHERE player_id = ?;
COMMIT;
```

此设计通过`FOR UPDATE`悲观锁保证同一拍卖标的不被并发事务重复处理。对于手游中的限时活动商店，通常采用Redis的`WATCH`+`MULTI`+`EXEC`事务命令实现乐观锁版本的原子库存扣减，因为手游流量峰值更高，悲观锁会导致大量请求超时。

## 常见误区

**误区一：认为应用层加互斥锁可以替代数据库事务**。部分开发者在游戏服务器代码中用`mutex`或`synchronized`保护交易逻辑，认为这足够保证数据安全。但应用层锁在服务器崩溃时无法保证持久性，也无法应对多节点部署（分布式游戏服务器）下的跨进程并发。数据库事务的持久性由WAL日志在内核层面保障，应用层锁做不到这一点。

**误区二：事务越大越安全**。部分开发者为"确保安全"将整个游戏回合结算（包括经验值、金币、道具、成就解锁）全部塞入一个长事务。这会导致该事务持有大量行锁长达数百毫秒，期间其他玩家操作相同数据（如同一公会仓库）将被阻塞，极大降低并发吞吐。正确做法是将可独立回滚的部分拆分为独立事务，不可拆分的部分最小化事务范围。

**误区三：NoSQL数据库无需考虑事务设计**。MongoDB在4.0版本之前仅支持单文档原子操作，跨文档操作不具备ACID保证。在此版本之前将玩家背包和钱包存储为两个独立MongoDB文档的游戏，无法用原生事务处理道具购买场景，必须引入补偿事务（Saga模式）手动模拟原子性。MongoDB 4.0起支持多文档事务，但性能开销约是单文档操作的3-5倍，游戏设计时需权衡。

## 知识关联

事务设计建立在**玩家数据模型**的基础上：只有当背包、钱包、角色属性等数据被正确建模为独立表或文档后，才能明确哪些操作需要跨表事务保护。若玩家所有数据存储在一个大JSON字段中，则单文档更新天然原子，但会失去独立查询和索引的能力。

事务设计与**经济系统保护**直接关联：金币转账、道具买卖等经济操作是事务设计最密集的应用场景，事务的原子性是防止经济系统出现金币复制BUG（Duplication Exploit）的最后一道技术防线。
