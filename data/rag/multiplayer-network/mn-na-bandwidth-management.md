# 带宽管理

## 概述

带宽管理（Bandwidth Management）是多人游戏网络架构中对数据传输量、发送速率与包优先级进行主动调控的技术体系。其核心目标是在有限网络带宽条件下，确保对游戏体验影响最大的数据包优先送达，同时避免链路拥塞引发的延迟峰值和丢包。

带宽管理的工程化应用可追溯至2000年代初。早期多人游戏如 *Quake III Arena*（1999）采用固定20 Hz发包率，不区分玩家网络条件，导致56K拨号用户与宽带用户体验严重失衡。Valve在开发Source引擎（2004）时引入可配置的 `rate` 参数（单位：字节/秒），允许客户端声明自身最大可接受下行带宽，服务器据此动态调整每帧下发的快照数据量，标志着游戏引擎级主动带宽管理的成熟。Glenn Fiedler在其广泛引用的网络编程系列文章（*Gaffer on Games*, 2004–2018）中将带宽预算管理列为多人游戏服务器最关键的工程挑战之一。

一个典型的64人竞技射击游戏场景可以量化带宽压力：若每个快照包大小为1200字节，服务器以64 Hz向所有在线玩家广播，则单条连接下行带宽需求约为 $1200 \times 8 \times 64 \approx 614\text{ Kbps}$，64路连接合计约需 **39.3 Mbps** 出口带宽。不加任何管理机制时，这一数字随玩家数线性增长，运营成本与丢包风险均急剧上升。

## 核心原理

### 流量整形：令牌桶与漏桶

流量整形（Traffic Shaping）限制数据发送速率，防止瞬时突发流量冲击链路。两种主流算法各有侧重：

**令牌桶算法（Token Bucket Algorithm）** 以固定速率 $r$（令牌/秒）向容量为 $b$ 的桶中注入令牌，发送 $n$ 字节消耗 $n$ 个令牌，桶满时多余令牌丢弃。其数学约束为：

$$
\text{在任意时间窗口 } T \text{ 内，允许发送的最大字节数} \leq r \cdot T + b
$$

游戏服务器通常将 $r$ 设为目标带宽上限（例如500 Kbps），将 $b$ 设为1到2帧的数据量（64 Hz下约为18750字节），从而在帧间平滑流量，同时允许偶发的关键事件包（如爆炸特效）在短时间内突发发送而不被截断。

**漏桶算法（Leaky Bucket Algorithm）** 则以严格固定速率向外"漏"出数据，无论输入多么突发，输出均匀恒定。漏桶更适合视频流等需要极度平滑的场景，但游戏逻辑由帧驱动，帧边界天然存在数据批量生成，令牌桶的突发容忍特性更契合游戏的发包模式。

### 优先级队列与加权公平调度

游戏数据包对玩家体验的影响差异巨大，不应等同对待。根据时效性和可替换性，可划分为三个优先级层：

| 优先级 | 典型数据类型 | 可替换性 | 延迟容忍 |
|--------|-------------|---------|---------|
| 高（P0） | 输入ACK、击中判定、死亡/复活事件 | 不可替换 | <50 ms |
| 中（P1） | 玩家位置快照、近距离实体状态 | 后帧覆盖前帧 | <100 ms |
| 低（P2） | 聊天消息、远距离实体更新、UI事件 | 可丢弃或延迟 | 数百 ms |

优先级队列的调度机制通常采用**加权公平队列（Weighted Fair Queuing, WFQ）**，将三个优先级的带宽权重比设为 **8:4:1**。这意味着在满载时，P0数据获得约61.5%带宽，P1获得约30.8%，P2获得约7.7%。WFQ的关键优势是即便P0、P1持续繁忙，P2数据也不会被完全饿死（Starvation），聊天消息最终仍能送达。

相比之下，严格优先级调度（Strict Priority Queuing）虽然保证高优先级零等待，但在极端情况下会完全饿死低优先级队列——对游戏场景通常不可接受。

### 拥塞检测与自适应降级

游戏服务器通过两个实时指标判断链路是否进入拥塞：**RTT（往返时延）** 和 **丢包率**。具体阈值参考：

- 若连续3个数据包的RTT相对于历史基准值增长超过**150%**，判定出现缓冲区膨胀（Bufferbloat）。
- 若最近1秒内丢包率超过**5%**，判定链路已过载。

检测到拥塞后，服务器按以下优先级顺序执行降级操作（参考《Multiplayer Game Programming》 Joshua Glazer & Sanjay Madhav, 2015，Chapter 7）：

1. **降低快照发送频率**：从64 Hz逐步降至20 Hz，带宽消耗减少约68%，是最直接有效的手段。
2. **缩小实体感知半径**：将位置更新的有效距离从100米压缩至50米，可减少约75%的需要序列化的实体数量（假设实体均匀分布）。
3. **降低坐标精度**：将32位浮点坐标压缩为16位定点数，单个三维坐标从12字节降至6字节，节省50%的位置数据体积。
4. **合并低优先级包**：将多帧内的P2级数据聚合为一个批量包发送，减少包头开销。

## 关键公式与算法

### 带宽预算计算

服务器为每个客户端维护一个每帧带宽预算（Budget），计算方式如下：

$$
B_{\text{frame}} = \frac{R_{\text{client}}}{f_{\text{tick}}}
$$

其中 $R_{\text{client}}$ 为客户端声明的最大下行速率（字节/秒），$f_{\text{tick}}$ 为服务器Tick频率（Hz）。例如客户端声明 $R = 128\text{ Kbps} = 16000\text{ 字节/秒}$，Tick率为 $64\text{ Hz}$，则每帧预算为 $B = 16000 / 64 = 250\text{ 字节/帧}$。服务器在序列化当帧快照时，按P0→P1→P2顺序填充，直到预算耗尽为止。

### 令牌桶实现示例

```python
import time

class TokenBucket:
    """
    令牌桶限速器：控制游戏服务器每条连接的发送速率。
    rate: 令牌注入速率（字节/秒），对应客户端声明的 rate 参数
    capacity: 桶容量（字节），对应允许的突发量
    """
    def __init__(self, rate: float, capacity: float):
        self.rate = rate          # 如 16000 字节/秒 (128 Kbps)
        self.capacity = capacity  # 如 3000 字节 (约2帧突发量 @64Hz)
        self.tokens = capacity
        self.last_refill = time.monotonic()

    def consume(self, bytes_to_send: int) -> bool:
        """尝试消耗令牌以发送数据包，返回是否允许发送。"""
        now = time.monotonic()
        elapsed = now - self.last_refill
        # 注入新令牌，不超过桶容量
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_refill = now

        if self.tokens >= bytes_to_send:
            self.tokens -= bytes_to_send
            return True  # 允许发送
        return False  # 令牌不足，延迟或丢弃该包
```

## 实际应用

### Source引擎中的rate/cmdrate/updaterate三元组

Valve Source引擎（CS:GO、Team Fortress 2等）将带宽管理参数暴露给用户：

- `rate`：客户端最大下行速率，单位字节/秒。CS:GO默认值为786432（约6 Mbps），历史版本长期默认30000（约240 Kbps），2016年更新后大幅提升以匹配现代宽带环境。
- `cl_updaterate`：客户端请求的服务器快照发送频率，最高64 Hz（Valve官方服务器），128 Hz（第三方FACEIT/ESEA服务器）。
- `cl_cmdrate`：客户端向服务器发送输入命令的频率。

三个参数共同决定了该客户端连接的实际带宽消耗。例如128 Hz服务器下，`rate=786432, cl_updaterate=128`时，下行峰值可达约1.2 Mbps，而早期默认配置仅约240 Kbps，这解释了为何老配置下玩家位置更新存在明显滞后。

### 案例：《火箭联盟》的自适应带宽管理

《火箭联盟》（Rocket League，Psyonix，2015）在服务器侧实现了基于RTT的动态Tick率调整。当检测到客户端RTT超过**100 ms**时，服务器将该客户端的快照发送频率从60 Hz降至30 Hz；RTT超过**200 ms**时进一步降至20 Hz。这一设计避免了高延迟客户端因接收队列积压导致的"延迟叠加"现象，同时为低延迟客户端保留了高频更新。

思考一下：如果服务器仅对高延迟玩家降低发包频率，而不对所有玩家统一降频，会对竞技公平性产生什么影响？不同更新率的玩家在感知同一游戏事件时，时间误差最大可能达到多少毫秒？

## 常见误区

### 误区一：带宽越大越不需要管理

即便客户端拥有1 Gbps光纤宽带，带宽管理仍不可或缺。原因在于：**服务器出口带宽是所有玩家共享的**。64人服务器若每人分配10 Mbps上限，总出口需求达640 Mbps，大多数游戏服务器的实际出口带宽远低于此。此外，过量发送数据会导致客户端接收缓冲区积压，反而增加排队延迟（Queue Delay），与减少延迟的目标相悖。

### 误区二：丢弃低优先级包等同于"不重要数据不用发"

低优先级数据（如远距离实体位置）虽然可以在拥塞时延迟，但长期不发送会导致客户端状态与服务器严重偏离，引发"幽灵实体"或对象突然跳变的问题。正确做法是为每类低优先级数据维护**发送时间戳**，超过最大延迟阈值（通常500 ms到1秒）后强制发送，即便此时带宽紧张。

### 误区三：令牌桶的桶容量越大越好

桶容量 $b$ 决定了允许的突发量。若 $b$ 过大（如设为10秒数据量），则短时间内允许发送大量数据，可能在路由器队列中造成瞬时拥塞，引发其他玩家的延迟峰值。游戏场景下，$b$ 通常设为**1到3帧**的数据量，足以吸收帧内数据批量生成的自然突发，同时不给下游链路造成压力。

## 知识关联

### 与连接生命周期的关系

带宽管理参数在连接建立阶段（握手期）完成协商：客户端在初始化握手包中声明自身的 `rate` 上限，服务器据此为该连接分配优先级队列和令牌桶实例。连接断开时，服务器释放对应的队列资源。在连接生命周期的"降级重连"场景中，重连后服务器需重新接收客户端的带宽声明，不能复用旧连接的参数（因为网络环境可能已改变）。
