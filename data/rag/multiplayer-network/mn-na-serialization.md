# 序列化协议

## 概述

序列化协议是将游戏内存中的结构化数据（如玩家坐标、血量、技能状态）转换为可跨网络传输的字节流，并在接收端还原为原始数据结构的编码规范。在多人游戏中，服务器每秒需要向数百个客户端广播游戏状态，序列化的速度和压缩比直接影响带宽消耗与延迟表现。

序列化协议的历史可追溯至早期网络游戏对 XML 和 JSON 的使用。这两种文本格式虽然可读性好，但一条包含玩家位置的 JSON 消息`{"x":1024,"y":768,"hp":100}` 需要约 28 字节，而同等信息用二进制协议只需 7 字节——开销差距近 4 倍。2008 年 Google 开源 Protocol Buffers（Protobuf）后，二进制序列化协议逐渐成为游戏行业主流选择。2014 年 Facebook 发布的 FlatBuffers 进一步解决了 Protobuf 必须完整解析才能读取字段的问题，允许零拷贝随机访问。

在竞技类游戏中，序列化协议的选型决定了逻辑帧数据包的大小上限。以 60Hz 同步频率计算，每秒需要序列化 60 次完整游戏帧，协议效率的微小差异在百万级 DAU 产品中会放大为可观的服务器带宽成本。

## 核心原理

### Protobuf 的 TLV 编码机制

Protobuf 使用 Tag-Length-Value（TLV）格式，每个字段由字段编号、数据类型和实际值组成。字段标签占 1 字节（低 3 位存类型，高 5 位存字段编号），小整数通过 Varint 变长编码压缩存储：数值小于 128 的整数只占 1 字节，而非固定的 4 字节 int32。一个典型的玩家位置消息（x、y 各为 int32，值为 100）在 Protobuf 编码后仅需 4 字节，相比 JSON 节省 85% 以上。Protobuf .proto 文件示例：

```protobuf
message PlayerPosition {
  int32 x = 1;
  int32 y = 2;
  uint32 player_id = 3;
}
```

字段编号 `= 1` 不可随意更改，一旦客户端已分发，修改字段编号会导致新旧版本之间数据解析错位，这是 Protobuf 向后兼容设计的关键约束。

### FlatBuffers 的零拷贝布局

FlatBuffers 在序列化时将数据直接按内存对齐方式写入缓冲区，读取时无需反序列化——接收端可以直接将指针偏移量指向缓冲区内的字段。这种"就地访问"特性使得在大型地图状态包（通常超过 4KB）的场景中，FlatBuffers 的读取速度比 Protobuf 快 2～10 倍。代价是写入时需要自底向上逐层构建数据，且缓冲区不可原地修改，适合读多写少的服务器→客户端广播场景。

FlatBuffers 使用 vtable 机制实现字段可选性：每个 table 对象头部存一个 vtable 偏移量，vtable 记录每个字段相对于对象起始地址的偏移，缺失字段的偏移值为 0，读取时检查偏移是否为 0 来判断字段是否存在。

### 自定义二进制协议设计

许多游戏（如《英雄联盟》早期版本、《魔兽世界》）选择手写自定义二进制协议以获得极致性能。典型的自定义协议头部格式为：

```
[2字节 消息长度] [2字节 消息类型ID] [1字节 序列号] [N字节 Payload]
```

消息类型 ID 预先在客户端和服务器之间硬编码映射，省去字段名传输。Payload 中浮点坐标常被量化为定点数：将 float 精度从 32 位压缩为 16 位（量化步长 = 地图尺寸 / 65536），在保证 1/65536 精度的前提下节省一半带宽。自定义协议需要维护一份严格的版本号表，客户端与服务器版本不一致时需强制重连，这是自定义方案最大的工程维护成本。

## 实际应用

**《王者荣耀》帧同步数据包**：每个逻辑帧仅同步玩家操作指令而非完整状态，指令消息（移动方向、技能释放）经自定义二进制协议编码后单帧数据量控制在 50 字节以内，在 4G 网络下实现稳定的 15Hz 帧同步。

**MMO 游戏的区域广播**：《最终幻想XIV》使用定制化二进制协议，将同一区域内所有玩家的位置更新打包为一条广播消息，单个数据包携带最多 32 个玩家的位置信息（每个玩家位置 12 字节：4字节 x，4字节 y，4字节 z）。服务器用 Protobuf 处理登录、背包等低频复杂消息，用紧凑二进制格式处理高频位置同步，这种混合方案兼顾了开发效率与运行效率。

**热更新场景**：使用 Protobuf 的游戏在新增字段时，只需给新字段分配新编号（如 `int32 shield_hp = 10`），旧版客户端收到包含该字段的消息时会自动忽略未知字段，实现不停服热更新。

## 常见误区

**误区一：Protobuf 的编码总是比 JSON 小**。当字段值为大整数（超过 268435455 的 int32）时，Protobuf Varint 编码需要 5 字节，而 JSON 字符串表示的相同数字可能只有 9 个字符。更关键的是，Protobuf 不压缩重复字符串——如果消息中反复出现相同的字符串值，JSON + gzip 的组合在某些场景下反而更小。序列化协议的体积优劣必须针对具体数据分布测量，而非凭直觉判断。

**误区二：FlatBuffers 在所有场景都优于 Protobuf**。FlatBuffers 的零拷贝优势在读取时显著，但其序列化（写入）速度由于需要手动管理 Builder 对象并自底向上构建，在对象嵌套层级深时代码复杂度高，且无法直接修改已序列化的缓冲区内容。对于客户端→服务器的写频繁场景（如玩家每帧上报输入），FlatBuffers 的工程代价往往超过其收益。

**误区三：字段编号可以复用已删除的编号**。在 Protobuf 中，删除某个字段后若不使用 `reserved` 关键字保留其编号，日后新增字段时可能意外复用该编号，导致老版本客户端将新字段数据解析为已删除字段的语义，引发静默数据错误，这类 bug 在版本混用的灰度发布阶段极难排查。

## 知识关联

序列化协议建立在**数据包结构**的基础上——数据包的头部字段（长度、类型ID、序列号）定义了封装序列化负载的外层容器，序列化协议负责将头部之后的 Payload 部分编码为紧凑字节流。理解数据包的分层结构有助于区分"协议头的固定字段"与"消息体的可变序列化内容"这两个不同层次的编码问题。

学习序列化协议之后，自然延伸到**网络数据压缩**。序列化产生的字节流可以进一步通过 zlib、LZ4 或 zstd 进行通用压缩，或针对游戏数据特征采用差分编码（仅传输相对上一帧变化的字段值）。LZ4 的压缩速度达到约 500 MB/s，适合游戏实时数据流；而 Protobuf 的 Varint 编码本身已经是一种针对小整数的特化压缩，在此之上叠加通用压缩算法时需要评估 CPU 开销与压缩收益的平衡点。
