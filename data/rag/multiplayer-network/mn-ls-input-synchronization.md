# 输入同步

## 概述

输入同步（Input Synchronization）是帧同步网络架构中负责收集、传输和分发所有玩家操作指令的核心机制。其基本原理是：游戏逻辑不直接响应本地输入，而是等待所有玩家在同一帧的输入数据全部到齐后，再统一驱动游戏状态向前推进一帧。这一机制保证了每台客户端上参与计算的输入集合完全一致，从而使确定性逻辑能够产生相同的游戏状态。

输入同步的思想最早在1990年代的局域网即时战略游戏中成型。Westwood Studios的《命令与征服》（1995年）采用了"锁步协议"（Lockstep Protocol），要求每台机器在发送自身输入的同时接收所有其他玩家的输入，只有当本帧输入集合完整时才推进模拟。这一设计避免了在网络上同步大量游戏对象状态的开销，因为战略游戏中单位数量庞大，状态同步的带宽代价远高于仅同步玩家指令。

输入同步之所以至今仍是格斗游戏、RTS和部分MOBA类游戏的首选方案，是因为它将网络流量压缩到极限：每帧每个玩家仅需传输几个字节的按键状态或操作指令，而非传输整个游戏世界的快照。然而，它也带来了一个根本性约束——任何一个玩家的输入延迟都会阻塞所有其他玩家的游戏推进，因此输入同步的工程实现必须在"等待完整性"与"容忍延迟"之间做出精细的权衡。

## 核心原理

### 输入帧打包与时间戳

每个客户端在每个逻辑帧（Tick）结束时，将本帧内采集到的所有玩家操作封装成一个**输入包（Input Packet）**。一个典型的输入包结构包含：帧序号（Frame Index，32位整数）、玩家ID（1字节）、输入位掩码（Input Bitmask，通常8~16位，每一位对应一个按键或方向）以及校验和。以格斗游戏为例，每帧输入包的大小通常只有4~8字节。帧序号是输入同步的关键字段，它告诉接收方该输入属于哪一个逻辑帧，防止因网络乱序导致输入被错误地应用到错误的帧上。

### 锁步等待与超时策略

经典锁步实现中，每台客户端在收到本帧所有玩家的输入包之前会**挂起游戏逻辑**，不推进帧计数器。假设游戏以20帧/秒运行（每帧50ms），如果某玩家的输入包在40ms内未到达，等待窗口还在容忍范围内；但如果等待超过一个帧周期（50ms），游戏就会出现明显卡顿，即俗称的"网络等待卡"（Lockstep Stall）。为了缓解这一问题，现代帧同步实现通常设定一个最大等待阈值（如2~3帧，即100~150ms），超过阈值后触发重连或断线处理流程，而不是无限期等待。

### 中继服务器与P2P拓扑下的输入分发

输入同步的网络拓扑直接影响延迟特性。在**P2P全连接拓扑**中，每个玩家直接向其他所有玩家广播自己的输入包，N个玩家需要维护N×(N-1)条连接，延迟取决于所有玩家对之间的最大RTT。在**中继服务器（Relay Server）拓扑**中，所有玩家的输入包先发送到中继服务器，由服务器收集完整帧输入后再广播给所有玩家，这额外增加了一个服务器单程延迟（通常10~30ms），但统一了分发时机，减少了各客户端之间的到达时间差。GGPO框架默认支持P2P拓扑，而《王者荣耀》等手游则更多采用中继服务器方案以便于反作弊。

### 输入确认与冗余传输

由于UDP丢包的存在，输入包可能在传输过程中丢失。输入同步通常采用**冗余打包（Redundant Input Packaging）**策略：每个输入包中不仅包含当前帧的输入，还附带前N帧（通常N=3~5）的历史输入。这样即使单个数据包丢失，接收方也能从后续包中恢复出丢失帧的输入数据。这一技术以轻微的带宽增加（约N倍）换取了极大的抗丢包能力，是帧同步网络层的标准实践。

## 实际应用

**格斗游戏中的输入同步**：以《街头霸王V》的帧同步实现为例，游戏以60帧/秒运行，每帧约16.7ms。每个玩家的输入状态用一个16位掩码表示（8个方向+8个按键），加上4字节帧序号，每帧输入包仅6字节。两名玩家的P2P连接每秒仅需传输约720字节的纯输入数据，带宽开销极低，但对延迟极度敏感，因此格斗游戏后来普遍引入了Rollback机制来进一步优化体验。

**RTS游戏中的多玩家输入同步**：《星际争霸II》支持最多16名玩家的多人对战。在16人局中，中继服务器需要在收到全部16个玩家的输入包后才广播，此时整体等待延迟由网络中RTT最大的那个玩家决定。Blizzard为此设计了动态帧率调节机制：当检测到某玩家持续拖慢同步节奏时，自动将全局逻辑帧率从16帧/秒降至8帧/秒，牺牲响应精度换取流畅度。

**移动端帧同步的输入压缩**：在手游帧同步实现中，虚拟摇杆的模拟量输入需要量化（Quantization）处理。以8位量化为例，将360度方向压缩为256个离散角度值，并将移动速度分为4档（0%、33%、66%、100%），使得每帧输入仅需1字节即可表达完整的移动指令，在带宽受限的移动网络环境下保持了输入同步的轻量特性。

## 常见误区

**误区一：输入同步等同于状态同步的替代方案**
输入同步与状态同步是两种不同的同步粒度，不是简单的替代关系。状态同步传输的是游戏对象的位置、血量等属性值，输入同步传输的是玩家的操作指令。输入同步依赖确定性逻辑在各端独立重算状态，如果游戏引擎中存在任何非确定性因素（如浮点数精度差异、随机数种子不一致），输入同步无法自动修复各客户端之间出现的状态分歧，反而会导致各端无声无息地跑偏——这是纯状态同步不会遭遇的问题。

**误区二：输入到达即可立刻执行**
许多初学者认为收到某帧输入后应立即执行该帧逻辑。但在帧同步中，必须等待**同一帧序号的所有玩家输入全部到达**后才能执行，否则本端以不完整的输入集合计算出的游戏状态将与其他玩家不同。提前以本地输入推进帧计数器是Rollback网络代码的特有行为，经典锁步输入同步中绝对不应这样做。

**误区三：输入同步只需处理按键按下事件**
输入同步必须在每一帧传输**完整的输入状态快照**，而不仅仅是按键按下/抬起的变化事件。这是因为帧同步的确定性要求每帧的输入集合绝对完整：若某帧因为没有按键变化就不发送输入包，接收方将无从判断该帧是"玩家维持上一帧的输入"还是"输入包丢失了"，从而破坏同步的可靠性。因此标准做法是每帧强制发送输入包，即使输入状态与上一帧完全相同。

## 知识关联

**前置概念——确定性帧同步**：输入同步的有效性完全建立在确定性逻辑的基础上。只有当相同的输入集合在所有客户端上必然产生相同的游戏状态时，输入同步才有意义。如果游戏逻辑本身不确定，即使输入同步完全正确，各端状态也会逐渐分歧。

**后续概念——Rollback网络代码**：Rollback是对经典锁步输入同步的改良，不再等待所有输入到达才推进帧，而是先以预测输入执行当前帧，收到真实输入后若预测错误则回滚重算。理解输入同步的等待机制是理解Rollback为何能改善延迟体验的前提。

**后续概念——输入延迟（Input Delay）**：输入延迟是另一种规避锁步卡顿的策略，它让本地玩家的输入延迟N帧后才真正生效，为远端输入的到达争取时间窗口。N的取值直接由网络RTT决定，通常设置为RTT/2除以帧时长后向上取整
