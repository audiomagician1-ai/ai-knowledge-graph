# 数据归档

## 概述

数据归档（Data Archiving）是指将不再频繁访问的历史数据从主业务数据库中迁移到独立存储系统的过程，核心目标是降低主数据库的数据量以提升查询性能，同时保留数据的完整性与可访问性。在网络多人游戏场景中，这一策略专门针对"冷热数据分离"问题——即将玩家近期活跃的"热数据"留在高速数据库中，而将90天以上未登录玩家的账号信息、已结束赛季的排行榜记录、过期活动的奖励发放日志等"冷数据"迁移至归档库。

数据归档的实践可追溯至1990年代大型主机时代的磁带归档，但在在线游戏领域的系统化应用始于2000年代MMORPG规模扩张时期。当时《魔兽世界》等游戏面临玩家角色表每年增长数亿行记录的压力，促使开发团队建立起周期性的历史数据归档机制。对于现代多人游戏而言，一个拥有1000万注册用户的游戏主角色表若不做归档处理，在三年运营后可能累积超过30亿条战斗日志记录，直接导致查询延迟从毫秒级升至秒级。

数据归档与数据备份有本质区别：备份是对数据的副本保留（原始数据仍在主库），而归档是将数据**迁移**出主库（主库中的原始记录被删除或标记为已归档）。理解这一区别是正确设计游戏数据库归档策略的前提。

## 核心原理

### 冷热数据划分标准

在多人游戏数据库中，"热数据"的判定依据通常是访问时间戳和访问频率两个维度。一个常见的工业标准是：**最近30天内有登录行为的玩家数据为热数据，31~90天为温数据，90天以上无活动则降级为冷数据**。具体的划分阈值需结合游戏类型调整——例如赛季制游戏（如《英雄联盟》）的赛季归档节点固定为每6个月一次，而开放世界游戏则更依赖用户行为频率。

游戏中常见的冷数据类型包括：
- **过期赛季排行榜**：赛季结束后，历史排名数据仅在申诉时才需要访问
- **已删除角色信息**：多数游戏保留已删除角色数据30~180天以支持误删恢复，之后归档
- **交易流水记录**：超过财务审计周期（通常为5~7年）之前归档至冷存储
- **系统操作日志**：GM操作日志、反作弊检测日志在3个月后访问频率趋近于零

### 归档数据库的技术选型

归档库与主业务库在技术选型上应有明显差异。主数据库通常选用 MySQL 或 PostgreSQL 等支持高并发写入的关系型数据库，而归档库常见选型包括：
- **列式数据库**（如 Apache Cassandra、ClickHouse）：适合按时间范围批量扫描的历史战斗统计数据
- **对象存储**（如 AWS S3、阿里云 OSS）：适合将归档数据序列化为 Parquet 或 Avro 格式后存储，单GB成本低至 $0.023/月（S3标准归档层）
- **冷存储归档服务**（如 AWS Glacier）：用于合规要求的长期保存，检索延迟为3~5小时，但每GB成本低至 $0.004/月

归档数据建议在迁移时增加 `archived_at` 时间戳字段和 `original_table` 来源标记字段，便于日后根据数据隐私法规（如GDPR的"被遗忘权"）精准定位并删除特定用户的全部历史记录。

### 归档迁移的执行策略

归档迁移不能简单地执行 `DELETE FROM player_logs WHERE created_at < '2023-01-01'`，这会在主库产生大量锁等待，影响在线玩家体验。安全的归档迁移采用**分批次迁移**策略：

```sql
-- 每批次迁移1000行，循环执行直至完成
INSERT INTO archive_db.player_logs_2022
SELECT * FROM player_logs
WHERE created_at < '2023-01-01'
LIMIT 1000;

DELETE FROM player_logs
WHERE id IN (
  SELECT id FROM player_logs
  WHERE created_at < '2023-01-01'
  LIMIT 1000
);
```

批次大小通常设定在500~5000行之间，并在每批次之间设置50~200毫秒的休眠间隔，以避免归档任务与正常业务查询产生资源竞争。整个归档作业应安排在游戏低谷时段（如凌晨2:00~6:00）执行。

## 实际应用

**赛季制游戏的周期归档**：以典型的3个月赛季制 MOBA 游戏为例，赛季结束当天需归档的数据包括：本赛季全部对战记录（约每用户100~500条）、段位积分变动历史、赛季专属称号授予记录。归档完成后，新赛季主数据库仅保留玩家基础档案和新赛季初始状态，查询延迟可下降 40%~60%。

**玩家账号注销的合规归档**：根据 GDPR 第17条（被遗忘权）和中国《个人信息保护法》第47条，玩家申请注销账号后，游戏运营商需在30日内从主数据库删除可识别个人身份的信息（PII），但为满足财务审计要求，虚拟货币充值记录需以脱敏形式保留5年。这意味着归档流程需要先对数据执行字段级脱敏（如将真实姓名替换为哈希值），再迁入长期归档库，而非整行删除——这正是归档与备份、归档与删除三者在游戏合规场景中交织的典型案例。

**反作弊日志的分级归档**：封禁玩家的作弊行为证据日志需要保留6~12个月以应对申诉，期间存放在温存储（访问延迟秒级）；12个月后若无申诉则迁入冷存储（Glacier级别，检索需数小时）；3年后彻底销毁。这种**两阶段归档**设计平衡了存储成本与申诉响应时效。

## 常见误区

**误区一：归档等同于备份，两套机制可以共用**
这是游戏数据库设计中最常见的混淆。备份的目的是灾难恢复，数据保留在主库中不变；归档的目的是主库瘦身，数据从主库中移除。如果将备份文件当作归档库，遇到"玩家申请查询3年前的充值记录"时，运维需要从全量备份中解压恢复，耗时数小时甚至数天，而独立归档库可以直接查询返回结果。两者应分别独立建设。

**误区二：冷数据归档后就不需要索引**
归档库中的历史数据确实访问频率低，但并非不需要索引。游戏运营中存在"客服查询玩家历史记录""财务审计特定时间段充值流水"等低频但对延迟敏感的查询场景。建议归档表至少保留 `player_id` 和 `created_at` 的复合索引，仅删除对归档场景无用的全文索引或覆盖索引以节约存储空间。

**误区三：归档策略一经制定无需调整**
游戏生命周期内，玩家活跃模式会发生变化。一款游戏在上线初期可能90%的用户是近30天活跃用户，但5年后大量休眠账号积累，若仍使用"90天无登录即归档"的标准，可能将大量实际上只是长假期后回归的老玩家数据误归档，导致其登录时出现数据加载延迟。归档阈值应每6~12个月根据实际用户留存曲线重新评估和调整。

## 知识关联

**与数据隐私合规的关系**：数据归档流程的设计必须以数据隐私合规为前提约束。GDPR 的"被遗忘权"和中国《个人信息保护法》的"最小必要原则"直接决定了归档数据中哪些字段可以完整保留、哪些必须脱敏、哪些必须在特定期限后彻底销毁。因此在设计归档表结构时，每个字段都应标注其对应的 PII 分类和法定保留期限，这是数据隐私合规知识在归档实践中的直接落地。

**为数据库性能优化提供基础**：实施了归档策略的主数据库具备更稳定的查询计划——因为表数据量不再随时间无限增长，数据库优化器的统计信息采样更加准确，索引的选择性维持在较高水平，B+树索引的高度不会随记录数增加而持续增加（理论上每增加一层节点可容纳的记录数为分支因子的幂次方，常见值约为100~1000）。这使归档成为长期稳定支撑大规模在线游戏服务的重要基础设施决策。
