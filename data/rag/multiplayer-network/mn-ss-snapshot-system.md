# 快照系统

## 概述

快照系统（Snapshot System）是网络多人游戏中一种以固定时间间隔将服务器世界状态打包并发送给客户端的同步机制。每一帧或每隔若干帧，服务器将当前世界中所有相关实体的位置、朝向、血量、速度等属性序列化为一个"快照"数据包，客户端收到后用该快照覆盖或插值本地状态。

快照系统的概念最早在 Quake（1996年）的网络代码中得到系统性实现，John Carmack 在 QuakeWorld（1996年底发布）中将该模型定型：服务器以 20Hz 频率发送世界快照，客户端基于快照进行预测与插值。这一设计奠定了此后大量第一人称射击游戏网络架构的基础，《半条命》《CSGO》等游戏的 Source 引擎延续了这一思路。

快照系统的核心价值在于其确定性与简单性：客户端不需要理解任何游戏逻辑，只需"显示"服务器告知的状态即可，服务器拥有绝对权威。与基于事件（Event-based）的同步方式不同，快照系统天然对丢包具有容错能力——即使一个数据包丢失，下一个快照仍包含完整的最新状态，不会出现状态永久偏离的问题。

## 核心原理

### 快照的数据结构

一个典型快照由**快照序号（Snapshot Sequence Number）**、**服务器时间戳**以及若干**实体记录（Entity Record）**组成。每条实体记录包含实体ID、实体类型及该类型对应的所有可同步字段。以一个简单射击游戏为例，一名玩家的实体记录可能包含：3轴位置（各4字节浮点）、3轴速度（各4字节）、yaw/pitch朝向（各2字节）、血量（1字节）、武器状态（1字节），单条记录约26字节。若场景中有64名玩家，仅玩家部分就需要约1664字节，加上其他动态对象后完整快照轻易超过数KB。

### 全量快照与差量快照

快照系统存在两种发送模式：**全量快照（Full Snapshot）**和**差量快照（Delta Snapshot）**。全量快照在首次连接或重同步时发送，包含所有实体的完整状态；差量快照则只包含自上一个已确认快照以来发生变化的字段。QuakeWorld 的差量系统以"基准快照编号"作为参考点，客户端在确认包（ACK）中告知服务器自己最后成功收到的快照序号，服务器据此计算差量。差量快照典型情况下可将带宽需求降低 60%–80%。

差量计算公式可描述为：

> **ΔSnapshot(n) = Snapshot(n) − Snapshot(base)**

其中 `base` 为客户端已确认的最新快照序号，`n` 为当前待发送快照序号。若客户端长时间未发送ACK或ACK丢失，服务器会退回发送全量快照以防止状态撕裂。

### 发送频率与插值

服务器快照频率（Tickrate）通常设置为 **20Hz、64Hz 或 128Hz**，例如 CS:GO 官方服务器默认 64Hz，竞技平台升至 128Hz。客户端渲染帧率往往高于 Tickrate，因此需要在两个相邻快照之间进行**线性插值（Lerp）**，公式为：

> **State(t) = State(snap_n) × (1−α) + State(snap_{n+1}) × α**

其中 `α = (t − t_n) / (t_{n+1} − t_n)`，`t` 为当前渲染时刻。为保证插值始终有"未来快照"可用，客户端通常引入约 **1–2 个快照周期（15–30ms）的延迟缓冲（Jitter Buffer）**。

### 快照的确认机制

快照系统依赖客户端持续向服务器发送的**用户指令包（User Command Packet）**隐式携带ACK信息。每个用户指令包头部包含"已收到的最高快照序号"字段，服务器用此字段管理每个客户端的差量基准。若某客户端超过约 **500ms** 未发送ACK，服务器通常重置其差量基准并强制发送全量快照。

## 实际应用

**《Overwatch》（守望先锋）**采用快照系统作为基础状态同步机制，服务器以 63Hz 发送快照，客户端维护约 60ms 的插值缓冲。该架构使每个客户端看到的是轻微"过去"的世界，换取了流畅的视觉表现。

**《火箭联盟》（Rocket League）**在快照系统上叠加了物理状态字段：车辆的刚体旋转以四元数（4×4字节）编码进快照，球的角速度同样包含其中，确保高速物理模拟在各客户端可视化一致。

**Valve 的 Source 引擎**提供了工程上的参考实现：`sv_updaterate`（服务器发送快照频率，默认64）和 `cl_interp_ratio`（插值比率，默认1，即1个快照周期的延迟）均为可调参数，开发者可直接在控制台观察快照系统的行为效果。

## 常见误区

**误区一：快照越大越全面，状态同步越准确。**  
快照包含的字段越多，带宽消耗越高，但并不一定提升同步质量。对于静止不动的实体，每帧重复发送相同数据是纯粹的浪费。差量快照正是为了解决这一问题——只有在字段值变化时才将其纳入差量包。在正确实现差量的情况下，增大快照频率（如从64Hz到128Hz）对带宽的增量影响远小于理论线性增长。

**误区二：快照系统可以完全替代客户端预测。**  
快照系统仅负责将服务器权威状态同步到客户端，它本身不消除玩家操作到画面反馈之间的延迟（RTT/2）。若客户端完全等待服务器快照再显示自身角色移动，100ms RTT 意味着每次按键都有明显卡顿感。因此快照系统通常必须与**客户端预测（Client-side Prediction）**配合，客户端本地立即执行自身输入，快照到达后再进行校正（Reconciliation）。

**误区三：丢失的快照必须重传。**  
由于每个全量或差量快照都包含特定时间点的完整差异集合，旧快照的丢失无需触发重传——下一个快照到达时，客户端只需以新的 base 序号建立更大的差量即可。这是快照系统相比基于可靠事件流（如TCP事件同步）的一大优势，也是其通常运行在 **UDP** 之上的根本原因。

## 知识关联

快照系统建立在**数据包结构**的基础之上：理解字段序列化方式、字节对齐与变长编码是实现高效快照序列化的前提，例如将32位浮点坐标压缩为16位定点数可将位置字段大小减半。

快照系统直接引出**差量压缩（Delta Compression）**这一进阶技术：如何用位掩码标记变化字段、如何对连续数值字段做XOR或预测差值编码，都是在全量/差量快照基础上的具体工程优化手段。

另一个后续主题**兴趣管理（AOI，Area of Interest）**解决的是"快照应该包含哪些实体"的问题：当世界规模达到数百个实体时，为每个客户端发送全局快照会导致带宽爆炸，AOI 系统通过空间划分或显式订阅筛选出每个客户端"可见"的实体子集，从而大幅压缩快照体积，是快照系统在大规模场景下的必要补充。
