# 快照系统

## 概述

快照系统（Snapshot System）是网络多人游戏状态同步中一种将世界状态定期打包成"快照"并发送给客户端的同步方案。与事件驱动型同步不同，快照系统不依赖"某件事发生了才发送"的触发逻辑，而是以固定频率（如每秒20次，即20Hz）将当前游戏世界的状态——包括所有实体的位置、朝向、速度、生命值等字段——编码进一个数据包并推送出去。

快照系统的雏形可以追溯到1990年代末的《Quake III Arena》网络架构。id Software 的工程师 John Carmack 在设计该引擎的网络层时，采用了基于快照差量（delta snapshot）的同步模型：服务器记录每次发送给特定客户端的快照内容，下一帧只发送与上一帧快照之间的差异部分。这一设计影响了此后大量竞技类游戏引擎，包括《Valve Source》引擎和虚幻引擎4中的 NetDriver 模块。

快照系统之所以在竞技游戏中被广泛采用，根本原因在于其**状态确定性**：无论客户端和服务器之间丢失了多少中间事件包，下一个完整快照都能将客户端强制对齐至服务器的权威状态，从而避免事件丢失导致的长期状态偏移。

## 核心原理

### 全量快照与差量快照

全量快照（Full Snapshot）将游戏世界中所有被追踪实体的全部字段一次性序列化。假设一个场景包含100个实体，每个实体携带32字节的状态数据，则单帧全量快照约为3200字节。以20Hz发送，带宽消耗约为 3200 × 20 = 64000 字节/秒（约512 kbps），这对于需要同时连接64个玩家的游戏服务器来说是不可接受的。

差量快照（Delta Snapshot）通过记录**基线快照编号（baseline snapshot ID）**解决带宽问题。服务器为每个客户端维护一个已确认接收的快照序列号（ACK sequence number）。下一帧发送时，只对比当前状态与该已确认帧的差异，仅传输发生变化的字段。在实践中，一个60人对战场景下，每帧实际变化的实体往往不超过总数的20%，差量快照可将带宽压缩至全量快照的10%~30%。

差量编码的基本公式如下：

```
Δstate = state_current XOR state_baseline
```

对于浮点数坐标，通常先量化为定点整数再做 XOR 差量，避免浮点表示不稳定导致的虚假差异。

### 快照序列号与重传策略

每个快照数据包包含一个单调递增的序列号（sequence number），以及该差量所依赖的基线序列号（baseline sequence number）。客户端收到快照后，通过 ACK 包将已确认的序列号回传给服务器。若服务器在若干帧内未收到 ACK（通常设置超时阈值为3~5帧），则回退到以某个已知可靠基线（或干脆退回全量快照）重新发送，防止差量链断裂导致客户端永久无法解码后续帧。

这套机制要求底层数据包结构（Packet Structure）必须包含序列号字段、基线引用字段和可选的确认掩码字段——这也是快照系统对前置概念"数据包结构"的直接依赖。

### 快照频率与插值窗口

快照发送频率（tick rate）直接决定游戏的响应延迟下限。《CS:GO》官方竞技服务器使用128 tick（每秒128次快照），而《Overwatch》服务器使用20Hz服务端快照配合客户端插值实现视觉流畅。客户端通常维护一个**插值缓冲区（interpolation buffer）**，存储最近2~3帧的快照，在快照帧之间对实体位置进行线性或三次插值，使得即使在20Hz快照频率下，玩家也能看到流畅的60fps或更高帧率的动画效果。

## 实际应用

**《Fortnite》的快照系统**：Epic Games 在虚幻引擎的 NetDriver 中采用属性复制（Property Replication）机制，本质上是基于字段级别的差量快照。每个 `AActor` 的 `UPROPERTY(Replicated)` 字段被追踪变化，服务器以固定 tick 周期扫描所有需要复制的属性并生成差量数据包。100人战场中，Epic 通过兴趣管理（AOI）裁剪后，单个客户端每帧实际接收的快照数据约为1~4 KB。

**《Rocket League》的帧同步快照**：Psyonix 使用8Hz的完整物理状态快照作为纠错锚点，配合客户端预测，即使在200ms延迟下也能保证球的物理轨迹最终与服务器对齐。快照中包括刚体的位置向量（3×float32）、四元数旋转（4×float32）、线速度和角速度共计56字节的核心物理状态。

## 常见误区

**误区一：快照频率越高，游戏同步质量一定越好。**
实际上，提高 tick rate 会线性增加服务器 CPU 负载和带宽消耗。128 tick 服务器相比64 tick 服务器，每秒需要处理的快照数量翻倍，在大规模战场（如128人）中可能导致服务器帧率下降，反而引入更大的抖动延迟。选择合适的 tick rate 需要平衡网络带宽、服务器算力与玩法对响应延迟的实际需求。

**误区二：差量快照一定比全量快照小。**
当客户端长时间未确认快照（例如因为网络中断）或基线快照对应的游戏状态变化极大（如玩家刚进入游戏、场景切换），差量与全量之间的差异可能接近甚至超过全量快照大小，此时服务器应主动降级发送全量快照或选择最近的可靠基线，而不是盲目追求差量压缩。

**误区三：快照系统能天然处理好实体的生成与销毁事件。**
快照系统负责同步持续存在实体的**状态字段**，但实体的生成（spawn）和销毁（destroy）属于离散事件，必须通过独立的可靠消息通道（而非不可靠的快照 UDP 包）发送，否则客户端可能因丢包而对某个实体的存在性产生持久性误判。

## 知识关联

**前置依赖——数据包结构**：快照系统的每一个发送单元本质上是一个按照特定格式编排的二进制数据包。理解数据包中序列号字段的偏移量、字节对齐方式、变长字段的长度前缀编码，是正确实现快照序列号追踪和差量基线引用的必要基础。没有规范的数据包结构，差量快照中的基线 ID 与 ACK 回传机制将无法可靠运作。

**后续扩展——差量压缩**：快照系统产生的差量数据（Δstate）可以进一步通过位域压缩、Huffman 编码或 zlib 压缩进行体积缩减。差量压缩是在快照系统已生成 XOR 差量的基础上，针对差量数据的熵特性专门设计的编码优化，能将快照差量数据再压缩30%~60%。

**后续扩展——兴趣管理（AOI）**：快照系统在大型场景中面临的核心瓶颈是：需要追踪和对比所有实体的状态变化。兴趣管理通过将地图划分为区域格或使用距离查询，筛选出每个客户端实际"感兴趣"的实体子集，从根本上减少快照系统需要序列化和差量对比的实体数量，使百人以上规模的在线游戏成为可能。
