# 房间列表浏览

## 概述

房间列表浏览（Room List Browser）是大厅系统中供玩家查看、搜索并加入现有游戏房间的功能模块，由前端列表UI与后端房间索引服务共同构成。玩家完成房间创建后，该房间的元数据（房间名、地图、人数、密码标志等）会被注册到大厅服务器的房间索引中，而房间列表浏览功能则负责将这批数据拉取、过滤并展示给其他玩家。

该功能最早在1990年代的对战平台上成型。Battle.net 1.0（1996年随《暗黑破坏神》上线）是最具代表性的早期实现，它首次将频道聊天室与可加入的游戏房间列表整合在同一界面，奠定了后续大厅系统的基本范式。Steam Matchmaking SDK 中对应的接口 `ISteamMatchmaking::RequestLobbyList()` 至今仍遵循相似的拉取-过滤-回调模型。

在网络多人游戏开发中，房间列表浏览的设计质量直接影响玩家发现和填充房间的效率。如果列表刷新延迟过高或筛选维度不足，会导致玩家无法快速定位合适房间，进而增加匹配等待时间，最终伤害在线人数分布。

## 核心原理

### 房间元数据的注册与索引

当一个房间被创建后，主机客户端或专用服务器会向大厅服务器发送一条 **RegisterRoom** 请求，携带若干键值对形式的元数据，例如：

```
RoomName: "老兵房 No.3"
Map: "de_dust2"
MaxPlayers: 10
CurrentPlayers: 4
HasPassword: false
GameMode: "竞技"
Ping: 32
```

大厅服务器在内存或 Redis 等缓存层中维护一张房间索引表，每条记录通常包含上述字段加上 **房间唯一ID**（UUID 或自增整数）。为保持数据时效性，房间主机需每隔固定间隔（常见值为 **5~30 秒**）发送心跳包（Heartbeat）更新 `CurrentPlayers` 和延迟数据；若大厅服务器超过 **60 秒**未收到心跳，则自动将该房间标记为过期并从列表中移除。

### 列表拉取协议与分页

客户端通过两种主流方式获取房间列表：

- **全量拉取（Full Pull）**：客户端发送一次请求，服务器返回符合过滤条件的全部房间，适合在线房间数量少于 200 的小型游戏。
- **分页拉取（Paginated Pull）**：服务器每页返回固定数量（如 **20 或 50 条**）记录，客户端通过翻页或滚动加载继续请求。Steam Matchmaking 的 `AddRequestLobbyListResultCountFilter(nMax)` 参数即用于控制单次返回上限。

响应数据格式通常为 JSON 或 Protocol Buffers，每条房间记录包含加入所需的最小字段集，以降低带宽消耗。对于同时在线 5000+ 个房间的大型平台，服务端会额外引入 **二级索引**（按地图、游戏模式分桶）加速过滤查询，将单次过滤查询耗时控制在 **10ms** 以内。

### 客户端筛选与排序逻辑

UI 层提供的筛选维度通常包括：
1. **关键词搜索**——对 RoomName 字段做前缀匹配或模糊匹配；
2. **延迟过滤**——隐藏 Ping 超过用户设定阈值（如 >150ms）的房间；
3. **人数过滤**——只显示"有空位"或"非空"的房间；
4. **游戏模式/地图**——下拉枚举筛选；
5. **密码房显示开关**——布尔标志过滤。

排序维度一般支持按 **延迟升序、人数降序、创建时间降序** 三种默认方案，部分游戏允许玩家自定义组合排序键。客户端侧排序在数据量小于 200 条时性能充足（JavaScript `Array.sort()` 对 200 条数据的排序耗时通常低于 **1ms**），数据量更大时建议将排序逻辑迁移至服务端。

### 实时刷新策略

房间列表的数据一致性与刷新频率之间存在权衡：

- **手动刷新**：玩家点击"刷新"按钮触发新一轮拉取，实现最简单，但数据时效性差；
- **定时轮询**：客户端每隔固定间隔（通常 **10~30 秒**）自动重新拉取，适用于大多数休闲类游戏；
- **服务端推送（SSE/WebSocket）**：大厅服务器在房间状态变更时主动推送差量更新（delta update），可将列表延迟降低至 **1~2 秒**，常见于竞技性较强的游戏，但对服务端连接维护成本更高。

## 实际应用

**《Minecraft》Java 版第三方服务器列表**采用定时轮询模型，官方主页每隔 5 分钟刷新一次服务器 Ping 与在线人数，并支持按游戏模式标签过滤，最多可展示数千条服务器记录，采用分页每页显示 20 条。

**Valve 的 CS:GO 社区服务器浏览器**提供完整的多维度筛选界面：地图、游戏模式、地区、延迟、反作弊状态均可独立过滤，并支持按标签（Tag）进行自定义字符串搜索。其后端基于 Steam Datagram Relay（SDR）网络，房间心跳间隔约为 **5 秒**。

**《Among Us》**则展示了极简房间列表设计：仅显示房间代码、当前人数/最大人数和地图名称三个字段，不提供延迟数据，针对其移动端主要受众降低了认知负担，整个列表 UI 的加载时间目标控制在 **2 秒**以内。

## 常见误区

**误区1：列表数据越实时越好**
开发者常误认为应将刷新频率设置得尽可能高。然而，过于频繁的拉取（如每秒刷新）在房间数量较多时会对大厅服务器造成显著的查询压力，且对玩家的实际加入决策几乎没有帮助——玩家浏览并选择房间的平均时间约为 **10~30 秒**，5~15 秒的刷新间隔通常已能满足体验需求。

**误区2：所有筛选都在服务端完成**
将延迟（Ping）筛选全部交给服务端处理是常见错误。Ping 值本质上是客户端到各房间主机的实时测量值，服务端存储的 Ping 数据是房间主机自报的估算值，存在偏差。正确做法是由客户端在拿到房间列表后，对候选房间的 IP 地址执行一次轻量级探针（ICMP ping 或 UDP 握手），用实测值覆盖服务端上报值，再进行延迟排序。

**误区3：房间列表等同于匹配大厅**
房间列表浏览是"玩家主动选择"模式，要求玩家自行浏览并决定加入哪个房间；而自动匹配（Matchmaking）是"系统被动分配"模式，玩家无需关心具体房间。两者的后端数据结构、刷新策略和UI流程均不相同，不应在架构设计上将两者复用同一套房间索引服务而不加区分。

## 知识关联

**前置概念——房间创建**：房间列表浏览的数据来源完全依赖房间创建流程产生的元数据。房间创建时写入索引的字段集合，决定了列表浏览能提供哪些筛选维度；若创建时未存储"地图"字段，列表端就无法按地图过滤。

**后续概念——服务器选择**：玩家在房间列表中点击某个房间后，进入服务器选择（或直接加入）流程。此时客户端需使用列表条目中携带的房间ID或服务器地址，向大厅服务器发送 JoinRoom 请求，涉及连接协商与地址解析，是房间列表浏览数据流的自然延续。

**后续概念——观战大厅**：部分游戏在房间列表中标记已满员但可加入旁观的房间，观战大厅功能需要房间列表浏览在元数据中额外维护 `SpectatorSlots` 字段，并在UI层提供"仅显示可观战房间"的筛选开关，两者共享同一套拉取和刷新基础设施。
