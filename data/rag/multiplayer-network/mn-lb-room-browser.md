# 房间列表浏览

## 概述

房间列表浏览是大厅系统中玩家在完成房间创建之后、正式匹配之前所使用的核心UI功能，它将服务器上所有公开房间的状态以结构化列表形式呈现给客户端，让玩家能够手动挑选并加入目标房间。与自动匹配（Matchmaking）不同，房间列表浏览赋予玩家完整的主动选择权——玩家可以看到房间名称、当前人数、地图、模式等元数据，然后决定是否加入。

该功能最早在1990年代末的局域网游戏大厅中成型，Battle.net 1.0（1996年发布）是最早大规模商用房间列表浏览系统之一，其频道+房间的二级结构直接影响了此后十余年的游戏大厅设计范式。GameSpy Arcade（2000年）进一步将该模式推广到第三方游戏，引入了按Ping值排序和地理位置筛选等筛选维度。

房间列表浏览的价值在于它是低延迟社交协作的入口：玩家可以寻找同公会成员开设的房间、指定特定规则集的房间，或观察某张地图的实时热度。对于开发者而言，房间列表请求的频率和筛选参数还是分析玩家偏好的重要数据来源。

## 核心原理

### 列表数据的拉取与推送机制

房间列表的数据同步有两种主要模式：**轮询拉取（Polling）**和**增量推送（Delta Push）**。轮询拉取是客户端每隔固定间隔（通常5~30秒）向大厅服务器发送一次 `LIST_ROOMS` 请求，服务器返回当前全量或分页的房间快照。增量推送则依赖服务器在房间状态变更时主动向订阅的客户端发送差异数据包，仅包含新增、删除或更新的房间条目，能将带宽消耗降低60%~80%。

现代游戏（如《Minecraft》基岩版的第三方服务器列表）通常采用混合策略：首次加载使用全量拉取，后续刷新使用增量推送或带 `ETag` 缓存头的条件请求，避免重复传输未变化的房间数据。

### 筛选与搜索的服务端实现

筛选条件在后端通常映射为数据库查询或内存索引查询。常见筛选字段包括：`game_mode`（游戏模式）、`map_name`（地图）、`player_count`（人数区间，如 `2<=n<=8`）、`has_password`（是否有密码）、`region`（地区）。以 Redis Sorted Set 为例，可以将房间按当前人数存入有序集合，实现 O(log N + M) 复杂度的人数区间查询，其中 N 为总房间数，M 为结果数量。

全文搜索房间名称通常使用前缀索引（Trie）或 Elasticsearch 的 `match_phrase_prefix` 查询，而不是数据库 `LIKE '%keyword%'` 全表扫描——后者在房间数超过10,000条时会造成明显延迟。

### 分页与排序策略

单次返回全部房间列表会导致数据包过大，因此需要分页。标准做法是**基于游标的分页（Cursor-based Pagination）**，客户端每次请求携带上次结果集末尾的 `cursor_id`，服务器返回该游标之后的 N 条记录。相比 `OFFSET/LIMIT` 分页，游标分页在列表频繁更新时不会出现跳行或重复显示的问题。

排序维度直接影响玩家的浏览体验。常见排序字段包括：房间人气（`player_count DESC`）、创建时间（`created_at DESC`）、Ping值（需客户端测量后上传）、以及"好友所在"（优先排列有好友的房间）。Valve 的 Steam 大厅系统通过 `ISteamMatchmaking::AddRequestLobbyListSortNearValueFilter` API 允许开发者指定自定义数值排序键，最多支持叠加16个筛选条件。

### 客户端UI渲染

房间列表UI通常使用虚拟滚动（Virtual Scrolling）技术，仅渲染当前视口内可见的房间条目，而非将全部数据生成DOM节点。例如，当列表包含5,000个房间时，虚拟滚动只实例化约20~30个列表项组件，随滚动位置复用节点，将内存占用和渲染时间压缩到与列表长度无关的常数级别。

## 实际应用

**《CS:GO》社区服务器浏览器**是房间列表浏览的经典实现案例。其服务器列表由 Valve 的 Steam Master Server 维护，服务器每隔60秒向 Master Server 发送心跳包以保持在线状态。客户端通过 UDP 协议的 `A2S_INFO` 查询每台服务器的实时状态（地图、人数、延迟），这些查询在客户端本地并行发出，而非由 Master Server 统一代理，从而将延迟检测的负担分散到客户端侧。

**《Among Us》** 的公开房间列表按地区分片存储，每个大区（北美、亚洲、欧洲）维护独立的房间列表服务。玩家选择地区后，客户端仅从对应分片拉取数据，单次请求返回最多30条房间，支持按房间代码直接搜索加入，绕过列表浏览直接精确定位目标房间。

**《Minecraft》Java版**的第三方服务器列表采用客户端主动Ping模式：服务器地址由玩家手动添加，客户端定期发送 `0xFE` 服务器列表Ping数据包，解析返回的 `MOTD`（Message Of The Day）字符串来获取人数和版本信息，整个流程不依赖任何中央大厅服务器。

## 常见误区

**误区一：房间列表刷新越频繁越好**。实际上，客户端刷新频率过高会对大厅服务器造成不必要压力。以100名同时在线玩家每秒刷新一次为例，服务器每秒需处理100次列表查询；若改为10秒轮询一次，同等负载降低90%。正确做法是根据玩家行为阶段动态调整刷新间隔：刚打开房间列表时可触发一次即时刷新，之后自动降频至10~30秒一次。

**误区二：筛选逻辑应放在客户端执行**。部分开发者为减少服务器压力，选择将全量房间列表下发到客户端再由前端JavaScript/Lua过滤。当房间数量较少（< 200条）时这种做法勉强可行，但一旦房间数量增长到数千条，初始下载数据量和内存占用将变得不可接受。正确架构是在服务端建立索引并执行筛选，客户端只接收符合条件的子集。

**误区三：房间列表中的人数显示可以完全信任**。房间列表中显示的当前人数是服务器上报的缓存值，并非实时状态。在玩家点击"加入"到实际连接服务器之间存在数秒的时间窗口，期间房间可能已满。因此加入逻辑必须在服务端再次校验人数上限，客户端需处理 `ROOM_FULL` 错误响应并向玩家展示相应提示，而不能仅凭列表中的人数判断能否加入。

## 知识关联

房间列表浏览直接依赖**房间创建**功能产生的数据结构：房间创建时写入大厅服务器的元数据字段（地图、模式、密码哈希、最大人数）正是房间列表中每个条目展示和筛选的依据。若房间创建阶段未定义某个元数据字段，该字段就无法出现在列表的筛选选项中。

在房间列表浏览之后，玩家面临两种路径：若手动选择了特定服务器节点，则进入**服务器选择**流程，需要处理节点地理位置、负载均衡和连接迁移等问题；若玩家进入房间后选择旁观而非参与，则触发**观战大厅**功能，该功能需要复用房间列表中的房间ID和连接凭证，以观察者身份而非参与者身份建立会话连接。房间列表作为玩家进入游戏会话前的最后一道手动决策界面，其设计质量直接影响玩家从浏览到加入的转化率。
