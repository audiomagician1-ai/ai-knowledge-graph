# 好友系统

## 概述

好友系统是网络多人游戏大厅中负责管理玩家社交关系的功能模块，具体包含三项核心能力：好友关系的添加与删除、实时在线状态的广播与接收、以及基于好友关系的组队邀请。与其他游戏功能不同，好友系统存储的是玩家账号之间的持久化双向关系，即使两名玩家同时下线，这段关系依然保存在服务器数据库中。

好友系统的标准化形态在2000年代初随《魔兽世界》（2004年发布）的好友列表功能而被广泛采纳，此后成为几乎所有多人在线游戏的标配模块。早期单机游戏没有好友系统，玩家只能通过IP地址手动连接对方，而好友系统的出现将"找人一起玩"这件事从技术操作变成了一键邀请的社交行为。

好友系统的设计直接影响玩家的留存率。根据游戏行业研究数据，拥有至少5名游戏内好友的玩家，其30日留存率比无好友玩家高出约3到5倍，因此好友系统不只是便利功能，而是游戏运营指标中的关键杠杆之一。

## 核心原理

### 好友关系的数据结构

好友关系在数据库中通常以两条互相指向的记录存储，而非单条记录。例如，玩家A（ID: 1001）添加玩家B（ID: 1002）为好友，数据库中会写入两行数据：`(1001, 1002, FRIEND)` 和 `(1002, 1001, FRIEND)`。这种双向冗余存储的好处是：查询某玩家的好友列表时，只需要 `WHERE user_id = 1001` 的单表查询，避免了每次查询都要做 `user_a OR user_b` 的复杂联合条件。

好友关系还存在"待确认"状态，即 `PENDING`，用于支持需要对方同意的好友申请流程。完整的状态机通常为：`NONE → PENDING → FRIEND`，以及从任意状态到 `BLOCKED` 的单向跳转。BLOCKED 状态比删除好友更严格，它会阻止对方再次发送申请。

### 在线状态的广播机制

好友列表中的在线状态并非客户端主动轮询服务器获取，而是由服务器主动推送（Push）给相关玩家。当玩家A上线时，游戏服务器会查询A的好友列表，找出其中当前在线的玩家，向他们的客户端推送"A已上线"的通知消息。这种发布-订阅（Pub/Sub）模型使得在线状态的延迟通常在1秒以内。

在线状态不只是二元的"在线/离线"，现代好友系统会细分为多种子状态，例如：`在线（大厅中）`、`游戏中（副本名称）`、`忙碌（请勿打扰）`、`离线`。其中"游戏中"状态会附带玩家当前所在的游戏场景信息，这是后续实现"加入好友所在房间"功能的数据基础。

### 邀请组队的实现逻辑

好友系统中的邀请功能本质上是一条从邀请方流向被邀请方的消息，消息体中携带邀请方的组队房间ID（RoomID）。被邀请的玩家点击"接受"后，客户端用该 RoomID 向大厅服务器发起加入请求，整个流程与陌生人手动输入房间号加入的流程在底层是完全相同的。

邀请消息需要设置有效期，通常为30秒到2分钟之间。超时未响应的邀请需要由服务器端定时任务清理，否则会出现玩家在游戏已经开始后还能点击邀请链接的异常情况。此外，若被邀请玩家的在线状态为"忙碌"，服务器应直接拒绝邀请请求，而不是将邀请消息静默投递后无人处理。

## 实际应用

**《英雄联盟》好友分组功能**：该游戏允许玩家将好友手动归类到自定义标签组（如"开黑队友"、"同事"），每个分组最多显示100名好友。这解决了好友数量超过100人时列表难以管理的问题，是大型用户量游戏中好友系统的成熟演进方向。

**跨平台好友系统（Epic Games Store）**：Epic 的好友系统通过统一账号体系，让在 PC、主机和移动端登录的玩家共享同一份好友列表，并能跨平台互相邀请。这需要在在线状态记录中额外存储玩家当前登录的平台类型（Platform字段），才能正确路由邀请消息到目标平台的客户端。

**《原神》好友留言板**：该游戏在基础好友关系之上附加了留言板功能，好友可以在彼此的主页留下文字消息，且消息对留言方不可见（即发出后不知道对方是否读了）。这是好友系统作为社交基础设施向异步通信扩展的典型案例。

## 常见误区

**误区一：好友关系是单向存储的**。初学者常常将好友关系设计为只存一条记录 `(A, B)`，然后在查询时用 `WHERE user_a = X OR user_b = X` 来获取好友列表。这在好友数量少时可以工作，但在数据量增大后，这类 OR 查询无法利用单列索引，会导致严重的查询性能问题。正确做法是双向冗余存储，以空间换取查询速度。

**误区二：在线状态变化用轮询（Polling）获取**。如果客户端每隔5秒向服务器查询一次好友在线状态，100名好友的玩家每分钟会产生约1200次不必要的请求。正确方案是服务器在状态变化时主动推送差量更新，客户端只在首次加载好友列表时请求完整状态快照，后续通过增量推送保持同步。

**误区三：好友系统与平台好友系统是等价的**。游戏自身维护的好友列表和 Steam、PlayStation Network 等平台好友列表是两套独立的数据，它们之间并不自动同步。许多开发者误以为接入了 Steam SDK 就自动拥有了好友系统，实际上平台 SDK 只提供读取平台好友列表的接口，游戏内好友关系的存储、状态广播和邀请逻辑仍需自行实现，或通过游戏服务器中间层桥接。

## 知识关联

好友系统是后续多个高级社交功能的数据基础。**邀请系统**在好友系统之上进一步规范化邀请的流程控制，包括邀请频率限制、跨服邀请路由等；没有好友系统提供的关系数据和在线状态，邀请系统无法判断邀请目标是否可达。

**在线状态**模块是从好友系统中拆分出来进行独立优化的方向，当游戏服务器规模扩大到多集群部署时，在线状态的跨节点同步问题（通常用 Redis Pub/Sub 或专用状态服务解决）需要单独设计，这正是好友系统在工程复杂度提升后自然演化出的专项课题。

**公会/战队系统**与好友系统共享同类型的关系型数据结构（多对多关系表），但公会成员关系是多人群体关系而非一对一关系，其权限分级（会长、管理员、成员）和群体广播逻辑比好友系统更为复杂。理解好友系统的双向关系存储和状态推送后，再学习公会系统时可以直接复用这两个基本概念。

**平台SDK集成**是好友系统走向工程完善的关键一步：当游戏需要在 Steam 或主机平台发行时，本地好友数据库必须能够与平台 SDK 提供的好友列表进行映射和双向同步，这对账号体系设计（本地账号ID与平台账号ID的绑定表）提出了具体要求。
