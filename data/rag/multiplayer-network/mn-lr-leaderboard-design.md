# 排行榜设计

## 概述

排行榜设计是多人在线游戏中将玩家按某一或多项数值指标排序并公开展示的系统架构。其核心任务是在可接受的延迟内，从数百万条玩家记录中快速检索出Top-N名次，并以直观的UI呈现名次、ID、分数和头像。排行榜本质上是一个有序集合（Sorted Set）的工程问题，数据量越大，读写效率与一致性的权衡越关键。

排行榜作为竞技功能最早可追溯至街机时代的"HIGH SCORE"屏幕，彼时只存本机10条记录。2003年前后随着宽带普及，《魔兽争霸III》的战网引入了基于胜率的全球天梯，开创了现代在线排行榜的基本范式。2010年后移动平台的Game Center和Google Play游戏服务将社交排行榜（好友榜）标准化，使"好友比较"成为留存驱动的核心设计工具之一。

排行榜的意义在于它是段位系统（Rank System）对外的"可视化窗口"。段位告诉玩家自己处于哪个区间，而排行榜告诉玩家在该区间内的具体名次。两者结合才能同时满足玩家对"我有多强"和"我比他强"这两种成就需求，从而驱动更长时间的重复游戏行为。

## 核心原理

### 排行榜类型与数据范围

**全球排行榜（Global Leaderboard）** 覆盖游戏全体活跃玩家，数据量通常达到百万乃至千万级别。由于读写并发极高，工业界普遍采用 Redis 的 `ZADD` / `ZRANK` 命令维护有序集合，其时间复杂度为 O(log N)，可在毫秒级完成单次读写。全球榜的价值在于为顶尖玩家提供可见的荣耀激励，但对中低分玩家几乎无激励效果，因为普通玩家无论如何都看不到自己的名字。

**好友排行榜（Friends Leaderboard）** 仅显示当前玩家好友关系链中的分数排序，通常从全局数据集中动态筛选。其设计要点是"可触达性"——玩家始终能看到自己周围±3名的好友，这一"接近效应"（Proximity Effect）被证明比全球排名更能刺激短期分数冲刺行为。Facebook 2012年发布的游戏数据分析报告中指出，引入好友榜后次日留存率平均提升12%—18%。

**区域排行榜（Regional Leaderboard）** 按服务器区（如亚服、欧服、美服）或地理位置分片，主要解决两个问题：一是减少跨洲际网络延迟对实时分数同步的影响；二是为玩家提供更具本地文化认同感的竞争环境。部分游戏（如《英雄联盟》）在大区内还会进一步按语言或国家细分。

### 更新频率与一致性策略

排行榜更新频率直接决定系统负载与数据新鲜度的权衡。常见策略分为三类：

- **实时更新（Real-time）**：每局结束立即写入，延迟 <1 秒。适用于日活百万以下的中小规模游戏，或仅更新好友榜等小数据集场景。
- **准实时更新（Near-real-time）**：采用消息队列（如 Kafka）异步写入，积压批量处理，通常延迟 10—60 秒。《王者荣耀》的全国段位榜采用此策略，在亿级用户规模下将峰值写入 QPS 压缩到可控范围。
- **定时快照（Scheduled Snapshot）**：每小时或每日重新计算一次排名，将结果缓存为静态数据。适用于"每日积分榜"等时效性要求低的场景，实现最简单但数据滞后明显。

### 展示层设计

展示层需要解决"无限长列表中定位玩家自身"的UX问题。通用方案是将排行榜分成两个视图区域：顶部固定显示 Top 10（或 Top 100），中部显示当前玩家为中心的 ±N 名邻域（通常 N=5）。这两个区域在数据库层面对应两次独立查询：`ZRANGE key 0 9` 获取头部，`ZRANK` 获取自身排名后再用偏移量取邻域。

排行榜条目的展示元素标准化为：排名序号、玩家昵称/头像、核心分数（主排序键）、辅助信息（胜率、段位图标）。颜色编码通常用金/银/铜区分前三名，这一视觉惯例来源于奥运会奖牌制度，在游戏玩家中具有极高的跨文化认知度。

## 实际应用

**《Clash Royale》的全球与俱乐部双轨榜**：该游戏同时运营全球奖杯榜（Top 200公开可见）和俱乐部内部贡献榜。俱乐部榜每周重置捐卡数，刺激成员每周活跃；全球奖杯榜采用实时更新，季末重置机制会将全体玩家分数削减约50%，确保榜单不会被早期玩家永久占据。

**《原神》的深渊统计榜**：虽然《原神》没有传统PvP排行榜，但其"深渊数据"页面本质上是一个"过关速度"的匿名聚合统计榜。该设计规避了竞争压力引发的负面情绪，同时通过"全服X%的玩家完成了X星"这类相对信息满足玩家的比较心理，是一种低冲突的排行榜变体。

**Redis ZSet 示例**（伪代码）：
```
ZADD leaderboard:season3:global <score> <player_id>
ZREVRANK leaderboard:season3:global <player_id>   # 获取降序排名
ZREVRANGE leaderboard:season3:global 0 9 WITHSCORES  # 获取Top10
```

## 常见误区

**误区一：全球榜覆盖面越广越好**。实际上，对于90%以上的非顶尖玩家，一个自己永远进不了的榜单毫无激励价值。《糖果传奇》的运营数据表明，好友榜的点击率是全球榜的7倍以上，这说明排行榜的竞争范围应当与玩家的"心理可及性"相匹配。

**误区二：更新频率越高越好**。实时更新的全球榜在百万玩家并发写入时会造成严重的数据库热点问题（Hot Key Problem），尤其是分数集中在少数几个数值附近时，Redis单分片的写入吞吐会成为瓶颈。更新频率应根据玩家对"分数刷新感知阈值"来设定——大多数玩家对30秒以内的延迟无感知差异。

**误区三：排行榜只需要展示分数**。缺乏上下文的分数数字对玩家意义有限。工业实践中，排行榜条目至少应附带段位标识和当前分数与第N+1名的分差（"+23分超越下一名"），这一"差距提示"设计能直接转化为短期内的游戏行为。

## 知识关联

排行榜设计依赖**段位系统**提供分数定义和区间划分逻辑：段位决定了排行榜是用ELO分、胜点数还是自定义积分作为排序键，不同排序键的分布特征会直接影响榜单的竞争密度。

从排行榜设计延伸出的进阶概念包括：**计分系统**（定义分数的计算规则和权重模型）、**分区排行榜**（将玩家按分段动态分组以提升竞争可触达性）、**赛季重置**（通过周期性清零或压缩分数维持榜单活力）以及**排行榜防刷**（识别和拦截脚本刷分、断线挂机等作弊行为）。这四个方向分别对应排行榜的"输入层"、"分层层"、"时间层"和"安全层"，共同构成完整的排行榜系统工程。
