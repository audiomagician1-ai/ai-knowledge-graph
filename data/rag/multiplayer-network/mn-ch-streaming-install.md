# 流式安装

## 概述

流式安装（Streaming Installation）是一种允许玩家在游戏资源尚未完全下载到本地的情况下，即可开始游玩的技术方案。其核心思路是将游戏资源按照玩家最可能的访问顺序进行优先级排序，优先下载"即将需要"的内容块，而非等待全部数据落盘后才启动游戏。PlayStation 4于2013年正式将流式安装纳入系统功能，规定发行商必须在玩家下载完首个数据块（通常不超过1GB）后即允许游戏启动，这使得"边玩边下载"从实验性功能演变为行业标准实践。

流式安装的必要性源于现代游戏体积的爆炸式增长。以《使命召唤：现代战争 2019》为例，其完整安装包超过200GB，若要求玩家完整下载后才能游玩，在100Mbps的家用带宽下也需要约4.5小时等待。流式安装通过将"能玩"与"完整安装"解耦，极大地降低了玩家从购买到首次体验的等待门槛，直接影响了用户留存率与付费转化率。

## 核心原理

### 资源依赖图与优先级队列

流式安装的关键技术基础是构建**资源依赖图（Asset Dependency Graph）**。开发者在构建阶段分析哪些资源属于"启动关键路径"，即游戏主菜单、新手教程或第一关卡所必须加载的内容，将这些资源打包为**Priority 0**数据块，优先推送至下载队列头部。其余资源按照游戏流程的先后顺序依次排列在队列中，下载管理器则在后台持续填充本地缓存。

技术实现上，每个数据块通常以固定大小切片（常见大小为32MB或64MB），并附带一份清单文件（Manifest），记录每个切片的哈希校验值、解压后大小以及游戏逻辑所需的最低到达时间（Deadline）。下载管理器会将当前下载速度与各切片Deadline进行实时对比，一旦发现某个关键切片有"来不及"的风险，便会提升其在CDN请求队列中的优先级。

### 数据块边界与游戏逻辑协作

流式安装并非纯粹的下载层技术，它要求游戏逻辑主动参与资源调度。游戏引擎需要在玩家触发场景切换（如进入新地图、选择新角色）前，向下载管理器查询所需资源包的到达状态。若对应数据块尚未落盘，引擎应以"Loading..."界面或过场动画进行遮掩，而不能直接调用缺失文件导致崩溃。Epic Games在《堡垒之夜》中将这一机制称为**Pak文件挂载策略**：每个Pak文件仅在其完整下载并通过SHA-1校验后，才被挂载到虚拟文件系统中，游戏逻辑仅能访问已挂载的Pak，从而在框架层面杜绝了访问未完成数据的可能性。

### CDN分发与动态调速

流式安装的效率高度依赖CDN分发策略。为保证玩家"玩到哪里，下到哪里"，CDN节点通常采用**HTTP Range请求**机制，允许下载管理器以字节范围（如`Range: bytes=67108864-134217727`）精确获取单个数据块，而无需从文件头开始下载整个资源包。与此同时，下载管理器还需实现**动态码率自适应**：当检测到网络带宽不足时，优先保障关键数据块的下载速率，可暂时降低非关键素材（如高清贴图的Mip最高层级）的下载优先级，确保游戏主流程不被中断。

## 实际应用

**Xbox Game Pass即玩即下（Play As You Download）功能**将流式安装的优先级逻辑暴露给运营层面：游戏发行后，Xbox平台工具链会要求开发者标记"第一阶段可玩包"，其体积上限为总安装包的30%，平台审核时会验证该阶段是否覆盖完整的新手体验或多人大厅功能。

**移动端游戏的首包+热更新模式**是流式安装在应用商店限制下的变体。由于iOS App Store和Google Play对初始包大小有200MB上限的约束，主流手游（如《原神》）选择发布一个轻量启动包，用户首次启动时由应用内下载管理器以流式方式拉取剩余4GB以上的资源包。游戏将主城与新手剧情所需资源排在下载序列最前端，玩家在等待下载期间即可完成角色创建和前30分钟的新手引导。

**PlayStation 5的SSD架构**进一步强化了流式安装的价值：PS5的定制SSD原始读取速度达5.5GB/s，系统可以将流式安装的"最小可玩数据块"压缩至几百MB，数秒内即可完成加载，使得流式安装在PS5上几乎感知不到延迟。

## 常见误区

**误区一：流式安装等同于云游戏串流**。云游戏（如Xbox Cloud Gaming）将渲染和计算全部放在服务器端，客户端仅接收视频流，本地无需存储游戏文件。流式安装则完全相反：游戏在本地硬件上运行，只是资源文件在后台异步下载填充，两者对网络带宽的依赖模式和延迟敏感性完全不同。

**误区二：只要把数据块切小就能实现流式安装**。若游戏引擎没有相应的资源到达状态查询接口，单纯将安装包切分为小块并不能带来任何体验提升——引擎仍会在启动时检索完整的资源清单，遇到未落盘文件即报错退出。流式安装必须是下载层与引擎层协同设计的结果，缺一不可。

**误区三：流式安装会增加总下载量**。部分开发者担心按需分块下载会因重复请求产生额外流量。实际上，设计良好的流式安装系统的总下载字节数与完整安装几乎相同，差异仅在于下载时序不同。唯一的额外开销是各数据块的Manifest清单文件，其大小相对游戏本体可忽略不计。

## 知识关联

流式安装建立在**下载管理器**的队列调度与断点续传能力之上：没有能够对单个数据块独立管理优先级、校验完整性并支持Range请求的下载管理器，流式安装的优先级分发机制将无从实现。下载管理器提供的多线程并发请求能力决定了流式安装能以多快的速度在CDN上并发拉取不同优先级的数据块。

在热更新体系中，流式安装与**补丁增量更新**存在技术交集：热更新的差分包也可以按照玩家当前所在的游戏区域或玩法模块进行拆分，优先推送当前会话所需的差分块，这本质上是将流式安装的分块优先级思想应用于版本更新场景。掌握流式安装的资源依赖图构建方法，是理解大型游戏版本管理与CDN调度策略的重要前提。
