# 时钟同步

## 概述

时钟同步（Clock Synchronization）是指通过测量网络往返延迟，计算出客户端与服务器之间的时钟偏差（Clock Offset），从而让双方在逻辑上共享同一条时间轴的技术手段。在多人游戏中，服务器的时间戳用于排序玩家输入、校验状态快照以及回滚操作，如果客户端的本地时间与服务器偏差超过一帧（例如在60Hz游戏中约16.7毫秒），就可能造成输入被服务器判定为"过期"而丢弃，直接导致玩家操作无效。

互联网时钟同步协议的先驱是**NTP（Network Time Protocol）**，由David Mills于1985年在RFC 958中首次规范化，后续经历多次修订（当前版本NTPv4规范于RFC 5905）。NTP的核心思想——用四个时间戳计算偏移量——被几乎所有游戏自定义时钟同步算法直接借鉴。游戏场景并不需要NTP所追求的UTC绝对精度（±1毫秒），只需要客户端与服务器之间的**相对偏差**足够小，通常游戏要求将该偏差控制在半帧时长以内。

时钟同步的意义在于它是预测回滚（Rollback Netcode）和输入延迟补偿的前提条件：服务器在处理"客户端时间戳为T的输入"时，必须能判断这个T相对服务器当前时间的位置，才能决定是回放历史还是直接应用。

## 核心原理

### NTP四时间戳公式

NTP算法使用四个时间戳来估算偏差：
- **t₀**：客户端发送请求的时刻
- **t₁**：服务器收到请求的时刻
- **t₂**：服务器发送响应的时刻
- **t₃**：客户端收到响应的时刻

由此推导出两个关键量：

**往返时延（RTT）**：
```
RTT = (t₃ - t₀) - (t₂ - t₁)
```

**时钟偏差（Offset）**：
```
Offset = ((t₁ - t₀) + (t₂ - t₃)) / 2
```

Offset为正表示客户端时钟落后于服务器，为负表示超前。客户端将本地时间加上此Offset即可得到与服务器对齐的"游戏时间"。该公式成立的前提假设是上行和下行网络延迟对称（各占RTT/2），而在实际广域网中这一假设往往不成立，这也是游戏自定义算法需要改进的核心痛点。

### 多样本滤波与异常值剔除

单次RTT采样受网络抖动影响极大，游戏中通常在玩家连接时连续发送**8到16个**同步包，对采集到的所有Offset样本进行处理。常见做法是：
1. **剔除高RTT样本**：丢弃RTT超过中位数150%的采样点，以排除网络拥堵瞬间的数据。
2. **取低RTT样本均值**：保留RTT最小的若干个样本（如最小的三分之一），对它们的Offset取平均值，原因是低RTT的样本意味着路由器队列延迟小，此时上下行对称假设更接近成立。
3. **低通滤波器平滑**：游戏运行过程中定期重新同步，使用指数加权移动平均（EWMA）对新旧Offset进行混合，避免时间轴突然跳变导致游戏逻辑出错。

### 时钟漂移补偿

即使初始同步完成，硬件晶振的频率误差会导致客户端时钟以略微不同的速率流逝，这一现象称为**时钟漂移（Clock Drift）**。廉价晶振的漂移率可高达每天约±1秒（约11.6 ppm）。游戏客户端通过定期重新发起同步探测（例如每30秒一次）来修正累积漂移，而不是依赖操作系统的系统时钟（`GetTickCount`或`clock_gettime`），因为后者可能受玩家手动调时或夏令时切换影响而发生跳变。

## 实际应用

**Valve的Source引擎**采用了一套自定义时钟同步机制：客户端维护一个名为`m_ClockOffsetFrames`的变量，单位是服务器帧数而非毫秒，这样时钟偏差与Tick率自然耦合。每隔约1秒，客户端发送一个`clc_RespondCvarValue`类型的同步包，服务器回复当前Tick编号，客户端据此计算本地Tick号应超前服务器多少帧以补偿网络延迟。

**Riot Games的《英雄联盟》**在其2012年发布的技术博文中披露，游戏客户端会在登录服务器完成后立即执行一轮时钟同步，目标是将客户端与游戏服务器的时间偏差控制在**±5毫秒**以内。超出此范围会导致技能命中判定时序不一致，表现为"明明点中了但没打到"的用户体验问题。

**移动网络场景**中时钟同步面临额外挑战：4G/5G网络的RTT抖动范围可从20ms到300ms不等，标准NTP四时间戳的对称假设更难成立。常见对策是增大采样次数至32个并使用更激进的离群值过滤（仅保留RTT最小的5个样本），代价是连接建立时间延长约1.5秒。

## 常见误区

**误区一：直接使用操作系统时间戳做同步**

许多初学者直接调用`Date.now()`（JavaScript）或`System.currentTimeMillis()`（Java）并传给服务器。这类系统时钟会受NTP校时跳变、玩家修改系统时间、夏令时切换等因素影响，发生非单调递增甚至向后跳跃的情况。游戏时钟同步应使用**单调时钟**（Monotonic Clock），如C++的`std::chrono::steady_clock`或Unity的`Time.realtimeSinceStartup`，它们只保证递增，不受系统时间修改影响。

**误区二：RTT/2 就等于单程延迟**

新手常将RTT除以2作为上行或下行的单程延迟，并据此计算Offset。然而移动网络、不对称宽带（上行1Mbps/下行100Mbps）的上下行路径物理路由可以完全不同，单程延迟比例可能是4:6甚至3:7而非1:1。直接使用RTT/2会导致Offset计算存在系统性误差，在高延迟环境下该误差可能达到**数十毫秒**。多样本采集配合低RTT筛选是缓解此问题的标准手段，而非完全消除。

**误区三：同步一次就够了**

时钟同步常被误解为连接建立时的一次性操作。由于时钟漂移的存在，客户端与服务器时钟的相对偏差会随时间缓慢增大。对于一场持续40分钟的MOBA对局，若不重新同步，仅时钟漂移本身就可能积累超过**50毫秒**的误差（以25 ppm漂移率估算：40×60×25×10⁻⁶ ≈ 0.06秒）。生产环境的游戏客户端必须在游戏中持续、周期性地重新同步时钟。

## 知识关联

时钟同步没有前置知识依赖，它本身是理解后续概念的基础测量工具。掌握时钟偏差的计算方式（特别是Offset公式中四个时间戳的物理含义）后，可以直接进入**网络Tick率**的学习：Tick率决定了服务器产生时间戳的最小粒度，而时钟同步的精度目标（如±5ms）必须与Tick率匹配——若Tick率为20Hz（每帧50ms），则将同步精度做到±1ms没有实际价值；若Tick率为128Hz（每帧约7.8ms，CS:GO竞技服务器标准），则同步精度必须远小于7.8ms才能保证帧级别的输入排序正确。两个概念共同构成状态同步系统中"时间维度"的完整描述。
