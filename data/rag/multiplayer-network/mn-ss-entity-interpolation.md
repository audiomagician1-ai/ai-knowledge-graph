# 实体插值

## 概述

实体插值（Entity Interpolation）是网络多人游戏中用于平滑渲染远端玩家或实体位置的技术，其核心思想是：客户端不直接渲染服务器传来的"当前帧"状态，而是渲染一段固定时间之前的历史状态，并在已知的两个历史快照之间做线性或曲线插值，从而消除网络抖动导致的视觉跳帧。

该技术由 Valve 工程师 Yahn Bernier 在 2001 年的 GDC 演讲及随后发布的 Source 引擎文档中系统阐述，在 Half-Life 和后来的 Counter-Strike 系列中得到了大规模工程验证。其根本动机在于：网络数据包的到达时间不均匀（即"网络抖动"），即使平均延迟只有 50ms，相邻两包的间隔也可能在 10ms 到 150ms 之间波动，如果直接用收到的位置渲染，实体会出现肉眼可见的"瞬移"或抖动。

实体插值与权威服务器模型直接配合——服务器以固定 Tick 率（如 64 tick/s 或 128 tick/s）广播世界状态，客户端将收到的每帧状态打上本地时间戳存入一个循环缓冲区，然后以"当前本地时间 - 插值延迟"这一时刻从缓冲区中重建平滑位置。这一机制以牺牲固定的额外延迟（通常 50–100ms）换取视觉连续性。

## 核心原理

### 快照缓冲区与时间轴偏移

客户端维护一个按服务器时间戳排序的快照队列，每条记录形如 `(t_server, position, rotation, ...)`。渲染时，客户端计算目标渲染时刻：

```
T_render = T_local_now - T_interp_delay
```

其中 `T_interp_delay`（插值延迟）通常设置为 **1 到 2 个服务器帧周期**。例如服务器 Tick 率为 64Hz（帧周期 ≈ 15.6ms），则插值延迟设为约 32ms。这意味着所有远端实体始终展示的是 32ms 前的状态，确保在正常网络条件下缓冲区内总能找到两个夹住 `T_render` 的快照。

### 线性插值公式

找到满足 `t₀ ≤ T_render ≤ t₁` 的相邻两个快照后，计算插值因子：

```
α = (T_render - t₀) / (t₁ - t₀)
```

然后对位置向量进行线性插值（LERP）：

```
P_render = P₀ + α × (P₁ - P₀)
```

对旋转量使用四元数球面线性插值（SLERP）而非直接 LERP，因为欧拉角直接插值在过零点（如从 359° 到 1°）会产生 358° 的错误旋转。

### 插值延迟的动态调整

固定延迟在高抖动网络下依然可能导致缓冲区"欠填充"（即 `T_render` 超出所有已知快照的右端）。此时有两种降级策略：
1. **外推（Extrapolation）**：用最后一个快照的速度向量预测位置，即临时切换为航位推算模式。此方法有累积误差风险，Source 引擎限制最多外推 **200ms**。
2. **冻结**：停止移动该实体，直到新快照到达。这在视觉上比外推的"橡皮筋"效果更保守，适用于低速移动物体。

动态延迟算法（如 CS:GO 的 `cl_interp_ratio`）会根据近 N=64 帧的包到达抖动标准差，自动将 `T_interp_delay` 上调至 **2σ**（σ为抖动标准差），以 99.7% 的概率保证缓冲区不欠填充。

## 实际应用

**Counter-Strike: Global Offensive** 的控制台变量 `cl_interp 0` 和 `cl_interp_ratio 2` 直接暴露了插值延迟参数。默认 `cl_interp_ratio=2` 对应 2 个 Tick 的延迟（128 tick 服务器下约 15.6ms，64 tick 服务器下约 31.25ms）。职业选手常将 `cl_interp_ratio` 设为 1 以减少延迟，但在抖动超过 8ms 的网络环境下会看到其他玩家出现轻微跳帧。

**Unreal Engine 5** 的 `NetUpdateFrequency`（默认 100Hz）和 `bReplicateMovement` 控制远端 Actor 的位置复制，引擎底层的 `UCharacterMovementComponent` 内置了基于上述 LERP 原理的平滑逻辑，并对非 Character 类型（如载具）提供了独立的 `SmoothNetUpdatePosition` 函数，其插值窗口默认为 **0.083 秒**（约 5 帧 @ 60fps）。

**Minecraft（Java Edition）** 服务器以 20 tick/s（50ms 间隔）广播实体位置，客户端对其他玩家的移动使用 3 步线性插值平滑：将差值分成 3 份，每渲染帧推进 1/3，这等价于一个固定步数而非固定时间的插值方案，在渲染帧率高于 20fps 时产生肉眼可见的"滑步"感，这也是 Minecraft 玩家远望他人移动时感觉"飘"的技术原因。

## 常见误区

**误区一：插值延迟越小越好**
许多初学者认为应将 `T_interp_delay` 调到最小（如 0ms）以降低延迟。实际上，插值延迟低于服务器帧周期时，缓冲区几乎总处于欠填充状态，渲染会持续触发外推或冻结，反而导致实体表现比不插值时更加卡顿。正确做法是将延迟设为至少等于服务器帧周期，并额外加上本地抖动余量。

**误区二：实体插值解决了"打人感"延迟问题**
实体插值仅优化"其他实体的视觉表现"，对本地玩家自身的操作响应没有帮助，也不影响命中判定。事实上，插值延迟会使对手的视觉位置落后于其在服务器上的真实位置，这正是服务器端延迟补偿（Lag Compensation）必须存在的原因——服务器需要"回滚"对手位置到射击者开枪时刻所见的插值位置再进行碰撞检测。

**误区三：旋转可以用欧拉角 LERP 代替 SLERP**
在角色朝向接近 ±180° 时，欧拉角从 179° 到 -179° 的 LERP 会插值出穿越 0° 的最长路径（旋转 358°），导致角色瞬间"转一圈"。SLERP 始终沿四元数超球面的短弧插值，保证最短旋转路径，这一差异在实际项目中曾多次造成难以排查的角色抖动 Bug。

## 知识关联

**前置：权威服务器模型** 提供了实体插值工作的基础结构——因为服务器以固定 Tick 广播带时间戳的权威状态，客户端才有可靠的历史快照队列可供插值使用。若客户端之间采用 P2P 拓扑，时间戳对齐本身就成为难题，插值的时间轴计算会复杂得多。

**后续：航位推算（Dead Reckoning）** 是实体插值在缓冲区欠填充时的自然降级方案，同时也是低频率更新网络（如移动游戏中 10–20 tick/s 的服务器）中对实体插值的替代策略。航位推算不依赖历史快照而是依赖物理模型进行前向预测，两种技术经常在同一系统中混用：插值区间内用 LERP，超出最新快照后切换为航位推算。

**后续：动画同步** 与实体插值密切耦合，因为角色的骨骼动画状态（如奔跑/跳跃/射击）需要与插值后的位置在同一时间轴上保持一致。若动画状态机以服务器 Tick 为单位同步，而位置以插值后时刻渲染，两者的时间偏差会导致"脚步动画与位移不匹配"（俗称"月球漫步"）问题，需要额外的动画时间轴偏移校正。
