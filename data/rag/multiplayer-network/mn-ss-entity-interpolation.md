# 实体插值

## 概述

实体插值（Entity Interpolation）是网络多人游戏中用于平滑渲染远程实体位置的客户端技术。其核心思想是：客户端不直接渲染服务器发来的"当前"状态，而是故意延迟渲染时间线，在两个已知快照之间进行插值，从而消除因网络抖动和丢包造成的视觉跳变。Valve在2001年发布的《Half-Life》网络模型技术文档中首次系统描述了这一算法，并将其命名为"Entity Interpolation"，此后成为第一人称射击游戏网络架构的标准组件。

实体插值之所以必要，是因为服务器通常以固定频率（如64 tick/秒或20 tick/秒）向客户端广播游戏状态，而客户端显示帧率往往高达60~144 FPS。若客户端每帧都直接使用最新收到的服务器状态，则远程玩家的位置会以"快照频率"跳跃更新，产生明显的抽搐感。插值通过在两个快照之间的时间轴上求中间值，将离散的网络更新转化为视觉连续的运动。

## 核心原理

### 时间轴延迟与插值窗口

实体插值的关键参数是**插值延迟（Interpolation Delay）**，通常设置为略大于一个发包周期的时间。例如服务器以64 tick运行时，每包间隔约15.6毫秒，客户端的插值延迟通常设为100毫秒（CS:GO默认值`cl_interp_ratio 2`对应约31.25ms，但实际延迟包含缓冲）。客户端维护一个**状态快照缓冲区**，存储最近若干帧的服务器状态，渲染时使用的时间戳 `T_render = T_now - interpolation_delay`。

### 线性插值公式

最基础的实体插值使用线性插值（LERP）：

```
P(t) = P_A + (P_B - P_A) × α
```

其中：
- `P_A` = 插值窗口起点快照的位置
- `P_B` = 插值窗口终点快照的位置
- `α = (T_render - T_A) / (T_B - T_A)`，值域为 `[0, 1]`

当 `T_render` 落在 `T_A` 和 `T_B` 之间时，`α` 自然在0~1范围内，插值结果严格处于两个已知位置之间，因此实体插值得到的位置永远不会超出历史已知范围——这与航位推算（Dead Reckoning）的本质区别在于，插值是在**已知数据之间**求值，而推算是在**已知数据之后**外推。

### 快照缓冲区管理

客户端需要维护一个按服务器时间戳排序的环形缓冲区（Ring Buffer）。Source引擎中该缓冲区默认保存约1秒的历史快照。每次渲染时，系统在缓冲区中二分查找满足 `T_A ≤ T_render ≤ T_B` 的相邻快照对。若找不到合适的快照对（例如严重丢包导致缓冲区断层），则退化为使用最近一次已知位置，或触发外推逻辑。丢包率超过5%时，这种退化会导致实体出现短暂"冻结"现象，这是插值算法固有的弱点。

### 旋转插值的特殊处理

位置插值使用LERP即可，但旋转的插值必须使用**球面线性插值（SLERP）**，否则在角度差较大时（如角色快速转身超过180°）会出现走捷径旋转（Gimbal Lock相关问题）。四元数SLERP公式为：

```
q(t) = sin((1-α)Ω)/sin(Ω) × q_A + sin(αΩ)/sin(Ω) × q_B
```

其中 `Ω = arccos(q_A · q_B)` 为两个方向之间的角度。

## 实际应用

**CS:GO/CS2 中的实体插值配置**：CS:GO暴露了`cl_interp`和`cl_interp_ratio`两个控制台参数供玩家调整。服务器以64 tick运行时，`cl_interp_ratio 1`对应约15.6ms插值延迟，`cl_interp_ratio 2`对应约31.3ms。职业比赛中玩家通常将其设置为最小值以降低"见到敌人"到"开枪命中"的感知延迟，但这在丢包时会导致敌方位置抖动。

**Minecraft中的实体插值**：Minecraft Java版服务器默认以20 tick/秒广播实体位置，客户端对其他玩家和生物均使用5步线性插值（即将服务器位置作为目标，每客户端tick移动1/5距离），这是一种简化的趋近插值（Approach Interpolation）而非标准的时间戳LERP，因此在服务器卡顿时表现较差。

**Overwatch的快照压缩**：Blizzard在GDC 2017演讲中披露，Overwatch对每个实体的位置量化精度为0.25cm，在插值前对位置数据进行了有损压缩，以在60Hz更新率下将每个客户端的带宽控制在约80~100 Kbps。

## 常见误区

**误区一：插值延迟越小越好**。降低插值延迟意味着缩短快照缓冲区的时间窗口，当网络抖动（Jitter）超过该窗口时，系统频繁找不到合适的快照对，反而导致更多视觉跳变。正确做法是将插值延迟设置为略大于平均网络抖动值，通常推荐 `interpolation_delay ≥ 1.5 × jitter`。

**误区二：实体插值会影响本地玩家的命中判定**。实体插值仅影响**其他玩家的视觉渲染位置**，本地玩家自身的位置由本地预测（Client-Side Prediction）处理，不经过插值。命中判定由权威服务器执行，服务器会用**延迟补偿（Lag Compensation）**将双方的历史位置回溯对齐，因此插值延迟确实会影响延迟补偿的回溯时间窗口，但不直接等同于"瞄准偏差"。

**误区三：插值等同于外推/预测**。两者在数学上完全不同：插值使用 `α ∈ [0,1]`，结果落在已知数据范围内；外推使用 `α > 1`，结果超出已知范围。在丢包或高延迟情况下，才需要从插值切换到外推（航位推算），两者是互补而非相同的技术。

## 知识关联

实体插值以**权威服务器模型**为前提：只有服务器是唯一状态权威来源，客户端收到的才是可信快照序列，插值才有意义。若使用P2P架构，各客户端收到的状态可信度不一，插值缓冲区的构建本身就存在问题。

掌握实体插值后，自然引出两个方向：其一是**航位推算**，解决快照缓冲区耗尽时如何用物理模型外推位置；其二是**动画同步**，解决在插值出的位置和旋转之外，如何同步动画状态机（如奔跑、跳跃、开枪动作），使插值后的实体不仅位置正确，动画表现也与服务器状态匹配。三者共同构成了远程实体表现层的完整解决方案。
