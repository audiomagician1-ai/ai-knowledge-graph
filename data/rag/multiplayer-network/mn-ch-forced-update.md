# 强制更新策略

## 概述

强制更新策略是指游戏运营方在检测到客户端版本号低于某个最低兼容版本（`min_supported_version`）时，阻断玩家进入游戏并强制其下载新版本的决策机制。与可选更新不同，强制更新会在登录验证阶段由服务器返回特定错误码（通常为 `UPDATE_REQUIRED` 或类似枚举值），客户端收到该码后无法绕过，必须完成更新流程才能继续游戏。

这一机制的出现与网络游戏协议兼容性问题直接相关。早期网络游戏（1990年代末至2000年代初）因客户端版本混乱导致服务器崩溃或数据损坏的事件屡见不鲜，促使开发者引入服务器端版本白名单机制，逐渐演变为今天成熟的强制更新策略体系。

强制更新策略在多人在线游戏中具有不可替代的工程意义：协议变更（如 Protobuf schema 升级）、反作弊模块替换、关键安全漏洞修补等场景都会造成新旧客户端无法共存于同一服务器，此时强制更新是保障所有在线玩家数据一致性的唯一手段。

## 核心原理

### 版本号比对逻辑

强制更新的触发依赖版本号的三段式结构，通常采用语义化版本 `MAJOR.MINOR.PATCH`。服务器维护两个关键字段：`latest_version`（最新版本）和 `min_supported_version`（最低兼容版本）。客户端在握手阶段上报自身版本号，服务器执行如下判断：

```
if client_version < min_supported_version:
    return ERROR_FORCE_UPDATE
elif client_version < latest_version:
    return NOTICE_OPTIONAL_UPDATE
else:
    return OK
```

`min_supported_version` 由研发团队在每次发布时手动或通过 CI/CD 流水线更新，其值的设定直接决定了多少比例的玩家会被触发强制更新。例如，将 `min_supported_version` 从 `1.2.0` 提升至 `1.5.0`，所有运行 `1.2.x`、`1.3.x`、`1.4.x` 的玩家都将被强制更新。

### 触发条件的决策分类

强制更新的触发并非任意为之，业界通常将其限定在以下三类必要场景：

**协议不兼容**：客户端与服务器通信协议发生破坏性变更（breaking change），例如消息字段被删除或类型改变，旧客户端发出的报文无法被新服务器解析，必须强制更新。

**安全修补**：发现高危漏洞（CVSS评分 ≥ 7.0 的漏洞）时，若补丁无法通过服务端热修复解决，则必须下发强制更新。例如 2021 年某款 Battle Royale 游戏因客户端内存越界漏洞被利用，在 6 小时内完成了全平台强制更新推送。

**资产校验失败**：反作弊系统（如 EAC、BattlEye）检测到客户端核心文件哈希与服务器白名单不符，需强制重新下载以排除篡改文件。

### 差量更新与全量更新的选择

强制更新的内容分发方式直接影响玩家流失率。研究数据显示，更新包每增加 100MB，在移动端导致的安装放弃率约上升 8%–12%。因此工程实践中优先使用 **VCDIFF** 或 **bsdiff** 算法生成差量补丁包，仅在差量包大小超过全量包 60% 时才退化为全量下载。CDN 分发时，差量包的 URL 路径通常格式为 `/patch/{from_version}_to_{to_version}.vpk`，客户端根据自身版本号构造请求地址。

## 实际应用

### 移动端强制更新的 UX 设计

移动端强制更新面临应用商店审核延迟（App Store 审核周期通常为 1–3 天）与服务器协议变更时间窗口不匹配的矛盾。常见解决方案是在强制更新弹窗中区分两种来源：**应用商店更新**（重定向至 App Store/Google Play）和 **CDN 热更新**（应用内直接下载资源包，无需过店）。《王者荣耀》等头部手游采用的方案是，非安装包级别的资产更新（如皮肤、地图数据）通过应用内 CDN 热更完成，无需重启应用；而涉及 so 库替换的更新则必须走商店渠道。

### PC 端后台静默更新

PC 客户端（如 Steam 平台）支持后台静默更新，可在玩家不在线时预下载补丁。强制更新策略在此场景下表现为：玩家启动游戏时 Steam API 已完成校验，直接进入游戏；若玩家禁用了自动更新，则在登录服务器时收到 `UPDATE_REQUIRED` 错误码，客户端触发 Steam 下载 UI。

### 灰度发布中的版本兼容窗口

灰度发布期间，新版本仅推送给 10%–20% 的玩家，此时 `min_supported_version` 不应立即升至新版本号，否则尚未收到更新推送的 80% 玩家将被误触发强制更新。标准做法是设置 48–72 小时的版本兼容窗口，待灰度全量铺开后再提升 `min_supported_version`。

## 常见误区

### 误区一：版本号越新就应立即强制全员更新

许多初级运营会在发布新版本后立即将 `min_supported_version` 设为最新版，导致大量玩家在游戏高峰期（如晚上 8–10 点）同时触发更新，CDN 带宽被瞬间打满，更新失败率飙升。正确做法是结合玩家版本分布数据（通常通过埋点上报），当旧版本玩家占比降至 5% 以下后再调整 `min_supported_version`。

### 误区二：强制更新弹窗不需要说明更新原因

玩家研究数据表明，提供明确更新原因（"此次更新修复了匹配系统崩溃问题"）比仅显示"需要更新才能继续游戏"的弹窗，玩家的更新完成率高出约 23%。强制更新 UX 中省略原因说明，会显著增加玩家投诉量和差评率。

### 误区三：热更新可以完全替代强制更新

热更新（通过 Lua、JS 脚本或资产替换实现）只能修改游戏逻辑层和资产层，无法替换已加载的 Native 代码（C++/C# 编译产物）、修改通信协议结构体，或更新操作系统级别的 SDK 依赖。因此凡涉及引擎升级、协议变更、反作弊 SDK 替换的场景，强制更新依然不可绕过。

## 知识关联

强制更新策略的判断逻辑直接建立在**版本管理**体系之上：若没有规范的语义化版本号体系和服务端版本白名单维护机制，强制更新的触发条件就无从定义。具体而言，`min_supported_version` 字段的更新通常集成在版本管理的发布流水线（Release Pipeline）中，与打包、签名、CDN 预热等步骤串联执行，确保版本控制的一致性与强制更新触发的精准性。

在实际工程链路中，强制更新策略还与 **CDN 分发架构**、**热更新资源管理**密切协作：CDN 决定补丁包的分发速度与覆盖范围，直接影响强制更新期间的玩家等待时长；热更新系统决定哪些内容可以在不触发强制更新的情况下静默推送，从而减少强制更新的触发频率，提升玩家体验连续性。
