# 自动扩缩容

## 概述

自动扩缩容（Auto Scaling）是游戏服务端架构中根据实时在线人数动态增减服务器实例数量的机制。当游戏玩家数量超过当前服务器集群的承载阈值时，系统自动启动新实例接入负载；当玩家离线导致利用率下降时，系统自动终止多余实例以节省成本。这一机制使游戏运营团队无需手动干预服务器资源，实现弹性供给。

自动扩缩容概念随云计算兴起而普及。AWS 于 2009 年正式推出 Auto Scaling 服务，此后 Google Cloud 和 Azure 相继提供类似能力。游戏行业率先大规模应用这一技术，因为游戏在线人数曲线极不规则——节假日、版本更新首日、直播带货活动期间的峰值流量可能是日常流量的 10 至 50 倍，若按峰值固定部署服务器，80% 以上的时间资源将严重浪费。

在网络多人游戏场景下，自动扩缩容尤为关键：一个爆款手游在新版本上线后的前 6 小时内，同时在线人数可能从 10 万骤升至 80 万，固定容量架构在这种冲击下要么崩溃、要么造成百万级别的冗余服务器成本。

## 核心原理

### 扩缩容触发指标

自动扩缩容的决策依赖量化指标。游戏服务端最常用的触发指标有三类：

- **CCU（Concurrent Users，同时在线人数）**：当单服 CCU 超过设定上限（例如每个房间服务器实例承载 200 人）时触发扩容；
- **CPU 利用率**：通常以 70% 作为扩容阈值、30% 作为缩容阈值；
- **消息队列深度**：游戏逻辑服的待处理消息数超过 500 条/秒时触发扩容。

扩缩容触发公式（以 CPU 利用率为例）：

```
目标实例数 = ceil( 当前实例数 × (当前平均CPU利用率 / 目标CPU利用率) )
```

例如当前 4 个实例平均 CPU 为 84%，目标 CPU 为 60%，则目标实例数 = ceil(4 × 84/60) = ceil(5.6) = 6，系统将新增 2 个实例。

### 冷却时间与扩缩容策略

游戏服务端必须为扩缩容动作配置**冷却时间（Cooldown Period）**，防止指标抖动导致实例频繁创建和销毁。一次扩容动作完成后，通常设置 120 至 300 秒的冷却窗口，期间不再触发新的扩缩决策。

扩缩容策略分为三种模式：

1. **响应式扩缩容**：指标超阈值后再扩容，延迟 1～3 分钟，游戏玩家可能在此期间遭遇排队或连接超时；
2. **预测式扩缩容**：基于历史 CCU 曲线（如每天晚上 20:00 在线人数必然上升）提前 15 分钟启动实例预热；
3. **定时扩缩容**：针对已知的活动节点（如春节首日 0 点开服）预设扩容计划，在活动开始前 30 分钟将实例数扩至峰值预估的 120%。

### 游戏服务端的缩容保护

普通 Web 服务缩容时直接终止实例即可，但游戏服务器存储着玩家的实时战斗状态，强制终止会导致掉线或存档丢失。因此游戏自动扩缩容必须实现**优雅缩容（Graceful Scale-in）**：

1. 目标实例被标记为"排水（Draining）"状态，停止接受新玩家分配；
2. 等待该实例上的现有房间内玩家全部完成对局或超时退出；
3. 实例 CCU 降至 0 后，再执行实际销毁操作。

这一流程通常配合服务器编排工具（如 Kubernetes 的 `terminationGracePeriodSeconds` 参数，游戏场景下常设为 600 秒）实现。

## 实际应用

**《英雄联盟》等对战类游戏**采用基于对局服务器（Game Server）粒度的扩缩容：每场对局对应一个独立进程，Agones（基于 Kubernetes 的游戏服务器管理框架）负责追踪空闲与占用的服务器实例，当空闲实例数量低于缓冲下限（Buffer Size，通常设为 5～10 个）时自动扩容，确保新对局请求能在 200ms 内分配到服务器。

**MMORPG 场景**（如《原神》的服务端）通常按地图分区服务器扩缩容：当某一地图区域内玩家数量超过区服承载上限（例如 500 人/分区），自动创建同地图的镜像分区并通过负载均衡将新进入玩家引流至新实例，老实例不中断，保证已在线玩家的游戏体验不受扩容影响。

**大型开服活动**场景下，运营团队会结合预测式扩缩容与定时扩缩容：提前 1 小时将最小实例数（Min Instance Count）设置为峰值预期的 80%，避免全靠响应式扩容追赶流量洪峰。

## 常见误区

**误区一：扩容越快越好，缩容速度不重要。** 实际上缩容过快在游戏场景中代价极高。如果缩容速度未受控制，正在进行对局的玩家会被强制踢出，引发大量投诉和退款请求。游戏运营商更愿意多保留 10%～20% 的冗余实例，也不接受因激进缩容导致的玩家体验损失。

**误区二：只需设置 CPU 阈值即可覆盖所有扩缩容场景。** 对于游戏逻辑服，CPU 利用率并不能准确反映实际负载。一个实例可能 CPU 仅 40% 但已承载了 300 个玩家房间（内存饱和），若只依赖 CPU 指标则不会触发扩容，导致服务响应迟缓。游戏服务端通常需要组合 CCU、内存利用率、网络带宽多维度指标进行扩缩容决策。

**误区三：自动扩缩容可以替代容量规划。** 自动扩缩容依赖云平台有充足的可用实例，在极端情况下（例如某区域突发事件导致大量游戏同时扩容）云厂商的特定实例类型可能出现配额不足。游戏团队仍需提前向云厂商申请实例配额预留（Reserved Capacity），并设置最大实例数上限防止因 Bug 或攻击触发无限扩容造成账单暴增。

## 知识关联

**前置概念——服务器编排：** 自动扩缩容的执行层依赖服务器编排系统（如 Kubernetes 或 Agones）提供实例的创建、注册与销毁接口。Kubernetes 的 Horizontal Pod Autoscaler（HPA）是自动扩缩容逻辑的具体实现载体，没有编排层，扩缩容指令无法落地为实际的容器操作。

**前置概念——负载均衡：** 扩容新增的实例必须通过负载均衡组件注册到服务发现中才能接收玩家流量；缩容时也需先从负载均衡的后端池中摘除，否则玩家仍会被路由到即将下线的实例。两者共同构成"流量分发 + 弹性供给"的完整链路。

**后续概念——多区域部署：** 单区域自动扩缩容解决了容量弹性问题，但无法应对整个区域级别的故障或海外玩家高延迟问题。多区域部署在自动扩缩容基础上进一步将扩缩容策略推广到多个地理区域，引入跨区域流量调度逻辑，是自动扩缩容在空间维度上的延伸。
