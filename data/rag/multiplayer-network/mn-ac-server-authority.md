# 服务器权威验证

## 概述

服务器权威验证（Server-Authoritative Validation）是指在多人游戏中，服务器对所有影响游戏状态的操作进行独立计算和验证，客户端提交的任何数据在未经服务器确认前不被视为有效。这意味着客户端仅充当输入来源和渲染终端，而不持有任何可信的游戏状态决定权。

这一机制的理论基础可追溯到1990年代早期的MUD（多用户地下城）服务器设计，开发者发现只要允许客户端自行汇报伤害数值或位置坐标，玩家便能通过修改本地内存轻易作弊。2000年代初期，《Quake III》等第一人称射击游戏因过度依赖客户端预测而遭遇大规模作弊问题，推动了行业对服务器权威模型的系统性采用。

服务器权威验证的重要性在于：攻击者无法仅通过修改本地客户端代码或网络数据包来影响其他玩家所见的结果。由于所有合法状态变更必须通过服务器逻辑生效，即便攻击者拦截并修改自己发出的数据包，服务器侧的验证会拒绝不符合规则的请求，从而将作弊的攻击面限制在服务器程序本身。

## 核心原理

### 请求-验证-广播流程

服务器权威验证遵循严格的单向信任流程：客户端发送"意图请求"（Intent Request），例如"我想对玩家B造成伤害"；服务器接收后，依据当前已知的权威游戏状态（服务器端快照）重新计算该操作是否合法；只有验证通过后，服务器才向所有相关客户端广播状态变更结果。客户端提交的任何中间计算结果（如伤害数值、治疗量）均被忽略，服务器自行推导最终值。

### 权威状态快照与时间戳校验

服务器维护一份不可被客户端直接写入的游戏状态快照，每帧或每个Tick（通常为16ms至64ms一个周期，对应60Hz至15Hz的服务器刷新率）更新一次。每个客户端请求必须携带客户端本地时间戳，服务器将其与服务器当前Tick进行比对。若时间差超过允许的延迟窗口（常见阈值为200ms至500ms），请求将被直接拒绝，以防止重放攻击（Replay Attack）——即攻击者将旧的合法操作包重复发送以触发多次效果。

### 可信伤害计算的实现

以射击游戏为例，服务器权威验证要求服务器独立持有所有玩家的位置、朝向、武器状态和弹药数量。客户端仅发送"在时间T按下射击键，准星朝向向量V"这类原始输入。服务器接收后，回溯至时间T对应的历史状态快照（即Lag Compensation技术中的历史帧），检查射线V是否与任何玩家碰撞体发生交叉，并按武器配置表中的伤害参数（而非客户端传入的数值）计算最终伤害。若客户端尝试传入伤害数值字段，服务器直接丢弃该字段。

### 状态不变量强制执行

服务器对每个操作检查一组业务不变量（Invariants），例如：玩家的生命值不得因单次操作降低超过武器最大伤害上限；玩家移动速度不得超过角色配置中的`maxMoveSpeed`常量；经济系统中物品栏数量不得凭空增加。任何违反不变量的请求均以错误码拒绝，并可触发反作弊日志记录。这类检查通常以断言（Assertion）形式内嵌于服务器业务逻辑，而非作为独立的"安全层"叠加，以避免遗漏新增逻辑路径。

## 实际应用

**Valve的Source引擎**在CS:GO中实现了典型的服务器权威验证：每个客户端的命令（UserCmd）包含按键位图和鼠标角度，服务器接收后在`CGameMovement::ProcessMovement()`中重新模拟物理运动，客户端预测的位置仅用于本地渲染平滑，不影响服务器的权威位置记录。

**MMO经济系统**中，《魔兽世界》采用服务器权威验证防止物品复制漏洞（Item Duplication）：所有物品转移操作（交易、邮件、拍卖行）均通过服务器端的事务性状态机执行，客户端无法直接写入角色背包数据，服务器在提交物品转移前先验证来源槽位存在该物品、目标槽位有空间，并以原子操作（Atomic Operation）完成状态更新，避免竞态条件被利用。

**移动游戏**场景下，《皇室战争》（Clash Royale）将所有单位AI计算和伤害结算放在服务器端，客户端只做动画插值，这使得针对本地内存的修改工具无法改变对战结果，因为服务器广播的才是双方客户端都显示的权威状态。

## 常见误区

**误区一：客户端预测等同于放弃服务器权威验证**。客户端预测（Client-Side Prediction）是为了掩盖网络延迟而在本地提前显示操作结果，但这并不意味着服务器放弃验证权。服务器仍然独立计算每个操作，若服务器结果与客户端预测不符，服务器结果强制覆盖客户端状态——这一机制称为"服务器纠正"（Server Reconciliation）。两者并不冲突：预测负责体验，验证负责安全。

**误区二：加密网络传输可以替代服务器权威验证**。TLS/SSL加密只能防止第三方窃听，无法阻止游戏客户端本身向服务器发送伪造的合法格式数据包。攻击者控制的是发送端，可以在加密前构造任意内容。只有服务器端对数据内容的语义合法性进行独立验证，才能抵御此类攻击。

**误区三：服务器权威验证会导致所有操作产生明显延迟**。实际上，服务器仅在需要影响他人的操作上执行强制验证，纯本地视觉效果（如技能动画启动）可无需等待服务器响应即时播放。通过将"视觉反馈"与"状态变更确认"解耦，可在保持权威验证的前提下实现低延迟游戏体验。

## 知识关联

服务器权威验证以**权威服务器模型**（Authoritative Server Model）为前提——只有当服务器被设计为唯一的状态仲裁者，验证机制才有意义；若架构采用P2P对等模式，则无法指定单一权威节点执行本文所述的验证逻辑。**经济系统保护**是服务器权威验证在货币和物品层面的直接应用，其原子事务设计依赖于服务器权威验证提供的单点写入保证。

在此基础上，**输入验证**进一步细化了服务器对原始用户输入的格式与范围检查（如按键频率上限、坐标数值范围），是服务器权威验证在输入层的具体化实现。**透视防护**则涉及服务器在发送地图实体数据前主动过滤玩家视野外的信息——这一机制同样依赖服务器拥有完整的权威世界状态作为判断依据，是服务器权威验证在信息发送侧的延伸应用。
