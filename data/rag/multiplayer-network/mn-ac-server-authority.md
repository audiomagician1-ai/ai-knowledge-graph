# 服务器权威验证

## 概述

服务器权威验证（Server-Authoritative Validation）是网络多人游戏反作弊体系的基础机制，其核心原则是：**游戏世界的最终状态由服务器决定，客户端的任何声明都必须经过服务器审核才能生效**。客户端发送的不是"我的血量变成了100"这样的状态更新，而是"我使用了一个治疗药水"这样的动作请求，服务器验证该动作是否合法后才修改权威数据。

这一机制在2000年代初期随着《反恐精英》《魔兽世界》等竞技类及大型多人在线游戏的普及而被系统化。早期游戏（如部分P2P架构的局域网游戏）允许客户端自报状态，导致"金手指"类外挂泛滥——玩家可以直接修改本地内存中的数值并广播给他人。服务器权威验证的出现从根本上切断了这条作弊路径：即使攻击者修改了本地进程中的所有数值，服务器仍然持有不受客户端干预的权威副本。

服务器权威验证之所以关键，在于它将信任边界从"信任每个客户端"转移为"只信任服务器"。这意味着哪怕客户端代码被完全逆向工程、游戏内存被任意读写，作弊者也无法单方面改变服务器的游戏状态，必须通过欺骗服务器的验证逻辑才能获利，而后者的难度远高于修改本地内存。

## 核心原理

### 请求-验证-广播循环

服务器权威验证的基本工作流遵循三步模型：

1. **客户端发送动作请求**（Action Request）：如"玩家A在坐标(120, 45)拾取了物品ID=5023"
2. **服务器独立验证合法性**：检查玩家是否真的在该坐标附近（通常允许±0.5米的网络误差容忍范围）、物品是否仍未被拾取、玩家背包是否有空间
3. **验证通过后服务器修改权威状态并广播结果**：所有客户端收到服务器确认消息后才更新本地显示

这一流程中，客户端的本地状态仅作为**预测（Prediction）**存在，用于维持流畅的视觉体验，但决不影响服务器的权威数据。

### 关键数值的服务端托管

服务器权威验证要求将所有"有作弊价值"的数值存储在服务器内存中，而非信任客户端上报。具体包括：

- **玩家属性**：HP、MP、等级、经验值、坐标位置
- **物品与货币**：背包内容、金币数量（这与经济系统保护直接关联）
- **技能冷却时间（Cooldown）**：服务器记录每个技能的上次使用时间戳，若客户端在冷却结束前再次请求使用，服务器拒绝该请求并返回错误码

以冷却验证为例，伪代码逻辑如下：

```
function validateSkillUse(playerId, skillId, clientTimestamp):
    lastUsed = serverDB.getLastUsedTime(playerId, skillId)
    cooldown = skillConfig[skillId].cooldownMs  // 例如：3000毫秒
    if (serverTime - lastUsed) < cooldown:
        return REJECT("冷却未结束")
    serverDB.setLastUsedTime(playerId, skillId, serverTime)
    return APPROVE
```

注意这里使用的是**服务器时间（serverTime）**而非客户端上报的时间戳，这是防止时间篡改攻击的关键细节。

### 速率限制与异常阈值

服务器权威验证还包含对请求频率的审查。例如，一名玩家正常移动速度上限为7米/秒，若服务器在500毫秒内收到两条位置更新消息，玩家声称移动了6米（等效速度12米/秒），服务器会拒绝该位置并将玩家传回上一个合法位置，同时记录一条速度违规日志。这一机制直接针对"加速外挂（Speed Hack）"。

## 实际应用

**《英雄联盟》伤害计算**：Riot Games的服务器在每次技能命中时独立计算伤害数值，所需参数包括攻击者的攻击力、防御穿透力、目标的护甲值——这些数值全部来自服务器权威副本，客户端发送的伤害请求不包含伤害数值本身，只包含"技能命中目标"的事件描述。

**《魔兽世界》拍卖行保护**：玩家在客户端点击"购买"后，服务器验证：①玩家金币是否足够；②拍卖物品是否仍上架（防止竞争条件下的重复购买）；③交易双方是否在同一服务器分区。三项验证全部通过才执行交易，任何一项失败均不扣款不转移物品。

**FPS游戏弹药管理**：射击游戏中，服务器为每名玩家维护权威弹药计数器。每次客户端发送"开枪"请求，服务器将弹药减1，若弹药为0时收到开枪请求，服务器拒绝该请求（无限子弹外挂因此无效）。《Valorant》的服务器端射击验证系统在其2020年上线时曾公开披露，服务器以每秒128次的Tick率处理此类验证。

## 常见误区

**误区一：客户端预测等于客户端权威**

许多开发者初次接触客户端预测（Client-Side Prediction）时，误以为允许客户端"自行决定"移动结果。实际上，预测只是本地的视觉优化，服务器每帧（或每Tick）仍会下发权威位置，若与客户端预测不符则触发**回滚（Rollback）**，将客户端状态强制纠正。客户端永远不能"告诉"服务器它的位置，只能告诉服务器它的输入指令。

**误区二：只验证敏感操作即可**

部分项目为减少服务器负载，仅对货币交易、等级提升等"高价值"操作做服务器验证，而对移动、普通攻击等高频操作信任客户端。这一妥协会留下严重漏洞：攻击者可以通过刷普通攻击频率绕过冷却时间限制，或通过位置操控接触到本不应接触的游戏内容。服务器权威验证的有效性依赖于覆盖所有有逻辑意义的操作，而不仅是"看起来重要"的操作。

**误区三：服务器验证等于防止所有作弊**

服务器权威验证能阻止改变服务器状态的作弊行为（如无限金币、无限血量），但对纯粹影响客户端感知的作弊行为无能为力，例如透视外挂（ESP）通过读取客户端已有的渲染数据获取敌方位置，服务器验证机制对此没有直接防护效果——这正是为什么透视防护需要作为独立机制在服务器权威验证之上单独实现。

## 知识关联

服务器权威验证建立在**权威服务器模型**的架构基础之上——只有在服务器持有唯一权威状态的前提下，验证机制才有意义；若采用P2P架构，"服务器"这一角色不存在，验证无法集中执行。同时，**经济系统保护**是服务器权威验证应用最密集的领域，货币数量、物品所有权等经济数据是最常见的作弊目标，其保护完全依赖服务器拒绝非法状态更新。

在服务器权威验证确立之后，**输入验证**作为其自然延伸，进一步规范了客户端向服务器发送的请求格式与内容——确保请求本身不包含非法参数（如负数金额、越界物品ID）。而**透视防护**则解决了服务器权威验证无法覆盖的纯客户端信息泄露问题，两者共同构成完整的反作弊防御层次。
