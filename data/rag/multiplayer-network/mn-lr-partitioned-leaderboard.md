# 分区排行榜

## 概述

分区排行榜（Partitioned Leaderboard）是将全局玩家数据按照特定维度切割成多个子集，每个子集独立排序、独立展示的排行榜系统。常见的分区维度包括地理区域（亚服、欧服、美服）、竞技段位（青铜/白银/黄金/铂金梯队）、社交关系（好友圈）、服务器分区，以及时间窗口（周榜、月榜）。与全局单一排行榜不同，分区排行榜的核心设计目标是让玩家在"可竞争"的范围内看到自己的相对位置，而非被数亿全球玩家的数据淹没。

分区排行榜的大规模实践最早出现在2003年前后的韩国MMO游戏中，当时《天堂II》等游戏因跨大陆延迟问题被迫按服务器物理分区，意外发现玩家在小范围内的活跃度和留存率显著高于统一全球榜时期。此后，好友圈排行榜随着Facebook平台游戏（2007-2012年黄金期）普及，成为休闲游戏的标配留存机制。段位分区排行榜则随《英雄联盟》在2012年推出Elo段位系统后被竞技游戏广泛借鉴。

分区排行榜的核心价值在于"心理可及性"——研究数据表明，当玩家在榜单中的可见排名位于前20%以内时，其次日留存率比排名在后50%的玩家高出约15-25个百分点。好友圈分区榜的效果尤为突出：看到"你比好友张三高37分"比看到"全球第1,847,293名"对玩家的激励效果强得多。

## 核心原理

### 分区键（Partition Key）的设计

分区排行榜的技术核心是为每个玩家分配一个或多个分区键，并以分区键作为数据存储和查询的基本单位。以Redis实现为例，一个典型的分区键格式为 `leaderboard:{game_id}:{partition_type}:{partition_value}:{time_window}`，例如 `leaderboard:LOL:region:KR:week_2024_03`。每个分区键对应一个独立的有序集合（Sorted Set），玩家分数通过 `ZADD` 写入，排名通过 `ZREVRANK` 查询，时间复杂度均为 O(log N)，其中 N 是该分区内的玩家数量。

分区粒度决定了榜单竞争密度。设分区内玩家总数为 P，则榜单前1%包含 P×0.01 个玩家。当全服100万玩家按10个地区分区后，每区约10万人，前1%约1000人；若再叠加5个段位层，每个最小分区约2万人，前1%仅约200人。过细的分区会导致榜单稀疏（玩家无法感受到竞争），过粗则失去分区意义，工程上通常将单分区活跃玩家数控制在1万到10万之间。

### 多维分区的数据一致性

当一个玩家的分数更新时，需要同时更新该玩家所属的所有分区排行榜。假设一名玩家同时参与：全局榜、地区榜、段位榜、好友榜，一次得分事件可能触发4-8个 Sorted Set 的写操作。好友圈分区榜的挑战尤为特殊——由于每个玩家的好友圈构成不同，好友榜实质上是一种**动态分区**，无法预先固化分区边界。常见的解决方案是采用"拉取合并"策略：不预先维护好友榜的完整 Sorted Set，而是在玩家请求时，实时拉取其所有好友的分数列表，在应用层做内存排序后返回。当好友数量超过500时，该策略的延迟会超过100ms，需要引入缓存或预计算优化。

### 跨分区的玩家晋升与归属变更

段位分区排行榜面临一个独特问题：玩家的分区归属会随游戏内行为动态变化。当玩家从黄金段位晋升到铂金段位时，必须从黄金榜中移除、加入铂金榜，同时其历史排名记录需要保留供成就系统引用。这个操作需要保证原子性，通常通过 Redis 的 `MULTI/EXEC` 事务或 Lua 脚本实现，确保玩家不会在短暂的中间状态下同时出现在两个段位榜上，或两个榜都不存在。地区分区的变更相对罕见（玩家换区），但需要处理跨数据中心的数据迁移问题。

## 实际应用

**《王者荣耀》的五维分区体系**将玩家分布在：全国榜、省级榜（34个省级行政区）、好友榜、英雄专项榜（按英雄使用率）、段位榜五个维度。其中省级榜和好友榜的刷新频率为每10分钟一次，全国榜每小时刷新，这种差异化刷新策略在保持实时感的同时降低了计算峰值压力。

**《Clash Royale》的联赛分区**采用"池化"机制：每个赛季开始时，将同段位玩家随机分配到约50人规模的小联赛池中，独立竞争。这种人工构建的小分区确保了每个池中必然有排名1-3的获胜玩家，从而保证奖励发放的稳定性，同时规避了因全局竞争过于激烈导致低活跃玩家完全躺平的问题。

**《Pokémon GO》的好友分区榜**在2019年社交功能上线后，采用了预计算好友积分快照的方案：每日凌晨2:00对所有玩家的好友关系图做一次全量快照，在快照基础上预计算并缓存好友榜数据，玩家白天查看的是缓存结果，当日新增好友关系不实时反映在榜单中。这一取舍将好友榜查询延迟控制在5ms以内，代价是数据存在最长23小时的滞后。

## 常见误区

**误区一：认为好友圈分区榜与全局榜的数据来源相互独立**。实际上，好友榜展示的是全局数据的一个动态子集视图，好友的分数数据仍然存储在全局的统一数据源中（如全局 Sorted Set 或关系型数据库），好友榜只是在查询时做了基于社交关系的筛选过滤。如果将好友榜实现为独立的数据副本，会带来严重的数据一致性问题——当一个玩家的分数更新时，需要找到所有将其加为好友的玩家，逐一更新其好友榜副本，这在大型游戏中是不可行的。

**误区二：段位分区排行榜等同于段位系统本身**。段位系统（Ranking System）是决定玩家竞技水平归类的算法，如Elo、MMR、TrueSkill；段位分区排行榜是以段位作为分区键组织展示数据的榜单功能。前者解决"玩家应该与谁配对"的问题，后者解决"玩家在同段位中排第几"的问题。铂金段位内的排行榜只有在铂金玩家数量足够多（通常建议至少5000名活跃玩家）时才有统计意义，这与段位划分本身的设计目标不同。

**误区三：分区越细，玩家体验越好**。过度细分会导致"虚假优越感"问题：当一个分区只有200名玩家时，很多中等水平玩家会长期占据前三名，失去继续提升的动力。《魔兽争霸3》天梯系统曾在2004年因分区过细（按地区×模式×地图三维交叉分区）导致部分小众地图分区长期只有个位数玩家，最终不得不合并分区。

## 知识关联

分区排行榜建立在基础排行榜设计的技术之上，但引入了分区键管理、多 Sorted Set 同步写入、以及动态分区（好友榜）等新的工程问题。掌握 Redis Sorted Set 的 `ZADD`/`ZREVRANK`/`ZRANGEBYSCORE` 操作是实现分区排行榜的基础技能，其时间复杂度分析直接影响分区粒度的工程决策。

在设计选型上，分区排行榜与**匹配系统（Matchmaking）**存在深度耦合：段位分区排行榜的分区边界往往与匹配池的边界一致，段位榜中玩家数量分布的均衡性直接影响匹配质量。好友圈排行榜则与**社交系统**紧密相连，好友关系数据的存储方案（邻接表 vs. 倒排索引）直接决定好友榜的可行实现方式。在实际游戏后端架构中，分区排行榜通常作为数据聚合层，消费来自战斗结算服务、任务系统等上游服务产生的玩家分数事件。
