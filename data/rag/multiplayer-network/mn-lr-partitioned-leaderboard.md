# 分区排行榜

## 概述

分区排行榜是将玩家群体按照特定维度（地理区域、竞技段位、好友社交圈等）切分成子集后，在每个子集内独立计算名次的排名系统。与单一全局排行榜不同，分区排行榜允许同一名玩家在不同分区中拥有不同名次——例如一名玩家可以在全球榜第 8,000 位，但在所属的地区榜（如"亚洲东区"）排名前 50。

这一机制最早在 2004 年前后随着 Xbox Live 和 Battle.net 的大规模商业化而普及。早期游戏仅有全球单一排行榜，但顶级玩家长期霸占前列导致普通玩家失去竞争动力，这一现象被业界称为"精英压制效应"（Elite Suppression Effect）。分区排行榜通过将玩家限制在更小的竞争池中，使得更大比例的玩家能看到自己处于有意义的名次范围内。

分区排行榜在技术实现上的核心挑战是：如何在保证各分区数据实时性的同时控制存储与查询成本。每个分区本质上是一个独立的有序集合，分区数量越多，系统维护的索引数量就越大，读写放大（Write Amplification）问题越突出。

## 核心原理

### 分区维度的选择

分区维度决定了玩家被分配到哪个"竞争池"。常见的三种维度各有计算逻辑：

- **地理区域分区**：依据玩家账号注册地或 IP 归属地划分，通常按大陆（如北美、欧洲、亚太）或国家建立子榜。《英雄联盟》的大区制度（LAN、EUW、KR 等）是典型实现，每个大区维护独立的积分排名。
- **段位分区**：按玩家当前竞技等级分层，例如《守望先锋》将玩家分为青铜/白银/黄金/铂金/钻石/大师/宗师七个段位，每段内部独立排名。段位分区的本质是通过 MMR（Match Making Rating）阈值切割连续数值空间为离散区间。
- **好友圈分区**：只展示玩家社交关系网络中的成员排名，竞争池大小等于好友数量（通常上限为 200-500 人）。Steam 好友排行榜、《糖果传奇》的 Facebook 好友榜均属此类。

### 数据存储结构

分区排行榜通常使用 **Redis Sorted Set** 作为底层数据结构，其 `ZADD` / `ZRANK` / `ZRANGE` 命令的时间复杂度分别为 O(log N)，N 为该分区内玩家数量。每个分区对应一个独立的 key，例如：

```
leaderboard:region:asia_east   → Sorted Set（按分数降序）
leaderboard:tier:diamond       → Sorted Set
leaderboard:friends:user_10234 → Sorted Set（动态构建）
```

好友圈分区是三种类型中维护成本最高的，因为它是**动态个性化分区**：每名玩家的好友列表不同，导致无法预先计算固定的排行榜，通常需要在查询时实时合并多个玩家的分数记录，或通过预计算缓存来平衡延迟。

### 名次归一化与跨区比较

由于不同分区的竞争激烈程度不同（亚洲区玩家密度远高于大洋洲区），单纯的绝对名次缺乏跨区可比性。百分位排名（Percentile Rank）是解决这一问题的标准方法：

> **百分位排名 = (分区内低于该玩家分数的人数 / 分区总人数) × 100%**

例如玩家在亚洲区 10 万人中排第 2,000 名，百分位排名为 98%；在大洋洲区 5,000 人中排第 100 名，百分位排名同样为 98%。两者竞争力等价。《Apex 英雄》的段位系统即使用百分位来校准不同平台（PC/主机）之间的段位边界。

## 实际应用

**《炉石传说》天梯赛季分区**：游戏按月赛季重置段位榜，同时维护地区榜（亚洲、美洲、欧洲）与传说级全球榜。只有到达传说段位的玩家才会进入跨区的全球前 1,000 实时榜，其余玩家始终在本区段位榜内竞争。这种"地区段位榜 + 全球精英榜"的双层结构兼顾了大众参与感与顶级竞技曝光度。

**《原神》多维度分区**：深境螺旋排行榜按服务器（国服/国际服/港澳台服）分区，同时在每个服务器内按通关时间排名。这里的分区边界与支付系统和内容版本更新节奏强绑定——不同服务器版本进度不同，跨服合并排名在技术和公平性上均不可行。

**移动端休闲游戏的好友圈榜**：《部落冲突》的好友捐兵积分榜将竞争范围限定在同一部落（Clan，上限 50 人）内，利用社交压力驱动玩家更频繁登录。这一设计使日活跃用户的次日留存率相比无好友榜版本提升约 12-18 个百分点（Supercell 公开数据）。

## 常见误区

**误区一：分区越细越好**
将玩家切分成过多小分区会导致每个分区内人数稀少，名次波动剧烈——一场胜负就可能使排名变化数十位，排名丧失稳定性与参考价值。实践中，单个地理分区建议维持至少 **1,000 名活跃玩家**才能使排名具有统计意义；段位分区中顶级段位（如大师/宗师）通常限制在全服前 0.1%-1% 以维持稀缺感与竞争密度。

**误区二：好友圈榜可以用全局榜过滤替代**
有些开发者认为只需在全局排行榜查询时加入"是否好友"的过滤条件即可实现好友榜，无需额外数据结构。但这种方式需要全表扫描，当全局玩家量达到百万级时，单次查询延迟将超过 500ms，远超排行榜 <100ms 的响应时间要求。正确做法是为每名玩家维护独立的好友分数集合，在好友关系变更时异步更新。

**误区三：段位分区排名等同于竞技实力评估**
段位分区榜只反映玩家在当前段位内的相对位置，而段位本身是通过晋级赛/积分阈值动态变动的。一名黄金段第 1 名的实际 MMR 可能低于铂金段末位玩家，两者的绝对排名不可直接比较，必须通过前述百分位方法或直接比较 MMR 原始值才能得出有意义的跨段位比较结论。

## 知识关联

分区排行榜以**排行榜设计**的基础概念为前提——包括分数存储格式、更新频率（实时榜 vs. 快照榜）、防刷分机制等。在理解全局排行榜的读写权衡之后，分区排行榜将这些问题乘以分区数量 N，使系统设计复杂度从 O(1) 套逻辑升级为需要处理分区路由、跨区数据一致性和动态分区（好友圈）三类新问题。

在游戏系统设计的更广泛语境中，分区排行榜与**赛季重置机制**紧密配合——赛季结束时通常对段位分区进行"软重置"（玩家 MMR 向中位数收缩），以防止高分玩家在段位榜中长期固化。此外，分区排行榜产生的分区间玩家分布数据，是游戏运营团队调整段位阈值和匹配参数的重要数据来源，形成"数据采集 → 参数调整 → 再分区"的闭环运营逻辑。
