# 加载同步

## 概述

加载同步（Loading Synchronization）是多人游戏大厅系统中的一种等待机制，其核心目标是确保所有玩家在同一时间点进入游戏场景。简而言之，当房间内任意一名玩家尚未完成地图或资源加载时，已完成加载的玩家需在加载屏幕上继续等待，直到房间内全体玩家均发出"加载完成"信号后，服务器才下发统一的"开始游戏"指令。

这一机制最早在局域网对战时代（1990年代末期的《星际争霸》等RTS游戏中）就已出现雏形，彼时称为"等待玩家（Waiting for Players）"对话框。进入宽带互联网时代后，不同玩家网络环境与硬件性能差异更加悬殊，加载同步逐渐演变为一套完整的握手协议，并成为网络多人游戏大厅系统设计的标准组成环节。

加载同步之所以不可或缺，是因为多人游戏的初始帧（Frame 0）必须在所有客户端同时执行，否则会导致游戏逻辑状态从第一帧起便产生分歧。即便游戏采用服务器权威（Server-Authoritative）架构，若某客户端在其他人已开始运行游戏逻辑时仍未完成场景初始化，该客户端将无法正确响应服务器下发的第一批游戏状态包，必然触发重同步（Resync）甚至断线。

## 核心原理

### 就绪状态上报与聚合

每个客户端在本地完成资源加载后，向服务器发送一条 **LoadComplete** 消息，消息体中通常携带：客户端ID、加载耗时（毫秒）、资源校验哈希（防止客户端使用不同版本资源包）。服务器维护一张就绪表（Ready Table），每收到一条 LoadComplete 消息就将对应玩家标记为 `Ready = true`。当就绪表中所有玩家条目均为 `true` 时，服务器广播 **GameStart** 指令。

就绪条件的判断公式可以简洁地表达为：

```
AllReady = (ReadyCount == ExpectedPlayerCount)
           AND (All checksum match)
```

其中 `ExpectedPlayerCount` 取自房间创建时锁定的玩家名单，而非实时在线人数，这可以防止因网络抖动导致人数短暂变化而误触发开始。

### 超时回退策略（Timeout Fallback）

纯等待机制存在一个关键问题：若某玩家硬盘过慢或卡死，会无限阻塞所有人。因此加载同步必须配合超时回退策略。业界常见做法是设置两段超时阈值：

- **软超时（Soft Timeout）**：通常为 60 秒，触发后向房间内所有玩家展示"等待某某玩家加载中（X%）"的进度提示，并允许房主手动踢人。
- **硬超时（Hard Timeout）**：通常为 120 秒，触发后服务器自动将未就绪玩家标记为断线，将其移出就绪表，然后以剩余已就绪玩家重新执行 AllReady 判断并启动游戏。

两段阈值的具体数值因游戏类型而异：重资产的开放世界游戏（如《命运2》）硬超时可延长至 180 秒，而轻量级休闲对战游戏通常压缩至 45/90 秒。

### 加载进度广播

为改善等待体验，许多实现方案在加载同步期间以固定间隔（如每 500ms 一次）向服务器上报加载进度百分比，服务器再将所有人的进度广播给房间内所有客户端，用于渲染"最慢玩家进度条"UI。该广播频率不宜过高（不应低于 200ms 间隔），否则在大房间（32人以上）中会产生不必要的服务器轮询压力。

## 实际应用

**《英雄联盟》的加载界面**是加载同步的典型实现案例。该游戏在加载屏幕中显示每位玩家的加载百分比，并以每秒约2次的频率刷新。系统在所有10名玩家均达到100%后，才下发进入游戏指令，并额外等待约1~2秒作为缓冲，确保网络传输延迟不导致某些玩家在GameStart到达之前仍处于渲染初始化阶段。

**房间人数变化的边界处理**也是实际开发中必须应对的场景。例如，房间共8人，7人已完成加载，第8人在加载过程中网络断线——此时其 LoadComplete 消息永远不会到达。服务器必须依赖心跳包（Heartbeat）检测到该玩家离线，将其从就绪表中移除，并重新触发 AllReady 判断，而不是永久卡在等待状态。

**离线模式的降级处理**：单机或私人房间场景下，若房间仅有1名玩家，加载同步可直接跳过聚合步骤，客户端本地加载完成后立即自行触发游戏开始，无需等待服务器确认，以减少不必要的网络往返延迟（RTT）。

## 常见误区

**误区一：加载完成后客户端可以自行进入游戏**

部分开发者认为，既然本地加载已完成，客户端可以不等待服务器的 GameStart 指令，直接切换到游戏场景以节省时间。这会导致该客户端的游戏逻辑从一个与其他客户端不对齐的时间点开始执行，若游戏使用确定性模拟（Deterministic Simulation），从第一帧起的输入序列将无法对齐，最终产生不可逆的状态分叉（Desync）。正确做法是客户端加载完成后必须保持在等待界面，直到收到服务器下发的统一 GameStart 信号。

**误区二：加载同步只需要等待最慢的一个人**

实际上，加载同步等待的不仅是"加载速度最慢"的玩家，还包括加载中途因网络原因导致资源下载卡顿的玩家，以及资源校验哈希不匹配（即客户端版本与房间版本不一致）而被阻塞的玩家。后者必须触发错误提示并将该玩家踢出，而非无限等待其哈希值变为正确值——这是一个逻辑错误，不是加载速度问题。

**误区三：超时后直接解散房间是最安全的做法**

部分实现选择在硬超时后直接解散整个房间，让所有人重新匹配。这对于已完成加载并等待了120秒的玩家体验极差。正确做法是仅移除超时未就绪的玩家，以剩余人数（若满足游戏最低人数要求）继续开始游戏，并在房间内广播移除原因（"玩家A加载超时，已离开房间"）。

## 知识关联

加载同步依赖**房间创建**阶段所锁定的玩家名单作为就绪表的初始数据来源。房间创建时确定的 `ExpectedPlayerCount` 和玩家ID列表，是服务器判断 AllReady 条件的基准——若房间创建阶段的名单不准确，加载同步的等待逻辑将无从正确终止。

加载同步在大厅系统中处于"游戏正式开始前"的最后一道屏障，其完成标志着大厅系统生命周期的结束和游戏会话（Game Session）生命周期的开始。与房间创建、匹配、准备确认等前序环节共同构成玩家从点击"开始游戏"到实际进入战斗的完整流程链路。对加载同步机制的调优（尤其是超时阈值与进度广播频率）直接影响玩家感知到的"进入游戏"速度，是大厅系统用户体验优化的重要量化指标之一。
