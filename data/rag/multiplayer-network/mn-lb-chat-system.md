# 聊天系统

## 概述

聊天系统是网络多人游戏大厅中负责玩家实时文字与语音通信的功能模块，其核心任务是将一条消息从发送端可靠地传递至一个或多个接收端，同时维护频道隔离、历史记录与过滤规则。与普通即时通讯软件不同，游戏聊天系统对延迟极为敏感——玩家在大厅组队时，超过500毫秒的消息延迟就会明显影响协作体验，因此系统设计必须在传输效率与功能丰富度之间取得平衡。

聊天系统的雏形出现在1990年代早期的MUD（Multi-User Dungeon）游戏中，当时仅支持单一全局频道的纯文字广播。随着《魔兽世界》（2004年）引入分层频道体系（综合、交易、本地防御、公会、队伍等），现代游戏聊天系统的频道设计范式逐渐成熟。今天的系统通常同时处理文字消息、语音流、表情包和系统通知四类不同数据流，每类数据对带宽、延迟和可靠性的要求各不相同。

理解聊天系统的价值在于，它直接影响游戏社区的形成质量。研究表明，拥有完善公会/频道聊天功能的游戏，玩家30日留存率平均比无聊天功能的同类游戏高出23%。同时，聊天系统也是作弊检测、违规举报和客服介入的第一线入口，其架构设计决定了这些运营能力能否有效落地。

## 核心原理

### 频道模型与订阅机制

游戏大厅的聊天系统通常采用**发布-订阅（Pub/Sub）**模型管理频道。每个频道是一个命名主题（Topic），服务器维护一张映射表：`Channel_ID → [Socket_ID列表]`。玩家进入大厅时自动订阅"大厅全局"频道；组队后额外订阅"队伍"频道；加入公会则订阅"公会"频道。一条消息被发布到某频道后，服务器遍历该频道的订阅列表，将消息推送至每个在线成员的连接。

频道之间必须严格隔离：队伍内部的战术讨论不应泄漏至大厅全局，交易频道的刷屏不应污染公会频道。实现隔离的关键是为每条消息打上`channel_id`标签，客户端根据当前激活频道过滤显示，而非在服务端阻止跨频道投递——后者实现复杂且影响性能。

### 消息结构与传输协议

一条标准文字聊天消息通常包含以下字段：

```
{
  msg_id:    uint64,   // 全局唯一消息ID，用于去重
  sender_id: uint64,   // 发送者账号ID
  channel_id: uint32,  // 目标频道
  timestamp: int64,    // Unix毫秒时间戳
  content:   string,   // UTF-8编码文字内容，上限通常为256字节
  msg_type:  uint8     // 0=文字, 1=系统通知, 2=表情
}
```

传输层选择上，文字聊天普遍使用**WebSocket**或**TCP长连接**而非UDP，原因是文字消息需要保序且不可丢失——丢失一条"全体集合"指令会导致严重误解。语音数据则相反，通常走UDP或WebRTC，因为单个音频帧丢失只影响50毫秒的音质，强行重传反而会破坏实时性。

### 消息限流与防刷屏

游戏聊天系统必须实现服务端限流（Rate Limiting），通常采用**令牌桶算法**：每个玩家账号每秒补充N个令牌（典型值N=3），发送一条消息消耗1个令牌，令牌上限为10。当令牌耗尽时，服务端拒绝消息并返回错误码，而非静默丢弃——静默丢弃会让发送者误以为消息已送达。全局频道（面向几千名玩家广播）的限流阈值通常比队伍频道（面向5人）更严格，常见配置为全局频道每5秒1条，队伍频道每秒3条。

服务端还需过滤违禁词。过滤器通常使用**Aho-Corasick算法**构建多模式匹配自动机，可在O(n)时间内对消息文本完成全词库扫描，其中n为消息长度，与违禁词数量无关，这对需要毫秒级响应的聊天系统尤为关键。

## 实际应用

**《英雄联盟》的频道架构**：该游戏将大厅聊天分为"英雄选择"频道（仅5名队友可见）和赛后"全局"频道（10名玩家均可见）。英雄选择阶段对己方禁用全局频道，防止策略泄露——这是通过服务端在`channel_id`层面隔离订阅列表实现的，而非客户端隐藏UI。

**跨服务器大厅聊天**：当游戏运行多个物理服务器分区时，玩家A在分区1，玩家B在分区2，同一公会频道的消息需要跨分区投递。常见解决方案是引入一个独立的**聊天网关服务**，各分区服务器将发往公会/全局频道的消息转发至网关，由网关统一广播至所有订阅了该频道的分区连接。《最终幻想XIV》的跨大区聊天即采用类似架构，支持超过10万人同时在线时的全服公告广播。

**历史记录的存储策略**：队伍聊天记录通常只保留内存中最近50条，房间解散后丢弃，无需持久化。公会聊天记录则写入数据库，保留7至30天，供离线玩家重新上线后回溯。这种差异化存储策略将数据库写入量减少了约80%，因为队伍消息量远大于公会消息量。

## 常见误区

**误区一：客户端过滤等同于服务端过滤**
许多初学者认为，在客户端屏蔽违禁词就足够了——"反正玩家自己看不见"。这是错误的。客户端过滤极易被修改版客户端绕过，违禁内容仍会在服务器日志和其他客户端上原文出现。违禁词过滤必须在**服务端消息入队前**执行，拦截后记录日志，客户端过滤只是额外的显示层保护。

**误区二：聊天系统可以直接复用游戏状态同步的UDP通道**
游戏状态（位置、技能）走UDP是因为允许丢包，旧状态会被新状态覆盖。文字聊天不具备这种自愈能力，"开团"指令被UDP丢包且不重传，将导致严重的游戏体验问题。混用传输层会引入不必要的可靠性实现负担，正确做法是为聊天系统维护独立的TCP/WebSocket连接，与游戏逻辑UDP通道并行。

**误区三：全局广播可以用单线程循环遍历推送**
当大厅有5000名玩家订阅全局频道时，单线程遍历推送5000个socket写操作会产生严重的线程阻塞。现代聊天服务器使用异步IO（如epoll或io_uring）或将广播任务分发至线程池，目标是将单条广播消息的端到端推送时间控制在200毫秒以内，无论频道订阅人数多少。

## 知识关联

聊天系统是学习**语音聊天**的前置基础——语音聊天在频道模型上与文字聊天完全共享同一套Pub/Sub架构，区别仅在于数据流从离散文字包变为连续音频流，传输协议从TCP切换至UDP/WebRTC，这个切换背后的原因需要先理解文字聊天为何必须用TCP。

**屏蔽与静音**功能建立在聊天系统的订阅层之上：屏蔽某玩家本质是在服务端或客户端为该`sender_id`添加过滤规则，使其消息不再路由到当前玩家。理解聊天消息的`sender_id`字段设计，是实现屏蔽功能的直接基础。

**消息队列**是聊天系统在高并发场景下的扩展方案：当聊天服务器收到消息后，不立即推送给所有订阅者，而是先写入Kafka或RabbitMQ等消息队列，由独立的推送工作进程消费队列并分发。这种解耦设计使聊天系统能横向扩展，支撑百万级同时在线——这是在理解聊天系统基础架构后，自然需要探索的下一层扩展问题。
