# 聊天系统

## 概述

聊天系统是网络多人游戏大厅中负责玩家实时文字与语音交流的基础设施，其核心职责是将一名玩家发出的消息，以低于200毫秒的感知延迟，准确投递给目标频道内的所有订阅者。与单人游戏内的聊天不同，大厅聊天系统需要在玩家尚未进入游戏房间之前就持续运转，因此它必须与游戏进程解耦，作为独立的长连接服务存在。

聊天系统在网络游戏中的大规模商业应用可追溯至1997年的《网络创世纪》（Ultima Online），该作首次引入了以`/`斜杠命令区分频道的机制。此后IRC（Internet Relay Chat，互联网中继聊天）协议的频道概念被广泛借鉴，形成了今天大厅聊天系统"全服频道 → 房间频道 → 私聊"这一三级频道层次结构的工业标准原型。

理解聊天系统的意义在于：它是大厅系统中并发订阅者数量最多、消息吞吐量最高的单一模块。一款拥有10万同时在线玩家的游戏，其全服频道每秒可产生数百条消息，每条消息需扇出投递至数万个连接，这使得聊天系统的扩展性设计直接决定了大厅后端的服务器成本与稳定性上限。

## 核心原理

### 频道模型与订阅机制

聊天系统以**频道（Channel）**为基本路由单元。每个频道维护一个订阅者列表（Subscriber List），玩家进入大厅时自动订阅全服频道（World Channel），加入房间时订阅房间频道（Room Channel），发起私聊时创建一对一的私信频道（Whisper Channel）。

频道的核心数据结构如下：

```
Channel {
  channel_id: uint64        // 唯一频道ID
  type: enum(WORLD, ROOM, WHISPER, GUILD)
  subscriber_set: Set<player_id>
  message_history: RingBuffer<Message>[50]  // 最近50条历史消息
}
```

消息投递遵循**发布-订阅（Pub/Sub）模式**：发送方只需将消息发布至频道，服务器负责将其推送给该频道内所有在线订阅者，发送方无需知道接收方的具体连接地址。这一设计使得玩家数量增加时，发送方的代码逻辑无需任何修改。

### 消息路由与扇出控制

全服频道的**扇出（Fan-out）**是聊天系统最大的性能挑战。若10万玩家同时订阅全服频道，单条消息需向10万个TCP连接推送数据包，单台服务器无法承担此负载。工业界通用解法是**分片扇出**：

1. 将全服频道按地理或哈希分成N个子分片（Shard），每个分片最多承载5000名订阅者。
2. 消息先发布至分片间的消息中间件（如Redis Pub/Sub或Kafka Topic）。
3. 各分片服务器独立从中间件消费消息，再向本分片内的订阅者推送。

这样单条全服消息的实际推送压力从"单点10万次写操作"降低为"N个分片各自5000次写操作"，每个分片服务器的压力是可控且可横向扩展的。

### 文字消息的传输协议

文字聊天通常通过**WebSocket**或游戏自定义的TCP长连接承载。消息体采用紧凑的二进制编码（如Protocol Buffers）以减少带宽消耗：

```protobuf
message ChatMessage {
  uint64 sender_id = 1;
  uint64 channel_id = 2;
  string content = 3;        // UTF-8编码，最大长度256字节
  int64  timestamp_ms = 4;   // 服务端盖戳的Unix毫秒时间戳
  uint32 sequence_num = 5;   // 频道内的消息序号，用于客户端去重
}
```

`sequence_num`字段至关重要：客户端重连后可通过发送自身持有的最后序号，请求服务端补发断线期间错过的消息（最多补发历史缓冲区内的50条），从而避免断线重连后的消息丢失问题。

### 发言频率限制（Rate Limiting）

若不加限制，恶意玩家可通过高频发消息淹没聊天频道（即"刷屏"攻击）。标准做法是在服务端对每个`player_id`实施**令牌桶（Token Bucket）**限速：

- 全服频道：每10秒补充1个令牌，桶容量为2（即最多允许短暂连发2条后进入冷却）。
- 房间频道：每2秒补充1个令牌，桶容量为5。
- 私聊频道：每1秒补充1个令牌，桶容量为10。

超出令牌的消息在服务端直接丢弃，并向发送方返回`RATE_LIMIT_EXCEEDED`错误码，客户端据此在UI上显示"发言过于频繁"的提示。

## 实际应用

**《英雄联盟》大厅聊天**采用了与游戏服务器完全分离的独立聊天集群，基于XMPP协议（Extensible Messaging and Presence Protocol）构建。每个玩家的好友列表即一个私信频道集合，匹配大厅中的队伍聊天则是容量为5人的临时房间频道，游戏结束后该频道自动销毁。

**《Dota 2》**的大厅聊天系统区分了"大厅频道"与"游戏内频道"两套独立管道。玩家在大厅发送的消息不会出现在游戏中，防止了赛前聊天记录污染游戏内通讯视图。这种频道隔离设计已成为竞技游戏的行业最佳实践。

**消息历史回放**是实际开发中容易忽略的需求：玩家打开聊天窗口时，应立即看到频道内最近的50条历史消息，而非等待新消息到来才看到内容。服务端为此需要为每个频道维护一个环形缓冲区（Ring Buffer），新消息入队时自动覆盖最旧的记录。

## 常见误区

**误区一：用HTTP轮询替代长连接实现聊天**

部分初学者会让客户端每隔1秒向服务器发一次HTTP请求询问是否有新消息（短轮询）。在1000名玩家同时在线的场景下，服务器每秒需处理1000次无效的空响应请求。正确做法是使用WebSocket或SSE（Server-Sent Events）建立持久连接，服务端在有消息时主动推送，将服务端请求处理量从O(玩家数×轮询频率)降低至O(新消息数)。

**误区二：由发送方客户端负责时间戳**

若使用客户端本地时间作为消息的`timestamp`，因各玩家设备时钟存在误差（实测偏差可超过30秒），同一频道内的消息在不同客户端显示时会出现乱序。正确做法是消息到达服务端后，由服务端统一盖戳，客户端收到消息后以服务端时间戳为准进行排序显示。

**误区三：全服频道对所有玩家完全平等地推送**

新手开发者往往直接将全服频道实现为单一的广播列表，导致服务器在高峰时段崩溃。实际上，全服频道的消息应经过优先级分级：普通玩家聊天消息可以容忍100-300毫秒的延迟批量推送，而系统公告（如服务器维护预警）则需要走高优先级通道立即投递。混淆这两类消息的优先级会导致系统公告被聊天消息的推送队列阻塞。

## 知识关联

聊天系统的频道订阅模型和消息序号机制，是理解**消息队列**这一后续概念的直接前提——消息队列（如Kafka）本质上是将频道的环形缓冲区持久化到磁盘，并支持多消费者独立追踪自己的消费位点，两者在逻辑模型上高度一致。

文字聊天系统中建立的**频道隔离**思想可直接延伸至**语音聊天**的实现：语音频道同样需要区分全队频道、全服频道和私密频道，只是传输的内容从文本消息包变为Opus编码的音频流，单包大小从几十字节增加到约1.5KB，对带宽和延迟的要求也随之大幅提高。

**屏蔽与静音**功能则建立在聊天系统的订阅机制之上：屏蔽某玩家的本质是在服务端过滤层中，将该玩家的`sender_id`加入当前玩家的屏蔽列表，服务端在执行频道扇出时跳过被屏蔽玩家的消息，而不是在客户端隐藏——服务端过滤可防止被屏蔽玩家通过其他手段绕过屏蔽机制。
