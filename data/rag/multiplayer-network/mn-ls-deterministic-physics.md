# 确定性物理

## 概述

确定性物理（Deterministic Physics）是帧同步网络游戏中用于保证所有客户端物理模拟结果按位完全一致的技术方案。其核心手段是将物理计算中的浮点数运算替换为定点数运算，消除因不同硬件架构、编译器优化和操作系统对 IEEE 754 浮点标准的差异实现而导致的计算结果分歧。

浮点数不确定性问题的根源可追溯至 1985 年发布的 IEEE 754 标准本身。该标准允许运算结果在末位存在精度误差（ULP，Unit in the Last Place），且不同平台对中间精度、舍入模式（Round-to-nearest、Round-toward-zero 等四种模式）的默认实现各不相同。x86 架构的 FPU 历史上使用 80 位扩展精度寄存器，而 ARM 和 SSE2 默认使用 64 位精度，即便是相同的乘法表达式 `a * b + c`，在 x86 和 ARM 上也可能产生最后几位不同的结果。这种微小差异在物理模拟的迭代积累下会导致帧同步游戏出现逻辑分叉（Desync），造成不同玩家看到完全不同的游戏状态。

确定性物理在帧同步架构中具有不可替代的地位，因为帧同步的正确性前提是"相同输入 → 相同状态"。《星际争霸》系列因早期未严格实现确定性物理，在跨平台对战时频繁出现不同步问题；而《王者荣耀》和《Lockstep 战争游戏》等现代帧同步游戏均在物理层完全采用定点数方案，以保证 iOS、Android、PC 三端状态完全一致。

## 核心原理

### 定点数表示与精度选择

确定性物理的基础是以整数类型模拟小数运算。最常见的方案是 Q16.16 或 Q32.32 格式：Q16.16 使用 32 位整数，其中高 16 位为整数部分，低 16 位为小数部分，精度约为 `1/65536 ≈ 0.0000153`；Q32.32 使用 64 位整数，精度约为 `2.3 × 10^-10`。游戏物理中常用 Q24.8（精度约 `1/256`，适合像素级碰撞）和 Q16.16（适合通用物理模拟）。

定点数乘法需要特别处理。两个 Q16.16 数相乘时，结果是 Q32.32 格式的 64 位中间值，必须右移 16 位截断回 Q16.16，公式为：
```
result = (int32_t)(((int64_t)a * b) >> 16)
```
这里 `a` 和 `b` 均为 Q16.16 格式的 `int32_t`。舍入策略必须在所有平台上统一，通常选择"截断（truncation）"而非"四舍五入"，因为截断在整数运算中天然一致。

### 跨平台数学库的构建

确定性物理需要重新实现 sin、cos、sqrt、atan2 等超越函数，因为标准库中这些函数的精度和实现因平台而异。常见做法是使用查找表（LUT，Look-Up Table）加线性插值：对 sin 函数建立 4096 个采样点的整数查找表，插值精度在 Q16.16 下误差不超过 `2^-14`。

平方根 sqrt 的确定性实现通常采用牛顿-拉弗森迭代法（Newton-Raphson）：从整数近似值 `x₀` 出发，迭代 `x_{n+1} = (x_n + a/x_n) / 2`，全程用定点整数完成，迭代次数固定（通常 4-6 次）以保证每次运行结果相同。碰撞检测中大量使用的向量归一化也依赖此确定性 sqrt。

### 物理引擎集成策略

将 Box2D 等开源物理引擎改造为确定性版本是常见工程路径。Box2D 的内部变量（`b2Vec2`、`b2Transform` 等结构体）中所有 `float` 替换为自定义定点数类型 `Fixed32`，运算符重载确保加减乘除均走定点路径。速度积分公式 `v = v₀ + a·Δt` 中，`Δt` 必须是固定步长（如 `1/30` 秒对应 Q16.16 表示为 `2185`），绝不使用可变帧时间。

碰撞响应中的分离速度公式 `v_rel = e × v_approach`（`e` 为恢复系数）里，`e` 也必须以定点数存储。物理步长固定为逻辑帧间隔（例如 66ms 对应 15Hz 物理频率，或 33ms 对应 30Hz），与渲染帧解耦，渲染层可用浮点数做插值表现，但逻辑物理层只走定点通道。

## 实际应用

**《王者荣耀》多端同步**：该游戏物理层使用自研定点数库，精度选用 Q16.16，支持 iOS（ARM64）、Android（ARMv7/ARM64）、PC（x86_64）三端逻辑帧完全同步。碰撞检测使用定点数 GJK 算法，向量运算全部替换为 `FixedVector2` 类型。

**RTS 游戏单位移动**：《帝国时代 2》高清版在复刻时将原版整数寻路保留，以维持与旧版录像的确定性兼容，这是确定性物理在历史遗留系统中的反例——若引入浮点物理则会导致旧录像回放失效。

**弹道计算**：射击类帧同步游戏中，抛体运动方程 `y = v₀t - ½gt²` 中的 `g`（重力加速度，如 `9.8 m/s²`）必须以定点数表示，例如 Q16.16 下 `g = 642252`（即 `9.8 × 65536`），所有客户端硬编码同一常量。

## 常见误区

**误区一：使用 `std::round` 统一浮点舍入即可解决问题**。部分开发者认为只要在每次计算后调用 `std::round` 对齐精度，就能消除平台差异。实际上 `std::round` 本身的实现在不同编译器下并不保证按位一致，且浮点加法的中间步骤在截断之前已经产生了平台相关的误差。只有彻底移除浮点数，才能根治不确定性。

**误区二：定点数精度越高越好**。Q32.32 格式确实精度更高，但 64 位乘法需要 128 位中间结果（在 32 位平台上需要软件模拟），性能开销是 Q16.16 的 3-5 倍。移动端物理密集场景中，Q32.32 可能导致帧率下降 20% 以上。应根据游戏具体精度需求选择合适格式，大多数动作游戏 Q16.16 已足够。

**误区三：物理引擎版本更新不影响确定性**。Box2D 3.0 相比 2.4 修改了碰撞求解器的迭代顺序，即便都改为定点数，不同版本引擎处理同一场景也会输出不同结果。帧同步游戏必须在所有客户端锁定同一物理引擎版本，任何升级都需要全端同步发布。

## 知识关联

确定性物理以**确定性帧同步**为直接前提，后者定义了"所有节点执行相同有序指令序列"的架构约束，确定性物理正是为满足这一约束而对物理层进行的工程实现。如果没有帧同步架构的锁步执行保证，确定性物理也失去了意义。

下一个学习重点是**定点数运算**，涵盖定点数的溢出处理（Q16.16 下整数部分仅有 16 位，物理量超过 32767 时需扩展或归一化）、饱和运算（saturating arithmetic）以及完整的定点数数学库实现细节。理解定点数运算的实现细节，能帮助开发者在精度损失、性能开销和跨平台一致性之间做出有据可查的权衡决策。
