# 确定性物理

## 概述

确定性物理（Deterministic Physics）是帧同步网络游戏中保证所有客户端物理计算结果逐比特完全一致的技术方案。其核心要求是：无论玩家使用何种操作系统、CPU架构或编译器，对于相同的初始状态和输入序列，物理仿真必须输出完全相同的结果。哪怕两台机器的计算结果仅相差 1 个 ULP（Unit in the Last Place，浮点数最小精度单位），也会在数百帧后引发"蝴蝶效应"，导致客户端状态彻底分歧（Desync）。

这一问题在2000年代中期随着 RTS 游戏的帧同步架构普及而进入工程视野。暴雪《魔兽争霸III》和 Riot《英雄联盟》早期版本均曾遭受浮点不一致导致的 Desync 问题。根本原因在于 IEEE 754 浮点标准本身的平台实现差异：x87 FPU 内部使用80位扩展精度寄存器，而 SSE2 使用64位，同一段 C++ 代码在不同编译器优化级别下可能产生不同的中间精度。

确定性物理的意义在于它是帧同步架构"无需发送状态、只发送输入"这一带宽优化策略的根基。一旦物理计算不具备确定性，帧同步就必须降级为状态同步，带宽开销可能增加数十倍。因此，所有依赖帧同步的竞技游戏——MOBA、RTS、格斗游戏——都必须解决这一问题。

## 核心原理

### 浮点数不确定性的根源

IEEE 754 标准规定了浮点数的格式，但并未强制规定所有中间运算的精度。典型问题有三类：

1. **FMA 融合乘加指令**：`a * b + c` 在支持 FMA 的 CPU 上会被优化为单条指令，仅做一次舍入；而在不支持的 CPU 上是两次独立运算、两次舍入。结果的最低有效位可能不同。
2. **x87 vs SSE2**：x87 FPU 的80位中间寄存器导致比 SSE2 的64位寄存器多出约3位精度，溢出行为不一致。
3. **编译器重排**：`-O2` 优化可能将 `(a + b) + c` 重排为 `a + (b + c)`，由于浮点加法不满足结合律，结果可能不同。

### 定点数方案的实现原理

解决上述问题的根本方法是完全放弃浮点运算，改用定点数（Fixed-Point Number）。定点数以整数形式存储，通过约定小数点位置来表示小数。最常见的格式是 **Q16.16** 或 **Q24.8**：

- **Q16.16**：32位整数，高16位为整数部分，低16位为小数部分，精度为 `1/65536 ≈ 0.0000153`，适合位置、速度等需要较高精度的量。
- **Q24.8**：32位整数，低8位为小数部分，精度为 `1/256 ≈ 0.0039`，适合角度或权重等精度要求较低的场景。

两个 Q16.16 数相乘时，结果是64位中间值，需右移16位并截断回32位：
```
result = (int32_t)((int64_t)a * b >> 16)
```
这个过程完全由整数指令完成，在任意平台上结果完全一致，因为整数运算不存在舍入模式差异。

### 确定性三角函数与开方

物理仿真中不可避免地需要 `sin`、`cos`、`sqrt` 等函数。标准库的 `sin()` 在 glibc 和 msvcrt 之间对同一输入可能返回不同的最后一位。确定性物理的解决方案：

- **查表 + 线性插值**：预先以 Q16.16 格式存储 4096 或 65536 个三角函数采样点，运行时通过整数索引和插值获得结果。表的精度完全由开发者控制，且所有平台读取同一张表。
- **牛顿迭代开方**：`sqrt(x)` 可用整数牛顿法迭代实现。以 Q16.16 为例，初始猜测后迭代公式为 `x_{n+1} = (x_n + S/x_n) / 2`，全程用整数除法实现，3~4次迭代即可达到 Q16.16 的全精度。

### 碰撞检测的确定性

AABB（轴对齐包围盒）和圆形碰撞检测天然适合定点数，因为它们只涉及加减乘法和比较运算。但凸多边形的 SAT（分离轴定理）检测中涉及向量归一化（除以长度，即 `sqrt`），必须使用上述确定性开方。GJK 算法同理，所有距离计算均需替换为定点数版本。

## 实际应用

**《星际争霸II》（StarCraft II）**使用了完整的定点数物理层。暴雪工程师在 GDC 2013 演讲中披露，游戏内所有单位位置使用 32 位定点数（精度约为地图单位的 1/256），速度和加速度同样全程用定点数运算，从根本上杜绝了跨平台 Desync。

**《王者荣耀》**的帧同步物理层采用了类似方案。腾讯技术团队公开文章指出，所有物理量使用自研的 `FP32`（Q16.16 变体）类型，并完全禁止在逻辑层调用 `Mathf.Sin` 等 Unity 浮点 API，改用预计算的定点数三角表。

**格斗游戏引擎 GGPO** 虽主要依赖回滚网络代码（Rollback Netcode）而非传统帧同步，但其推荐的物理实现同样要求确定性：所有碰撞判定盒坐标必须为整数像素，速度以整数像素/帧存储，彻底回避浮点问题。

## 常见误区

**误区一：设置 `_controlfp` 或 `/fp:strict` 就能解决浮点确定性问题**

部分开发者认为只要统一浮点控制字（设置 x87 精度为64位）或使用严格浮点编译选项，就能跨平台保证一致性。这是不完整的方案。`/fp:strict` 只能禁止编译器重排，却无法阻止 CPU 在 x87 和 SSE2 路径间的差异，也无法保证不同版本 glibc 的 `sin` 实现完全一致。只有彻底放弃浮点数才能从根本上解决问题。

**误区二：定点数精度足够高就不会影响游戏体验**

Q16.16 的小数精度约为 0.0000153 游戏单位，听起来已经极高，但在涉及连续积分（速度→位置）的物理模拟中，每帧的截断误差会累积。若帧率为 20 FPS（帧同步常见逻辑帧率），60 秒内最多积累约 72000 次截断误差。设计时必须有意识地选择合适的定点格式，并在高精度场景下使用64位定点数（如 Q32.32）。

**误区三：Unity/Unreal 引擎的 Physics 组件经过配置后可用于帧同步**

Unity 的 PhysX 和 Unreal 的 Chaos 物理引擎均基于 IEEE 754 浮点数，其内部优化（包括 SIMD 向量化）可能在不同平台产生不同结果。这些引擎从未以"逐比特确定性"为设计目标。在帧同步项目中，必须完全绕过引擎物理层，使用自行实现的定点数物理库（如开源项目 `FixedMath.Net` 或 `libfixedmath`）。

## 知识关联

确定性物理建立在**确定性帧同步**的架构约束之上——帧同步要求所有客户端以相同输入推演出相同状态，确定性物理正是实现这一约束在物理层面的具体手段。没有确定性物理，帧同步的"轻量级输入同步"优势便无从成立。

学习确定性物理后，自然引出**定点数运算**这一专题。定点数的加法、乘法、除法的溢出处理策略，64位中间值的使用时机，以及 Q 格式的选型原则，都是实现确定性物理库时必须精通的具体技能。可以说，确定性物理规定了"为什么必须用定点数"，而定点数运算回答了"定点数具体如何运算而不出错"。

此外，确定性物理与**状态快照与回滚**机制密切相关：在基于回滚的帧同步变体（如 GGPO 风格）中，确定性物理使得状态快照可以精确复现，回滚后重新仿真的结果与原始仿真完全一致，这是回滚正确性的物理层保证。
