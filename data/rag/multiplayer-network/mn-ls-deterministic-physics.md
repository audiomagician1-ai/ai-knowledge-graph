# 确定性物理

## 概述

确定性物理（Deterministic Physics）是帧同步网络游戏中用于保证所有客户端物理计算结果逐比特一致的技术方案。其核心手段是将所有物理量（位置、速度、加速度等）从浮点数（float/double）表示改为定点数（Fixed-Point Number）表示，从而消除不同硬件平台、不同编译器版本在IEEE 754浮点运算中产生的精度差异。

浮点数不确定性问题的根源在于IEEE 754标准允许在中间计算中使用80位扩展精度，而x86处理器的FPU与ARM处理器的NEON单元在处理相同运算时可能产生最后一位（ULP, Unit in the Last Place）的偏差。这个看似微小的误差会通过帧同步的逐帧累积，在数百帧后导致客户端之间的游戏状态完全分歧，即"状态爆炸（State Explosion）"或"蝴蝶效应"问题。

确定性物理最早在经典RTS游戏开发中被系统化提出。《星际争霸》和早期《魔兽争霸3》均采用了定点数物理以保证局域网对战的一致性。随着移动跨平台游戏（如iOS与Android混合对战）的普及，确定性物理从可选方案变为帧同步游戏的强制技术要求。

## 核心原理

### 定点数编码方案

确定性物理的基础是选择合适的定点数格式。最常见的格式是 **Q16.16**：用一个32位有符号整数存储物理量，其中高16位表示整数部分，低16位表示小数部分。其精度为 `1 / 65536 ≈ 0.0000153`，表示范围为 `[-32768, 32767.99998]`。对于需要更大范围的场景（如大世界地图），也可采用 **Q20.12** 格式，牺牲精度换取整数范围扩展至 `[-524288, 524287]`。

加法运算在定点数下直接执行整数加法，无精度损失。乘法则需要额外处理：两个Q16.16数相乘，中间结果为64位（Q32.32），必须右移16位才能还原为Q16.16格式。完整乘法公式为：

```
result = (int32_t)(((int64_t)a * (int64_t)b) >> 16)
```

若省略64位中间步骤而直接使用32位乘法，则会发生整数溢出，导致结果错误。这是初学者最常犯的实现错误。

### 数学函数的确定性实现

标准库的 `sin()`、`cos()`、`sqrt()` 等函数在不同平台下返回结果可能存在ULP级别差异，因此必须用查表法或定点数迭代算法替代。

**平方根**通常采用牛顿-拉弗森迭代法实现定点数版本：从初始估计值 `x₀` 出发，按 `x_{n+1} = (x_n + S/x_n) / 2` 迭代4~6次即可达到Q16.16精度要求。**三角函数**常用256项或1024项的查表法，对应角度分辨率分别为 `360/256 ≈ 1.406°` 和 `360/1024 ≈ 0.352°`，线性插值可进一步提升精度。

### 碰撞检测的确定性保障

AABB（轴对齐包围盒）碰撞检测因只含加减比较运算，天然适合定点数化。圆形碰撞检测需要计算距离的平方，可避免 `sqrt()` 调用：判断两圆是否碰撞只需比较 `dx²+dy²` 与 `(r1+r2)²`，两侧均为定点数整数乘法，完全确定性。

多边形分离轴定理（SAT）碰撞检测中需要计算法向量的点积，所有运算均可通过Q16.16乘法替代。物理引擎Box2D有专门的确定性移植版本 **Box2D-fp**，将其内部所有 `float` 替换为定点数类型，并作为帧同步游戏的开源参考实现被广泛引用。

### 帧物理步长的固定化

确定性物理要求物理模拟步长（timestep）必须是固定值而非动态帧率。通常设置为 `1/20秒（50ms）` 或 `1/30秒（33.33ms）` 的逻辑帧。将可变帧率渲染与固定步长物理分离，渲染层做插值，物理层严格按固定步长推进，这是防止因帧率差异导致积分误差的关键设计。

## 实际应用

**《王者荣耀》**的帧同步架构完整采用了Q16.16定点数物理，其逻辑帧率为15帧/秒（66.67ms/帧），所有英雄位置、技能弹道均以定点数表示。这保证了iOS设备与Android设备在同一局游戏中物理计算的逐帧一致，使录像回放功能只需保存输入序列而无需存储每帧状态快照。

**弹幕射击游戏**（如东方系列同人网络版）中，数百个子弹的抛物线运动均需确定性物理。每颗子弹的速度分量 `vx = speed * cos(angle)`，使用Q12.20定点数三角函数表计算，保证2名玩家在不同机器上子弹位置完全一致，避免"本地判定打中但对方判定未命中"的同步冲突。

**格斗游戏**帧同步实现中，角色碰撞箱（Hitbox）和受击箱（Hurtbox）的坐标使用Q24.8格式（精度约0.004像素），在像素级精度下保持多平台一致性，支持竞技级别的输入回滚（Rollback）与重播（Replay）系统。

## 常见误区

**误区一：在C#/Unity中用 `decimal` 代替定点数**
`decimal` 类型虽为十进制定点数且跨平台一致，但其运算速度比 `float` 慢约10~20倍，在需要每帧处理数千次物理运算的游戏场景中性能不可接受。正确做法是用 `int` 或 `long` 封装自定义定点数结构体，运算速度与原生整数指令相当。

**误区二：只替换物理量，忘记随机数确定性**
部分开发者替换了物理计算为定点数，却仍使用 `System.Random` 或 `UnityEngine.Random` 等平台相关的随机数生成器处理暴击、闪避等随机事件。这同样会导致客户端分歧。确定性物理方案必须配合确定性随机数生成器（如线性同余生成器LCG或梅森旋转算法Mersenne Twister，且所有客户端使用相同种子）。

**误区三：认为关闭浮点数优化就能保证确定性**
即使通过编译器标志 `/fp:strict`（MSVC）禁用浮点优化，不同架构CPU（如x86_64与ARM64）仍然无法保证完全相同的浮点结果，因为它们的浮点单元微架构不同。只有彻底切换为整数定点数运算才能在跨平台场景下达到真正的逐比特一致性。

## 知识关联

确定性物理以**确定性帧同步**为前提：帧同步框架负责保证所有客户端收到相同的输入序列，而确定性物理负责保证相同输入在不同平台上产生相同的物理状态输出。两者分工明确——前者解决"同步什么"，后者解决"如何相同地计算"。

掌握确定性物理后，下一步需深入学习**定点数运算**的完整实现细节，包括溢出防护、精度损失分析、超越函数（如 `atan2`、`exp`）的定点数近似算法，以及如何为现有物理引擎（Box2D、Chipmunk）进行定点数移植。此外，确定性物理的调试依赖**状态校验和（State Checksum）**技术——通常在每逻辑帧对所有物理对象的定点数状态进行CRC32或MD5摘要，客户端间对比校验和可快速定位首次分歧帧，这是确定性物理工程实践中不可缺少的调试手段。
