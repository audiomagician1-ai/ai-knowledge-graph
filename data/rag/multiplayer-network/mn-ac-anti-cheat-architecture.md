# 反作弊架构

## 概述

反作弊架构（Anti-Cheat Architecture）是指在多人网络游戏中，将多种检测手段、验证逻辑和响应机制按照职责分层、协同运作的整体系统设计。它不是单一模块，而是一套将服务器端验证、客户端完整性检查、行为分析引擎和人工审核流程整合在一起的分层框架。典型商业实现如 VAC（Valve Anti-Cheat）、EAC（Easy Anti-Cheat）和 BattlEye 各自采用不同的架构权重，但均遵循"纵深防御"原则——即假设任意单层防御均可被突破，因此必须设计多层独立屏障。

这一架构概念在 2000 年代初随着《Counter-Strike》等竞技游戏的普及而逐渐成型。彼时单纯依赖服务器端校验已无法应对内存注入、瞄准辅助等客户端侧作弊手段，开发者被迫将防御边界扩展至客户端进程层。2010 年后，随着机器学习技术引入行为分析，反作弊架构正式形成了"客户端探针 + 服务器验证 + 云端分析"三层并行的现代范式。

理解反作弊架构的重要性在于：游戏作弊行为直接影响匹配公平性和玩家留存率。Irdeto 2020 年调查数据显示，约 60% 的玩家因遭遇作弊行为而放弃某款游戏，而每封禁一批作弊账号可使相关游戏区服的活跃玩家数在短期内回升 5–15%。架构设计的合理性直接决定了检测覆盖率与误封率之间的平衡。

## 核心原理

### 三层防御分工模型

现代反作弊架构通常划分为三个职责层，每层独立运作且具备不同的信任假设：

- **L1 客户端探针层**：在玩家本地运行（通常以内核驱动或用户态进程形式存在），负责检测内存注入、DLL 劫持、调试器附加和文件哈希篡改。BattlEye 和 EAC 均采用内核级驱动（Ring 0）扫描，以获取比普通进程更高的系统访问权限。客户端层的核心局限是：任何运行在目标机器上的代码都可能被有权访问该机器的作弊者绕过，因此该层数据不可完全信任。

- **L2 服务器权威验证层**：游戏服务器对所有关键游戏状态（位置、伤害、命中判定）保持权威版本，拒绝接受客户端提交的不合理数值。例如玩家在单帧（约 16ms）内位移超过物理上限，服务器应直接丢弃该数据包并记录异常。这一层是架构中唯一真正可信的防御边界。

- **L3 云端行为分析层**：汇聚大量对局数据，使用统计模型或机器学习算法识别长期异常行为模式（如爆头率持续超过 85%、反应时间始终低于人类反应极限 150ms）。该层延迟较高（可能数小时至数天），但误判率低，适合执行永久封禁决策。

### 信任边界与数据流设计

架构设计中最关键的概念是**信任边界**（Trust Boundary）：所有跨越该边界传入服务器的数据必须被视为不可信输入。客户端向服务器发送的应仅包含**玩家意图**（如移动方向、开枪指令），而非游戏结果（如"我击中了敌人"）。服务器根据接收到的意图自行计算结果。这一设计原则将作弊空间压缩为仅影响客户端显示，而无法改变服务器端真相。

### 检测覆盖率与误封率的量化权衡

反作弊架构设计需要在两个对立指标之间寻求平衡：
- **检测率（True Positive Rate）**：正确识别作弊玩家的比例
- **误封率（False Positive Rate）**：将正常玩家误判为作弊者的比例

设检测阈值为 θ，则提高 θ 可降低误封率但会放过更多作弊者；降低 θ 则反之。成熟架构通常采用分级置信度机制：置信度低时仅静默标记（Shadow Flag），置信度高时触发自动封禁，介于两者之间则加入人工审核队列。Valve 的 Overwatch 系统正是这一思路的产物——由高段位玩家组成的陪审团对可疑录像进行人工裁定。

## 实际应用

**《Valorant》与 Vanguard 的架构选择**：Riot Games 为《Valorant》开发的 Vanguard 反作弊系统采用开机自启动的 Ring 0 内核驱动，这是客户端探针层的极端形式。其设计理由是：如果等到游戏启动后再加载驱动，作弊程序可能已在更早阶段劫持系统。这一决策引发了较大的隐私争议，但从架构角度看，它将 L1 层的检测时间窗口从"游戏运行期间"扩展到"系统运行全程"。

**《PUBG》的服务器端反速度作弊**：《绝地求生》早期饱受速度作弊困扰，后来在 L2 层引入位移合法性校验：服务器记录每个玩家的上一帧位置 P₀，当前帧位置 P₁，若 |P₁ - P₀| > V_max × Δt（V_max 为该状态下最大合法速度，Δt 为帧时间间隔），则该移动被服务器拒绝并将玩家回滚至 P₀。此修改在不增加客户端代码的情况下封堵了大量速度作弊。

## 常见误区

**误区一：客户端反作弊越强，系统越安全**。实际上，客户端代码运行在攻击者控制的环境中，无论加多少混淆或驱动级保护，均可被逆向或绕过。过度依赖 L1 层会使开发者忽视更可靠的 L2 服务器验证，形成架构上的单点依赖。正确做法是将 L1 视为增加作弊成本的手段，而非根本防线。

**误区二：一套架构可以同时覆盖所有作弊类型**。瞄准辅助（Aimbot）、透视（Wallhack）和经济作弊（无限资源）需要不同层次的检测逻辑。Aimbot 可以通过行为分析（L3）识别，但实时透视必须在服务器端通过视域剔除（PVS，Potentially Visible Set）来消除根本信息泄露，靠 L1 或 L3 无法有效对抗。架构设计必须针对不同威胁模型分配不同层次的对策。

**误区三：反作弊架构与游戏逻辑完全解耦**。事实上，有效的 L2 验证要求服务器完整运行游戏物理和命中判定逻辑，这意味着反作弊需求会直接影响游戏服务器的计算开销和架构设计，两者无法真正分离。

## 知识关联

**前置概念——漏洞修补流程**：漏洞修补流程为反作弊架构提供了动态更新机制。当 L1 层的某种检测特征被新版作弊工具绕过时，修补流程决定了热更新签名库还是推送驱动版本更新；修补的响应速度直接影响架构中"检测窗口期"的长短。

**后续概念——服务器权威验证**：服务器权威验证是反作弊架构 L2 层的具体技术实现，包括状态回滚、输入合法性校验和延迟补偿机制的详细设计，是将架构原则落地为可执行代码的核心步骤。

**后续概念——客户端完整性**：客户端完整性检查是 L1 层的具体实现技术，涵盖代码签名验证、内存扫描和进程隔离等方法，回答了"如何在客户端本地检测篡改行为"这一 L1 层的工程问题。
