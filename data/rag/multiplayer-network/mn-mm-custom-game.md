# 自定义房间

## 概述

自定义房间（Custom Room）是网络多人游戏匹配系统中的一种玩家主导的游戏组织模式。与自动匹配（Matchmaking）不同，自定义房间允许玩家手动创建一个独立的游戏会话，并通过房间码、好友邀请或公开列表等方式邀请其他玩家加入，整个流程完全绕过了 MMR（匹配分）算法的干预。

自定义房间的设计最早出现在 1990 年代末的局域网对战平台，例如暴雪的 Battle.net（1997年上线）就提供了"创建游戏"和"加入游戏"两个核心按钮，玩家可以自行设置地图、人数和密码。这一设计后来成为几乎所有网络多人游戏的标配功能，但其与自动匹配系统的关系、入口设计和权限控制在不同游戏中差异显著。

自定义房间存在的核心价值在于它满足了三类自动匹配无法覆盖的需求：有组织的团队内部练习（如电竞战队5v5训练赛）、跨服或非标准规则的游戏模式（如《我的世界》自定义地图），以及社交驱动的联机体验（朋友之间约好时间一起玩）。因此，自定义房间本质上是匹配系统的"旁路入口"，其流量规模通常远小于自动匹配，但对玩家留存的贡献不可忽视。

## 核心原理

### 房间的创建与参数配置

玩家创建房间时，系统会在服务器端分配一个唯一的 **房间ID**（通常为6位字母数字混合码，例如 `AB3K7Q`），并将房主（Host）的身份与该ID绑定。房主可配置的参数一般包括：最大玩家人数、是否公开可见、是否设置密码、游戏模式、地图或场景选项等。这些参数在技术层面存储于专用的 **房间状态服务器（Room State Server）** 上，而非游戏逻辑服务器，这样可以在游戏开始前低成本地维护房间信息。

### 公开列表与私密邀请两种入口模式

自定义房间有两种典型的玩家加入入口设计：

- **公开列表模式**：所有未满员且无密码的房间显示在一个可搜索、可筛选的大厅列表中。《魔兽争霸3》的游戏大厅是此模式的经典实现，玩家可以按地图名称、房主ID或人数筛选。这种模式下，系统需要处理"房间列表的实时刷新"问题，过于频繁的轮询（Polling）会给服务器带来压力，业界常见方案是将刷新间隔设为3～5秒。

- **邀请码/好友邀请模式**：房间不对外公开，仅凭6位房间码或通过游戏内好友系统发送邀请链接加入。《英雄联盟》的自定义对局和《Among Us》均采用此模式。邀请码设计的关键在于码的有效期（通常为30分钟至房间关闭）和防滥用机制（同一账号每小时创建房间数量上限）。

### 房主权限与房间生命周期管理

房主拥有其他成员不具备的权限集合，包括：踢出玩家、修改房间参数、转让房主身份、以及触发游戏开始。这套权限模型决定了房间的生命周期逻辑：

```
创建 → 等待中（Lobby） → 游戏进行中（In Game） → 结算 → 解散
```

当房主意外掉线时，系统必须有明确的"房主迁移"策略，常见做法是按加入顺序将房主身份自动转移给下一位玩家，《CS:GO》的自定义房间采用的是此策略。若所有玩家均离开则房间立即从服务器上销毁，以释放内存资源。

## 实际应用

**电竞练习赛场景**：职业或半职业队伍使用自定义房间进行内部5v5训练时，会要求房间支持**观战席位**（通常最多可加入额外的4～8名观察者）和**延迟观战**（Delay Spectate，一般设置为2分钟延迟防止作弊）。《Dota 2》的自定义大厅提供了详细的观战延迟设置，最短1分钟，最长10分钟。

**娱乐自制内容场景**：《Among Us》在2020年爆红的核心驱动力之一，正是其简洁的6位房间码设计——主播可以在直播中直接公布房间码，观众凭码进入，形成了自定义房间与内容创作的强绑定。这个设计让自定义房间同时成为了游戏的传播渠道。

**跨平台联机场景**：部分支持跨平台游戏的作品（如《火箭联盟》2021年上线跨平台功能后）通过自定义房间码作为跨平台联机的主要入口，因为好友系统在不同平台间互通存在障碍，而房间码天然是平台无关的。

## 常见误区

**误区一：认为自定义房间只是"没有匹配分的自动匹配"**

自定义房间与自动匹配在架构上属于两个独立的服务模块。自动匹配依赖匹配队列（Queue）和撮合算法（如 Elo 或 TrueSkill），而自定义房间的服务器分配逻辑通常是"先到先得"地直接指派一个可用的游戏服务器实例，不经过任何队列调度。两者的服务器资源池在很多游戏的实际部署中也是分开管理的。

**误区二：认为公开列表模式适合所有游戏**

公开房间列表在在线玩家数量超过10万时会产生严重的列表拥挤问题——玩家面对数千个房间无从选择。《Halo 2》早期的自定义大厅就因房间列表过长而遭到玩家批评，后续版本改为以好友邀请为主。因此，公开列表模式更适合同时在线玩家数量在数千人量级的中小型游戏或特定游戏模式。

**误区三：自定义房间不需要反作弊措施**

由于自定义房间绕过了MMR系统，部分开发者认为其中的作弊行为危害较小。但实际上，《CS:GO》的职业选手曾在自定义服务器上遭遇外挂玩家录制训练视频窃取战术，且利用自定义房间进行的"刷成就/刷统计数据"行为在多款游戏中均被记录在案。因此，成熟的自定义房间系统仍需对异常行为（如击杀速率异常）进行基础检测。

## 知识关联

自定义房间的前置知识是基本的客户端-服务器网络通信模型，理解"客户端发送房间创建请求→服务器分配房间ID→其他客户端通过ID查询并连接"这一流程即可上手本概念的设计讨论。

在匹配系统的整体架构中，自定义房间与**自动匹配**构成两条并行的入局路径，两者共享游戏服务器资源池，但各自维护独立的玩家会话管理逻辑。理解自定义房间的权限模型（房主-成员二级结构）有助于后续学习**组队系统（Party System）**的设计，因为组队本质上是一种"可以携带进入自动匹配的小型自定义房间"。此外，自定义房间的公开列表设计与**游戏大厅（Game Lobby）** 的 UI/UX 设计高度重叠，两个概念可对比学习。
