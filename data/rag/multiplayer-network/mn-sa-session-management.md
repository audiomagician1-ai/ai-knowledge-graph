# 会话管理

## 概述

会话管理（Session Management）是网络多人游戏服务端负责追踪已连接玩家身份状态的机制。服务器无法仅凭一条 TCP/UDP 数据包判断发送方是谁，因此需要在玩家登录成功后创建一个会话对象，将该玩家的身份信息与后续所有网络请求绑定在一起。这个会话对象在玩家从登录到断开的整个生命周期内持续存在于服务端内存中。

会话管理的思想来自 Web 应用领域，HTTP 协议本身无状态，早期开发者在 1994 年前后引入 Cookie + Session ID 解决身份追踪问题。多人游戏服务端借鉴了这套思路，但由于游戏使用长连接（持久 TCP 或 UDP + 自定义可靠层），会话的生命周期比 Web 请求长得多，通常贯穿整个游戏对局，可能持续数分钟到数小时。

对于游戏服务端而言，会话管理直接决定了作弊防护能力与服务器资源消耗的平衡点。没有正确过期机制的会话会导致"幽灵会话"堆积，占用服务器内存；而过于激进的过期策略则会把网络抖动中的真实玩家误踢出局。

## 核心原理

### 会话令牌的生成与结构

玩家通过账号密码（或 OAuth 第三方登录）完成鉴权后，服务器生成一个**会话令牌（Session Token）**。令牌通常是一个 128 位或 256 位的随机字节串，以十六进制或 Base64 编码表示，例如 `a3f9c2d1...`。令牌必须使用密码学安全的随机数生成器（如 `/dev/urandom` 或 `crypto.randomBytes()`），普通的 `rand()` 函数熵值不足，容易被穷举攻击。

服务端内部的会话对象通常包含以下字段：

```
Session {
    token:        string      // 128-bit 随机令牌
    player_id:    uint64      // 与账号数据库对应的玩家唯一ID
    created_at:   timestamp   // 创建时刻（Unix 时间戳，秒）
    last_active:  timestamp   // 最后一次收到有效数据包的时刻
    state:        enum        // AUTHENTICATING / IN_LOBBY / IN_MATCH
    room_id:      uint32?     // 若在房间中，关联当前房间ID
}
```

玩家每次发送游戏数据包时，数据包头部都携带该 token，服务端通过哈希表（O(1) 查找）快速定位对应会话。

### 会话鉴权流程

鉴权（Authentication）发生在会话创建阶段，大致分为三步：

1. **握手阶段**：客户端发送登录请求，包含玩家凭证（账号密码的 SHA-256 哈希值，或第三方平台的 Access Token）。
2. **验证阶段**：服务端查询账号数据库，校验凭证有效性。若使用第三方平台（如 Steam、PlayStation Network），则向该平台 API 发起服务端到服务端的 token 验证请求，避免客户端伪造身份。
3. **会话建立**：验证通过后服务端生成 Session Token，将会话对象写入内存（Redis 或本地内存哈希表），并将 token 下发给客户端。

此后客户端每个数据包都在帧头附带 token，服务端在业务逻辑处理前先执行**会话中间件**校验 token 合法性，非法或已过期的 token 直接丢弃，不进入后续逻辑。

### 会话维持与心跳机制

长连接游戏中，玩家可能因为网络中断暂时无法发送数据，服务端需要一套规则判断连接是"临时中断"还是"玩家真正离开"。常见做法是**双向心跳（Bidirectional Heartbeat）**：

- 客户端每隔固定间隔（通常 5 秒）发送一个心跳包，服务端收到后更新 `last_active` 字段。
- 服务端每隔 30 秒扫描所有会话，若某会话的 `last_active` 距当前时刻超过**超时阈值**（典型值为 60 秒），则将该会话标记为过期并执行清理。

对于对局中的玩家，超时阈值往往设置得更宽松（如 90 秒），因为网络抖动在移动端游戏中非常普遍；而在大厅等待状态的玩家则可以设置更短的超时（如 30 秒），节约服务器资源。

### 会话过期与资源清理

当会话过期或玩家主动断开时，服务端必须执行一系列清理操作，而不仅仅是删除 Session 对象：

1. 从会话哈希表移除该 token 对应的条目。
2. 若玩家在房间中，通知房间逻辑层处理"玩家离开"事件（例如广播给其他玩家、判断是否触发房间解散）。
3. 若服务器使用 Redis 存储会话（分布式场景），需调用 `DEL session:{token}` 命令释放 Redis 内存。
4. 更新账号数据库中的"最后在线时间"字段。

## 实际应用

**手机 MOBA 游戏的重连机制**：以《王者荣耀》类游戏为例，玩家对局中断线后 App 重启，客户端会携带原始 Session Token 重新连接服务端。服务端找到对应会话且 `state == IN_MATCH`，则允许玩家重新加入正在进行的对局，而无需重新登录。这依赖于会话在短暂断线期间未被过期清除，因此对局会话的超时阈值通常比大厅会话长 2-3 倍。

**反作弊令牌绑定**：某些服务端会在 Session 对象中额外记录玩家登录时的 IP 地址和设备指纹。若同一 token 在 15 分钟内从两个不同 IP 段发来请求，服务端可判定 token 可能被盗用，强制该会话失效并要求重新鉴权。

**分布式游戏服务器的会话共享**：当游戏服务器是一个集群（多台物理机）时，玩家可能连接到不同节点。此时会话数据存储在集中式 Redis 中，所有节点共享同一份会话表，保证玩家切换节点（如从大厅服切换到战斗服）无需重新登录。

## 常见误区

**误区一：会话管理等同于登录验证**  
登录验证只是会话生命周期的起点，负责确认"这个人是谁"；而会话管理贯穿连接全程，持续负责"这个已确认的人现在处于什么状态"。混淆两者会导致开发者只在登录时做安全校验，对后续数据包疏于验证，给数据包注入攻击留下空间。

**误区二：超时阈值越短越安全**  
将超时阈值设为 10 秒会在移动网络环境下频繁误踢正常玩家，因为 4G/5G 网络在切换基站时可能产生 5-15 秒的短暂中断。正确做法是区分场景：大厅状态用较短阈值回收资源，对局状态用较长阈值保障体验。

**误区三：Session Token 可以包含可解码的玩家信息**  
部分开发者会将 `player_id` 直接 Base64 编码作为 token，这意味着攻击者可以遍历 `player_id` 伪造任意玩家的 token。正确做法是令牌本身是随机无意义的，玩家信息只存储在服务端的 Session 对象中，通过查表才能获得。

## 知识关联

**前置概念——游戏服务器生命周期**：服务器的启动、运行、关闭阶段决定了会话存储的初始化和清空时机。服务器重启时若未将会话数据持久化到 Redis，所有在线玩家的会话会瞬间失效，需要重新登录。理解服务器生命周期有助于正确设计会话的持久化策略。

**后续概念——房间创建**：房间创建依赖会话管理的结果。只有持有合法会话且 `state == IN_LOBBY` 的玩家才能发起创建房间请求；房间创建成功后，服务端会将相关玩家的会话 `state` 更新为 `IN_MATCH` 并写入 `room_id`，实现会话与房间的绑定。这也是为什么会话管理需要在房间系统之前设计完成——房间逻辑的准入校验直接依赖会话状态字段。
