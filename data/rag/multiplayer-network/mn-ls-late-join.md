# 中途加入（Mid-game Join / Late Join）

## 概述

中途加入（Mid-game Join，又称 Late Join）是帧同步网络游戏中允许新玩家在对局已经开始后进入、并通过特定机制快速同步到当前游戏状态的技术方案。帧同步架构的根本特征是：整个游戏世界的状态由从第 0 帧开始的**完整输入帧序列**唯一决定——给定相同的初始状态与相同的输入序列，任意两台机器的模拟结果完全一致（确定性原则）。正因如此，新玩家无法像状态同步（State Sync）游戏那样直接接收一份当前世界快照并立即开始游玩，而必须以某种方式重建出与已有玩家**逐比特相同**的确定性状态。

对中途加入需求最早变得迫切的是 RTS 品类。《星际争霸：母巢之战》（1999）与《魔兽争霸 3：混乱之治》（2002）的局均时长超过 30 分钟，Battle.net 的掉线重连机制迫使开发团队设计出"观战追帧（Catch-up Replay）"机制。移动端帧同步普及后，《王者荣耀》（2015年上线）将断线重连的可接受等待时间定在 **8 秒以内**，这对追帧速度提出了工程量化目标。腾讯 GWB 技术博客（2018）披露，该游戏采用每 **5 秒（约 150 帧，逻辑帧率 30 fps）** 存储一次完整快照的策略，断线玩家最多需要重放 149 帧的输入日志。

中途加入的核心矛盾在于：新玩家追帧所需时间与已在局内玩家的体验之间的冲突——追帧越彻底、快照频率越高，服务器存储与带宽开销越大；若追帧过慢，玩家等待时间超过心理阈值（业界经验值约 **10～15 秒**）就会流失。

## 核心原理

### 快照下发 + 快进追帧

中途加入的基本流程如下：

1. 服务器（或权威客户端）向新加入玩家发送最近一个关键帧快照（Snapshot），记为第 $K$ 帧；
2. 服务器同时下发第 $K+1$ 帧至当前帧 $F$ 之间的全部输入帧记录；
3. 新玩家客户端加载快照，从第 $K+1$ 帧开始以**最大速度**重放输入帧直到追上 $F$；
4. 追帧完成后切入正常帧率游玩。

快进追帧阶段**跳过渲染**，仅执行游戏逻辑（即调用 `LogicUpdate()` 而不调用 `Render()`），单核 CPU 每秒可处理的逻辑帧数通常是正常逻辑帧率的 **15～40 倍**。以逻辑帧率 15 fps 的游戏为例，快进时每秒可消化约 300 帧，相当于追赶 **20 秒**的游戏进度。

新玩家需追帧的帧数取决于：快照帧 $K$ 距当前帧 $F$ 的距离，即 $F - K$。若快照间隔为 $N$ 帧，则最差情况下需重放 $N-1$ 帧。总追帧耗时（秒）可粗略估算为：

$$T_{catchup} = \frac{F - K}{R_{fastforward}}$$

其中 $R_{fastforward}$ 为快进时每秒可处理的逻辑帧数。若 $F - K = 150$，$R_{fastforward} = 400$ 帧/秒，则 $T_{catchup} \approx 0.375$ 秒，远低于玩家感知阈值。

### 快照的完整性要求

中途加入的快照必须包含**全部影响后续确定性的状态量**，任何遗漏都会导致加入玩家与其他玩家的游戏状态悄然分叉（Desync）。必须序列化的内容包括：

| 类别 | 具体字段示例 |
|------|-------------|
| 实体状态 | 每个单位的位置（定点数坐标）、血量、魔法值、移动速度、朝向 |
| 技能/CD | 每个技能的剩余冷却帧数、施法状态机状态 |
| Buff 列表 | Buff 类型、剩余持续帧、层数、来源实体 ID |
| RNG 种子 | 随机数生成器的完整内部状态（如 LCG 的当前值或 MT19937 的状态数组） |
| 物理系统 | 碰撞检测网格版本号、所有动态刚体的速度向量 |
| 计时器 | 游戏已运行的精确逻辑帧号 $K$（不可用毫秒时间替代） |

其中 **RNG 种子**是最常被遗漏的字段。若快照中未保存随机数生成器的完整内部状态，新玩家加入后的随机事件序列（暴击判定、技能随机效果、AI 行为树分支）将与已有玩家不同，产生局部逻辑分叉——这种 Desync 极难调试，因为它只影响部分逻辑分支而非立即崩溃。《Glenn Fiedler's Game Networking》（Fiedler, 2015）将 RNG 状态遗漏列为帧同步实现中排名第一的隐性 Desync 来源。

### 服务器端帧历史缓存策略

服务器必须维护一个**滑动快照窗口**，以便随时响应中途加入请求。典型策略为：每隔 $N$ 帧生成一次完整快照并持久化，同时保留该快照之后到当前帧的全部输入帧日志。当新玩家请求加入时，服务器取出最新快照帧 $K$，将快照数据包（通常压缩后为 **50～500 KB**，视游戏复杂度而定）与输入帧日志一并下发。

快照间隔 $N$ 的选取需权衡三个指标：

- **追帧等待时间**：$N$ 越小，$F-K$ 越小，追帧越快；
- **服务器存储开销**：若 $N = 150$ 帧（5秒），一场 20 分钟对局需存储约 **240 个快照**，按每个快照 200 KB 计算，单局占用约 **48 MB** 服务器内存；
- **快照生成 CPU 开销**：序列化完整状态会引发一次性能峰值，$N$ 过小会使峰值过于频繁。

实践中，$N$ 通常取 **150～300 帧**（5～10 秒），可在上述三项指标间取得合理平衡。

## 关键公式与代码示例

以下为追帧主循环的伪代码，展示了分批追帧以避免主线程阻塞的实现方式：

```csharp
// 中途加入追帧主循环（Unity C# 伪代码）
IEnumerator CatchUpCoroutine(GameSnapshot snapshot, List<InputFrame> inputLog) {
    // Step 1: 加载快照，恢复到第 K 帧状态
    GameLogic.LoadSnapshot(snapshot);
    int currentFrame = snapshot.frameIndex; // K
    int targetFrame  = inputLog.Last().frameIndex; // F

    const int FRAMES_PER_BATCH = 100; // 每批次处理帧数，避免单帧耗时过长

    // Step 2: 分批快进，每批处理 100 帧后让出协程
    foreach (var inputFrame in inputLog) {
        GameLogic.ApplyInput(inputFrame);   // 只执行逻辑，不调用 Render()
        currentFrame++;

        if (currentFrame % FRAMES_PER_BATCH == 0) {
            float progress = (float)(currentFrame - snapshot.frameIndex)
                           / (targetFrame - snapshot.frameIndex);
            UI.UpdateLoadingBar(progress);  // 更新加载进度条
            yield return null;              // 让出一帧给渲染线程
        }
    }

    // Step 3: 追帧完成，切入正常游戏
    GameLogic.SetMode(GameMode.Normal);
    UI.HideLoadingScreen();
}
```

注意第 `yield return null` 处：协程在此处暂停一个渲染帧，使加载进度条动画能够正常刷新，避免玩家看到"假死"的界面。每批次 100 帧对应约 **3.3 秒**的游戏时长（15 fps 逻辑帧率），单批次执行时间约 **8～25 ms**（视逻辑复杂度），在 16.7 ms 的渲染帧预算内基本可控。

## 实际应用案例

### 《王者荣耀》断线重连

《王者荣耀》的帧同步实现中，断线重连本质上是中途加入的一个子集（玩家 ID 已在局，但客户端状态丢失）。该游戏将目标等待时间控制在 **8 秒**以内，具体参数如下（据腾讯 GWB 2018 技术分享）：

- 逻辑帧率：**15 fps**
- 快照间隔：**150 帧**（10 秒）
- 快照大小：约 **80～120 KB**（压缩后）
- 最大需追帧数：**149 帧**
- 快进速率：约 **500 帧/秒**（跳过渲染后）
- 理论最大追帧耗时：$149 / 500 \approx 0.3$ 秒（加上网络下载约 1～3 秒）

### 《魔兽争霸 3》观战与重连

《魔兽争霸 3》的网络代码（Battle.net 2002 年版本）采用全量重放追帧方案而非快照方案——即从第 0 帧开始重放全部输入历史。这使得 30 分钟对局的重连等待可达 **2～5 分钟**，是早期帧同步中途加入体验差的典型案例。正是这一痛点推动了后续业界向"快照 + 增量追帧"方案迁移（参见《Multiplayer Game Programming》Grossman, 2013，第 9 章）。

## 常见误区

**误区一：快照只需保存实体列表，不需要保存 RNG 状态**

这是最危险的遗漏。以 MT19937 随机数生成器为例，其内部状态由 **624 个 32 位整数**构成，若快照中缺失这 2496 字节，后续所有随机调用将产生与其他客户端完全不同的序列，引发 Desync。修复方法：在快照序列化时显式调用 `rng.Serialize()` 将完整状态写入快照字节流。

**误区二：用系统时钟（毫秒时间戳）作为快照的帧号标识**

帧同步中，帧号必须是离散的**逻辑帧计数器**（整数），而非浮点毫秒时间。若用毫秒时间戳标识快照位置，在帧率波动（如帧跳过）时会出现帧号对不上的情况，导致新玩家从错误的帧开始重放。正确做法是快照头部显式存储 `int frameIndex = K`。

**误区三：中途加入与观战（Spectator）共用同一套追帧逻辑即可**

观战模式的追帧允许渲染，因为观众不参与操作，延迟 10～30 秒进入也可接受。中途加入的追帧必须**跳过渲染**以保证速度，且追帧完成后玩家的操作输入需立即生效并上行至服务器。两者逻辑层相似但接入点不同，混用会导致中途加入玩家在追帧完成前意外接收本地输入事件。

**误区四：快照间隔 N 越小越好**

$N = 1$（即每帧保存快照）虽然可以让追帧代价趋近于零，但快照序列化本身每次需要 **5～20 ms**（视状态规模），每帧都触发将使游戏逻辑帧率从 30 fps 骤降至 **10 fps 以下**，得不偿失。实际 $N$ 的下限由"快照序列化耗时不超过单帧预算的 10%"确定。
