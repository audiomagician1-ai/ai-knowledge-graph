# 完整性校验

## 概述

完整性校验是指在客户端下载热更新资源包后，通过哈希算法对文件内容进行数学验证，确认下载到本地的字节数据与服务器原始文件完全一致的技术手段。最常用的算法是 MD5（生成128位/32个十六进制字符的摘要）和 SHA-256（生成256位/64个十六进制字符的摘要），二者均具有"雪崩效应"——即原文件哪怕只改变1个比特，输出的哈希值也会完全不同。

该技术的工程背景源于互联网传输的不可靠性。当 CDN 节点缓存资源时，可能因网络抖动导致文件截断，或因节点磁盘故障造成静默数据损坏（Silent Corruption）；客户端在移动网络下下载时也频繁遭遇断点续传拼接错误。游戏热更新场景中，一个损坏的 Lua 脚本包或 AssetBundle 文件一旦被加载，轻则引发 nil 指针崩溃，重则造成存档数据损坏，因此在安装资源前必须先通过校验关卡。

在实际热更新流程里，完整性校验不仅保护玩家设备上的资源正确性，还承担着防止资源被中间人篡改的安全职责。服务器下发的版本清单（通常是一个 JSON 或 CSV 文件）会为每个资源文件记录其预期哈希值，客户端校验失败时必须重新触发下载而非直接使用损坏文件。

## 核心原理

### 哈希摘要的生成与比对

MD5 算法将任意长度的输入按512位分组处理，经过4轮64步运算后输出固定的128位摘要。SHA-256 则采用256位内部状态，安全强度更高但计算耗时约为 MD5 的1.5倍。游戏热更新中常见的做法是：构建服务器在打包资源时对每个文件计算哈希，将结果写入版本清单；客户端下载完成后在本地对同一文件重新计算哈希，与清单中的预期值逐字节比对，完全匹配则校验通过，否则视为损坏。

```
预期哈希（来自清单）:  a3f5c2d1e8b094...
实测哈希（本地计算）:  a3f5c2d1e8b094...  → 通过
实测哈希（损坏文件）:  9e12fa47c03b81...  → 失败，触发重下载
```

### 版本清单的结构设计

热更新版本清单通常包含三个字段：`file_path`（资源在包内的相对路径）、`size`（文件字节数，用于快速预检）、`hash`（MD5 或 SHA-256 值）。先用文件大小做第一道过滤可以跳过大量无需重新哈希的未变更文件，仅对大小匹配但内容可能变化的文件才启动完整哈希计算，显著降低每次启动时的校验耗时。Cocos Creator 的热更新组件 `AssetsManager` 内置了这一两阶段校验逻辑，其清单格式称为 `project.manifest`。

### 损坏修复策略

当某个文件校验失败时，修复流程分三步执行：首先删除本地损坏文件（避免后续意外加载）；其次从 CDN 重新请求同一 URL，若同一 CDN 节点连续两次返回损坏文件则切换备用节点（Origin 回源）；最后对重新下载的文件再次执行哈希校验，确认修复成功才写入本地缓存目录。若重试超过3次仍失败，客户端通常会提示玩家检查网络并上报错误日志，而不会让损坏资源进入游戏运行时。

对于分片下载（将大型 AssetBundle 拆成多个512KB 的分片）的场景，每个分片有独立的哈希值，损坏时只需重下失败的分片而非整个文件，可将大文件损坏时的修复流量降低至原来的 1/N（N 为分片数）。

## 实际应用

**Unity AssetBundle 热更新**：游戏打包时使用 `BuildPipeline.BuildAssetBundles()` 输出的 `.manifest` 文件自带每个 Bundle 的 CRC32 校验值。但 CRC32 仅32位，碰撞概率高于 MD5，安全要求高的游戏会在自有构建管线里额外计算 MD5 并写入自定义清单，以替代官方 `.manifest` 中的 CRC。

**移动端弱网环境下的断点续传校验**：使用 HTTP Range 请求分段下载时，整体文件的哈希需要在所有分段拼接完成后才能计算。为了提前发现损坏，可在服务端对每个分段单独计算哈希并写入清单，客户端每收到一个分段立即校验，失败则当场重传该分段，避免等到全部下载完毕才发现问题。

**CDN 缓存污染检测**：2019年某款国产 MMORPG 曾遭遇 CDN 边缘节点将旧版本资源错误缓存为新版本 URL 的事故，导致部分玩家下载到哈希不匹配的文件。因为客户端部署了 MD5 校验，损坏文件被拦截在加载环节之前，游戏未发生大规模崩溃，而是触发了自动回源重下流程完成修复。

## 常见误区

**误区一：文件大小相同就代表内容正确**  
文件大小只能检测截断类损坏，无法检测字节替换类损坏。攻击者或 CDN 故障完全可能产生大小相同但内容不同的文件，此时只有哈希校验才能发现问题。仅依赖大小检查会给损坏资源留下进入运行时的通道。

**误区二：MD5 已被破解，游戏热更新中不应使用**  
MD5 的"破解"是指密码学意义上可以构造出哈希碰撞，但在热更新场景中威胁模型不同——我们的目标是检测意外损坏而非抵御定向伪造攻击。对于普通网络传输错误和 CDN 缓存问题，MD5 的128位摘要已足够可靠，且其计算速度比 SHA-256 快约30%。如果需要防止资源被恶意替换（如防外挂篡改），则应升级为 SHA-256 并配合签名验证。

**误区三：校验应在后台线程中一次性对所有文件完成**  
全量哈希计算会在游戏启动时造成明显卡顿，尤其是包含数百个 AssetBundle 的大型项目。正确做法是将校验分散到下载流水线中（下载完一个立即校验一个），并在首次校验通过后将结果缓存到本地数据库，后续启动时只对"上次检查后被修改过的文件"（通过文件修改时间戳判断）重新计算哈希，而非每次全量重算。

## 知识关联

完整性校验建立在**资源分发**流程的基础上：CDN 节点负责将资源下发到客户端，而完整性校验则在资源抵达本地后立即对其"验收"，二者在时序上是紧密衔接的。没有可靠的资源分发渠道，哈希清单本身也可能被篡改，因此版本清单文件通常通过 HTTPS 下载或附带数字签名，而非与资源文件使用同一条无保护的 HTTP 通道。

在技术栈层面，完整性校验与**断点续传**、**差量更新（bsdiff/xdelta）**关系密切——差量补丁应用完成后同样需要对最终合并文件执行哈希校验，确认补丁计算过程无误。掌握完整性校验的原理，是理解整个热更新管线中质量保障机制的关键环节。
