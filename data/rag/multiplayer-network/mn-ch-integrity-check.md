# 完整性校验

## 概述

完整性校验是指在客户端下载游戏资源后，通过对文件内容进行哈希计算并与服务器预存的哈希值比对，来判断文件是否完整、未被篡改或损坏的技术手段。在热更新流程中，这一步骤发生在文件落盘之后、资源加载之前，是决定是否允许进入游戏的"最后一道门"。

MD5（Message Digest Algorithm 5）和 SHA（Secure Hash Algorithm）是完整性校验中最常用的两种哈希算法。MD5 生成 128 位（32个十六进制字符）的摘要值，SHA-256 生成 256 位（64个十六进制字符）的摘要值。MD5 自1992年由 Ron Rivest 设计发布，计算速度快，长期被游戏热更新系统广泛采用；SHA-256 则在安全要求更高的场景中逐渐取代 MD5，例如需要防止恶意文件替换的竞技类游戏客户端。

完整性校验之所以在 CDN 热更新中不可缺少，是因为 CDN 节点在分发文件时存在缓存击穿、边缘节点同步延迟等问题，客户端极有可能下载到截断的或版本错误的资源包。如果没有校验直接加载，轻则游戏资产显示异常，重则导致客户端崩溃或逻辑错误，严重影响玩家体验。

## 核心原理

### 哈希摘要的生成与比对

完整性校验的数学基础是单向哈希函数：对于任意长度的输入数据 M，哈希函数 H(M) 输出固定长度的摘要 d，且在计算上不可逆。游戏构建服务器在打包资源时，会对每一个 `.bundle` 或 `.patch` 文件计算其哈希值，将结果写入一个清单文件（通常命名为 `manifest.json` 或 `fileinfo.csv`）并一并上传至 CDN。

客户端完成文件下载后，在本地重新计算同一文件的哈希值，再与清单中记录的哈希值进行字符串比对。若两者完全一致（32位或64位十六进制字符串逐字符相同），则校验通过；任意一位不匹配，均视为文件损坏。以一个典型的 SHA-256 校验为例：

```
期望值: 3b4c9e1fa2d88c6e4f7a0b5d29e31c87a4f60e2d1b9c7f3a8d5e4b2c6a9f1e0d
实际值: 3b4c9e1fa2d88c6e4f7a0b5d29e31c87a4f60e2d1b9c7f3a8d5e4b2c6a9f1e0d  ✓
```

若实际值与期望值有任何差异，则判定下载失败。

### 校验时机与粒度控制

完整性校验可以在三个层次执行：**文件级别**、**分片级别**和**字节流级别**。

- **文件级别**：下载完成后整体校验，是热更新的最常见方式。适用于单文件小于 20MB 的资源包，校验成本低，但一旦失败须整体重下。
- **分片级别**：将大文件切分为固定大小的分片（常见为 512KB 或 1MB），每个分片独立计算 MD5。下载失败时只需重下损坏的分片，节省带宽。这种方式被 CDN 的断点续传协议广泛采用。
- **字节流级别**：在下载过程中使用流式哈希（Streaming Hash），边接收数据边累积哈希状态，完成时直接得到最终摘要，无需二次读取文件，适合内存受限的移动端游戏。

### 损坏修复流程

当校验失败时，标准修复流程分为三步：

1. **删除损坏文件**：将本地校验失败的文件标记并删除，防止下次启动误用。
2. **重试下载**：通常配合退避重试策略（Exponential Backoff），首次立即重试，第二次等待2秒，第三次等待4秒，最多重试3～5次。超出次数后弹出用户提示。
3. **源站回源**：若同一 CDN 节点反复校验失败，客户端应切换请求到备用 CDN 节点或直接回源站下载，避免因边缘节点缓存损坏导致反复失败的死循环。

## 实际应用

**Unity AssetBundle 热更新**：Unity 的 `UnityWebRequest.GetAssetBundle()` 接口支持传入 CRC32 校验码，下载完成后引擎自动比对，校验失败会返回 `isNetworkError = true`，开发者无需手动计算哈希。然而许多项目为了更严格的控制，仍选择手动下载字节流再用 `System.Security.Cryptography.MD5` 类进行 MD5 校验。

**《原神》资源下载器**：miHoYo 的客户端更新程序在每个资源包下载完毕后，对照 `pkg_version` 清单文件中的 MD5 字段逐一校验，失败文件会被加入重下队列并在日志中记录 `[CheckFail] filename: xxx, expected: abc123, actual: def456`，方便排查 CDN 污染问题。

**Cocos Creator 热更新组件**：Cocos 内置的 `AssetsManager` 类在其 `checkUpdate` 和 `update` 阶段均会对比 MD5 值，清单文件格式为 JSON，每条记录包含 `md5`、`size`、`compressed` 三个字段，其中 `size` 字段作为下载完成的初步检测，`md5` 字段才是最终可信的完整性凭据。

## 常见误区

**误区一：文件大小相同就代表文件完整**

很多初学者认为只要下载的文件字节数与预期一致，就可以跳过哈希校验。然而网络中间设备（如透明代理、运营商 DNS 劫持）可能替换文件内容而保持文件大小不变。哈希碰撞在 MD5 中已有实际攻击案例（2004年王小云团队证明），更不必说单纯比较大小的方法。正确做法是大小检查仅作为快速预筛，哈希校验才是最终判据。

**误区二：校验失败一定是服务器或 CDN 的问题**

校验失败的原因可能在客户端本地：设备存储空间不足导致写入截断、App 在写文件过程中被系统强杀、SD卡坏块等都会造成落盘数据损坏，和 CDN 无关。错误处理逻辑不应只切换 CDN 节点，还应检查本地磁盘可用空间（建议阈值 50MB 以下时主动提示）并捕获文件写入的 IO 异常。

**误区三：每次启动都对所有本地资源做全量校验**

对成百上千个 bundle 文件在启动时逐一重新计算 MD5，会造成几秒甚至十几秒的额外等待。正确实践是仅对本次热更新涉及的新下载文件进行校验，已通过校验且未被修改的文件通过"文件修改时间 + 大小"做快速缓存命中判断，只有命中异常时才触发重新哈希计算。

## 知识关联

完整性校验依赖**资源分发**阶段提供的清单文件和 CDN 下载链路：只有资源被正确打包并上传、清单中的哈希值与实际文件对应，校验流程才有意义。若清单文件本身被篡改（例如 CDN 被劫持），则校验会通过但加载的是错误资源，这是 HTTPS + 代码签名需要解决的更高层次问题，超出完整性校验本身的能力范围。

在工程实践中，完整性校验与**断点续传**机制紧密配合：分片校验允许续传时跳过已验证通过的分片，而不是重新下载整个文件。掌握完整性校验的具体实现细节，是构建健壮热更新系统的必要基础，也是排查线上玩家"资源加载失败"问题时最先检查的环节。
