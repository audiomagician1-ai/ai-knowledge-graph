# 服务发现

## 概述

服务发现（Service Discovery）是微服务架构中使服务实例能够自动注册自身网络位置、并让其他服务能够动态查找这些位置的机制。在游戏服务端中，一个大厅服务需要找到可用的战斗服务器时，不能将 IP 地址硬编码在配置文件里——因为战斗服务器实例会随负载动态扩缩容，其 IP 和端口随时变化。服务发现正是解决这一问题的基础设施。

服务发现的概念随着 Netflix 在 2012 年前后将其单体架构拆分为微服务而被广泛实践，Netflix 开源的 Eureka 是最早被大规模使用的服务注册中心之一。在游戏行业，随着云原生架构的普及，像 Consul（HashiCorp，2014年发布）和 Kubernetes 内置的 DNS 服务发现也成为主流选项。

对于网络多人游戏而言，服务发现尤为关键：一场百人对战可能需要动态启动新的战斗服务器 Pod，匹配服务必须在毫秒级别内感知到这个新实例的存在并将玩家路由过去。若服务发现延迟过高或信息过期，玩家可能被分配到已崩溃的实例，导致连接失败。

## 核心原理

### 注册表与服务注册

服务发现的核心是一个集中式或分布式的**服务注册表**（Service Registry），存储每个服务实例的元数据，包括：服务名称、IP 地址、端口号、健康状态和自定义标签（如区服、地图类型）。

注册方式分两种：
- **自注册（Self-Registration）**：服务实例启动时主动向注册中心发送 HTTP POST 或 gRPC 请求，写入自身信息，并以心跳（Heartbeat）维持存活状态。Consul 要求客户端每隔 10 秒发送一次心跳，超过 TTL（Time-To-Live）未收到心跳则将实例标记为不健康。
- **第三方注册（Third-Party Registration）**：由部署平台（如 Kubernetes）负责检测容器启动事件，自动向注册中心写入信息，服务自身无需感知注册逻辑。Kubernetes 的 Endpoints 对象就扮演这一角色。

### 客户端发现 vs 服务端发现

**客户端发现**：调用方（如匹配服务）直接查询注册中心获取战斗服务器列表，然后在本地运用负载均衡算法（如随机、轮询、最少连接）选择一个实例进行调用。Netflix Ribbon 库采用此模式。优点是调用路径短、延迟低；缺点是每个客户端都需要实现负载均衡逻辑。

**服务端发现**：调用方将请求发送给一个负载均衡器（如 AWS ALB 或 Nginx），由负载均衡器查询注册中心并转发。Kubernetes 的 Service 对象 + kube-proxy 就是服务端发现模式。优点是客户端逻辑简单；缺点是负载均衡器本身可能成为瓶颈，且引入额外一跳的网络延迟（通常 1-2ms）。

### 健康检查与失效摘除

注册表中的信息必须与实际运行状态保持同步，否则调用方会拿到已宕机实例的地址。健康检查机制包括：
- **TCP 检查**：注册中心定期尝试建立 TCP 连接到服务端口，连接失败则标记不健康。
- **HTTP 检查**：注册中心定期 GET 服务的 `/health` 端点，检查返回 HTTP 200。
- **心跳超时**：如上述 Consul 的 TTL 机制。

在游戏服务端中，一个战斗服务器在游戏结束后需要主动向注册中心发起**注销（Deregister）**请求，而不是仅仅停止心跳——等待超时摘除会造成数秒的延迟窗口，期间新玩家仍可能被路由到该实例。

## 实际应用

**多人游戏房间分配流程**：玩家通过客户端请求匹配时，匹配服务（Matchmaking Service）向 Consul 查询标签为 `game_mode=battle_royale` 且 `status=waiting` 的战斗服务器实例列表。Consul 返回满足条件的 3 个实例地址，匹配服务选择当前玩家数最少的那个，通知玩家客户端直连该 IP:Port。整个查询过程通过 Consul HTTP API 完成，典型响应时间在 5ms 以内。

**弹性扩容场景**：当在线玩家数量在晚高峰激增时，Kubernetes HPA（Horizontal Pod Autoscaler）新启动了 5 个战斗服务器 Pod。得益于 Kubernetes 内置的服务发现，这 5 个 Pod 的 Endpoint 在就绪探针（Readiness Probe）通过后（约 10-30 秒）自动加入 Service 的后端列表，无需人工干预，匹配服务立即可以将玩家路由到新实例。

**跨区服服务发现**：若游戏同时运营亚服和欧服，可以为每个注册实例打上 `region=asia` 或 `region=eu` 标签。网关层根据玩家 IP 的地理位置查询对应 region 的服务实例，实现地理感知的服务路由，避免玩家被错误路由到高延迟区服。

## 常见误区

**误区一：服务注册后不需要心跳维护**
很多初学者认为服务只需注册一次即可。实际上，注册信息是有生命周期的。以 Consul 为例，若服务进程因 OOM 被操作系统强杀，进程没有机会执行注销逻辑，此时注册表里仍有该实例的记录。只有持续心跳机制才能让注册中心在 TTL 超期后（通常 15-30 秒）自动将其摘除。游戏服务必须在单独的后台线程中维持心跳，而非依赖进程正常退出时的主动注销。

**误区二：服务发现能替代负载均衡**
服务发现只负责"找到谁在线"，返回的是实例列表，而不是自动决定将流量发给哪个实例。选择具体实例（负载均衡策略）是独立的步骤。例如对于有状态的游戏房间服务器，不能用无状态的轮询策略——必须根据各实例当前承载的玩家数量来决策，这需要在服务元数据中额外记录负载信息并在客户端实现定制化的选择逻辑。

**误区三：注册中心挂掉则所有服务通信中断**
采用客户端发现模式时，客户端通常会本地缓存上一次查询到的服务列表（如 Consul 客户端默认缓存 30 秒）。注册中心短暂宕机期间，已缓存的路由信息仍可使用，已建立的连接不受影响。真正危险的是注册中心长时间不可用导致缓存过期后无法更新，这是为何 Consul 和 etcd 都要求部署 3 或 5 个节点组成 Raft 共识集群来保证高可用。

## 知识关联

**前置知识——微服务架构**：理解服务发现需要先掌握微服务的基本拆分思想。正是因为微服务将大厅、匹配、战斗、排行榜等功能分拆为独立进程，每个进程独立部署和扩缩容，服务实例的 IP 地址才变成了动态的，静态配置文件中的 IP 列表才变得不可维护，服务发现机制才有存在的必要。

**横向关联——API 网关**：在游戏服务端中，API 网关通常是服务发现的重度使用者。网关收到客户端请求后，需要查询服务注册表来决定将请求转发给哪个下游微服务的哪个实例，同时结合服务发现提供的健康状态信息来实现熔断（Circuit Breaker）——当某服务实例连续健康检查失败时，网关停止向其发送流量。

**工具链**：常见的服务注册中心包括 Consul（支持多数据中心，内置健康检查）、etcd（Kubernetes 使用，强一致性）、ZooKeeper（Kafka 生态常用，较重）。对于中小型游戏项目，直接利用 Kubernetes 内置的 Service + CoreDNS 服务发现是最低成本的选择，无需额外部署和运维独立注册中心。
