# 相关性系统

## 概述

相关性系统（Relevancy System）是网络多人游戏中用于动态调整实体同步频率与优先级的机制，其核心目标是：**并非所有游戏对象都需要以相同频率同步给所有玩家**。以一款100人同时在线的射击游戏为例，若服务器将全部实体状态以30次/秒的频率广播给每位玩家，带宽消耗将呈平方级增长；相关性系统通过评估"某实体对某玩家的重要程度"，将有限带宽分配给真正重要的信息。

该系统起源于Quake引擎时代（1996年前后）的PVS（Potentially Visible Set）技术，该技术通过BSP树预计算哪些区域在视觉上可见，从而决定同步范围。但纯粹基于可见性的判断过于简单——一个在你背后即将向你开枪的玩家，即使不在视野内也极为重要。现代相关性系统因此从单一的空间裁剪演进为综合考量距离、方向、游戏逻辑状态的多维评分体系。

相关性系统相较于兴趣管理（AOI）解决的是更细粒度的问题：AOI回答"哪些实体需要被感知"（二值判断），而相关性系统回答"每个实体应以多高的优先级、多快的频率被更新"（连续量化评分）。这一区别使得在同一AOI范围内的实体也能获得差异化的带宽分配。

## 核心原理

### 相关性评分函数

每个实体对每个客户端的相关性以一个标量分数 **R ∈ [0.0, 1.0]** 表示，服务器按帧（或按固定间隔，如100ms）重新计算该分数。典型的评分公式为：

```
R = w₁·f(distance) + w₂·f(direction) + w₃·f(gamestate) + w₄·f(velocity)
```

其中：
- `f(distance)`：距离衰减项，距离超过某阈值（如150米）后快速趋近0
- `f(direction)`：朝向项，面对玩家的实体得分更高
- `f(gamestate)`：游戏状态项，例如"正在对玩家开枪"的敌人直接获得满分1.0
- `w₁~w₄`：各项权重，总和为1.0

R值随后被映射到一个**更新频率表**：R > 0.8 对应每帧同步，R ∈ [0.4, 0.8] 对应每2帧同步，R < 0.2 则可能每10帧才同步一次甚至暂时休眠。

### 优先级队列与带宽预算

相关性系统的实现通常与**带宽预算（Bandwidth Budget）**机制结合。服务器为每条客户端连接设定一个每帧最大可发送字节数（例如Epic的Unreal Engine默认值为 `NetServerMaxTickRate` 结合 `NetClientTicksPerSecond` 共同控制，典型值约为10~20 KB/s）。当一帧内需要同步的数据超出预算时，系统维护一个优先级队列，按相关性分数从高到低依次填充数据包，低优先级实体的更新被推迟到下一帧。

被推迟的实体会累积一个**饥饿惩罚（Starvation Penalty）**：每被推迟一帧，其优先级分数额外增加一个固定值（如0.05），防止低优先级实体永远得不到同步机会。这一机制确保了即使是远处静止的实体，也会在几秒内至少完成一次状态更新。

### 动态休眠与唤醒

R分数低于阈值（通常为0.05）的实体会进入**网络休眠（Network Dormancy）**状态：服务器停止为该实体生成差量更新包，但客户端保留其最后已知状态继续进行本地模拟（如继续执行待机动画）。当某一触发条件满足时（实体移动、被攻击、进入新的游戏状态），服务器发送一个**唤醒包（Wake Packet）**，内含完整快照而非增量数据，以避免客户端因错过中间状态而出现视觉错误。

Unreal Engine 4/5中通过`AActor::SetNetDormancy()`接口控制这一行为，`DORM_DormantAll`状态意味着该Actor对所有连接都进入休眠，服务器完全跳过其Replication计算，CPU与带宽双重节省。

## 实际应用

### 大型多人战场游戏

在《Battlefield》系列此类64~128人的大地图对战游戏中，相关性系统会为"当前正在被我瞄准的玩家"分配最高优先级（R = 1.0），确保其位置更新延迟不超过一帧（约16ms @ 60Hz服务器）。而地图另一端正在驾车行驶的敌人可能每500ms才获得一次位置更新，客户端使用航位推测法填补中间帧。

### MMO中的人群密度管理

在城镇中心聚集大量玩家时，相关性系统按玩家与观察者的距离分层：0~10米范围内的玩家以完整帧率同步所有属性（血量、技能冷却、坐骑状态）；10~50米的玩家仅同步位置与基础动画；50米以外的玩家仅同步位置，每秒更新2次。这种分层策略使单个客户端在100人同屏场景下的接收带宽控制在3 MB/s以内。

### 交互式物件的状态驱动优先级

游戏中的可破坏门窗等物件在静止状态下保持休眠（Dormant），其R值为0。当玩家走近并触发交互检测时，门的R值立即跳变为0.9，系统发送完整状态唤醒包，同步当前耐久度、开关状态等关键属性，确保所有客户端在互动发生前已接收到最新状态。

## 常见误区

### 误区一：相关性系统等同于视锥剔除

许多开发者将相关性系统简单等同于"屏幕外的东西不同步"。实际上，视锥剔除是客户端渲染层的概念，而相关性系统运行在服务器端的网络层。背后接近的敌人、即将到来的技能投射物等对玩家有即时影响的实体，无论是否在视野内，都应获得高R值并高频同步。过度依赖视锥结果会导致"回头一枪"类型的明显同步错误。

### 误区二：R值越精确计算越好

相关性评分的计算本身也有CPU开销。若服务器每帧为1000个实体×100个玩家 = 100,000对组合重新计算完整R值，其开销可能超过同步本身。实践中通常采用**分摊计算策略**：相关性分数每100~200ms更新一次，而非每帧更新。对于移动缓慢或状态稳定的实体，甚至可以将计算周期延长至500ms。精确度与计算频率之间的权衡必须通过性能分析（Profiling）数据决定，而非凭直觉追求极致精确。

### 误区三：休眠实体无需占用服务器资源

进入`DORM_DormantAll`的实体并不是零开销的。服务器仍需监听能够触发唤醒的事件（碰撞、技能影响范围等），并在实体属性变化时将其标记为"脏"（Dirty），下一帧发送唤醒包。若游戏中有大量可交互物件（如10,000个可破坏物体），事件监听列表本身就需要精心管理，否则反而增加了内存与遍历开销。

## 知识关联

**前置概念——兴趣管理（AOI）**：AOI系统确定了每个玩家"可能关心"的实体集合，为相关性系统提供输入候选列表。相关性系统的R值计算只针对已在AOI范围内的实体展开，AOI是粗粒度的空间过滤器，相关性系统是其后的细粒度量化评分层。若AOI的半径设置过小，相关性系统将永远无法感知到距离较远但游戏逻辑上仍高度相关的实体（如远距离狙击手），因此两个系统的参数需要协调配合。

**横向关联——delta压缩与快照插值**：相关性系统决定"以多高频率同步"，而delta压缩决定"同步时发送多少数据"。低频同步的实体由于两次快照之间的时间差较大，状态变化量更多，delta压缩的效率相应降低；唤醒包因包含完整状态无法使用delta压缩，这要求协议层能够区分增量包与全量包的格式。理解这一联动关系有助于在整体网络架构中做出合理的设计权衡。
