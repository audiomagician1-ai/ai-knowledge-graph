# 相关性系统

## 概述

相关性系统（Relevancy System）是网络多人游戏中用于动态控制哪些游戏对象应当被同步给特定客户端、以及以何种频率和优先级进行同步的机制。它不是简单地开关某个对象的可见性，而是持续评估每个服务器对象对每个连接客户端的"相关性评分"，并据此分配有限的网络带宽资源。

相关性系统的概念在虚幻引擎（Unreal Engine）1998年的《虚幻》原版中已经出现雏形，其引擎内部函数 `IsNetRelevantFor()` 至今仍是该系统的核心入口。Epic Games 在后续版本中将相关性判断扩展为可覆盖的多因子评估框架，开发者可以为每类 Actor 自定义相关性逻辑。

这一机制直接决定了大型多人游戏能否在有限服务器出口带宽（通常每连接 20–100 KB/s）下支撑数十甚至数百个并发玩家。若没有相关性系统，服务器需要向每个客户端广播场景中所有对象的状态变更，带宽消耗随玩家数量呈 O(n²) 增长；引入相关性过滤后，实际同步量可降至 O(n·k)，其中 k 为每个客户端平均关注的对象数量。

## 核心原理

### 相关性评分的多因子计算

相关性系统对每个 <客户端, 对象> 对计算一个综合评分，常见的评分公式形如：

```
Relevancy = w₁·f(距离) + w₂·g(视野) + w₃·h(逻辑关联) + w₄·p(优先级标记)
```

其中 `f(距离)` 通常是距离倒数或指数衰减函数，`g(视野)` 在对象处于玩家视锥内时附加加成（典型加成系数为 1.5–2.0），`h(逻辑关联)` 捕捉队友、交战目标、任务 NPC 等逻辑上必须感知的关系，`p(优先级标记)` 允许设计师为关键剧情对象设置始终高相关性的硬性标记。权重 w₁–w₄ 通常通过游戏类型调优，例如第一人称射击游戏会给视野因子更高的权重。

### 同步频率分级

评分计算完毕后，系统将对象划分为若干同步优先级档位，不同档位对应不同的更新周期：

- **高相关（评分 > 阈值 H）**：每帧或每 66ms（约 15Hz）同步一次，用于近距离交战目标。
- **中相关（阈值 L < 评分 ≤ H）**：每 200–500ms 同步一次，用于同一区域但非交战的对象。
- **低相关（评分 ≤ L）**：每 2–5 秒同步一次或仅同步重要状态变更，用于远距离背景角色。
- **不相关（评分为 0 或对象被剔除）**：完全不发送该对象的数据，客户端将其销毁或休眠。

虚幻引擎的 `NetUpdateFrequency` 和 `MinNetUpdateFrequency` 属性正是这一分级机制的具体实现，默认高频率为 100Hz，低频率可降至 2Hz。

### 相关性的动态更新与滞后处理

相关性评分并非每帧重新计算——服务器通常每 100–200ms 重新评估一次全体对象的相关性，以平衡计算开销。这带来了"滞后窗口"问题：一个快速移动的玩家可能在评估间隔内穿越相关性边界。为解决此问题，系统引入**滞后容忍带**（Hysteresis Band）：对象从不相关变为相关的激活阈值高于从相关变为不相关的休眠阈值，两个阈值之间形成一条缓冲带，防止对象在边界附近频繁上下线造成的网络抖动（通常缓冲带宽度为激活距离的 10–15%）。

## 实际应用

### 大型战场游戏中的带宽分配

在《PUBG》（绝地求生）这类百人大逃杀游戏中，服务器同时管理最多 100 个玩家角色和大量可交互物品。相关性系统将每个玩家的高相关半径设置为约 200 米（游戏内坐标），在该范围内以高频同步角色动画、射击事件和载具状态；200–800 米的中相关区域仅同步位置和大型载具；800 米以外完全不同步，客户端显示静态占位模型。这使得即使在激战区域，单个客户端每秒接收的状态更新包数量也能控制在 300–500 个以内，而非理论上的数万个。

### MMORPG 中的逻辑关联覆盖

在 MMORPG 场景中，相关性系统需要处理纯距离以外的逻辑关联。例如一个玩家的宠物或召唤物即使超出常规相关距离，也必须与主人保持完整同步；团队副本中的 Boss 对所有团员始终高相关，无论其在房间中的位置。这类"逻辑关联"覆盖通过在 `h(逻辑关联)` 因子中注入玩家社会关系图（组队、公会、仇恨列表）来实现，确保战斗完整性不受距离剔除干扰。

## 常见误区

### 误区一：相关性系统等同于 AOI（兴趣区域）

兴趣管理（AOI）解决的是"哪些对象在空间上需要感知"，输出的是一个二元集合（在/不在感知范围内）。相关性系统在 AOI 的基础上进一步区分同步的**频率和优先级**，输出的是连续评分和分级同步计划。同一个对象可以"在 AOI 范围内"但被相关性系统降级为每 2 秒才同步一次；也可以"不在 AOI 空间范围内"但因逻辑关联被相关性系统强制保持高频同步。两者解决的是不同层次的问题。

### 误区二：降低相关性阈值总是能节省带宽

激进地缩小高相关范围或提高剔除速度，确实减少了稳态下的同步数量，但会增加对象频繁进出相关性区域时产生的**重新激活开销**。每次对象从不相关变为相关，服务器需要发送该对象的完整初始状态快照（Initial State Snapshot），其数据量可能是常规增量更新的 10–50 倍。在玩家密集对战区域，过于激进的剔除策略反而会因大量快照传输导致带宽峰值激增。

### 误区三：相关性评分可以完全自动化，无需人工调参

相关性系统的权重和阈值高度依赖具体游戏类型的数值设计。一款回合制策略游戏和一款 Battle Royale 游戏对"高相关"的定义完全不同，前者可能距离 50 格仍需高频同步，后者 500 米外即可剔除。直接使用引擎默认参数而不进行专项调优，往往导致带宽浪费或可感知的同步延迟，必须结合实际压力测试数据对 `NetUpdateFrequency`、距离衰减曲线和逻辑关联权重进行迭代标定。

## 知识关联

相关性系统依赖**兴趣管理（AOI）**提供初始的空间候选集——AOI 的输出（空间上可能相关的对象列表）是相关性评分计算的输入，避免了对场景中所有对象进行全量评估的 O(n²) 计算开销。没有 AOI 预筛选，相关性系统在万人同服场景下将面临不可接受的 CPU 负载。

从相关性系统出发，可以延伸至**预测与插值**（客户端在低相关对象更新间隔内如何平滑显示状态）以及**优先级队列与网络调度**（当带宽上限到达时如何在多个高相关对象之间分配发包顺序）。相关性评分本身也可以作为**服务器负载均衡**的依据——相关性密集区域可触发动态分区迁移，将高负载区域分配给专用计算节点。
