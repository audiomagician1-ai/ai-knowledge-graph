# 相关性系统

## 概述

相关性系统（Relevancy System）是网络多人游戏中用于动态决定"哪些实体需要同步给哪些客户端、以何种频率同步"的机制。与兴趣管理（AOI）系统只回答"是否在范围内"不同，相关性系统回答的是一个连续的问题：每个实体对每个客户端的重要程度是多少，并据此分配有限的网络带宽。典型实现中，相关性值是一个0.0到1.0之间的浮点数，服务器根据这个值决定同步频率（如每秒更新1次还是20次）和数据优先级。

该系统的雏形出现在1999年前后的早期网络引擎中，虚幻引擎（Unreal Engine）在其网络架构中率先将相关性（Relevancy）作为一等概念正式化，Actor类至今仍保留`IsNetRelevantFor()`虚函数供开发者重写。Epic的设计选择直接影响了后续大量商业游戏引擎的网络层设计。

相关性系统的核心价值在于解决带宽的非均匀分配问题。一个100人的服务器房间中，若所有实体以相同频率向所有客户端广播状态，网络流量与玩家数量的平方成正比；而通过相关性裁剪，实际同步流量可降低至线性甚至更优，使大规模战场类游戏（如战地系列支持128人的Battlefield 2042）在有限带宽下成为可能。

## 核心原理

### 相关性评分的计算

相关性评分通常由多个子因素加权求和得出，公式形如：

```
Relevance = w₁·DistanceFactor + w₂·VisibilityFactor + w₃·GameplayFactor + w₄·OwnerFactor
```

其中`DistanceFactor`为距离衰减项（通常使用反比或指数衰减），`VisibilityFactor`为视线检测结果（被遮挡的敌人可获得极低相关性），`GameplayFactor`为游戏逻辑加成（如当前被玩家锁定的目标得分提升），`OwnerFactor`为所有权标记（玩家自身控制的Pawn始终保持最高相关性1.0）。权重`w₁~w₄`由设计师配置，且各子项可按需增删。

### 同步频率的分级映射

计算出相关性评分后，系统将其映射到离散的同步频率档位，而非连续调整（离散化有助于调度器批处理）。虚幻引擎默认使用`NetUpdateFrequency`和`MinNetUpdateFrequency`两个参数界定频率区间，Actor的实际同步频率在两者之间根据相关性线性插值。例如设置`NetUpdateFrequency=20`（每秒20次）、`MinNetUpdateFrequency=2`（每秒2次），相关性0.5的实体将以约11次/秒同步，而相关性0.1的实体接近最低档2次/秒。

### 相关性的时间持久性与冷却机制

单帧相关性抖动会导致实体在客户端频繁出现和消失，产生"幽灵闪烁"问题。相关性系统通常引入滞后阈值（Hysteresis）：实体激活相关性的阈值（如0.3）高于失活阈值（如0.1），并附加最短保持时间（如500毫秒）。这意味着一个刚进入视野边缘的角色不会因为一帧的遮挡计算失误而立即被标记为无关并停止同步，有效平衡了带宽节省与表现平滑性。

### 强制相关性与特殊事件推送

某些游戏事件要求绕过相关性评分，强制向特定客户端发送一次完整更新，称为强制相关性（Force Relevancy）。典型触发场景包括：玩家刚进入游戏房间时需要接收完整世界快照、爆炸等高影响事件发生时附近所有客户端必须收到通知、以及管理员频道的系统广播。虚幻引擎通过`bAlwaysRelevant`布尔标志和`bOnlyRelevantToOwner`标志实现两个极端情形，开发者也可在`IsNetRelevantFor()`中手动返回`true`触发单次强制同步。

## 实际应用

**大规模Battle Royale场景**：在100人大逃杀地图中，服务器将每个玩家周围约150米内的实体标记为高相关性（评分>0.6），对应同步频率15~20次/秒；150~300米范围内降至中等相关性，对应3~8次/秒；300米以外仅保留极低频率的位置同步（1次/秒）用于雷达显示，武器开火等状态则完全不同步。这一三级划分使单个服务器对单个客户端的出站带宽控制在约30~80 Kbps的可接受范围。

**MMORPG副本中的Boss战**：当一个团队Boss进入战斗状态时，所有参战玩家的客户端对该Boss的相关性评分因`GameplayFactor`大幅提升而接近1.0，Boss的技能释放、仇恨列表、血量变化全部以最高频率同步。战斗结束后，相关性自动回落，同步频率降至心跳检测级别（约0.5次/秒）。

**车辆系统的分部件同步**：赛车游戏中，车辆被拆分为车身、轮胎、引擎状态等多个可独立配置相关性的子组件。距离1000米以上的车辆只同步车身位置（评分0.05），进入500米后额外同步轮胎转动（评分0.3），进入50米内才同步引擎损坏状态和视觉特效参数（评分0.8以上）。

## 常见误区

**误区一：相关性系统等同于AOI裁剪**。AOI（兴趣管理）是二值判断——实体要么在兴趣区域内，要么不在；相关性系统是连续评分，同一个"在AOI范围内"的实体会因距离、遮挡、游戏逻辑而获得不同的同步频率。相关性系统通常建立在AOI之上：AOI先做粗筛（淘汰明显无关的实体），相关性再做细粒度的带宽分配。将两者混为一谈会导致设计时误用AOI阈值来替代相关性权重，进而出现"要么全量同步要么完全不同步"的带宽浪费或信息缺失问题。

**误区二：相关性值越高越好**。将过多实体的相关性拉满（始终为1.0）会使相关性系统退化为全量广播，完全失去带宽节省效果。`bAlwaysRelevant=true`标志应当仅用于游戏规则必须要求所有玩家实时感知的实体（如旗帜、计分板相关对象），不应因"更安全"的心态滥用。一个100人服务器中若有30%的实体被设为永久全量相关，带宽消耗可轻易超出同类型服务器的3倍以上。

**误区三：相关性系统可以完全替代状态压缩**。相关性系统控制的是"同步多频繁"，状态压缩控制的是"每次同步传多少字节"。即使相关性正确裁剪了80%的低优先级更新，若单次同步包因字段冗余而过大，网络峰值仍然存在。两者需配合使用，而非互相替代。

## 知识关联

相关性系统直接依赖兴趣管理（AOI）提供的候选实体集合——AOI的格子划分或KD树查询输出的近邻列表是相关性评分的输入前提，没有AOI的粗筛，相关性系统需要对场景中所有实体进行评分，计算开销随实体数量线性增长，在大规模场景中无法实用。

在状态同步的技术栈中，相关性系统位于"决定同步什么"的决策层，其输出直接驱动下游的增量序列化（Delta Serialization）和可靠/不可靠通道分配逻辑。理解相关性系统有助于正确配置虚幻引擎中`NetDriver`的`MaxInternetClientRate`（默认10000 bytes/s）等带宽上限参数——这些参数的合理值取决于相关性裁剪后的实际期望流量，而非盲目设置。对于希望进一步优化的开发者，状态预测（Client-Side Prediction）和兴趣组广播（Interest Group Multicast）是在相关性系统确定同步对象后进一步提升效率的进阶方向。
