# 代码规范与风格

## 概述

代码规范（Coding Standards）是指在软件开发中约定的一套书写代码的格式、命名和组织规则，目的是让不同开发者写出的代码具有统一的可读性。与个人写作风格不同，代码规范是团队或语言社区共同认定的标准，例如 Python 社区于2001年发布的 PEP 8 文档，至今仍是 Python 代码风格的主要参考标准。

代码规范之所以重要，是因为程序员平均花费约 70% 的时间在阅读代码而非编写代码。在 AI 工程项目中，数据预处理脚本、模型训练代码和推理服务常由多人协作完成，若每人按自己的习惯书写，合并代码时会产生大量理解成本。统一的代码风格能将代码审查（Code Review）时间缩短，并降低因误读变量含义而引入 bug 的概率。

代码规范不仅是格式问题，还涉及代码的可维护性、安全性和可测试性。一份遵循规范的代码文件，即使作者离职，接手者也能在较短时间内理解其逻辑，这在持续迭代的 AI 模型项目中尤为关键。

## 核心原理

### 命名规范

命名是代码规范中最直接影响可读性的部分。Python 中约定使用 `snake_case`（下划线分隔小写字母）命名变量和函数，如 `learning_rate`、`load_dataset()`；使用 `PascalCase`（每个单词首字母大写）命名类，如 `TransformerModel`；使用全大写加下划线命名常量，如 `MAX_EPOCH = 100`。JavaScript 社区则惯用 `camelCase` 命名变量和函数，如 `learningRate`。

命名的核心原则是"见名知意"：变量 `x` 在线性代数公式中可接受，但在数据加载函数中应写为 `input_tensor` 或 `batch_data`。布尔变量应以 `is_`、`has_`、`can_` 开头，例如 `is_trained`、`has_label`，避免用 `flag` 或 `tmp` 这类无语义的名称。

### 缩进与空白

Python 语言规范（PEP 8）明确要求使用 **4个空格**作为一级缩进，禁止混用制表符（Tab）和空格，因为不同编辑器对 Tab 宽度的解释不同（2、4或8个字符），混用会导致代码在其他人的环境中显示错乱甚至运行错误。

运算符两侧应加空格：`loss = criterion(output, target)` 而非 `loss=criterion(output,target)`。函数定义之间保留两个空行，类内部方法之间保留一个空行。行长度 PEP 8 建议不超过 **79个字符**，这一限制源于早期终端每行显示宽度，现代项目有时放宽至120个字符，但超长行仍会降低并排比较代码时的体验。

### 注释规范

注释分为三类，各有适用场景：

- **行内注释**：紧跟代码行末，解释非显而易见的逻辑，如 `lr *= 0.1  # 每10个epoch衰减学习率`
- **块注释**：位于代码段上方，解释整个逻辑块的意图
- **文档字符串（Docstring）**：位于函数或类定义的第一行，使用三引号，描述功能、参数和返回值，供 `help()` 函数和自动化文档工具（如 Sphinx）提取

一个规范的 Python 函数 Docstring 示例：
```python
def normalize(tensor, mean, std):
    """
    对输入张量进行标准化处理。

    Args:
        tensor (torch.Tensor): 输入数据，形状为 (N, C, H, W)
        mean (list): 各通道均值
        std (list): 各通道标准差

    Returns:
        torch.Tensor: 标准化后的张量
    """
```

注释应解释"为什么"而非"是什么"——`i += 1  # i加1` 毫无意义，而 `i += 1  # 跳过CSV文件的标题行` 则有实际价值。

### 自动化检查工具

代码规范可通过工具强制执行，而非依赖人工记忆：

- **Flake8**：Python 代码风格检查工具，检测 PEP 8 违规和常见错误
- **Black**：Python 代码自动格式化工具，直接修改文件，不给出选择，号称"不妥协的格式化器"
- **isort**：自动排序 Python import 语句，规定标准库 → 第三方库 → 本地模块的顺序
- **ESLint**：JavaScript/TypeScript 的代码规范检查工具

这些工具通常集成到 Git 的 pre-commit 钩子中，在每次提交前自动运行，不合规的代码无法提交到代码仓库。

## 实际应用

**AI 训练脚本的规范实践**：在一个典型的模型训练文件中，超参数应集中定义为常量或配置字典，而非散布在代码各处。例如将 `BATCH_SIZE = 32`、`LEARNING_RATE = 1e-4`、`NUM_EPOCHS = 50` 统一写在文件顶部，修改时只需改一处，避免出现同一参数在不同地方取不同值的矛盾。

**团队协作中的规范统一**：GitHub 上的主流开源 AI 项目（如 Hugging Face Transformers）在仓库根目录放置 `.flake8` 配置文件和 `pyproject.toml`，规定了项目专属的行长限制和忽略规则，所有贡献者克隆仓库后运行相同的格式化工具，确保代码风格一致。

**代码审查中的风格反馈**：在 AI 团队的 Pull Request 中，审查者应区分"功能性错误"和"风格问题"——前者必须修改，后者可以在自动化工具未覆盖的情况下提出，但不应成为阻塞合并的强制条件，除非团队有明确约定。

## 常见误区

**误区一：注释越多越好**
过度注释会造成"注释噪音"，使真正重要的注释被淹没。函数 `def add(a, b): return a + b` 上方若写"# 这个函数计算两个数的和并返回结果"，等于重复了代码本身，反而增加阅读负担。规范的做法是为复杂算法、业务逻辑和非直觉性代码添加注释，对自解释的代码保持沉默。

**误区二：规范是个人习惯问题，无需统一**
代码规范的价值在团队规模扩大时才充分体现。单人项目中，混用驼峰命名和下划线命名虽然影响美观，但不会造成严重问题；一旦项目有5人以上协作，不一致的命名会导致同一概念（如"学习率"）出现 `lr`、`learning_rate`、`learningRate` 三种写法，代码搜索和理解成本显著上升。

**误区三：格式化工具会损坏代码逻辑**
Black 等格式化工具只修改空白、换行和引号等纯格式元素，不会改变代码的抽象语法树（AST），即代码的执行逻辑不受任何影响。开发者对工具"乱改代码"的担忧通常来自对工具行为不熟悉，实际上 Black 的设计原则之一就是格式化结果具有确定性——相同输入永远产生相同输出。

## 知识关联

代码规范是编写任何程序的基础习惯，直接影响后续的**调试基础**学习效果。规范书写的代码在调试时更容易定位问题：变量名语义清晰时，观察调试器中的变量值能立刻判断是否异常；缩进规范时，代码块的嵌套关系一目了然，不会将逻辑错误与格式混淆。

学习调试技巧后，会进一步体会到好的命名和适当注释的价值——当程序在第300行抛出 `KeyError`，函数名 `preprocess_input_features()` 比 `func2()` 能在第一时间缩小排查范围。代码规范的习惯越早养成，在复杂的 AI 工程项目中受益越大。
