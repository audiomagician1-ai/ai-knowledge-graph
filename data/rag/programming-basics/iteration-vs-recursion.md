# 迭代 vs 递归

## 概述

迭代（Iteration）与递归（Recursion）是解决重复性计算问题的两种根本不同的程序设计范式。迭代通过循环结构（`for`、`while`）在同一栈帧内反复执行代码块，每次循环更新状态变量直到满足终止条件；递归则通过函数自我调用，将大问题分解为同类型的小问题，每次调用都会在调用栈上压入一个新的栈帧，直到触达基础情形（base case）后逐层返回。

从历史脉络看，迭代对应冯·诺依曼架构中最自然的指令执行模型，是命令式编程的基石；递归则源自1930年代阿隆佐·邱奇（Alonzo Church）的 λ 演算，是函数式编程的理论基础。1960年代 Lisp 语言将递归带入实际编程，成为 AI 领域早期算法表达的主流方式。

理解二者的差异对 AI 工程至关重要，因为神经网络训练中的反向传播（Backpropagation）本质上是链式递推，而批次梯度下降的 epoch 循环是典型迭代。选错范式不仅影响代码可读性，更会导致栈溢出（Stack Overflow）或不必要的性能损耗，直接决定模型训练流水线能否稳定运行。

## 核心原理

### 调用栈与空间复杂度的本质差异

迭代使用 **O(1)** 的额外空间（循环变量固定占用），而朴素递归的空间复杂度为 **O(n)**（调用深度），因为每次递归调用都要在栈上保存当前函数的局部变量、参数和返回地址。以计算斐波那契数列第 n 项为例：

- **迭代版本**：两个临时变量 `a, b` 交替赋值，空间 O(1)，时间 O(n)
- **朴素递归版本**：树状展开，时间 O(2ⁿ)，栈深度 O(n)
- **尾递归版本**：经编译器优化后等价于迭代，空间降至 O(1)

Python 默认递归深度限制为 **1000**（`sys.getrecursionlimit()` 可查），超过即抛出 `RecursionError`。这意味着处理超过1000层的树结构或深度搜索时，直接递归会崩溃，必须改用显式栈迭代。

### 尾递归优化（TCO）与语言支持

尾递归是指递归调用是函数体中**最后一个操作**，形如 `return f(n-1, acc * n)`，其中 `acc` 是累加器参数。此时编译器可以复用当前栈帧而非新建一个，使递归的空间行为等同于迭代。

然而，**Python 和 Java 均不实现 TCO**，这是刻意的设计决策——Python 之父 Guido van Rossum 明确表示不希望 TCO 隐藏调用栈信息，以保留调试可追溯性。相比之下，Haskell、Scala、Scheme 原生支持 TCO。在 AI 工程的 Python 生态中，这意味着所有"看起来是尾递归"的代码实际上仍会消耗栈空间，必须手动转换为迭代。

### 相互转换的规律

任何递归算法都可以机械转换为迭代，方法是用**显式栈（`collections.deque` 或 `list`）模拟调用栈**。转换步骤如下：

1. 将初始参数压入栈
2. 循环弹出栈顶元素，执行原递归函数体的逻辑
3. 将子问题参数压回栈，代替递归调用
4. 用显式变量替代返回值的传递

反向转换（迭代→递归）在**分治结构清晰**时更自然，如归并排序、树的遍历。但若迭代依赖多个相互依赖的状态变量（如动态规划的 DP 表），强行递归化会造成代码可读性下降，且极易引入重复子问题。

## 实际应用

**决策树遍历**：sklearn 的 `DecisionTreeClassifier` 在预测时，内部用迭代（while 循环）沿树路径下降，而非递归，原因正是生产环境中树深度可能超过 Python 栈限制，且迭代版本避免了函数调用开销，推理延迟降低约 15-30%。

**深度优先搜索（DFS）**：在知识图谱或计算图遍历中，递归版 DFS 代码仅需 5-6 行，逻辑直观；但图节点数超过数千时需改为显式栈迭代版。PyTorch 的计算图反向传播正是用迭代拓扑排序实现，而非递归，确保大模型的梯度计算不受栈深度限制。

**LLM 推理中的 Token 生成**：自回归语言模型每步生成一个 token 并将其追加到上下文中，这是**纯迭代过程**——同一前向传播函数被反复调用，上下文长度是循环变量。若误将其理解为递归（"每步依赖上一步"），会错误地期望尾递归优化，实际上代码必须写成显式循环。

## 常见误区

**误区一："递归总比迭代优雅，应优先选递归"**
递归在树/图的结构性分解上确实更直观，但对于线性遍历（如数组求和、字符串处理），递归引入了无谓的 O(n) 栈空间和函数调用开销。Python 中对长度为 10000 的列表求和，递归版本直接触发 `RecursionError`，迭代版本则无任何问题。合理标准是：**问题自然具有递归子结构（树、图、分治）时用递归，线性重复时用迭代**。

**误区二："尾递归在 Python 中等于迭代，不用担心栈溢出"**
这是最危险的误区之一。Python 解释器不做 TCO，即使函数写成标准尾递归形式，调用栈仍会随递归深度线性增长。验证方法：在尾递归函数末尾打印 `len(traceback.extract_stack())`，可见每次递归调用该值均递增1，证明新栈帧始终在被创建。

**误区三："将递归改为迭代一定会提升性能"**
对于**记忆化递归（Memoized Recursion）**，若子问题数量远小于递归调用深度，递归+缓存的性能完全可以与迭代 DP 持平，甚至因为只计算实际访问的子问题（稀疏 DP）而更快。盲目改写为迭代 DP 会导致填充大量无用的表格项，在某些 NLP 动态规划（如 CKY 句法分析）中反而浪费内存。

## 知识关联

**与递归的关系**：本文建立在对递归基础概念（base case、递归关系、调用栈）的理解之上，进一步量化了递归的代价（栈深度、时间复杂度指数爆炸）并提供了系统性的转换方法。递归是工具，迭代vs递归的选择是工程判断。

**通向动态规划**：动态规划恰好处于两者的交汇点——它的**状态转移方程**（如 `dp[i] = dp[i-1] + dp[i-2]`）是递归定义，而**实现**通常用迭代填表（Bottom-Up DP）或递归+记忆化（Top-Down DP）。理解迭代与递归的空间/时间权衡，是选择两种 DP 实现方式的核心依据。

**通向函数式编程**：Haskell、Scala 等函数式语言没有 `for`/`while` 循环，所有重复计算均通过递归（配合 TCO）或高阶函数（`map`、`fold`、`reduce`）表达。`fold` 函数本质上是将迭代累积操作抽象化，理解迭代的本质（状态更新 + 终止条件）有助于直接映射到 `foldLeft`/`foldRight` 的参数设计。
