# 迭代 vs 递归

## 概述

迭代（Iteration）与递归（Recursion）是实现重复计算逻辑的两种根本不同的方法。迭代通过循环结构（`for`、`while`）在同一栈帧内反复执行，直到终止条件满足；递归则通过函数调用自身，每次调用都会在调用栈上压入新的栈帧，直到触达基础情形（base case）后逐层返回。两者在计算能力上是等价的——任何递归程序都可以改写为迭代程序，反之亦然，但在代码可读性、内存开销和执行效率上往往有显著差异。

递归的思想最早可追溯到1950年代，LISP语言（1958年由John McCarthy设计）将递归作为核心控制结构，几乎完全取代循环。而命令式语言如C、Python则以迭代为主流。这一历史分野至今仍影响着函数式编程与命令式编程的设计哲学。在AI工程中，理解两者的区别直接关系到树遍历、动态规划展开、神经网络反向传播等核心算法的实现选择。

## 核心原理

### 调用栈与内存模型

递归的本质代价在于调用栈（Call Stack）的增长。每次递归调用都会压入一个新的栈帧，保存当前函数的局部变量、返回地址和参数。以计算 `factorial(n)` 为例，递归实现需要 O(n) 的栈空间；而迭代实现只需 O(1) 的额外空间，因为只有一个循环变量在持续更新。

Python默认的递归深度限制为 **1000层**（可通过 `sys.setrecursionlimit()` 修改），超过此限制会抛出 `RecursionError`。这意味着对一个长度为5000的链表做递归遍历，在Python中直接崩溃，而迭代版本毫无压力。

### 尾递归优化（TCO）

尾递归（Tail Recursion）是递归的一种特殊形式：函数的最后一步操作是递归调用自身，且返回值直接是该调用的结果，中间不做任何额外计算。支持尾递归优化（Tail Call Optimization, TCO）的编译器或解释器可将尾递归转化为等价的循环，使栈空间降为 O(1)。

```python
# 普通递归（非尾递归）：最后一步是乘法，不能优化
def factorial(n):
    if n == 1: return 1
    return n * factorial(n - 1)  # 返回后还需做乘法

# 尾递归：最后一步只有递归调用
def factorial_tail(n, acc=1):
    if n == 1: return acc
    return factorial_tail(n - 1, acc * n)  # 纯调用，无后续操作
```

**关键陷阱**：Python **不支持** TCO（Guido van Rossum明确拒绝过该特性），所以即使写成尾递归形式，Python中也不会节省栈空间。Scheme、Haskell、Scala则原生支持TCO。

### 时间复杂度的隐藏代价

递归中存在一类特殊问题：重叠子问题导致指数级冗余计算。朴素递归计算斐波那契数列 `fib(n) = fib(n-1) + fib(n-2)` 的时间复杂度为 O(2^n)，因为 `fib(3)` 会被重复计算指数次。迭代版本从底部累加，时间复杂度为 O(n)，空间复杂度为 O(1)。这不是风格偏好，而是算法效率的本质差距，正是这一问题引出了动态规划（将递归+记忆化改写为迭代填表）的需求。

### 代码结构对比

| 维度 | 迭代 | 递归 |
|------|------|------|
| 状态管理 | 显式维护循环变量 | 隐式依赖调用栈 |
| 可读性 | 树/图遍历代码较繁琐 | 树形问题自然映射问题结构 |
| 栈空间 | O(1) | O(深度) |
| 调试难度 | 步骤直观 | 需追踪多层栈帧 |

## 实际应用

**深度优先搜索（DFS）**：递归写法天然契合树和图的嵌套结构，代码仅需5-10行；迭代写法需要手动维护一个显式栈（`stack = []`），代码量翻倍，但在节点深度超过Python递归限制时是必须选项。

**AI决策树推断**：Scikit-learn的决策树预测函数在生产环境中使用迭代遍历而非递归，原因是推断时树深度可能达到数十层，且需要在极低延迟下批量处理数千条样本，迭代消除了函数调用开销。

**反向传播（Backpropagation）**：神经网络的梯度计算在概念上是递归的（链式法则逐层传递），但PyTorch和TensorFlow的实现是迭代的——按拓扑排序遍历计算图节点，用显式的梯度累积变量代替递归调用，避免深层网络的栈溢出风险。

## 常见误区

**误区1：递归一定比迭代慢**
递归的性能劣势来自两方面：函数调用开销（压栈/出栈）和可能的重复计算。如果递归没有重叠子问题（如归并排序的递归划分），且语言支持TCO，递归的实际性能可与迭代持平。Haskell的递归列表处理经过编译器优化后，性能与C语言的循环相当。

**误区2：递归转迭代只需加一个循环**
将递归改写为迭代，通常需要**手动模拟调用栈**，把递归隐式保存的"待处理工作"显式地存入一个栈数据结构。对于二叉树的后序遍历（左→右→根），迭代实现需要两个栈或额外标记节点状态，逻辑比递归版本复杂得多，不是简单替换语法的问题。

**误区3：能写尾递归就等于没有栈溢出风险**
如前所述，TCO是语言/编译器特性，不是递归本身的性质。在不支持TCO的语言（如Python、Java标准JVM）中，尾递归与普通递归的栈消耗完全相同。在Java中，每次调用方法的栈帧大小约为数百字节，JVM默认线程栈大小为512KB，大约能支撑约1000-2000层递归。

## 知识关联

**前置概念——递归**：本文档是在递归基础上的延伸对比。理解递归的基础情形（base case）、递推关系和调用栈模型，是分析两者差异的必要前提。

**后续概念——动态规划**：动态规划是本文讨论的"递归+记忆化"向"迭代填表"转化的系统化方法论。`fib(n)` 的递归→迭代优化正是DP中"自底向上"（Bottom-Up）方法的最简例证。掌握迭代与递归的等价转换，是理解DP状态转移方程如何变成填表循环的直接基础。

**后续概念——函数式编程入门**：函数式编程（如Haskell、Clojure）偏好用递归表达循环逻辑，并依赖TCO或惰性求值保证效率。理解为何Python中递归受限而Haskell中递归是主流，需要从本文档的TCO和语言设计原则出发进行分析。
