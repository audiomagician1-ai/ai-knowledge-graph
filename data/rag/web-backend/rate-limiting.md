# 限流策略

## 概述

限流策略（Rate Limiting Strategy）是一种通过控制单位时间内允许通过的请求数量，保护后端服务免受过载、滥用或恶意攻击的机制。其核心目标是在保证合法用户正常访问的前提下，拒绝或延迟超出阈值的请求。与防火墙的包过滤不同，限流策略专注于请求频率维度，可以细化到每个用户、每个IP、每个API Key甚至每个接口路径的维度进行独立控制。

限流的工程实践可追溯至1990年代互联网早期的TCP拥塞控制机制，但现代API限流体系的标准化主要发生在2010年前后，随着Twitter在2013年公开其REST API的限流规范（每15分钟窗口最多15次请求）、GitHub在2011年引入OAuth Token级别的每小时5000次限制，才逐步形成业界通行的实践规范。这些早期案例奠定了"速率窗口+响应头反馈"的基本模式。

在AI工程的Web后端场景中，限流策略尤为重要，因为大语言模型推理的单次调用成本远高于普通API调用（一次GPT-4调用可能消耗数百毫秒GPU算力），一旦缺乏限流保护，少量并发用户即可耗尽服务资源，导致所有用户请求超时。

## 核心原理

### 固定窗口算法（Fixed Window）

固定窗口算法将时间轴切分为等长的独立窗口，每个窗口内维护一个独立计数器。例如，设置"每分钟最多100次请求"时，计数器在每分钟的第0秒重置为0，窗口内每次请求到达时计数器加1，超过100则拒绝。该算法实现极简，Redis中用 `INCR` + `EXPIRE` 两条命令即可完成，但存在**边界爆发问题**：若用户在第59秒发送100次、在下一分钟的第1秒再发送100次，则2秒内实际通过了200次请求，是阈值的两倍。

### 滑动窗口算法（Sliding Window）

滑动窗口算法消除了固定窗口的边界爆发问题。其精确实现使用有序集合（Sorted Set）存储每次请求的时间戳，公式为：

> **当前窗口请求数 = count(timestamps ∈ [now - window_size, now])**

每次请求到达时，删除窗口外的旧时间戳，统计窗口内请求总数，若超过阈值则拒绝。Redis命令序列为：`ZREMRANGEBYSCORE` → `ZCARD` → `ZADD`。精确滑动窗口的内存消耗与窗口内请求数成正比，在高并发场景下可引入**滑动窗口计数器近似算法**：将当前窗口计数与前一窗口计数按时间比例加权，公式为：

> **estimated_count = prev_count × (1 - elapsed/window) + curr_count**

该近似算法将内存开销降为O(1)，误差通常在0.1%以内。

### 令牌桶算法（Token Bucket）

令牌桶算法以固定速率（rate tokens/second）向桶中投放令牌，桶的最大容量为 `burst_size`。每个请求到达时消耗一枚令牌，桶空则拒绝请求。与固定窗口不同，令牌桶天然支持**突发流量**：若用户在空闲期积累了满桶令牌（如容量200），可在短时间内消费完，随后恢复到 `rate` 速率。AWS API Gateway默认采用令牌桶机制，其账户级默认参数为每秒10,000次稳态速率、5,000次突发容量。

### 漏桶算法（Leaky Bucket）

漏桶算法以恒定速率处理请求，超出桶容量的请求直接丢弃。与令牌桶的关键区别在于：漏桶不允许任何突发流量通过，输出速率严格恒定，因此更适合需要平滑流量的场景（如向下游AI推理服务转发请求），而令牌桶允许短时突发，更适合面向终端用户的API层。

## 实际应用

**OpenAI API的分层限流实践**是业界经典案例。OpenAI对不同等级的账户设置了多维度并发限流：GPT-4的免费层限制为每分钟3次请求（RPM）、每分钟40,000个Token（TPM）；Tier 1付费用户为每分钟500次RPM、800,000 TPM。系统同时在RPM和TPM两个维度独立计数，任意一个触发上限均返回HTTP 429状态码，响应头中包含 `x-ratelimit-remaining-requests` 和 `x-ratelimit-reset-requests` 字段，客户端可据此实现指数退避重试。

**AI推理服务的差异化限流**是另一典型应用场景。在构建AI后端时，通常需要为不同模型配置不同的限流参数：轻量级Embedding接口（如text-embedding-3-small）可配置每分钟1000次，而需要长时间推理的流式对话接口（如32K上下文窗口的模型）可能仅配置每用户每分钟5次，并针对Token消耗量（TPM）而非请求次数进行计量限流。

**限流响应的标准化处理**：合规的限流实现应在HTTP响应中返回 `Retry-After` 头（RFC 7231标准），告知客户端需等待多少秒后重试，避免客户端盲目重试加剧拥塞。

## 常见误区

**误区一：认为限流只需要计数器，不需要维度设计。** 许多初学者用单一全局计数器实现限流，导致一个爬虫用户触发全局限流后，所有合法用户也无法访问。正确做法是按"限流键（Rate Limit Key）"分别计数，常见键的设计为：`IP地址`、`用户ID`、`API Key`、`IP + 路径` 的组合。不同键粒度对应不同防护目标：IP级防DDoS，用户ID级防滥用，API Key级实现商业配额管理。

**误区二：混淆限流（Rate Limiting）与熔断（Circuit Breaker）的职责边界。** 限流是在请求进入服务之前进行流量控制，作用方向是"入口保护"，阈值由服务容量决定；熔断是检测到下游服务故障率超标（如50%请求超时）后主动停止调用，作用方向是"故障隔离"。在AI推理后端中，两者需协同使用：限流控制请求速率防止GPU过载，熔断在推理服务响应延迟过高时保护上游不被拖垮。

**误区三：认为返回HTTP 429就足够，无需实现退避提示。** 若服务仅返回429而不携带 `Retry-After` 或 `x-ratelimit-reset` 头，客户端通常会立即重试，形成"限流→重试→加剧限流"的恶性循环。正确的限流响应必须给出明确的等待时间，并在响应体中说明被限流的维度（是RPM还是TPM），便于客户端和开发者诊断。

## 知识关联

**与API认证（JWT/OAuth）的衔接**：JWT认证完成后，服务端从Token中解析出 `user_id`、`plan_tier`（套餐等级）等字段，这些字段直接作为限流键和限流阈值的来源。例如，解析JWT Payload中的 `"tier": "pro"` 字段，映射到Redis中对应的限流配置（如pro用户每分钟1000次）。如果没有完成认证，限流策略就失去了"用户身份"维度，只能退化为粗粒度的IP级限流。

**为设计限流器做铺垫**：掌握上述四种算法的时间复杂度与空间复杂度权衡——固定窗口O(1)空间最优但有边界漏洞，精确滑动窗口O(n)空间最准确但高负载下有压力，令牌桶和漏桶O(1)空间且语义明确——是进行限流器系统设计（包括分布式限流中的Redis Lua脚本原子操作、Nginx限流模块配置、以及多节点同步问题）的必要基础。
