# 缓存策略

## 概述

缓存策略（Cache Policy）是指在Web后端系统中，决定数据如何存储于高速介质、何时失效、以及如何更新的一套规则体系。其本质是在响应速度、数据新鲜度与存储成本之间寻找最优平衡点。缓存的物理载体通常是内存（如Redis）而非磁盘，因为内存访问延迟约为100纳秒，而典型数据库查询延迟可达数十毫秒，差距超过1000倍。

缓存策略的理论基础来自1968年Peter Denning提出的"工作集模型"（Working Set Model），该模型描述了程序在一段时间内倾向于访问固定的数据子集，即"时间局部性"和"空间局部性"原理。Web后端缓存正是利用这两个原理：被频繁请求的热点数据只需从数据库加载一次，后续从内存中直接返回。

在AI工程的Web后端语境下，缓存策略的重要性尤为突出：一次大型语言模型推理调用可能耗时2-10秒，若对相同提示词（Prompt）的重复请求都走完整推理链路，服务器成本将极其高昂。通过对推理结果进行语义缓存，可将命中请求的响应时间压缩至毫秒级。

## 核心原理

### 缓存淘汰算法

当缓存空间不足时，必须按照既定算法删除旧数据，这类算法称为缓存淘汰策略。最常用的三种是：

- **LRU（Least Recently Used，最近最少使用）**：淘汰最长时间未被访问的数据。Redis 4.0以后将其实现为近似LRU，通过随机采样16个候选键选最旧的删除，将时间复杂度从O(n)降至O(1)。
- **LFU（Least Frequently Used，最不经常使用）**：淘汰访问频次最低的数据，适合访问分布长期稳定的场景。Redis 4.0引入LFU时使用了Morris计数器，用8位整数模拟频率，避免了真实计数的内存开销。
- **TTL过期（Time-To-Live）**：每条缓存记录附带一个存活时间，到期自动失效。Redis通过`EXPIRE key seconds`命令设置，背后采用惰性删除+定期删除两种机制并行运行。

### 缓存更新模式

缓存与数据库之间的数据同步方式直接决定系统的一致性水平：

**Cache-Aside（旁路缓存）** 是最常见的读写模式：读取时先查缓存，未命中则查数据库并回填缓存；写入时先更新数据库，再删除缓存（注意是删除而非更新，因为更新操作存在并发竞争风险）。绝大多数使用Redis的Web应用采用此模式。

**Write-Through（同步写穿）**：每次写操作同时更新缓存与数据库，两者始终一致，但写操作延迟增加，适合读多写少场景。

**Write-Behind（异步写回）**：写操作只更新缓存，由后台异步批量刷回数据库，写性能极高但存在数据丢失风险，MySQL的InnoDB Buffer Pool即采用此思路。

### 缓存失效与雪崩防护

三种典型的缓存失效问题需要特别设计应对方案：

- **缓存穿透**：查询的Key在缓存和数据库中均不存在，导致每次请求都打到数据库。解决方案是布隆过滤器（Bloom Filter）预先过滤无效Key，或对空结果也缓存一个短TTL（如60秒）的null值。
- **缓存击穿**：单个热点Key的TTL恰好在高并发时刻到期，大量请求同时穿透到数据库。解决方案是互斥锁（Mutex Lock）重建缓存，或使用永不过期策略配合异步更新。
- **缓存雪崩**：大批Key同时过期，通常是因为它们被同时写入且TTL相同。解决方案是在基础TTL上叠加`random(0, 300)`秒的随机抖动，使过期时间分散。

## 实际应用

**AI推理结果的语义缓存**：OpenAI的GPT Prompt Cache功能对超过1024 token的提示词前缀进行缓存，命中时API调用费用降低50%。自建系统中，可以使用向量数据库（如Qdrant）将用户问题转换为Embedding向量，通过余弦相似度阈值（通常设为0.95）判断是否命中语义缓存，返回已有答案而无需重新推理。

**HTTP缓存头控制**：Web后端通过响应头 `Cache-Control: max-age=3600, stale-while-revalidate=60` 指示CDN或浏览器：资源在3600秒内新鲜可直接使用，过期后的60秒内仍可用旧资源但须异步刷新。这种策略使用户始终获得快速响应，同时后台静默更新内容。

**数据库查询结果缓存**：以电商商品详情页为例，商品基本信息（名称、图片）可设置TTL为24小时，而库存数量因变动频繁需设置为30秒甚至不缓存。在Redis中，可使用Hash结构`HSET product:1001 name "手机" price 3999`存储多字段，再以`EXPIRE product:1001 86400`设置整体过期。

## 常见误区

**误区一：缓存更新时应该"更新缓存"而非"删除缓存"**。很多初学者认为写数据库后应立即将新值写入缓存，但这在高并发场景下会产生"脏数据"：线程A更新数据库后写缓存，线程B恰好在两个操作中间读取了旧数据库值并覆盖缓存，导致缓存长期存储过期值。正确做法是删除缓存，让下一次读操作重新从数据库加载（Cache-Aside模式）。

**误区二：TTL设置越短，系统越安全**。TTL过短意味着缓存频繁失效，缓存命中率（Hit Rate = 命中次数 / 总请求次数）大幅下降，数据库承受的压力反而急剧上升。合理的TTL应根据数据的业务更新频率与可接受的数据延迟来设定，而非一律设为几秒钟。

**误区三：本地内存缓存（JVM内存/Python字典）可以完全替代Redis**。进程内缓存速度确实快于Redis（无网络RTT），但在多实例部署时（如3个后端Pod），各实例的本地缓存相互独立，更新一个实例的缓存不会同步到其他实例，导致不同请求路由到不同实例时返回不一致的数据。Redis作为集中式缓存可解决此问题。

## 知识关联

掌握**服务器基础概念**（尤其是进程、内存、TCP连接池）是理解缓存为何能提速的前提——缓存本质上是用内存换CPU时间和网络I/O时间。**前端性能优化**课程中学习的HTTP缓存头（ETag、Last-Modified）与后端缓存策略形成完整链路：浏览器→CDN→反向代理→应用层缓存→数据库，每一层都有独立的缓存策略协同工作。

后续的**分布式缓存**课程将解决单机Redis容量瓶颈问题，引入Redis Cluster的一致性哈希分片机制（16384个哈希槽分配给多个节点），以及主从复制与哨兵模式的高可用架构。**CDN内容分发**则是将缓存策略的应用场景从单个数据中心扩展到全球边缘节点，重点学习如何通过`Cache-Control`响应头与Vary头字段精确控制哪些内容可被CDN缓存、如何触发缓存失效（Purge）。
