# 通用状态机

## 概述

通用状态机（General State Machine）是游戏脚本系统中用于描述对象行为逻辑的有限结构模型，其核心思想是：任何时刻，对象只处于有限集合中的**一个**状态，并通过定义好的**转换条件**在状态之间跳转。游戏引擎中的NPC AI、动画控制器、关卡流程管理等都依赖状态机来管理复杂的行为分支，避免用嵌套`if-else`堆砌出难以维护的"意大利面代码"。

状态机的理论基础来自1943年McCulloch和Pitts提出的神经网络模型，以及1956年George Mealy和Edward Moore分别定义的**Mealy机**与**Moore机**。在游戏工业中，状态机真正普及是在1990年代的格斗游戏时代——《街头霸王II》的每个角色都使用了显式的有限状态机（FSM）管理站立、跳跃、蹲下、攻击等状态。现代引擎如Unity的Animator Controller和Unreal的AnimGraph，本质上都是这一思想的图形化实现。

理解通用状态机之所以关键，在于它将**行为逻辑的时序性**与**条件分支**统一在一个可视化、可序列化的数据结构中。相比纯代码逻辑，状态机让游戏设计师无需编程知识也能参与行为调整，同时其明确的状态边界使得调试时能够精确定位"对象在错误的状态中停留"这类问题。

## 核心原理

### 有限状态机（FSM）的形式定义

FSM可用五元组正式表示：**M = (Q, Σ, δ, q₀, F)**，其中：
- **Q**：有限状态集合，例如 `{Idle, Walk, Run, Attack, Dead}`
- **Σ**：输入符号集合（即触发转换的事件或条件），例如 `{速度>0, 受到攻击, 血量=0}`
- **δ**：转换函数 `δ: Q × Σ → Q`，规定在状态`q`下接收输入`σ`后跳转到哪个状态
- **q₀**：初始状态
- **F**：接受状态集合（游戏中通常不强调此项，而关注所有运行时状态）

在游戏实现中，每个状态节点通常包含三个回调钩子：`OnEnter()`（进入状态时执行一次）、`OnUpdate()`（每帧执行）、`OnExit()`（离开状态时执行一次）。这三个钩子的划分使得"播放进入动画"、"持续检测敌人"、"停止音效"等逻辑得以精确归位。

### 分层有限状态机（HFSM）

当FSM的状态数量超过约7～10个时，转换连线会形成"状态爆炸"——一个有10个状态的FSM理论上最多需要90条转换边。**HFSM（Hierarchical FSM）**通过引入**超状态（Superstate）**解决这一问题：子状态可以继承父状态的转换规则。

例如，NPC的`Combat`超状态可以包含`Chasing`、`Attacking`、`Retreating`三个子状态，而"玩家死亡→返回巡逻"这一转换只需定义在`Combat`层级，三个子状态自动继承。Unity Animator的"Any State"节点是HFSM继承语义的近似实现——从任意子状态都能响应某个全局条件。HFSM的代价是**历史状态（History State）**管理变得复杂：当对象从`Combat`退出后再次进入，引擎需要决定是恢复到上次离开时的子状态，还是始终从默认子状态开始。

### 下推自动机（Pushdown Automaton，PDA）

下推自动机在FSM基础上增加了一个**状态栈**，使得状态可以被"暂停"并在之后恢复，而不是被覆盖。其转换函数扩展为 **δ: Q × Σ × Γ → Q × Γ\***，其中Γ是栈字母表。

在游戏中，PDA最典型的应用是**UI菜单系统**：玩家在`Playing`状态按下ESC，引擎将`Playing`压栈并进入`PauseMenu`；在暂停菜单打开`OptionsMenu`，将`PauseMenu`再次压栈；按下返回键则弹栈恢复到`PauseMenu`。这种机制保证了"关闭选项菜单后必然回到暂停菜单而非游戏内"，是纯FSM无法天然保证的**有序返回语义**。用栈实现的PDA每次压栈操作的时间复杂度为O(1)，但需要防止无限递归导致栈溢出。

## 实际应用

**NPC巡逻与战斗AI**：一个典型的游戏NPC使用4状态FSM：`Patrol → Alerted（发现玩家足迹）→ Chase（视野内确认玩家）→ Attack（距离≤2米）`，外加从任意战斗状态回到`Patrol`的计时器（通常设为失去目标后8秒）。这种结构在Unreal Engine 4的行为树出现之前，是FPS游戏中敌人AI的标准实现方案。

**动画状态机**：Unity Animator Controller使用的正是HFSM语义。`Base Layer`中的`Locomotion`子状态机包含`Idle`、`Walk`、`Run`，通过`Speed`参数驱动；`Attack`层可以用`Avatar Mask`叠加在`Locomotion`上，实现边移动边攻击。参数类型分为Float、Int、Bool、Trigger四种，其中Trigger是一次性消耗型布尔值，专门解决"动画事件只应触发一次"的竞态问题。

**关卡流程管理**：大型开放世界游戏中，主线任务进度本身就是一个FSM——每个任务阶段是一个状态，玩家的特定行为（到达地点、对话、拾取物品）触发转换。《塞尔达传说：旷野之息》的神庙谜题系统使用类似PDA的机制，允许玩家暂存当前谜题状态、临时离开神庙补充体力后返回继续，而非重置进度。

## 常见误区

**误区一：认为FSM与HFSM只是规模上的区别**。实际上两者在语义上存在本质差异：HFSM中子状态对超状态的转换具有**继承性**，一个超状态的`OnExit()`在任意子状态退出超状态时都会被调用。这意味着如果在FSM中将HFSM的逻辑平铺，开发者需要在每个子状态手动复制相同的退出逻辑，不仅冗余，还容易因漏写一处导致Bug。HFSM减少的不只是边的数量，而是消除了**逻辑重复**。

**误区二：下推自动机可以用普通FSM加一个"返回状态"变量替代**。这种方案在两层嵌套时勉强可行，但当菜单层级超过2层时，单个"返回状态"变量无法记录完整的历史路径。例如从`A→B→C`后需要依次返回`B`再到`A`，但变量只能保存最近一次来源。PDA的栈结构天然支持任意深度的有序回退，而"返回变量"方案的可扩展性上限是固定的。

**误区三：状态机会导致性能瓶颈**。这种担忧来自对状态机实现方式的误解。基于虚函数表的状态对象实现（每个状态是一个类）确实存在间接调用开销，但在游戏中每帧通常只有数百个活跃状态机，其调用开销远低于渲染或物理计算。更轻量的实现方式是用枚举+switch-case的平铺FSM，其性能与手写if-else相当，现代CPU的分支预测可将其开销压缩到纳秒级别。

## 知识关联

通用状态机建立在**脚本系统概述**所介绍的脚本生命周期（Awake/Start/Update）之上——状态机的`OnUpdate()`需要挂载到引擎的每帧回调中才能驱动转换检测。理解脚本组件的Update执行顺序，有助于避免"状态切换和渲染帧不同步"导致的视觉抖动。

在掌握通用状态机之后，行为树（Behavior Tree）是自然的进阶方向：行为树将FSM的状态概念替换为**任务节点**，用选择节点（Selector）和序列节点（Sequence）的组合代替显式的转换函数δ，更适合表达具有**优先级**和**并发性**的复杂AI行为。Unreal Engine 4正是在2012年将Behavior Tree作为官方AI框架推出，部分取代了早期项目中手写FSM的工作流。此外，脚本系统中的**事件/委托机制**与状态机的转换条件触发紧密配合——理解事件如何跨帧传递，能帮助开发者正确实现"受到伤害立即触发状态切换"而非"下一帧才检测到伤害"的即时响应语义。
