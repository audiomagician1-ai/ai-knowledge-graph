# 命令模式

## 概述

命令模式（Command Pattern）是一种行为型设计模式，其核心思想是将一个操作请求封装成一个独立的对象（称为"命令对象"），该对象包含执行操作所需的全部信息：接收者对象、方法名称以及参数。这与策略模式的区别在于：策略模式封装的是"如何做"的算法，而命令模式封装的是"做什么"这一具体动作本身，使得发出请求的调用者与执行请求的接收者完全解耦。

命令模式最早由 Gang of Four（GoF）在1994年出版的《设计模式：可复用面向对象软件的基础》中正式命名并系统化。在游戏开发领域，命令模式因其天然支持 Undo/Redo（撤销/重做）机制而成为关卡编辑器、脚本系统和输入系统的标配方案。Unity 编辑器的操作历史栈、Unreal Engine 的蓝图事务系统，本质上都是命令模式的实现。

在游戏引擎脚本系统中，命令模式解决了一个关键问题：脚本动作需要被记录、回放、序列化传输以及逆向执行。如果直接调用函数，这些需求几乎无法实现；但将每个脚本动作封装为命令对象后，只需维护一个命令队列，上述所有需求都自然得到满足。

## 核心原理

### 命令接口与具体命令类

命令模式的最小接口只需定义两个方法：`execute()` 和 `undo()`。每个具体命令类实现这两个方法，并在内部保存执行时的快照数据。以游戏角色移动为例：

```cpp
class MoveCommand : public ICommand {
    Actor* actor;
    Vector3 oldPos;  // undo 时需要的快照
    Vector3 delta;
public:
    void execute() override {
        oldPos = actor->position;
        actor->position += delta;
    }
    void undo() override {
        actor->position = oldPos;
    }
};
```

`oldPos` 在 `execute()` 时才被记录，这一点至关重要——命令对象创建时不保存状态，执行时才捕获快照，避免了预先计算无效数据的开销。

### Undo/Redo 栈的数据结构

标准的撤销/重做系统使用**双栈结构**：一个撤销栈（undo stack）和一个重做栈（redo stack）。

- 执行新命令：将命令压入 undo 栈，**同时清空** redo 栈（因为在历史中间插入新操作会使之前的 redo 分支失效）
- 撤销：从 undo 栈弹出命令，调用其 `undo()`，再压入 redo 栈
- 重做：从 redo 栈弹出命令，调用其 `execute()`，再压入 undo 栈

游戏编辑器通常限制撤销栈深度（Unity 默认为 200 步），超出限制时从栈底丢弃最旧的命令，以控制内存占用。

### 命令队列与宏命令

命令队列（Command Queue）是命令模式在实时游戏逻辑中的延伸应用。脚本系统将玩家输入或 AI 决策产生的命令对象推入一个 FIFO 队列，在每帧的固定时间点统一执行。这种设计使得输入处理与逻辑更新严格分离，便于实现帧确定性——网络同步对战游戏要求所有客户端在相同帧执行完全相同的命令序列。

宏命令（Macro Command）是命令模式的组合变体：将多个命令对象打包为一个复合命令对象，其 `execute()` 依次调用子命令的 `execute()`，其 `undo()` **逆序**调用子命令的 `undo()`。这一"逆序撤销"细节是宏命令实现中最容易出错的地方。

## 实际应用

**关卡编辑器操作历史**：在 Unreal Engine 的编辑器中，每次拖动 Actor、修改属性或添加组件都会生成一个具体命令对象并推入事务栈。按下 Ctrl+Z 时，编辑器从栈顶取出命令对象调用其 `undo()` 方法，所有对象恢复到操作前状态，而无需编辑器框架了解每种操作的具体实现细节。

**游戏录像与回放**：将玩家每帧的输入封装为命令对象并序列化存储到磁盘，回放时只需按时间戳重新 `execute()` 这些命令对象，即可完整还原游戏过程。这比录制屏幕视频节省了数百倍存储空间，且支持从任意帧开始重放。

**脚本系统的延迟执行**：在 Lua 脚本与 C++ 引擎核心通信时，脚本触发的命令不应立即跨线程修改引擎对象，而应被封装为命令对象投递到主线程命令队列，由主线程在帧末统一安全执行，避免竞态条件。

## 常见误区

**误区一：认为命令模式必然增加大量类**
初学者看到需要为每种操作创建一个类时会感到繁琐。但在现代 C++ 或 C# 中，可以用 lambda 表达式配合闭包捕获上下文来实现轻量级命令对象，例如 `new LambdaCommand([actor, delta]{ actor->move(delta); }, [actor, old]{ actor->setPos(old); })`，无需为每个操作单独创建命名类。

**误区二：以为所有命令都必须支持 Undo**
并非所有游戏操作都需要 Undo 支持。例如"播放音效"命令通常实现空 `undo()` 方法（无法让声音"未被听到"），只有改变游戏状态的命令才需要真正的撤销逻辑。强行为不可逆操作设计 `undo()` 是过度设计。

**误区三：混淆命令对象的创建时机与执行时机**
命令对象的参数（如目标位置）在**创建时**绑定，但命令执行前的状态快照（如当前位置）必须在**执行时**捕获。若在创建命令时就保存当前状态作为 undo 快照，当同一个命令被延迟执行时，快照可能已经过期，导致撤销后恢复到错误状态。

## 知识关联

命令模式建立在**设计模式概述**中介绍的"封装变化点"原则之上——命令模式将"操作的种类"这一变化点封装为可替换的对象体系。与**策略模式**相比，两者都使用对象封装行为，但策略模式中客户端主动选择算法并立即生效，而命令模式强调操作的延迟执行、排队与可逆性，这三个特性是策略模式不具备的。

在**脚本系统概述**中已介绍的脚本执行流水线，是命令队列最直接的应用场景：脚本解释器将解析出的每条指令生成命令对象，引擎主循环从队列中取出并执行，形成完整的脚本驱动游戏逻辑的闭环。掌握命令模式后，可进一步研究事件溯源（Event Sourcing）架构——该架构将命令模式推向极致，用命令序列的完整历史来表示系统状态，而非直接存储当前状态值。
