# 数据驱动设计

## 概述

数据驱动设计（Data-Driven Design）是一种将游戏逻辑的**可变参数和行为规则从代码中剥离出来**，存储在外部数据表、配置文件或资产中的架构方法。在传统硬编码做法中，一个敌人的生命值、移动速度、攻击范围等数值直接写在 C++ 或 C# 源码里；而数据驱动设计则把这些数值放入 JSON、XML、CSV 或专属的二进制表格中，程序运行时动态读取。

这一理念在20世纪90年代中期随着商业游戏规模扩大而兴起。id Software 在开发《Quake》（1996年）时大量使用脚本文件配置关卡实体行为，是早期数据驱动思想的典型案例。暴雪在《魔兽争霸III》中引入了 `.slk` 电子表格格式管理单位数据，使策划人员无需修改引擎代码即可调整全部230余种单位的属性。此后，虚幻引擎（Unreal Engine）通过 `DataTable` 资产、Unity 通过 `ScriptableObject` 将这一模式固化为引擎级别的标准工具。

数据驱动设计之所以在游戏引擎脚本系统中被广泛采用，根本原因在于它将**迭代速度**从"重新编译"压缩到"热重载配置"。一个数值策划修改角色伤害数值，在纯硬编码项目中可能需要等待数分钟的编译链接；在数据驱动项目中只需保存 CSV 文件并触发热重载，整个流程通常在5秒内完成。

## 核心原理

### 数据与逻辑的分离边界

数据驱动设计的核心操作是划定"**什么是数据，什么是逻辑**"的边界。逻辑是不变的行为算法——例如"角色受伤后扣除生命值并播放受伤动画"；数据是驱动逻辑执行的可变输入——例如"扣除多少生命值""播放哪个动画片段"。正确的分离原则是：**凡是策划或美术可能在一次迭代中需要改变的值，都应当外化为数据**。

以虚幻引擎的 `UDataTable` 为例，开发者首先定义一个 `USTRUCT`：

```cpp
USTRUCT(BlueprintType)
struct FEnemyData : public FTableRowBase {
    float MaxHealth;      // 最大生命值
    float MoveSpeed;      // 移动速度（单位：cm/s）
    float AttackRange;    // 攻击范围（单位：cm）
};
```

然后在编辑器中挂载对应 CSV 文件，运行时通过 `FindRow<FEnemyData>` 按行名查询，整个敌人属性系统只需一套代码即可支持数百种敌人变体。

### 配置格式的选型与权衡

游戏引擎脚本系统常见的数据存储格式各有适用场景：

- **CSV/Excel**：策划人员友好，适合数值型表格数据（如武器属性、技能冷却）；缺点是无法表达嵌套结构，不支持注释。
- **JSON**：支持嵌套对象和数组，适合描述复杂配置（如 AI 行为树参数）；文件体积约比二进制格式大3～5倍。
- **二进制格式（如 FlatBuffers、Protocol Buffers）**：解析速度最快，FlatBuffers 读取时无需内存拷贝，适合性能敏感的运行时热点路径；代价是可读性差，需要额外工具链。
- **ScriptableObject（Unity专属）**：作为引擎资产存储，支持 Inspector 可视化编辑，且天然参与引擎的资产引用系统，适合需要引用其他资产（如贴图、音频）的复合数据。

### 数据驱动与行为驱动的组合

纯数据驱动只能描述"静态属性"，当行为规则本身也需要可配置时，数据驱动会与**行为表**（Behavior Table）或**状态机配置**结合。例如，Unity 的 Animator Controller 本质上是一张以 JSON/二进制存储的状态机数据表，动画过渡条件（如"Speed > 0.1 进入跑步状态"）都是可由编辑器修改的数据，而状态机求值算法则固化在引擎运行时代码中。这种"算法固定、规则可配"的模式是数据驱动设计在行为层面的延伸。

## 实际应用

**RPG 技能系统**：将每个技能的伤害公式系数、冷却时间、消耗魔法值、作用范围写入一张 `skills.csv`，每行对应一个技能 ID。当游戏平衡性测试发现"火球术"伤害过高时，策划直接修改表中的 `BaseDamage` 字段，无需开发介入，当天即可出测试版本。《暗黑破坏神：不朽》的技能数据库包含超过600行的技能配置，正是依赖此类结构维护的。

**本地化文本**：所有玩家可见字符串绝不硬编码在源文件中，而是存储在以语言代码为键的字符串表（如 `strings_zh-CN.json`）里。引擎在启动时根据系统语言加载对应表，同一套逻辑代码无修改地支持十余种语言。

**关卡实体配置**：《虚幻竞技场》系列通过 `.ini` 配置文件控制每个关卡的重生点位置、道具刷新时间、Bot AI 难度参数，地图设计师无需访问 C++ 代码即可完整配置关卡行为。

## 常见误区

**误区一：把所有东西都做成数据**。数据驱动并非意味着消灭所有硬编码。物理碰撞检测算法、渲染管线排序逻辑等纯计算型代码若强行数据化，会引入不必要的解析开销和维护复杂度。正确做法是只将**有业务变更需求**的部分外化，例如碰撞检测算法留在代码中，但碰撞层级掩码（Layer Mask）写入配置。

**误区二：数据驱动等同于脚本语言**。Lua、Python 等嵌入式脚本语言是把**逻辑**移出编译代码，而数据驱动设计迁移的是**参数和规则描述**。二者可以组合使用，但概念不同：一个用 CSV 配置的技能系统是数据驱动的，即使它完全没有使用任何脚本语言。

**误区三：数据驱动必然带来性能损失**。频繁在运行时解析 JSON 确实有性能代价，但在项目启动阶段或关卡加载阶段完成一次性数据解析并缓存为原生结构体（如 `FEnemyData`），运行时的访问开销与直接读取成员变量相同，约为纳秒级别，可忽略不计。

## 知识关联

数据驱动设计建立在**脚本系统概述**所介绍的"引擎与游戏逻辑分离"思想之上——脚本系统将逻辑从编译代码中解耦，而数据驱动进一步将参数从逻辑中解耦，形成"引擎运行时 → 脚本逻辑 → 数据配置"三层递进结构。掌握这一设计方法后，开发者能够直接着手构建具体的游戏系统（如技能系统、AI配置系统），因为数据驱动设计为这些系统提供了面向策划人员的可编辑接口，而无需每次修改游戏规则都依赖程序员重新编译发布。理解数据与逻辑的分离边界，也是后续学习 ECS（Entity-Component-System）架构的重要前置认知——ECS 的 Component 本质上就是纯数据容器，与数据驱动设计在哲学上高度一致。
