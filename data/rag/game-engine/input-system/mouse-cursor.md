# 鼠标光标处理

## 概述

鼠标光标处理是游戏引擎输入系统中专门管理光标可见性、位置约束和渲染方式的子系统。它解决的核心问题是：游戏运行时，操作系统默认的箭头光标往往破坏沉浸感，而某些玩法（如第一人称射击）需要完全隐藏光标并将鼠标移动转化为纯粹的视角旋转增量，另一些玩法（如策略游戏）则需要自定义光标图案来显示当前操作模式。

光标处理机制在早期DOS游戏时代几乎不存在——游戏直接操控硬件中断向量，绕过操作系统完全接管输入。进入Windows图形界面时代后，操作系统的窗口管理器开始统一管理光标，游戏引擎因此必须通过平台API来显示或隐藏光标、修改光标图案、以及锁定光标位置。DirectX 5.0（1997年）引入的独占全屏模式曾经简化了这一问题，但随着现代游戏普遍支持窗口化和无边框窗口模式，光标处理的复杂度大幅提升。

准确处理光标状态对游戏体验有直接的量化影响：研究表明，当玩家在第一人称游戏中按下Alt+Tab切换后，若光标未能正确恢复到桌面模式，用户流失率会显著上升。对于引擎开发者来说，光标处理还与操作系统焦点事件、多显示器设置和平台差异（Windows/macOS/Linux行为各不相同）紧密交织。

## 核心原理

### 光标显示与隐藏

操作系统提供两种控制光标可见性的机制。在Windows上，`ShowCursor(FALSE)` 使用一个内部引用计数器而非简单的布尔开关——每调用一次 `ShowCursor(FALSE)` 计数器减1，只有当计数为负数时光标才真正消失。这意味着若引擎多次调用隐藏而没有对应的恢复调用，窗口失焦后桌面光标也会消失，造成严重的用户体验问题。Unity引擎通过 `Cursor.visible = false` 封装了这一平台差异，内部会追踪调用深度。Godot 4 则提供了枚举值 `Input.MOUSE_MODE_HIDDEN` 和 `Input.MOUSE_MODE_VISIBLE` 来控制可见性，保证跨平台行为一致。

### 光标锁定（Cursor Confinement 与 Cursor Capture）

光标锁定分为两种独立的概念，常被混淆。**限制（Confine）**是将光标限定在窗口矩形内移动，鼠标仍有绝对位置；**捕获（Capture）**则将光标固定在屏幕中心某一点，操作系统每帧将其强制移回该位置，游戏读取的是每帧的相对位移量（Delta）而非绝对坐标。第一人称游戏使用捕获模式，策略游戏使用限制模式。在Windows上，捕获模式对应 `ClipCursor()` + 每帧调用 `SetCursorPos()` 将指针移回中心；在SDL2库中，对应标志为 `SDL_SetRelativeMouseMode(SDL_TRUE)`，该函数直接启用操作系统级别的相对鼠标输入，精度高于手动移回中心点的方案，延迟可降低约1帧。

### 软件光标与硬件光标

**硬件光标（Hardware Cursor）**由操作系统或GPU驱动直接绘制，不经过游戏的渲染管线。其核心优势是即使游戏帧率下降到15fps，光标仍以显示器刷新率（如144Hz）流畅更新，延迟极低。硬件光标的局限是尺寸受限（Windows标准为32×32像素，扩展支持最高64×64），且颜色格式和动画帧数有平台限制。

**软件光标（Software Cursor）**是在隐藏系统光标后，由引擎自行在每帧渲染一个精灵（Sprite）到光标位置。软件光标可以任意尺寸、任意动画效果，并可与游戏世界发生交互（如悬停在敌人身上时光标变色并播放粒子效果）。代价是光标的位置更新受帧率限制，在低帧率下会出现明显滞后感，这种滞后通常在帧时间超过33ms（即帧率低于30fps）时被玩家察觉。虚幻引擎5默认使用软件光标方案（`UUserWidget` 派生的光标Widget），并在Slate渲染器层面单独对光标进行合成以减少滞后。

### 焦点丢失时的光标恢复

窗口失去焦点时，引擎必须自动恢复光标到正常状态，否则玩家无法操作桌面。标准做法是监听操作系统的 `WM_KILLFOCUS`（Windows）或 `SDL_WINDOWEVENT_FOCUS_LOST` 事件，在回调中执行：解除光标锁定、显示系统光标、释放输入捕获。重新获得焦点时再按游戏当前状态重新施加约束。Unity的 `OnApplicationFocus(bool hasFocus)` 回调正是为此目的设计的接口。

## 实际应用

**第一人称射击（FPS）游戏**：游戏启动后立即设置为捕获+隐藏模式，光标完全不可见，鼠标移动量作为摄像机旋转的输入源。按下Escape打开菜单时切换为限制+显示软件光标模式，以显示自定义的游戏内菜单光标。

**实时策略（RTS）游戏**：光标始终可见并限制在游戏窗口内。光标图案根据状态动态切换：默认为箭头，悬停在可攻击单位上变为十字准星（32×32硬件光标），悬停在可采集资源上变为手型图案，拖动框选时隐藏光标改为纯框选叠加层。《星际争霸2》使用了一套包含17种状态的光标状态机。

**鼠标感度校准与原始输入**：某些引擎在捕获模式下默认读取经过操作系统加速处理的鼠标坐标。若要绕过Windows的"指针精确度"（鼠标加速），必须通过 `RegisterRawInputDevices()` 注册原始输入设备，从 `WM_INPUT` 消息中读取未经加速的HID报告数据。这在竞技游戏中至关重要，因为鼠标加速会导致相同物理位移在不同移动速度下产生不同的游戏内位移，破坏肌肉记忆。

## 常见误区

**误区一：隐藏光标等同于锁定光标**
隐藏光标只是让光标图案不可见，鼠标仍在移动。如果不同时启用捕获模式，光标会静默移出窗口边界，下一帧读取到的绝对坐标可能已经在桌面的另一个角落，导致视角突变。正确做法是：隐藏（`Cursor.visible = false`）与捕获（`Cursor.lockState = CursorLockMode.Locked`）必须同时设置。

**误区二：软件光标总是比硬件光标效果更好**
开发者常因为软件光标支持更复杂的视觉效果而全面采用软件光标。但在性能较差的设备上，游戏帧率低于30fps时，软件光标的视觉滞后感会让玩家误以为鼠标硬件出现问题，大幅降低可用性。正确策略是：UI界面和菜单使用硬件光标，仅在确认帧率稳定（>60fps）的游戏玩法阶段使用具有复杂效果的软件光标。

**误区三：光标处理是跨平台透明的**
macOS在沙盒应用中对 `CGWarpMouseCursorPosition`（相当于Windows的 `SetCursorPos`）有严格限制，过于频繁调用会被系统节流。Linux下的X11和Wayland对光标锁定的API完全不同，Wayland协议通过 `zwp_pointer_constraints_v1` 扩展实现锁定，而X11使用 `XGrabPointer()`。直接调用平台API而不通过抽象层（如SDL2、GLFW）极易导致跨平台行为不一致。
