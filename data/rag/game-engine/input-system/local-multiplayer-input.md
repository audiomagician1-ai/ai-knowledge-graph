# 本地多人输入

## 概述

本地多人输入（Local Multiplayer Input）是游戏引擎输入系统中专门处理多个玩家在同一台设备上同时进行游戏的机制，核心任务是将物理输入设备（手柄、键盘分区、触控分区等）与玩家逻辑角色进行绑定和隔离。与单人输入不同，本地多人输入必须解决"哪个输入来自哪个玩家"这一根本问题，并确保玩家1的按键操作绝对不会影响玩家2的角色。

这一技术在1990年代家用主机时代随着双手柄接口的普及而逐步标准化。早期的《超级马里奥兄弟3》（1988年，FC平台）依赖硬件层面的两个独立手柄端口实现双人输入，每个端口直接映射到固定玩家编号。现代游戏引擎如Unity（PlayerInput组件，2019年随Input System 1.0正式推出）和Unreal Engine（GamePad Player Index枚举）则在软件层面提供了完整的多玩家输入管理框架，支持热插拔、设备重新分配和最多同时8个玩家的本地管理。

本地多人输入的正确实现直接影响分屏格斗游戏、合作冒险游戏和派对游戏等品类的核心体验。若设备分配逻辑出现错误，玩家1的摇杆输入可能被错误路由给玩家2，造成控制混乱，这是本地多人游戏最常见的严重Bug来源之一。

## 核心原理

### 设备到玩家槽位的映射

本地多人输入的基础是维护一张**设备-玩家映射表**，将每个已连接的物理输入设备分配给一个玩家槽位（Player Slot，通常从索引0开始编号）。该映射表的数据结构可以简化为：

```
PlayerSlot[i] → InputDevice deviceHandle
```

当第3号手柄产生按键事件时，系统先查询该手柄的 `deviceId`，再通过映射表反查到对应的玩家槽位索引，最后将该事件分发到该槽位绑定的玩家角色控制器。整个路由过程必须在每一帧的输入采集阶段完成，通常发生在引擎主循环的 `PollInput()` 步骤中。

### 设备加入与退出协议

多人游戏通常需要实现**加入流程（Join Flow）**：玩家按下特定按键（如Xbox手柄的Start键或任意键）向游戏宣告"我想加入"。Unity Input System对此提供了 `PlayerInputManager` 组件，其 `joinBehavior` 属性可配置为 `JoinPlayersWhenButtonIsPressed`，此时系统监听所有未绑定设备上的任意按键，一旦触发即自动创建新玩家实例并完成设备绑定。

设备断开（手柄电量耗尽或线缆拔出）同样需要特殊处理。最常见的策略有两种：
- **暂停策略**：检测到设备断开时自动暂停游戏并弹出重连提示，适合竞技类游戏。
- **踢出策略**：将对应玩家标记为非活跃状态，适合派对游戏。

Unreal Engine通过 `OnControllerConnectionChange` 委托提供设备断开回调，开发者需在此处实现上述策略的具体逻辑。

### 键盘分区输入（单键盘双人）

当游戏只有键盘可用时，可通过**键盘分区（Keyboard Split）**让两个玩家共享一个键盘。典型方案是：
- 玩家1使用 WASD + 空格键
- 玩家2使用方向键 + Enter键

这要求输入绑定层为每个玩家槽位维护独立的键位映射（Action Map），而不是全局共享同一套映射。Unity Input System中通过为不同 `PlayerInput` 实例分配不同的 `InputActionAsset` 来实现，或在同一Asset内创建多套 `Control Scheme`（如 "Keyboard&Mouse Left" 和 "Keyboard&Mouse Right"）。

### 分屏视口与输入区域的对应关系

分屏（Split Screen）是本地多人游戏的典型视觉表现，但视口分割本身属于渲染层面的概念，不直接等同于输入分配。然而在触控屏设备上，需要额外建立**触控区域到玩家槽位的映射**：左半屏触控事件路由给玩家1，右半屏触控事件路由给玩家2。触控点的X坐标除以屏幕宽度可得归一化位置，以0.5为分界点确定归属玩家。Unity中可通过在 `TouchControl` 的 `interactions` 层面添加位置过滤器实现此需求。

## 实际应用

**格斗游戏（如《街头霸王》系列）**：两个玩家各绑定一个手柄，系统在 `Player1ActionMap` 和 `Player2ActionMap` 中分别维护 `Light Punch`、`Heavy Kick` 等动作，两套映射完全独立。玩家1手柄的"圆圈键"触发玩家1的出拳动画，与玩家2的同名按键产生的事件无任何交叉。

**合作冒险游戏（如《It Takes Two》）**：游戏支持同屏双人和分屏双人两种模式。在分屏模式下，视口在水平方向一分为二，但输入系统的玩家槽位分配在进入分屏之前已经完成，切换到分屏模式仅改变摄像机渲染目标，不改变已建立的设备-玩家映射。

**派对游戏（如《马里奥派对》）**：最多支持4个手柄同时接入，且允许途中加入和退出。游戏大厅界面通常展示4个玩家槽位，已绑定设备的槽位显示对应手柄图标，空槽位显示"按A键加入"提示，这正是上文提及的加入流程的典型UI实现。

## 常见误区

**误区1：手柄物理编号等于玩家编号**

操作系统给手柄分配的设备ID（如XInput的 `XUSER_INDEX_0` 到 `XUSER_INDEX_3`）是纯粹的硬件枚举顺序，与游戏内的玩家1/玩家2没有固定对应关系。如果玩家1使用的手柄因为插拔顺序而被系统分配到索引2，游戏必须通过加入流程或设备分配逻辑动态建立映射，而不能直接用设备索引作为玩家索引。

**误区2：分屏视口分割等于输入分割**

许多初学者认为设置了左右分屏之后，左侧摇杆输入自动属于玩家1，右侧属于玩家2。实际上，分屏是纯粹的渲染/摄像机概念，输入路由完全独立于视口位置。若不正确设置设备绑定，两个玩家可能同时看到"自己的"画面，但输入却都控制同一个角色。

**误区3：单手柄可以不做玩家槽位管理**

即使是单人游戏，若后续需要扩展为多人模式，从一开始就使用玩家槽位抽象（即使只有Slot 0）会大幅降低重构成本。直接将全局唯一输入Manager与角色控制器耦合，在改造为多人输入时往往需要对整个输入-角色通信链路进行重写。

## 知识关联

**前置概念：输入设备抽象**
本地多人输入建立在输入设备抽象之上。设备抽象层已经将物理手柄、键盘、触控屏统一表达为具有相同接口的逻辑设备对象，本地多人输入在此基础上增加了"将哪个逻辑设备分配给哪个玩家"的管理层。若没有设备抽象，直接操作原始硬件API，多设备管理的复杂度将呈指数级上升——开发者需要为XInput、DInput、HID等不同协议分别实现独立的多玩家路由逻辑。

**横向关联：输入重映射（Input Remapping）**
在多人场景中，每个玩家的按键重映射配置必须独立存储和应用。玩家1修改自己的"跳跃键"不应影响玩家2的同名动作绑定，这要求重映射系统与玩家槽位系统深度集成，每个槽位持有自己独立的重映射覆盖层（Override Layer）。
