# 输入延迟优化

## 概述

输入延迟（Input Latency）是指从玩家按下按键到屏幕上出现对应画面变化所经历的总时间。这条链路横跨硬件、操作系统、游戏引擎和显示器四个层级，任何一个环节的积累都会让玩家感受到操作"滞后"。对于竞技游戏而言，职业选手普遍认为超过 **50ms** 的输入延迟会明显影响反应判断，而格斗游戏社区通常将 **2帧（约33ms@60fps）** 作为可接受的上限阈值。

这一优化领域在2010年代随着竞技游戏的兴起而受到系统性关注。早期《街头霸王IV》移植版因输入延迟高达 **8帧** 而遭到格斗游戏社区的强烈批评，促使开发者开始将输入延迟作为独立的技术指标进行度量和优化。此后，《街头霸王V》将目标控制在 **4帧**，而《Guilty Gear Strive》则通过大量技术改造将本地延迟压缩至 **3帧**。

理解输入延迟优化的价值不仅在于竞技游戏体验，VR游戏对此要求更为苛刻——当头显追踪与画面更新之间的延迟超过 **20ms** 时，大多数用户会产生晕动症（motion sickness），因此VR领域将"运动到光子延迟"（Motion-to-Photon Latency）视为首要性能指标。

## 核心原理

### 全链路延迟分解模型

输入延迟由以下各阶段延迟线性叠加而成：

```
总延迟 = T_硬件 + T_驱动 + T_轮询 + T_逻辑帧 + T_渲染 + T_显示
```

- **T_硬件**：按键物理触发到信号发出，机械键盘约 **1ms**，薄膜键盘可达 **5ms**
- **T_驱动/轮询**：USB HID设备默认轮询率为 **125Hz（8ms间隔）**，高端游戏外设可提升至 **1000Hz（1ms间隔）**
- **T_逻辑帧**：游戏主循环处理输入的等待时间，60fps下最坏情况为 **16.7ms**
- **T_渲染**：GPU绘制当前帧所需时间，通常 **2-8ms**
- **T_显示**：显示器从接收信号到像素点亮的响应时间，加上扫描线延迟可达 **1-10ms**

典型配置下（USB 125Hz + 60fps游戏 + 普通显示器），全链路延迟轻易超过 **80ms**。

### 输入采集策略：轮询时机的优化

传统游戏引擎在帧开始时读取一次输入状态（称为"帧头采集"），这意味着帧中任何时刻按下的按键都要等到下一帧才被处理，造成最坏 **1帧**的额外延迟。

更优的策略是**帧末采集**（Late Latching）：在逻辑处理完成、即将提交渲染指令前的最后时刻再读取输入。这样输入到画面响应的窗口可以缩短至不足半帧。Valve的Source引擎和Epic的Unreal Engine 5均支持此模式，UE5中通过设置 `r.OneFrameThreadLag 0` 配合低延迟模式来实现帧末输入采集。

更进一步的技术是**异步输入采样**：将输入线程与游戏逻辑线程分离，以远高于帧率的频率（如每 **1ms**）持续读取设备状态，确保当帧提交时能使用最新的输入值。

### 渲染管线中的延迟来源

现代渲染管线通常采用"三缓冲"或"双缓冲 + Vsync"，这两种方案都会在渲染到显示之间引入额外的帧缓冲延迟：

- **双缓冲 + Vsync**：最坏情况下增加 **1帧** 延迟（约 16.7ms@60fps）
- **三缓冲**：提升流畅度但延迟最坏可达 **2帧**（约 33ms@60fps）

**NVIDIA Reflex** 和 **AMD Anti-Lag** 等技术通过精确控制CPU提交渲染指令的时机（即在GPU即将空闲时才提交），避免渲染队列积压，实测可将"渲染器延迟"降低 **30-50%**。NVIDIA官方数据显示，在《堡垒之夜》中启用Reflex后系统延迟从约 **55ms** 降低至约 **30ms**。

### 预测与补偿技术

当延迟无法从源头消除时，引擎可使用**输入预测**（Input Prediction）来视觉补偿。具体做法是基于当前输入状态预测下一帧的游戏世界状态，并立即渲染预测结果。若预测正确，玩家感受到的延迟接近零；若预测错误则进行回滚校正（Rollback）。

《龙珠斗士Z》和《街头霸王6》的网络同步使用了Rollback Netcode，其核心就是将本地输入延迟压缩至 **0帧**，通过预测+回滚来处理网络对战中的对手输入不确定性。

## 实际应用

**格斗游戏的帧数精确性要求**：《Guilty Gear Strive》开发团队在主机版优化中专门测量了PS4与PS5的输入延迟差异。PS5版通过利用DualSense控制器的 **250Hz USB轮询**（相比PS4的 **125Hz**）将延迟减少约 **4ms**，配合游戏内的帧末采集，最终实现3帧延迟目标。

**FPS游戏的鼠标响应优化**：《Valorant》要求游戏以 **128tick**（约7.8ms/帧）运行服务端，并在客户端实现异步输入读取，确保鼠标移动数据在每帧渲染前都以最高分辨率采集。Riot Games公开的技术博客中指出，仅将鼠标读取从帧头移至帧末就带来了约 **8ms** 的延迟改善。

**移动端触摸输入优化**：iOS设备的触摸采样率为 **120Hz**，而Android设备差异较大（60-240Hz不等）。Unity移动游戏开发中，可通过 `Input.GetTouch()` 的 `deltaPosition` 结合固定时间步长来插值触摸轨迹，补偿低采样率设备上的输入精度损失。

## 常见误区

**误区一：提高帧率就等于降低输入延迟**

帧率提升确实缩短了每帧等待时间（240fps下每帧约4.2ms），但如果引擎仍在帧头采集输入，输入到画面的延迟还包括整个渲染+显示链路。实测中，运行于 **60fps但启用帧末采集**的游戏，输入延迟可能低于运行于 **120fps但使用帧头采集+三缓冲**的游戏。帧率和输入延迟是可以独立优化的两个维度。

**误区二：关闭Vsync总是能减少延迟**

关闭Vsync确实消除了等待垂直同步信号的延迟，但同时引入画面撕裂（Tearing）。更重要的是，无限帧率模式下GPU满载会导致渲染队列积压（Queue Stuffing），反而增加延迟。正确做法是关闭Vsync同时配合**帧率上限**和**NVIDIA Reflex/AMD Anti-Lag**，避免队列积压。

**误区三：输入延迟是固定值**

实际上输入延迟是一个**分布范围**，不是固定数值。USB轮询的随机相位、帧渲染时间的波动、显示器扫描线位置的不同，都会导致每次按键的实际延迟在最小值和最大值之间随机分布。优化目标应同时包括降低**平均延迟**和缩小**延迟抖动（Jitter）**，后者对玩家的"手感一致性"同样重要。

## 知识关联

本文所讨论的各项技术建立在**输入系统概述**中介绍的设备轮询机制和事件队列模型之上——理解USB HID轮询协议和操作系统消息队列是分析T_驱动和T_轮询这两个延迟来源的前提。

输入延迟优化与**游戏主循环设计**紧密相关：帧末采集策略要求引擎在Update()阶段结束后、提交渲染命令前存在明确的"输入重读"钩子，这需要主循环在架构层面为此预留插入点。此外，本文提到的Rollback Netcode技术延伸至**网络同步**领域，是理解帧同步与状态同步延迟权衡的重要基础。
