# 运动输入

## 概述

运动输入（Motion Input）是指游戏引擎通过读取设备内置的物理传感器——主要是陀螺仪（Gyroscope）和加速度计（Accelerometer）——将玩家身体或设备的物理运动转化为游戏内可用控制信号的输入方式。与按键输入不同，运动输入捕获的不是离散的"按下/松开"状态，而是连续变化的六轴空间数据（三轴角速度 + 三轴线性加速度）。

运动输入在消费级游戏领域的普及始于2006年任天堂Wii的发布。Wii Remote内置加速度计，让玩家通过挥动手臂模拟网球击球等动作，首次将体感控制带入主流市场。2010年前后，索尼PS Move和微软Kinect相继出现，进一步扩展了体感控制的技术路线。现代移动设备（iOS/Android）和主机手柄（如Nintendo Switch Joy-Con、PS5 DualSense）已将陀螺仪与加速度计作为标配硬件。

运动输入在游戏引擎中的重要性体现在它能提供普通摇杆无法实现的精细指向控制。例如在第一人称射击游戏中，陀螺仪辅助瞄准可以实现比摇杆更精确的微调，《塞尔达传说：荒野之息》正是利用这一特性设计了需要倾斜设备解开的平衡谜题。

## 核心原理

### 加速度计的工作原理与数据格式

加速度计测量设备沿三个正交轴（X、Y、Z）的线性加速度，单位通常为 **m/s²** 或以重力加速度 **g（≈9.8 m/s²）** 为单位。其输出向量为：

```
a = (ax, ay, az)
```

当设备静止平放时，加速度计会读取到约 `(0, 0, 1g)` 的值，这是地球重力在Z轴上的分量。游戏引擎可以利用这个静止时的重力向量推算设备的倾斜角度（俯仰角Pitch和滚转角Roll），但**无法通过加速度计单独推算偏航角（Yaw）**，因为绕重力方向旋转不改变重力分量的分布。

### 陀螺仪的工作原理与积分漂移

陀螺仪测量设备绕三轴旋转的**角速度**，单位为度/秒（°/s）或弧度/秒（rad/s）。要从角速度得到绝对旋转角度，需要对时间积分：

```
θ(t) = θ(t-1) + ω × Δt
```

其中 `ω` 是当前帧的角速度，`Δt` 是帧间时间差。这个积分过程会引入**累积误差（漂移，Drift）**——即使设备完全静止，陀螺仪也会因微小的噪声信号持续积分而逐渐偏移。这是运动输入处理中最核心的技术难点之一。实践中通常使用**互补滤波器（Complementary Filter）** 或 **卡尔曼滤波器（Kalman Filter）** 将加速度计的低频稳定数据与陀螺仪的高频精准数据融合，公式形如：

```
θ_fused = α × (θ_gyro) + (1 - α) × θ_accel
```

其中 `α` 通常取 0.96–0.98，赋予陀螺仪更高权重以保留瞬间响应灵敏度。

### 游戏引擎中的运动输入API抽象

在游戏引擎的输入系统层面，运动输入数据通过专用API暴露给开发者。Unity引擎提供 `Input.gyro`（陀螺仪对象）和 `Input.acceleration`（加速度向量），开发者需要先调用 `Input.gyro.enabled = true` 显式启用陀螺仪，因为持续读取传感器会增加设备功耗。Unreal Engine则通过 `GetMotionValue()` 节点在蓝图系统中访问运动数据，返回包含倾斜（Tilt）、旋转速率（RotationRate）、重力（Gravity）和加速度（Acceleration）四组向量的结构体。

值得注意的是，不同平台对坐标系的定义并不统一——iOS以设备竖屏方向为基准，Android以横屏为基准，引擎通常在底层封装平台差异，但开发者在处理屏幕旋转时仍需手动映射坐标轴。

## 实际应用

**陀螺仪辅助瞄准**：《喷射战士2》（Splatoon 2）是陀螺仪运动输入在射击游戏中应用的典型案例。玩家用摇杆进行大范围粗略转向，同时通过倾斜手柄做精细微调。这种"双重输入叠加"模式已被证明在目标追踪精度上接近鼠标，远超纯摇杆控制。实现上，引擎将陀螺仪角速度直接映射为摄像机旋转增量，而非尝试重建绝对空间姿态，从而完全规避了漂移累积问题。

**设备倾斜控制方向**：赛车手游常用加速度计的X轴分量控制转向，将 `[-1g, +1g]` 的加速度范围线性映射到方向盘的 `[-30°, +30°]` 转角区间，同时设置死区（Dead Zone）约0.05g以过滤手持抖动。

**计步与摇晃检测**：部分RPG游戏通过检测加速度信号的周期性峰值（频率约1.5–2.5Hz对应正常步行节奏）实现计步功能，或通过检测加速度瞬时幅值超过阈值（如2g）触发特殊动作，例如《Pokemon GO》的孵蛋计步机制。

## 常见误区

**误区一：加速度计可以追踪设备的绝对空间位置**
很多初学者认为对加速度二次积分就能得到设备的移动距离。理论上 `x = ∫∫a dt²` 成立，但加速度计的噪声误差经过二次积分后会以平方速度发散，几秒内位置误差就会累积到米级。消费级惯性传感器**无法单独实现可靠的室内位置追踪**，需要结合光学定位（如PS VR的摄像头追踪）才能使用。

**误区二：所有平台的陀螺仪采样率相同**
不同设备的传感器采样率差异显著：iOS设备通过Core Motion可提供100Hz采样率，而部分低端Android设备仅有50Hz甚至更低。如果游戏逻辑按固定帧率（如60fps）读取运动数据而非使用传感器回调，在低采样率设备上会丢失高频运动信息，导致快速甩动时陀螺仪响应出现"跳变"。正确做法是以传感器自身频率回调采集数据，再在渲染帧中聚合。

**误区三：关闭陀螺仪辅助瞄准意味着移除运动输入代码**
在《塞尔达传说：旷野之息》的Switch版中，陀螺仪功能可以被玩家关闭，但引擎仍在持续读取传感器数据用于其他系统（如振动反馈的姿态感知）。开发者应区分"功能禁用"与"传感器停止采集"——频繁开关传感器采集会产生延迟，应保持传感器持续工作并在上层逻辑过滤不需要的数据流。

## 知识关联

运动输入建立在**输入设备抽象**的概念之上：正是因为引擎将物理传感器封装为统一的数据接口（加速度向量、角速度向量），开发者才无需关心底层MEMS传感器的硬件型号差异。输入设备抽象层负责将iOS的Core Motion、Android的SensorManager、以及各主机SDK的传感器API统一呈现为相同的数据结构，运动输入逻辑在此之上编写一次即可多平台运行。

运动输入与**触摸输入**存在天然互补关系——移动游戏常将两者组合，触摸屏处理UI交互与精确点击，加速度计/陀螺仪处理连续的方向与姿态控制，两套输入流在引擎的输入管理器中并行处理互不干扰。在VR/AR开发领域，运动输入的传感器融合算法（互补滤波、卡尔曼滤波）是头显姿态追踪的技术基础，理解这部分原理有助于后续学习六自由度（6DoF）追踪系统的设计。
