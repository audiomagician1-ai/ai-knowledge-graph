# 运动输入

## 概述

运动输入（Motion Input）是游戏引擎输入系统中专门处理物理运动传感器数据的子系统，负责采集并转换来自陀螺仪（Gyroscope）、加速度计（Accelerometer）和磁力计（Magnetometer）的原始信号，将其转化为游戏可用的旋转角度、倾斜方向或手势动作。与按键输入不同，运动输入提供的是连续的模拟量数据，每帧都会产生一组浮点向量。

运动输入在游戏中的大规模应用始于2006年任天堂发布的 Wii 遥控器（Wiimote），其内置的三轴加速度计让体感控制进入主流市场。2009年索尼 PlayStation Move 和2010年微软 Kinect 相继推出后，陀螺仪与加速度计的组合成为移动和主机平台的标准配置。如今 iOS 的 `CMMotionManager`、Android 的 `SensorManager` 以及 Unity 的 `Input.gyro` 和 `Input.acceleration` 接口都统一封装了这类传感器。

运动输入在移动游戏中尤为关键：赛车游戏通过倾斜设备控制方向，AR 应用依赖陀螺仪数据稳定叠加图层，VR 头显的头部追踪完全依赖运动传感器的低延迟姿态解算。

## 核心原理

### 1. 陀螺仪数据结构

陀螺仪测量设备绕三个轴旋转的**角速度**，单位为弧度/秒（rad/s）。游戏引擎每帧接收到的原始数据是一个三维向量 `(ωx, ωy, ωz)`，分别对应 Pitch（俯仰）、Yaw（偏航）、Roll（滚转）三个轴。

要从角速度恢复出绝对姿态角，需要对角速度进行时间积分：

```
θ(t) = θ(t-1) + ω × Δt
```

其中 `Δt` 是帧间时间间隔（单位秒）。这种积分方法称为**欧拉积分**，长时间累积会产生漂移误差，因此实际引擎会结合加速度计数据用互补滤波或卡尔曼滤波（Kalman Filter）修正。

### 2. 加速度计数据结构

加速度计测量设备所受的比力（Specific Force），单位为 m/s²，静止时会读到约 **9.8 m/s²** 的重力加速度，方向指向地面。游戏引擎将此向量归一化后，可以直接得出设备相对于地面的倾斜角度。

Unity 中 `Input.acceleration` 返回的是已归一化到 g 单位（1g = 9.81 m/s²）的三维向量。手机平放桌面、屏幕朝上时，该向量约为 `(0, 0, -1)`；竖直握持时约为 `(0, 1, 0)`。加速度计对低频、静态的倾斜姿态精确，但对快速抖动和冲击动作噪声很大。

### 3. 传感器融合：互补滤波

单独使用陀螺仪会漂移，单独使用加速度计会抖动，实际项目通常采用**互补滤波**（Complementary Filter）将两者融合：

```
θ_fused = α × (θ_gyro + ω × Δt) + (1 - α) × θ_accel
```

其中 `α` 通常取 **0.98**，表示陀螺仪数据占 98%，加速度计修正占 2%。这个公式在 Arduino/移动平台上被广泛使用，计算代价仅为一次加权平均，远低于完整卡尔曼滤波，但在长时间运动场景下仍足够准确。

### 4. 坐标系与平台差异

各平台运动传感器的坐标系定义不完全一致：iOS 以设备自然持握方向（竖屏）为基准，Z 轴垂直屏幕向外；Android 同样以竖屏为基准但 Y 轴朝上。Unity 统一将加速度数据转换到 Unity 世界坐标系，但陀螺仪的 `Input.gyro.attitude` 返回的是 `Quaternion`（四元数），代表设备在世界空间中的绝对朝向，需要乘以一个 **-90°绕X轴** 的修正四元数才能对齐 Unity 摄像机坐标系。

## 实际应用

**移动赛车游戏的倾斜转向**：读取 `Input.acceleration.x` 的值，映射到方向盘旋转角。通常设定 ±0.3g 为死区，±0.7g 为最大转向，超出范围裁剪（Clamp）。为避免抖动，需用低通滤波对加速度计数据做平滑：`smoothed = Mathf.Lerp(smoothed, raw, 0.15f)`，每帧更新一次。

**VR 头显的头部追踪**：Oculus Quest 和 PlayStation VR2 均内置六自由度（6DoF）IMU，陀螺仪采样率高达 **1000 Hz**，通过传感器融合输出 4×4 变换矩阵供渲染引擎每帧调用。延迟须控制在 20ms 以内才能避免晕动症。

**体感挥拍检测**：检测加速度计合力突然超过阈值（通常 **2g**）并伴随陀螺仪角速度峰值，即可判断玩家做出了挥动手腕的动作。Wii Sports 网球采用的正是此类双传感器阈值判断逻辑。

## 常见误区

**误区一：把加速度计当作速度或位移传感器直接使用**
加速度计测量的是比力而非纯加速度，对其积分得到速度、再积分得到位移会在数秒内产生数米级的累计漂移，这是因为微小的噪声经过二次积分会被时间平方放大。在没有外部参考（GPS、光学追踪）的情况下，单靠加速度计无法准确追踪绝对位置。

**误区二：陀螺仪精度永远优于加速度计**
陀螺仪在快速旋转（角速度超过量程，如消费级 MEMS 陀螺仪常见量程为 **±2000°/s**）时会饱和截断，此时积分结果完全错误。加速度计在设备静止或缓慢倾斜时反而更为精确，这正是互补滤波必须混合两者的原因。

**误区三：Unity 的 `Input.gyro.attitude` 可以直接赋给摄像机旋转**
直接赋值会导致摄像机旋转轴与屏幕方向不对齐，表现为画面倾斜 90°。正确做法是乘以修正四元数：`Quaternion.Euler(90, 0, 0) * Input.gyro.attitude`，同时还需要将四元数从右手系（iOS/Android传感器标准）转换为 Unity 的左手坐标系。

## 知识关联

**前置知识——输入设备抽象**：输入设备抽象层将硬件传感器封装为统一的轮询接口，运动输入正是在这一层之上工作的。理解设备抽象有助于解释为何同一段 `Input.acceleration` 代码可以在不同品牌手机上产生一致的坐标轴约定，而不需要开发者处理各厂商 SDK 的差异。

**延伸应用——姿态解算与四元数**：运动输入的融合算法输出的是四元数（Quaternion），后续需要学习四元数乘法和球面线性插值（SLERP）才能平滑地将传感器姿态应用到摄像机或角色骨骼上。VR 应用中的头部追踪预测（ATW/ASW 技术）也建立在对运动传感器数据的外推预测之上，是运动输入在高性能渲染管线中的进阶用途。
