# 运动输入

## 概述

运动输入（Motion Input）是指游戏引擎通过读取陀螺仪（Gyroscope）、加速度计（Accelerometer）和其他惯性传感器的数据，将物理设备的旋转与位移转换为游戏内可用控制信号的输入机制。与按键或摇杆不同，运动输入捕捉的是设备在三维空间中的姿态变化，而非离散按压或轴向偏移。

运动输入在游戏领域的大规模普及始于2006年任天堂 Wii 遥控器（Wiimote）的发布。Wiimote 内置三轴加速度计，使玩家可以通过真实挥舞动作打网球或投保龄球，打破了传统手柄操作的边界。此后 2009 年 Wii MotionPlus 配件追加了陀螺仪，弥补了纯加速度计无法检测旋转的缺陷。智能手机的普及进一步将陀螺仪和加速度计内置到每一台移动设备中，成为移动游戏不可忽视的输入维度。

运动输入的意义在于它提供了一个连续的六自由度（6DoF）数据流，覆盖线性加速度的 X/Y/Z 三轴与角速度的 Roll/Pitch/Yaw 三轴。这让游戏能够实现直觉化的方向瞄准、体感挥击和重力感应解谜等机制，而这些机制用传统离散输入极难等价还原。

## 核心原理

### 加速度计的工作方式

加速度计测量设备受到的比力（Specific Force），单位为 m/s²，通常以 g（1g ≈ 9.8 m/s²）表示量程。消费级设备的量程常见为 ±2g 至 ±16g。当设备静止平放时，加速度计会读到约 1g 的向上分量，这是重力加速度的反作用力。游戏引擎利用这一特性可以在静止状态下直接估算设备的倾斜角（Tilt Angle），公式为：

> `pitch = atan2(ay, sqrt(ax² + az²))`
> `roll  = atan2(-ax, az)`

其中 ax、ay、az 分别是三轴加速度读数。但在设备剧烈运动时，重力分量与运动加速度叠加，导致仅凭加速度计无法稳定解算姿态。

### 陀螺仪的工作方式

陀螺仪测量角速度（Angular Velocity），单位为度/秒（°/s）或弧度/秒（rad/s）。消费级 MEMS 陀螺仪的典型量程为 ±250°/s 至 ±2000°/s，采样率一般为 100–1000 Hz。游戏引擎对陀螺仪数据进行时间积分（Integrate over time）可得到旋转角度：

> `angle += gyro_rate × Δt`

这一方法在短时间内精度较高，但由于 MEMS 传感器存在固有的零偏漂移（Bias Drift），长时间积分会导致角度累积误差（Drift），使游戏内视角或姿态缓慢偏移。这也是单独依赖陀螺仪进行长时姿态追踪时必须处理的技术挑战。

### 互补滤波与传感器融合

为了兼顾加速度计的长期稳定性和陀螺仪的短期精度，游戏引擎通常使用互补滤波器（Complementary Filter）或卡尔曼滤波器（Kalman Filter）对两路传感器数据进行融合。互补滤波器的经典公式为：

> `angle = α × (angle + gyro × Δt) + (1 - α) × accel_angle`

其中 α 通常取 0.96–0.98，高频旋转信号由陀螺仪贡献，低频重力方向修正由加速度计贡献。Unity 引擎的 `Input.gyro` 和 `Input.acceleration` API 分别暴露这两路原始数据，开发者可以自行实现融合逻辑，也可调用平台层已融合好的 `Input.gyro.attitude`（四元数姿态）直接使用。

### 体感控制的手势识别

运动输入不仅提供原始数值，还可通过滑动窗口检测加速度突变来识别挥击手势。典型实现是当合成加速度向量的模（`|a| = sqrt(ax²+ay²+az²)`）在单帧内超过阈值（如 2.5g）时触发挥击事件。任天堂 Switch Joy-Con 手柄内置的 HD 震动与六轴传感器以 200 Hz 采样，正是利用这一原理实现了《1-2-Switch》中的挤奶、乒乓等细腻体感交互。

## 实际应用

**移动游戏方向控制**：《神庙逃亡》（Temple Run）系列使用加速度计的 Roll 轴（左右倾斜）控制角色转向，灵敏度通常映射为 ±30° 对应满量程转向输入，比虚拟摇杆提供更直觉的物理反馈。

**主机游戏精准瞄准**：《塞尔达传说：荒野之息》在 Nintendo Switch 上将陀螺仪用于弓箭精准瞄准的微调（Gyro Aim），玩家先用右摇杆粗略移动瞄准点，再通过倾斜手柄进行像素级精确修正。这种"粗调+微调"的双层输入设计已成为主机射击游戏的参考范式。

**AR 与重力解谜**：《纪念碑谷》类游戏的后继作品及众多 AR 应用使用融合后的四元数姿态将手机屏幕的朝向映射到虚拟摄像机，使玩家通过物理转动设备探索三维空间，而无需触摸任何虚拟按钮。

**PC 平台的陀螺仪辅助**：Steam 的 Gyro-to-Mouse 功能允许支持陀螺仪的手柄（如 DualSense、Switch Pro Controller）将旋转角速度直接映射为鼠标光标移动量，使 FPS 游戏在手柄上获得接近鼠标的精准度。

## 常见误区

**误区一：加速度计可以直接计算位移**。许多开发者认为对加速度积分两次即可得到设备位移，从而实现手持追踪。实际上，MEMS 加速度计的噪声经过二次积分后误差会以 t² 的速度爆炸性增长，仅数秒内累计位移误差便可达到米级，完全不可用。真正的位置追踪需要结合光学（如 PS Move 摄像头）或磁力计进行额外修正。

**误区二：陀螺仪数据可以直接当角度使用**。陀螺仪输出的是角速度（°/s），不是角度。若不进行时间积分而直接将读数视为角度偏移，会导致设备旋转速度越快、光标移动越快，静止时光标完全不动——这恰好是正确的速度映射行为，而非位置映射。开发者需要明确自己的游戏需要的是速率控制（如飞行游戏翻滚）还是绝对姿态控制（如 AR 摄像机朝向），两者的处理管线截然不同。

**误区三：所有平台的运动输入 API 行为一致**。iOS 的 `CMMotionManager` 默认输出已经过传感器融合的姿态四元数，而 Android 的原始 `Sensor.TYPE_GYROSCOPE` 输出未融合的角速度。Unity 的 `Input.gyro.attitude` 在 iOS 上可靠，在部分 Android 设备上存在坐标系差异（Android 使用右手坐标系，Z 轴朝上；Unity 使用左手坐标系），需要手动旋转四元数进行适配，通常需要乘以 `Quaternion.Euler(90, 0, 0)` 进行修正。

## 知识关联

运动输入建立在**输入设备抽象**的基础之上——输入设备抽象层负责将各平台（iOS CoreMotion、Android SensorManager、Nintendo Switch SDK、Sony DualSense SDK）的底层传感器 API 统一封装为引擎内部一致的数据格式，使运动输入模块无需关心硬件差异即可获取标准化的三轴数值或融合姿态。

在运动输入内部，原始传感器数据与滤波算法、坐标系转换、手势阈值检测共同构成一条完整的处理管线：原始采样 → 滤波融合 → 坐标系对齐 → 游戏逻辑映射。掌握这条管线上每个环节的数学含义（积分、互补滤波、四元数旋转），是在不同类型游戏中正确运用运动输入的前提。由于运动输入在当前课程体系中已是终端概念，其知识延伸方向指向更广泛的空间计算领域，包括 VR 头显的头部追踪（Head Tracking）和 XR 手部追踪（Hand Tracking），这些系统在原理上是运动输入的高维扩展。
