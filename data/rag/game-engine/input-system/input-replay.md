# 输入录制与回放

## 概述

输入录制与回放（Input Recording & Playback）是一种将玩家的原始输入序列保存为数据文件，并在之后以完全相同的顺序重新注入游戏引擎输入层的技术。其核心前提是**确定性执行**：只要初始游戏状态相同、输入序列相同，游戏世界的演进结果必须逐帧一致。这与视频录制截然不同——视频只保存像素输出，而输入录制保存的是"导致那些像素出现的操作序列"，数据量通常小 100 倍以上。

该技术的工程化应用最早可追溯至 1990 年代的街机格斗游戏演示模式（Attract Mode）。《毁灭战士》（DOOM，1993）内置了 `.lmp` 格式的 Demo 录制功能，将每 tic（约 35Hz）的玩家按键状态压缩记录，这是主流 PC 游戏中最早的系统性输入回放实现之一。现代游戏引擎如 Unreal Engine 5 的 `InputRecord` 子系统和 Unity 的 Input System Package 均内置了类似机制，主要服务于 QA 自动化测试、速通（Speedrun）验证和 AI 幽灵（Ghost）数据三大场景。

输入录制的价值在于它允许 QA 工程师将一个可复现的 Bug 编码为一段几 KB 的输入文件，而不是一段几 GB 的屏幕录像。同样地，赛车游戏中的"幽灵赛车"本质上是对手或自己最佳成绩的输入序列回放，引擎将其驱动一个不参与碰撞的幽灵对象，让玩家实时对比差距。

## 核心原理

### 确定性约束与时间戳

输入录制文件的最小结构是一个 `(frame_index, input_state)` 键值对列表。`input_state` 包含当帧所有轴（Axis）的浮点值和所有按键（Button）的 bool 状态。为保证回放确定性，引擎必须满足三个硬性条件：

1. **固定步长更新（Fixed Timestep）**：物理与逻辑帧必须以固定 dt（如 1/60 s）推进，不允许可变帧率影响游戏状态计算。
2. **伪随机数种子一致**：回放开始时必须将 RNG 种子重置为录制开始时的相同值，否则 AI 行为、粒子生成等随机过程将发散。
3. **初始状态快照**：录制文件头部通常存储一份精简的世界快照（World Snapshot），包含玩家初始位置、速度、生命值等关键变量，而非每帧都存储完整状态。

如果任何一条被破坏，即便在同一台机器上，回放也会在数秒内与原始录制发生可观测偏差，游戏界称之为"回放漂移（Desync）"。

### 输入数据的编码格式

一条典型的每帧输入记录可以表示为结构体：

```
struct InputFrame {
    uint32_t frame;          // 帧序号，从0开始
    uint16_t buttons;        // 位掩码，每位对应一个按键
    int16_t  axis[4];        // 左摇杆XY、右摇杆XY，定点数 [-32768, 32767]
    uint8_t  triggers[2];    // L2/R2，[0, 255]
};
```

将轴值从 `float` 降采样为 `int16_t` 是有意为之的精度损失——它消除了不同平台浮点运算的微小差异（如 x87 vs SSE 的 80-bit 中间精度问题），保证跨平台回放一致性。整帧大小约为 16 字节；以 60 Hz 录制 10 分钟的游戏仅需约 576 KB，远比视频紧凑。

对于稀疏输入（如 RTS 游戏中大多数帧无操作），可改用**增量编码**：只记录输入状态发生变化的帧，并在每条记录中存储该状态持续到下一条记录前。这能将文件大小再缩减 60%–90%。

### 回放注入机制

回放时，引擎将正常的硬件输入轮询（如 `SDL_GetGamepadState()`）替换为从文件读取 `InputFrame` 的虚拟函数。游戏逻辑层完全无感知——它收到的 `InputState` 对象与真实玩家操作时完全相同。这种设计要求输入系统在架构上将**采集层**与**消费层**解耦：

```
[硬件/文件] --> [InputCollector] --> [InputState Buffer] --> [游戏逻辑]
```

在 Unreal Engine 中，这通过替换 `UPlayerInput` 的 `ProcessInputStack()` 调用实现；Unity Input System 则通过将设备替换为 `InputDevice` 的子类 `InputEventTrace` 回放器来完成。

## 实际应用

### QA 自动化回归测试

QA 工程师遭遇崩溃或行为异常后，提交 Bug 时附带一个 `.inputrec` 文件和初始世界快照。CI/CD 流水线在每次代码提交后自动回放该文件，若回放再次触发相同崩溃，测试失败并阻断合并。《火箭联盟》（Rocket League）的开发商 Psyonix 曾公开介绍，他们使用确定性回放覆盖了数百个碰撞判定 Bug，每条 Bug 对应一个平均不超过 30 秒的输入录像。

### 幽灵赛车数据

《马力欧赛车》系列将幽灵数据（Ghost Data）存储为输入序列而非位置关键帧序列。回放时，幽灵车辆从相同起点出发，以相同输入驱动，物理引擎自然还原其轨迹。这比插值位置关键帧更精确，也更省存储：《马力欧赛车 8 豪华版》的单圈幽灵文件约为 4–8 KB。

### 速通验证

主流速通平台（如 Speedrun.com）对部分游戏要求提交输入录制文件（BIOS 版本、存档状态、输入序列）作为防作弊凭据。验证者在相同环境下回放文件，若结果一致则确认成绩有效。这对《超级马力欧 64》等具有帧级精确技巧的游戏尤为重要。

## 常见误区

### 误区一：输入录制等同于视频录制

许多初学者认为两者可以互换。实际上，视频录制是**输出驱动**的，不含任何游戏逻辑信息；输入录制是**输入驱动**的，不含像素信息。视频在任何游戏版本上都能播放，而输入录制对游戏版本极度敏感——哪怕修复了一个浮点计算的精度问题，也可能导致旧录制文件在 30 秒后完全偏离原始轨迹。

### 误区二：网络同步帧（Lockstep）与输入录制是同一机制

帧同步（Lockstep）网络模型的确也传输输入序列，但其目的是在多个客户端间实时同步，而非存储回放。输入录制不关心网络延迟，它是单向写入后离线读取的。两者虽共用"确定性"这一数学前提，但应用层的实现目标、错误处理逻辑和文件格式完全不同。

### 误区三：浮点精度问题可以忽略

新手常假设同一台机器上的回放天然是确定性的。实际上，若游戏使用多线程物理（如 PhysX 的多核调度），线程交错顺序可能因系统负载而变化，导致浮点累加结果不同。解决方案是回放期间强制单线程物理求解，或使用整数定点数学库（如 Box2D 的定点分支）替代浮点物理。

## 知识关联

输入录制与回放直接建立在**输入系统概述**中介绍的输入轮询模型之上——只有当输入被抽象为每帧一次的状态查询（而非事件回调）时，替换采集源才能做到对游戏逻辑完全透明。理解输入系统的分层架构（硬件层、抽象层、动作映射层）是正确设计录制注入点的前提，错误地在动作映射层之后注入会导致玩家配置（键位绑定）变更时录制文件失效。

该技术与**确定性物理模拟**和**网络帧同步**存在深度交叉：后两者同样依赖固定 dt 和 RNG 种子管理，QA 工程师在调试输入回放漂移时往往同时需要调试物理层的不确定性来源。在工具层，输入录制系统通常与**引擎的调试与分析（Profiling）工具链**集成，例如将输入回放与 CPU 性能采样同步，以复现性能毛刺。
