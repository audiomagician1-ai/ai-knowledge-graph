# 输入预测

## 概述

输入预测（Input Prediction）是网络游戏引擎中的一种延迟补偿技术，允许客户端在未收到服务器确认之前，先行本地执行玩家的输入操作并渲染结果。其核心假设是：玩家自己的输入指令极大概率会被服务器接受，因此客户端可以"乐观地"立即响应，而无需等待一次完整的网络往返（RTT，Round-Trip Time）。

该技术最早在1996年 Quake 的网络代码中由 John Carmack 实现并推广，称为"客户端预测"（Client-side Prediction）。在典型的100ms往返延迟下，如果没有客户端预测，玩家每次按下移动键都要等待100ms才能看到角色移动，操作手感极为迟滞。输入预测解决了这一根本性的延迟感知问题。

在现代多人游戏引擎中，Unreal Engine 的 `UCharacterMovementComponent` 和 Unity 的 Mirror/Netcode for GameObjects 框架都内置了输入预测机制。理解输入预测的工作流程，是实现流畅多人游戏体验的必要前提。

## 核心原理

### 客户端预测循环

输入预测的基本执行流程分为三个阶段：**预测（Predict）→ 提交（Submit）→ 校正（Reconcile）**。

每一帧，客户端将玩家输入打包为一个"输入命令帧"（Input Command Frame），包含时间戳、输入序列号（Sequence Number）和具体操作（如方向向量、按键状态）。客户端立即将该命令应用到本地角色状态上并渲染，同时将该命令发送给服务器。服务器接收后执行相同的物理/逻辑模拟，并将权威结果（Authoritative State）连同对应序列号一起回传给客户端。

客户端维护一个"未确认输入缓冲区"（Unacknowledged Input Buffer），存储所有已发出但尚未被服务器确认的命令帧。一旦收到服务器的权威状态，客户端执行校正。

### 状态校正与重放（Reconciliation & Replay）

当服务器返回某序列号 `N` 对应的权威位置时，客户端执行如下步骤：

1. 将本地状态回滚（Rollback）到序列号 `N` 时刻的状态
2. 清除缓冲区中序列号 ≤ N 的所有已确认帧
3. 将缓冲区中序列号 > N 的所有未确认命令帧**重新模拟一遍**（Replay）
4. 若重放后的位置与当前渲染位置偏差超过阈值（例如 Quake 3 中使用约 1 单位距离），则执行瞬移或平滑插值修正

这一过程保证了：即使网络出现延迟，最终显示的状态始终以服务器权威结果为准，同时视觉上的抖动被最小化。

### 预测误差的来源

预测误差（Misprediction）主要来自以下几类情况：

- **碰撞阻挡**：客户端预测角色可以前进，但服务器判定前方有其他玩家（移动中的障碍物），导致位置不一致
- **服务器拒绝指令**：角色进入眩晕/冻结状态时，服务器忽略移动命令，而客户端不知道该状态变化
- **浮点精度差异**：客户端与服务器使用不同平台或优化级别编译，累积浮点误差导致模拟结果逐渐漂移

为了减少最后一类误差，Valve 的 Source 引擎在服务器和客户端均使用固定步长（`sv_tickrate`，默认64tick或128tick）进行物理模拟，确保相同输入在相同步长下产生完全一致的结果。

## 实际应用

### 第一人称射击游戏（FPS）

在 CS:GO 的128tick服务器中，服务器每秒处理128次物理帧，每帧约7.8ms。客户端预测使玩家的移动、跳跃等操作立即生效，而非等待至少7.8ms的服务器确认。当网络RTT为60ms时，未确认缓冲区中平均积压约8帧的输入命令，客户端需在收到服务器回包后对这8帧进行重放校正。

### 格斗/平台游戏的预测限制

对于需要精确帧判定的游戏，输入预测的重放机制可能造成视觉上的"橡皮筋效应"（Rubber-banding）。因此《街头霸王5》的联网模式改为采用回滚网络代码（Rollback Netcode，如 GGPO），这是输入预测思想的一个变体——对**双方**的输入都进行预测，而非只预测本地玩家。

### Unreal Engine 中的实现

在 Unreal Engine 5 中，`FSavedMove_Character` 结构体负责存储每一帧的输入快照，包含速度、加速度、旋转等字段。`UCharacterMovementComponent::ReplicateMoveToServer()` 方法将本地预测结果与服务器校正结果进行对比，误差超过 `KINDA_SMALL_NUMBER`（1e-4）时触发重放。

## 常见误区

### 误区一：输入预测会使作弊更容易

有些开发者担心，客户端自行执行运动逻辑意味着玩家可以修改客户端绕过服务器限制。实际上，输入预测的核心是"预测但不授权"——客户端的预测结果仅用于本地渲染，服务器始终重新模拟并返回权威结果。如果客户端伪造位置，服务器校正会将其纠正回合法位置。真正的反作弊压力在于服务器端的合法性校验，而非预测逻辑本身。

### 误区二：提高服务器Tick率可以替代客户端预测

提高Tick率（如从64tick升至128tick）缩短了每次服务器校正的延迟粒度，但无法消除网络RTT引入的延迟。即使服务器每秒处理1000帧，RTT为100ms的玩家仍然需要等待100ms才能获得服务器确认。客户端预测解决的是RTT延迟感知问题，而Tick率提升解决的是模拟精度问题，两者作用维度不同，不可互相替代。

### 误区三：所有游戏状态都应该进行预测

在实践中，只有玩家**直接控制**的本地角色适合进行输入预测。对于其他玩家、NPC、投射物等实体，客户端通常使用**实体插值**（Entity Interpolation）而非输入预测，因为客户端无法得知这些实体的未来输入，盲目预测会引入严重误差。混淆两种技术的适用对象，是多人网络游戏开发中最常见的架构错误之一。

## 知识关联

输入预测建立在**输入系统概述**所描述的输入采集与帧同步机制之上——只有当输入被封装为带时间戳的命令帧后，未确认输入缓冲区的序列号管理才能正确工作。理解输入事件的轮询（Polling）与回调（Callback）差异，直接影响预测循环中命令帧的生成粒度。

与输入预测紧密配合的相关技术包括：**实体插值**（用于处理远程玩家的平滑表现）、**延迟补偿射线检测**（Lag Compensation，服务器将碰撞检测回滚到客户端开枪时刻以确保命中判定公平性），以及**确定性物理模拟**（Deterministic Physics，保证相同输入在客户端和服务器产生完全一致的数值结果，是预测校正误差最小化的前提条件）。
