# 输入预测

## 概述

输入预测（Input Prediction）是网络游戏引擎中用于消除网络延迟对操控体验负面影响的技术机制。其核心思想是：客户端不等待服务器确认，而是立即将本地玩家的输入应用到本地游戏状态，让角色动作"看起来"即时响应，同时后台继续与服务器进行状态同步。没有输入预测的网络游戏，每一次按键都必须等待往返延迟（RTT）之后才能看到效果——在50ms延迟下按下跳跃键，玩家要等至少50ms才能看到角色离地，这种延迟感在格斗游戏或FPS中是完全不可接受的。

输入预测技术的系统化应用最早可追溯到Valve于1996年发布的《Quake》引擎，其cl_cmdrate和cl_updaterate参数体系奠定了基础架构。到2004年，Valve在Half-Life 2的Source引擎中将这套机制正式命名为"客户端预测（Client-Side Prediction）"，并在官方技术文档中详细公开，成为后来无数引擎参考的标准实现。此后Unreal Engine和Unity等主流引擎均内置了类似系统（分别称为CharacterMovement预测和Netcode Prediction）。

输入预测之所以关键，在于它直接决定"手感"——即玩家按下操作键到屏幕反馈之间的感知延迟（Perceived Latency）。研究显示，人类对输入延迟的感知阈值约为100ms，超过该值玩家会明显感到"卡顿"或"不跟手"。输入预测将感知延迟压缩至本地帧时间（通常16ms~33ms），从根本上解决了该体验问题。

## 核心原理

### 预测执行流程

输入预测的基本执行步骤如下：

1. **本地采集输入**：在当前帧采集键盘、手柄等原始输入，生成带时间戳和序列号的输入命令包（Input Command Packet），格式通常为 `{seq, tick, moveForward, moveRight, yaw, pitch, jump}`。
2. **本地立即模拟**：客户端使用与服务器完全相同的物理/运动逻辑，将该输入命令应用到本地预测状态，立即渲染新位置。
3. **发送至服务器**：同一输入命令包异步发送给服务器，服务器在收到后用权威逻辑重新计算状态。
4. **接收权威状态**：服务器返回经过验证的权威状态（包含服务器处理的最后序列号 `lastProcessedSeq`）。
5. **对比与回滚重放**：客户端用权威状态替换对应时刻的本地状态，然后将所有序列号大于 `lastProcessedSeq` 的未确认输入命令**重新应用一遍**，得到新的当前预测状态。

### 预测缓冲区与命令序列管理

客户端必须在本地维护一个**未确认命令环形缓冲区**（Unacknowledged Command Buffer），存储所有已发送但尚未被服务器确认的输入命令。缓冲区容量通常取决于最大可接受延迟：

```
bufferSize = ceil(maxRTT_ms / tickInterval_ms)
```

例如，服务器以64 tick运行（tickInterval = 15.6ms），最大容忍RTT为200ms，则缓冲区至少需存储 `ceil(200 / 15.6) ≈ 13` 条命令。当服务器确认序列号N时，客户端丢弃所有序列号 ≤ N 的命令，仅保留后续未确认命令用于重放。

### 预测误差与校正（Reconciliation）

当服务器权威状态与客户端预测状态之间存在差异时，称为**预测误差（Prediction Error）**。差异来源主要有两类：一是服务器端存在其他玩家碰撞、技能干扰等客户端无法预知的事件；二是客户端与服务器运动逻辑存在浮点精度差异（不同平台累积误差可达数毫米）。

校正策略分为两类：
- **硬校正（Hard Correction）**：直接将角色位置跳变到服务器权威位置。实现简单但会产生明显的位置跳变闪烁，仅在误差超过阈值（如0.5米）时触发。
- **软校正（Smooth Correction）**：将预测位置与权威位置的差值通过线性插值或指数衰减在数帧内平滑过渡，典型公式为：

```
correctedPos = predictedPos + (authorityPos - predictedPos) * smoothFactor
```

其中 `smoothFactor` 通常取 `0.2~0.4`，表示每帧消化20%~40%的误差，约5~10帧完成全部校正。

## 实际应用

**FPS游戏中的角色移动预测**：《CS:GO》与《Valorant》均在客户端完整复制服务器的角色移动逻辑（包括加速度、摩擦系数、空中控制参数）。玩家按下W键后，本地角色立即以权威速度移动，同时命令包发往服务器。若服务器因被对手子弹击中而修改了速度状态，则下一次状态同步会触发软校正。

**格斗游戏的帧级输入预测**：《街头霸王6》采用的GGPO回滚网络码（Rollback Netcode）是输入预测的极端形式：预测对手输入（通常假设对手"持续上一帧的输入"），渲染一帧，若收到真实输入后发现预测错误，则**回滚整个游戏状态**至误差帧并重新模拟。GGPO最多可预测8帧，约133ms（60fps下），允许高延迟环境下仍保持流畅对战。

**Unity NetCode for GameObjects中的实现**：Unity 2022版本的NGO（Netcode for GameObjects）提供`NetworkTransform`组件，可配置`InterpolatePosition`和`Extrapolate`选项。开发者需在`NetworkBehaviour`子类中重写`OnNetworkTransformInterpolated`回调来实现自定义预测逻辑。

## 常见误区

**误区一：输入预测等于延迟隐藏**

部分开发者认为输入预测只是"视觉欺骗"，实际上它是真实的物理模拟——客户端确实用与服务器相同的运动方程运算出了正确位置，服务器只是做权威验证。仅当发生碰撞等不可预知事件时，本地模拟结果才会被修正。把它理解为"延迟隐藏"会导致开发者跳过运动逻辑同步，造成大量预测误差。

**误区二：只要做了输入预测，延迟就完全透明**

输入预测能消除本地输入响应延迟，但无法消除**玩家间交互的延迟感知**——例如射击其他玩家时，对方的位置仍受延迟影响（需要另外依靠服务器端倒带/延迟补偿机制解决）。混淆这两个问题会导致开发者以为实现了输入预测就万事大吉，忽略了弹道判定不准的问题。

**误区三：预测频率越高越好**

提高客户端发送输入命令的频率（cmdrate）确实能减少服务器处理延迟，但超过服务器tick率后毫无意义，反而增加带宽消耗。例如64-tick服务器上，cmdrate超过64（即每包间隔＜15.6ms）的输入命令会被服务器合并处理，额外的命令包纯属浪费。

## 知识关联

**前置依赖——输入系统概述**：输入预测的工作基础是输入系统能以精确时间戳记录每帧的原始输入状态（按键状态位图、轴值等），并将这些状态封装成可序列化的命令结构体发往网络层。若底层输入采集存在帧级抖动或时间戳不准确，预测序列号对齐将出错。

**关联概念——服务器端延迟补偿（Lag Compensation）**：输入预测解决的是"我的操作看起来即时"，而延迟补偿解决的是"我打到的位置在服务器上是否有效"。两者协同工作：输入预测负责本地角色，延迟补偿负责对其他玩家命中判定的时间回溯。在Source引擎中，延迟补偿最大回溯时间由`sv_maxunlag`控制（默认1.0秒），超过该值的高延迟玩家无法获得命中补偿。

**关联概念——实体插值（Entity Interpolation）**：其他玩家（非本地）的位置通过插值而非预测来平滑显示，插值缓冲区通常为100ms，与输入预测的即时显示策略形成对比。两者共同构成完整的网络表现层。
