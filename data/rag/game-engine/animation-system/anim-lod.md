# 动画LOD

## 概述

动画LOD（Level of Detail，细节层级）是游戏引擎动画系统中根据角色在屏幕上的显示大小或与摄像机的距离，自动降低动画计算复杂度的技术。当一个角色占据屏幕面积不足1%时，为其执行完整的逆向运动学（IK）求解和64根骨骼的完整更新是严重的性能浪费——动画LOD正是解决这一矛盾的机制。

该技术最早在PS2/Xbox时代（2000年代初）因CPU性能瓶颈而被广泛采用。当时Bungie为《光环2》中的大规模战场场景设计了三级动画LOD方案，使战场中同屏敌兵数量从约20个提升到50个以上。现代游戏引擎如Unreal Engine 5和Unity均内置了动画LOD框架，开发者可直接在编辑器中配置各级阈值。

动画LOD之所以重要，在于动画更新在CPU帧预算中往往占据15%到30%的份额。一个开放世界游戏同屏可能存在数百个NPC，若每个NPC都执行完整骨骼姿态计算，帧率将无法维持。通过LOD分级，距离较远的角色可以以1/4甚至更低的频率更新动画，或完全切换为预烘焙的顶点动画，从而在视觉损失极小的前提下大幅降低计算开销。

## 核心原理

### 距离与屏幕尺寸的驱动方式

动画LOD的触发条件有两种主要度量方式：**世界空间距离**和**屏幕覆盖率（Screen Coverage）**。

- **世界空间距离**：以摄像机到角色骨骼根节点（通常是Hips或Root骨骼）的欧氏距离为基准，设定阈值如LOD0 = 0~10米、LOD1 = 10~30米、LOD2 = 30~60米、LOD3 = 60米以上。
- **屏幕覆盖率**：计算角色包围盒投影到屏幕后占据的像素比例，例如覆盖率 > 5%为LOD0，0.5%~5%为LOD1，< 0.5%为LOD2。

屏幕覆盖率方式更精准，因为它同时考虑了距离和视场角（FOV），在FOV变化时不会出现距离方法的误判。Unreal Engine的骨骼网格体组件默认使用屏幕尺寸（Screen Size）阈值，其计算公式为：

```
ScreenSize = BoundingSphereRadius / (Distance × tan(FOV/2))
```

其中`BoundingSphereRadius`为角色包围球半径，`Distance`为摄像机距离，`FOV`为视场角。

### 各LOD级别的动画简化策略

不同LOD级别采用不同的简化手段，级别越高（距离越远）简化程度越深：

**LOD0（近距离，完整计算）**：执行全骨骼更新、IK求解（手部/足部接触）、布料模拟、面部BlendShape、程序化动画（如随风摆动的头发）。骨骼数量通常为50~100根。

**LOD1（中等距离）**：禁用IK和布料，面部动画停止更新，骨骼数量削减至30~50根（移除末端手指骨骼、面部精细骨骼）。动画更新频率可保持30Hz不变，或根据帧预算降至20Hz。

**LOD2（较远距离）**：仅保留躯干、四肢主要骨骼（10~20根），动画更新频率降至10~15Hz，使用插值填充帧间缺失姿态。程序化细节（脚步IK、头部朝向追踪）全部关闭。

**LOD3（极远距离）**：完全切换为**顶点动画纹理（Vertex Animation Texture，VAT）**，预先将角色动画烘焙为贴图序列，在Shader中采样还原位移，CPU完全不参与动画计算。更新频率可低至8Hz。

### 更新频率节流（Tick Rate Throttling）

骨骼动画LOD中另一个关键机制是**动画节流（Animation Throttling）**：在较低LOD级别，不必每帧都执行完整的动画图（AnimGraph）求值，而是每N帧求值一次，中间帧使用上一帧姿态或线性插值。

Unity的`Animator`组件通过`cullingMode`属性控制此行为：
- `AlwaysAnimate`：始终完整计算
- `CullUpdateTransforms`：不在视野内时停止骨骼Transform写入
- `CullCompletely`：不在视野内时完全停止所有动画计算

Unreal Engine则通过`URO（Update Rate Optimizations）`系统实现类似功能，可按距离配置每秒最大更新次数。

## 实际应用

**开放世界NPC群体**：《刺客信条：奥德赛》的城市场景中，街道上的普通市民在超过25米后进入LOD1状态，50米后进入LOD2，此时市民的行走动画仅保留骨盆、大腿、小腿、上臂4个骨骼节点的计算，实现了同屏300+市民的流畅渲染。

**大规模战斗场景**：策略类游戏如《全面战争：三国》的兵团中，单兵距离超过40米后自动切换到仅8根骨骼的简化骨架，动画更新频率降至10fps，而玩家的关注焦点在于阵型层面，个体动画精度的下降几乎察觉不到。

**移动端游戏**：《原神》在iOS/Android端对距离超过15米的角色启用骨骼压缩，手指骨骼（每只手5根手指×3节=15根骨骼）全部停用，这一项优化在中低端设备上节省约8%的CPU帧时间。

## 常见误区

**误区一：LOD切换会导致明显的动画"跳变"**

很多初学者担心LOD级别切换时角色动画会出现突变。实际上，正确实现的动画LOD会在切换时进行姿态混合过渡（通常0.1~0.3秒），且骨骼数量减少是通过将精细骨骼绑定到父骨骼上实现的，而非直接删除，所以几何体不会塌陷变形。只要阈值设置合理，切换过渡在正常游玩时完全不可见。

**误区二：动画LOD和网格体LOD（Mesh LOD）是同一回事**

网格体LOD降低的是顶点数量和三角面数量（影响GPU），而动画LOD降低的是骨骼计算量和更新频率（影响CPU）。二者独立配置，级别不必对齐——一个角色可以在网格体切换到LOD2时仍保持动画LOD1，或反之。混淆二者会导致优化策略错位：GPU瓶颈时应优先优化网格体LOD，CPU动画瓶颈时才应收紧动画LOD阈值。

**误区三：LOD3的顶点动画纹理（VAT）适合所有角色**

VAT在Shader端还原动画时，每帧采样纹理的开销固定，但它无法响应运行时的任何动态变化——碰撞反应、布娃娃物理、程序化IK均不可用。因此VAT只适合完全被动的背景角色（如远处的观众人群），对需要与玩家交互的NPC，即使距离很远也不应使用LOD3切换到VAT模式。

## 知识关联

学习动画LOD需要先理解**骨骼动画系统**的基本结构——骨骼层级、蒙皮权重和动画图的求值流程，因为LOD的简化策略（如削减骨骼数量、禁用IK节点）直接操作的正是这些底层组件。不理解骨骼动画的更新管线，就无法判断某个LOD级别的节省量是否合理。

动画LOD的配置与**角色着色器**和**骨骼网格体LOD（Skeletal Mesh LOD）**紧密协作：在同一距离段，通常需要同步降低网格体面数和骨骼动画精度，才能同时释放GPU和CPU压力。此外，**动画压缩**（Animation Compression）技术与动画LOD互补——前者在磁盘和内存层面压缩关键帧数据，后者在运行时减少动画求值次数，两者结合才能达到最佳的内存与性能平衡。
