# 动画压缩

## 概述

动画压缩是游戏引擎动画系统中将骨骼动画数据进行精简存储的技术集合，其目标是在保持视觉效果可接受的前提下，大幅降低动画片段（Animation Clip）所占用的内存和磁盘空间。一个典型的人形角色骨架含有60至100根骨骼，每块骨骼在每帧记录位移、旋转、缩放三种通道的关键帧数据，若以32位浮点数存储60fps的10秒动画，原始数据量轻易超过数十MB，因此压缩是生产环境中不可省略的步骤。

动画压缩技术在游戏工业中随硬件限制的演变而成熟。2000年代早期主机内存不足128MB，开发者已开始大规模使用线性插值关键帧精简方案；2010年后移动平台兴起，内存更加紧张，促使了量化（Quantization）方案的普及。2019年，育碧蒙特利尔工作室开源了专门针对游戏动画的压缩库 **ACL（Animation Compression Library）**，成为目前工业界使用最广泛的参考实现，Unreal Engine 5从5.0版本开始将ACL作为默认动画压缩后端。

动画压缩之所以重要，不仅因为它节省内存，还因为压缩方式直接影响运行时的解压（Decompression）开销。游戏主循环每帧需要对数十个角色同时进行骨骼采样，解压速度若过慢会成为CPU瓶颈。优秀的压缩方案必须同时满足高压缩比与低解压延迟两个约束。

## 核心原理

### 曲线简化（Curve Simplification）

曲线简化通过丢弃冗余关键帧来缩减数据量，最常用的算法是 **Ramer–Douglas–Peucker（RDP）** 算法。该算法对动画曲线按误差阈值递归删除不必要的中间关键帧：若某帧的真实值与相邻两帧线性插值结果的偏差小于阈值 ε，则删除该帧。对于旋转通道，ε 的单位通常为角度（如0.5°），对于位移通道则为世界空间距离（如0.001米）。

Unity Editor的动画导入器中，"Rotation Error"默认值为0.5度，"Position Error"默认值为0.5毫米，正是RDP阈值的直接体现。曲线简化的压缩比取决于动画本身的频率特性：缓慢移动的骨骼可压缩90%以上的关键帧，而高频抖动骨骼（如布料、头发）压缩效果有限。

### 数值量化（Quantization）

量化将浮点数映射到低位整数以节省每个关键帧的存储字节数。对于旋转数据，四元数的四个分量值域在 [-1, 1] 之间，可以用16位整数（Short）表示，精度约为 1/32768 ≈ 0.00003，引入的误差远小于人眼可分辨阈值。

旋转数据常见的进一步优化是**"最小三分量"（Smallest Three）** 编码：由于单位四元数满足 w² + x² + y² + z² = 1，可丢弃绝对值最大的分量，只存储其余三个，并用2位记录被丢弃的分量索引。这样一个旋转从128位（4×32位）压缩至约48位（3×15位+2位符号+2位索引），压缩比接近2.67倍。

位移通道量化通常先计算该骨骼在动画中的值域范围 [min, max]，然后将实际值线性映射到 [0, 2ⁿ-1] 的整数范围，n 常取16或24位。解压时通过反向映射公式还原：

> **value = min + (quantized / (2ⁿ - 1)) × (max - min)**

其中 `quantized` 为存储的整数，`n` 为量化位数，`min` 和 `max` 为该通道的范围边界，作为元数据随动画一起存储。

### ACL库的分段压缩策略

ACL（Animation Compression Library）采用**分段（Segment）** 架构：将一条动画在时间轴上切分为若干小段（默认每段约16帧），对每段独立计算值域并量化。相比全局量化，分段量化能更精确地适应局部运动幅度，从而在相同位误数下获得更低的误差，或在相同误差下使用更少位数。

ACL还实现了**范围缩减（Range Reduction）**：在分段内先计算每根骨骼的局部最小/最大值，将数值归一化到 [0, 1] 后再量化，确保每一段的动态范围都被充分利用。ACL官方基准测试数据显示，在误差阈值0.01mm的条件下，对标准人形动画数据集，ACL的平均压缩后大小约为原始数据的2%到5%，同时每骨骼采样时间低于100纳秒（在现代x86-64 CPU上）。

## 实际应用

**Unreal Engine 5中的ACL集成：** 在UE5的动画序列（Animation Sequence）资产中，可在"压缩设置"中选择"ACL"压缩方案，并暴露"默认虚拟顶点距离"（Default Virtual Vertex Distance）参数（默认1.0cm）作为误差阈值。开发者可通过UE5的"Animation Compression"命令行工具批量重压缩项目内所有动画，并查看压缩前后的大小对比报告。

**Unity的动画压缩等级：** Unity的AnimationClip在导入设置中提供Optimal、Keyframe Reduction、Off三档。选择Optimal时，Unity自动对每条曲线按误差阈值分别应用密集关键帧删除和浮点量化，并可在Inspector中分别调整旋转、位移、缩放三个通道的误差容限，实现精细化控制。

**移动游戏的16位骨骼限制：** 部分移动平台GPU的蒙皮着色器只支持16位骨骼索引，开发者在压缩流程中也需同步控制骨骼数量，与动画压缩结合进行整体内存预算管理。

## 常见误区

**误区一：压缩阈值越小越好。** 许多开发者为了"安全"将误差阈值设为极小值（如0.0001°），但过小的阈值会导致大量原本可删除的关键帧被保留，压缩率大幅下降，动画文件反而比必要值大5到10倍。正确做法是在目标平台分辨率下做视觉验证，逐步放大阈值直到出现可见瑕疵，取上一档阈值作为最终设置。

**误区二：量化误差可以完全叠加到蒙皮误差中一并忽略。** 量化误差在层级骨架中会沿骨骼链传播并累积：根骨骼的旋转误差 θ 在经过3层子骨骼后，末端效应器的位置误差最大可达 θ × L（L为骨骼链总长）。因此靠近根部的骨骼应使用更高的量化精度，而末端小骨骼（如手指最末节）可容忍更大误差——ACL的"adaptive bitrate"功能正是基于这一原理为不同骨骼自动分配不同位宽。

**误区三：ACL压缩只在磁盘上生效。** ACL的压缩数据在运行时同样以压缩格式驻留内存，采样时对压缩数据直接进行解压计算，而非先解压全部数据再采样。这意味着运行时内存占用与磁盘大小几乎相同，这与传统"运行时解压后缓存"方案有本质区别。

## 知识关联

动画压缩以**动画片段**（Animation Clip）为直接操作对象：动画片段定义了每根骨骼随时间变化的浮点曲线，压缩算法正是对这些曲线施加简化和量化处理。理解动画片段中旋转通道使用四元数表示、位移通道使用向量3表示的数据结构，是理解量化位数分配逻辑的前提——四元数四分量有界且已归一化，因此可以用较少位数量化；而位移无先验范围，需要存储每段min/max元数据。

动画压缩与**动画混合树**（Blend Tree）和**骨骼蒙皮**（Skinning）紧密协作：混合树在运行时对多个压缩片段同时采样并插值，解压速度直接决定混合操作的CPU开销；蒙皮着色器消耗的骨骼矩阵数据来源于采样结果，压缩引入的角度误差最终体现为顶点位置偏移，从而影响角色的视觉还原度。
