# 动画压缩

## 概述

动画压缩是游戏引擎动画系统中将原始关键帧数据体积缩小的技术集合，主要通过**曲线简化**（删除冗余采样点）、**量化**（降低数值精度）和**专用编解码库**（如ACL）三类手段实现。一个未压缩的60帧骨骼动画，若含100根骨骼，每帧每根骨骼存储位置（3个float）、旋转（4个float）、缩放（3个float），总计需要约 `60 × 100 × 10 × 4 = 2.4 MB`，而经过现代压缩后通常可缩减至原始大小的5%~20%。

动画压缩技术的规模化应用始于2000年代初期，当时主机平台内存限制（PlayStation 2仅有32 MB RAM）迫使开发者寻求激进的数据精简方案。早期方法依赖人工设置关键帧，后来逐渐被自动化的曲线拟合算法取代。2019年育碧发布了开源的**Animation Compression Library（ACL）**，将这一技术领域的最佳实践以MIT许可证公开，目前已被Unreal Engine 5作为默认动画压缩后端集成。

动画压缩的重要性体现在两个维度：其一，现代3A游戏的动画数据量可达数GB（如《赛博朋克2077》的动捕数据），压缩直接决定安装包体积；其二，解压速度影响运行时CPU占用，压缩格式设计不当会在角色大量出现时造成明显帧率下降。

## 核心原理

### 曲线简化（Curve Reduction）

原始动捕数据通常以固定频率（60Hz或120Hz）采样，产生大量在时间轴上分布均匀的关键帧，但相邻帧之间的变化往往极小。曲线简化的核心算法是**道格拉斯-普克算法（Douglas-Peucker）**，其原理是递归地检测中间采样点对直线拟合的最大误差，若误差低于阈值 `ε` 则删除该点。

误差评估公式为：

```
error = max_distance(P_i, line(P_start, P_end))
```

其中 `P_i` 是待删除区间内的采样点，`line()` 为两端点确定的插值线。阈值 `ε` 通常在世界单位0.001~0.01厘米（位置）或0.001~0.1度（旋转）之间取值。对于几乎不运动的骨骼（如背景道具骨骼），整段曲线可被简化为单一常量值，节省极为可观的空间。

旋转曲线简化时需特别处理四元数的连续性问题——相邻四元数可能因符号翻转而产生虚假的"大幅变化"，需在简化前统一翻转使相邻四元数点积为正值。

### 量化（Quantization）

量化是将高精度浮点数（32位float）替换为低精度整数表示的过程。旋转数据量化最常见的方案是**"最大分量省略"（Largest Component Omitted）格式**：由于单位四元数满足 `x² + y² + z² + w² = 1`，省略绝对值最大的分量，其余三个分量各用16位有符号整数存储（每个分量范围映射到 `[-1/√2, 1/√2]`），共48位即可替代原来的128位，压缩比约为2.67倍，且解码时可精确恢复第四个分量。

位置数据量化通常采用**局部范围归一化**：先记录每条曲线的最小值和最大值（各32位float），再将每个采样值映射为16位或甚至10位的归一化整数。例如一个范围为`[-1.2, 1.2]`的平移轨道，用16位整数量化后精度为 `2.4 / 65535 ≈ 0.000037`，远超实际渲染所需精度。

### ACL库的编解码架构

ACL（Animation Compression Library）采用的是**基于片段（Segment-based）的压缩架构**，将一个动画片段切分为若干16帧长的子片段（Segment），每个子片段独立记录局部值域范围并量化。这使得快速变化与缓慢变化的动画段可各自获得最优精度分配，避免全局值域导致的精度浪费。

ACL的核心解压路径在现代CPU上（如x86 SSE4或ARM NEON）针对SIMD进行了深度优化，解压一条128骨骼、30秒的动画，在PC端仅需约 **0.5毫秒**。ACL还内置了**误差度量框架**，可自动选择每根骨骼使用16位还是8位量化，以满足用户指定的最大视觉误差（默认0.01厘米骨骼末端位移）。

## 实际应用

**Unreal Engine 5中的ACL集成**：UE5通过`Anim Compression Settings`资产配置ACL参数，开发者可设置`Default Virtual Vertex Distance`（默认2厘米，用于末端骨骼误差测量）和`Safe Virtual Vertex Distance`（默认10厘米）。对于手部骨骼等视觉敏感区域，可单独降低误差阈值。

**Unity的压缩级别**：Unity提供Optimal、Dense、Keyframed三档动画压缩预设，其中Optimal模式对位置、旋转、缩放分别使用不同精度量化，Float曲线默认误差阈值为0.0001。开发者在`Animation Import Settings`中可单独为每种轨道类型覆盖误差值。

**带宽敏感场景的极限压缩**：网络游戏中服务器下发动画状态时，有时将旋转量化至每分量8位（共24位表示四元数），位置量化至每轴12位，单帧100骨骼仅需约450字节传输，代价是可见的关节抖动，需配合客户端插值掩盖。

## 常见误区

**误区一：压缩比越高越好**。提高量化精度损失或增大曲线简化阈值确实缩小体积，但骨骼链末端的误差会被父骨骼误差累积放大。一个5级骨骼链中，若每级旋转误差0.1°，末端位移误差可能超过数厘米，在近距离特写镜头下明显可见。ACL的误差度量框架正是为解决此问题而计算**蒙皮顶点实际位移**而非直接比较关节角度。

**误区二：曲线简化只删除关键帧，不影响播放质量**。实际上，删除关键帧后运行时需通过插值（Lerp/Slerp）重建中间帧，若原始动画包含急剧速度变化（如击打瞬间）而采样点恰好被删除，插值会"抹平"该动作细节，产生肉眼可见的迟钝感。建议对动作突变帧手动标记"不可删除"或使用更小的局部阈值。

**误区三：一次性离线压缩后无需再关注运行时开销**。ACL等库在解压时仍需CPU计算，尤其是同时播放数百个NPC动画时（如RTS或开放世界人群场景），解压总开销不可忽视。Unreal Engine提供了`Animation Budget Allocator`组件，可在帧预算不足时降低远处角色的动画更新频率，这是与压缩技术配合使用的运行时优化手段。

## 知识关联

动画压缩以**动画片段（Animation Clip）**作为直接处理对象——片段中存储的每根骨骼的位置、旋转、缩放随时间变化的曲线数据，正是曲线简化和量化操作的输入。理解动画片段的内部结构（轨道数量、采样率、骨骼层级深度）是评估压缩方案效果的前提：骨骼数量越多、采样率越高的片段，压缩收益越大，但骨骼链末端误差累积风险也越高。

在骨骼层级（Skeleton Hierarchy）方面，父骨骼的压缩误差会通过正向运动学传导至所有子骨骼，这一传播关系直接影响误差阈值的设置策略——ACL对此提供了基于蒙皮网格顶点位移的端到端误差验证，而非孤立地检查每根骨骼的旋转偏差。
