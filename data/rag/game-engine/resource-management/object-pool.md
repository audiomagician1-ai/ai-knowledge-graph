# 对象池

## 概述

对象池（Object Pool）是一种通过预先分配并复用固定数量对象来避免频繁内存申请与释放的资源管理技术。它的核心思想是：与其每次需要对象时向操作系统申请内存、使用完毕后再释放，不如维护一个"备用仓库"，用完的对象归还仓库而不是销毁，下次需要时直接从仓库取出复用。

这一模式最早在20世纪90年代的实时图形与游戏开发领域被广泛采用，原因在于当时的游戏需要每帧处理数百颗子弹、粒子或敌人单位，而C++的`new`/`delete`操作符在频繁调用时会引发严重的内存碎片和帧率抖动。《雷神之锤》（Quake，1996年）的源码中就已能看到类似思想的手工内存块管理。

在游戏引擎中，对象池的意义远超"优化技巧"的范畴。子弹、爆炸粒子、音效实例、UI飘字等对象的生命周期极短但创建频率极高，如果不用池化管理，一个粒子系统每秒可能触发数千次堆内存分配，直接导致垃圾回收（GC）卡顿或内存碎片累积，最终表现为游戏帧率的周期性下降（"GC Spike"）。

## 核心原理

### 池的生命周期：初始化、借出、归还

对象池在初始化阶段一次性创建 N 个对象并将其标记为"空闲（idle）"。当游戏逻辑请求一个新对象时，池从空闲队列取出一个，将其状态重置后标记为"活跃（active）"并返回给调用方。当对象使用完毕（如子弹飞出屏幕），不调用析构函数，而是调用池的`Release()`方法，对象被重置并重新放回空闲队列。整个过程中，操作系统层面的内存分配只发生在初始化阶段，运行时的"创建"与"销毁"实际上是 O(1) 的状态切换操作。

### 容量策略：固定池与动态扩容池

**固定大小池（Fixed Pool）** 预分配恰好 N 个槽位，若空闲队列耗尽则拒绝新请求或覆盖最旧的活跃对象。固定池的优点是内存占用完全可预测，适合主机平台的严格内存预算；缺点是需要事先调参确定 N 值，Unity 引擎的 `ParticleSystem` 最大粒子数上限字段即属此类。

**动态扩容池（Growing Pool）** 在空闲队列耗尽时以批量方式（如每次翻倍或追加固定块）申请新内存，兼顾灵活性与分配频率的降低。Unreal Engine 的 `TObjectPool<T>` 模板类默认采用此策略，初始容量通过构造参数指定。

### 对象重置（Reset）的正确性

对象池最容易被忽视的设计要点是：归还池的对象必须被彻底重置，否则下一个借用方会接收到携带上一个使用者"脏数据"的对象。一颗"复用子弹"若不清除上次的速度向量、碰撞标志和特效引用，就会在新一轮发射中产生不可复现的错误行为。标准做法是定义一个 `Reset()` 接口方法，在对象归还时强制调用，而非寄希望于借用方在使用前自行初始化。

## 实际应用

**子弹与投射物系统**：射击游戏中子弹的创建速率可达每秒数十枚，而每颗子弹的存活时间往往不足1秒。使用对象池后，发射逻辑从`new Bullet()`改为`bulletPool.Get()`，飞出边界的子弹调用`bulletPool.Release(bullet)`。以一款每秒发射30发子弹、子弹存活0.5秒的游戏为例，稳定状态下池内只需保持约15个活跃对象，预分配30个槽位即可覆盖峰值。

**粒子特效**：Unity 的粒子系统内置对象池逻辑，每个粒子实例在 `ParticleSystem` 组件内部以结构体数组（SoA，Structure of Arrays）形式池化存储，而非独立的堆对象，这是引擎层面对对象池思想的深度整合。

**音效实例池**：同一时刻可能有多个相同音效并发播放（如多个爆炸同时触发），AudioSource 组件的复用池可避免反复挂载/卸载组件的开销。Unreal Engine 推荐为高频音效预分配至少8个 AudioComponent 实例组成的池。

**UI飘字**：MMORPG 战斗界面中伤害数字飘字每秒可能出现数十条，每条存活约1.5秒，使用对象池可将 `Instantiate`/`Destroy` 调用完全消除，在中低端移动设备上帧率提升可达15%~30%。

## 常见误区

**误区一：认为对象池总能降低内存占用**
对象池的目标是降低分配频率和GC压力，而非节省内存。固定池预分配的 N 个对象始终占据内存，即便当前只有2个处于活跃状态。如果 N 设置过大，对象池反而比按需分配消耗更多内存。正确的做法是根据性能分析数据（Profiler）确定峰值并发数量，将 N 设为峰值的1.2~1.5倍作为余量，而不是随意设一个"够大的数"。

**误区二：对所有对象都使用池化管理**
对象池只在对象创建/销毁频率高且对象构造代价较大时才值得引入。对于整个游戏会话期间只创建一次的配置对象、场景管理器等单例，引入对象池不仅没有收益，还增加了代码复杂度和 `Reset()` 方法遗漏导致的状态污染风险。经验法则：当某类对象在1秒内的创建销毁次数超过10次，或性能分析器显示该类型的GC Alloc不可忽视时，才考虑池化。

**误区三：池化对象仍可被外部持有引用**
当一个对象被归还给池后，若外部代码仍持有对它的引用并继续访问，就会读写到已被复用的对象，这是对象池特有的"悬空引用"问题。解决方案包括：归还时将外部引用置为null、使用"版本号（Generation ID）"机制（每次借出时递增版本号，访问时校验版本号是否匹配），Unreal Engine 的 `FPooledRenderTarget` 即采用了引用计数结合版本号的双重保护机制。

## 知识关联

对象池建立在**资源管理概述**所介绍的"内存分配成本"与"运行时开销"基础认知之上——理解为何频繁堆分配代价高昂，是理解对象池存在意义的前提。具体而言，堆分配涉及系统调用、内存对齐计算和碎片整理，而对象池将这一成本摊销到初始化阶段的一次批量分配中。

在技术实现层面，对象池与**内存分配器**（如线性分配器、块分配器）概念相邻：对象池在语义层面管理"对象的逻辑生命周期"，内存分配器在物理层面管理"字节的归属"，两者可以组合使用——对象池内部的连续槽位本身可由一个块分配器（Block Allocator）统一持有，进一步提升缓存局部性（Cache Locality）。

对象池也是**组件系统（ECS，Entity-Component-System）** 架构的思想基础之一：ECS 中组件以紧密排列的数组形式存储，本质上是对同类对象的池化管理，只是将"活跃/空闲标志"升级为了实体（Entity）的存在性判断。
