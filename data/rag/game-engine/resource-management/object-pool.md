# 对象池

## 概述

对象池（Object Pool）是一种通过预先分配并复用对象来避免频繁内存分配与释放的资源管理技术。其核心思想是：将暂时不用的对象放入"池"中保存，而不是销毁它们；当需要新对象时，从池中取出已有对象并重置其状态，而不是重新创建。这一技术在游戏引擎中针对子弹、粒子、敌人单位等生命周期极短、创建频率极高的对象尤为关键。

对象池模式最早在20世纪90年代随实时游戏开发的兴起而普及。C++中的 `new`/`delete` 操作在频繁调用时会引发堆内存碎片化，并触发操作系统级的内存分配调用，这在每帧需要产生数百次对象创建的场景（如粒子系统、射击游戏子弹）中会造成明显的帧率抖动。Unity引擎在2021.1版本中正式内置了 `ObjectPool<T>` 类，印证了这一模式在工业级引擎中的地位。

对象池真正解决的问题是**GC压力**（垃圾回收停顿）。在使用托管内存的引擎（如Unity的C#运行时）中，频繁分配小对象会填满Gen0堆，触发垃圾回收，导致游戏出现几毫秒到几十毫秒的卡顿峰值。对象池通过将对象的生命周期从"创建→使用→销毁"改为"借出→使用→归还"，使堆上的对象数量保持稳定，从根本上消除这类停顿。

## 核心原理

### 基本数据结构

对象池内部通常维护一个容器来存放空闲对象，最常见的实现使用**栈（Stack）**或**队列（Queue）**。栈的优势在于借出和归还操作均为 O(1)，且刚归还的对象仍在CPU缓存中（局部性更好），因此性能略优于队列。一个最简化的对象池结构如下：

```
Pool {
    freeStack: Stack<T>       // 空闲对象栈
    activeCount: int          // 当前借出数量
    maxSize: int              // 池容量上限
}
```

`Get()` 操作：若 `freeStack` 非空则弹出顶部对象，否则实例化新对象（若未超过 `maxSize`）。  
`Release(obj)` 操作：重置对象状态后将其压入 `freeStack`。

### 预热与容量策略

对象池通常在场景加载阶段进行**预热（Warm-up）**，即提前创建 N 个对象填充池，避免游戏运行中第一次 `Get()` 时触发实例化。预热数量的选择依赖对历史并发活跃量的分析——例如一个横版射击游戏中子弹的峰值并发量约为80发，则预热数量可设为100，留有20个冗余。

当池中空闲对象耗尽时，有三种策略可选：
1. **扩容（Grow）**：动态分配新对象加入池，池大小无上限，内存占用会增长。
2. **失败返回null**：严格限定上限，调用方必须处理返回空的情况，适合硬件内存极度受限的平台（如主机开发）。
3. **强制回收最旧对象**：从活跃对象中强制回收使用时间最长的那个，适合粒子系统（最旧的粒子视觉影响最小）。

### 对象重置机制

对象池的正确性依赖于**归还时的状态重置**。被归还的对象可能携带上一次使用的速度向量、碰撞标记、事件监听器等残留状态。标准做法是为池化对象定义一个 `Reset()` 或 `OnReturnToPool()` 接口方法，在 `Release()` 内部自动调用。在Unity中，`IObjectPool<T>` 接口要求实现者提供 `actionOnGet`（借出回调）和 `actionOnRelease`（归还回调）两个委托，分别处理激活和停用逻辑（如 `gameObject.SetActive(true/false)`）。

## 实际应用

**子弹系统**：第一人称射击游戏中，玩家可能以每秒10发的速度持续射击，每颗子弹飞行0.5秒后消失。若直接实例化与销毁，每秒产生10次 `Instantiate` 和10次 `Destroy`。使用对象池后，池中维持约6~10个子弹对象循环使用，堆内存分配次数降为0（仅预热时分配一次）。

**粒子特效**：爆炸特效可能一次性产生200个粒子对象。现代游戏引擎通常为不同类型的粒子（火花、烟雾、碎片）各建立独立的对象池，而不是混用一个通用池。Unity的VFX Graph和Particle System内部均使用了类似对象池的粒子缓冲区来管理粒子实例。

**UI元素复用**：在无限滚动列表（如背包界面）中，可见区域只有10个格子，但背包可能有500件物品。对象池在此场景下只创建12~15个格子UI对象，滚动时将移出视野的格子归还池中，再借出给新进入视野的物品使用，内存占用从500个对象降低为约15个。

## 常见误区

**误区一：认为对象池适合所有对象类型。**  
对象池对创建成本低、生命周期长的对象（如场景中唯一存在的玩家角色）没有收益，反而增加了代码复杂度。对象池的收益与对象的**创建频率**成正比——每帧创建少于1次的对象通常不值得池化。

**误区二：归还对象后忘记重置状态，导致"脏对象"复用。**  
这是对象池实现中最常见的Bug来源。例如，子弹对象被归还时若未清空其"已命中目标"标志位，下次借出时该子弹可能在飞出的第一帧就触发命中逻辑。解决方案是在 `Release()` 中强制调用重置方法，而不依赖借出方在 `Get()` 后自行初始化。

**误区三：将对象池的池容量设置得过大。**  
如果一个子弹池预热了500个对象但实际并发峰值只有20个，那么多余的480个对象会长期占用内存而毫无收益。正确做法是通过性能分析工具（Profiler）测量实际并发峰值后，设置略高10%~20%冗余量的池容量。

## 知识关联

**前置概念**：对象池建立在资源管理概述所讲的**内存分配策略**基础上——只有理解了堆内存分配（`malloc`/`new`）的开销以及GC的触发机制，才能量化对象池带来的性能提升。资源管理概述中介绍的"资源生命周期"概念（加载→使用→卸载）在对象池中被改写为"预分配→借出→归还"这一新的循环模型。

**横向关联**：对象池与**内存池（Memory Pool）**相关但不同——内存池管理的是原始字节块，对象池管理的是已构造好的类型化对象实例。在引擎架构中，二者可以组合使用：对象池负责对象级别的复用，内存池负责为这些对象提供底层的连续内存块，从而同时消除GC压力和内存碎片。对象池的思想也延伸到了**线程池**（复用线程而非内存对象）和**连接池**（数据库连接复用）等更广泛的系统设计领域。
