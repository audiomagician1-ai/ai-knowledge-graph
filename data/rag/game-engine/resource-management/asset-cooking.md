# 资源烘焙

## 概述

资源烘焙（Asset Baking）是游戏引擎资源管理流程中的一个预处理阶段，指将美术人员创作的源文件（如 Photoshop 的 .psd、Blender 的 .blend、未压缩的 .wav）转换为目标平台可以直接高效读取的二进制格式。这个转换过程不是简单的格式重命名，而是可能涉及颜色空间转换、Mipmap 生成、顶点缓冲布局重排等深度处理操作。烘焙后的资源通常无法反向还原为可编辑的源文件，因此"烘焙"一词非常贴切——就像面包烤好后无法变回面粉一样。

资源烘焙的概念随着游戏机平台的多样化而变得愈发重要。早期 PC 游戏时代，程序员往往在运行时直接解析 BMP 或 WAV 文件，性能损耗被接受。进入 2000 年代后，PlayStation 2、Xbox 等主机平台具有各自特定的纹理格式（如 PS2 的 GS 纹理格式），开发者被迫引入离线转换步骤。Unreal Engine 3（2006年）将这一流程系统化为"Cook"（烹饪）操作，Unity 则将其称为"Build"阶段的资源处理，行业逐渐形成了标准化的烘焙管线概念。

烘焙的核心价值在于将计算成本从运行时转移到构建时。一张 4096×4096 的源纹理在烘焙阶段生成完整的 Mipmap 链只需数秒，而如果放到运行时处理，会在首帧造成明显卡顿，甚至在某些嵌入式平台上根本不具备实时生成能力。

## 核心原理

### 平台目标格式映射

资源烘焙最基础的工作是维护一张"平台-格式"映射表。以纹理为例，同一张漫反射贴图在不同目标平台上会被烘焙为完全不同的压缩格式：PC（DirectX）使用 BC7 或 BC3（DXT5）格式，Android 使用 ASTC 4×4 或 ETC2，iOS 使用 PVRTC 或 ASTC，Nintendo Switch 使用定制的 NVN 纹理格式。烘焙系统需要在构建管线中维护这些规则，确保同一份源资源可以为五个平台各自生成一份对应的二进制文件，存储在各自的构建输出目录中。

### 依赖追踪与增量烘焙

全量烘焙一个中型项目可能耗费数小时，因此高效的烘焙系统必须实现增量构建。实现方式通常是为每个资源记录一个内容哈希值（如 MD5 或 xxHash），并存储在一个清单文件（manifest）中。当源文件的哈希值与清单记录一致时，跳过该资源的烘焙。依赖追踪还需向上传播：一张材质球 `.mat` 引用了一张纹理，若纹理源文件发生变更，依赖该纹理的所有材质球也需要标记为"脏"（dirty）并触发重烘。Unreal Engine 的 Derived Data Cache（DDC）系统正是为此设计，它使用资源内容加上烘焙参数共同计算缓存键。

### 烘焙参数与资源标注

源资源本身通常需要携带元数据来指导烘焙行为。以纹理为例，一张法线贴图和一张漫反射贴图在源文件层面都是普通图片，但烘焙时法线贴图需要：禁用 sRGB 颜色空间转换、选择保留双通道精度的压缩格式（如 BC5）、可能需要重新归一化每个像素的 XY 通道。Unity 的 TextureImporter 设置、Unreal 的 Texture 资源属性面板都是这种元数据的存储载体。音频资源同样有类似的烘焙参数，例如是否将立体声压缩为单声道（节省约 50% 体积）、目标采样率是否从 44100Hz 降为 22050Hz。

### 烘焙输出结构

烘焙完成后，输出文件通常以扁平化或内容哈希命名的方式组织，而非保留源文件的目录结构。Unity 的 StreamingAssets 路径之外的所有资源最终都会被打包进 `.assets` 文件，资源以内部 GUID 索引而非路径引用。这种结构使得后续的内容包（AssetBundle）系统可以直接从烘焙输出中按需拆分打包，而无需重新处理源数据。

## 实际应用

**多平台发行管线**：一款同时发行 PC、PS5 和 Switch 版本的游戏，在 CI/CD（持续集成）系统中会配置三条并行的烘焙任务。PC 任务输出 BC7 压缩纹理和 Ogg Vorbis 音频；PS5 任务输出符合 PlayStation 认证要求的 GNF 纹理格式（PlayStation 专有格式）；Switch 任务输出 ASTC 纹理并将部分音频转为 Nintendo ADPCM 格式。三条任务可以在构建服务器上并行执行，互不干扰。

**光照烘焙（Lightmap Baking）**：这是资源烘焙的一个特殊子类，指将全局光照计算结果预先写入光照贴图纹理。Unity 的 Enlighten 和 Progressive Lightmapper 都在编辑器离线阶段将光线追踪或辐射度计算结果"烘焙"进分辨率通常为 512×512 到 2048×2048 的 Lightmap 纹理。运行时渲染器只需采样这张静态贴图，而无需实时计算复杂的间接光照，可将移动端帧率从 20fps 提升至 60fps 以上。

**Shader 变体预编译**：现代引擎也将 Shader 编译纳入烘焙阶段。Unity 的 Shader Variant Collection 和 Unreal 的 PSO Cache 都在构建时将 Shader 源码针对目标 GPU API（Vulkan、Metal、DirectX 12）预编译为二进制，避免运行时首次着色器编译导致的卡顿（业界称为"Shader Stutter"）。

## 常见误区

**误区一：烘焙只是格式转换，与源文件无关**
许多初学者认为可以随意修改烘焙参数而不影响源文件。实际上，将法线贴图的"sRGB"选项错误勾选后，引擎会在烘焙阶段对其进行 Gamma 2.2 的颜色空间转换，导致法线向量被扭曲，渲染出错误的光照结果。这类错误不会报错，只会表现为视觉异常，排查成本极高。烘焙参数是资源定义的一部分，必须纳入版本控制。

**误区二：增量烘焙总是安全的**
当烘焙工具链本身（如纹理压缩库 ispc_texcomp 的版本）升级后，旧的缓存条目可能使用过时算法生成，但哈希校验不会发现工具版本变更。正确做法是在缓存键计算中加入工具链版本号，或在工具链升级后强制执行一次全量烘焙（Full Rebuild）并清空 DDC 缓存。Unreal Engine 在 DDC 键中包含引擎版本字符串正是出于此原因。

**误区三：烘焙后的资源等同于最终包内资源**
烘焙输出只是构建流程的中间产物，后续还需要经过内容包系统的打包和压缩格式的二次处理。一个 2MB 的 BC7 纹理烘焙输出，经过 LZ4 或 Zstandard 压缩后打入内容包，最终在包内可能仅占 800KB。将烘焙输出大小误作最终包大小会导致包体估算严重偏差。

## 知识关联

学习资源烘焙需要掌握**资源管理概述**中的源资源与运行时资源的区分概念——烘焙正是连接这两个阶段的处理步骤。理解烘焙输出的结构，是深入学习**内容包系统**（如 Unity AssetBundle 或 Unreal Pak 文件）的前提，因为内容包系统直接以烘焙输出为输入来组织打包策略。而烘焙阶段选择何种中间格式，又直接决定了**压缩格式**章节中讨论的 GPU 压缩格式（ASTC、BC7、ETC2）与通用压缩算法（Zstd、LZ4）如何配合使用——两者并非竞争关系，而是在构建管线的不同层次各自发挥作用。
