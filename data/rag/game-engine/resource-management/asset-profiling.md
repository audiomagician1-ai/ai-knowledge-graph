# 资源性能分析

## 概述

资源性能分析是游戏引擎资源管理流程中专门用于度量、审查和优化游戏资产质量与效率的技术手段，核心工具包括 Size Map（尺寸图）、引用审计（Reference Audit）和冗余检测（Redundancy Detection）。其目标是定位哪些资产占用了过多内存或磁盘空间、哪些资产被过度或错误引用，以及哪些内容在包体中重复存在。

该方向在 Unreal Engine 4 引入 Size Map 工具（约 2016 年随 4.12 版本成熟）后开始被开发者系统化使用。在此之前，团队通常只能依靠 cooked 包大小的手工比对来推断资产问题，效率极低。随着主机与移动端对包体大小限制日趋严格（Google Play 最初限制 APK 为 50MB，现为 150MB；主机 DLC 也有严格预算），资源性能分析从辅助工作变为每个项目发布周期的必要环节。

在实际项目中，一张 4096×4096 的纹理在 DXT1 压缩下约占 8MB 显存，若该纹理仅在一个次要场景的小道具上使用，其性价比极差。资源性能分析的意义正在于通过工具化手段将这类"隐形浪费"系统性地暴露出来，而不是依赖开发者的主观经验逐一排查。

## 核心原理

### Size Map：可视化资产磁盘占用

Size Map 以树状矩形图（Treemap）的形式展示资产及其所有传递依赖（Transitive Dependencies）的磁盘占用比例。在 Unreal Engine 中，右键任意资产选择"Size Map"即可展开完整依赖链，每个矩形面积正比于该资产 cooked 后的实际大小。

其计算逻辑分两层：**独立大小（Exclusive Size）**仅统计该资产本身；**包含大小（Inclusive Size）**则递归累加所有被引用子资产的大小。一个蓝图 Actor 的 Inclusive Size 可能高达 200MB，而其 Exclusive Size 仅有 12KB，这说明它引用了大量重型资产。Size Map 使这种差距在视觉上立即清晰可见，是排查"意外重量级资产"最直接的入口。

### 引用审计：追踪资产引用关系

引用审计通过遍历资产注册表（Asset Registry）中存储的依赖图，检查每个资产被哪些对象引用（硬引用 Hard Reference vs 软引用 Soft Reference）。硬引用意味着当持有者加载时，目标资产**必须**同步加载到内存；软引用（FSoftObjectPath）则允许按需异步加载。

审计的关键指标包括：
- **硬引用深度**：某资产被多少层硬引用链追溯到启动关卡，深度越浅越危险，因为它可能在游戏启动时就强制常驻内存。
- **孤儿资产（Orphan Asset）**：引用计数为 0 的资产——没有任何对象引用它，但它仍存在于内容目录中并被 cook 进包体。
- **循环引用风险**：A 引用 B，B 又引用 A，导致两者永远无法单独卸载，内存泄漏由此产生。

Unreal Engine 提供 `ReferenceViewer`（引用查看器）作为可视化工具，同时可通过 `AssetRegistry.GetDependencies()` API 在编辑器脚本中批量导出引用数据供离线分析。

### 冗余检测：识别重复内容

冗余检测针对三种典型重复场景：

1. **内容重复**：两张纹理像素数据完全相同但文件名不同，通常由美术人员误操作复制资产而非创建实例引用所致。检测方法为计算资产二进制内容的哈希值（如 SHA-256），对比碰撞列表。
2. **Mipmap 配置冗余**：同一纹理在不同材质中分别设置了不同的 LODBias，导致实际加载到显存的 mip 级别不一致，产生额外显存开销。
3. **派生数据冗余（DDC Duplication）**：Unreal 的 Derived Data Cache 存储资产编译后的平台特定数据，若 DDC 未正确共享（如本地 DDC 与共享 DDC 不同步），不同开发者会各自重复编译相同资产，浪费构建时间。

冗余检测的量化指标通常用**冗余率 = （重复数据总量 / 总资产数据量）× 100%** 表示，目标是将其控制在 5% 以下。

## 实际应用

**场景一：移动端包体瘦身**

某移动游戏项目 APK 超出 150MB 限制，通过 Size Map 扫描发现一个 UI 蓝图的 Inclusive Size 占全包的 18%，追踪后发现它硬引用了一张用于过场动画的 2048×2048 UI 纹理，而该纹理的实际显示尺寸在屏幕上从未超过 256×256 像素。将其降级为 512×512 并改为软引用后，包体节省约 12MB。

**场景二：主机内存泄漏排查**

PS5 版本出现关卡切换后内存持续升高的问题。通过引用审计发现某武器系统蓝图存在循环硬引用：武器蓝图引用特效管理器，特效管理器又持有对武器蓝图类的硬引用，导致两者在卸载关卡后均无法释放。将其中一条引用改为 `TSoftClassPtr` 软引用后，关卡卸载时内存正确回落约 40MB。

**场景三：冗余纹理批量清理**

通过编辑器 Python 脚本对全项目纹理进行 MD5 哈希比对，在 3000 张纹理中发现 127 对哈希相同的资产，合并后节省 230MB cooked 包体空间，同时减少了 cook 时间约 8 分钟。

## 常见误区

**误区一：Exclusive Size 小就代表该资产"轻量"**

许多开发者看到某个蓝图的 Exclusive Size 只有几KB，便认为它对包体没有影响。事实上若其 Inclusive Size 远大于 Exclusive Size，该蓝图作为"入口"将拖入数十乃至数百MB的依赖资产，应优先检查 Inclusive Size 而非 Exclusive Size。

**误区二：删除孤儿资产一定安全**

引用审计显示引用计数为 0 的资产并非全部可以安全删除。部分资产通过字符串路径（如 `LoadObject<UTexture2D>(nullptr, TEXT("/Game/Textures/T_Foo"))`）在代码中被动态加载，这类软字符串引用不会被资产注册表记录，因此在审计中显示为孤儿，但实际运行时仍被使用。删除前必须结合代码全文搜索进行二次确认。

**误区三：冗余检测只需在发布前执行一次**

冗余问题会随着美术资产的持续迭代不断产生，仅在发布前执行一次检测往往已积累大量冗余难以逐一追溯责任。正确做法是将哈希比对脚本集成到 CI/CD 流水线中，每次提交后自动扫描并生成冗余报告，当冗余率超过阈值（如 3%）时触发警告。

## 知识关联

资源性能分析直接依赖**资产注册表**提供的依赖图数据——若资产注册表未正确扫描或索引不完整，引用审计的结果将出现遗漏，导致孤儿资产判断错误或硬引用链追踪中断。资产注册表中的 `FAssetData` 结构存储了每个资产的类型、路径及标签信息，这些字段是 Size Map 和引用审计脚本的数据来源基础。

在工具链层面，资源性能分析的产出（如孤儿资产列表、冗余资产对、过重引用链报告）直接驱动后续的**资产清理（Asset Cleanup）**和**LOD策略调整**决策。Size Map 发现的过重纹理应进入纹理优化流程，引用审计发现的循环依赖应由架构师介入重构引用结构，而冗余检测的报告则直接指导美术规范的制定与执行。三种分析手段协同使用时，通常按照"先用 Size Map 定位热点 → 再用引用审计追溯原因 → 最后用冗余检测清理存量问题"的顺序进行。
