# 压缩格式

## 概述

压缩格式是游戏引擎资源管理中将数据以更小的字节体积存储和传输的技术手段，通过特定算法消除数据中的冗余信息，使同等内容的资产文件占用更少的磁盘空间和内存带宽。游戏行业常用的压缩格式包括 Oodle、LZ4 和 Zlib，它们在压缩率、解压速度和 CPU 消耗三个维度上各有侧重，开发者需根据资源类型和加载场景选择合适方案。

压缩格式的应用在游戏行业随着物理介质速度瓶颈的演变而变化。早期 CD-ROM 读取速度仅约 150 KB/s，压缩主要用于减少碟片容量需求；进入 SSD 时代后，NVMe 硬盘顺序读取速度可达 5000 MB/s 以上，此时 CPU 解压速度成为新的瓶颈，驱动了以 Oodle Kraken 为代表的高速解压算法的兴起。PlayStation 5 的定制 SSD 与专用解压硬件协同工作，正是这一趋势的体现。

游戏中的纹理、音频、关卡几何体、脚本字节码等资产在经过资源烘焙后输出为二进制格式，压缩格式决定了这些二进制数据最终在 PAK 包或 Bundle 中的存储形式，直接影响游戏的安装包体积和运行时加载延迟。

## 核心原理

### LZ 系列算法基础

LZ4 和 Zlib 都源于 Lempel-Ziv 压缩理论，核心思想是用"向前引用"替代重复出现的字节序列。LZ4 的工作单元是"序列（sequence）"，每个序列由字面量字节和匹配引用构成，匹配引用通过偏移量（offset）和长度（match length）两个字段表达。LZ4 的最小匹配长度为 4 字节，匹配窗口固定为 64 KB，这使得其解压速度可在现代 CPU 上达到 **4–5 GB/s**，远高于磁盘读取速度，几乎不构成 I/O 瓶颈。

Zlib 基于 Deflate 算法，结合了 LZ77 匹配搜索和 Huffman 熵编码两步压缩，压缩率高于 LZ4，但解压速度通常只有 **300–500 MB/s**。Zlib 的压缩级别参数范围为 0（无压缩）到 9（最大压缩），级别 6 是通用默认值，在压缩率与 CPU 耗时之间取得平衡。游戏引擎（如 Unity 的旧版 AssetBundle）历史上大量使用 Zlib，但因解压延迟过高已逐渐被 LZ4 取代。

### Oodle 的分层架构

Oodle 是 RAD Game Tools 开发的商业压缩库，目前被 Epic Games 收购并集成于 Unreal Engine。Oodle 提供多个编解码器，其中 **Kraken** 针对高压缩率与高解压速度的平衡进行了优化，解压速度约为 **1.5–3 GB/s**，压缩率接近 Zlib 的最高级别。**Selkie** 牺牲约 5% 的压缩率换取接近 LZ4 的解压速度，适合频繁随机访问的场景。**Leviathan** 则优先追求最大压缩率，解压速度约为 500 MB/s，适合安装包中不常访问的大型资产。

Oodle 还内置了 **Oodle Texture**（针对 BC/ASTC 纹理块的专用压缩）和 **Oodle Network**（针对游戏网络数据包的字典压缩），形成覆盖多场景的压缩工具链。

### 块压缩与流式压缩的区别

游戏资源管理中常将大型资产切分为固定大小的"压缩块（chunk）"分别压缩，典型块大小为 **64 KB 或 256 KB**。块压缩允许引擎在不解压整个文件的前提下随机访问某一块数据，例如流式加载开放世界地形时只需解压当前区块，而无需将整个地形文件完整解压到内存。Unreal Engine 的 PAK 文件默认使用 256 KB 块大小配合 Oodle Kraken，而 Unity 的 LZ4 AssetBundle 同样采用分块方案以支持按需加载。

流式压缩（streaming compression）则适用于音频等连续播放数据，如 Ogg Vorbis 或 Opus 格式本身已内嵌压缩，引擎在播放时逐帧解码，无需将整个音频文件预先解压到内存。

## 实际应用

**安装包体积优化**：使用 Oodle Kraken 对游戏 PAK 包内的脚本字节码和关卡数据进行压缩，通常比 Zlib 压缩率提高 **8–12%**，同时解压速度高出约 4–5 倍。Epic Games 在《堡垒之夜》的更新包中引入 Oodle 后，更新包体积减少约 30%。

**纹理数据的双层压缩**：GPU 使用的 BC7 或 ASTC 格式已是一种有损块压缩，将其再用 LZ4 进行无损压缩通常额外节省 **15–25%** 的包体空间，因为相邻纹理块之间存在相关性。注意此处 GPU 纹理压缩与文件级压缩是两个独立层次，GPU 纹理压缩在运行时由硬件解码，文件级压缩由 CPU 在加载时解压。

**按资产类型选择格式**：音频文件通常已经过 Ogg/Opus 压缩，对其再叠加 LZ4 压缩几乎不产生额外收益（压缩率提升不足 1%），反而徒增 CPU 负担；而未经压缩的 JSON 配置文件或本地化字符串文本，使用 Zlib 压缩可减少 **60–80%** 的体积。

## 常见误区

**误区一：压缩率越高，加载速度就越快**。这一判断忽略了解压 CPU 开销。在机械硬盘（HDD）时代，存储 I/O 速度约为 100–150 MB/s，CPU 解压速度远超 I/O，更高压缩率意味着更少的 I/O 时间，净效果是更快加载。但在 NVMe SSD 上，I/O 速度可达 3000 MB/s 以上，若使用 Zlib（解压约 400 MB/s），解压 CPU 会成为瓶颈，实际加载速度反而低于使用 LZ4。

**误区二：对所有资产统一使用同一种压缩格式**。游戏引擎资源种类繁多，已压缩的 Ogg 音频、GPU 纹理（BC/ASTC）、视频（H.264/H.265）等格式本身已利用了数据的统计规律，再叠加通用文件压缩的收益极低。对这类文件强制启用 Zlib 压缩不仅浪费构建时间，还会在加载时产生无效的 CPU 消耗。

**误区三：Oodle 与 LZ4/Zlib 是同类可直接替换的开源库**。LZ4 和 Zlib 是开源库，可免费使用，而 Oodle 是需要商业授权的付费库，授权费用与游戏销售收入挂钩。虽然 Unreal Engine 用户通过引擎订阅可直接获得 Oodle 使用权，但独立引擎开发者需单独评估授权成本与压缩收益的比值。

## 知识关联

资源烘焙阶段产出的二进制资产（如序列化后的材质、网格、蓝图字节码）是压缩格式的直接输入对象。烘焙时确定的资产二进制布局会影响压缩效率：若烘焙输出中将相似数据（如同类型的顶点属性数组）紧密排列，LZ 系列算法的重复字节检测命中率更高，压缩率可提升 10–20%。因此烘焙阶段的数据布局设计与压缩格式的选择是相互影响的决策。

在运行时资源管理层面，压缩格式的解压策略直接影响内存池的分配行为——分块压缩需要为每个 chunk 分配临时解压缓冲区，而流式解压需要维护滑动窗口缓冲区；理解这一点是设计高效资源流式加载系统的前提。
