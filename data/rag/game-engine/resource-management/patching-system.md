# 补丁系统

## 概述

补丁系统（Patch System）是游戏引擎资源管理中负责将已发布版本增量更新至新版本的机制，其核心思想是只传输"变化的部分"而非完整的资源包。与重新下载完整内容包相比，补丁系统通过计算两个版本之间的二进制差异（binary diff），生成体积极小的差异文件（delta file），玩家只需下载并应用这份差异即可完成更新。以一款典型的30GB游戏为例，一次小版本更新可能只涉及200MB的实际改动，若强制全量下载则极大浪费带宽，而补丁系统可将玩家实际下载量压缩至20-50MB。

补丁系统的工业化实践可追溯到1990年代的软件分发领域，bsdiff算法由Colin Percival于2003年首次发表，至今仍是游戏行业二进制差分的主流基础算法之一。游戏行业的特殊性在于资源文件以自定义二进制格式存储（如纹理的DDS文件、音频的OGG/OPUS文件、网格数据的自定义二进制格式），这些文件经过压缩后相邻版本之间的字节相似度很低，直接对压缩后的包做diff效果极差，因此游戏补丁系统通常需要在**解压缩状态**下计算差异再重新压缩，以获得最优的差异包大小。

## 核心原理

### 二进制差分算法

bsdiff算法的基本原理是将旧文件视为查找字典，通过后缀排序（suffix sorting）找到新文件中每段字节在旧文件中的最佳匹配位置，输出三条数据流：差异数据（diff data）、额外数据（extra data）和控制数据（control data）。生成的补丁文件格式为：

```
控制块（每个控制块包含三个整数 x, y, z）：
  x = 从旧文件复制并混合差异的字节数
  y = 从额外数据直接写入的字节数
  z = 旧文件指针的偏移量
```

bsdiff算法时间复杂度为 O(n log n)（其中 n 为旧文件大小），空间复杂度较高，对于4GB以上的大型游戏资源包，服务端构建补丁时内存消耗可达文件大小的5-6倍，因此补丁生成必须在服务器端离线完成，客户端仅负责**应用**补丁（bspatch），后者仅需 O(n) 时间和极低的内存开销。

### 版本链与补丁图（Patch Graph）

游戏版本号通常采用三段式格式（如 1.4.2），补丁系统需要维护一张**补丁图**，记录哪些版本对之间存在可用的差分补丁。朴素策略是为每个版本仅提供从上一版本到当前版本的单步补丁（线性版本链），但这要求玩家必须逐步升级（1.0→1.1→1.2→1.3），若玩家跳过了多个版本则需依次应用多个补丁，累积下载量可能超过全量包。

优化策略是构建**跨版本补丁**：同时提供 v1.0→v1.3 的大跨度补丁和 v1.2→v1.3 的小步补丁。版本管理服务器根据客户端上报的当前版本号，动态选择使总下载量最小的补丁路径。Steam 的 SteamPipe 系统和虚幻引擎的 PAK 补丁机制均采用了类似的多路径补丁策略。

### 资源清单与完整性校验

补丁系统依赖**资源清单文件**（Manifest）记录每个文件的哈希值（通常使用SHA-1或XXHash64）、大小和版本归属。客户端在应用补丁前必须校验本地文件的哈希值与清单一致，防止因玩家手动修改文件或磁盘错误导致补丁应用失败甚至客户端损坏。应用补丁后同样需要对输出文件重新计算哈希并比对，验证 bspatch 操作的正确性。

清单文件本身也需要版本化管理：新版清单包含全量文件列表，客户端通过比对本地清单与新版清单，确定哪些文件需要下载新的差分补丁、哪些文件未变化可以跳过、哪些文件被删除需要清理。这一流程要求内容包系统（AssetBundle/PAK等）在打包时保证**确定性构建**（deterministic build）——相同输入在不同构建机器上产生完全相同的二进制输出，否则即使资源内容未变，构建产物的字节差异也会导致差分包体积虚增。

## 实际应用

**虚幻引擎的 PAK 补丁流程**：UE4/UE5 中，补丁包被打包为带有版本号后缀的额外 PAK 文件（如 `pakchunk0_p.pak`），引擎在启动时按优先级加载所有 PAK，高版本 PAK 中的资源自动覆盖低版本。开发团队在 Project Launcher 中配置"为补丁生成差异文件列表"，系统自动比对两次 cook 结果的清单，仅将变更资产打入补丁 PAK，最终补丁 PAK 通过 bsdiff 或 zstd delta 进一步压缩后分发。

**移动端热更新场景**：移动游戏受应用商店审核周期限制（iOS审核平均1-3天），大量改动通过补丁系统绕过审核直接下发给客户端。补丁包通常被拆分为不超过5MB的小块（chunk），支持断点续传，以适应移动网络的不稳定性。补丁应用失败时系统自动回退并触发全量重下该资源块，保证更新原子性。

## 常见误区

**误区一：对压缩后的包文件直接做 diff 可以得到更小的补丁**。实际情况完全相反。LZ4、Zlib、Oodle 等压缩算法的输出对输入微小变化极为敏感——原始资源中改动1个字节可能导致压缩包中数十KB的字节发生变化，bsdiff 在压缩包上找不到有效的字节匹配，生成的差分文件甚至可能比新版完整压缩包更大。正确做法是先解压旧包，对原始数据做 diff，再对 diff 结果压缩传输。

**误区二：补丁系统可以替代版本控制系统（VCS）**。补丁系统的差分是面向**二进制终端分发文件**设计的，生成的 delta 文件对人类不可读、不可合并，只能单向应用（旧版→新版）。它解决的是"玩家客户端增量升级"问题，而非"多名开发者并发修改同一资源"的冲突解决问题，两者在设计目标上完全不同。

**误区三：补丁系统只需在客户端实现**。补丁包的生成（diff计算）是计算密集型任务，必须在持续集成（CI）流水线中由服务端构建系统完成，并存储于 CDN。客户端需实现的是补丁应用（patch apply）、完整性校验和失败回滚，若将 diff 计算也放在客户端则会导致更新过程中CPU占用率飙升，严重影响用户体验。

## 知识关联

补丁系统以**内容包系统**（AssetBundle/PAK）为直接前置依赖——只有资源被确定性地打包成标准格式后，补丁系统才能对两次打包的产物做有意义的字节级比对。若内容包的打包过程是非确定性的（如包含随机生成的GUID或时间戳），则每次构建即使资源未变也会产生巨大差异，补丁优化效果完全失效。

在资源管理的整体架构中，补丁系统处于分发层的末端，其上游是构建流水线（负责生成清单和差分文件），其下游是客户端的资源加载器（负责在加载时正确路由到已打补丁的最新版本文件）。理解 bsdiff 算法的内存模型有助于在评估补丁服务器规格时做出合理的硬件选型——对于拥有数十GB内容的AAA游戏，构建一次全量跨版本补丁图可能需要数百GB内存和数小时的计算时间，这是补丁方案选型时不可忽视的工程约束。
