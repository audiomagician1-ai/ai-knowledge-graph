# 流式系统

## 概述

流式系统（Streaming System）是游戏引擎资源管理模块中的一种机制，负责在游戏运行时根据玩家的位置、视野或游戏进度，**动态决定哪些资产需要加载进内存、哪些可以从内存中卸载**。与一次性将所有资产打包进内存的静态加载方式不同，流式系统将资产的生命周期从"启动时加载、结束时释放"缩短为以帧为单位的持续调度过程。

流式系统的工业化应用可追溯到 PS2 时代的赛车游戏，如《Gran Turismo 3》（2001年）为了在 32MB 主内存限制下呈现连续赛道，工程师将赛道切割为分段资产包，当摄像机前进时依次流入新段并淘汰旧段。现代引擎如 Unreal Engine 5 的 Nanite 与 Virtual Shadow Maps 同样依赖流式系统，以 GPU 可见性剔除结果驱动每帧的显存页面调度。

流式系统的价值在于它使"内存预算"与"游戏世界规模"解耦。没有流式系统，一款包含 100GB 美术资产的开放世界游戏将无法在 16GB 内存的主机上运行；有了流式系统，引擎只需在任意时刻维持约 2–4GB 的活跃资产集合，剩余资产留在磁盘或 SSD 上等待调度。

## 核心原理

### 优先级驱动的调度器

流式系统的核心是一个**优先级队列调度器**，它为每一个候选资产计算一个加载优先级分值 `P`。常见公式形如：

```
P = W_dist × (1 / distance²) + W_vel × predictedFrames + W_lod × lodBias
```

其中 `distance` 是玩家到资产包围盒的距离，`predictedFrames` 是基于玩家移动速度预测该资产进入视野所需的帧数，`lodBias` 是手动配置的层级权重。优先级最高的请求优先占用 I/O 带宽，保证玩家走到某区域时所需资产已在内存中就绪，而不是看到模糊的占位网格（pop-in）。

### 内存预算与驱逐策略

流式系统维护一块固定大小的**资产内存池**，例如 Unreal Engine 的 `r.Streaming.PoolSize` 默认值为 1000MB。当新资产请求进入队列而内存池已满时，系统必须执行**驱逐（Eviction）**：将当前优先级最低的已加载资产标记为可释放，等其引用计数归零后归还内存页。常用驱逐算法包括 LRU（最近最少使用）和基于距离的贪心驱逐。驱逐过于激进会导致"抖动"（thrashing）——同一资产反复加载卸载，产生大量 I/O 开销；驱逐过于保守则导致内存溢出崩溃。

### I/O 异步管道与依赖解析

流式系统并不直接调用同步文件读取，而是向**异步 I/O 层**提交读取请求（这正是它以异步加载为前置知识的原因）。一个纹理资产可能依赖于特定的压缩格式解码器（如 BC7 或 ASTC），而一个角色网格可能依赖于骨架资产先行就绪。流式系统维护一张**依赖图（Dependency Graph）**，只有当某资产的所有依赖均已完成加载，该资产才会从"待解析"状态切换为"可用"状态，并通知渲染线程替换占位资产。

### LOD 与 Mip 流式

对于大型纹理，流式系统通常不等待最高分辨率的 Mip0 加载完成再显示，而是**从低分辨率 Mip 开始逐级加载**。Unreal Engine 的纹理流式系统允许引擎先在 60ms 内加载 Mip4（原始纹理的 1/16 尺寸），再在后续 200–500ms 内将 Mip3、Mip2 依次流入，使玩家看到纹理从模糊逐渐变清晰，而非长时间等待后突然跳出高清版本。

## 实际应用

**开放世界游戏地形分块**：《荒野大镖客：救赎 2》将游戏地图分割为数千个流式地块（Streaming Cell），每个地块包含地形网格、草地实例、建筑物等资产。当玩家骑马以约 12m/s 的速度穿越时，流式系统需提前约 300 帧（5秒）开始加载前方地块，以掩盖 HDD 约 100–150MB/s 的读取延迟。

**角色换装系统**：在 MMO 游戏中，玩家换装时旧装备的纹理和网格应立即进入候选驱逐列表，新装备资产则以最高优先级插入加载队列。流式系统通过引用计数确认旧资产已无渲染引用后，才将其内存页面回收并分配给新装备资产，整个过程对玩家仅表现为约 200–400ms 的换装延迟。

**过场动画预加载**：在触发剧情过场前，游戏逻辑层会向流式系统发送"预热请求"，强制将过场所需的高精度角色模型和 4K 纹理提升优先级。这种**主动预流（Proactive Streaming）**模式比被动的基于距离的触发提前 3–10 秒开始 I/O 操作，避免过场画面首帧出现资产缺失。

## 常见误区

**误区一：流式系统等同于异步加载**。异步加载只解决"如何在不阻塞主线程的情况下读取文件"，而流式系统还额外负责"何时加载、何时卸载、加载哪个 LOD 级别、内存不足时驱逐谁"。异步加载是流式系统的 I/O 执行层，流式系统是更高层的调度决策系统，两者处于不同的抽象层级。

**误区二：增大流式内存池能彻底消除 pop-in**。Pop-in 的主要原因不是内存不足，而是 I/O 带宽不足或预测算法不准确。将内存池从 1GB 扩大到 4GB 可以减少驱逐频率，但如果玩家突然传送到陌生区域，再大的内存池也无法在 16ms 内完成磁盘读取，pop-in 依然会发生。真正的解决方案是传送时插入加载过渡界面，给流式系统 1–2 秒的填充窗口。

**误区三：流式系统卸载资产时会立即释放内存**。多数引擎实现中，被标记为驱逐的资产会进入一个**冷却队列（Cool-down Queue）**，保留约 2–5 秒再实际释放，以应对玩家快速折返的情况。开发者若直接监控内存分配器的峰值数据，会误以为内存泄漏，但实际上只是冷却队列尚未清空。

## 知识关联

**前置概念——异步加载**：流式系统的所有 I/O 请求均通过异步加载接口提交，理解异步 I/O 的回调机制、完成事件通知以及线程安全的引用计数是使用流式系统 API 的必要基础。不掌握异步加载就配置流式参数，容易在 I/O 完成回调中产生数据竞争。

**后续概念——关卡流式加载**：关卡流式加载是流式系统在"关卡（Level）"粒度上的具体实现，它将整个子关卡（Sublevel）作为一个原子加载单元，并在流式系统之上添加了关卡可见性状态机（Unload → Loading → Loaded → Visible）。掌握流式系统的优先级调度和内存驱逐逻辑后，理解关卡流式加载中为何需要"加载完成但不可见"这一中间状态将变得直观——这正是为了让资产完成所有依赖解析后再切换渲染状态。
