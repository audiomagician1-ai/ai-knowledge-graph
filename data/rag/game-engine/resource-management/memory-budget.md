# 内存预算

## 概述

内存预算（Memory Budget）是指游戏在目标平台上可使用的物理内存总量上限，以及如何在纹理、网格、音频、代码、运行时数据等各类资源之间进行合理分配的策略体系。超出内存预算会直接导致操作系统强制终止进程（iOS的"Jetsam"内存回收机制）或触发页面交换（PC平台的虚拟内存），两种情况都会造成游戏崩溃或严重卡顿。

内存预算概念随主机游戏开发的成熟而逐步形成体系。PlayStation 2时代开发者只有32MB RAM可用，必须精确到KB级别地规划每类资源的占用量，由此催生了系统化的预算管理方法。现代平台的内存容量虽已大幅提升，但移动端（如iOS设备对单个App的内存警告阈值通常在总RAM的45%~50%）和主机端（PS5操作系统保留约2.5GB RAM）依然存在严格的实际可用上限，使内存预算成为资源管理的基础约束。

内存预算的意义在于它是一个不可压缩的硬性限制——与帧率或画质不同，内存溢出不会优雅降级，而是直接导致游戏崩溃。因此在项目立项阶段就必须确定各平台的预算数字，并在整个开发周期内持续监控实际占用量。

## 核心原理

### 各平台内存限制的实际数值

不同平台的可用内存差异悬殊，开发者必须掌握具体数字。PS5拥有16GB GDDR6统一内存，但操作系统保留约2.5GB，游戏实际可用约13.5GB。Xbox Series X同样为16GB，可用量与PS5相近。Nintendo Switch拥有4GB RAM，其中操作系统占用约1GB，游戏可用约3GB，掌机模式下GPU内存共享进一步压缩可用量。iOS设备方面，iPhone 15 Pro的8GB RAM中，单个游戏App收到`didReceiveMemoryWarning`的阈值通常在1.4GB~2.0GB之间，超过约2.8GB则面临被系统强杀的风险。Android碎片化严重，低端机型（2GB RAM）的游戏可用内存可能仅有约700MB。

### 内存分类与分配比例策略

内存预算需要按资源类型进行分区管理，典型的分配框架将总预算划分为以下几类：**纹理内存**通常占总预算的40%~60%，是最大的单类消耗；**网格与骨骼数据**占10%~15%；**音频**占5%~10%；**代码与引擎运行时**（包括Unreal Engine的基础开销约200MB~400MB）占10%~20%；**运行时动态分配**（粒子、物理、AI数据）需预留10%~15%的余量。这些比例并非固定，开放世界游戏的纹理占比可能高达70%，而音乐游戏的音频占比则可能超过30%。

### 内存预算的测量与追踪方法

准确测量内存占用需要区分两个关键指标：**驻留内存（Resident Set Size, RSS）**是操作系统实际分配的物理页面总量，**虚拟内存（Virtual Memory Size, VMS）**则包含已映射但未加载的部分。游戏引擎中常用的预算追踪公式为：

```
可用余量 = 平台硬限制 - 引擎基础开销 - 当前场景已加载资源 - 运行时动态内存
```

Unreal Engine提供`stat memory`命令显示分类内存占用；Unity的Memory Profiler工具可抓取快照并对比两帧之间的内存变化。开发阶段应在目标设备上每日进行内存快照采集，而非依赖编辑器中的估算值，因为编辑器版本的引擎本身额外消耗约200MB~500MB内存。

### 流送（Streaming）作为预算控制手段

当场景所需资源总量超过内存预算时，流送系统负责动态加载和卸载资源。内存预算直接决定流送系统的设计参数：预算越紧张，流送粒度越细，I/O开销越大，出现"资源弹出（Pop-in）"的风险越高。PS5的SSD读取速度达5.5GB/s，使得更激进的小预算+高速流送策略成为可能；而Switch的eMMC读取速度仅约100MB/s，迫使开发者维持更大的内存驻留量。

## 实际应用

**移动端崩溃率控制**：某款移动RPG在iOS上的崩溃报告显示90%的崩溃由内存溢出触发。团队通过将纹理预算从512MB压缩至380MB、将音频流式加载替换为全内存加载的混合策略，使iOS崩溃率从1.2%降至0.15%。

**主机平台分级预算**：开发跨平台游戏时，通常为不同平台设立独立的预算配置文件。以一个动作游戏为例：PS5配置文件允许8GB纹理预算，Xbox One版本压缩至4GB，Switch版本进一步限制至1.8GB。美术团队为每个预算级别维护不同的纹理Mipmap设置和网格LOD链。

**关卡加载时内存峰值管理**：关卡切换时会出现新旧资源同时驻留的内存峰值（Double-buffer loading peak），峰值可能超出平台限制。解决方案是在卸载旧关卡和加载新关卡之间插入一个强制GC（垃圾回收）阶段，将峰值控制在预算内。

## 常见误区

**误区一：用开发机内存衡量目标平台**。开发用PC通常配备32GB或64GB内存，引擎在此环境下不会触发内存限制，导致资源体量在开发后期才暴露超标问题。正确做法是从项目初期就在真实目标设备上建立内存基准线，并在CI/CD流程中加入自动化内存检测。

**误区二：内存预算等于RAM总量**。初学者常将平台标称的RAM总量当作可用预算，忽视操作系统保留、引擎基础开销和必要的碎片化余量。Switch的4GB RAM实际游戏可用量不超过3GB，如果不预留10%~15%的碎片化缓冲，内存分配器在运行时极易失败。

**误区三：一次性制定预算后不再调整**。内存预算应随项目阶段动态修订。Alpha阶段的预算分配往往不能反映最终功能规模——UI系统、存档数据、在线功能等都会带来额外开销，需要在里程碑节点重新审核预算分配比例。

## 知识关联

内存预算建立在**资源管理概述**所介绍的资产生命周期（加载、驻留、卸载）基础上——只有理解资源何时进入和离开内存，才能有效规划预算的动态变化。内存预算作为总量约束，直接拆解为下游的**纹理预算**（管理纹理资源的具体内存策略，通常是预算中占比最大的单项）和**网格预算**（管理几何体数据的内存分配）。这两个子预算都必须在总内存预算的框架内独立制定上限，且三者之和加上其他类别开销不得超过平台硬限制。掌握内存预算的数值约束，是理解LOD系统、纹理流送和资源打包策略等后续主题的必要前提。
