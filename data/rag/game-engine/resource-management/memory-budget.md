# 内存预算

## 概述

内存预算（Memory Budget）是指游戏在目标平台上可以安全使用的内存总量上限，以及将这一总量按资源类型和系统模块进行分配的策略。不同于PC平台的虚拟内存机制，主机和移动平台的物理内存是严格有限且不可超量的——例如PlayStation 4的系统RAM为8GB GDDR5，但游戏开发者实际可用量约为5.5GB，其余被操作系统和社交功能占用。超出内存预算会直接导致游戏崩溃（Out of Memory Kill，简称OOM Kill），而非仅仅性能下降，因此内存预算是资源管理中唯一具有硬性约束的维度。

内存预算的概念随着主机游戏的发展而逐渐形成。早期如NES平台仅有2KB系统RAM，开发者通过硬件技巧手动管理每一个字节。进入PS3/Xbox 360时代（2005-2006年），两款主机分别提供256MB和512MB RAM，内存预算成为跨平台开发中必须量化管理的工程约束，现代化的内存预算分配工具和流水线开始出现。如今在移动端，iOS设备会在应用内存超过设备物理RAM的约40%~60%时触发低内存警告并最终终止进程，使得内存预算对移动游戏的稳定性至关重要。

## 核心原理

### 平台内存限制的构成

各平台的可用内存并非等于硬件标称值。以Nintendo Switch为例，主机模式下总RAM为4GB，游戏可用上限约为3.2GB；掌机模式下系统会额外保留资源，可用量进一步缩减至约2.8GB。这种"物理总量 − 系统保留量 = 开发者预算"的公式是内存预算计算的起点。

| 平台 | 物理RAM | 游戏可用上限 |
|------|---------|-------------|
| PS4 | 8GB GDDR5 | ~5.5GB |
| Xbox One | 8GB DDR3 | ~5GB |
| Nintendo Switch（主机模式） | 4GB | ~3.2GB |
| iOS（中端机型） | 3GB | ~1.2~1.5GB |

移动平台尤其特殊，因为iOS和Android不公开具体内存门限值，开发者需要使用Xcode Instruments或Android Memory Profiler实测触发OOM的阈值，再以该阈值的80%作为实际内存预算的安全上限。

### 内存预算的分配分层

确定总预算后，需将其分配给引擎的各个子系统。业界通用做法是采用"分层预算表"，典型的一张分配表如下所示（以PS4的5.5GB为例）：

- **纹理资源**：约2.0~2.5GB（占比最大，因纹理在GPU内存和系统内存中均有拷贝）
- **网格与几何体**：约300~500MB
- **音频流和音频缓冲**：约200~400MB
- **动画数据**：约150~300MB
- **引擎运行时（代码、堆、栈）**：约300~500MB
- **预留缓冲（Safety Margin）**：约200~300MB（约占总预算的5%~10%，用于应对运行时峰值）

分配比例并非固定，开放世界游戏会向纹理和流式加载倾斜，而叙事向游戏可能会给音频和动画分配更多份额。

### 静态预算与动态预算

内存预算分为**静态预算**和**动态预算**两种管理模式。静态预算在项目立项时由技术总监（Technical Director）确定各系统的硬性上限，并在引擎资产管道（Asset Pipeline）中通过断言（Assert）强制执行，任何超出配额的资产提交都会被自动拒绝。动态预算则根据游戏当前所在场景调整各系统的实时配额——例如在室内战斗场景中，渲染纹理预算可以压缩，将释放出的内存临时划拨给物理模拟和AI寻路系统。

虚幻引擎5的`FPlatformMemory::GetStats()`接口以及Unity的`Profiler.GetRuntimeMemorySizeLong()`均提供了运行时查询已用内存的API，是实现动态预算调节的基础工具。

## 实际应用

**跨平台移植中的内存预算适配**：当一款游戏从PS4移植到Nintendo Switch时，总可用内存从5.5GB下降到约3.2GB，缩减了约42%。Panic Button在将《毁灭战士：永恒》移植到Switch时，通过降低流式纹理分辨率、压缩音频采样率（从48kHz降至32kHz）、减少同屏可见网格LOD层级等手段，将内存占用压缩至Switch的预算范围内。

**移动游戏的动态内存预警系统**：《原神》在iOS端会监听`UIApplication.didReceiveMemoryWarningNotification`通知，一旦触发即卸载当前场景中非必要的角色纹理Mipmap高级别，主动释放约150~200MB内存，从而将进程内存维持在安全预算线以下，避免系统强制终止。

**开发阶段的预算追踪**：在持续集成（CI）管道中，每次资产提交后自动运行内存分析脚本，将各类资产的内存占用汇总写入每日报告。如果某一类别的累计占用超过预先定义的预算阈值（如纹理超过2GB），流水线会向资产团队发出警告邮件，而非等到最终打包时才发现问题。

## 常见误区

**误区一：PC开发中内存预算不重要**
部分开发者认为PC有虚拟内存和几乎无限的交换空间，无需管理内存预算。实际上，当GPU显存（VRAM）超出限制时，DirectX/Vulkan会将资源降级到系统内存，导致帧时间（Frame Time）从2ms暴增至20ms以上，造成严重卡顿。即便不崩溃，超出VRAM预算的代价也是玩家可感知的性能劣化。

**误区二：内存预算等于峰值内存占用**
内存预算是需要主动规划的上限目标，而峰值内存占用是被动测量的结果。实际开发中，场景切换瞬间（新场景开始加载但旧场景资产尚未卸载）会出现内存占用的双峰叠加，这一瞬态峰值往往比稳定运行时高出30%~50%。内存预算必须为这一加载过渡期专门预留空间，而不能按正常运行时的平均占用来制定。

**误区三：内存预算只与程序员相关**
内存预算对美术资产的制作规范有直接约束。一张4096×4096像素的未压缩RGBA纹理占用64MB显存，而同尺寸使用BC7压缩格式后仅占约16MB。美术团队若不了解各资产类型在其预算分配中的权重，就无法在视觉质量和内存成本之间做出有据可依的决策。

## 知识关联

**前置概念**：资源管理概述建立了资源生命周期（加载、使用、卸载）的基本框架，内存预算正是在此框架下量化"使用"阶段的上限约束，并倒推对加载策略（是否流式、何时卸载）的要求。

**后续概念**：**纹理预算**是内存预算中占比最大的子分类，专注于GPU显存中纹理Mipmap的分配与压缩格式选择；**网格预算**则管理顶点缓冲区和索引缓冲区的内存分配。理解整体内存预算的分层结构之后，学习纹理预算和网格预算时便能清楚地知道每项技术决策（如纹理压缩、LOD层级）最终服务于哪个量化目标，而非孤立地理解单项优化技术。
