# 角色控制器

## 概述

角色控制器（Character Controller）是游戏物理引擎中专门用于驱动可操控角色（玩家或NPC）移动的组件，它不依赖刚体的力和力矩来产生位移，而是通过直接指定速度或位移向量来精确控制运动轨迹。与普通刚体物理对象不同，角色控制器会主动检测并响应周围几何体，同时屏蔽掉物理引擎中惯性、摩擦力等会破坏手感的真实物理模拟。

角色控制器的概念最早随着3D动作游戏的兴起而成形。在早期的《雷神之锤》（Quake，1996年）时代，开发者就发现用标准刚体模拟玩家角色会导致移动"滑溜"且难以精确控制，因此抽象出独立的角色移动逻辑。现代引擎中，Unity的`CharacterController`组件和Unreal Engine的`UCharacterMovementComponent`都是这一思路的成熟实现，两者均以胶囊体作为默认碰撞形状。

角色控制器之所以重要，是因为游戏中玩家角色的移动手感直接决定游戏体验。一个好的角色控制器能够无缝处理上下台阶、走斜坡、卡在墙角时自动滑开等边界情况，让玩家专注于游戏操作而非与环境碰撞做斗争。

## 核心原理

### 胶囊体碰撞形状

角色控制器几乎universally采用**胶囊体（Capsule）**作为碰撞形状，而不是盒体或球体。胶囊体由一个圆柱体加上两端的半球体组成，在Unity中由`radius`（半径）和`height`（总高度）两个参数定义，例如典型的人形角色设置为 radius=0.5m、height=2.0m。

选择胶囊体的原因非常具体：
- **底部半球**可以让角色在接触地面的小凸起或台阶边缘时自然"滑过"，避免被毫米级的几何细节卡住。
- **侧面无棱角**意味着角色贴墙行走时，接触点会沿曲面光滑滑动，不会像盒体那样在棱角处产生异常旋转力矩。
- **顶部半球**让角色在低矮障碍物下方自动偏折，而不是直接停下。

相比之下，若使用盒体碰撞，角色走过两块地板砖之间的微小缝隙时极容易"绊倒"停住，这在实际开发中是高频BUG来源。

### 楼梯步进（Step Offset）

楼梯步进是角色控制器中专门处理台阶的机制，通过一个称为`stepOffset`（步高阈值）的参数来控制。Unity中该参数默认值为0.3m。当角色前进方向检测到高度差**小于等于stepOffset**的垂直障碍时，控制器会自动将角色"抬起"越过该障碍，而非将其视为不可通过的墙壁。

步进检测的具体流程分三步：
1. 在角色底部以水平射线检测前方是否存在障碍。
2. 若障碍高度 ≤ stepOffset，则在内部将角色垂直位移至障碍顶部高度。
3. 继续执行水平移动，完成"上台阶"的平滑过渡。

`stepOffset`数值设定需要谨慎：设得过大（如0.8m）会导致角色能"飞跃"明显的高台；设得过小（如0.05m）则会在标准台阶前停住。实际游戏中台阶高度通常在0.15m至0.25m之间，因此0.3m是留有余量的合理默认值。

### 斜面处理（Slope Limit）

角色控制器通过`slopeLimit`参数定义可行走斜面的最大角度，Unity默认值为45度。当地面法线与垂直方向的夹角超过该阈值时，控制器将斜面视为墙壁，角色会从斜面上滑落而无法站立。

斜面处理的内部计算依赖于接触法线向量 **n̂** 与世界向上向量 **(0,1,0)** 的点积：

```
θ = arccos(n̂ · (0,1,0))
```

若 θ > slopeLimit，则该接触面被标记为不可站立，角色控制器会沿斜面的切线方向施加一个向下滑动的速度分量，使角色从陡坡上溜走。这一行为可防止玩家靠在超过45度的悬崖边缘"挂住"不动的穿模现象。

此外，角色在斜面上行走时，水平移动速度也需要投影到斜面切平面上，否则角色会在缓坡上表现出异常的加速或减速感。

## 实际应用

**第一人称射击游戏（FPS）**：《反恐精英》系列中，玩家在楼梯处不会有明显的"跳帧"感，依靠的正是stepOffset的平滑步进逻辑配合相机插值。

**开放世界地形行走**：在Unity Terrain（地形系统）上行走时，角色控制器的斜面检测会实时计算地形网格的局部法线，使角色在山坡上自动停步，无需手动编写"是否在坡上"的判断逻辑。

**调用接口**：Unity的角色控制器核心API是`CharacterController.Move(Vector3 motion)`，该方法每帧接收一个位移向量，返回一个`CollisionFlags`枚举值，其中`CollisionFlags.Below`表示角色正站在地面上，开发者通过判断此标志来控制跳跃逻辑：只有`isGrounded == true`时才允许起跳。

## 常见误区

**误区一：认为角色控制器会自动处理重力**
角色控制器本身**不会自动施加重力**。开发者必须在每帧的Move调用中手动累加向下的速度分量（如 `velocity.y -= 9.8f * Time.deltaTime`）。初学者常常发现角色"漂浮"在空中，原因正是遗漏了重力计算，而非角色控制器的Bug。

**误区二：混淆stepOffset与物理碰撞的关系**
步进机制是角色控制器的**运动学逻辑**，不经过物理引擎的碰撞响应系统。这意味着一个高度为0.2m的障碍物，对于角色控制器（stepOffset=0.3m）是透明的，但对场景中其他的刚体对象仍然是实体障碍。不理解这一点会导致设计关卡时出现"为什么刚体小球滚不上去，玩家却能走上去"的困惑。

**误区三：用角色控制器模拟车辆或物理交互**
角色控制器屏蔽了力和冲量，导致它无法对其他刚体施加物理推力。如果在角色控制器组件的基础上直接推动场景中的木箱，木箱不会移动——必须通过代码检测`OnControllerColliderHit`回调，手动调用`Rigidbody.AddForce`才能实现推箱效果。

## 知识关联

角色控制器以**碰撞形状**为基础：胶囊体是碰撞形状中的一种，理解碰撞形状的几何参数（如radius、height）是正确配置角色控制器的前提。胶囊体与环境几何体的相交检测（Overlap Test）是步进和斜面处理算法能够运行的底层依据。

在更广泛的物理引擎体系中，角色控制器与**运动学刚体（Kinematic Rigidbody）**概念相邻——两者都不受力驱动，但角色控制器额外封装了步进、斜坡、皮肤厚度（`skinWidth`，Unity默认0.08m）等游戏常用逻辑，是运动学刚体在角色移动场景下的专用封装。掌握角色控制器后，开发者通常会进一步研究自定义角色移动系统（如基于射线检测的手写控制器），以突破内置组件在处理动态平台、游泳、攀爬等特殊状态时的局限性。
