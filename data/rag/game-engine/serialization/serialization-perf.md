# 序列化性能

## 概述

序列化性能是指游戏引擎在将对象状态转换为字节流（序列化）或将字节流还原为对象（反序列化）时所消耗的时间、CPU 周期和内存空间的量化指标。与普通的逻辑运算不同，序列化操作在游戏中频繁发生于存档读写、网络同步和资源加载三大场景，低效的实现直接导致加载卡顿或帧率下降。

JSON 格式由于需要逐字符解析文本，其读写速度通常比二进制格式慢 3～10 倍；而 Protocol Buffers（protobuf）采用 varint 编码和固定偏移访问，在相同数据量下编解码延迟可低至 JSON 的 1/5。游戏引擎选择序列化方案时，性能数据必须通过 Benchmark（基准测试）来量化，而非凭经验估算。

序列化性能之所以受到特别关注，是因为游戏的实时性要求极高。一款 60 FPS 的游戏每帧仅有约 16.6 毫秒的预算，如果单次地图加载的反序列化耗时超过 500 毫秒，玩家会明显感知到卡顿。Unreal Engine 的 `.uasset` 文件采用自定义二进制格式，正是因为其反序列化速度比通用格式快约 40%。

## 核心原理

### 读写速度的影响因素

序列化速度由三个主要因素决定：**字段访问开销**、**内存分配次数**和**编码方式**。

- **字段访问开销**：反射（Reflection）驱动的序列化（如 Unity 的 `JsonUtility`）需要在运行时查询字段元数据，每次字段查询可能引入数十纳秒的额外开销。相比之下，手写的序列化代码或代码生成方案直接通过内存偏移访问字段，省去了反射查询。
- **内存分配次数**：每次调用 `new`/`malloc` 都会触发堆分配，在序列化一个包含 1000 个节点的对象图时，频繁分配小对象可导致 GC 压力暴增。FlatBuffers 采用"零拷贝"读取设计，反序列化时不需要额外分配堆内存，直接从字节缓冲区读取，因此其反序列化速度接近于**零开销**。
- **编码方式**：varint 对小整数（如 0～127）只用 1 字节编码，但对大整数需要 2～5 字节，且存在额外的分支判断。定长编码（如 fixed32）则无分支，适合值域固定的字段。

### 大小优化策略

序列化数据的体积直接影响磁盘 I/O 时间和网络带宽消耗。常见的压缩策略包括：

- **Delta 编码**：仅记录与上一帧或参考状态的差值。在网络同步场景中，若角色位置每帧变化不超过 1 单位，用 Delta 编码后每次同步的位置数据可从 12 字节（3 × float32）压缩至 3 字节（3 × int8 增量）。
- **位字段打包（Bit Packing）**：布尔型数组中每个 bool 通常占 1 字节，将 8 个 bool 打包进 1 个字节后，相同信息量的存储空间缩减为 1/8。
- **字符串池（String Interning）**：重复出现的字符串（如道具名称 `"Sword"`）只存储一次，其余位置用整数索引代替，适合存档文件中大量重复的标签字段。

典型的二进制游戏存档在经过位字段打包和 Delta 编码后，体积可比原始 JSON 缩减 60%～80%。

### Benchmark 方法论

游戏引擎中评估序列化性能需要遵循严格的基准测试规范，否则测试结果会因 CPU 缓存预热、GC 时机不同而产生数倍的误差。标准做法包括：

1. **预热（Warmup）阶段**：至少运行 1000 次迭代后丢弃结果，确保 JIT 编译（针对 C#/Java 运行时）和 CPU 缓存已稳定。
2. **测量指标**：同时记录**吞吐量**（MB/s 或 ops/s）和**延迟百分位**（P50、P95、P99），因为平均值会掩盖偶发的长尾耗时。
3. **数据代表性**：使用接近真实游戏的测试数据（如一个包含 500 个实体、各带 10 个组件的场景），而不是简单的整数数组，因为字符串和嵌套对象的序列化特性与基础类型差异显著。

在 Unity 中可使用 `System.Diagnostics.Stopwatch` 配合 `GC.Collect()` 手动控制 GC，确保测量的是纯序列化耗时而非垃圾回收时间。

## 实际应用

**游戏存档系统**：RPG 游戏的存档文件通常包含玩家属性、背包道具、任务状态等大量字段。《Minecraft》的区块数据使用 NBT（Named Binary Tag）格式而非 JSON，单个区块文件的读取速度因此比等效 JSON 快约 5 倍，这是其能够流畅加载大型世界的基础之一。

**网络状态同步**：在每秒 20 次状态更新（20 Hz）的多人游戏服务器上，若每个玩家每帧同步数据包为 200 字节，则 100 名玩家每秒产生的序列化数据为 200 × 20 × 100 = 400 KB/s。通过 Delta 压缩将单包从 200 字节降至 40 字节，服务器带宽消耗直接减少 80%。

**资源热加载（Hot Reload）**：开发阶段频繁修改资源时，编辑器需要快速重新序列化和反序列化资源文件。Unreal Engine 的 Hot Reload 之所以在修改 C++ 代码后仍能在数秒内完成，部分原因在于其资源的二进制序列化格式支持**增量写入**，只重写发生变更的数据块而非整个文件。

## 常见误区

**误区一：文本格式"够快了"，没必要优化**

开发者常用 JSON 实现原型系统，并在测试时认为读写速度可以接受。但此类测试往往使用的是极小的数据集（数十个字段），而实际游戏存档包含数千个对象。当数据量从 50 个实体增长到 5000 个实体时，JSON 解析的时间复杂度近似线性增长，延迟可能从 5 毫秒跃升至 500 毫秒，触碰玩家感知阈值。

**误区二：越小的体积越好，压缩越激进越优**

高压缩率通常以 CPU 时间为代价。LZ4 压缩算法的压缩速度约为 500 MB/s，解压速度约为 2 GB/s；而 LZMA（7-Zip 使用的算法）压缩率更高但压缩速度仅约 20 MB/s。若游戏在加载时需要实时解压，选择 LZMA 可能使加载时间比 LZ4 长 10 倍以上，得不偿失。正确做法是测量**整体 I/O + 解压时间**，而非单独优化体积。

**误区三：手写序列化代码一定比框架快**

不经优化的手写序列化代码可能因错误的内存访问顺序（非缓存友好）或过多的条件分支而比成熟框架更慢。protobuf 的 C++ 实现经过多年的 SIMD 指令优化，其编解码速度在连续内存布局下可达约 1 GB/s，普通手写代码难以超越。

## 知识关联

**前置概念**：序列化概述建立了"对象→字节流"转换的基本模型，本文的所有性能讨论都建立在对该转换过程已有清晰认知的基础上。若不理解字段如何被逐一编码，就无法评估 Delta 编码或位字段打包的具体优化意义。

**横向关联**：序列化性能与**内存管理**和**文件 I/O** 密切交织。零拷贝反序列化（FlatBuffers）的性能优势完全依赖于内存映射文件（mmap）机制；而磁盘顺序读写比随机读写快约 100 倍的特性，决定了游戏引擎通常将同类资源连续打包（如 `.pak` 文件），以最大化序列化数据的 I/O 效率。

**工程决策参考**：选择序列化格式时，建议使用 Google 开源的 `benchmark` 库或引擎自带的 Profiler 采集真实场景数据，再结合读写速度、数据体积和代码维护成本三个维度做权衡，而非仅凭单一指标决策。
