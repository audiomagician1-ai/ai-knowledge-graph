# 序列化性能

## 概述

序列化性能是指游戏引擎在将内存中的对象状态转换为字节流（序列化）或将字节流还原为对象（反序列化）时所消耗的时间、内存以及最终产出的数据体积大小。在游戏引擎语境中，序列化性能直接影响场景加载时长、存档写入帧率波动、网络同步延迟三个关键指标，因此它不是一个可以事后再优化的边缘问题，而是设计数据格式时必须优先考量的约束条件。

序列化性能的量化研究随着游戏规模的扩大而成为工程焦点。2010年前后，随着开放世界游戏兴起，单个场景文件从几 MB 膨胀至数百 MB，Unity 引擎的 YAML 文本序列化格式在加载 500 个以上游戏对象时出现明显卡顿，促使 Unity 5 引入了二进制序列化的 `.assets` 格式以替代部分场景存储需求。同期，Unreal Engine 也针对 `.uasset` 包文件的反序列化引入了 async loading 机制，将主线程阻塞时间缩短至单帧 1ms 以内。

衡量序列化性能通常使用三个维度的 Benchmark 指标：**吞吐量**（Throughput，单位 MB/s）、**延迟**（Latency，单位 μs/ms）和**压缩比**（Compression Ratio = 原始大小 / 序列化后大小）。游戏引擎开发者需要根据具体场景选择在这三者之间取得平衡，而非盲目追求任意一个极值。

## 核心原理

### 二进制格式 vs 文本格式的吞吐量差异

文本格式（如 JSON、YAML、XML）需要将每个数值转换为 ASCII 字符串再写入，一个 `float` 数值 `3.14159` 在内存中占 4 字节，序列化为文本后至少占 7 字节，且解析时需要字符串转浮点数的额外运算，通常需要 `atof()` 或等价函数调用。相比之下，二进制格式直接将 4 字节内存内容写入字节流，解析时只需一次内存拷贝（`memcpy`），无需任何字符转换。

基准测试数据表明，对同一个包含 10,000 个 Transform（位置 + 旋转 + 缩放，共 9 个 float）的场景对象列表，MessagePack 二进制格式的序列化吞吐量约为 800 MB/s，而 JSON 格式约为 120 MB/s，差距约 **6.7 倍**。FlatBuffers 由于零拷贝设计，反序列化速度可达 **2,000 MB/s** 以上，因为它根本不进行数据解包，直接在字节缓冲区内寻址。

### 内存分配对序列化延迟的影响

序列化过程中的堆内存分配（heap allocation）是隐藏的延迟杀手。每次调用 `new` 或 `malloc` 分配动态缓冲区都会触发内存分配器的锁竞争与碎片化。以 C# 中的 `BinaryFormatter` 为例，它在序列化时会为每个字段创建临时 `byte[]` 对象，导致大量 GC 压力。Unity 的 `JsonUtility` 相较于 `Newtonsoft.Json` 的优势之一就在于前者通过 C++ 层避免了逐字段的托管堆分配，实测在序列化 1,000 个 `Vector3` 时，`JsonUtility` 耗时约 0.8ms，`Newtonsoft.Json` 约 4.5ms。

优化策略是使用**预分配缓冲池**（Buffer Pool）：在引擎初始化时分配固定大小的字节缓冲区，序列化时直接向该缓冲区写入，避免运行时的动态分配。`System.Buffers.ArrayPool<byte>` 就是 .NET 生态中的标准实现，它将同尺寸缓冲区复用率提升至接近 100%。

### 数据布局与压缩比优化

序列化后的数据体积影响磁盘 I/O 时间和网络带宽消耗。数据体积优化有两个层次：**结构级优化**和**压缩算法**。

结构级优化包括使用变长整数编码（Varint）——对于值小于 128 的整数只需 1 字节，而固定 `int32` 始终占 4 字节，Protocol Buffers 使用该技术对稀疏整数字段实现平均 40%~60% 的体积缩减。还包括去除冗余字段（例如枚举类型不需要存储字段名称）和使用 Delta 编码（只存储与上一帧的差值，在动画关键帧数据中极为有效）。

压缩算法层面，LZ4 压缩算法以约 400 MB/s 的解压速度著称，适合实时加载；Zstandard（zstd）在压缩比和速度之间取得更好平衡，压缩比通常比 LZ4 高 20%~30%，解压速度约 200~300 MB/s，Unreal Engine 5 的 `.pak` 文件默认支持选择这两种压缩后端。

## 实际应用

**游戏存档系统**：玩家按下存档时，通常处于游戏主循环中，如果存档序列化操作超过 16.6ms（60fps 的单帧预算），就会产生明显的卡顿帧。解决方案是将序列化分为两步：第一步在主线程做数据快照（拷贝状态到临时结构体），耗时通常在 1~2ms；第二步在工作线程异步执行磁盘写入，主线程完全不阻塞。

**网络状态同步**：在多人游戏中，每帧（通常 30Hz 或 60Hz）需要向所有客户端广播玩家位置。一个包含 `position (3×float) + rotation (4×float quaternion) + velocity (3×float)` 的状态包若用 JSON 序列化约 120 字节，用 Protocol Buffers 约 40 字节，用手写二进制打包约 40 字节，但若加入 Delta 编码只发变化量则可缩减至平均 10~15 字节，在 100 名玩家的服务器上每秒节约带宽达 (40-12.5)×100×60 ≈ **165 KB/s**。

**资产热重载（Hot Reload）**：编辑器中修改材质或脚本后，引擎需要快速反序列化新资产并替换内存中的旧版本。此场景对反序列化延迟极为敏感，通常要求单个资产的反序列化完成时间低于 5ms，因此编辑器资产格式（如 Unity 的 `.meta` + 二进制 `.asset`）专门为增量更新设计了局部反序列化能力，只重新解析变更的字段。

## 常见误区

**误区一：压缩算法越强越好**
许多开发者直觉上认为压缩率更高的算法（如 LZMA/7-zip）对游戏更有益。但 LZMA 的解压速度约 20~50 MB/s，比 LZ4 慢 8~20 倍。对于需要在游戏运行时频繁加载的资产，过强的压缩会导致解压时间超过磁盘 I/O 节省的时间，净效果为负。LZMA 只适合用于打包阶段的安装包压缩，而非运行时资产加载。

**误区二：序列化性能主要取决于格式选择**
格式选择（JSON vs Binary）固然重要，但实际瓶颈经常在于**数据结构的设计**而非格式本身。一个包含深层嵌套指针引用的对象图，即便用 FlatBuffers 序列化，也会因为图遍历时的缓存未命中（Cache Miss）而性能低下。将同类型数据按数组结构（Array of Structs → Struct of Arrays）重新排布，可以使 CPU 缓存命中率从不足 40% 提升至 90% 以上，这种布局优化往往比切换序列化格式带来更大的性能提升。

**误区三：Benchmark 结果可以跨场景直接套用**
公开的序列化 Benchmark（如 github.com/nicowillis/serialization-benchmarks）通常使用简单的扁平结构体测试，而游戏引擎中的真实数据往往包含大量字符串（资产路径、组件名称）、嵌套对象和可空字段。字符串的序列化性能在不同格式间差异可达 3~5 倍，与整数/浮点数场景完全不同。因此引擎开发者必须使用**代表自身数据特征的 Benchmark 数据集**进行测量，而非直接引用通用测试数据。
