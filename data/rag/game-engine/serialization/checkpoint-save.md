# 检查点存档

## 概述

检查点存档（Checkpoint Save）是游戏引擎序列化系统中的一种自动触发机制，在玩家到达特定地理位置、完成特定事件或经过固定时间间隔时，无需玩家手动操作即自动将游戏状态写入持久化存储。与手动存档不同，检查点存档由游戏逻辑主动发起，玩家通常只能选择从最近检查点恢复，而非任意时刻。

这一机制最早在1980年代街机游戏中以"关卡继续"形式出现，进入家用主机时代后，《超级马里奥兄弟》的旗杆存档点和《索尼克》系列的星环存档标志着检查点设计的成熟。现代3A游戏如《黑暗之魂》系列将篝火存档点发展为核心玩法机制，而非单纯的技术工具。

检查点存档解决了一个具体的工程问题：在不打断游戏沉浸感的前提下，以最低的玩家操作成本保护游戏进度。与完整存档相比，检查点通常只保存当前关卡内的局部状态，序列化数据量往往在1KB到数百KB之间，远小于完整游戏存档的MB级别。

## 核心原理

### 触发条件与检查点注册

检查点存档的触发分为三类：**位置触发**（玩家角色进入触发体积 Trigger Volume）、**事件触发**（剧情节点、Boss击杀、谜题完成）以及**时间触发**（每隔固定秒数自动写盘，常见于生存类游戏）。在引擎实现中，每个检查点对象需要在关卡加载时向存档管理器（SaveManager）注册自己的唯一ID，防止同一关卡多个检查点写入相同槽位。

```
CheckpointID = LevelID (16 bit) | SequenceIndex (8 bit) | Type (8 bit)
```

这条32位ID编码方案被虚幻引擎的多款第三方插件采用，可在不使用字符串哈希的情况下快速索引检查点槽位。

### 快照数据的范围裁剪

检查点存档不序列化整个游戏世界，而是通过**脏标记（Dirty Flag）**机制只写入自上一个检查点以来发生变化的状态。常见的快照数据包括：玩家坐标与朝向（通常3个float，12字节）、当前生命值与资源值、已激活的敌人死亡列表（使用位掩码压缩，每个敌人占1 bit）、以及场景中可交互物体的状态差异。

Unity中实现这一机制时，常将可存档组件标记为`ICheckpointSaveable`接口，在`OnCheckpointReached()`回调中各自序列化自身的差异状态，最终由中央管理器将所有组件数据合并为一个二进制块写入磁盘或内存缓冲区。

### 快速保存（Quick Save）的特殊性

快速保存是检查点存档的玩家主动变体，通常绑定到F5键（PC端约定俗成的快捷键）。与自动检查点的关键区别在于：快速保存需要捕获**完整的即时状态**，包括敌人的当前AI状态、弹道位置、以及所有正在播放的动画帧数，因为玩家可能在战斗中途触发保存。这使得快速保存的序列化复杂度显著高于位置检查点，存档大小通常是后者的5到20倍。部分引擎选择在快速保存时暂停游戏循环（Pause Game Loop）确保数据一致性，另一些则使用写时复制（Copy-on-Write）内存快照以维持游戏不暂停的体验。

### 恢复时的状态重建

从检查点恢复时，引擎执行的顺序至关重要：首先重新加载关卡基础场景（加载持久层），然后将检查点差异数据覆盖写回相应组件，最后触发`OnCheckpointRestored()`事件让各系统（如音频、粒子、UI）重置自身表现层状态。若顺序错误，例如在关卡加载完成前写入敌人死亡状态，会导致已死敌人的碰撞体仍然存在的"幽灵碰撞"Bug。

## 实际应用

**战斗区域自动存档**：在《战神4》（God of War 2018）中，每个竞技场入口均设置隐藏检查点，玩家进入后立即保存。这意味着玩家死亡后只需重新尝试当前战斗，而非重玩整个探索路段。这类检查点通常被设计为"单向写入"——进入战斗区域覆盖存档，但战斗胜利后才写入下一个检查点，防止玩家在Boss战中途意外触发覆盖存档的问题。

**开放世界的稀疏检查点**：《荒野大镖客：救赎2》（Red Dead Redemption 2）在主线任务中每隔约5分钟触发一次自动检查点，在开放世界探索时则依赖玩家手动存档。这种混合策略体现了检查点密度与存储频率之间的设计权衡——过密的检查点会使玩家放弃手动存档习惯，过稀则增加进度丢失风险。

**移动游戏的云端快照**：移动端检查点存档通常在关卡完成时触发，并通过GameCenter或Google Play Games API异步上传至云端。由于移动设备可能随时被强制终止进程，部分引擎（如Cocos Creator 3.x）在`onHide`生命周期回调中额外触发一次紧急检查点写入本地缓存，数据大小通常被限制在64KB以内以保证写入速度。

## 常见误区

**误区一：检查点存档与完整存档可以互相替代**
检查点存档本质上是"关卡内局部状态"的记录，它通常不包含玩家的全局资产（如背包物品、全局剧情标志、技能树状态）。若开发者将所有进度依赖检查点存档而忽略完整存档，玩家关闭游戏再打开后，全局状态会恢复到进入当前关卡之前的版本，导致资源回退Bug。正确做法是检查点存档与主存档文件分离，前者只覆盖当前会话的局部进度。

**误区二：检查点越密集游戏体验越好**
在《黑暗之魂》系列中，稀疏的篝火检查点（两个篝火之间平均距离约5到10分钟游戏时长）是难度设计的核心组成部分，而非缺陷。过于密集的检查点会消除玩家因死亡产生的代价感，使得风险-回报的核心游戏循环失效。检查点密度需与游戏类型和难度曲线匹配，而非单纯追求便利性最大化。

**误区三：快速保存对性能无影响**
快速保存触发时，主线程需要序列化包括物理模拟状态在内的大量数据。若在60fps的游戏中于主线程同步写入磁盘，单次写入耗时超过16.67ms就会造成明显卡顿。正确实现应将序列化操作提交至工作线程（Worker Thread），主线程只负责生成内存快照，磁盘写入异步完成。

## 知识关联

**前置概念——存档系统**：检查点存档建立在通用存档系统提供的文件IO抽象和序列化协议之上。脏标记机制和差异序列化是存档系统中的基础能力，检查点机制只是在其上增加了自动触发逻辑和作用域限制。掌握二进制序列化格式（如MessagePack或Protocol Buffers）和文件槽位管理是理解检查点内部实现的必要基础。

**横向概念——状态机与场景管理**：检查点在关卡加载和卸载的生命周期中必须与场景管理器协作。检查点数据的有效期与关卡场景的生命周期绑定，当关卡被卸载时，未提交到主存档的检查点数据应当被正确清理，否则会造成内存泄漏或下次进入关卡时读取到陈旧的检查点ID。
