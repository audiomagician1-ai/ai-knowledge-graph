# 二进制序列化

## 概述

二进制序列化是将程序内存中的数据结构直接转换为紧凑字节流的过程，其输出结果是人类不可读的二进制编码，而非JSON或XML那样的文本格式。与文本序列化相比，二进制序列化通常能将数据体积压缩至原始大小的20%~50%，同时将读写速度提升5到10倍，这对于需要频繁进行磁盘IO或网络传输的游戏引擎来说至关重要。

这一技术在游戏开发中有悠久历史。早期主机游戏（如1990年代的SNES时代）受限于卡带容量，就已经将玩家存档直接以固定宽度的内存布局写入SRAM，本质上是最原始的二进制序列化。现代游戏引擎如Unreal Engine和Unity在资产管理、网络同步和存档系统中均大量依赖二进制序列化，Unreal的`.uasset`格式和Unity的`.assets`包文件都是其典型产物。

理解二进制序列化的关键在于掌握三个紧密相关的问题：**如何将内存结构映射为字节流**、**不同机器如何统一字节排列顺序**，以及**数据格式升级后旧文件如何继续可用**。这三个问题对应本文的三个核心原理。

## 核心原理

### 结构体内存布局与字节对齐

当程序将一个C++结构体写入字节流时，最直接的方法是将结构体的内存块整体复制（`memcpy`）到缓冲区。例如以下结构体：

```cpp
struct PlayerData {
    int32_t  health;    // 4字节
    float    posX;      // 4字节
    float    posY;      // 4字节
    uint8_t  level;     // 1字节
    // 编译器会自动填充3字节以对齐
};  // 总占用：16字节
```

编译器为满足CPU对齐要求，会在`level`字段后自动插入3个填充字节（padding），使结构体总大小从13字节变为16字节。如果直接序列化包含padding的内存块，不仅浪费空间，在不同编译器或平台上padding的位置也可能不同，导致数据错误。因此专业的二进制序列化库会逐字段手动写入，跳过padding，或使用`#pragma pack(1)`强制取消对齐再序列化。

### 字节序（Endianness）问题

同一个32位整数`0x01020304`，在大端（Big-Endian）机器上存储为字节序列`01 02 03 04`，而在小端（Little-Endian）机器上存储为`04 03 02 01`。x86/x64 CPU使用小端，而网络标准协议（如TCP/IP）规定使用大端，部分游戏主机（如早期PowerPC架构的Xbox 360和PS3）也使用大端。

这意味着同一份存档文件从PC移植到PS3时，若不进行字节序转换，所有多字节数值将全部错误。转换公式使用位运算实现，以32位整数为例：

```cpp
uint32_t SwapEndian32(uint32_t val) {
    return ((val & 0xFF000000) >> 24) |
           ((val & 0x00FF0000) >>  8) |
           ((val & 0x0000FF00) <<  8) |
           ((val & 0x000000FF) << 24);
}
```

Unreal Engine在其序列化层中通过`FArchive`类的`ByteOrderSwap`函数统一处理字节序，游戏开发者无需手动调用上述转换，但理解其原理是排查跨平台数据损坏的基础。

### 版本兼容性设计

游戏上线后必然需要更新数据格式，版本兼容性是二进制序列化中最容易出错的环节。常见策略有三种：

**版本号标记法**：在二进制文件头部写入一个整数版本号（如`uint32_t version = 5`），反序列化时先读取版本号，再根据版本执行不同的读取逻辑分支。Unreal Engine的`FPackageFileSummary`结构体头部就包含`LegacyFileVersion`字段。

**字段标签法（Tag-Length-Value）**：每个字段写入前先写一个ID（tag）和长度（length），反序列化时遇到未知tag直接跳过对应长度的字节，从而实现前向兼容。这是Protobuf二进制格式的核心机制。

**不可删除原则**：在版本兼容设计中，已发布版本中的字段只能废弃（读取后丢弃），不能从序列化流中真正删除，否则后续字段的偏移量会发生错位，导致整个文件解析失败。

## 实际应用

**游戏存档系统**：一个典型的RPG存档文件通常由文件头（魔数`0x53415645`即ASCII "SAVE"、版本号、校验和CRC32）和数据块组成。数据块按玩家状态、背包物品列表、地图状态等分区存储。使用二进制格式可使一个包含数百件物品的存档文件控制在几十KB以内，而等效JSON文件可能超过500KB。

**网络状态同步**：在多人游戏中，每帧需要同步数百个游戏对象的位置和状态。使用二进制序列化将一个包含位置（3×float=12字节）、旋转（4×float=16字节）、速度（3×float=12字节）的Transform结构压缩为40字节，并可进一步通过量化（Quantization）将每个float压缩为2字节的定点数，使单个对象同步数据降至12字节。

**UAsset资产文件**：Unreal Engine的`.uasset`文件是二进制序列化的复杂应用，其文件头包含导入表（Import Table）和导出表（Export Table），均以二进制格式存储对象之间的引用关系，使引擎可在不加载全部资产的情况下快速解析依赖图。

## 常见误区

**误区一：直接memcpy结构体就是安全的二进制序列化**

许多初学者认为用`fwrite(&myStruct, sizeof(myStruct), 1, file)`写入结构体就完成了序列化。这在同一机器、同一编译选项下确实可行，但一旦换用不同的编译器版本（padding规则可能改变）、不同架构（字节序不同）或修改了结构体定义（字段偏移变化），所有历史文件将全部无法读取，且不会有任何错误提示，只会产生静默数据损坏。

**误区二：二进制序列化不需要考虑浮点数精度**

整数的二进制表示是精确的，但IEEE 754浮点数的二进制序列化存在平台兼容风险。极少数嵌入式平台使用非IEEE 754浮点格式，序列化的float值在这些平台上读取会产生错误结果。此外，将`double`（8字节）降级序列化为`float`（4字节）以节省空间时，必须在设计阶段明确记录精度损失的可接受范围，例如地图坐标使用厘米级精度的float（误差约±0.03米在10000米范围内）是否满足游戏需求。

**误区三：添加字段只需在结构体末尾追加**

在没有版本号保护的情况下，即使只在结构体末尾新增一个字段，旧版程序读取新版文件时会因为文件长度超出预期而触发读取错误；新版程序读取旧版文件时会因为末尾字段缺失而读到垃圾数据。正确做法是先递增版本号，再用`if (version >= NEW_VERSION)`条件包裹新字段的读写逻辑。

## 知识关联

学习二进制序列化需要先掌握**序列化概述**中的基本概念——序列化/反序列化的对称性，以及为何需要将内存状态持久化，这是理解本文中字段对齐和版本兼容问题的前提。

掌握二进制序列化后，可以进入**存档系统**的学习，存档系统是将本文的版本兼容和文件头设计原则落地为具体游戏功能的工程实践。**UAsset格式**是Unreal Engine对二进制序列化的深度定制，其导入/导出表结构直接建立在本文字节布局知识之上。**网络序列化**在二进制序列化基础上增加了带宽限制、丢包容忍和增量更新等约束，量化压缩技术也在此展开。**Protobuf/FlatBuffers**则是解决本文版本兼容问题的工业级方案，其Tag-Length-Value机制与手写版本号方案形成对比，有助于理解标准化二进制协议的设计权衡。
