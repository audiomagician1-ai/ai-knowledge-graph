# 二进制序列化

## 概述

二进制序列化是将游戏数据结构直接编码为紧凑字节流的过程，与JSON或XML等文本序列化不同，二进制格式不包含任何字段名称字符串或标签符号，仅存储原始数值。一个包含位置、旋转、缩放的Transform结构体，JSON表示可能需要120字节，而二进制表示只需36字节（3个float向量 × 3分量 × 4字节/float），压缩率约为3.3:1。

二进制序列化的历史可追溯到早期主机游戏时代，FC（NES）时代的存档数据就是将RAM中的固定地址字节块直接写入电池供电的SRAM芯片，这是最原始的二进制序列化形式。现代游戏引擎如Unreal Engine的`.uasset`格式和Unity的`.asset`格式均建立在二进制序列化基础上。游戏引擎选择二进制序列化的核心原因有三：读写速度比文本格式快10-100倍、存储体积显著更小、可直接用`memcpy`映射到内存结构体。

## 核心原理

### 结构体内存布局与序列化映射

C++结构体在内存中的字节排布由编译器决定，其中最关键的概念是内存对齐（alignment）。例如以下结构体：

```cpp
struct PlayerData {
    uint8_t  level;    // 1字节，偏移0
    // 3字节填充（padding）
    float    health;   // 4字节，偏移4
    uint32_t score;    // 4字节，偏移8
};                     // 总计12字节，非8字节
```

直接用`fwrite(&player, sizeof(PlayerData), 1, file)`写入文件时，填充字节也会被写入，造成空间浪费且可能导致跨平台不兼容。因此专业的二进制序列化通常不使用直接内存转储（memory dump），而是逐字段手动序列化，或使用`#pragma pack(1)`消除对齐填充后再序列化。

### 字节序（Endianness）问题

字节序指多字节整数在内存中的存储顺序。小端序（Little-Endian）将低位字节存于低地址，大端序（Big-Endian）相反。例如32位整数`0x12345678`：
- 小端序存储：`78 56 34 12`（x86/x64 PC架构）
- 大端序存储：`12 34 56 78`（PowerPC，早期PS3/Xbox 360）

早期PS3游戏（基于PowerPC大端序）移植到PC时，若直接读取二进制存档，所有多字节数值都会错误解析。网络字节序（Network Byte Order）规定使用大端序，`ntohl()`和`htonl()`函数负责在主机字节序与网络字节序之间转换。现代游戏引擎通常选定一种字节序作为文件标准，并在读取时按需转换。Unreal Engine的序列化系统中，`FArchive`类封装了`ByteSwap`操作，通过编译宏`PLATFORM_LITTLE_ENDIAN`自动处理字节序差异。

### 版本兼容性机制

二进制序列化最棘手的工程问题是版本兼容，因为没有字段名称，读取端完全依赖字段的字节偏移位置。版本兼容通常通过以下方式实现：

**文件头版本号**：每个二进制文件开头写入4字节的魔数（Magic Number）和版本号。例如Unreal的uasset文件以`0x9E2A83C1`（反转后为`C1832A9E`）作为魔数标识，后跟版本整数。读取器检查版本号后选择对应的解析逻辑。

**追加式字段扩展**：只允许在结构体末尾添加新字段，从不删除或重排已有字段。旧版读取器遇到更长的新格式数据时，只读取自己理解的字段，跳过末尾多余字节；新版读取器遇到旧格式数据时，检测字节流剩余长度不足后为新字段填入默认值。

**字段标签（Tag）方案**：每个字段序列化时前缀一个1-2字节的ID和长度，读取端通过ID而非位置识别字段，遇到未知ID时按长度跳过。这种方案接近Protobuf的核心思想，但在纯自定义二进制格式中也广泛使用。

## 实际应用

**游戏存档系统**：角色扮演游戏的存档包含大量数值型数据（属性、背包物品ID、地图坐标），二进制序列化比JSON存档体积小4-8倍，且加载时无需字符串解析，在主机平台有限的IO带宽下优势明显。《黑暗之魂》系列的`.sl2`存档文件即为二进制格式，并在此基础上叠加了AES加密。

**网络数据包**：多人游戏中玩家状态同步每帧发送，带宽极度敏感。一个玩家状态若用二进制紧凑序列化，位置用3×16bit定点数（6字节），旋转用压缩四元数（4字节），速度用3×16bit（6字节），合计约20字节；等效JSON则需约150字节，差距7.5倍。

**资产文件格式**：Unreal Engine的`FBulkData`系统将纹理Mipmap的原始像素数据直接以二进制块存储在uasset中，读取时直接`memcpy`到GPU上传缓冲区，避免任何解析开销。

## 常见误区

**误区一：`sizeof(struct)`等于序列化字节数**。由于内存对齐填充，`sizeof`的结果往往大于手动逐字段序列化的字节数。在实现跨平台存档时，若直接用`fwrite`写入结构体，不同编译器或不同`#pragma pack`设置会导致文件格式不同，造成无法互读。正确做法是为序列化专门定义无填充的紧凑布局，或手动逐字段写入。

**误区二：二进制序列化天然安全**。二进制格式只是不可读，并非加密。对于玩家存档文件，若不加密，玩家可通过十六进制编辑器定位并修改数值（例如找到连续出现的整数找到金币数）。二进制序列化与加密、校验码（CRC32、MD5）是独立的关注点，需单独叠加。

**误区三：版本兼容只需保证向后兼容**。游戏上线后经常需要热更新或服务器回滚，此时旧版程序需要读取新版数据（向前兼容），这要求序列化格式设计时就必须规划好未知字段的跳过机制，而不能假设数据总是旧格式。

## 知识关联

学习二进制序列化需要先理解序列化概述中建立的"序列化-反序列化"基本模型，以及为何游戏引擎需要将运行时对象持久化。

掌握二进制序列化后，**存档系统**是最直接的应用方向，存档的加密、压缩和差量存储都建立在二进制字节流操作之上。**UAsset格式**是Unreal Engine对二进制序列化的具体工程实现，包含资产依赖表、导入/导出表等复杂结构，是理解引擎资产管线的关键。**网络序列化**在二进制序列化基础上增加了Delta压缩和优先级调度，专门处理高频低延迟场景。**Protobuf/FlatBuffers**则是对手写二进制序列化的工程化抽象，理解了手写二进制格式的字节布局和版本兼容问题，才能真正理解Protobuf的field tag机制和FlatBuffers的零拷贝设计所解决的根本问题。
