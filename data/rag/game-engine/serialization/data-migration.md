# 数据迁移

## 概述

数据迁移（Data Migration）在游戏引擎序列化体系中，是指将旧版本格式的存档或资产文件转换为新版本格式的完整过程。当游戏迭代开发导致数据结构发生变化——例如角色属性新增字段、关卡格式重构或物理参数单位换算——玩家的历史存档和美术资产必须经过迁移才能被新版本引擎正确读取。

数据迁移的概念伴随着游戏的版本化更新需求而产生。早期游戏（如1990年代的RPG）往往采用硬编码的二进制存档，版本不兼容时直接让玩家删档重来，这在现代游戏中已不可接受。2000年代中期，Unreal Engine和Unity等商业引擎开始引入系统化的资产迁移管线，将数据升级从手工操作变为可自动化的流程。

数据迁移的重要性体现在两个维度：对玩家，保留存档进度是基本服务承诺；对开发者，资产向后兼容性决定了能否在不重建工作流的情况下升级引擎或工具链。一次处理不当的迁移可能导致数千个场景文件损坏，造成无法估量的制作损失。

## 核心原理

### 版本链式迁移（Chain Migration）

链式迁移是处理跨多个版本升级的标准策略：不直接从版本 N 跳转到版本 M，而是按顺序执行 N→N+1→…→M 的每一步迁移函数。每个迁移步骤（称为 Migration Step 或 Migrator）只负责相邻两个版本间的转换逻辑。

```
migrate_v1_to_v2(data) → migrate_v2_to_v3(data) → migrate_v3_to_v4(data)
```

这种设计使每个 Migrator 的代码量极小、测试覆盖简单。Unity 的 `AssetDatabase` 内部维护一个迁移函数注册表，在导入资产时自动检测版本号并调用相应的升级链。假设当前引擎为版本 12，读取版本 7 的资产时，系统会依次调用 `Migrator[7]` 到 `Migrator[11]` 共 5 个函数。

### 迁移函数的三种模式

**字段重命名与类型转换**：最常见的场景。将旧字段 `hitPoint`（整型）迁移为 `health`（浮点型），迁移函数执行 `new.health = (float)old.hitPoint`。必须处理类型精度损失，例如整型转浮点后的小数部分填充规则。

**字段添加与默认值填充**：新版本增加了 `armorRating` 字段，旧数据中不存在该键。迁移函数需为其填入语义正确的默认值——此处"默认值"并非简单的 `0`，而是应与旧版本的隐含行为等价。例如旧版本中所有角色等效于无护甲，则默认填入 `0.0f`；但若旧版本中主角硬编码有 10 点防御，则玩家存档迁移时该字段应填 `10.0f`，而非模板默认值。

**字段删除与数据压缩**：旧版本冗余存储了 `maxHealthCache` 作为缓存字段，新版本改为运行时计算。迁移时直接丢弃该字段，同时验证运行时计算结果与旧缓存值的误差是否在容许范围（通常 ±0.001）内，超差则记录警告供QA排查。

### 迁移的原子性与回滚

迁移操作必须满足原子性：要么全部成功，要么回滚到原始状态，不能产生半迁移状态的损坏文件。工程实现上，通常先将旧文件完整备份（或在内存中保留原始字节流），在临时副本上执行迁移，验证通过后才原子替换原文件。Unreal Engine 的 `FObjectAndNameAsStringProxyArchive` 在迁移失败时会抛出异常并恢复原始序列化数据，而不是写入半成品。

对于资产批量迁移（如升级整个项目的数千个 `.uasset` 文件），通常采用事务日志（Transaction Log）记录每个文件的迁移状态，支持中断后从断点续传，而非从头重跑。

### 懒迁移与预迁移

**懒迁移（Lazy Migration）**：文件在被访问时才执行版本升级，适合存档文件（玩家加载存档时触发）。优点是不占用启动时间；缺点是首次加载延迟增加，且线上服务中若多个客户端并发访问同一资产可能产生竞争。

**预迁移（Eager Migration）**：在发布新版本时，构建管线（Build Pipeline）批量升级所有资产。Unity 的 `AssetDatabase.ForceReserializeAssets()` API 即触发预迁移，将项目内所有资产升级到当前编辑器格式。预迁移产生大量版本控制（如 Git）的文件变更提交，需要团队协商时间窗口统一执行。

## 实际应用

**RPG游戏存档升级**：某手游在 1.5 版本将装备系统从"攻击力+1"的整型叠加改为百分比浮点系统。迁移函数将旧存档中每件装备的 `attackBonus: 15`（整型）转换为 `attackBonusPercent: 0.075f`（基于旧版线性公式 `bonus * 0.005f`），确保玩家战力数值平滑过渡，避免数值膨胀投诉。

**Unity 场景文件迁移**：Unity 从 2019 升级到 2021 时，`GameObject` 的 `m_TagString` 字段被废弃，改用 `m_TagIndex`（整型索引）。项目升级时，编辑器自动遍历所有 `.unity` 场景文件，将字符串标签通过内置标签表查找对应索引写入新字段。若标签字符串在新版标签表中不存在，则降级为 `"Untagged"` 并在控制台输出警告。

**二进制格式的偏移表修复**：某引擎的地图文件格式升级时，增加了一个 16 字节的文件头扩展区。旧版文件中所有内嵌的偏移量（offset）值均需加上 `+16`，否则所有数据块的起始位置计算错误。迁移函数对整个偏移量表执行批量修正，这一操作需要深度了解旧格式的二进制布局，是最易出错的迁移类型之一。

## 常见误区

**误区一：用默认构造值代替语义等价值**

许多开发者在迁移新增字段时，直接填写该字段的构造器默认值（如 `0`、`false`、`""`）。然而旧版本数据往往隐含了特定行为语义，直接填 `0` 可能导致旧存档中角色的某项能力"凭空消失"。正确做法是在迁移设计阶段显式文档化每个新字段对于旧数据的等价语义，而非依赖程序默认值。

**误区二：跳版本迁移（Skip-Version Migration）**

为了"优化"，有人编写直接从版本 1 跳到版本 5 的迁移函数，绕过中间三个版本。这在短期内减少了函数调用次数，但一旦未来需要在版本 3 中插入新的历史迁移逻辑，跳版本函数就变成了难以维护的特例，需要手动拆分或打补丁。版本 2 的测试用例也无法覆盖这条跳版本路径，隐患极大。

**误区三：迁移函数修改共享资产而未通知引用方**

当一个材质资产（Material）的 GUID 未变但内部参数结构已迁移，引用该材质的所有场景文件可能依赖旧的参数布局。仅迁移材质文件本身而不同步触发引用方的重新序列化，会导致场景加载时出现参数错位的渲染错误。正确的做法是迁移后触发引用资产的**依赖脏标记（Dependency Dirty Flag）**，强制引用方在下次保存时重新序列化其对该材质的引用数据。

## 知识关联

数据迁移直接依赖 **Schema 版本控制** 所建立的版本号体系。Schema 版本控制决定了版本号如何嵌入文件（文件头字节偏移位置、JSON顶层字段名），以及如何判断某文件需要迁移；数据迁移则是在版本号识别完成后，执行实际的字段级转换逻辑。若 Schema 版本控制选用了语义化版本号（Semantic Versioning），迁移策略还需区分主版本（Major）变更（破坏性字段删除，需强制迁移）与次版本（Minor）变更（新增向后兼容字段，迁移可选）的不同处理路径。

在序列化体系的整体架构中，数据迁移是保证**向后兼容性（Backward Compatibility）**的核心机制。没有迁移系统的序列化框架，每次 Schema 变更都会迫使开发团队在"冻结数据格式"与"重置所有存档"之间二选一，严重制约了游戏的迭代速度。掌握链式迁移、懒迁移与预迁移的选型逻辑，以及原子性保证的工程实现，是构建生产级
