# 数据迁移

## 概述

数据迁移（Data Migration）在游戏引擎序列化系统中，特指当存档文件或资产文件的格式版本发生变化时，将旧格式数据自动转换为新格式的完整过程。这与数据库领域的迁移概念相似，但游戏引擎的迁移需要在运行时或离线工具中处理二进制/JSON/XML等多种序列化格式，并且必须保证玩家存档在游戏更新后仍能正确读取。

历史上，Unreal Engine 在4.0版本（2014年）引入了系统性的资产版本号机制（`FPackageFileSummary` 中的 `FileVersionUE4` 字段），Unity则依赖YAML序列化格式中的 `m_ObjectHideFlags` 等隐式版本信息演化出自己的资产迁移管道。早期游戏开发中缺乏正式迁移机制，导致《上古卷轴：晨风》等游戏的存档在补丁后直接失效，引发大量玩家投诉——这促使行业逐步将迁移策略纳入引擎设计。

数据迁移之所以重要，在于游戏生命周期内格式变更不可避免：新增武器属性字段、重构背包数据结构、调整物理参数精度等操作都会破坏旧格式兼容性。没有迁移系统，每次格式变更都会造成存档丢失，严重损害玩家体验和游戏口碑。

## 核心原理

### 版本号驱动的迁移链

迁移系统的基础是在序列化数据头部嵌入版本号，例如存档文件首4字节存储 `uint32 schemaVersion`。当加载器检测到文件版本（如 `v3`）低于当前版本（如 `v5`）时，会依次执行 `v3→v4` 和 `v4→v5` 两个独立迁移函数，形成迁移链（Migration Chain）。每一步迁移只负责相邻两个版本之间的转换，这样新版本只需新增一个迁移函数，而不必修改历史代码。

迁移链的核心逻辑伪代码如下：

```
while (data.version < CURRENT_VERSION) {
    migrations[data.version](data);  // 执行对应版本的迁移函数
    data.version++;
}
```

这种设计确保了一个存了三年的存档（版本 `v1`）依然能通过逐步迁移升级到最新的 `v12` 格式。

### 迁移函数的三类操作

每个迁移函数通常执行三类具体操作：

**字段增删**：新增字段时需为旧数据补填默认值，例如角色存档新增 `stamina` 字段时，迁移函数将其设为 `100`（满值）而非 `0`（会让所有老存档角色体力耗尽）。删除字段时直接跳过即可，多余数据会在下次序列化时自然消失。

**字段重命名与类型变换**：将旧字段 `hp_int`（整型）转换为新字段 `health_float`（浮点型）时，迁移函数执行 `data.health_float = (float)data.hp_int`，并删除 `hp_int` 键。类型变换是最容易引入精度误差的操作，需特别注意数值范围。

**结构重组**：旧格式用平铺列表 `items[0..99]` 存储背包，新格式改为嵌套结构 `inventory.slots[0..99]`。迁移函数需遍历旧列表并重新封装到新结构中，同时保留物品数量、耐久度等子字段。

### 向前兼容与向后兼容的取舍

迁移系统通常只保证**向后兼容**（旧存档能在新版游戏中读取），而刻意放弃**向前兼容**（新存档无法在旧版游戏中读取）。原因在于：向前兼容要求新版格式不破坏旧读取器的解析逻辑，这意味着格式演化受到严重约束，例如永远不能移除字段、不能改变字段顺序。Unreal Engine 的 `.uasset` 格式选择只维护向后兼容，通过 `VER_UE4_*` 系列宏（累计超过500个版本常量）追踪每一个格式变更点。

## 实际应用

**RPG游戏存档迁移**：某RPG游戏在1.3版本将技能系统从整型技能ID改为字符串型技能Key（`skill_id: 42` → `skill_key: "fireball_lv2"`）。迁移函数维护一张 `int→string` 的映射表，遍历存档中所有技能槽位执行替换。若旧ID在映射表中不存在（可能是已删除技能），则将该槽位设为空，而非崩溃报错。

**Unity资产管道迁移**：Unity在从版本2019升级到2020时，`ParticleSystem` 组件的 `startLifetime` 字段从单一浮点值改为支持曲线的 `MinMaxCurve` 结构。Unity的资产迁移器（Asset Upgrader）自动将旧的 `float startLifetime = 2.5` 转换为 `MinMaxCurve { mode=Constant, constant=2.5 }`，保持粒子效果视觉不变。

**离线批量迁移工具**：对于资产数量庞大的项目（如包含10000+个 `.prefab` 文件的大型游戏），在游戏启动时逐一迁移会严重拖慢加载速度。常见方案是提供离线迁移CLI工具，在版本发布前对整个资产库执行一次性批量迁移并提交到版本控制，运行时只需验证版本号一致即可跳过迁移逻辑。

## 常见误区

**误区一：迁移函数可以直接修改生产存档**

部分开发者在调试时直接对玩家存档运行迁移函数，未做备份。迁移函数并非无损操作——类型转换可能丢失精度，字段删除是不可逆的，结构重组可能引入Bug。正确做法是迁移前自动备份原始文件（如 `save_v3.bak`），迁移后写入新文件，确认无误后再删除备份。Steam Cloud等平台的存档同步机制也可能覆盖备份，需在本地先完成迁移。

**误区二：为旧字段保留"兼容层"会减少迁移工作量**

一些团队倾向于在新格式中同时保留旧字段和新字段（如同时存储 `hp_int` 和 `health_float`），认为这样可以让新旧代码都能运行。但这会导致数据冗余、字段同步问题（修改一个忘记修改另一个），并且随着版本增多，格式会积累大量"僵尸字段"。正确做法是迁移时彻底替换，用严格的版本号管理代替字段共存。

**误区三：忽略迁移函数的幂等性**

若加载代码存在Bug导致迁移函数被调用两次，应确保结果不变（幂等性）。例如将 `damage *= 1.5` 的迁移执行两次会让伤害变为原来的2.25倍。正确写法是先检查目标字段是否已存在或版本号是否已更新，确认未迁移后再执行转换，避免双重迁移破坏数据。

## 知识关联

数据迁移直接依赖 **Schema版本控制** 提供的版本号基础设施——没有准确的版本号，迁移系统无法判断需要执行哪些迁移步骤，也无法确认数据是否已经是最新格式。Schema版本控制解决了"数据是哪个版本"的问题，而数据迁移解决了"如何从旧版本变为新版本"的问题，两者分工明确：版本控制负责标记，迁移负责转换。

在序列化系统的整体架构中，数据迁移处于反序列化流程的最前端——读取文件头版本号后、解析具体字段之前。这个位置设计是刻意的：迁移完成后，后续所有反序列化代码只需针对当前版本格式编写，无需在每个字段的读取逻辑中混入版本判断，大幅降低了序列化代码的复杂度。掌握数据迁移后，开发者能够自信地对存档格式进行重构，而不必因担心破坏老存档而保留历史技术债务。
