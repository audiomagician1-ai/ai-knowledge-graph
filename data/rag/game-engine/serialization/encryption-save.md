# 存档加密

## 概述

存档加密（Save File Encryption）是游戏引擎序列化系统中用于保护玩家存档数据不被非法读取、篡改或伪造的技术手段。其核心目标是通过密码学方法确保存档文件的机密性（Confidentiality）和完整性（Integrity），使作弊者无法直接修改金币数量、角色等级或任务进度等关键数据。

存档加密并非现代游戏独有的概念。早在1990年代，卡带游戏便通过校验和（Checksum）方式防止存档被修改——例如《精灵宝可梦红/绿》在1996年使用了简单的字节求和校验。随着PC游戏和在线排行榜的普及，攻击者使用Cheat Engine等内存修改工具或十六进制编辑器直接篡改存档文件的行为日益猖獗，游戏开发者不得不引入对称加密、哈希验证等更强手段来应对。

对于单机游戏，存档加密主要对抗本地作弊行为；对于带有全球排行榜或成就系统的游戏，被篡改的存档若能同步至云端，会破坏整个玩家社区的公平性。因此，存档加密是单机体验完整性和多人公平竞技之间的重要防线。

## 核心原理

### 对称加密与密钥管理

最常见的存档加密方案使用 AES（Advanced Encryption Standard）对称加密算法，通常选用 AES-128 或 AES-256 模式。加密流程为：将序列化后的存档字节流作为明文，以一个固定或动态生成的 128/256 位密钥进行 CBC 或 GCM 模式加密，输出密文写入磁盘。

密钥管理是最大的挑战。若将密钥硬编码于可执行文件中，攻击者可通过逆向工程提取。常见的缓解方案包括：
- **设备绑定密钥**：将设备唯一标识（如 Windows 的 MachineGuid 或 Android 的 `Settings.Secure.ANDROID_ID`）与游戏安装时生成的随机盐值（Salt）混合，通过 PBKDF2 派生出加密密钥，使得存档只能在原设备上解密。
- **远程密钥下发**：联网游戏可在登录时从服务器获取每用户唯一密钥，存档离线后无法被解密。

### 哈希完整性验证

加密保证了数据的机密性，但完整性需要独立校验。常用方案是在存档末尾附加 HMAC-SHA256 签名：

```
HMAC = HMAC_SHA256(key, plaintext_save_data)
```

其中 `key` 为服务器持有或设备绑定的密钥，`plaintext_save_data` 为加密前的原始存档字节流。游戏加载存档时，先解密数据，再重新计算 HMAC 并与存储值比对——若不一致，则判定存档被篡改，拒绝加载或重置为默认存档。

相比早期《精灵宝可梦》时代使用的简单加法校验和，HMAC-SHA256 在密钥未泄露的前提下具有密码学意义上的不可伪造性，攻击者即使修改了数据内容，也无法在不知道密钥的情况下生成合法的签名。

### 数据混淆与反调试补充措施

纯密码学方案对抗动态修改存在盲点——攻击者可以在游戏运行时直接修改内存，无需破解存档文件。因此存档加密常与以下手段配合：

- **运行时数据混淆**：关键变量（如金币数量）在内存中以异或掩码形式存储，真实值 = 存储值 XOR 随机掩码，写入存档前还原为真实值再加密。Unity 的 PlayFab SDK 和第三方插件（如 SavegameManager Pro）均提供此类运行时混淆。
- **版本哈希绑定**：将游戏版本号（如 `v1.3.2` 的哈希）纳入 HMAC 计算，防止玩家将旧版本生成的存档迁移至新版本以绕过数值平衡调整。

## 实际应用

**Unity 中的 AES 存档加密实现**：开发者通常使用 `System.Security.Cryptography.Aes` 类，将 `JsonUtility.ToJson()` 序列化后的 UTF-8 字节流进行 AES-128-CBC 加密，IV（初始化向量）随机生成并拼接在密文头部（前 16 字节），整体文件再经 Base64 编码后写入 `Application.persistentDataPath`。

**Steam 云存档防篡改**：Valve 的 Steam 云同步本身不提供加密，存档文件以明文存于本地 `userdata` 目录。《黑魂3》等游戏通过在存档中内嵌角色数据的 SHA-1 校验块，服务器在 PvP 匹配时验证存档合法性，检测到异常哈希的账号会被 SoftBan（仅与其他作弊者匹配）。

**移动端存档保护**：Android 平台上，存档若存储于 `getExternalFilesDir()` 路径则对有 Root 权限的设备完全可读，因此安全敏感的存档（如付费道具进度）应存储于内部私有目录并附加 AES + HMAC 保护。Google Play Games 的云存档 API 在传输层使用 TLS，但落盘前的本地加密仍需开发者自行实现。

## 常见误区

**误区1：Base64 编码等同于加密**
许多初学者将序列化后的存档进行 Base64 编码，误认为这样玩家无法直接阅读数据。实际上，Base64 是可逆编码而非加密，任何在线工具可一键还原为明文。真正的加密要求在不知道密钥的情况下，破解所需计算量在数学上不可行——例如暴力破解 AES-256 密钥需要约 2²⁵⁶ 次运算。

**误区2：仅对存档文件加密即可高枕无忧**
存档加密只保护磁盘上的静态文件。若游戏不防内存读写，攻击者可在游戏运行过程中直接修改内存中的变量值，存档写出时虽然加密了"已被修改的数据"，加密形式上完全合法。完整的反作弊体系还需要运行时内存保护（如前述 XOR 混淆）或外挂检测服务（如 Easy Anti-Cheat）的配合。

**误区3：强加密必然带来不可接受的性能开销**
一些开发者为了避免性能问题放弃加密。实测数据表明，对一个典型的 10KB 游戏存档进行 AES-128-CBC 加密在现代 CPU 上耗时不足 1 毫秒，对玩家几乎无感知。性能敏感的场景（如自动存档每隔 30 秒触发一次）也完全可以接受这一开销，不应以此为由省略加密步骤。

## 知识关联

存档加密建立在**存档系统**的基础上：首先需要存档系统完成游戏状态的序列化（将对象转换为字节流），随后加密层才对该字节流进行处理。因此，存档文件的格式选择（二进制 vs JSON vs Protocol Buffers）直接影响加密性能——二进制格式更紧凑，加密耗时更短，JSON 格式因含有大量冗余字符，建议在加密前先执行 zlib 或 LZ4 压缩以减少密文大小。

在系统架构上，存档加密与**序列化版本控制**紧密相关：当游戏更新导致存档格式变化时，需要同时处理旧密钥对旧版存档的解密迁移问题，否则玩家升级游戏后可能因无法解密旧存档而丢失进度。常见方案是在存档头部存储版本号和对应的密钥派生参数，迁移流程为"旧密钥解密 → 数据升级 → 新密钥重加密"。
