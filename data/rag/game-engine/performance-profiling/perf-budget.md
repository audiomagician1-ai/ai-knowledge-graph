# 性能预算制定

## 概述

性能预算制定（Performance Budgeting）是游戏开发中将目标帧率转化为各子系统毫秒级时间配额的规划方法。其核心思路是：既然每帧可用的总时间是固定的，那么就必须像财务预算一样，将这块"时间蛋糕"按照优先级切分给渲染、物理、AI、音频、脚本等各个子系统，确保它们的总耗时不超过帧时间上限。

该方法的实践化发展主要源于主机游戏开发的严格性能要求。PlayStation 2时代开始，AAA工作室就已建立帧时间分配表格，用于跟踪各模块的CPU/GPU开销。如今无论是目标60fps的动作游戏、30fps的开放世界，还是120fps的VR应用，性能预算制定都是制作初期必须完成的技术规划文档。

为什么它重要？因为没有预算的优化是被动救火，有预算的开发才能主动防控。当一名程序员知道自己负责的寻路模块每帧最多只能用0.8ms，他会在设计阶段就选择合适的算法，而不是等到上线前崩溃才压缩。

## 核心原理

### 帧时间上限的计算

帧时间（Frame Budget）由目标帧率直接决定，公式为：

**T_frame = 1000ms ÷ 目标帧率（fps）**

常见换算结果：
- 30fps → **33.33ms** 每帧
- 60fps → **16.67ms** 每帧
- 120fps → **8.33ms** 每帧

但这并非全部可用时间。操作系统调度、垂直同步等开销会消耗约1-2ms，因此60fps游戏的**实际可分配预算通常按15ms计算**，为交换缓冲和驱动调用保留缓冲区。

### CPU与GPU的双路预算

现代游戏存在CPU时间线和GPU时间线并行执行的特性，两者都需要独立制定预算，最终帧时间取二者中**较慢的那条**作为瓶颈。

**CPU预算分配示例（60fps，目标15ms）：**

| 子系统 | 典型配额 |
|---|---|
| 渲染命令生成（DrawCall排序、剔除） | 3.0ms |
| 物理模拟（碰撞检测、刚体积分） | 2.0ms |
| AI与寻路 | 1.5ms |
| 动画更新（骨骼混合、IK） | 1.5ms |
| 游戏逻辑与脚本 | 2.0ms |
| 音频混音 | 0.5ms |
| UI更新 | 0.5ms |
| 预留缓冲 | 4.0ms |

**GPU预算**则通常按以下比例切分：不透明几何体绘制约占40%、光照与阴影约占30%、后处理特效约占20%、透明物体与UI约占10%。

### 预算分级与优先级规则

性能预算不是平均分配，而是根据游戏类型确立优先级。一款格斗游戏的物理预算可以是策略游戏的3倍，但其AI预算可以压缩到0.2ms。制定预算时需明确两类限制：

- **硬上限（Hard Budget）**：绝对不可超越，一旦超出就产生掉帧。例如整体帧时间15ms是硬上限。
- **软上限（Soft Budget）**：作为警告阈值，通常设为硬上限的80%。当某子系统在连续10帧内超出软上限时，CI系统自动标记为性能回归（Performance Regression）。

## 实际应用

**案例一：Unity移动端60fps游戏**
在Unity Profiler中，开发团队将目标帧时间设为16.6ms。通过前期Profiling发现，移动端GPU填充率是瓶颈，于是将屏幕分辨率渲染降为0.85倍（即动态分辨率），为后处理节省出约2.3ms，确保在中端Android设备上维持流畅。

**案例二：Unreal Engine 5的Lumen全局光照预算**
在Nanite与Lumen同时开启时，Lumen的屏幕空间光照每帧在PS5上约消耗3-4ms GPU时间。Epic官方建议将Lumen的"Final Gather Quality"调低至0.6，配合时域降噪，可将GPU预算压缩至2.5ms以内，为阴影和反射保留更多配额。

**案例三：VR游戏的严格120fps预算**
Meta Quest 2运行120fps时每帧仅有8.33ms。开发者通常将Draw Call严格限制在每帧100个以内，每个Draw Call约占0.05-0.08ms CPU时间，从而让DrawCall生成控制在5ms以内，为应用逻辑留出约3ms空间。

## 常见误区

**误区一：将帧率目标等同于帧时间上限**
很多初学者认为60fps=16.67ms完全可用，便将全部子系统时间填满至16.67ms。这忽略了操作系统中断、GC（垃圾回收）刺激和驱动提交等不可控消耗。正确做法是预留10%-15%的缓冲，即60fps实际按14-15ms规划。

**误区二：CPU预算达标就等于帧率达标**
由于GPU与CPU并行，有时CPU只用了8ms，但GPU渲染阴影贴图耗费了22ms，导致帧时间超标。性能预算必须同时覆盖CPU和GPU两条时间线，单看一侧数据会产生误判。

**误区三：预算一旦制定就不需要更新**
随着关卡复杂度变化，原先给物理分配的2ms到了开放世界大场景时可能完全不够。性能预算应该是动态文档，在每个里程碑节点（Alpha、Beta、Gold）根据实测数据重新调整各子系统配额，而非一成不变的初始设定。

## 知识关联

**前置知识衔接**：性能剖析概述建立了"通过采样和测量识别耗时"的能力，而性能预算制定则将这种测量结果转化为**规范性约束**——剖析告诉你"现在花了多少时间"，预算告诉你"允许花多少时间"。两者共同构成性能管理的闭环：先用剖析工具（如RenderDoc、Nsight、Unity Profiler）采集基线数据，再以此为依据制定合理的毫秒分配表。

**向后扩展方向**：掌握预算制定后，自然延伸至各子系统的具体优化技术，例如DrawCall合批（Batching）、LOD分级、物理层次时间步（Substep）控制等。这些优化手段的优先级与力度，恰好由预算超支的幅度来驱动——超支0.5ms用轻量优化，超支3ms则需要架构级重构。
