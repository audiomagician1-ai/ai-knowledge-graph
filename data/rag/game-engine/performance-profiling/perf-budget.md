# 性能预算制定

## 概述

性能预算制定（Performance Budgeting）是游戏开发中将帧时间（frame time）按毫秒单位分配给各个子系统的规划过程。开发团队首先确定目标帧率——例如60fps或30fps——然后将对应的帧时间总量切分为若干份额，分别分配给渲染、物理、动画、AI、音频和脚本逻辑等模块。这种分配不是估算，而是带有约束力的指标：一旦某子系统超出配额，必须优化，而不是挤占其他模块的时间。

性能预算的概念在主机游戏开发中得到系统化实践，最早可追溯到PS2/Xbox时代，当时硬件资源极为固定，开发团队被迫将CPU和GPU周期精确分配到每条管线。随着移动端和VR的普及，预算制定变得更加严格：VR设备要求90fps（帧时间约11.1ms），超过20ms的延迟会引发眩晕感，容错空间极小。

这一做法之所以重要，在于它将"我们的游戏跑得慢"这种模糊问题转化为"AI模块在本帧超支了2.3ms"这种可执行的工程任务，使性能优化工作有明确的目标和责任边界。

## 核心原理

### 帧时间计算与目标设定

一切预算的基础是帧时间公式：

$$T_{frame} = \frac{1000}{FPS} \text{ (ms)}$$

- 目标 **60fps** → 每帧总预算 **16.67ms**
- 目标 **30fps** → 每帧总预算 **33.33ms**
- 目标 **90fps（VR）** → 每帧总预算 **11.11ms**

但这个总量并非全部可用。操作系统调度抖动、驱动层开销、垂直同步等因素通常消耗0.5~1ms，因此实际可分配的"净预算"通常比理论值少约1ms。以60fps为例，工程上常用16ms而非16.67ms作为设计上限。

### 子系统分配比例

在典型3D动作游戏的60fps目标下，一套常见的16ms分配方案如下：

| 子系统 | 预算（ms） | 说明 |
|---|---|---|
| 渲染（CPU端提交） | 4.0 | DrawCall提交、状态切换 |
| 渲染（GPU） | 6.0 | 着色、光照、后处理 |
| 物理模拟 | 2.0 | 碰撞检测与响应 |
| 动画系统 | 1.5 | 骨骼混合、IK求解 |
| AI与寻路 | 1.0 | 行为树、A*查询 |
| 音频 | 0.5 | 混音、3D空间化 |
| 脚本/游戏逻辑 | 1.0 | Lua/C#业务逻辑 |

**总计约16ms**。这一分配并非通用标准，需要根据游戏类型调整：策略游戏的AI预算可能达到4ms，而格斗游戏可能压缩AI至0.2ms以将更多时间留给动画。

### CPU预算与GPU预算的独立性

一个关键原则是：CPU帧预算和GPU帧预算必须**分开核算**，因为两者可以并行工作。如果CPU在3ms内完成提交，GPU需要10ms完成渲染，实际帧时间由GPU决定（取两者中较长者）。这意味着：

- **CPU-bound场景**：减少DrawCall数量、使用instancing合批、简化AI逻辑
- **GPU-bound场景**：降低分辨率、减少overdraw、使用LOD削减顶点数

错误地把两者放在同一条预算线上会导致优化方向完全错误。

### 预算的动态分级

游戏场景并非均匀：空旷场景和高密度战斗场景的负载差距可达3倍以上。成熟的预算方案会设置**三档预算**：

- **稳态预算**：日常场景的常规分配
- **峰值预算**：允许短暂（≤3帧）超支，用于爆炸、技能特效等
- **警戒线**：任何子系统不得突破的硬性上限，通常设为稳态预算的150%

## 实际应用

**案例1：手机游戏30fps目标**  
某手游目标机型为中端安卓（骁龙730），目标30fps，33ms帧时间。由于GPU性能有限，GPU预算仅分配15ms（分辨率降至720p），物理降为固定步长每帧只计算1次碰撞检测（约1ms），AI采用每2帧更新一次的分帧策略，每帧实际AI开销从2ms降至1ms。

**案例2：PC VR项目90fps**  
某VR射击游戏，总预算11ms，GPU预算7ms（借助Foveated Rendering节省约2ms）。物理引擎使用固定步长0.0111s与帧同步，不允许使用浮动步长。所有后处理效果须通过"VR后处理白名单"审核，Bloom和景深被禁用，仅保留色调映射（约0.3ms）。

**案例3：预算违规工作流**  
在Unity Profiler或Unreal Insights中设置自定义性能标记（Custom Profiler Marker），当物理模块超过2ms时触发警告日志。CI/CD流水线集成自动化性能测试，若连续3次构建物理耗时均超过2.5ms，则阻断合并。

## 常见误区

**误区1：把GPU时间算入CPU预算**  
初学者常看到Profiler显示"帧时间16ms"就认为CPU忙碌了16ms。实际上，如果CPU提交DrawCall仅用了4ms，后续12ms是GPU在工作，CPU处于等待状态。正确做法是分别查看CPU timeline和GPU timeline，针对真正的瓶颈端优化。

**误区2：预算只需制定一次**  
性能预算会随版本迭代失效。美术资源增加、新功能上线、目标机型扩展——这些都会重塑各子系统的实际耗时分布。正确做法是在里程碑节点（Alpha、Beta、Gold）各重新评估一次，并结合真实设备的Profiler数据而非编辑器数据来修订预算。

**误区3：所有平台共用一套预算**  
PC、主机、手机的硬件差异导致同一子系统的绝对耗时可能相差5~10倍。为手机制定的1ms物理预算在PC上可能只需0.2ms，但GPU渲染预算在手机上可能严格十倍。不同平台必须持有独立的预算文档。

## 知识关联

学习性能预算制定需要先掌握**性能剖析概述**中的基本工具使用——如果看不懂Profiler的火焰图和时间轴，就无法获取各子系统的实际耗时数据，预算制定便成为无源之水。具体而言，需要能够区分CPU主线程、渲染线程、Worker线程各自的时间条，才能正确归类各项开销。

完成预算制定后，自然延伸到各子系统的具体优化技术：渲染预算超支时需要研究DrawCall批处理和LOD技术；物理预算紧张时涉及固定步长、空间分区（BVH/四叉树）的选择；AI预算不足时引出分帧更新和行为树剪枝策略。性能预算制定是这些优化工作的前置量化依据——没有明确的毫秒目标，优化工作缺乏收敛条件。
