# 内存性能分析

## 概述

内存性能分析是游戏引擎开发中专门针对内存使用行为的量化诊断方法，聚焦于三类核心问题：内存分配（Allocation）、内存泄漏（Leak）和内存碎片化（Fragmentation）。与CPU性能分析关注执行时间不同，内存性能分析追踪的是字节级别的资源占用、生命周期管理错误以及堆空间的碎裂程度。

内存分析工具的历史可追溯至1992年 Purify 工具的诞生，它是第一个在二进制层面插桩检测内存错误的商业工具。现代游戏引擎如 Unreal Engine 5 内置了 `LLM (Low-Level Memory Tracker)` 系统，可将内存划分为超过100个命名标签类别，精确到每个子系统的用量；Unity 则提供了 Memory Profiler 包，自版本1.0.0起支持快照对比（Snapshot Diff）功能，直接显示两帧之间新增和释放的对象。

内存问题之所以在游戏开发中格外关键，是因为主机平台（如 PS5 的 16GB GDDR6 统一内存）不支持虚拟内存换页，一旦超出物理限制即直接崩溃，而移动设备（如 iOS 对单应用的内存警告阈值通常在 1.2GB～1.8GB 之间）更会触发系统强制终止进程。

## 核心原理

### 内存分配分析

每次调用 `malloc`/`new` 或引擎封装的分配函数时，分配器会从堆中取出一块连续地址空间。分析工具通过**替换分配器**或**插桩钩子**的方式记录每次分配的：调用栈（Call Stack）、分配字节数、分配时间戳和线程ID。

评估分配压力的关键指标是**分配速率（Allocation Rate）**，单位为 MB/s 或次/秒。以 Unreal Engine 为例，若每帧的瞬时分配超过 1MB（60fps 下即 60MB/s），通常被视为需要优化的红线。高频小块分配（每次 < 64 字节但每帧数千次）比低频大块分配对性能的伤害更大，因为每次分配都涉及锁竞争和缓存污染。

### 内存泄漏检测

内存泄漏指已分配但永远不会被释放的内存块。检测原理是在程序退出时（或某个生命周期结束点）检查所有分配记录中尚未对应 `free`/`delete` 的条目。工具如 Valgrind 的 `--leak-check=full` 选项会在进程退出时输出每处泄漏的完整调用栈和字节数。

在游戏引擎中，泄漏往往不是单次大块泄漏，而是**累积型泄漏**：每次关卡切换泄漏 500KB，经过20次加载后累积 10MB。检测这类问题的标准手法是**内存快照对比**：在关卡加载前后各取一次快照，过滤掉应当保留的长期对象，剩余未释放的即为候选泄漏点。Unity Memory Profiler 的 Diff 视图直接以红色高亮显示第二张快照中新增且未配对释放的 Native 对象。

### 内存碎片化分析

碎片化分为**外部碎片**（堆中存在足够总量的空闲内存，但没有一块连续空间满足分配需求）和**内部碎片**（分配块大于实际需求，多余字节被浪费）。衡量碎片化程度的指标是**碎片率**，公式为：

$$
\text{碎片率} = 1 - \frac{\text{最大连续空闲块大小}}{\text{总空闲内存大小}}
$$

碎片率接近1.0表示内存高度碎片化。在长期运行的游戏（尤其是开放世界游戏）中，频繁的资产流式加载/卸载会将堆分裂成数百个细小空洞。解决方案包括使用**内存池（Memory Pool）**预分配固定大小的块、或使用**线性分配器（Linear Allocator）**在帧内按帧重置，Unreal 的 `FMemStack` 正是基于此原理。

## 实际应用

**场景一：开放世界资产流式加载的碎片化诊断**  
在《荒野大镖客2》这类开放世界游戏的引擎实现中，地形块和植被资产每秒可能触发数十次分配/释放。开发者使用内存分析工具抓取运行30分钟后的堆布局，若发现碎片率超过0.4，则需审查是否可以将同类资产统一使用固定大小的池分配器管理。

**场景二：移动游戏的泄漏追踪**  
一款手机 RPG 在切换10次战斗场景后内存占用从 800MB 增长到 1.1GB，触发 iOS 内存警告。开发者用 Unity Memory Profiler 对比第1次和第10次场景切换后的快照，发现每次切换都残留约 30MB 的 `Texture2D` Native 对象未被释放，定位到场景卸载时漏掉了对动态加载的 UI 图集调用 `Addressables.Release()`。

**场景三：分配热点优化**  
通过 Unreal Insights 的内存通道，发现粒子系统每帧分配 2.3MB 的临时缓冲区，将其改为使用预分配的 `TArray` 并调用 `Reserve()`，帧内分配降至 0MB，同时消除了对应的堆锁争用。

## 常见误区

**误区一：内存占用稳定就代表没有泄漏**  
实际上，若泄漏速率与 GC 回收或资产卸载的速率相当，内存总量会呈现表面平稳。正确做法是在完全停止所有动态活动（暂停游戏逻辑）后强制触发GC，再对比前后快照。Unity 的 `GC.Collect()` 配合内存快照可以暴露这类被GC节奏掩盖的泄漏。

**误区二：只需关注峰值内存，不需关注分配频率**  
峰值内存决定是否崩溃，但分配频率决定帧率稳定性。每次 `malloc` 在多线程环境下需要获取分配器锁，1000次/帧的细碎分配即便总量只有 50KB，也可能导致 0.5ms 以上的锁等待，在 16.67ms 的帧预算中造成明显卡顿。

**误区三：内存池一定能解决碎片化**  
固定大小内存池消除的是该尺寸的外部碎片，但若分配请求的大小分布极度不均匀（如同时存在 16B、1KB、10MB 三种请求），为每种大小维护独立的池反而增加内存预留浪费，形成严重的内部碎片。实践中通常按大小范围分档（如 TLSF 分配器的两级分层方案），而非对所有尺寸使用同一种池策略。

## 知识关联

在学习内存性能分析之前，需要掌握**性能剖析概述**中的采样与插桩两种基本剖析方法——内存分析工具正是以插桩方式在分配器调用点注入记录逻辑，理解插桩原理才能正确解读工具的开销与精度权衡（插桩版分配器本身会带来约10%~30%的额外内存和时间开销）。

内存性能分析与 **CPU 缓存性能**存在直接联系：内存布局决定缓存命中率，高碎片化的堆导致对象在内存中散乱分布，增加 L1/L2 Cache Miss。内存分配策略（如将同类组件数据紧密排布的 ECS 架构）既是内存分析的优化结论，也是 CPU 数据局部性优化的前提条件。此外，**渲染资源管理**（纹理、顶点缓冲区的 GPU 内存）是内存泄漏的高发区，GPU 内存分析是本主题向图形领域的自然延伸。
