# 天气系统

## 概述

天气系统是游戏引擎场景管理模块中负责模拟大气环境效果的子系统，具体涵盖降雨、降雪、雾气、动态天空盒等视觉与物理效果的实时生成与控制。与静态场景装饰不同，天气系统中的每一个元素都需要在运行时动态计算，并响应时间、位置、玩家行为等变量的实时变化。

天气系统最早在1990年代的3D赛车游戏（如1994年的《Ridge Racer》）中以简单形式出现，主要表现为单一的雨滴粒子覆盖屏幕的2D效果。随着GPU性能提升，2000年代《孤岛危机》（Crytek, 2007）率先引入动态光照与天气过渡机制，将天气从纯视觉装饰提升为影响玩家战斗策略的核心环境要素——暴雨会降低AI的视野侦测距离达40%。现代引擎如Unreal Engine 5的"Lumen"全局光照系统已能在每帧实时响应云层厚度变化对漫反射间接光照的影响，无需预烘焙光照贴图。

天气系统直接决定场景的视觉氛围和玩家的游戏体验。《荒野大镖客：救赎2》（Rockstar, 2018）中记录了超过9种不同的天气过渡状态，每种状态对NPC行为模式、能见度半径和环境音效独立产生影响；暴风雪状态下玩家马匹的移动速度被强制削减15%，形成了天气与涌现玩法的直接联结。

## 核心原理

### 粒子系统驱动的降水效果

雨和雪本质上是粒子系统的特化形式。雨粒子通常由大量细长的半透明线段（Billboard Quad）构成，每个粒子被赋予向下的初速度，并叠加风力扰动：

```glsl
// 雨粒子速度计算（顶点着色器伪代码）
vec3 velocity = vec3(wind_x, -9.8 * time_delta, wind_z);
position += velocity * particle_lifetime;
// wind_x, wind_z 范围通常为 -2.0 ~ 2.0 m/s
```

降雪粒子则使用更大的雪花纹理贴图（分辨率通常为64×64像素的alpha透明图），并对每个粒子施加随机旋转角速度（0.5～3.0 rad/s），同时将重力系数压低至0.1～0.3之间，以模拟雪花因空气阻力产生的飘落感。

性能是粒子降水的关键挑战。为避免在玩家视野中同时维护数万个世界空间粒子，现代引擎采用"相机跟随粒子体积"技术：粒子不在世界空间固定生成，而是在一个跟随摄像机移动的包围盒（典型尺寸为30m×30m×20m）内循环复用。当粒子离开该包围盒下边界时，立即被传送至上边界重生，从而以约2000个粒子实例模拟无限延伸的降雨视觉效果。Unity粒子系统通过设置 `Simulation Space: World` 并结合 `Inherit Velocity` 模块实现这一效果。

### 体积雾与深度雾的实现差异

雾效有两种主要实现方式，性能开销相差数十倍。**深度雾（Depth Fog）**通过指数平方公式对每个屏幕像素计算雾化强度：

$$fog\_factor = e^{-(density \times distance)^2}$$

其中 `density` 通常取值0.01～0.05，`distance` 为像素对应的世界空间深度值。该方式GPU开销极低（每像素仅需1次指数运算），但无法表现雾气的三维体积感，适用于移动端或低端PC平台。

**体积雾（Volumetric Fog）**将摄像机视锥体划分为三维纹理体素网格——Unreal Engine 5默认分辨率为144×80×64体素——对每个体素独立执行光线步进（Ray Marching）积分，计算来自定向光、点光源、聚光灯的散射贡献量。这一过程能呈现丁达尔效应（Tyndall Effect）：光束穿过局部浓雾时产生可见的光柱。《赛博朋克2077》（CD Projekt Red, 2020）的霓虹夜雾正是基于体积雾技术，使街道霓虹灯的光锥在雾中产生半径约3米的真实散射光晕，该场景体积雾渲染单帧GPU时间约为2.4ms（RTX 2080Ti基准）。

### 动态天空盒与昼夜循环

传统静态天空盒使用一张Cubemap贴图，无法响应时间或天气变化。动态天空盒的核心是**程序化大气散射着色器**，学界标准实现参考 Preetham et al. 的论文《A Practical Analytic Model for Daylight》（Preetham, 1999, SIGGRAPH）和后续改进的 Hosek-Wilkie 模型（Hosek & Wilkie, 2012）。这两个模型均以太阳天顶角 $\theta_s$ 作为主要输入，实时输出天空各方向的光谱辐射值。

$$L(\theta, \gamma) = L_0 \cdot \frac{(1 + A \cdot e^{B/\cos\theta})(1 + C \cdot e^{D\gamma} + E \cdot \cos^2\gamma)}{(1 + A \cdot e^{B})(1 + C \cdot e^{D\theta_s} + E \cdot \cos^2\theta_s)}$$

其中 $\theta$ 为观察方向天顶角，$\gamma$ 为观察方向与太阳方向的夹角，$A$～$E$ 为拟合大气条件的系数。当太阳仰角从90°（正午头顶）降低至5°（接近地平线）时，光线穿越大气层的路径长度增加约38倍，短波蓝光被优先散射殆尽，天空呈现橙红色，即物理意义上的瑞利散射（Rayleigh Scattering）效应。

昼夜循环通常由一个范围0.0～24.0的浮点时间变量 `world_time` 驱动，同步控制以下四个维度：①定向光（DirectionalLight）的旋转四元数插值；②天空盒着色器的 $\theta_s$ 输入参数更新；③环境光（Ambient Light）颜色与强度的Bezier曲线采样；④雾气颜色从日间白雾向夜间蓝灰雾的关键帧插值。

### 天气状态机与过渡混合

实际项目中天气系统以**有限状态机（FSM）**管理多种离散天气状态，并通过参数化混合实现平滑过渡。以下为一个精简的C++状态定义示意：

```cpp
enum class WeatherState {
    CLEAR,       // 晴天：能见度 > 500m，云覆盖率 0~15%
    OVERCAST,    // 阴天：能见度 200~500m，云覆盖率 70~100%
    RAIN_LIGHT,  // 小雨：粒子密度 800/帧，风速 1.5 m/s
    RAIN_HEAVY,  // 暴雨：粒子密度 4000/帧，风速 8.0 m/s，闪电触发概率 0.02/s
    SNOW,        // 降雪：粒子密度 2000/帧，重力系数 0.15
    FOG_DENSE    // 浓雾：能见度 < 30m，体积雾密度 0.08
};

struct WeatherBlendParams {
    float rain_intensity;    // 0.0 ~ 1.0
    float fog_density;       // 0.0 ~ 0.1
    float cloud_coverage;    // 0.0 ~ 1.0
    float wind_speed;        // m/s
};
```

过渡期间，系统对 `WeatherBlendParams` 的各字段执行独立的线性插值（Lerp），典型的晴转暴雨过渡时长设置为90～180秒，以避免突变造成的沉浸感割裂。

## 关键公式与技术参数

### 指数雾衰减公式

深度雾的线性变体与指数变体在视觉表现上有显著差异，实践中常用指数平方形式：

$$F_{linear} = \frac{d_{end} - d}{d_{end} - d_{start}}, \quad F_{exp} = e^{-density \cdot d}, \quad F_{exp2} = e^{-(density \cdot d)^2}$$

其中 $F_{exp2}$ 在近处几乎无雾、远处急剧加浓的曲线特性最符合真实大气视觉，`density` 推荐取值：晴天室外场景0.005，阴天0.015，浓雾场景0.05～0.08。

### 雨滴碰撞溅射

当雨粒子检测到与地面碰撞时，系统在碰撞点生成溅射子粒子（通常4～8个），赋予半径向外、与地面夹角15°～45°的初速度，生命周期0.2～0.4秒，半径扩展至0.1～0.3米后消散。高品质实现还会在地面着色器中叠加波纹法线贴图动画，通过对两张以不同速度（0.8x 和 1.3x）滚动的涟漪法线贴图进行混合，模拟水坑涟漪效果，参考《GPU Gems》第2章（Fernando, 2004, NVIDIA）的水面法线动画技术。

## 实际应用

### 开放世界的天气驱动玩法

《荒野大镖客：救赎2》将天气数据直接输入NPC行为树：暴雨触发NPC执行"寻找遮蔽物"行为，晴天提高野外动物出现频率15%～25%。玩家可以利用大雨状态的低能见度（能见度下降至80米）规避敌人巡逻，这是天气系统与涌现玩法模块直接联动的典型案例。

### 移动端天气系统的性能裁切策略

在移动端（如iOS Metal、Android Vulkan），体积雾通常被完全禁用，改为屏幕空间雾（Screen Space Fog）；降雨粒子数量上限从PC端的4000个削减至800个；动态天空盒着色器改用预烘焙的16帧Cubemap序列插值，将每帧天空渲染开销从约1.2ms压缩至0.15ms。Unity的URP管线通过Quality Settings中的`Render Scale`和自定义`RenderFeature`实现这套分级裁切。

### 天气音效的同步绑定

视觉天气必须与音效系统同步，否则产生感知冲突。Wwise和FMOD的常见做法是将 `rain_intensity`（0.0～1.0）作为RTPC参数实时传递给音效引擎，驱动雨声音量曲线、低频滤波截止频率（暴雨时低频增强）以及雷声的随机触发概率。雷声触发延迟依据闪电视觉位置与玩家距离按声速340m/s计算，每300米延迟约0.88秒，保持视听物理一致性。

## 常见误区

**误区一：粒子雨不随摄像机旋转而更新方向。** 若粒子在世界空间静止生成，玩家快速转身时雨滴方向会瞬间穿帮。正确做法是在粒子速度中叠加摄像机的移动速度向量，或使用相机空间粒子系统。

**误区二：昼夜循环只旋转太阳光方向，忽视Ambient Light同步。** 若环境光不跟随太阳位置更新，会出现"夜晚场景仍被日间强度环境光照亮"的穿帮现象。环境光颜色应从日间的 `(1.0, 0.95, 0.8)` 插值至夜间的 `(0.05, 0.08, 0.15)`，同时强度从1.0降至0.05。

**误区三：体积雾开销被低估。** 体积雾的性能开销与体素网格分辨率呈立方关系——将Unreal Engine的默认144×80×64分辨率提升至288×160×128，GPU开销增加约8倍，而非2倍。在性能预算不足时，应优先降低Z轴（深度方向）分辨率，因为玩家对水平方向雾气细节更敏感。
