# 场景图

## 概述

场景图（Scene Graph）是游戏引擎中用于组织和管理游戏对象的树形数据结构，其核心特征是通过父子节点关系实现变换的层级继承。一个游戏对象（节点）的世界空间位置、旋转和缩放，由其自身的局部变换与所有祖先节点变换累积相乘得到。这一设计最早在1980年代的3D图形系统PHIGS中被形式化，后来被OpenInventor、Java3D等图形框架广泛采用，并成为Unity、Unreal Engine等现代游戏引擎的标准场景管理方式。

场景图解决的核心问题是"相对运动的自动传播"。在没有场景图的情况下，若要让一辆坦克炮塔跟随车身旋转，开发者需要手动在每一帧计算炮塔的世界坐标。场景图将炮塔作为车身节点的子节点，车身的变换自动传播给炮塔，开发者只需维护炮塔相对于车身的局部偏移即可。

## 核心原理

### 变换矩阵的层级累乘

场景图中每个节点存储一个4×4的局部变换矩阵（Local Transform Matrix），表示该节点相对于父节点的位移、旋转和缩放。某节点的世界变换矩阵（World Transform Matrix）由以下公式计算：

```
M_world = M_parent_world × M_local
```

其中 `M_parent_world` 是父节点的世界变换矩阵，`M_local` 是当前节点的局部变换矩阵。对于根节点，`M_world = M_local`。这意味着树的深度每增加一层，就需要额外进行一次矩阵乘法运算。在Unity中，这个世界矩阵通过 `Transform.localToWorldMatrix` 属性暴露给开发者。

矩阵乘法不满足交换律，因此变换顺序至关重要：标准顺序为先缩放（Scale）、再旋转（Rotation）、后位移（Translation），即TRS变换模型。若顺序错误，一个绕自身中心旋转45度的子对象，将变成绕父节点原点公转45度的错误行为。

### 脏标记机制（Dirty Flag）

朴素实现中，每帧对场景图所有节点重新计算世界矩阵代价极高。现代引擎使用脏标记（Dirty Flag）优化：当某节点的局部变换被修改时，引擎将该节点及其所有后代节点标记为"脏"（dirty = true），但不立即重新计算。只有在节点的世界变换被实际访问时（例如提交给渲染管线），才触发自顶向下的矩阵重新计算。Unity在内部使用此机制，这也是为什么频繁读取 `Transform.position` 比缓存结果消耗更多CPU时间的原因。

脏标记的传播方向与变换计算方向一致：从父节点向子节点单向传播。一个节点被标记为脏后，其所有子孙节点也被连带标记，但父节点不受影响。

### 节点的父子关系操作

场景图的拓扑操作包括挂载（Attach）、解绑（Detach）和重新父化（Re-parent）。重新父化是一个需要特别注意的操作：将节点从一个父节点移动到另一个父节点时，若希望对象在世界空间中的位置保持不变，必须同时更新其局部变换矩阵。

设节点原世界变换为 `M_world`，新父节点的世界变换为 `M_new_parent_world`，则新局部变换为：

```
M_new_local = M_new_parent_world_inverse × M_world
```

Unity的 `Transform.SetParent(parent, worldPositionStays: true)` 正是自动执行了这个逆矩阵计算过程。若传入 `worldPositionStays: false`，则直接复用旧的局部矩阵，对象会"跳位"到新父节点坐标系下的对应位置。

## 实际应用

**角色装备系统**：RPG游戏中，武器、盾牌挂载为角色骨骼节点的子节点。当骨骼动画驱动手部关节旋转时，武器自动跟随，无需额外代码。武器的局部变换仅需存储相对于握柄骨骼的偏移，通常是一个固定的位移向量，例如 `(0, 0.3, 0.1)` 米。

**摄像机绑定**：第三人称跟随摄像机作为玩家角色的子节点，存储固定偏移 `(0, 3, -6)` 米。角色移动时摄像机自动跟随，而摄像机自身的旋转操作只修改其局部旋转矩阵，不影响父节点（角色）的变换。

**UI层级组织**：游戏的HUD界面通常也采用场景图管理。一个血条容器节点作为父节点，血条背景、血条填充图、数值文本均作为子节点。移动容器位置时，所有子元素整体偏移，符合UI模块化设计需求。Unreal Engine的UMG和Unity的Canvas均采用此结构。

## 常见误区

**误区一：认为场景图深度不影响性能**。实际上，场景图的深度直接影响世界矩阵的计算链长度。一个深度为10层的节点，访问其世界变换时最坏情况需要向上遍历10次矩阵乘法。在包含数千动态节点的移动游戏项目中，不合理的深层嵌套（如将地形细节对象嵌套7-8层）会导致明显的帧率下降。通常建议将频繁移动的对象保持在场景图的浅层（3层以内）。

**误区二：缩放的继承会产生非均匀缩放问题**。若父节点在X轴缩放2倍而Y轴不缩放，子节点即便自身设置均匀缩放，其世界矩阵也会包含非均匀缩放成分。非均匀缩放下，法线向量不能直接用模型矩阵变换，必须使用法线矩阵（模型矩阵左上3×3子矩阵的逆转置矩阵）进行修正，否则光照计算将出现错误的高光方向。这是场景图中父节点带缩放时最易忽视的渲染问题。

**误区三：混淆局部坐标与世界坐标**。初学者常在局部空间中做世界空间的运算，例如直接将两个不同父节点下的子对象的 `localPosition` 相减来计算距离，这在父节点存在旋转时会给出完全错误的结果。正确做法是先将两个节点的位置转换为世界空间坐标（即应用世界变换矩阵），再进行向量运算。

## 知识关联

场景图建立在**场景管理概述**中介绍的游戏对象模型之上：每个游戏对象成为场景图中的一个节点，节点持有局部变换数据。理解TRS变换矩阵和矩阵乘法是掌握场景图变换传播的先决条件。

掌握场景图后，下一步学习**空间分区**（如四叉树、八叉树、BVH）时会发现一个重要区别：场景图按逻辑从属关系组织节点（炮塔属于坦克），而空间分区按几何位置组织对象（炮塔在地图左上区域）。两者服务于不同目的——场景图负责变换继承与逻辑分组，空间分区负责碰撞检测与可见性剔除的加速查询。现代引擎（如Unity）同时维护这两种结构，场景图的层级变换结果作为输入，驱动空间分区结构的动态更新。
