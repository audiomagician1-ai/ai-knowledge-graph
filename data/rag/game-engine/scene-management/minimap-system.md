# 小地图系统

## 概述

小地图系统（Minimap System）是游戏引擎场景管理模块中用于将大型游戏世界压缩呈现于屏幕角落的功能子系统，其核心职责是将三维或大规模二维场景以俯视视角实时渲染成一张缩略图，供玩家快速定位自身位置、感知周边地形与敌友分布。最早出现在1980年代的即时战略游戏中，《Dune II》（1992年）被公认为首批将小地图系统规范化运用的商业游戏之一，此后几乎所有开放世界、RTS（即时战略）与MOBA类游戏都将其列为必备UI组件。

小地图之所以重要，在于人类视野与游戏世界尺度之间的巨大差距。以《魔兽争霸III》为例，单张地图尺寸可达256×256格，而玩家主视野一次仅能看到约20×15格的范围，小地图将剩余约98%的区域信息压缩到屏幕一角，使战略决策周期从"需要移动视野确认"缩短为"瞬时扫视"。

## 核心原理

### 正交投影渲染

小地图通常使用**正交投影（Orthographic Projection）**而非透视投影来生成缩略图像。正交投影的变换矩阵不包含透视除法，平行线在投影后依然平行，因此地图上任意两点之间的像素距离与世界空间距离成固定比例关系：

```
缩放比 = 小地图像素尺寸 / 世界空间范围
例：128像素 / 2048单位 = 0.0625 像素/单位
```

实现上，引擎会在场景中放置一个专用的**俯视正交摄像机（Top-Down Orthographic Camera）**，其视锥体宽高等于整张地图的世界空间尺寸，远近裁剪平面设置为覆盖全部地形高度。该摄像机以固定帧率（通常每秒10~30帧，远低于主摄像机的60帧）将场景渲染到一张**RenderTexture**上，再将该纹理作为UI图像显示在屏幕HUD层。降低刷新率是小地图性能优化的最直接手段，因为玩家对小地图的实时性要求远低于主视野。

### 战争迷雾（Fog of War）

战争迷雾（Fog of War，简称FoW）是小地图系统中控制信息可见性的核心机制，将地图格子划分为三种状态：
- **未探索（Unexplored）**：完全黑色遮盖，玩家对此区域一无所知；
- **已探索但不可见（Explored/Hidden）**：显示地形但不显示动态单位，通常以半透明灰色蒙版叠加；
- **当前可见（Visible）**：己方单位视野范围内，实时显示所有信息。

技术实现上，FoW常用一张与地图等分辨率的**遮罩纹理（Mask Texture）**来存储每个格子的可见状态，每个像素的R/G/B通道分别编码三种状态。每当单位移动时，CPU或Compute Shader重新计算视野圆形区域并更新该纹理，渲染时将遮罩纹理与小地图RenderTexture做乘法混合（Multiply Blend）。《星际争霸II》使用GPU端的视野计算，使其在单位数量超过200时依然保持流畅。

### 标记与图标系统

小地图上的**标记（Marker）**是叠加在背景缩略图之上的一组2D精灵图标，用于指示己方单位、敌方单位、兴趣点（POI）、队友位置等动态信息。每个标记的屏幕坐标由世界坐标通过以下线性映射计算得出：

```
minimap_x = (world_x - map_origin_x) / map_width  × minimap_width
minimap_y = (world_z - map_origin_z) / map_height × minimap_height
```

图标尺寸通常固定在8×8至16×16像素之间，不随摄像机缩放变化，以保证在任何缩放级别下都清晰可辨。MOBA游戏（如《英雄联盟》）还引入了**优先级剔除**规则：当多个标记重叠时，英雄图标优先级高于小兵，绿色（友方）与红色（敌方）采用不同颜色编码，玩家可通过点击小地图直接下达移动命令，此时引擎将小地图坐标逆映射回世界坐标，作为导航目标点输入给寻路系统。

## 实际应用

**Unity中的小地图实现**：在Unity中，创建小地图的标准做法是新建一个Camera组件，将Projection设为Orthographic，Size设为地图半径（如1024），Culling Mask仅勾选"MinimapLayer"层，Target Texture指向一张128×128或256×256的RenderTexture，最后用UI RawImage组件将该RenderTexture显示在Canvas上，搭配Circular Mask组件裁剪为圆形。

**《艾尔登法环》风格的动态缩放**：开放世界游戏常允许玩家在主界面按下Tab键展开全图，此时小地图的正交摄像机的Orthographic Size从局部值（如200单位）动态插值到全图值（如4096单位），同时缩放精灵图标以维持可读性，这一过程在约0.3秒的动画过渡内完成。

**《DOTA 2》的小地图攻击提示**：当敌方在小地图区域发动攻击时，系统在被攻击位置短暂渲染一个红色闪烁圆圈标记，持续约1.5秒后淡出，提醒玩家注意该方位，这是标记系统与游戏事件系统深度整合的典型案例。

## 常见误区

**误区一：小地图应每帧更新以保证准确**。实际上，以每帧更新小地图RenderTexture会显著增加GPU负担，尤其在使用FoW时，视野遮罩计算本身已有成本。行业标准是将小地图刷新率设为10~20 FPS，玩家几乎感知不到延迟，但GPU开销下降幅度可达50%~70%。

**误区二：战争迷雾只是一张全屏半透明黑色遮罩**。初学者常用一张纯黑图片简单遮盖未探索区域，这种方法无法区分"已探索"与"未探索"状态，丢失了地形信息的渐进式揭露。正确实现需要维护一张持久化的"历史可见"纹理，记录玩家曾经到达过的所有格子，将其与当前可见纹理合并后才能正确显示灰色的"迷雾已探索"区域。

**误区三：小地图点击坐标可以直接用作世界坐标**。屏幕上的小地图UI坐标需要先换算到小地图的归一化坐标（0~1范围），再乘以地图世界空间尺寸并加上地图原点偏移，才能得到正确的世界坐标。忽略地图原点偏移是新手最常见的bug来源，尤其当地图不以世界原点(0,0,0)为中心时。

## 知识关联

小地图系统直接依赖**场景管理概述**中建立的概念：场景分层（Layer）机制决定了哪些对象会被小地图摄像机捕捉，场景坐标系的世界空间定义是标记坐标映射公式中`map_origin`与`map_width`参数的来源。理解场景中摄像机的投影类型差异（透视 vs 正交）是实现小地图的前置知识，因为混淆两种投影会导致地图比例失真。

在技术深度上，小地图系统可进一步扩展至**程序化地图生成**（将PCG生成的地形同步反映到小地图纹理）以及**网络同步**（多人联机时如何在客户端正确呈现各玩家的FoW状态而不泄露信息），但这些属于更高难度的工程课题。对于引擎初学者而言，能在Unity或Godot中实现一个带有静态FoW遮罩与动态单位标记的功能性小地图，即可视为对本概念的完整掌握。
