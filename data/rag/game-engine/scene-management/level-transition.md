# 关卡过渡

## 概述

关卡过渡（Level Transition）是游戏引擎场景管理中负责将玩家从一个游戏区域切换到另一个区域的技术机制。其核心问题是：如何在释放旧场景内存、加载新场景资源的同时，保证玩家体验的连续性或至少减少等待时间的心理负担。根据实现方式不同，主要分为三类方案：无缝过渡（Seamless Transition）、加载屏幕过渡（Loading Screen Transition）和流式加载过渡（Streaming Transition）。

关卡过渡技术的演进与硬件存储速度密切相关。早期 NES 时代游戏受 ROM 容量限制，关卡间切换依赖简单的画面闪烁和短暂黑屏。进入 PS1/N64 时代，CD-ROM 读取速度约 150 KB/s，强制引入了加载屏幕来遮蔽数十秒的磁盘读取。随着 SSD 普及和 PS5 的 DirectStorage 架构（理论带宽 5.5 GB/s），加载时间压缩至不足 1 秒，促使开发者重新考量是否仍需要传统加载屏幕。

理解关卡过渡对游戏开发者的核心价值在于内存管理与玩家沉浸感的平衡。一款 AAA 游戏的单个场景资源可能高达 2–4 GB，在切换时若处理不当，会出现内存峰值导致的崩溃，或因 GPU 显存驱逐造成的材质闪烁。正确选择过渡方案直接决定了游戏的性能稳定性和叙事沉浸感。

## 核心原理

### 加载屏幕过渡

加载屏幕过渡是最直接的方案：渲染一张静态或动画界面，在后台完整地卸载当前场景并加载目标场景，完成后再切换回游戏画面。其典型实现流程为：

1. 触发过渡事件 → 淡出当前画面
2. 在主线程之外启动异步加载协程（如 Unity 的 `LoadSceneAsync`，返回 `AsyncOperation` 对象）
3. 轮询 `AsyncOperation.progress`，该值从 0 升至 0.9（Unity 刻意在 0.9 处暂停，等待 `allowSceneActivation = true`）
4. 加载完毕后激活新场景，淡入游戏画面

这种方案内存管理最为清晰，因为新旧场景不共存，内存峰值等于最大单场景占用量加上加载屏幕自身资源。缺点是打断叙事，适合章节感强、关卡边界明确的游戏（如《黑暗之魂》系列的火炉传送）。

### 无缝过渡

无缝过渡要求玩家在过渡期间全程可操控角色，视觉上不出现任何加载界面。实现方法是在触发过渡条件前，将目标场景在后台预加载完毕，然后在单帧内执行"场景替换"：卸载旧场景中不复用的对象，激活已静默加载的新场景对象。

关键挑战是"双场景共存期"的内存峰值：此时内存占用 = 旧场景 + 新场景 + 角色/UI 持久对象。为此，设计上通常在连接两个场景的"过渡走廊"（Transition Corridor）中放置几何体，走廊长度需确保玩家走完所需时间 ≥ 后台加载完成时间。《黑暗之魂》中连接两区域的电梯，以及《战神 2018》中始终保持连续镜头的洞穴段落，均使用此策略遮盖后台加载。

### 流式加载过渡

流式加载（Streaming）不以"关卡"为单位加载，而是以区块（Chunk）或流关卡（Streaming Level / Sub-Level）为单位，随玩家位置动态加载和卸载。在 Unreal Engine 中，World Composition 系统将大世界划分为若干子关卡，每个子关卡有独立的 `StreamingBounds`；当玩家与子关卡中心距离小于 `LoadingRange` 时触发加载，大于 `UnloadingRange` 时触发卸载，两个阈值之间设有滞后区间（Hysteresis），防止玩家在边界反复横跳导致颠簸加载。

流式加载的核心参数是视距（View Distance）与内存预算的权衡。Unreal 引擎通过 `r.Streaming.PoolSize`（单位 MB）限制纹理流送池大小；若实际需求超过该预算，引擎会降低 Mip 等级，出现材质模糊的"LOD Pop"现象。典型 AAA 游戏的 Streaming Pool 设定在 512–2048 MB 之间，视目标平台显存而定。

## 实际应用

**《原神》的区域切换**采用混合方案：同一大区域内使用流式加载保持无缝探索，跨大区域（如从蒙德传送至璃月）使用约 3 秒的加载屏幕，同时在加载期间展示提示文字以缓解等待感。这是典型的"流式 + 加载屏兜底"组合策略。

**Unity 中的 Additive 场景加载**是实现流式过渡的实用手段。调用 `LoadSceneAsync("ChunkB", LoadSceneMode.Additive)` 可在不卸载当前场景的情况下叠加加载新场景；再通过 `UnloadSceneAsync("ChunkA")` 异步卸载离开的区域。相比 Single 模式加载，Additive 模式允许开发者精确控制对象生命周期，是开放世界游戏场景管理的标配做法。

**加载屏幕的交互设计**本身也影响过渡体验。Namco 于 1995 年在《山脊赛车》中首创"在加载屏幕期间运行可玩 Demo"的专利，该专利于 2015 年到期，此后《最后生还者 第一章》等游戏广泛采用加载期间播放过场动画或显示可交互提示的方式降低等待感。

## 常见误区

**误区一：流式加载可以完全取代加载屏幕**。流式加载依赖玩家移动速度给加载系统留出时间；一旦玩家通过传送点瞬间跨越大距离，流式系统来不及完成加载，仍需退化为加载屏幕。《艾尔登法环》在快速旅行时即显示加载界面，正是因为目标区域不在当前流式加载范围内。

**误区二：无缝过渡总比加载屏幕更好**。无缝过渡的内存峰值可能是加载屏幕方案的 1.5–2 倍。在内存受限平台（如 Nintendo Switch 仅 4 GB 统一内存）上强行使用无缝过渡，反而可能导致系统内存不足崩溃，得不偿失。

**误区三：`AsyncOperation.progress == 1.0` 意味着场景已可用**。在 Unity 中，该值仅代表资源读取完毕，场景内 `Awake()`/`Start()` 的初始化逻辑尚未执行。若不等待 `SceneManager.sceneLoaded` 事件回调，直接尝试访问新场景内的对象，将引发空引用异常。

## 知识关联

关卡过渡直接建立在**世界组合（World Composition）**的概念之上：世界组合定义了场景如何被划分为子关卡和区块，而关卡过渡则是在运行时动态调度这些子关卡加载与卸载的执行层。没有清晰的子关卡划分，流式过渡将无法确定加载/卸载边界；反之，设计了合理的世界组合结构后，关卡过渡方案的选择（无缝、加载屏或流式）便取决于玩家移动模式和目标平台内存预算。

在渲染管线层面，关卡过渡还与**遮挡剔除（Occlusion Culling）**和**LOD 系统**协作——流式过渡激活新区块后，遮挡数据需要重新构建，LOD 系统需要在新场景对象完全加载前以低精度模型占位，避免画面空洞。掌握关卡过渡的三种方案及其内存特征，是实现稳定开放世界游戏的基础工程能力。
