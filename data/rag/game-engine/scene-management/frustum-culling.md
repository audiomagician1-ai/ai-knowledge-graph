# 视锥剔除

## 概述

视锥剔除（Frustum Culling）是游戏引擎渲染管线中的一种加速技术，其核心思路是：在将场景物体提交给 GPU 绘制之前，先在 CPU 端判断哪些物体位于摄像机视锥体（View Frustum）的范围之外，并将这些不可见物体完全跳过，从而避免无效的 Draw Call 和顶点处理开销。

视锥剔除的概念伴随着 3D 实时渲染的发展而逐步成熟。早在 1984 年，Sutherland-Hodgman 算法就已经提出了针对凸多边形的裁剪思路；到 1990 年代，随着家用游戏机和 PC 3D 游戏的普及，开发者开始在场景管理层面引入整体物体的视锥剔除，而非仅仅依赖 GPU 的硬件裁剪。如今，Unity、Unreal Engine 等主流引擎均在渲染循环的最前端默认执行视锥剔除。

视锥剔除的意义在于其剔除时机极早——发生在 CPU 向 GPU 提交渲染命令之前。一个复杂场景可能包含数千个网格对象，而摄像机在任意帧实际可见的往往不足 20%。如果缺少视锥剔除，所有对象都会进入 GPU 流水线，造成顶点着色器和光栅化阶段的大量浪费。

## 核心原理

### 视锥体的六平面表示

透视摄像机的视锥体是由六个平面围成的截锥体，分别是：近裁剪面（Near）、远裁剪面（Far）、左面（Left）、右面（Right）、上面（Top）和下面（Bottom）。每个平面用平面方程 **n⃗ · x⃗ + d = 0** 来表示，其中 **n⃗** 是平面的单位法线（朝向视锥体内部），**d** 是平面到原点的有符号距离。判断一个点 **P** 是否在该平面的"内侧"，只需计算 **n⃗ · P + d ≥ 0**。对六个平面全部通过则点在视锥体内。

正交摄像机的视锥体退化为长方体（AABB），因此其六个平面两两平行，剔除计算量更小，常用于 UI 相机或 2D 游戏的正交投影场景。

### 包围体测试

直接对物体的每个顶点做六平面测试过于昂贵。实践中通常使用简化的包围体来近似物体的空间占用：

- **包围球（Bounding Sphere）**：仅需计算球心到每个平面的有符号距离，再与球半径 *r* 比较。若球心距某平面的有符号距离 **< −r**，则球完全在视锥体外侧，可直接剔除。包围球测试只需 6 次点积 + 6 次比较，速度极快。
- **轴对齐包围盒（AABB）**：将 AABB 的 8 个角点与六平面逐一检测，若某平面的"正侧极端点"（p-vertex）满足 **n⃗ · p_vertex + d < 0**，则整个 AABB 在该平面外，物体可剔除。AABB 测试的误判率（把可见物体误判为不可见）接近于零，比包围球更紧。

两种包围体各有取舍：包围球计算速度更快但包裹不紧（细长物体浪费率高），AABB 包裹更紧但需要额外处理旋转后的重新计算。

### 与空间分区结合的批量剔除

单独对每个物体做视锥剔除时，复杂度为 O(N)，N 是场景物体总数。将视锥剔除与八叉树（Octree）或 BVH（层次包围盒）结合后，可实现树状早退出（Early-Out）：当一个节点的包围体完全在视锥体外时，其所有子节点直接跳过，无需逐一检查。典型场景下，这可以将需要逐一测试的对象数量从数千个降低到数十个，从 O(N) 降至 O(log N) 量级。

Unreal Engine 的场景管理使用 BVH 结合视锥剔除，在每帧 `FSceneRenderer::PreVisibilityDetermine` 阶段批量计算可见性，结果存储在 `FViewInfo::PrimitiveVisibilityMap` 位数组中，后续渲染阶段直接查表，避免重复计算。

## 实际应用

**开放世界地形渲染**：在《荒野大镖客：救赎 2》等开放世界游戏中，地形被划分为若干 Chunk，每个 Chunk 拥有独立的 AABB。玩家摄像机在移动时，视锥剔除每帧可跳过大约 60%–80% 的地形块提交，使 Draw Call 数量维持在可控范围内。

**Unity 中的内置支持**：Unity 的 `Camera` 组件默认开启视锥剔除，开发者可通过 `GeometryUtility.CalculateFrustumPlanes(camera)` 获取六个裁剪平面，再配合 `GeometryUtility.TestPlanesAABB(planes, bounds)` 手动对自定义对象做测试，返回 `true` 表示 AABB 与视锥体相交（需要渲染），`false` 表示完全在外（可跳过）。

**实例化渲染（GPU Instancing）的前置剔除**：当使用 `Graphics.DrawMeshInstanced` 渲染大量草丛或石块时，需要在 CPU 端先用视锥剔除筛选出可见实例，再将可见实例的矩阵列表传给 GPU，否则实例数组超过 1023 个的硬上限也会造成多次 Draw Call 分批，剔除可显著减小数组规模。

## 常见误区

**误区一：视锥剔除能处理遮挡关系**
视锥剔除只判断物体是否在摄像机视野范围之内，无法判断物体是否被其他物体遮挡。一堵厚墙背后的整栋建筑，视锥剔除无法将其剔除——这需要额外的遮挡剔除（Occlusion Culling）技术。将两者混淆会导致开发者误以为视锥剔除已经足够，而忽略了被遮挡物体仍在消耗渲染资源的问题。

**误区二：视锥剔除的包围体越紧越好**
包围体越紧（如用 OBB 替代 AABB），空间浪费越少，但包围体本身的测试计算量也越大。对于静态场景中的规则网格，AABB 的计算开销远低于 OBB，且由于物体密度高时剔除效率本已很高，改用更复杂的包围体往往得不偿失。只有当物体极度细长（如长桥、长围栏）且 AABB 误判率极高时，才值得考虑 OBB 或包围胶囊体。

**误区三：视锥剔除在 GPU 管线中自动完成，无需 CPU 介入**
GPU 硬件裁剪（Clipping）发生在顶点着色之后的图元装配阶段，此时顶点已经被处理，Draw Call 已经发出。CPU 端的视锥剔除发生在提交 Draw Call 之前，完全避免了这些顶点进入 GPU 流水线，两者处于流水线的不同阶段，作用互补而非替代。

## 知识关联

视锥剔除依赖**空间分区**（如八叉树、BVH）来实现高效的批量测试。没有空间分区时，视锥剔除仍可工作，但退化为 O(N) 的线性遍历，在大场景下性能不足。空间分区为视锥剔除提供了树状遍历结构，两者配合才能在开放世界规模的场景中保持帧率稳定。

在视锥剔除确定"哪些物体在视野内"之后，**可见性系统**（Visibility System）进一步处理"哪些视野内的物体实际上能被看见"。可见性系统通常包含遮挡剔除（Occlusion Culling）、Portal-Cell 可见性查询等技术，是视锥剔除的下一步精化。视锥剔除的输出结果（候选可见集合）正是可见性系统的输入，两者共同构成场景管理中从粗到细的多级剔除流水线。
