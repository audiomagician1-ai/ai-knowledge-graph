# 场景持久状态

## 概述

场景持久状态（Scene Persistent State）是指在游戏引擎的场景管理中，将场景内物体的运行时数据（如位置、血量、开关状态、动画帧等）序列化保存到持久化存储介质，并在需要时将这些数据反序列化回场景的完整机制。与仅记录玩家角色数据的简单存档不同，场景持久状态追踪的是**场景中所有相关实体**的状态快照，包括已被摧毁的箱子、打开的门、触发过的机关、掉落在地板上的道具等。

这一机制最早在1990年代的RPG游戏引擎中被系统化处理。早期《最终幻想》系列使用固定内存地址存储地图标记位（flag bit），每个场景事件对应一个二进制位。现代引擎如Unity和Unreal Engine 5则通过组件序列化系统或Level Instance保存架构来实现更细粒度的场景状态管理。理解场景持久状态的意义在于：如果缺少这套机制，玩家每次重新加载场景时都会看到"满血复活"的敌人和"原封未动"的场景，彻底破坏游戏世界的连续感。

## 核心原理

### 可持久化对象的标识与筛选

并非场景中所有GameObject都需要持久化。引擎通常通过**唯一持久ID（Persistent GUID）**来标记需要保存的对象。Unity中每个资源和预制体实例在Editor阶段会生成128位的GUID，但运行时动态生成的对象则需要开发者手动分配稳定ID。

筛选策略分为白名单和黑名单两种：白名单仅保存带有`ISaveable`接口或特定标签的对象；黑名单则排除纯装饰性粒子、临时特效等无需恢复的对象。场景中一个典型的可持久化物体清单包括：可破坏环境、NPC当前状态、门锁开关、收集物（已拾取/未拾取）。

### 状态数据的序列化格式

场景持久状态的序列化通常使用**差量存储（Delta Storage）**而非全量存储。全量存储保存场景内每个物体的全部属性，数据量随场景规模线性增长；差量存储仅记录与初始预设值不同的字段，可将数据体积压缩70%~90%。

一个具体的差量状态数据示例（JSON格式）：
```json
{
  "sceneId": "dungeon_level_02",
  "timestamp": 1698423600,
  "objects": [
    { "id": "chest_007", "destroyed": true },
    { "id": "door_east", "isOpen": true, "rotation": [0, 90, 0] },
    { "id": "enemy_goblin_03", "hp": 0, "position": [12.5, 0, -3.2] }
  ]
}
```
其中`sceneId`用于与特定场景文件绑定，`id`对应上文的持久GUID。恢复时引擎按`id`查找场景中已加载的对象并注入保存的字段值；若对象标记为`destroyed`则在Awake阶段立即销毁该实例。

### 场景加载时的状态恢复流程

场景持久状态的恢复必须在正确的生命周期节点执行。以Unity为例，标准恢复流程遵循以下顺序：

1. **SceneManager.LoadScene()触发** → 场景所有GameObject执行Awake()
2. **存档系统读取对应sceneId的状态块**
3. **在Start()之前完成状态注入**（若在Start()之后注入，组件可能已基于默认值执行了初始化逻辑）
4. **销毁标记为destroyed的对象**
5. **对存活对象更新Transform、组件数值、动画状态机参数**

这个时序至关重要：如果在Awake()之前注入数据，部分组件尚未初始化会导致空引用；如果在Start()之后注入，NPC的巡逻路径计算可能已经基于错误的初始HP或位置运行过一帧。

### 跨场景切换时的状态缓存

当玩家在多个场景之间频繁切换时（如城镇→副本→城镇），已卸载场景的状态需要缓存在内存中或写入磁盘。内存缓存策略使用一个以`sceneId`为键的字典，保存最近N个（通常3~5个）已访问场景的状态块；超出限制时将最久未访问的状态写入磁盘并从内存移除（LRU淘汰策略）。这避免了每次切换场景都触发磁盘I/O的性能开销。

## 实际应用

**开放世界游戏中的资源刷新控制**：《塞尔达传说：旷野之息》使用场景持久状态记录每株草丛、每块矿石的采集状态，并配合游戏内时间系统实现72小时（游戏时间）后的条件刷新。其内部将采集状态存储为每个区块对应的位标志（bit flag）数组，而非逐对象存储，大幅降低了内存占用。

**恐怖游戏的场景破坏保留**：恐怖游戏要求玩家回到之前探索过的房间时，之前翻倒的家具、打破的窗户依然保持破坏状态，以维持紧张感。实现上，每个可破坏物体维护一个`destructionState`枚举（Intact/Damaged/Destroyed）作为最小状态集，序列化时仅写入非Intact状态的对象。

**多人联机场景同步**：在服务器权威的多人游戏中，场景持久状态由服务器维护主版本（Master State），客户端加入时服务器推送完整状态快照（Full Snapshot），后续仅同步增量变更（Delta Update）以节省带宽。每条Delta Update包含`objectId`、`fieldName`、`newValue`三元组。

## 常见误区

**误区一：把场景持久状态等同于玩家存档**
场景持久状态管理的是场景内所有相关实体的状态，而玩家存档通常只记录玩家角色的属性（背包、等级、任务进度）。两者在数据归属、触发时机和存储格式上均不同：场景持久状态在每次场景卸载时自动触发序列化，而玩家存档通常在玩家主动操作或到达存档点时触发。将二者混入同一数据结构会导致存档文件无限膨胀，并使"只读取玩家数据而不加载场景状态"的需求难以实现。

**误区二：对所有对象使用Transform全量序列化**
初学者常犯的错误是将场景中每个GameObject的完整Transform（position 3个float + rotation 4个float + scale 3个float = 40字节/对象）全部序列化。一个含500个可交互对象的场景若全量存储Transform，单帧快照就超过20KB。正确做法是只序列化**发生过变化**的字段，并对旋转数据使用欧拉角（3个float）而非四元数（4个float）以节省空间，仅在需要插值的场景才保留四元数格式。

**误区三：认为场景持久状态可以完全依赖引擎内置序列化**
Unity的`JsonUtility`或Unreal的`SaveGame`类提供了通用序列化，但它们不能自动处理以下情况：运行时动态实例化的对象（没有预设GUID）、对其他场景对象的引用（跨场景引用在反序列化时对象可能尚未加载）、以及自定义着色器属性等非标准组件数据。这些边界情况需要开发者在序列化层上实现额外的引用解析（Reference Resolution）逻辑。

## 知识关联

场景持久状态建立在**场景管理概述**所介绍的场景生命周期（Load→Active→Unload）之上——只有明确了Unload时机，才能确定序列化应在哪个回调中执行（Unity中对应`OnSceneUnloaded`事件）。同时，场景持久状态的磁盘写入部分直接依赖**存档系统**提供的文件路径管理、加密和版本兼容机制：存档系统决定"数据写到哪里、怎么读回来"，而场景持久状态决定"要写哪些数据、用什么结构组织"。两套系统通过明确的接口解耦：场景持久状态只调用存档系统提供的`Save(key, bytes)`和`Load(key): bytes`接口，不关心底层是写入`PlayerPrefs`、本地JSON文件还是云端数据库。
