# 场景持久状态

## 概述

场景持久状态（Scene Persistent State）是游戏引擎场景管理中用于**保存和恢复场景内物体运行时状态**的机制。当玩家保存游戏、切换场景或重新启动程序后，场景中的门是否被打开、箱子是否被搜刮、NPC 是否已死亡等信息不会丢失，而是通过持久状态系统记录下来，待下次加载时原样恢复。

这一概念伴随着角色扮演游戏（RPG）的发展在 1980 年代逐渐成型。早期文字冒险游戏如《Zork》（1977）已有简单的房间状态记忆，到了《创世纪》（Ultima）系列，开放世界中每个区域的物件状态需要独立跟踪，这直接推动了结构化场景状态存储的出现。

在现代 3D 游戏引擎（如 Unity、Unreal Engine 4/5）中，场景持久状态的重要性体现在：一旦处理不当，玩家会遇到已清空的怪物营地"复活"、拾取过的物品"归位"等破坏沉浸感的 Bug。正确实现场景持久状态是开放世界和关卡重访类游戏品质的直接保障。

## 核心原理

### 可持久化数据的识别与分类

并非场景中所有数据都需要持久化，错误地保存过多状态会导致存档文件膨胀。一般将场景物体的状态分为三类：

- **静态配置数据**：物体的初始位置、模型资源引用——这些数据已存在于场景文件本身，**无需重复保存**。
- **运行时动态状态**：物体是否被激活（`isActive`）、血量、门的开关角度、NPC 的对话进度标志位——这些是场景持久状态的核心保存目标。
- **临时过程数据**：动画播放进度、粒子系统当前帧——通常不需要持久化，加载后重新初始化即可。

### 唯一标识符（GUID）机制

场景持久状态系统必须解决一个根本问题：存档数据如何与场景中具体物体对应？解决方案是为每个需要持久化的场景物体分配一个**全局唯一标识符（GUID）**。

以 Unity 为例，编辑器会为每个 GameObject 生成一个 128 位的 GUID 并存储在 `.meta` 文件中。运行时可自定义一个 `PersistentID` 组件，在 `Awake()` 阶段向场景状态管理器注册自身，格式通常为：

```
SceneID_ObjectGUID → StateData
```

例如 `"dungeon_01_door_A3F2"` 映射到 `{ isOpen: true, rotation: 90.0 }`。这样即使场景重新加载，物体通过 GUID 就能精准找回自己的状态。

### 序列化与反序列化流程

场景持久状态的存取遵循固定的生命周期：

1. **保存阶段**（触发时机：玩家手动存档、切换场景前）：遍历场景中所有带 `IPersistable` 接口的物体，调用各自的 `CaptureState()` 方法，将返回的状态对象序列化为 JSON 或二进制格式，写入以场景名称为键的存档字典。
2. **加载阶段**（触发时机：场景加载完成后的 `Start()` 或 `OnSceneLoaded` 回调中）：从存档字典读取对应场景的状态数据，反序列化后按 GUID 分发给各物体，各物体调用 `RestoreState()` 应用数据。
3. **首次进入处理**：若存档字典中不存在该场景的条目，说明玩家首次进入，使用场景文件中的默认配置，无需任何恢复操作。

一个典型的状态接口定义如下：
```csharp
interface IPersistable {
    object CaptureState();          // 返回可序列化的状态对象
    void RestoreState(object state); // 从状态对象恢复物体
}
```

### 状态数据的存储结构

场景持久状态数据通常以**嵌套字典**结构组织，外层键为场景标识符（如 `"Level_Forest"`），内层键为物体 GUID，值为序列化的状态负载：

```json
{
  "Level_Forest": {
    "chest_01_B2E9": { "isLooted": true },
    "door_gate_CC41": { "isOpen": false, "lockBroken": true }
  }
}
```

这种结构使得不同场景的状态相互隔离，查找复杂度为 O(1)，且便于按场景整体清除（如游戏中的"重置副本"功能）。

## 实际应用

**开放世界中的资源采集**：《塞尔达传说：旷野之息》中砍倒的树木会在特定条件下重新生长，而已开启的宝箱则永久保持打开状态。这两种行为都依赖场景持久状态：宝箱的 `isOpened` 标志永久写入存档，而树木状态绑定到游戏内时间，超过阈值后持久状态被重置。

**副本与关卡重访**：在 Dungeon 类游戏中，副本未完成时离开，再次进入需要恢复到离开前的状态——哪些怪物已死、哪些机关已触发。这要求在"场景卸载前"（`OnDestroy` 或场景切换钩子）触发一次完整的 `CaptureState()` 保存，而不是仅在玩家手动存档时才保存。

**对话与任务标志位**：NPC 的对话分支选择结果（如已告知过玩家某情报）以布尔或枚举值存储在该 NPC 的持久状态中，避免玩家重进场景后 NPC 重复说出首次见面台词。

## 常见误区

**误区一：把场景持久状态等同于全局存档**
许多初学者将所有游戏数据（玩家背包、技能树、游戏进度）与场景状态混存在同一个存档文件中，导致结构混乱。正确做法是区分"玩家全局存档"（跨场景的玩家数据）和"场景持久状态"（特定场景内物体的本地状态），两者分开存储，按需加载。

**误区二：在场景加载完成前调用 RestoreState**
常见错误是在 `Awake()` 而非 `Start()` 中触发状态恢复，此时部分依赖物体可能尚未初始化，导致 `NullReferenceException`。正确时序是：场景所有物体完成 `Awake()` 初始化后，在 `Start()` 阶段（或 `OnSceneLoaded` 回调的帧末）统一下发恢复指令。

**误区三：每帧持续写入状态**
将 `CaptureState()` 放在 `Update()` 中每帧调用，导致频繁的序列化和字典写入，造成明显的性能损耗。场景持久状态的捕获应为**事件驱动**——仅在玩家存档、场景卸载、关键事件发生时触发，而非持续轮询。

## 知识关联

场景持久状态建立在**场景管理概述**所描述的场景生命周期之上——`OnSceneLoaded`、`OnSceneUnloaded` 等事件钩子正是触发状态保存与恢复的时机节点，不理解场景的加载/卸载顺序就无法正确安排持久化的时序。

与**存档系统**的关系是：存档系统负责将数据持久化到磁盘（文件 I/O、加密、版本兼容），而场景持久状态负责从场景内物体中**提取和注入**运行时数据——前者是存储层，后者是数据采集与分发层。两者协作时，场景状态字典作为中间层，由存档系统统一序列化落盘，加载时由存档系统读取后再交由场景状态系统分发还原。
