# 场景优化综合

## 概述

场景优化综合是游戏引擎场景管理中将**合批（Batching）、视锥剔除（Frustum Culling）、流式加载（Streaming）与层级细节（HLOD）**四类技术协同运用的整体策略。单独使用任一技术只能解决渲染管线的局部瓶颈，而综合策略的目标是使 CPU DrawCall 数量、GPU 像素填充率、内存驻留量三项指标同时控制在目标平台的预算范围内。

该策略的工程化实践始于 2010 年代大型开放世界游戏的爆发期。《荒野大镖客：救赎2》（2018）的技术分享披露，其场景管理系统需要在同一帧内同时维护约 30 万个场景对象的可见性状态，仅靠单一剔除手段无法满足 30 FPS 的目标帧率，正是四类技术的综合调度才使其成为可能。

理解场景优化综合的意义在于：单项技术之间存在**相互约束**关系——激进的合批会破坏逐对象的剔除精度，过细的 HLOD 分层会增加流式加载的 I/O 次数，开发者必须量化各技术的收益与副作用，才能制定出适合具体项目的综合参数方案。

## 核心原理

### 1. 合批与剔除的耦合关系

静态合批（Static Batching）将多个网格合并为单一 VBO，使 DrawCall 从 N 降为 1，但代价是丧失了对合批内单个子网格的逐对象剔除能力。若合批后的包围盒（AABB）跨越相机视锥边界，渲染器必须提交整个合批，即使其中 80% 的子网格已位于视锥之外。

解决此问题的标准做法是**分块合批（Chunk-Based Batching）**：将世界划分为固定边长的单元格（Unreal Engine 默认 Cluster 半径为 300 cm），仅在单元格内部执行合批。这样剔除的粒度从单对象降为单元格，在保留大部分 DrawCall 收益的同时，将无效提交比例控制在 15% 以内（Unity DOTS 文档给出的经验值）。

GPU Instancing 是合批的另一种形式，它不合并几何体，而是用一次 DrawInstanced 调用渲染 N 个相同网格，并可对每个实例独立进行遮挡查询，因此与剔除系统的兼容性优于静态合批。

### 2. HLOD 与流式加载的分层协作

HLOD（Hierarchical Level of Detail）为场景中的建筑群、植被群等生成多层代理网格。以 Unreal Engine 5 的 World Partition 系统为例，HLOD 0 层代理使用原始三角形数的约 10%，HLOD 1 层进一步降至 1%。流式加载系统根据相机距离决定哪一层 HLOD 驻留于显存：

- **距离 < 加载半径**：卸载 HLOD 代理，流式加入原始高精度资产
- **距离 ≥ 加载半径**：原始资产从显存驱逐，HLOD 代理接管渲染

两套系统必须共享同一套**距离阈值参数**，否则会出现"弹出（Popping）"现象——原始网格尚未加载完成时 HLOD 代理已经卸载，造成场景闪烁。解决方案是设置 10%~20% 的迟滞区间（Hysteresis Zone），在过渡距离附近同时驻留两级资产并做 Alpha 淡出。

### 3. 可见性系统作为综合调度的驱动核心

本文所有技术依赖从可见性系统获得的**每帧可见集合（Visible Set）**作为输入。可见性系统通过 BVH（Bounding Volume Hierarchy）进行视锥剔除，再经遮挡剔除（Occlusion Culling）二次过滤，将场景对象数从 30 万压缩到实际需要渲染的数千个。

综合策略在此基础上增加两个调度逻辑：
1. **合批失效检测**：若某单元格的可见性在连续 3 帧发生变化（对象进出视锥），则暂停对该单元格的 GPU Instancing 更新，避免每帧重建 Instance Buffer 带来的 CPU 开销。
2. **流式优先级队列**：根据可见集合中对象的屏幕覆盖像素数（Screen Coverage）对待加载资产排序，优先加载屏幕占比 > 5% 的对象，抑制玩家视野之外的后台流式请求。

## 实际应用

### 开放世界游戏的典型参数方案

以 PC 平台中高画质为目标，一套常见的综合参数配置如下：

| 技术 | 参数 | 典型值 |
|------|------|--------|
| 分块合批 | 单元格边长 | 200～400 cm |
| HLOD 0 层切换距离 | 世界单位 | 5000～8000 |
| 流式内存预算 | 显存占比 | ≤ 60% |
| 遮挡剔除深度图分辨率 | 像素 | 512×512（软件光栅化） |

### 移动平台的差异化策略

移动端 GPU 带宽约为桌面的 1/5，因此流式加载优先级远高于 HLOD 精度。常见做法是取消 HLOD 1 层，仅保留 0 层代理，将节省出的显存预算（约 20 MB）分配给更激进的纹理流式压缩（ASTC 6×6 而非 4×4）。合批策略也从静态合批转向 GPU Instancing，因为移动端场景对象总量少，逐对象剔除的精度收益大于合批的 DrawCall 收益。

## 常见误区

**误区一：HLOD 可以替代流式加载**
部分开发者认为只要设计足够多的 HLOD 层级，就不需要将原始资产驱逐出显存。实际上 HLOD 代理网格本身仍占用显存，一个包含 500 栋建筑的城市场景，即便全部替换为 HLOD 1 层代理，总显存消耗仍可达到 300 MB 以上，超出大多数移动设备预算。流式加载是唯一能做到"按需驻留"的机制。

**误区二：合批越大越好**
将全场景合为一个 DrawCall 理论上最优，但在实测中，超过 10 万三角形的单个合批会导致 GPU 顶点处理阶段无法并行，反而使帧时间上升。Unreal Engine 的实测数据显示，单个静态合批的最优三角形数上限约为 **5 万个**，超出后收益递减。

**误区三：综合优化是一次性配置**
综合策略的各项阈值与玩家移动速度、相机 FOV、目标平台 I/O 带宽强相关。在赛车游戏中（相机速度可达 100 m/s），流式加载的预加载距离需比 RPG 游戏大 3～5 倍，HLOD 切换迟滞区间也需相应扩大，否则玩家会在行进中看到大量 Popping。

## 知识关联

**前置依赖：可见性系统**
本文所有四类技术的调度均以可见性系统输出的 Visible Set 为起点。若 BVH 构建质量低（如 SAH 代价函数参数错误），导致剔除过于保守，则合批收益会被过多的无效提交抵消，HLOD 切换判定也会产生误差。因此在实施综合策略前，必须先验证遮挡剔除的准确率（通常用"已提交但实际未覆盖任何像素的 DrawCall 占比"衡量，健康值应 < 5%）。

**横向关联：渲染管线预算分配**
场景优化综合的最终产出是 CPU/GPU/内存三项预算的联合分配结果。CPU 节省（来自 DrawCall 减少）可以部分转移为物理模拟或 AI 的计算时间；GPU 节省（来自剔除）可以用于后处理效果的提升。这种预算的跨系统调配是大型项目技术总监需要在综合优化完成后做出的工程决策，而非场景管理系统本身能自动完成的事。
