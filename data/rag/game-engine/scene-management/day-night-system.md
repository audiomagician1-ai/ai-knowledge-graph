# 昼夜系统

## 概述

昼夜系统（Day-Night Cycle System）是游戏引擎场景管理中负责模拟现实世界24小时光照变化的功能模块，通过在时间轴上动态驱动太阳/月亮位置、天空颜色、环境光强度等参数，使游戏世界呈现出随时间流逝的真实感。典型的昼夜系统会将游戏内一天压缩为现实时间的20分钟至2小时不等，例如《我的世界》默认将一个游戏日设定为现实20分钟，《上古卷轴5》则将一天设定为现实72分钟。

昼夜系统的概念最早出现在1990年代的开放世界RPG中，彼时仅能实现简单的贴图切换（白天贴图换夜晚贴图）。随着可编程渲染管线的普及，现代昼夜系统已发展为基于物理的大气散射模型驱动，能实时计算丁达尔效应、瑞利散射和米氏散射，产生黎明橘红色和正午蓝天的自然过渡。

对游戏设计者而言，昼夜系统不仅是视觉工具，它直接影响NPC行程、怪物刷新规则和玩家生存压力，将单一静态场景变为随时间演变的动态世界，显著延长游戏的重复可玩性。

## 核心原理

### 时间驱动的太阳轨迹计算

昼夜系统的核心是一条太阳运动曲线。技术上，太阳方向向量通常通过对游戏内时间参数 `t`（范围0.0～1.0，代表0:00到24:00）进行三角函数映射来计算：

```
SunDirection = (cos(t × 2π - π/2), sin(t × 2π - π/2), 偏移量Z)
```

其中仰角分量 `sin(t × 2π)` 在 t=0.25（正午）时达到最大值1.0，在 t=0.75（午夜）时降至-1.0。当太阳方向向量Y分量小于0时，切换至月亮光源接管主方向光。Unreal Engine的定向光组件（Directional Light）通过挂载 `SkyAtmosphere` Actor实现这一映射，Unity的HDRP管线则使用 `PhysicallyBasedSky` 体积组件完成同等计算。

### 光照参数的时间梯度插值

昼夜系统需要管理至少5类随时间变化的光照参数：

1. **主方向光颜色与强度**：正午白色光（色温约6500K），日落时衰减至橙红色（约2000K），夜间月光切换为冷蓝色（约4100K）；
2. **天空盒颜色梯度**：通常用1D渐变纹理（Gradient Texture）存储一天24小时的天空颜色，以游戏时间为UV横坐标采样；
3. **环境光（Ambient Light）强度**：夜间环境光降至白天的15%～30%，避免场景完全黑暗；
4. **雾（Fog）参数**：清晨雾浓度高于正午，可用曲线（Curve）资产单独控制雾起始距离和密度；
5. **阴影长度**：太阳仰角低时阴影拉长，通过方向光旋转自动实现，无需额外计算。

所有参数均通过关键帧曲线（Timeline/Curve）存储，在每帧 `Tick` 中以当前游戏时间为输入进行线性或样条插值，输出到对应渲染组件。

### 游戏时间管理器

昼夜系统需要一个全局时间管理器来维护"游戏时间"与"现实时间"的映射比率。设现实秒数为 `RealSeconds`，时间倍率为 `TimeScale`，则游戏内经过的秒数为：

```
GameSeconds = RealSeconds × TimeScale
```

《上古卷轴5》的 `TimeScale` 默认值为20，意味着现实1秒等于游戏内20秒。时间管理器还需负责处理快速旅行后的时间跳跃、室内场景禁用太阳渲染（但保持时间流逝）、以及存档读取时恢复到正确时刻等边界情况。

## 实际应用

### Gameplay机制联动

昼夜系统最直接的Gameplay应用是NPC日程驱动。《荒野大镖客：救赎2》中，NPC商店在游戏时间20:00至6:00关门，酒吧NPC在22:00后才会出现打架行为，这些逻辑均订阅了昼夜系统的时间事件（Time Event）。开发者通常在时间管理器中实现一个"时间段枚举"：黎明（5:00-7:00）、白天（7:00-18:00）、黄昏（18:00-20:00）、夜晚（20:00-5:00），Gameplay逻辑层监听枚举变化而非每帧轮询时间值，以降低耦合度。

### 怪物刷新与危险等级

在生存类游戏中，昼夜系统直接控制敌人刷新规则。典型实现是将"危险度系数"绑定到时间曲线：夜间（0.0-0.25和0.75-1.0区间）危险系数为1.5，怪物刷新数量乘以该系数。《泰拉瑞亚》中血月事件（Blood Moon）在游戏午夜12:00准时触发，且持续至游戏7:30日出，是昼夜系统触发特殊事件的经典案例。

### 动态天气与昼夜系统集成

更复杂的场景管理中，昼夜系统会作为天气系统的时间输入。例如夜间湿度参数升高，触发雾天概率增加；清晨6:00-8:00若当天天气为"晴"，则激活丁达尔光束（God Ray）效果的渲染Pass。两系统通过事件总线（Event Bus）通信，避免直接引用造成的循环依赖。

## 常见误区

### 误区一：天空盒切换等同于昼夜系统

许多初学者将"白天天空盒换夜晚天空盒"误认为是完整的昼夜系统实现。实际上，仅切换静态天空盒既无法产生日落的渐变过渡，也无法驱动场景内物体的实时阴影方向变化。真正的昼夜系统必须同步更新方向光旋转角度、环境光颜色以及场景中所有依赖时间状态的Gameplay组件，天空盒替换充其量是其中一个输出端。

### 误区二：时间倍率（TimeScale）越高越好

高时间倍率会导致昼夜循环过快，使玩家在一次游戏会话中经历数十个昼夜，昼夜变化失去仪式感和规律感。《我的世界》将默认日长设为20分钟，是经过大量用户测试后得出的平衡值：足够短让玩家感受到时间流逝的压迫感，又足够长让玩家在单个白天内完成一次完整的探索-回家循环。建议将 `TimeScale` 设计为可配置参数，由难度系统在运行时动态调节，而非写死在代码中。

### 误区三：昼夜系统只影响视觉表现

昼夜系统的光照变化会显著影响游戏性能指标。夜间场景中点光源（如火把、灯笼）的数量往往大幅增加，因为设计师会在夜间关卡中布置更密集的人工光源来保证可见性，这会导致Forward Rendering下的光照计算压力剧增。正确的做法是在昼夜系统的夜间阶段自动提高LOD剔除激进程度，或启用专门的夜间灯光光照贴图（Light Map）组来分摊实时光照计算压力。

## 知识关联

昼夜系统建立在**场景管理概述**中全局组件（Global Component）和场景生命周期（Scene Lifecycle）概念之上——时间管理器本身就是一个随场景加载而初始化、随场景卸载而销毁的全局单例（Singleton），其 `Tick` 更新频率和暂停行为必须遵循场景管理的统一调度规则。理解场景树中节点的挂载与激活顺序，有助于避免太阳Actor在场景其他光照组件初始化完成前就开始驱动参数变化而导致的一帧闪烁问题。

在技术延伸方向上，昼夜系统为**动态天气系统**和**季节系统**提供了时间轴基础设施：天气系统可以复用昼夜系统的曲线插值框架来管理云量、降水量等参数；季节系统则在昼夜循环之上叠加一个以现实天为单位的长周期，控制植被颜色和日照时长（冬季太阳仰角更低，日照时段缩短至游戏内10小时）。这两个系统在引擎架构层面都以昼夜系统的时间管理器作为统一时钟源。
