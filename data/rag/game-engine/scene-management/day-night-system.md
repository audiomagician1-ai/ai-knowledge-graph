# 昼夜系统

## 概述

昼夜系统（Day-Night System）是游戏引擎场景管理中负责模拟真实世界时间流逝的模块，通过驱动太阳/月亮位置、天空颜色、环境光照强度随时间连续变化，使游戏世界产生动态的早晨、正午、黄昏、夜晚等视觉效果。与静态场景不同，昼夜系统的存在让同一片地图在不同时段呈现截然不同的外观与氛围。

该系统最早在开放世界游戏中得到成熟应用。2002年发售的《Morrowind》已包含基本的太阳轨迹与光照变化，而2011年的《Skyrim》将其与NPC日程安排深度绑定，正式确立了昼夜系统作为Gameplay驱动层的地位。现代引擎如Unity和Unreal Engine 5均内置了昼夜系统的基础工具（Unity的Procedural Sky与UE5的Sky Atmosphere组件），开发者可在数小时内搭建可运行的原型。

昼夜系统之所以重要，不仅在于视觉表现，更在于它是连接"时间流逝"与"游戏逻辑"的桥梁——敌人巡逻频率、商店营业状态、特定任务触发窗口都可以绑定到系统暴露的时间参数上，以低成本大幅提升世界的真实感与可玩深度。

## 核心原理

### 太阳轨迹计算

昼夜系统的驱动核心是一个**游戏内时钟**，通常以"一天等于现实若干分钟"的比例运行（常见设定为现实20分钟 = 游戏1天）。时钟的当前值被归一化为 `t ∈ [0, 1)`，其中 `0` 代表午夜0:00，`0.25` 代表6:00日出，`0.5` 代表正午，`0.75` 代表18:00日落。

太阳方向光（Directional Light）的旋转角通过以下公式驱动：

```
θ_sun = (t - 0.25) × 360°
```

其中 `θ_sun = 0°` 时太阳位于地平线东方，`θ_sun = 90°` 时位于正上方。月亮通常偏移半圈：`θ_moon = θ_sun + 180°`。部分系统还加入**纬度偏移参数**，使极昼/极夜成为可能——只需将最低仰角设置为正值，太阳便永不落下。

### 光照参数随时间插值

仅旋转太阳不足以还原真实光感，光照强度与颜色必须随 `t` 变化。实现方式是为以下参数各自定义一条**时间-数值曲线（Gradient / AnimationCurve）**：

| 参数 | 正午典型值 | 黄昏典型值 | 夜晚典型值 |
|------|-----------|-----------|-----------|
| 方向光强度（lux） | 100,000 | 8,000 | 0（禁用） |
| 方向光色温（K） | 6,500 K（白） | 2,000 K（橙红） | — |
| 环境光（Ambient）强度 | 1.0 | 0.4 | 0.05 |
| 天空盒曝光值 | 1.2 | 0.6 | 0.02 |

Unity中使用 `Gradient` 资产；UE5中使用 `Curve Linear Color` 资产。系统每帧对当前 `t` 在曲线上采样并赋值，整个过程对CPU的开销极低，通常不超过0.1 ms/帧。

### 天空与大气渲染

现代昼夜系统通常挂接物理天空散射模型。UE5的**Sky Atmosphere**组件基于Hillaire 2020年提出的预计算大气散射方法，能自动根据太阳高度角计算瑞利散射（蓝天）和米氏散射（地平线橙红），无需手工绘制天空颜色曲线。Unity的**HDRI Sky**则是用一张EXR格式全景贴图替代物理散射，配合光照探针烘焙实时变化的反射光照。两种方案各有侧重：物理散射适合开放世界，HDRI适合对性能敏感的移动端项目。

## 实际应用

**开放世界RPG中的敌人生成逻辑**：《艾尔登法环》类型的游戏中，昼夜系统向Gameplay层暴露 `IsNight` 布尔值，夜晚时特定精英怪刷新、环境伤害（如暗影区域）激活。开发者只需订阅系统的 `OnTimeChanged` 事件，即可在无需修改生成器本身的情况下实现行为差异。

**NPC日程系统**：《星露谷物物语》将NPC的移动路径直接绑定到游戏内小时数，商店在游戏内9:00~17:00开放。这要求昼夜系统提供稳定的整点事件（`OnHourChanged`），而不仅仅是连续的浮点时间值。

**性能优化策略**：在光照变化较慢的时段（如正午前后1小时），可将每帧更新降为每5帧采样一次（即从60 Hz降至12 Hz），人眼无法察觉，但在低端机型上可节省约0.3 ms/帧的插值计算。

## 常见误区

**误区一：夜晚只需把方向光强度归零**
直接将方向光强度设为0而不同时降低环境光（Ambient）和天空盒亮度，会造成场景在"没有太阳"的情况下依然明亮如白昼。正确做法是所有光照参数协同变化，缺一不可。月光通常以0.1~1 lux强度的冷色调方向光单独模拟。

**误区二：昼夜速度可以随意设置**
游戏内1天=现实20分钟是大多数开放世界的黄金比例，短于10分钟会导致玩家频繁体验夜晚（可见度下降）而产生游戏体验中断感；长于60分钟则让时间变化失去对Gameplay的实际影响力。如果项目需要冻结时间推进（如剧情过场），务必提供`PauseTime()`接口而非直接设置 `t` 为固定值，否则会触发光照突变。

**误区三：昼夜系统仅影响画面**
忽视将时间变量开放给Gameplay层是最常见的设计失误。没有将 `currentHour` 暴露给AI、任务系统、天气系统的昼夜实现，只是一个"装饰性动画"，无法为游戏玩法增加实质深度。

## 知识关联

昼夜系统直接依赖**场景管理概述**中建立的场景树与组件更新循环——太阳位置的每帧刷新、光照探针的动态更新都必须挂载在场景的Tick流程中才能生效。掌握场景管理中对象生命周期（Awake/Start/Update）的执行顺序，有助于正确选择昼夜控制器的初始化时机，避免第一帧出现光照闪烁问题。

昼夜系统向上为**天气系统**提供太阳高度角与当前时段信息（例如"正在日落"时才可能触发晚霞云彩效果），向右为**NPC行为树**与**任务触发系统**提供时间事件订阅接口，是构建具有时间感的活跃世界的基础技术模块。
