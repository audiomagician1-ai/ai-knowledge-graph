# 世界原点偏移

## 概述

世界原点偏移（World Origin Shifting，有时称为 Floating Origin）是一种专门用于解决超大开放世界中浮点精度丢失问题的场景管理技术。其核心思想是：当玩家（或摄像机）距离世界坐标系原点 (0, 0, 0) 过远时，将整个世界的所有对象坐标统一向原点方向平移，使玩家始终保持在靠近原点的位置，从而保证浮点数运算的精度。

该技术最早在航空飞行模拟器中被广泛应用，因为飞行模拟需要覆盖数百公里甚至更大的地图范围。游戏行业中，《精英：危险》（Elite: Dangerous，2014年）和《无人深空》（No Man's Sky，2016年）等涉及星际尺度的游戏将该技术推广至主流引擎开发实践。虚幻引擎（Unreal Engine）在其 `AWorldSettings` 类中内置了对该功能的支持，可通过 `bEnableWorldOriginRebasing` 参数启用。

世界原点偏移之所以必要，是因为 IEEE 754 单精度浮点数（float）在距离原点超过约 **16,000 米（1.6 × 10⁴ 米）**时，相邻可表示数值之间的间距已超过 1 毫米；当距离达到 **100,000 米**时，精度间距超过 1 厘米；到达星际尺度时，精度损失会导致物体抖动、碰撞失效、动画撕裂等严重视觉和物理问题。

## 核心原理

### 浮点精度损失的数学根源

单精度浮点数（32位）使用 1 位符号、8 位指数、23 位尾数表示数值。在坐标值 x 处，最小可分辨间距（机器精度 ε）约为：

```
ε(x) ≈ x × 2^(-23) ≈ x × 1.19 × 10^(-7)
```

当 x = 10,000 米时，ε ≈ 0.00119 米（约 1.2 毫米），已接近人眼可察觉的抖动阈值。当 x = 1,000,000 米时，ε ≈ 0.119 米，物理碰撞计算误差达到分米级，完全不可接受。世界原点偏移正是通过将 x 本身压缩回接近 0 的小值来规避这一问题。

### 偏移触发与执行机制

偏移操作通常在玩家距离当前世界原点超过某一阈值时触发。以虚幻引擎为例，默认阈值为 **HALF_WORLD_MAX = 2,097,152 厘米（约 20,971 米）**。触发后，引擎执行以下操作序列：

1. 计算偏移向量 `ΔP = -PlayerPosition`（将玩家平移回原点）
2. 遍历场景中所有 `AActor`，对其 `RootComponent` 的世界坐标应用 `ΔP`
3. 对物理模拟体（Rigid Body、布料、流体）同步施加相同偏移
4. 更新导航网格（NavMesh）的基准坐标
5. 广播 `ApplyWorldOffset` 事件，通知所有子系统（粒子系统、音频监听器、流送管理器）

整个过程在单帧内完成，玩家感知不到任何位移。

### 双精度坐标的替代与配合方案

部分引擎（如 Unity 的 DOTS 架构）采用"双精度世界坐标（double-precision world position）+ 单精度本地坐标"的混合方案。在这种设计中，每个实体存储一个 `double3` 类型的绝对坐标，渲染时计算该实体相对于摄像机的**相对坐标**（camera-relative rendering），结果是一个精度安全的小数值 `float3` 再传入 GPU。世界原点偏移与这一方案并不互斥——双精度方案解决了 CPU 逻辑层的精度问题，而原点偏移额外解决了 GPU 着色器中 float 精度不足的问题，两者常结合使用。

## 实际应用

**开放世界 RPG 中的玩家移动**：在《异星探险家》类型的行星探索游戏中，地图半径可达 50 公里。当玩家从星球南极走向北极时，引擎每隔约 10 公里执行一次原点偏移，将玩家坐标重置到 (0, 0, z)，其他地形块的坐标相应减去该偏移量。地形流送系统依据新的原点重新判断哪些地形 Tile 需要加载，哪些可以卸载，偏移操作与流送操作天然融合。

**多人联机的服务器协调问题**：在多人游戏中，不同客户端的玩家可能触发不同频率的原点偏移，导致各客户端的本地坐标不再一致。解决方案是服务器维护唯一的**权威绝对坐标（Authoritative Absolute Position）**，各客户端保存自己的 `LocalOriginOffset` 向量，收到网络同步数据后先加上 `LocalOriginOffset` 再使用本地坐标显示。虚幻引擎的 `UNetDriver` 在处理 `RepMovement` 数据时内置了该补偿逻辑。

**粒子与特效系统的特殊处理**：GPU 粒子系统的模拟在着色器中以世界空间坐标进行，偏移时若不通知粒子系统，已发射的粒子会瞬间位移。Niagara 系统通过在偏移事件回调中调用 `FNiagaraWorldManager::OnWorldCleanup()` 并重建粒子模拟上下文来解决此问题。

## 常见误区

**误区一：认为偏移会导致玩家"感觉到"瞬移**。实际上，偏移向量精确等于玩家当前位置的负值，玩家自身的本地坐标在偏移后变为 (0, 0, 0)，摄像机相对于角色的位置完全不变，渲染结果与偏移前完全连续。只有在调试模式下打开世界坐标可视化，才能看到坐标数字的跳变。

**误区二：认为只需对玩家角色坐标做偏移即可**。这是最危险的实现错误。物理引擎（PhysX、Chaos）的内部状态（约束求解器中存储的绝对位置、速度积分起点）若未同步偏移，会在下一帧产生巨大的虚假力，导致物体飞出场景。正确实现必须调用物理引擎提供的专用偏移接口，如 PhysX 的 `PxScene::shiftOrigin(PxVec3 shift)`。

**误区三：认为使用 64 位双精度（double）可以完全替代世界原点偏移**。双精度浮点数在 ε(x) 的公式中将 23 位尾数扩展至 52 位，精度极大提升，但 GPU 着色器的 `float` 类型仍为 32 位，顶点坐标在传入渲染管线时依然面临精度丢失。因此，纯双精度方案只解决了 CPU 侧的逻辑问题，无法解决渲染侧的顶点抖动，依然需要 camera-relative rendering 或原点偏移配合。

## 知识关联

**前置概念——世界组合（World Composition）**：世界原点偏移与世界组合紧密配合。世界组合将超大地图切分为多个子关卡（Sub-Level），每个子关卡有自己的流送距离和加载状态。当执行原点偏移时，子关卡的流送边界阈值必须随偏移向量一起更新，否则流送判断会基于错误的相对距离加载或卸载不应该变化的地块。虚幻引擎的 `UWorldComposition::UpdateStreamingState()` 在收到 `OnWorldOriginLocationChanged` 事件后会自动重新计算所有子关卡的距离。

**延伸应用方向**：掌握世界原点偏移后，可进一步研究**行星级曲率渲染**（将大地坐标系投影到局部欧氏坐标系）、**相对坐标渲染管线（Camera-Relative Rendering）** 的着色器实现，以及在 ECS/DOTS 架构下如何高效批量处理 `ChunkEntity` 的坐标偏移以避免 CPU 缓存失效。这些技术共同构成了现代超大开放世界引擎的精度管理体系。
