# Agent工具创造

## 概述

Agent工具创造（Tool Creation by Agent）是指大语言模型驱动的Agent在执行任务时，不仅能够调用预定义的工具，还能够自主生成、测试和优化新工具代码的能力。与传统的Function Calling不同，工具创造要求Agent主动识别"当前工具库无法有效解决当前问题"这一缺口，并通过生成可执行代码来填补这一缺口，从而扩展自身的行动空间。

这一能力的研究集中出现在2023年前后。2023年8月，普林斯顿大学发布了**LATM（Large Language Models as Tool Makers）**框架，首次系统性地将工具创造任务分为"制造者（Maker）"和"使用者（User）"两个角色，前者负责生成Python函数，后者负责调用。同年，**CREATOR框架**进一步引入了四阶段流程（创建→决策→执行→精化），将工具创造与Agent反思机制深度耦合。这两个框架标志着Agent能力从"工具调用"向"工具制造"的范式转变。

在实践意义上，工具创造解决了预置工具库的覆盖边界问题。当任务涉及特定格式转换、领域计算或非标准API封装时，静态工具库必然存在盲区，而具备工具创造能力的Agent可以在零样本条件下生成对应工具，并在工具库中持久化存储以供复用，从而实现知识的累积式增长。

## 核心原理

### LATM框架：分角色协作生成

LATM的核心设计是**角色分离**：使用高能力模型（如GPT-4）作为Tool Maker，使用低成本模型（如GPT-3.5-turbo）作为Tool User。Maker接收任务描述和少量示例，生成一个带有完整文档字符串（docstring）的Python函数；User则仅接收函数签名和文档字符串，调用该工具完成具体实例。

Maker生成工具的提示词结构要求严格：必须包含函数功能描述、输入输出类型声明（Python类型注解）、至少一个可验证的测试用例。LATM在BigBench Hard测试集上的实验显示，对于适合工具化的任务类别，Tool Maker+Tool User的组合在推理成本上比纯GPT-4方案降低约**10倍**，同时保持相近的准确率。

### CREATOR框架：四阶段闭环

CREATOR将工具创造拆解为四个可追踪的阶段：

1. **创建（Creation）**：Agent根据任务生成一个通用Python工具函数，要求函数参数化程度高，不硬编码具体输入值。
2. **决策（Decision）**：Agent决定是否直接调用新工具，或需要进一步细化工具接口。
3. **执行（Execution）**：在沙箱环境中运行工具，捕获标准输出和异常信息（`stdout` + `traceback`）。
4. **精化（Rectification）**：若执行结果与预期不符，Agent读取错误信息并修改工具代码，形成循环。

CREATOR在MATH数据集上的实验表明，四阶段流程相较于直接代码生成（PAL方法）提升了约**8个百分点**的解题准确率，核心原因在于精化阶段消除了初版代码中的逻辑边界错误。

### 工具持久化与索引机制

工具创造的长期价值依赖于**工具库的持久化存储与检索**。新生成的工具以JSON或Python模块形式存入工具库，并为每个工具生成向量嵌入（embedding），以支持语义检索。当后续任务到来时，Agent首先用任务描述向量检索工具库（余弦相似度阈值通常设为0.75~0.85），若命中则直接复用，若未命中则触发工具创造流程。这一机制使工具库随使用量增长而增长，形成自扩展的能力基础。

工具存储条目的标准结构包含：工具名称、自然语言功能描述、函数签名、通过测试的示例输入输出对、生成时间戳。其中通过测试的示例输入输出对是判断工具可信赖程度的核心字段，未经测试验证的工具不应被直接入库。

### 代码生成与安全沙箱

工具创造必须在隔离环境中执行生成的代码，以防止恶意代码或意外副作用。常见实现方式包括：使用`RestrictedPython`库限制可调用的内置函数、在Docker容器内运行工具、设置执行超时（通常为5~30秒）。代码生成阶段还需过滤包含`os.system()`、`subprocess`、`eval()`等高风险调用的生成结果。

## 实际应用

**数学竞赛题求解**：CREATOR在GSM8K和MATH数据集上的实验中，Agent面对组合数学题时，自动创建了一个计算斯特林数（Stirling Numbers）的工具函数，该函数被后续9道相似题目复用，避免了重复推理的累积误差。

**数据分析流水线构建**：在企业级Agent应用中，当用户请求"对Excel文件按季度汇总并计算环比增长率"时，Agent创建一个接受`filepath`和`date_column`参数的Pandas封装工具，而非硬编码文件路径，使该工具可在同类任务中通用。

**API适配器自动生成**：当Agent需要调用未被预置工具覆盖的第三方REST API时，可根据API文档（OpenAPI Spec）自动生成封装函数，包含认证头设置、参数序列化和响应解析逻辑，整个过程无需人工介入。

## 常见误区

**误区一：工具创造等同于代码生成（Code Generation）**
代码生成针对单次任务输出可执行脚本，不考虑复用性；工具创造要求生成的函数必须参数化、有明确接口、通过测试验证并存入工具库。CREATOR框架明确区分了"针对实例的代码"与"可复用的工具函数"——前者将具体数值写入代码体，后者将其作为函数参数传入。将两者混淆会导致工具库中充斥着无法泛化复用的一次性脚本。

**误区二：工具创造可以绕过Function Calling基础**
工具创造依赖于Agent对函数签名、参数类型、调用约定的准确理解，这些能力直接来源于Function Calling的训练。若Agent无法可靠地解析`{"name": "tool_name", "arguments": {...}}`格式的调用结果，那么在精化阶段将无法正确将执行反馈映射回代码修改指令，导致闭环断裂。

**误区三：精化次数越多，工具质量越高**
实验数据显示，CREATOR中工具质量在第1~2次精化后趋于稳定，超过3次精化往往意味着工具设计本身存在根本性缺陷（如任务不适合工具化），继续精化会产生过拟合到当前错误信息的代码，反而降低泛化能力。因此，主流实现设置最大精化轮次为**3次**，超出则回退至直接推理策略。

## 知识关联

**前置概念——工具调用（Function Calling）**：工具创造的调用阶段（Tool User）完全依赖Function Calling机制，Agent需要能够解析工具的JSON Schema描述并正确格式化调用参数。若Function Calling的稳定性不足（工具调用成功率低于85%），工具创造的精化闭环将频繁因调用格式错误而中断，而非因逻辑错误触发有效的代码修改。

**前置概念——Agent反思机制**：CREATOR的精化阶段本质上是一种结构化的反思：Agent读取`traceback`错误信息，识别出代码中的具体错误行，生成修改建议，再重新执行。这与通用Agent反思（如Reflexion框架）的区别在于：工具创造的反思对象是函数代码体，反馈信号来自Python解释器，而非自然语言批评，因此反思的可操作性更强、噪声更低。

工具创造目前是Agent自主进化研究链条上的前沿节点，其延伸方向包括：多Agent协作分工创造工具（一个Agent专职测试工具质量）、工具创造与知识图谱结合（将工具能力映射为结构化知识节点）以及工具自动废弃与版本管理机制。
