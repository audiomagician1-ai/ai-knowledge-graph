# Agent工作流编排

## 概述

Agent工作流编排是指将多个LLM推理步骤、工具调用和数据处理节点组织成有向无环图（DAG），通过显式定义节点间的依赖关系、条件分支规则和并行执行策略，使复杂任务得以结构化、可追踪地自动完成的工程方法。与单次LLM调用不同，工作流编排的最小单元是"节点"（Node），每个节点封装一个原子操作（如调用GPT-4生成摘要、执行Python代码、查询向量数据库），边（Edge）则定义执行顺序与数据传递方式。

该领域的系统化工程实践兴起于2023年，LangGraph（2024年1月正式发布）、LlamaIndex Workflows和Temporal等框架的出现，将过去以线性Chain为主的编排模式扩展到支持循环、回溯和动态分支的有状态图结构。关键转折点在于：ReAct框架揭示了Agent需要在Reasoning和Acting之间多次迭代，而简单的链式调用无法表达这种循环依赖，必须引入图结构才能正确建模。

工作流编排的核心工程价值体现在三个可量化维度：通过并行执行将总延迟从串行累加压缩为关键路径长度（Critical Path Length）；通过检查点（Checkpoint）机制在节点级别实现断点续跑，避免因单节点失败导致整个长时任务重启；通过条件分支将不同输入路由至专门化的子图，避免通用提示在特定任务上的性能损耗。

## 核心原理

### DAG结构与状态传递

工作流的拓扑基础是有向无环图，节点之间的执行顺序由拓扑排序决定。形式化表示为 G = (V, E)，其中 V 是节点集合，E ⊆ V × V 是有向边集合，且图中不存在环路。每条边附带一个状态对象（State Schema），定义了从上游节点传递到下游节点的字段：例如，一个研究型Agent的State可能包含 `query: str`、`retrieved_docs: List[str]`、`draft: str`、`critique: str` 四个字段，每个节点只读取自己依赖的字段、写入自己产出的字段。

LangGraph采用TypedDict定义全局状态，节点函数签名为 `def node(state: State) -> dict`，返回值是对State的部分更新（Partial Update），框架负责合并。这种设计避免了节点间通过全局变量耦合，使每个节点可以独立测试。Reducer函数（如 `operator.add` 用于列表追加）解决了并行节点向同一字段写入时的冲突合并问题。

### 条件分支的路由机制

条件分支通过路由函数（Router Function）实现，该函数接受当前State，返回下一个节点的名称或名称列表。路由逻辑可以是：基于规则的字段判断（如 `if state["confidence"] < 0.7: return "clarification_node"`）、LLM分类调用（让模型输出结构化的路由标签）、或工具调用结果的解析。

在LangGraph中，条件边通过 `add_conditional_edges(source_node, router_fn, path_map)` 注册，其中 `path_map` 是路由函数返回值到目标节点名的映射字典。路由函数必须是纯函数（无副作用），以确保工作流在回放（Replay）时的确定性。一个典型的客服Agent路由可能将意图分类为 `["billing", "technical", "escalate"]` 三条路径，分别指向不同专业化的子图。

条件分支支持"扇出-扇入"（Fan-out / Fan-in）模式：一个节点可以通过条件边同时激活多个下游节点（扇出），这些并行节点完成后汇聚到一个聚合节点（扇入）。聚合节点的触发条件是所有上游并行节点均已写入其等待的字段，该机制等价于并发编程中的 `await Promise.all()`。

### 并行执行与关键路径优化

并行执行的前提是识别图中无数据依赖的节点集合（拓扑层级相同的节点）。以一个文章分析工作流为例：摘要生成节点、关键词提取节点、情感分析节点三者均只依赖原始文本，可以并行执行。若每个节点耗时约2秒，串行总耗时6秒，并行后关键路径长度降为2秒，加速比为3×。

框架层面，LangGraph通过Python的 `asyncio` 实现并行节点的异步调度，节点函数声明为 `async def` 即可参与并行调度。实际工程中需注意LLM API的并发限制（如OpenAI Tier 1账户的RPM上限），过度并行可能导致 429 Rate Limit 错误，需在编排层实现令牌桶（Token Bucket）限速。

循环（Loop）结构是工作流编排超越标准DAG的扩展，通过允许图中存在"受控回边"（Controlled Back-edge）实现。LangGraph用 `END` 特殊节点和条件边组合来表达"继续迭代"与"终止"的分支：当批评节点判断草稿质量不足时，路由函数返回写作节点的名称（回边），否则返回 `END`。为防止无限循环，工程实践中必须在State中维护 `iteration_count` 字段并设置上限（通常为5-10次）。

### 检查点与容错机制

检查点（Checkpoint）在每个节点执行完毕后将State序列化并持久化存储（支持SQLite、PostgreSQL、Redis等后端）。当工作流因网络超时、API限额或运行时异常中断时，可从最近一个成功检查点恢复执行，而非从头重跑。对于包含30个节点、总耗时可能超过10分钟的复杂研究Agent，检查点机制是生产可用性的基础要求。

## 实际应用

**多步骤代码审查Agent**：工作流包含并行的语法检查节点、安全漏洞扫描节点和风格规范节点（三者独立执行），汇聚后由综合评估节点生成最终报告。条件分支判断是否存在安全漏洞：若存在，路由至强制修复子图并阻断合并流程；若无，路由至可选优化建议节点。

**文献综述自动生成**：DAG结构包含文献检索节点（并行查询arXiv、Semantic Scholar两个来源）→ 去重与相关性过滤节点 → 并行的逐篇摘要节点（每篇文献独立调用LLM）→ 主题聚类节点 → 综述写作节点 → 自我批评与修订循环（最多3次迭代）。完整工作流节点数约15个，通过并行化将处理20篇文献的时间从约40分钟压缩到约8分钟。

**客户支持分诊系统**：入口路由节点调用意图分类模型，将工单分发到计费、技术、退款三个专业子图。每个子图内部有独立的上下文检索和回复生成流程。对于分类置信度低于0.6的工单，路由至人工审核节点而非自动处理，避免错误路由导致的客户体验损失。

## 常见误区

**误区一：将所有节点设计为串行以简化开发**。开发者常因并行调度的复杂性而放弃并行化，将可独立执行的节点强制串行排列。这不仅增加了无谓的延迟，还使工作流对单节点失败更加脆弱（串行链中任一节点失败即中断后续所有步骤）。正确做法是在设计阶段绘制显式的依赖图，仅对存在数据依赖的节点添加顺序边。

**误区二：在State中存储完整的LLM原始响应而非结构化抽取结果**。将数千Token的原始文本全部放入State，导致每次节点调用时传递的上下文过大，同时使下游节点的输入解析逻辑复杂化。正确做法是在每个节点内部完成结构化抽取（通过Pydantic模型或函数调用），State中只保留后续节点实际需要的字段，通常可将State体积减少70%-90%。

**误区三：混淆工作流编排与Agent自主规划的适用边界**。工作流编排适用于步骤可预先定义的任务（如文档处理流水线），此时显式DAG比让LLM动态规划更快、更可靠、更易调试。而对于步骤事先未知的探索性任务（如开放域研究），才需要让Agent自主决定下一步行动。在实际系统中，常见的错误是对结构化程度高的任务引入过度自主性，导致LLM在已知最优路径上仍产生不必要的规划偏差，增加Token消耗和错误率。

## 知识关联

Agent工作流编排建立在**提示链（Prompt Chaining）**的基础上：提示链定义了单一线性的LLM调用序列，而工作流编排将其泛化为支持分支和并行的图结构，解决了提示链无法表达条件逻辑和循环迭代的局限。

与**Agent规划与分解**的关系在于：规划模块（如LLM生成的任务分解列表）是工作流的动态输入，编排框架负责将规划结果映射为可执行的节点调度。规划决定"做什么"，编排决定"如何并发、如何容错地执行"。
