# Agent规划与分解

## 概述

Agent规划与分解（Agent Planning & Decomposition）是指让AI Agent将复杂的高层目标拆解为可执行的子任务序列，并通过动态调度这些子任务来逐步完成原始目标的技术体系。与单次推理不同，规划与分解赋予Agent跨多个时间步骤协调行动的能力，使其能处理需要数十乃至数百次工具调用才能完成的长程任务。

这一概念的技术基础可以追溯到1987年STRIPS（Stanford Research Institute Problem Solver）规划系统，现代版本则在2022年以后随大语言模型的崛起而重新焕发活力。2023年3月发布的ReAct论文（Reason + Act）和2023年5月发布的HuggingGPT（也称TaskMatrix.AI）分别从"推理交织行动"和"模型即工具"两个维度奠定了现代Agent规划架构的技术范式。Agent规划能力直接决定了系统能否突破单次上下文窗口的限制，处理真实业务中的端到端自动化场景。

## 核心原理

### 任务分解的三种主要策略

**自顶向下分解（Top-Down Decomposition）** 是最常见的策略：Agent首先生成一个高层执行计划，再对每个步骤递归细化。典型实现是HuggingGPT的四阶段流程——任务规划、模型选择、任务执行、结果汇总。该策略适用于目标明确、结构化程度高的任务，但面对中途出现新信息时调整成本较高。

**动态规划（Dynamic/Reactive Planning）** 允许Agent在执行过程中根据工具调用的反馈实时修改后续步骤。ReAct框架中每个（Thought, Action, Observation）三元组结束后，Agent都会更新对当前状态的理解，再决定下一步行动。这种方式的本质是将规划与执行交织在一个循环中，而非预先生成完整计划。

**并行分解（Parallel Decomposition）** 将相互独立的子任务识别出来并发执行，显著缩短完成时间。LangGraph框架中通过有向无环图（DAG）建模任务依赖关系，拓扑排序后入度为零的节点可以并发启动。一个典型例子是数据分析任务：同时调用网络搜索和数据库查询这两个无依赖关系的子任务，最后汇聚结果。

### 规划器的提示工程结构

现代LLM-based规划器通常要求模型输出结构化的JSON任务列表，每个任务包含四个字段：`id`（唯一标识）、`task`（自然语言描述）、`depends_on`（前置任务id数组）、`tool`（调用的工具名称）。`depends_on`字段是实现任务依赖建模的关键，当该数组为空时表示该任务可立即执行。这种格式由OpenAI的Function Calling规范和2023年底的Assistants API Tool Use标准所推广。

### 子任务粒度控制

分解的粒度过粗会导致单步操作复杂度过高而失败，粒度过细则会引发大量冗余的LLM调用和上下文膨胀。业界经验规则（来自LangChain文档）是：每个子任务应能在单次LLM调用加最多3次工具调用内完成。此外，子任务的描述需满足"可验证完成条件"原则——即存在明确的完成判断标准，如"返回包含revenue字段的JSON对象"而非模糊的"获取财务数据"。

### 计划可行性验证

部分高级实现在正式执行前增加一个"预规划验证"步骤：使用另一个LLM实例（或相同模型的第二次调用）检查计划中是否存在循环依赖、无法获得的工具调用、或逻辑矛盾。AutoGPT早期版本因缺少此步骤而频繁陷入无限规划循环，这一教训推动了后续框架对规划验证的重视。

## 实际应用

**软件开发Agent**：GitHub Copilot Workspace（2024年）将用户的自然语言需求分解为：1）理解代码库结构，2）定位相关文件，3）生成修改方案，4）编写测试用例，5）执行测试并修复报错，共5个有序子任务。每个步骤的输出直接成为下一步的上下文输入，体现了严格的线性依赖链。

**数据分析Agent**：在企业BI场景中，用户提问"今年Q3相比Q2利润下降的原因"可被分解为：并行执行Q2和Q3数据提取 → 计算差值并排序维度 → 针对差值最大的维度（如某产品线）深入查询 → 汇总叙述。整个流程涉及4-6次SQL工具调用和2-3次LLM推理调用。

**Web Research Agent**：Perplexity的深度研究功能（Deep Research，2024年初上线）将一个研究问题分解为多个子查询，分别检索不同角度的信息，再通过聚合步骤合成最终报告。其官方介绍显示该功能平均执行约30次搜索才完成一次深度研究任务。

## 常见误区

**误区一：将规划等同于思维链（CoT）**。CoT是让模型在单次推理中展开中间步骤，其输出是文字推理过程；而规划与分解产生的是可执行的任务调度结构，子任务需要真实调用外部工具或触发真实操作。混淆两者会导致开发者误以为让模型"想清楚步骤"就等于完成了Agent规划，但实际上缺少了执行层的反馈闭环。

**误区二：子任务越细越好**。过度分解会导致"规划幻觉"——模型生成大量细粒度步骤，但许多步骤在实际执行中因工具不支持或上下文缺失而失败。GPT-4在处理超过15个子任务的计划时，错误积累率显著上升（研究中称为"compounding error"问题）。合理的分解层次通常不超过3层递归深度。

**误区三：静态计划足以应对复杂任务**。在真实环境中，工具调用返回意外结果、资源不可用、中间步骤失败等情况频繁发生。纯静态计划（在执行前生成完整步骤序列并逐一执行）在这类场景中失败率极高。解决方案是引入Agent反思机制（Reflection），在检测到执行结果偏差时触发局部重新规划，这也是Agent规划天然连接反思机制的原因。

## 知识关联

**前置概念的支撑关系**：Agent规划是在感知-推理-行动循环基础上增加了跨循环的协调层——单个循环处理单个子任务，规划层负责编排多个循环的执行顺序。思维链（CoT）为规划器的推理步骤提供了内部展开方式，思维树（ToT）则为需要在多个规划方案中择优的场景（如BFS/DFS搜索最优执行路径）提供了搜索框架。

**通向后续概念的路径**：掌握单Agent规划后，多Agent协作系统将规划能力扩展到多个专业化Agent之间的任务分配，每个子Agent接收规划器分配的子任务后独立执行。Agent工作流编排（如LangGraph、Temporal）则是将规划与分解从动态生成转向预定义有向图结构，提升系统的可靠性和可调试性。Agent反思机制处理规划执行过程中的错误检测与局部修正，是使规划系统在真实环境中鲁棒运行的必要补充。
