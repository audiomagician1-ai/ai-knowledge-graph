# Agent规划与分解

## 概述

Agent规划与分解（Agent Planning & Decomposition）是指将一个复杂的高层任务拆解为一系列可执行的子任务序列，并为Agent提供明确行动路径的机制。它解决的核心问题是：当单步推理无法完成目标时，如何让Agent在多步骤、长时程场景中保持方向性和连贯性。典型代表如2023年发布的ReAct框架（Reasoning + Acting），将推理轨迹与行动步骤显式交织，使Agent每次行动前都能输出可审查的思考过程。

规划与分解的概念源自经典AI中的HTN（分层任务网络，Hierarchical Task Networks），最早在1990年代的SHOP和SHOP2规划器中得到系统实现。现代LLM驱动的Agent将HTN思想与语言模型的涌现能力结合，使规划过程从硬编码的符号逻辑转向基于自然语言的软性推导。2023年AutoGPT等自主Agent项目的爆发，将这一话题从学术研究推向工程实践前沿。

规划质量直接决定Agent在长任务中的成功率。斯坦福大学的实验数据表明，在需要10步以上操作的任务中，有显式规划步骤的Agent比直接行动的Agent任务完成率高约40%。因此，规划与分解不是可选优化项，而是长程Agent系统可用性的基本保障。

## 核心原理

### 任务分解的层次结构

任务分解遵循目标→子目标→行动三层结构。顶层目标（Goal）通常由用户自然语言描述，例如"撰写一份竞品分析报告"。Agent需将其分解为中层子目标（Subgoal），如"收集竞品数据"、"分析功能差异"、"生成报告框架"。每个子目标再对应具体的原子行动（Action），如调用搜索工具、读取网页内容、写入文件等。

分解的粒度是工程中的关键权衡：分解过粗导致单步行动仍然复杂，模型容易出错；分解过细则产生极长的规划序列，累积误差增大。经验法则是将每个叶节点行动控制在"单次工具调用可完成"的粒度，即每步行动应映射到一个明确的函数签名。

### 静态规划与动态规划

**静态规划**（Plan-then-Execute）在执行前一次性生成完整计划。优点是结构清晰、易于审查；缺点是计划对环境反馈不敏感，当第3步工具返回意外结果时，后续步骤可能全部失效。LangChain中的`PlanAndExecute` Agent采用此模式，适合环境稳定、任务确定性高的场景。

**动态规划**（Interleaved Planning）在每次行动后根据观察结果重新规划下一步。ReAct框架的核心公式正是如此：

```
Trajectory = (Thought₁, Action₁, Observation₁, Thought₂, Action₂, Observation₂, ...)
```

其中每个`Thoughtₜ`都可以修正原有计划。这种方式的信息利用率更高，但Token消耗也更大，每轮都需要携带完整历史上下文。

### 规划策略的三种范式

**前向规划（Forward Planning）**：从当前状态出发，逐步推导行动序列直至达成目标。大多数LLM Agent默认采用此方式，配合CoT提示可显著提升推理深度。

**逆向规划（Backward Planning / Goal Regression）**：从目标状态逆推所需的前置条件，直到与当前状态对接。适合约束明确的场景，例如代码生成中先确定函数签名再填充实现逻辑。

**分层规划（Hierarchical Planning）**：结合ToT的树状搜索思想，在不同抽象层次上并行维护多个候选子计划，通过评分函数选择最优分支。LLM+Planner架构（如LLM-MCTS）将蒙特卡洛树搜索嵌入LLM推理过程，在2023年Game benchmark上取得超越纯CoT方案约15%的得分提升。

## 实际应用

**代码生成Agent**：GitHub Copilot Workspace将用户的自然语言需求先分解为"需求理解→文件改动计划→逐文件实现"三阶段，用户可在执行前审查并修改分解结果，体现静态规划的可干预性优势。

**信息研究Agent**：GPT-Researcher将"写一篇关于X的研究报告"分解为：生成研究子问题列表→并行搜索各子问题→抓取并总结来源→综合写作。其并行子任务设计将研究耗时从串行的数十分钟压缩至约2分钟，依赖的关键技术是识别哪些子任务之间没有数据依赖，可并发执行。

**操作系统Agent**：微软UFO项目针对Windows GUI操作，将"帮我把桌面上所有PDF合并"分解为：定位文件→选择合并工具→执行操作→验证输出。每一步都需要感知当前屏幕状态后才能确定下一步行动，是典型的动态规划场景。

**工具调用顺序规划**：当Agent拥有十余个工具时（搜索、计算器、数据库查询、代码解释器等），规划模块还需决定工具调用拓扑顺序，避免因依赖关系混乱导致的空指针型错误（如尚未获取数据便尝试分析数据）。

## 常见误区

**误区一：规划越详细越好**
许多工程师倾向于在第一步就要求模型输出极度细化的计划，但这会导致两个问题：其一，初始信息不足时模型会幻构不存在的中间步骤；其二，过长的初始计划会占用大量上下文窗口，挤压后续行动的观察信息。正确做法是采用"粗粒度规划+细粒度执行"的渐进式分解，顶层计划只保留3-7个子目标。

**误区二：将规划与反思混为一谈**
规划是前向的——在执行前或执行中决定"下一步做什么"；反思（Reflection）是回溯性的——在遭遇失败后评估"哪里出了问题"。Reflexion框架明确区分了两者：规划模块输出行动序列，反思模块在Episode结束后生成文字总结并存入长期记忆，影响下一个Episode的规划输入，两者形成闭环但各司其职。混淆两者会导致Agent在执行中途陷入无休止的自我质疑循环，无法推进任务。

**误区三：认为LLM的规划能力等同于推理能力**
实验证明，同一个LLM在"解释计划"时表现优秀，在"严格执行计划并追踪状态"时却频繁出错。这是因为规划需要持久化状态追踪（已完成哪些步骤、哪些待完成），而LLM的无状态自回归特性并不天然支持此能力。工程上的解法是在Agent框架中引入独立的**计划状态追踪器**（Plan State Tracker），以结构化数据（如JSON）存储计划执行进度，而非依赖模型自行记忆。

## 知识关联

**与前置概念的衔接**：Agent循环（感知-推理-行动）是规划与分解的执行容器，每个子任务分解到原子行动层后，正好对应一次完整的感知-推理-行动循环。CoT为单步推理提供了深度，而规划与分解则在CoT之上提供了多步推理的宽度与结构；ToT的树搜索机制直接对应分层规划中的候选分支评估，是规划策略在搜索空间上的扩展形式。

**向后续概念的延伸**：多Agent协作系统依赖规划分解的输出——当某个子任务被识别为专项能力要求时，协作架构将其分派给专门的子Agent，任务分解图直接演变为Agent协作拓扑图。Agent工作流编排（如LangGraph、Prefect）本质上是将规划结果持久化为有向无环图（DAG），实现规划与执行的解耦和可观测性。Agent反思机制则在规划失败时介入，形成"规划→执行→反思→修正规划"的完整自主循环。
