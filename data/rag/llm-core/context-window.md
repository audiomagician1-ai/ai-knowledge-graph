# 上下文窗口与长文本

## 概述

上下文窗口（Context Window）是指大语言模型在单次推理中能够同时处理的最大 token 数量。这个限制源于 Transformer 的自注意力机制：每个 token 必须与窗口内所有其他 token 计算注意力分数，因此内存占用随序列长度呈 **O(n²)** 增长。早期 GPT-2 的上下文窗口仅为 1024 tokens，GPT-3 扩展至 4096 tokens，而 GPT-4 Turbo 和 Claude 2.1 分别支持 128K 和 200K tokens，几年间增长了约 200 倍。

这个限制的根本原因在于 KV Cache 的显存需求。在推理时，模型需要保存每一层每个 token 的 Key 和 Value 矩阵，显存消耗公式为：`KV Cache = 2 × L × H × d × n × dtype_bytes`，其中 L 为层数，H 为注意力头数，d 为每头维度，n 为序列长度。对于 LLaMA-2 70B 这样的模型，处理 4K tokens 时 KV Cache 已消耗约 8GB 显存，线性延伸到 128K 时直接爆显存。

长文本处理的重要性在于实际工程需求：法律合同审查动辄数万字，代码库分析需要跨文件理解调用关系，多轮对话需要维持完整历史。上下文窗口不足时，模型会丢失早期信息，产生"遗忘"效应，严重降低任务质量。

## 核心原理

### 位置编码的扩展瓶颈

原始 Transformer 使用绝对正弦位置编码，位置信息与序列长度强绑定，训练时未见过的位置索引会导致性能骤降。RoPE（旋转位置编码）通过将位置信息编码为复数旋转角度解决了部分问题：位置 m 和 n 之间的相对位置通过旋转角 `θ = m × base^(-2i/d)` 表达，其中 base 默认为 10000。然而当序列长度超过训练长度时，旋转角度的分布发生外推偏移，模型依然失效。

**RoPE 扩展的核心方案**是调整 base 值或对位置索引做线性缩放：
- **位置插值（Position Interpolation, PI）**：将新位置索引线性压缩到训练范围，例如将 8K 长度映射到 4K 训练窗口，代价是损失近距离分辨率。
- **YaRN（Yet another RoPE extensioN）**：对不同频率的旋转维度采用不同的缩放因子，高频维度（捕捉局部关系）保持不变，低频维度（捕捉全局关系）做插值，实现更平滑的长度泛化。LLaMA-3 系列使用了类似策略将上下文扩展至 128K。

### 注意力的"迷失在中间"现象

2023 年 Stanford 的研究论文《Lost in the Middle》实验表明，当上下文长度增加时，GPT-3.5-Turbo 和 Claude 在回答多文档问答时，对于**放置在上下文中间位置**的关键信息，准确率比放在开头或结尾低约 20 个百分点。这是因为注意力分数在长序列中存在"首因效应"和"近因效应"，中间部分的 token 获得的累积梯度更新更少。这一现象对 RAG 系统的文档排序策略有直接工程影响：最相关的片段应尽量放在上下文的首尾。

### 滑动窗口与分块策略

当文档长度超过模型最大窗口时，工程上常用**滑动窗口（Sliding Window）**策略：设定窗口大小 W 和步长 S，将文档切成重叠的片段依次处理。重叠区域（W - S tokens）确保跨片段的语义不被截断。但滑动窗口无法捕捉跨越整个窗口的长程依赖，例如文档第 1 页的定义与第 50 页引用之间的关联会丢失。

Mistral 7B 引入了固定大小为 4096 tokens 的**滑动窗口注意力（SWA）**，每个 token 只与前后 2048 tokens 内的 token 计算注意力，而非全序列。通过多层堆叠，第 k 层的感受野可达 `k × window_size`，在 32 层模型中理论感受野覆盖约 131K tokens，但实际全局信息传播依赖于层数累积，存在信息衰减。

## 实际应用

**代码助手的跨文件理解**：GitHub Copilot 和 Cursor 在处理大型代码库时面临典型的长文本问题。实际工程方案是将当前打开文件（约 2K tokens）、相关函数定义（从 AST 解析，约 1K tokens）和最近编辑历史（约 500 tokens）组合为约 4K tokens 的上下文，而非简单截断整个代码仓库。

**RAG 系统的分块设计**：在检索增强生成中，文档分块大小直接影响检索精度和上下文质量。实践中常见 chunk_size=512 tokens 配合 chunk_overlap=64 tokens，既能保持段落完整性，又避免单一片段语义稀释。多个检索片段拼入上下文时需注意总长度不超过模型限制，同时将最相关片段置于首尾以规避"迷失在中间"问题。

**长文档摘要的 Map-Reduce 模式**：对于超出窗口的超长文档（如完整书籍），常用 Map-Reduce 策略：先对每个分块独立生成摘要（Map阶段），再对所有摘要拼接后做二次压缩（Reduce阶段）。LangChain 的 `MapReduceDocumentsChain` 即实现了此模式。其缺点是两阶段处理丢失了段落间的细粒度交叉引用信息。

## 常见误区

**误区一：上下文越长，模型表现越好**
扩大上下文窗口不等于提升性能。实验表明，即使模型支持 128K tokens，在 100K tokens 处填充无关文本后，模型在关键信息提取任务上的准确率仍会下降。长上下文扩大了"干扰噪声"，模型需要更强的注意力聚焦能力才能从海量文本中定位关键信息。盲目塞入全部文档往往不如精心设计的 2K tokens RAG 上下文。

**误区二：微调可以直接扩展上下文窗口**
很多工程师认为在长文本数据上继续微调即可扩展窗口，但这忽略了 RoPE 位置编码的外推失效问题。如果没有同步调整 base 值或使用 YaRN/PI 等扩展方案，模型在训练长度之外的序列上位置编码会产生数值溢出，loss 不降反升。正确做法是先修改位置编码的缩放参数，再在长文本上做少量继续预训练（通常需要约 1000 步以上的微调才能稳定收敛）。

**误区三：KV Cache 压缩不影响输出质量**
H2O（Heavy-Hitter Oracle）等 KV Cache 压缩方案通过丢弃低注意力分数的历史 token 来节省显存，但"低注意力分数"并不等于"信息不重要"。某些 token 在当前位置注意力分数低，在未来生成步骤中可能变得关键（如合同中的日期条款）。KV Cache 激进压缩会导致随机性丢失，在需要精确引用原文的任务中表现明显下降。

## 知识关联

**前置基础**：理解上下文窗口的限制必须建立在 GPT 解码器架构之上。自回归解码过程中每生成一个 token 都需要重新读取整个历史序列，正是这个机制使 KV Cache 成为必要优化，同时也使得序列长度成为推理延迟的主要瓶颈。注意力权重矩阵 `softmax(QKᵀ/√d_k)` 的 n² 复杂度是窗口限制的数学根源。

**后续延伸**：上下文窗口的 O(n²) 瓶颈直接催生了稀疏注意力（Sparse Attention）的研究方向。Longformer 引入局部滑动窗口注意力与全局 token 注意力的混合机制，BigBird 用随机注意力+局部注意力+全局注意力的组合将复杂度降至 O(n)，这些方案都是针对长文本 O(n²) 问题的系统性解决方案，是上下文窗口扩展技术的直接演进路径。
