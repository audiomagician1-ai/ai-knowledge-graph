# 数据库锁机制

## 概述

数据库锁机制是并发控制的核心手段，通过对数据资源加锁防止多个事务同时修改同一数据时产生脏读、不可重复读或幻读等一致性问题。锁的本质是一种互斥协议：当事务A持有某资源的排他锁时，事务B必须等待A释放后才能访问该资源，从而将并发操作串行化为安全的执行序列。

锁机制的概念最早随1970年代关系型数据库的出现而系统化。IBM的System R项目（1974-1979年）在实现SQL原型时首次正式引入了两阶段锁协议（2PL），规定事务在释放任何锁之前必须先获取所有需要的锁，这一协议至今仍是MySQL InnoDB、PostgreSQL等主流数据库引擎的并发控制基础。

理解锁机制对AI工程师尤为重要：在特征工程流水线、模型训练数据预处理或在线推理服务写回数据库时，错误的锁策略会导致死锁（Deadlock）或严重的锁等待超时，直接影响高并发推理服务的吞吐量。例如，一个每秒处理10万次推理请求的服务，若每次写回预测结果都触发表级锁，系统将几乎完全串行化，QPS可能骤降至原来的1/100。

## 核心原理

### 锁的粒度：行锁与表锁

锁粒度描述了锁保护数据范围的大小，是并发性与系统开销之间的权衡。

**表锁（Table Lock）** 一次锁定整张表，实现简单，MySQL的MyISAM引擎默认使用表锁。执行`LOCK TABLES t WRITE`时，所有其他事务对该表的读写请求全部阻塞。表锁的内存开销极低（每张表仅需一个锁结构），适合批量导入或全表更新场景，但并发写入性能极差。

**行锁（Row Lock）** 仅锁定被操作的具体行，InnoDB引擎通过在索引记录上附加锁标记实现行锁。关键细节：**InnoDB的行锁是加在索引上的，而非加在行记录本身**。若查询条件列没有索引，InnoDB将退化为全表扫描并锁定所有行，效果等同于表锁。行锁支持高并发，但每个锁需要独立的内存结构，锁定100万行会消耗显著更多内存。

**间隙锁（Gap Lock）** 是InnoDB在可重复读隔离级别下独有的锁类型，锁定索引记录之间的"间隙"而非记录本身。例如，若索引中存在值10和20，间隙锁可锁定(10, 20)这个开区间，阻止其他事务在此范围内插入新行，专门用于防止幻读。间隙锁与行锁结合形成**临键锁（Next-Key Lock）**，锁定范围变为左开右闭区间，如(10, 20]。

### 悲观锁与乐观锁

**悲观锁（Pessimistic Lock）** 假设并发冲突频繁发生，每次访问数据时都先加锁再操作。在SQL中通过`SELECT ... FOR UPDATE`实现，该语句会在查询结果集上加排他行锁，其他事务的读写操作均被阻塞直到持锁事务提交。悲观锁适用于银行转账、库存扣减等写冲突概率高的场景，但持锁时间过长会直接导致系统吞吐量下降。

**乐观锁（Optimistic Lock）** 假设冲突罕见，不在读取时加锁，而是在提交更新时检测是否有其他事务修改了同一数据。最常见的实现是**版本号（Version）机制**：

```sql
-- 读取时记录版本号
SELECT id, stock, version FROM inventory WHERE id = 1;

-- 更新时校验版本号是否未变
UPDATE inventory 
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 5;  -- 若version已被他人改为6，此语句影响行数=0，表示冲突
```

若`UPDATE`影响行数为0，说明发生冲突，业务层需重试。乐观锁适用于读多写少场景，如AI系统中的模型版本管理表。另一种实现是使用时间戳列替代版本号，语义相同。

### 两阶段锁协议（2PL）与死锁

两阶段锁协议规定：事务分为**加锁阶段**（只获取锁，不释放）和**解锁阶段**（只释放锁，不获取）。这保证了串行化可调度性，但也是死锁产生的根源。

死锁的经典场景：事务A先锁定行R1再请求行R2，事务B先锁定行R2再请求行R1，两者互相等待形成循环依赖。InnoDB通过**等待图（Wait-for Graph）** 检测死锁，当检测到有向环时，选择回滚代价较小的事务（通常是修改行数较少的那个）并返回错误码1213。可通过`SHOW ENGINE INNODB STATUS`查看最近一次死锁的详细信息。

## 实际应用

**AI模型特征写入场景**：在实时推荐系统中，用户行为特征需要高频写入特征数据库。若使用悲观锁（FOR UPDATE），大量并发请求会形成锁队列，P99延迟可能超过1秒。更合理的方案是使用乐观锁+版本号，配合`INSERT ... ON DUPLICATE KEY UPDATE`语法批量合并写入，冲突时自动重试，将锁竞争转化为应用层重试逻辑。

**分布式训练数据分片**：多个训练Worker并发读取数据集时，为防止重复读取同一批次数据，可对任务表中的`status`字段使用乐观锁：Worker读取`status='pending'`的记录后，通过`UPDATE ... WHERE status='pending' AND version=N`将其标记为`processing`，若失败则重新选取其他记录，实现无锁化的任务分发。

**模型A/B测试配置更新**：配置表写入频率低但读取极高，适合使用意向锁（Intention Lock）优化。InnoDB在行锁基础上自动为表添加意向行锁（IX），使得全表操作（如Schema变更）能快速检测到是否存在行级锁冲突，而不必逐行扫描，将O(n)的冲突检测降为O(1)。

## 常见误区

**误区一：乐观锁"没有锁"所以性能一定优于悲观锁**。乐观锁在高冲突场景下会产生大量失败重试，每次重试都是一次完整的读-校验-写操作，总体数据库访问次数可能远超悲观锁。在冲突率超过20%的写密集场景中，实测悲观锁的吞吐量往往更高。选择依据是**实际冲突概率**，而非"直觉上哪个更轻量"。

**误区二：`SELECT ... FOR UPDATE`一定会锁定查询结果集中的所有行**。当WHERE条件列无索引时，InnoDB无法精确定位行，会锁定全表所有行的Next-Key Lock，而不仅仅是结果集中的行。这是行锁意外退化为表锁的典型原因，在生产环境中曾多次引发严重的锁等待事故。务必用`EXPLAIN`确认查询走索引后再使用FOR UPDATE。

**误区三：加锁即可保证数据一致性，事务隔离级别无关紧要**。锁控制的是并发访问的时序，而隔离级别决定了事务能"看到"哪个版本的数据。即使对行加了排他锁，在读未提交（Read Uncommitted）隔离级别下，其他事务仍可能读到未提交的修改（脏读）。锁机制与隔离级别必须协同配置才能实现正确的并发语义。

## 知识关联

**与事务ACID的关系**：锁机制是实现ACID中**隔离性（Isolation）** 的主要手段之一。不同隔离级别对应不同的锁策略：读提交（RC）使用语句级别的快照读+行锁，可重复读（RR）额外引入间隙锁来防止幻读。理解事务的四个隔离级别（RU/RC/RR/Serializable）是正确配置锁行为的前提。

**与索引原理的关系**：如前所述，InnoDB行锁依附于索引实现。聚簇索引（Clustered Index）上的行锁直接锁定数据行，而二级索引（Secondary Index）上的行锁需要回表后再在聚簇索引上加锁，因此一次加锁操作可能涉及两棵B+树上的锁。索引设计不合理会直接导致锁粒度升级，这是性能调优必须同时审查SQL和锁行为的原因。

**向MVCC的延伸**：传统锁机制在读写并发时仍会相互阻塞（读需要等写锁释放）。MVCC（多版本并发控制）通过为每行数据维护多个历史版本（Undo Log链），使读操作可以访问事务开始时的一致性快照而无需加锁，从根本上解决了读写互斥问题。InnoDB在RR级别下将锁机制与MVCC结合使用：快照读走MVCC，当前读（FOR UPDATE）走锁。
