# 数据库备份恢复

## 概述

数据库备份恢复是指通过系统化手段将数据库在某一时间点的状态完整保存，并在数据丢失、损坏或灾难事件发生后将数据库还原至可用状态的技术体系。它直接决定了系统的恢复时间目标（RTO，Recovery Time Objective）和恢复点目标（RPO，Recovery Point Objective）——这两个指标量化了业务对数据丢失和停机时间的容忍上限。

备份恢复技术最早伴随磁带存储的普及于1970年代形成规范。PostgreSQL在8.0版本（2005年）引入了连续归档（Continuous Archiving）机制，将WAL日志归档与基础备份结合，实现了任意时间点恢复（PITR）。MySQL的InnoDB引擎则通过redo log和binlog的组合支持类似能力。现代AI工程中，训练数据集、模型权重和特征存储数据库同样依赖这套机制防止因硬件故障或误操作导致不可逆损失。

备份策略的选择直接影响存储成本、备份窗口时长和最终恢复速度。一个设计不当的备份策略可能导致在数据中心故障时需要数小时甚至数天才能完成恢复，而精心设计的策略可以将RPO压缩至秒级，将RTO压缩至分钟级。

## 核心原理

### 全量备份（Full Backup）

全量备份将数据库在某一时刻的所有数据页完整复制，生成一个自包含的快照文件。PostgreSQL的 `pg_basebackup` 工具执行全量备份时，会先触发检查点（checkpoint）将所有脏页刷盘，再逐文件复制数据目录。对于一个100GB的数据库，全量备份通常耗时15至60分钟，具体取决于磁盘I/O性能。全量备份的优势是恢复流程最简单——只需将备份文件还原到目标目录即可；缺点是存储空间消耗大、备份窗口长，通常每周执行一次而非每日。

### 增量备份（Incremental Backup）与差异备份（Differential Backup）

增量备份仅保存自**上次备份（无论全量还是增量）**以来发生变化的数据块。差异备份则保存自**上次全量备份**以来的所有变更。假设周日做全量备份，周一至周六每日做增量备份：恢复周三数据需要依次应用全量 + 周一增量 + 周二增量 + 周三增量，共4份文件；若使用差异备份，则只需全量 + 周三差异，共2份文件，但周三的差异文件体积大于周三的增量文件。Percona XtraBackup支持MySQL的增量热备份，通过跟踪InnoDB的LSN（Log Sequence Number）确定哪些页面已被修改。

### WAL（Write-Ahead Logging）与连续归档

WAL是数据库保证ACID特性的底层机制：所有数据修改在写入实际数据文件之前，必须先将变更记录写入WAL日志文件。PostgreSQL的WAL文件默认大小为16MB一段（segment），文件命名格式为 `000000010000000000000001` 这样的24位十六进制时间线编号。通过将WAL段文件持续归档至对象存储（如S3），再配合全量基础备份，即可实现PITR——恢复命令如下：

```
recovery_target_time = '2024-03-15 14:30:00'
restore_command = 'aws s3 cp s3://backup-bucket/wal/%f %p'
```

这意味着即使最近一次全量备份是昨天，也能将数据库精确还原到今天某一分钟的状态，RPO可达到秒级。

### RTO 与 RPO 的量化设计

在AI工程的特征数据库场景中，RPO和RTO的计算公式为：

- **RPO** = 当前时间 − 最近一次成功备份的时间戳（或WAL归档延迟）
- **RTO** = 备份传输时间 + 数据还原时间 + 数据库启动与验证时间

若要求RPO ≤ 5分钟，则WAL归档频率必须 ≤ 5分钟一次。若要求RTO ≤ 30分钟，则全量备份文件大小须控制在可在该时间内完成网络传输和磁盘写入的范围内——对于千兆内网，约125MB/s的传输速率意味着全量备份不宜超过200GB。

## 实际应用

**AI训练数据库的每日备份方案**：在存储模型训练元数据和实验日志的PostgreSQL数据库上，通常配置每周日凌晨2点执行 `pg_basebackup` 全量备份，每5分钟通过 `archive_command` 将WAL段文件上传至S3。当某次错误的数据预处理脚本误删了特征表中30%的行时，DBA可以执行PITR，将数据库恢复至删除操作发生前10秒的状态，而无需从业务逻辑层重建数据。

**MySQL binlog用于逻辑备份恢复**：`mysqldump` 生成的是逻辑备份——SQL语句的集合，而非数据页的物理副本。配合MySQL binlog（`--log-bin` 开启），可以实现类似PITR的效果：先还原最近的 `mysqldump` 文件，再用 `mysqlbinlog --start-datetime` 和 `--stop-datetime` 参数重放指定时间段内的binlog事件。逻辑备份的恢复速度远慢于物理备份——一个50GB的数据库，`mysqldump` 恢复可能需要2小时，而Percona XtraBackup的物理备份恢复只需20分钟。

**灾难恢复演练**：合规标准（如SOC 2）要求每季度执行一次备份可用性验证，即在隔离环境中实际执行完整的恢复流程，验证备份文件的完整性（checksum）和数据库启动后的逻辑一致性。仅依赖备份程序的日志报告而不做实际恢复测试，是导致灾难时恢复失败的最常见原因之一。

## 常见误区

**误区1：WAL归档等同于备份**。WAL归档文件不能独立用于恢复——必须有对应的全量基础备份作为起点，WAL日志只是用于"追赶"基础备份之后的变更。若基础备份损坏或丢失，再完整的WAL归档也无法恢复数据库。正确做法是将基础备份与对应时间段的WAL文件绑定存储，并验证两者的LSN连续性。

**误区2：备份频率越高，数据越安全**。备份频率影响的是RPO，而不直接影响恢复成功率。若所有备份文件都存储在与生产库相同的物理机器或同一可用区，一次机房级故障会同时摧毁生产数据和所有备份——这违反了"3-2-1备份原则"（3份副本、2种介质、1份异地存储）。AI工程中正确做法是将备份文件同步至不同云区域（如从 `ap-northeast-1` 同步至 `us-east-1`）。

**误区3：逻辑备份可以替代物理备份**。`mysqldump` 或 `pg_dump` 生成的逻辑备份不包含时间点恢复能力，无法恢复到备份文件创建时刻之后的任意状态。对于需要RPO ≤ 1小时的生产系统，必须同时配置物理备份加WAL/binlog归档，逻辑备份只适合作为结构迁移或小型数据集的补充手段。

## 知识关联

**与数据库复制的关系**：数据库复制（Replication）通过将数据实时同步到从节点提供高可用性，但它不能替代备份——若主库执行了误操作（如 `DROP TABLE`），该操作会立即同步到所有从库，复制无法保护免受逻辑错误的影响。备份与复制互补：复制解决硬件单点故障的RTO问题，WAL归档解决逻辑错误的RPO问题。PostgreSQL的流复制本身就基于WAL传输机制，因此理解WAL的段文件结构和LSN序号是同时掌握两者的基础。

**与数据库基本概念的衔接**：ACID中的持久性（Durability）保证正是由WAL实现的——数据库承诺"事务提交后数据不丢失"，其底层依赖是WAL日志在提交前必须成功写入磁盘。备份恢复策略的设计须从业务的RPO/RTO需求出发，反推所需的备份类型组合和归档频率，这一设计过程将数据库的物理存储结构、事务日志机制和运维策略联结为一个可量化的工程方案。
