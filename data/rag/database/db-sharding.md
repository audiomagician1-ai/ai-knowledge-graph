# 分库分表

## 概述

分库分表（Database Sharding）是一种将单一大型数据库水平拆分为多个较小数据库或数据表的架构技术。当单表数据量超过千万级别（通常以 1000 万行作为警戒线，超过 5000 万行则几乎必须拆分）或单库 QPS 超过数千时，B+ 树索引深度增加导致查询性能急剧下降，分库分表通过将数据分散到多个物理节点来解决这一瓶颈。

该技术在 2000 年代中期随着互联网业务爆发式增长而成熟，淘宝的 TDDL（Taobao Distributed Data Layer）和 Facebook 的 MySQL 分片实践是标志性案例。2016 年前后，随着 ShardingSphere、MyCat 等中间件框架的普及，分库分表从大厂专属技术走向工程化标准实践。

在 AI 工程场景中，分库分表直接影响特征存储、模型训练样本库和推理日志库的访问效率。一个千亿级行为日志表若不进行分片，单次全表扫描可能耗时数小时；合理分库后，同等查询可缩短至分钟级，这对在线学习系统的数据管道延迟有决定性影响。

## 核心原理

### 水平拆分与垂直拆分的区别

**垂直分库**是按业务模块将不同的表拆分到不同数据库，例如将用户信息库、订单库、商品库分离到三个独立数据库实例。**垂直分表**则是将一张宽表按列拆分，例如将 `user` 表中的热字段（id、name、status）和冷字段（简介、头像 URL、扩展 JSON）分到两张表，减少单行数据宽度，提升热字段的缓存命中率。

**水平分库**和**水平分表**才是通常所说"分库分表"的核心，二者均是将同一张表的数据按行分散：水平分库将数据分布到不同数据库实例（不同机器），水平分表则在同一实例内创建多张结构相同的子表（如 `order_0` 到 `order_255`）。

### 分片键的选择与路由算法

分片键（Shard Key）的选择决定数据分布是否均匀，是分库分表设计中最关键的决策。常见路由策略包括：

- **哈希取模**：`shard_id = hash(user_id) % N`，其中 N 为分片数。优点是分布均匀，缺点是扩容时需要迁移大量数据（扩容比例越大，迁移量越大）。
- **一致性哈希**：将节点映射到 0~2³²-1 的哈希环上，新增节点时平均只需迁移约 `1/N` 比例的数据，显著降低扩容代价。
- **范围分片**：按 `user_id` 区间（如 1~1000000 归 shard0）或时间范围分片，查询连续范围时效率高，但可能导致热点集中（如最新时间段的写入全部打到同一分片）。

选择分片键时需满足三个条件：①该字段在绝大多数查询中作为 WHERE 条件出现；②字段值分布足够离散，避免数据倾斜；③字段值一经写入不允许更新，因为修改分片键意味着数据需要跨分片迁移。

### 跨分片查询与分布式事务的代价

分库分表最显著的副作用是**跨分片查询**的性能损耗。当 SQL 包含非分片键的 `ORDER BY`、`GROUP BY` 或 `JOIN` 操作时，中间件必须向所有分片下发查询，再在内存中做归并排序（Merge Sort），查询复杂度从 O(log N) 退化为 O(k × log(n/k))，其中 k 为分片数，n 为总数据量。

分布式事务方面，跨分片的写操作无法直接使用数据库本地事务，必须借助 2PC（两阶段提交）或柔性事务（如 TCC、Saga 模式）。2PC 在协调者宕机时存在阻塞问题，而 TCC 要求业务代码实现 Try、Confirm、Cancel 三个接口，开发成本翻倍。因此实践中强烈建议将需要强一致性的操作数据路由到同一分片（即通过合理选择分片键规避跨片事务）。

## 实际应用

**AI 特征存储分片**：以推荐系统的用户实时特征表为例，按 `user_id` 的哈希值分为 256 张子表（`feature_0` ~ `feature_255`），每张子表约存储 100 万用户特征向量。在线推理服务根据请求中的 `user_id` 计算 `user_id % 256` 直接定位到目标子表，将特征读取 P99 延迟控制在 5ms 以内。

**训练样本库的时间范围分片**：模型训练通常只需要最近 N 天的样本，使用按天范围分片（每天一张分区表，如 `sample_20240315`）可以将冷数据直接归档删除，而无需全表扫描删除，`DROP TABLE sample_20240101` 的速度远快于 `DELETE FROM sample WHERE date = '2024-01-01'`。

**MyCat 中间件配置示例**：在 MyCat 的 `rule.xml` 中配置哈希分片规则，指定 `columns` 为分片键字段名，`algorithm` 为 `murmur`（MurmurHash 算法），并在 `schema.xml` 中配置 dataHost 连接到 8 个物理 MySQL 节点，每节点承载 32 张子表，共 256 分片。

## 常见误区

**误区一：分片数越多性能越好。** 很多工程师认为分片数翻倍则性能翻倍，实则不然。分片数增加会导致跨分片查询的归并开销线性增长，同时中间件连接池的连接数也等比增加（256 分片 × 每片 10 连接 = 2560 个连接），可能导致连接耗尽。通常建议分片数为 2 的幂次（便于后续成倍扩容），初始规模设置为 32 或 64 即可满足大多数场景。

**误区二：分库分表后可以不建索引。** 部分开发者认为数据已经分散到小表，全表扫描代价降低，索引可以省略。实际上单分片的数据量仍可能达到千万级，B+ 树索引仍是必要的；更危险的是，缺少索引的分片查询会触发中间件对所有分片的全表扫描广播，放大效果与分片数成正比，性能比未分片时更差。

**误区三：全局唯一 ID 可以继续用数据库自增主键。** 分库分表后各分片独立维护自增序列，`AUTO_INCREMENT` 会在不同分片上产生相同的 ID 值，导致 ID 冲突。必须改用分布式 ID 方案，如 Twitter 的 Snowflake 算法（64 位 = 1 位符号位 + 41 位时间戳毫秒 + 10 位机器 ID + 12 位序列号，支持同一毫秒内单机生成 4096 个不重复 ID）或 UUID。

## 知识关联

**前置依赖——数据库复制**：理解分库分表需要先掌握 MySQL 主从复制机制，因为每个分片节点自身通常也是一个主从复制集群（一主两从），分库是在复制基础上进一步做水平扩展。复制解决的是读扩展和高可用问题，而分库分表解决的是写瓶颈和存储容量问题，两者在实际系统中往往同时存在。

**后续概念——数据分区策略**：分库分表属于应用层（中间件层）的手动分片方案，而 MySQL 8.0 的 PARTITION BY 子句和 PostgreSQL 的声明式分区提供了数据库原生的分区能力。理解了分库分表的分片键选择和路由原理之后，学习数据分区策略时能够直接类比 `RANGE PARTITION`、`HASH PARTITION` 等分区类型，并理解原生分区在跨分片事务和全局索引方面相比中间件方案的优势与限制。
