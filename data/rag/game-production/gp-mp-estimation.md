# 工时估算

## 概述

工时估算（Effort Estimation）是指在游戏制作里程碑规划中，对完成特定任务所需人力工时进行量化预测的系统性方法。工时的单位通常以"人天"（person-day）或"人时"（person-hour）表达，例如"制作一个Boss战关卡的美术资产需要8人天"。准确的工时估算直接决定了里程碑节点能否按期交付，是排期表（Schedule）的数值基础。

工时估算方法在软件工程领域有较长的发展历史。1958年，NASA在北极星导弹项目中首次系统性地应用了PERT（Program Evaluation and Review Technique）方法，将概率统计引入工程工时预测。游戏行业在2000年代后期逐渐从传统软件项目管理中借鉴了这套方法体系，并结合游戏开发的迭代性特征加以调整。

在游戏制作的里程碑规划语境下，工时估算解决的核心问题是：WBS分解出的每个工作包（Work Package）究竟要消耗多少人力资源。若估算偏低，团队将在Alpha或Beta阶段出现严重的加班与功能削减；若估算偏高，则会导致发行商或内部预算审批受阻。国际游戏开发者协会（IGDA）2019年调查显示，约67%的延期项目将"前期工时估算不足"列为主要原因之一。

## 核心原理

### 类比估算法（Analogous Estimation）

类比估算法依赖历史项目数据，将当前任务与过去已完成的相似任务进行对标。例如，某工作室曾花费12人天制作一个包含4个房间的地牢关卡，则估算一个结构相近的新关卡时，可以直接以12人天为基准，再根据复杂度差异乘以调整系数（通常为0.8–1.5之间）。

这一方法的精度高度依赖历史数据的质量。如果工作室维护了结构化的"完工任务库"，类比估算误差可控制在±20%以内；若无历史记录则误差可超过50%。类比估算适用于概念阶段（Concept Phase）的快速粗估，因为此时WBS细节尚不完整，无法支撑更精细的方法。

### 参数估算法（Parametric Estimation）

参数估算法建立数学模型，通过已知参数计算工时。在游戏美术生产中，常见的参数模型形如：

**工时 = 资产数量 × 单资产基准工时 × 复杂度系数**

例如，建模一个环境道具的基准工时为0.5人天，复杂度系数分三档：简单道具×1.0、中等道具×1.8、高精度道具×3.5。若某关卡需要20个中等道具，则参数估算结果为：20 × 0.5 × 1.8 = **18人天**。

参数模型需要定期用实际完工数据进行校准（Re-calibration）。通常在每个里程碑结束后，项目经理会比对估算值与实际值，更新系数表，使下一阶段的预测更加准确。

### 三点估算法与PERT公式

三点估算法承认工时预测存在不确定性，要求估算者给出三个数值：

- **O（Optimistic，乐观值）**：一切顺利时的最短工时
- **M（Most Likely，最可能值）**：正常情况下的预期工时
- **P（Pessimistic，悲观值）**：出现障碍时的最长工时

PERT方法在此基础上计算**加权期望值（E）**和**标准差（σ）**：

$$E = \frac{O + 4M + P}{6}$$

$$\sigma = \frac{P - O}{6}$$

以一个游戏关卡的脚本编写任务为例：O = 3人天，M = 5人天，P = 11人天，则：

$$E = \frac{3 + 4×5 + 11}{6} = \frac{34}{6} ≈ 5.67\ \text{人天}$$

$$\sigma = \frac{11 - 3}{6} ≈ 1.33\ \text{人天}$$

最可能值M被赋予4倍权重，反映了正态分布中众数附近概率较高的统计规律。标准差σ可用于构建置信区间：E ± σ覆盖约68%的概率，E ± 2σ覆盖约95%的概率。里程碑规划中常用E + 2σ作为**缓冲工时**（Buffer）的依据。

## 实际应用

**角色动画生产排期**：某动作游戏的动画团队需要为主角制作32套战斗动作。项目经理使用参数估算法，设定基准工时为0.75人天/套，最终Boss专属动作设复杂度系数2.0，普通连招系数1.0。总计估算：28套×0.75×1.0 + 4套×0.75×2.0 = 21 + 6 = **27人天**，据此将动画制作里程碑设定在Alpha前5周。

**关卡设计预生产阶段**：一个独立工作室使用三点估算对首个Demo关卡进行预测：O = 10天，M = 18天，P = 30天，PERT期望值E = (10 + 72 + 30)/6 = 18.67天。团队取E + σ = 18.67 + 3.33 ≈ **22天**作为里程碑承诺日期，成功留出了约15%的缓冲。

**多平台移植工时对比**：类比估算在移植项目中尤为实用。某工作室将PC版游戏移植至主机平台时，参考同类型游戏移植的历史数据——过去3个项目分别耗费85、110、95人天——取均值97人天作为当前项目的初始估算基准。

## 常见误区

**误区一：把工时估算等同于日历时长**。工时（Effort）衡量的是人力投入总量，日历时长（Duration）则取决于人员分配和并行工作安排。一项12人天的任务，如果由2名美术并行完成，日历时长约为6天，而非12天。将两者混淆会直接导致里程碑节点错误。

**误区二：PERT的M（最可能值）等于团队的"希望值"**。实践中，估算者常倾向于给出过于乐观的M值，导致PERT计算结果系统性偏低。正确做法是要求估算者在给出M时，明确描述"正常工作条件"的假设前提（如：无外部依赖阻塞、无临时人员调动），使M锚定在客观情境而非主观期望上。

**误区三：估算精度随WBS层级的细化而无限提升**。将任务分解到过细的粒度（例如细化到每个UI图标的工时）会产生大量估算开销，且每个小任务的单独误差率并不低于粗粒度任务。通常WBS分解到"工作包可由1–2人在2–10天内完成"的粒度时，三点估算的边际精度回报已趋于饱和。

## 知识关联

工时估算的输入直接来源于**WBS工作分解**的结果——每个被识别出的工作包是工时估算的最小计量单元。若WBS划分过粗，则参数估算的系数难以准确对标；若WBS划分过细，则类比估算找不到对应的历史参照，因此两者需要在粒度上相互匹配。

完成工时估算后，下一步是将各任务的工时数值与依赖关系输入**关键路径法（CPM）**分析，识别出决定项目最早完成日期的任务链。没有量化的工时数据，关键路径计算无法得出有意义的浮动时间（Float）。同时，工时估算也是**成本估算**的前置输入：将各角色的工时数量乘以对应的人力成本费率，即可得出每个里程碑阶段的预算需求。
