# 依赖关系图

## 概述

依赖关系图（Dependency Map）是一种将游戏项目中各团队、任务、资产之间的前置条件与输出关系以节点-边的形式可视化呈现的工具。图中每个节点代表一个具体的团队或任务单元，每条有向边则代表"A团队的输出是B团队开始工作的前提"这一依赖事实。通过依赖关系图，制作人和项目经理可以一眼看出哪些节点是多条依赖链的交汇点，这些交汇点通常就是项目的潜在瓶颈。

依赖关系图的方法论源于20世纪50年代诞生的关键路径法（Critical Path Method，CPM），由杜邦公司和雷明顿兰德公司于1957年联合开发。游戏行业在2000年代中期大型3A项目规模扩张后，开始广泛借鉴CPM思想，将静态的甘特图升级为能够实时反映跨部门依赖的有向无环图（DAG，Directed Acyclic Graph）。

在游戏制作的跨部门协作场景中，依赖关系图的核心价值在于将"谁在等谁"这一隐性信息转化为可测量、可追踪的显性结构。当美术团队等待技术美术交付着色器、关卡设计团队等待程序完成工具链时，如果没有依赖关系图，这些等待往往直到排期滑坡时才被察觉。

## 核心原理

### 节点与有向边的定义规则

依赖关系图中的每个节点必须对应一个有明确完成标准的可交付物（Deliverable），而非一个模糊的职能部门。例如，不能只写"美术"，而应写"角色A基础骨骼绑定完成"。有向边的方向遵循"箭头指向依赖方"的约定：若节点X→节点Y，表示Y必须在X完成后才能启动。

图的合法形态要求必须是有向无环图（DAG），即图中不能存在环路。如果出现"程序等待策划，策划等待程序"的循环依赖，就会在图中形成环，这在DAG中是禁止的，也说明现实中存在需要立即打破的沟通死锁。

### 关键路径计算

关键路径是依赖关系图中从项目起点到终点耗时最长的那条路径，其公式为：

**关键路径长度 = max( Σ 路径上每个节点的持续时间 )**

具体计算使用正向遍历与反向遍历两次扫描：正向遍历计算每个节点的最早开始时间（ES）和最早完成时间（EF）；反向遍历计算最晚开始时间（LS）和最晚完成时间（LF）。当某节点的 LS - ES = 0（即总浮动时间为零）时，该节点位于关键路径上，任何延误都将直接推迟整体交付日期。

### 瓶颈识别：入度与出度分析

在游戏制作的依赖关系图中，入度（Indegree）高的节点——即被多个团队所依赖的节点——是最常见的瓶颈来源。例如，一个技术美术团队同时为角色、场景、特效三个部门提供材质管线支持，其节点入度为0但出度为3，所有三条下游任务链都挂在它身上。识别此类节点的经验阈值通常是：出度≥3的节点需要单独进行风险评估并分配缓冲时间（Buffer）。

## 实际应用

**示例：开放世界游戏的关卡制作管线**

以某开放世界游戏一个区域关卡的制作为例，依赖关系图可能包含以下链条：

1. 世界观文档（叙事）→ 地形白盒（关卡设计）→ 地形精模（3D美术）→ 植被散布（技术美术）→ 性能优化（程序）→ 最终发布
2. 世界观文档（叙事）→ 怪物设定（策划）→ 怪物模型与动画（美术+动画）→ AI行为树接入（程序）→ 关卡刷新逻辑（关卡设计）→ 最终发布

在这张图中，"世界观文档"节点出度为2，是两条链的共同起点；而"最终发布"节点入度为2，是两条链的汇聚点。通过计算，若路径1总耗时为8周，路径2总耗时为11周，则关键路径为路径2，项目最短周期为11周。路径1有3周的总浮动时间，意味着地形美术团队在不影响发布的前提下最多可延迟3周。

**工具实践**

在实际制作中，依赖关系图常在Miro或Lucidchart中手动绘制，也可在Jira中通过"Epic Link"和"Blocks/Is Blocked By"关系字段自动生成依赖视图。每周Sync会议中应将图中红色标注（已延误节点）作为首要议题，而非遍历所有任务。

## 常见误区

**误区一：把依赖关系图当成甘特图的替代品**

依赖关系图展示的是任务之间的逻辑顺序关系，而甘特图展示的是任务在时间轴上的排布。两者功能不同：依赖关系图回答"谁必须先完成"，甘特图回答"什么时候完成"。将两者混淆会导致制作人只关注时间节点而忽略逻辑先后，或反之只知道任务顺序而无法分配具体日期资源。正确做法是先建立依赖关系图确定关键路径，再将结果输入甘特图安排日期。

**误区二：依赖关系图只需在项目立项时画一次**

依赖关系在游戏制作过程中会随着方案变更、人员调整频繁改变。若某个系统在里程碑M2被砍掉，所有依赖该系统的下游节点的依赖边都必须被删除或重定向。静态的依赖关系图在制作中期往往已严重失真，必须建立每两周更新一次的维护节律，否则图本身会成为误导决策的噪声来源。

**误区三：将双向沟通关系误标为依赖关系**

程序团队和策划团队之间存在频繁的双向沟通，但这并不意味着两者之间存在硬性依赖。依赖关系图中的边只能代表"没有X的输出，Y无法开始"的强依赖，而非"X和Y需要保持对齐"的协作关系。把软性沟通需求误标为依赖边，会人为制造环路或夸大关键路径长度，导致排期虚胀。

## 知识关联

依赖关系图的有效绘制，建立在**沟通机制设计**所确立的信息流路径之上。只有当各团队之间已经有明确的输出物定义和交付节奏（例如每周五提交资产包），依赖关系图中的边才能对应到真实可操作的交接行为，而不是停留在概念层面。如果沟通机制尚未建立，依赖关系图往往只能反映制作人的主观预期，而非实际工作流。

在明确了依赖关系图所揭示的任务顺序与瓶颈后，下一步需要设计**交接流程**——即当一个节点完成时，其输出物如何以标准化的方式传递给下游依赖团队。依赖关系图告诉我们"谁传给谁"，交接流程则规定"以什么格式、通过什么渠道、附带哪些验收标准传递"。两者共同构成跨部门协作中任务流转的完整闭环。
