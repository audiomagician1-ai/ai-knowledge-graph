# 策划-开发协作

## 概述

策划-开发协作是游戏制作流程中，游戏设计师（策划）与程序开发工程师之间将设计意图转化为可执行代码功能的工作机制。其核心任务是将策划文档中的玩法逻辑、数值系统和交互规则，以精确无歧义的形式传递给开发团队，使最终实现的功能与设计意图保持一致。

这一协作模式在20世纪90年代末随着游戏项目规模扩大而逐渐规范化。早期小型团队中，策划与程序往往是同一人，但当《魔兽争霸II》（1995年）、《最终幻想VII》（1997年）这类百人级别项目出现时，设计与实现的分工变得不可避免，功能规格书（Functional Specification Document，FSD）作为跨角色沟通工具开始系统化使用。

策划-开发协作质量直接影响"设计漂移"（Design Drift）程度——即最终游戏功能与原始设计意图的偏离量。据业内统计，沟通不畅导致的返工约占单个功能总开发时间的30%~40%，而一份合格的功能规格书可以将需求变更导致的代码重写概率从62%降低至约18%。

## 核心原理

### 功能规格书（FSD）的结构要素

功能规格书是策划-开发协作的核心文档载体，不同于高层次的游戏设计文档（GDD），FSD聚焦于**单一功能模块**的实现细节。一份标准FSD必须包含以下五个字段：

- **功能边界定义**：明确该功能覆盖的范围与排除项（In-Scope / Out-of-Scope）
- **输入/输出规格**：例如"玩家按下跳跃键→角色施加 `初速度 = 8.5 m/s` 的垂直冲量，重力加速度为 `g = 20 m/s²`"
- **状态机描述**：列出功能涉及的所有状态及转换条件，如"站立→起跳→空中→落地→着地硬直（0.15秒）"
- **异常处理预期**：当边界条件触发时（如跳跃时被击飞）期望的行为
- **验收标准（AC）**：可被测试的通过/失败判断标准，如"在60帧/秒下跳跃弧线与参考视频误差≤5%"

缺少验收标准是FSD最常见的致命缺陷——没有AC，开发者无从判断实现是否"完成"。

### 设计意图传达的三层模型

策划向开发传达信息时存在三个层次，混淆层次是产生误解的主要来源：

1. **体验目标层**（Why）：玩家应感受到"爬行时的紧张窒息感"
2. **功能逻辑层**（What）：玩家进入匍匐状态时移速降低至步行速度的40%，视角高度从170cm降至50cm
3. **实现建议层**（How）：可通过碰撞体缩放或摄像机骨骼绑定实现

策划的核心职责是清晰表达第1层和第2层，**不强制规定第3层**。当策划越级规定实现方式（如指定必须使用某种算法）而未说明背后的体验目标时，开发者遇到技术瓶颈时无法自主做出等效替代方案，协作效率急剧下降。

### 迭代反馈机制

有效的迭代反馈遵循**双向闭环**结构，而非单向提需求模式。其标准流程为：

```
策划提交FSD → 开发提出技术可行性评审（TFR）→ 策划根据TFR调整设计
→ 开发输出原型（Prototype）→ 策划填写反馈单（含"意图符合度"与"体验感受"两项独立评分）
→ 开发输出修改版本 → 进入Sprint验收
```

反馈单中"意图符合度"（0-5分）与"体验感受"（0-5分）分开打分的原因在于：有时实现完全符合FSD描述，但策划发现原设计本身存在问题（此时意图符合度=5，体验感受=2），这需要走**策划端修正流程**而非让开发返工。将两者合并打分会导致责任归因混乱，平均增加1.2个Sprint的无效沟通。

## 实际应用

**案例一：战斗系统的连击窗口设计**

某动作游戏策划设计了一套连击系统，要求"第二段攻击在第一段攻击动作播放到70%时可以输入触发"。策划在FSD中仅写"攻击可以连击"，开发者将窗口设置为动作完全结束后才可输入，导致手感偏重偏慢。问题根源是FSD缺少精确的帧数规格。修正后的条目为："输入窗口开始帧 = 当前攻击动画总帧数 × 0.70，窗口持续帧数 = 12帧，缓冲队列长度 = 8帧"。开发者依此实现后，一次通过AC验收。

**案例二：大世界任务触发器的边界歧义**

开放世界项目中，策划描述NPC触发对话的条件为"玩家靠近时"。开发者实现为半径8米的球形碰撞体，而策划预期的是NPC面朝方向前方5米×3米的矩形区域。FSD中缺少触发体形状与朝向参数导致3次返工。此后团队规定：所有触发器类FSD必须附带俯视图草图，标注长、宽、高三轴尺寸及坐标原点位置。

## 常见误区

**误区一：口头确认等同于文档确认**

策划与开发在站会上口头讨论并达成共识，但未同步更新FSD。当原始开发者休假、离职或进入下一Sprint后，替换人员按旧文档实现，造成功能回退。口头变更必须在24小时内体现于FSD版本变更记录（Change Log），并注明变更日期、变更人和变更原因。

**误区二：技术可行性由策划判断**

部分策划会在FSD中预先标注"此功能技术上应该不难"或直接规避可能复杂的功能，而不经过正式的TFR流程。实际上，某功能的技术复杂度取决于现有代码架构，例如在一个已有刚体物理系统的项目中添加"游泳"状态，可能需要对角色控制器进行架构级重构（工期3~5天），而非策划预估的"改几个参数"（0.5天）。跳过TFR导致的工期误判是Sprint失败的第二大原因。

**误区三：迭代反馈等于无限修改**

部分策划将原型阶段的迭代理解为可以随意更改设计方向的机会。实际上，一旦功能进入**集成阶段**（其他系统开始依赖该功能的接口），设计变更的影响范围会呈指数级扩大。业界通行做法是将功能开发分为"沙盒期"（允许设计变更）和"锁定期"（仅允许Bug修复），沙盒期通常不超过该功能总开发周期的前50%。

## 知识关联

**前置概念衔接：美术-技术协作**

美术-技术协作建立了资产规格化（Asset Specification）的基本意识——美术资产必须满足技术约束（多边形面数、贴图尺寸等）才能被引擎使用。策划-开发协作将这一规格化思维扩展到**逻辑层面**：设计意图必须满足可实现性约束才能被代码执行。功能规格书中的"输入/输出规格"部分，与美术规格书中的"导出设置"部分在方法论上具有直接对应关系。

**后续概念延伸：音频集成协作**

音频集成协作将引入第三个专业角色（音频设计师），使双向协作模式扩展为三角协作。策划在音频需求中的核心任务变为：用FSD中相同的"状态机描述"格式，为音频工程师定义音效触发时机（如"攻击命中帧+2帧时触发命中音效"），与策划-开发协作中定义动画输入窗口的方法高度一致。掌握FSD的精确描述规范后，音频触发逻辑的文档化将成为自然延伸。
