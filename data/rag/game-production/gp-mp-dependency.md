# 任务依赖管理

## 概述

任务依赖管理是游戏开发里程碑规划中用于描述和控制任务执行顺序的系统方法。其核心是通过四种标准化的依赖关系类型（FS、SS、FF、SF）来建立任务之间的逻辑约束，确保美术资产、程序功能、音效制作等不同职能团队的工作能够按照正确的顺序推进，避免因前置工作未完成导致下游任务无法开始或频繁返工。

这套依赖类型体系来源于项目管理学中的**箭线图法（PDM，Precedence Diagramming Method）**，由美国建筑和工程行业在20世纪60年代发展成熟，后被PMI纳入PMBOK标准。游戏行业从软件工程借鉴这套方法，用于解决多学科并行开发时的协调问题——例如，关卡设计师需要等待引擎渲染功能完成，但角色动画可以与地图灰盒同步推进，这类复杂的时序关系必须用依赖图才能清晰表达。

正确建立依赖关系可以直接缩短项目关键路径（Critical Path）的总长度。一个典型的3A游戏项目可能包含数千个任务节点，若依赖关系定义不准确，项目经理将无法识别哪些任务延误会影响最终交付日期，进而导致里程碑节点的连锁滑期。

## 核心原理

### 四种依赖类型的定义与公式

PDM定义了四种任务依赖类型，每种类型都可以附加**时滞（Lag）**或**时超（Lead）**参数来微调时序关系：

**FS（Finish-to-Start，完成到开始）** 是最常见的类型：前置任务完成后，后续任务才能开始。例如"角色概念设计"完成后，"角色3D建模"方可开始。表达式为：`后续任务开始时间 ≥ 前置任务完成时间 + Lag`。在游戏开发中约70%的依赖关系属于FS类型。

**SS（Start-to-Start，开始到开始）** 表示前置任务开始后，后续任务才能开始，两者可以并行推进。例如"关卡白盒制作"开始后，"AI寻路测试"即可同步启动，无需等待全部白盒完成。公式为：`后续任务开始时间 ≥ 前置任务开始时间 + Lag`。

**FF（Finish-to-Finish，完成到完成）** 要求后续任务的完成不得早于前置任务完成。例如"配音录制"完成后，"口型动画制作"才能最终完成，因为新增的对话录音可能随时触发口型修订。公式为：`后续任务完成时间 ≥ 前置任务完成时间 + Lag`。

**SF（Start-to-Finish，开始到完成）** 最为罕见：前置任务开始后，后续任务才能完成。在游戏生产中的典型场景是"新版本引擎上线"（前置）触发"旧版本维护任务"（后续）的正式关闭，即引擎切换的交接期管理。

### 时滞（Lag）与时超（Lead）

Lag是在依赖约束之上额外增加的等待时间，值为正数。例如 `FS+3天` 表示前置任务完成后还需等待3天（如材质烘焙渲染冷却期）后续任务才能开始。Lead是负值的Lag，表示允许后续任务提前开始，常见于内测版本的文档撰写可以在功能开发完成前2天（`FS-2天`）提前准备。误用Lead容易造成虚假的进度提前感，实际上是将风险前移。

### 依赖链分析与关键路径识别

将所有任务节点和依赖关系绘制成有向无环图（DAG）后，需要通过**前向计算（Forward Pass）**和**后向计算（Backward Pass）**两轮遍历来求解每个任务的最早开始时间、最早完成时间、最晚开始时间和总浮动时间（Total Float）。总浮动时间为0的任务链即为关键路径，该路径上任何一个任务延误1天，项目里程碑日期就会推迟1天。

## 实际应用

**案例1：角色系统生产线的依赖链设计**

在一个角色从概念到可用资产的完整生产链中，典型的依赖结构如下：
- "概念设计批准" → FS → "高模雕刻"（必须等概念批准）
- "高模雕刻" → SS+5天 → "低模制作"（高模开始5天后低模可同步推进）
- "低模制作" → FS → "UV展开" → FS → "贴图绘制"
- "贴图绘制" → FF → "材质设置"（贴图更新时材质设置也必须同步完成）

通过SS依赖将"高模雕刻"与"低模制作"并行，可将角色资产生产周期从串行的18天压缩到约13天，缩短约28%。

**案例2：Alpha里程碑前的依赖收口管理**

通常Alpha里程碑要求所有核心玩法可运行。在Sprint计划中，"战斗系统功能封锁"这一任务会被设为多条下游任务的共同前置（FS依赖），包括"AI行为树接入战斗系统"、"伤害数值平衡测试"和"战斗UI绑定数据接口"。若战斗系统功能封锁日期滑期，这三条并行任务全部被推迟，直接压缩Alpha前的测试窗口期。识别这种"扇形前置任务"是里程碑风险管理的关键。

## 常见误区

**误区1：将所有任务关系默认设为FS依赖**

许多初级制作人习惯性地用FS串联所有任务，因为这是最直觉的"完成再开始"逻辑。但这会人为延长关键路径，使项目计划比实际需要的时间多出10%~30%。识别哪些任务可以改为SS依赖并行推进，是里程碑压缩（Schedule Compression）中"快速跟进（Fast Tracking）"技术的核心操作，而不是简单地压缩每个任务的工期估算。

**误区2：混淆硬依赖与软依赖**

并非所有依赖关系都是技术上不可打破的"硬依赖"（Mandatory Dependency）。例如"游戏本地化文本翻译"在逻辑上依赖"UI字符串冻结"，但在资源充足时可以先翻译已确定的80%内容。将软依赖（Discretionary Dependency）误标为硬依赖，会导致可并行的任务被人为串行化，浪费并行开发的时间窗口。在依赖图维护中，应为每条依赖标注其类型（强制/资源/外部），以便在进度压力下快速识别可调整的空间。

**误区3：设置过多的Lead值来"优化"进度**

当项目经理面临里程碑压力时，容易在依赖关系上大量叠加Lead（负延迟）来制造进度提前的假象。例如将"QA集成测试"设为在"功能开发"完成前5天开始（FS-5天）。但若功能开发实际延误，QA团队的提前准备工作将全部变成等待，Lead实质上将风险转移给了下游任务而非消除了它。正确做法是先评估任务本身是否可拆分为独立的早期验证阶段，再决定是否调整依赖约束。

## 知识关联

**与甘特图的关系**：甘特图是任务依赖关系的可视化载体。在MS Project或Shotgun（现ShotGrid）等工具中，FS、SS、FF、SF依赖关系直接渲染为甘特图上连接任务条的箭线，箭线形状和起止锚点的位置对应不同的依赖类型。没有正确的依赖定义，甘特图仅是任务列表的图形化呈现，无法支持关键路径计算。

**与进度追踪的关系**：进度追踪阶段需要持续更新实际完成百分比，而依赖管理决定了某个任务的延误如何传播到下游任务。在进度追踪中引入"赢得值分析（Earned Value Analysis）"时，依赖链的准确性直接影响SPI（进度绩效指数）的计算精度。一旦关键路径上的任务出现偏差，依赖网络会自动重新计算受影响任务的最早开始时间，为制作人提供准确的滑期预警信号。
