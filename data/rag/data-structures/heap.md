# 堆（优先队列）

## 概述

堆（Heap）是一种满足堆序性质的完全二叉树，分为最大堆（Max-Heap）和最小堆（Min-Heap）两种形式。最大堆要求每个节点的值不小于其子节点的值，即根节点存储全局最大值；最小堆则相反，根节点存储全局最小值。优先队列（Priority Queue）是堆的抽象数据类型表示——堆是优先队列最常见的底层实现结构，支持以 O(log n) 的时间复杂度完成插入和提取最值操作。

堆的概念由 J. W. J. Williams 于 1964 年在提出堆排序算法时正式引入。与普通二叉搜索树不同，堆并不保证左右子节点之间的大小关系，只保证父节点与子节点之间的偏序关系。正因如此，堆的结构维护成本远低于 AVL 树或红黑树，却足以高效支持"取最值"这一核心操作。

在 AI 工程领域，堆是诸多算法的基础组件。A* 路径搜索算法使用最小堆管理开放列表，以便每次以 O(log n) 时间取出 f 值最小的节点；Top-K 问题（如推荐系统中取评分最高的 K 个候选项）通过维护一个大小为 K 的最小堆，可在 O(n log K) 时间内完成全量数据扫描，显著优于全排序的 O(n log n)。

## 核心原理

### 完全二叉树的数组表示

堆利用完全二叉树的层序遍历特性，将树结构紧凑地映射到一维数组中，无需存储指针。设数组下标从 1 开始，对于下标为 i 的节点：

- 父节点下标：`⌊i / 2⌋`
- 左子节点下标：`2i`
- 右子节点下标：`2i + 1`

这一映射使得堆的空间利用率极高，n 个元素的堆恰好占用数组的前 n 个位置，缓存局部性也优于链式二叉树结构。

### 上浮（Sift-Up）与下沉（Sift-Down）

堆的所有操作都依赖两个核心过程：

**上浮（Sift-Up / Bubble-Up）**：插入新元素时，将其放置于数组末尾（即树的最后一个位置），然后反复与父节点比较：若违反堆序性质则交换，直到满足条件或到达根节点。最坏情况下需比较 ⌊log₂ n⌋ 次，时间复杂度 O(log n)。

**下沉（Sift-Down / Heapify-Down）**：提取根节点后，将数组末尾元素移到根位置，然后反复与两个子节点中的较大者（最大堆）或较小者（最小堆）比较，若违反堆序则交换，直到叶节点或满足条件。同样时间复杂度 O(log n)。

### 建堆的线性时间算法

给定一个无序数组直接建堆，朴素做法是逐一插入 n 个元素，耗时 O(n log n)。但 Floyd 于 1964 年证明，从最后一个非叶节点（下标 ⌊n/2⌋）开始向前依次对每个节点执行 Sift-Down，总时间复杂度为 **O(n)**，而非直觉上的 O(n log n)。

证明依赖于各层节点数的几何级数求和：高度为 h 的节点最多需要 h 次交换，且高度为 h 的节点数约为 n / 2^(h+1)，累加得 Σ h · n/2^(h+1) = O(n)。这一线性建堆是堆排序达到 O(n log n) 总时间的关键前提。

### 时间复杂度汇总

| 操作 | 时间复杂度 |
|------|-----------|
| 插入（Insert） | O(log n) |
| 提取最值（Extract-Min/Max） | O(log n) |
| 查看最值（Peek） | O(1) |
| 建堆（Build-Heap） | O(n) |
| 任意元素删除 | O(log n) |

## 实际应用

**Top-K 问题**：在推荐系统或搜索引擎中，从 10 亿条记录里找评分最高的 K=100 个条目。维护容量为 100 的最小堆：遍历每条记录，若其评分高于堆顶（当前第 100 名），则弹出堆顶并插入新元素。总时间 O(n log K)，空间仅 O(K)。

**Dijkstra 最短路径算法**：使用最小堆（优先队列）存储"待松弛节点及其当前最短距离"，每次以 O(log V) 提取距离最小的节点。配合邻接表，总时间复杂度为 O((V + E) log V)，远优于朴素的 O(V²) 版本，在稀疏图上表现突出。

**流式数据中位数**：使用两个堆——一个最大堆存储较小的一半数据，一个最小堆存储较大的一半数据——保持两堆大小差不超过 1。每次插入后调整平衡，O(log n)；取中位数时直接查看堆顶，O(1)。Python 的 `heapq` 模块在 LeetCode 第 295 题"数据流的中位数"中即采用此方案。

**任务调度**：操作系统或 AI 推理服务中，CPU/GPU 任务按优先级排队。Linux 内核的完全公平调度器（CFS）使用红黑树而非堆，但许多简化调度场景（如 Celery 的延迟任务）直接使用最小堆按时间戳排序。

## 常见误区

**误区一：认为堆是有序结构**
堆仅保证父节点与子节点之间的偏序关系，同层节点之间没有任何大小保证。例如最大堆中，下标 2 的节点（左子）与下标 3 的节点（右子）大小完全不确定。因此，堆不支持 O(log n) 的任意元素查找，查找特定值需要 O(n) 线性扫描。将堆误当"有序数组"使用会导致错误的算法设计。

**误区二：混淆最大堆与 Python heapq 的关系**
Python 标准库 `heapq` 模块仅实现最小堆。若需最大堆，常见做法是将所有元素取负数后入堆，取出后再取负还原。直接使用 `heapq` 求 Top-K 最大值时，若不取负而是用 `nlargest()` 函数，其内部实现是维护最小堆而非最大堆，时间复杂度仍为 O(n log K)，而非错误预期的 O(n)。

**误区三：建堆时选择逐一插入而非 Floyd 算法**
对已知全量数据建堆时，许多初学者习惯循环调用 `heappush`，导致 O(n log n) 的时间开销。正确做法是使用 `heapq.heapify(list)`，该函数内部采用 Floyd 线性建堆，时间复杂度 O(n)。当 n 达到百万量级时，两者性能差距可超过 20 倍。

## 知识关联

**前置知识——二叉树**：堆是完全二叉树的特化形式，理解二叉树的层次结构、节点父子关系和树高概念（h = ⌊log₂ n⌋）是掌握堆的下标计算公式和 O(log n) 操作分析的直接基础。完全二叉树保证树高最小化，这正是堆效率的来源。

**后续概念——堆排序**：堆排序直接建立在本文所述的建堆（O(n) Floyd 算法）和反复 Extract-Max（O(log n) Sift-Down）两个操作之上，通过 n 次提取操作完成原地排序，总时间 O(n log n)，额外空间 O(1)。堆排序是唯一同时满足最坏情况 O(n log n) 且原地排序的经典比较排序算法。
