# 单调队列

## 概述

单调队列（Monotonic Deque）是一种基于双端队列（deque）实现的特殊数据结构，其内部元素始终保持严格单调递增或严格单调递减的顺序。与普通队列不同，单调队列在插入新元素时会主动弹出队尾所有违反单调性的元素，从而保证队列头部始终存储当前滑动窗口内的最大值（递减队列）或最小值（递增队列）。

该数据结构最早因解决"滑动窗口最大值"（Sliding Window Maximum）问题而广泛传播，该问题由 LeetCode 第 239 题收录，并频繁出现于字节跳动、阿里巴巴等大厂面试中。在此之前，朴素的滑动窗口最值算法需要对每个窗口内的 k 个元素逐一比较，时间复杂度为 O(n×k)；单调队列将该问题的时间复杂度压缩至 O(n)，因为每个元素在整个处理过程中至多入队一次、出队一次。

单调队列的价值不仅限于滑动窗口最值，它还是解决"带限制的区间最优子结构"问题（如多重背包的单调队列优化、跳跃游戏变体）的核心工具，在 AI 工程的特征工程、时间序列预处理、强化学习回放缓冲区分析中均有直接应用。

## 核心原理

### 单调性维护：插入时的"清洗"操作

单调队列的关键操作发生在**入队**阶段。以维护窗口最大值的递减队列为例，当新元素 `x` 入队前，队列会从队尾持续弹出所有**严格小于** `x` 的元素，直到队尾元素大于等于 `x` 或队列为空，然后将 `x` 加入队尾。这一操作的直觉是：若元素 `a` 比 `x` 小，且 `a` 比 `x` 更早进入窗口（即更早会被移除），则 `a` 永远不可能成为任何后续窗口的最大值，因此可以安全丢弃。

伪代码逻辑如下（以递减队列为例）：
```
入队(x):
    while deque 非空 且 deque.back() < x:
        deque.pop_back()
    deque.push_back(x)

出队(过期元素索引 i):
    if deque.front() == array[i]:
        deque.pop_front()

查询最大值:
    return deque.front()
```

注意：队列中存储的通常是**元素的下标**而非值本身，以便判断队头元素是否已滑出窗口边界。

### 时间复杂度证明：均摊 O(1)

整个长度为 n 的数组处理完毕后，每个元素恰好经历一次 `push_back` 和至多一次 `pop_back`（被更大元素清洗时）以及至多一次 `pop_front`（滑出窗口时）。因此三类操作的总次数上限均为 n，总操作次数 ≤ 3n，整体时间复杂度为 **O(n)**，空间复杂度为 **O(k)**（k 为窗口大小，队列中最多同时存储 k 个元素的下标）。

### 窗口边界管理

由于队列存储的是下标，每次查询前需检查队头下标 `deque.front()` 是否满足 `front ≥ current_index - k + 1`（当前窗口左边界）。若队头下标已超出窗口左边界，执行 `pop_front` 将其移除。这一步骤使得单调队列能够同时处理"元素过期"和"单调性维护"两个约束，缺一不可。

## 实际应用

### 应用一：滑动窗口最大值（LeetCode 239）

给定数组 `nums = [1,3,-1,-3,5,3,6,7]`，窗口大小 `k=3`，输出每个窗口的最大值 `[3,3,5,5,6,7]`。使用递减单调队列，依次处理每个元素时保持队头为当前窗口最大值的下标。当 `i ≥ k-1` 时，将 `nums[deque.front()]` 加入结果数组。

### 应用二：时间序列特征工程中的滚动极值

在 AI 工程的特征工程阶段，对长度为百万级的时间序列（如金融 tick 数据、传感器流数据）计算固定时间窗口内的最高价/最低价时，O(n×k) 的暴力法不可接受。单调队列方案可将 1,000,000 条记录、窗口 k=200 的滚动最大值计算从约 2×10⁸ 次比较降至 ≤ 3×10⁶ 次操作，实测耗时从数十秒降至毫秒级。

### 应用三：多重背包问题的单调队列优化

多重背包中，对于每种容量 `c` 的同余类，状态转移方程 `dp[j] = max(dp[j-v], dp[j-2v], ..., dp[j-k*v])` 本质上是在同余组内做长度为 `k+1` 的滑动窗口最大值，使用单调队列可将该问题的时间复杂度从 O(N×V×K) 优化至 O(N×V)，其中 N 为物品种数，V 为背包容量。

## 常见误区

### 误区一：混淆"存值"与"存索引"

初学者常将元素值而非下标存入队列，导致无法判断队头元素是否已滑出窗口左边界。例如对数组 `[5, 3, 5]`，窗口 k=2，若队列存值则两个 `5` 无法区分，出现"错误地移除仍在窗口内的最大值"的 bug。正确做法是**始终存储下标**，通过下标访问原数组获取值进行比较，通过下标与窗口边界比较判断过期。

### 误区二：严格单调 vs 非严格单调的选择错误

对递减队列，弹出条件应为 `deque.back() < x`（严格小于），这样遇到相等元素时保留较新的（下标更大的），确保窗口左移时不会过早移除仍有效的最大值元素。若误用 `deque.back() <= x`（严格递减），当窗口内有重复最大值时，队列中仅保留最新一个，可能在前一个下标尚未过期时已被清洗，导致结果错误。具体选择需根据题目对重复元素的处理要求决定。

### 误区三：认为单调队列只适用于最大值问题

单调队列完全对称地支持最小值查询——只需将插入时的比较方向取反，维护递增队列（入队时弹出所有大于新元素的队尾元素）即可。许多动态规划优化问题（如"绝对值不超过 t 的子数组和"）同时需要维护滑动窗口的最大值和最小值，需要同时维护两个单调队列分别处理。

## 知识关联

**前置知识**：
- **双端队列（deque）**：单调队列的物理载体，需熟悉 `push_front`、`push_back`、`pop_front`、`pop_back` 四种 O(1) 操作。C++ STL 中使用 `std::deque`，Python 中使用 `collections.deque`。
- **滑动窗口**：单调队列解决的正是滑动窗口最值这一具体问题，需理解窗口的左右指针推进逻辑以及窗口内元素的生命周期管理。

**扩展方向**：
- 将单调队列与**动态规划**结合，处理带有长度约束的区间最优化问题，如"长度在 [L, R] 之间的最大子数组和"可通过前缀和数组上的滑动窗口最小值转化为单调队列问题。
- 在 AI 工程的**在线学习**场景中，单调队列可作为流式数据的近似分位数估计的基础组件，配合遗忘机制实现对历史窗口统计量的高效更新。
