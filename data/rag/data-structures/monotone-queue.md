# 单调队列

## 概述

单调队列（Monotonic Deque）是一种在双端队列基础上施加单调性约束的数据结构，其队列内部的元素值始终保持严格递增或严格递减的顺序。它专门为解决**滑动窗口最值查询**问题而设计，将朴素的 O(n·k) 暴力扫描降低到 O(n) 线性时间，其中 n 是数组长度，k 是窗口大小。

单调队列的雏形出现在算法竞赛领域，最早被系统总结于2000年代初期的信息学奥林匹克竞赛题解中。与同样利用双端队列的普通滑动窗口技术不同，单调队列的关键创新在于**主动淘汰对未来答案无用的元素**，而不只是被动地在窗口滑出时删除旧元素。

在AI工程和机器学习的数据预处理流水线中，单调队列被广泛用于时间序列特征提取，例如在长度为百万级的传感器数据流中，以固定窗口计算局部最大功率值或局部最小温度值，能将特征工程阶段的耗时从分钟级降至秒级。

## 核心原理

### 单调性维护机制

单调队列的操作规则与普通双端队列的根本区别在于：**入队时执行条件性弹出**。以维护窗口最大值为例，当新元素 `a[i]` 准备从队尾入队时，若队尾元素 `a[deque.back()]` 满足 `a[deque.back()] ≤ a[i]`，则将队尾元素弹出，重复此过程直至队列为空或队尾元素严格大于 `a[i]`。这一步骤保证了队列从队头到队尾始终保持**严格递减**（求最大值时）或**严格递增**（求最小值时）。

被弹出的元素并非数据丢失——它们是被"遮蔽"的元素：若一个元素 `a[j]` 在位置 `j < i` 处的值小于等于 `a[i]`，则在 `a[j]` 仍在窗口内的所有时刻，`a[i]` 也在窗口内且更大，因此 `a[j]` 永远不可能成为任何窗口的最大值答案，可以安全丢弃。

### 双端删除策略

单调队列存储的是**元素的下标而非值**，这是正确实现的关键细节。队列维护两类删除操作：

- **队尾删除（入队时）**：维护单调性，淘汰被新元素遮蔽的旧元素（如上所述）。
- **队头删除（窗口滑动时）**：当 `deque.front() ≤ i - k`（队头下标已超出窗口左边界）时，从队头弹出过期元素。

每个元素在整个算法执行过程中**最多入队一次、出队一次**，因此 n 个元素产生的总操作次数为 O(2n)，即整体时间复杂度 O(n)，空间复杂度为 O(k)（队列中最多同时存在 k 个有效下标）。

### 完整算法伪代码（滑动窗口最大值）

```
输入：数组 a[0..n-1]，窗口大小 k
输出：max_val[0..n-k]，其中 max_val[i] = max(a[i..i+k-1])

初始化双端队列 dq 为空
for i in 0 to n-1:
    // 步骤1：队头过期检查
    while dq 非空 且 dq.front() <= i - k:
        dq.pop_front()
    // 步骤2：队尾单调性维护
    while dq 非空 且 a[dq.back()] <= a[i]:
        dq.pop_back()
    dq.push_back(i)
    // 步骤3：窗口已满时记录答案
    if i >= k - 1:
        max_val[i - k + 1] = a[dq.front()]
```

注意步骤1与步骤2的顺序不可交换：应先清理过期队头，再做单调性维护，最后推入新下标。

## 实际应用

### LeetCode 239：滑动窗口最大值

这道题是单调队列最经典的直接应用。给定 `nums = [1,3,-1,-3,5,3,6,7]`，k=3，输出每个窗口的最大值 `[3,3,5,5,6,7]`。用暴力法需要约 8×3=24 次比较，而单调队列仅需 8 次入队操作和若干次出队操作（共不超过16次），线性完成。

### 时间序列特征工程

在AI工程的滚动特征计算场景中，假设有一条长度为 10,000,000 的股票 tick 价格序列，需要计算每个时刻前60秒的最高价和最低价。用单调队列可在约 0.3 秒内完成两遍扫描（最大值+最小值各一遍），而用 Pandas 的 `rolling().max()` 默认实现在 CPython 环境下耗时约 4-8 秒，差异主要来自于单调队列避免了重复比较。

### 多维度扩展：二维滑动窗口最值

单调队列可以扩展到矩阵的二维滑动窗口最值问题（如图像处理中的局部最大池化 Max Pooling）：先对每一行用单调队列做一维压缩（时间 O(n·m)），再对结果矩阵的每一列再次应用单调队列，总复杂度 O(n·m)，远优于暴力的 O(n·m·k²)。

## 常见误区

### 误区1：混淆存储值与存储下标

初学者常直接将元素值而非下标压入队列。这会导致在执行"队头过期检查"时无法判断该元素对应的原始位置是否已滑出窗口——因为相同的值可能出现在数组的多个位置，无法区分。正确做法是队列中始终存储**下标 i**，需要比较大小时通过 `a[dq.front()]` 取值。

### 误区2：入队条件使用非严格不等号导致重复元素错误

在维护递减队列时，若将入队条件写为 `a[dq.back()] < a[i]`（严格小于），则当出现相等元素时会保留队尾的旧元素。虽然在求最大值时结果仍然正确，但会造成队列中存在冗余元素，在某些变体问题（如求使窗口最大值等于某值的最小窗口长度）中会产生错误答案。标准写法是 `a[dq.back()] <= a[i]` 以保证严格单调。

### 误区3：认为单调队列能处理动态更新的滑动窗口

单调队列只支持**顺序移动**的固定步长窗口（窗口右边界每次向右移动1步），不支持随机访问或跳跃式窗口移动。若窗口可以任意扩张或收缩（如双指针可变长窗口中同时查询当前窗口的最值），单调队列的队头过期机制失效，需要改用线段树或稀疏表（Sparse Table）来维护区间最值。

## 知识关联

**与双端队列的关系**：单调队列直接使用双端队列的 `push_front`、`push_back`、`pop_front`、`pop_back`、`front`、`back` 六个接口，但在语义上限制了调用模式——`push_front` 在单调队列中从不使用，新元素只从队尾入队，旧元素过期时只从队头删除。理解双端队列的 O(1) 两端操作是理解单调队列复杂度分析的基础。

**与滑动窗口的关系**：经典的滑动窗口双指针技术维护的是窗口内元素的某种聚合状态（如计数、求和），而单调队列在此基础上额外维护窗口内的**极值及其位置**。两者共享"窗口右边界递增扫描"的框架，但单调队列的队尾淘汰逻辑是双指针法中没有的新机制。

**向上延伸至线段树与稀疏表**：当查询不再局限于固定步长滑动窗口而需要任意区间 `[l, r]` 的最值时，需要学习稀疏表（O(n log n) 预处理，O(1) 查询，不支持更新）或线段树（O(n) 建树，O(log n) 查询与更新）。单调队列解决的是这两者覆盖范围之外的一个特殊而高效的子集：**连续固定长度滑动窗口的在线最值查询**。
