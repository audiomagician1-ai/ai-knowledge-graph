# 哈希表

## 概述

哈希表（Hash Table）是一种通过**哈希函数**将键（Key）映射到存储位置的数据结构，实现平均O(1)时间复杂度的插入、删除和查找操作。其核心思想是将任意长度的键转换为固定范围的整数索引，直接定位到数组中的存储槽位，从而绕过逐一比较的线性搜索。

哈希表的概念可追溯至1953年，IBM工程师Hans Peter Luhn在内部备忘录中首次描述了哈希链接技术。1954年，Arnold Dumey在论文中正式提出将哈希函数用于地址计算的思想。此后，Donald Knuth在其1973年出版的《计算机程序设计艺术》第三卷中系统化整理了哈希表的理论基础，奠定了现代实现的标准范式。

在AI工程场景中，哈希表是特征存储、词汇表索引、去重运算和缓存系统的基础支撑结构。例如，NLP系统中将词汇表（词汇量可达数十万量级）映射到词向量索引时，哈希表比二叉搜索树快约40倍（对比O(1) vs O(log n)，n=100,000时差距约17倍操作数）。Python的`dict`、Java的`HashMap`以及Redis的内部哈希结构，均基于哈希表实现。

## 核心原理

### 哈希函数

哈希函数 `h(k)` 将键 `k` 映射到槽位索引 `i`，公式为：

```
i = h(k) mod m
```

其中 `m` 为哈希表的槽位总数（通常选取质数以减少冲突）。对于字符串键，一种常见的多项式滚动哈希为：

```
h = (c₀ × p⁰ + c₁ × p¹ + ... + cₙ × pⁿ) mod m
```

其中 `p` 常取31（处理小写字母）或131（处理ASCII），`cᵢ` 为第i个字符的ASCII值。一个好的哈希函数需满足：计算高效（O(key长度)）、分布均匀（各槽被命中概率接近1/m）、雪崩效应强（输入微小变化导致输出大幅变化）。

### 冲突处理

**哈希冲突**指两个不同键产生相同索引，是哈希表设计的核心挑战。主流解决方案有两类：

**链地址法（Separate Chaining）**：每个槽位存储一个链表，冲突键追加到链表尾部。Python的`dict`在3.6版本之前使用此方法的变种。最坏情况下（所有键冲突），查找退化为O(n)。

**开放寻址法（Open Addressing）**：冲突时按照探测序列寻找下一个空槽。线性探测（步长固定为1）实现简单但易产生**主聚集**（primary clustering）；双重哈希（使用第二个哈希函数决定步长）可有效缓解聚集问题，探测序列为 `h(k, i) = (h₁(k) + i × h₂(k)) mod m`。

### 负载因子与扩容

**负载因子**（Load Factor）定义为：

```
α = n / m
```

其中 `n` 为已存储键值对数量，`m` 为槽位总数。当 `α` 过高时，冲突概率急剧上升。Java的`HashMap`默认负载因子为**0.75**，超过此阈值时触发扩容（rehashing）：分配2倍大小的新数组，并将所有键重新哈希插入，此操作时间复杂度为O(n)。Python的`dict`扩容阈值为2/3（约0.67）。扩容后平均查找性能恢复到O(1)，这是哈希表"均摊O(1)"保证的来源。

## 实际应用

**AI特征工程中的特征哈希（Feature Hashing / Hashing Trick）**：在大规模机器学习中，原始特征（如用户ID、商品类目）种类可达数百万，直接独热编码内存不可承受。特征哈希将特征名直接哈希到固定维度（如2²⁰ = 1,048,576维）的向量索引，无需维护显式的特征词典，被scikit-learn的`HashingVectorizer`和Vowpal Wabbit广泛采用。

**词向量查找表**：Word2Vec、GloVe等模型训练完成后，词汇表通过哈希表存储`词→向量索引`的映射，推理时每次词查找耗时稳定在微秒级，不受词汇量（通常30万~100万词）影响。

**去重与集合运算**：数据预处理阶段对数十亿条URL或文档进行去重时，基于哈希表的集合（`HashSet`）相比排序后扫描方案节省O(n log n)排序开销，内存访问模式更适合流式处理。

## 常见误区

**误区1：哈希表查找总是O(1)**

严格来说，哈希表的O(1)是**平均期望复杂度**，而非最坏情况保证。当攻击者精心构造大量哈希冲突的键（哈希洪泛攻击，Hash DoS），所有键落入同一链表，查找退化为O(n)。Python从3.3版本起引入**SipHash-1-3**作为默认字符串哈希函数，并加入随机盐值（hash randomization），正是为了防御此类攻击。

**误区2：负载因子越小越好**

降低负载因子确实减少冲突，但代价是内存利用率下降和缓存局部性变差。α=0.1时，90%的槽位为空，内存浪费严重，且扩容触发更频繁（每次O(n)的rehashing）。工程实践中0.65~0.75是经过大量benchmark验证的平衡点。

**误区3：哈希表与Python字典完全等价**

Python 3.7+的`dict`保证**插入顺序**（通过将哈希表与独立的紧凑键列表结合），这是标准哈希表不提供的特性。此外，Python的`dict`对整数键使用恒等哈希（`hash(n) == n`，模m后取余），对字符串则使用SipHash，这与"哈希表使用统一哈希函数"的教科书描述存在差异。

## 知识关联

**与前置概念的衔接**：字典/映射（Dictionary/Map）是哈希表在抽象层面的接口定义，规定了`put`、`get`、`delete`等操作语义，而哈希表是这一接口最常用的底层实现。理解字典的键值对语义是正确使用哈希表的前提，但字典也可由平衡二叉搜索树（如C++ `std::map`）实现，两者时间复杂度不同。

**通向布隆过滤器**：布隆过滤器可视为哈希表的空间极端压缩版本——它使用多个哈希函数将键映射到一个位数组，彻底放弃存储具体值，仅保留"键是否存在"的概率性判断（存在假阳性，无假阴性）。掌握哈希冲突概率的计算方式，是理解布隆过滤器误判率公式 `(1 - e^{-kn/m})^k` 的直接基础。

**通向LRU缓存**：LRU（最近最少使用）缓存的标准实现结合了哈希表（O(1)键查找）和双向链表（O(1)节点移动），两者缺一不可。哈希表负责存储`键→链表节点指针`的映射，使得访问任意缓存条目后能立即定位并调整其在链表中的位置，实现整体O(1)的`get`和`put`操作。
