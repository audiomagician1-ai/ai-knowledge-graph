# 邻接表

## 概述

邻接表（Adjacency List）是图数据结构的一种链式存储表示方式，由图论与计算机科学的交叉发展而来，在1960年代随着链表数据结构的普及逐步成为图的标准实现方案之一。其核心思想是：为图中每个顶点维护一条链表（或动态数组），链表中存储所有与该顶点直接相邻的顶点编号（以及可能的边权重）。

邻接表的存储空间复杂度为 **O(V + E)**，其中 V 是顶点数，E 是边数。这与邻接矩阵的 O(V²) 形成鲜明对比。对于稀疏图（边数远小于 V²），邻接表能节省大量内存——例如，一个拥有 10,000 个顶点、20,000 条边的社交网络图，邻接矩阵需要约 10⁸ 个存储单元，而邻接表只需约 30,000 个。

邻接表在 AI 工程领域中广泛应用于知识图谱、图神经网络（GNN）的数据预处理、推荐系统的用户-物品图构建，以及路径规划算法（Dijkstra、BFS、DFS）的底层实现。理解邻接表的内存布局与访问模式，直接影响大规模图算法的工程效率。

## 核心原理

### 基本结构定义

邻接表由两部分组成：一个长度为 V 的顶点数组（头数组），以及挂载在每个顶点下的邻居链表。对于无向图中的一条边 (u, v)，需要在 u 的链表中插入 v，同时在 v 的链表中插入 u，因此总链表节点数为 **2E**。对于有向图，每条有向边 u→v 仅在 u 的链表中插入 v，总链表节点数为 **E**。

以 Python 字典实现为例：
```python
graph = {
    0: [1, 2],
    1: [0, 3],
    2: [0, 3],
    3: [1, 2]
}
```
这表示一个4顶点无向图，每条边在双方链表中各记录一次，总存储8个邻居条目。

### 带权邻接表

当图中每条边具有权重时（如路网中的距离、知识图谱中的置信度），链表节点需同时存储**目标顶点**与**边权**，结构变为：

```
Node { int neighbor; float weight; Node* next; }
```

在 Python 中常用列表嵌套元组实现：`graph[u].append((v, w))`，其中 `w` 为边权。此时 Dijkstra 算法的时间复杂度为 **O((V + E) log V)**，需配合最小堆来高效提取最短路径。

### 静态邻接表（CSR格式）

在高性能图计算和 GNN 框架（如 PyG、DGL）中，常用**压缩稀疏行（CSR, Compressed Sparse Row）**格式替代动态链表。CSR 由两个数组构成：
- `indptr`：长度 V+1，`indptr[i]` 存储顶点 i 的邻居在 `indices` 中的起始位置
- `indices`：长度 E，顺序存储所有边的目标顶点

访问顶点 i 的所有邻居：`indices[indptr[i] : indptr[i+1]]`，时间复杂度 O(deg(i))，且内存连续访问对 CPU 缓存友好。PyTorch Geometric 内部使用 `edge_index`（形状为 [2, E] 的张量）本质上是 CSR 的变体。

### 时间复杂度对比

| 操作 | 邻接表 | 邻接矩阵 |
|------|--------|----------|
| 检查边 (u,v) 是否存在 | O(deg(u)) | O(1) |
| 遍历顶点 u 的所有邻居 | O(deg(u)) | O(V) |
| 添加一条边 | O(1) | O(1) |
| 空间占用 | O(V+E) | O(V²) |

## 实际应用

### 知识图谱的构建

在 AI 知识图谱中，实体（entity）是顶点，关系（relation）是有向带标签的边。用邻接表存储时，每个实体的链表记录 `(目标实体ID, 关系类型ID)` 二元组。以 Freebase 数据集为例，其包含约 1,200 万个实体和 1.2 亿条关系边，若用邻接矩阵存储需要约 10^14 个单元，显然不可行，而邻接表仅需 O(1.2亿) 级别存储。

### 图神经网络的消息传递

GNN 的消息传递机制本质上是沿邻接表遍历：对于每个顶点 v，聚合其所有邻居 u ∈ N(v) 的特征向量。公式为：

$$h_v^{(k)} = \text{AGGREGATE}\left(\left\{h_u^{(k-1)} : u \in \mathcal{N}(v)\right\}\right)$$

其中 $\mathcal{N}(v)$ 即顶点 v 在邻接表中对应的邻居集合。邻接表的遍历效率直接决定了 GNN 每轮迭代的计算速度。

### BFS 在推荐系统中的应用

电商推荐系统中，用户-商品交互可建模为二部图，用邻接表存储。基于 BFS 进行 K 跳邻居采样（如 GraphSAGE 算法中 K=2，每跳采样25和10个邻居），邻接表能以 O(采样数) 的时间完成局部子图提取，而邻接矩阵需扫描整行。

## 常见误区

**误区1：认为邻接表总是优于邻接矩阵**
邻接表不适合**稠密图**（边数接近 V²）。此时链表指针的额外内存开销反而比邻接矩阵更大。更重要的是，判断两点间是否存在边时，邻接表需遍历链表，时间复杂度为 O(deg(u))，可能高达 O(V)；而邻接矩阵只需 O(1) 的随机访问。在Floyd-Warshall全源最短路算法中，邻接矩阵反而更高效。

**误区2：无向图只需存储一方的链表**
无向图中边 (u,v) 必须在 u 和 v 的链表中**各存储一次**。若只存一方，在遍历 v 的邻居时将遗漏 u，导致 BFS/DFS 结果错误，图的连通分量计算也会出错。这是初学者在手动构建无向图邻接表时最常见的 bug。

**误区3：混淆链式邻接表与 CSR 格式的适用场景**
动态链表实现的邻接表支持 O(1) 的边插入，适合**图结构频繁变化**的场景（如实时社交网络）；CSR格式的静态邻接表不支持高效的边插入（需重建整个数组），但内存访问连续，适合**图结构固定、需要高性能遍历**的场景（如离线 GNN 训练）。将两者混用会导致性能问题或实现错误。

## 知识关联

**前置概念——图（数据结构）**：邻接表是图的存储实现，必须先理解图的顶点、边、有向/无向、权重等基本概念。图的稀疏性（sparsity）直接决定了选择邻接表还是邻接矩阵：当 E ≪ V² 时选邻接表，当 E 接近 V² 时选邻接矩阵。图的度（degree）的定义与邻接表的链表长度直接对应：顶点 v 的出度 = v 的邻接链表长度。

**延伸方向**：掌握邻接表后，可进一步学习基于邻接表实现的经典算法——BFS（时间复杂度 O(V+E)，依赖邻接表的逐层遍历）、DFS（递归遍历邻接表）、Dijkstra（带权邻接表 + 最小堆）。在 AI 工程方向，邻接表是理解 CSR 稀疏矩阵格式和 PyTorch Geometric 中 `edge_index` 数据格式的直接基础，也是图采样算法（GraphSAGE、FastGCN）在工程实现层面的核心数据结构。
