# 邻接表

## 概述

邻接表（Adjacency List）是一种用于表示图结构的数据结构，它为每个顶点维护一个链表（或动态数组），存储与该顶点直接相邻的所有顶点。对于一个含有 V 个顶点、E 条边的图，邻接表的总空间复杂度为 O(V + E)，这与邻接矩阵的 O(V²) 形成鲜明对比，使邻接表在稀疏图场景下具备明显的内存优势。

邻接表的概念最早伴随图论的计算机实现而出现，在1960年代的早期算法教材中已有描述。Dijkstra最短路径算法（1959年发表）和深度优先搜索算法在描述其实现时，均将邻接表列为首选底层存储结构。与邻接矩阵相比，邻接表查询"顶点 u 的所有邻居"仅需遍历其链表，时间为 O(degree(u))，而不是扫描整行 O(V)。

邻接表在AI工程领域的重要性体现在知识图谱存储、图神经网络（GNN）的消息传递机制以及搜索算法（如A\*、BFS）的底层实现中。现实世界的图往往是稀疏图——例如社交网络中每人平均只有数百个朋友，而用户总数可达亿级——此时邻接表能将内存消耗从 O(V²) 降至 O(V + E)，节省数个数量级的空间。

## 核心原理

### 基本数据结构与内存布局

邻接表本质上是一个长度为 V 的数组（或哈希表），数组下标 i 对应顶点 i，其值是一个存储邻居顶点编号的列表。在 Python 中最常用 `defaultdict(list)` 或 `List[List[int]]` 实现：

```python
# 无向图，5个顶点
adj = [[] for _ in range(5)]
adj[0].append(1)
adj[1].append(0)   # 无向边需双向添加
adj[0].append(3)
adj[3].append(0)
```

对于带权图，链表节点需同时存储目标顶点和边权重，通常表示为 `(neighbor, weight)` 元组。每添加一条无向边需写入两次，有向图只需写入一次。

### 时间复杂度分析

邻接表各操作的时间复杂度如下：

| 操作 | 时间复杂度 |
|------|-----------|
| 检查边 (u, v) 是否存在 | O(degree(u)) |
| 遍历顶点 u 的所有邻居 | O(degree(u)) |
| 添加一条边 | O(1) 摊销 |
| 删除一条边 | O(degree(u)) |
| 遍历整图所有边 | O(V + E) |

检查边存在性时，邻接表需要线性搜索链表，而邻接矩阵只需 O(1) 的索引操作。这是邻接表唯一劣于邻接矩阵的操作场景。当需要频繁执行"判断两顶点是否相邻"时，可改用邻接集合（Adjacency Set），将链表替换为哈希集合，使边存在性检查降至平均 O(1)。

### 有向图与无向图的构建差异

有向图中，边 (u → v) 只在 `adj[u]` 中追加 v，不修改 `adj[v]`。这使得邻接表天然支持区分出度（`len(adj[u])`）和入度（需要额外维护反向邻接表 `in_adj`）。若要同时支持 O(1) 的入度查询，需维护两个邻接表：正向表 `adj` 用于出边，反向表 `radj` 用于入边。Kosaraju算法求强连通分量时，就需要同时使用这两个方向的邻接表，先对原图做 DFS 并记录完成顺序，再在反向图上按该顺序做第二次 DFS。

### 邻接表在图神经网络中的稀疏表示

PyTorch Geometric 和 DGL 等GNN框架使用 COO（Coordinate List）格式存储边，本质是邻接表的变体——用两个长度为 E 的数组分别存储所有边的源顶点和目标顶点：

```
edge_index = [[0, 0, 1, 3],   # 源顶点
              [1, 3, 0, 0]]   # 目标顶点
```

这种格式与邻接表等价，但更利于GPU向量化运算。消息传递过程中，每条边对应一次"源节点特征 → 目标节点聚合"的操作，直接从 `edge_index` 读取即可，无需解析嵌套链表。

## 实际应用

**BFS与DFS的邻接表遍历**：使用邻接表实现BFS时，从起始顶点 s 出发，将 `adj[s]` 中所有顶点入队，再依次展开。整体时间复杂度为 O(V + E)，若用邻接矩阵则退化为 O(V²)，对百万级顶点的图相差可达百倍。

**Dijkstra最短路径算法**：标准实现使用最小堆 + 邻接表，时间复杂度为 O((V + E) log V)。遍历顶点 u 的邻居时，直接迭代 `adj[u]` 中的 `(neighbor, weight)` 对，无需扫描整行矩阵。对于电商推荐系统中的用户-商品二部图（亿级顶点、百亿级边），邻接表是唯一可行的存储方案。

**知识图谱存储**：Neo4j等图数据库内部以邻接表形式存储实体关系，每个实体节点直接持有指向其关系链表的指针，实现O(1)的邻居访问（称为"无索引邻接"，Index-Free Adjacency）。这使得多跳查询（如"找出某演员合作过的导演所导的其他电影"）无需全表扫描，直接沿指针链路完成。

## 常见误区

**误区一：认为邻接表总是优于邻接矩阵**

邻接表在稀疏图（E ≪ V²）中内存高效，但在稠密图（E 接近 V²）中，邻接矩阵的内存开销相当甚至更小（因为链表节点有指针开销），且边存在性查询为 O(1)。Floyd-Warshall全源最短路径算法需要反复进行三重循环的随机访问，配合邻接矩阵的缓存局部性更优，若硬套邻接表反而性能下降。

**误区二：添加无向边时只写单向**

新手在构建无向图时，常常只执行 `adj[u].append(v)` 而忘记 `adj[v].append(u)`，导致从 v 出发无法遍历到 u，产生"单向无向图"的逻辑错误。BFS/DFS 结果会出现顶点遗漏，且错误极难直接从输出定位，需要在构建时封装成专门的 `add_edge(u, v)` 函数来防范。

**误区三：将邻接表的 degree 等同于边数**

对于无向图，邻接表中所有链表长度之和 = 2E（每条边被计数两次），而非 E。计算图的实际边数时必须将总和除以2。有向图则不同：所有 `len(adj[i])` 之和 = E（等于总出度之和）。混淆这一点会导致边数统计错误，进而影响算法的终止条件判断。

## 知识关联

**前驱依赖——图的基础概念**：邻接表是"图（Graph）"这一抽象概念的具体存储实现。掌握顶点（Vertex）、边（Edge）、有向/无向图、度（Degree）的定义是使用邻接表的前提——邻接表的每个槽位对应一个顶点，每个列表元素对应一条出边，这些概念直接映射到图的数学定义。

**并列结构——邻接矩阵**：邻接表与邻接矩阵是图的两大主流存储方式，选择标准取决于图的稀疏程度和主要操作类型：稀疏图 + 邻居遍历主导 → 邻接表；稠密图 + 边存在性查询主导 → 邻接矩阵。两者在空间和时间上的权衡是图算法设计的基础决策之一。

**后续应用——图算法族**：邻接表直接支撑了拓扑排序（Kahn算法需遍历每个顶点的出边）、强连通分量（Kosaraju/Tarjan算法）、最短路径（Dijkstra/Bellman-Ford）等一系列图算法的高效实现。在GNN框架中，理解邻接表有助于掌握消息传递网络（Message Passing Neural Network, MPNN）的边索引机制，这是Graph Attention Network（GAT）和GraphSAGE等模型的底层数据流基础。
