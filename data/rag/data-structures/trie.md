# 字典树（Trie）

## 概述

字典树（Trie），又称前缀树或数字搜索树，是一种专门用于处理字符串集合的树形数据结构。与哈希表或二叉搜索树不同，Trie 的每个节点不存储完整的键值，而是将键的每个字符分散存储在从根到叶的路径上。例如，存储英文单词时，从根节点到某个标记了"单词结束"的节点所经过的所有边上的字符拼接起来，就构成一个完整的单词。

Trie 这一名称源自英文单词 "retrieval"，由 Edward Fredkin 于 1960 年在论文中正式命名。其理论基础早于此已被使用，但 Fredkin 将其系统化为一种独立的数据结构。Trie 在自然语言处理、搜索引擎自动补全、IP 路由表（使用二进制 Trie）等领域有着极其广泛的应用，是 AI 工程中处理大规模字符串检索任务的首选结构之一。

Trie 的核心优势在于其时间复杂度与字符集大小和键的总数无关，查找一个长度为 `m` 的字符串的时间复杂度为 **O(m)**，而与字典中存储的单词总数 `n` 完全无关。这一特性使其在词汇量达到百万级别时，相较于平衡二叉搜索树的 O(m log n) 依然保持高效。

## 核心原理

### 节点结构与字符映射

Trie 的每个节点包含两个核心字段：一个大小为字符集大小 `|Σ|` 的子节点数组（或哈希表），以及一个布尔值 `is_end`，用于标记该节点是否为某个单词的结尾。对于仅包含小写英文字母的场景，每个节点持有一个长度为 **26** 的指针数组，数组下标通过 `char - 'a'` 计算得出。

```python
class TrieNode:
    def __init__(self):
        self.children = [None] * 26  # 固定大小 26
        self.is_end = False
```

当字符集扩展至 Unicode 时，通常改用哈希表代替固定数组，以避免每个节点占用过多内存。这一设计取舍直接影响 Trie 的空间复杂度：固定数组实现的空间复杂度为 **O(n × m × |Σ|)**，其中 `n` 为单词数量，`m` 为平均长度，`|Σ|` 为字符集大小。

### 插入与查找操作

**插入操作**从根节点出发，逐字符遍历目标字符串。对于每个字符 `c`，若 `children[c - 'a']` 为空，则创建新节点；否则沿已有路径前进。遍历完最后一个字符后，将当前节点的 `is_end` 置为 `True`。整个过程严格执行 `m` 次节点访问，时间复杂度为 O(m)。

**查找操作**分为两种语义：
- **精确匹配（search）**：遍历完所有字符后，检查当前节点的 `is_end` 是否为 `True`。
- **前缀匹配（startsWith）**：遍历完前缀的所有字符后，只要路径存在即返回 `True`，无需检查 `is_end`。

这两种操作的区分是 Trie 与普通树的关键差异所在——`is_end` 标记使得 Trie 能在同一路径上同时表示 "app" 和 "apple" 两个独立单词。

### 前缀共享与压缩 Trie

Trie 最本质的优化来自**前缀共享**：所有拥有相同前缀的字符串共用同一段路径。例如，同时存储 "cat"、"car"、"card" 三个单词，前两个字符 "ca" 对应的两个节点只被创建一次。当字符串集合中存在大量公共前缀时，这一机制极大压缩了存储空间。

**压缩 Trie**（Compressed Trie，又称 Patricia Trie 或 Radix Tree）是 Trie 的变体，将只有单个子节点的连续节点合并为一个节点，节点上存储字符串片段而非单个字符。Linux 内核的路由表实现、Go 语言的 `net/http` 路由器（httprouter）均采用基于 Radix Tree 的路径匹配，是压缩 Trie 在工业界的典型落地。

## 实际应用

**搜索引擎自动补全**：当用户输入查询前缀时，系统在 Trie 中定位到前缀末尾节点，然后通过 DFS 或 BFS 收集该节点下所有 `is_end = True` 的路径，返回候选词列表。Google 搜索的实时补全功能背后使用了类似机制的变体结构。

**拼写检查与纠错**：Trie 结合编辑距离算法，可在 O(m × n) 内完成模糊匹配（m 为输入长度，n 为 Trie 中单词数量的替代参数）。具体实现是：在 Trie 上执行带容忍度的 DFS，每步允许最多 `k` 次字符替换、插入或删除，`k` 通常取 1 或 2。

**IP 地址路由（二进制 Trie）**：IPv4 的 CIDR 前缀匹配使用二进制 Trie，字符集缩减为 `{0, 1}`，树的深度最大为 **32**（IPv6 为 128）。路由器的最长前缀匹配（Longest Prefix Match）算法在二进制 Trie 上的时间复杂度固定为 O(32)，与路由表条目数量无关。

**AC 自动机（Aho-Corasick）**：这是 Trie 在多模式字符串匹配中的扩展，在 Trie 基础上增加失败指针（failure link），可在 O(n + m + z) 时间内完成文本 n 中所有 m 个模式的匹配，z 为匹配次数。微信内容审核、敏感词过滤系统广泛使用该算法。

## 常见误区

**误区一：认为 Trie 总是比哈希表快**。实际上，哈希表在单次精确查找时平均时间复杂度为 O(1)（假设哈希冲突极少），而 Trie 的 O(m) 需要逐字符遍历，当字符串很长（如 URL）时，Trie 的实际常数因子较大。Trie 的优势是**前缀查询**和**按字典序枚举**，这两项操作哈希表无法高效支持。

**误区二：忽略 Trie 的空间开销**。对于英文小写字母，每个节点持有 26 个指针，若每个指针占 8 字节（64 位系统），单个节点至少占用 **26 × 8 = 208 字节**。存储 10 万个平均长度为 8 的单词，最差情况下（无公共前缀）需要约 **800,000 个节点**，总内存超过 160MB。在内存受限场景下应优先考虑压缩 Trie 或双数组 Trie（Double-Array Trie）。

**误区三：混淆 `search` 与 `startsWith` 的语义**。一个常见的实现错误是：在 `search` 中遍历到路径末尾就返回 `True`，而忘记检查 `is_end`。这会导致查询 "app" 时，因 "apple" 的存在而错误返回命中，实际上 "app" 并未被插入。`is_end` 标记的正确维护是 Trie 正确性的关键保障。

## 知识关联

Trie 的节点本质上是一棵多叉树，因此**二叉树**中关于 DFS、BFS、递归遍历的所有技巧可以直接迁移。Trie 插入和查找的递归实现方式与二叉搜索树的节点插入逻辑高度相似，但分叉数量由 2 扩展到了 `|Σ|`。理解二叉树的指针操作和递归栈帧，是手写 Trie 的必要前提。

在 AI 工程的具体应用中，Trie 经常与**倒排索引**（Inverted Index）配合使用：Trie 负责前缀检索，定位候选词集合；倒排索引则完成从词到文档的映射。此外，Trie 也是构建**词典自动机（DAWG，Directed Acyclic Word Graph）**的基础，DAWG 在 Trie 基础上合并后缀相同的子树，使空间复杂度进一步从 O(n×m) 降至接近单词数量的线性规模，常用于移动端离线词典应用。
