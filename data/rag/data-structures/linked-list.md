# 链表

## 概述

链表（Linked List）是一种通过指针将独立内存节点串联起来的线性数据结构，其根本特征在于节点的物理存储不必连续，逻辑顺序完全由指针域（pointer field）维护。最基本的单链表节点包含两个域：存放业务数据的数据域（data field）和指向后继节点的指针域（next pointer）。这一看似简单的设计赋予了链表数组所不具备的动态伸缩能力——插入和删除节点无需搬移任何现有数据。

链表的历史起点可以精确追溯到 1955–1956 年：John McCarthy 在设计 Lisp 语言时将 cons cell 作为核心数据单元引入计算机科学，每个 cons cell 由 `car`（存储数据或指向子树）和 `cdr`（指向下一个 cons cell 的指针）组成，本质上就是一个二元链表节点（McCarthy, 1960）。与此同时，Allen Newell、Herbert Simon 和 Cliff Shaw 在开发 IPL（Information Processing Language）时独立设计了类似的链式存储机制，用于实现人工智能早期符号推理程序。此后，链表渗透进操作系统内核——Linux 内核 2.6 以来的 `include/linux/list.h` 中定义了 `struct list_head`，以循环双向链表管理进程描述符、文件系统挂载点等数千个内核对象，是工业级链表应用规模最大的公开代码库之一。

在数据结构体系中，链表是跳表、LRU 缓存、邻接表、哈希表的拉链法冲突链等高级结构的底层构件，其 O(1) 定点插删特性是这些结构实现高效操作的直接物质基础。

## 核心原理

### 节点结构与三种变体

**单链表**的节点定义（Python 伪代码）：

```python
class ListNode:
    def __init__(self, val=0):
        self.val  = val   # 数据域
        self.next = None  # 指针域，指向后继节点
```

**双向链表**（Doubly Linked List）在每个节点中额外维护 `prev` 指针，指向前驱节点。这使得在已知节点引用的前提下，向前遍历和原地删除均可在 O(1) 内完成，代价是每节点多耗费一个指针大小的内存（在 64 位系统上为 8 字节）。C++ STL 的 `std::list`、Java 的 `java.util.LinkedList` 均以双向链表实现。

**循环链表**（Circular Linked List）的尾节点 `next` 指针不指向 `nullptr`，而是回指头节点，形成闭环。操作系统的轮转调度（Round-Robin Scheduling）天然契合循环链表的语义，Linux 内核的进程调度就使用循环双向链表串联同一优先级的就绪态进程。

### 时间与空间复杂度分析

链表与数组最根本的性能差异由内存布局决定。数组元素在物理上连续存储，CPU 缓存行（cache line，通常为 64 字节）可一次预取多个相邻元素，顺序遍历具有极高的空间局部性；链表节点散布在堆内存的随机位置，顺序遍历会频繁产生缓存缺失（cache miss），实际常数因子远大于 O 符号所表达的理论值。

| 操作 | 单链表 | 双向链表 | 数组 |
|------|--------|----------|------|
| 头部插入/删除 | $O(1)$ | $O(1)$ | $O(n)$ |
| 已知指针处插入/删除 | $O(1)$ | $O(1)$ | $O(n)$ |
| 按索引随机访问 | $O(n)$ | $O(n)$ | $O(1)$ |
| 尾部追加（有尾指针） | $O(1)$ | $O(1)$ | $O(1)$ 均摊 |
| 查找特定值 | $O(n)$ | $O(n)$ | $O(n)$ |

链表头部插入只需执行 `new_node.next = head; head = new_node`，两步指针操作即完成，与链表长度 $n$ 完全无关；数组头部插入则需将全部 $n$ 个元素后移一位，这是 $O(1)$ 与 $O(n)$ 差距的直接机制。

### 哨兵节点（Dummy Node）技术

工程实践中，链表操作最常见的 bug 根源在于对头节点和空链表的特判逻辑。引入不携带业务数据的哨兵节点（Sentinel / Dummy Node）可彻底消除这一问题：令 `dummy.next = head`，所有插入、删除统一写作 `prev.next = curr.next`，无需分支判断被操作节点是否为真实头节点。LRU 缓存实现中，哨兵头节点和哨兵尾节点配合使用，使得"将节点移至链表头部"和"删除链表尾节点"的代码路径完全统一，避免空指针异常。

## 关键算法与公式

### Floyd 判圈算法（Floyd's Cycle Detection）

Robert Floyd 于 1967 年提出使用快慢双指针检测链表环的经典算法，又称"龟兔赛跑算法"（Knuth, 1997, *The Art of Computer Programming*, Vol. 2, Section 3.1）：

- **慢指针** $\text{slow}$：每步移动 1 个节点，$\text{slow} = \text{slow.next}$
- **快指针** $\text{fast}$：每步移动 2 个节点，$\text{fast} = \text{fast.next.next}$

**环存在的充要条件**：若链表存在长度为 $\lambda$ 的环，设环入口距头节点距离为 $\mu$，则快慢指针在慢指针走了 $\mu + k\lambda$（$k \geq 1$）步后必然在环内某点相遇。

**定位环入口**的关键等式推导如下：设相遇时慢指针共走 $d$ 步，快指针走了 $2d$ 步。快指针比慢指针多走的 $d$ 步恰好是环长的整数倍，即 $d = n\lambda$。此时将慢指针重置回头节点，快慢指针以相同速度（每步 1 格）同步前进，再次相遇点即为环入口。证明：从头节点到环入口距离为 $\mu$，从相遇点到环入口（沿环方向）距离同样为 $\mu$（可由 $d = n\lambda$ 推出），两者恰好在环入口相遇。

$$
\text{设相遇点距环入口距离为 } x,\quad \mu + x = d = n\lambda \Rightarrow x = n\lambda - \mu \equiv -\mu \pmod{\lambda}
$$

该算法时间复杂度 $O(n)$，空间复杂度 $O(1)$，是链表环检测的最优方案。

### 链表反转的迭代公式

将链表 $A \to B \to C \to D \to \text{None}$ 反转为 $D \to C \to B \to A \to \text{None}$，迭代算法维护三个指针 $\text{prev}$、$\text{curr}$、$\text{next}$，每步执行：

$$
\text{next} \leftarrow \text{curr.next},\quad \text{curr.next} \leftarrow \text{prev},\quad \text{prev} \leftarrow \text{curr},\quad \text{curr} \leftarrow \text{next}
$$

整体遍历一次链表即完成反转，时间复杂度 $O(n)$，空间复杂度 $O(1)$。递归反转的调用栈深度为 $O(n)$，在链表长度超过数万节点时存在栈溢出风险，工程实践中应优先选择迭代版本。

### 归并排序在链表上的优越性

归并排序（Merge Sort）在链表上的时间复杂度同样为 $O(n \log n)$，但其空间复杂度仅为 $O(\log n)$（递归栈），而无需数组归并所需的 $O(n)$ 辅助空间——合并两个有序链表只需修改指针，不涉及数据拷贝。快速排序在链表上则因无法 $O(1)$ 随机访问而使 partition 过程退化，因此**归并排序是链表排序的首选算法**（Cormen et al., 2009, *Introduction to Algorithms*, 3rd ed., Chapter 2.3）。

## 实际应用

### LRU 缓存：双向链表 + 哈希表

LRU（Least Recently Used）缓存要求 `get` 和 `put` 操作均在 $O(1)$ 内完成。单独使用双向链表可实现 $O(1)$ 的节点移动，但查找特定 key 需要 $O(n)$ 遍历；单独使用哈希表可 $O(1)$ 查找，但无法维护时序关系。两者结合：哈希表存储 `key → 链表节点指针` 的映射，双向链表按访问时序排列节点（最近访问的节点移至头部，链表尾部的节点为最久未使用），即可同时满足两个 $O(1)$ 要求。这是 LeetCode 第 146 题的标准解法，也是 Redis `maxmemory-policy allkeys-lru` 的底层思路（Redis 实际使用近似 LRU 以降低开销）。

例如：缓存容量为 3，操作序列为 `put(1,A) → put(2,B) → put(3,C) → get(1) → put(4,D)`。`get(1)` 后链表顺序变为 `[1,3,2]`，随后 `put(4,D)` 触发淘汰，尾节点 key=2 被驱逐，链表变为 `[4,1,3]`。整个过程每步仅涉及常数次指针操作。

### 跳表（Skip List）的底层链表层

跳表由 William Pugh 于 1990 年在论文《Skip Lists: A Probabilistic Alternative to Balanced Trees》中提出，其底层是一条包含所有元素的有序单链表，上方各层是通过随机化算法建立的"快速通道"链表。查找、插入、删除的期望时间复杂度均为 $O(\log n)$，最坏情况为 $O(n)$。Redis 的有序集合（Sorted Set）内部使用跳表而非红黑树，正是因为跳表的链表层天然支持高效的范围查询（`ZRANGEBYSCORE` 命令），且实现复杂度远低于平衡树。

### 操作系统内核中的链表

Linux 内核的 `struct list_head` 采用了一种"侵入式链表"（intrusive linked list）设计：链表节点嵌入在宿主结构体内部，而非持有指向宿主的指针。这避免了为每个链表节点单独分配内存，消除了一次内存分配的开销和潜在的内存碎片。通过 `container_of` 宏可从 `list_head` 指针反算出宿主结构体的地址，实现零额外内存的多链表复用。

## 常见误区

**误区一：链表一定比数组慢**。这一判断忽略了操作类型。对于频繁在中间位置插删的场景（如文本编辑器的行缓冲区、操作系统的任务队列），链表的 $O(1)$ 定点插删远优于数组的 $O(n)$ 移位。但对于以随机访问和顺序遍历为主的场景，数组因缓存行预取的空间局部性优势，实测吞吐量往往是链表的 3–5 倍（Drepper, 2007, *What Every Programmer Should Know About Memory*, Section 6.3.1）。

**误区二：双向链表一定优于单链表**。双向链表支持 $O(1)$ 反向遍历和无需前驱指针的原地删除，但每个节点多消耗 8 字节（64 位系统），且修改操作需同时维护 `prev` 和 `next` 两个指针，出错概率加倍。若业务场景不需要反向遍历，单链表是更轻量的选择。

**误区三：用链表实现栈或队列一定比数组实现高效**。链表实现的栈每次 `push` 涉及堆内存分配（`malloc`/`new`），其延迟不可预测且受内存碎片影响；基于数组的循环队列（ring buffer）在容量固定的场景下完全避免动态内存分配，在高频操作下吞吐量通常高出链表实现数倍。
