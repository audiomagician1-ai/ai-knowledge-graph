# 链表

## 概述

链表（Linked List）是一种通过指针将若干个独立内存节点串联起来的线性数据结构。与数组将元素存储在连续内存地址不同，链表的每个节点包含两个字段：**数据域**（存储实际值）和**指针域**（存储下一个节点的内存地址）。正因如此，链表可以在不连续的内存空间中动态扩展，无需预先分配固定大小。

链表的概念最早随 LISP 语言在 1958 年前后普及，LISP 的核心数据结构 cons cell 本质上就是一个单向链表节点，由 `car`（数据）和 `cdr`（指向下一节点的指针）构成。这一设计深刻影响了后续编程语言和数据结构的演化。

在 AI 工程领域，链表是实现 LRU 缓存、图的邻接表表示以及跳表索引的底层骨架。理解链表的指针操作和时间复杂度差异，直接决定了能否写出高效的缓存替换策略和图遍历算法。

## 核心原理

### 节点结构与内存模型

链表的最小单元是节点（Node），以单向链表为例，其结构定义如下：

```python
class ListNode:
    def __init__(self, val=0):
        self.val = val
        self.next = None  # 指向下一个节点，末尾节点指向 None
```

每个节点在堆内存中独立分配，节点之间的"连接"仅靠 `next` 指针维持。删除某节点时，只需将前驱节点的 `next` 跳过该节点指向其后继，原节点即成为垃圾回收候选，**不需要移动任何其他元素**。

### 时间复杂度对比

链表与数组的核心性能差异体现在以下操作上：

| 操作 | 数组 | 链表（单向） |
|------|------|-------------|
| 随机访问第 k 个元素 | O(1) | O(n) |
| 头部插入/删除 | O(n) | O(1) |
| 尾部插入（有尾指针）| O(1) | O(1) |
| 中间插入（已知位置）| O(n) | O(1) |
| 搜索特定值 | O(n) | O(n) |

链表**牺牲了随机访问能力（O(1) → O(n)）**，换取了任意位置插入/删除的 O(1) 时间（前提是已持有该位置的指针）。

### 链表的三种主要变体

**单向链表**：每个节点只有一个 `next` 指针，遍历只能从头到尾，反向访问需要 O(n)。

**双向链表**：每个节点同时持有 `prev` 和 `next` 两个指针，支持 O(1) 的双向移动。Python 标准库中的 `collections.OrderedDict` 底层即采用双向链表，用于维护插入顺序并支持 O(1) 的头尾移动操作——这正是 LRU 缓存的关键设施。

**循环链表**：尾节点的 `next` 指针指回头节点，而非 `None`。操作系统的进程调度（Round-Robin 算法）中常使用循环链表表示就绪队列，使调度器可以无限循环遍历所有进程。

### 关键指针操作技巧

处理链表问题时，**哨兵节点（Dummy Node）** 技术可以消除对头节点的特殊判断。在头部插入伪节点后，所有插入/删除操作都变为"操作某节点的后继"，代码分支减少约 30%。

**快慢指针**是链表特有的双指针技法：令快指针每次走 2 步，慢指针每次走 1 步。当快指针到达链表末尾时，慢指针恰好停在中间节点——这是检测环（Floyd 判圈算法）和寻找中间节点的标准方法，时间复杂度 O(n)，空间复杂度 O(1)。

## 实际应用

**LRU 缓存实现**：LRU（Least Recently Used）缓存淘汰策略要求每次访问后将节点移至"最近使用"端，并在容量满时淘汰"最久未使用"端的节点。使用**双向链表 + 哈希表**组合，可使 `get` 和 `put` 操作均达到 O(1)。哈希表存储 key 到链表节点的映射，双向链表维护访问顺序。Python 的 `functools.lru_cache` 装饰器底层即采用此结构，默认缓存大小为 128 个条目。

**图的邻接表表示**：在 AI 中构建知识图谱或神经网络计算图时，邻接表是稀疏图的首选存储方式。每个顶点对应一条链表，链表节点代表与该顶点相邻的边。相比邻接矩阵，邻接表将空间复杂度从 O(V²) 降至 O(V+E)，对于边数远少于顶点数平方的稀疏图效果显著。

**PyTorch 自动微分**：PyTorch 构建的计算图本质上是一个以张量为节点、以梯度函数为指针的链式结构。反向传播时，`grad_fn` 指针链从输出节点逐级追溯到叶子节点，这一逆向遍历机制正是链表指针语义的直接体现。

## 常见误区

**误区一："链表插入操作一定是 O(1)"**
这是最常见的错误认知。链表在**已持有目标位置指针**的前提下，插入才是 O(1)。若需先找到第 k 个位置再插入，寻址本身就要 O(k)，总体仍是 O(n)。相比之下，数组的中间插入是 O(n) 因为要移动元素，而不是因为寻址。两者的 O(n) 来源完全不同，不能混淆。

**误区二："链表比数组更省内存"**
实际上链表每个节点除存储数据外，还需额外存储 1 个（单向）或 2 个（双向）指针。在 64 位系统中，每个指针占 8 字节。若存储 n 个整数（各 4 字节），单向链表的内存开销是数组的 3 倍（4 + 8 = 12 字节/节点 vs 4 字节/元素）。链表的优势是**动态按需分配**，而非节省空间。

**误区三："循环链表检测可以用 visited 集合"**
用哈希集合记录已访问节点地址确实能判断环，但空间复杂度为 O(n)。Floyd 判圈算法（快慢指针）利用链表的指针特性，以 O(1) 空间解决同一问题，是链表场景下的最优解。将通用集合方法直接套用于链表，忽视了链表结构本身提供的遍历契机。

## 知识关联

**与数组的关系**：链表是数组的"对立互补"结构——数组以连续内存换取 O(1) 随机访问，链表以指针开销换取 O(1) 动态插删。理解这一 trade-off 是选择合适数据结构的前提：当场景需要频繁随机访问时用数组，需要频繁头部/中间插删时用链表。

**通向图结构**：图的邻接表直接由链表构成，每个顶点的邻居列表就是一条链表。掌握链表的指针遍历后，图的 DFS 和 BFS 中的邻居迭代逻辑将自然延伸——区别仅在于从线性遍历变为分支遍历。

**通向跳表**：跳表在单向链表基础上增加多级"快速通道"指针，使搜索复杂度从 O(n) 降至 O(log n)。Redis 的有序集合（Sorted Set）底层即用跳表实现，平均层数约为 log₁/ₚ n（p=0.25 时约为 log₄ n）。跳表是链表指针操作的直接进阶。

**支撑 LRU 缓存**：LRU 缓存是双向链表最典型的工程应用，也是面试高频题。理解双向链表的 `prev/next` 操作后，LRU 的节点移动逻辑（断开旧连接、重建新连接，共涉及 6 个指针赋值）将变得直观可操作。
