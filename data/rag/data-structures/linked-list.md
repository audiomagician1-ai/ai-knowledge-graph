# 链表

## 概述

链表（Linked List）是一种通过指针将独立内存节点串联起来的线性数据结构。与数组在内存中连续存储不同，链表的每个节点可以分散存储在内存的任意位置，节点之间靠指针域（pointer field）相互引用。最基本的链表节点包含两个域：存放数据的数据域（data field）和指向下一个节点的指针域（next pointer）。

链表由 John McCarthy 在 1955-1956 年开发 Lisp 语言时首次系统性引入计算机科学。Lisp 的核心数据结构 cons cell 本质上就是一个二元链表节点，由 car（数据）和 cdr（指针）组成。此后链表成为操作系统、编译器和数据库设计中的基础构件，例如 Linux 内核中的 `list_head` 结构广泛使用循环双向链表管理进程队列。

在 AI 工程的数据结构语境中，理解链表至关重要，因为许多高级结构——LRU 缓存、跳表、邻接表实现的图——底层都依赖链表节点的动态插删能力。链表的 O(1) 插入和删除特性是这些结构实现高效操作的直接原因。

## 核心原理

### 节点结构与指针组织

单链表节点的定义如下（以 Python 伪代码表示）：

```python
class Node:
    def __init__(self, data):
        self.data = data   # 数据域
        self.next = None   # 指针域，指向下一节点
```

双向链表节点额外增加 `prev` 指针，指向前驱节点，因此每个节点占用的指针空间是单链表的两倍。循环链表的尾节点的 `next` 指针不指向 `None`，而是指回头节点，形成闭环。

### 时间复杂度对比

链表与数组最根本的性能差异由其内存布局决定：

| 操作 | 链表 | 数组 |
|---|---|---|
| 头部插入/删除 | O(1) | O(n) |
| 任意位置插入/删除（已知指针）| O(1) | O(n) |
| 按索引随机访问 | O(n) | O(1) |
| 尾部追加（有尾指针） | O(1) | O(1)均摊 |

链表在头部插入只需更改头指针，无需移动任何现有元素；而数组头部插入需要将 n 个元素全部后移一位，这是两者 O(1) 与 O(n) 差距的直接根源。

### 哨兵节点（Dummy Node）技术

为了消除链表操作中对"空链表"或"头节点"的特判逻辑，工程实践中常引入一个不存储业务数据的哨兵节点作为链表头。删除节点时统一执行 `prev.next = curr.next`，无需单独判断被删节点是否为真实头节点。LeetCode 链表题目约 60% 以上可通过引入哨兵节点大幅简化代码。

### 快慢指针（Floyd 判圈算法）

Floyd 在 1967 年提出用两个速度不同的指针检测链表中是否存在环：慢指针每步走 1 个节点，快指针每步走 2 个节点。若链表存在环，则快慢指针必然在环内相遇；若不存在环，快指针会先到达 `None`。相遇后，将慢指针重置到头节点，两指针同速前进，再次相遇点即为环的入口节点。该算法空间复杂度为 O(1)，是链表环检测的标准方法。

## 实际应用

**LRU 缓存实现**：LRU（Least Recently Used）缓存要求以 O(1) 时间完成查找、插入和删除。标准实现使用哈希表配合双向链表：哈希表提供 O(1) 的键值查找，双向链表维护访问时序。每次访问某个缓存项时，将对应节点从链表中间摘除并移至链表头部，这一"摘除+头插"操作因为有前驱指针（`prev`）的存在，仅需 O(1) 时间。Python 标准库中的 `collections.OrderedDict` 底层即用此结构实现。

**深度学习框架的计算图**：PyTorch 的 autograd 引擎在反向传播时使用链式结构存储梯度函数（grad_fn）。每个张量的 `grad_fn` 属性指向生成它的操作节点，该节点再通过 `next_functions` 元组指向输入张量的梯度函数，形成单向链式结构，与单链表的遍历逻辑完全一致。

**内存池管理**：操作系统的空闲内存块通常用单链表或双向链表组织，称为空闲链表（free list）。dlmalloc 分配器将不同大小的空闲块按大小分级，每级维护一条链表，`malloc` 时从合适大小的链表头取节点，`free` 时将节点重新插回对应链表头部，两个操作均为 O(1)。

## 常见误区

**误区一：认为链表的内存使用比数组更省**

链表每个节点除了存储业务数据外，还需额外存储 1 至 2 个指针（在 64 位系统上每个指针占 8 字节）。存储 n 个整数（4 字节/个）的单链表，指针开销是数据本身的 2 倍。同时，链表节点分散在堆内存各处，CPU 缓存（cache line 通常为 64 字节）无法在一次加载中覆盖多个节点，导致缓存命中率远低于数组，实际访问性能可能比数组慢 3-5 倍。

**误区二：双向链表一定比单链表更优**

双向链表允许 O(1) 删除任意已知节点和从尾部逆向遍历，但代价是每个节点增加 8 字节指针空间，且每次插入/删除需维护两个方向的指针（4 条赋值语句而非 2 条），稍有疏漏即造成指针断链。在只需从头到尾单向遍历的场景（如哈希表的链地址法冲突链）中，单链表更轻量，强行使用双向链表是过度设计。

**误区三：链表头插法总是安全的**

在多线程环境下，两个线程同时对同一链表执行头插法会导致经典的"ABA 问题"：线程 A 读取头节点为 X，被挂起；线程 B 删除 X 并插入 Y，再插入 X；线程 A 恢复后 CAS 操作认为头节点仍为 X 而成功，但链表结构已被破坏。Java 的 `HashMap` 在 JDK 1.7 中使用链表头插法处理扩容，在并发场景下正是因此产生了死循环 bug，JDK 1.8 改为尾插法修复了该问题。

## 知识关联

**前置知识——数组/列表**：理解链表的关键前提是熟悉数组的连续内存模型。数组的随机访问公式为 `address = base_address + index × element_size`，正是因为数组需要保持这种连续性，插入和删除才必须移动元素；而链表彻底放弃了这种连续性，用指针换取了插删的灵活性，两者是一组对偶的设计取舍。

**后续概念——跳表**：跳表在单链表基础上增加多层索引链表（第 1 层每 2 个节点抽取 1 个，第 2 层再抽取 1 个，以此类推），使查找复杂度从 O(n) 降至期望 O(log n)，同时保留了链表 O(1) 的插入删除特性。Redis 中有序集合（Sorted Set）的底层实现之一即为跳表，节点层数按概率 p=0.25 随机生成，最大层数限制为 32 层。

**后续概念——图（邻接表表示）**：图的邻接表表示中，每个顶点对应一条链表，链表中存储该顶点的所有邻居节点。对于稀疏图（边数远小于 V²），邻接表比邻接矩阵节省 O(V²) 空间，且遍历某顶点的所有邻居时间为 O(度数) 而非 O(V)。

**后续概念——LRU 缓存**：如上文"实际应用"中所述，LRU 缓存是双向链表与哈希表组合使用的经典范例，是链表 O(1) 插删特性在工程中最直接的落地场景，也是大厂面试中高频考察链表综合运用能力的题型（对应 LeetCode 第 146 题）。
