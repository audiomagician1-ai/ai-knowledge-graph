# 稀疏表（Sparse Table）

## 概述

稀疏表是一种专为静态数组上**区间最值查询（Range Minimum/Maximum Query，RMQ）**设计的数据结构，通过 O(n log n) 的预处理空间和时间换取 O(1) 的单次查询效率。它的核心思路是利用"任意区间可以被两个长度为 2 的幂次方的子区间覆盖"这一特性，将区间最值的计算分解为两个已预计算结果的重叠查询。

稀疏表由 Bender 和 Farach-Colton 在2000年的论文中系统化用于 RMQ 问题，但其思想源于更早的倍增（Binary Lifting）技巧。它在算法竞赛（如 ACM-ICPC）和工业系统（如数据库查询优化、时间序列分析）中被广泛使用，因为它是已知解决静态 RMQ 问题在**常数级查询**方面最简洁的方案——相比线段树的 O(log n) 查询，稀疏表在查询频繁、数据不修改的场景下有数量级的优势。

稀疏表仅适用于满足**可重叠性（Overlap-friendly）**的操作，即对同一元素重复计算不影响结果。最小值（min）、最大值（max）、最大公约数（gcd）满足此性质，而求和（sum）则不满足，这直接决定了稀疏表的使用边界。

## 核心原理

### 预处理：倍增填表

稀疏表用二维数组 `st[i][j]` 存储从下标 `i` 开始、长度为 `2^j` 的子数组的最小值：

$$\text{st}[i][j] = \min(A[i],\ A[i+1],\ \ldots,\ A[i + 2^j - 1])$$

递推关系为：

$$\text{st}[i][j] = \min\bigl(\text{st}[i][j-1],\ \text{st}[i + 2^{j-1}][j-1]\bigr)$$

含义是：长度为 `2^j` 的区间可以拆成左半段 `[i, i+2^{j-1}-1]` 和右半段 `[i+2^{j-1}, i+2^j-1]` 两个长度为 `2^{j-1}` 的子区间，各取已存好的最小值再取 min。

预处理的外层循环遍历 `j` 从 1 到 `⌊log₂n⌋`，内层遍历起点 `i` 从 0 到 `n - 2^j`，总计算量为 O(n log n)。对于 n = 10⁶ 的数组，`j` 最大取到 19（因为 2¹⁹ = 524288 < 10⁶ < 2²⁰），预处理耗时约 2000 万次操作，实践中极快。

### 查询：两段重叠覆盖

对于查询区间 `[l, r]`，令 `k = ⌊log₂(r - l + 1)⌋`，则：

$$\text{query}(l, r) = \min\bigl(\text{st}[l][k],\ \text{st}[r - 2^k + 1][k]\bigr)$$

两个子区间 `[l, l+2^k-1]` 和 `[r-2^k+1, r]` 合并后完整覆盖 `[l, r]`，且由于 min 操作的可重叠性，中间可能重复计算的元素不影响正确性。整个查询**不涉及任何循环**，仅需一次对数计算和两次表查找，时间复杂度严格 O(1)。

`k` 的计算可以用 `__builtin_clz`（GCC内置函数）实现：`k = 31 - __builtin_clz(r - l + 1)`，避免 `log` 函数的浮点误差，在工程实现中更安全。

### 空间布局与缓存友好性

稀疏表的二维数组大小为 `n × (⌊log₂n⌋ + 1)`，对于 n = 10⁵，需约 `10⁵ × 17 ≈ 170` 万个整数，占用约 6.8 MB（int32）。在 C++ 中推荐将 `j`（幂次）作为**第一维**，即 `st[j][i]`，使查询时两次访问 `st[k][l]` 和 `st[k][r-2^k+1]` 位于同一行，提升 CPU 缓存命中率，实测可加速 20%-30%。

## 实际应用

**在线 RMQ 系统（时间序列监控）**：假设一个监控平台存储了 24 小时内每秒的 CPU 使用率（共 86400 个数据点），需要频繁响应"过去某段时间内的峰值"查询。使用稀疏表，预处理一次（约 86400 × 17 步），此后每次查询均为 O(1)，完全满足毫秒级响应需求，而线段树每次查询需 log₂(86400) ≈ 17 步比较。

**LCP 数组上的最近公共祖先（Suffix Array + LCP + RMQ）**：后缀数组（Suffix Array）中，两个后缀的最长公共前缀（LCP）等于其在 LCP 数组上某段区间的最小值。对 LCP 数组建稀疏表后，任意两后缀的 LCP 查询变为 O(1)，这是字符串搜索和文本相似度计算的核心加速手段，广泛应用于 DNA 序列比对工具（如 BWA）中。

**数据库范围查询优化**：在列式存储数据库（如 Apache Parquet 格式的查询引擎）中，对不更新的分区数据建立稀疏表索引，可将 `MIN/MAX` 聚合查询从全扫描 O(n) 降至 O(1)。

## 常见误区

**误区一：认为稀疏表可以支持动态更新**
稀疏表**只能处理静态数组**。一旦数组中任意元素发生修改，整张 `st` 表的受影响行均需重新计算，最坏情况下代价仍是 O(n log n)，丧失了增量更新的意义。需要支持单点修改+区间查询时，应改用线段树（O(log n) 更新+查询）或 BIT（Binary Indexed Tree）。稀疏表的价值严格局限于"构建一次、查询多次"的场景。

**误区二：认为所有可交换结合运算都适用稀疏表**
稀疏表的正确性依赖**幂等性（Idempotency）**：`f(x, x) = x`，而不仅仅是交换律或结合律。区间求和 `sum(l, r)` 不满足幂等性——两个重叠区间相加会重复累计中间部分，导致结果错误。`min`、`max`、`gcd` 均满足幂等性，`sum`、`product` 则不满足，后者必须用前缀数组（prefix sum）处理。

**误区三：混淆稀疏表与跳表（Skip List）**
二者名称相近但毫无关联。跳表是一种用于有序集合的动态查找数据结构，基于随机化层级指针；稀疏表是静态 RMQ 的解决方案，基于确定性的倍增预处理。仅凭"稀疏"二字混淆两者是对其各自结构和目标的根本误解。

## 知识关联

**动态规划**是稀疏表预处理的直接方法论基础——`st[i][j]` 的递推公式 `st[i][j] = min(st[i][j-1], st[i+2^{j-1}][j-1])` 正是区间 DP 在倍增方向上的具体应用，状态定义为"起点+长度的幂次"，转移方向沿 `j` 递增。

**二分查找**与稀疏表的查询逻辑共享"将区间长度表示为2的幂次"的思想：`k = ⌊log₂(r-l+1)⌋` 的计算本质上是找到不超过区间长度的最大幂次，这与二分查找中不断折半的边界分析方式直接同源。理解二分查找的边界推导，有助于正确计算 `k` 并验证两段覆盖的完整性（需保证 `2^k ≤ r-l+1`，即两段合并后不短于原区间）。

稀疏表是 RMQ 问题族的终点之一：从朴素 O(n) 查询，到线段树 O(log n)，再到 Sparse Table O(1)（静态），最终到 Farach-Colton & Bender 算法实现 O(n) 预处理 + O(1) 查询的理论最优，稀疏表处于这条路径的关键节点，是工程实践中性能与实现复杂度最佳平衡点。
