# 稀疏表（Sparse Table）

## 概述

稀疏表是一种专为**静态数组区间最值查询（Range Minimum/Maximum Query，RMQ）**设计的数据结构，通过 O(n log n) 的预处理空间和时间，将单次查询的时间复杂度降至 O(1)。其核心思想是利用"任意区间可被两个等长的2的幂次区间覆盖"这一性质，从而在查询时直接返回两个预计算子区间的最值，无需遍历或递归。

稀疏表的理论基础来自于 20 世纪 70~80 年代对 LCA（最近公共祖先）和 RMQ 问题的研究。Bender 与 Farach-Colton 在 2000 年的论文中将 RMQ 与 LCA 的线性时间互规约正式化，使稀疏表成为 RMQ 的经典解法之一。与线段树相比，稀疏表不支持动态修改，但其查询无需任何递归或栈操作，在 CPU 缓存命中率方面表现更优，因此在竞赛和工程中的只读场景下被广泛采用。

稀疏表之所以重要，在于它展示了"用空间换时间"与"幂次覆盖"两种思维的极致结合：一旦数组确定不变，任意区间最值都可在常数时间内回答，这对需要海量 RMQ 查询的 AI 特征工程（如滑动窗口统计）和时间序列分析具有实际价值。

## 核心原理

### 1. 预处理：构建二维稀疏表

定义二维数组 `st[i][j]`，表示**从下标 i 开始、长度为 2^j 的子数组的最小值**（以区间最小值为例）：

$$
\text{st}[i][j] = \min\bigl(\text{st}[i][j-1],\ \text{st}[i + 2^{j-1}][j-1]\bigr)
$$

- **i**：子区间起始下标，范围 0 到 n-1
- **j**：表示区间长度的幂次，范围 0 到 ⌊log₂n⌋
- **边界条件**：`st[i][0] = arr[i]`（长度为1的子区间即数组本身）

预处理过程按 j 从小到大迭代，j 每增加 1，区间长度翻倍，总共需要填充约 n × ⌊log₂n⌋ 个格子。对于 n = 10⁶ 的数组，⌊log₂n⌋ ≈ 20，因此预处理空间约为 2000 万个整数，内存占用约 80 MB（每个 int 4 字节），工程中需提前评估。

### 2. 查询：O(1) 的区间覆盖技巧

查询区间 [l, r] 的最小值时，令 k = ⌊log₂(r - l + 1)⌋，则：

$$
\text{query}(l, r) = \min\bigl(\text{st}[l][k],\ \text{st}[r - 2^k + 1][k]\bigr)
$$

两个长度均为 2^k 的子区间分别从 l 和 r-2^k+1 出发，**二者合并后必然完整覆盖 [l, r]**，且允许重叠。这种"可重叠子区间"的技巧是稀疏表区别于线段树的本质：因为求最值的操作具有**幂等性**（即对同一元素重复取最值不改变结果），所以区间重叠不会引入误差。

k 值可通过预先构建 log 表（`log_table[i] = ⌊log₂i⌋`）在 O(1) 时间内查到，整个查询过程只需 2 次数组访问和 1 次比较，无任何分支跳转。

### 3. 幂等性约束：稀疏表的使用边界

稀疏表的 O(1) 查询依赖于操作的幂等性：`f(f(a, b), b) = f(a, b)`。**区间最大值、区间最小值、区间按位与、区间按位或**均满足此性质，故可使用稀疏表。但**区间求和**不满足幂等性（重叠部分会被重复计算），因此稀疏表**不能**用于区间求和，此类场景应使用前缀和或线段树。

## 实际应用

**案例一：时间序列滑动窗口极值**  
在 AI 特征工程中，常需对传感器时序数据计算任意时间段内的温度峰值或谷值。使用稀疏表对长度 n = 10⁶ 的序列预处理后，10⁸ 次 RMQ 查询可在约 1 秒内完成，而朴素遍历需要约 100 秒。

**案例二：LCA 问题的 RMQ 转化**  
树上最近公共祖先（LCA）可通过欧拉序将问题转化为 RMQ：对树做欧拉序遍历后得到长度为 2n-1 的深度序列，在该序列上建立稀疏表，即可 O(1) 回答任意两节点的 LCA 查询。这是稀疏表在图算法中的经典应用。

**案例三：字符串算法中的后缀数组**  
在后缀数组（Suffix Array）求最长公共前缀（LCP）时，LCP(i, j) 等价于 height 数组某区间的最小值，使用稀疏表后每次 LCP 查询从 O(n) 降至 O(1)，是字符串匹配引擎的重要优化手段。

## 常见误区

**误区一：认为稀疏表可以处理动态更新**  
稀疏表的 `st[i][j]` 一旦构建完成，若数组中任意元素发生修改，最坏情况下需要重新预处理全部 O(n log n) 个格子。稀疏表**严格适用于静态数组**；若需支持单点更新，应改用线段树（单次更新 O(log n)）或 Fischer-Heun 结构（支持 O(1) 更新的理论方案）。

**误区二：将 O(1) 查询误解为不需要预计算 k 值**  
查询公式中的 k = ⌊log₂(r - l + 1)⌋ 若每次用 `(int)(log2(r-l+1))` 浮点函数计算，会引入约 50~200 ns 的浮点运算开销，且存在精度风险（如 log₂(8) 可能返回 2.9999...）。正确做法是在预处理阶段用递推公式 `log_table[i] = log_table[i/2] + 1` 建立整数 log 表，查询时直接 O(1) 查表。

**误区三：混淆稀疏表与跳表（Skip List）**  
两者名称相近但完全不同：跳表是一种基于概率的有序链表，支持 O(log n) 的插入、删除、搜索；稀疏表是基于2的幂次区间划分的静态只读结构，专用于 RMQ，不支持任何修改操作，也不维护元素顺序。

## 知识关联

**依赖动态规划的状态转移思想**：稀疏表的预处理公式 `st[i][j] = min(st[i][j-1], st[i+2^(j-1)][j-1])` 是一个典型的二维 DP 递推，其中 j 为阶段变量，每一阶段的状态由上一阶段的两个子问题合并而来，无后效性正是该递推成立的前提。

**与二分查找共享"对半分"的幂次思维**：二分查找每次将搜索范围缩小一半，稀疏表的区间划分则将长度扩大一倍，两者都建立在 2 的幂次对数级别迭代的基础上。理解二分查找中"k 次迭代覆盖 2^k 范围"的思路，有助于快速建立稀疏表预处理中 j 层含义的直觉。

**向更复杂 RMQ 结构的延伸**：稀疏表的 O(n log n) 空间在 n 极大时（如 n = 10⁸）会成为瓶颈。Fischer-Heun Structure（2006）将空间降至 O(n)、查询仍为 O(1)，其核心是将稀疏表应用于"块间最值"，再对块内进行笛卡尔树分类，是稀疏表的直接理论延伸。
