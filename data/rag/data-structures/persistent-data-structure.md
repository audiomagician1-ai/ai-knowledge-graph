# 持久化数据结构

## 概述

持久化数据结构（Persistent Data Structure）是一类在修改操作后仍能保留其所有历史版本的数据结构。与普通的"临时性"（ephemeral）数据结构不同，对持久化数据结构执行任何修改，不会销毁旧版本，而是创建一个新版本，新旧版本均可被独立访问。这一特性在函数式编程语言中天然契合"不可变性"（immutability）原则，也因此在ML系统的特征存储、实验追踪以及版本化模型参数等场景中具有重要工程价值。

持久化数据结构的理论基础由James R. Driscoll、Neil Sarnak、Daniel D. Sleator和Robert E. Tarjan于1986年在论文《Making Data Structures Persistent》中系统建立。该论文提出了将任意临时数据结构转化为持久化结构的通用方法，并证明了在时间和空间上均可以达到接近O(log n)的操作复杂度。Clojure语言的创始人Rich Hickey在2007年将持久化向量和持久化哈希映射引入Clojure标准库，使该概念在工业界获得广泛应用。

在AI工程场景下，持久化数据结构解决了一个实际痛点：当需要对同一份数据集的多个变体进行并行实验时，简单的深拷贝会带来O(n)的内存开销。持久化数据结构通过**结构共享**（structural sharing）将大部分历史版本的节点复用，使每次"修改"的额外内存开销降低到O(log n)级别。

## 核心原理

### 路径复制（Path Copying）

路径复制是实现持久化树结构最基础的技术。以持久化二叉搜索树为例，当修改某个叶节点时，不修改原节点本身，而是沿从根节点到目标节点的路径，依次创建各节点的副本，最终生成一棵新树。新树与旧树共享所有**未被修改路径上的节点**。

对于一棵高度为h的平衡二叉树，每次修改仅复制h个节点。若树包含n个节点，h = O(log n)，因此单次修改的空间开销为O(log n)，而非O(n)。以一棵包含100万个节点的平衡二叉树为例，每次修改仅需复制约20个节点，结构共享率超过99.998%。

### 胖节点（Fat Node）方法

胖节点方法是另一种实现持久化的策略：不复制路径，而是在每个节点内部存储一个**修改记录列表**（modification log），每条记录包含字段名、新值和版本号。查询时，根据目标版本号从节点的修改列表中检索最新的有效值。

胖节点方法的版本查询时间复杂度为O(log v)，其中v是版本总数（使用二分查找修改列表），而写入每个新版本仅需O(1)的额外存储（向对应节点追加一条修改记录）。Driscoll等人1986年的论文证明，胖节点方法在摊销意义下的时间复杂度与原结构相同。

### 持久化向量与HAMT

Clojure的持久化向量基于32叉树（branching factor = 32）实现，这意味着树高仅为⌈log₃₂(n)⌉。对于包含10亿个元素的向量，树高仅为6层。每次在索引i处修改一个元素，需复制的节点数等于树高，即最多6个节点。

持久化哈希映射则基于**哈希数组映射字典树**（Hash Array Mapped Trie，HAMT）实现。HAMT使用键的哈希值逐5位（5 bits per level）作为索引，在每层最多分出32个分支。节点内部使用一个32位位图（bitmap）记录哪些子节点实际存在，避免为稀疏节点分配全部32个指针槽位，从而节省内存。修改一个键值对的额外空间为O(log₃₂ n)个新节点。

## 实际应用

**AI实验追踪中的特征版本管理**：在特征工程阶段，数据科学家频繁对同一份特征表进行不同预处理（归一化、分桶、填充策略各异）。使用持久化哈希映射存储特征配置，每个实验分支仅需存储与基准配置差异的O(log n)个新节点，而不是复制整张特征表。Delta Lake和Apache Iceberg的快照隔离机制正是借鉴了这一原理，通过维护元数据的持久化树来实现表的时间旅行查询（time-travel query）。

**强化学习中的状态树搜索**：蒙特卡洛树搜索（MCTS）在探索博弈树时，需要同时维护数千条从根节点出发的不同路径状态。若使用持久化树结构存储游戏状态，在每个节点处"分叉"时，两条子路径自动共享父节点以上的所有历史状态，无需显式拷贝。AlphaGo的早期实现中即使用了类似的共享状态树来优化内存占用。

**不可变计算图与自动微分**：PyTorch的`autograd`机制在构建计算图时，每个张量操作生成一个新的图节点而不修改已有节点，整张计算图是隐式的持久化有向无环图（DAG）。当调用`.detach()`或在`no_grad()`上下文中操作时，实质上是创建了图的一个"新版本"，与原图共享所有叶节点（即模型参数张量）。

## 常见误区

**误区一：持久化等同于不可变，不可变等同于持久化**。不可变（immutable）仅意味着对象创建后不能被修改；持久化则进一步要求所有历史版本均可访问。Python中的`tuple`是不可变的，但它不是持久化数据结构——你无法访问它"修改前"的某个版本，因为每次创建的都是全新对象，旧对象若没有引用则被垃圾回收。持久化数据结构必须维护一个**版本索引**（version index），使任意历史版本均可被检索。

**误区二：路径复制导致内存泄漏，必须手动管理版本**。由于持久化数据结构依赖结构共享，节点的生命周期由引用计数或垃圾回收管理，当某个版本不再被任何外部引用持有时，该版本独有的节点会被自动回收。Clojure的持久化集合完全由JVM的GC管理，无需手动释放旧版本。然而，若将所有版本的根节点存入一个永久列表，则所有版本均不会被GC回收，**版本积累导致的内存增长是使用方的责任，而非结构本身的泄漏**。

**误区三：持久化数据结构在所有操作上均比可变结构慢**。持久化向量的随机访问复杂度为O(log₃₂ n)，对于n = 10⁹时仅需6次指针跳转，而基于哈希表的可变映射理论上是O(1)但存在缓存未命中。在多线程AI推理服务场景中，持久化结构因消除了锁竞争，实际吞吐量有时反而高于加锁的可变结构。

## 知识关联

**与二叉树的关系**：路径复制技术直接构建在二叉搜索树的路径概念上。理解从根到叶的路径长度为O(log n)是推导持久化操作复杂度的前提。红黑树的持久化版本（Okasaki在《Purely Functional Data Structures》1998年书中给出的实现）在插入时最多复制O(log n)个节点，且旋转操作同样通过路径复制实现，无需修改任何已有节点。

**与函数式编程的关系**：函数式编程中的纯函数（pure function）要求相同输入产生相同输出且无副作用，这直接要求函数操作的数据结构不能被原地修改。持久化数据结构是在函数式环境中实现高效数据操作的唯一可行方案——若每次操作均进行完整拷贝，则列表操作的空间复杂度退化为O(n²)。Haskell的`Data.Map`模块使用的AVL树变体和Clojure的所有核心集合均是持久化结构，这使得纯函数式代码可以"修改"数据而不违反不可变性约束。
