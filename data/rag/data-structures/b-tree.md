# B树与B+树

## 概述

B树（B-tree，注意不是"B减树"）由Rudolf Bayer和Edward McCreight于1972年在波音研究实验室提出，最初用于解决磁盘文件系统中大规模数据的高效检索问题。其核心设计动机是：当数据量超过内存容量时，AVL树或红黑树每次比较都可能触发一次磁盘I/O，而磁盘寻道时间（约10ms量级）比内存访问慢10万倍以上，导致树高成为性能瓶颈。B树通过让每个节点存储多个键值（即"多路"特性），将树高从O(log₂N)压缩至O(log_m N)，其中m为节点的最大子节点数（阶数），大幅减少磁盘访问次数。

B+树是B树的变体，在数据库领域（尤其是MySQL InnoDB、PostgreSQL等主流引擎）中比B树应用更为广泛。两者的根本区别在于：B树的每个节点（包括内部节点）都存储完整数据记录，而B+树的内部节点仅存储键（索引），完整数据只存储在叶子节点中，且所有叶子节点通过双向链表串联。这一设计差异直接决定了两者在范围查询和全表扫描场景下截然不同的性能表现。

## 核心原理

### B树的结构定义与约束

一棵m阶B树满足以下精确约束：每个节点最多有m棵子树，最少有⌈m/2⌉棵子树（根节点除外，根节点至少有2棵子树）；每个节点最多包含m-1个键，最少包含⌈m/2⌉-1个键。对于一个含有k个键的节点，恰好拥有k+1棵子树，且子树中的所有键值严格满足：左子树 < 当前键 < 右子树。以5阶B树为例，每个非根内部节点至少存2个键、最多存4个键，至少3个子节点、最多5个子节点。B树的高度上界为 h ≤ log₍⌈m/2⌉₎((N+1)/2)，其中N为总键数。

### B+树与B树的关键差异

B+树在B树基础上做出三个结构性修改：第一，**内部节点不存储数据**，只存储键的副本作为路由索引，因此相同大小的磁盘页（通常为4KB或16KB）能放入更多键，使树更"矮胖"；第二，**所有数据记录只存在叶子节点**，且叶子节点包含所有键（而B树的键分散在各层）；第三，**叶子节点通过指针串成有序双向链表**，支持O(1)代价的相邻叶节点跳转。以MySQL InnoDB默认16KB页为例，若键为8字节整数、指针6字节，单个内部节点可存约1170个键，三层B+树可索引约1170² × 叶节点记录数 ≈ 数千万条记录。

### 插入与分裂机制

B树和B+树的插入都遵循"下溢预防"逻辑：向下查找叶节点，在叶节点插入键后，若节点键数超过m-1，则触发**分裂**：将节点的中间键（第⌈m/2⌉个键）上推至父节点，左右各保留一半键。B树分裂时中间键直接上升到父节点并从原节点删除；B+树分裂时中间键**同时保留**在新右节点中（因为叶节点需存所有键），上推的只是键的副本。这导致B+树的内部节点键集合是叶节点键集合的子集，而B树则不然。分裂可能向上递归传播至根，若根也满了，则创建新根，树高加1——这是B树/B+树高度增长的唯一方式。

### 范围查询效率对比

B树执行范围查询`WHERE id BETWEEN 100 AND 200`时，必须通过中序遍历（In-order traversal）在树中来回跳转，每次跨节点访问都可能是一次随机磁盘I/O，最坏情况需要访问O(k log N)个节点（k为结果集大小）。B+树只需：①定位起始叶节点（O(log N)次I/O）；②沿叶节点链表顺序扫描至终止键（O(k/m)次I/O，因为链表访问是顺序I/O，硬盘顺序读吞吐量可达随机读的100倍以上）。这一差距在OLAP场景中极为显著。

## 实际应用

**MySQL InnoDB聚簇索引**：InnoDB的主键索引就是一棵B+树，叶节点直接存储完整行数据（聚簇）。辅助索引的叶节点存储主键值而非行地址，查询辅助索引后需"回表"再查一次主键B+树——这解释了为何InnoDB建议主键尽量短小（如INT比UUID更优，节省辅助索引存储和回表开销）。

**文件系统目录索引**：Linux的ext4、XFS文件系统使用B树变体（HTree）管理大目录下的文件名查找；NTFS的MFT（Master File Table）同样采用B+树组织文件记录，支持对含数十万文件的目录进行O(log N)级别的查找。

**MongoDB的WiredTiger引擎**：WiredTiger从3.2版本起使用B+树作为默认存储结构，其B+树页大小默认为4KB，在内存压力下会触发page eviction机制将B+树节点写回磁盘，验证了B+树节点粒度与OS页缓存（Page Cache）配合的高效性。

## 常见误区

**误区一："B-树"是"B减树"**。"-"在英文原名B-tree中只是连字符，中文写法"B树"才是正确表达。"B减树"这一说法在中文教材中广泛流传，导致初学者误以为B树与B+树是"减法"关系，实际上两者是独立命名的不同结构。

**误区二：B树比B+树更通用，B+树只是优化版**。B树在某些场景下反而更优：当查询几乎全为等值查询（point query）且数据与键一起存储可减少一次I/O（不需要"到叶节点取数据"）时，B树有微弱优势。MongoDB早期版本（WiredTiger之前的MMAPv1引擎）就使用了接近B树语义的结构。选择B+树还是B树取决于读写模式，不存在绝对优劣。

**误区三：B+树叶节点链表使遍历复杂度为O(1)**。叶节点链表让**相邻叶节点间跳转**成为O(1)指针追踪，但整个范围查询的总复杂度仍是O(log N + k)（N为总记录数，k为结果集大小），O(log N)的树搜索开销不可消除。单纯认为"有链表就快"忽略了定位起始节点这一必要步骤。

## 知识关联

从二叉搜索树（BST）到AVL树，再到B树/B+树，这条演进路径的主线是：BST解决有序查找问题，AVL树通过旋转保证O(log N)高度，但两者都是二叉结构，每个节点只有1个键和2个子树，在磁盘场景下每层对应一次I/O，树高过大。B树的本质是将AVL树的平衡思想推广到m路情形：同样维护严格的高度平衡（所有叶节点在同一层），但通过增大分支因子m（通常取100至1000量级）使树高降至2~4层，将磁盘访问次数控制在个位数。掌握B+树之后，自然延伸至**LSM树**（Log-Structured Merge Tree，RocksDB/LevelDB所用），LSM树用顺序写换随机写，与B+树的随机写友好型设计形成对比，两者分别适合写密集型与读密集型负载——理解这一权衡是数据库存储引擎选型的基础判断力。
