# 跳表

## 概述

跳表（Skip List）是由William Pugh于1990年在论文《Skip Lists: A Probabilistic Alternative to Balanced Trees》中提出的一种基于有序链表的概率性数据结构。它通过在原始链表上建立多层"快速通道"（索引层），使得查找、插入、删除操作的平均时间复杂度均达到O(log n)，与平衡二叉树相当，但实现复杂度远低于红黑树或AVL树。

跳表的核心思想来自于"二分查找"在链表上的推广。普通有序链表无法实现O(log n)的查找，因为链表不支持随机访问。跳表通过给部分节点附加多级"向上指针"，使得遍历时可以跳过大量节点，从而模拟二分查找的效率。Redis的有序集合（Sorted Set）底层即使用跳表实现，是该数据结构在工业界最广为人知的应用案例。

跳表之所以在AI工程的数据结构选型中受到重视，原因在于它支持高效的范围查询（range query）——给定区间[a, b]，跳表可以在O(log n + k)时间内返回所有k个满足条件的元素，这一特性在向量数据库的近似最近邻搜索候选集筛选和特征存储系统中具有实际价值。

## 核心原理

### 多层索引结构

跳表由多层链表叠加构成，最底层（第0层）是完整的有序链表，包含全部n个元素。第1层仅保留约n/2个节点，第2层保留约n/4个节点，以此类推，直到最顶层通常只剩2~3个节点。每个节点在第i层拥有一个指向同层下一个节点的指针`forward[i]`。

标准跳表的期望层高为`L(n) = log_{1/p}(n)`，其中p为每个节点晋升到上一层的概率，通常取p=0.5或p=0.25。Redis的跳表实现将最大层数（ZSKIPLIST_MAXLEVEL）设定为64，p取0.25，在实际工作负载下层数期望约为16。

### 概率性晋升机制

节点的层数不是由排序算法决定的，而是在插入时通过抛硬币（随机数）确定。伪代码如下：

```
randomLevel():
    level = 1
    while random() < p AND level < MAXLEVEL:
        level += 1
    return level
```

这一机制使跳表无需在插入/删除后进行旋转或再平衡操作，直接规避了平衡树维护的复杂逻辑。以p=0.5为例，节点恰好有i层的概率为`(1-p) × p^{i-1}`，即层数服从几何分布。

### 查找算法

查找值x时，从最高层的头节点出发，执行如下逻辑：

1. 若当前节点的`forward[i]`所指节点的值 < x，则在本层前进（水平移动）；
2. 否则，下降一层（垂直移动）；
3. 重复至第0层，判断当前节点的下一个节点是否等于x。

这一过程每次下降前都在本层尽量前进，保证了每层最多遍历`1/p`个节点。总比较次数的期望为`(log_{1/p} n) / p`，在p=0.5时约为`2 log₂ n`。

### 插入与删除

插入时，先通过查找过程记录每层的"前驱节点"数组`update[]`，然后用`randomLevel()`决定新节点层数，最后在各层拼接指针，时间复杂度O(log n)。删除时同样利用`update[]`数组，在各层解除对目标节点的引用，并回收内存，无需任何结构重平衡操作。

## 实际应用

**Redis Sorted Set**：Redis 7.x中的有序集合在元素数量超过`zset-max-listpack-entries`（默认128）或单个元素长度超过`zset-max-listpack-value`（默认64字节）时，自动从listpack升级为跳表+哈希表的双索引结构。跳表负责按score排序并支持`ZRANGEBYSCORE`等范围命令，哈希表负责O(1)的成员查找。

**LevelDB/RocksDB的MemTable**：LevelDB使用跳表作为内存写缓冲区（MemTable）的核心数据结构。写入的键值对先插入跳表，待MemTable达到阈值（默认4MB）后以有序方式flush到磁盘形成SSTable。跳表天然有序的特性使得这一flush过程无需额外排序。

**近似最近邻搜索（ANNS）辅助索引**：在部分向量数据库实现中，跳表被用于对标量字段（如时间戳、类别ID）建立辅助索引，在执行混合查询（向量相似度 + 标量过滤）时，先用跳表快速筛选出满足标量条件的候选集，再在候选集内执行向量检索，大幅减少暴力扫描的开销。

## 常见误区

**误区一：跳表的最坏时间复杂度与平衡树相同**
跳表的O(log n)是期望值，最坏情况下所有节点只有1层，退化为O(n)的普通链表。而红黑树的最坏时间复杂度严格保证为O(log n)。在对延迟有严格上界要求的实时系统中，跳表的概率性保证不等同于确定性保证，这一区别不可忽视。

**误区二：层数越多，跳表一定越快**
层数设置过高会导致内存浪费和指针维护开销增大。以p=0.25、n=10^6为例，期望最大层数约为log₄(10^6)≈10，Redis设定MAXLEVEL=64是为了应对极端大规模场景，并非在所有场景下64层都会被实际使用。对于n<10^4的小规模数据集，2~4层的跳表通常已足够，盲目增大MAXLEVEL反而浪费内存。

**误区三：跳表适合所有需要有序数据的场景**
跳表的每个节点需要存储多个`forward`指针（期望层数约为`1/(1-p)`），内存开销高于普通链表。在内存极度受限的嵌入式环境，或数据已知且不频繁更新的静态场景中，预排序数组+二分查找在缓存命中率和内存占用上均优于跳表。

## 知识关联

跳表在结构上直接依赖**有序链表**的基础操作：节点间的指针拼接与解除引用逻辑与单链表完全一致，理解`forward[0]`层就是在操作一条有序链表，是读懂跳表插入/删除代码的前提。

跳表与**平衡二叉搜索树（AVL/红黑树）**构成直接的算法选型对比：两者实现相同的O(log n)期望复杂度，但跳表以概率性换取了代码简洁性（实现行数通常不足红黑树的1/3），而红黑树以旋转操作换取了确定性的最坏复杂度保证。

在AI工程的更大上下文中，跳表是理解**LSM-Tree（日志结构合并树）**的重要前置知识，MemTable的跳表实现直接影响LevelDB/RocksDB在高写入吞吐量场景下的性能表现；同时也是深入研究**Redis内部实现**时无法绕开的数据结构，`t_zset.c`中的`zslInsert`和`zslDelete`函数是阅读Redis源码的典型入口。
