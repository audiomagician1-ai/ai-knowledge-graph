# 跳表

## 概述

跳表（Skip List）是由William Pugh于1990年在论文《Skip Lists: A Probabilistic Alternative to Balanced Trees》中提出的一种随机化数据结构。它在有序链表的基础上增加多层索引，使得查找、插入和删除操作的平均时间复杂度从链表的O(n)降低到O(log n)，与平衡二叉搜索树（如红黑树、AVL树）相当，但实现复杂度远低于这些树结构。

跳表的命名来自其核心思想：通过多级索引"跳跃"式地跳过大量节点，直接定位到目标区域，而非逐个遍历。每一层索引都是下层的子集，最底层（第0层）包含所有元素，越高层包含的元素越稀疏，通常每层元素数量约为下层的1/2或1/p（p为提升概率参数）。

跳表在工程中被广泛采用，Redis的有序集合（Sorted Set）底层实现之一就是跳表，LevelDB和RocksDB的内存表（MemTable）也使用跳表存储数据。相比红黑树，跳表的并发实现更加自然，因为修改操作只涉及局部节点的指针调整，不需要复杂的旋转操作。

## 核心原理

### 多层索引结构

跳表的物理结构由多条链表叠加而成。设最大层数为MaxLevel，第0层是完整的有序链表，第k层是第k-1层的随机子集。每个节点的层数由随机算法决定：生成一个随机数，若该随机数小于提升概率p（William Pugh的原始论文建议p=1/4或p=1/2），则将节点提升到更高层，重复直至不满足条件或达到MaxLevel。

节点的数据结构包含：元素值、多个forward指针数组（每层一个前向指针）。设跳表中存储n个元素，建议的最大层数MaxLevel = log_{1/p}(n)。例如n=65536且p=1/4时，MaxLevel=8层足够；若p=1/2，则MaxLevel=16层。

### 查找过程与时间复杂度

查找值x时，从最高层的头节点出发，在每一层向右移动，直到下一个节点的值大于x，则下降一层继续查找，直到在第0层找到或确认不存在。查找路径的期望长度为O(log n)。

具体推导：在p=1/2时，期望比较次数C(n) ≈ 2log₂n。这是因为每层预期节点数为上层的2倍，从最高层到第0层共约log₂n层，每层最多比较常数次（期望为2次），总期望比较次数约为(2/p)×log_{1/p}(n)。

### 插入与删除操作

**插入**操作分三步：①用查找过程记录每层的前驱节点（存入update数组）；②随机生成新节点层数level；③在第0层到level层中，将新节点插入到update[i]之后，调整forward指针。插入操作仅需修改相邻节点的指针，不涉及全局平衡调整。

**删除**操作同样先记录update数组，然后从第0层到最高层逐层将目标节点从链表中摘除：若update[i].forward[i] == 目标节点，则执行`update[i].forward[i] = target.forward[i]`。删除后还需更新跳表的当前最大层数。

### 随机层数生成

```
function randomLevel():
    level = 1
    while random() < p AND level < MaxLevel:
        level = level + 1
    return level
```

这里random()返回[0,1)之间均匀分布的随机数，p通常取0.25（Redis实际使用值）。由于层数提升是随机独立事件，跳表的结构具有概率平衡性而非强制平衡性，这使得最坏情况理论上可能退化，但期望性能稳定在O(log n)。

## 实际应用

**Redis有序集合（ZSET）**：当元素数量超过128个或单个元素长度超过64字节时，Redis使用跳表+哈希表的组合存储有序集合。跳表负责按分值（score）排序，支持ZRANGE、ZRANK等范围查询命令；哈希表负责O(1)的成员查找。Redis跳表的MaxLevel设为32，p=0.25。

**数据库内存索引**：LevelDB的MemTable使用跳表而非B树，因为跳表的顺序写性能优异（只需修改局部指针），且易于实现无锁并发读取。写操作完成后，MemTable内容会按顺序刷写到磁盘SSTable文件，跳表的有序性使得这一过程无需额外排序。

**范围查询场景**：跳表在执行范围查询[a, b]时，只需先定位到a的位置，然后在第0层顺序遍历至b，时间复杂度为O(log n + k)，其中k为区间内元素数量。这比哈希表（不支持范围查询）和平衡树（范围查询需要中序遍历）在工程实现上更直观。

## 常见误区

**误区一：跳表查找最坏情况为O(log n)**
跳表是概率性数据结构，其O(log n)是期望时间复杂度，理论最坏情况为O(n)（例如所有节点均只有第0层）。但这一最坏情况发生的概率极低，具体地，n个节点全部处于第0层的概率为(1-p)^n。对n=1000且p=0.25，该概率约为10^{-125}，在实践中可忽略不计。与红黑树的O(log n)最坏保证相比，跳表牺牲了确定性换取了实现简洁性。

**误区二：增加层数一定能提升性能**
盲目增加MaxLevel反而会增加内存占用和头节点遍历开销。MaxLevel应与实际存储元素数量匹配：对于n个元素，最优MaxLevel = ⌈log_{1/p}(n)⌉。若n=1000且p=0.5，MaxLevel=10已足够；设为32层会导致查找时从高层开始做大量无效遍历（因高层几乎为空）。

**误区三：跳表节点层数越高越好**
高层节点虽能加速查找，但每个节点每增加一层就额外占用一个指针空间（64位系统上8字节）。若p=0.5，期望每个节点占用约2个指针；若p=0.25，期望约1.33个指针，内存效率更高。Redis选择p=0.25正是为了在查找速度与内存占用之间取得平衡——相比p=0.5，p=0.25的查找比较次数仅多约33%，但内存节省约33%。

## 知识关联

**前置知识——链表**：跳表的第0层本质上就是一条有序单链表，跳表中的forward指针数组是链表next指针的直接推广。理解链表的节点插入（修改prev.next和node.next）是理解跳表多层指针修改的必要基础。跳表的update数组记录的就是每一层中待修改节点的前驱，与链表插入时找前驱节点的逻辑完全对应。

**横向对比——平衡树**：跳表与AVL树、红黑树解决同一问题（有序动态集合的高效增删查），但维护平衡的机制截然不同。红黑树通过严格的颜色约束和旋转操作保证O(log n)最坏性能；跳表通过概率独立性保证O(log n)期望性能。在并发环境下，跳表的局部指针修改使其更适合实现无锁（lock-free）并发数据结构，这是Redis、LevelDB选择跳表的重要工程原因。

**延伸应用——数据库索引设计**：掌握跳表后，可进一步研究LSM树（Log-Structured Merge Tree）的完整架构，其中MemTable的跳表实现是理解整个LSM写入路径的关键节点，也是理解LevelDB、RocksDB、Cassandra等系统写性能优势的基础。
