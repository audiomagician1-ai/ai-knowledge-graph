# 红黑树

## 概述

红黑树（Red-Black Tree）是一种自平衡二叉搜索树，由鲁道夫·拜尔（Rudolf Bayer）于1972年提出，最初称为"对称二叉B树"（Symmetric Binary B-Tree）。1978年，莱奥尼达斯·吉巴斯（Leonidas Guibas）和罗伯特·塞奇威克（Robert Sedgewick）将其重新命名为红黑树，并确立了现代定义中使用红黑着色的规则体系。

红黑树通过对每个节点附加一个颜色属性（红色或黑色），并强制维护五条颜色规则，确保树的最长路径不超过最短路径的两倍。这一约束使得红黑树在最坏情况下的查找、插入、删除时间复杂度均为 O(log n)。与 AVL 树相比，红黑树允许稍微宽松的平衡条件——AVL 树要求任意节点左右子树高度差不超过1，而红黑树只保证树高不超过 2·log₂(n+1)——换来的是插入和删除操作中旋转次数更少，平均性能更优。

红黑树在工程系统中极为重要：Linux 内核用红黑树管理进程调度（CFS 调度器）和虚拟内存区域；Java 的 `TreeMap`、`TreeSet` 以及 JDK 8 之后的 `HashMap`（链表退化时）均基于红黑树实现；C++ STL 中的 `std::map` 和 `std::set` 底层也是红黑树。

## 核心原理

### 五条红黑性质

红黑树必须严格满足以下五条性质，缺少任何一条都不构成合法的红黑树：

1. **每个节点是红色或黑色。**
2. **根节点是黑色。**
3. **所有叶子节点（NIL 哨兵节点）是黑色。**（NIL 节点是虚拟的外部节点，不存储数据）
4. **红色节点的两个子节点必须都是黑色。**（即不存在两个相邻的红色节点）
5. **从任意节点到其所有后代叶子节点的路径上，黑色节点数量相同。**（称为"黑高"，Black-Height）

由性质4和性质5可推导出：树中最长路径为红黑交替路径，长度至多为最短全黑路径的两倍。设黑高为 bh，则红黑树节点数 n 满足 n ≥ 2^bh − 1，从而树高 h ≤ 2·log₂(n+1)。

### 插入修复：旋转与重着色

新节点默认涂为**红色**（以避免破坏黑高）。插入后可能违反性质4（两个相邻红节点），需要根据叔父节点（Uncle）的颜色分三种情况修复：

- **情况1：叔父节点为红色。** 将父节点和叔父节点改为黑色，祖父节点改为红色，然后将祖父节点视为新插入节点向上递归检查。此操作不涉及旋转，仅重着色，代价为 O(1)。
- **情况2：叔父节点为黑色，新节点为"内侧"子节点（LR 或 RL 型）。** 先对父节点做一次旋转（左旋或右旋），转化为情况3。
- **情况3：叔父节点为黑色，新节点为"外侧"子节点（LL 或 RR 型）。** 对祖父节点做一次旋转，并交换父节点与祖父节点颜色。此操作完成后局部子树恢复合法，无需继续向上传播。

插入操作最多执行 **2次旋转**（情况2 + 情况3各一次），这是红黑树相比 AVL 树的重要优势——AVL 树插入后最多也是2次旋转，但删除时 AVL 最多需要 O(log n) 次旋转，而红黑树删除最多只需 **3次旋转**。

### 删除修复：双黑节点处理

删除操作比插入更复杂。删除节点后若出现"双黑"（Double Black）节点（即某节点的黑高少了1），需要根据兄弟节点颜色及其子节点颜色分四种情况递归修复。关键数量结论：删除修复最多执行 **3次旋转**，但重着色可能传播至根节点，故重着色步骤仍为 O(log n)。

## 实际应用

**Linux CFS 调度器（完全公平调度）：** Linux 内核使用红黑树按虚拟运行时间（vruntime）组织可运行进程。每次调度选取 vruntime 最小的进程（即树最左节点），时间复杂度 O(log n)。进程入队/出队对应红黑树插入/删除，保证调度决策的一致性能。

**Java HashMap 树化：** JDK 8 起，当哈希桶中链表长度超过 **8** 时，链表转换为红黑树，使最坏情况查找从 O(n) 降至 O(log n)。当节点数降至 **6** 时再退化回链表，避免频繁转换。

**数据库索引辅助结构：** 部分内存数据库（如 Redis 的有序集合 ZSet 跳表替代方案讨论中）曾评估红黑树，最终因缓存局部性劣于 B+ 树而未用于磁盘索引，但在内存键值管理（如 Redis 内部字典的某些模块）中仍有应用。

**区间树扩展：** 红黑树可扩展为区间树（Interval Tree），每个节点额外存储子树最大右端点 max，支持 O(log n) 的区间重叠查询，常用于操作系统虚拟内存段管理（Linux `vm_area_struct` 的 `mm_rb`）。

## 常见误区

**误区1：认为红黑树比 AVL 树"更平衡"。**
事实相反。AVL 树的平衡条件更严格（高度差≤1），树高更低，查找性能略优于红黑树。红黑树的优势在于**写操作**（插入/删除）的旋转次数更少，适合写多读少的场景。若数据集静态或查询密集，AVL 树可能是更好的选择。

**误区2：以为 NIL 叶子节点可以省略不实现。**
NIL 哨兵节点是性质3和性质5成立的基础。若实现时用 `null` 指针替代，需要在所有旋转和修复逻辑中特判 `null`，极易引入 bug。标准实现（如 CLRS 算法导论中的伪代码）使用一个全局共享的 `T.nil` 哨兵对象，颜色为黑，避免边界判断错误。

**误区3：插入修复中情况1的重着色不会破坏黑高。**
情况1将祖父节点从黑色改为红色，这确实可能使祖父的父节点出现两个相邻红色节点，因此需要向上递归。但黑高始终保持不变——因为该情况将父和叔父从红改黑的同时将祖父从黑改红，祖父到其子节点路径上的黑色节点数量在修复前后完全一致。

## 知识关联

**与 AVL 树的关系：** 红黑树可视为对 AVL 树平衡约束的放宽版本。学习 AVL 树后理解的单旋/双旋操作（LL旋、LR旋等）在红黑树中直接复用，区别在于红黑树以颜色规则代替高度差规则来触发旋转。掌握 AVL 的旋转正确性证明，有助于理解为何红黑树在情况3中仅需一次旋转即可恢复局部合法性。

**延伸方向：** 红黑树是理解**B树/B+树**的重要前置直觉——2-3-4树（4阶B树）与红黑树存在精确的一一对应关系：红黑树中一个黑节点与其红色子节点构成一个2-3-4树节点，两棵树的结构变换完全等价。此外，红黑树可扩展为**持久化红黑树**（Persistent Red-Black Tree），通过路径复制（Path Copying）实现不可变版本，时间空间复杂度均为 O(log n)，在函数式编程语言（如 Haskell）的标准库中有直接应用。
