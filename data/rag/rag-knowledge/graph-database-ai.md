# 图数据库在AI中的应用

## 概述

图数据库（Graph Database）是以图结构存储数据的非关系型数据库，其中数据以节点（Node）、边（Edge/Relationship）和属性（Property）三要素组织。与关系型数据库在多跳查询中需要执行昂贵的JOIN操作不同，图数据库通过指针直接遍历相邻节点，使得三跳以上的关联查询性能提升可达1000倍以上。在AI工程领域，图数据库承担的核心角色是为知识图谱提供持久化存储和高效检索能力，同时也为RAG系统中的结构化知识推理提供底层支撑。

图数据库的商业化发展始于2007年Neo4j的首次发布。Neo4j引入了属性图模型（Property Graph Model）和专用查询语言Cypher，后者于2015年通过openCypher项目开源，成为事实上的图查询语言标准。2019年后，随着知识图谱增强型RAG（Graph RAG）研究的兴起，Amazon Neptune、TigerGraph、ArangoDB等系统相继被引入AI工作流，使图数据库从传统的社交网络和欺诈检测场景扩展至大语言模型的知识增强管道中。

在AI工程实践中，图数据库解决了向量数据库无法独立解决的**关系推理问题**。纯向量检索依赖语义相似度，无法准确回答"A的上级部门是否与B存在合作协议"这类需要沿边遍历的问题。将图数据库与向量数据库结合，构成了当前Graph RAG架构的技术基础。

## 核心原理

### 属性图模型与RDF图模型的选择

AI应用中使用的图数据库主要遵循两种数据模型。**属性图模型**（如Neo4j、TigerGraph）允许在节点和边上直接附加键值对属性，例如一个"论文"节点可同时携带`title`、`year`、`embedding`等属性；**RDF三元组模型**（如Apache Jena、Stardog）将一切表示为`主语-谓语-宾语`三元组，查询语言为SPARQL。在RAG场景中，属性图模型更为常用，因为它能在同一节点上同时存储语义向量（`embedding: [0.23, -0.71, ...]`）和结构信息，无需跨系统同步。RDF模型则在需要严格本体推理（OWL推理）的医疗或法律知识图谱中更具优势。

### Cypher查询语言与多跳检索

Cypher是图数据库中用于AI检索管道的核心查询接口。其语法模式`(n:Node)-[:RELATION]->(m:Node)`直观模拟图的视觉表示。以下查询示例展示了在GraphRAG中典型的两跳检索：

```cypher
MATCH (q:Question)-[:RELATED_TO]->(e:Entity)-[:MENTIONED_IN]->(d:Document)
WHERE e.name = 'transformer架构'
RETURN d.content, d.source
LIMIT 10
```

可变长度路径语法`-[:REL*1..3]->`允许在1到3跳范围内自动搜索，无需预定义路径深度，这对于探索性知识检索尤为重要。在性能方面，Neo4j的原生图存储使得百万节点规模下，两跳查询的平均响应时间可控制在100ms以内。

### 图向量混合索引

现代AI应用要求图数据库同时支持**结构化遍历**和**语义向量搜索**。Neo4j 5.x版本引入了原生向量索引，支持通过`db.index.vector.queryNodes()`在余弦相似度基础上检索节点，然后沿图边进行精确关系筛选。这种模式称为**向量+图混合检索（Vector-Graph Hybrid Retrieval）**，其工作流程如下：

1. 将用户问题的嵌入向量与图中所有实体节点的`embedding`属性做近似最近邻（ANN）搜索，召回Top-K个候选实体。
2. 以这些实体为起点，沿预定义边类型（如`SUPPORTS`、`CONTRADICTS`、`DERIVED_FROM`）执行Cypher多跳遍历，扩展上下文。
3. 将遍历结果拼接为结构化上下文，注入LLM的提示词。

此方法相比纯向量RAG，在需要跨文档推理的问答任务上，答案准确率提升约15%-30%（依据Microsoft Graph RAG 2024年论文的基准测试数据）。

## 实际应用

**医疗知识图谱问答**：在药物相互作用查询中，图数据库存储`药物-[抑制]->酶-[催化]->代谢通路`的多层关系。当医生询问"患者同时服用华法林和氟康唑是否安全"时，GraphRAG系统通过两跳Cypher查询找到"氟康唑抑制CYP2C9酶，而CYP2C9是华法林的主要代谢酶"这一关键路径，LLM据此生成警示答案，而纯文本向量检索往往会漏掉这一间接推理链。

**企业知识管理RAG**：某大型金融机构将5年内的内部文档拆分为段落，每段作为图节点，同时建立`作者-属于->部门`、`文档-更新自->历史版本`、`段落-引用->法规条文`等边关系。查询"合规部关于反洗钱的最新政策"时，系统先向量检索相关段落，再沿`文档-更新自`边找到最新版本，确保信息时效性。

**代码知识图谱**：将代码库构建为图（函数节点、类节点、调用关系边、继承关系边），结合AST静态分析，AI代码助手可以回答"哪些模块调用了已废弃的API"——这需要反向遍历`CALLS`边，是向量检索完全无法实现的查询类型。

## 常见误区

**误区一：图数据库可以完全替代向量数据库**。图数据库擅长关系遍历和精确属性匹配，但其原生向量搜索能力（如Neo4j Vector Index）在超大规模（千万维度级别）下性能不及专用向量数据库（如Pinecone、Weaviate）。实际生产系统通常采用Neo4j负责结构化关系存储、Pinecone负责高并发向量检索的双库架构，而非单一替换。

**误区二：知识图谱构建完成后图结构就固定了**。在RAG系统中，图结构需要随文档更新而动态演化。若新增文档引入了对旧实体的新关系描述，图中对应边必须更新，否则LLM会基于过时的关系推理。Neo4j的`MERGE`语句支持幂等更新，是维护动态知识图谱的关键操作，很多初学者误用`CREATE`导致重复节点累积。

**误区三：图查询越复杂检索效果越好**。无限制的可变长度路径（如`-[:REL*]->`不设上限）会在密集图中引发笛卡尔积式膨胀，导致查询超时甚至内存耗尽。GraphRAG的最佳实践是将最大遍历深度限制在2-3跳，超出此范围的隐性关联应依赖LLM的语言推理能力而非图遍历。

## 知识关联

**前置概念的具体衔接**：图数据结构中的邻接表表示直接对应图数据库的原生存储方式——Neo4j的无索引邻接（Index-Free Adjacency）本质上就是基于指针的邻接表，这使得它比基于索引的关系型数据库在图遍历中快出数个数量级。NoSQL概述中学习的CAP定理在图数据库中的体现是：Neo4j单机版优先保证CP（强一致性+分区容错），而分布式版Neo4j 4.x的因果一致性（Causal Consistency）则是在可用性与一致性间的工程权衡。

**通向知识图谱+RAG的桥梁**：本主题建立了图数据库的技术能力边界——Cypher查询、属性图模型、向量混合索引。这些能力直接对应知识图谱+RAG架构中的三个核心模块：实体抽取后的图入库流程（使用`MERGE`语句）、检索时的图遍历策略（Cypher多跳查询）、以及检索结果如何序列化为LLM可理解的自然语言上下文。掌握图数据库的查询性能特征，是设计低延迟GraphRAG管道的技术前提。
