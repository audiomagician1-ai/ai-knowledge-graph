# 相似度搜索与重排

## 概述

相似度搜索（Similarity Search）是RAG系统中从向量数据库检索语义相关文档的核心机制，其基本原理是将查询文本编码为高维向量后，计算该向量与数据库中所有已索引向量之间的距离，返回距离最近的Top-K个文档片段。与传统关键词匹配不同，相似度搜索能捕捉"用户询问发烧怎么办"与存储着"退热药使用指南"之间的语义关联，即便两者没有任何共同词汇。

重排（Reranking）是在相似度搜索之后加入的二阶段精排步骤，由Nogueira等人于2019年在论文《Passage Re-ranking with BERT》中系统化提出并验证其有效性。第一阶段（召回阶段）用向量相似度快速筛选出候选文档集（通常50-200条），第二阶段（精排阶段）用计算更重量级的交叉编码器（Cross-Encoder）模型对查询与每个候选文档的组合重新打分，输出真正与查询最相关的Top-N条（通常3-10条）。

这种两阶段架构解决了纯向量搜索的致命缺陷：双编码器（Bi-Encoder）在生成Embedding时查询和文档是独立编码的，无法建模两者之间的细粒度词语交互；而重排模型将查询与文档拼接为一个序列`[CLS] query [SEP] document [SEP]`，让注意力机制充分计算跨序列的词级相关性，从而显著提升最终答案质量。

## 核心原理

### 相似度度量函数的选择

向量相似度有三种主要度量方式，选择错误会直接导致检索质量下降：

- **余弦相似度（Cosine Similarity）**：`cos(θ) = (A·B) / (‖A‖·‖B‖)`，消除向量长度的影响，适合OpenAI `text-embedding-ada-002`等L2归一化后的Embedding，Pinecone默认使用此度量。
- **点积（Dot Product）**：`A·B = Σ(aᵢ·bᵢ)`，当向量未归一化时与余弦相似度结果不同，Google的ScaNN和FAISS的内积索引（IndexFlatIP）使用此度量，计算速度比余弦稍快。
- **欧氏距离（L2 Distance）**：`d = √Σ(aᵢ-bᵢ)²`，距离越小越相关，Milvus的IVF_FLAT索引默认使用此度量，适合未归一化的密集向量。

对于同一批Embedding，错误地在归一化向量上使用L2距离，与使用余弦相似度的排序结果往往不同，这是生产系统中常见的配置陷阱。

### 近似最近邻（ANN）算法与精确搜索的权衡

精确最近邻搜索（Exact KNN）的时间复杂度为O(n·d)，其中n为向量数量，d为维度。当数据库存储了1000万条1536维向量时，单次查询需要扫描约154亿次浮点运算，延迟往往超过秒级，不可接受。

ANN算法通过牺牲少量召回率换取查询速度的大幅提升：

- **HNSW（Hierarchical Navigable Small World）**：Milvus和Weaviate广泛使用，构建多层图索引，查询复杂度降至O(log n)，在典型基准测试（如ANN-Benchmarks）上可达到95%+的召回率，查询延迟降至毫秒级。
- **IVF（Inverted File Index）**：将向量空间划分为`nlist`个聚类，查询时只搜索最近的`nprobe`个聚类，nprobe/nlist的比值直接控制速度与召回率的权衡。
- **FAISS的IndexIVFPQ**：结合乘积量化（Product Quantization）压缩存储，可将内存占用压缩至原始大小的1/8到1/32，代价是轻微损失召回率。

### 交叉编码器重排的工作机制

重排模型的输入格式为：`input = [CLS] + query_tokens + [SEP] + doc_tokens + [SEP]`，将该拼接序列送入BERT类模型后，取`[CLS]`位置的输出向量，接一个线性层输出单个相关性分数。

常用的开源重排模型包括：

- `cross-encoder/ms-marco-MiniLM-L-6-v2`：在MS MARCO段落检索数据集上微调，适合英文问答场景，模型大小仅22MB，推理延迟低。
- `BAAI/bge-reranker-large`：智源研究院发布的中英双语重排模型，在CMTEB中文检索基准上表现突出，适合中文RAG系统。
- Cohere的`rerank-english-v3.0` API：商业化重排服务，每1000次调用约1美元，无需自部署。

重排阶段不使用向量索引，必须对每个候选文档单独执行一次前向传播，因此当候选集为100条时，重排的计算量是单次Embedding推理的100倍，这是为何必须先用ANN缩小候选集再重排的根本原因。

## 实际应用

**企业知识库问答**：某法律科技公司构建合同审查RAG系统时，第一阶段用HNSW从8万条合同条款向量中召回Top-50候选，第二阶段用`bge-reranker-large`精排后取Top-5送入GPT-4生成建议。引入重排后，系统在内部测试集上的答案准确率从61%提升至79%，主要提升来自重排能正确处理"违约金条款"与"赔偿责任条款"这类语义相近但法律含义不同的细粒度区分。

**电商搜索增强**：某电商平台将商品描述编码为768维向量存储于Milvus，用户搜索"防滑耐磨户外鞋"时，纯向量搜索的Top-1结果常返回"户外运动袜"（因语义向量空间接近），加入Cohere Rerank后，有鞋底纹理描述的商品得分显著上升，Top-3命中率提升23个百分点。

**文档摘要生成的上下文窗口优化**：在长文档RAG中，向量搜索召回的50个候选片段总Token数可能超过32K，无法全部塞入LLM上下文窗口。通过重排取Top-5后，仅需约2000 Tokens，同时保留了最高质量的证据片段，使生成内容的引用准确性显著优于随机截断方案。

## 常见误区

**误区一：相似度分数越高，文档越适合回答问题**

向量相似度高只代表语义空间接近，不代表文档能回答该问题。例如查询"苹果手机的内存多少"，一篇讨论"内存条发展史"的文档可能与查询有较高的余弦相似度（因共享"内存"这一语义域），但实际无法回答问题。这正是重排存在的意义：相关性分数（relevance score）不等于相似性分数（similarity score），前者建模的是"是否能回答该问题"，后者建模的是"是否在同一语义领域"。

**误区二：增大Top-K召回数量一定能提高最终答案质量**

将ANN召回数从50增至200，重排后的Top-5质量并不一定更高，原因在于：（1）更大的候选集引入更多噪声，超出重排模型的有效分辨边界；（2）重排延迟线性增长，200条候选意味着4倍于50条的推理时间；（3）在Milvus的IVF索引中，盲目增大nprobe也会消耗更多计算资源。实践中，对于多数RAG场景，召回Top-50到100后重排是延迟与质量的合理平衡点。

**误区三：重排模型可以用双编码器的Embedding余弦相似度替代**

有开发者尝试用双编码器分别计算查询和文档的Embedding，再用余弦相似度排序，将此作为"轻量重排"。但双编码器无法捕捉查询中具体词语在文档中的精确匹配，对于包含数字、专有名词或否定语义（如"不包含X条款"）的查询，其分辨能力远弱于交叉编码器，在BEIR基准测试中，交叉编码器的NDCG@10通常比双编码器高8-15个百分点。

## 知识关联

**与前置概念的关联**：相似度搜索依赖文本Embedding的质量——如果Embedding模型对中文法律文本未经过领域微调，则向量空间本身的语义结构就不合理，无论使用何种相似度度量或ANN算法，召回质量都会受限。向量数据库（Pinecone/Milvus）提供了HNSW和IVF等ANN索引的实际存储与检索能力，是相似度搜索的执行载体，其索引参数（如HNSW的`ef_construction`和`M`值）直接影响构建时间、内存占用与查询召回率三者的平衡。

**向混合检索的延伸**：纯向量相似度搜索在处理精确关键词（如产品型号"iPhone 15 Pro Max"、法条编号"第52条第3款"）时召回率不稳定，这推动了混合检索（向量相似度 + BM25关键词检索）
