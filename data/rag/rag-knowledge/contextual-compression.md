# 上下文压缩

## 概述

上下文压缩（Contextual Compression）是RAG管道中的一种后检索优化技术，其核心操作是：在将检索到的文档块传入LLM之前，从中提取与用户查询语义最相关的句子或段落，剔除无关内容，从而缩短最终输入到LLM的上下文长度。这一技术由LangChain团队在2023年提出并在其框架中正式实现，对应的核心类为`ContextualCompressionRetriever`，它将一个基础检索器与一个"压缩器"（Compressor）组合成新的检索器链。

上下文压缩要解决的问题是：标准RAG检索出的文档块（通常512至1024 tokens）往往只有其中一两个句子与查询直接相关，其余内容构成噪声。当这些噪声被塞进提示词时，LLM的注意力会被稀释，出现"迷失在中间（Lost in the Middle）"的现象——研究显示，将关键信息置于长上下文的中部比置于首尾时，GPT-4的准确率平均下降约20%。通过压缩，有效信息密度显著提升，同时减少了token消耗和推理延迟。

与Reranking不同，Reranking只对文档块重新排序而不改变其内容；上下文压缩则直接修改文档块的内容，是一种破坏性操作，需要谨慎评估信息丢失的风险。

## 核心原理

### LLM压缩器（LLMChainExtractor）

最直接的压缩方式是用一个小型LLM充当"摘录员"。对于每一个检索到的文档块，压缩器向LLM发送如下结构的提示：

```
给定以下文档内容：
[文档块原文]

根据查询：[用户问题]

仅提取文档中与查询相关的内容，逐字引用，不要添加任何解释。
如果没有相关内容，返回"NO_OUTPUT"。
```

LLM被要求逐字引用原文而非改写，这样可以保留原始文档的措辞，避免幻觉。LangChain的`LLMChainExtractor`正是采用这一机制，在实测中，它将平均每个文档块从约800 tokens压缩到150-200 tokens，压缩比约为4:1。代价是需要为每个检索到的文档块（通常3-5个）额外发起一次LLM调用，延迟增加明显。

### 嵌入式过滤器（EmbeddingsFilter）

为了规避LLM压缩的高延迟，嵌入式过滤器提供了一种轻量替代方案。其算法如下：

1. 将文档块拆分成句子（使用句号或换行作为分隔符）
2. 为用户查询和每个句子分别生成嵌入向量
3. 计算每个句子与查询的余弦相似度
4. 仅保留相似度超过阈值 θ（默认值通常设为0.76）的句子
5. 将保留的句子按原文顺序重新拼接

公式为：保留句子 sᵢ 当且仅当 cos(embed(sᵢ), embed(q)) ≥ θ，其中 q 为查询文本。这种方式无需额外LLM调用，延迟与向量相似度计算相当，通常在毫秒级别完成。缺点是纯粹依赖语义相似度，无法理解句子间的逻辑依赖关系，例如"如前所述"此类指代性句子本身相似度低但语义重要。

### 管道压缩器（DocumentCompressorPipeline）

在生产场景中，压缩步骤通常被组合成管道：先用`EmbeddingsFilter`进行粗粒度过滤（去除相似度极低的整个句子群），再用`LLMChainExtractor`对保留内容做精细提炼。LangChain的`DocumentCompressorPipeline`允许将多个压缩器串联，典型配置为：

```
EmbeddingsFilter(θ=0.70) → CharacterTextSplitter → EmbeddingsFilter(θ=0.80)
```

第一层过滤降低噪声，中间层重新分割文本粒度，第二层提高精度阈值，最终输出密度更高的片段。这种两阶段嵌入过滤可以在不增加LLM调用的前提下，将token消耗降低60-70%。

## 实际应用

**法律文书问答系统**：一份合同可能有50页，用户提问"违约金条款是什么"。标准RAG检索出的块可能包含整个第三章（约2000 tokens），其中仅第12条（约200 tokens）与违约金相关。使用`LLMChainExtractor`后，系统仅将第12条传入最终LLM，GPT-3.5-turbo的回答准确率从71%提升到89%（基于内部测试数据），token费用同时减少约75%。

**多跳问题**：对于需要跨文档综合信息的查询，压缩尤为关键。假设查询为"A公司的CEO在哪所大学读的本科"，检索可能返回三个块：A公司介绍、CEO履历、大学排名。只有CEO履历中的教育背景段落有用，其余均可剔除。若不压缩，LLM可能将排名信息与问题混淆。

**阈值调优**：在使用`EmbeddingsFilter`时，θ=0.76是通用默认值，但在专业垂直领域（如医疗、法律），领域术语的嵌入向量与通用查询的余弦相似度天然偏低，建议将阈值下调至0.65-0.70，否则会过度过滤掉关键专业内容。

## 常见误区

**误区一：压缩等于摘要**。很多初学者将上下文压缩与文档摘要混淆。摘要是对内容的重新表述和归纳，会引入模型的主观解释；上下文压缩应该是**原文摘录**，即从原文中直接剪裁出相关片段，不改写。`LLMChainExtractor`的提示词中明确要求"逐字引用"正是出于这一原因。若使用摘要替代压缩，等效于在检索结果上叠加了一层幻觉风险。

**误区二：压缩一定提升效果**。当原始文档块的分块粒度已经足够细（如128 tokens/块）时，强行压缩可能截断完整的语义单元，反而降低质量。上下文压缩最适合的场景是块粒度较粗（512 tokens以上）且查询高度针对性的情况。若已在分块阶段使用了语义分块（Semantic Chunking），压缩的边际收益会显著缩小。

**误区三：用单一阈值θ处理所有查询类型**。事实型查询（"某公司成立于哪年"）与开放式查询（"请分析这家公司的发展策略"）对上下文需求的广度截然不同。前者适合高阈值（θ≥0.80）严格过滤，后者需要低阈值（θ≤0.70）保留更多背景信息。将θ固定为单一值会在两类查询中各损失一部分准确率。

## 知识关联

上下文压缩直接依赖**文档分块策略**的输出：块的大小决定了压缩的必要性和压缩比的上限——块越大，可压缩空间越多；若块本身只有128 tokens，压缩能带来的提升极为有限。分块时设定的元数据（如页码、章节标题）在压缩后通常需要保留，以便LLM在回答时能正确引用来源。

**Reranking**通常作为上下文压缩的前置步骤：先用交叉编码器（Cross-Encoder）对检索结果重排序，确保最相关的文档块排在前面，再对top-K个文档块执行压缩。这样压缩器只需处理已经筛选过的高质量输入，减少因原始检索噪声造成的压缩偏差。若跳过Reranking直接压缩，低相关度的文档块进入压缩器后可能返回空输出（NO_OUTPUT），浪费额外的LLM调用开销。在LangChain中，典型的完整后检索链路为：`BaseRetriever → Reranker → ContextualCompressionRetriever → LLM`。
