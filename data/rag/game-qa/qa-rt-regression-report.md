# 回归报告

## 概述

回归报告是游戏项目回归测试阶段的正式输出文档，用于汇总每轮回归测试的通过率、用例覆盖率和历史趋势，供开发团队、测试负责人和发布决策者直接参考。与单次缺陷报告不同，回归报告的核心价值在于**纵向对比**——通过连续多个版本的数据横排，让团队看清质量是在收敛还是在恶化。

该报告格式在游戏行业随着持续集成流水线（CI/CD）的普及逐步标准化。早期（2010年前后）回归报告多以邮件形式手动发送，内容随意；进入2015年后，主流游戏工作室开始采用测试管理平台（如TestRail、JIRA Zephyr）自动生成结构化回归报告，固定了"通过率 / 失败率 / 阻塞率 / 未执行率"四象限格式。

回归报告直接服务于**发布质量标准**的判断：当报告显示连续三个版本通过率低于90%时，发布门禁通常会自动拦截该版本进入外发或上线流程。因此报告的准确性、及时性和可读性直接影响发布节奏，不能将其视为纯粹的归档文件。

## 核心原理

### 1. 三大核心指标及计算公式

回归报告必须包含以下三项量化指标，缺一不可：

**通过率（Pass Rate）**
$$\text{通过率} = \frac{\text{通过用例数}}{\text{已执行用例总数}} \times 100\%$$

注意分母是"已执行"而非"全部用例"，未执行用例不计入通过率但必须单独列明，否则会高估版本质量。

**用例覆盖率（Case Coverage）**
$$\text{覆盖率} = \frac{\text{本轮执行用例数}}{\text{回归用例库总数}} \times 100\%$$

当覆盖率低于80%时，通过率数据的参考价值大幅下降，报告中需标注原因（如时间不足、环境故障）。

**缺陷回归率（Defect Reopen Rate）**
$$\text{回归率} = \frac{\text{已修复但重新打开的缺陷数}}{\text{本轮验证的已修复缺陷总数}} \times 100\%$$

回归率超过15%通常被视为开发侧代码质量警告信号，需在报告中单独高亮。

### 2. 不稳定用例的处理与标注规则

回归报告必须区分"真实失败"与"不稳定用例（Flaky Case）"导致的失败。若将不稳定用例的失败计入通过率分母，会系统性低估版本质量。标准做法是：

- 在报告中单独列出"不稳定用例列表"，注明用例ID、历史失败频率和本次结果
- 计算"净通过率"时从已执行总数中剔除被标记为Flaky的失败用例
- 但原始通过率（含Flaky失败）也必须保留，供不稳定用例管理流程使用

这两个通过率数值同时存在于报告中，分别服务于不同读者：发布决策看净通过率，质量改进看原始通过率。

### 3. 趋势图的必要性与配置规范

单次报告数据孤立价值有限，趋势图是回归报告的核心可视化组件。标准配置要求：

- **X轴**：连续版本号或日期（最少展示最近5个版本）
- **Y轴左**：通过率折线（%）
- **Y轴右**：失败用例绝对数量柱状图
- **阈值红线**：在通过率90%处画出发布门禁基准线

当趋势图显示通过率连续下滑超过2个版本，报告需自动触发"质量预警"标签，提示团队介入分析而非等到版本临近发布才发现问题。

### 4. 报告模板的五段式结构

一份合格的游戏项目回归报告包含以下五个固定区块：

| 区块 | 内容 |
|---|---|
| **执行摘要** | 通过率、覆盖率、重大失败用例数，3行以内 |
| **数据明细** | 按功能模块分组的用例结果表 |
| **失败用例清单** | 含缺陷ID、严重级别、负责人 |
| **不稳定用例附录** | 本轮Flaky用例及历史频率 |
| **趋势对比** | 最近N版本通过率 / 覆盖率折线图 |

## 实际应用

**场景一：大型手游每周版本回归**
某MMORPG项目每周四上线更新版本，回归报告在周三18:00截止前必须产出。报告模板预设了"核心玩法模块（战斗、副本、商城）"和"新增功能模块"两个分组，核心玩法要求100%用例覆盖、通过率≥95%，新增功能通过率≥85%，两项标准分开计算，避免新功能拉低核心模块指标。

**场景二：热修复（Hotfix）回归报告**
Hotfix场景时间极度压缩，回归报告模板需预设"精简版"格式：只统计受影响模块的用例子集（通常为全量用例的10%~20%），并在报告头部强制标注"本次为精简回归，覆盖率XX%，未覆盖模块风险自担"，确保发布决策者知晓范围限制。

**场景三：端游项目测试报告自动化**
通过将TestRail API与CI流水线集成，每次自动化回归结束后，系统在5分钟内自动生成并发送回归报告至项目频道，报告中的趋势图取数自数据库中存储的历史结果，无需人工整理。

## 常见误区

**误区一：用通过率单一指标评估版本质量**
通过率95%听起来优秀，但如果覆盖率只有40%（另60%的用例未执行），该数据几乎没有意义。回归报告必须同时呈现通过率和覆盖率，读者应始终将二者结合阅读，任何单独引用通过率做出发布决策的行为均属于信息不完整决策。

**误区二：将回归报告与测试周报混用**
测试周报包含缺陷发现趋势、用例新增数量等开发过程数据；回归报告仅聚焦当前版本的回归执行结果。两者受众和决策用途不同，合并为一份文档会导致发布决策者需要从大量过程数据中筛选关键指标，增加误判风险。大型项目中二者应严格分开。

**误区三：回归报告只需在发布前产出一次**
部分团队习惯在版本末期才出回归报告，此时即使发现问题也没有时间修复。正确做法是在每轮回归（通常每日或每个迭代）结束后即时出报告，通过趋势图提前预判质量走向，而非将报告作为最终的通关文书。

## 知识关联

回归报告的数据准确性依赖**不稳定用例管理**的前置工作：只有建立了Flaky用例台账并持续更新历史失败率，报告中的"净通过率"与"原始通过率"分离才有数据基础；若不稳定用例未被标记，两个通过率将混同，导致净通过率无法计算。

回归报告的阈值设置与**发布质量标准**直接绑定：发布质量标准文档中定义的通过率门禁值（如核心功能≥95%）必须以硬编码方式写入回归报告模板，而不是靠人工逐次对照。报告产出后，发布负责人通过比对报告中的实际值与标准值，完成"是否允许发布"的最终判断。因此回归报告是连接测试执行结果与发布决策的最终环节文档。
