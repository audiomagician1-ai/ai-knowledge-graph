# 不稳定测试治理

## 概述

不稳定测试（Flaky Test）是指在代码未发生任何变更的情况下，多次执行同一测试用例时出现结果不一致的现象——有时通过，有时失败。在游戏自动化测试中，Flaky Test 的典型表现包括：UI 渲染帧率抖动导致截图比对失败、网络延迟波动使异步接口超时、以及多线程场景下的竞态条件（Race Condition）引发的随机崩溃。Google 内部研究数据显示，约 16% 的测试用例在其 CI 流水线中至少展现过一次不稳定行为，这一比例在游戏测试领域因渲染和物理引擎的非确定性特征而更高。

不稳定测试治理（Flaky Test Governance）是一套系统化管理 Flaky Test 全生命周期的方法体系，涵盖检测、隔离、修复与预防四个阶段。若放任不稳定测试存在，测试团队会逐渐产生"测试结果不可信"的惯性思维，出现频繁手动重跑（Re-run）来掩盖问题的反模式。Meta 的工程团队曾记录到，一个被忽视的 Flaky Test 平均每周消耗工程师 2.4 小时的无效排查时间。

在游戏 QA 流水线中，Flaky Test 治理具有特殊优先级，因为游戏测试中大量依赖基于录制回放生成的测试用例，而录制回放捕获的环境状态（如随机种子、时间戳、物理帧序列）若未被严格复现，必然产生系统性的不稳定测试集群。

## 核心原理

### 不稳定性的根因分类

游戏自动化测试中的 Flaky Test 根因可分为五类：

1. **时序依赖（Timing Dependency）**：测试用例硬编码了固定等待时间，例如 `sleep(500ms)` 等待动画播放完成，但在低端设备或高负载环境下动画实际耗时为 800ms，导致断言提前触发。
2. **共享状态污染（Shared State Pollution）**：测试用例之间共享游戏存档、内存缓存或数据库实例，前一个用例的残留数据影响后续执行结果。
3. **非确定性系统调用**：调用 `Random.Next()`、`DateTime.Now` 或物理引擎的浮点运算，在不同硬件上产生不同结果。
4. **网络与外部服务依赖**：测试直接调用游戏后端 API，受服务器响应时间影响；CDN 资源加载超时导致资产缺失断言失败。
5. **并发与资源竞争**：多线程渲染任务与测试断言线程之间存在竞态，在 GPU 异步计算场景尤为突出。

### 检测方法：重跑采样与稳定性评分

检测 Flaky Test 的核心手段是**多次重跑分析**。标准做法是对同一用例在相同代码基线下执行 N 次（通常 N=10 或 N=50），计算其**不稳定率（Flakiness Rate）**：

```
Flakiness Rate = 失败次数 / 总执行次数
```

当 Flakiness Rate > 0 且 < 1 时，该用例被标记为 Flaky。实践中常设置阈值，例如 Flakiness Rate ≥ 5%（即 50 次中有 2+ 次不一致）触发隔离流程。

另一种高效检测方式是**差异化重跑（Bisect Re-run）**：当 CI 流水线中某个测试失败时，自动在相同 commit 上重跑 3 次，若有 1 次通过则判定为 Flaky，而非真实 Bug 导致的失败，从而区分"代码缺陷"与"测试缺陷"。

### 隔离机制：隔离队列与标签体系

确认为 Flaky 的测试用例应立即从主干 CI 的阻断流程中移出，转入专用的**隔离队列（Quarantine Queue）**。隔离队列以独立频率运行（例如每日深夜一次），不阻塞主干构建，但持续监控 Flaky Test 的状态变化。

标签体系是隔离管理的基础设施，常见标签包括：
- `@Flaky`：标记该用例已被隔离，不参与主干阻断
- `@FlakyReason=TimingDependency`：记录根因类型，便于批量修复
- `@FlakyOwner=CharacterTeam`：指定负责团队，防止所有权模糊化
- `@FlakyDeadline=2024-03-15`：设置强制修复或删除的截止日期

若一个 Flaky Test 在隔离队列中连续 30 天（或设定周期）未被修复，应触发自动删除或强制升级（Escalation）流程，防止测试负债无限累积。

### 修复策略

针对不同根因的修复方案各有差异：

- **时序依赖修复**：将 `sleep(N ms)` 替换为**轮询等待（Polling Wait）**，即循环检查目标条件是否满足，最大等待时间设上限（如 5000ms），避免硬编码延迟。Unity 测试框架提供 `yield return new WaitUntil(() => condition, timeout)` 即为此类机制。
- **共享状态修复**：在每个测试用例的 `setUp` 和 `tearDown` 阶段重置游戏状态，使用数据库事务回滚或内存快照还原，确保测试间的**幂等性（Idempotency）**。
- **非确定性修复**：注入固定随机种子（如 `Random.Seed = 42`），将时间戳替换为可控的模拟时钟（Mock Clock），将浮点比较改为带容差的断言（如误差范围 ±0.001f）。

## 实际应用

**游戏战斗系统回放测试**：某 MMORPG 项目的技能释放自动化测试中，伤害数值断言频繁失败。根因分析发现，物理引擎的碰撞检测使用了设备相关的浮点精度，在 iOS 与 Android 设备上产生不同的暴击触发序列。修复方案是在录制回放时锁定随机种子并强制禁用浮点优化，使两端结果一致，将该用例组的 Flakiness Rate 从 23% 降至 0%。

**开放世界加载测试**：地图区块流式加载测试在低内存设备上偶发超时，Flakiness Rate 约 12%。工程师将固定超时断言（`assertLoaded(3000ms)`）改造为指数退避轮询，最大重试 5 次，间隔分别为 500ms、1000ms、2000ms、4000ms、8000ms，彻底消除了加载速度波动导致的假失败。

**多人联机同步测试**：服务器帧同步测试用例依赖真实网络环境，每当测试服务器负载升高时失败率飙升。解决方案是引入网络模拟层（使用 `tc netem` 工具模拟固定 50ms 延迟和 0.1% 丢包），使测试不再依赖真实网络条件，彻底消除外部依赖带来的不稳定性。

## 常见误区

**误区一：Flaky Test 只需重跑就能解决**
许多团队在 CI 配置中加入"失败自动重跑 3 次"的规则，将最终结果以最后一次通过为准，认为此举解决了 Flaky 问题。实际上这只是掩盖症状：重跑会显著增加 CI 时间（3× 执行时间），且当真实 Bug 出现时，重跑机制会掩盖真正的失败，造成 Bug 漏检。正确做法是将重跑仅用于**检测 Flaky Test**（识别阶段），而非作为长期容忍策略。

**误区二：Flaky Test 是测试代码的问题，与业务代码无关**
部分 Flaky Test 根因确实来自测试框架的时序处理，但也有相当比例源于被测系统（SUT）本身的并发 Bug 或非确定性逻辑。例如，某个技能 CD 计算偶发返回负值，正是因为业务层存在线程不安全的计数器操作。将所有 Flaky Test 归类为"测试代码缺陷"会导致真实 Bug 被错误隔离而不是被修复。

**误区三：隔离即等于修复，搁置隔离队列不影响质量**
隔离机制是临时保护措施，不是终态。若隔离队列中积累了大量未修复的 Flaky Test（如超过总用例数的 10%），则覆盖率数据将严重失真，业务风险实际上已无法被这些用例检测。必须配合截止日期和所有权追踪机制，确保隔离队列持续清零。

## 知识关联

**与录制回放的关联**：录制回放生成的测试用例天然携带时间戳、帧序列等环境相关数据，是游戏自动化测试中 Flaky Test 的高发来源。在回放执行时，若未能精确复现录制时的随机状态（随机种子、物理引擎参数），录制回放用例会系统性地成为 Flaky Test，因此不稳定测试治理的隔离与修复方法直接应用于回放用例的质量保障。
