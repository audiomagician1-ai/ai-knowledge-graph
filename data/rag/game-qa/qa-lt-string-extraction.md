# 字符串提取验证

## 概述

字符串提取验证（String Extraction Verification）是游戏本地化测试的一个专项流程，专门用于确认游戏中所有可见的UI文本、对话内容、错误提示、教程说明等文字元素已被正确地从代码中分离出来，写入外部资源文件（如`.po`、`.xliff`、`.csv`或引擎自定义的`.loc`格式），而非以硬编码（hardcoded string）形式嵌入源代码之中。

这一验证方式起源于软件国际化（i18n）工程实践，随着游戏产业在2000年代初期开始大规模推进多语言同步发行（Simultaneous Ship，简称SimShip），字符串提取验证逐渐成为本地化测试周期中的标准检查项。早期未执行此验证的项目曾出现大量"残留英文"（English bleed-through）问题——即日语或德语版本界面中突然出现英文按钮标签，根本原因正是开发阶段遗留的硬编码字符串未被捕获。

字符串提取验证之所以不可或缺，在于它直接决定了翻译文件能否被完整应用到游戏的每一处角落。一个未被提取的字符串无论翻译质量多高，都永远不会出现在本地化版本中；而一旦进入认证（Certification/Lotcheck）阶段才发现硬编码字符串，修复成本将比开发早期高出数十倍。

## 核心原理

### 硬编码字符串的识别机制

硬编码字符串是指直接写在源代码逻辑中、无法通过修改资源文件来替换的文字。典型的C#硬编码例子如下：

```csharp
// 硬编码（错误做法）
buttonLabel.text = "Start Game";

// 已提取（正确做法）
buttonLabel.text = LocalizationManager.GetString("UI_MAINMENU_STARTBUTTON");
```

验证人员通过静态代码分析工具（如`grep`正则扫描、专用i18n linter）或在游戏运行时配合伪本地化环境进行检测。正则表达式模式如`"[A-Za-z][A-Za-z\s]{3,}"`可批量定位代码中潜在的英文字符串字面量，但需与合法的逻辑常量（如枚举名、文件路径）区分过滤。

### 资源文件覆盖率指标

字符串提取验证的核心量化目标是达到**100%字符串外部化率**，即游戏中所有面向玩家显示的文本都必须存在于可翻译资源文件中。测试团队通常使用**字符串覆盖率报告**来追踪进度，公式为：

> **覆盖率 = 已提取字符串数 ÷ 全部可见字符串总数 × 100%**

在实际项目中，当覆盖率低于98%时，通常要求开发团队立即提交修复；100%是进入翻译交付阶段（Handoff）的前置条件之一。未覆盖的字符串每发现一条，需在缺陷跟踪系统（如Jira）中建立独立缺陷单，标注所在文件路径、代码行号及上下文截图。

### 验证与伪本地化的协同工作

字符串提取验证通常紧接在伪本地化阶段之后执行。伪本地化会将所有已提取的字符串替换为特殊标记字符（例如在英文原文外包裹`[!!! Ŝţàŕţ Ĝàmé !!!]`），使得游戏运行时凡是仍显示为普通英文的文字，均可立即被视觉识别为未提取的硬编码字符串。这种方法比纯代码扫描更有效，因为动态生成的字符串（如通过字符串拼接`"Level " + levelNum`构造的文本）往往能躲过静态分析，却无法在伪本地化界面中隐藏。

## 实际应用

**案例：RPG游戏库存界面的硬编码发现**

某款RPG游戏在本地化准备阶段，测试人员启用伪本地化模式后发现，玩家背包界面的物品数量提示仍显示为`"You have 5 items"`而非伪本地化文本。追查后确认，该字符串通过运行时拼接（`"You have " + count + " items"`）生成，本地化键值仅为`"You have"`和`"items"`两个片段，未使用支持参数占位符的完整句式键（如`"UI_INVENTORY_ITEMCOUNT"`对应译文模板`"您拥有{0}件物品"`）。此缺陷导致所有语言的语序均告失效，需重新设计字符串结构。

**工具使用：Unity Localization Package的字符串审计**

在Unity引擎项目中，使用Unity Localization Package提供的`String Table`可视化界面，测试人员能够执行"String Table Check"功能，自动扫描场景中所有`TextMeshPro`和`Text`组件，标出未绑定本地化键的文本字段。该工具可直接输出CSV格式报告，列出组件路径、当前硬编码内容及所在场景名称，极大提升了字符串提取验证的效率。

## 常见误区

**误区一：通过代码审查即可完成验证**

许多开发团队认为让程序员自查即可保证字符串全部外部化。实际上，动态拼接字符串、第三方中间件内嵌文本、平台SDK错误提示（如主机端系统弹窗）均属于代码审查的盲区。字符串提取验证必须结合运行时测试（即实际启动游戏并逐一遍历所有界面），才能捕获这类隐性硬编码。

**误区二：只需检查UI界面**

字符串提取验证的范围并不限于主菜单和HUD等明显UI区域。成就名称、系统通知、游戏结束画面统计数据标签、调试/开发者模式下的提示文字（若未完全隐藏于发行版），乃至Trophy/Achievement描述，均在验证范围之内。遗漏调试字符串导致其出现在发行版中，在历史上曾多次引发玩家投诉和媒体报道。

**误区三：一次验证即可覆盖整个开发周期**

字符串提取验证不是一次性任务，而是需要配合每次构建（Build）持续执行的回归测试项。每当开发团队新增功能、修改UI布局或集成第三方插件时，都可能引入新的硬编码字符串。建议在CI/CD流水线中集成自动化字符串扫描脚本，在每次提交后自动生成硬编码字符串警告报告。

## 知识关联

**前置知识：伪本地化**
字符串提取验证高度依赖伪本地化所建立的视觉反馈机制——伪本地化将已提取字符串转化为特殊字符，使硬编码字符串在界面中原形毕露。没有伪本地化环境，字符串提取验证只能依赖代码扫描，漏检率会显著上升。

**后续概念：上下文测试**
完成字符串提取验证并确保100%外部化后，下一步是上下文测试（In-context Review）：翻译人员或本地化测试员在实际游戏界面中检查译文是否在特定UI尺寸、字体和排版条件下呈现正确。字符串提取验证保证了"所有文本都可被翻译"，而上下文测试则进一步保证"翻译后的文本在界面中合适地显示"。两者形成从文本可达性到文本展示质量的完整验证链条。
