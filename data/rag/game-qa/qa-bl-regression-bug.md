# 回归Bug

## 概述

回归Bug（Regression Bug）是指已经被确认修复并关闭的Bug，在后续版本迭代或代码变更后重新出现的现象。区别于首次发现的新Bug，回归Bug的特征是它曾经存在过完整的修复记录，包括根因分析、修复提交（Commit）以及验收通过的测试记录，却因某种原因"死灰复燃"。在游戏QA领域，回归Bug的出现往往意味着某次代码合并、配置覆盖或资源替换打断了原有修复逻辑。

回归Bug这一术语在软件测试领域最早随"回归测试（Regression Testing）"概念一同被正式化，后者在1980年代随持续集成实践的推广而普及。在游戏开发中，由于游戏项目的版本分支复杂（主干Branch、DLC分支、平台适配分支等并行存在），不同分支的修复补丁未能同步Cherry-pick，是导致回归Bug的最高频原因之一。

回归Bug的危害不仅在于它本身，更在于它会严重消耗QA团队的信任资产。测试人员曾经验证通过的内容再次失效，会破坏对"已完成"状态的置信度，并且在临近上线节点时，回归Bug的重新发现往往触发紧急修复流程，带来额外的风险窗口。准确识别、追踪和预防回归Bug是衡量一个游戏QA流程成熟度的重要指标。

## 核心原理

### 回归Bug的成因分类

回归Bug的产生来源可归纳为以下四类，每一类在游戏项目中均有典型场景：

1. **代码覆盖（Code Overwrite）**：开发者A在分支B1修复了一个战斗伤害计算公式的Bug，但开发者B随后将分支B2的旧代码合入主干，直接覆盖了A的修复内容。这是最常见的机制，占据回归Bug成因的约60%。
2. **依赖变更（Dependency Change）**：修复逻辑依赖某个底层函数或数据结构，而该依赖在后续迭代中被重构或删除。例如，角色技能冷却的Bug修复依赖于`CooldownManager.Reset()`方法，但该方法在重构后被替换为`SkillTimerController.Clear()`，旧修复代码失效。
3. **配置/资源覆盖（Config Overwrite）**：游戏中大量Bug以调整数值配置表（如Excel或JSON数据文件）的方式修复，若美术或策划重新导出资源时未携带已修改的配置项，回归即发生。这在回归Bug中属于"无代码变动却复现"的典型情形。
4. **环境差异（Environment Divergence）**：测试环境与生产环境的差异导致Bug在某一环境重新触发。这与前置知识"环境信息"直接相关——记录Bug首次修复时的完整环境快照（引擎版本、SDK版本、平台OS版本、服务器配置），能够在环境差异导致的回归发生时快速定位。

### 回归Bug的识别标志

在Bug跟踪工具（如JIRA、Tapd）中，识别回归Bug需要核对以下三个条件同时成立：
- Bug状态历史中存在"已关闭（Closed）"或"已验证（Verified）"节点；
- 当前复现的症状与历史记录的症状描述一致（复现步骤重合度≥80%）；
- 复现发生在历史关闭时间**之后**的新版本Build上。

当三个条件均满足时，应在Bug工单中将类型标记为`Regression`，并强制填写"首次修复版本（First Fixed Build）"字段，而非开立新的Bug单。重复开单是回归Bug管理中最常见的流程错误，会导致统计数据失真。

### 回归测试与回归Bug检测的关系

预防回归Bug的核心手段是建立**回归测试用例库（Regression Test Suite）**。每一个被修复的Bug，在关闭前必须将其复现步骤转化为一条或多条可执行的测试用例，归入回归测试集。在后续每个版本的冒烟测试（Smoke Test）或专项测试轮次中，该测试集需被执行。

对于游戏QA而言，建议按优先级将回归用例分为两层：
- **P0回归集**：历史上引发线上事故（如服务器宕机、付费系统崩溃）的Bug对应的用例，每次提测必跑；
- **P1回归集**：主要玩法流程中的功能性Bug，按版本类型选择性执行。

若项目具备自动化测试条件，Unity项目可借助**Unity Test Framework**将P0回归用例转化为自动化脚本，在每次CI构建后自动运行，将回归检出时间从数天压缩至数小时。

## 实际应用

**案例1：技能连击系统回归Bug**

某手游战斗系统在版本2.3.0修复了一个"连击技能第3段伤害为0"的Bug，根因是连击索引越界。修复后关闭Bug单。在版本2.5.0，策划调整了技能数据配置表并重新导表，但配置表模板未包含2.3.0修复时新增的`combo_index_cap`字段，导致该字段被默认值0覆盖，Bug回归。测试人员通过比对环境信息中的配置文件MD5校验值，在30分钟内定位到根因。

**案例2：跨平台分支同步遗漏**

某主机游戏PC版本在Patch 1.2修复了存档文件路径写入权限Bug，但PS5适配分支未Cherry-pick该修复的Pull Request（PR #4471），PS5版本在Patch 1.3上线后玩家存档丢失，该问题作为回归Bug被处理，并追溯发现PR合并清单审核环节缺失"跨平台同步确认"步骤。

## 常见误区

**误区1：认为回归Bug是全新Bug，重新开单处理**

许多初级测试人员在发现症状相似的问题时，倾向于开立新单而非查阅历史记录。这会导致：（a）开发者进行重复的根因分析，浪费时间；（b）Bug统计数据中新增Bug数量虚高，回归率指标失真，影响项目健康度评估。正确做法是在开单前，先在Bug库中以关键词搜索历史已关闭单，确认是否属于回归。

**误区2：认为修复代码合入即可不建回归用例**

部分团队在Bug修复后直接关闭，不沉淀回归测试用例，导致相同Bug在后续4-5个版本内反复出现。以某RPG项目实际数据为例，未建立回归用例库的项目，回归Bug占总Bug数的比例可高达18%-25%；建立P0+P1回归用例库并坚持执行的项目，该比例可降至5%以下。

**误区3：认为自动化测试能100%覆盖所有回归**

自动化测试脚本本身存在维护成本，若游戏UI或玩法逻辑发生较大重构，旧的自动化脚本可能失效，形成"测试覆盖假象"——脚本通过了，但实际功能已回归。因此回归测试应保持自动化与手动探索性测试（Exploratory Testing）的结合，尤其在大版本迭代后针对历史高风险模块进行人工复核。

## 知识关联

回归Bug的准确识别高度依赖**环境信息**的完整记录。Bug首次修复时所在的Build号、引擎版本、平台版本、关键配置文件的哈希值，构成判断"同一Bug是否再次出现"的基准线。若环境信息缺失，测试人员将无法区分"相同Bug在新环境回归"与"新环境引发的全新Bug"，导致错误分类。

在下游知识领域，回归Bug的数据是**Bug统计分析**的重要原始输入。回归率（Regression Rate = 回归Bug数 / 已关闭Bug总数 × 100%）是衡量代码变更管理质量和回归测试覆盖率的核心KPI指标。高回归率往往指向代码审查（Code Review）流程薄弱、分支合并策略混乱或回归测试执行频率不足等系统性问题，需要在Bug统计分析阶段进行专项归因。
