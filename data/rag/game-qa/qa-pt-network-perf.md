# 网络性能测试

## 概述

网络性能测试是针对多人游戏、在线功能及云游戏等场景的专项测试方法，通过模拟真实网络环境中的延迟（Latency）、带宽限制（Bandwidth）、丢包（Packet Loss）和抖动（Jitter）四类核心指标，量化这些网络状况对玩家体验的具体影响。与加载时间测试关注本地磁盘/内存吞吐不同，网络性能测试的被测对象是客户端与游戏服务器之间的数据传输质量。

该测试方法随网游产业成熟而标准化。2004年前后，随着《魔兽世界》等大型MMO上线，游戏公司开始系统性地建立网络测试规范，并引入网络仿真工具（如Netem、Clumsy）制造可复现的劣质网络条件。时至今日，这套测试体系已扩展至移动游戏、实时对战（RTS/FPS/BR）以及云游戏流媒体传输等场景。

对游戏QA而言，网络性能测试的核心价值在于：通过在受控环境下注入具体的网络损伤参数，找出游戏在哪个网络阈值下会出现操作失灵、角色位置漂移、同步错误或崩溃，从而为开发团队提供可量化的优化依据，而非仅凭玩家投诉定位问题。

## 核心原理

### 四大网络指标的定义与游戏影响阈值

**延迟（RTT/Ping）**是从客户端发送数据包到收到服务器响应的往返时间，单位为毫秒（ms）。对FPS类游戏（如《CS:GO》《Valorant》），通常认为：
- 0–60ms：正常，玩家几乎无感知
- 61–120ms：轻微延迟，射击判定可见偏差
- 121–200ms：明显卡顿，客户端预测（Client-side Prediction）补偿失效
- 200ms+：操作明显滞后，角色橡皮筋现象（Rubber-banding）频繁出现

**带宽（Bandwidth）**决定单位时间内可传输的数据量，常见多人射击游戏上行带宽需求约为32–128Kbps/玩家，MMORPG因地图状态同步可达512Kbps以上。当带宽被限制到低于游戏最低需求时，优先级低的数据包（如环境音效、非关键NPC位置）会被丢弃，测试需记录哪些游戏逻辑在带宽压缩下最先降级。

**丢包（Packet Loss）**以百分比表示，即在传输过程中未到达目标的数据包比例。实测数据显示，对实时对战游戏，丢包率超过**3%**时玩家开始感知操作不流畅；超过**10%**时大多数游戏的UDP同步机制会触发重传逻辑或状态回滚，严重影响对局公平性。

**抖动（Jitter）**是延迟的标准差，即每个数据包到达时间的波动幅度。即使平均延迟为50ms，若抖动达到±30ms，数据包会以20ms和80ms交替到达，导致游戏内动画插值（Interpolation）出现跳帧，其主观体验往往比稳定的高延迟更差。

### 网络损伤注入方法

测试工程师使用专用工具在本地或测试机器上人工制造网络损伤：
- **Clumsy**（Windows）：图形界面，可实时调节丢包率、延迟（ms）、乱序（Out-of-order）比例
- **tc-netem**（Linux）：内核级别的网络仿真，命令示例：`tc qdisc add dev eth0 root netem delay 100ms 20ms loss 5%`，表示添加100ms±20ms延迟和5%丢包
- **Charles Proxy / Fiddler**：用于HTTP长连接或游戏内购流程的带宽限速测试

测试时须同时在服务端和客户端部署监控，因为单侧延迟与双侧延迟对游戏逻辑的影响机制不同（上行丢包影响玩家输入上报，下行丢包影响游戏状态同步）。

### 游戏协议层分析

绝大多数实时对战游戏使用**UDP**而非TCP，因为UDP无内置重传机制，延迟更低但需要游戏引擎自行处理丢包补偿。测试人员需验证游戏的UDP可靠层（如ENet、QUIC、Raknet）在设计的丢包率范围内是否能正确触发状态对齐，而不是无限重传导致内存溢出。部分游戏内置网络诊断面板（如《Apex英雄》的网络统计叠加层），测试时可直读实时RTT、Tickrate和包大小数据。

## 实际应用

**FPS游戏延迟阈值回归测试**：以20ms为步长，从20ms递增至300ms，每档延迟下记录命中判定一致性（客户端命中率 vs 服务端确认命中率之差）、角色移动预测失误次数及客户端崩溃日志。《Valorant》官方公布其服务器Tickrate为128Hz，对应每帧同步间隔约7.8ms，因此测试延迟阶梯需细化到16ms精度才能覆盖关键帧边界。

**移动网络切换测试**：模拟4G→Wi-Fi或5G信号弱区场景，通过Clumsy注入持续2–5秒的100%丢包窗口，验证游戏是否在断线重连后能正确恢复进度，而不是将玩家踢出对局或保留异常数据。

**带宽竞争场景**：在同一局域网内同时运行8个游戏客户端，将路由器出口带宽限制到10Mbps，测试服务器是否启动带宽公平调度（Fair Queuing），以及低带宽用户是否被游戏判定为"作弊"而踢出（误判问题）。

## 常见误区

**误区一：用Ping值等同于游戏延迟体验**
ICMP Ping测量的是网络层RTT，而游戏实际体验延迟还叠加了服务器处理时间（Server Processing Time）、Tickrate周期等待时间和客户端帧渲染时间。一个Ping为30ms的玩家在Tickrate为20Hz（50ms/帧）的服务器上，实际输入响应延迟可高达80ms以上。测试报告应区分"网络RTT"和"端到端输入延迟"这两个不同指标。

**误区二：丢包率越低越好，0%丢包无需测试**
即使在局域网0%丢包环境下，游戏仍需通过网络性能测试验证：乱序数据包（Packet Reordering）是否触发错误的状态回滚，以及数据包大小突然增大（如进入高密度战场）是否导致MTU分片引发隐性延迟。仅在零丢包条件下通过的游戏，上线后在真实互联网环境中往往暴露深层协议缺陷。

**误区三：抖动问题可以用提高缓冲区解决**
加大接收缓冲区（Jitter Buffer）确实能平滑抖动，但代价是引入固定的额外延迟（Buffer Delay）。测试必须验证游戏采用的抖动缓冲策略是否在"平滑度"与"响应延迟"之间取得可接受的平衡，而不是无限扩大缓冲区掩盖问题。

## 知识关联

**前置依赖**：加载时间测试建立了性能数据采集的基础框架（帧计时、日志埋点）；多人游戏测试验证了游戏逻辑在多客户端并发下的正确性，这两者共同为网络性能测试提供了"正常基线"——只有确认本地性能和逻辑正确后，才能将测试中暴露的问题准确归因到网络层。

**后续发展**：网络性能测试的测试规模扩大即演变为压力测试（Stress Testing）——当成百上千个并发连接同时施加网络损伤时，不再只观察单个玩家体验，而是验证服务器在极端网络条件下的整体容量上限、级联故障模式，以及负载均衡策略是否能在高丢包环境中自动隔离问题节点。
