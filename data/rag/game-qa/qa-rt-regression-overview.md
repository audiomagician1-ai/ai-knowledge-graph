# 回归测试概述

## 概述

回归测试（Regression Testing）是指在软件发生修改之后，重新运行已有的测试用例集合，以验证原有功能未因本次变更而遭到破坏的测试方法。"回归"二字来源于数学统计中的"向均值回归"概念，在软件工程语境中特指代码变更导致系统"退步"或"倒退"到错误状态的现象。1979年，Glenford Myers 在其著作 *The Art of Software Testing* 中系统描述了这一现象，将其定名为回归缺陷（Regression Bug），回归测试随之成为独立的测试类别。

回归测试的核心价值在于：每一次代码提交都是一次潜在的风险引入点。游戏开发中，一名程序员修复了角色跳跃手感的 Bug，可能无意中影响了下蹲、爬行等使用同一物理运算模块的功能；一次资源包热更新可能导致旧存档读取失败。回归测试的存在使团队可以在这些问题对玩家可见之前主动发现并修复它们，而不是依赖玩家的投诉报告。

在游戏质量保障（QA）体系中，回归测试占据版本发布前工作量的相当大比重——业界调查数据显示，成熟项目的回归测试用例数量通常占总用例库的 60%~80%，其执行频率也远高于其他测试类型。

## 核心原理

### 变更触发机制

回归测试的启动条件是"变更"，具体包括：代码修改（Bug 修复、新功能开发、性能优化）、配置文件更改（如调整角色数值平衡表）、依赖库或引擎版本升级，以及资源替换（模型、音效、Shader 等）。并非所有变更都需要全量回归——识别哪些变更对应哪些功能模块是回归测试策略设计的关键，这一过程称为"影响分析"，是回归测试的直接下游知识点。

### 回归缺陷的产生原理

回归缺陷通常由三类原因产生：
1. **直接依赖破坏**：被修改的函数 A 被函数 B、C 直接调用，A 的接口或行为改变导致 B、C 出错。例如修改 `CalculateDamage()` 函数的返回值精度从 `int` 改为 `float`，调用该函数的伤害结算 UI 显示出现异常。
2. **共享状态污染**：两个功能共享同一全局变量或单例对象，修改其中一个功能时意外改变了共享状态的初始化时机，导致另一功能在特定顺序执行时出错。
3. **编译/构建副作用**：在 C++ 游戏项目中，头文件的修改会触发大量依赖该头文件的源文件重新编译，若修改引入了宏定义冲突或预处理器行为变化，回归问题可能出现在完全不相关的模块中。

### 回归测试与全量测试的区别

全量测试（Full Testing）是对产品所有功能进行完整验证，理论上覆盖率接近100%，但在大型游戏项目中执行一次全量测试可能需要数天时间，无法适应每日多次提交的开发节奏。回归测试通过选择与本次变更相关的子集用例来换取执行速度——一个设计良好的回归测试套件应在 30 分钟至 2 小时内完成执行，以配合持续集成（CI）流水线的节奏。

## 实际应用

**场景一：每日构建验证（Daily Build Smoke Regression）**
某手游项目采用 Jenkins CI，每次代码合并到主干分支后自动触发一套包含约 200 个自动化用例的冒烟回归套件，覆盖登录、关卡加载、战斗核心逻辑、存档读写四个最高优先级模块。该套件平均运行时长为 18 分钟，若失败则通知提交者立即处理，阻止后续合并。

**场景二：版本发布前全量回归**
在游戏正式版本（Gold Master）提交前，QA 团队会执行完整的回归测试包，其中包含历史版本中出现过的所有已修复 Bug 对应的回归用例（称为 Bug Regression Case）。这些用例的核心价值在于防止"已知缺陷复发"——据统计，约有 15%~20% 的 Bug 在后续版本中会因为代码重构而再度出现。

**场景三：DLC 与热更新后的局部回归**
当游戏发布内容更新包（DLC）或热补丁时，回归测试不需要覆盖整个游戏，而是聚焦于更新涉及的关卡、角色、系统。例如新增角色时，需对战斗系统、成就系统、存档系统、UI 系统进行针对性回归，而不触碰渲染管线等无关模块。

## 常见误区

**误区一：回归测试等同于重复测试，价值低于探索性测试**
这是新手 QA 最常见的认知偏差。回归测试用例并非随意的重复，它们是历史缺陷和风险点的结晶——每一条用例背后对应一个真实发生过的线上问题或高风险路径。探索性测试擅长发现新类型缺陷，而回归测试专门防御已知风险的复发，两者功能互补，不可替代。

**误区二：自动化覆盖率越高，回归测试就越完善**
自动化脚本只能验证可被程序化断言的结果，例如数值、状态标志、崩溃与否。游戏中大量依赖视觉感受的内容——如角色动画过渡的流畅度、光照效果的还原度、音效的播放时机——难以被自动化脚本准确验证。因此，成熟的游戏回归测试体系通常采用"自动化+人工手动核查"的混合模式，而非纯自动化。

**误区三：没有新功能就不需要执行回归测试**
即使本次迭代仅包含 Bug 修复，也必须执行回归测试。Bug 修复是代码变更，代码变更就存在引入回归缺陷的可能性。实际上，针对复杂系统的 Bug 修复（如修改游戏底层物理引擎参数）所引发的回归风险，有时远高于新增简单独立功能所带来的风险。

## 知识关联

学习回归测试概述不需要任何测试领域的前置知识，它本身即是理解游戏 QA 流程的起点。掌握"什么是回归测试、为何执行、何时执行"之后，下一步需要学习**影响分析（Impact Analysis）**——即如何系统性地判断一次代码变更影响了哪些功能模块，从而决定回归测试的范围与优先级，避免既遗漏高风险区域，又不浪费资源执行与本次变更完全无关的测试用例。影响分析将回归测试从"无差别重跑所有用例"提升为精准、高效的工程实践。
