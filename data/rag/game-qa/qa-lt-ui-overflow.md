# UI文本溢出

## 概述

UI文本溢出（Text Overflow / String Truncation）是指游戏界面中的控件容器无法容纳翻译后文本时，产生截断、换行异常或视觉边界突破等排版故障的现象。这一问题在本地化测试中尤为突出，因为不同语言的字符信息密度差异极大：德语单词平均比英语长30%~40%，芬兰语因其黏着语属性可能产生超过25个字母的合成词，而中文和日文虽然单字符信息量高，但全角字符宽度是半角的两倍，同样容易在固定宽度容器中触发溢出。

文本溢出问题的工程根源来自UI设计阶段使用英语作为基准语言。英语原文通常是所有主要目标语言中字符数最短的语言之一，UI控件的尺寸依据英语字符串规划，导致翻译版本普遍面临"空间不足"的结构性矛盾。1990年代末期，随着跨区域同步发售（Simultaneous Shipment）策略的普及，开发商开始建立专门的本地化QA流程，文本溢出检测成为其中的必检项目。

在实际测试中，文本溢出可造成玩家无法阅读完整的任务说明、技能描述被截断导致数值信息缺失（如"每秒造成15点…"被截为"每秒造成"），严重损害游戏体验并引发用户投诉。对于ESRB/PEGI等分级审查机构的内容要求，若警告文字因溢出被截断，还可能带来法律合规风险。

## 核心原理

### 溢出的三种表现形式

文本溢出在UI层面有三种典型表现，测试人员需逐一区分：

**截断（Truncation）**：文本超出容器边界后被强制切断，通常以省略号（…）或直接消失两种形式出现。省略号模式下，玩家知晓内容不完整；无省略号的硬截断则最危险，玩家可能误以为信息完整。

**意外换行（Unintended Line Break）**：容器允许多行显示，但文本过长导致行数超过UI设计的最大行数限制，底部文字被剪裁。例如英语版为2行的技能描述，德语版因文本扩展（Text Expansion）膨胀至3行，第3行落在控件可视区外。

**控件溢出（Widget Overflow）**：文本撑破控件边界，覆盖到相邻UI元素（按钮、图标、血条等）上，造成视觉重叠或点击区域失效。这在对话框中的按钮文字（如"确认"对应德语"Bestätigen"）上极为常见。

### 文本扩展率参考公式

本地化工程界使用**文本扩展率（Text Expansion Rate）**量化溢出风险。IBM于1994年在内部本地化规范中提出了一套基于源字符串长度的估算公式：

- 源字符串 ≤10个字符：扩展率约为 **200%~300%**
- 源字符串 11–20个字符：扩展率约为 **180%~200%**
- 源字符串 21–30个字符：扩展率约为 **140%~180%**
- 源字符串 >70个字符：扩展率约为 **130%**

公式意义：若一个按钮标签英语原文为"OK"（2字符），目标语言对应译文空间需预留最多原来3倍的像素宽度。测试人员在编写测试用例时，应优先对源字符串≤15字符的短标签进行溢出专项验证。

### 溢出检测的测试层级

**静态字符串检测**：在实际游戏运行前，通过解包游戏资源提取所有本地化字符串，使用脚本计算每条字符串的像素宽度（需结合目标语言的字体文件和字号参数），与UI设计文档中的控件最大宽度进行比对，标记超出阈值（通常设为控件宽度的90%）的条目。

**运行时视觉验证**：直接在游戏内截图，使用图像差分工具或人工目视检查每个界面，确认文字是否超出控件边界。该方法能捕捉到动态加载内容（如玩家名称、服务器端配置文本）的溢出，是静态检测的必要补充。

## 实际应用

**RPG技能描述界面**：《最终幻想XIV》在欧洲多语言版（法语、德语）上线初期，大量技能描述文本在德语版中因文本扩展超出技能图鉴的固定高度容器，导致效果描述末尾数值被截断。测试规范要求每条技能字符串需以最长预估语言（通常是德语）为基准验证容器高度。

**按钮与菜单标签**：游戏主菜单的"多人游戏"按钮在英语下为12字符，在俄语下译文"Многопользовательская игра"达26个西里尔字符，加上西里尔字体字形通常比拉丁字体宽，实际渲染像素宽度可能超出按钮控件的150%，导致文字溢出按钮边框或被强制缩小到不可读的字号。

**移动端竖屏UI**：手机游戏受屏幕宽度限制（通常为360dp~414dp），横向空间极为有限。阿拉伯语RTL（从右到左）排版不仅要求文本方向翻转，其字母连写特性使某些词组渲染宽度比英语同义词宽20%~35%，在移动端常见的单行标签控件中极易触发截断。

**动态生成字符串**：许多游戏使用变量插值格式如`"你已击败了 {enemy_name}!"`，当enemy_name为游戏内的中文怪物名（可能因版本更新随时变更）时，静态测试无法覆盖所有情况，需要使用边界值测试方法，以已知最长怪物名字符串注入测试环境进行专项验证。

## 常见误区

**误区一：翻译审校通过即代表无溢出问题**
语言质量测试（LQT）确认译文语义准确、语法正确，但不负责验证排版适配。一段翻译质量优秀的德语文本完全可以同时触发UI溢出。文本溢出属于**功能性本地化缺陷（Functional Localization Bug）**，而非翻译错误，需要独立的UI测试流程覆盖，不能以语言审校替代。

**误区二：只测试最长语言即可**
许多测试工程师误认为德语是"最坏情况"，测过德语就等于覆盖了所有语言。事实上，泰语和缅甸语的垂直排版需求（部分游戏使用竖排布局）、阿拉伯语的双向文本（BiDi）渲染、韩语音节字符的全角宽度，都可能在德语未触发溢出的控件位置引发溢出。正确做法是对**每个目标语言单独进行文本溢出验证**，或使用伪本地化（Pseudo-localization）工具生成包含变音符、宽字符的模拟字符串进行早期批量扫描。

**误区三：省略号省略就是"处理了溢出"**
开发者在控件中配置了`text-overflow: ellipsis`或等效的游戏引擎属性（如Unity的`TextMeshPro`的`Overflow: Truncate`模式），认为有省略号就不算Bug。实际上，若截断后玩家无法获知完整信息（尤其是数值型技能参数、操作说明），该省略号截断仍属于高优先级缺陷，需要提供可展开查看全文的交互设计或调整UI布局。

## 知识关联

**前置概念——语言质量测试**：语言质量测试负责确认各语言译文的准确性，为UI文本溢出测试提供已确认的最终译文字符串。若在溢出测试阶段发现文本过长，有时可通过申请**简短替代译文（Short Alternative String）**在不修改UI的前提下解决问题，这需要LQT团队配合评估缩短后的译文是否仍满足语义完整性要求。

**后续概念——字体渲染**：文本溢出的像素宽度计算依赖具体字体的字形度量数据（Glyph Metrics），同一段文字在不同字体下渲染宽度差异可达10%~20%。学习字体渲染后，测试人员能理解为何字体文件更换会导致原本通过测试的布局重新出现溢出，以及如何利用字体回退（Font Fallback）机制的宽度特性预测溢出风险。

**后续概念——分辨率与宽高比**：UI控件的绝对像素尺寸随分辨率和宽高比变化。在1920×1080分辨率下未溢出的文本，在1280×720或21:9超宽屏下因UI缩放策略不同，控件宽度可能减少15%~30%，重新触发溢出。分辨率与宽高比测试是文本溢出验证的多维度扩展，两个测试维度需交叉覆盖。
