# 复数/性别变化

## 概述

复数/性别变化是游戏本地化测试中专门验证不同语言对名词数量形式及语法性别的正确处理能力的测试领域。当游戏UI显示"你获得了1个金币"与"你获得了5个金币"时，英语只需在词尾加"-s"，但波兰语要求测试人员验证单数（1 moneta）、小数量复数（2-4 monety）、大数量复数（5+ monet）三种完全不同的词形，任何一种处理错误都会导致语法错误直接呈现给玩家。

这一测试需求源于20世纪90年代游戏全球化浪潮中的惨痛教训——早期开发者习惯将英语"1 item / N items"的简单二元复数逻辑硬编码到字符串系统中，导致俄语、阿拉伯语游戏版本出现大量语法错误。Unicode CLDR（Common Locale Data Repository）项目为解决这一混乱，于2005年开始系统整理各语言复数规则，目前已收录74种不同语言的复数类别标准，测试人员需以此为权威参考。

在角色扮演游戏和策略游戏中，物品数量、角色计数、伤害数值的动态显示几乎无处不在，加之德语、法语、西班牙语等主流发行市场均存在语法性别体系，性别变化错误会影响形容词、冠词、过去分词的形态，使一句话中多个单词同时出错，测试覆盖的复杂度远超简单的字符显示验证。

## 核心原理

### 复数规则的语言分类差异

Unicode CLDR将世界语言的复数规则划分为六个类别：zero（零）、one（单数）、two（双数）、few（少量）、many（多量）、other（其他）。英语仅使用one和other两类，而阿拉伯语全部六类均有使用，测试阿拉伯语版本时必须构造0、1、2、3、11、100这六个不同数值的测试用例，才能覆盖所有形态变化。

波兰语的复数规则尤为典型，其规则公式可描述为：n=1时用单数形（sztuka），n%10在2-4范围内且n%100不在12-14范围内时用少量复数形（sztuki），其余情况用多量复数形（sztuk）。这意味着游戏中显示"22个物品"与"12个物品"所用的词形截然不同，测试人员必须专门准备12、13、14这组"陷阱数值"验证正确的例外处理。

俄语与波兰语共享斯拉夫语系的三级复数结构，但细节规则不同：俄语中n=21时重新使用单数形（21 монета），而非像英文那样在21处继续使用复数形，测试用例必须覆盖21、22、25、111、112这类边界数值，不能仅测试个位数。

### 语法性别对本地化字符串的级联影响

德语名词拥有阳性（der）、阴性（die）、中性（das）三种语法性别，且性别属性是任意的而非语义推导的——"女孩"（das Mädchen）为中性，"钥匙"（der Schlüssel）为阳性。当游戏动态插入道具名称时，形容词词尾必须根据道具的语法性别进行屈折变化：`ein großer Schwert`（错误）应为`ein großes Schwert`（正确，"剑"为中性），测试需为每类语法性别至少各选一件代表道具构造测试串。

法语动态语句中，过去分词与性别的一致性问题是高频错误点。"角色已升级"若用代词指代女性角色，需写成"Elle est montée de niveau"，其中montée带有-e后缀，而指代男性角色时为monté。游戏若直接复用同一字符串模板而未根据角色性别切换变体，法语母语玩家会立刻察觉语法错误。

### 字符串变体系统的测试验证方法

现代游戏引擎（如Unity的Localization Package和Unreal的FText系统）提供了`Plural Form`和`Gender Form`变体表，开发者预先为每个复数/性别类别填写对应译文。测试人员的职责包括：一是验证变体表中每个槽位均已填写且无遗漏（空槽位通常显示为英文原文或乱码）；二是通过构造边界数值确认代码层的选择逻辑与变体表匹配；三是核查本地化字符串中的占位符顺序——德语句法允许将数字放在名词后方，若开发者硬编码了`{0} {1}`的顺序则会产生语序错误。

## 实际应用

《魔兽世界》在推出俄语版本（2008年）初期曾因复数处理不足出现大量"5 штук меч"（5把剑）的语法错误，后续版本引入了专用的俄语复数选择器模块修正了该问题，此案例成为业界讲解斯拉夫语复数测试的标准参考案例。

在实际测试流程中，针对西班牙语版本应专门测试物品描述中的性别一致性：西班牙语中"红色的盾牌"为"escudo rojo"（阳性），而"红色的剑"若"espada"为阴性则需写成"espada roja"。测试人员需建立一份覆盖游戏内全部可动态组合形容词的道具性别对照表，逐一核验UI中的形容词词尾。

移动游戏中的每日任务描述是另一个高密度测试区域，例如"完成3场战斗"，波兰语中3对应少量复数，应使用"bitwy"而非单数"bitwa"或多量复数"bitew"，测试任务完成数量从0到5的每一个值都能正确切换词形。

## 常见误区

**误区一：认为只需测试单数和复数两种情况。** 许多初级测试人员沿用英语思维，只验证数量为1和数量为2（或10）的显示结果，完全忽略了波兰语、俄语、捷克语等语言存在三个以上复数级别的事实。CLDR文档明确指出，至少需要针对目标语言的每一个复数类别（zero/one/two/few/many/other）分别构造测试数值，阿拉伯语最少需要6个独立测试用例。

**误区二：将语法性别等同于自然性别，认为人物名词性别问题仅出现在角色相关文本中。** 德语、法语的语法性别附着于所有名词（包括道具、地名、技能名称），与现实性别无关。游戏道具"护甲"在德语中是阴性（die Rüstung），若测试人员只检查角色对话中的性别一致性而忽略装备描述文本，会漏掉大量形容词词尾错误。

**误区三：假设游戏引擎自动处理了复数选择逻辑，本地化文件只需填写一种形式。** 实际工程中，Unity旧版本（2020.3之前）的本地化插件对非英语复数规则支持有限，译者填写了多个变体但引擎始终调用"other"槽位的情况时有发生。测试人员必须在真实游戏构建中运行验证，而不能仅审查本地化电子表格。

## 知识关联

在掌握输入法兼容测试的基础上，测试人员已熟悉不同语言字符集在游戏文本框中的输入和显示机制，复数/性别变化测试在此之上进一步要求验证动态拼接字符串的语法正确性，而非仅关注静态字符的渲染。输入法测试确保了字符能正确进入游戏系统，复数/性别测试则确保系统在运行时将这些字符以符合语法的形态呈现给玩家。

复数/性别变化测试的大量测试用例设计（边界数值列表、性别对照表、变体槽位清单）本身具有高度的规律性和可脚本化特征，这为进阶到本地化自动化测试奠定了直接的实践基础——自动化框架可以读取CLDR规则配置、自动生成覆盖所有复数类别的测试数值、通过截图比对验证UI字符串，从而将原本依赖人工逐一核查的复数形态验证转化为可持续运行的回归测试套件。
