# 测试框架设计

## 概述

测试框架设计是自动化测试的骨架，它规定了测试代码的组织方式、数据流向和复用策略。在游戏QA领域，测试框架不是单一脚本的堆砌，而是一套预定义的规则体系，决定着测试人员如何编写用例、如何传递参数、以及如何在多个游戏版本间复用逻辑。框架选型错误会导致日后维护成本呈指数级增长——当游戏UI改版时，如果没有框架层的隔离，可能需要修改数百个测试脚本。

测试框架的三大经典模式——Page Object Model（POM）、关键字驱动（Keyword-Driven）和数据驱动（Data-Driven）——分别诞生于不同的行业需求背景。POM由Selenium核心贡献者Simon Stewart在2009年前后提出，最初用于Web测试，后被游戏客户端UI测试广泛借鉴。关键字驱动最早可追溯至1990年代的HP WinRunner工具链，其核心思想是将操作语义（如"点击登录按钮"）与底层API解耦。数据驱动则随着参数化测试工具（如Python的`pytest.mark.parametrize`）的成熟而普及，允许同一套测试逻辑跑遍数十种角色配置、装备组合或地图参数。

在游戏QA中，正确选择框架模式直接影响项目交付速度。以一款含100个UI界面的手游为例，若采用无框架的线性脚本，界面改版时平均每个界面需修改7~15处代码；而POM框架下只需修改对应的Page类，改动量降至1~3处。

## 核心原理

### Page Object Model（POM）

POM的核心思想是"每一个可测试的界面对应一个类"，该类封装该界面的元素定位器和操作方法，测试用例只调用Page类的方法而不直接操作底层控件。以游戏登录界面为例：

```python
class LoginPage:
    USERNAME_INPUT = "//input[@id='username']"
    PASSWORD_INPUT = "//input[@id='password']"
    LOGIN_BTN     = "//button[@id='login']"

    def login(self, user, pwd):
        self.driver.find(self.USERNAME_INPUT).send_keys(user)
        self.driver.find(self.PASSWORD_INPUT).send_keys(pwd)
        self.driver.find(self.LOGIN_BTN).click()
```

当登录按钮的ID从`login`改为`btn_login`时，只需修改`LOGIN_BTN`一处常量，所有调用`login()`的用例自动生效。POM的层级通常分三层：**定位器层**（XPath/控件树路径）、**操作层**（Page类方法）、**业务层**（测试用例）。

### 关键字驱动框架

关键字驱动将测试动作抽象为业务语义词汇表，例如`点击背包`、`拖拽装备到槽位1`、`验证角色血量等于500`。测试用例以表格形式编写，每行包含**关键字名称**和**参数列表**，由框架引擎查表映射到具体函数执行。

| 步骤 | 关键字 | 参数1 | 参数2 |
|------|--------|-------|-------|
| 1 | 打开背包 | — | — |
| 2 | 拖拽装备 | 长剑 | 槽位1 |
| 3 | 验证属性 | 攻击力 | 150 |

这种模式的最大优势是**非编程人员也能编写测试用例**，适合游戏策划或测试策划参与用例设计。引擎层通常用Python字典或Robot Framework的关键字库实现，关键字到函数的映射表即框架的核心注册机制。

### 数据驱动框架

数据驱动将测试逻辑与测试数据完全分离，数据存储在外部文件（CSV、Excel、JSON或数据库）中，框架读取数据后对同一测试逻辑执行多次迭代。游戏中典型场景是**角色养成路径回归测试**：同一套"充值→购买道具→验证属性变化"逻辑，需要覆盖6个职业、3个服务器区域、4种支付渠道，共72种组合。数据驱动框架通过参数化注入，无需编写72个独立用例。

数据驱动的公式可以简化为：

```
用例总数 = 逻辑模板数 × 数据行数
```

维护时只需更新数据文件，逻辑脚本零修改。pytest中实现方式为：

```python
@pytest.mark.parametrize("role,server,pay", load_csv("testdata.csv"))
def test_purchase(role, server, pay):
    ...
```

### 混合框架（Hybrid Framework）

实际游戏项目中，三种模式常组合使用：以POM管理界面对象，以关键字驱动暴露给策划团队，以数据驱动覆盖多配置组合。例如网易某手游自动化测试项目采用"POM + 数据驱动"的混合架构，将用例数量从手工维护的3200个压缩到约400个参数化模板，覆盖率反而提升了18%。

## 实际应用

**副本关卡战斗验证**：游戏中有50个副本，每个副本的战斗逻辑相似但参数（怪物血量、掉落物、通关条件）不同。使用数据驱动框架，将50组副本参数存入Excel，单一测试函数循环执行50次，每次验证通关后的掉落物ID和角色状态值是否与策划配表一致。

**商店UI改版回归**：游戏大版本更新时商城界面重构，采用POM框架后，QA团队只需更新`ShopPage`类的12个元素定位器，无需改动调用了`ShopPage`的38个测试用例，当天完成回归测试。

**跨语言版本测试**：游戏面向中、英、日、韩四个市场，关键字驱动框架中"验证按钮文本"关键字封装了语言判断逻辑，测试用例本身不含任何硬编码字符串，同一套用例表驱动四语言版本的UI回归。

## 常见误区

**误区一：POM等同于把所有代码塞进Page类**。POM的Page类应只包含元素定位和单步操作，不应包含断言逻辑或多步业务流程。一旦Page类中出现`assert`或复杂的条件分支，说明业务逻辑下沉到了定位层，造成层次污染，后续很难单独维护。

**误区二：数据驱动的数据越多越好**。数据驱动的价值在于覆盖有意义的边界值和等价类，而非穷举。游戏装备强化测试中，强化等级1~15级若无特殊阈值，仅需测试等级1、7（中间值）、14（最大-1）、15（最大）和16（超出范围）共5个典型值，而非15个连续用例。数据行数过多会使每次CI执行时间超出预算，导致框架被放弃。

**误区三：关键字驱动框架能完全替代编程知识**。关键字库的**开发和维护**仍需要编程能力。当游戏新增一种操作类型（如陀螺仪旋转控制），必须由自动化工程师扩展关键字库，测试策划才能使用新关键字。误认为"只需维护表格"会导致关键字库长期滞后于游戏功能迭代。

## 知识关联

**前置知识**：测试框架设计的实现依赖脚本语言的面向对象特性——POM中的Page类需要类继承和属性封装，Python的`class`语法或Lua的元表（metatable）机制是基础。关键字驱动框架的注册表本质是字典（dict）数据结构，数据驱动依赖文件I/O操作，这些均属脚本语言选择阶段应掌握的能力。

**后续应用**：本章设计好的框架架构将直接影响CI/CD集成的实现难度。数据驱动框架的用例生成策略需与Jenkins/GitHub Actions的矩阵构建（Matrix Build）特性对应，POM框架的层次结构决定测试报告中失败定位的粒度，合理的框架设计能使每次代码提交后的自动化验证管道在5~15分钟内给出反馈，这是移动游戏快速迭代周期的基本要求。
