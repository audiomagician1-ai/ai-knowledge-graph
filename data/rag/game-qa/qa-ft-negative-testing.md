# 负面测试

## 概述

负面测试（Negative Testing）是游戏功能测试中专门针对**非预期输入、非法操作和极端边界条件**的系统健壮性验证方法。与验证"正确流程是否正常工作"的正面测试不同，负面测试的核心目标是验证游戏在遭受错误操作时**不会崩溃、不会产生脏数据、不会给玩家带来可利用的异常状态**。例如，测试玩家在背包已满时强制丢弃物品、在对话触发期间强制关闭游戏，或输入超出数值边界的角色属性值。

负面测试的系统化实践可追溯至1990年代软件容错工程领域，游戏行业在2000年代主机世代逐渐将其纳入QA流程，原因是主机平台的认证要求（如索尼PlayStation的Submission Requirements）明确规定游戏在异常输入下不得出现系统崩溃或死机。时至今日，主流游戏发行平台的上线审核均包含负面测试场景的强制验证项目。

在游戏开发语境中，负面测试直接决定游戏的**防外挂能力、容错体验和数据安全性**。一次遗漏的负面测试场景——例如未验证网络断线时交易系统的回滚逻辑——可能导致玩家物品凭空消失或复制，造成经济系统崩溃，这类事故的修复成本远高于测试本身的投入。

## 核心原理

### 边界值攻击（Boundary Value Attack）

负面测试最常用的技术是针对数值边界施加极端输入。以游戏角色属性为例，若攻击力有效范围为1–9999，测试用例必须覆盖：输入0、输入10000、输入-1、输入小数（如99.5）、输入字符串（如"MAX"）。当输入值为整数上界+1时，系统应返回错误提示而非将数值截断为9999或产生整数溢出（Integer Overflow）——后者在32位有符号整数下会使9999+1变为-2147483648，直接导致角色攻击力为负数，属于高危Bug。

### 非法状态切换测试

游戏内存在大量状态机逻辑，负面测试需要验证**跳过中间状态的强制切换**是否会导致系统异常。典型场景包括：在载入界面未完成时按下确认键、在技能释放动画播放的第3帧时强制切换场景、在NPC交互对话窗口打开期间触发角色死亡事件。这类测试通常需要测试人员以每秒15次以上的操作频率连续触发，或借助自动化脚本模拟快速状态切换，目的是暴露状态机设计中未处理的边界过渡（Unhandled Transition）。

### 异常网络与资源条件

对于联机游戏，负面测试必须模拟以下网络异常：网络延迟突增至500ms以上、数据包丢失率达30%、在服务器写入操作进行中断开连接。本地单机游戏则需测试磁盘写满（可用空间为0字节）时存档操作的处理方式——合格的处理结果是弹出明确错误提示并保留上一次存档完整性，而非静默失败或覆盖损坏存档。测试工具层面，可使用Network Link Conditioner（macOS）或Clumsy（Windows）人为制造上述网络条件。

### 输入注入与非法字符测试

针对所有文本输入框（角色名称、公会名称、聊天频道），负面测试需提交以下内容验证过滤机制：超长字符串（如256个汉字，覆盖UTF-8编码的3字节场景）、SQL注入字符串（`'; DROP TABLE users;--`）、HTML/JavaScript标签（`<script>alert(1)</script>`）、全角空格与零宽字符的组合。其中，零宽字符（U+200B）常被玩家用于创建视觉上相同但实际不同的角色名，绕过重名检测逻辑，属于负面测试需要专项覆盖的安全漏洞类型。

## 实际应用

**背包与仓库系统测试**：在背包容量恰好为满格时（如格子数=50/50），同时发起拾取操作和任务奖励发放，验证系统是否会出现"超量写入"导致实际格子数变为51/50。此类边界并发场景在正面测试中永远不会覆盖，但在正式运营中因玩家数量庞大而极易触发。

**技能连招与取消帧测试**：测试在技能前摇的最后1帧触发背包界面开启，或在治疗技能飞行中途退出游戏再登录，验证技能效果是否仍然正确结算。部分游戏因此类场景未做负面测试，导致玩家可通过特定帧操作无限积累BUFF层数，成为严重数值漏洞。

**支付与货币系统测试**：在商城购买流程中，验证用户钻石余额不足时点击购买按钮的行为；验证在支付成功回调与客户端确认之间强制关闭App后，货物是否仍正确发放且不会重复发放。这两种场景分别对应"不足拒绝"和"幂等性保障"两类负面测试目标。

## 常见误区

**误区一：认为负面测试只需测"输入非法值"**。很多初级测试人员将负面测试等同于在输入框填写错误数据。实际上，操作时序异常（如并发操作、快速连击）、环境异常（断网、低内存）、权限异常（普通账号访问GM指令接口）都是负面测试的重要维度，仅覆盖输入层会遗漏60%以上的真实故障场景。

**误区二：负面测试发现问题后不需要记录预期结果**。正面测试的预期结果是"功能成功执行"，而负面测试的预期结果同样需要精确定义——例如"输入空角色名时，界面应显示'角色名不能为空'的Toast提示，且确认按钮保持不可点击状态"。若仅记录"应该报错"，则无法判断系统弹出崩溃弹窗是否算作测试通过，导致测试结论失效。

**误区三：负面测试与压力测试相同**。负面测试验证的是**单个非法操作的系统响应正确性**，而压力测试关注系统在**高并发合法请求**下的性能表现。两者都可能导致系统崩溃，但触发原因和修复方向完全不同，不可混用测试报告或合并缺陷跟踪。

## 知识关联

**前置知识——新手引导测试**：在新手引导测试阶段，测试人员已熟悉游戏各功能模块的正常交互流程和界面状态机逻辑。负面测试建立在此基础上，将已知的正常状态转换路径反向推导出非法跳转路径。没有新手引导测试建立的功能地图，负面测试用例设计会遗漏大量状态机节点。

**后续概念——冒烟测试**：完成负面测试后，团队进入冒烟测试阶段。冒烟测试（Smoke Testing）的通过标准之一是"核心流程在正常输入下不崩溃"，这隐含要求已知的高危负面场景均已修复。因此负面测试产生的P0/P1级Bug清单，直接构成冒烟测试的入场检查项——若存在未修复的崩溃类负面Bug，冒烟测试应立即中止并退回版本。两者的执行顺序体现了从"系统健壮性底线验证"到"核心功能可用性验证"的测试层次递进。
