# 图片本地化

## 概述

图片本地化（Image Localization）是指对游戏中包含嵌入文字的图片、UI纹理、品牌Logo以及图标进行多语言替换，并验证替换后的视觉效果符合目标语言市场标准的测试流程。与音频本地化不同，图片本地化处理的是静态或动态的视觉资源，这些资源无法通过简单的字符串替换完成翻译——文字直接"烙印"在像素层面，必须重新制作或替换整张图片资源。

该测试类型在2000年代初主机游戏向欧洲多语言市场扩张时开始系统化。日本游戏发行商如Square Enix和Capcom在将游戏本地化为德语、法语、西班牙语（EFIGS组合，即英语+法语+意大利语+德语+西班牙语五语）时，发现大量UI纹理中包含硬编码的英文文字，由此推动了图片本地化测试规范的形成。

图片本地化测试的核心价值在于防止"视觉错语"——即界面上文字已翻译，但图片内嵌文字仍为原始语言，导致目标市场玩家看到语言混乱的画面。以一款RPG游戏为例，物品描述框翻译成了中文，但物品图标上的"POISON"标签仍保持英文，这种不一致会直接影响玩家体验和产品评审评分。

## 核心原理

### 资源类型识别与分类

图片本地化测试首先需要识别游戏中所有含文字的图片资源，通常分为四类：
- **硬编码文字纹理**：文字直接烘焙进PNG/DDS/TGA等纹理文件，如技能图标上的冷却时间数字"CD:30s"；
- **动态合成图片**：运行时将文字渲染叠加在底图上，此类若字体资源已本地化，图片本身无需替换；
- **品牌与Logo素材**：游戏片头的发行商Logo、加载界面的"PRESS START"提示图等；
- **场景内嵌文字**：游戏世界中的路牌、书本内页、广告牌等3D场景贴图中的文字。

测试人员需借助资源管理工具（如Unity的Asset Bundle Viewer或虚幻引擎的Content Browser）系统遍历所有图片资源，建立含文字图片清单（Image Localization Asset List），标注每张图片的语言版本对应关系。

### 替换验证的三项标准

图片本地化验证不仅确认文字已替换，还需验证以下三项标准：

**1. 字符完整性（Character Integrity）**：验证替换图片中的目标语言文字无截断、无乱码。例如，德语单词"Vergiftungsresistenz"（毒性抵抗）的长度远超英文"Poison Resist"，若图片画布尺寸未扩展，文字会被裁剪。测试时应使用像素尺量工具确认文字边界与图片边框至少保留4像素安全边距。

**2. 文字方向适配（Text Direction Compliance）**：针对阿拉伯语（从右至左书写，RTL）和希伯来语版本，图片内的文字排列方向必须完全镜像。仅替换文字内容而未调整排列方向是阿拉伯语本地化中最常见的图片错误。

**3. 文化符号合规性（Cultural Symbol Compliance）**：某些图片虽不含文字，但包含在特定地区受限的符号，如德国版本须将卍字纹章替换为中性符号，中国版本须将骷髅图标替换为骨骼或其他形象。这类替换同属图片本地化测试范畴。

### 分辨率与色彩一致性检查

替换后的本地化图片必须与原版图片保持相同的分辨率规格和色彩深度。例如，若原始英文UI纹理为512×256像素、RGBA 32位格式，替换后的日文版本必须保持相同规格，否则游戏引擎在加载时会产生缩放失真或透明通道丢失问题。测试工程师通常使用自动化脚本批量比对资源元数据：

```
检查公式：原始图片尺寸 == 本地化图片尺寸 AND 原始色彩深度 == 本地化色彩深度
不符合条件的资源标记为 [IMG_SIZE_MISMATCH] 或 [IMG_COLOR_MISMATCH]
```

## 实际应用

**案例一：《赛博朋克2077》多语言UI纹理**
该游戏包含大量世界观广告牌纹理（约占全部图片资源的18%），本地化至简体中文时需将广告牌上的英文/日文文字替换为中文。测试团队需在游戏实机中逐一触发显示这些广告牌的场景，确认替换纹理在不同光照条件（HDR、夜间霓虹灯）下的文字可读性。

**案例二：手机卡牌游戏卡面本地化**
以《炉石传说》为代表的卡牌游戏，每张卡的技能描述文字印制在卡面纹理上。当新增俄语版本时，俄文单词普遍比英文长15%–30%，卡面设计师必须重新排版，测试人员需验证每张卡（《炉石传说》标准模式约有1600张卡）的文字均未超出卡面边框，且字号不低于可读阈值（通常为游戏内显示时8pt等效大小）。

**案例三：游戏内教学截图**
部分游戏的新手引导使用预渲染截图（非实时UI）作为教学图片，这些截图直接截取自英文版界面，在本地化版本中必须重新截取目标语言界面生成新的引导图片，并进行替换验证。

## 常见误区

**误区一："文字图层分离就不需要测图片本地化"**
部分开发者认为，只要UI系统采用了文字与底图分离的架构（文字通过TTF字体实时渲染），图片本地化测试就可以跳过。这是错误的——即便主UI如此设计，游戏中仍大概率存在预渲染的过场截图、手绘风格的教程卡通图、以及引擎外部制作的营销材料被嵌入游戏包体中，这些资源全部需要单独的图片本地化验证。

**误区二："图片替换只要视觉上差不多就行"**
测试新手常凭肉眼确认文字"看起来"已替换即通过，忽略了压缩格式差异导致的质量损失。例如，原始纹理为无损PNG，本地化版本误用有损JPEG压缩（压缩率80%），文字边缘会出现JPEG块状伪影，在游戏高清模式下清晰可见。正确做法是验证文件格式（扩展名）以及引擎导入设置中的压缩选项完全匹配原始资源配置。

**误区三："Logo不属于图片本地化范围"**
Logo和品牌徽标通常被测试人员视为"不可动"资源而跳过检查。但在中国大陆版本中，国家新闻出版总署要求游戏不得展示境外评级机构（如ESRB）的Logo图片；部分中东版本要求替换含有特定宗教形象的Logo素材。跳过Logo类图片的本地化验证是导致产品在特定市场无法上架的常见原因之一。

## 知识关联

图片本地化与**音频本地化**共同构成游戏非文本资源本地化的两大支柱。音频本地化处理的是WAV/OGG等音频文件的替换验证，而图片本地化处理的是PNG/DDS/TGA等图形文件；两者的测试流程设计相似（资源清单→替换验证→质量检查），但缺陷类型完全不同——音频关注时长对齐和口型同步，图片关注像素尺寸和文字渲染质量。

完成图片本地化测试的基础上，团队可以进入**伪本地化（Pseudo-localization）**阶段。伪本地化会将所有图片中的文字替换为刻意拉长的乱码字符串（如将"OK"替换为"[ÒŔŔòôÒ_OK]"），以此一次性暴露所有图片资源中的文字截断风险，而无需等待真实翻译文件就绪。图片本地化测试建立的"含文字图片资产清单"可直接复用为伪本地化的输入文件列表，显著提升后续测试的覆盖效率。
