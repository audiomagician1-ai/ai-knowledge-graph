# 基准测试设计

## 概述

基准测试设计（Benchmark Design）是指在游戏性能测试中，创建可重复执行的标准化场景，并通过自动化手段采集帧时间、GPU 占用率、内存分配等关键指标的系统性方法。它与一般性压力测试的本质区别在于"可复现性"——同一基准场景在不同时间、不同硬件配置上运行，产生的环境条件必须严格一致，使得结果具备可比较性。

基准测试的概念最早在游戏行业大规模应用是在 3DMark 系列工具（1998年发布首版）普及之后。开发商借鉴其思路，开始在内部构建项目专属的基准序列（Benchmark Sequence），将特定的游戏内场景录制为固定的相机路径或输入回放，确保每次测试触发完全相同的渲染调用和逻辑运算。这种做法使得提交新代码前后的性能对比误差从随机测试的 ±15% 压缩到 ±2% 以内。

基准测试设计的价值在于它将"感觉卡顿"这一主观描述转化为可追踪的数字曲线。当一次提交导致基准场景的 P99 帧时间（第99百分位帧耗时）从 16.7ms 上升到 22ms 时，CI/CD 管线可以自动标记该提交为性能回归，无需人工介入评估。

## 核心原理

### 场景选取：最坏情况与代表情况并存

基准场景的选取遵循"覆盖峰值与均值"原则，通常需要至少两类场景：**压力场景**（Stress Scene）和**代表场景**（Representative Scene）。压力场景选取渲染负载最高的地图区域，例如粒子特效密集的 Boss 战房间或同屏角色数量最多的城镇广场；代表场景则对应玩家 75% 时间所在的普通战斗区域。只选取压力场景会导致测试过于保守，只选取代表场景则会遗漏尖刺型卡顿。两类场景的比例通常建议为 1:2 到 1:3。

### 相机路径录制与输入回放

可复现的基准测试必须固定"刺激源"。做法分为两种：

- **相机路径回放（Camera Path Replay）**：使用引擎内置的 Cinematic Tool 或 Sequencer 录制固定飞行路径，每帧的视锥体（Frustum）完全一致，从而保证每次触发的 Draw Call 集合相同。Unreal Engine 的 `stat unit` 结合 `-benchmark` 命令行参数可直接启用此模式。
- **输入序列回放（Input Replay）**：录制玩家操作序列（按键、鼠标移动），适用于需要模拟物理交互或 AI 行为的场景。但因 AI 随机种子需固定（例如通过 `FMath::RandInit(42)` 强制设定），实现复杂度更高。

相机路径回放的实现成本低，复现误差更小，是大多数游戏项目首选方式。

### 自动化采集管线

基准测试设计不只是场景设计，还包括数据采集的自动化链路。完整管线包含以下四个环节：

1. **触发**：通过 CI 脚本（如 Jenkins Pipeline 或 GitHub Actions）在每次代码合并后自动启动基准进程。
2. **采集**：使用引擎内置性能计数器（Unreal 的 `FPlatformTime::Cycles64()`，Unity 的 `Profiler.BeginSample()`）或平台层工具（PC 端 ETW、主机端 PIX）以每帧粒度记录数据，输出为 CSV 或 JSON 格式。
3. **聚合**：计算每次运行的 P50（中位帧时间）、P95 和 P99 帧时间，而非仅使用平均帧率（平均帧率会掩盖偶发的帧耗时尖刺）。
4. **比对**：将结果与基准基线（Baseline）对比，超出阈值（例如 P99 上升超过 10%）则触发报警。

### 硬件矩阵与环境隔离

基准场景本身设计再严格，若测试机器运行着后台进程或散热状态不同，结果仍会漂移。实践中需要：将测试机器设置为"锁频模式"（CPU/GPU 频率固定，禁用 Turbo Boost 和动态调频）；使用独立的专用裸机而非共享虚拟机；每次测试前执行"预热跑"（Warm-up Pass，通常 30 秒），让 GPU 和 CPU 进入热稳态后再开始采集。锁频操作在 Windows 上可通过电源计划设置 + `powercfg /setacvalueindex` 命令实现。

## 实际应用

**开放世界游戏的流式加载基准**：以某款 RPG 为例，设计一条穿越四个地形区块边界的固定飞行路径，全程 90 秒。每次经过区块边界时 CPU 流式解压线程会出现负载尖刺，基准测试专门监控这段时间的 `AsyncTask` 线程占用率和主线程 Stall 时长，用于追踪地形加载优化的效果。

**多人游戏帧同步基准**：对于帧同步（Lockstep）对战游戏，基准场景录制为一段包含 16 名玩家同时施放技能的回放文件，固定帧号范围为第 1800 帧至第 2400 帧，重点采集每帧的逻辑计算时间（Logic Frame Time）而非渲染帧时间。这类基准可在无 GPU 的服务器上运行，降低硬件成本。

**移动端热耗散基准**：针对 iOS 设备，基准测试持续 20 分钟以触发芯片的温控降频（Thermal Throttling）。采集目标不是稳定态帧率，而是芯片进入降频状态所需的时间以及降频后的稳定帧率，以此评估不同渲染质量配置的热功耗差异。

## 常见误区

**误区一：用平均帧率代替百分位帧时间**  
平均帧率（如"平均 60fps"）会将偶发的 100ms 长帧均摊掉，玩家感受到的卡顿完全不可见。基准测试应以 P99 帧时间为主要指标，因为人眼对单次超过 50ms 的帧间隔（约为 20fps 对应的帧时间）极为敏感，而这类事件在平均帧率中仅贡献约 0.1% 的权重。

**误区二：基准场景越复杂越好**  
初学者倾向于设计覆盖所有系统的"终极基准场景"，结果是当测试报告显示性能回归时，无法判断是渲染、物理还是 AI 引起的。正确做法是将基准场景分层：一个全系统综合场景 + 若干单系统隔离场景（纯 GPU 渲染场景、纯物理模拟场景等），定位问题时先查综合场景发现异常，再用隔离场景确认根因。

**误区三：基准基线永不更新**  
若基准基线长期不更新，随着版本迭代导致场景内容变化（新增建筑、调整光照），基线会与当前场景脱节，比对结果失去意义。应建立"基线版本号"机制：每次场景内容发生结构性变更时，由性能负责人审批后更新基线，并在版本历史中标注变更原因，确保任意两个版本的对比有明确的参照系。

## 知识关联

**前序概念——可扩展性测试**：可扩展性测试确定了游戏在不同硬件档位（低/中/高）下的配置矩阵，基准测试设计则以这套配置矩阵为输入，为每个硬件档位制定独立的基准场景及合格阈值。若可扩展性测试尚未完成，基准测试就缺乏"测试目标"，无法确定 P99 帧时间的合格上限应设为 33ms（30fps 目标）还是 16.7ms（60fps 目标）。

**后续概念——性能预算**：基准测试长期运行后积累的历史数据（各子系统的典型耗时分布）是制定性能预算的实证依据。性能预算将整体帧时间（如 16.7ms）分配给渲染（8ms）、逻辑（3ms）、物理（2ms）等模块，而这些分配比例的合理性正是通过基准测试中各模块的实测耗时来校验的。两者形成"测量→预算→测量验证"的闭环。
