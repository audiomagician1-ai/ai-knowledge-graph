# 影响分析

## 概述

影响分析（Impact Analysis）是回归测试中确定代码变更所波及范围的系统性方法，通过追踪源代码修改与程序功能、模块之间的依赖关系，精确划定哪些测试用例必须重新执行。其核心目标是在保证质量的前提下，避免对所有历史测试用例进行全量重跑——在大型游戏项目中，全量回归往往需要数十小时，而经过影响分析后的定向回归可将执行时间压缩至原来的20%~40%。

影响分析的概念最早在1980年代软件工程文献中被正式提出。Leung与White于1990年在IEEE Transactions on Software Engineering上发表的论文将其定义为"给定一组变更，识别程序中受影响实体的集合"，奠定了该领域的理论基础。在游戏QA领域，由于游戏系统间高度耦合（例如战斗系统的伤害公式可能同时影响成就系统、排行榜系统和新手引导逻辑），影响分析显得尤为重要。

影响分析的价值在于它将"哪些东西可能坏了"这一问题转化为可操作的检查清单。一次看似微小的改动——比如将某个角色的基础攻击力从100调整为105——可能触发连锁反应，波及伤害计算、战斗日志、成就解锁条件、平衡性数值等多个子系统。只有通过系统化的影响分析，测试人员才能将这些隐性关联显现出来。

## 核心原理

### 依赖关系图（Dependency Graph）构建

影响分析的基础是建立变更代码与程序其他部分之间的依赖关系图。依赖类型主要分为三类：

- **数据依赖**：模块B读取模块A写入的变量，如`PlayerStats.hp`被战斗模块、UI模块和存档模块共同引用；
- **控制依赖**：模块A的条件分支决定模块B是否被执行，如技能冷却时间检查决定技能释放逻辑是否运行；
- **接口依赖**：模块B调用模块A暴露的函数接口，函数签名或返回值变化将直接破坏调用方。

实际操作中，可借助静态代码分析工具（如游戏项目常用的Visual Studio的"查找所有引用"或专用的依赖分析插件）自动生成依赖关系图。

### 变更集合（Change Set）的精确界定

影响分析必须从一个明确的**变更集合**出发，即本次迭代中被修改的所有文件、函数或配置项的完整列表。在实践中通过Git Diff命令获取两个提交之间的差异：

```
git diff <base_commit> <head_commit> --name-only
```

变更集合不仅包括直接修改的代码文件，还应包括被修改代码所引用的配置文件（如Lua脚本中的技能数值表）以及自动生成代码（如Protobuf生成的网络协议类）。遗漏配置文件是游戏QA中影响分析失效的最常见原因之一。

### 影响传播的层次模型

影响从变更点向外扩散时呈现层次化特征，通常用**传播深度（Propagation Depth）**来度量：

| 传播层次 | 含义 | 游戏示例 |
|---|---|---|
| 第0层 | 变更点本身 | 修改伤害计算函数`CalcDamage()` |
| 第1层 | 直接调用或引用变更点的模块 | 普攻模块、技能模块 |
| 第2层 | 依赖第1层模块的模块 | 战斗日志模块、击杀判定模块 |
| 第3层+ | 更深层间接依赖 | 成就系统、排行榜数据上报 |

研究表明，超过80%的变更影响范围集中在第0层至第2层，因此在资源有限时优先覆盖前三层可以获得最佳的风险/收益比。

## 实际应用

### 游戏战斗系统版本迭代中的影响分析示例

假设某手机RPG游戏本次版本修改了`SkillManager.cs`中的技能命中率计算逻辑（将命中公式由 `HitRate = BaseHit / (BaseHit + EnemyDodge)` 改为引入等级差修正系数）。影响分析流程如下：

1. **确定变更集合**：`SkillManager.cs`中的`CalculateHitRate()`函数；
2. **追踪第1层依赖**：通过IDE搜索发现`BattleController.cs`、`AutoBattleAI.cs`、`BattleReplayRecorder.cs`三个文件直接调用该函数；
3. **追踪第2层依赖**：`BattleController.cs`的战斗结果被`QuestProgressTracker.cs`（任务进度）和`AchievementChecker.cs`（成就系统）监听；
4. **生成受影响测试用例列表**：命中率相关的27个TC、自动战斗AI行为的15个TC、战斗录像回放的8个TC、任务完成判定的12个TC，合计62个TC，而全量TC数量为480个。

经过影响分析，测试范围从480个缩减至62个，节省了约87%的执行工作量。

### 游戏配置变更中的影响分析

当策划修改Excel数值表（如武器基础伤害表）而不涉及代码变更时，影响分析同样适用。此时需要追踪哪些代码模块在运行时加载并解析该配置文件，以及哪些游戏系统的平衡性依赖于这些数值。

## 常见误区

### 误区一：影响分析等同于只测试被修改的文件

许多初级测试人员认为"只改了A文件，所以只测A的功能"。这种思路忽略了**间接依赖**的存在。`SkillManager.cs`的变更可能根本不影响`SkillManager.cs`自身对应的技能释放流程，但会破坏依赖其返回值的成就解锁逻辑。影响分析的本质是追踪**数据流和控制流的传播路径**，而不是停留在文件层面的字面匹配。

### 误区二：影响分析可以完全替代全量回归

影响分析存在固有的**遗漏风险（Escape Risk）**，原因在于静态分析工具无法识别某些运行时动态依赖（如通过字符串反射调用的函数、事件总线上的订阅关系）。因此在重大版本发布或底层框架改动时，仍需安排全量回归作为最终兜底手段。影响分析的适用场景是迭代频繁的日常版本，而非一年一度的大版本上线。

### 误区三：影响分析只需要测试人员单独完成

影响分析的质量高度依赖开发人员提供准确的变更信息。开发人员有时会提交未在变更日志中说明的"顺手修改"，导致测试人员的影响分析基于不完整的变更集合。规范的影响分析流程要求开发人员在提交代码时填写结构化的变更描述（包括修改原因、影响模块的自评估），测试人员在此基础上进行独立验证和补充。

## 知识关联

影响分析以**回归测试概述**中建立的基本概念为前提，尤其是对"回归缺陷（Regression Bug）"的定义——即变更引入的、使原本通过的测试失败的缺陷。理解回归缺陷的成因（代码耦合、副作用、状态共享）是正确执行影响分析的思维基础。

完成影响分析后，下一步是**测试选择策略**，即在已知影响范围的前提下，决定从影响集合中优先选取哪些测试用例执行。测试选择策略需要影响分析的输出作为输入——没有边界清晰的影响范围，选择策略就无从谈起。两者共同构成了"做哪些测试"这一问题的完整回答：影响分析负责划定候选集合，测试选择策略负责在候选集合内进行优先级排序。
