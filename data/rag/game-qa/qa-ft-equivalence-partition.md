# 等价类划分

## 概述

等价类划分（Equivalence Class Partitioning）是一种黑盒测试设计技术，将程序的输入域划分为若干子集，同一子集内的任意数据被认为具有等效的测试能力——即如果某个值能发现缺陷，该子集中的其他值理论上也能发现同类缺陷。这一假设使测试人员可以从每个子集中仅选取一个代表值，而非穷举所有可能的输入。

该方法由迈尔斯（Glenford J. Myers）在1979年出版的《软件测试技术》（*The Art of Software Testing*）中系统化阐述，成为功能测试领域最早被标准化的用例设计方法之一。其核心价值在于用最少的测试用例覆盖最大的输入空间——对于一个接受1到999整数的输入字段，穷举需要999条用例，而等价类划分只需3条左右即可覆盖主要场景。

在游戏QA场景中，等价类划分尤其适用于角色属性校验（如等级范围1-100）、道具数量输入（如购买1-99件）、昵称长度限制（如2-16个字符）等功能点。通过划分等价类，测试人员可以在版本迭代快、功能点密集的游戏项目中，以结构化方式高效覆盖输入验证逻辑。

## 核心原理

### 有效等价类与无效等价类的定义

等价类分为两大类型：**有效等价类**（Valid Equivalence Class）指满足程序规格说明的合法输入集合；**无效等价类**（Invalid Equivalence Class）指不满足规格说明的非法输入集合。以游戏角色命名系统为例，若规则要求昵称长度为2至16个中文或英文字符，则有效等价类为"长度在2到16字符之间的昵称"，无效等价类包括"长度为0或1的过短昵称"、"长度超过16的过长昵称"、"含特殊符号的昵称"等多个子集。

无效等价类并非只有一个——每种不同类型的违规输入应独立划分，因为它们触发的错误处理逻辑可能完全不同。长度超限的提示文案与含非法字符的提示文案通常来自不同的校验分支，必须分别用例覆盖。

### 划分规则与等价类识别方法

等价类的划分遵循以下具体规则：

- **区间型输入**：若规格说明规定输入在区间 [a, b] 内，则产生1个有效等价类（a ≤ x ≤ b）和2个无效等价类（x < a 以及 x > b）。
- **枚举型输入**：若输入为有限集合中的某个值（如职业选择：战士/法师/弓手），则每个合法值构成一个有效等价类，所有其他值合并为一个无效等价类。
- **布尔型输入**：产生1个有效等价类和1个无效等价类。
- **字符串型输入**：通常需同时考虑长度、字符类型两个维度，各自独立划分等价类。

以游戏充值功能为例，充值金额（单位：元）若规定为6、30、68、128、328、648六档固定值，则这六个值各构成独立的有效等价类，非以上金额的任意数字构成无效等价类。

### 测试用例生成规则

确定等价类后，按照以下规则生成测试用例：

1. 为每个**有效等价类**设计至少1条测试用例，且允许单条用例同时覆盖多个有效等价类（以减少用例总量）。
2. 为每个**无效等价类**设计**独立的**测试用例，每条用例只覆盖一个无效等价类，防止一个错误掩盖另一个错误的检测（即"错误屏蔽"效应）。

以游戏背包物品堆叠上限（1-99件）为例，完整的等价类划分结果为：

| 等价类编号 | 类型 | 取值范围 | 代表值 |
|---|---|---|---|
| EC1 | 有效 | 1 ≤ 数量 ≤ 99 | 50 |
| EC2 | 无效 | 数量 < 1 | 0 |
| EC3 | 无效 | 数量 > 99 | 100 |
| EC4 | 无效 | 非整数 | 1.5 |

## 实际应用

**游戏登录密码校验**：某手游要求密码为8到20位，且必须包含字母和数字。等价类划分产生1个有效类（8-20位含字母和数字），以及"少于8位"、"超过20位"、"纯数字"、"纯字母"、"含特殊字符（视规则而定）"等多个无效类。测试人员从每类中各取一个代表值，共约6条用例即可覆盖主要输入验证逻辑。

**游戏段位积分系统**：若某段位需要积分范围在0到2000分之间才能参与匹配，有效类为0≤积分≤2000，无效类为积分<0（负分异常数据）和积分>2000（超段位上限数据）。实际测试时，代表值可取1000、-1、2001三个值，分别对应三条测试用例。

**游戏地图坐标输入**：地图编辑器若允许X坐标范围为-512到512的整数，则需覆盖有效整数（如0）、低于下限的值（如-513）、高于上限的值（如513）、非整数（如3.14）、以及非数字字符（如"abc"）共五个等价类。

## 常见误区

**误区一：将有效等价类内的所有值视为完全等价**
等价类划分假设同类内的值等价，但这一假设在边界附近会失效。例如，在"1到99"的有效等价类中，取50作为代表值是合理的，但值1和99处于边界，其行为可能与50完全不同。等价类划分本身不检测边界缺陷，这正是需要配合边界值分析使用的原因。

**误区二：无效等价类只有"比最小值小"和"比最大值大"两个**
测试人员常只考虑数值越界，忽略数据类型越界（如输入小数、字母、空字符串、空格）。以游戏角色等级输入框为例，用户可能输入"abc"、"１"（全角数字）、"-1"、"0"、"101"等多种非法值，每种类型应归入不同的无效等价类，对应不同的错误处理分支测试。

**误区三：用单条测试用例同时覆盖多个无效等价类**
这是违反等价类划分规则的典型错误。若某用例输入一个"超长度且含特殊字符"的昵称，当程序提示错误时，测试人员无法判断是哪条校验规则生效，也无法排除其中一条规则实际上未被正确实现。每个无效等价类必须用独立用例覆盖，这是等价类划分方法的强制要求，不是可选建议。

## 知识关联

等价类划分的前置知识是**测试用例设计**的基本框架——测试人员需要先理解测试用例由输入、执行步骤、预期结果三部分构成，才能在等价类划分完成后正确填充用例模板中的代表值和对应预期输出。

等价类划分与**边界值分析**构成互补关系：等价类划分解决"从哪些区域取值"的问题，边界值分析解决"在等价类边界处取哪些精确值"的问题。标准做法是先用等价类划分确定输入区间，再用边界值分析在每个区间的上下边界点（如对于区间[1,99]，取值1、2、98、99以及越界值0和100）补充测试用例。两者结合使用，才能同时覆盖等价类内部逻辑和边界处的off-by-one类型缺陷——而这类边界错误在游戏属性计算和范围判断代码中极为常见。
