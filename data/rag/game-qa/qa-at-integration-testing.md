# 集成测试

## 概述

集成测试（Integration Testing）是在单元测试完成之后，将多个已通过单元测试的模块组合在一起，专门验证这些模块**之间的接口、数据传递和调用协议**是否正确的测试阶段。它既不关心单个模块的内部逻辑（那是单元测试的职责），也不模拟完整的用户操作流程（那是端到端测试的职责），而是聚焦于模块边界处发生的一切。

集成测试的概念最早可追溯到1970年代的结构化软件工程实践，V模型（V-Model）将其正式定位为单元测试与系统测试之间的独立阶段。在游戏QA领域，集成测试解决的典型问题是：角色系统（CharacterSystem）在调用背包系统（InventorySystem）的 `AddItem(itemId, count)` 接口时，传递的数据格式是否符合约定，背包容量溢出后的返回码是否被正确处理。

在游戏开发中，集成测试的重要性体现在：游戏模块之间的耦合往往比企业软件更紧密，战斗系统要同时调用伤害计算模块、状态效果模块和网络同步模块。任何一处接口契约被违反，都会产生"模块本身测试通过，但组合起来行为异常"的典型集成缺陷。

## 核心原理

### 接口契约验证

集成测试的第一要务是验证两个模块之间的**接口契约**（Interface Contract）。契约包含三个要素：输入参数的类型与范围、返回值的结构与含义、以及异常/错误码的处理约定。例如，游戏中的技能系统调用音效系统时，契约规定 `PlaySFX(clipId: int, volume: float)` 中 `volume` 范围为 `[0.0, 1.0]`；集成测试需要覆盖传入 `1.5` 时音效系统能否安全降级，而不是直接崩溃。

测试用例设计通常采用**边界值分析**结合**等价类划分**：针对接口参数设计正常值、边界值（如 `0`、`MAX_INT`）和越界值三类输入，断言被调用模块的响应符合契约描述。

### 集成策略：自顶向下与自底向上

集成测试有两种主流策略，在游戏QA中各有适用场景：

- **自顶向下（Top-Down）**：从最高层模块（如 `GameManager`）开始，用**桩（Stub）**替代尚未集成的子模块。这种方式适合游戏主流程验证较早，但需要为每个被替换的子模块编写 Stub，工作量较大。
- **自底向上（Bottom-Up）**：先集成底层基础模块（如数学库、资源加载器），用**驱动程序（Driver）**调用它们，逐层向上组合。游戏的物理引擎、渲染模块等底层系统通常适合这种策略。

另有**大爆炸集成（Big-Bang）**策略——将所有模块一次性集成后测试，但由于缺陷定位困难，游戏项目中几乎不使用。

### 数据流追踪

游戏的集成测试必须追踪**跨模块的数据流**。以一个典型例子说明：玩家触发击杀敌人事件时，数据需依次流经 `CombatSystem → QuestSystem → AchievementSystem → UISystem`。集成测试要验证每一个传递节点：`CombatSystem` 输出的击杀事件对象（包含 `enemyType`、`damageDealt`、`timestamp` 等字段）是否被 `QuestSystem` 正确解析，以及任务进度更新后触发的回调是否能被成就系统正确接收。

测试断言的形式通常为：给定输入事件 → 执行链路 → 断言链路末端（UI层）的显示数值与预期一致，误差容忍度为 `0`（游戏数值逻辑不允许浮点累积误差传播到最终呈现层）。

## 实际应用

**场景一：战斗伤害链集成测试**

游戏QA团队针对 `AttackModule` + `DefenseModule` + `HealthModule` 三个模块编写集成测试。测试用例：攻击力为 `150`，防御力为 `50`，伤害减免公式为 `实际伤害 = 攻击力 × (1 - 防御力/300)`，预期实际伤害 = `125`。测试断言 `HealthModule.currentHP` 在调用链执行后等于 `初始HP - 125`。这个测试只验证三个模块组合后数值是否正确流转，不关心各模块内部如何实现减免逻辑。

**场景二：存档系统集成测试**

`SaveSystem` 调用 `SerializationModule` 将 `PlayerData` 对象序列化后写入磁盘，再调用 `FileIOModule` 完成持久化。集成测试验证：序列化后的字节流被 `FileIOModule` 完整写入，再经反序列化后 `PlayerData` 的每个字段（包括背包列表 `List<Item>`）与原始对象完全一致。这类测试在存档格式版本升级时（如从 `v2` 升级到 `v3`）尤其关键。

**场景三：网络同步模块集成测试**

多人游戏中，`ClientGameState` 模块与 `NetworkSyncModule` 集成测试需验证：客户端发送位置更新数据包（包含 `x, y, z` 坐标和 `sequenceId`）后，`NetworkSyncModule` 正确处理乱序包（`sequenceId` 不连续）时，`ClientGameState` 接收到的坐标仍为最新帧的值。

## 常见误区

**误区一：用集成测试替代单元测试**

部分游戏QA工程师因为集成测试能"发现更多问题"，倾向于跳过单元测试直接写集成测试。这会导致严重后果：当集成测试失败时，无法快速判断是哪个模块的内部逻辑有误还是接口调用有误，排查时间显著增加。正确做法是保持测试金字塔结构——单元测试数量 > 集成测试数量，集成测试专注于**接口和数据流**，而非重复验证模块内部逻辑。

**误区二：将"调用多个函数"等同于"集成测试"**

在同一个类或同一个模块内部调用多个方法，即使测试了多个函数的组合行为，仍然属于**单元测试**的范畴，而非集成测试。集成测试的定义特征是存在**跨越模块边界**的交互——通常意味着跨越不同的程序集（Assembly）、服务进程，或至少是经过明确接口隔离的不同子系统。

**误区三：忽略异步与事件驱动接口的测试**

游戏中大量接口是异步的（如资源加载完成回调、网络响应回调）。集成测试如果只测试同步调用路径，会遗漏"模块A触发事件后，模块B的回调未被注册"这类缺陷。针对异步接口，测试框架需要使用等待机制（如 Unity Test Framework 的 `[UnityTest]` 配合 `yield return` ）来正确等待异步操作完成后再执行断言。

## 知识关联

**与单元测试的衔接**：单元测试使用 Mock 对象隔离外部依赖，使每个模块可独立验证。集成测试则**移除这些 Mock**，替换为真实模块，此时 Mock 阶段掩盖的接口不兼容问题才会暴露出来。完成单元测试后，集成测试的第一步往往是检查各模块的接口定义（如 C# 的 `interface` 声明或 Lua 的模块导出表）是否与实际实现一致。

**通往端到端测试**：端到端测试（E2E Test）模拟真实玩家从启动游戏到完成某一游戏流程的完整操作路径。集成测试为此奠定基础：只有各子系统的接口集成已通过验证，端到端测试的失败才能被快速归因为"用户流程逻辑问题"而非"底层接口数据错误"。在游戏QA流水线中，通常的顺序是：单元测试 → 集成测试 → 端到端测试，三层测试共同覆盖从局部到整体的质量保障。
