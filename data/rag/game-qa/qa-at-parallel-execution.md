# 并行执行

## 概述

并行执行（Parallel Execution）是自动化测试中将多个测试用例在同一时间段内同时运行的技术策略，与传统的串行执行相比，并行执行通过多线程、多进程或多机器分布式部署，将原本需要顺序完成的测试任务切分为可同时进行的独立单元。在游戏QA领域，一套完整的回归测试套件往往包含数百乃至数千个测试用例，串行执行耗时可能长达数小时，而并行执行可将总耗时压缩至原来的 1/N（N为并行度），显著缩短版本迭代的反馈周期。

并行执行的概念最早随持续集成（CI）工具的普及而兴起。2011年前后，Jenkins 引入了并行构建插件，随后 Selenium Grid 在2.0版本正式支持多节点并发运行浏览器测试，这一架构随即被移植到游戏自动化测试框架中。在游戏QA实践中，并行执行不仅用于功能测试，也广泛应用于性能压测场景，例如同时模拟数千个虚拟玩家并发登录以检测服务器承载极限。

在游戏版本发布节奏加快的背景下（主流手游往往每两周发布一个版本），测试窗口通常被压缩至4到8小时以内，并行执行成为在有限时间内完成全量回归的必要手段，而非可选优化。

## 核心原理

### 测试用例的独立性要求

并行执行的基本前提是每个测试用例必须具备**状态隔离性**：用例A的执行结果不能影响用例B的初始状态。游戏测试中常见的隔离手段包括：为每个并行工作线程分配独立的账号（例如测试账号池中预备 50 个独立账号供 50 个并行线程各自占用），在每个用例开始前通过 API 重置角色背包、货币和进度数据，以及使用沙箱服务器或 Docker 容器隔离游戏服务端状态。若测试用例之间存在隐式依赖（如共享同一个公会或同一个副本实例），并行执行极易触发竞态条件（Race Condition），导致测试结果不稳定。

### 并行架构模型：主从模式与对等模式

游戏自动化测试常用两种并行架构：

**主从模式（Master-Worker）**：由一个调度节点（Master）持有全部测试用例队列，动态分配任务给若干执行节点（Worker）。Worker 完成用例后主动向 Master 请求下一个任务，实现负载均衡。Selenium Grid 和 Appium Grid 均采用此模式；Master 节点同时负责收集各 Worker 的执行结果并汇总至测试报告。

**分片模式（Sharding）**：将测试用例集合按数量平均切分为 K 份，每台机器负责第 i 份（i = 0, 1, …, K-1）。分片公式为：`用例分配到第 i 片 = 用例索引 % K == i`。此模式部署简单，适合 CI 流水线中机器数量固定的场景，GitHub Actions 的 `matrix` 策略即使用此思路。但若各分片内用例执行时长差异较大，会出现木桶效应，总耗时取决于最慢的那一片。

### 并行度的计算与资源约束

理论加速比受 Amdahl 定律约束：`S(N) = 1 / (F + (1-F)/N)`，其中 F 为不可并行化的串行比例，N 为并行线程数。若一套游戏测试套件中有 10% 的用例因依赖特定全局资源（如唯一的充值沙箱账号）必须串行执行，则即便 N 趋于无穷大，理论最大加速比也不超过 10 倍。实践中，游戏测试的并行度受限于以下因素：测试设备或模拟器数量、游戏服务器并发连接数限制、以及 CI 机器的内存容量（每个 Android 模拟器实例通常需要 2\~4 GB 内存）。

### 分布式执行的协调机制

在多台物理机或云实例上运行分布式测试时，需要一个中央协调层来处理用例分发、心跳检测和故障转移。当某台 Worker 节点在执行过程中崩溃（游戏测试中模拟器崩溃较为常见），Master 必须能在超时后（通常设置为 120\~300 秒）将该用例重新分配给其他健康节点，避免整套测试因单点故障卡死。

## 实际应用

**手游回归测试提速**：某款 MMORPG 手游的回归测试套件共有 800 个 UI 自动化用例，在单台机器上串行执行需要约 6 小时。通过部署 10 台云手机（Cloud Phone）并行执行，总耗时降至约 40 分钟（考虑到约 8% 的串行初始化开销），满足了每日两次全量回归的要求。

**战斗系统并行压测**：游戏服务器的战斗逻辑接口可通过并行执行多个无状态 HTTP 请求进行压测，例如使用 Locust 框架同时启动 500 个并发用户模拟技能释放请求，观察服务器在峰值并发下的响应时间分布（P95、P99 延迟指标）。

**多平台同步验证**：同一款游戏在 Android、iOS 和 PC 端需同时通过功能验证。利用并行执行，三个平台的测试套件可在同一 CI 流水线中并发触发，总时间等于最慢平台的测试时间，而非三个平台时间之和。

## 常见误区

**误区一：并行度越高越好**。部分团队将并行线程数设置为与用例总数相同，期望"一次全部跑完"。但这忽略了游戏服务器的并发连接上限——若服务器设置了最大 200 个测试账号同时在线的限制，超出限制的连接会被拒绝，导致大量用例因登录失败而报错，误判为功能缺陷。合理的并行度应通过压测游戏服务器的承载能力后确定上限。

**误区二：将不稳定测试（Flaky Test）归咎于并行执行**。在引入并行执行后，如果测试失败率上升，团队往往将原因直接归结为"并行冲突"，而忽略了原本就存在的不稳定测试问题。事实上，并行执行会放大已有的用例状态隔离缺陷：原本在串行模式下偶发的竞态问题，在并行下会以更高频率暴露。正确做法是先完成不稳定测试治理（即在串行环境下将 Flaky Rate 降低至 2% 以下），再引入并行执行。

**误区三：分片模式天然均衡**。简单按用例数量分片并不等于按执行时间均衡。游戏战斗系统的端到端测试用例可能单个耗时 3 分钟，而 UI 点击用例仅需 10 秒，若不按历史耗时数据进行加权分片，分片间的时间差异可能超过 5 倍，导致大部分 Worker 空闲等待最慢的那一片完成。

## 知识关联

并行执行直接依赖**不稳定测试治理**的成果：只有当测试套件的 Flaky Rate 被控制在可接受范围内，并行执行的结果才具有可信度。不稳定用例在并行环境中会以更高频率失败，产生大量误报，污染并行执行的价值。如果跳过不稳定测试治理直接引入并行执行，团队将难以区分"并行冲突导致的失败"与"用例本身不稳定"两类问题。

并行执行完成后，多个 Worker 节点各自产生的测试结果需要被聚合为统一的**测试报告**。这一聚合过程并非简单的数据拼接：需要去重（同一用例被重试多次的情况）、合并执行时间线、以及生成按模块分类的覆盖率统计。并行执行的架构设计（如 Master 节点的结果收集接口格式）直接影响测试报告工具（如 Allure Report、ReportPortal）的集成方式，因此在设计并行方案时需同步考虑报告层的数据结构需求。
