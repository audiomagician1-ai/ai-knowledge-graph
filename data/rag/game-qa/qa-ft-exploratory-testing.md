# 探索性测试

## 概述

探索性测试（Exploratory Testing）是由软件测试先驱 Cem Kaner 于 1984 年首次正式提出的测试方法，其核心定义为：测试人员同时进行测试设计、执行和学习，而非事先编写完整的测试用例文档再逐一执行。在游戏 QA 领域，这意味着测试员像一名真实玩家一样自由操作游戏，凭借对游戏机制的理解和个人经验，主动寻找未被预设脚本覆盖的问题。

探索性测试与传统脚本化测试的根本区别在于：脚本化测试回答"系统是否按设计工作"，而探索性测试回答"系统在非预期操作下会发生什么"。在游戏测试中，玩家的实际行为几乎无法被预测脚本完全覆盖——有玩家会在最终 Boss 战前反复跳跃、倒退走、同时触发两个剧情事件——这些边缘操作恰好是探索性测试最擅长捕捉的场景。

对游戏 QA 团队而言，探索性测试通常在冒烟测试确认基础功能可运行之后展开，因为在游戏频繁崩溃的早期版本中进行自由探索效率极低。一旦冒烟测试通过，测试员即可将自由探索时间集中于核心玩法循环、过场动画触发条件、存档读取等高风险区域。

---

## 核心原理

### 时间盒（Time-Boxing）机制

探索性测试通常以"章节式时间盒"（Session-Based Test Management，SBTM）为组织单位，每个测试会话固定时长为 **60 至 90 分钟**。超过这个时长，测试员的注意力和创造力会明显下降，发现新 Bug 的概率也随之降低。每次会话前，测试员确定一个明确的"章程"（Charter），例如"专注于第三关所有可交互物件的物理碰撞行为"，而非漫无目的地游玩。章程使探索保持聚焦，同时不限制测试员在会话中的自由度。

### 启发式测试思维

有经验的游戏测试员在探索时会运用一套心理模型，称为**测试启发式（Test Heuristics）**。常用的启发式框架之一是 James Bach 提出的 **SFDIPOT**（Structure, Function, Data, Integration, Platform, Operations, Time），帮助测试员系统性地切换测试视角。例如针对游戏背包系统：Structure（背包 UI 布局）、Function（拖拽、丢弃、叠加道具）、Data（最大堆叠数量、0 个道具、负数道具）、Integration（存档时背包状态是否正确写入磁盘）。这一框架防止测试员只盯着"常规玩法路径"而遗漏数据边界问题。

### 同步记录与实时笔记

探索性测试要求测试员在游玩的同时完成记录，这不同于事后复盘。实用做法是保持一个"双轨记录"：屏幕录制软件（如 OBS）在后台持续录像，测试员同步在文本文件中记录时间戳和操作序列。当 Bug 发生时，测试员立刻写下形如"12:34 — 在背包满载状态下拾取任务道具，道具消失但任务未完成"的记录。这种即时记录使 Bug 的**复现路径（Reproduction Path）**不依赖记忆，为后续提交 Bug 报告奠定准确基础。

## 实际应用

**开放世界地图边界测试**：测试员在《某开放世界 RPG》中选定"尝试离开地图边界"作为会话章程，60 分钟内采用冲刺跳跃、骑乘坐骑、利用地形卡位等多种方式尝试穿越边界。脚本化测试通常只验证官方设置的边界墙是否存在，而探索性测试员可能发现：在特定山坡角度骑马冲刺可以飞跃边界墙，跌入无底洞后游戏不会触发死亡判定，导致角色永久卡死。这类问题只有自由探索才能有效暴露。

**多人联机同步漏洞挖掘**：在两人联机测试会话中，一名测试员专门执行"与另一名测试员操作时间差攻击"：当队友正在触发存档点动画时，自己强制断网重连，观察存档状态是否发生错乱。这种协作式探索专门针对网络同步层的竞态条件（Race Condition），是任何预先编写的测试用例都难以精确描述的场景。

**UI/UX 快速压力测试**：测试员在菜单界面进行"快速反复点击"探索，例如在加载过渡动画播放期间疯狂点击确认键，检测按钮事件是否被正确节流（Debounce）。多个游戏曾因此暴露出重复触发购买确认逻辑、货币被多次扣除的严重 Bug。

## 常见误区

**误区一：探索性测试等于随机乱点**
许多初级测试员认为探索性测试就是"想怎么测就怎么测"，结果陷入反复测试自己感兴趣的区域，而忽略高风险模块。真正的探索性测试要求每次会话有明确章程，并在会话结束后产出一份简短的"会话报告"，注明覆盖区域、发现的 Bug、以及未来值得深挖的线索。没有章程的探索本质上是在浪费测试时间。

**误区二：探索性测试不需要任何前置知识**
部分人认为探索性测试"门槛低，谁都能做"。实际上，高质量的探索性测试高度依赖测试员对**游戏类型机制的先验知识**。一名从未玩过 MOBA 游戏的测试员无法凭直觉想到"同时击杀与被击杀会不会导致双方同时获得击杀分"这类边缘场景。正因如此，游戏公司在招募探索性测试岗位时，通常要求候选人具备相应类型游戏的深度玩家经验。

**误区三：探索性测试的结果无法量化**
一些团队认为探索性测试产出不可衡量，因此不将其纳入正式测试计划。事实上，SBTM 框架提供了明确的量化指标：每次会话的**Bug 发现率（Bugs per Session）**、**测试覆盖章程完成比例**、**未解答问题数量**均可统计。Cem Kaner 本人的研究数据显示，在相同时间投入下，训练有素的探索性测试员发现的 Bug 数量比纯脚本测试员高出约 **30%–40%**。

## 知识关联

探索性测试建立在冒烟测试的基础之上：冒烟测试确认游戏版本可稳定运行后，探索性测试才能高效展开；若游戏在探索中频繁崩溃，测试员无法完成连贯的操作序列，会话价值大幅降低。

探索性测试中发现的 Bug 直接引出两个后续技能：其一是**复现方法**——探索过程中发现的 Bug 往往路径复杂且偶发，测试员需要掌握系统性的复现技巧，将"刚才不知道做了什么导致崩溃"转化为可重复执行的步骤序列；其二是**验收测试**——探索性测试帮助团队发现真实玩家可能踩中的问题，验收测试则在此基础上正式评估游戏是否满足发布标准，两者共同构成游戏上线前的最终质量门控。
