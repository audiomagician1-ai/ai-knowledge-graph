# 更新兼容

## 概述

更新兼容（Update Compatibility）是游戏质量保证领域中的一项专项测试类别，专门验证玩家在不同版本切换时游戏数据、存档、功能能否正常延续。其核心覆盖三个维度：新旧版本客户端在同一设备上的共存行为、增量补丁的正确叠加逻辑，以及玩家历史存档在版本升级后的迁移正确性。

这一测试类型随着"服务型游戏"（Games as a Service, GaaS）模式的普及而变得不可回避。2013年前后，《英雄联盟》《炉石传说》等在线游戏将"热更新"机制推向主流，开发团队开始系统化地处理版本差异带来的兼容问题。如果更新兼容验证不充分，玩家可能面临存档损坏、进度回滚、物品丢失或游戏直接崩溃等严重后果，影响用户留存。

更新兼容测试的特殊性在于，它处理的是"时序状态"——同一个游戏对象（角色、道具、地图）在不同版本生命周期内的状态序列化与反序列化是否一致。这与纯功能测试或区域设置兼容有本质区别：后者处理空间维度的差异，而更新兼容处理时间维度的数据演化。

## 核心原理

### 版本号语义与兼容策略

更新兼容测试首先依赖语义版本控制（Semantic Versioning，SemVer）规范，格式为 `MAJOR.MINOR.PATCH`。其中：
- **PATCH 升级**（如 1.2.3 → 1.2.4）：仅修复 Bug，存档和协议必须完全向后兼容；
- **MINOR 升级**（如 1.2.x → 1.3.0）：新增功能，存档需向后兼容但可扩展新字段；
- **MAJOR 升级**（如 1.x → 2.0）：允许破坏性变更，必须提供存档迁移脚本或转换工具。

测试团队需要针对每类升级路径设计不同严格程度的用例矩阵，而非用同一套测试覆盖所有情况。

### 存档迁移的序列化模式

游戏存档通常以二进制（如 MessagePack）或文本格式（如 JSON、XML）序列化。更新兼容测试中，最危险的场景是**字段删除**和**字段类型变更**。例如，假设某 RPG 游戏在 v2.0 版本将玩家金币字段从 `int32`（最大值约 21 亿）扩展为 `int64`，持有超过 2,147,483,647 金币的玩家在新版本首次读取旧存档时，若迁移脚本遗漏类型转换，将直接导致数值溢出或读取失败。

测试策略应包含**边界存档构造**：人工制造含极值数据的旧版存档，在新版本中加载并验证数据完整性。

### 增量更新的叠加顺序验证

增量更新（Delta Patch）机制要求补丁必须按固定顺序叠加，跳过任何中间版本都可能导致资源文件的哈希校验失败。例如，玩家若从 v1.0 直接跳到 v1.3（跳过 v1.1 和 v1.2），游戏内某个角色模型的顶点缓冲区可能因差量计算基准错误而出现渲染破面。

更新兼容测试需构建**版本跨度矩阵**，验证所有可能的升级路径，包括：
- 连续升级路径（v1.0 → v1.1 → v1.2 → v1.3）
- 跨版本升级路径（v1.0 → v1.3、v1.1 → v1.3）
- 全量重新安装路径（直接安装 v1.3 后与增量升级结果比对）

三条路径安装完成后的游戏文件哈希值必须完全一致。

### 新旧版本客户端共存

部分平台（如 PC Steam Early Access）允许玩家在同一设备同时保留 beta 分支和正式分支。共存场景下，两个客户端共用的注册表键值、共享缓存目录或全局配置文件可能互相污染。测试需验证注册表写入路径是否按版本隔离（如 `HKCU\Software\GameTitle\v2.0` 与 `HKCU\Software\GameTitle\v1.9` 分别独立），以及卸载旧版本是否会误删新版本所依赖的共享 DLL。

## 实际应用

**案例一：《原神》大版本存档兼容**
《原神》每隔约 6 周发布一个大版本（如从 4.x 升至 5.0），涉及新增祈愿记录字段与角色天赋数据结构调整。更新兼容测试需验证：旧版本中已达满命（6命）角色的命座激活位图在新版本中能否被正确读取，以及未出现的新属性字段能否被旧存档以默认值正确填充而不触发异常。

**案例二：主机游戏的跨代存档传输**
《使命召唤》系列在 PS4 → PS5 升级时，测试团队需要验证玩家武器皮肤解锁记录（以 bitmask 存储）能否在新版本解包后仍然对应正确的外观 ID，尤其是当新版本调整了 ID 排列顺序时，需有映射转换表保证旧 ID 指向相同外观资产。

**案例三：手游 OTA 热更新失败回滚**
手游项目中，若某次 OTA（Over-The-Air）热更失败（如网络中断），客户端通常保留上一个完整版本的快照。更新兼容测试需模拟"下载到 40% 时中断"的场景，验证游戏能否正确回滚至上一个完整快照版本并正常启动，而不是卡在一个混合了两个版本资源的损坏状态。

## 常见误区

**误区一：只测最新版到当前版的升级路径**
许多团队仅测试"上一个版本 → 当前版本"的直接升级，忽略了仍有大量玩家停留在更早版本（例如长期未登录玩家可能从 v1.0 直接升级至 v3.5）。正确做法是维护一个**最小支持版本**策略，并对所有大于此跨度的升级路径进行实测，而非假设"链式升级等价于直接跨版升级"。

**误区二：存档兼容等同于功能正常**
存档成功加载并不代表数据语义正确。一个典型错误是：v1.0 中角色技能冷却时间以帧数（整数）存储，v2.0 改为秒数（浮点数）存储，迁移脚本直接复制旧值而未除以帧率（如 30 fps），导致玩家技能冷却被错误读取为原来 30 倍。测试必须验证数据**语义的正确性**，而非仅校验文件能否被解析。

**误区三：增量补丁测试等同于安装包测试**
增量补丁经过差量算法（如 bsdiff 或 xdelta3）生成，其产物与全量安装包是完全不同的代码路径。全量安装包测试通过并不能替代增量路径的验证，两者必须分别执行，且需对比最终文件系统状态是否完全一致。

## 知识关联

**前置概念：区域设置兼容**
区域设置兼容确立了"同一游戏在不同配置下的静态兼容性"基准测试框架，更新兼容在此基础上引入了时间维度——需要同时处理区域变量（如语言包路径）和版本变量，例如中文区玩家从 v1.0 跨版升级至 v2.0 时，语言包迁移逻辑与版本迁移逻辑必须同时被验证，不能仅单独测试其中一个维度。

**后续概念：模拟器测试**
模拟器测试是更新兼容测试的重要执行环境之一。在验证老旧版本存档（如 3 年前的 Android 系统版本上生成的存档）时，真实设备往往难以获取，模拟器可以还原指定 API Level（如 Android 7.0 / API 24）的系统环境以重现旧版本行为。掌握更新兼容测试的矩阵设计方法，是有效规划模拟器测试资源分配的前提条件。
