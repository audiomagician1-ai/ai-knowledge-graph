# 磁盘IO测试

## 概述

磁盘IO测试（Disk I/O Testing）是游戏性能测试中专门针对存储子系统的测量手段，核心目标是量化游戏在运行过程中对磁盘的读写速率、延迟表现以及存储空间占用情况。游戏客户端的所有资产——包括贴图、模型、音效、脚本——在首次使用前必须从磁盘加载到内存，IO性能直接决定了关卡加载时间、开放世界流式加载的卡顿频率以及存档读写的响应速度。

历史上，游戏IO测试的标准化需求在PS4/Xbox One世代（2013年前后）急剧上升，因为该世代主机普遍使用5400RPM机械硬盘，顺序读取速度约为50–100 MB/s，成为性能瓶颈的主要来源之一。PlayStation 5的定制SSD（标称5.5 GB/s原始读取速度）出现后，IO测试的关注点从"如何掩盖加载延迟"转变为"如何充分利用高带宽"，推动了整套测试方法论的更新。

对于采用流式加载（Streaming）架构的开放世界游戏，磁盘IO测试的质量直接影响玩家是否会在高速移动时遭遇资产弹出（Pop-in）或强制加载卡顿（Stutter），因此它是发布前认证（Submission Certification）的必要检查项之一。

## 核心原理

### 顺序读取与随机读取的区别

磁盘IO测试必须区分两种访问模式：**顺序读取（Sequential Read）**和**随机读取（Random Read/4K Random Read）**。顺序读取模拟大型关卡包（.pak文件）整体加载，SATA SSD的顺序读取通常达到500–550 MB/s；而随机4K读取（每次仅读取4096字节的随机位置）则模拟游戏引擎并发请求多个小型资产的场景，同一块SSD的随机4K IOPS可能仅为90,000–100,000，换算为带宽约为350 MB/s。对机械硬盘而言，随机读取性能会因磁头寻道（寻道时间约8–12ms）下降到顺序读取的1/100甚至更低。测试时若仅跑顺序读取而忽略随机访问，会严重低估真实游戏场景下的IO负担。

### 流式加载压力测试

流式加载压力测试的标准做法是在游戏内构造"最坏路径（Worst-Case Traversal）"——即以最高移动速度穿越相邻LOD区块边界密度最大的区域，同时使用工具（如Unreal Insights或自研IO监控模块）记录每帧的IO请求队列深度（Queue Depth）、待处理字节数（Pending Bytes）以及发生IO等待的帧数百分比（IO Stall Frame %）。行业经验值要求IO Stall Frame % 低于0.5%为合格，超过1%即需要优化资产打包策略或预加载逻辑。关键指标公式为：

```
IO带宽利用率 = 实际传输字节数 / (测试时长 × 设备标称带宽) × 100%
```

若利用率持续低于设备峰值带宽的30%，通常说明IO请求碎片化严重，需要合并资产包。

### 存储空间占用测试

存储空间占用测试不仅统计安装包总体积，还需追踪**运行时临时文件**（Shader Cache、Texture Cache、Save Data）的增长曲线。以PlayStation平台为例，SCE的认证要求游戏必须在安装页面精确声明所需空间，允许误差不超过实际值的5%。测试员需在游戏100%存档进度时测量ShaderCache体积（部分游戏的ShaderCache在完整游玩后可增长至2–4 GB），并验证卸载后临时文件是否被完整清除，防止出现"幽灵占用"问题。

## 实际应用

**开放世界关卡边界测试**：在《荒野大镖客：救赎2》这类地图超过100 km²的游戏中，QA团队会编写自动化脚本让角色以马速（约15 m/s）沿地图对角线连续骑行，实时采集IO吞吐量日志，并标注发生植被/建筑弹出的GPS坐标，反查对应时间点的IO队列积压情况。

**主机认证IO压力测试**：索尼的PS5技术要求（TRC R4163）规定游戏冷启动到可操作状态不得超过特定秒数阈值（通常为20–30秒，具体值随品类而异），QA会在完全冷机状态（清空SSD读取缓存）下重复测量10次取平均值，并对比HDD兼容模式（若支持）下的数据。

**DLC增量包IO验证**：DLC发布后，新资产若与基础包资产存在读取竞争，可能导致原本合格的IO指标恶化。测试需在安装DLC后重跑全部IO基准测试用例，对比增量前后的随机读取IOPS变化。

## 常见误区

**误区一：用PC测试结果直接代表主机IO表现**。PC测试环境通常使用NVMe SSD（读取速度3500 MB/s以上），而目标平台若为PS4（5400RPM HDD，读取约50–80 MB/s）则差异超过40倍。曾有项目因仅在开发用PC上通过IO测试，上线主机版本后出现超过60秒黑屏加载的严重问题。测试必须在目标硬件上执行，或使用IO速率限速工具（如Windows的磁盘限速功能设置为目标平台带宽上限）模拟。

**误区二：认为SSD普及后IO测试不再重要**。即便在PS5上，游戏若将数百万个小文件散列存放（每个文件＜16 KB），IO请求数量会使文件系统元数据访问成为瓶颈，实际有效带宽可能跌至标称值的10%以下。流式加载架构的资产打包策略仍然是IO测试的核心检查对象。

**误区三：存档（Save Data）IO不值得单独测试**。部分游戏在存档时序列化几十MB的游戏状态数据，若在战斗高峰帧存档（Auto-Save触发），IO写入会与渲染资产读取竞争带宽，导致帧率瞬时下降5–10 fps。存档IO需在游戏逻辑高负载状态下单独测量写入延迟（Write Latency），而不是在空闲主菜单测试。

## 知识关联

**与热量测试的关系**：热量测试验证了设备在持续高负载下的散热稳定性，磁盘IO测试则需要在热量测试通过后的"热机状态"下复测，因为NVMe SSD在高温时会触发温度节流（Thermal Throttling），导致持续顺序读取速度从3500 MB/s降至1500 MB/s左右，影响IO测试结论的可靠性。若先跑IO测试再跑热量测试，存在遗漏热节流导致IO性能下降的风险。

**与渲染管线分析的关系**：磁盘IO测试完成后，工程师掌握了资产加载进内存的速率边界。渲染管线分析则在此基础上研究资产进入GPU后的处理效率——IO决定了纹理能否按时送达显存，渲染管线分析则追踪纹理在显存中的采样效率和带宽占用。两者的衔接点是**纹理流送（Texture Streaming）**系统：IO测试验证磁盘→内存的传输速率，渲染管线分析验证内存→显存的传输速率和实际渲染使用情况。
