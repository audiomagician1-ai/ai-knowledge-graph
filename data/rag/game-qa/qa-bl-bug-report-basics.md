# Bug报告基础

## 概述

Bug报告（也称缺陷报告或问题单）是游戏QA测试人员将发现的软件缺陷记录下来并传递给开发团队的正式文档。一份合格的Bug报告必须包含六大核心要素：**标题、复现步骤、预期结果、实际结果、测试环境、证据附件**。缺少任何一项，开发工程师都可能无法定位或复现该问题，导致Bug被退回或修复方向错误。

Bug报告的规范化实践起源于1970年代软件工程体系化管理的兴起。在游戏行业，随着项目规模扩大、跨团队协作增多，非结构化的口头反馈已经完全无法支撑工作流程。现代游戏项目中常用的缺陷跟踪工具如Jira、Bugzilla、TestRail，都以这六要素为基础字段构建其表单模板。一个典型的AAA游戏项目在发布前可能积累数千条Bug记录，规范的报告格式使得这些记录可被搜索、分类、分配和追踪。

对于入门级QA测试人员，Bug报告写作是最基础也是最频繁使用的技能。一份填写混乱的报告不仅浪费开发工程师的时间，还会降低测试团队的可信度，影响整个缺陷修复流程的效率。

## 核心原理

### 1. 标题（Title）

标题是Bug报告的"门面"，必须在不超过80个字符的限制内准确描述缺陷的**对象**和**症状**。推荐格式为：`[模块/场景] + 触发条件 + 异常现象`。

- ❌ 差示例：`游戏崩溃了`
- ✅ 好示例：`[主城商店] 点击"购买"按钮后游戏崩溃，返回至主菜单界面`

好的标题让开发工程师在不展开详情的情况下就能判断Bug所属模块，方便快速分配责任人。

### 2. 复现步骤（Steps to Reproduce）

复现步骤是Bug报告中技术含量最高的部分。每一步必须以动词开头，且精确到**操作对象**和**具体动作**，不能跳过任何前置条件。步骤编号从第1步开始，建议控制在10步以内，超过则说明需要寻找更短的复现路径。

示例（以移动游戏商城Bug为例）：
1. 启动游戏，登录账号（服务器：Asia-01）
2. 点击主界面右下角「商城」图标
3. 在"限时礼包"分类下，点击售价648元的「至尊礼包」
4. 在弹出的购买确认窗口中，连续快速点击「确认购买」按钮3次
5. 观察扣款金额与库存变化

若步骤填写不完整，该报告复现率低于50%，工程师往往直接标注"无法复现"并关闭问题单。

### 3. 预期结果与实际结果（Expected vs. Actual Result）

这两项必须**成对出现**，是判定是否构成Bug的逻辑依据。

- **预期结果**：依据游戏设计文档（GDD）、策划案或用户常识，操作完成后应该发生的事。
- **实际结果**：测试人员亲眼观察到的真实现象，包括具体的错误码、异常数值或界面状态。

错误写法：实际结果写成"出现问题"；正确写法：`实际结果：扣款3次共计1944元，但背包中仅出现1份礼包内容，库存显示从10变为9（应变为7）`。

### 4. 测试环境（Environment）

游戏Bug的复现往往与硬件、操作系统、游戏版本强相关。环境信息必须包含以下字段：

| 字段 | 示例值 |
|------|--------|
| 游戏版本 | v2.3.1 Build 40821 |
| 操作系统 | iOS 17.2 / Android 13 |
| 设备型号 | iPhone 15 Pro / Samsung Galaxy S23 |
| 网络环境 | WiFi / 4G |
| 测试服务器 | 内网测试服 QA-Server-03 |

如果缺少版本号，即便同名Bug在新版本中已被修复，也可能被重复提交，造成冗余记录。

### 5. 证据附件（Evidence）

证据分为三种形式：
- **截图**：标注箭头指向异常区域，推荐使用Snagit或系统截图工具；
- **视频录像**：复现过程完整录制，移动平台可使用系统内置录屏，时长建议控制在30秒至2分钟；
- **日志文件**：崩溃类Bug必须附带`crash.log`，服务器连接类Bug需附带网络抓包记录。

没有证据的Bug报告在争议发生时无法自证，且视觉类Bug（UI错位、贴图异常）若无截图几乎无法核实。

## 实际应用

**场景：多人在线射击游戏中的碰撞体积Bug**

测试人员发现角色可以穿透某地图的一面具体墙壁，并在应用六要素后写出如下报告：

- **标题**：`[地图：废弃工厂] 角色从西侧入口冲刺时可穿越坐标(X:340, Y:12, Z:88)处的混凝土墙进入墙体内部`
- **步骤**：选择突击手角色→进入废弃工厂地图→移动至西侧入口标记点→按住Shift键冲刺并斜向撞击该墙
- **预期**：角色被墙壁阻挡，无法进入墙体内部
- **实际**：角色进入墙体，可在墙内向外射击而不被对方命中，构成严重竞技公平性漏洞
- **环境**：PC版 v1.7.0 Build 33201，Windows 11，DirectX 12
- **证据**：附30秒操作录像 + 坐标截图

这条报告使开发工程师无需额外沟通即可定位并复现问题。

## 常见误区

**误区1：将预期结果写成"正常运行"或"不报错"**

"正常运行"不是具体的预期结果，它没有描述任何可以被验证的状态。正确做法是对照设计文档写出明确的行为，例如"玩家金币减少648，背包内增加1份至尊礼包，礼包数量上限提示消失"。

**误区2：把复现步骤与实际结果混写在一起**

很多初学者会在步骤的最后一步写"然后游戏崩溃"，却忘记在实际结果字段中单独描述崩溃的完整表现（如崩溃弹窗内容、错误码0xC0000005）。这会导致字段信息重叠，搜索和统计时产生数据混乱。

**误区3：认为只要能复现就不需要附录证据**

部分测试人员认为步骤写清楚了就足够。但视觉类Bug（如UI元素轻微偏移2px）极度依赖截图，且证据是后续版本验证修复时的基准对照物。没有"修复前"的截图，修复后就无法确认改动是否满足预期效果。

## 知识关联

Bug报告基础是整个Bug生命周期的起点。一旦掌握六要素的填写规范，下一步自然进入**严重性与优先级**的判断——因为严重性（Severity）和优先级（Priority）正是在报告提交时需要同步填写的关键字段。严重性描述Bug对游戏功能的破坏程度（如S1崩溃、S2功能失效），而优先级则决定开发资源的调度顺序。理解Bug报告六要素后，你才能在填写严重性字段时准确依据"实际结果"中描述的破坏范围来判断等级，而不是凭直觉猜测。
