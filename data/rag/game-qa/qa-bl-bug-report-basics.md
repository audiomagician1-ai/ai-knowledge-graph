# Bug报告基础

## 概述

Bug报告（Bug Report）是游戏QA测试人员将发现的缺陷以书面形式记录下来、传递给开发团队的标准文档。一份合格的Bug报告必须包含六个核心要素：**标题、复现步骤、预期结果、实际结果、测试环境、证据附件**。缺少任何一项，开发人员都可能无法定位或复现该缺陷，导致Bug被搁置或错误修复。

Bug报告作为一种工程实践，最早在软件行业的系统测试流程中形成规范。1992年，Cem Kaner在其著作《Testing Computer Software》中首次系统性地定义了有效缺陷报告的结构要求，这套框架后来被游戏行业广泛沿用。游戏测试中，一个项目的Bug库少则数百条，多则上万条，清晰的报告结构直接决定了开发团队处理缺陷的效率。

对于游戏QA新人而言，Bug报告质量的高低往往比发现Bug数量更重要。一条写得模糊的报告可能让程序员花费数小时仍无法复现问题；而一条要素完整、表达精确的报告，通常15分钟内就能被定位并分配修复任务。

## 核心原理

### 标题：一句话定位问题

Bug标题要遵循"**位置 + 触发条件 + 现象**"的三段式结构。例如：  
- ❌ 差的标题：「游戏崩溃了」  
- ✅ 好的标题：「主城商店界面——点击购买按钮后——客户端崩溃并返回桌面」

标题的长度建议控制在25～50个汉字之间，既要让开发人员一眼看出问题出在哪个模块，又不能把复现步骤塞进标题里。同一个系统中的多个标题放在列表里时，任何人应能通过扫读标题列表快速区分每条Bug的唯一性。

### 复现步骤：逐步可操作的指令链

复现步骤必须从一个明确的**起点状态**开始，例如"新建角色，等级1，站在新手村出口坐标(120, 45)"。每个步骤用数字编号，每步只做一个操作动作，绝不写"进行一系列操作后"这类模糊描述。

一份合格的复现步骤应满足**可重复性原则**：任何一名初次接触该项目的测试员按照步骤操作，复现率应达到100%（必现Bug）或明确标注复现率如"约5次中出现3次（60%）"（概率性Bug）。对于概率性Bug，还需注明测试者执行了多少次才总结出这个概率。

### 预期结果与实际结果：对比揭示偏差

预期结果（Expected Result）必须有依据，通常来源于**策划文档、UI设计稿或用户常识**，不能写测试员的个人喜好。例如：「根据战斗设计文档v2.3第8页，普通攻击对同级敌人的基础伤害应为角色攻击力的100%」。

实际结果（Actual Result）要描述客观观察到的现象，包含具体数值。例如：「实测伤害数值显示为攻击力的47%，且伤害数字颜色变为灰色（正常为白色）」。"游戏表现不对"或"效果很奇怪"这类描述无法写入实际结果栏，必须量化或具体化。

### 测试环境：精确定位复现条件

游戏Bug的复现往往与运行环境强相关。测试环境一栏需记录：
- **硬件平台**：如 iPhone 14 Pro / Windows 11 + RTX 3070
- **操作系统版本**：如 iOS 16.4.1 / Android 13
- **游戏版本号**：如 Build 0.8.23-rc2（不能只写"最新版"）
- **网络状态**：WiFi / 4G / 本地包

同一个Bug在不同GPU驱动或不同分辨率下可能表现完全不同。漏写版本号是新手最常见的错误，当该Bug被修复后还需要对应版本号进行回归测试验证。

### 证据附件：让开发人员亲眼看见

视频录屏是游戏Bug报告中最重要的证据形式，尤其对于动画异常、帧率问题、音效错误等难以用文字描述的缺陷。截图需标注箭头或红框指出异常位置，分辨率应不低于1080p。日志文件（Log File）在崩溃类Bug中是必须附带的，因为程序员需要通过堆栈信息（Stack Trace）才能定位代码位置。证据文件应按"Bug编号_描述_日期"命名，例如`BUG-2341_购买崩溃_20240315.mp4`。

## 实际应用

**场景：移动游戏的角色穿模Bug**

某手游内测期间，测试员发现角色在特定地形会陷入地面。一份完整的报告写法如下：

- **标题**：森林副本第3关——角色跳跃落地时——角色下半身穿入地形网格消失
- **步骤**：①进入森林副本第3关；②走到坐标(88, 12)处的斜坡边缘；③按跳跃键后立即向左移动；④角色落地瞬间观察模型表现
- **预期**：角色正常站立于地面，模型完整显示
- **实际**：角色腰部以下穿入地面，仅显示上半身，持续约2秒后弹回正常位置
- **环境**：Samsung Galaxy S21 / Android 12 / 游戏版本Build 1.2.7 / WiFi
- **证据**：附录屏视频BUG-0892_穿模_20240210.mp4（时长12秒）

这条报告让负责碰撞检测的程序员在收到报告后20分钟内即复现并定位到斜面碰撞体设置错误。

## 常见误区

**误区1：把"预期结果"写成"希望修复"**  
初学者常在预期结果栏写「希望点击按钮后不要崩溃」或「建议修改此处逻辑」。预期结果是基于设计文档的客观标准描述，修复建议应放在独立的备注栏，两者混淆会让开发人员误以为这是优化需求而非缺陷。

**误区2：跳过中间步骤，认为"显而易见"**  
测试员熟悉流程后会自动省略某些操作步骤，比如省略"先完成第一章主线任务"这类前置条件。一旦开发人员复现不了，该Bug往往会被直接标注为"无法复现（Cannot Reproduce）"并关闭，等于白报。复现步骤要假设读者对游戏一无所知。

**误区3：一条报告夹带多个Bug**  
发现角色穿模的同时发现音效也有异常，部分新人会在同一条报告里描述两个问题。Bug追踪系统（如Jira、Mantis）以单条Bug为单位进行分配、修复和关闭，混合报告会导致其中一个问题被忽略，或两个问题被不同的人重复处理。**一条报告，仅描述一个缺陷**，这是行业通行准则。

## 知识关联

Bug报告基础是进入游戏QA领域的第一项硬技能，所有后续的测试工作都建立在能够书写合格报告的前提上。

掌握六要素写法之后，下一步需要学习**严重性（Severity）与优先级（Priority）**的区分与标注。严重性描述Bug对游戏功能的破坏程度（如S1崩溃级到S4界面级），优先级描述修复的紧迫程度，两者都需要在Bug报告提交时由测试员给出初始评估，填写在报告的元数据字段中。正确标注严重性是本节六要素之外的第七个常规字段，是Bug报告从"合格"走向"专业"的关键一步。
