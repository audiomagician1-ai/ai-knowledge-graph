# 发布质量标准

## 概述

发布质量标准（Release Quality Criteria）是游戏QA流程中用于决定某个版本是否具备发布条件的量化判断体系，通俗称为"Go/No-Go决策标准"。它将Bug统计数据转化为明确的发布准入门槛：例如规定版本上线前P0级（游戏崩溃/无法启动）Bug数量必须为零，P1级（核心玩法阻断）Bug不超过3个且有明确规避方案，P2级（功能受损但可绕过）Bug不超过15个。

这套标准的雏形来自软件工程领域1970年代的"零缺陷"运动，但游戏行业在2000年代主机游戏时代将其系统化，原因在于主机游戏的物理光盘一旦压盘便无法打补丁（早期Xbox/PS2时代），发布后的Bug修复成本极高。随着数字发行和热更新普及，标准变得更灵活，但核心Go/No-Go框架保留至今。

发布质量标准的核心价值在于将"能不能发"这个带有主观压力的决定转化为可核查的数字指标，避免项目压力下团队以"差不多能玩"为由强行发布存在严重问题的版本。当PM或发行方要求提前发布时，QA Lead可以用标准文件说明"当前P1 Bug为7个，超出准入阈值3个，当前状态为No-Go"，而非依赖个人判断进行争论。

## 核心原理

### Bug严重性分级与发布阈值的对应关系

发布质量标准必须建立在明确的Bug优先级分类之上。行业常见的四级体系中，P0（Blocker/Critical）指游戏完全无法运行、存档丢失、付费功能失效等；P1（Major）指主线流程卡死、核心战斗系统异常；P2（Minor）指非核心功能瑕疵；P3（Trivial）指文本错误、视觉细节问题。

Go/No-Go决策公式可表示为：

**发布状态 = f(N_P0, N_P1, N_P2, N_P3)**

其中最常见的硬性规则为：
- **N_P0 = 0**（P0级Bug必须清零，无例外）
- **N_P1 ≤ T1**（T1为团队预设阈值，典型值为0~5，视项目规模）
- **N_P2 ≤ T2**（T2通常为10~30，具体取决于游戏类型）
- **P3不作为阻断条件**，但会进入Known Issues列表随版本发布说明公开

手机游戏与主机/PC游戏的阈值往往不同：手机游戏因可随时热更，P2阈值通常放宽到50~80个，而买断制PC游戏首发版本P1阈值通常严格控制在3个以内。

### Known Issues列表与豁免机制

当某个P1或P2级Bug在发布截止日前无法修复时，并非自动触发No-Go，而是进入**豁免流程**：QA需提供该Bug的复现概率（如"需要特定操作序列，复现率约12%"）、影响范围（影响全体玩家/仅影响特定机型）、以及规避方案（如"重启游戏后可恢复"）。项目负责人、QA Lead和发行方代表三方确认签字后，该Bug被列入Known Issues，并在发布说明中公示，方可豁免。

豁免的Bug不会消失于统计，而是标记为"Accepted Risk"状态，仍计入版本Bug密度报告，供下次迭代优先处理。

### Bug密度指标与历史基线对比

单纯的Bug数量不足以全面判断质量，还需结合**Bug密度（Bug Density）**：即每千行代码（KLOC）或每个功能模块内的Bug数量。一个拥有50万行代码的大型RPG，P2级Bug 20个与一个10万行的休闲游戏P2级Bug 5个，实际质量状况可能相当。

更重要的是与该项目**历史版本基线**对比：若本项目前三个版本发布时P2平均为12个，而当前版本为28个，即便未超过绝对阈值，也应触发额外的质量审查流程（Extended QA Cycle），这被称为"趋势异常报警"机制。

## 实际应用

**场景一：手机游戏版本节点评审**

某手机MMORPG在春节活动版本（v2.3.0）上线前48小时召开Go/No-Go会议。QA统计显示：P0 Bug 0个，P1 Bug 2个（其中1个已有客户端规避补丁，1个已提交修复待审批），P2 Bug 34个，P3 Bug 127个。参照该项目质量标准文件（P1≤3为Go条件），当前状态为**条件Go**：P1中的已修复项需在上线前2小时通过回归验证，否则回退到No-Go。

**场景二：主机DLC审核（平台认证要求）**

索尼PlayStation和微软Xbox均有各自的Technical Requirements Checklist（TRC/XR）认证标准，其中索尼PS5平台要求所有提交版本必须通过约400项强制检查项目，任何一项不通过均为Blocker。在这种情况下，发布质量标准必须同时满足内部阈值**和**第三方平台硬性清单，两者取更严格者为准。

## 常见误区

**误区一：认为"P0清零"就等于可以发布**

P0 Bug清零只是必要条件，而非充分条件。若P1 Bug数量超出阈值，即便P0为零，正确决策仍是No-Go。许多新人QA会将"没有崩溃Bug"误读为"可以发布"，忽略了P1级核心玩法阻断问题同样可能严重损害玩家体验和评分。

**误区二：发布质量标准是固定不变的**

发布质量标准应随项目阶段调整：Alpha阶段的P2阈值可以设为100，Beta阶段收紧到30，黄金版本（Gold）严格控制到10。用同一套标准贯穿整个开发周期会导致早期过度紧张（浪费修复资源）或后期标准失真（低质量发布）。

**误区三：No-Go决定等于QA否决发布**

Go/No-Go是由包括制作人、QA Lead、发行方代表在内的多方共同签字的**决策机制**，而非QA单方面的否决权。QA的职责是提供准确的Bug统计数据和状态报告，最终发布决定权归项目负责人。QA在会议上的角色是"数据提供者+标准守护者"，而非"发布裁判"。

## 知识关联

发布质量标准的输入直接来自**Bug统计分析**：没有准确的Bug数量统计、优先级分布图和趋势曲线，Go/No-Go会议就无法基于事实进行判断。Bug统计分析为质量标准提供了原始数据，而质量标准将这些数据转化为二元决策。

掌握发布质量标准后，自然衍生出两个方向：一是**Bug沟通技巧**，即如何在Go/No-Go会议上向非技术的制作人或发行方清晰表达当前质量状况、阈值差距和风险影响；二是**回归报告**，它是Go/No-Go会议的核心支撑文件，记录了每个修复项是否通过回归验证，直接决定P1 Bug能否从"待验证"状态变更为"已关闭"从而满足发布阈值。
