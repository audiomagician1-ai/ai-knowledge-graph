# 多平台回归

## 概述

多平台回归是指在游戏发布或更新时，针对同一版本的构建物（Build）在多个目标平台（如 PC、iOS、Android、PlayStation、Xbox、Switch）上同步或并行执行回归测试的策略。其核心目标是在单次测试周期内验证跨平台代码共享层与各平台特有渲染/输入/存档逻辑是否存在行为差异，而不是对每个平台依次单独走完完整回归流程。

多平台回归的需求随着跨平台游戏引擎（如 Unity 和 Unreal Engine 4/5）的普及而急剧上升。2010 年代初期，大多数工作室仍以"平台顺序测试"为主流，导致 PC 版通过后 Console 版延迟数周上线的情况极为常见。现代 AAA 工作室普遍在 CI/CD 流水线中为每个平台分配独立的测试节点，要求所有平台的回归测试结果在同一 Sprint 窗口（通常 24–48 小时）内收敛。

多平台回归的价值在于提前暴露"平台差异缺陷"（Platform-Divergent Bug），即同一逻辑在某一平台表现正常、在另一平台表现异常的问题。这类缺陷如果延迟到单平台测试结束后才发现，修复成本将成倍增加，因为需要重新触发已通过平台的增量回归。

## 核心原理

### 平台矩阵与测试覆盖分层

多平台回归并非将完整测试用例集在每个平台上全量执行，而是根据"平台差异风险矩阵"将测试用例分为三类：

- **全平台强制用例**：涉及共享游戏逻辑、存档兼容性、网络同步的用例，在所有目标平台上全量执行，占总用例量约 30%–40%。
- **平台特有用例**：涉及特定平台 SDK（如 PlayStation Network 成就、Switch Joy-Con 振动）的用例，仅在对应平台执行。
- **抽样对比用例**：针对渲染输出、帧率稳定性的截图比对用例，采用代表性场景抽样，典型比例为总场景数的 15%–20%。

这种分层策略使总测试工时相比"全量×平台数"方案降低约 50%–60%，同时保留了跨平台差异的检出能力。

### 并行构建同步机制

多平台回归的前提是各平台 Build 的时间戳对齐。通常在 CI 系统（如 Jenkins、TeamCity）中设置"平台构建门控"：只有当所有平台的构建任务均成功完成，才触发统一的回归测试任务分发。若某平台构建失败，其余平台的测试结果会被标记为"不可比较状态"，防止用单平台通过结论掩盖跨平台问题。

版本标识采用统一的 Build Hash（例如 `game-build-20240315-abc123`），该 Hash 在所有平台 Build 中保持一致，确保回归测试报告聚合时能够按同一版本基线进行跨平台对比，而非误将不同代码版本的测试结果混为一谈。

### 差异检测与基准比对

多平台回归引入"跨平台一致性基准"（Cross-Platform Baseline）概念：以某一主平台（通常为 PC）的输出作为参考基准，其他平台的对应输出与之进行结构化比对。比对维度包括：

- **逻辑输出**：相同输入序列下游戏状态机的最终状态值（如角色坐标、血量、物品栏数据）必须在允许误差 ±0.001f 内一致（浮点精度差异除外）。
- **渲染输出**：使用像素差异率（PDR，Pixel Difference Rate）衡量，行业常见容忍阈值为 PDR < 2%，超出则触发人工审查。
- **性能指标**：各平台帧率基准不同（PC 目标 60 FPS、Switch 目标 30 FPS），因此性能回归判定采用平台独立阈值，而非跨平台统一值。

## 实际应用

**场景一：移动端双平台同步回归**
某手游工作室在每次热更新后，对 iOS 和 Android 双平台执行多平台回归。由于两个平台的图形 API 不同（Metal vs. OpenGL ES/Vulkan），渲染相关的 30 个用例会在两端并行运行并互相比对截图 PDR。2023 年某次更新中，该机制提前发现了仅在 Android Vulkan 驱动下出现的 UI 层叠错误，避免了该版本进入公测。

**场景二：主机平台三端回归矩阵**
某跨平台 RPG 维护 PS5、Xbox Series X、PC 三端回归矩阵。全平台强制用例集包含 820 条用例，平台特有用例共 340 条（PS5 成就系统 130 条、Xbox 成就系统 130 条、PC Mod 支持 80 条）。每次版本分支合并后，三端并行测试总耗时控制在 36 小时以内，满足双周迭代的发布节奏要求。

**场景三：与热修复回归的衔接**
在紧急热修复（Hotfix）场景下，多平台回归通常只执行"全平台强制用例"子集（约 40% 用例量），跳过平台特有和抽样对比用例，以便在 6–8 小时内完成所有平台的基线验证，快速决策是否可以全平台同步推送修复包。

## 常见误区

**误区一：以单平台通过结果代替多平台回归**
部分团队认为"PC 通过即等于逻辑层通过，控制台只需测渲染"。实际上，主机平台的内存管理策略（如 PS5 的 UMA 架构与 PC 的独立显存）会导致相同代码在内存压力下触发不同的缓存淘汰路径，进而引发只在主机端出现的崩溃或数据损坏问题。逻辑层用例必须在每个目标平台上独立执行。

**误区二：认为多平台回归只需对比最终画面**
仅做截图 PDR 比对会遗漏大量非渲染层的平台差异缺陷，包括输入延迟差异、存档序列化字段顺序差异（不同平台字节序问题）、网络 NAT 穿透行为差异等。多平台回归需要覆盖逻辑、渲染、性能、网络四个维度，而非仅关注视觉输出。

**误区三：所有平台使用同一套通过/失败阈值**
将 PC 的性能回归阈值（如帧率波动容忍 ±5 FPS）直接套用到 Switch 或中低端 Android 设备上，会导致大量误报（False Positive），降低测试置信度。多平台回归必须为每个平台预先标定独立的性能基准和容忍区间。

## 知识关联

**前置概念——热修复回归**：热修复回归建立了最小化用例集的筛选方法论，多平台回归在热修复场景下直接复用这套"最小强制用例集"逻辑，将其扩展为"多端并行执行"模式。热修复回归中对用例优先级的分类（P0/P1/P2）在多平台回归的平台矩阵分层中同样作为强制执行与抽样执行的分级依据。

**后续概念——回归排期**：多平台回归的并行测试耗时数据（各平台用例执行时长、构建等待时间）是制定回归排期的关键输入。排期规划需要基于历史多平台回归的时间分布，为不同发布类型（热修复、小版本、大版本）预留合理的平台测试窗口。

**后续概念——跨平台一致性**：多平台回归是跨平台一致性验证的执行机制，跨平台一致性则定义了"何为可接受的平台间差异"的标准规范，包括逻辑等价性定义、PDR 阈值制定、平台独立性测试边界划定等内容，是多平台回归判定通过/失败的理论依据。
