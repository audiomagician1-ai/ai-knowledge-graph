# 内存检测

## 概述

内存检测（Memory Profiling）是游戏性能测试中专门针对RAM使用情况的系统性分析方法，涵盖运行时内存占用追踪、内存泄漏定位、堆碎片化分析以及内存预算管理四个核心维度。与GPU Profiling关注显存和渲染管线不同，内存检测的对象是系统主内存（RAM）以及托管堆（Managed Heap）中的对象生命周期。

内存检测作为独立测试流程的形成，与主机平台的严格内存限制密切相关。PS4和Xbox One均配备8GB统一内存（GDDR5），其中操作系统预留约2.5GB，游戏可用内存约5.5GB。在这种硬性上限下，内存预算管理从"可选优化"变为"发布门槛"，推动了系统化内存检测方法的成熟。

在游戏质量保证流程中，内存检测能够捕捉到仅凭帧率监控无法发现的问题——例如游戏运行2小时后因积累的内存泄漏导致的崩溃，或大型开放世界场景切换时瞬间峰值超出平台限制的内存分配。这类问题若未被检测发现，往往直到玩家投诉才暴露，修复成本极高。

## 核心原理

### 内存占用追踪与快照对比

内存追踪通过定期采集内存快照（Memory Snapshot）并对比两个时间点的对象分配差异来定位增长来源。Unity的Memory Profiler（Package版本1.0.0+）支持捕获详细的托管堆快照，记录每个对象实例的类型、大小和引用链。标准检测流程是：在场景加载完成后取基准快照（Baseline Snapshot），执行标准游戏操作序列后取第二张快照，再通过Diff视图找出新增且未释放的对象。

Unreal Engine则使用`stat memory`命令实时显示各内存池的占用，并可通过`memreport -full`生成详细报告，输出包含每个UObject实例的内存占用细节。对于主机平台测试，PS5提供的`SceKernelGetDirectMemorySize()`系统调用可精确返回可用物理内存量，QA工程师通常在测试脚本中每60秒记录一次该数值。

### 内存泄漏检测

内存泄漏指程序分配的内存因未被正确释放而永久占用，在长时运行测试（通常为2小时以上的压力测试）中表现为内存曲线持续单调递增。检测泄漏的核心指标是：关卡往返加载后（Load → Play → Unload → Load），内存应回到与首次加载后相近的水平，允许误差通常设定在±5MB以内。若每次往返后内存净增长超过此阈值，则判定存在泄漏。

C++游戏代码中常见的泄漏模式包括：监听器注册后未注销（尤其是事件系统中的Lambda捕获导致的循环引用）、异步加载资源后句柄未释放，以及单例类持有场景对象引用。Valgrind的`--leak-check=full`选项在PC开发版本上能精确报告泄漏的字节数和分配调用栈，但其运行时开销约为原程序的10-30倍，仅适合专项泄漏排查。

### 堆碎片化分析

堆碎片化（Heap Fragmentation）发生在频繁分配和释放不同大小内存块之后，导致虽然总可用内存充足但无法满足单次大块连续分配请求的情况。游戏中常见场景是在战斗系统中每帧创建销毁大量短生命周期对象（如子弹、特效粒子），数千次操作后托管堆可能形成"碎片夹心"结构。

碎片率（Fragmentation Ratio）可通过公式评估：

```
碎片率 = 1 - (最大可用连续块 / 总空闲内存)
```

该值越接近1，碎片化越严重。Unity的GC使用Boehm垃圾回收器，在碎片化严重时会触发更频繁的GC收集，进而产生帧率尖刺（GC Spike）。针对碎片化问题，通常采用对象池（Object Pool）模式预分配固定数量对象，将动态分配频率降低90%以上。

### 内存预算管理

内存预算是依据目标平台可用内存对各系统进行定量分配的规划文档。典型的主机游戏内存预算划分示例如下：纹理资源40%、网格与动画25%、音频15%、代码与逻辑10%、其他（UI、物理、AI数据）10%。QA测试中需验证每个独立场景在峰值状态下不超过各自的预算上限，并在测试报告中标注峰值发生的具体游戏事件（如触发大型爆炸特效时）。

## 实际应用

**开放世界流式加载测试**：在《原神》类型的开放世界游戏中，QA工程师会沿预定路线匀速行走，每隔30秒记录一次内存占用，重点关注区块（Chunk）加载卸载时的内存曲线，确认已离开视野的区块内存是否在15秒内完全释放，防止流式系统积累过多已卸载区块的残留数据。

**多人联机大厅测试**：在同时容纳100个玩家角色的场景中，单次测试从0逐步加入到满员，记录每加入10名玩家时的内存增量。若增量非线性增长（例如前50人每人增加2MB，后50人每人增加5MB），则提示存在随玩家数量非线性扩张的数据结构，需与程序员确认是否符合设计预期。

**长时间会话压力测试**：使用自动化脚本模拟玩家连续游玩6小时（进行商店浏览、背包操作、多次副本进出），每10分钟记录一次托管堆大小。若6小时后堆大小超过初始值的150%且趋势未见收敛，则记录为高优先级内存泄漏缺陷。

## 常见误区

**误区一：GPU显存占用高等同于内存占用高**。显存（VRAM）和系统内存（RAM）是独立的资源池，在非统一内存架构（如PC的独立显卡）上，一个纹理同时占用显存（供GPU采样）和系统内存（供CPU访问）的双倍空间。内存检测工具报告的是系统内存数据，若仅依赖GPU Profiler中的显存指标来判断内存是否超标，会完全遗漏CPU侧的内存状态。

**误区二：内存在测试结束时恢复正常就代表无泄漏**。部分泄漏具有"阶段性积累"特征：每次完整游戏会话结束时操作系统回收了所有进程内存，表面看起来正常，但在单次会话内部执行某个特定操作（如读取存档）30次以上后，内存会超出平台限制导致崩溃。因此必须在不重启进程的前提下执行长时测试，才能捕捉到此类泄漏。

**误区三：内存占用越低越好**。预分配内存是主流主机游戏的常见优化策略——启动时一次性申请大块内存并自行管理分配，使运行时内存曲线呈"高位平稳"而非"动态波动"。QA工程师若见到内存占用在启动后立即达到4GB并保持稳定，不应将其记录为异常，而应确认该值是否在预算范围内且曲线无持续增长趋势。

## 知识关联

内存检测与GPU Profiling（前置知识）构成互补关系：GPU Profiling中发现的高Draw Call数量往往对应着CPU侧大量渲染状态对象的内存分配，在内存检测时需重点核查这些对象的生命周期是否受控。例如，通过GPU Profiler发现大量重复材质实例（Material Instance）后，在内存快照中可以量化这些实例占用的总字节数，为合并材质的优化工作提供数据依据。

内存检测的结论直接影响**加载时间测试**（后续概念）的分析方向：若内存检测发现场景切换时内存释放缓慢，则加载时间测试中观察到的加载耗时增加，很可能源于新场景资源等待旧场景内存释放完成，而非磁盘I/O或资源解压本身的瓶颈。掌握内存检测数据能帮助QA工程师在加载时间测试中更准确地将耗时归因到正确的性能瓶颈上。
