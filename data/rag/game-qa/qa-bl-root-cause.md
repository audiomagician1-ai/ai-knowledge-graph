# 根因分析

## 概述

根因分析（Root Cause Analysis，简称RCA）是一套在Bug已被稳定复现后，通过结构化方法追溯问题**根本成因**的诊断流程。它的目标不是修复表面症状，而是找到导致Bug出现的最深层原因，从而在代码或设计层面做出一次性修复，防止同类问题反复出现。

根因分析方法论起源于20世纪50年代日本丰田汽车的质量管理实践，由工程师大野耐一将"5-Why追问法"系统化。随后，石川馨（Kaoru Ishikawa）在1960年代将因果图（鱼骨图）引入工业质量控制，该方法后来被广泛移植到软件工程和游戏QA领域。

在游戏Bug生命周期中，根因分析处于"复现确认"之后、"Bug评审会"之前这一关键节点。如果跳过根因分析直接提交修复，开发团队往往只打补丁而不治根本，导致同一模块在不同版本中持续产出同类缺陷，拉高整体Bug密度，最终影响发布质量。

## 核心原理

### 5-Why追问法

5-Why方法要求分析人员对一个Bug症状连续追问"为什么"至少5次，每一次回答都成为下一次追问的对象，直到找到无法再被追问的根本原因。

**游戏场景示例：**
- Bug现象：玩家在副本BOSS战中帧率骤降至15 FPS。
- Why 1：为什么帧率骤降？→ Draw Call数量在0.3秒内从200暴增至1200。
- Why 2：为什么Draw Call暴增？→ BOSS释放技能时动态生成了大量粒子特效对象。
- Why 3：为什么粒子对象导致Draw Call暴增？→ 每个粒子使用了独立材质实例，未走GPU Instancing合批。
- Why 4：为什么未走合批？→ 特效美术创建材质时未勾选"Enable GPU Instancing"选项。
- Why 5：为什么会漏勾？→ 特效资产提交规范中没有将该选项列为强制检查项。

**根本原因**：特效资产提交检查清单存在规范漏洞。修复方向：在Asset Pipeline的CI检查中增加GPU Instancing的自动化验证规则，而非仅仅修改那一个材质。

### 鱼骨图（石川图）分析法

鱼骨图以Bug表现作为"鱼头"，从"鱼骨"延伸出多条主因分支，游戏QA领域通常采用以下6个分支维度（6M分类）：

| 分支 | 游戏QA对应含义 |
|------|---------------|
| Man（人） | 策划需求描述不清、程序理解偏差 |
| Machine（机器） | 目标机型GPU/CPU性能差异 |
| Method（方法） | 错误的算法或数据结构选型 |
| Material（素材） | 资产规格不符合引擎限制 |
| Measurement（测量） | 测试用例覆盖不足、边界值遗漏 |
| Environment（环境） | 特定操作系统版本、驱动兼容性问题 |

鱼骨图的实操价值在于它能**防止分析人员过早锁定单一假设**。分析师在独立填写各分支后再集体讨论，往往能发现5-Why单线追问容易忽略的并发成因。例如，某游戏的存档丢失Bug，5-Why追问指向写文件API调用时序错误，而鱼骨图同时揭示出测试团队未覆盖"低存储空间"边界场景（Measurement分支），两者共同构成完整的根因图谱。

### 代码审查定向法

当5-Why或鱼骨图将根因范围缩小到具体模块后，定向代码审查（Targeted Code Review）负责在源代码层面找到精确的引入点（Introduction Point）。与常规代码审查不同，RCA中的代码审查聚焦于以下三个维度：

1. **Git Blame追溯**：定位引入问题代码的具体提交（Commit Hash），确认是新增功能、重构还是热修复导致的回归。
2. **调用链分析**：针对逻辑错误类Bug，逐帧或逐调用栈追踪数据流向，找到数值在哪一步发生了非预期变化。
3. **边界条件验证**：对照原始需求文档，检查代码是否对所有约定的输入范围做了防御性处理，尤其关注`null`检查、数组越界、整数溢出（如HP值存储在`int16`时超过32767的溢出Bug）等。

三种方法在实践中通常组合使用：鱼骨图确定分析范围 → 5-Why缩小到具体模块 → 代码审查定位精确行号。

## 实际应用

**案例：RPG游戏装备强化成功率显示为NaN**

测试工程师通过复现步骤确认：当强化材料数量为0时，成功率显示"NaN"（Not a Number）。

应用5-Why：Why 1→除数为零；Why 2→材料数量允许输入0进入计算函数；Why 3→强化UI层未做数量下限校验；Why 4→策划文档中强化功能的前置条件章节仅写"需要足够材料"，未明确"材料数量≥1"的硬性约束；Why 5→需求评审模板中没有"前置条件数值边界"的必填栏位。

根因：**需求模板缺陷**。修复行动：更新需求模板并在UI逻辑层补充`materialCount >= 1`的输入拦截。

## 常见误区

**误区一：把第一个"为什么"的答案当作根因**

很多分析人员发现"代码写错了"就停止追问，直接提Bug修复单。但"代码为何写错"才是更有价值的问题——是注释误导了开发者、单元测试覆盖不足，还是需求文档存在歧义？停在第一层只能解决单个Bug，找到第4、5层才能修复产出Bug的系统性缺陷。

**误区二：根因分析等于责任追究**

RCA的产出是**流程或技术改进方案**，而非判断某位程序员或策划"犯了错误"。如果分析会议演变成追责会议，团队成员会倾向于隐瞒信息，导致鱼骨图的多个分支无法被如实填写，分析结论失真。正式的RCA文档中只记录"哪个环节出现了什么类型的缺陷"，不记录个人姓名。

**误区三：跳过鱼骨图直接进行代码审查**

对于复杂Bug，如果跳过多因素分析直接扎进代码，很容易陷入"修了一个引用，漏了另一个路径"的循环。某款MMO曾因跳过鱼骨图分析，对一个掉线Bug做了3次补丁修复，最终才通过鱼骨图发现根因是服务端心跳包超时阈值与客户端重连等待时间的配置不匹配——这属于"Method"（方法/配置）分支的问题，纯粹靠代码审查难以发现。

## 知识关联

**前置依赖——复现方法**：根因分析必须建立在稳定复现的基础上。若Bug复现率低于100%，5-Why的每一步追问都可能因现象不稳定而得出错误假设。复现步骤文档（STR）直接为代码审查的"调用链分析"提供入口，决定了从哪个函数开始追踪。

**后续概念——Bug评审会**：根因分析的输出物（根因报告、改进行动项）是Bug评审会的核心输入材料。评审会需要根因报告来决策修复优先级——同一Bug模块如果根因指向架构设计问题，评审优先级通常会升级为P1；若根因仅为单行输入校验遗漏，则可能维持P3处理。没有完整的根因报告，评审会的优先级决策将缺乏依据。
