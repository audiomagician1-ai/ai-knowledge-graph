# 根因分析

## 概述

根因分析（Root Cause Analysis，简称RCA）是在游戏QA流程中，通过系统性追溯手段找出导致Bug出现的最深层原因，而非仅修复表面症状的方法论。游戏Bug往往呈现"治标不治本"的问题：例如某款角色扮演游戏中NPC寻路失效，若只修复当前地图的碰撞体积，同类地图上的相同Bug会在两周后再次出现。根因分析的目标是找到那个"如果修复了它，同类Bug将不再出现"的根本缺陷。

根因分析方法在工业制造领域由丰田生产系统（TPS）于20世纪50年代系统化，其中"5-Why分析法"由大野耐一提出。这套方法在2000年代初随着敏捷开发和游戏工业化进入游戏QA领域，成为Bug生命周期中"分析阶段"的标准工具。对于游戏测试而言，根因分析尤其重要，因为游戏系统之间的耦合极强——一个物理引擎参数的根本错误，可能同时引发碰撞、动画、音效三个模块的Bug表现。

根因分析通常在Bug已被稳定复现之后展开，复现方法确认了Bug的触发路径，而根因分析则沿着这条路径向代码或设计的源头溯源。其输出结果将直接服务于后续的Bug评审会，为开发团队提供修复优先级和改进方案的决策依据。

## 核心原理

### 5-Why分析法

5-Why分析法的操作规则是：针对一个Bug的直接现象，连续追问"为什么"至少5次，每一个答案成为下一个"为什么"的输入，直到抵达无法再向上追溯的根本原因。

以实际游戏案例说明：某射击游戏玩家报告"子弹穿透了墙体"。  
- 第1个Why：为什么子弹穿透墙体？→ 碰撞检测未触发。  
- 第2个Why：为什么碰撞检测未触发？→ 子弹移动速度超过了碰撞检测的计算频率。  
- 第3个Why：为什么子弹速度超过检测频率？→ 策划将子弹速度数值从800调整为2400而未通知程序。  
- 第4个Why：为什么策划修改数值时无需通知程序？→ 参数表缺少数值范围限制和跨部门变更通知字段。  
- 第5个Why：为什么参数表没有这些约束？→ 游戏设计文档（GDD）中未规定物理参数的变更审批流程。  

根因即为：**缺少针对物理参数修改的跨部门审批规范**。修复此根因后，未来类似数值超限导致的穿透类Bug会被流程拦截，而非等到QA发现。

### 鱼骨图（石川图）分析法

鱼骨图由日本质量管理专家石川馨于1943年提出，其结构将Bug现象置于"鱼头"，将可能的原因分支绘制为"鱼刺"，每条主刺代表一个大类原因，支刺代表具体因素。在游戏QA中，常用的6个主刺分类为：**代码逻辑、数据配置、引擎版本、硬件环境、测试流程、需求文档**。

以某手机游戏UI在特定机型上显示错位的Bug为例，鱼骨图会同时展开6个方向的原因假设：代码方向检查适配分辨率的计算公式（如是否使用了硬编码的`1920×1080`基准值）；数据配置方向检查UI锚点参数；硬件环境方向对比该机型的屏幕宽高比是否为非标准的`21:9`；引擎版本方向确认Unity的`Canvas Scaler`设置是否随版本升级被重置。鱼骨图的价值在于防止分析团队陷入单一假设，强制进行多维度的并行排查。

### 代码审查定向分析法

当5-Why或鱼骨图将根因定位到代码层面时，需要对与Bug相关的代码模块进行定向审查（Targeted Code Review）。与常规代码审查不同，根因分析中的代码审查有明确的排查范围：以Bug首次出现的版本号为基准，通过Git的`git bisect`命令进行二分查找，将"可疑提交"的范围从数百次缩小到1-3次。

具体步骤为：`git bisect start` → 标记当前Bug版本为`bad` → 标记最近正常版本为`good` → 通过逐步测试将范围二分缩小。此方法在拥有1000次提交历史的项目中，最多只需测试`log₂(1000)≈10`次即可定位问题提交。找到目标提交后，审查其diff内容，结合5-Why的逻辑链条确认根因代码行。

## 实际应用

**案例：开放世界游戏的掉帧Bug**  
某开放世界RPG在玩家进入特定区域时帧率从60fps骤降至18fps。复现方法已确认触发条件为"角色站在X:1240、Z:880坐标附近"。根因分析小组先用5-Why追问：帧率下降→Draw Call数量在该区域激增至3200次（正常区域为400次）→该区域的草地使用了非实例化渲染→场景美术导入草地资源时未勾选GPU Instancing选项→项目资源导入规范文档中未包含植被类资源的Instancing强制检查项。根因定位为**资源导入规范的缺失**，而非代码Bug，修复方向是更新规范并添加自动化检查脚本。

**案例：多人游戏的同步错误**  
某多人竞技游戏中，客户端A和B对同一技能伤害值的显示相差15点。鱼骨图分析后，在"代码逻辑"分支发现：伤害计算公式中浮点数精度处理在两端使用了不同的取整方式（一端`Math.Floor`，另端`Math.Round`），这一差异在基础伤害为整数时不显现，但在暴击倍率介入后被放大。根因为**多端计算逻辑未统一规范**，修复方案为将所有伤害计算迁移至服务端权威计算。

## 常见误区

**误区一：将直接原因等同于根因**  
许多测试人员在找到第一个"技术原因"后便停止追溯，例如发现"碰撞层级设置错误"即认为找到根因并提交修复。实际上，还需继续追问"为什么碰撞层级会被错误设置"——若答案是"无人负责审查物理层级的变更"，则根因是流程缺失，仅修改该碰撞层级设置，同类错误会在1-2个版本后重现。衡量是否抵达根因的标准是：**修复此原因后，同类Bug的发生概率应趋近于零**。

**误区二：鱼骨图流于形式，只填写已知假设**  
部分团队在压力下将鱼骨图变成"答案已知，形式上填充"的文档工作。真正有效的鱼骨图分析要求每个分支都必须经过实际验证或明确排除，而非凭主观判断直接忽略某分支。例如，不能因为"这是新机型问题"就跳过对代码逻辑分支的核查——某款游戏的"新机型崩溃"Bug，最终根因恰恰是代码中一处未处理的空指针异常，而非硬件兼容性问题。

**误区三：根因分析等同于追责**  
在游戏团队中，根因分析的目的是改善系统和流程，而非认定责任人。若分析会议演变为追责会议，团队成员会在5-Why的第3步便开始刻意引导方向，导致根因被人为遮蔽。正确的根因分析输出应为：**一条可操作的流程改进建议或技术规范更新**，而非"某开发人员操作失误"这类结论。

## 知识关联

根因分析以"复现方法"的成果为起点：稳定的复现步骤是5-Why分析能够进行连续追问的前提条件，若Bug无法稳定复现，则无法验证每个"为什么"的答案是否成立，分析链条会在第2或第3步断裂。建议将复现率低于80%的Bug推迟进行根因分析，等待更充分的复现数据。

根因分析的输出文档——包含根因描述、修复建议和预防措施的RCA报告——是Bug评审会的核心输入材料。评审会需要基于根因分析的结论决定：该Bug需要代码修复还是流程整改，修复的优先级是P0紧急还是纳入下一迭代，以及是否需要对相似模块进行专项扫描测试。没有完整根因分析支撑的Bug评审会，容易陷入主观判断和重复讨论同类问题的困境。
