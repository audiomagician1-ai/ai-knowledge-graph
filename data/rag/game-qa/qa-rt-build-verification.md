# 构建验证测试

## 概述

构建验证测试（Build Verification Test，简称BVT）是游戏QA流程中针对每一个新构建版本（Build）执行的自动化冒烟测试套件，其核心目标是在15分钟至30分钟内判断该构建是否具备进入完整回归测试的基本资格。BVT并不验证功能的完整正确性，而是验证构建本身是否"活着"——游戏能否启动、核心系统能否初始化、关键入口点是否可达。

BVT这一概念最早由微软在1990年代的Windows开发流程中正式化，当时的内部规定是：每日构建（Daily Build）必须通过一套固定的冒烟用例才能分发给测试团队。游戏行业在2005年前后随着主机游戏认证流程的标准化，开始大规模采用类似机制，尤其是索尼PS平台的TRC（Technical Requirements Checklist）对BVT的通过率有明确的准入要求。

在游戏QA的回归测试体系中，BVT扮演的角色是"门卫"而非"检察员"。一个未通过BVT的构建会被标记为Broken Build，整个回归排期中安排好的测试任务将暂停执行，直到该构建被修复并重新提交。这一机制防止了测试团队在一个根本无法运行的版本上浪费大量人力资源。

## 核心原理

### BVT用例的选取标准

BVT用例集必须满足三个硬性约束：单次执行时间不超过30分钟、覆盖所有关键崩溃路径、用例失败时能精确定位到具体模块。具体到游戏项目，BVT通常包含以下几类验证点：游戏主进程能否在目标平台（PC/主机/移动端）成功启动并到达主菜单；存档系统能否完成一次读写循环而不抛出异常；核心网络模块（如房间创建、匹配队列）能否完成握手；以及基础渲染管线能否在不崩溃的情况下渲染第一帧。

BVT用例数量通常控制在50至150条之间。用例数量过多会导致执行时间超过触发回归测试的等待容忍阈值；用例过少则会让明显的系统级缺陷漏过验证门槛。许多团队采用"P0用例即BVT"的简化规则，即将优先级为P0（游戏无法启动或主流程完全阻断）的缺陷所对应的验证场景全部纳入BVT。

### 自动化执行流程

BVT在CI（持续集成）服务器上以全自动方式触发，典型的触发条件是代码库主分支（main/master）每次合并完成后自动启动构建，构建产物生成后立刻拉起BVT任务。整个流程可以用以下步骤描述：

1. **构建编译**：编译服务器拉取最新代码，生成可执行构建包
2. **部署到测试设备**：自动将构建推送到预配置的目标设备池（Device Farm）
3. **执行BVT脚本**：自动化框架（如Unity Test Framework、Unreal Automation Tool或自研框架）按序执行预定义用例
4. **结果上报**：测试结果通过API写入测试管理平台（如TestRail或JIRA），同时向构建负责人发送通知
5. **门控决策**：若通过率低于预设阈值（通常为100%通过或允许不超过2条非关键用例失败），该构建被标记为Rejected，阻断后续回归任务下发

BVT报告必须包含每条用例的通过/失败状态、失败时的堆栈截图或日志片段，以及与上一个通过构建的对比差异，方便开发人员快速定位引入缺陷的提交。

### 通过标准与失败分级

BVT的失败并非等同于"构建作废"，需要根据失败类型进行分级处理。**硬阻断（Hard Block）**：游戏进程崩溃、无法启动、BVT执行环境本身错误——此类情况构建直接Rejected，所有下游测试停止。**软失败（Soft Failure）**：单一非核心用例失败但游戏进程依然存活——此类情况构建可有条件进入部分回归测试，但失败用例对应的功能模块测试暂停，并同时创建缺陷单追踪。这种分级机制确保一个轻微缺陷不会导致整个测试排期全线停摆。

## 实际应用

**案例一：移动端每日构建验证**
在一个移动端多人对战游戏项目中，开发团队每天产出2至3个构建（早间构建、午间热修复构建、晚间构建）。BVT配置为在每个构建上传至内部分发服务器后15分钟内自动执行，覆盖启动、登录、进入对局大厅三个核心路径共计72条用例。若早间构建在10:00前通过BVT，QA团队即可在10:30前开始当日回归测试，保障每日6小时有效测试时间不被浪费在破损构建上。

**案例二：主机认证前的BVT强化**
在向索尼提交PS5版本进行洛特检查（Lot Check）前的两周内，BVT用例集会临时扩展，加入所有TRC必检项目的自动化验证（约30条），使BVT总量临时升至180条。这一阶段BVT的通过标准从"100%通过"升级为"0容忍"，任何失败直接触发全员Blocker处理流程。

## 常见误区

**误区一：BVT通过代表构建质量良好**
BVT通过仅代表构建具备基本可测性，而非功能正确性。一个BVT全绿的构建可能包含数十个P1/P2级缺陷，只是这些缺陷没有触碰到BVT覆盖的路径。将BVT结果作为"构建质量报告"对外沟通会严重误导项目管理决策。BVT是准入门槛，不是质量证书。

**误区二：BVT用例越多越好**
扩充BVT用例集以提高覆盖率的冲动很常见，但会导致执行时间超过可接受范围。若BVT需要2小时才能完成，开发人员在等待结果期间会积累更多提交，增加问题定位难度；测试团队的启动时间也会随之推迟。BVT的价值在于快速反馈，而非全面覆盖——全面覆盖是后续完整回归测试套件的职责。

**误区三：BVT自动化脚本无需维护**
BVT脚本需要随游戏版本迭代持续维护。当UI布局改变、场景加载顺序调整或网络协议升级后，原有BVT脚本可能因定位元素失效而产生假失败（False Failure）。若团队忽视脚本维护，BVT结果的可信度会迅速下降，最终演变为"狼来了"——开发人员开始忽视BVT失败报告，失去了门控机制的全部价值。

## 知识关联

BVT建立在**回归排期**所确定的构建节奏和优先级基础之上——回归排期定义了"何时需要哪个构建通过验证"，BVT则是执行这一判断的自动化手段。如果回归排期设定为每日两次构建窗口，BVT的执行时间上限就必须匹配这一节奏，通常不超过单个构建窗口间隔的25%。

BVT通过后，测试流程进入**功能开关回归**阶段——在使用Feature Flag管理功能发布的项目中，BVT通常只验证所有开关处于默认状态下的行为，功能开关的组合验证属于下一阶段的专项工作。此外，BVT自动化脚本的触发机制和结果上报直接依赖**CI/CD集成**所提供的流水线基础设施，包括构建触发器配置、设备池管理和测试报告API对接，这是BVT从手动验证转变为真正自动化门控的技术前提。
