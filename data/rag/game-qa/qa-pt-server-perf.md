# 服务器性能

## 概述

服务器性能是指游戏服务器在承载玩家并发请求时所表现出的处理能力，通常通过 TPS（Transactions Per Second，每秒事务数）、QPS（Queries Per Second，每秒查询数）、响应时间（Response Time）和最大承载量（Capacity）四个核心指标来量化。与客户端性能不同，服务器性能的瓶颈直接影响所有在线玩家的游戏体验——单台服务器的崩溃可导致数百乃至数千名玩家同时掉线。

该领域的系统化测试方法兴起于2000年代中期，随着 MMORPG 游戏（如《魔兽世界》2004年上线时因服务器过载导致大规模排队事件）的爆发式增长而得到重视。现代游戏后端通常采用分布式微服务架构，使得服务器性能测试从单点压测演进为多节点协同压测，测试复杂度大幅提升。

游戏服务器性能测试的核心价值在于：在正式开服或版本更新前，通过模拟真实玩家行为（包括登录风暴、副本并发、交易市场高频操作等场景）来确定系统的承载上限，防止因容量规划不足而导致的服务质量下降。

## 核心原理

### TPS 与 QPS 的计算与区别

TPS 指服务器每秒完成的完整事务数，一个事务通常包含请求发送、服务器处理、数据库读写、响应返回四个阶段；QPS 则仅统计每秒收到并处理的查询请求数，不要求完整事务闭环。在游戏场景中，玩家释放一次技能可能触发 1 次 TPS（伤害结算事务），但同时产生 5～10 次 QPS（查询技能数据、校验 CD、读取装备属性等）。

计算公式如下：

> **TPS = 并发用户数 × 每用户每秒发起事务数**  
> **平均响应时间 = 并发用户数 / TPS**（利特尔定律，Little's Law）

利特尔定律（Little's Law）是服务器容量规划的数学基础：`L = λ × W`，其中 `L` 为系统中的平均请求数，`λ` 为平均到达速率（即 QPS），`W` 为平均响应时间。若目标响应时间为 100ms，QPS 为 10,000，则系统中同时存在的请求数应控制在 1,000 以内。

### 响应时间分层分析

游戏服务器响应时间通常被拆解为三层：**网络传输时延**（客户端到服务器的 RTT，典型值 20～80ms）、**服务器处理时延**（逻辑计算 + 线程调度，目标值 < 50ms）、**数据库 I/O 时延**（MySQL 单次查询目标值 < 10ms，Redis 缓存命中目标值 < 1ms）。

测试时需使用百分位数（Percentile）而非平均值来评估响应时间，因为游戏玩家对高延迟的感知来自长尾请求。行业通用标准是：P50 ≤ 50ms，P95 ≤ 200ms，P99 ≤ 500ms。若 P99 超过 1 秒，通常意味着存在锁竞争、慢查询或 GC 停顿问题。

### 承载量测试的阶梯加压模型

承载量测试不采用一次性并发到峰值的方式，而是使用**阶梯加压法（Step Load Testing）**：每个阶梯持续 5～10 分钟，每次增加 500 或 1,000 个并发连接，同时监控 CPU 使用率、内存占用、TPS 变化和错误率。当 TPS 不再随并发数增加而线性提升，或错误率超过 1%，即判定为服务器已达到性能拐点（Knee Point）。

游戏行业常见的服务器承载目标为：单个战斗服务器承载 200～500 个同区玩家，单个世界服务器承载 5,000～20,000 名在线玩家，具体数值因游戏类型（MOBA、MMORPG、FPS）差异显著。

## 实际应用

**大型副本开放压测场景**：某 MMORPG 版本更新引入 100 人团队副本，测试团队需模拟 100 名玩家在 5 秒内同时进入副本并高频释放技能。测试结果显示副本服务器 TPS 峰值达 8,500，但此时 P99 响应时间从正常的 120ms 飙升至 2,300ms，定位发现是副本初始化时对 MySQL 的 100 条并发写入触发了行锁竞争，优化方案是改用批量写入（Batch Insert）并引入异步队列，将 P99 压降至 380ms。

**开服登录风暴测试**：新服开服时，玩家在前 10 分钟内集中登录，QPS 峰值可达日常的 10～20 倍。测试工具（如 JMeter、Locust）需预先录制登录协议包（包含账号验证、角色加载、好友列表拉取等步骤），模拟 5,000 个账号在 60 秒内完成登录。若登录服务 CPU 超过 85% 或认证服务响应时间超过 3 秒，则需扩容网关节点或优化 Token 验证逻辑。

**交易行高频查询测试**：在游戏交易市场场景下，单个玩家每秒可发起 3～5 次物品搜索请求，千人在线时 QPS 轻易突破 3,000。测试重点在于验证 Redis 缓存层的命中率（目标 > 95%）与 MySQL 读库的查询性能，防止缓存穿透导致数据库雪崩。

## 常见误区

**误区一：用平均响应时间代替百分位响应时间**  
许多团队报告服务器平均响应时间为 80ms 便认为性能达标，但实际上 5% 的请求可能响应时间超过 2 秒（P95 = 2,000ms），这 5% 的玩家会体验到明显卡顿。正确做法是始终同时关注 P50、P95、P99 三个分位数，并在测试报告中明确列出。

**误区二：把并发用户数等同于同时在线人数（CCU）**  
游戏服务器中，同时在线 10,000 名玩家并不意味着同一时刻产生 10,000 个并发请求。真实的并发请求数取决于玩家行为模式：挂机玩家每 30 秒才发一次心跳包，而高强度战斗玩家每秒可能发出 20 个请求。压测时应根据玩家行为分布建立混合流量模型，而非直接将 CCU 作为并发数输入。

**误区三：只测单节点而忽视分布式一致性开销**  
在微服务架构下，单节点压测结果优异，但上线后多节点协作时性能大幅下降的情况十分常见。原因在于节点间的 RPC 调用（如战斗服向数据服同步玩家状态）、分布式锁竞争以及消息队列堆积均会引入额外延迟。服务器性能测试必须在接近生产环境的完整集群上进行，而非仅在单台测试机上完成。

## 知识关联

**与移动端性能的关联**：移动端性能测试关注单设备的 CPU/GPU/内存占用，而服务器性能测试关注的是多用户并发下的服务端处理能力，二者分别代表客户端与服务端的性能维度。在联网游戏的完整性能评估中，移动端帧率下降有时并非客户端瓶颈，而是服务器响应延迟（P99 过高）导致客户端逻辑等待，因此两类测试结果需要交叉比对。

**与性能报告的关联**：服务器性能测试产出的 TPS 曲线、响应时间分布图、承载量拐点数据，是性能报告中服务端章节的核心内容。性能报告需将测试数据与预设的性能基线（Baseline）对比，明确标注系统在哪个并发规模下开始出现性能退化，为运维团队的扩容决策和开发团队的优化优先级排序提供量化依据。
