# 测试用例设计

## 概述

测试用例（Test Case）是描述"如何验证某个功能是否正确"的最小可执行单元，由前置条件、操作步骤和预期结果三个必要字段共同构成。在游戏QA领域，一个完整的测试用例不是随意的操作记录，而是可被不同测试人员重复执行并得到一致结论的标准化文档。

测试用例设计作为正式方法论，最早在1950至60年代随软件工程兴起而系统化，IEEE 829标准（1998年发布，2008年修订为IEEE 29119）明确规范了测试用例文档的格式要求。在游戏测试中，由于一款商业游戏上线前往往需要覆盖数百乃至数千个功能点，没有结构化的测试用例，团队就无法追踪哪些功能已被验证、哪些尚未覆盖。

测试用例的核心价值在于将"测试什么"与"怎么测"从测试人员的个人经验中剥离出来，变成团队共享的知识资产。一名新加入的测试人员拿到一套写好的用例，无需额外培训即可执行测试并判断结果是否通过，这正是标准化用例的关键作用。

## 核心原理

### 测试用例的三段式结构

每个测试用例必须包含以下三个字段，缺一不可：

**前置条件（Preconditions）**：执行本用例之前系统必须处于的状态。例如，测试"玩家购买道具"的用例，其前置条件应写明"账号已登录、角色等级≥10级、背包剩余格数≥1、金币余额≥500"。前置条件写不清楚，会导致不同人执行同一用例却得到不同结果——这是用例失效最常见的原因。

**操作步骤（Steps）**：按顺序编号的具体操作，每一步只描述一个动作。正确写法："1. 打开商店界面；2. 点击'消耗品'分类；3. 选中'初级回血药'；4. 点击'购买'按钮，数量输入1"。错误写法："进入商店买一个药水"——这种模糊描述无法保证两个测试员执行的是同一路径。

**预期结果（Expected Result）**：系统应该产生的可观测、可判断的输出。预期结果必须具体，例如"背包中增加1个'初级回血药'，金币减少500，右上角弹出提示文字'购买成功'"，而不是"购买成功"这样无法量化的描述。

### 测试用例的可追溯性

每个用例应携带一个唯一的用例ID（如`TC-SHOP-001`），并通过需求编号与对应的游戏功能需求挂钩。这种追溯关系使得当需求变更时，测试负责人能快速定位哪些用例需要更新。例如商店系统的需求文档编号为`REQ-SHOP-003`，则所有覆盖该需求的用例均在"关联需求"字段中标注该编号。

### 用例设计的四项基本原则

1. **原子性**：一个用例只验证一个具体行为，不要在一个用例里同时验证"购买"和"退款"两个功能。
2. **确定性**：对于同一个正确实现的功能，任意时间执行该用例都应得到相同的预期结果，不能依赖随机事件（例如不能把"随机掉落率"的验证写成单次执行即可判断通过的用例）。
3. **独立性**：用例A的执行结果不能成为用例B的前置条件——若A失败则B无法执行，导致覆盖率虚假下降。
4. **可重复性**：用例执行完毕后，应有恢复步骤（teardown）将环境复原，方便下次执行。

## 实际应用

### 游戏登录功能的测试用例示例

| 字段 | 内容 |
|------|------|
| 用例ID | TC-LOGIN-003 |
| 测试标题 | 使用正确的账号密码登录成功 |
| 前置条件 | 账号"testuser01"已注册且未被封禁；客户端版本1.2.5；网络连接正常 |
| 操作步骤 | 1. 启动游戏客户端；2. 在账号框输入"testuser01"；3. 在密码框输入"Test@1234"；4. 点击"登录"按钮 |
| 预期结果 | 3秒内进入游戏主界面，左上角显示角色名"测试角色01"，登录等待动画消失 |
| 关联需求 | REQ-LOGIN-001 |

### 技能系统的边界验证用例

在验证游戏技能CD（冷却时间）时，假设某技能的冷却时间设计值为5秒，测试人员应设计以下用例：TC-SKILL-011验证"在CD结束前0.5秒再次点击技能无效"，TC-SKILL-012验证"在CD恰好结束后立即点击技能可正常释放"。这两个用例针对5秒这个具体数值的边界进行检验，而不是泛泛测试"技能可以使用"。

## 常见误区

### 误区一：将预期结果写成操作描述

初学测试人员常把预期结果写成"点击购买按钮"或"系统处理请求"——这是对操作步骤的重复，而非系统应产生的结果。预期结果必须站在用户或系统输出的角度描述，回答"完成操作后，界面/数据/声音/网络请求发生了什么变化"。

### 误区二：前置条件过于宽泛导致用例不可复现

"账号已登录"作为前置条件是不充分的。在游戏测试中，角色等级、服务器区服、当前活动状态、背包内物品数量等都可能影响测试结果。遗漏这些条件会导致用例在不同环境下执行结果不一致，测试团队误判为"偶现bug"，浪费定位时间。

### 误区三：一个用例塞入多个验证点

某些测试人员为了"提高效率"，在一个用例中同时验证正常购买、余额不足、背包满三种场景。这违反了原子性原则——一旦用例失败，无法判断是哪个场景导致的失败，bug定位成本大幅上升。三种场景应分别写为TC-SHOP-001、TC-SHOP-002、TC-SHOP-003三个独立用例。

## 知识关联

**前置知识——功能测试基础**：功能测试基础课程介绍了"验证功能是否符合需求规格"的概念，测试用例设计是将这一目标转化为可执行文档的具体方法。没有功能测试的概念框架，测试人员不知道用例应该以需求为锚点来设计，而非仅凭个人经验随意测试。

**后续知识——等价类划分**：掌握了测试用例的三段式结构之后，下一个问题是"对于一个输入框，要设计多少个用例才算充分"。等价类划分技术正是回答这个问题的方法——它将无限的输入空间分割为有限个等价类，每个等价类只需选取1个代表值设计用例。例如一个接受1至99级玩家等级的输入框，等价类划分会告诉你需要设计覆盖有效范围（如50级）、下边界（1级）、上边界（99级）和无效范围（0级、100级）共5类用例，而不是盲目地写几十个用例。
