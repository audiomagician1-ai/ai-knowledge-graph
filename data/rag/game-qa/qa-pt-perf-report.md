# 性能报告

## 概述

性能报告（Performance Report）是将游戏性能测试数据（Profiling Data）转化为结构化文档的过程，其核心目的是将 CPU 帧时间、GPU 渲染耗时、内存占用峰值等原始数字转化为开发团队可直接执行的优化行动项。一份完整的游戏性能报告通常包含测试环境描述、性能指标汇总表、瓶颈定位分析、逐优先级排列的优化建议四个固定模块。

这一文档实践在 2000 年代主机游戏开发中逐步规范化，起因是 PlayStation 2 时代开发商发现口头汇报导致修复优先级混乱，进而引入书面报告制度。如今各大游戏公司（如 Naughty Dog、Ubisoft）都设有专门的性能工程师（Performance Engineer）岗位，其核心产出物就是定期提交的性能报告。

性能报告对游戏 QA 团队的价值在于：它将 Unity Profiler、Unreal Insights、RenderDoc 等工具输出的海量数据"翻译"成程序员和美术能够理解的语言，使帧率卡顿（Frame Spike）、内存泄漏（Memory Leak）等问题具备可追溯性和可验证性。若无规范报告，同一个 GPU Overdraw 问题可能在不同版本中被重复提交三到四次。

## 核心原理

### 性能指标的选取与阈值设定

撰写性能报告的第一步是明确**关键性能指标（KPI）**及其通过/失败阈值。常见指标包括：

- **帧率（FPS）**：移动端游戏通常以 30 FPS（33.3ms/帧）为最低达标线，主机/PC 端以 60 FPS（16.6ms/帧）为目标。
- **帧时间（Frame Time）**：比帧率更精确，需记录平均值（Average）、第 95 百分位（P95）和第 99 百分位（P99），其中 P99 > 50ms 通常定义为"严重卡顿"。
- **内存占用**：iOS 设备上超过设备物理内存的 50% 即触发系统警告，报告中须注明测试设备型号（如 iPhone 12，4GB RAM）。
- **发热与电量消耗**：移动端报告必须包含测试 15 分钟后的设备温度与电池消耗百分比。

报告中每个指标必须附带**基准值（Baseline）**，即上一个稳定版本或立项初期制定的目标值，偏差百分比用公式 `Δ = (当前值 - 基准值) / 基准值 × 100%` 表达。

### 可视化图表的规范构建

性能报告的图表不是装饰，而是承载信息密度最高的部分。标准的报告可视化包含以下几类：

1. **帧时间折线图（Frame Time Graph）**：横轴为游戏运行时间（秒），纵轴为每帧耗时（ms），需用红色水平线标注 16.6ms 目标线。图中所有超过阈值的尖峰（Spike）需用序号标注，并在文字说明中一一对应解释成因。

2. **CPU/GPU 分时饼图或条形图**：将总帧时间拆解为渲染线程（Render Thread）、游戏逻辑线程（Game Thread）、物理（Physics）、动画（Animation）等子系统耗时，直观呈现性能瓶颈所在。

3. **内存趋势图（Memory Trend Graph）**：横轴为测试持续时间，纵轴为内存占用（MB），若曲线单调递增而未出现回落，则可初步判定存在内存泄漏，需在报告中标注斜率（MB/分钟）。

4. **热区截图（Hotspot Screenshot）**：使用 Unreal Insights 的 GPU Visualizer 或 Unity 的 Frame Debugger 截取最耗时的单帧，标注具体 Draw Call 数量（如"该场景 Draw Call = 1243，超出目标值 800 的 55.4%"）。

### 优化建议的撰写规范

优化建议是报告的核心输出，每条建议必须遵循"**问题 → 证据 → 建议 → 预期收益**"四段式结构：

- **问题**：明确描述现象，如"场景 Level_03 在进入时帧率跌落至 18 FPS，持续约 2 秒"。
- **证据**：引用具体数据，如"CPU Profiler 显示 `LoadAssetAsync()` 在主线程耗时 68ms，占该帧总耗时的 81%"。
- **建议**：给出可操作步骤，如"将资产加载迁移至异步子线程，参考 Unreal 的 `FStreamableManager::RequestAsyncLoad` 接口"。
- **预期收益**：估算优化后的改善幅度，如"预计可将进场帧时间降至 20ms 以内，恢复至 30 FPS 以上"。

每条建议还需标注**优先级（P0/P1/P2）**：P0 为阻断发布的性能崩溃问题，P1 为影响玩家体验的卡顿，P2 为超出目标但尚在可接受范围的优化项。

## 实际应用

### 移动端游戏低端机性能报告案例

在测试一款卡牌手游的 Android 端性能时，测试工程师使用 Android GPU Inspector 采集数据后撰写报告，发现红米 Note 9（Snapdragon 662）设备在战斗界面的平均帧率为 21.4 FPS，P95 帧时间高达 72ms。报告中的 CPU 分时图显示粒子系统（Particle System）更新占用主线程 28ms，是头号瓶颈。优化建议列为 P1，具体建议将活跃粒子数上限从 512 压缩至 128，并为 GameObject 粒子系统添加 LOD 策略。报告附带了修改后的预期帧时间模拟计算：`28ms × (128/512) ≈ 7ms`，理论上可将帧率提升至 28 FPS 以上。

### 服务器端性能报告中的接口响应时段分析

针对多人游戏的服务器性能测试，报告需统计各 API 接口的平均响应时延（Latency）和错误率。例如匹配服务（Matchmaking Service）在并发 2000 人时的 P99 延迟为 4.2 秒，超出设计目标（1.5 秒）的 180%，报告需以时序图展示服务器在高峰时段的 CPU 利用率与数据库查询耗时的正相关关系，并建议引入 Redis 缓存层以减少重复数据库查询。

## 常见误区

### 误区一：只报告平均值，忽略百分位数

许多初级测试人员撰写报告时只写"平均帧率 58 FPS"，看似达标，但若 P99 帧时间高达 80ms，玩家每 100 帧就会感受到一次明显卡顿。正确做法是同时报告 Average、P50、P95、P99 四个统计量，特别是帧时间的长尾分布对用户体验影响远大于均值。

### 误区二：优化建议缺乏量化的预期收益

"建议优化 Shader 以提升性能"这类表述在专业性能报告中不可接受，因为它无法帮助开发团队进行排期决策。每条建议必须估算具体收益（如"预计减少 GPU 耗时 3-5ms"），即便这个数字是基于经验推算，也要注明推算依据，否则报告失去优先级排序的客观依据。

### 误区三：图表与文字说明脱节

报告中插入了帧时间折线图，但文字部分直接跳转讨论优化方案，完全没有解释图中标注的 Spike ①②③分别对应什么操作。读者必须自行猜测图表含义，导致报告的实际传达效率极低。正确做法是在图表标题下方紧跟 2-3 句对图中关键异常点的文字说明，形成图文一一对应。

## 知识关联

性能报告直接依赖**服务器性能**测试中采集的原始指标数据，包括 Tick Rate、并发连接数、数据库查询延迟等服务器侧指标，这些数据若未在测试阶段被正确记录，报告中的服务器性能章节就无从撰写。客户端侧则需要熟练操作 Unity Profiler 的 CPU Usage 模块和 Memory Profiler，以及 Unreal Insights 的 Timing Insights 视图，才能获取帧时间分解数据作为报告素材。

在游戏 QA 工作流中，性能报告是性能测试阶段的**最终交付物**，通常提交给性能工程师、主程序员和制作人三方，并在版本迭代后与上一份报告进行对比分析，形成纵向的性能健康度追踪（Performance Health Tracking）机制。掌握性能报告的撰写规范，能够使测试人员的工作成果从"发现问题"升级为"推动问题解决"，这是 QA 职能从执行层向分析层晋升的关键能力之一。
