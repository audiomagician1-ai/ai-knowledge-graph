# 通道打包

## 概述

通道打包（Channel Packing）是一种将多张灰度贴图合并存储进单张RGBA纹理的四个独立颜色通道中的数据压缩策略。在游戏特效领域，一张普通的PNG或TGA图像拥有R（红）、G（绿）、B（蓝）、A（透明）四个8-bit通道，每个通道可独立存储0~255范围的灰度值，这意味着一张贴图理论上可以同时承载四张完全独立的灰度数据序列。

通道打包最早在次世代游戏引擎的材质系统中被广泛应用，尤其是在GGX-PBR工作流普及后（约2014年前后），美术团队开始将粗糙度、金属度、AO等灰度图打包进同一张贴图以减少采样次数。在序列帧特效领域，这一策略被进一步延伸：当水花、爆炸、烟雾等特效需要同时驱动多种视觉变量时，通道打包可将原本需要4张独立序列帧贴图压缩为1张，显著降低GPU的纹理带宽占用和Draw Call数量。

在移动端游戏特效中，通道打包的价值尤为突出。一套完整的水花特效往往需要透明度蒙版、扭曲强度图、流动方向图和泡沫遮罩四种数据，若分别存储则需要读取四次纹理，而通道打包后仅需一次采样，在Mali和Adreno等移动端GPU架构上可减少约30%~40%的纹理带宽开销。

## 核心原理

### 通道独立性与数据类型匹配

RGBA四个通道在硬件层面完全并行读取，每个通道的8-bit精度（256级灰阶）相互不干扰。通道打包的前提是被打包的数据必须是**单通道灰度信息**，即原始数据本身只含一维变化量。例如，水花序列帧中的透明度蒙版、波纹扭曲强度均为标量场，天然适合单通道存储；而RGB颜色信息由于已经占用三个通道，不适合参与打包。

Shader端读取打包数据的典型代码结构如下：

```hlsl
float4 packed = tex2D(_PackedTex, uv);
float opacity    = packed.r;  // R通道：不透明度蒙版
float distortion = packed.g;  // G通道：扭曲强度
float foam       = packed.b;  // B通道：泡沫遮罩
float flowSpeed  = packed.a;  // A通道：流速控制图
```

每个通道通过`.r .g .b .a`分量单独提取，不产生任何额外计算开销。

### 精度损失与数据范围映射

8-bit精度意味着每个通道只有256个离散级别，对于需要高精度的数据（如法线向量的分量）存在精度不足的问题。通道打包前需要对数据进行**值域重映射**：若原始数据范围为`[-1, 1]`（如扭曲方向），必须通过公式 `stored = (value + 1.0) / 2.0` 压缩到`[0, 1]`后再存入贴图，Shader读取时执行逆变换 `value = stored * 2.0 - 1.0`。

对于序列帧特效，0~255的精度通常足够处理透明度、强度等感知类数据，但流速控制图若需要表达细微的速度梯度变化，则建议将数据集中在`[64, 192]`区间以避免端点精度损失。

### 序列帧UV坐标的通道复用策略

在序列帧特效的通道打包中，存在一个特殊场景：多套序列帧动画共享同一套UV翻转节奏。以水花特效为例，如果R通道存储主体蒙版序列（8×8 = 64帧），G通道存储泡沫叠加序列（也是8×8），两者使用完全相同的FlipBook UV，则只需在Shader中计算一次翻页UV，再分别对两个通道采样，从而避免重复计算翻书逻辑的算术开销。

若两套序列帧帧数不同（如R通道32帧、A通道16帧），则需要分别维护两套独立的`frameIndex`变量，通道打包的节省效益会相应降低，此时需要权衡是否值得打包。

## 实际应用

### 水花特效的四通道分配方案

在一套标准的水花溅射特效中，常见的通道分配如下：

- **R通道**：水花主体透明度蒙版序列帧（控制水花整体出现与消散）
- **G通道**：水面法线扭曲强度图（驱动折射偏移量，值越大扭曲越强）
- **B通道**：泡沫与气泡遮罩（叠加白色泡沫细节）
- **A通道**：流速渐变图（控制水花边缘向外扩散的速度权重）

这四种数据分别制作时各需一张512×512的灰度序列帧，打包后仅需一张512×512的RGBA贴图，内存占用从约4MB降至约1MB（未压缩状态下）。

### 爆炸烟雾特效的通道复用

在爆炸特效中，R通道存储烟雾的Emission自发光蒙版序列，A通道存储烟雾的透明度序列，B通道存储用于颜色渐变的热度梯度图（从橙红到灰黑）。Shader中通过将B通道值映射到颜色渐变曲线（Gradient），实现烟雾颜色随时间从亮橙变暗灰，而无需单独的颜色序列帧贴图。

### 工具链中的打包流程

在实际制作中，使用Photoshop的"通道"面板或Substance Designer的"RGBA Merge"节点可手动或程序化地将四张灰度PNG分配到RGBA各通道并导出。导出格式选择TGA（32-bit）或PNG（32-bit）可保留完整的四通道数据；若使用DXT5/BC3压缩格式，需注意A通道单独压缩块可获得较好精度，而RGB通道共享压缩块会对B通道的细节造成约10%的精度损失。

## 常见误区

### 误区一：将RGB彩色图打包进通道

通道打包只适用于灰度（单通道）数据。有美术会尝试将一张彩色贴图的R通道直接复用作打包数据，但如果该彩色贴图的R通道本身带有颜色信息的混叠，读出的灰度值并不代表任何物理意义的强度，会导致特效表现错乱。正确做法是先将彩色图转换为所需的灰度语义图（如亮度图、法线强度图），再进行打包。

### 误区二：打包后忽略sRGB与Linear色彩空间冲突

引擎默认将贴图作为sRGB颜色空间处理（用于颜色贴图的Gamma校正），但通道打包的数据通道存储的是线性数值（如强度0.5就是0.5，不应被Gamma转换为0.214）。若打包贴图的Import Settings中勾选了"sRGB"选项，Shader采样到的数值会经过Gamma 2.2的逆变换，导致中间值偏暗。必须将打包贴图设置为**Linear**（取消勾选sRGB），否则所有通道数据将产生系统性偏差。

### 误区三：认为通道打包一定优于分贴图方案

当四个通道的贴图尺寸或帧数差异极大时，打包反而造成浪费。例如主体蒙版需要1024×1024分辨率，而泡沫遮罩仅需128×128，若强行打包到同一张1024×1024贴图中，泡沫通道占用了大量无效内存。合理的做法是将分辨率相近、语义相关的灰度图打包在一起，分辨率差距超过4倍时应考虑独立存储。

## 知识关联

通道打包建立在对**水花序列帧**基础原理的熟练掌握之上——只有理解了序列帧如何通过UV翻书驱动动画，才能准确判断哪些数据适合共享同一UV坐标系进行打包，哪些数据需要独立的UV动画驱动。水花序列帧中的透明度蒙版是最典型的R通道打包候选数据，是进入通道打包学习的直接入口。

掌握通道打包后，自然延伸至**法线序列帧**的学习。法线序列帧将切线空间法线的XY分量分别存入RG通道（Z分量可由`sqrt(1 - R² - G²)`重建），本质上是通道打包思想在法线数据上的特化应用；但由于法线数据的精度要求更高，法线序列帧还需要额外处理BC5/ATI2N等专用压缩格式与16-bit精度提升方案，是通道打包知识的直接进阶。
