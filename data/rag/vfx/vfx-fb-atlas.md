# 图集制作

## 概述

图集制作（Sprite Sheet / Texture Atlas）是将多帧动画图像拼合到单张纹理图片中的技术流程，专门用于序列帧特效的资源准备阶段。其核心任务是将原本分散的帧图像按照行列网格或紧凑排布方式合并，使GPU在渲染时只需绑定一张纹理，通过UV坐标偏移来访问不同帧的内容。

这一技术最早由街机游戏行业在1980年代确立，当时硬件限制了纹理切换次数，开发者被迫将角色动画拼入单张位图。现代引擎（如Unity、Unreal Engine、Cocos2d-x）继承并标准化了这种做法，Unity的Sprite Atlas系统和TexturePacker工具均以此为基础构建。

图集制作的质量直接影响GPU的纹理采样效率、内存占用和序列帧播放的精度。若UV计算出现亚像素偏差，会导致帧边缘出现"出血"（bleeding）现象，使相邻帧的像素渗入当前帧，破坏特效视觉效果。

## 核心原理

### 网格排列与帧编号规则

最常见的图集排列方式是**均匀网格**（Uniform Grid），所有帧的宽度和高度完全一致，按从左到右、从上到下的顺序编号（第0帧位于左上角）。若图集共有 N 帧，排布为 C 列 × R 行，则第 i 帧的UV起始坐标计算公式为：

```
u = (i % C) / C
v = floor(i / C) / R
```

其中 `%` 为取余运算，`floor` 为向下取整。每帧的UV尺寸为 `(1/C, 1/R)`。这个公式是播放控制模块实现帧切换的直接依据，因此图集的列数C必须在导出时记录到元数据中。

### 图集尺寸的二次幂限制

GPU硬件对纹理尺寸有**POT（Power of Two）**要求，即宽高必须是2的整数次幂：128、256、512、1024、2048、4096像素。若提交非POT尺寸的图集，OpenGL ES 2.0及以下版本将无法使用纹理重复和Mipmap功能，部分驱动会自动填充空白像素至最近的POT尺寸，导致实际内存占用反而增加。因此在规划图集时，应提前计算所有帧的总像素量，选择能容纳所有帧且最接近的POT尺寸。例如，一个20帧、每帧128×128像素的爆炸特效，5列4行恰好需要640×512像素，但必须扩展至1024×512才符合POT要求。

### 间距（Padding）与出血区域

每帧之间必须预留2~4像素的间距（Padding），以防止双线性过滤（Bilinear Filtering）采样时跨越帧边界。若两帧直接相邻（Padding = 0），当帧的UV坐标接近边缘时，GPU的双线性插值会将相邻帧的像素混入，产生可见的色彩闪烁。

间距的推荐值根据纹理过滤模式不同而变化：
- **最近邻过滤（Nearest）**：Padding = 1像素即可
- **双线性过滤（Bilinear）**：Padding = 2像素
- **三线性过滤 + Mipmap**：Padding = max(2^(mip层数), 4) 像素，通常设为8像素

### 格式选择与透明通道

序列帧特效通常含有大量透明区域，因此图集格式的透明通道支持至关重要：

- **PNG**：无损压缩，支持完整Alpha通道，适合原始资源存储，但文件体积较大
- **ETC2（Android）/ PVRTC（iOS）**：有损硬件压缩格式，ETC2支持独立Alpha通道，内存占用约为PNG的1/4，但ETC1不支持Alpha，需要将RGB和Alpha分离为两张图集配合使用
- **ASTC（现代移动端）**：自适应可伸缩纹理压缩，支持4x4至12x12多种压缩块尺寸，质量和体积可按需平衡，是目前移动端特效图集的推荐格式

## 实际应用

**粒子爆炸特效图集**：一个典型的爆炸效果由32帧图像组成，每帧256×256像素，采用8列4行排列。图集总尺寸为2048×1024像素，格式选择ASTC 6x6压缩（每帧约压缩至原始大小的11%），最终图集文件约为350KB。制作时在每帧之间添加2像素Padding，实际每帧占用区域为260×260像素，外加Padding后图集需要扩展至8×260+余量，最终仍取2048×1024。

**TexturePacker工具配置**：使用TexturePacker（版本6.x）时，关键参数设置为：Max Size = 2048×2048，Algorithm = Basic（均匀网格保证列数一致），Border Padding = 2，Shape Padding = 2，勾选"Force Squared"确保POT输出，导出格式选择JSON Array以记录每帧的精确UV坐标供引擎读取。

**UI特效图集分组**：同一技能的多个特效（施法光效、命中光效、消散光效）可以合并到同一张2048×2048图集中，减少渲染批次（Draw Call）。Unity的Sprite Atlas支持将这些帧归组打包，在运行时自动管理纹理绑定。

## 常见误区

**误区一：帧数越多越好，尽量塞满图集**

在单张2048×2048图集中堆入100帧以上的内容，会导致每帧的有效分辨率过低。例如将100帧均匀排入10×10网格，每帧仅剩204×204像素，接近边界的间距侵蚀后有效内容区域不足200像素，在近距离特效中会出现明显模糊。应根据特效的屏幕占用大小决定单帧分辨率，再反推图集尺寸。

**误区二：Padding可以省略以节省空间**

省略Padding在Nearest过滤模式下可能暂时不可见，但当特效缩放或在不同分辨率设备上运行时，浮点UV坐标的精度误差会超过单像素，导致出血现象在低端设备上爆发式出现。这是一个"在开发机上正常、在发布设备上花屏"的典型原因。

**误区三：PNG格式等同于最高质量，应作为最终格式**

PNG是无损的中间格式，在GPU中使用时必须解压为原始RGBA数据存入显存，一张2048×2048的PNG图集实际消耗显存为2048×2048×4字节 = 16MB，而ASTC 6x6压缩版本仅需约1.4MB显存。将PNG直接作为运行时纹理格式会大幅增加显存压力，导致移动端频繁触发纹理换出（Texture Swapping）。

## 知识关联

图集制作依赖**纹理采样**的UV坐标系统——正是因为GPU通过归一化UV（0.0到1.0）访问纹理，才使得帧偏移计算（u = i%C / C）成为可能。如果对双线性过滤的采样半径不熟悉，就无法正确理解为何Padding是必须的。

掌握图集制作后，**播放控制**模块的实现变得直接：播放控制器读取图集的列数C、行数R和帧率FPS三个参数，在每个时间步长内递增帧索引并代入UV公式，驱动材质的纹理偏移属性（如Unity的`_MainTex_ST`或Shader中的`offset`参数）。图集规格（尤其是C值和每帧尺寸）是播放控制的输入契约，两者必须严格对应。
