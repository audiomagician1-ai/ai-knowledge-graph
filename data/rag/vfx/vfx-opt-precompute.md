# 预计算策略

## 概述

预计算策略（Precomputation Strategy）是指在游戏或应用程序运行前，将计算量密集的模拟结果提前离线生成并存储为静态数据，运行时直接读取这些缓存数据替代实时计算的优化手法。这一方法的核心价值在于：模拟一次流体爆炸粒子场可能需要30分钟CPU时间，但播放预烘焙结果只需每帧几毫秒的纹理采样开销。这种时间上的"不对等交换"使得原本无法实时运行的高精度特效成为可能。

预计算策略在游戏引擎中的大规模应用始于2000年代中期，随着HDR光照烘焙技术（如Lightmass）的普及而被广泛接受。早期的离线烟雾模拟（如《魔兽世界》2004年发布时的部分体积云效果）就已使用预烘焙翻页动画（Flipbook）技术将64帧流体模拟压缩进一张2048×2048纹理图集。这一技术路径沿用至今，并衍生出向量场烘焙、骨骼缓存动画等多个变体。

在特效优化的成本谱系中，预计算策略属于"以存储换算力"的典型取舍。对于爆炸、水面、布料等物理模拟类特效，实时解算一帧的Navier-Stokes方程可能消耗5ms以上的GPU计算时间，而预烘焙方案可将运行时开销压缩至0.2ms以下，代价是增加数十MB甚至数百MB的资产体积。

## 核心原理

### Flipbook翻页动画烘焙

最基础的预计算形式是将粒子或流体模拟输出为Flipbook纹理。具体做法是将N帧模拟结果排列在一张纹理图集中，常见规格为8×8（64帧）或16×16（256帧）布局。运行时shader通过如下UV偏移公式采样：

```
UV_offset = vec2(frame % cols, floor(frame / cols)) / vec2(cols, rows)
```

其中`cols`和`rows`为Flipbook的列数与行数，`frame`为当前帧索引。这种方法的关键局限是每个Flipbook实例必须共享同一套模拟结果，无法产生差异化表现，因此通常配合随机时间偏移（Random Start Frame）缓解重复感。

### 向量场（Vector Field）烘焙

向量场烘焙将粒子运动的速度信息预计算为3D纹理（通常为32³或64³分辨率的体积纹理），存储每个体素位置对应的运动向量。运行时，粒子在空间中根据自身位置采样向量场并叠加速度，从而模拟风场、爆炸冲击波等复杂力场效果。Unreal Engine 4引入了FVectorFieldVolume组件原生支持此格式，一个64³的向量场以FP16精度存储仅需约3MB，却能驱动数万粒子的自然飘散运动。

### 几何缓存（Alembic Cache）

对于布料撕裂、破碎物理等拓扑结构会发生变化的特效，无法用骨骼动画表达，此时使用Alembic格式将每帧的顶点位置序列化存储。一个3秒钟的破碎动画，若网格顶点数为5000，帧率30fps，则顶点位置数据量约为：5000顶点 × 3轴 × 4字节 × 90帧 ≈ 5.4MB（未压缩）。实际使用时会配合顶点增量压缩（Delta Compression）将体积缩减至原始的30%~50%。

### 预烘焙与实时LOD的协作

预计算资产通常不以单一精度存在，而是按距离生成多级LOD版本。例如，主炮爆炸特效可设计为：近距离（0-10米）使用256帧高精度Flipbook，中距离（10-50米）使用64帧低精度版本，远距离（50米以外）退化为单张颜色渐变贴图。这种分级烘焙策略能使平均内存占用降低约60%，同时不影响视觉焦点区域的表现质量。

## 实际应用

**火焰与爆炸特效**：《战地5》的载具爆炸特效采用预烘焙Flipbook方案，在PS4平台上实现了单帧0.3ms的渲染开销，而等效的实时粒子模拟需要8ms以上。美术人员使用Houdini离线模拟约4小时，输出为128帧2048分辨率的Flipbook纹理图集。

**体积云与烟雾**：移动游戏中的营地篝火烟雾特效若使用实时体积渲染，在中端手机上帧时间会增加12ms。改用预烘焙向量场驱动的Flipbook粒子后，开销降至0.8ms，视觉品质损失在2米以内几乎不可察觉。

**水面破碎泡沫**：开放世界游戏中的海浪破碎泡沫使用预烘焙的Flowmap（流向图）技术，将64帧流体模拟的流速方向存储为RG两通道纹理，运行时通过双帧混合（Dual Frame Blending）实现无缝循环，消除了16帧Flipbook常见的跳帧闪烁问题。

## 常见误区

**误区一：预烘焙一定比实时计算省性能**
这一判断忽略了读取开销与数据传输成本。当Flipbook纹理分辨率极高（如4096×4096）且同屏实例数量超过200个时，纹理采样带宽的消耗可能反而超过对应的实时粒子计算成本。正确的决策应当基于具体平台的显存带宽限制（如PS5带宽为448GB/s，移动端Adreno 650约68GB/s）进行实测对比。

**误区二：预烘焙结果不支持交互响应**
许多开发者认为预计算资产是"死的"，无法响应玩家行为。实际上，向量场可以在运行时叠加额外的动态力场（如爆炸冲击）；Flipbook的播放速率、透明度可由游戏逻辑动态调整；多套预烘焙序列还可通过权重混合产生差异化外观，从而在保留预计算性能优势的同时实现有限度的交互表现。

**误区三：存储体积无关紧要，空间很便宜**
在主机和PC平台，原始存储成本确实较低，但预烘焙资产的体积会影响流媒体加载时间（streaming latency）和内存常驻成本。一套完整的爆炸特效预烘焙资产（含向量场+多级LOD Flipbook）如果未经压缩可达80MB，在下载包体预算严格的手游中是不可接受的。应使用ASTC（移动端）或BC7（PC/主机）等GPU原生压缩格式，通常可将体积压缩至原始的25%~12%。

## 知识关联

**前置概念衔接**：理解光源特效开销后，可以看到预计算策略解决的是同类问题——将高频计算分摊至离线阶段。光源烘焙（Light Baking）正是预计算策略在光照领域的具体实例，两者共享"以磁盘空间换运行时帧时间"的底层逻辑。学习过光源特效开销的读者对"实时vs.预计算"的成本权衡框架已有感性认识，可直接将这一框架迁移到粒子与流体模拟领域。

**后续概念延伸**：预计算策略是构建特效预算系统（Budget System）的重要数据基础。预算系统需要为每种特效类型设置帧时间上限（如爆炸类≤2ms/帧），判断某特效是否适合实时计算或必须采用预烘焙，依赖的正是对预计算资产读取开销与实时模拟开销的精确测量数据。掌握预计算策略的成本模型，是在预算系统中做出正确分级决策的前提。
