# 运行时碎裂

## 概述

运行时碎裂（Runtime Fracturing）是指在游戏或模拟运行过程中，根据实际受力情况实时计算物体破碎方式的技术。与预碎裂（Pre-Fracturing）不同，运行时碎裂不依赖预先烘焙的破碎形态，而是在碰撞或爆炸发生的瞬间，根据撞击点位置、冲击力大小和材质属性动态生成裂缝几何体。这意味着同一个石墙被不同弹道击中，将产生完全不同的碎片形状和飞散轨迹。

该技术在2006年前后随着Voronoi分割算法的GPU化实现而逐渐进入商业引擎。早期代表是Nvidia的PhysX Destruction模块，后来Houdini的实时破碎求解器和Unreal Engine的Chaos Physics系统将其发展为主流工业工具。运行时碎裂的意义在于它能将破碎结果与玩家行为精确绑定——子弹的入射角度、爆炸的超压波形都会反映在最终的碎片图案中，这是任何预碎裂资产库无法复现的物理真实性。

在性能预算极为有限的实时应用中，运行时碎裂面临的核心挑战是：单帧内必须完成几何体切割、新网格拓扑生成、物理代理体绑定和渲染资源上传这一完整流水线，而这些操作在单个高精度网格上可能消耗2毫秒以上，直接威胁60Hz帧率目标。

## 核心原理

### Voronoi实时分割

运行时Voronoi碎裂的基本流程是：在撞击点周围随机或按力学规律播撒种子点集合，然后对目标网格执行半空间求交运算（Half-Space Intersection）。每个碎片的几何形状由到最近种子点距离小于到所有其他种子点距离的体素集合确定，数学上表达为：

> 碎片区域 Rᵢ = { x ∈ Ω | d(x, sᵢ) ≤ d(x, sⱼ), ∀j ≠ i }

其中 Ω 为原始网格体积，sᵢ 为第 i 个种子点坐标，d 为欧氏距离函数。种子数量直接决定碎片粒度：16个种子产生粗碎块效果，256个种子接近混凝土粉碎效果，但计算量以 O(n log n) 增长。

### 表面内部网格生成

原始模型通常只有外表面网格，运行时碎裂必须为每个新生成的碎片实时填充内部截面（Interior Cap Mesh）。这一步骤包含三角剖分（Triangulation）和UV展开两个子操作，其中UV展开必须在单帧内完成，因此实践中普遍采用近似平面投影而非精确参数化，代价是内截面贴图存在一定拉伸误差。Unreal Engine Chaos中使用的Constrained Delaunay Triangulation算法将单个截面的网格生成时间控制在0.3毫秒以内（在标准测试场景：4000顶点多边形网格，i7-9700K处理器）。

### 力场响应与碎片分布

运行时碎裂的种子点分布不是纯随机的，而是根据冲击波传播模型加权生成。撞击点附近区域采用高密度种子以产生细小碎片，远离撞击点的区域种子稀疏以保留大块结构，这符合应力集中（Stress Concentration）的物理规律。具体实现中，种子密度函数通常设计为以撞击点为中心的指数衰减分布：

> ρ(r) = ρ₀ · e^(-λr)

其中 r 为到撞击点的距离，λ 为材质衰减系数（混凝土典型值约0.8，玻璃约2.5，软木约0.3），ρ₀ 为撞击点处最大种子密度。

### 物理代理体的实时绑定

几何碎片生成后，引擎必须立即为每个碎片创建简化的凸包碰撞代理体（Convex Hull Proxy）。运行时碎裂区别于离线流水线之处在于此步骤的精度限制：实时凸包计算通常限定最多32个面以保证求解速度，这意味着不规则形状的碎片碰撞体积存在最多15%的几何偏差。Chaos Physics采用了分级细化策略：碎裂发生后前0.5秒使用粗糙代理体，之后根据该碎片与摄像机的距离决定是否升级为精确碰撞体。

## 实际应用

**第一人称射击游戏中的墙面破碎**：在《战地》系列的Frostbite引擎中，子弹击中混凝土墙时的运行时碎裂被限制在每次事件最多生成32个碎片，且碎片体积低于原始体积0.5%的部分直接被转换为粒子特效而非物理刚体，以避免物理世界中刚体数量爆炸性增长。

**载具碰撞变形**：赛车游戏中车身碰撞的运行时碎裂需要与蒙皮网格（Skinned Mesh）的变形系统协同工作，而非纯刚体切割。此时碎裂算法需要在骨骼权重边界处优先产生断裂，否则蒙皮变形会导致碎片拓扑错误。

**VR场景中的交互式破坏**：VR应用对运行时碎裂的延迟要求尤为苛刻，破碎事件必须在8毫秒内完成整个流水线以避免帧率降至72Hz以下。为此，部分VR引擎预先在后台线程中维护一个候选碎裂网格缓存池，当交互触发时直接从缓存中取用最匹配的几何结果而非完整重算。

## 常见误区

**误区一：种子点越多碎裂效果越好**。许多初学者认为增加Voronoi种子数量必然提升视觉质量。实际上超过128个种子的碎片中，大量碎片体积小于渲染最小阈值，不仅无法被玩家感知，还会同时触发128个以上的物理刚体激活，在单次破碎事件中消耗超过5毫秒的物理更新时间。正确做法是将大量细小碎片替换为GPU粒子系统。

**误区二：运行时碎裂可以完全替代预碎裂**。这两种技术针对的场景不同，而非替代关系。对于需要重复触发、角色可以多次交互的可破坏物体（如整关中玩家反复经过的箱子），运行时碎裂每次产生不同形态的代价是无法利用任何几何缓存，每次破碎事件都是冷启动计算。对于此类高频可破坏物，预碎裂仍然是性能更优的选择。

**误区三：运行时碎裂的内截面贴图可以使用高分辨率材质**。由于内截面UV是近似生成的，即使使用4K贴图也无法消除截面的拉伸伪影，反而增加贴图采样开销。实践中内截面材质通常使用与外表面材质相同的世界空间三面投影（Triplanar Mapping）以掩盖UV不精确性，同时保持合理的纹理密度。

## 知识关联

运行时碎裂的前置知识是预碎裂工作流——理解预碎裂中Voronoi图的离线生成方式和碎片连接图（Connectivity Graph）的构建逻辑，才能理解运行时碎裂为何必须将这些步骤压缩到单帧预算中以及因此做出的每项精度妥协。预碎裂中每个碎片的邻接关系是静态预计算的，而运行时碎裂必须在新几何体生成后即时构建这一关系，这是两者在数据结构上的根本差异。

运行时碎裂是通往Chaos物理系统的直接路径。Unreal Engine 5的Chaos Fracture工具集将运行时碎裂与场场景查询、约束求解器、几何体集合（Geometry Collection）三套系统深度集成，使得运行时碎裂产生的碎片能够参与更复杂的二次破碎链（Secondary Fracture Chain）和结构性坍塌模拟。掌握运行时碎裂的种子分布控制和性能预算管理，是有效使用Chaos Field系统中的外部应力场（External Strain Field）驱动程序化破碎的必要基础。
