# 纹理预算

## 概述

纹理预算（Texture Budget）是指在实时渲染特效系统中，为粒子、烟雾、火焰、闪光等视觉特效所分配的显存（VRAM）上限。这一上限通常由平台显存总量、其他系统（模型、UI、场景）的占用共同决定——在移动端项目中，特效纹理预算往往被限制在整体显存预算的 10%~15% 以内，而主机或PC端可放宽至 20% 左右。

纹理预算的概念随GPU可编程管线的普及而正式进入游戏开发流程。2000年代中期，随着粒子特效大量采用 512×512 乃至 1024×1024 的序列帧纹理，显存溢出导致的卡顿和闪退问题愈发突出，促使技术美术（Technical Artist）岗位开始系统性地管理特效纹理开销。Unity 和 Unreal 引擎的 Memory Profiler 工具将每张纹理的实际显存占用可视化，进一步推动了"预算制"管理模式的落地。

管理纹理预算的核心价值在于：同屏多特效并发时，GPU 显存带宽（Memory Bandwidth）是瓶颈，而不是计算单元。一张未经压缩的 512×512 RGBA 纹理在显存中占用 1 MB，若同屏同时激活 30 个粒子系统且各自引用独立纹理，瞬间显存增量即可超出移动端硬件的承受阈值。

## 核心原理

### 分辨率选择与显存开销计算

特效纹理的显存占用可用以下公式精确计算：

> **显存占用（字节）= 宽 × 高 × 每像素字节数 × Mipmap系数**

其中，完整 Mipmap 链的系数约为 **1.333**（即原图的 4/3）。对于一张 256×256 的 RGBA8 纹理：256 × 256 × 4 × 1.333 ≈ **349 KB**。

特效纹理不需要像角色贴图那样保留精细的高频细节，因此分辨率降级收益极高。将 512×512 降为 256×256，显存占用直接缩减至原来的 **1/4**，而画面质量损失在运动模糊和粒子透明叠加的遮蔽下往往肉眼难以察觉。通用经验：粒子纹理超过 256×256 需要有充分的艺术理由，序列帧图集（Sprite Sheet）超过 2048×2048 则几乎一定是过度开销。

### 压缩格式选型

不同平台对压缩格式支持不同，错误选型会导致 GPU 解压开销抵消掉显存节省的优势：

- **移动端（Android）**：优先使用 **ETC2**（支持 Alpha 通道），RGBA 通道压缩比约 4:1，即 256×256 RGBA 从 256 KB 降至约 64 KB。
- **移动端（iOS）**：优先使用 **ASTC 6×6** 或 **ASTC 8×8**，ASTC 8×8 压缩比约 16:1，是目前移动端特效纹理压缩收益最高的格式。
- **PC/主机端**：使用 **BC3（DXT5）** 处理含 Alpha 的特效纹理，或 **BC7** 获得更高质量，压缩比均为 4:1。

特效纹理的 Alpha 通道存储方式值得特别关注：将 Alpha 通道打包进 RGB 通道的某一分量（如 R 通道存 Alpha，再用 **BC4** 单通道压缩），可在只需软粒子遮罩时将显存压缩至 BC3 方案的一半。

### 纹理共享与图集策略

纹理共享是在预算有限条件下最大化特效种类的关键手段。同一个特效系统中的火焰核心、火焰边缘、烟雾可以合并为一张 **1024×512 的图集**，三种素材各占 512×256，显存占用仅为单独存储三张 256×256 纹理的 **2/3**（因为 Mipmap 合并降低了冗余）。

序列帧特效（Flipbook）的图集行列数选择直接影响显存效率：一个 8×8=64 帧的序列帧存储在 2048×2048 图集中，每帧分辨率为 256×256，若帧率为 24fps 则动画时长约 2.67 秒。若将同一图集改为 4×4=16 帧、1024×1024，显存从约 5.3 MB（RGBA8 未压缩）降至约 1.3 MB，适合对显存更敏感的移动平台。

## 实际应用

**项目案例——移动端 MOBA 游戏技能特效**：某技能特效包含 12 张独立粒子纹理，总显存占用 8.4 MB（RGBA8 未压缩）。优化流程：① 将同色系烟雾纹理合并图集，减少至 7 张；② 全部转为 ASTC 6×6，显存降至约 0.9 MB；③ 对不需要逐像素 Alpha 的发光纹理改用 BC4+着色器重建 Alpha，最终总显存降至 **0.6 MB**，优化幅度超过 92%。

**Unity 中的纹理预算监控**：在 Unity Memory Profiler（2021.3 LTS 以后版本提供详细的 GPU 对象视图）中，可筛选 `Texture2D` 类型，按显存大小排序，找出超出单张 1 MB 阈值的特效纹理逐一审查。此外，Unity 的 `TextureImporter` API 支持在 Editor 脚本中批量检查 `maxTextureSize`、`textureCompression` 设置，实现自动化预算合规审查。

**Unreal Engine 中的流送控制**：UE5 的 Nanite 不管理 2D 纹理流送，Niagara 粒子纹理仍依赖传统 Texture Streaming Pool。将特效纹理的 `Texture Group` 设置为 `TEXTUREGROUP_Effects`，并在 `Engine.ini` 中设置 `r.Streaming.PoolSize` 上限，可防止特效纹理挤占角色和场景的流送预算。

## 常见误区

**误区一：特效纹理不需要 Mipmap**

部分开发者认为粒子纹理距相机较近时不需要 Mipmap，因此关闭以节省 33% 显存。然而，当粒子系统包含从近到远的生命周期运动时，没有 Mipmap 的纹理在远距离采样时会产生严重的闪烁噪点（Aliasing），GPU 反而需要更多采样次数来抗锯齿。正确做法是保留 Mipmap，但将 `Mipmap Bias` 设为 +1 或 +2，让 GPU 优先使用更小的 Mip 级别，同时在序列帧纹理上启用 Mipmap 时注意相邻帧不要因低级 Mip 产生颜色串扰。

**误区二：压缩格式统一用 DXT5/BC3 就够了**

在移动端用 DXT5 会导致硬件无法原生解压，GPU 会在着色器内进行软件解压，反而增加计算开销。正确做法是按平台分别设置：iOS 用 ASTC，Android 用 ETC2 或 ASTC（需检测硬件支持，2017 年后主流 Android 设备均支持 ASTC）。一套特效资产至少需要维护 PC 和移动两套压缩设置。

**误区三：纹理预算只需关注首次加载时的峰值**

特效纹理的显存压力往往不在加载阶段，而在多特效**并发激活**的瞬间。例如，群战场景中 20 个玩家同时释放技能，即使每个技能纹理仅 200 KB，20 个技能各含 5 张纹理，瞬间新增显存占用就达到 20 MB，超出移动端特效预算上限。解决方案是在设计阶段限制同一帧内可激活的不同纹理集合数量，或通过纹理引用计数（Reference Counting）系统在粒子系统失活时及时释放显存。

## 知识关联

**前置概念——特效 LOD**：特效 LOD 系统决定了在不同距离下激活哪个粒子系统层级，直接影响哪些纹理会被实际加载进显存。纹理预算管理需要与 LOD 距离阈值配合——LOD2 以上的远距离特效应强制使用低分辨率或高压缩比纹理，避免 LOD 切换时产生不必要的纹理流送。

**后续概念——DrawCall 优化**：纹理图集策略同时服务于纹理预算和 DrawCall 减少两个目标。将多个粒子系统引用的不同纹理合并进同一图集后，粒子渲染可以在一次 DrawCall 中提交，GPU 无需频繁切换纹理状态（State Change）。因此，图集的分区规划需要在预算优化和批次合并两个维度上同步考量，图集内纹理的 UV 偏移量由粒子系统的 Sub UV 参数控制，两者共同决定最终的渲染效率。
