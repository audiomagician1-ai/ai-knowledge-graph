# 图集共享

## 概述

图集共享（Atlas Sharing）是一种将多个特效所需的纹理图片合并到同一张纹理图集（Texture Atlas）中，使不同粒子特效共同引用该图集的优化技术。通过这种方式，GPU 在渲染多个特效时无需频繁切换纹理采样器绑定状态，从而减少 Draw Call 中的纹理切换开销，同时降低显存中重复纹理数据的占用量。

该技术的起源可以追溯到早期 2D 游戏的精灵图（Sprite Sheet）优化实践，后被引入三维粒子系统。在现代游戏引擎（如 Unity 的 Sprite Atlas 系统和 Unreal Engine 的 TextureGroup 机制）中，图集共享已成为特效美术与技术美术协作的标准流程之一。对于移动端项目而言，由于 GPU 的纹理采样单元（TMU）数量有限，通常只有 8 至 16 个，过多的独立纹理绑定会直接导致性能瓶颈。

图集共享之所以在特效优化中备受重视，是因为粒子特效的纹理通常尺寸小（如 64×64 或 128×128 像素）但数量众多。将十几张小纹理散布在独立文件中，每张均需占用一个采样器槽位；而将它们打包进一张 1024×1024 的图集后，所有特效共享同一个采样器绑定，GPU 状态切换次数可降低 80% 以上。

## 核心原理

### 纹理图集的 UV 重映射机制

图集共享的核心数学基础是 UV 坐标的线性重映射。假设原始纹理的 UV 坐标为 `(u, v)`，该子纹理在图集中的起始偏移为 `(offset_x, offset_y)`，占整张图集的比例宽高为 `(scale_x, scale_y)`，则重映射后的图集 UV 为：

```
u_atlas = offset_x + u × scale_x
v_atlas = offset_y + v × scale_y
```

例如，一张 128×128 的火焰纹理放置在 1024×1024 图集的左上角，则 `scale_x = scale_y = 128/1024 = 0.125`，原始 UV `(1.0, 1.0)` 映射到图集中的 `(0.125, 0.125)`。粒子系统的 Shader 需要预先接收每个子图的偏移与缩放参数（通常通过 `Vector4` 类型的材质属性传递），才能在渲染时正确采样。

### 采样器槽位与 Draw Call 合并

移动端 GPU（如 Mali-G78 或 Adreno 660）在执行渲染管线时，纹理采样器槽位是有限的共享资源。当两个特效使用不同纹理时，它们无法被合并为同一个 Draw Call，即使它们使用相同的 Shader 和混合模式。图集共享的关键在于：只要多个粒子系统引用同一张图集纹理，渲染系统就可以将它们合并到同一批次（Batch）中提交，将原本 N 次 Draw Call 压缩为 1 次。实测数据表明，在包含 5 个共享同一图集的粒子特效的场景中，Draw Call 数量从 12 次降至 3 次（含其他渲染对象），帧时间减少约 0.8ms（基于高通 Snapdragon 865 设备测试）。

### 图集分辨率与子图布局策略

图集的总分辨率必须为 2 的幂次（Power of Two），常用规格为 512×512、1024×1024 或 2048×2048。子图的排布推荐使用行列均匀分割（Grid Layout），便于计算偏移参数；对于尺寸差异较大的子纹理，可使用打包算法（如 MaxRects 算法）提高图集填充率。图集中的子纹理之间需要保留 2 至 4 像素的"出血边（Bleeding Margin）"，防止在 Mipmap 降级时相邻子图的像素渗入彼此边缘，产生纹理污染（Texture Bleeding）的视觉瑕疵。

## 实际应用

**场景一：UI 特效与战斗特效混合图集**
在移动端 ARPG 项目中，界面技能特效（光晕、闪光）与角色攻击粒子特效往往需要相同的基础图案（如圆形光斑、星形粒子）。将这两类特效的纹理合并进同一张 1024×1024 图集后，UI 层和世界层的特效渲染均可引用同一纹理资源，显存中该资源仅加载一份，节省约 0.5MB 至 1MB（取决于压缩格式）。

**场景二：序列帧动画与静态粒子共存图集**
火焰序列帧动画（16 帧，每帧 128×128）与烟雾静态粒子（256×256）可以共同打包进一张 1024×1024 图集。序列帧在图集中占据上半区域（4×4 排列），烟雾占据下半区域。粒子系统通过逐帧更新 UV 偏移参数实现动画播放，同时烟雾粒子持续采样其对应区域，二者共享一个纹理采样器绑定。

**场景三：多特效预制件打包验证**
在 Unity 中，可以在 Particle System 的 Renderer 模块下指定 Material，并通过 Sprite Atlas 将多个精灵子图注册到同一图集。使用 Frame Debugger 可以验证多个特效预制件是否真正共享了同一个纹理 ID（对应同一 Draw Call Batch），若 Frame Debugger 中显示相邻两个粒子渲染指令使用了同一 Texture Handle，则图集共享生效。

## 常见误区

**误区一：图集越大越好**
部分开发者认为使用 2048×2048 甚至 4096×4096 图集能打包更多纹理，从而实现更大范围的共享。但移动端 GPU 对单张纹理的最大支持尺寸通常为 4096×4096，且加载一张 4096×4096 的未压缩 RGBA 纹理需要占用 64MB 显存。更重要的是，过大的图集会导致图集中大量区域空置（填充率低），或强制将本不需要同时存在于内存中的纹理一起加载。合理的做法是按特效使用频率分组，将同一场景或同一技能组的特效纹理打包进同一图集，通常 1024×1024 是移动端特效图集的推荐上限。

**误区二：图集共享必然导致 UV 动画失效**
使用 UV 滚动（UV Scrolling）或序列帧动画的粒子特效在打包进图集后，若 Shader 直接对 UV 执行 `frac()` 取整循环操作，会导致采样越界到相邻子图区域，出现画面错误。解决方案是在 Shader 中先对局部 UV（范围 0 至 1）执行 `frac()` 运算，再应用图集的偏移与缩放映射，而非对图集 UV 直接取余。这是图集共享在动画特效中最常见的技术陷阱。

**误区三：不同混合模式的特效可以共享同一图集**
图集共享能合并 Draw Call 的前提是参与合批的粒子使用相同的渲染状态，包括相同的混合模式（Blend Mode）。将使用 Additive（加法混合）的光效纹理与使用 Alpha Blend（透明混合）的烟雾纹理打包进同一图集，并不能使这两者合并 Draw Call，因为它们的 GPU 混合状态不同，渲染队列必须分开提交。图集共享只解决纹理采样器槽位与显存的问题，不能绕过混合模式差异导致的批次分割。

## 知识关联

图集共享建立在**移动端优化**的基础上——正是因为移动端 GPU 的采样器槽位限制和显存带宽敏感性，才使图集共享成为必要的手段而非可选的锦上添花。移动端优化章节中涉及的纹理压缩格式（ETC2、ASTC）同样适用于图集纹理本身，二者协同使用效果最佳：图集减少采样器绑定次数，压缩格式减少显存带宽消耗。

在掌握图集共享之后，下一个需要学习的概念是**模拟空间（Simulation Space）**。模拟空间决定粒子特效的坐标计算是在世界空间还是局部空间执行，这直接影响粒子系统能否参与动态批处理（Dynamic Batching）。即使多个粒子系统已经共享了同一图集，若它们分别使用世界空间模拟，引擎仍可能无法将其合并为同一 Draw Call，因此图集共享与模拟空间设置需要配合调整才能发挥最大的批处理收益。
