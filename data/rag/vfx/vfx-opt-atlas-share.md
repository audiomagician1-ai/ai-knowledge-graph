# 图集共享

## 概述

图集共享（Atlas Sharing）是指多个特效资产将各自的纹理贴图合并到同一张纹理图集（Texture Atlas）中，使这些特效在渲染时共用同一个纹理采样器绑定，从而降低 GPU 的采样器占用数量和显存中的重复纹理加载。与为每个特效单独分配独立纹理相比，图集共享可在多特效同屏场景下显著减少 Draw Call 中的纹理状态切换开销。

这一优化策略源于早期移动游戏开发对显存带宽的极度敏感需求。2010年代初期，iOS 设备的 OpenGL ES 2.0 规范规定每个片元着色器最多绑定 8 个纹理采样器，促使开发者将多张小纹理打包进单张大图集，以突破采样器数量限制。如今这一技术在 Unity、Unreal 等引擎的特效系统中已成为标准优化手段，尤其在移动端特效密集型场景（如技能释放、群体粒子）中仍然不可或缺。

图集共享的核心价值在于：将原本需要多次纹理绑定的渲染批次合并为单次绑定，配合粒子系统的批次合并（Batching），最终实现多个特效在同一 Draw Call 内完成绘制。

## 核心原理

### 纹理采样器绑定与图集布局

GPU 渲染管线中，每次切换纹理绑定都会导致渲染状态改变，产生额外的 CPU 指令开销。图集共享的做法是将多张特效子纹理（如火焰帧序列、烟雾贴图、闪光贴片）拼合进一张分辨率较大的图集，例如将 8 张 128×128 的特效纹理合并为一张 512×512 的图集，总像素数不变但采样器绑定次数从 8 次降为 1 次。

图集中每张子纹理的位置通过 UV 偏移（UV Offset）和 UV 缩放（UV Scale）参数访问，计算公式为：

```
最终UV = 子图UV × uvScale + uvOffset
```

其中 `uvScale = (子图宽度/图集宽度, 子图高度/图集高度)`，`uvOffset = (子图左边界/图集宽度, 子图下边界/图集高度)`。特效材质的着色器需支持这两个参数的动态传入。

### 渲染批次合并的前提条件

图集共享能够触发 Draw Call 合并，需同时满足三个条件：所有特效使用**同一材质实例**、所有特效的粒子几何体类型相同（如均为 Billboard），以及渲染队列相同（如均为 Transparent 队列中的相同层级）。若两个特效使用同一图集但材质实例不同，则无法合批，图集共享的性能收益会大幅缩水。Unity 的粒子系统在使用 GPU Instancing 或 Renderer Batching 时，会自动检测上述条件并决定是否合并批次。

### 图集尺寸与像素填充率权衡

图集的最大尺寸受移动端 GPU 限制，主流 Android 设备支持最高 4096×4096 的纹理，但实际推荐不超过 2048×2048，因为超大图集在纹理采样时会增加缓存未命中（Cache Miss）的概率。子图之间必须保留 2 至 4 像素的出血区（Bleed/Padding），防止 Mip Map 采样时相邻子图像素互相渗透，产生可见的颜色溢出边缘瑕疵。图集内的子图排列应使用矩形装箱算法（Rectangle Bin Packing）最大化填充率，通常优质工具链（如 TexturePacker）可将填充率控制在 90% 以上。

## 实际应用

**技能特效场景**：一款 MMORPG 的职业技能系统含 12 种普攻特效，每种特效原本使用独立的 256×256 火花纹理，共占用 12 个采样器绑定。将这 12 张纹理合并至一张 1024×1024 图集后（填充率约 75%），采样器绑定降为 1 次，多个技能特效同时触发时可合并为单次 Draw Call，在 Samsung Galaxy S10 上的测试中帧耗时从 4.2ms 降低至 1.8ms。

**UI特效与世界特效分图集**：实际工程中应将 UI 层特效（渲染队列 Overlay）与世界空间特效（渲染队列 Transparent）分别维护独立图集，两者混用同一图集会因渲染队列不同而无法合批，反而造成图集内存浪费。

**帧序列动画共享**：粒子系统中的帧序列动画（Flipbook）本身已是微型图集——将多帧动画排列在单张纹理上。图集共享的进一步做法是将多套 Flipbook 动画放入同一更大图集，并通过不同的 uvOffset 让不同粒子播放不同动画，整体仍在同一 Draw Call 内完成。

## 常见误区

**误区一：图集越大，性能越好**  
有开发者认为将所有特效纹理塞入一张 4096×4096 图集可获得最优性能。实际上，超大图集在生成 Mip Map 时每级内存占用成倍增长（4096×4096 全 Mip 链约占 22MB，而两张 2048×2048 图集仅约 11MB × 2 = 22MB 但缓存局部性更好）。更关键的问题是，超大图集中许多子图在当前帧根本未被使用，但整张图集仍需常驻显存，造成不必要的内存压力。

**误区二：共享图集等同于共享材质**  
图集共享是纹理层面的合并，材质共享是材质实例层面的合并，两者相互独立。即使两个特效引用同一张图集纹理，若它们使用了不同的材质属性（如不同的发光颜色参数），Unity 会为每个实例创建独立的 Material Property Block，Draw Call 仍可能无法合并。真正实现合批需要将可变参数通过顶点色或自定义顶点数据传入，而非通过材质属性覆盖。

**误区三：图集共享对所有特效类型均有效**  
使用了 Trail Renderer（拖尾渲染器）或 Line Renderer 的特效，由于其几何体动态生成方式与 Billboard 粒子不兼容，即便共享同一图集也无法与粒子系统合批。图集共享的合批收益主要集中在标准 Billboard 粒子和 Mesh 粒子两种类型上。

## 知识关联

图集共享建立在移动端优化的纹理压缩基础之上——图集本身应使用 ETC2（Android）或 ASTC（iOS 及新型 Android）格式压缩，若图集未经压缩，合并采样器的收益可能被过高的显存带宽消耗所抵消。实践中需先确认图集的压缩格式兼容性，再评估图集共享的实际收益。

图集共享完成后，下一步的优化方向是**模拟空间**的选择——粒子的模拟空间（世界空间 vs 本地空间）直接决定粒子的变换矩阵是否可以合批。若特效使用世界空间模拟，不同发射器位置不同导致变换矩阵各异，即便已完成图集共享，Draw Call 合并也可能受阻。因此图集共享与模拟空间配置需协同考虑，才能实现完整的特效渲染批次优化链路。
