# 序列帧概述

## 概述

序列帧（Frame Sequence / Flipbook Animation）是将连续动画预先离散化为若干静态图像帧，按照时间顺序排列并以固定帧率逐帧切换播放，从而利用人眼的视觉暂留效应（Persistence of Vision）还原运动感知的一种实时特效技术。在游戏引擎和实时渲染管线中，序列帧文件通常以PNG、EXR或WebP格式存储，每帧图像代表动画在某一离散时刻的完整画面快照，播放时引擎依次切换纹理或UV偏移，以24fps、30fps或60fps等标准速率呈现视觉连续感。

序列帧技术的工程应用可追溯至1960至1970年代的赛璐珞动画数字化存档，而其在实时游戏特效领域的系统化使用，最早见于街机时代——Midway公司于1978年发布的《Space Invaders》中，外星人行走动画便由2帧精灵图交替显示构成，这一机制奠定了后续三十年2D游戏角色动画的基本范式。进入实时3D特效时代，序列帧被大量应用于烟雾、火焰、爆炸、水花、法术光效等物理模拟效果的轻量化呈现。此类效果若采用实时粒子模拟，仅一次中规模烟雾计算便可能占用10ms以上的CPU时间；而以等价视觉品质的序列帧替代，整个播放过程仅需一次纹理采样，耗时通常低于0.1ms（Real-Time Rendering, Akenine-Möller et al., 2018）。

序列帧在移动端特效管线中占据不可替代的地位，根本原因在于它以可控的存储成本换取了运行时几乎为零的计算开销，同时完整保留了美术师手工绘制或高精度离线渲染所具备的视觉表现力。这一特性使其成为连接高质量离线艺术创作与实时硬件约束之间的关键桥梁。

## 核心原理

### 视觉暂留与帧率感知阈值

人眼对静止画面的视觉暂留时间约为1/15秒，这意味着当序列帧播放帧率高于15fps时，大脑即可将离散图像感知为连续运动。电影工业标准为24fps，游戏特效通常采用24fps至30fps，而需要表现高速冲击感的爆炸或闪光效果有时会使用12fps甚至8fps以制造顿挫感。帧率并非越高越好——一段烟雾飘散动画在12fps下反而比30fps更具体积感，因为低帧率与烟雾的惰性视觉特征相契合。

序列帧播放的数学本质是按固定时间间隔切换帧索引。设动画总帧数为 $N$，目标播放帧率为 $f_{play}$（fps），则每帧停留时间为：

$$\Delta t = \frac{1}{f_{play}}$$

引擎每次更新时累加经过时间 $t_{elapsed}$，当 $t_{elapsed} \geq \Delta t$ 时，当前帧索引 $i$ 自增1，同时将 $t_{elapsed}$ 减去 $\Delta t$（而非归零，以保持亚帧精度）：

$$i_{current} = \left\lfloor \frac{t_{total} \cdot f_{play}}{1} \right\rfloor \mod N$$

例如，以24fps播放一段32帧爆炸动画，总时长为 $32 \div 24 \approx 1.333$ 秒；若将播放速率设为0.5倍（即慢动作），等效帧率降至12fps，同一段动画将历时2.667秒播完。

### 图像存储方式：独立帧文件与图集打包

序列帧的图像数据有两种主流存储形式，其性能差异在移动端极为显著。

**独立帧模式**将每帧存为独立文件，命名约定通常为`vfx_explosion_00.png`至`vfx_explosion_31.png`。这种方式便于美术单独修改某一帧，但在运行时会产生大量Draw Call——30帧独立纹理意味着每次播放最多触发30次纹理状态切换，在移动端GPU上每次纹理绑定的驱动开销约为2至5μs，叠加后对帧预算的侵占不容忽视。

**图集模式**（Texture Atlas / Sprite Sheet）将所有帧紧密排列于一张大纹理中，播放时通过实时修改UV坐标偏移选取对应帧区域。无论动画包含多少帧，整个序列的全程播放仅需1次Draw Call。Unity的Particle System、Unreal Engine的Flipbook节点以及Cocos Creator的SkeletonAnimation组件均原生支持此模式。

### UV坐标偏移的着色器实现

在图集模式下，设图集排布为 $C$ 列、$R$ 行（共 $C \times R$ 帧），则第 $i$ 帧（从0开始计数）在归一化UV空间中的偏移量为：

$$u_{offset} = \frac{i \bmod C}{C}, \quad v_{offset} = \frac{\lfloor i / C \rfloor}{R}$$

在GLSL着色器中，对应采样代码如下：

```glsl
// 图集参数
uniform float u_totalFrames;  // 总帧数 N
uniform float u_cols;         // 列数 C
uniform float u_rows;         // 行数 R
uniform float u_currentFrame; // 当前帧索引 i

void main() {
    float col = mod(u_currentFrame, u_cols);
    float row = floor(u_currentFrame / u_cols);
    vec2 frameSize = vec2(1.0 / u_cols, 1.0 / u_rows);
    vec2 atlasUV = v_uv * frameSize + vec2(col, row) * frameSize;
    gl_FragColor = texture2D(u_mainTex, atlasUV);
}
```

以8列×4行共32帧的爆炸图集为例：播放第13帧时，$col = 13 \bmod 8 = 5$，$row = \lfloor 13/8 \rfloor = 1$，UV偏移为 $(5/8,\ 1/4) = (0.625,\ 0.25)$，Shader将在图集第2行第6列的区域采样，完全无需CPU重新提交顶点数据。

### 适用场景的工程判断准则

序列帧并非万能方案，其适用性由以下三个维度决定：

**视角依赖性**：序列帧本质是二维画面的时间序列，无法响应三维视角变化。固定视角的UI技能特效、全屏闪光、地面贴花爆炸等场景完全适用；而需要从任意角度观察的体积烟雾或粒子群，则必须结合Billboard朝向或真正的3D粒子系统。

**预算比较**：以一段16帧、256×256像素的火焰动画为例，图集纹理占用内存约为 $16 \times 256 \times 256 \times 4 \text{ bytes} = 4\text{ MB}$（未压缩RGBA），使用ETC2压缩后约降至1MB。对比实时流体模拟所需的向量场计算（通常需要独立计算着色器Pass），序列帧以1MB存储成本实现了可忽略不计的运行时开销，在移动端这一取舍极具价值。

**视觉控制精度**：手绘逐帧爆炸或经Houdini离线模拟渲染的烟雾序列帧，其视觉质量是程序化粒子系统无法复现的。当美术向导确定了特定的笔触风格或流体细节层次时，序列帧是唯一能够完整还原创作意图的实时方案。

## 关键参数与制作规范

### 分辨率与帧数的权衡

序列帧图集的纹理尺寸必须为2的幂次方（Power of Two），以确保在所有GPU上的兼容性与Mipmap生成的正确性。常见规格如下：

| 图集尺寸 | 每帧分辨率（8×8排布） | 适用场景 |
|----------|----------------------|----------|
| 2048×2048 | 256×256 | 近景高精度爆炸、BOSS技能 |
| 1024×1024 | 128×128 | 中景通用特效 |
| 512×512 | 64×64 | 远景/大量同屏实例 |

帧数方面，24帧是覆盖1秒24fps循环动画的最小完整单元；16帧（4×4排布）则是移动端性价比最高的配置，在视觉流畅度与纹理占用之间取得平衡（Valve, VFX Production Guidelines, 2019）。

### 预乘Alpha与混合模式

序列帧纹理在存储时应使用**预乘Alpha**（Premultiplied Alpha）格式，即将RGB通道值预先乘以Alpha值：$R' = R \times A$，$G' = G \times A$，$B' = B \times A$。这一处理消除了线性插值时半透明边缘出现的"黑边"伪影（Dark Fringe Artifact），对火焰、光晕等高亮半透明特效尤为关键。在Cocos Creator中，对应混合模式应设置为`SRC=ONE, DST=ONE_MINUS_SRC_ALPHA`而非默认的`SRC_ALPHA`模式。

## 实际应用案例

### 案例一：移动端MOBA游戏技能特效

以某移动端MOBA游戏（2019年）的冰霜新星技能特效为例：美术组在Houdini中模拟了一段高精度冰晶炸裂流体动画，离线渲染输出为32帧、512×512像素的序列帧图集（4096×4096纹理，ETC2压缩后约2MB）。在游戏内以30fps播放，技能全程约1.06秒，Draw Call数量为1，GPU耗时在Snapdragon 865设备上测量为0.3ms，而同等视觉效果的实时粒子模拟版本在同设备上耗时约8ms——序列帧版本将GPU开销降低了约96%。

### 案例二：横版动作游戏的火焰循环

横版动作游戏常需要持续燃烧的火焰背景元素。美术师手绘12帧、128×128的火焰序列帧，以12fps循环播放（恰好因低帧率强化了火焰的跳跃感）。由于火焰动画循环，首末帧需保持视觉连续——制作时要求第0帧与第12帧（即下一循环第0帧）的火焰形态保持相近的整体轮廓，避免循环点出现跳帧感。

思考：如果同一技能特效需要同时在手机端（低端机）和PC端（高端机）上呈现不同画质，序列帧系统应如何设计LOD降级策略，以在保持相同Draw Call数量的前提下动态切换不同精度的图集资源？

## 常见误区

**误区一：帧数越多视觉越流畅**。帧数提升会使图集纹理尺寸呈平方增长，从16帧升至64帧，纹理面积增大4倍。超过某一视觉感知阈值后（通常为24fps），继续增帧的收益趋近于零，而存储与加载成本持续上升。正确做法是先确定目标帧率，再以最少帧数满足该帧率。

**误区二：序列帧无法做空间变换**。序列帧Quad可以在三维空间中进行位移、旋转、缩放，并可附着于粒子系统的每个粒子上（每个粒子独立播放序列帧动画的不同偏移起始帧），Unreal Engine的Flipbook材质节点正是基于此原理，允许大规模粒子群各自播放相位不同的同一段序列帧。

**误区三：图集必须排满所有格子**。当动画帧数不是整数倍完整填满图集时（例如20帧填入4×8=32格的图集），剩余12个格子为空白。着色器中必须通过`u_totalFrames`参数限制最大帧索引为19，防止采样到空白区域导致闪烁或透明帧跳出。

**误区四：序列帧适合所有风格的特效**。写实向的序列帧依赖高分辨率保存细节，而卡通风格的序列帧对低帧率和低分辨率的容忍度更高。强行用32帧512px图集制作卡通爆炸，是对存储预算的浪费；而用16帧128px图集表现写实火焰，则会暴露明显的像素化跳帧感。
