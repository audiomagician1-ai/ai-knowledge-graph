# 模拟空间

## 概述

模拟空间（Simulation Space）是粒子系统中决定粒子运动计算所在坐标系的设置项，分为**Local（本地空间）**和**World（世界空间）**两种模式。在 Local 模式下，粒子的位置、速度均相对于发射器自身的变换矩阵进行计算；在 World 模式下，粒子一旦发射便进入世界坐标系，其运动不再跟随发射器的位移或旋转。

这一概念源于 3D 图形学中坐标变换的基本体系。Unity 的 Shuriken 粒子系统在 2010 年引入 Local/World 模拟空间选项时，就将其作为影响粒子系统性能批次合并（Batching）的关键参数之一。选择错误的模拟空间不仅产生视觉错误，还会导致 Draw Call 数量成倍增加。

在特效优化领域，模拟空间的选择直接影响 GPU 实例化（GPU Instancing）和动态合批（Dynamic Batching）能否生效。使用 World 空间的多个相同粒子系统，若其材质和纹理图集相同，可以合并为单次 Draw Call；而 Local 空间的粒子系统因各自持有独立的变换偏移，通常无法与其他系统合批。

## 核心原理

### Local 空间的计算方式

在 Local 模拟空间中，每个粒子的位置向量 **P_local** 存储在发射器的本地坐标系内。引擎在渲染阶段将其乘以发射器的模型矩阵（Model Matrix）转换为世界坐标：

```
P_world = M_emitter × P_local
```

其中 `M_emitter` 包含发射器的平移、旋转和缩放信息。这意味着当发射器移动时，所有存活粒子会整体跟随移动，视觉上粒子仿佛"附着"在发射器上。这一特性适合角色身上的持续光环、武器拖尾等需要粒子紧随物体的效果。

### World 空间的计算方式

World 模拟空间下，粒子在发射瞬间即被赋予世界坐标位置，后续更新仅依赖速度、重力、噪声等物理参数，与发射器的变换矩阵完全解耦。当发射器快速移动时，已发射的粒子会"留在原地"继续飞散，形成自然的轨迹拖尾效果。火焰、爆炸烟雾通常使用 World 空间，因为现实中气体离开燃点后不会跟随物体运动。

### 性能差异的量化影响

Local 空间的核心性能代价在于：每帧必须将粒子本地坐标批量变换到世界空间，当单个粒子系统存活粒子数超过 **500** 个时，这一 CPU 端矩阵乘法开销开始变得可测量。更关键的是，Local 空间的粒子系统在 Unity 中**默认无法参与 GPU Instancing 合批**，因为每个实例的变换偏移各不相同，需要额外的 per-instance 数据传递。

相比之下，World 空间的多个粒子系统若共享同一图集材质（即前置知识"图集共享"所建立的合批条件），GPU Instancing 可将它们压缩为 **1 次 Draw Call**，在场景中存在 10 个以上同类特效时，帧率提升可达 30%~60%（具体数据因硬件而异）。

### 缩放对模拟空间的额外影响

Local 空间下，若发射器的 Scale 在运行时发生非均匀变化（如 X:1, Y:2, Z:1），粒子的速度方向也会被该缩放矩阵扭曲，导致粒子轨迹出现非预期的拉伸。这是 Local 空间特有的陷阱，World 空间粒子不受发射器缩放影响。Unity 提供了 `Scaling Mode` 子选项（Hierarchy / Local / Shape）来部分缓解此问题，但根本上无法消除非均匀缩放带来的速度扭曲。

## 实际应用

**移动端角色技能特效**：在移动平台上，一个角色释放技能时可能同时激活 3~5 个粒子系统。若这些子系统均使用 Local 空间且不能合批，Draw Call 将直接为 3~5 个。将非跟随性粒子（如爆炸碎片、冲击波）切换至 World 空间并共享图集后，可合并为 1 次 Draw Call，在中低端 Android 设备上减少约 2ms 的 CPU 提交时间。

**载具尾焰效果**：飞机或赛车的引擎尾焰需要粒子在飞行物离开后继续在空中消散，必须使用 World 空间。若误用 Local 空间，尾焰会像"固定在引擎口"的火苗而不是向后喷射的气流，视觉效果完全错误。

**UI 特效的特殊处理**：挂载在 Canvas 上的粒子系统若使用 World 空间，粒子会渲染在世界坐标原点附近而非 UI 面板上，因为 Canvas 的世界矩阵与屏幕位置存在偏差。此类场景必须使用 Local 空间，同时将 `Render Mode` 设置为 `Screen Space`。

## 常见误区

**误区一：World 空间一定比 Local 空间性能好**

许多开发者认为 World 空间因为不需要跟随变换计算所以更快。实际上，World 空间粒子每帧仍需在 CPU 或 GPU 上更新位置（速度积分、力场计算等），其 Simulation 开销与 Local 空间相当。性能差距主要来自**合批能力**而非模拟本身。孤立的单个 World 空间粒子系统与单个 Local 空间粒子系统，帧耗时几乎相同。

**误区二：Local 空间粒子系统移动发射器不影响已有粒子**

实际上 Local 空间的粒子存储的是本地坐标，发射器一旦移动，所有存活粒子的世界坐标同步跟随变化。如果需要"发射后脱离"的效果（如抛出后继续飞行的火球），必须使用 World 空间，Local 空间无法实现该行为。

**误区三：切换模拟空间不需要重新测试粒子行为**

模拟空间从 Local 切换到 World（或反向）后，粒子的起始速度方向、重力响应、旋转行为都会发生实质性变化，因为原始速度向量的参考系改变了。在一个朝 Y 轴负方向运动的 Local 空间系统中，若发射器本身旋转了 90°，粒子在 World 空间中的实际运动方向会完全不同，需要重新调整所有速度和力场参数。

## 知识关联

**前置概念——图集共享**：图集共享（Texture Atlas）确保多个粒子系统使用同一张纹理贴图，这是合批的材质层面前提。模拟空间则是合批的坐标系层面前提：即使图集已共享，Local 空间的粒子系统仍因独立的变换矩阵而无法合批。两者缺一不可，必须同时满足才能实现多系统的单次 Draw Call 渲染。

**后续概念——碰撞开销**：碰撞检测（Collision）的计算模式同样分为 World 和 Local 两种，并且其性能代价与模拟空间强相关。World 空间粒子的碰撞检测直接在世界坐标系中进行射线投射，而 Local 空间粒子需要先将碰撞体变换到本地坐标系后再求解，在粒子数量较大时会额外增加矩阵逆运算的开销。理解模拟空间的坐标变换逻辑是分析碰撞开销来源的直接前提。
