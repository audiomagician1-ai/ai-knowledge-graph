# 基准测试

## 概述

基准测试（Benchmark Testing）是通过在受控条件下运行固定场景、记录量化性能指标，从而为特效系统建立性能参照基线的方法。与一般的功能测试不同，基准测试关注的不是"特效是否正常播放"，而是"特效在标准硬件上消耗了多少 GPU 时间、粒子数量峰值是多少、帧率下降了几个百分点"。这些数字构成一条可追溯的历史曲线，使每次代码或资源修改的性能影响都可以被客观度量。

基准测试在游戏特效领域的系统化应用约兴起于 2010 年代中期，随着移动平台硬件碎片化加剧，仅凭主观感受判断特效性能代价越来越不可靠。Unreal Engine 4 在 4.14 版本（2016 年）引入了内置的 `Automation Test Framework`，使特效团队可以将性能测试脚本纳入 CI/CD 流水线，这标志着特效基准测试从"手动抓帧"走向"自动化回归"的重要节点。

建立基准测试的核心价值在于它能与预算系统配合：如果项目规定单个爆炸特效的 GPU 耗时预算上限为 2ms，那么基准测试就是验证每次迭代是否仍在 2ms 以内的唯一客观手段。没有基准数据，预算系统就只是一纸规定，无法驱动实际优化决策。

## 核心原理

### 测试场景的标准化构建

基准测试的有效性依赖于场景的高度可重复性。测试场景必须固定以下变量：摄像机位置与朝向、光源数量与方向、触发特效的时间戳、同屏其他渲染物数量。在 Unity 中，常用做法是创建一个专用的 `BenchmarkScene`，场景内只放置被测特效与统一的灰色背景板，使用 `Time.captureFramerate = 30` 将模拟帧率锁定为固定步长，避免因主机负载波动导致粒子模拟步进不一致。如果同一特效每次测试时触发位置有 0.1 米偏差，反射探针采样结果就会不同，数据噪声可能掩盖真正的性能变化。

### 核心指标的采集与计算

特效基准测试通常采集以下四类指标：

- **GPU 渲染耗时**（单位：ms）：使用 GPU 性能查询接口（如 D3D11 的 `ID3D11Query` 或 Vulkan 的 `VK_QUERY_TYPE_TIMESTAMP`）在特效 Draw Call 前后插入时间戳，两者差值即为该特效的 GPU 耗时。
- **粒子峰值数量**：记录整个特效生命周期内同时存活粒子的最大值，这是衡量粒子系统内存压力的直接指标。
- **Draw Call 增量**：对比特效播放前后的 Draw Call 总数，每个子发射器或材质变体都可能引入额外 Draw Call。
- **帧率跌落百分比**：在基准帧率（如锁定 60FPS）下，记录因特效触发导致帧率跌至 60FPS 以下的帧数占比，公式为 `跌落率 = 跌落帧数 / 总采样帧数 × 100%`。

通常连续采样 300 帧，取其中第 5 至第 95 百分位的数据（丢弃最高和最低 5% 的异常帧）计算平均值，这是 P90 稳定性指标的常见做法。

### 自动化测试流水线集成

手动跑基准测试无法保证每次提交都被覆盖。自动化方案的关键在于将测试脚本与版本控制系统的提交钩子（commit hook）绑定。一套完整的自动化基准测试流水线包含三个阶段：①由 CI 服务器（如 Jenkins 或 GitLab CI）在每次 Push 后自动启动目标平台的构建；②在专用的测试机器（与开发机硬件规格固定一致）上无头运行基准场景并输出 JSON 格式的性能报告；③将本次数据与主干分支的历史基线对比，若 GPU 耗时回归超过 15%，则自动将此次提交标记为"性能回归"并通知责任人。这一阈值（15%）是业界常见的合理警戒线，低于此值可能为统计噪声，高于此值通常是真实的代码或资源变更导致。

## 实际应用

**案例一：粒子系统重构前后的对比验证**

某项目对烟雾特效的 Shader 进行了从 Lit 到 Unlit 的降级优化。优化前基准测试记录 GPU 耗时为 3.8ms（峰值粒子数 200），优化后降至 1.2ms，降幅达 68.4%。基准数据同时显示 Draw Call 数量从 12 降至 4，这一结论支持了将该优化方案推广至项目中其他 23 个同类烟雾特效的决策。

**案例二：移动端特效预算回归检测**

在 Android 中低端机（骁龙 660 级别）上，项目设定技能特效的帧率跌落率不超过 5%。某次美术提交将火焰特效的贴图分辨率从 256×256 升至 512×512 后，自动化基准测试检测到跌落率从 3.2% 升至 11.7%，超过预算阈值，流水线在 20 分钟内自动拦截了该提交，避免了问题进入正式分支。

## 常见误区

**误区一：在开发机上跑基准测试即可代表目标平台**

开发机通常配备高端 GPU，其基准数据与目标平台（如中端 Android 手机）可能相差 5 倍以上。正确做法是在与发布目标规格相同的专用测试设备上跑基准，并且该设备的硬件配置必须版本锁定，不得随意升级。若测试设备更换，必须重新建立全量基线而不是沿用历史数据。

**误区二：只记录平均值，忽略帧时间的方差**

两个特效可能拥有相同的 GPU 耗时均值（如均为 2ms），但其中一个的标准差为 0.1ms（非常稳定），另一个的标准差为 0.8ms（帧时间波动剧烈）。后者在实际游戏中更容易造成肉眼可感知的卡顿（Hitch）。基准报告中必须同时包含 P50（中位数）、P95、P99 三个分位数，而不仅仅是算术平均值。

**误区三：基准测试通过就意味着实际游戏性能达标**

基准场景是高度简化的孤立环境，而实际游戏中特效会与角色蒙皮、阴影、后处理等多个系统同时运行。基准测试是用来追踪特效自身的性能变化趋势，而不是预测特效在完整游戏场景中的最终表现。验证最终场景需要配合 Profiler 在实际关卡中进行专项的场景性能分析。

## 知识关联

**与预算系统的关系**：预算系统定义了各类特效的性能上限数值（如"UI 特效 GPU 耗时不超过 0.5ms"），基准测试则是持续验证这些数值是否被遵守的执行机制。两者是"规则制定"与"规则执行"的关系，预算系统的数字直接作为基准测试中判断回归的阈值来源。

**通向平台差异的桥梁**：建立了基准测试体系后，下一步自然面对的问题是：同一套特效资源在 PC、iOS、Android 上的基准数据差异如何量化、如何决策？这就进入了平台差异的议题——需要为不同平台维护独立的基线数据库，并基于各平台的基准测试结果决定是否启用平台特定的 LOD 变体或 Shader 分级方案。
