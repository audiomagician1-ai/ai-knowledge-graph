# 基准测试

## 概述

基准测试（Benchmark Testing）是通过在受控条件下重复执行特定特效场景，记录帧率、Draw Call 数量、GPU 耗时等量化指标，从而建立性能基线（Baseline）的系统化方法。与普通性能观察不同，基准测试要求每次运行的场景内容、粒子数量、摄像机路径完全固定，确保不同版本之间的数据可比较性。

基准测试的概念源自硬件行业，1992 年 SPEC（标准性能评估组织）建立了首套 CPU 基准体系。游戏特效领域在 2010 年代移动平台爆发后开始系统引入这一方法，因为移动设备碎片化严重，同一粒子系统在不同 GPU 上帧率差距可达 5 倍以上，单靠主观"感觉流畅"完全无法指导优化工作。

对于特效优化而言，基准测试解决的核心问题是：优化措施究竟带来了多少可量化的改善？例如将粒子贴图从 512×512 压缩至 256×256，显存带宽降低了多少、对应场景帧率提升了几毫秒，这些数字只有通过标准化基准测试才能可靠地回答。

## 核心原理

### 测试场景的固定化设计

基准测试场景必须"无随机性"——所有粒子的初始种子（Random Seed）必须锁定为固定值，例如 Unity Particle System 中将 `Stop Action` 设为 `None` 并固定 `Random Seed = 12345`，确保第 1 帧到第 300 帧的粒子行为在任何机器、任何时间完全一致。摄像机路径通常录制为预烘焙的 Dolly 轨道，而非实时玩家输入，以消除视角差异对 GPU 裁剪结果的影响。测试时长建议至少覆盖 10 秒完整粒子生命周期，包含粒子从 Spawn 到 Die 的完整过程，避免只测量稳态而遗漏发射峰值。

### 性能指标的采集与计算

基准测试关注以下四类指标，每类指标的采集方式各不相同：

- **帧时间（Frame Time）**：以毫秒为单位，常用 P50、P95、P99 百分位数而非平均值。P99 帧时间代表最差的 1% 帧，对应玩家感知的"卡顿"，比平均帧时间更能反映特效峰值开销。
- **GPU Marker 耗时**：通过 GPU Profiler（如 RenderDoc 或 Mali Graphics Debugger）插入命名标记，精确测量特效 Draw Pass 的 GPU 执行时间，单位通常为微秒（μs）。
- **Draw Call 数量**：统计每帧特效专属的 Draw Call，合批（Batching）优化前后的 Draw Call 数量差异是基准测试最直接验证的指标之一。
- **填充率（Fill Rate）消耗**：计算公式为 `总像素写入量 = 粒子数 × 平均粒子屏幕面积（像素） × 层叠系数（Overdraw）`，基准测试需在固定分辨率（如 1920×1080）下运行以保证填充率数据可比较。

### 自动化测试流程

手动运行基准测试效率低且容易遗忘，成熟的特效团队会将基准测试集成到 CI/CD 流水线中。典型做法是在每次 Git 提交后由构建服务器自动执行以下步骤：①加载固定基准场景；②以 `Time.captureFramerate = 60` 锁帧运行 600 帧；③将帧时间序列写入 JSON 文件；④与上一次通过的基线 JSON 做差值比较；⑤若 P95 帧时间劣化超过阈值（常见设定为 **1.5ms**）则发出警告或阻断合并。这一流程可以在特效 Bug 引入后的数小时内自动发现性能回归，而不是等到 QA 阶段才察觉。

## 实际应用

**案例一：粒子数量上限的确定**
某手机游戏技术美术在制作爆炸特效时，通过基准测试对 50、100、200、400 个粒子分别跑完整测试序列，记录到在目标机型（骁龙 730G）上，粒子数从 200 增至 400 时 P95 帧时间从 8.2ms 跳升至 14.7ms，超过了 16.7ms（60fps 对应帧预算）的一半，由此确定单次爆炸粒子上限为 200。这一结论直接写入项目的特效预算规范文档，避免后续美术反复试错。

**案例二：Shader 变体的性能对比**
在比较 Alpha Blending 与 Alpha Test 两种透明度方案时，基准测试在相同粒子数（128 个）下分别运行，结果显示 Alpha Test 方案因提前 Discard 片元，GPU 帧时间降低了 **0.8ms**，但在 Adreno GPU 上由于 Early-Z 优化差异，反而比 Alpha Blending 慢了 **0.3ms**。此类跨平台结论只能通过在各目标平台独立运行基准测试才能得到，而非靠通用经验推断。

**案例三：LOD 切换距离的校准**
通过编写自动化测试脚本，让摄像机沿预设路径从距离粒子源 5 米移动至 50 米，每米记录一次 GPU 耗时，从而绘制出"距离-帧时间"曲线。根据此曲线将 LOD 切换距离从经验值 20 米调整至 28 米，在不影响视觉质量的前提下，中远景区域 GPU 特效耗时降低约 22%。

## 常见误区

**误区一：用平均帧率代替百分位帧时间**
很多初学者只记录"平均 58fps"就认为测试通过，但特效峰值（如粒子集中爆发的第 0.5 秒）可能瞬间下降到 30fps。P99 帧时间会直接暴露这类瞬时峰值，而平均帧率会将其稀释隐藏。正确做法是始终同时记录 P50、P95、P99 三档数据。

**误区二：在编辑器内运行基准测试**
Unity Editor 本身有大量后台线程（资产导入、日志系统、Inspector 刷新）会干扰 GPU 调度，同一场景在 Editor 中测得的帧时间可能比真机高出 **30%~50%**。基准测试必须使用 Development Build 或 Release Build 在目标硬件上执行，Editor 数据只能作为相对参考，不能作为基准数值存档。

**误区三：跨不同系统温度状态比较数据**
移动 GPU 在高温下会触发降频（Thermal Throttling），同一场景在设备冷启动后测得 6.1ms，而连续运行 10 分钟后可能升至 9.4ms。标准化基准测试要求统一预热时间（通常先运行 2 分钟暖机场景），再开始正式记录，以确保所有测试处于相同热状态下。

## 知识关联

**与预算系统的关系**：预算系统定义了特效帧时间、Draw Call、显存的上限数值（如"特效每帧不超过 4ms"），而基准测试是验证当前特效是否符合这些上限的测量工具。没有预算系统，基准测试的数据缺乏判断标准；没有基准测试，预算系统的数字无从落地核查。

**通向平台差异的路径**：在单一平台上完成基准测试方法论的建立之后，下一步自然面临的问题是：同一套基准场景在 iOS（Apple GPU）、Android 高通、Android Mali、PC 上会产生多大差异，以及如何为不同平台维护各自的基线数据库。这正是"平台差异"专题的核心内容——基准测试提供了跨平台性能比较所必需的统一测量语言。
