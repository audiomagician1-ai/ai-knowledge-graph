# 数据库架构

## 概述

数据库架构是指在软件系统中，对数据存储层的整体组织方式进行设计与规划，涵盖数据库类型选型（SQL或NoSQL）、物理部署拓扑、数据分布策略（分库分表）以及读写分离方案。一个数据库架构决策通常在系统上线后极难更换，因此在项目初期就必须依据业务数据模型、访问模式、一致性要求和规模预期做出明确设计。

关系型数据库（SQL）的历史可追溯至1970年Edgar Codd发表的论文《A Relational Model of Data for Large Shared Data Banks》，此后MySQL、PostgreSQL等成为Web应用的主流选择。NoSQL概念于2009年在一次技术会议上被正式命名，代表产品包括MongoDB（文档型）、Redis（键值型）、Cassandra（宽列型）和Neo4j（图型），它们针对特定访问模式提供了关系型数据库难以企及的扩展性。

数据库架构的重要性在于它直接影响系统的查询延迟、吞吐量上限和运维复杂度。以Twitter为例，其早期使用单一MySQL架构，在用户增长至亿级后出现严重性能瓶颈，最终通过引入Cassandra和分库分表方案将时间线读取延迟从秒级降至毫秒级。

## 核心原理

### SQL与NoSQL的选型依据

SQL数据库以ACID（原子性、一致性、隔离性、持久性）为核心保障，适用于资金转账、订单状态等强一致性场景。其表结构预先定义（Schema-on-Write），JOIN操作可跨表关联，适合数据关系复杂且需要复杂查询的业务，如ERP、银行核心系统。

NoSQL数据库遵循BASE理论（基本可用、软状态、最终一致性），以牺牲部分一致性换取水平扩展能力。选型时需匹配数据模型：若数据天然为JSON文档（如用户配置、商品详情），选MongoDB；若需要极低延迟的键值查询（如Session缓存），选Redis；若写入吞吐量极高且需跨数据中心多活（如IoT传感器数据），选Cassandra。

关键选型指标：若单表数据量预估超过5000万行且查询模式固定，应优先考虑NoSQL；若业务涉及多表事务且查询条件多变，优先保留SQL。

### 分库分表策略

**垂直拆分**是按业务模块将不同表拆分到不同数据库实例，例如将用户库（user_db）、订单库（order_db）、商品库（product_db）分离部署。垂直拆分减少了单库的表数量和连接竞争，但跨库JOIN需要在应用层通过多次查询拼装，不能依赖数据库原生JOIN。

**水平分表（Sharding）**是将同一张表按某个分片键（Shard Key）将数据行分散到多个物理表或实例中。常用分片算法有两种：

- **取模分片**：`shard_id = user_id % N`，其中N为分片数。优点是分布均匀，缺点是扩容时需要迁移约50%的数据（例如从N=4扩容到N=8）。
- **范围分片**：按user_id范围划分，如0~999999存储在shard_0。优点是扩容只需新增分片，缺点是可能产生热点（新注册用户集中在最后一个分片）。
- **一致性哈希**：将分片键映射到0~2³²的环形空间，新增节点时平均只迁移1/N的数据，适合动态扩缩容场景。

分片键的选择极为关键：以电商订单表为例，若按`order_id`分片，则查询某用户所有订单需要扫描所有分片（全分片扫描）；若按`user_id`分片，则用户订单聚合查询高效，但单个大用户可能成为热点分片。

### 读写分离架构

读写分离通过一主多从（Master-Replica）拓扑将写操作路由至主库、读操作路由至从库，可将读吞吐量线性扩展。MySQL的主从复制基于Binlog，存在复制延迟（通常1~100ms），因此对读取实时性要求高的场景（如支付后立即查询余额）必须强制走主库，避免读到旧数据。

## 实际应用

**电商平台订单系统**：通常采用MySQL按`user_id`水平分表，分为1024个分表（orders_0000到orders_1023），分片键为`user_id % 1024`。商品库存由于需要强一致性（防止超卖），使用MySQL行级锁或Redis分布式锁；商品详情因读多写少且结构灵活，迁移至MongoDB存储。

**社交平台消息系统**：微信消息存储采用按`user_id`分库的策略，每个用户的所有消息存于同一分库，保证单用户消息查询无需跨库。同时引入HBase存储历史消息，利用其行键（RowKey）的字节序排序特性，以`user_id + timestamp`作为RowKey实现时间范围扫描。

**实时计数器场景**：点赞数、浏览量等高频写入计数器，若存储于MySQL会产生严重锁竞争。标准方案是先写入Redis（`INCR`命令，时间复杂度O(1)），再定期批量同步至MySQL持久化，读取时优先读Redis。

## 常见误区

**误区一：认为NoSQL天然比SQL性能更好。**  
NoSQL的性能优势仅针对特定访问模式。Redis在简单键值查询上确实比MySQL快10~100倍，但MongoDB在执行多字段联合查询且缺少索引时，全集合扫描性能反而劣于MySQL的全表扫描，因为MySQL的B+树索引结构对磁盘顺序读更为友好。性能优劣取决于查询模式与数据结构是否匹配，而非数据库类型本身。

**误区二：分库分表越早做越好。**  
分库分表引入了事务跨库问题（需使用分布式事务框架如Seata，或改为最终一致性补偿方案）、全局唯一ID生成问题（不能再依赖数据库自增主键，需引入Snowflake等算法）以及运维复杂度的大幅提升。在MySQL单表数据量未超过2000万行、单实例QPS未超过5000之前，通过索引优化和读写分离往往已经足够。过早分库分表会显著增加开发成本。

**误区三：分片键选好就一劳永逸。**  
业务初期按`user_id`分片的电商订单表，在平台引入商家管理后台时会发现：商家需要查询其名下所有用户的订单，而按`user_id`分片使得这类查询必须扫描所有分片。这种情况下需要引入**冗余双写**策略，同时维护一张按`merchant_id`分片的订单索引表，通过双写保持数据同步，以存储冗余换取查询效率。

## 知识关联

数据库架构与**CAP定理**直接挂钩：CAP指出分布式系统中一致性（Consistency）、可用性（Availability）、分区容忍性（Partition tolerance）三者不可兼得。在选择Cassandra时即是选择了AP，意味着接受最终一致性；选择传统MySQL主从时在分区发生时选择CP，主库不可用则停止写服务。理解CAP定理是理解各类数据库在分布式场景下行为差异的直接前提。

在分层架构中，数据库架构向上对接**缓存架构**（如Redis旁路缓存模式，Cache-Aside Pattern），向下涉及存储介质选型（SSD vs HDD对Cassandra的压缩存储策略有直接影响）。分库分表方案还与**微服务架构**密切关联：每个微服务拥有独立数据库（Database per Service）是微服务的基本原则，其本质即一种垂直分库的架构落地形式。
