# 架构决策记录

## 概述

架构决策记录（Architecture Decision Record，简称 ADR）是一种轻量级文档格式，用于捕获软件架构中每一个重要技术决策的背景、理由与后果。每份 ADR 聚焦于单一决策，例如"选择 PostgreSQL 而非 MongoDB 作为主数据库"或"采用 REST 而非 gRPC 作为服务间通信协议"，并以固定结构记录做出该决策时的约束条件和权衡取舍。

ADR 格式由 Michael Nygard 于 2011 年在其博客文章《Documenting Architecture Decisions》中首次提出并推广。Nygard 认为，架构文档最大的失败不是缺少设计图，而是缺少"为什么这样设计"的解释——六个月后没有人记得当初为何选择某项技术。ADR 正是为了填补这一空白而生，它不记录"是什么"，而专门记录"为什么"。

ADR 的价值在于将架构决策从个人记忆外化为团队共享的文字资产。新成员加入团队时，通过阅读 ADR 目录可以迅速理解系统演进的脉络；当需要推翻旧决策时，ADR 提供了明确的依据，避免重复犯同样的错误——这种现象在软件行业被称为"重蹈覆辙反模式"（Groundhog Day Anti-pattern）。

## 核心原理

### ADR 的标准结构

一份标准 ADR 文档通常包含以下五个字段：

1. **标题（Title）**：简短描述决策内容，格式一般为"ADR-0012: 使用 Kafka 作为事件总线"，编号便于引用。
2. **状态（Status）**：取值为 `Proposed`（提议中）、`Accepted`（已接受）、`Deprecated`（已废弃）或 `Superseded by ADR-XXXX`（被某编号取代）。状态字段是 ADR 区别于普通技术文档的关键特征，它明确标注决策的生命周期。
3. **背景（Context）**：描述做出该决策时面临的技术约束、业务需求和外部限制，例如"团队只有 3 名工程师，均无 Kubernetes 运维经验"。
4. **决策（Decision）**：用一句话明确陈述最终选择，例如"我们将采用 Docker Compose 而非 Kubernetes 管理容器编排"。
5. **后果（Consequences）**：分别列出正面结果（例如"部署流程简化，新人上手时间从两周缩短至两天"）和负面结果（例如"无法支持跨节点自动扩容"）。

### 技术选型评估框架的嵌入

ADR 的背景和后果两个字段天然构成了一个技术选型评估框架。在撰写背景时，工程师需要识别**决策驱动因素（Decision Drivers）**，即约束选择空间的硬性条件；在撰写后果时，需要评估**质量属性权衡**，例如选择 SQLite 获得了可移植性（Portability），但牺牲了并发写入性能。

一些团队在 ADR 中额外加入**备选方案（Options Considered）**字段，列出所有曾经评估过的选项及其优缺点。例如在选择前端框架时，同时评估了 React、Vue 和 Svelte，并用表格对比了生态系统成熟度、团队学习曲线和打包体积三个维度。这种结构使 ADR 兼具决策记录与选型报告的双重功能。

### ADR 的存储与管理

ADR 通常以 Markdown 文件形式存储在代码仓库的 `docs/adr/` 目录下，与代码共同进行版本控制。这一选择本身也是一个架构决策：将文档与代码放在同一仓库，确保两者同步演进，避免出现代码已更新但文档仍描述旧方案的情况。

编号规则建议采用四位递增整数（如 `ADR-0001`），而非按日期命名，因为日期格式在不同地区存在歧义（MM/DD 与 DD/MM 的混淆问题），且编号顺序更直观反映决策的先后关系。工具层面，`adr-tools`（GitHub 上的开源 CLI 工具）可自动生成符合 Nygard 格式的 ADR 模板文件。

## 实际应用

**场景一：微服务数据库隔离决策**
某电商团队在拆分单体应用时，需要决定是否为每个微服务配备独立数据库。ADR-0008 记录了这一决策：背景中说明当时订单服务与库存服务共用同一个 MySQL 实例导致的锁竞争问题；决策字段明确写明"每个微服务拥有独立的数据库 Schema，禁止跨服务直接查询对方数据库表"；后果中诚实记录了引入的新问题——跨服务数据一致性需要通过 Saga 模式解决，增加了约 20% 的开发工作量。

**场景二：废弃旧决策**
同一团队半年后发现 REST API 无法满足低延迟需求，需要引入 gRPC。他们新建 ADR-0023，状态标注为 `Superseded ADR-0005`（ADR-0005 是当初选择 REST 的记录）。ADR-0023 在背景中引用 ADR-0005，解释当时选择 REST 的理由已不再成立（P99 延迟从 50ms 升至 320ms），从而完整保留了决策演进的历史轨迹。

## 常见误区

**误区一：将 ADR 写成技术调研报告**
ADR 不是研究论文，背景字段应控制在 3-5 句话以内，重点是"此时此地的约束"，而不是技术方案的全面介绍。若背景字段超过 500 字，通常意味着作者混淆了 ADR 与技术规范文档（Technical Specification）的边界。ADR 记录决策的瞬间，技术规范描述方案的细节。

**误区二：只记录成功的决策，回避失败的选择**
部分团队只为"正确"的决策创建 ADR，导致后来者看不到历史上曾经尝试并放弃的路径。ADR 的 `Deprecated` 状态专门用于标记那些被证明错误的决策，保留这些"失败的 ADR"与保留"成功的 ADR"同等重要——它们共同构成团队的架构认知资产。

**误区三：等到决策完全确定后才写 ADR**
`Proposed` 状态的存在正是为了支持在决策过程中就开始撰写 ADR。在团队评审阶段，一份 `Proposed` 状态的 ADR 可以作为讨论的基础文档，比口头讨论更容易聚焦，也方便异步参与的团队成员（如不同时区的同事）提交书面意见。

## 知识关联

ADR 建立在软件架构概述所介绍的"架构是一系列重要决策的集合"这一核心命题之上——ADR 正是将这些决策从隐性知识转化为显性文档的具体操作手段。理解软件架构中质量属性（如可用性、可维护性、性能）之间的权衡关系，是撰写高质量 ADR"后果"字段的前提，因为每个架构决策本质上都是在多个质量属性之间进行取舍。

在团队实践层面，ADR 与架构治理（Architecture Governance）机制紧密配合：治理机制规定哪类决策需要强制创建 ADR（例如"所有引入新外部依赖的决策"），ADR 则是治理流程中的证据留存工具。随着 ADR 积累到一定数量（通常超过 30 份），团队可从中提炼出系统性的架构原则文档，完成从单点决策记录向整体架构知识体系的升华。
