# 领域驱动设计

## 概述

领域驱动设计（Domain-Driven Design，简称 DDD）是由 Eric Evans 于 2003 年在其同名著作《Domain-Driven Design: Tackling Complexity in the Heart of Software》中系统提出的软件建模方法论。其核心主张是：软件系统的设计应当以业务领域的知识模型为首要驱动力，而非以数据库表结构或 UI 流程为出发点。

DDD 诞生于企业级软件开发遭遇"大泥球"架构困境的背景下——大型单体系统中业务逻辑与技术实现相互纠缠，导致代码难以理解和修改。Evans 提出通过构建精确反映业务语义的"领域模型"，让代码能够直接表达业务规则，使软件系统成为领域专家与开发者之间共同思考的工具，而非翻译层。

DDD 对现代微服务架构影响深远。微服务的"服务边界如何划分"这一核心问题，正是通过 DDD 的**限界上下文**概念获得了可操作的答案。理解 DDD 能解释为什么两个微服务中都可以出现名为 `Customer` 的类，却不需要共享同一张数据库表。

## 核心原理

### 通用语言（Ubiquitous Language）

通用语言是 DDD 中开发者与领域专家在同一限界上下文内共同使用、精确约定的词汇体系。"通用"不代表跨越全公司统一，而是指在该上下文边界内，代码命名、需求文档、口头讨论必须使用完全一致的术语。

举例来说，在电商系统中，"订单取消"对于仓储上下文意味着"释放库存锁定"，对于财务上下文意味着"触发退款流程"。如果两个上下文强行共用同一个 `Order.cancel()` 方法，业务含义就会模糊。通用语言要求每个上下文内部都有自己精确的动词和名词，例如仓储上下文用 `Allocation.release()`，财务上下文用 `Payment.refund()`，各自清晰。

在代码层面，通用语言直接体现为类名、方法名和变量名，开发者应该能把代码朗读给领域专家听，并期望对方能够理解。这是评估通用语言质量的实用标准。

### 限界上下文（Bounded Context）

限界上下文是一个明确的边界，在该边界内，某个领域模型的定义和含义保持一致。同一个业务实体（如"用户"）在不同上下文中可以拥有完全不同的属性和行为，这不是设计错误，而是 DDD 对真实业务复杂性的正确建模。

限界上下文之间的协作通过**上下文映射（Context Map）**描述，常见关系包括：
- **防腐层（Anti-Corruption Layer，ACL）**：保护己方模型不被上游模型污染
- **开放主机服务（Open Host Service）**：通过协议发布服务供他人集成
- **共享内核（Shared Kernel）**：两个上下文共享一小部分模型，需双方共同维护

一个典型的电商平台可以划分为：商品目录上下文、库存上下文、订单上下文、支付上下文、物流上下文，每个上下文对"商品"的建模内容完全不同——商品目录关心描述和图片，库存关心 SKU 数量，物流关心体积和重量。

### 聚合根（Aggregate Root）

聚合（Aggregate）是一组具有内聚性的领域对象的集合，聚合根是该集合对外的唯一访问入口。外部对象只能持有聚合根的引用，不得直接持有聚合内部实体的引用，这一规则保证了聚合内部不变量（Invariant）的强制执行。

以订单为例：`Order`（订单）是聚合根，`OrderLine`（订单行）是聚合内部实体。外部代码不能绕过 `Order` 直接修改 `OrderLine` 的数量——必须调用 `Order.updateItemQuantity(itemId, newQty)`，由 `Order` 负责校验修改后总金额是否超出信用额度等不变量。

聚合根还确定了**事务边界**：一次数据库事务只应修改一个聚合的状态。如果需要跨聚合协调，必须通过领域事件（Domain Event）异步传递，而非在一个事务中同时修改两个聚合。这一规则直接推动了事件驱动架构的采用。

聚合的大小设计是 DDD 实践中最棘手的问题。过大的聚合导致并发冲突频繁（多个操作竞争同一聚合的锁），过小的聚合则难以维护不变量。经验法则是：**将聚合设计得尽可能小，仅包含需要在同一事务内保持一致的对象**。

## 实际应用

在微服务架构下，限界上下文与微服务存在自然对应关系，通常建议一个限界上下文对应一个微服务（或少数几个），而不是反过来把多个限界上下文塞进一个服务。

以在线银行系统为例：账户开户流程涉及"客户身份上下文"和"账户产品上下文"。身份上下文中 `Customer` 包含 KYC 认证状态、证件信息；产品上下文中 `AccountHolder` 只包含账户关联的客户 ID 和联系方式。两者通过防腐层集成：产品上下文订阅身份上下文发出的 `CustomerVerified` 领域事件，在本地创建对应的 `AccountHolder` 记录，而非直接查询身份服务的数据库。

在 Python 代码实践中，聚合根通常作为 Repository 的唯一操作对象：`order_repository.save(order)` 保存整个聚合，框架（如 SQLAlchemy 的 Unit of Work 模式）负责追踪聚合内部实体的变更，开发者无需为 `OrderLine` 单独编写保存逻辑。

## 常见误区

**误区一：将数据库表结构映射为领域模型**。初学者常从 ORM 模型出发，把 `orders` 表和 `order_lines` 表直接映射为领域对象。这导致领域模型充满 getter/setter 而无业务方法，退化为"贫血模型（Anemic Domain Model）"——Evans 本人曾多次批评这是 DDD 的反模式。正确做法是先建模领域行为，再考虑如何持久化。

**误区二：聚合根等同于数据库主表**。有开发者认为只要是主键所在的表对应的实体就是聚合根。实际上聚合根的判断标准是：哪个对象负责维护这组对象的业务不变量，哪个就是聚合根，与是否有外键关系无直接关联。`Address`（地址）可以是 `User` 聚合的值对象，尽管它在数据库中可能有自己的表。

**误区三：限界上下文必须跨越物理服务边界**。DDD 中限界上下文是逻辑概念，在系统规模较小时，多个限界上下文可以部署在同一个进程内，通过 Python 包或 Java 模块隔离。不必因为"没有做成微服务"就认为没有实践 DDD。

## 知识关联

DDD 与**六边形架构**（Hexagonal Architecture）天然互补：六边形架构定义了领域层与基础设施层之间的端口/适配器分离方式，而 DDD 填充了领域层内部的建模规则——聚合、实体、值对象、领域服务、领域事件构成了六边形架构"内六边形"的具体内容。没有 DDD，六边形架构的领域层容易退化为只有服务类和数据类的贫血设计；没有六边形架构，DDD 的领域模型容易被框架和数据库细节污染。

在分布式系统方向，聚合边界与事务边界的对应关系，直接引出 **Saga 模式**（分布式长事务的编排/协调方案）和**事件溯源（Event Sourcing）**的必要性——当聚合状态只能通过重放领域事件推导时，系统获得了完整的审计日志和时间旅行能力，这在金融领域尤其重要。
