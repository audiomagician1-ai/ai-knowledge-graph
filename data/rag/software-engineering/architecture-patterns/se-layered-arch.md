# 分层架构

## 概述

分层架构（Layered Architecture）是一种将软件系统按照职责划分为若干水平层次的设计模式，每一层只能与相邻层进行通信，上层调用下层提供的服务，下层不感知上层的存在。最经典的形式是三层架构（3-Tier Architecture），由表示层（Presentation Layer）、业务逻辑层（Business Logic Layer）和数据访问层（Data Access Layer）组成。

分层思想最早可以追溯到1960年代的操作系统设计。Dijkstra在1968年发表的THE多道程序系统设计论文中，首次系统性地将操作系统划分为六个层次，从硬件抽象到用户程序逐层堆叠。这一思想在1990年代随着企业级Web应用的兴起被广泛采用，Microsoft的DNA架构和Sun的J2EE平台都将三层架构作为标准参考模型推广。

分层架构之所以重要，在于它直接解决了"关注点分离"问题：开发者修改数据库查询逻辑时无需触碰界面代码，UI设计师调整页面布局时无需理解SQL。这种隔离使得每一层可以独立替换，例如将数据库从MySQL切换为PostgreSQL只需修改数据访问层，其他层代码保持不变。

## 核心原理

### 严格分层与松散分层

分层架构存在两种变体：**严格分层**（Strict Layering）规定请求只能逐层传递，第1层只能调用第2层，不可跨越直接调用第3层；**松散分层**（Relaxed Layering）允许跨层调用，例如表示层可以直接访问数据访问层。Martin Fowler在《企业应用架构模式》（2002年）中记录了这两种模式的权衡：严格分层保障了隔离性但会引入中间层的无效传递代码（Pass-through代码），松散分层减少样板代码但层边界逐渐模糊，容易退化为意大利面条代码。

### 三层架构的职责边界

三层架构中每层的职责有精确定义：

- **表示层**：负责接收用户输入、渲染视图、处理HTTP请求/响应。它不包含任何业务判断，仅做格式验证（如"手机号必须是11位数字"）。
- **业务逻辑层**：包含应用的核心规则，如"订单金额超过500元打九折"、"用户余额不足时拒绝下单"。这一层不知道数据存储在哪里，不持有任何SQL语句。
- **数据访问层**：封装所有数据库操作，对外暴露面向对象的接口（如`getUserById(int id)`），内部实现可以是JDBC、ORM框架或REST调用。

一个常见的数字：在典型的Java EE企业应用中，业务逻辑层的代码量通常占整体代码量的40%～60%，因为它承载了最复杂的领域知识。

### 依赖规则与单向依赖

分层架构的核心约束是**依赖方向必须单向向下**。用依赖关系表示为：

```
表示层 → 业务逻辑层 → 数据访问层 → 数据库
```

箭头表示"依赖/调用"关系。数据访问层绝不能调用业务逻辑层的代码，否则会形成循环依赖，导致编译失败或运行时行为不可预测。在Java项目中，这通常通过Maven模块划分来强制执行：`dal`模块不引入`service`模块的依赖，违反则编译报错。

### N层架构的扩展

当三层不足以描述系统时，可以扩展为N层。常见的四层架构在业务逻辑层和数据访问层之间增加**领域模型层**（Domain Layer），将纯粹的业务实体（如`Order`、`User`类）从业务规则服务中独立出来。五层架构还可以在表示层下方增加**应用服务层**（Application Service Layer），专门处理用例编排（事务管理、权限校验调用顺序）而不包含领域逻辑本身。

## 实际应用

**电商订单系统**是分层架构的典型场景。表示层的`OrderController`接收HTTP POST请求，将请求参数转换为`CreateOrderRequest`对象后传入业务逻辑层。业务逻辑层的`OrderService`执行库存检查、价格计算、优惠券核销等规则，调用数据访问层的`OrderRepository`和`InventoryRepository`完成持久化。前端的表现形式从Web页面换为移动App时，只需增加一个新的Controller，`OrderService`代码零修改。

**ASP.NET MVC框架**是分层架构的框架级实现范例，它通过命名约定将代码强制分入`Controllers`、`Services`、`Repositories`三个目录，配合依赖注入容器在运行时组装各层对象，开发者只需遵循约定即可天然获得分层隔离的好处。

## 常见误区

**误区一：将DTO（数据传输对象）误用于层间传递**。很多开发者直接将数据库实体类（如Hibernate的`@Entity`）传递到表示层，导致表示层与数据库表结构耦合。正确做法是每个层边界都应定义独立的数据对象：数据访问层返回`UserEntity`，业务逻辑层转换为`UserDO`，表示层再转换为`UserVO`，虽然有转换成本但层间独立性得到保障。

**误区二：把所有逻辑堆入业务逻辑层形成"贫血模型"**。三层架构并不要求业务实体（如`Order`类）只有getter/setter方法而没有任何行为，将`order.calculateTotalPrice()`这样的逻辑写在`OrderService`而不是`Order`类中，会导致`Service`类膨胀到数千行。分层架构约束的是**层间依赖方向**，并不限制每层内部的面向对象设计。

**误区三：认为分层架构天然支持高并发**。分层架构是代码组织模式，与部署拓扑是两个独立维度。三层架构的代码可以打包成单个进程部署（单体），也可以将每一层部署为独立服务（分布式）。将架构模式与扩展性混淆，会导致错误地为低流量系统引入不必要的分布式复杂度。

## 知识关联

分层架构建立在软件架构概述中"模块化"和"关注点分离"两个概念之上，是这些抽象原则在水平维度上的具体落地形式。理解分层架构中"依赖方向单向向下"的约束，是学习**六边形架构**的必要前提——六边形架构（又称端口与适配器架构）正是为了解决分层架构中"业务逻辑层依赖数据访问层"这一问题而提出的，它通过依赖倒置原则（DIP）使业务层不再依赖任何外部层。**Clean Architecture**则将这种倒置思想推向极致，定义了四个同心圆层次，其核心规则"源代码依赖只能由外层指向内层"是对分层架构单向依赖规则的一次根本性翻转——依赖方向由"上到下"改为"外到内"，领域实体层位于最内圆，不依赖任何框架或数据库代码。
