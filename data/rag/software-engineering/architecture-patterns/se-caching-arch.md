# 缓存架构

## 概述

缓存架构是一种通过在数据源与数据消费者之间插入高速存储层来减少访问延迟和后端负载的系统设计模式。其核心思想是利用局部性原理（Locality of Reference）：程序在一段时间内倾向于反复访问同一批数据，因此将热点数据存储在比原始数据源更快的介质中，可以显著提升系统吞吐量。

缓存这一概念最早来自1960年代的CPU硬件设计，IBM System/360在1968年引入了高速缓冲存储器（Cache Memory）的概念。软件领域将该思想延伸到分布式系统中，形成了今天以Redis、Memcached为代表的分布式缓存体系，以及CDN（内容分发网络）等全球化缓存基础设施。

在互联网高并发场景下，缓存架构的价值极为突出。以电商大促为例，数据库每秒仅能处理数千次查询，而Redis单节点每秒可处理10万次以上的读操作。若不引入缓存层，数据库将成为系统瓶颈，导致请求积压和服务崩溃。缓存架构的设计质量直接决定系统能否承受突发流量。

## 核心原理

### 多级缓存层次结构

实际生产系统通常采用L1/L2/L3三级缓存模型：

- **L1（本地内存缓存）**：部署在应用服务器进程内，如Java的Guava Cache或Caffeine，读取延迟低于1毫秒，但容量受单机内存限制，通常为数百MB，且多实例间数据不共享。
- **L2（分布式缓存）**：以Redis Cluster或Memcached集群为代表，跨进程共享，读取延迟约1～5毫秒，容量可扩展至数TB。
- **L3（CDN/边缘缓存）**：针对静态资源或可公开的API响应，部署在离用户最近的边缘节点，将延迟压缩至10毫秒以内。

请求按照 L1 → L2 → L3 → 数据库的顺序依次穿透，只有在各层均未命中时才会触达数据库，这种设计使得数据库承受的真实流量往往不足总请求量的5%。

### 缓存读写策略

缓存架构中存在四种经典的读写模式，各有适用场景：

1. **Cache-Aside（旁路缓存）**：应用程序先查缓存，未命中则查数据库并将结果写入缓存。写操作时先更新数据库，再删除缓存（而非更新缓存）。删除而非更新的原因是避免并发写入导致缓存存储了旧值。
2. **Read-Through**：缓存层透明地负责从数据库加载数据，应用程序仅与缓存交互。
3. **Write-Through**：每次写操作同时写入缓存和数据库，保证强一致性，但写延迟增加。
4. **Write-Behind（Write-Back）**：写操作仅更新缓存，由缓存层异步批量刷写数据库，写性能最优但存在数据丢失风险，适合日志计数等允许少量丢失的场景。

### 缓存一致性保证

缓存与数据库之间的数据不一致是缓存架构最棘手的问题。常见方案如下：

- **TTL过期机制**：为每条缓存记录设置生存时间（Time-To-Live），过期后自动失效。TTL的选择是关键权衡：TTL=0意味着不缓存，TTL=∞意味着永不失效。实践中对用户个人信息可设置TTL为300秒，对商品价格等敏感数据则设置为30秒甚至更短。
- **主动失效（Cache Invalidation）**：数据更新时主动删除或更新缓存条目。但在分布式环境中，若多个缓存节点存储了同一数据的副本，需要广播失效消息，此时可借助Redis的Pub/Sub机制实现多节点同步失效。
- **双删策略**：在Cache-Aside模式下，为防止并发读写导致缓存被"回写"脏数据，推荐的操作顺序为：①删除缓存 → ②更新数据库 → ③延迟500毫秒后再次删除缓存，以覆盖期间可能被并发读操作重建的旧缓存值。

### 缓存失效经典问题

三类经典失效问题是缓存架构面试和生产中的必考点：

- **缓存穿透**：查询数据库中根本不存在的key（如恶意请求`id=-1`），每次都穿透到数据库。解决方案是对空结果也缓存（TTL设为60秒），或使用Bloom Filter在缓存层前过滤必定不存在的key。
- **缓存击穿**：某个热点key在高并发访问时恰好过期，大量请求同时穿透到数据库。解决方案是使用互斥锁（Mutex Lock）让只有第一个请求重建缓存，其余请求等待，或对热点key设置永不过期。
- **缓存雪崩**：大量缓存key在同一时刻集中过期，导致数据库瞬间承压。预防方案是在TTL基础上加随机偏移量，如 `TTL = 基础时间 + Random(0, 300秒)`，将过期时间打散。

## 实际应用

**微博热搜排行榜**：使用Redis的Sorted Set（有序集合）数据结构存储热搜词条及其热度分值，每次微博被发布时通过`ZINCRBY`命令更新分值，`ZRANGE`命令以O(log N)时间复杂度获取TopN排名。整个排行榜计算和读取均在内存中完成，数据库仅用于持久化归档。

**电商商品详情页**：采用"本地缓存+Redis二级缓存"的双层方案。商品基础信息（标题、图片）变动少，存入Caffeine本地缓存，TTL设为10分钟；商品实时库存变动频繁，仅存入Redis并设置TTL为30秒，同时通过MQ（消息队列）监听库存变更事件主动失效缓存。

**会话（Session）管理**：Web应用将用户登录Token存入Redis，key格式为`session:{userId}`，TTL设为7200秒（2小时）。用户每次请求后通过`EXPIRE`命令刷新TTL，实现滑动过期窗口，避免活跃用户被强制下线。

## 常见误区

**误区一：缓存更新应该"写入新值"而非"删除旧值"**  
很多初学者在数据库更新后，会将新值同步写入缓存。但在并发场景下，若线程A更新数据库，线程B同时读到旧值并准备写缓存，执行顺序可能为：A更新DB → B写旧值入缓存 → A写新值入缓存。若A先于B写缓存，缓存最终存储的是B的旧值，造成长期脏数据。**正确做法是删除缓存**，让下一次读操作重建缓存，规避并发写冲突。

**误区二：缓存命中率越高越好，应该无限扩大缓存容量**  
缓存的存储介质（内存）成本远高于磁盘，盲目扩大缓存容量会显著增加基础设施成本。工程上通常以"80%的请求由20%的热点数据贡献"（即Pareto原则）为依据，优先为高频访问数据分配缓存空间，并设置LRU（最近最少使用）淘汰策略，而非追求100%命中率。实际生产中，60%～80%的命中率往往已能满足性能目标。

**误区三：本地缓存与分布式缓存效果相同，选一种即可**  
本地缓存（进程内）无需网络IO，读取延迟是分布式缓存的100倍以上，但在多实例部署场景下，各实例的本地缓存独立存储，会导致同一数据被多次从数据库加载，且各实例缓存内容可能不同步。分布式缓存解决了共享问题但引入了网络开销。生产系统的最佳实践是二者配合使用，以本地缓存承接极高频读取，以分布式缓存保证跨实例一致性。

## 知识关联

**与数据库索引的关系**：缓存和数据库索引都以牺牲空间换取查询速度，但索引仍在数据库内部运作，而缓存将数据提升到数据库之外的更快介质。在设计缓存key时，需要结合数据库查询模式选择缓存粒度，避免缓存了用不到的大对象。

**与消息队列的关联**：在主动缓存失效场景中，数据库的变更事件往往通过Kafka或RocketMQ等消息队列广播给缓存失效消费者，形成"数据库变更 → MQ → 缓存失效"的解耦链路，避免缓存失效逻辑侵入业务代码。

**向分布式一致性的延伸**：多级缓存中每一层都是数据的一个副本，这本质上是分布式系统的副本一致性问题。深入学习缓存架构后，可自然过渡到CAP定理、最终一致性模型以及分布式事务等更复杂的一致性议题，这些概念共享"副本同步延迟"这一核心矛
