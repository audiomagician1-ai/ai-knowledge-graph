# 对象池

## 概述

对象池（Object Pool）是一种通过预先分配固定数量的对象并循环复用这些对象，从而避免运行时频繁执行 `new`/`delete`（或 `malloc`/`free`）的编程模式。其核心思想是：与其每次需要对象时向堆申请内存，不如在游戏启动阶段一次性批量创建对象并存入池中，需要时从池中"借出"，用完后"归还"而非销毁。

该模式最早在数据库连接池（1990年代）中得到广泛应用，后被游戏开发社区系统化。Robert Nystrom 在2014年出版的《Game Programming Patterns》一书中将对象池列为与命令模式、观察者模式并列的核心游戏模式，并以粒子系统作为标准示例详细分析了其实现细节。

在游戏编程中，对象池的价值集中在两点：第一，消除托管语言（如 C#/Unity）中垃圾回收器（GC）因大量短生命周期对象积累而触发的卡顿；第二，使内存布局连续，提升 CPU 缓存命中率。子弹、粒子、敌人、音效实例等每帧可能大量生灭的对象，是对象池最典型的应用场景。

## 核心原理

### 池的基本数据结构

对象池内部维护一个固定大小的对象数组（而非链表），每个槽位携带一个 `inUse` 布尔标志或枚举状态。以粒子系统为例，典型实现如下：

```cpp
class ParticlePool {
    static const int POOL_SIZE = 100;
    Particle particles_[POOL_SIZE];  // 连续内存块，一次性分配

public:
    Particle* create() {
        for (int i = 0; i < POOL_SIZE; i++) {
            if (!particles_[i].inUse()) {
                particles_[i].init();
                return &particles_[i];
            }
        }
        return nullptr;  // 池已满，返回空
    }
};
```

数组而非链表的选择至关重要：100个粒子连续存储时，一次缓存行（通常64字节）可以装入多个粒子对象，批量更新时的缓存缺失次数远少于分散在堆上的指针链表。

### 空闲链表优化（Free List）

顺序扫描查找空闲槽位的时间复杂度为 O(n)，当池容量达到数千时性能下降明显。优化方案是在未被使用的对象内部存储"下一个空闲对象的索引"，构成一条嵌入式空闲链表（intrusive free list）。`create()` 只需取链表头节点，`destroy()` 只需将节点插回链表头，两者均为 O(1)。这种技巧直接复用了对象自身的内存来存放元数据，不需要额外的指针数组。

### 对象生命周期管理

对象池中的对象从未真正被构造或析构（在 C++ 意义上），取而代之的是 `init()` 和 `reset()` 方法显式初始化和清理状态。这意味着：对象的构造函数仅在程序启动时调用一次，析构函数在程序退出时调用一次；游戏运行期间只有普通成员函数调用，完全绕开了 GC 和内存分配器。在 Unity C# 中，`UnityEngine.Pool.ObjectPool<T>` 类（Unity 2021 起内置）通过 `OnGet` 和 `OnRelease` 两个回调委托实现等效的初始化与重置逻辑。

## 实际应用

**子弹与投射物系统**：射击游戏中玩家每秒可能发射数十颗子弹，每颗子弹平均存活 0.5–2 秒。若不使用对象池，每秒将产生数十次堆分配与释放，在 Unity 中极易触发 Gen 0 GC。使用容量为 200 的子弹对象池后，整个游戏会话期间堆分配次数降为 0（初始化完成后），GC 卡顿从平均每30秒一次降为零。

**粒子特效**：爆炸、火焰、烟雾等粒子系统单次可激活 50–500 个粒子对象。现代引擎（Unreal Engine 的 Niagara、Unity 的 VFX Graph）在底层均使用对象池管理粒子实例，并将池大小暴露为 `Max Particles` 参数供开发者调整。

**音效实例池**：同一音效在同一帧被多个事件触发（如多颗子弹同时命中）时，音频引擎需要多个独立的播放实例。预分配 32 个 AudioSource 组件构成音效池，是 Unity 游戏中防止 `AddComponent` 卡顿的标准做法。

## 常见误区

**误区一：池越大越好**
池容量并非越大越好。预分配 10000 个粒子对象而游戏实际峰值只需 200 个，会造成内存浪费，且初始化耗时增加启动时间。正确做法是通过 profiling 测量峰值用量，再加 20%–30% 的余量设定池大小。池满时返回 `nullptr` 并跳过该次生成（而非强制扩容）在游戏中往往是可接受的降级策略——少一颗粒子不影响游戏体验，但GC卡顿会。

**误区二：对象归还后立即重用是安全的**
将对象归还池后，若代码其他地方仍持有该对象的指针或引用，下一次从池中取出并重新初始化该对象时，旧引用将指向一个状态已被覆盖的"僵尸对象"。这是对象池特有的悬空引用问题，与普通 `delete` 后的悬空指针性质相同但更难察觉，因为对象的内存地址并未失效。解决方案是使用"世代索引"（generation index）：对象携带一个版本号，引用方同时持有索引和版本号，归还时版本号递增，持有旧版本号的引用自动失效。

**误区三：对象池能替代所有堆分配优化**
对象池只解决"同类型对象频繁分配/释放"的问题，对于大量不同类型对象混合分配的场景，需要的是池分配器（Pool Allocator）或线性分配器（Linear Allocator），而非对象池。将游戏中所有对象都纳入对象池管理会导致代码耦合度急剧上升，只有生命周期短且数量波动大的对象类型才是对象池的适用目标。

## 知识关联

对象池依赖对 **GC 优化**的理解才能体会其价值：不了解 .NET/Mono 分代垃圾回收机制的开发者很难量化对象池带来的性能收益，也不知道何时必须引入它。对象池在接口层面常与**服务定位器**模式配合——通过服务定位器获取全局唯一的粒子池或子弹池实例，避免将池引用直接传递给每个生产者对象。

从对象池可以自然延伸到**池分配器**：对象池管理的是特定类型的对象，而池分配器管理的是原始内存块，是更底层、更通用的版本，可以为多种不同类型的对象提供快速分配。**双缓冲**模式则解决了对象池的另一个挑战：当一帧内产生的粒子需要在下一帧才能被安全读取时，双缓冲提供了无锁的帧间状态分离机制。**空间分区**（如四叉树）经常以对象池为基础构建分区节点的内存管理，避免树节点分裂/合并时的动态内存操作。
