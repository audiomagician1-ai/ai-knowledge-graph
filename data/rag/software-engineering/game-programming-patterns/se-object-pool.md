# 对象池

## 概述

对象池（Object Pool）是一种通过预先分配固定数量的对象并在需要时复用它们，而非反复创建和销毁，从而降低垃圾回收（GC）压力的内存管理模式。其核心思想是将对象的生命周期管理权从运行时的动态分配机制转移到程序员手中，使内存分配操作在游戏启动或关卡加载阶段一次性完成。

对象池模式最早在游戏开发领域因子弹、粒子、敌人单位等高频生成销毁的场景中被广泛采用。在早期内存受限的主机平台（如PlayStation时代），GC停顿或内存碎片化会导致帧率明显抖动，对象池成为保证16ms帧时间预算的关键手段。在Unity等现代引擎中，频繁调用`Instantiate()`和`Destroy()`会触发托管堆的GC分配，而对象池通过`SetActive(false/true)`绕过了这一过程。

在游戏主循环每帧都在运行的情景下，哪怕一次几毫秒的GC停顿也会打破60fps的流畅体验。以射击游戏为例，若每秒发射10发子弹、每颗子弹生命周期约2秒，则每秒需销毁10个对象，不使用对象池时单局游戏内可能累积数千次动态分配，对象池将这个数字压缩为零。

## 核心原理

### 预分配与初始化

对象池在初始化时一次性分配N个对象（N由开发者根据峰值需求估算），所有对象默认处于"未激活"状态，存储在一个空闲列表（Free List）中。以C#伪代码为例：

```csharp
// 初始化：预分配20个子弹对象
for (int i = 0; i < 20; i++) {
    GameObject bullet = Instantiate(bulletPrefab);
    bullet.SetActive(false);
    pool.Enqueue(bullet);
}
```

初始化时产生的内存分配是一次性的托管堆写入，之后的"创建"和"销毁"实际上仅是从队列中取出/归还引用，时间复杂度为O(1)，不涉及任何新的堆内存申请。

### 借用与归还机制

对象池对外暴露两个关键操作：`Acquire()`（借用）和`Release()`（归还）。借用时从空闲列表取出一个对象并重置其状态；归还时将对象放回空闲列表而非销毁。状态重置（Reset）是正确性的关键——归还的子弹对象必须清零速度向量、重置碰撞状态，否则复用时会携带上一次使用的残留数据，产生难以追踪的逻辑Bug。

归还时机通常通过两种方式触发：对象自身在`Update()`中检测到生命周期结束时主动调用`Release()`，或由外部管理器在检测到对象越界/碰撞后强制回收。两种方式的选择影响到对象与池之间的耦合程度。

### 池容量与溢出策略

当所有对象都处于激活状态时，`Acquire()`会面临空池问题。常见的三种溢出策略各有代价：
1. **返回null**：调用方需做空值检查，逻辑最简单，但可能导致静默失败（如子弹不出现）。
2. **动态扩容**：临时`Instantiate()`一个新对象，容量扩大1，破坏了"零GC分配"的承诺，但保证功能完整性。
3. **复用最旧对象**（循环缓冲区方案）：强制回收池中存活时间最长的对象，适合粒子效果等视觉非关键对象，Unity的`ParticleSystem`内部即采用此策略，最大粒子数默认上限为1000。

## 实际应用

**子弹与投射物**：这是对象池最经典的应用场景。横版射击游戏中，玩家每次按下射击键触发`BulletPool.Acquire()`，子弹飞出屏幕或命中目标后触发`BulletPool.Release()`。池大小通常设为"最大同屏子弹数×1.2"作为安全余量。

**粒子特效**：爆炸效果由数十个粒子对象组成，若使用动态分配，一次爆炸即产生50次以上的堆分配。对象池将粒子预先存储为长度50的数组，`PlayExplosion()`仅需修改每个粒子的位置、速度、生命周期字段，GC分配次数为0。

**UI伤害数字弹出**：MMORPG中每次攻击触发一个飘动的伤害数字文本对象（TextMeshPro），若玩家AOE技能同时命中30个敌人，每秒可能产生数百个文本对象。使用对象池时，屏幕外消失的数字被立即归还池中供下一次弹出复用，保持UI帧率稳定。

**Unity 2021.1内置对象池**：Unity从2021.1版本起在`UnityEngine.Pool`命名空间中提供了官方的`ObjectPool<T>`类，其内部使用`Stack<T>`实现空闲列表，并提供`countAll`（总对象数）、`countActive`（借出数）、`countInactive`（池中可用数）三个只读属性供监控使用。

## 常见误区

**误区一：对象归还后不重置状态**  
将一个速度为`(100, 0, 0)`的子弹对象归还到池中，下次借出时若忘记在`Acquire()`内调用`Reset()`，该子弹会以上次的速度立即飞出，导致方向错误的子弹凭空出现。正确做法是将重置逻辑封装在池的`Acquire()`方法内部而非依赖调用方手动调用，避免因疏忽导致Bug。

**误区二：池大小设置为无限扩容等同于不用池**  
有些实现将溢出策略设为无限动态扩容，认为"反正能用"。但这在GC压力最大的战斗高峰期（对象池最易溢出时）反而引入了动态分配，恰好在最不应该触发GC的时候触发了GC。正确做法是通过性能分析工具（如Unity Profiler的Memory标签）测量峰值需求，将初始池大小设置为高于峰值需求的固定值。

**误区三：对所有对象都使用对象池**  
对象池仅对高频创建销毁的轻量级对象（子弹、粒子、音效播放器）有显著收益。对于每局游戏只创建一次的Boss对象或复杂场景物件，对象池引入了额外的管理代码却几乎没有GC收益，反而增加了内存常驻占用。选择标准是：若单类对象在1秒内的创建销毁次数超过10次，则值得引入对象池。

## 知识关联

**前置概念衔接**：理解对象池需要掌握GC优化中托管堆（Managed Heap）分代回收的原理——Gen0堆存储短生命周期对象，高频分配导致Gen0频繁触发，对象池的价值正在于将子弹、粒子等天然短命对象从Gen0回收循环中移除。策略模式为对象池的溢出策略（返回null / 扩容 / 循环复用）提供了结构支撑，三种策略可通过`IPoolOverflowPolicy`接口进行运行时切换。

**后续概念延伸**：空间分区（Spatial Partition）与对象池常常配套使用——空间分区负责高效查询场景中的活跃对象，而对象池管理这些对象的内存生命周期，两者共同支撑大量动态实体的高性能场景。池分配器（Pool Allocator）是对象池思想在原生内存层面的延伸，它直接管理固定大小内存块的分配与回收，绕过操作系统的`malloc`调用，是对象池从托管层下沉到非托管层的进阶形式。
