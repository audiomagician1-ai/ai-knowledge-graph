# 事件队列

## 概述

事件队列（Event Queue）是一种将消息的**发送时刻**与**处理时刻**解耦的设计模式。发送方将事件压入队列尾部，接收方在合适时机从队列头部取出并处理，两者无需同步等待。这与直接回调或函数调用的本质区别在于：队列引入了时间维度上的缓冲层，使得"触发"和"响应"可以发生在不同的游戏帧中。

该模式最早在操作系统的消息泵（Message Pump）设计中广泛使用——Windows 的 `GetMessage` / `DispatchMessage` 循环正是事件队列的经典实现。游戏编程领域将其引入，主要是为解决以下问题：当一个系统在帧中途触发事件时，若立即处理会导致栈深度失控或破坏当前帧的更新逻辑顺序，而将事件推迟到下一帧帧首统一处理则安全得多。Robert Nystrom 在《游戏编程模式》一书第15章对此有系统性论述。

在游戏中，事件队列的典型用途包括：音效请求去重（同一帧内多处触发"爆炸音效"只播放一次）、跨子系统通知（物理系统通知动画系统角色着地）以及 UI 事件的帧间缓冲。其重要性在于它是实现松耦合多系统协同的基础工具，而不仅是一个辅助机制。

## 核心原理

### 队列的数据结构与循环缓冲区

游戏中的事件队列通常使用**环形缓冲区（Ring Buffer）**而非动态数组，原因是游戏对内存分配极度敏感。环形缓冲区预分配固定大小的数组（如 128 个事件槽），用两个索引 `head` 和 `tail` 追踪读写位置：

```
入队：queue[tail % SIZE] = event; tail++;
出队：event = queue[head % SIZE]; head++;
队满条件：(tail - head) == SIZE
```

当 `tail - head == SIZE` 时队列已满，新事件应被丢弃或触发断言，而不是无限扩容。这种设计将单次入队/出队操作控制在 O(1) 时间复杂度，且零堆分配。

### 事件结构体的设计

每个事件通常是一个紧凑的 POD（Plain Old Data）结构体，包含事件类型枚举和联合体（union）形式的载荷数据：

```cpp
struct Event {
    EventType type;      // 枚举：SOUND_PLAY, COLLISION, UI_CLICK 等
    union {
        SoundId   soundId;
        EntityId  entityId;
        Vector2   position;
    } data;
};
```

使用 `union` 而非继承的原因是所有事件实例大小一致，环形缓冲区的内存布局才能保持连续且固定。若改用多态指针，则需要额外的堆分配，破坏了环形缓冲区的零分配优势。

### 帧循环中的处理时机

游戏主循环的标准写法会在每帧开头（或帧末）集中 drain（排空）队列：

```
while (game running):
    processInput()
    drainEventQueue()   ← 统一处理上一帧积累的事件
    update(dt)
    render()
```

`drainEventQueue` 会循环调用 `dequeue()` 直到队列为空。关键约束是：**在 drain 过程中产生的新事件不应在本轮 drain 中处理**，否则可能出现无限循环。实现方式是在 drain 开始时记录当前 `tail` 值，只处理到该位置为止。

### 与观察者模式的本质差异

观察者模式（Observer）中，`notify()` 调用是**同步的**——发布者调用 `notify` 时，所有订阅者的处理函数立即在同一调用栈上执行完毕。事件队列则是**异步的**：发布者调用 `enqueue` 后立即返回，订阅者的处理被推迟。这意味着事件队列中的事件到达处理时，发送方的状态可能已经改变，处理函数不能假设发送时的上下文仍然有效。

## 实际应用

**音效去重**：音频系统是事件队列最经典的游戏应用场景。假设一帧内角色踩到陷阱、碰撞墙壁、拾取道具各触发一次 `SOUND_PLAY(explosion)`，若直接调用则同一音效叠放三次。通过事件队列，可以在 drain 阶段遍历队列，对相同 `soundId` 的事件进行合并（dedup），只实际播放一次，避免音频爆音问题。

**物理到动画的跨系统通知**：物理引擎在模拟步骤中检测到角色刚体落地，此时动画系统可能正处于更新管线中间。直接调用动画 API 会打乱其内部状态机。将 `ENTITY_LANDED {entityId: 42}` 推入队列，动画系统在下一帧帧首处理该事件并触发落地动画，保证了系统边界的清晰。

**UI 按钮防抖**：移动端触摸事件在同一帧可能触发多个 `BUTTON_PRESSED`，事件队列的帧间缓冲天然起到防抖作用——只需在 drain 时检测队列中是否已存在同类型事件即可去重，无需在输入层额外实现防抖逻辑。

## 常见误区

**误区一：认为事件队列可以替代所有同步调用**。事件队列的异步特性意味着发送方无法获得返回值。对于需要立即得到响应结果的场景（如碰撞查询"当前位置是否可通行"），强行使用事件队列会导致逻辑被拆成两帧，引入单帧延迟，反而增加复杂度。事件队列适合"通知"语义，不适合"查询"语义。

**误区二：无限制地扩大队列容量**。初学者常将环形缓冲区改为动态数组，认为"永远不会丢事件更安全"。然而在游戏的实时系统中，如果队列持续增长说明处理速度跟不上生产速度，这是一个需要修复的架构问题，而不是通过扩容掩盖的问题。固定容量配合满队报警，能更早暴露系统负载异常。

**误区三：在 drain 循环内产生新事件并期望本帧处理**。如前文所述，不加保护地在处理函数内再次 `enqueue`，若新事件又触发相同处理函数，将导致当帧队列永远不为空，形成死循环。正确做法是在 drain 前快照 `tail`，或使用双缓冲队列（两个队列交替作为"当前处理"和"当前接收"）。

## 知识关联

**前置概念连接**：命令模式（游戏）将操作封装为对象，事件队列则是存储和调度这些命令对象的容器机制——命令模式解决"如何表达一个操作"，事件队列解决"何时执行这个操作"。组件模式将实体拆分为独立组件，各组件之间若使用直接引用通信会产生强耦合，而通过事件队列传递 `{type, entityId}` 可让组件间完全无需互相持有引用。

**后续概念连接**：服务定位器（Service Locator）常与事件队列配合——事件队列本身需要一个全局可访问的入口供各系统调用 `enqueue`，服务定位器提供了定位该入口而无需直接依赖具体实现的机制。掌握事件队列的帧间缓冲思想后，理解服务定位器中的"空服务（Null Service）"模式——在服务不可用时将请求静默丢弃——会更加自然。
