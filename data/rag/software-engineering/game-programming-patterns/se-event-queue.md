# 事件队列

## 概述

事件队列（Event Queue）是一种将事件的**发送时刻**与**处理时刻**解耦的设计模式。发送方将事件推入队列末尾，接收方在合适的时机从队列头部取出并处理。这种"先进先出（FIFO）"的缓冲机制使得游戏中不同系统之间可以异步通信，而不需要在事件发生的瞬间立刻响应。

该模式在游戏开发领域的雏形可以追溯到 GUI 框架时代——Windows 消息循环（Message Loop）从 1985 年的 Windows 1.0 起就以 `MSG` 结构体队列驱动整个程序。游戏引擎借鉴了这一思想，将事件队列嵌入游戏主循环，使音频、物理、AI 等子系统无需直接耦合就能相互通知状态变化。

事件队列与直接函数调用最本质的区别在于**时序隔离**：当角色死亡时调用 `PlayDeathSound()` 是同步耦合，而向队列压入 `{type: SOUND_PLAY, clip: "death.wav"}` 则允许音频系统在下一帧或音频线程中独立消费该事件，彻底消除调用栈上的跨系统依赖。

## 核心原理

### 环形缓冲区（Ring Buffer）实现

游戏中最常见的事件队列底层结构是**环形缓冲区**，而非动态数组。固定容量（如 256 个事件槽）预分配在内存中，用 `head` 和 `tail` 两个下标指针维护队列状态：

```
入队：buffer[tail] = event; tail = (tail + 1) % MAX_EVENTS;
出队：event = buffer[head]; head = (head + 1) % MAX_EVENTS;
队满条件：(tail + 1) % MAX_EVENTS == head
```

环形缓冲区避免了每帧动态分配内存，在 60 fps 的主循环中每帧最多产生数百个事件的场景下，这一点对 GC 压力至关重要。Unity 的 `UnityEvent` 内部处理以及 Unreal Engine 的 `UGameplayMessageSubsystem` 均采用类似的固定容量策略。

### 事件聚合（Event Aggregation）

队列天然支持**事件合并**：当同一帧内连续压入多个 `MOUSE_MOVED` 事件时，处理前可将它们折叠为一个，只保留最终位置。这与命令模式中命令对象的合并逻辑相似，但事件队列在批量消费阶段统一执行合并，而不是在每次 `execute()` 时判断。具体实现是在入队时检查 `tail` 前方是否存在相同 `type` 的待处理事件，若有则覆盖数据而非新增槽位。

### 帧间通信与处理时机控制

游戏主循环通常在每帧固定位置（典型做法是 `Update()` 开始前或结束后）统一清空事件队列，称为**泵送（Pumping）**。这一机制保证了帧内状态一致性：本帧物理模拟产生的碰撞事件不会在同一帧内立即触发角色状态机跳转，而是延迟到下一帧处理，避免同帧内级联修改导致的不确定行为。

帧间通信还涉及**优先级队列**变体：紧急事件（如玩家输入）可以插队到普通事件（如 NPC 对话触发）之前。此时底层结构从 FIFO 队列改为最小堆，以事件优先级值作为排序键。

### 多生产者/单消费者线程模型

在多线程游戏引擎中（如使用 Job System 的 Unreal Engine 5），物理线程、渲染线程可能同时向事件队列压入事件。此时入队操作需要**无锁原子操作（CAS, Compare-And-Swap）**保护 `tail` 指针，而主线程作为唯一消费者读取 `head` 时无需加锁，形成典型的 SPSC（Single Producer Single Consumer）或 MPSC（Multiple Producers Single Consumer）无锁队列模式。

## 实际应用

**游戏内音频系统**是事件队列最经典的应用场景。爆炸发生时，游戏逻辑线程压入 `{PLAY_SOUND, "explosion_large", pos: (120, 0, 45), volume: 0.8}`，音频线程以自己的速率消费队列，与渲染帧率完全解耦。若音频线程繁忙，事件在队列中等待，而游戏逻辑不会阻塞。

**成就系统**是另一个说明时序隔离价值的案例。当玩家击杀第 100 个敌人时，战斗模块仅压入 `{ENEMY_KILLED, enemyType: "goblin"}`，成就模块作为独立订阅者在队列泵送时接收此事件并判断是否解锁"百夫长"成就。战斗模块完全不知道成就系统的存在，二者通过事件队列实现零耦合通信。

**UI 事件路由**在手机游戏中也依赖此模式：触摸事件由操作系统回调压入队列，UI 框架在主线程帧起始处统一处理，防止触摸回调线程直接修改 UI 状态导致的竞态条件。

## 常见误区

**误区一：将事件队列等同于观察者模式**。观察者模式（Observer）是**同步**的——事件发生时立即遍历所有监听者并调用其回调，调用栈未返回前下一行代码不会执行。事件队列则是**异步**的，压入队列的操作立即返回，监听者在未来某帧才被调用。混淆两者会导致开发者误以为帧内压入的事件会在同帧产生效果，从而引入帧延迟 Bug。

**误区二：队列越大越安全**。将缓冲区容量设为 65536 并非保险做法。队列满载意味着事件生产速率长期超过消费速率，这是系统设计失衡的信号。正确做法是在入队时检测满队状态并记录日志或丢弃低优先级事件，而非无限扩容。《游戏编程模式》（Robert Nystrom, 2014）明确指出：队列积压是性能问题的诊断指标，而非通过增大容量掩盖的细节。

**误区三：事件队列可以替代所有跨系统通信**。对于需要**立即返回值**的查询（如"当前玩家血量是否大于 50"），事件队列无法胜任，因为它不携带同步返回通道。此类通信应当使用直接访问或服务定位器模式，而非强行异步化。

## 知识关联

**与命令模式的关系**：命令模式（游戏）将操作封装为对象，事件队列则为这些命令对象提供了延迟执行的传输通道。命令对象（如 `MoveCommand`）可以直接作为事件压入队列，由处理器在适当帧取出执行。事件队列相当于给命令模式增加了时序缓冲层。

**与组件模式的关系**：组件模式将游戏实体拆分为多个独立组件（物理组件、渲染组件、AI 组件），各组件之间天然需要通信但又不应直接持有彼此引用。事件队列为组件间通信提供了标准接口：物理组件压入碰撞事件，AI 组件订阅并消费，两者仅通过事件类型字符串或枚举值耦合，契合组件模式的低耦合设计目标。

学习完事件队列后，开发者掌握了游戏引擎中异步消息分发的完整机制，可以进一步探索 ECS（实体-组件-系统）架构中基于事件队列的系统间通信，以及 Unreal Engine 的 `Gameplay Message Router` 等工业级实现方案。
