# 空间分区

## 概述

空间分区（Spatial Partitioning）是游戏编程中用于加速空间查询的一类数据结构与算法集合，其核心思想是将游戏世界的连续空间递归或均匀地切割为离散区域，使得"距离查询"、"碰撞检测"、"视锥剔除"等操作的时间复杂度从暴力枚举的 O(n²) 降低到 O(n log n) 乃至 O(n + k)，其中 k 为实际命中数量。

这一思想的工程应用可追溯至1980年代早期的计算机图形学。Fuchs、Kedem 和 Naylor 于 **1980年** 正式发表了 BSP 树（Binary Space Partitioning Tree）论文，而四叉树（Quadtree）概念由 Finkel 与 Bentley 于 **1974年** 提出，最初用于二维点集的近邻搜索。此后这些结构被游戏工业大规模采用：id Software 在《毁灭战士》（1993）和《雷神之锤》（1996）中使用 BSP 树来实现静态几何体的快速可见性判断。

空间分区之所以重要，在于现代游戏场景动辄包含数千乃至数万个可交互实体，若每帧都对所有实体两两进行碰撞检测，60 FPS 下每秒需执行的检测次数为 n(n-1)/2 × 60，当 n=1000 时约为 3000 万次。空间分区将每次查询的候选集从全局缩减到局部邻域，是实时游戏物理与渲染管线得以运行的基础工程保障。

## 核心原理

### 均匀网格（Grid）

均匀网格将世界坐标系等分为固定尺寸的单元格，每个实体根据其 AABB（轴对齐包围盒）的中心坐标计算出所属格子索引：`index = floor(x / cellSize) + floor(y / cellSize) * gridWidth`。碰撞检测时只需检查目标格子及其相邻的 8 个（2D）或 26 个（3D）格子内的对象。均匀网格的插入与查询时间均为 O(1)，代价是内存占用随格子数量线性增长——当场景稀疏而分辨率高时会浪费大量空间。cellSize 的最佳选择通常等于场景中最大对象直径的 2 倍，以避免单个对象跨越过多格子。

### 四叉树与八叉树（Quadtree / Octree）

四叉树对二维空间进行自适应细分：每当一个节点中的对象数量超过预设阈值（常见值为 **4 或 8**），该节点分裂为四个等面积子象限。三维场景则使用八叉树，每次沿 X、Y、Z 三轴均分，产生 8 个子节点。深度通常限制在 **8～12 层**，防止极端分布下的无限递归。与均匀网格相比，四叉树在对象密度不均匀的场景（如开放世界地图）中更节省内存，但节点分裂与重平衡带来额外开销，动态对象移动时需要从旧节点删除并重新插入，实现复杂度高于网格。

### BSP 树（Binary Space Partitioning Tree）

BSP 树使用任意方向的超平面递归地将空间一分为二，并非局限于轴对齐切割。每个内部节点存储一个分割平面方程 `ax + by + cz = d`，左右子树分别对应平面正负两侧的几何体。BSP 树最大优势在于可以预计算出场景中所有多边形的**前后排序**，使渲染时不需要深度缓冲（画家算法），这正是《毁灭战士》能在没有硬件深度缓冲的 386 CPU 上实时渲染的关键。其代价是构建 BSP 树本身需要 O(n²) 时间且多边形可能被分割平面切割，故 BSP 树通常只适用于**静态几何体**的离线预处理。

### BVH（包围体层次结构，Bounding Volume Hierarchy）

BVH 将对象本身包裹在层次化的包围体（常用 AABB 或包围球）中，父节点的包围盒是所有子节点包围盒的并集。与 BSP 不同，BVH 的分割依据是**对象集合**而非空间平面，因此不会切割多边形，且对动态对象更友好——移动一个对象只需向上更新其祖先节点的包围盒，而非重建整棵树。现代实时光线追踪（如 NVIDIA RTX 的 DXR API）在硬件层面加速了 BVH 遍历，成为光追游戏的核心加速结构。SAH（Surface Area Heuristic，表面积启发式）是构建高质量 BVH 时最常用的分割策略，其目标是最小化 `C = C_trav + (SA(L)/SA(P)) × N_L × C_isect + (SA(R)/SA(P)) × N_R × C_isect`。

## 实际应用

**碰撞检测的宽阶段（Broad Phase）**：Unity 的 PhysX 集成默认使用 SAP（Sweep and Prune）结合 BVH 作为宽阶段加速，将潜在碰撞对筛选后再进行精确的窄阶段检测（GJK 算法）。在一个有 500 个刚体的场景中，宽阶段可将候选对数量从 124750 对压缩到数十对。

**视锥剔除（Frustum Culling）**：游戏引擎遍历场景的八叉树或 BVH，对每个节点的包围盒执行视锥体相交测试——若父节点完全在视锥体外，其所有子节点直接跳过。虚幻引擎 5 的 Nanite 系统在此基础上进一步使用 GPU 驱动的 HZB（Hierarchical Z-Buffer）剔除，将可见性测试并行化到数百万个 Cluster 级别。

**寻路网格的区域查询**：实时战略游戏中大量单位需要快速找到附近的敌人，通过均匀网格可将"半径 R 内的所有单位"查询限制在 `ceil(R/cellSize)²` 个格子内，《星际争霸 II》的 AI 系统使用类似机制每帧处理数百个单位的威胁评估。

## 常见误区

**误区一：所有场景都应使用最复杂的结构**。BVH 和 BSP 的构建与维护成本高，对于对象数量少于 100 个的简单场景，均匀网格或甚至暴力枚举（带简单排序）反而更快，因为避免了树遍历的 cache miss 开销。选择空间分区结构时，必须根据对象密度分布（均匀 vs 聚集）、动静比例（纯静态 vs 高频移动）来决策。

**误区二：空间分区可以直接替代对象池**。空间分区本身不管理对象的内存生命周期，它只存储对象的引用或指针。在高频创建/销毁对象的场景（如子弹、粒子）中，必须配合对象池使用：对象池负责内存复用，空间分区负责位置索引。若不配合对象池，每次从四叉树删除并重新插入一个新建对象都会触发堆分配，反而抵消空间分区带来的查询加速。

**误区三：四叉树天然优于均匀网格**。四叉树在**稀疏大地图**场景中内存效率高，但其树节点的指针间接访问模式导致严重的 CPU cache 不友好。现代游戏引擎在对象密度较高或内存不是瓶颈时，往往选择内存连续的均匀网格，因为其线性内存布局能充分利用 L1/L2 cache，实测吞吐量经常比四叉树高 2～5 倍。

## 知识关联

空间分区在工程实现上与**对象池**直接耦合：四叉树或 BVH 节点本身是频繁分配释放的小对象，应将节点对象预先存入对象池，避免每次树重平衡时触发系统堆分配，这与空间分区整体加速查询的目标形成闭环。

在渲染管线方向，空间分区是 LOD（Level of Detail）系统和遮挡剔除（Occlusion Culling）的前置基础：只有通过八叉树或 BVH 快速确定对象距摄像机的距离区间，LOD 系统才能高效地在不同精度网格间切换。在物理引擎方向，BVH 是现代连续碰撞检测（CCD）和光线投射（Raycast）的标准加速后端。掌握空间分区后，可进一步研究 GPU 并行化的空间结构，如 LBVH（Linear BVH，基于 Morton Code 排序）——它利用 Z 曲线将三维坐标映射为一维整数后并行构建，是现代实时光追引擎 BVH 构建速度达到毫秒级的核心技术。
