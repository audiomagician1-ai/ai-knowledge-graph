# 空间分区

## 概述

空间分区（Spatial Partitioning）是游戏编程中用于加速空间查询的一类数据结构模式，其核心思想是将游戏世界的空间递归地或规则地划分为更小的区域，使得碰撞检测、视锥裁剪、射线投射等操作的时间复杂度从 O(n²) 降低到 O(n log n) 甚至 O(log n)。对于拥有数千个对象的游戏场景，这种复杂度差异意味着每帧节省数十毫秒的计算时间，直接影响游戏是否能稳定运行在 60fps。

空间分区的思想最早可追溯到 1967 年 Schumacker 等人在飞行模拟器中使用的多边形排序技术，随后 1969 年 Warnock 算法正式将递归四叉分割引入计算机图形学。二叉空间分割树（BSP Tree）由 Fuchs、Kedem、Naylor 于 1980 年提出，后来成为《毁灭战士》（DOOM，1993年）和《雷神之锤》（Quake，1996年）的核心渲染技术。此后随着 3D 游戏的普及，四叉树、八叉树和层次包围盒（BVH）相继成为行业标准工具。

空间分区对于游戏开发者的实用价值在于：未经优化的暴力碰撞检测需要对每对对象执行 n(n-1)/2 次测试，当场景中有 1000 个对象时，每帧需要约 50 万次测试；而使用四叉树后，同样场景通常只需要几千次测试。这一数量级的差异使得开放世界游戏、RTS 大规模战斗等场景变得可行。

## 核心原理

### 均匀网格（Uniform Grid）

均匀网格是最简单的空间分区形式，将世界划分为等大小的格子，每个对象根据其位置注册到对应的格子中。查询某区域内的对象时，只需遍历该区域覆盖的格子列表。格子尺寸的选择至关重要：格子过大则每格内对象过多，查询退化为暴力搜索；格子过小则对象跨格情况增多，内存占用激增。**经验法则**是将格子边长设为最大对象直径的 2 倍左右。均匀网格的插入与删除复杂度均为 O(1)，查询复杂度为 O(k)（k 为目标区域内格子数），适合对象分布均匀的场景，如 2D 射击游戏中的子弹管理。

### 四叉树与八叉树（Quadtree / Octree）

四叉树将 2D 空间递归四分，八叉树将 3D 空间递归八分。每个节点存储一个区域，当该区域内对象数量超过阈值（通常为 4~8 个）时进行分裂。标准四叉树节点的判断条件如下：

```
if (node.objectCount > MAX_OBJECTS && node.depth < MAX_DEPTH):
    subdivide(node)  // 分裂为4个子节点
    redistribute(node.objects)
```

四叉树的查询复杂度为 O(log n + k)，其中 k 是结果集大小。其关键优势是**自适应密度**：对象密集区域（如城市中心）会自动产生更多层级，稀疏区域（如空旷原野）保持较少层级，避免均匀网格在稀疏区浪费内存的问题。Unity 引擎的 Physics2D 模块内部使用动态四叉树进行 2D 碰撞宽相检测。

### 二叉空间分割树（BSP Tree）

BSP 树用任意平面（在 2D 中为直线）将空间二分，分割平面通常沿着场景几何体的面片选取，因此 BSP 树能精确表示静态关卡几何体。DOOM 引擎利用 BSP 树在软件渲染时代实现了 O(n) 的可见面排序（Painter's Algorithm），无需深度缓冲。BSP 树的主要缺点是**构建代价高**（离线预处理可能耗时数分钟），且不适合动态对象，因此现代游戏仅将其用于静态关卡数据（如 Valve 的 .bsp 地图格式）。

### 层次包围盒（BVH，Bounding Volume Hierarchy）

BVH 将对象的包围盒（AABB 或 OBB）组织成树形层次，父节点的包围盒恰好包含所有子节点的包围盒。射线与场景的求交测试时，若射线不与父节点包围盒相交，则直接跳过整棵子树。BVH 的优势在于**支持动态对象**：移动对象只需更新从叶节点到根节点路径上的包围盒，复杂度为 O(log n)。Nvidia OptiX 和 DirectX Raytracing (DXR) 的硬件光线追踪加速结构正是 BVH 的硬件实现，RTX 显卡中的 RT Core 单元专门负责 BVH 遍历。

## 实际应用

**RTS游戏单位寻路与攻击范围检测**：《星际争霸II》场景中同时存在数百个单位，每个单位每帧需要查询攻击范围内的敌方单位。使用四叉树后，每次范围查询只需遍历附近少量节点，而非扫描全部单位列表。

**开放世界LOD管理**：《GTA V》等开放世界游戏使用八叉树管理场景对象的加载与卸载。摄像机视锥裁剪时，八叉树的节点包围盒与视锥体的相交测试可快速剔除视野外整块区域的渲染请求，减少 GPU draw call 数量通常超过 80%。

**物理引擎宽相检测**：Bullet Physics 和 Box2D 均提供 DBVT（Dynamic Bounding Volume Tree，动态 BVH 变体）作为默认宽相检测结构。宽相负责筛选**可能**发生碰撞的对象对，窄相再做精确几何测试。BVH 将宽相候选对数量从 O(n²) 压缩到实际碰撞附近的少量对象对。

**游戏地图射线投射**：FPS 游戏中，子弹射线每帧可能有数十条，使用 BSP 或 BVH 的射线求交，每次测试复杂度约为 O(log n)，而暴力测试所有多边形的复杂度为 O(n)。对于含 10 万面片的关卡，差异相当于每次射线测试从约 10 万次运算降至约 17 次。

## 常见误区

**误区一：四叉树总是比均匀网格好**。实际上，当对象分布高度均匀（如经典弹幕射击游戏中满屏子弹）时，均匀网格的常数开销更小，因为它没有树遍历和动态分裂的代价。四叉树的自适应性仅在对象分布**不均匀**时才体现优势。在对象分布均匀的场景中盲目使用四叉树，反而会因为频繁的节点分裂与合并操作引入额外开销。

**误区二：空间分区结构应频繁重建**。许多初学者在对象移动后完全重建整棵四叉树，这会使每帧重建的代价抵消查询优化的收益。正确做法是对动态对象使用**增量更新**：对象移动时，先从旧节点移除，再插入新节点，而非整体重建。对于高速移动对象，也可以每隔若干帧（如每 3 帧）才更新一次空间分区结构，用轻微的精度损失换取性能。

**误区三：空间分区能替代对象池**。空间分区管理的是对象的**空间索引**，与对象自身的内存分配无关。若子弹频繁创建销毁，即使有四叉树加速空间查询，堆内存分配仍是性能瓶颈。空间分区与对象池应当协同使用：对象池解决内存分配问题，空间分区解决空间查询问题，两者针对不同层次的性能开销。

## 知识关联

**与对象池的关系**：对象池（前置概念）解决了对象频繁创建销毁的内存碎片问题。在空间分区实现中，四叉树节点本身也会频繁创建和销毁（分裂与合并时），因此**四叉树节点应当从节点池中分配**，而非每次 new/delete。将对象池技术应用于空间分区的内部节点管理，能够消除树操作中的内存分配瓶颈，这是两个模式在实践中最直接的结合点。

不同空间分区结构之间存在明确的适用边界：均匀网格适合分布均匀的动态对象；四叉树/八叉树适合分布不均匀的动静混合场景；BSP 树适合静态关卡几何体的精确表示与渲染排序；BVH 适合支持光线追踪和动态碰撞检测的场景。现代游戏引擎（如 Unreal Engine 5）通常同时使用多种空间分区结构：用八叉树管理世界分区（World Partition）的流式加载，用 BVH 支持 Lumen 全局光照的光线追踪加速。
