# 更新方法

## 概述

更新方法（Update Method）是游戏编程中用于统一管理实体行为的一种设计模式。其核心思想是：游戏世界中的每一个活跃对象都实现一个名为 `update()` 的方法，游戏循环每帧调用所有对象的这个方法，使每个对象能够独立处理属于自己的那一帧逻辑，而无需主循环了解每个对象的具体行为细节。

该模式最早在1990年代的动作游戏和角色扮演游戏引擎中广泛出现，并在 Robert Nystrom 2014年出版的《Game Programming Patterns》一书中被正式命名和系统化阐述。它之所以重要，在于它将"每帧做什么"的决策权从主循环下放到了各个实体本身，彻底解耦了游戏对象的行为逻辑与帧驱动的调度机制。

在实际引擎中，Unity 的 `MonoBehaviour.Update()`、Unreal Engine 的 `AActor::Tick()` 以及 Godot 的 `_process()` 都是这一模式的直接体现，足以说明该模式在现代游戏开发框架中的普及程度。

## 核心原理

### 接口统一与多态调度

更新方法依赖于面向对象的多态机制。通常定义一个基类或接口，声明纯虚函数 `virtual void update() = 0`，所有游戏实体（如敌人、子弹、道具）继承该接口并实现各自的 `update()` 逻辑。主循环持有一个 `vector<Entity*>` 容器，每帧只需遍历调用 `entity->update()`，无需关心对象类型。这种调度方式使得添加新的实体类型时，主循环代码**零修改**。

### 帧时间与状态推进

每个 `update()` 方法通常接收一个参数 `double deltaTime`，即自上一帧到当前帧经过的秒数（例如 60fps 时约为 0.01667 秒）。对象内部的状态变更——例如骷髅巡逻兵向左移动，位置计算为 `x += speed * deltaTime`——都在这个方法内完成。对象自身维护私有状态变量（如当前动画帧编号、剩余冷却时间、行为状态机当前状态），`update()` 每帧推进这些状态，形成连续的行为表现。

### 对象生命周期管理的边界问题

更新方法本身不处理对象的创建与销毁，这是该模式一个需要特别注意的边界。当一个 `update()` 调用中途销毁了列表中的另一个对象（如子弹击中敌人），若主循环正在遍历同一容器，会导致迭代器失效。常见解决方案是引入"待删除队列"：在 `update()` 内标记对象为 `isDead = true`，主循环在本帧所有 `update()` 调用完成后，统一清除标记对象。这一延迟删除策略是更新方法在工程实现中的标准做法。

### 更新顺序依赖

对象在容器中的排列顺序决定了 `update()` 的调用顺序，而某些逻辑对顺序敏感。例如，物理碰撞检测依赖所有对象先完成位置更新，若对象 A 的 `update()` 读取了对象 B 尚未在本帧更新的位置数据，结果会有一帧延迟误差。解决此问题的方法是将更新分为多个阶段（如先统一更新物理，再统一处理逻辑），或采用双缓冲状态（Double Buffer Pattern）存储上一帧与当前帧的状态。

## 实际应用

**敌人 AI 行为**：一个格斗游戏中的 Boss 实体实现 `update()`，内部维护一个状态枚举 `{IDLE, PATROL, ATTACK, STUNNED}`。每帧的 `update()` 根据当前状态执行对应逻辑：处于 `PATROL` 时计算路径点移动，进入 `ATTACK` 时触发攻击动画并检测命中。整个 AI 行为树的每步推进都封装在这一个方法内，主循环不感知 Boss 处于哪个阶段。

**粒子系统**：烟雾、火焰等粒子效果中，每个粒子对象的 `update()` 负责将剩余寿命减去 `deltaTime`，更新透明度和位置，并在寿命归零后标记自身为待删除。一个典型的粒子系统可能同时存在数百个粒子对象，每个对象通过自己的 `update()` 独立演化，主循环无需为粒子写任何专用代码。

**游戏计时器与冷却**：技能冷却可通过在技能对象的 `update()` 中执行 `cooldown -= deltaTime` 实现，当 `cooldown <= 0` 时允许释放技能。这使得冷却逻辑完全自包含，不污染玩家输入处理代码。

## 常见误区

**误区一：认为 `update()` 越多越好，无限细分对象**。更新方法的调用本身有开销：虚函数调用、缓存不命中（Cache Miss）在对象数量达到数千时会显著影响性能。一个常见教训是将每颗草、每块地砖都做成独立的带 `update()` 的实体，导致帧时间暴增。正确做法是只有**需要逐帧变化行为**的对象才实现 `update()`，静态场景元素不应参与更新循环。

**误区二：在 `update()` 内部直接修改迭代中的实体列表**。如前文所述，在遍历容器时调用 `push_back()` 或 `erase()` 会破坏迭代器，导致未定义行为（Undefined Behavior）。许多初学者在子弹生成逻辑（`update()` 中 `spawn new Bullet`）时踩中此坑，程序随机崩溃。正确做法是使用"待添加队列"，在本轮遍历结束后统一合并。

**误区三：将 `update()` 与渲染逻辑混写**。更新方法应只负责状态变更（位置、血量、动画帧索引），绘制调用（`draw()`、`render()`）应分离在独立的渲染阶段执行。混写会导致逻辑帧与渲染帧耦合，无法实现固定物理步长（Fixed Timestep）与可变渲染帧率的分离架构。

## 知识关联

更新方法直接建立在**游戏循环**（Game Loop）之上：游戏循环提供每帧的驱动脉冲，而更新方法定义了每帧脉冲到达时各对象如何响应。没有游戏循环，更新方法的"每帧调用"语义就无从依附。理解游戏循环中 `deltaTime` 的计算方式，是正确使用 `update(deltaTime)` 参数的前提。

更新方法与**双缓冲模式**（Double Buffer）配合使用时，可解决更新顺序导致的一帧读写不一致问题——每个对象读取上一帧的快照状态，写入当前帧缓冲，彻底消除顺序依赖。此外，**组件模式**（Component Pattern）通常将更新方法下放到各组件（物理组件、AI组件、动画组件各自持有 `update()`），使单一实体的行为可以通过组合不同组件灵活构建，是更新方法在大型项目中的进阶应用形态。
