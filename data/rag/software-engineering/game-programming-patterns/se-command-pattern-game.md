# 命令模式（游戏）

## 概述

命令模式在游戏开发中是将"玩家意图"封装为独立对象的设计手法。与通用软件中的命令模式不同，游戏场景下的命令对象不仅记录"做什么"，还必须记录"谁在什么时间以什么参数执行"——这三个维度共同构成了游戏命令对象的完整语义。

命令模式最早由 GoF（Gang of Four）在 1994 年《设计模式》中提出，但其在游戏中的大规模应用是随 RTS（实时战略）游戏的网络对战需求而兴起的。《星际争霸》等早期 RTS 游戏正是依赖类似"命令帧同步"机制在拨号网络上实现多人对战的。Robert Nystrom 的《游戏编程模式》（2014）将其列为游戏架构中价值最高的几个模式之一，专门强调了其在输入映射、撤销和回放三个游戏专有场景下的威力。

在游戏中，命令模式解决的核心矛盾是：玩家的原始输入（按键/鼠标/手柄）与游戏角色的具体行为之间存在多对多的动态映射关系，而且这种映射必须可被记录、回溯和传输。如果没有命令对象作为中间层，撤销一步操作或回放一段录像将需要侵入角色的所有状态逻辑。

## 核心原理

### 命令接口的最小设计

游戏命令对象通常只需要实现两个方法：`execute()` 和 `undo()`。`execute()` 执行动作并在对象内部保存执行前的状态快照（例如角色的位置坐标 `(x0, y0)`），`undo()` 利用这个快照将世界还原。

```cpp
class MoveCommand : public Command {
    Actor* actor_;
    float dx_, dy_;
    float prevX_, prevY_;  // 执行前快照
public:
    void execute() override {
        prevX_ = actor_->x();
        prevY_ = actor_->y();
        actor_->moveTo(actor_->x() + dx_, actor_->y() + dy_);
    }
    void undo() override {
        actor_->moveTo(prevX_, prevY_);
    }
};
```

这一结构使命令对象同时充当"操作记录"和"快照容器"，是游戏中区别于一般命令模式的关键设计决策。

### 输入映射层

游戏中输入处理器不直接调用 `actor->jump()`，而是维护一张"按键→命令工厂"的映射表。当按下空格键时，系统从表中取出 `JumpCommand` 实例并调用 `execute()`。改键位只需修改映射表，角色逻辑代码零改动。更重要的是，同一个 `JumpCommand` 类可以绑定到不同的 Actor，实现"玩家1控制A角色、玩家2控制B角色、AI 也发出相同格式的命令"三者共用同一套命令体系。

典型的映射结构如下：
```
InputHandler::handleInput() {
    if (isPressed(BUTTON_A)) return new JumpCommand();
    if (isPressed(BUTTON_B)) return new DuckCommand();
}
```
`handleInput()` 返回命令对象而不是立即执行，这一延迟执行特性是实现回放和网络同步的物质基础。

### 撤销栈与回放队列

游戏中的撤销系统使用一个有界栈（通常容量为 10～50 步）存储已执行的命令对象。每次调用 `execute()` 后将命令压栈；调用撤销时弹栈并调用 `undo()`。重做（Redo）则维护第二个栈，当撤销时将命令从撤销栈移入重做栈，重做时反向操作。

回放功能的实现逻辑更简洁：录制时将每帧产生的命令序列（连同帧编号）序列化存储；回放时从初始状态开始，按帧编号顺序重新调用每个命令的 `execute()`。《星际争霸》的录像文件本质上就是一系列玩家命令的时间戳记录，而非画面数据，文件通常仅有几十 KB。

### 网络帧同步中的命令传输

在帧同步网络模型中，每位玩家的本地输入处理器在每个逻辑帧（通常 16ms 或 33ms）将命令对象序列化后广播给所有客户端。每个客户端收齐本帧所有玩家的命令后，按相同顺序执行，保证游戏状态一致。命令对象必须是确定性的（Deterministic）——相同输入在任何机器上产生完全相同的状态变化，这要求命令内部不能使用 `rand()` 或浮点数精度不确定的运算。

## 实际应用

**策略游戏的单位操作**：玩家框选10个士兵并右键下达移动指令时，系统生成10个 `MoveToCommand` 对象，每个对象绑定一个士兵和目标位置。若玩家撤销，10个命令依次出栈，所有士兵回到移动前的位置，无需外部保存任何状态。

**格斗游戏的输入缓冲**：街霸、铁拳等格斗游戏允许玩家提前输入下一个动作（Buffer Window 通常为 3～6 帧）。这通过将命令对象放入一个短暂的命令队列实现：当前动作还在执行时，命令对象在队列中等待，而不是直接被丢弃。

**关卡编辑器的完整撤销**：Unity、Unreal 等引擎的编辑器中每个编辑操作（移动物体、修改材质、添加组件）都是一个命令对象。Unreal Engine 内部的 `UEditorEngine` 维护一个 `UTransactionBuffer`，其本质就是一个支持多级撤销的命令历史栈，默认深度为 100 步。

## 常见误区

**误区一：命令对象应该只保存"操作"而不保存"状态"**
初学者常认为命令对象越轻量越好，不愿在其中存储快照数据。但游戏撤销功能的实现要求命令对象在 `execute()` 时必须记录足够的前置状态，否则 `undo()` 无处可还原。对于复杂操作（如删除一个含有子节点的场景树节点），命令对象可能需要序列化整个子树——这是正确设计，而非过度设计。

**误区二：回放只能通过录制画面实现**
很多初学者将游戏回放与视频录制混淆。基于命令模式的回放录制的是"输入流"而非"像素流"，存储成本差异是数量级级别的。但这也意味着回放的正确性完全依赖于游戏逻辑的确定性——如果游戏存在任何非确定性计算（如物理引擎使用了与平台相关的浮点行为），命令回放将产生偏差，最终画面与原始录制不同。《魔兽争霸3》早期版本的录像文件在特定版本更新后就因此出现乱码问题。

**误区三：命令模式可以直接代替状态机处理角色行为**
命令模式描述的是"离散的意图触发"，而角色状态机描述的是"持续状态下的行为逻辑"。奔跑、跳跃的物理更新每帧都需要执行，这属于状态机的职责；而"发起一次跳跃"的触发才是命令的职责。两者是互补关系：命令对象将状态机从当前状态推入下一状态，但状态机内部的持续逻辑不是命令的管辖范围。

## 知识关联

**前置概念——命令模式（GoF）**：游戏中的命令模式在通用命令模式的 `execute()` 接口基础上，强制要求实现 `undo()` 方法，并将"接收者（Receiver）"从构造时注入改为执行时注入（传入 Actor 引用），以支持同一命令类绑定不同角色。

**后续概念——事件队列**：命令对象的延迟执行特性（`handleInput()` 只返回命令而不立即执行）自然导向事件队列模式。当命令需要跨帧延迟、优先级排序或在不同线程间传递时，将命令对象放入事件队列是顺理成章的演进。事件队列本质上是一个异步的命令调度器，其中每个事件可以看作一个待执行的命令对象。
