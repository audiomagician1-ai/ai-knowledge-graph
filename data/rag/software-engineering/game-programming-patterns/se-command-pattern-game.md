# 命令模式（游戏应用）

## 概述

命令模式在游戏开发中，是将"玩家意图"与"游戏逻辑执行"彻底解耦的编程手法。其核心做法是：把每一个可执行的操作（移动、攻击、使用道具）封装成一个独立的对象，该对象携带执行所需的全部数据，而不是在输入检测处直接调用游戏逻辑函数。

命令模式在游戏领域的广泛应用可追溯到1990年代的策略游戏与RPG开发。Robert Nystrom在2014年出版的《Game Programming Patterns》中将其列为游戏中最具实用价值的设计模式之一，并以"输入处理"章节为切入点详细阐述了其四种核心应用场景。

命令模式在游戏中的价值不止于代码整洁：它直接**使能**了撤销、回放、AI控制以及网络同步四大功能。这四个功能若不借助命令对象，实现成本将以指数级上升。

## 核心原理

### 输入映射与按键重绑定

传统游戏代码直接在输入轮询处写死逻辑：

```cpp
if (isKeyDown(KEY_W)) player.moveForward();
```

命令模式的改法是将按键与命令对象分离：

```cpp
Command* buttonW = new MoveForwardCommand();
// 玩家重绑定时只需：
buttonW = new MoveBackCommand();
```

每帧的输入处理函数返回一个 `Command*`，主循环负责调用 `command->execute(actor)`。这样一来，同一套命令对象既可由玩家键盘触发，也可由AI逻辑或网络消息触发，角色（actor）对象本身完全感知不到命令来源。

### 撤销（Undo）机制

命令对象在 `execute()` 之外额外实现一个 `undo()` 方法，并在 `execute()` 时保存执行前的状态快照。游戏维护一个命令历史栈（`std::stack<Command*>`）：执行时压栈，撤销时弹栈并调用 `undo()`。

以回合制战棋游戏为例，`MoveUnitCommand` 在执行时记录单位的起始坐标 `(fromX, fromY)`；`undo()` 被调用时，将单位移回该坐标并恢复其移动点数。这种设计只需命令对象保存**差量数据**（delta），而非整个游戏世界快照，内存开销极小。

### 回放系统

游戏回放的本质不是录制每一帧的画面，而是录制每一帧的命令序列。StarCraft、Warcraft III的.rep回放文件，存储的正是各玩家在每个游戏逻辑帧提交的命令流（操作类型 + 目标参数 + 时间戳），文件体积因此仅为视频录制的万分之一量级。

回放时，游戏以相同的随机数种子重新启动，并按时间戳顺序重新 `execute()` 录制的命令，游戏世界便完全确定性地重演。这要求命令对象必须是**纯数据**的——不能包含对当前帧运行时状态的指针引用，否则回放时指针已失效。

### 网络同步

RTS与格斗游戏的延迟锁步（Lockstep）网络模型，正是命令模式的直接应用。每个客户端在一个逻辑帧内收集本地命令，广播给所有对等端，待所有端的命令汇齐后，各端以相同顺序 `execute()` 这批命令，从而保持游戏状态完全一致，无需传输任何状态数据。

格斗游戏的帧优势机制（frame advantage）中，每一帧的攻击/防御命令必须精确同步，命令对象通常携带字段：`uint8_t playerId`、`uint16_t frameNumber`、`uint8_t actionType`，总大小仅4字节，极大降低网络带宽需求。

## 实际应用

**编辑器中的Ctrl+Z**：Unity编辑器的操作撤销系统，每次用户变换物体位置、修改材质，引擎内部都生成一个 `UndoPropertyModification` 命令对象并压入编辑器历史栈。Unity文档明确指出其编辑器支持最多200步撤销历史。

**RPG技能回放**：《黑暗之魂》类游戏的死亡回放功能，记录玩家最后N秒的输入命令序列，死亡时用幽灵角色重新执行这批命令，展示死亡过程。回放角色与真实角色共享命令执行逻辑，无需额外代码。

**AI行为复用**：命令模式使AI单位与玩家角色共享完全相同的 `execute()` 路径。AI行为树输出的是命令对象，而非直接操纵角色状态，这意味着AI的行为自动具备了撤销与回放能力，可用于AI调试录制。

## 常见误区

**误区一：认为命令对象必须是无状态的**。恰恰相反，支持撤销的命令对象**必须携带状态**——即执行前的快照数据（如移动前坐标、攻击前HP值）。无状态命令只能支持执行，无法支持撤销。

**误区二：将命令模式与事件系统混为一谈**。命令对象是主动调用的、有明确执行者（actor）的操作封装，调用方知道命令会被执行；事件是被动广播的通知，发布方不关心有没有人处理。游戏输入流水线中，命令先于事件产生——输入产生命令，命令执行后可能**派发**事件。

**误区三：认为回放只需录制随机数种子即可省去命令录制**。仅记录随机数种子只对完全确定性游戏有效。一旦游戏逻辑中存在浮点数运算的跨平台差异、多线程时序不确定，或中途加载存档等操作，纯种子回放立刻失效；命令录制则始终有效，因为它直接记录的是意图而非推演结果。

## 知识关联

**前置概念——命令模式（通用）**：通用命令模式定义了 `Command` 接口及 `execute()` 方法签名；游戏应用在此基础上强制要求 `undo()` 方法，并引入命令历史栈和命令序列化需求，是对通用模式的有方向性的扩展。

**后续概念——事件队列**：命令模式解决的是"如何封装单次操作"，而事件队列解决的是"如何在时间轴上调度一批命令/事件的执行顺序"。理解命令对象的生命周期（创建→入队→执行→销毁/入历史栈）是理解事件队列调度逻辑的直接铺垫——事件队列本质上是命令对象的异步执行容器。
