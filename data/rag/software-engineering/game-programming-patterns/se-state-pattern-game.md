# 状态模式（游戏）

## 概述

状态模式在游戏编程中特指用有限状态机（Finite State Machine，FSM）来管理角色、AI或游戏对象的行为逻辑。其核心思想是：一个游戏实体在任意时刻只处于一种明确的状态（如"站立"、"跳跃"、"攻击"），每种状态封装了该状态下的输入响应逻辑和行为表现，状态之间通过明确定义的转换条件互相切换。

这一模式在游戏编程领域的普及源于1990年代的动作游戏开发实践。早期横版格斗游戏（如《街头霸王》系列）的角色控制代码面临"意大利面式if-else链"的困境——当角色状态超过10种时，嵌套条件判断变得极难维护。开发者随后将软件工程中的经典状态模式引入游戏角色控制，形成了今天游戏FSM的标准实践。Robert Nystrom在《游戏编程模式》（2014年出版）一书中将此列为游戏程序员必须掌握的第一个行为模式。

在实际游戏项目中，几乎每一个有行为逻辑的游戏实体都需要FSM：玩家角色的动作状态机、敌人AI的决策状态机、UI界面流转、动画控制器本质上都是FSM的具体体现。Unity引擎的Animator Controller本质上就是一个内置的FSM可视化编辑器。

## 核心原理

### 有限状态机的数学模型

游戏FSM可以形式化定义为一个五元组：**FSM = (S, Σ, δ, s₀, F)**，其中：
- **S** 是有限状态集合，例如 {Idle, Run, Jump, Attack, Dead}
- **Σ** 是输入符号集（触发条件），例如 {按跳跃键, 落地, 受伤, 血量为0}
- **δ: S × Σ → S** 是状态转换函数，定义"当前状态 + 输入 = 下一状态"
- **s₀** 是初始状态，通常为 Idle
- **F** 是终态集合（如 Dead 状态）

以一个具体的平台跳跃游戏角色为例，转换函数δ的部分映射如下：δ(Idle, 按方向键) = Run，δ(Run, 按跳跃键) = Jump，δ(Jump, 碰撞地面) = Idle。这个映射表完整定义了角色的所有合法行为转换，任何不在表中的转换都不会发生。

### 代码实现方式

游戏中FSM最常见的实现分为两代：**枚举+switch版本**和**状态类（State Class）版本**。

枚举+switch版本将所有状态行为写在一个大型switch语句中，每个case对应一种状态。这种写法简单直接，适合状态数量少于5种的简单对象，但状态一旦增多（超过8-10种），代码文件会膨胀到数百行且难以扩展。

状态类版本遵循软件工程状态模式的标准结构：定义一个`IState`接口，包含`Enter()`、`Execute()`、`Exit()`三个核心方法；每种状态实现为一个独立的类（如`IdleState`、`JumpState`）；角色类持有一个`currentState`引用，每帧调用`currentState.Execute()`。当状态切换时，先调用旧状态的`Exit()`，再调用新状态的`Enter()`，最后更新`currentState`引用。Unity官方的角色控制器示例代码正是采用这一结构。

### 层次状态机与并行状态机

基础FSM存在**状态爆炸**问题：若角色有"地面/空中"两种大状态，每种大状态下有5种小状态，则总状态数为2×5=10种，且存在大量重复逻辑。**层次状态机（Hierarchical FSM，HFSM）**通过状态嵌套解决这个问题，子状态可以继承父状态的默认行为，只覆盖差异部分。《光环》系列的敌人AI就使用了多层HFSM来管理战术决策与具体动作的分层逻辑。

**并行状态机**允许一个实体同时运行多个独立的FSM，例如角色的"移动状态机"和"武器状态机"可以完全独立运转，避免因两个维度的组合而产生状态爆炸。

## 实际应用

**玩家角色控制**是FSM最典型的应用场景。以《超级马里奥》风格的平台游戏为例，Mario的状态机至少包含：Idle、Walk、Run、Jump、Fall、Crouch、Die共7个状态，以及至少12条转换边。每次按键输入都会被当前状态的`HandleInput()`方法捕获，决定是否触发转换。"在空中按攻击键"和"在地面按攻击键"可以映射到完全不同的行为，而不需要在动作逻辑中埋入大量`if(isGrounded)`判断。

**敌人AI决策**是FSM的另一个核心战场。一个标准的巡逻敌人拥有三种状态：Patrol（沿路径巡逻）、Chase（追逐玩家）、Attack（攻击玩家）。转换条件为：玩家进入感知范围（圆形半径通常设为8-15个单位）触发Patrol→Chase，玩家进入攻击范围（1-3个单位）触发Chase→Attack，玩家离开感知范围触发Chase→Patrol。《吃豆人》中的四个鬼魂（Blinky、Pinky、Inky、Clyde）每个都有Scatter、Chase、Frightened三种状态，状态切换由全局计时器驱动，这是游戏史上最著名的FSM AI设计之一。

**UI流程管理**同样依赖FSM：主菜单、游戏中、暂停、游戏结算等界面之间的切换逻辑，用FSM管理可以避免界面状态混乱、按钮响应错误等常见bug。

## 常见误区

**误区一：在FSM内部直接引用其他状态进行硬编码跳转。** 许多初学者在`JumpState`的代码中直接写`character.currentState = new LandState()`，这使得每个状态类都持有对其他状态类的直接依赖，修改任何一个状态都可能引发连锁修改。正确做法是通过状态机的`Transition()`方法或传递状态标识符（枚举值）来切换，保持状态类之间的解耦。

**误区二：用FSM处理需要记忆历史的复杂行为。** FSM是"无记忆"的——它不记录如何到达当前状态。当敌人AI需要"在被攻击3次后变为狂暴状态"这类需要计数的逻辑时，单纯依赖FSM转换条件会导致设计复杂化。正确做法是在角色数据层维护计数器变量，FSM只负责读取该变量作为转换条件，两者职责分离。

**误区三：将所有行为都塞入同一个FSM。** 角色的"生命值状态（正常/受伤/濒死）"和"动作状态（奔跑/跳跃/攻击）"是两个正交维度，若强行合并，会产生"受伤奔跑"、"受伤跳跃"等组合状态，导致状态数量以乘法方式爆炸。识别出独立的行为维度并使用并行FSM分别管理，是游戏FSM设计的重要技巧。

## 知识关联

**前置知识衔接**：通用状态模式（GoF设计模式）定义了`Context`、`State`接口和`ConcreteState`的三角关系，游戏FSM的状态类版本直接继承了这一结构，`Character`类扮演`Context`角色，`IState`接口即为`State`接口。理解游戏编程模式概述中"将行为与数据分离"的核心思想，有助于理解为何将每种角色状态封装为独立类而非集中在角色类内部。

**横向关联**：游戏FSM与动画系统高度耦合，Unity的Animator Controller使用与代码FSM相同的状态-转换模型，两者通常需要同步设计。行为树（Behavior Tree）是FSM在敌人AI领域的进化替代品，《光晕3》的AI系统从FSM迁移到行为树是游戏AI发展史上的标志性事件，了解FSM的局限性（状态爆炸、转换线爆炸）有助于理解行为树解决了哪些具体问题。
