# 脏标记

## 概述

脏标记（Dirty Flag）是一种通过布尔状态位来追踪数据是否发生变化、从而避免不必要重复计算的优化模式。其核心思路是：用一个额外的 `bool` 字段（即"脏位"）记录某份数据自上次计算结果被缓存以来是否被修改过。只有当脏位为 `true` 时，才重新执行开销较大的推导计算；否则直接返回缓存值。

该模式最早可追溯到操作系统的页表管理机制，CPU 的 TLB（Translation Lookaside Buffer）使用 Dirty Bit 标记内存页是否被写入，以决定是否需要回写磁盘。在游戏编程领域，Robert Nystrom 在其 2014 年出版的《Game Programming Patterns》一书中将其作为一种独立的设计模式加以归纳，使该模式在游戏开发社区中广泛传播。

脏标记在游戏编程中的价值体现在以下具体场景：场景图（Scene Graph）中每个节点的世界坐标需要由本地变换矩阵与祖先节点的变换矩阵连乘得到，若每帧都对所有节点重新计算这一连乘结果，计算量随节点数量线性增长；使用脏标记后，只有被移动过的节点分支才触发矩阵重算，可将实际计算量降低至仅需更新的子树范围。

## 核心原理

### 两类状态的分离

脏标记模式将数据分为两类：**原始数据**（Primary Data）和**派生数据**（Derived Data）。原始数据是直接由用户操作或逻辑修改的值，例如游戏对象的本地坐标 `localPosition`；派生数据是由原始数据经过某种计算得到的结果，例如世界坐标 `worldPosition`（需要将本地坐标与父节点世界坐标组合）。脏标记 `isDirty` 与派生数据绑定，一旦原始数据被写入，立即将 `isDirty` 置为 `true`，而不立刻触发重新计算。

### 延迟计算（Lazy Evaluation）机制

读取派生数据时，检查 `isDirty`：若为 `false`，直接返回已缓存的 `cachedWorldPosition`；若为 `true`，则执行矩阵变换计算，将结果存入缓存，同时将 `isDirty` 重置为 `false`。这种"写时标记、读时计算"的流程用伪代码表示如下：

```
Vector3 getWorldPosition():
    if isDirty:
        cachedWorld = parent.getWorldPosition() * localTransform
        isDirty = false
    return cachedWorld
```

其中 `*` 表示矩阵乘法组合变换。该模式将计算时机从"写操作时"推迟到"读操作时"，当一帧内某节点被修改多次但只被读取一次时，计算次数从 N 次（每次写操作各一次）降低为 1 次。

### 脏标记的传播规则

在层级结构中，父节点的原始数据变化必须向下传播脏状态到所有子节点。具体策略有两种：**即时传播**（在父节点被修改时，递归将所有子孙节点的 `isDirty` 置 `true`）和**惰性传播**（子节点在读取时检查父节点是否为脏，再决定是否重算）。即时传播在节点数量极大时可能造成一次修改触发大量标记写操作；惰性传播则将开销分摊到读取时刻，但需要向上递归查询父链。大多数游戏引擎（如 Unity 的 Transform 组件）采用即时传播策略，因为读操作通常比写操作更频繁，需要保证读取路径最短。

### 触发条件与清除时机

脏位被置为 `true` 的时机称为**污染点**（Dirtying Point），被重置为 `false` 的时机称为**清除点**（Cleaning Point）。两个常见清除时机是：①每次读取时按需清除（On-demand），适合读操作分散的场景；②每帧统一清除（Per-frame batch），适合渲染前统一提交变换矩阵的流程，例如在 `LateUpdate` 阶段统一遍历脏节点列表并重算，避免同一帧内多次读取反复触发计算。

## 实际应用

**Unity Transform 组件**：Unity 引擎内部的 `Transform` 组件使用脏标记追踪本地变换变化。当调用 `transform.position = newPos` 时，引擎内部将该节点及其所有子节点标记为脏，下一次访问 `localToWorldMatrix` 时才重新计算 4×4 变换矩阵。这也是为什么在性能敏感代码中应缓存 `transform.position` 读取结果，而非在循环中反复访问。

**文字排版引擎**：字体渲染器（如游戏 UI 系统中的文字组件）将文字布局结果（字符位置、换行点）视为派生数据。当字符串内容或字体大小改变时置脏，仅在实际渲染前重新执行昂贵的文字排版算法（时间复杂度约为 O(n²) 的换行计算），避免每次更新文字内容都立即执行排版。

**保存文件状态提示**：桌面应用程序（如文本编辑器）的"文件已修改未保存"提示同样是脏标记的直接应用：`isDocumentDirty` 在任何编辑操作后置 `true`，在执行保存操作后清除为 `false`，窗口标题的星号（`*`）仅反映该布尔值，不涉及任何实际文件比对。

## 常见误区

**误区一：脏标记总是比直接计算快**。脏标记本身引入了额外的内存读写（每次写操作需额外写一个 `bool`）和条件分支（每次读操作需检查 `bool`）。当派生数据的计算本身非常廉价（例如两个整数相加），且每次写操作后几乎必然紧跟一次读操作时，脏标记的开销可能超过它节省的开销。该模式仅在**写操作远多于读操作**，或**派生计算代价远大于标记维护代价**的场景下有净收益。

**误区二：忘记在层级结构中传播脏标记**。仅将直接被修改的节点置脏，而忽略其子节点，会导致子节点缓存的世界坐标基于过时的父节点数据，产生视觉错位 Bug。这类 Bug 通常在运行时表现为子对象位置短暂"滞后"一帧跟随父对象，难以在静态代码审查中发现。正确实现要求：任何可能影响派生数据的原始数据变更，都必须覆盖所有依赖该数据的下游节点。

**误区三：将脏标记与观察者模式混淆**。观察者模式在数据变化时**主动推送通知**给监听者，立即触发响应逻辑；脏标记则是**惰性拉取**——只记录"有变化"这一事实，等到外部主动请求派生数据时才消费这一记录。两者方向相反：前者是 Push，后者是 Pull。混淆两者会导致在需要即时响应的场景（如碰撞检测触发器）错误地使用脏标记，造成一帧延迟。

## 知识关联

脏标记与**缓存模式**（Caching Pattern）密切相关：缓存模式关注如何存储和复用计算结果，而脏标记解决的是"何时令缓存失效"这一配套问题——没有脏标记的缓存会永久返回过时数据，没有缓存的脏标记只有标记而无值可返回。两者通常成对出现。

在游戏场景图的实现中，脏标记通常与**组件模式**（Component Pattern）结合使用：Transform 组件持有脏位和缓存的世界矩阵，Entity 持有 Transform 组件，脏位的传播沿 Entity 的父子关系链进行，形成脏标记在组件化架构中的标准用法。

对于需要追踪**多种独立变化**的场景，单一布尔脏位可扩展为**脏位掩码**（Dirty Mask），使用整型的各个二进制位分别代表不同属性的脏状态（如位 0 = 位置已变，位 1 = 旋转已变，位 2 = 缩放已变），从而实现更细粒度的按需重算，避免修改旋转时不必要地重新计算不依赖旋转的派生量。
