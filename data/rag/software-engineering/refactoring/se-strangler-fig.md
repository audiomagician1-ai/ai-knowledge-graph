# 绞杀者模式

## 概述

绞杀者模式（Strangler Fig Pattern）由 Martin Fowler 于 2004 年在其博客文章《Strangler Application》中正式提出，得名于热带雨林中的绞杀榕树——这种植物从宿主树的顶部开始生长，逐渐向下包裹宿主，最终完全替代宿主树并独立存活。这一生物学现象与软件系统替换的过程高度类似：新系统像藤蔓一样逐步包裹并接管旧系统的功能，直到旧系统可以被安全移除。

绞杀者模式的核心思路是**渐进式替换**，而非一次性的"大爆炸重写"（Big Bang Rewrite）。大爆炸重写要求团队在某个时间点冻结旧系统开发、全力构建新系统，历史上此类项目有极高的失败率——Netscape 4.x 到 Mozilla 的重写耗时近三年且差点拖垮公司，就是著名的反面案例。绞杀者模式通过让新旧系统并行运行、逐步转移流量，将风险分散到若干个小的、可回滚的迁移步骤中。

这一模式在现代软件工程中尤其适用于将单体应用（Monolith）迁移到微服务架构的场景。由于可以按业务功能模块逐一迁移，团队能在不中断线上服务的前提下持续交付价值，每次迁移一个有界上下文（Bounded Context），使整个替换周期可以跨越数月甚至数年而不影响业务连续性。

## 核心原理

### 代理层拦截与路由

绞杀者模式的关键技术手段是在旧系统前引入一个**路由代理层**（Strangler Facade），所有请求先到达该代理层，再由代理决定转发给旧系统还是新系统。这个代理层通常以反向代理（如 Nginx）或 API 网关实现。初始状态下 100% 流量流向旧系统；每当新系统完成某一功能模块的实现并通过验证，代理层的路由规则即更新，将对应路径的流量切换至新系统。这种方式保证了任意时刻系统对外表现一致，调用方无需感知替换过程。

### 功能分块与优先级排序

绞杀者模式要求将遗留系统按**功能边界**切分为若干独立的迁移单元，每个单元具备独立的输入/输出接口。切分粒度通常以领域驱动设计中的有界上下文为依据，也可按 HTTP 路径前缀或消息队列主题划分。优先迁移的候选模块通常满足以下条件之一：变更频率高（遗留代码中频繁出问题的模块）、业务价值大、与其他模块耦合度低。低耦合意味着迁移时需要处理的接口契约数量少，可显著降低迁移工作量。

### 双写与数据同步策略

当新旧系统并行运行时，数据一致性是最难处理的问题。绞杀者模式通常配合**双写策略**：写操作同时写入旧数据库和新数据库，读操作根据路由规则决定从哪个数据库读取。在流量切换初期，可采用"新系统写、旧系统为主读"的模式，通过比对两侧读取结果来验证新系统的正确性（即 Shadow Mode 验证）。双写阶段结束后，旧系统的写路径关闭，数据完全由新系统管理，此时才可以着手清理旧系统代码和旧数据库表。

### 绞杀完成判定

当路由代理层中所有规则均指向新系统、旧系统不再接收任何生产流量，且旧系统的数据存储已完成迁移时，绞杀过程宣告完成。此时代理层本身也可被移除或简化为普通的负载均衡器，旧系统代码库可以归档删除。一般建议旧系统在零流量状态下保留至少 2～4 周的观察窗口，确认无任何回滚需求后再执行删除操作。

## 实际应用

**电商平台单体迁移微服务**：某零售公司将其运行了 10 年的 Java EE 单体应用迁移至基于 Spring Boot 的微服务架构。团队首先在 Kong API 网关前部署绞杀者代理，第一阶段仅迁移"商品目录"模块（对应 `/api/products/*` 路径），历时 6 周上线。随后按季度节奏依次迁移"购物车"、"订单"、"支付"模块，整个迁移历时 18 个月，旧系统在此期间从未中断服务。

**银行核心系统替换**：一家欧洲银行使用绞杀者模式替换其 COBOL 核心银行系统。由于监管要求不允许服务中断，团队在 IBM MQ 消息层之上实现路由代理，先将"账户查询"这一只读功能迁移至新系统，验证数据一致性达标（新旧系统查询结果差异率低于 0.001%）后，再逐步开放"转账"等写操作的迁移。

**前端 AngularJS 到 React 迁移**：绞杀者模式同样适用于前端重构。通过在主路由层按页面路径分流，保留旧 AngularJS 应用处理未迁移页面，新 React 应用处理已迁移页面，两套前端应用共享同一个顶层 HTML Shell，用户在不同页面间跳转时无缝切换。

## 常见误区

**误区一：把代理层当成永久架构**。有些团队在绞杀完成后保留了 Strangler Facade，并在其中堆积了越来越多的业务逻辑，导致代理层本身变成了新的遗留包袱。代理层应保持轻薄，仅承载路由规则，一旦迁移完成应尽快拆除或退化为普通网关配置。

**误区二：过早切断旧系统的写入路径**。部分团队在新系统上线第一天就关闭旧系统写入，导致一旦新系统出现 Bug 需要回滚时，数据无法复原。正确做法是在双写阶段至少运行完一个完整的业务周期（如一个结算周期），验证新旧数据一致性无误后再切断旧写路径。

**误区三：将绞杀者模式等同于"模块化重构"**。绞杀者模式强调的是**运行时的新旧系统并存与流量切换**，而不仅仅是对旧代码做拆分重组。如果新模块和旧模块仍然共享同一个部署单元和数据库，而没有实现真正的路由隔离，则不符合绞杀者模式的定义，其风险管理优势也无法体现。

## 知识关联

**前置概念——遗留代码重构**：绞杀者模式的应用前提是能够识别遗留系统中的功能边界，这依赖于对遗留代码的阅读与分析能力。只有在梳理清楚遗留系统的模块依赖图（通常借助静态分析工具如 jdeps 或 SonarQube）之后，才能合理规划绞杀迁移的切分单元与顺序。

**后续概念——重构模式**：绞杀者模式解决的是系统级替换策略，而在每个被迁移的新模块内部，仍需运用具体的重构模式（如提炼函数、移动方法、引入参数对象等）来保证新代码的质量。两者结合构成了从宏观迁移策略到微观代码改善的完整实践体系。绞杀者模式还与**特性开关（Feature Toggle）**紧密配合，通过开关控制流量切换的粒度，实现比路由规则更细腻的灰度发布能力。
