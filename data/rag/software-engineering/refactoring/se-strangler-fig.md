# 绞杀者模式

## 概述

绞杀者模式（Strangler Fig Pattern）得名于热带雨林中的绞杀榕（Ficus aurea）——这种植物从宿主树的树冠开始生长，逐渐向下延伸根系，最终完全包裹并替代原有树木，而宿主树则在此过程中慢慢消亡。Martin Fowler 于2004年首次将这一自然现象类比到软件系统的渐进式迁移中，提出了绞杀者模式这一重构策略。

该模式的核心思想是：在旧系统（被绞杀者）仍然运行的同时，逐步构建新系统（绞杀者），并通过一个中间层（通常称为"绞杀者外墙"或 Facade）将流量从旧系统切换到新系统。与"大爆炸重写"（Big Bang Rewrite）相反，绞杀者模式要求新旧系统在一段时间内并行共存，直到旧系统的所有功能都被新系统完整接管后才彻底废弃旧系统。

绞杀者模式在微服务迁移场景中被广泛应用。当企业需要将一个运行多年的单体应用拆解为微服务架构时，无法承受数月停机或大规模并行开发的风险，绞杀者模式允许团队以功能模块为单位，一次只迁移一个功能，持续交付业务价值，同时将技术债务逐步清偿。

## 核心原理

### 绞杀者外墙（Strangler Facade）

绞杀者模式的关键基础设施是一个位于客户端与新旧系统之间的代理层。所有外部请求首先到达这个外墙，由它决定将请求路由到旧系统还是新系统。外墙本身不实现业务逻辑，只做路由决策。常见的实现方式包括 HTTP 反向代理（如 Nginx）、API 网关或消息中间件。以 Web 应用迁移为例，可以在 URL 路径级别做路由：`/orders/*` 已迁移完成则转发到新服务，`/inventory/*` 尚未迁移则仍转发到旧系统。

### 三阶段迁移生命周期

绞杀者模式的迁移过程可以细分为三个明确阶段：

1. **共存阶段（Co-existence）**：新旧系统同时运行，外墙将特定功能的流量切换至新系统，但旧系统仍处理其余请求。此阶段可通过金丝雀发布（Canary Release）将1%~5%的流量先导向新系统进行验证。
2. **节流阶段（Throttling）**：随着新系统模块逐一通过验证，外墙将越来越多的路由规则指向新系统，旧系统承接的流量比例持续下降。
3. **终止阶段（Termination）**：旧系统的所有功能均已被新系统覆盖，外墙所有规则全部指向新系统，旧系统实例可安全关闭。

### 功能切割粒度的选择

每次迁移的功能模块大小直接影响绞杀者模式的成败。粒度过大则单次迁移风险高、耗时长；粒度过小则迁移次数过多，外墙路由规则管理复杂。业界经验表明，以"用户故事"或"业务子域"为单位进行切割是可操作的基准，例如将"用户认证"、"订单管理"、"支付处理"作为三个独立的迁移批次，而非按技术层（如数据访问层、控制层）切割。按技术层切割会导致新旧系统在同一业务流程中深度耦合，增加数据一致性问题。

### 数据迁移与双写策略

绞杀者模式中最棘手的问题是数据层的迁移。当新服务接管某个功能模块时，它通常需要独立的数据存储，而旧系统可能仍在读写同一份数据。解决这一问题的常见策略是**双写（Dual Write）**：在迁移期间，写操作同时写入旧数据库和新数据库，读操作则根据当前迁移阶段选择来源。双写期间需要通过对账脚本定期比对两个数据源的差异，确保数据一致性。一旦新数据库的数据校验通过，即可切断旧数据库的写入并完成数据主权转移。

## 实际应用

**案例：Shopify 的单体应用迁移**

Shopify 工程团队在将其 Ruby on Rails 单体应用向服务化架构迁移时，明确采用了绞杀者模式。他们首先抽离"礼品卡"功能作为第一个独立服务，通过在旧 Rails 应用前部署路由层，将 `/gift_cards/*` 的所有请求转发至新服务，同时旧代码冻结该功能的开发。此次迁移历时约6个月完成单一功能的完整替换，积累了可复用的迁移流程，之后其他模块的迁移速度明显加快。

**案例：遗留 COBOL 系统的现代化**

金融行业中，绞杀者模式被用于替换运行数十年的 COBOL 主机系统。通常做法是在 COBOL 系统前部署一个中间件层（如 IBM MQ 或现代 API 网关），将新开发的 Java 或 Go 微服务逐步接管各类交易处理功能，而 COBOL 程序在后台继续处理尚未迁移的批量任务。这种方式避免了一次性替换风险，也允许对 COBOL 系统的隐性业务逻辑进行逐步梳理。

## 常见误区

**误区一：认为绞杀者模式等同于在旧代码上叠加层次**

一些团队错误地将绞杀者模式理解为在旧系统之上持续添加新的抽象层，而从不删除旧代码。真正的绞杀者模式要求每完成一个功能模块的迁移，必须从旧系统中**物理删除**对应的旧代码和旧路由规则。如果旧代码持续保留，则外墙的路由规则会无限膨胀，系统复杂度实际上在增加而非降低，这是对该模式最常见的执行错误。

**误区二：把绞杀者模式当成解决所有遗留代码问题的万能方案**

绞杀者模式的前提是旧系统能够被明确地按功能边界切割，并且存在可拦截的请求入口（HTTP 请求、消息队列等）供外墙代理。对于高度耦合、功能边界模糊的遗留系统，或者以批处理为主的系统（无明确的请求/响应入口点），直接应用绞杀者模式可能比大爆炸重写更费力。此时需要先通过"修缮者模式"（Renovation Pattern）或依赖注入等手段建立清晰的模块边界，再启动绞杀。

**误区三：忽视旧系统的"退出成本"**

团队往往在规划绞杀者迁移时只计算新系统的开发成本，而低估旧系统的维护成本和最终下线成本。旧系统在共存期间仍需接受安全补丁和关键 bug 修复，相当于团队同时维护两套系统。如果迁移计划无明确的时间节点（如"12个月内完成全部迁移"），旧系统很容易陷入永久性"半绞杀"状态，导致双重维护成本长期叠加。

## 知识关联

绞杀者模式直接建立在**遗留代码重构**的基础上：识别遗留代码中的"接缝"（Seam）——即可以在不修改周边代码的情况下替换行为的位置——正是为绞杀者外墙确定拦截点的核心技能。Michael Feathers 在《修改代码的艺术》中提出的接缝概念与绞杀者模式的路由切割点在本质上是同一问题的两个视角。

在微服务架构领域，绞杀者模式常与**反腐败层**（Anti-Corruption Layer，ACL）配合使用：当新服务调用旧系统的接口时，ACL 负责翻译两套系统之间的领域模型差异，防止旧系统的概念污染新系统的设计。此外，**功能开关**（Feature Toggle）是绞杀者模式在流量切换环节的常用辅助工具，通过开关控制外墙的路由策略，可以在无需重新部署的情况下实时调整新旧系统的流量分配比例。
