# 绞杀者模式

## 概述

绞杀者模式（Strangler Fig Pattern）由Martin Fowler于2004年在其博文《Strangler Fig Application》中正式提出，灵感来源于热带雨林中的绞杀榕（Ficus aurea）——这种植物从宿主树的树冠开始生长，逐步向下蔓延，最终用自身的结构完全替代原有宿主，而宿主在这一过程中持续为新植物提供支撑。绞杀者模式将同样的生长策略应用于软件系统替换：新系统如寄生植物一般逐步包围旧系统，直到旧系统可以被安全移除。

这一模式专门针对"大爆炸重写"（Big Bang Rewrite）的核心风险而设计。大爆炸重写要求团队停止向旧系统添加功能，投入数月乃至数年开发全新系统，最终在某一天一次性完成切换。历史上，许多此类项目因低估旧系统的隐性行为（如未文档化的业务规则）而失败或严重超期。绞杀者模式通过将迁移拆解为一系列独立可部署的小步骤，使得新旧系统在生产环境中长期共存，每一步均可验证正确性并在出错时独立回滚。

绞杀者模式在现代系统迁移中的核心价值在于：迁移过程中旧系统始终处于可用状态，业务连续性不受中断，同时新系统的功能覆盖范围随时间单调递增，最终在某个功能完全转移的节点上，旧系统自然退出——如同被完全绞杀的宿主树消失于雨林地表。

## 核心原理

### 门面层（Facade Layer）拦截机制

绞杀者模式的实施依赖一个关键的中间代理层——通常称为绞杀者门面（Strangler Facade）。所有来自客户端的请求首先到达该门面，由门面根据路由规则决定将请求转发至旧系统还是新系统。这一机制在HTTP架构中通常以反向代理（如Nginx或API Gateway）实现，在消息驱动架构中则以消息路由器实现。门面层的存在使得路由规则的变更无需修改任何业务代码即可生效。

### 功能迁移的三阶段循环

每个独立功能模块的迁移遵循严格的三阶段循环：

1. **共存阶段**：新实现在独立部署单元中完成开发，门面层仍将该功能的100%流量路由至旧系统。此时新实现可在影子模式（Shadow Mode）下并行接收请求，其输出与旧系统对比但不返回给客户端，用于验证行为一致性。

2. **切流阶段**：通过功能开关（Feature Toggle）或流量权重配置，逐步将该功能的请求比例从旧系统迁移至新系统，例如按照1% → 10% → 50% → 100%的梯度推进，每个梯度持续观察错误率与延迟指标。

3. **清除阶段**：当新系统承接100%流量且运行稳定后，从门面层删除旧系统对应的路由规则，并删除旧系统中该功能的代码与部署资源。

### 数据层的同步挑战

绞杀者模式中最复杂的工程问题发生在数据层。当新旧系统同时处于活跃状态时，两者可能操作不同的数据存储，导致数据不一致。常见解决方案包括：**双写策略**（新系统写入新数据库同时同步至旧数据库）和**变更数据捕获（CDC）**同步（如使用Debezium监听旧数据库的binlog并实时同步至新数据库）。选择哪种策略取决于旧系统是否允许被修改以添加双写逻辑。

## 实际应用

**电商平台结账模块迁移**：某零售公司将一套运行15年的单体PHP系统迁移至基于Node.js的微服务架构。团队首先识别出结账（Checkout）功能作为第一个迁移目标，因为其接口边界清晰且与其他模块耦合度相对较低。他们在AWS API Gateway上建立了绞杀者门面，初期将结账相关的所有API路由至旧PHP应用。新Node.js结账服务完成开发后，首先以影子模式运行两周验证计算结果一致性，随后通过API Gateway的权重路由功能在三天内将流量从0%逐步切至100%。旧系统的结账代码在三个月的稳定观察期后正式被删除。整个迁移过程中，用户未感知到任何服务中断。

**主机银行系统现代化**：一家欧洲银行将运行30年的COBOL主机系统迁移至Java微服务，整个工程历时5年。由于账户余额数据不能在两个系统中独立维护，工程师采用了"旧系统作为唯一数据源（Single Source of Truth）"策略：新Java服务本身不保存账户余额，所有余额查询和更新仍通过集成层调用COBOL主机，直到该数据域的完整迁移工作就绪才切换数据来源。这一策略以牺牲部分性能为代价，换取了迁移期间数据层的绝对一致性。

## 常见误区

**误区一：绞杀者模式适用于任何规模的变更**。实际上，绞杀者模式对功能边界的清晰度有较高要求。当旧系统模块间存在大量隐式共享状态（如全局数据库事务跨越多个业务域）时，建立有效的门面层拦截点本身就极为困难。在这种情况下，需要先对旧系统做足够的模块化改造（例如引入防腐层，Anti-Corruption Layer），才能为绞杀者模式创造条件，而非将绞杀者模式视为可以绕过模块化工作的捷径。

**误区二：门面层一旦建立就应该永久保留**。部分团队在完成全量迁移后仍保留门面层，将其作为"以备回退"的保险。这会导致额外的网络跳转延迟长期存在（通常增加5-20ms的P99延迟），且门面层本身成为新的技术债务来源。绞杀者模式的完成标志是：旧系统被完全移除后，门面层也随之被移除，客户端直接访问新系统。

**误区三：切流比例的推进速度越快越好**。部分团队在新系统通过测试后迅速将流量比例推至100%，跳过中间梯度。然而，旧系统中往往存在仅在特定时间（如月末批处理）或特定用户行为序列下触发的隐性逻辑，快速切流会导致这类边缘场景来不及在生产流量中被观察到。Martin Fowler在原始文章中强调，切流节奏应与旧系统行为的已知覆盖度挂钩，而非单纯由时间表驱动。

## 知识关联

绞杀者模式建立在遗留代码重构的核心方法论之上：在对旧系统代码进行任何替换之前，必须对其行为建立足够的测试覆盖（特征测试，Characterization Tests），以便在新实现中验证行为一致性。没有这一基础，影子模式阶段的对比验证便无从进行，绞杀者模式的安全性承诺也无法兑现。

在架构模式层面，绞杀者模式与防腐层（Anti-Corruption Layer）、功能开关（Feature Toggles）和蓝绿部署（Blue-Green Deployment）存在紧密协作关系：防腐层隔离新旧系统的领域模型差异，功能开关控制切流粒度，蓝绿部署提供快速回滚的基础设施能力。三者共同构成绞杀者模式在工程落地时的技术支撑体系。对于计划采用绞杀者模式进行微服务拆分的团队，《Building Microservices》（Sam Newman，2015）第13章专门讨论了绞杀者模式与微服务迁移策略的结合方法，可作为进阶参考。
