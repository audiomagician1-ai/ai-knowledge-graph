# 提取类

## 概述

提取类（Extract Class）是一种重构手法，专门用于解决"上帝类"（God Class）问题——即一个类承担了过多职责、包含过多字段和方法，违反了单一职责原则（SRP）。其核心操作是：将原类中属于同一逻辑子领域的字段和方法，迁移到一个新建的独立类中，再让原类持有对新类的引用。

该手法由Martin Fowler在1999年出版的《重构：改善既有代码的设计》（第一版）中正式命名和系统化描述，编号为"Extract Class"，属于"将大类拆分"这一重构类别。Fowler给出了判断是否需要提取类的经典标准：当你无法用一句话描述一个类的职责，或者描述时不得不使用"以及（and）"这个词时，就说明该类已经承担了多个职责。

提取类的价值在于直接降低原类的认知复杂度。一个包含20个字段、50个方法的上帝类，在被拆分后，每个子类可能只需关注5-8个字段，使得测试覆盖、独立修改和代码复用都变得更加可行。

## 核心原理

### 识别候选字段群

提取类的第一步是在原类中识别出"字段子集"，这些字段往往总是被同一组方法操作，或在业务语义上属于同一概念。例如，一个`Person`类同时包含`name`、`age`、`areaCode`、`officeNumber`、`mobileNumber`等字段，可以发现后三个字段都与联系方式相关，它们构成一个候选提取集——应被提取为`TelephoneNumber`类。字段之间的高内聚性（cohesion）是识别边界的主要信号。

### 新类的创建与接口设计

确定候选字段后，创建新类并将相关字段迁移进去。此时需要决定新类与原类之间的可见性策略：新类是作为原类的**私有实现细节**（调用方无法感知），还是作为**可对外暴露的值对象**。以`TelephoneNumber`为例，如果外部代码需要直接传递电话号码对象，则应将其设计为公开类并提供合理的访问器（getter）；如果仅内部使用，可暂时保持包级私有。这一决策直接影响后续是否需要进行"委托隐藏"（Hide Delegate）重构。

### 迁移步骤的原子性

提取类的标准操作序列包含以下四步，每步之后都应执行测试以确保行为不变：

1. 在源文件中创建新类，声明需迁移的字段（此时两个类同时拥有这些字段，代码可编译）；
2. 在原类中添加`private NewClass newClass = new NewClass()`类型的字段作为引用；
3. 将原类中操作这些字段的方法逐一改为委托给`newClass`对象；
4. 删除原类中已迁移的字段，仅保留对`newClass`的引用。

这种渐进式迁移（每次只移动一个字段或方法）可将大规模重构的风险控制在单个方法级别，避免一次性修改引发大面积编译错误。

### 双向依赖的处理

若提取过程中发现新类中的方法需要反向调用原类的方法，就产生了双向引用（circular reference），这是提取类中最棘手的情形。Fowler明确建议：尽量避免双向引用，因为它会使两个类无法独立变化。解决方案是重新审视职责划分边界——通常意味着某些方法被分配到了错误的类中，需要将产生回调需求的方法整体移回原类，或进一步引入第三个类来打破循环。

## 实际应用

**订单类拆分案例**：一个电商系统中的`Order`类同时管理商品列表、收货地址和支付信息，字段多达30个。应用提取类后，`ShippingAddress`类接管`province`、`city`、`street`、`zipCode`四个字段及地址格式化方法；`PaymentInfo`类接管`cardNumber`、`expiryDate`、`cvv`字段及支付验证逻辑。原`Order`类仅保留订单业务流程相关的字段，行数从约400行降至150行。

**Java标准库中的实例**：`java.net.URL`类在历史版本中因职责过重而被批评，后续版本通过引入`URI`类分离了部分统一资源标识符的解析逻辑，这正是提取类思想在标准库层面的体现。

**配合IDE工具执行**：IntelliJ IDEA和Eclipse均内置了Extract Class重构操作（IntelliJ快捷键：右键 → Refactor → Extract Delegate），工具会自动生成委托代码，但仍需人工确认字段分组是否合理，工具无法替代人对业务语义的判断。

## 常见误区

**误区一：按方法数量均分而非按职责聚合**。初学者常将一个有40个方法的类机械地拆成各20个方法的两个类，而不是按业务职责聚合。这样拆出的类内部方法缺乏内聚性，新类的命名往往是`OrderHelper`或`OrderUtils`这类模糊名称，无法反映任何业务概念，实质上只是将混乱分散到两个文件。判断标准应是：新类是否能被一个清晰的名词（代表业务概念）来命名。

**误区二：将提取类与提取超类（Extract Superclass）混淆**。当两个已有类具有相似行为时，应使用提取超类将公共行为上移；而提取类是针对**单个**过于庞大的类进行纵向切分，产生的是"整体-部件"关系（原类持有新类的实例），而非"父类-子类"的继承关系。误用继承来拆分上帝类会导致滥用继承的代码异味。

**误区三：提取后忽略原类的"委托方法"清理**。提取完成后，原类为了保持对外接口不变，常保留大量转发方法（如`String getAreaCode() { return telephoneNumber.getAreaCode(); }`）。若外部调用方可以直接访问新类，应进行"内联委托（Inline Delegate）"或开放新类引用，而非无限积累空壳委托方法，否则原类代码体积不降反增。

## 知识关联

提取类以"代码异味"中的**上帝类**和**数据泥团**（Data Clumps，指总是一起出现的字段组合）为前置识别条件，能够准确发现这两种异味是应用本手法的必要能力。同时，提取类通常需要配合"提取方法"的前期工作——先将原类中的长方法拆解为若干短方法，才能更清晰地看出哪些方法属于同一职责簇。

完成提取类之后，自然引出"**移动方法/字段**（Move Method / Move Field）"手法：当发现原类中有某个方法更多地操作新类的数据而非原类的数据时，应直接将该方法移动到新类中，而不是留在原类通过委托调用。移动方法是提取类的精细化后续步骤，两者共同构成了处理上帝类问题的完整工具链。
