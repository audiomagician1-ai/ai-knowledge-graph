# 可复现构建

## 概述

可复现构建（Reproducible Builds）是指在给定相同源代码、相同构建环境和相同构建指令的前提下，无论何时何地重复执行构建过程，都能产生**逐字节完全相同**的输出文件（二进制、压缩包等）的构建实践。衡量标准非常精确：两次构建产物的 SHA-256 哈希值必须完全一致，差一个字节都算失败。

可复现构建的概念在 2014 年由 Tor Project 团队系统化提出并率先实践，他们意识到用户无法验证官方分发的 Tor Browser 二进制包是否真正由公开源代码编译而来——攻击者可以在构建服务器上植入后门，而源代码审计形同虚设。此后，Debian 项目于 2015 年启动大规模的可复现构建改造工程，目标是让所有软件包都可被第三方独立验证，截至 2023 年已有超过 93% 的 Debian 软件包实现可复现。

可复现构建在供应链安全领域具有独特价值。2020 年的 SolarWinds 攻击事件中，攻击者正是在构建阶段注入恶意代码，而最终用户收到的是已签名的"合法"二进制文件，无从察觉。如果 SolarWinds 的构建流程是可复现的，任何独立第三方重新构建后发现哈希不一致，即可立即发出警报。

## 核心原理

### 不确定性来源的消除

传统构建流程中存在大量产生非确定性输出的因素，必须逐一消除：

- **时间戳嵌入**：编译器（如 GCC）默认会将 `__DATE__` 和 `__TIME__` 宏展开为构建时的系统时间，并写入目标文件。解决方案是设置环境变量 `SOURCE_DATE_EPOCH`（一个 Unix 时间戳整数），告知构建工具链使用该固定时间替代系统时间。`SOURCE_DATE_EPOCH` 规范由 reproducible-builds.org 于 2015 年正式提出，现已被 GCC、Clang、dpkg、rpm 等数十种工具原生支持。
- **文件系统遍历顺序**：`find`、`glob` 等命令列出目录内容时，不同操作系统或文件系统（ext4 vs btrfs）返回顺序不同。构建脚本必须显式添加 `sort` 排序步骤，如 `find src -name "*.c" | sort`。
- **绝对路径嵌入**：GCC 编译时默认将源文件绝对路径写入调试信息（DWARF），导致 `/home/alice/project` 和 `/home/bob/project` 编译结果不同。解决方案是使用 `-ffile-prefix-map=/home/alice/project=.` 编译选项将路径重映射为相对路径。
- **并发不确定性**：多线程构建中，链接顺序或归档文件内文件排列顺序可能随线程调度而改变，需要强制排序或使用确定性链接参数（如 `ar` 的 `-D` 选项禁用时间戳）。

### 构建环境的封装

仅消除不确定性还不够，构建环境本身也需要严格锁定。容器化是常见方案：使用 Docker 或 Nix 将编译器版本（如 `gcc-12.3.0`）、libc 版本、构建工具版本全部固定。Nix 包管理器通过对每个依赖的内容哈希寻址来确保环境精确复现，其核心公式可表示为：

```
output_hash = f(source_hash, deps_hashes[], build_script_hash)
```

所有输入都是内容寻址的，输出哈希由所有输入唯一决定，与安装路径、时间无关。

### 独立验证机制

可复现构建的安全价值来自**多方独立重建（Multi-Party Rebuild）**：至少两个互相不信任的构建者（例如官方维护者 + 社区志愿者）分别在隔离环境中构建同一源码，对比哈希值。Debian 项目运营 `buildinfo.debian.net` 服务，存储每次构建的 `.buildinfo` 文件，记录构建输入的完整元数据（编译器版本、环境变量、dpkg 包列表），供第三方验证使用。工具 `diffoscope` 专为比较两个不可复现构建的差异而设计，能递归拆解 ELF 文件、ZIP 包、tar 包，精确定位差异字节所在的数据结构字段。

## 实际应用

**Debian Linux 软件包验证**：Debian 用户可以使用 `reprotest` 工具对任意软件包执行可复现性测试，该工具会在改变时间、路径、用户名、语言环境等 25+ 种环境变量的条件下重复构建，检验结果是否一致。

**Bitcoin Core 的 Gitian 构建**：Bitcoin Core 自 2011 年起使用 Gitian 系统，要求多名核心开发者分别在 Ubuntu LTS 虚拟机中构建发布包并公开签名，最终发布的二进制文件必须有至少 3 名开发者签名的哈希相互一致才被认可，直接对抗单点构建服务器被攻陷的风险。

**F-Droid Android 应用商店**：F-Droid 坚持自行从源代码构建所有 Android 应用，并将构建结果的哈希与开发者上传的 APK 进行比对。自 2022 年起，F-Droid 引入"可复现构建"徽章，标记那些开发者自签名 APK 与 F-Droid 重建结果哈希相同的应用，赋予用户安装开发者原版的选择权。

## 常见误区

**误区一：代码签名等同于可复现构建**
很多开发者认为对发布包进行数字签名就足以保证安全性。实际上，代码签名只证明"这个文件是由持有私钥的人签名的"，但如果签名者的构建服务器已被植入后门，签名反而为恶意代码背书。可复现构建证明的是"这个二进制文件确实来自这段源代码"，两者保护的威胁模型完全不同。

**误区二：只需锁定依赖版本号即可**
使用 `package-lock.json` 或 `Cargo.lock` 锁定依赖版本号是依赖管理的良好实践，但不等于可复现构建。版本号相同的包可能因 npm registry 内容被替换（依赖混淆攻击）而内容不同；而且即使依赖完全相同，构建过程中的时间戳嵌入、路径差异等因素仍会导致输出不一致。真正的可复现构建需要同时锁定内容哈希（如 `integrity` 字段的 `sha512` 值）和消除构建过程的非确定性。

**误区三：可复现构建必须与构建环境完全相同**
这是一个常见的过度要求。`SOURCE_DATE_EPOCH` 和路径重映射等技术的意义正在于：在**不同机器、不同路径、不同时间**下构建出相同结果。如果需要环境完全一致才能复现，那称为"冻结环境"，而非可复现构建，因为它无法用于独立第三方验证。

## 知识关联

**与哈希函数的关系**：可复现构建直接依赖 SHA-256 等密码学哈希函数的确定性和抗碰撞性。正因为 SHA-256 对相同输入必然产生相同的 256 位输出，逐字节一致的二进制文件才能用哈希值相等来高效验证，而不必传输完整文件。

**与持续集成（CI）系统的关系**：在 CI 流水线中集成可复现性检查，通常做法是在同一 CI 流程中触发两次构建（使用不同的 `BUILD_PATH` 和 `SOURCE_DATE_EPOCH`），调用 `diffoscope` 对比输出，若发现差异则构建失败。这将可复现性从一次性审计变为持续回归防护。

**与软件物料清单（SBOM）的关系**：SBOM 记录软件由哪些组件构成，而可复现构建提供了验证 SBOM 准确性的技术手段——通过重建并比对哈希，可以确认 SBOM 中声明的源代码确实对应分发的二进制文件，二者结合构成完整的供应链可验证性。
