# 回滚策略

## 概述

回滚策略（Rollback Strategy）是指当新版本部署后出现故障时，将系统恢复至上一个稳定版本的计划与执行方案。在CI/CD流水线中，回滚不是偶发的补救措施，而是每次发布计划中必须提前设计的一部分——经验法则是"能快速部署，就必须能快速回滚"。

回滚的概念随着持续交付实践的成熟而系统化，特别是在2010年代DevOps运动兴起后，Netflix、Amazon等公司将自动回滚能力内建至部署工具中，使"回滚时间（Time to Rollback，TTR）"成为与部署频率并列的可靠性指标。现代团队通常将TTR的目标设定在5分钟以内。

回滚策略的核心价值在于控制"平均恢复时间（MTTR，Mean Time to Recovery）"。如果一次有问题的部署需要花费2小时修复代码再重新发布，而回滚只需1分钟恢复服务，那么有效的回滚策略直接将用户受影响时长压缩了99%以上。

## 核心原理

### 自动回滚（Automatic Rollback）

自动回滚依赖于预设的健康检查指标触发回滚动作，无需人工干预。常见的触发条件包括：HTTP 5xx错误率超过阈值（如5%）、应用启动探针（Readiness Probe）连续失败3次、或新版本容器的崩溃重启次数（CrashLoopBackOff）超过指定次数。

在Kubernetes中，Deployment控制器原生支持自动回滚：当新的Pod无法通过就绪检查时，滚动更新会自动暂停并执行 `kubectl rollout undo deployment/<name>`，将副本集（ReplicaSet）切换回上一版本。Argo Rollouts等工具进一步扩展了此能力，支持基于Prometheus指标的分析规则，例如"若新版本的p99延迟超过200ms持续3分钟，则自动回滚"。

自动回滚的关键限制是**它只能回到上一个版本**，不支持跨多个版本的历史回滚，且对数据库迁移（Migration）引起的故障无能为力。

### 手动回滚（Manual Rollback）

手动回滚由工程师主动发起，适用于两类场景：一是故障原因不明确、需要评估后再决策；二是自动回滚条件未被触发但业务指标已经恶化（如转化率下降但错误率正常）。

操作上，手动回滚通常通过以下三种途径实现：
1. **重新触发流水线**：在CI/CD系统（如Jenkins、GitLab CI）中手动指定上一个稳定的Git commit SHA重新执行部署任务。
2. **直接切换制品**：在Artifact Registry中将生产环境指向上一个镜像标签（tag），例如从 `v2.3.1` 切换回 `v2.2.9`。
3. **特性开关（Feature Flag）关闭**：若新功能通过LaunchDarkly等工具管控，可在不重新部署的情况下通过关闭Flag实现功能级别的"软回滚"。

### 版本锁定（Version Pinning）

版本锁定是回滚策略的预防性手段，通过固定依赖项的精确版本号，确保在执行回滚时能够完整复现历史环境。在 `package.json` 中使用 `^1.2.3` 表示允许自动升级小版本，而 `package-lock.json` 或 `yarn.lock` 文件则将整个依赖树的每个包锁定到特定哈希值——这正是版本锁定文件必须提交到版本库的根本原因。

对于容器化应用，版本锁定意味着Docker镜像标签应使用不可变的摘要（digest），例如 `nginx@sha256:abc123...` 而非 `nginx:latest`。使用 `latest` 标签会导致回滚时实际拉取的镜像与历史版本不同，使回滚失去确定性。

Helm Chart通过 `Chart.lock` 文件对子Chart的版本进行锁定，执行 `helm dependency update` 后生成的锁文件保证了每次部署的依赖一致性。

## 实际应用

**蓝绿部署中的回滚**：蓝绿部署天然具备最快速的手动回滚能力。假设蓝色环境是当前生产（v1.0），绿色环境部署了新版本（v2.0），若发现问题，只需将负载均衡器或DNS记录从绿色切回蓝色，回滚完成时间通常在30秒内，且蓝色环境的数据状态完全未受影响。

**金丝雀发布中的自动回滚**：以Flagger框架为例，金丝雀分析每隔1分钟（可配置）检查一次指标。若连续5次分析失败，Flagger自动将流量权重从金丝雀版本（如20%）归零，并将Deployment恢复至主版本，整个过程不需要任何人工操作，受影响的最大用户比例被限制在金丝雀流量比例内。

**数据库迁移与回滚的冲突**：若v2.0版本执行了一条不可逆的DDL语句（如 `DROP COLUMN old_field`），回滚代码到v1.0后服务会因为找不到该字段而崩溃。此场景下正确做法是采用"展开-收缩（Expand-Contract）"模式：v2.0仅新增字段，v3.0迁移数据，v4.0才删除旧字段，每一步都保持向后兼容，使任意步骤都可安全回滚。

## 常见误区

**误区一：保留旧版本制品等于拥有了回滚能力**
仅仅在镜像仓库中保存了旧版本镜像，并不意味着可以顺利回滚。回滚还需要：对应版本的基础设施配置（Terraform状态）、当时的环境变量与密钥版本、以及经过验证的回滚操作手册。没有定期演练回滚流程的团队，在真正需要时往往会发现流程早已失效。

**误区二：回滚总是比前滚（Roll Forward）更安全**
对于已经发生了数据写入或外部API调用的故障，回滚代码版本并不能撤销这些副作用。此时"前滚"——即快速发布一个修复补丁——反而比回滚更安全，因为它不会引入数据状态与代码逻辑不匹配的问题。回滚vs前滚的决策应基于故障性质，而非默认偏好其中一种。

**误区三：自动回滚可以完全替代手动回滚**
自动回滚依赖于可观测到的技术指标，但业务层面的故障（如推荐算法产生不当内容、定价逻辑错误）往往不会触发任何技术报警。这类问题只能通过人工监控业务指标发现，然后手动触发回滚，两种机制缺一不可。

## 知识关联

回滚策略直接建立在**部署策略**的基础之上：蓝绿部署、金丝雀发布和滚动更新各自对应不同的回滚机制和回滚速度。蓝绿部署的回滚最快但成本最高（需要双倍资源）；金丝雀发布的爆炸半径最小；滚动更新的回滚过程本身也是滚动的，中间状态存在新旧版本并存的时间窗口。

在可观测性体系中，回滚的触发决策依赖于指标（Metrics）、日志（Logs）和追踪（Traces）三类数据——健康的回滚策略必须明确规定"什么指标达到什么阈值触发回滚"，而不是依赖工程师的主观感受。

版本锁定的习惯进一步延伸至**供应链安全（Supply Chain Security）**领域，SLSA（Supply-chain Levels for Software Artifacts）框架将制品的不可变性和可溯源性列为软件安全的基础要求，这与回滚策略中对镜像摘要锁定的要求高度一致。
