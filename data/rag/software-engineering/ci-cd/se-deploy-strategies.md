# 部署策略

## 概述

部署策略是指将软件新版本从开发环境发布到生产环境时所采用的具体方法，核心目标是在最小化停机时间的同时降低因新版本缺陷引发的线上风险。不同策略本质上是在**发布速度**、**资源消耗**、**风险范围**三个维度之间做权衡取舍。

现代部署策略的体系化形成于2000年代末云计算和持续交付实践兴起之后。Netflix、Amazon等公司在大规模微服务架构下，将蓝绿部署和金丝雀部署推广为行业标准模式。2010年前后，Etsy等公司在其持续部署实践中公开记录了每天数十次发布的经验，使这些策略从理论走向主流工程实践。

选择正确的部署策略直接影响故障影响范围（Blast Radius）：一次全量部署的故障可能波及100%用户，而金丝雀部署可将初始影响范围压缩至1%~5%的流量，这一数量级的差距在高并发系统中意味着数万乃至数百万用户的服务体验差异。

## 核心原理

### 蓝绿部署（Blue-Green Deployment）

蓝绿部署维护两套完全相同的生产环境：**蓝环境**运行当前版本，**绿环境**部署新版本。切换时通过修改负载均衡器或DNS指向，将全部流量从蓝环境瞬间切换至绿环境，整个切换过程理论停机时间为0。

关键特性：**切换是原子操作**，要么100%流量在蓝，要么100%在绿，不存在中间状态。回滚同样是秒级操作——只需将流量切回蓝环境即可。代价是需要**双倍基础设施资源**，且切换瞬间若新版本存在数据库 Schema 不兼容问题，可能导致全量故障而非局部故障。

### 金丝雀部署（Canary Deployment）

名称来源于19世纪矿工携带金丝雀入矿探测毒气的做法。金丝雀部署将新版本先发布给一小部分真实用户（如5%或1000名用户），在收集足够监控数据确认稳定后，再逐步扩大流量比例，最终完成全量替换。

流量分配公式通常基于**请求头、用户ID哈希或地理位置**：  
```
is_canary = hash(user_id) % 100 < canary_percentage
```
这确保同一用户在观察窗口期内始终路由到同一版本，避免用户体验割裂。金丝雀部署需要系统能够**同时运行两个版本**，因此要求新旧版本的API和数据库模型在过渡期内保持向下兼容。

### 滚动部署（Rolling Deployment）

滚动部署将服务实例逐批升级，例如一个10节点集群每次升级2个节点（20%批次），升级完成并健康检查通过后再进行下一批。Kubernetes的`RollingUpdate`策略默认参数为`maxSurge: 25%`、`maxUnavailable: 25%`，即最多允许25%实例同时处于升级中且不可用。

滚动部署的特点是**资源消耗最低**，不需要额外的空闲环境。但在升级过程中，集群中同时存在新旧两个版本处理请求，若新旧版本之间存在不兼容的业务逻辑（如消息队列格式变更），可能引发难以排查的跨版本交互问题。回滚时同样需要逐步替换，耗时比蓝绿部署长。

### A/B 部署（A/B Testing Deployment）

A/B部署常与金丝雀部署混淆，但两者目标本质不同：金丝雀部署的目标是**验证技术稳定性**，A/B部署的目标是**验证业务指标假设**（如点击率、转化率）。A/B部署按照用户细分属性（地区、设备类型、账户等级）将流量精确路由到不同版本，并通过统计显著性检验（通常需要p值＜0.05）来判定版本优劣，而非依赖错误率等工程指标。

## 实际应用

**电商大促前的发布场景**：某电商平台在双十一前需要上线新的优惠券计算引擎。由于大促期间不允许出现全量故障，团队选择金丝雀部署，先将1%的流量（约5万请求/分钟）路由至新版本，监控P99延迟和优惠计算错误率30分钟后，若无异常再按5%→20%→50%→100%的梯度扩大，整个发布过程历时约4小时。

**数据库Schema变更场景**：当新版本需要在数据库中新增一列时，蓝绿部署需要特别谨慎——切换前必须确保旧版本代码能够忽略新列（向后兼容写入），否则回滚时旧版本会因不认识新列而报错。行业惯例是将Schema变更拆分为**Expand-Contract**（扩展-收缩）两步：先加列不删旧列，新旧版本并存后再删旧列。

**Kubernetes滚动部署配置示例**：在生产环境中，设置`minReadySeconds: 30`可确保每个新Pod启动后至少稳定运行30秒再进行下一批升级，这一参数对于防止启动慢的服务被过早判定为健康至关重要。

## 常见误区

**误区一：认为蓝绿部署一定比滚动部署更安全**。实际上蓝绿部署的切换是全量原子性的，一旦新版本存在未被测试覆盖的缺陷，故障会瞬间影响100%流量；而滚动部署在批次升级过程中，缺陷仅影响已升级的比例（如第一批20%的实例）。对于无法并发运行两个版本的有状态服务，滚动部署有时风险更可控。

**误区二：将金丝雀部署与A/B测试等同**。金丝雀部署是以**尽快完成全量发布**为目标的风险管控手段，一旦验证通过即全量推进，通常持续数小时至数天；A/B测试以**收集足够统计样本**为目标，两个版本可能需要并行运行数周，且不以某一版本最终取代另一个为必然结果——有时结论是两个版本无统计显著差异。

**误区三：认为滚动部署天然支持快速回滚**。在Kubernetes中，`kubectl rollout undo`命令触发的回滚本质上仍是一次新的滚动部署（将镜像版本改回旧版），而非瞬间切换。若新版本已导致数据损坏或消息队列消费了不可逆的消息，滚动回滚无法还原这些副作用。

## 知识关联

**与回滚策略的关系**：每种部署策略都对应不同的回滚机制和回滚窗口期。蓝绿部署保留了完整的旧环境，回滚等同于再次切换，是最干净的回滚形式；金丝雀部署在扩量过程中可随时停止并缩容新版本流量；滚动部署的回滚时间与集群规模成正比。理解部署策略是设计可靠回滚方案的前提。

**与Feature Flags的关系**：Feature Flags（功能开关）可以看作一种代码层面的"软性部署策略"。当部署策略解决的是**哪些服务器实例运行新代码**的问题时，Feature Flags解决的是**新代码中哪些功能对哪些用户生效**的问题。两者可以组合使用：先通过金丝雀部署将新代码发布到5%实例，再通过Feature Flags在这5%实例上对0.1%用户开启新功能，实现双重风险隔离。
