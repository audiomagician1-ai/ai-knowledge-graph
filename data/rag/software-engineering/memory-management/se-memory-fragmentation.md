# 内存碎片化

## 概述

内存碎片化（Memory Fragmentation）是指系统中存在大量无法被有效利用的内存空间的现象。这些空间本身是空闲的，但由于其大小、位置或不连续性，导致新的内存分配请求无法被满足——即使系统总空闲内存远大于请求大小。碎片化是堆内存管理中的顽固问题，直接导致内存利用率下降和分配失败。

碎片化问题最早在1960年代随着分时操作系统的兴起而被正式记录。当时的分区内存管理方案（如IBM OS/360的固定分区）频繁遭遇此问题，促使研究者在1970至1980年代开发出伙伴系统（Buddy System）、slab分配器等针对性解决方案。Knuth在《计算机程序设计艺术》第一卷（1968年）中对碎片化做了早期系统性分析，将其描述为动态内存分配的核心困境。

碎片化直接影响长时间运行的服务进程。一个典型的Java或C++后端服务在运行数小时后，堆内存碎片率可达30%~50%，导致即使剩余内存充足，仍触发OOM（Out of Memory）错误或被迫进行耗时的GC压缩操作。理解碎片化机制是进行内存调优、选择分配算法的前提。

## 核心原理

### 外部碎片与内部碎片

内存碎片分为两种截然不同的类型，必须区分对待：

**外部碎片（External Fragmentation）**发生在分配器层面。系统中空闲块分散在堆的各个位置，每块单独太小无法满足新请求，合并后才够用。例如，堆中有10个各为64字节的空闲块（共640字节），但一个512字节的连续分配请求仍会失败。外部碎片是首次适应（First Fit）、最佳适应（Best Fit）等算法的固有副产品。

**内部碎片（Internal Fragmentation）**发生在已分配块内部。分配器为满足对齐要求或最小块限制，分配了比请求更大的空间。例如，请求分配65字节，但分配器按8字节对齐分配了72字节，多出的7字节既无法被其他分配使用，也被当前对象"占用"。固定大小内存池（如slab）中内部碎片尤其显著。

理论上，外部碎片率可用公式量化：

```
外部碎片率 = (1 - 最大可分配块 / 总空闲内存) × 100%
```

若总空闲内存为1GB，但最大连续空闲块只有256MB，则外部碎片率为75%。

### 碎片化的产生机制

碎片化本质上由**分配与释放操作的不对称性**引起。当程序以不规律的顺序释放大小各异的对象时，堆结构逐渐变得如同"瑞士奶酪"。以下场景尤为典型：

- **短寿命对象与长寿命对象交错分配**：短寿命对象释放后留下空洞，但空洞两侧的长寿命对象阻止了合并。
- **大对象分配失败触发扩堆**：系统中明明有足够总量的空闲内存，但因碎片化无法满足大块连续分配，堆被迫向OS申请更多内存。
- **内存池大小类别不匹配**：分配128字节对象到256字节的size class，浪费128字节，当系统中有数百万此类对象时，浪费量以GB计。

### 紧凑化（Compaction）

紧凑化是消除外部碎片的直接手段，通过将所有存活对象移动到堆的一端，从而在另一端形成大块连续空闲区域。Java的G1 GC和ZGC均使用紧凑化策略。G1 GC在Mixed GC阶段选择碎片率最高的Region进行紧凑，ZGC则通过引用着色（Reference Coloring）技术实现并发紧凑，将STW（Stop-The-World）停顿控制在1ms以内。

紧凑化的代价是**对象移动开销**：需要更新所有指向被移动对象的引用，时间复杂度与存活对象数量成正比。C++的默认堆分配器（如ptmalloc2、jemalloc）不支持紧凑化，因为C++原生指针不可被运行时透明修改。

## 实际应用

**jemalloc的size class策略**：Facebook开发的jemalloc通过将内存请求对齐到约200个预定义的size class，将内部碎片率控制在2%以内（相比glibc的ptmalloc约10%的碎片率有显著改善）。jemalloc将内存组织为run和chunk，同一size class的对象集中存放，有效减少了外部碎片。Redis、Firefox均采用jemalloc作为默认分配器。

**Go的内存分配器**：Go运行时使用受tcmalloc启发的分配器，对小对象（≤32KB）按67个size class分配，每个P（逻辑处理器）拥有独立的mcache，避免多线程竞争的同时减少跨size class的外部碎片。Go的GC在并发标记清除后不做即时紧凑化，因此长期运行的Go程序仍可能积累外部碎片，通常通过`runtime.GC()`配合`debug.FreeOSMemory()`触发归还。

**Linux内核的slab分配器**：内核中频繁分配固定结构（如`struct task_struct`、`struct inode`）时，slab分配器为每种结构预先建立缓存池，完全消除外部碎片并将内部碎片控制在结构体大小与slab对齐要求之差内。

## 常见误区

**误区一：总空闲内存足够就不会发生分配失败**

许多开发者认为只要监控显示"剩余内存充足"就不会OOM。实际上，当外部碎片严重时，即使系统剩余2GB空闲内存，一个需要连续1GB内存的请求仍会失败。在C/C++程序中，`malloc`返回`NULL`并不总意味着内存耗尽，而可能是碎片化导致找不到足够大的连续块。

**误区二：垃圾回收完全解决了碎片问题**

GC消除了内存泄漏风险，但不等于消除碎片化。标记-清除（Mark-Sweep）GC（如早期的JVM CMS收集器）会产生大量外部碎片，这正是CMS在Java 14中被彻底移除的原因之一——其碎片化问题导致在高负载下触发Full GC的频率难以接受。只有带紧凑化的GC（如Serial GC、G1 GC）才能主动整合碎片。

**误区三：使用内存池就不会有碎片**

内存池消除了外部碎片，但引入了内部碎片。若内存池的块大小设计不合理，内部碎片率可能超过使用通用分配器的情况。例如，用512字节的池块存放平均只有100字节的对象，内部碎片率高达80%。

## 知识关联

内存碎片化建立在**内存分配器**原理之上：理解首次适应、最佳适应、伙伴系统等分配算法，是分析碎片产生模式的前提——伙伴系统通过二进制对齐将外部碎片限制在特定边界，而最佳适应算法虽减少了空间浪费但会产生大量细小空洞，加速外部碎片化。对**GC优化**的理解则直接影响如何在托管运行时中控制碎片：选择G1还是ZGC，本质上是在吞吐量与碎片率之间做取舍。

向上延伸至**虚拟内存**机制：操作系统通过页表将物理内存碎片对应用层透明化，应用程序看到的是连续的虚拟地址空间，物理层面的碎片由MMU和OS调度处理。这也是为什么即使物理内存碎片化严重，64位系统上的用户态程序仍能分配到"大块连续"内存——实际上是连续的虚拟地址映射到了不连续的物理页。理解这一层次关系，是进行系统级内存调优的关键视角。
