# 字符串池化

## 概述

字符串池化（String Interning）是一种将重复字符串统一存储在一个共享内存区域的技术，使得内容相同的字符串只保留一份副本。程序中出现大量值相同的字符串时，若每次都分配独立内存，会造成严重浪费；字符串池化通过维护一张"字符串到内存地址"的哈希映射表，将相同内容的字符串引用指向同一块内存，从而消除冗余副本。

这一机制最早在 Lisp 语言的符号（Symbol）系统中就有体现，后来在 Java 虚拟机（JVM）中得到广泛推广。Java 在 JDK 1.0 起便内置了字符串常量池（String Constant Pool），位于方法区（Method Area），所有编译期的字符串字面量（如 `"hello"`）都自动进入池中。Python 同样对短字符串和标识符形式的字符串默认执行 interning，可通过 `sys.intern()` 函数手动触发。

字符串池化的价值在两个维度上体现得最为直接：其一，大幅降低堆内存占用——在处理数百万条日志、XML 标签、JSON 键名等重复字符串场景下，内存节省比例可达 40%～80%；其二，将字符串相等性判断从 O(n)（逐字符比较）优化为 O(1)（指针/引用比较），因为池化后相同字符串必然共享同一地址。

## 核心原理

### 哈希表驱动的去重机制

字符串池的本质是一张全局哈希表，键为字符串内容（通过哈希函数计算摘要），值为该字符串在内存中的唯一地址。当程序请求使用某个字符串时，池化逻辑首先计算其哈希值，在表中查找是否已存在相同内容的条目：若命中，则直接返回已有地址；若未命中，才分配新内存并将地址注册进哈希表。整个流程的查找复杂度为 O(1)（平均），插入复杂度同为 O(1)。

### 引用比较代替内容比较

池化后的核心收益是等值判断可退化为引用比较。以 Java 为例，`==` 运算符比较的是对象引用地址，而 `.equals()` 比较字符串内容。未池化时，两个值均为 `"user_id"` 的字符串对象，`==` 返回 `false`，必须调用 `.equals()` 逐字符对比；池化后二者指向同一对象，`==` 直接返回 `true`，省去 O(n) 的遍历开销。在每秒需要执行数百万次字符串比较的高频场景（如数据库列名匹配、HTTP 头字段解析），这一差异具有显著的性能意义。

### Java 字符串常量池的具体实现

Java 中字符串字面量在编译后存入 `.class` 文件的常量池（Constant Pool），JVM 加载类时将其迁移至运行时字符串池。开发者可调用 `String.intern()` 方法手动将运行时动态生成的字符串也纳入池中——该方法若池中已有相同内容的字符串则返回池中实例，否则将当前字符串加入池并返回。Java 7 之前字符串池位于永久代（PermGen），大量调用 `intern()` 可能导致 `OutOfMemoryError: PermGen space`；Java 7 起字符串池迁移至堆（Heap），垃圾收集器可以回收不再被引用的池中字符串，彻底解决了这一问题。

### 池化的内存模型

设程序中出现 k 种不同字符串，每种字符串的平均长度为 L 字节，各字符串平均出现 m 次，则：
- **未池化**总占用约为 `k × m × L` 字节
- **池化后**总占用约为 `k × L + k × P` 字节（P 为每个指针大小，64 位系统为 8 字节）

当 m 远大于 1 时，节省比例接近 `1 - 1/m`。若同一字符串平均出现 100 次，理论节省约 99%（不计哈希表自身开销）。

## 实际应用

**XML/HTML 解析器中的标签名复用**：SAX、DOM 等解析器在解析文档时会反复遇到相同的标签名（如 `<div>`、`<span>`、`class`），解析器通常内置字符串池化，使得整个文档树中所有 `"div"` 字符串共用一个实例，大型 HTML 文档的内存占用可减少数十倍。

**Java 编译器与 JVM 字节码**：Java 源码中所有写成字面量的字符串（如 `String s = "hello"`）在编译阶段即合并至类文件常量池。同一 JVM 进程中加载的所有类共享同一字符串池，因此不同类文件中相同的字面量字符串在运行时只占用一份内存。

**Python 标识符池化**：Python 解释器对满足标识符规则（仅含字母、数字、下划线，且不以数字开头）的字符串自动执行 interning。因此变量名、函数名、字典键（当键为合法标识符时）的比较均为 O(1)，这也是 Python 字典键查找在使用字符串键时能保持高效的原因之一。

**数据库列名与表名管理**：关系型数据库引擎（如 PostgreSQL、MySQL）在解析 SQL 查询时，对表名、列名、别名等元数据字符串执行池化，使得查询计划生成阶段的符号比较可直接通过指针比较完成，提升查询编译速度。

## 常见误区

**误区一：认为所有字符串都应手动 intern**
部分开发者在 Java 中对所有字符串调用 `intern()`，期望全面提升性能。实际上，池化本身有维护哈希表的开销，且动态生成的长字符串（如 UUID、Base64 编码结果）几乎不会重复出现，强行 intern 只会浪费哈希表空间，并在 Java 7 以前引发 PermGen 溢出。池化仅对高重复率、短至中等长度的字符串有益。

**误区二：混淆字符串池化与字符串常量折叠**
编译器的常量折叠（Constant Folding）是在编译期将 `"foo" + "bar"` 直接合并为 `"foobar"` 的优化，发生在编译阶段且不维护运行时数据结构；而字符串池化是运行时去重机制，维护动态哈希表，两者解决的问题和工作阶段均不同。Java 中 `"foo" + "bar" == "foobar"` 为 `true` 是因为编译期折叠后同为字面量进入常量池，而非运行时 interning 的结果。

**误区三：认为池化后字符串不可变是池化的前提**
字符串不可变（Immutability）是池化**安全运行**的必要条件，而非池化的充分条件。若字符串可被修改，则修改某一引用处的内容会影响所有共享该地址的引用，导致错误。Java 将 `String` 设计为不可变类（`final` 字节数组），正是为了保证池化的安全性。但不可变字符串本身并不自动池化，二者是独立的设计决策。

## 知识关联

字符串池化建立在**哈希表**（Hash Table）这一基础数据结构之上，理解哈希函数的碰撞处理（开链法 vs. 开放寻址法）有助于理解不同语言实现字符串池时的性能特征差异。

在内存管理层面，字符串池化是**享元模式**（Flyweight Pattern）在字符串域的具体实现——享元模式通过共享细粒度对象减少内存分配，字符串池化正是其最典型的工业级案例。掌握字符串池化的原理后，可以自然延伸到对象池（Object Pool）、整数缓存（Java 的 `Integer.valueOf()` 对 -128～127 的缓存）等类似机制的理解。

对于需要处理高并发场景的开发者，还需关注字符串池在多线程环境下的线程安全问题：Java 的 `String.intern()` 是线程安全的（JVM 内部使用同步机制），但自行实现字符串池时必须显式处理并发写入的竞态条件，通常采用 `ConcurrentHashMap` 或读写锁来保护共享池结构。
