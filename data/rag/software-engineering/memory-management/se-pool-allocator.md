# 池分配器

## 概述

池分配器（Pool Allocator）是一种专门针对**固定大小对象**设计的内存管理方案，通过预先分配一整块连续内存区域，再将其切分为等大小的"槽"（slot），使每次对象的分配和回收操作均可在 **O(1)** 时间复杂度内完成。与 `malloc`/`free` 这类通用分配器相比，池分配器完全绕过了内存碎片搜索和元数据维护等耗时步骤。

池分配器的思想最早可追溯到20世纪70年代操作系统内核的 slab 分配器雏形。Linux 内核在 1994 年（2.0 版本前后）引入的 **SLAB 分配器**（由 Jeff Bonwick 在 Solaris 中首次系统化提出，1994 年 USENIX 论文发表）即是池分配器理念的工程化实现，专门用于内核对象（如 `task_struct`、`inode`）的快速复用。

对于游戏引擎、网络服务器、实时系统等需要**高频率、短生命周期**对象创建销毁的场景，通用分配器引入的延迟抖动是不可接受的。池分配器将分配延迟从通常的数百纳秒压缩到**个位数纳秒级别**，同时消除了堆碎片化风险，这是它在性能敏感系统中被广泛采用的根本原因。

## 核心原理

### 空闲链表（Free List）机制

池分配器的核心数据结构是**侵入式空闲链表**（intrusive free list）。初始化时，分配器将整块内存区域按对象大小 `S` 字节切分成 `N` 个槽，并在**每个空闲槽的头部**写入指向下一个空闲槽的指针，形成单链表。这种"侵入式"设计意味着不需要额外的元数据存储空间——空闲槽的内存本身就充当链表节点。

```
[ptr→slot2][   slot1  ] → [ptr→slot3][   slot2  ] → [NULL][   slot3  ]
```

**分配**：取出链表头节点，将 `head` 指针移动到下一个空闲槽，返回当前槽地址。伪代码：
```
allocate():
    if head == NULL: return NULL  // 池已满
    ptr = head
    head = *head                  // head 指向下一个空闲槽
    return ptr
```

**回收**：将归还的槽插入链表头部：
```
deallocate(ptr):
    *ptr = head    // 在 ptr 处写入旧 head 地址
    head = ptr     // 更新 head
```
两个操作均只涉及**1~2次指针赋值**，故时间复杂度严格为 O(1)。

### 内存布局与对齐要求

池分配器要求每个槽的大小至少为 `sizeof(void*)` 字节（在64位系统上为8字节），因为空闲槽需要存储一个指针。若请求的对象大小小于8字节，分配器必须将槽大小**向上对齐到8字节**。此外，为了保证 SIMD 指令或 `double` 类型的正确访问，槽起始地址通常还需按 **16字节边界**对齐，因此实际槽大小计算公式为：

```
slot_size = align_up(max(object_size, sizeof(void*)), alignment)
```

其中 `align_up(x, a) = (x + a - 1) & ~(a - 1)`。

### 池扩容策略

单块内存池耗尽时，有两种处理策略：**固定池**直接返回 `NULL`，由上层决定失败处理逻辑，适用于实时嵌入式系统（如 FreeRTOS 的 `heap_4.c` 实现）；**动态扩展池**则向底层分配器申请新的内存块，将多个块组成**块链表**（block list），每个块内部仍维持 O(1) 的空闲链表。扩容操作本身是 O(1) 的块申请加 O(N) 的新块初始化（N 为新块槽数），但均摊到每次分配仍接近 O(1)。

## 实际应用

**游戏引擎中的粒子系统**：Unreal Engine 的粒子模块使用固定大小的 `FParticle`（约128字节）结构体，通过池分配器维护一个预分配的粒子对象池，在每帧可能产生数千个粒子的场景下避免堆碎片，保证帧率稳定。

**网络服务器的报文缓冲区**：Nginx 使用 `ngx_pool_t` 结构管理请求级别的小对象。其中对于固定大小的连接对象 `ngx_connection_t`，使用池分配器预分配，使高并发下（如 10 万并发连接）的连接建立延迟保持可控。

**C++ 标准库适配器**：Boost.Pool 库提供的 `boost::pool<>` 模板类是池分配器的典型实现，可通过 `pool_allocator` 与 `std::list`、`std::set` 等容器集成，替换其默认的 `std::allocator`，在频繁增删节点时将分配耗时降低约 **3~5 倍**（依据 Boost 官方基准测试）。

**操作系统内核**：Linux 的 `kmem_cache_alloc()` 函数本质上就是对特定大小对象池的分配接口，例如 `kmem_cache_create("sock", sizeof(struct sock), ...)` 为 socket 对象创建专属池，使内核网络栈每秒可处理百万级套接字创建销毁。

## 常见误区

**误区一：认为池分配器适用于任意大小对象**。池分配器的 O(1) 性能来源于所有槽**尺寸完全相同**这一前提。若将其用于可变大小对象，就必须按最大对象尺寸对齐所有槽，造成严重的内部碎片浪费。例如，若对象大小在 8~256 字节之间变化，所有槽都必须是 256 字节，内存利用率可能低至 3%。可变大小场景应使用**分级池**（slab 的 size class 机制）或 `jemalloc` 等通用分配器。

**误区二：回收时无需清零内存，因此安全性等同于 `free`**。池分配器回收后的槽仍位于进程地址空间内，且空闲槽头部被**复写为链表指针**。如果存在悬空指针（use-after-free），攻击者可以通过读取已回收槽的头部字节获取下一个空闲槽的地址，进而实施**堆布局攻击**（heap feng shui）。安全敏感场景需在回收时主动清零槽内容，或使用带 canary 的调试版实现。

**误区三：池分配器完全消除了内存碎片**。池分配器消除的是**外部碎片**（external fragmentation），但如果大量槽被分配后仅少数归还，整个内存块无法释放回操作系统，形成**内部保留浪费**。例如一个拥有 1000 个槽的池，若始终有 1 个槽被占用，则整块内存（假设每槽 64 字节，共 64KB）永远无法归还给 OS，这与通用分配器的行为不同。

## 知识关联

**前置概念**：理解**内存分配器**（`malloc` 的 first-fit/best-fit 算法及其 O(n) 搜索开销）能直接解释为何池分配器的 O(1) 是可贵的——通用分配器维护复杂的空闲链表元数据，而池分配器用"尺寸单一"换来了元数据极简化。**对象池**（Object Pool）模式是池分配器的上层抽象：对象池负责对象的构造/析构生命周期管理，而池分配器只负责原始内存的分配/回收；两者分工不同，实际工程中对象池通常以池分配器作为底层存储后端。

**横向关联**：与池分配器并列的还有**线性分配器**（Bump Allocator / Arena Allocator），后者支持任意大小分配但只能整体一次性回收，无法单独释放槽；以及**伙伴系统**（Buddy System），用于操作系统页级别的 2^n 大小分配，具有 O(log n) 复杂度。三者的选择取决于对象大小均匀性和生命周期模式。
