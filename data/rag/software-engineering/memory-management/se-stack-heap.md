# 栈与堆

## 概述

栈（Stack）与堆（Heap）是程序运行时内存的两种主要分配区域，分别服务于自动生命周期变量和动态生命周期变量。栈由编译器自动管理，函数调用时向低地址方向增长（x86架构），函数返回时自动收缩；堆则由程序员或运行时库手动控制，通过 `malloc`/`free`（C语言）或 `new`/`delete`（C++）显式申请与释放。两者共享同一块物理内存，但分配策略截然不同。

栈的概念源于1946年图灵对子程序调用机制的描述，现代CPU专门设计了栈指针寄存器（x86-64中为 `rsp`）以硬件级别支持栈操作。堆的动态分配则随ALGOL 60语言（1960年）引入，后来演化为C标准库中 `malloc` 的实现形式。二者的存在使程序既能高效管理短暂的局部变量，又能灵活处理运行时才能确定大小的数据结构。

理解栈与堆的区别，直接影响程序的性能调优和内存安全决策。在现代服务器程序中，频繁的堆分配是导致系统延迟抖动的常见原因之一，而栈溢出（Stack Overflow）则是嵌入式系统崩溃的高频故障类型。

## 核心原理

### 栈的分配机制：线性时间恒定

栈分配的时间复杂度为 **O(1)**，本质上只需移动一次栈指针（`rsp -= N`）。局部变量的大小在编译期确定，编译器在函数序言（prologue）中一次性为所有局部变量预留空间。例如：

```c
void foo() {
    int a = 1;      // rsp - 4
    double b = 2.0; // rsp - 8
}  // 函数返回时 rsp 自动复原
```

典型的线程栈大小为 **8 MB**（Linux默认值，可通过 `ulimit -s` 查看），超出即引发段错误。栈帧（Stack Frame）包含：返回地址、保存的寄存器、局部变量、传参区域四部分，由调用约定（如 System V AMD64 ABI）严格规定布局。

### 堆的分配机制：动态但代价昂贵

堆分配调用 `malloc` 时，底层需要在空闲链表（Free List）中搜索满足大小要求的块，常见算法包括首次适配（First Fit）、最佳适配（Best Fit）和伙伴系统（Buddy System）。Linux 的 `glibc` 使用 **ptmalloc2**，Windows 使用低碎片堆（LFH, Low Fragmentation Heap）。分配一次 `malloc` 的平均耗时约为 **50–200纳秒**，而栈分配仅需 **1–2条CPU指令**（约0.3纳秒）。

堆内存存在元数据开销：glibc 的每个堆块头部占用 **16字节**（64位系统），记录块大小和占用状态，这意味着申请1字节实际消耗至少32字节（含对齐填充）。

### 内存碎片与局部性

堆的长期使用会产生**外部碎片**：多次申请释放后，空闲空间被切割成不连续的小块，导致即使总空闲内存充足也无法满足大块申请。栈则因线性增长/收缩不存在碎片问题。

从CPU缓存角度，栈变量天然具备良好的**空间局部性**——相邻的局部变量地址连续，容易命中L1缓存（通常32 KB）。堆对象因分散分布，cache miss率更高，这是堆访问比栈访问慢的深层原因之一。

## 实际应用

**场景一：递归深度限制**  
递归函数每层调用在栈上压入一个新的栈帧。若递归深度过大，如对10万节点的链表递归遍历，栈帧累积可能突破8 MB限制。解决方案是将递归改写为使用堆分配的显式栈结构，将无限递归的风险转移到可控的堆内存。

**场景二：大型缓冲区的选择**  
在处理网络数据包时，若缓冲区大小固定（如512字节），优先放在栈上以利用O(1)分配速度；若需根据HTTP头部动态确定缓冲区大小，则必须使用堆分配。`alloca()` 函数可在栈上动态分配，但因不安全（无边界检查）在生产代码中慎用。

**场景三：性能敏感路径**  
游戏引擎的每帧渲染循环（目标60 fps，即每帧预算约16.7毫秒）中，频繁调用 `malloc` 会引入不确定延迟。Unreal Engine、Unity 等引擎为此实现**帧内存池（Frame Allocator）**：在帧开始时从堆预分配一大块内存，帧内的临时对象全部在这块内存上以栈方式线性分配，帧结束后批量释放，兼顾堆的灵活性和栈的速度。

## 常见误区

**误区一："堆比栈大，所以大数据应总放堆上"**  
堆虽然上限更大（通常受虚拟地址空间限制，64位系统理论上达128 TB），但这不代表"所有较大数据都适合放堆"。对于大小固定且生命周期与函数一致的数组（如 `char buf[4096]`），放栈上是完全合理的，避免了堆分配的延迟和碎片风险。真正需要堆的条件是：**大小运行时才确定**，或**生命周期需要超出当前函数作用域**。

**误区二："栈分配一定比堆快，所以应尽量用栈"**  
这忽略了栈大小的硬性限制。在多线程程序中，每个线程独立持有一个8 MB栈，创建1000个线程仅栈内存就消耗8 GB。滥用栈（如在栈上声明超大数组）会导致栈溢出，其症状往往是静默地覆盖相邻栈帧数据，引发极难调试的行为异常。

**误区三："free/delete 之后内存立刻归还操作系统"**  
`free()` 通常只是将内存块标记为空闲并归还给 `malloc` 的内部空闲链表，并不立即触发 `sbrk()` 或 `munmap()` 系统调用归还给OS。只有当堆顶连续空闲区域超过特定阈值（glibc默认128 KB）时，才触发实际收缩。这意味着进程的RSS（Resident Set Size）在free后可能不会立即下降。

## 知识关联

**前置概念——内存管理概述**：栈与堆是虚拟地址空间布局中"栈段"和"堆段"的具体实现，理解虚拟内存分段是掌握两者边界行为的前提。

**后续概念——智能指针（`std::unique_ptr`/`std::shared_ptr`）**：智能指针本身在栈上分配（控制块对象），但所管理的资源在堆上。`unique_ptr` 的析构函数在栈帧弹出时自动调用 `delete`，正是利用了栈的确定性生命周期来管理堆资源。

**后续概念——RAII**：RAII 惯用法的核心逻辑依赖栈对象的确定性析构顺序（后进先出），将堆资源的释放绑定到栈对象的生命周期终点，从根本上解决了手动 `free` 遗漏的问题。

**后续概念——内存分配器**：自定义内存分配器（如 jemalloc、tcmalloc）的设计目标之一，正是通过线程本地缓存（Thread-Local Cache）和大小类（Size Class）减少堆分配与栈分配之间的性能差距，最终在堆上模拟类似栈的快速分配路径。
