# 栈与堆

## 概述

栈（Stack）和堆（Heap）是程序运行时内存空间中两种截然不同的区域，分别承担着自动生命周期管理和动态生命周期管理的职责。栈由编译器自动管理，以后进先出（LIFO）顺序分配和释放内存；堆则由程序员（或垃圾回收器）显式控制，允许在运行时任意申请和释放内存块。

栈内存的概念可以追溯到1946年Alan Turing对子程序调用机制的描述，而现代操作系统为每个线程分配独立的栈空间——在Linux x86-64系统上，默认栈大小通常为8MB（可通过`ulimit -s`查看）。堆则对应进程的堆段（heap segment），由C标准库的`malloc`/`free`或C++的`new`/`delete`操作符管理。

理解两者的差异对于编写高性能、无内存泄漏的程序至关重要：选错存储区域可能导致栈溢出（stack overflow）、堆碎片化，甚至程序崩溃。

## 核心原理

### 栈的分配机制与地址增长方向

栈的分配本质上是移动栈指针（Stack Pointer，寄存器`rsp`在x86-64中）。进入函数时，编译器生成`sub rsp, N`指令将栈指针向低地址方向移动N字节（x86架构下栈向低地址生长），在栈帧中为局部变量预留空间；函数返回时，`add rsp, N`恢复指针，这一操作的时间复杂度为O(1)，耗时通常不超过几个时钟周期。

```c
void foo() {
    int a = 10;       // 栈上分配：rsp -= 4
    double b = 3.14;  // 栈上分配：rsp -= 8
}   // 函数返回时，rsp自动恢复，a和b的内存同时被回收
```

这种机制决定了栈变量具有严格的作用域生命周期：离开作用域即被销毁，无法在函数外部安全访问。

### 堆的分配机制与碎片化问题

堆由内存分配器（如glibc的`ptmalloc2`，或tcmalloc、jemalloc等替代实现）管理。调用`malloc(size)`时，分配器需要在空闲块链表中找到足够大的空闲块，这一搜索操作的时间复杂度在最坏情况下为O(n)。堆的地址空间通常从低地址向高地址增长，由系统调用`brk()`或`mmap()`从操作系统获取新内存页。

反复申请、释放不同大小的内存块会产生**外部碎片**（external fragmentation）：空闲块分散在堆中，即使总空闲内存充足，也可能无法满足一次较大的连续内存请求。例如，交替分配16字节和32字节的块后随机释放，碎片率可超过50%。

### 性能差异的量化比较

| 指标 | 栈 | 堆 |
|------|----|----|
| 分配速度 | ~2纳秒（移动寄存器） | ~100纳秒（ptmalloc2平均） |
| 最大空间 | 8MB（Linux默认） | 受虚拟地址空间限制（64位系统约128TB） |
| 线程安全性 | 天然隔离，无需锁 | 需要互斥锁（ptmalloc2使用arena锁） |
| 缓存局部性 | 极高（栈顶数据常驻L1缓存） | 较低（碎片化导致缓存未命中率上升） |

堆分配比栈分配慢约50倍的原因不仅是搜索空闲块的开销，还包括多线程环境下的锁竞争以及缓存行污染。

### 变量存储的决定规则

编译器根据以下规则决定变量分配位置：
- **栈**：函数内的局部变量、函数参数、返回地址（如`int x = 5;`、`char buf[64];`）
- **堆**：通过`malloc`/`new`动态分配的内存（如`int* p = new int(5);`）
- **栈上逃逸到堆**：Go语言中若变量生命周期超出函数范围，编译器通过**逃逸分析**（escape analysis）自动将其移至堆上

## 实际应用

**场景一：递归深度限制**  
递归函数每次调用都在栈上创建新帧。以默认8MB栈为例，若每个栈帧占用1KB（包含局部数组），最大递归深度约为8192层。超过此限制将触发段错误（SIGSEGV），即常见的"stack overflow"。解决方案是将深度递归改写为显式栈（使用`std::stack`在堆上模拟），或使用尾递归优化。

**场景二：大型缓冲区的位置选择**  
处理视频帧时，一帧1920×1080的RGB图像需要约6MB内存，超过Linux默认栈大小，必须使用堆分配（`malloc(6220800)`）。若误放栈上，程序会在函数入口处立即崩溃，这是实际开发中常见的启动期崩溃原因。

**场景三：多线程下的分配性能**  
高并发服务器中，多线程频繁调用`malloc`会因ptmalloc2的全局锁造成性能瓶颈。Facebook的jemalloc通过为每个线程维护独立的**线程缓存（tcache）**，将线程局部的小对象分配（≤14KB）的锁竞争降至接近零，这是Redis 3.0以后切换至jemalloc的核心原因。

## 常见误区

**误区一："堆比栈慢，所以应尽量用栈"**  
这一说法在小对象场景下成立，但忽略了一个事实：当需要存储的对象体积超过几百字节或生命周期需要跨越函数边界时，栈根本无法胜任。正确的判断标准应综合考虑对象大小、生命周期、访问模式三个维度，而非单一地追求栈分配。

**误区二："局部变量一定在栈上"**  
在开启优化的编译器中，小的局部变量可能被完全优化进寄存器（register allocation），从不占用栈空间。反之，在Go、Java中，局部变量经逃逸分析后可能被分配到堆上。"局部变量在栈上"只是C/C++在未开启激进优化时的默认行为，并非通用定律。

**误区三："free/delete之后内存立即归还操作系统"**  
`free()`通常只将内存归还给分配器管理的空闲链表，不会立即触发`brk()`缩减或`munmap()`系统调用。这意味着进程的RSS（Resident Set Size）可能长时间不降低，这是使用`top`监控内存时容易误判为"内存泄漏"的原因。jemalloc相比glibc默认分配器更积极地将内存归还OS，这也是其在内存敏感型服务中受欢迎的原因之一。

## 知识关联

**前置知识**：理解栈与堆需要先掌握**进程内存布局**（代码段、数据段、栈段、堆段的位置关系）以及函数调用时寄存器的保存机制，这些属于**内存管理概述**中的基础内容。

**后续概念**：**智能指针**（`std::unique_ptr`、`std::shared_ptr`）正是为了解决手动管理堆内存容易出现的泄漏和悬空指针问题而设计——它们在栈上创建对象包装器，利用栈的自动销毁特性触发堆内存的释放，将两种内存区域的优势结合起来。**RAII**（Resource Acquisition Is Initialization）模式的核心也依赖于栈对象析构的确定性：将堆资源的生命周期绑定到栈对象的作用域上。而**内存分配器**（tcmalloc、jemalloc）则专门针对堆分配器的碎片化和锁竞争问题提供优化实现，是本主题性能分析的直接延伸。
