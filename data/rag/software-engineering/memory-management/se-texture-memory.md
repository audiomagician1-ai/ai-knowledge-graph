# 纹理内存管理

## 概述

纹理内存管理是GPU显存资源调度的专项技术，专注于控制游戏或图形应用中图像贴图数据在CPU内存与GPU显存之间的分配、传输与释放。与通用内存管理不同，纹理数据具有高度不连续的访问模式（相邻像素在屏幕空间与纹理空间的映射关系不固定），因此需要专门的缓存结构和压缩格式支持。

纹理内存管理技术随3D图形硬件发展而演进。1999年Quake III Arena引入了MIP贴图的LOD偏置控制，2000年代初DirectX 9引入了DXT压缩格式（即S3TC标准），将纹理存储密度提升至原来的4至8倍。2015年前后，随着虚拟纹理（Virtual Texturing）和稀疏纹理技术（DirectX 11.2的Tiled Resources，OpenGL的ARB_sparse_texture）成熟，纹理内存管理进入按需映射的时代。

纹理内存往往是移动平台与主机平台最紧张的显存类型。以PlayStation 5为例，其16 GB GDDR6显存中，AAA游戏的纹理预算通常占据6至10 GB，超出预算将触发强制降级或画面撕裂。精准的纹理内存管理决定了在固定硬件预算下能呈现的视觉保真度上限。

## 核心原理

### Mipmap层级与显存占用的计算

Mipmap是为同一纹理预生成的一系列分辨率递减的版本，每个层级的宽高各减半。对于一张1024×1024的RGBA8纹理（每像素4字节），原始数据为4 MB；完整Mipmap链从1024×1024一直到1×1共11个层级，总占用为原大小的4/3倍，即约5.33 MB。公式为：

**总显存 = 原始大小 × (1 + 1/4 + 1/16 + ...) ≈ 原始大小 × 4/3**

当相机距离物体较远时，GPU自动采样低层级Mipmap，减少纹理缓存失效（cache miss），提升渲染性能。禁用Mipmap看似节省33%显存，但实际会导致GPU纹理缓存命中率下降30%至50%，渲染耗时反而上升。

### 压缩格式选择策略

不同压缩格式针对不同平台和纹理类型设计，选择错误会导致显存浪费或画质损失：

- **BC1（DXT1）**：无Alpha通道或1-bit Alpha，压缩比8:1，每像素0.5字节，适合漫反射颜色纹理。
- **BC3（DXT5）**：带平滑Alpha通道，压缩比4:1，每像素1字节，适合带透明度的UI元素或植被。
- **BC5**：双通道格式，压缩比4:1，专用于法线贴图（仅存储X/Y分量，Z分量通过计算还原），比BC3存储法线贴图节省50%显存且减少色差伪影。
- **BC7**：高质量RGBA压缩，压缩比4:1，适合需要高保真的漫反射或自发光纹理，压缩速度较BC1慢约100倍，因此应离线预压缩而非运行时压缩。
- **ASTC（Adaptive Scalable Texture Compression）**：移动平台标准，支持4×4至12×12的可变块大小，压缩比从2:1到36:1可调，iOS A8芯片起和Android Vulkan设备均原生支持。

选择原则：法线贴图固定用BC5，移动端所有纹理优先用ASTC 6×6（压缩比约9:1），PC端漫反射用BC7，透明纹理用BC3。

### 流式加载（Texture Streaming）机制

流式加载的核心思想是根据相机与物体的距离，仅在GPU显存中保留当前可见Mipmap层级所需的数据，其余层级留在系统内存或磁盘上。Unreal Engine 4的Texture Streaming系统以每帧为单位计算每张纹理的"所需Mip级别"（Required Mip Level），计算公式基于屏幕占用像素数（Screen Size）与纹理分辨率的比值：

**所需MIP = log₂(纹理宽度 / 屏幕覆盖像素数)**

流式加载需维护一个优先队列，高优先级纹理（距离近、屏幕占比大）优先上传至显存。上传带宽是瓶颈：PCIe 4.0的CPU→GPU传输峰值约16 GB/s，但实际可调度带宽通常限制为2至4 GB/s以避免阻塞渲染管线。对于超大开放世界，还需结合"纹理池（Texture Pool）"机制设置硬性显存上限，当池满时强制驱逐最低优先级的Mipmap层级。

## 实际应用

**《赛博朋克2077》的流式纹理分级**：该游戏将纹理按类型分为三个显存优先级层：角色皮肤与近景道具（最高优先级，常驻2至3个最高Mip层级），中景建筑（中等优先级，按需加载），远景贴花（最低优先级，仅保留Mip 3及以下）。这一策略使4 GB显存显卡与16 GB显存显卡在近景视觉质量上差距缩小至可接受范围。

**移动游戏的ASTC分档**：《原神》在iOS上对角色纹理使用ASTC 4×4（高质量），对地形使用ASTC 8×8（节省70%显存），对UI图集使用ASTC 6×6（平衡画质与内存）。这种分档策略使游戏在2 GB RAM的低端设备上将纹理总占用控制在300 MB以内。

**虚拟纹理在地形中的应用**：Unity HDRP的Virtual Texturing系统将地形纹理按128×128像素的Page分割，仅将当前帧相机可见区域的Page上传至一张固定大小的Physical Texture Cache（默认1024×1024），从而使理论上无限大的地形纹理在显存中的占用固定为16 MB以内。

## 常见误区

**误区一：压缩纹理一定会损失质量，应尽量使用未压缩格式**

这是错误认知。BC7和ASTC 4×4的压缩伪影在大多数漫反射纹理上人眼不可察觉，而使用未压缩RGBA8代替BC7意味着4倍显存消耗。在显存受限平台上，未压缩格式反而会因显存超出预算而导致频繁的Mip降级，画质总体更差。真正不适合压缩的纹理仅有高精度HDR环境贴图（使用BC6H）和需要精确数值的数据纹理（如骨骼动画的查找表纹理）。

**误区二：流式加载会消除所有加载卡顿，纹理预算因此可以忽略**

流式加载依赖异步上传，当玩家快速移动或场景切换速度超过IO带宽时，仍会出现纹理在低分辨率Mip层级停留的"纹理弹入（Texture Pop-in）"现象。2023年《星空》发售时频繁被玩家批评的画质问题之一，正是过于激进的流式策略导致中景纹理长时间停留在Mip 3。正确做法是设置合理的显存预算下限，保证高频可见纹理的前2至3个Mip层级常驻显存，而非完全依赖流式系统动态调度。

**误区三：Mipmap始终应该生成完整链至1×1**

对于UI纹理和粒子特效纹理，完整Mipmap链反而有害：UI元素通常以像素精确（Pixel Perfect）方式渲染，低层级Mip会导致缩放时出现模糊；粒子纹理若采样到低层级Mip会产生颜色偏移。正确做法是对UI纹理禁用Mipmap并开启双线性过滤，对粒子纹理将Mipmap层数限制为2至3级（最大512×512至256×256），而非生成至1×1。

## 知识关联

纹理内存管理建立在**内存预算管理**的基础上：必须首先明确平台总显存预算（如主机16 GB中纹理分配8 GB），才能为流式加载设置有意义的纹理池上限和驱逐阈值。内存预算管理提供的"高水位线（High-Water Mark）"监控方法直接用于检测纹理内存泄漏——当纹理池持续增长而不回落时，通常是流式卸载回调未被正确触发。

在纹理内存管理之上，工程师可进一步研究**虚拟几何体（Nanite/Virtual Geometry）**技术，该技术将类似的流式加载思路扩展至几何体网格数据；以及**渲染管线状态管理**，其中描述符堆（Descriptor Heap）的设计方式与纹理内存的绑定效率直接相关。掌握BC格式压缩原理后，还可延伸至**视频解码内存管理**，两者共享块压缩（Block Compression）的空间局部性优化思路。
