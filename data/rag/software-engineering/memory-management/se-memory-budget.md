# 内存预算管理

## 概述

内存预算管理（Memory Budget Management）是游戏开发中预先规划并严格约束各系统内存用量的工程实践。与普通软件不同，游戏必须在固定硬件平台上同时运行渲染、音频、物理、AI等多个子系统，每个子系统的内存消耗需要在开发初期就明确上限，否则任何一个系统的"超支"都可能导致整体崩溃或严重卡顿。

这一实践伴随主机游戏的商业化而成熟。PlayStation 2（2000年发布）仅拥有32 MB主内存和4 MB视频内存，开发者被迫精确记录每一个系统的内存占用，由此催生了系统化的预算文档和分配流程。现代平台的内存容量虽然大幅提升（PS5拥有16 GB统一内存），但游戏内容量的增长速度与之同步，内存预算管理依然是避免内存超限（Out-of-Memory，OOM）崩溃的关键工程手段。

内存预算管理的核心价值在于：它将"内存不足"这个运行时崩溃问题，转化为开发阶段可量化、可审查的工程指标。通过在立项阶段就制定分配表，团队可以在添加新功能时立即评估内存代价，而不是等到发布前才发现超限。

## 核心原理

### 平台内存限制与可用预算计算

每个目标平台都有固定的物理内存上限，但游戏实际可用的内存远小于这个数字。以PS4为例：物理内存8 GB中，约1 GB由操作系统和社交功能占用，另有约512 MB用于截图和流媒体缓冲，开发者真正可用的预算约为5.5 GB。这意味着可用预算 = 物理内存总量 - 系统保留 - 运行时开销。

在PC平台，由于用户硬件差异巨大，通常以最低配置要求（如"最低8 GB RAM"）为基础，预留操作系统约2 GB占用后，制定低配预算方案，再根据高配硬件提供动态扩展选项。

### 预算分配表结构

标准内存预算表将总可用内存按子系统拆分为若干固定额度（hard budget）。以一款典型的PS4游戏为例，一个可能的分配方案如下：

| 子系统 | 分配额度 |
|--------|---------|
| 纹理与图形资源 | 2000 MB |
| 游戏逻辑与实体 | 512 MB |
| 音频资源 | 256 MB |
| 物理模拟 | 128 MB |
| 动画数据 | 384 MB |
| 网络与存档缓冲 | 64 MB |
| 调试与工具（最终版本移除）| 256 MB |
| 安全冗余 | 100 MB |

安全冗余（slack）通常保留总预算的1%～3%，用于吸收内存碎片化（fragmentation）带来的实际占用波动，避免因碎片导致连续内存分配失败。

### 静态内存与动态内存的分离管理

内存预算分为两类：启动时一次性分配的**静态内存**（如引擎核心数据结构、常驻UI资源）和运行时按需分配的**动态内存**（如关卡资产、NPC数据）。

静态内存需要在引擎初始化阶段使用内存池（memory pool）预先分配，其总量必须严格固定，不允许后期增长。动态内存则通过分配器（allocator）在关卡加载/卸载时管理，但其峰值（peak usage）同样受预算约束——开发者需要测量每个关卡的峰值内存占用，确保最重的关卡也不超过该子系统的分配额度。

使用公式表示：`Peak(关卡i，子系统j) ≤ Budget(子系统j)`，对所有关卡i和子系统j必须同时成立。

## 实际应用

**预算追踪工具集成**：大多数商业引擎（如Unreal Engine 5）内置了内存统计面板（Memory Insights），开发者可实时查看各类别内存占用并与预算对比。在自研引擎中，通常会在每日构建（daily build）的自动化测试中加入内存回归检测：若某次提交导致任意子系统超预算，CI/CD流水线会自动标记该提交为失败并通知相关开发者。

**关卡资产的流式加载预算**：在开放世界游戏中，关卡资产无法全部常驻内存，需要通过流式加载（streaming）动态换入换出。内存预算管理在此场景中表现为：为流式缓冲区分配固定大小（例如256 MB），引擎在该缓冲区满时必须卸载距离玩家最远的资产才能加载新资产，而非无限扩展占用。

**多人游戏中的额外开销**：网络对战游戏通常需要为每位连接玩家额外分配约4～16 MB的状态同步缓冲，预算表必须以最大玩家数为基准（如16人房间 × 8 MB = 128 MB），而非按平均在线人数计算，否则满员时会超限崩溃。

## 常见误区

**误区一：把平台总内存当作可用预算**。许多初学者看到PS5拥有16 GB统一内存，便以此为基础规划资产。实际上扣除系统占用、GPU驱动保留、运行时堆栈等开销后，游戏实际可支配的内存通常只有标称值的60%～70%。以总内存为预算上限会在最终集成测试时导致系统性超限，届时削减资产代价极高。

**误区二：只检查平均内存，忽视峰值**。内存崩溃发生在峰值时刻，而非平均状态。一个关卡的平均纹理内存可能是1.5 GB，但在加载屏幕结束、新区域资产全部载入而旧区域尚未卸载的瞬间，峰值可能达到2.2 GB。内存预算管理必须以峰值为约束目标，测量工具也必须记录峰值而非均值。

**误区三：调试内存在发布前必然全部节省出来**。开发阶段的调试工具（如内存统计、碰撞可视化、日志缓冲）会占用数百MB，但发布版本并非完全归零——崩溃报告模块、成就系统SDK、平台社交功能等第三方中间件通常需要保留50～100 MB，如果预算表在开发阶段没有明确区分"调试专用"和"发布版本保留"两类开销，很容易在送审阶段发现内存超限。

## 知识关联

内存预算管理不依赖复杂的前置知识，掌握基本的字节单位换算（1 GB = 1024 MB）和加减法即可开始制定预算表，这也是它在内存管理学习路径中位于最前端的原因。

在此基础之上，**纹理内存管理**是直接延伸的下一步：纹理资源在大多数游戏中占据最大单项预算（通常达到总可用内存的35%～50%），需要专门的压缩格式选择（如BC7、ASTC）、mipmap策略和流式加载技术来在预算约束内最大化视觉品质。纹理内存管理本质上是在执行"纹理子系统预算的精细化运营"——内存预算管理制定了这个子系统的总额度，而纹理内存管理研究如何在该额度内做到最优分配。
