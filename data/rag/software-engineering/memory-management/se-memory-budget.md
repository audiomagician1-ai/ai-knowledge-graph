# 内存预算管理

## 概述

内存预算管理是指在游戏或实时软件开发中，根据目标运行平台的物理内存上限，将有限的内存空间预先划分为若干固定额度（budget），分配给渲染、音频、游戏逻辑、物理模拟等不同子系统的策略体系。与通用软件"按需分配"的思路不同，游戏内存预算管理要求开发团队在立项初期就确定每个子系统能够使用的最大字节数，并在整个开发周期内严格执行，以防止在发布阶段出现因内存超限导致崩溃或性能急剧下降的问题。

这一概念起源于20世纪90年代的游戏主机开发实践。任天堂Super Famicom仅拥有128 KB工作内存，开发者必须对每个变量的存储位置精确计划；PlayStation 1的主RAM为2 MB，索尼官方开发指南中明确要求开发商提交内存使用分配表（Memory Map）才能通过审核。进入Xbox 360和PS3时代，两台主机分别提供512 MB统一内存和256 MB XDR主内存+256 MB GDDR显存的架构，内存预算管理从非正式规范升级为正式的项目管理流程。

内存预算管理的重要性在于：移动平台（iOS/Android）的内存警告机制（`didReceiveMemoryWarning` / `onTrimMemory`）会在应用占用内存超过系统阈值时强制终止进程，这一阈值在低端Android设备上可能仅有192 MB；主机平台如PS5虽然拥有16 GB GDDR6统一内存，但其中约1.5 GB被操作系统保留，超出系统可用额度的游戏将直接无法通过技术认证（TCR/TRC）。

## 核心原理

### 平台内存上限与可用空间计算

每个目标平台的实际可用内存并非等于硬件标称值。以Nintendo Switch为例，其搭载4 GB LPDDR4内存，但操作系统（Horizon OS）保留约约1 GB，游戏最多可申请3 GB，而其中GPU使用的显存与CPU主内存共享这一池，因此渲染预算和逻辑预算直接竞争同一物理资源。开发者必须从平台SDK文档获取"可用内存上限"，而非依赖硬件规格表。

可用预算总量的计算公式为：

```
可用总预算 = 硬件总内存 - OS保留 - 运行时引擎开销 - 安全余量
```

安全余量通常设定为可用总量的5%～10%，用于应对内存碎片（fragmentation）、异步加载缓冲区峰值以及未预料的驱动层分配。在实际项目中，Unreal Engine 5在PS5平台的引擎自身开销约为300～400 MB（依据启用的子系统不同而浮动）。

### 子系统预算划分方法

将可用总预算分配给各子系统有两种主流方法：**静态硬限制**（Static Hard Limit）和**动态预算池**（Dynamic Budget Pool）。

静态硬限制为每个子系统设定编译期常量，例如：
- 渲染子系统：占总预算的40%～50%
- 音频子系统：64 MB（固定值，涵盖流式缓冲区和解码缓冲区）
- 游戏逻辑/AI：100～150 MB
- 物理模拟：32 MB

当某子系统试图超出其静态上限时，内存分配器应触发断言（assert）或返回失败码而非静默扩展，这样可以在开发阶段而非测试阶段暴露问题。

动态预算池则允许子系统在运行时向中央内存管理器申请临时借用其他子系统的空闲额度，但借用须设定归还期限，适用于开放世界游戏中区域切换时渲染资源暂时膨胀的场景。

### 内存追踪与报告机制

仅划分预算而不追踪实际使用量，等同于没有预算管理。专业的内存追踪系统会为每次`malloc`/`new`操作附加分配标签（allocation tag），记录其所属子系统、分配调用栈和分配时刻。Unreal Engine提供的`Stat Memory`命令可以按分类实时显示各子系统内存占用；Radeon Developer Panel和RenderDoc则支持分析GPU显存分配。

项目应设立每日构建的自动化内存报告，若某子系统当日内存占用超过其预算的90%，应自动触发警告邮件通知责任人。这一阈值设在90%而非100%，是因为从发现超限到完成优化通常需要1～3个开发日。

## 实际应用

在移动游戏《原神》的iOS版本中，米哈游针对不同设备档次设定了三套内存预算方案：低端设备（2 GB RAM）限制纹理总量在350 MB以内，中端设备（4 GB RAM）放开至700 MB，高端设备（6 GB及以上）允许使用1.2 GB以上的纹理。这套分档策略直接体现了内存预算管理与硬件分级（Device Tier）的结合。

在主机游戏开发中，育碧团队在制作《刺客信条：奥德赛》时采用了以关卡区域为单位的预算管理：每个可见游戏区域的资源总量不得超过1.8 GB（PS4 Pro版本），区域切换时通过异步流式加载（async streaming）实现资源的换入换出，而非将所有资源常驻内存。

## 常见误区

**误区一：认为内存预算只需在项目末期优化时才考虑。** 实际上，若在项目开发中期才发现某子系统超出预算50%，往往需要重新制作大量美术资产（压缩纹理、减少多边形），改动成本远高于立项时即设定合理预算并实时监控。索尼第一方工作室的内部规范要求在Pre-Production阶段（通常为项目启动后3个月内）提交完整的内存预算文档。

**误区二：认为设备RAM越大，预算管理越不重要。** iOS系统在设备内存不足时，会以`JETSAM`机制按优先级杀死后台进程，但当前前台游戏自身占用过高时，同样会被系统终止。实测数据表明，占用超过设备总RAM 70%的应用在iPhone 14（6 GB RAM）上的崩溃率显著上升。大内存设备改变的是绝对数值，而非对预算纪律的需求。

**误区三：将内存预算等同于内存优化。** 预算管理是事前规划与持续监控，内存优化是针对已有超限问题的事后修复（如使用BC7格式将纹理压缩至原大小的25%）。两者均不可缺少，但预算管理侧重于建立约束框架，优化侧重于在约束内降低实际占用。

## 知识关联

内存预算管理是学习纹理内存管理的直接前提。纹理资源在典型3D游戏中占据渲染子系统内存预算的60%～80%，理解"渲染子系统被分配了多少总额度"之后，才能有意义地讨论如何在该额度内通过Mipmap、纹理图集（Texture Atlas）和流式加载调度纹理资源。没有预算框架，纹理内存管理的优先级决策（哪些纹理必须常驻、哪些可以卸载）将失去量化依据。此外，内存预算管理与虚拟内存分页、内存池分配器（Memory Pool Allocator）等底层机制密切相关，了解预算如何映射到物理页帧，有助于深入理解为什么碎片化会侵蚀有效可用预算。
