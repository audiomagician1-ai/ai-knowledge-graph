# 模板方法模式

## 概述

模板方法模式（Template Method Pattern）是一种行为型设计模式，其核心思想是在抽象基类中定义一个算法的固定骨架（即"模板方法"），而将算法中某些可变的步骤推迟到子类中实现。模板方法本身被声明为 `final`（在Java中）以防止子类修改整体流程，但骨架中调用的若干钩子方法或抽象方法则由子类按需覆写。

该模式由GoF（Gang of Four）在1994年出版的《设计模式：可复用面向对象软件的基础》中正式收录，归类于行为型模式。其动机来源于软件开发中一类典型问题：多个类共享相同的算法结构，但各自在某些具体步骤上存在差异，若每个子类独立实现完整算法，则会产生大量重复代码，且对公共流程的修改需要散点式地改动每个子类。

模板方法模式通过"好莱坞原则"（Hollywood Principle）实现控制反转——"不要调用我们，我们会调用你"。抽象父类掌控整体流程，在合适时机主动调用子类的覆写方法，而非由子类驱动父类行为。这使得公共逻辑集中在父类一处，变化被隔离在各自的子类覆写方法中。

## 核心原理

### 模板方法与钩子方法的角色划分

模板方法模式的类结构包含两类角色：`AbstractClass`（抽象类）和`ConcreteClass`（具体子类）。抽象类中包含：
- **模板方法（Template Method）**：`final` 方法，定义算法的完整步骤序列，不可被子类覆写。
- **抽象步骤方法（Abstract Step）**：声明为 `abstract`，子类**必须**实现，代表算法中强制可变的部分。
- **钩子方法（Hook Method）**：提供默认实现（通常为空或返回 `true`），子类**可选择性**覆写，用于在不改变算法流程的前提下插入额外行为。

以下是模板方法骨架的伪代码示意：

```java
abstract class DataProcessor {
    // 模板方法，final 锁定流程
    public final void process() {
        readData();       // 抽象步骤
        processData();    // 抽象步骤
        if (needsExport()) { // 钩子：默认 true
            exportData();
        }
    }
    protected abstract void readData();
    protected abstract void processData();
    protected void exportData() { /* 默认空实现 */ }
    protected boolean needsExport() { return true; } // 钩子
}
```

### 与继承机制的绑定关系

模板方法模式**强依赖继承**，这与策略模式依赖组合形成鲜明对比。子类与抽象父类之间存在强耦合：子类覆写的方法名称和签名必须与父类声明完全一致。这意味着一旦父类的模板方法修改了步骤顺序或新增了必须步骤，所有子类都会受到影响。因此，该模式特别适合那些**变化点明确且稳定**的算法骨架，而非频繁演化的流程。

### 抽象步骤的数量与粒度

GoF建议模板方法中的可覆写步骤数量应尽可能少——通常控制在**2至5个**为宜。步骤过多会导致子类实现负担沉重，且模板方法与子类之间的协作协议变得复杂难以理解。步骤粒度设计原则是：每个步骤应只负责一项独立的、内聚的操作，步骤之间不应通过父类的实例字段隐式传递中间状态，而应通过方法返回值显式传递。

## 实际应用

### Java标准库中的经典案例

Java标准库中 `java.util.AbstractList` 是模板方法模式的典型实现。`AbstractList` 为 `List` 接口提供了大量方法的默认实现，如 `indexOf`、`iterator` 等，其内部均依赖对 `get(int index)` 这一抽象方法的调用。子类 `ArrayList` 只需实现 `get`、`set`、`size` 等少量抽象方法，即可获得完整的列表行为。

### 测试框架中的setUp/tearDown流程

JUnit 3.x 的 `TestCase` 类使用模板方法模式组织测试生命周期：`runBare()` 是模板方法，它按顺序调用 `setUp()`、`runTest()`、`tearDown()`。测试开发者只需继承 `TestCase` 并覆写这三个方法，框架保证即使 `runTest()` 抛出异常，`tearDown()` 也一定会被执行——这种资源清理保障正是通过模板方法的结构实现的。

### 报表生成流程

企业应用中，生成不同格式（PDF、Excel、HTML）的报表往往共享相同的流程：读取数据源 → 格式化数据 → 渲染表头 → 渲染行数据 → 输出文件。可将此流程定义为模板方法，其中"渲染表头"和"渲染行数据"声明为抽象方法，由 `PdfReportGenerator`、`ExcelReportGenerator` 分别覆写，而"读取数据源"和"输出文件"则作为钩子方法提供默认实现。

## 常见误区

### 误区一：将模板方法设计为非final方法

许多初学者在父类中定义模板方法时忘记使用 `final` 修饰，导致子类可以意外地覆写整个流程而非仅覆写指定步骤。这从根本上破坏了模板方法模式的契约——算法骨架应当是稳定不变的，只有步骤是开放可变的。`final` 关键字不是可选的编码风格，而是该模式的结构性保障。

### 误区二：混淆模板方法模式与策略模式的使用场景

两者均用于封装算法的变化部分，但切换算法的代价截然不同。模板方法模式通过继承在**编译期**绑定算法变体，一个类实例只能对应一种算法变体；策略模式通过组合在**运行期**替换算法，同一对象可以在运行时切换不同策略。若业务要求在运行时动态更换处理逻辑，应选择策略模式，而非强行用模板方法模式的子类切换来模拟。

### 误区三：过度使用钩子方法导致流程不可预测

有些开发者为了"灵活性"在模板方法中添加大量布尔型钩子（如 `shouldDoStepA()`、`shouldDoStepB()`），使模板方法的实际执行路径高度依赖钩子的返回值组合。当钩子数量超过3个时，算法流程的可能分支数呈指数级增长（最多达 2^n 种），导致行为难以预测和测试。正确做法是将复杂条件逻辑提取为独立的子类，而不是在同一模板方法中堆砌钩子。

## 知识关联

### 与策略模式的对比

模板方法模式是学习策略模式之后自然延伸的比较对象。策略模式使用**对象组合**封装算法族，客户端持有一个可替换的策略接口引用；模板方法模式使用**类继承**封装算法骨架，父子类之间共享固定的调用协议。两者解决相似问题但采用相反的扩展机制，GoF原则"优先使用组合而非继承"暗示在多数新设计中策略模式的适应性更强，而模板方法模式则在算法骨架极其稳定、变化点有限的场景中更为简洁直接。

### 在设计模式体系中的位置

理解模板方法模式后，可以更准确地辨识工厂方法模式（Factory Method Pattern）——工厂方法模式本质上是模板方法模式在对象**创建**场景下的特化应用，模板方法负责对象创建流程，而具体产品的实例化由子类覆写的工厂方法决定。此外，在框架设计领域，模板方法模式是"控制反转"思想最直接的表达，理解它有助于掌握Spring框架中 `JdbcTemplate`、`RestTemplate` 等工具类的设计意图——这些类的 `execute` 方法正是将资源管理（打开连接、异常处理、关闭连接）固化为模板，将用户逻辑通过回调接口（即变体策略）插入其中。
