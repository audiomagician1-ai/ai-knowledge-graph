# 责任链模式

## 概述

责任链模式（Chain of Responsibility Pattern）是一种行为型设计模式，其核心机制是将多个处理对象串联成一条链，请求沿链逐节传递，直到某个处理者决定处理该请求为止。每个处理者持有对下一个处理者的引用，并自行决定是处理当前请求还是将其转发给后继者。这种结构将请求的发送方与实际处理方彻底解耦——发送方只需将请求投入链的入口，无需关心最终由哪个节点响应。

该模式最早由 GoF（Gang of Four）在 1994 年出版的《设计模式：可复用面向对象软件的基础》中正式定义。GoF 以图形界面中的帮助系统为例：用户点击某个控件时，若该控件自身无法提供帮助文档，请求便向上传递至父容器，直至找到能处理的节点。这一思想此后被大量移植到后端架构中，演变为现代 Web 框架中无处不在的"中间件管道"（Middleware Pipeline）概念，例如 Express.js 和 ASP.NET Core 均以此模式构建请求处理流程。

责任链模式在软件工程中的价值在于它天然支持"开闭原则"：向链中插入或移除一个处理节点，无需修改其他节点的代码，也无需改动请求的发送方。这使得审批流程、过滤器链、日志分级写入等场景中的规则扩展极为低成本。

## 核心原理

### 基本结构与三个角色

责任链模式包含三个固定角色：

- **Handler（抽象处理者）**：定义处理请求的接口，通常包含 `setNext(handler)` 方法和 `handle(request)` 方法，并持有一个指向下一处理者的引用 `successor`。
- **ConcreteHandler（具体处理者）**：实现 `handle` 方法，判断自身是否能处理该请求；若不能，则调用 `successor.handle(request)` 将请求沿链传递。
- **Client（客户端）**：负责组装链的顺序，并将请求送入链的头部节点。

以 Java 伪代码表达核心逻辑：

```java
abstract class Handler {
    protected Handler successor;
    public void setNext(Handler h) { this.successor = h; }
    public abstract void handle(int level);
}
```

处理条件通常以数值阈值、枚举类型或请求类型标识来表达，例如"只处理 level <= 3 的请求，其余转发"。

### 链的传递与终止条件

请求在链上存在两种终止方式：**被某节点消费**（该节点处理后不再调用 `successor`），或**到达链尾仍未被处理**（此时需决定是静默丢弃还是抛出异常）。区分这两种行为至关重要：纯过滤器链（如 Servlet Filter）通常要求每个节点调用 `chain.doFilter()` 以让请求继续流动，即所有节点均参与处理；而传统的审批链则是"一旦有节点处理即终止"，二者在语义上截然不同，混淆会导致逻辑漏洞。

### 与命令模式的协作关系

责任链模式的处理者通常接收的是一个封装了请求数据的对象，这正是**命令模式**（Command Pattern）所提供的"命令对象"。将请求封装为命令对象后，链上的每个处理者可以无副作用地检查命令内容（如读取命令类型、优先级字段），而不必直接依赖原始调用参数，这极大地增强了处理者之间的独立性。若链上存在大量相似的、状态无关的处理者实例，则可借助**享元模式**（Flyweight Pattern）共享这些实例的内部状态，避免为每条请求重复创建相同的处理者对象，降低内存开销。

### 动态链与静态链

链的组装时机影响系统灵活性。**静态链**在应用启动时一次性配置，适合规则固定的场景（如固定的日志级别过滤器）；**动态链**允许在运行时根据配置文件或数据库记录动态添加/删除/重排节点，例如电商平台的促销规则引擎，不同时段激活不同的折扣处理节点。动态链实现时，通常用链表或数组存储处理者序列，而非硬编码 `setNext` 调用。

## 实际应用

**Web 中间件管道**：Express.js 的中间件机制是责任链模式最典型的工业级实现。每个 `app.use(fn)` 注册的中间件函数接收 `(req, res, next)` 三个参数，调用 `next()` 即将请求传递给链上的下一个中间件，不调用则终止链。鉴权中间件、日志中间件、压缩中间件、路由处理器依次串联，形成完整的请求处理管道。

**企业审批流程**：OA 系统中，报销申请金额不超过 1000 元由组长审批，1000～10000 元由部门经理审批，超过 10000 元由 CFO 审批。每个审批节点是一个 ConcreteHandler，链的顺序即权限的层级顺序，新增审批层级只需插入新节点，现有节点代码零修改。

**Java Servlet Filter 链**：`javax.servlet.FilterChain` 接口的 `doFilter(request, response)` 方法即责任链的传递调用。容器按 `web.xml` 或注解声明的顺序构建过滤器链，字符编码过滤器、XSS 防护过滤器、Session 验证过滤器依次执行，每个过滤器均可在请求到达 Servlet 之前或之后插入自定义逻辑。

**日志框架分级输出**：Log4j 中，Logger 的日志事件若在当前 Logger 未被 Appender 完全消费，会沿父 Logger 传递（Additivity 机制），这正是责任链的变体应用，链的节点为各级 Logger 对象。

## 常见误区

**误区一：认为链上每个节点都必须处理请求**。实际上，责任链模式的本意是"有能力处理则处理，否则转发"，并非所有节点强制参与。混淆这一点往往是将"纯管道模式"（每个节点必须处理）与"纯传递模式"（只有一个节点处理）的语义边界模糊所致。在实现前必须明确本系统属于哪种语义，否则会出现请求被重复处理或漏处理的 Bug。

**误区二：将链的组装逻辑写入具体处理者内部**。部分初学者让每个 ConcreteHandler 在构造函数中直接 `new` 下一个处理者并设置为 `successor`，导致处理者之间产生硬依赖，无法单独复用或单元测试单个节点。正确做法是由 Client 或专门的 Builder/Factory 负责组装链，处理者自身只关心"处理或转发"的逻辑。

**误区三：忽视链过长时的性能问题**。责任链的时间复杂度在最坏情况下为 O(n)（n 为链的长度），若链中有 50 个处理节点且每次请求都需遍历到链尾，在高并发场景下会带来不可忽视的延迟。应根据实际命中率统计，将高频命中的处理者排在链的前段，或引入索引机制（如按请求类型直接路由到对应节点）以绕过不必要的遍历。

## 知识关联

**前置概念**：责任链模式在实现时经常以**命令模式**的命令对象作为链上传递的载体，理解命令对象的封装方式（将请求参数、执行者、操作统一包装）有助于设计出接口稳定的处理者。**享元模式**则在处理者数量庞大且处理者本身无状态时发挥作用，通过共享处理者实例减少对象创建开销，两种模式与责任链结合可在高吞吐系统中显著降低 GC 压力。

**后续概念**：学习责任链模式后，**访问者模式**（Visitor Pattern）是自然的延伸方向。访问者模式同样涉及将操作与对象结构解耦，但其解耦维度不同：责任链解耦的是"谁来处理请求"，而访问者解耦的是"对一组固定结构的对象施加哪种操作"。两者在 AST（抽象语法树）遍历等场景中常结合使用——链式结构负责节点间的遍历调度，访问者负责在每个节点上执行具体的分析或转换逻辑。
