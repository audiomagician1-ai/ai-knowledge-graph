# 中介者模式

## 概述

中介者模式（Mediator Pattern）是一种行为型设计模式，其核心思想是引入一个**中介对象**，将多个组件（称为"同事类"）之间的直接通信全部重定向至该中介对象，由中介者统一协调和转发，从而消除组件间的直接依赖关系。没有中介者时，N个组件之间的通信链路最多有 N×(N-1)/2 条；引入中介者后，每个组件只需与中介者建立1条连接，总链路数降至 N 条，网络拓扑从网状变为星状。

该模式由 GoF（Gang of Four）在1994年出版的《设计模式：可复用面向对象软件的基础》中正式归纳，灵感来源于航空交通管制系统——飞机不直接与彼此通信，而是统一通过控制塔（中介者）收发指令。这一现实类比精准描述了中介者模式的工作机制。

中介者模式在GUI框架、聊天室系统和游戏事件总线中被广泛使用。它最直接的价值在于：当项目中某个类因承担过多"协调职责"而膨胀时，中介者模式提供了一种将这些职责合法地提取到专门对象中的标准化方案。

## 核心原理

### 参与角色的四元结构

中介者模式由以下四个角色构成：

- **Mediator（抽象中介者）**：定义同事类与中介者通信的接口，通常包含 `notify(sender, event)` 方法。
- **ConcreteMediator（具体中介者）**：持有所有同事对象的引用，实现协调逻辑。
- **Colleague（抽象同事类）**：持有中介者的引用，只通过该引用发出通知。
- **ConcreteColleague（具体同事类）**：实现自身业务逻辑，状态变化时调用 `mediator.notify(this, event)` 通知中介者，而非直接操作其他同事。

这四个角色确保了**同事类之间零直接耦合**——ConcreteColleague A 无需 import 或持有 ConcreteColleague B 的引用。

### 通信流的单向汇聚

中介者中的协调逻辑遵循固定流程：同事类触发事件 → 中介者接收 `notify` 调用 → 中介者根据发送者和事件类型决策 → 中介者调用目标同事类的方法。以一个对话框为例：当用户勾选"记住密码"复选框时，复选框调用 `mediator.notify(this, "checked")`，具体中介者在 `notify` 内判断事件来源，然后调用密码输入框的 `setEnabled(false)` 方法。整个过程中，复选框与密码输入框之间没有任何代码依赖。

### 与观察者模式的结构性区别

中介者模式常被与观察者模式混淆，但两者结构目标完全不同。观察者模式中，Subject 广播事件，所有 Observer 自行决定如何响应，事件流是**一对多、去中心化**的。中介者模式中，事件必须经过中介者的**有状态判断**后才分发给指定同事，逻辑集中在中介者内，是**多对多、集中决策**的。一个具体区分标准：如果协调逻辑需要根据多个组件的**组合状态**（如"当 A 处于状态X 且 B 处于状态Y 时才触发 C"）做出决策，则应使用中介者模式而非观察者模式。

## 实际应用

**GUI 对话框协调**：Java Swing 或 Qt 的复杂对话框中，按钮、下拉框、输入框之间存在大量联动逻辑（如"仅当选择了某个选项时某按钮才可用"）。将 Dialog 本身实现为 ConcreteMediator，所有控件的事件回调都调用 `dialog.componentChanged(this)`，Dialog 内部集中处理所有启用/禁用逻辑。这是 GoF 原书中给出的标准案例。

**聊天室系统**：多用户聊天室中，用户 A 发送消息不应直接调用用户 B 的接收方法。ChatRoom 类作为中介者，持有所有 User 的列表。User 调用 `chatRoom.send(message, this)`，ChatRoom 负责将消息转发给除发送者外的所有用户。用户数从 2 人增至 100 人，协调逻辑只需修改 ChatRoom，User 类代码零改动。

**游戏事件总线（EventBus 模式）**：Unity 或 Unreal 引擎中，GameManager 充当中介者，接受 Player、Enemy、UI 等组件发出的事件（如 `PlayerDied`、`EnemySpawned`），并统一决定触发哪些后续动作。这一架构使得添加新的游戏系统时不需要修改现有的 Player 或 Enemy 代码。

## 常见误区

**误区一：中介者模式与外观模式等同**。外观模式（Facade）为复杂子系统提供**单向的简化接口**，外观对象不依赖调用方；中介者模式中，中介者与同事类之间是**双向依赖**——中介者持有同事的引用，同事也持有中介者的引用，两者相互协作。若将一个只负责"封装调用顺序"的类称为中介者，则是对该模式的误用。

**误区二：中介者模式必然消除复杂性**。将所有同事的协调逻辑集中到中介者后，中介者本身会随着系统增长而变成"上帝类"（God Class）。当 ConcreteMediator 中的 `notify` 方法出现超过10个 `if-else` 分支时，应考虑将中介者按职责拆分为多个中介者，或引入状态模式来管理中介者内部的复杂条件判断。中介者模式转移了复杂性，而不是消除它。

**误区三：所有同事必须使用同一个中介者接口**。实际上，一个系统中可以存在多个独立的中介者实例，分别负责不同功能域的协调（如 UIMediator、NetworkMediator、AudioMediator），同事类按照功能域持有对应中介者的引用。强行将所有协调逻辑塞进单一中介者是对该模式的过度使用。

## 知识关联

**前置概念——观察者模式**：观察者模式中事件订阅和广播的机制是理解中介者模式通信流的基础。中介者内部的"向目标同事发出调用"本质上是一种受控的单次通知，相当于观察者模式中 Subject 只通知指定 Observer 而非广播给全体。掌握观察者模式中"解耦发布与订阅"的思路，有助于理解中介者为何能打断同事间的直接引用链。

**前置概念——迭代器模式**：迭代器模式训练了"将遍历逻辑从集合中分离"的思维，这与中介者"将协调逻辑从同事类中分离"的思路结构一致，都属于职责分离的行为型模式。具体中介者在向多个同事分发事件时，常常需要遍历内部维护的同事列表，此处迭代器的使用是自然的组合。

**后续概念——享元模式**：享元模式关注对象实例的共享与内存优化。当中介者需要管理大量同事对象（如聊天室中的数千个用户）时，若这些同事对象包含大量相同的内部状态，享元模式可以与中介者模式结合使用——中介者持有享元工厂而非持有独立的同事实例，在保留集中协调架构的同时显著降低内存开销。
