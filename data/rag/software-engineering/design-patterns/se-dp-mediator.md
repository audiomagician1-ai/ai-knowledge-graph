# 中介者模式

## 概述

中介者模式（Mediator Pattern）是一种行为型设计模式，其核心思路是引入一个中介者对象来封装多个组件之间的交互逻辑，使各组件不再直接引用彼此，而是只与中介者通信。原本 N 个组件之间可能存在 N×(N-1)/2 条直接通信路径，引入中介者后，所有通信路径收敛为 N 条，每个组件只需知道中介者的存在。

中介者模式由 GoF（Gang of Four）在 1994 年出版的《设计模式：可复用面向对象软件的基础》中正式定义。最早的工程实践原型来自航空管制领域——机场的空中交通管制塔台充当所有飞机之间的调度中心，飞机无需两两通信，只需与塔台交换信息。这一现实类比至今仍是解释该模式最直观的例子。

在软件工程中，中介者模式解决的是"对象网状依赖"问题。当一个 UI 表单中有十几个控件（输入框、下拉框、按钮）相互依赖时，若让控件直接互相引用，任何一个控件的修改都可能引发连锁变更。引入表单级中介者后，每个控件只通知中介者"我发生了变化"，由中介者决定如何响应其他控件，从而将复杂交互逻辑集中在一处管理。

## 核心原理

### 参与角色与结构

中介者模式由四个角色构成：

- **Mediator（抽象中介者）**：定义组件与中介者通信的接口，通常包含 `notify(sender, event)` 这样的方法签名。
- **ConcreteMediator（具体中介者）**：实现协调逻辑，持有对所有具体组件的引用，并在组件触发事件时做出调度决策。
- **Component（组件基类）**：持有对中介者的引用，只通过中介者接口发出通知，不直接调用其他组件。
- **ConcreteComponent（具体组件）**：如按钮、输入框等，在自身状态变化时调用 `mediator.notify(this, "eventName")`。

组件与中介者之间的依赖是双向但受控的：组件持有抽象中介者引用，具体中介者持有具体组件引用。这种结构确保组件层不需要了解彼此的具体类型。

### 通信流程

当某个组件（如"提交按钮"）触发事件时，流程如下：

1. 具体组件调用 `mediator.notify(this, "submit")`；
2. 具体中介者在 `notify` 方法内，根据 sender 和 event 的组合判断应执行的逻辑；
3. 中介者直接调用其他组件的方法，例如禁用输入框、清空验证提示。

整个过程中，"提交按钮"对"输入框"的存在一无所知。中介者内部的条件判断（通常是 `if/switch` 结构）集中承载了所有跨组件协调规则。

### 与观察者模式的本质区别

中介者模式与观察者模式在代码形态上有相似之处，但意图截然不同。观察者模式建立的是"一对多"的单向通知关系：一个主题向多个订阅者广播，主题不关心订阅者之间如何协作。中介者模式建立的是"多对多"的协调关系：中介者主动了解所有组件并编排它们之间的复杂交互，通信是双向的。换句话说，观察者解耦"发布"与"订阅"，中介者解耦"所有参与者之间的相互依赖"。在实现上，中介者内部有时会使用观察者机制让组件向它注册，但这只是实现细节，两个模式的职责范畴不同。

## 实际应用

### GUI 对话框控件协调

这是 GoF 书中给出的经典场景。一个用户注册对话框中：选择"企业账号"后，"公司名称"输入框应变为可用；若"密码"与"确认密码"不一致，"注册"按钮应保持禁用。若让每个控件持有其他控件的引用，代码将形成蜘蛛网式依赖。以 `DialogMediator` 作为中介者，所有控件变更事件统一汇报给它，由它执行上述联动规则，每个控件的代码量可缩减约 60%~70%。

### 聊天室服务器

即时通讯系统中，服务器扮演中介者角色。每个客户端只与服务器建立连接，发送消息时调用 `server.send(this, message, targetRoom)`，服务器决定将消息转发给哪些客户端。这使得客户端之间完全解耦，客户端数量从 1 个扩展到 10000 个，每个客户端的代码无需任何修改。

### 航班调度与微服务编排

在微服务架构中，工作流编排器（如 Netflix Conductor）承担类似中介者的职责：各微服务不直接调用彼此，而是向编排器汇报执行结果，由编排器决定下一步触发哪个服务。这与中介者模式的思想高度一致，属于该模式在分布式系统层面的工程落地。

## 常见误区

### 误区一：中介者变成"上帝对象"

中介者集中了所有协调逻辑，如果不加约束，随着业务增长，具体中介者的代码会迅速膨胀成数百行密密麻麻的 `if-else`，反而成为最难维护的类。正确做法是将中介者的职责严格限定在"协调交互"，不让它承担业务逻辑计算；必要时可以将不同场景的协调逻辑拆分到多个具体中介者实现中。

### 误区二：将中介者与外观模式混淆

外观模式（Facade）提供一个简化接口来访问子系统，子系统内部各组件之间仍可直接通信，外观对象的目的是隐藏复杂性、提供便利入口。中介者模式的目的是切断组件之间的直接引用、改变通信拓扑结构，它与组件之间是双向依赖关系，而外观对象通常只持有子系统引用，子系统不知道外观的存在。

### 误区三：所有多对多通信都必须引入中介者

中介者模式的引入是有成本的：增加了一个额外的抽象层，且中介者内部逻辑难以做单元测试（因为它通常需要持有真实的组件实例）。当组件数量只有 2~3 个且交互规则简单时，直接引用反而更清晰。只有当组件间存在复杂的多向依赖，且未来变更频率较高时，引入中介者的收益才能超过成本。

## 知识关联

**前置知识——观察者模式**：学习中介者模式前需要掌握观察者模式中的"事件通知"机制，因为中介者内部常使用类似的 `notify/subscribe` 机制让各组件向它注册。但理解两者的区别是关键：观察者是去中心化的广播，中介者是中心化的调度，前者的"主题"不主动操控订阅者，而后者的中介者会主动操控各组件的状态。

**横向对比——命令模式与责任链模式**：命令模式将操作封装为对象以支持撤销/重做，责任链将请求沿处理链传递，两者都在处理"请求的路由"问题，但均不涉及多个对等组件之间的相互协调。中介者模式的独特价值在于处理"多个平等组件相互感知"的场景，而非单向的请求传递场景。

**架构层面延伸**：中介者模式的思想在更高抽象层次上体现为消息总线（Message Bus）和事件驱动架构（Event-Driven Architecture）。例如，Apache Kafka 在微服务中承担类似中介者的职责，各服务通过 Topic 发布和订阅事件，彼此不直接调用。理解中介者模式的组件解耦原理，是理解这类分布式消息系统设计理念的基础。
