# 享元模式

## 概述

享元模式（Flyweight Pattern）是一种结构型设计模式，其核心思想是**通过共享相同状态的细粒度对象来减少内存消耗**。当系统中需要创建大量相似对象，且这些对象的大部分状态可以外部化时，享元模式将对象状态分为"内部状态"（intrinsic state）和"外部状态"（extrinsic state），只在内存中保存不变的内部状态，从而让多个逻辑对象共享同一个物理对象实例。

享元模式由 GoF（Gang of Four）在 1994 年出版的《设计模式：可复用面向对象软件的基础》中正式归纳，其名称"Flyweight"来自拳击中的"次最轻量级"，寓意用极小的内存代价支撑大量对象的存在。该模式最早被用于解决文本编辑器中字符对象过多的问题——一篇文章中每个字母都创建独立对象会导致内存爆炸，而通过共享字符对象可将内存占用降低至原来的 1/100 甚至更少。

享元模式在内存受限环境中尤为重要。以一款地图应用为例，若地图上有 10 万棵树需要渲染，每棵树如果独立存储树种纹理、颜色等数据，每个对象占用 1MB，总共需要约 100GB 内存；而使用享元模式共享树种数据后，内存需求可降至数十 MB 量级。

## 核心原理

### 内部状态与外部状态的划分

享元模式的关键设计决策是将对象状态一分为二：

- **内部状态（Intrinsic State）**：不随环境变化的共享数据，存储在享元对象内部。例如字符的字体、字号；树木的纹理贴图、颜色。
- **外部状态（Extrinsic State）**：随上下文变化的非共享数据，由客户端在调用时传入。例如字符在文本中的坐标位置；树木在地图上的 x/y 坐标。

正确划分的原则是：**内部状态必须对所有使用该享元对象的客户端保持一致**，若某个属性在不同使用场景下会变化，它就必须作为外部状态处理。

### 享元工厂（Flyweight Factory）

享元模式引入一个**享元工厂**来管理享元对象池（pool）。工厂维护一个以内部状态为键的哈希表，当客户端请求一个特定内部状态的享元对象时，工厂先检查池中是否已存在，若存在则返回已有实例，若不存在则新建并缓存后返回。以下是核心逻辑的伪代码：

```
class FlyweightFactory:
    pool = {}

    def get_flyweight(key):
        if key not in pool:
            pool[key] = new ConcreteFlyweight(key)
        return pool[key]
```

这一机制确保相同内部状态的享元对象在整个应用生命周期中只被实例化**一次**。

### 对象结构与内存计算

享元模式涉及以下角色：

| 角色 | 职责 |
|------|------|
| Flyweight（抽象享元） | 定义接受外部状态参数的接口 `operation(extrinsicState)` |
| ConcreteFlyweight（具体享元） | 存储内部状态，实现 `operation` 方法 |
| FlyweightFactory（享元工厂） | 管理享元对象池，提供获取接口 |
| Client（客户端） | 持有外部状态，调用享元对象时传入 |

内存节省公式为：

$$\text{节省率} = 1 - \frac{|\text{唯一内部状态种数}| \times S_{内部} + N \times S_{外部}}{N \times (S_{内部} + S_{外部})}$$

其中 $N$ 为逻辑对象总数，当 $N$ 远大于唯一内部状态种数时，节省率趋近于 1。

## 实际应用

### Java 字符串常量池

Java 的 `String.intern()` 机制和字符串字面量池本质上就是享元模式的实现。JVM 在字符串常量池（String Pool）中存储字符串的唯一副本，当两个变量赋值相同字符串字面量时，它们指向同一个堆内存地址。例如：

```java
String a = "hello";
String b = "hello";
System.out.println(a == b); // 输出 true，指向同一对象
```

### Java Integer 缓存

Java 的 `Integer.valueOf()` 方法对 **-128 到 127** 之间的整数使用享元模式缓存，这 256 个 Integer 对象在 JVM 启动时预先创建，后续请求直接返回缓存实例，而不是每次 `new Integer()`。

### 棋类游戏棋子渲染

中国象棋程序中，棋盘上最多 32 枚棋子，但"车"、"马"、"炮"等棋子在视觉上相同。享元对象存储棋子的图片纹理（内部状态），外部状态（棋子在棋盘上的列/行坐标、属于哪方）在调用渲染方法时传入，避免为每一枚棋子重复存储相同图片数据。

## 常见误区

### 误区一：享元模式等同于单例模式

两者都涉及"只创建一个实例"，但目的和结构完全不同。单例模式保证**某个类**全局只有一个实例；享元模式是针对**同一内部状态**的对象只共享一个实例，不同内部状态会对应不同的享元对象，享元工厂可以管理**多个**享元实例。例如字体享元池中，"Arial-12px"和"Arial-14px"是两个不同的享元对象。

### 误区二：所有属性都可以放入内部状态

初学者容易贪图简便，将本应是外部状态的属性（如对象位置、运行时动态计算的值）强行存入享元对象内部。这会导致**线程安全问题**——多个客户端并发修改同一个享元对象的"外部状态字段"时会产生竞态条件。正确做法是享元对象设计为**不可变对象（immutable）**，外部状态只能通过方法参数传递。

### 误区三：享元模式一定能提升性能

享元工厂的哈希表查找和外部状态的传递带来额外的运行时开销。当对象数量较少（如只有几十个）或对象的内部状态种类极多（接近对象总数）时，使用享元模式不仅不省内存，还因为工厂管理和参数传递增加了 CPU 负担。该模式适合的前提是：**对象总数 $N$ 远大于唯一内部状态种数 $K$**，通常 $N/K > 10$ 时才值得引入。

## 知识关联

**与组合模式的协作**：享元模式常与组合模式结合使用，将组合树中的**叶节点**设计为共享的享元对象，从而让拥有大量叶节点的树形结构（如文档中的每个字符）显著节省内存。

**与工厂方法模式的区别**：享元工厂表面上类似工厂方法，但工厂方法每次调用都创建新对象，而享元工厂的核心职责是**复用已有对象**，两者意图截然相反。

**下一个学习方向——游戏场景中的享元模式**：掌握基础享元模式后，可进一步学习如何在游戏引擎中应用该模式处理粒子系统、地形块（Tile）渲染和大规模 NPC 生成等场景，这些场景中外部状态通常包含 3D 变换矩阵，享元对象共享的是网格（Mesh）和材质（Material）数据，是本模式最典型的工程级应用之一。
