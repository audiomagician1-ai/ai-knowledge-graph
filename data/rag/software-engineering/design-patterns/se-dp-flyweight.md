# 享元模式

## 概述

享元模式（Flyweight Pattern）是一种结构型设计模式，其核心思想是通过共享大量细粒度对象来降低内存消耗。当程序需要创建数量极多的相似对象，且这些对象会消耗大量内存时，享元模式通过提取对象中可共享的状态（内部状态），将其存储在享元对象内部并反复复用，从而将成千上万个对象压缩为少数几个共享实例。

享元模式由 GoF（Gang of Four）在1994年出版的《设计模式：可复用面向对象软件的基础》中正式定义，英文名"Flyweight"本身借用了拳击运动中最轻量级别（次最轻量级别，≤112磅）的命名，强调对象的极度轻量化。该模式的实际动机来源于文字处理器的字符渲染问题：一篇文档中同一个字母"A"可能出现数千次，若每次出现都创建一个独立对象，内存开销将极为惊人。

享元模式在大规模对象场景下能产生数量级的内存节省，例如在一个渲染10,000棵树的森林场景中，每棵树若独立存储纹理和网格数据（各占数MB），总内存可能超过数十GB；而使用享元模式后，纹理和网格被共享，实际内存占用可降至数十MB。这种量级的差距使其在游戏开发、GUI框架和文本渲染等领域中不可或缺。

## 核心原理

### 内部状态与外部状态的分离

享元模式的关键设计决策是将对象状态拆分为两类：**内部状态（Intrinsic State）**和**外部状态（Extrinsic State）**。内部状态是不随上下文变化的固定信息，存储在享元对象内部并被所有使用者共享；外部状态是随上下文改变的可变信息，由客户端在使用时传入，不存储在享元对象中。

以文字处理器为例，字符"A"的字体、字号、字形（如Arial 12pt）属于内部状态，可被文档中所有"A"字符共享；而每个"A"出现的位置坐标（x, y）则属于外部状态，在渲染时由调用方传入。这种分离保证了享元对象的**不可变性（Immutability）**，使其可以安全地被多处引用而不产生副作用。

### 享元工厂与对象池

享元模式通常配合**享元工厂（Flyweight Factory）**使用，工厂维护一个享元对象池（通常使用 `HashMap` 实现）。当客户端请求一个享元对象时，工厂先检查池中是否已存在具有相同内部状态的实例；若存在则直接返回该实例，若不存在则新建一个并加入池中。

以Java伪代码表示核心逻辑：

```java
private Map<String, Flyweight> pool = new HashMap<>();

public Flyweight getFlyweight(String key) {
    if (!pool.containsKey(key)) {
        pool.put(key, new ConcreteFlyweight(key));
    }
    return pool.get(key);
}
```

这个查找-复用机制使得 N 个外观相同的对象只需占用 1 份内存，对象复用率随总请求量增加而趋近于 100%。

### 参与者结构

享元模式包含四个明确角色：
- **Flyweight（抽象享元）**：声明接受外部状态参数的操作接口，如 `operation(extrinsicState)`。
- **ConcreteFlyweight（具体享元）**：实现抽象享元接口，存储内部状态，要求内部状态必须是可共享且不可变的。
- **UnsharedConcreteFlyweight（非共享具体享元）**：并非所有享元子类都需要被共享，某些场景允许不共享的实例存在。
- **FlyweightFactory（享元工厂）**：创建和管理享元对象，确保合理地共享享元。

## 实际应用

**Java 字符串常量池**是享元模式最广为人知的内置实现。JVM 维护一个字符串池，字面量 `"hello"` 无论在代码中出现多少次，都指向常量池中同一个 `String` 对象。`String.intern()` 方法可以手动将字符串加入常量池，正是享元工厂的显式调用形式。

**Java 包装类缓存**同样采用享元模式：`Integer.valueOf(int)` 对 -128 到 127 范围内的整数返回缓存对象，这意味着 `Integer.valueOf(100) == Integer.valueOf(100)` 结果为 `true`，而 `Integer.valueOf(200) == Integer.valueOf(200)` 结果为 `false`，因为200超出了缓存范围。这一特性是Java面试中的经典考点，直接体现了享元模式的边界效应。

**围棋程序**是教学中常用的示例：棋盘上最多有 361 个棋子，但棋子的颜色（黑/白）就是内部状态，只需创建两个享元对象；而每个棋子的位置（行列坐标）则是外部状态，每次落子时作为参数传入 `draw(int row, int col)` 方法。整个棋局过程中，程序只需维护 2 个棋子对象而非最多 361 个。

## 常见误区

**误区一：将所有对象属性都视为内部状态**。初学者常犯的错误是试图把对象的所有字段都放入享元对象，结果导致享元对象反而因携带大量状态而无法共享。正确做法是严格识别哪些属性在整个生命周期内固定不变（适合作内部状态），哪些随位置或上下文变化（必须作外部状态）。如果内部状态与外部状态的区分不清晰，共享就会破坏对象的语义一致性。

**误区二：享元模式适用于所有对象数量多的场景**。享元模式只有在对象数量极大（通常指万级以上）且内部状态种类远少于对象总数时才值得应用。若一批对象的内部状态各不相同（如每个用户对象都有唯一的ID和属性），则享元工厂的查找开销不仅无法节省内存，反而会引入额外的 `HashMap` 查找和管理成本。衡量指标是**共享率 = 共享实例数 / 请求总数**，该值越接近1，享元模式收益越大。

**误区三：享元对象可以被修改**。由于享元对象被多处引用，任何对内部状态的修改都会影响所有使用该享元的上下文，引发难以排查的 bug。享元对象必须设计为不可变的（使用 `final` 字段），外部状态的变化应通过方法参数传递，而非修改享元对象本身。

## 知识关联

享元模式以对象组合和接口抽象为基础，理解类与实例的区别是使用该模式的前提。享元工厂本质上是一种特殊的**工厂模式**，额外增加了对象池缓存逻辑；同时享元对象本身对外部状态保持无状态，与**单例模式**的全局唯一性有相似之处，但享元允许同一类型存在多个实例（每种内部状态对应一个）。

在后续应用中，**游戏场景中的享元模式**是本概念的直接延伸，将在粒子系统、地形渲染、NPC批量创建等更复杂的实战场景中深化对内外部状态分离的理解，并结合对象池（Object Pool）模式探讨性能优化的组合策略。
