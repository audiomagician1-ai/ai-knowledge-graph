# 外观模式

## 概述

外观模式（Facade Pattern）是一种结构型设计模式，其核心思想是为一组复杂的子系统提供一个统一的、简化的高层接口，使客户端无需了解子系统内部的复杂结构即可完成操作。Facade 一词来自法语，原意为"建筑物的正面外墙"——就像大楼的外墙对外展示简洁门面、隐藏内部复杂管道布线一样，外观模式向调用方呈现一个干净的接口层。

该模式由 GoF（Gang of Four）在 1994 年出版的《设计模式：可复用面向对象软件的基础》一书中正式提出，属于"结构型模式"分类中实现难度最低的模式之一（本文将其定级为 2/9）。它解决的典型问题场景是：一个复杂子系统由十几个甚至数十个类协同完成功能，客户端若直接调用这些类则需要了解它们的初始化顺序、依赖关系和调用协议，极易出错且维护成本极高。

外观模式的价值在于"解耦客户端与子系统"。通过引入外观类（Facade Class），子系统内部可以自由重构、替换、拆分，而客户端代码无需任何修改。这种单向依赖关系（客户端 → 外观类 → 子系统）是软件分层架构的基本构建单元。

## 核心原理

### 结构组成与类图关系

外观模式涉及三类角色：
- **外观类（Facade）**：持有子系统各类的引用，将客户端请求委托给适当的子系统对象处理。
- **子系统类（Subsystem Classes）**：实现子系统实际功能，对外观类的存在一无所知，相互之间可以有依赖。
- **客户端（Client）**：仅与外观类交互，不直接引用任何子系统类。

典型的依赖方向：`Client → Facade → {SubsystemA, SubsystemB, SubsystemC}`。子系统类不持有 Facade 的引用，因此属于单向依赖，不存在循环引用问题。

### 最小示例：家庭影院场景

以家庭影院为例，子系统包含 `Amplifier`（功放）、`DVDPlayer`（播放器）、`Projector`（投影仪）、`Lights`（灯光）四个独立类，每个类各有多个初始化和控制方法。没有外观模式时，客户端需要：

```
lights.dim(10)
projector.on()
projector.setInput(dvd)
amplifier.on()
amplifier.setVolume(5)
dvd.on()
dvd.play(movie)
```

引入 `HomeTheaterFacade` 后，客户端只需调用：

```
homeTheater.watchMovie("星际穿越")
```

外观类内部封装了上述 7 个步骤的正确顺序，客户端从 7 行代码减少到 1 行，且子系统各类的初始化细节完全透明。

### 与适配器模式的结构差异

前置知识适配器模式（Adapter Pattern）解决的是**接口不兼容**问题，通常是将一个类的接口转换为另一个类期望的接口，涉及 1 对 1 的接口转换；而外观模式解决的是**复杂度暴露**问题，是将多个类的功能组合封装在一个新接口后面，属于 N 对 1 的聚合关系。适配器不改变功能语义，外观类则往往会定义新的业务语义方法名（如 `watchMovie` 而非 `play`）。两者虽然都属于结构型模式，但适配器的 `target` 接口是已存在的，外观类的接口是新设计的。

### 外观模式不封闭子系统访问权

这是外观模式的一个重要设计决策：外观类的存在**不禁止**客户端直接访问子系统类。若某些高级用户需要精细控制，他们仍可绕过外观类直接使用子系统。这与"不透明外观"（Opaque Facade）的变体相对，后者会将子系统类设为 `package-private`（Java 中）或置于内部命名空间，强制所有访问必须经过外观类。

## 实际应用

**1. Java 标准库中的 `javax.faces.context.FacesContext`**
JavaServer Faces 框架将 HTTP 请求处理、组件树管理、EL 表达式解析等十余个子系统统一暴露在 `FacesContext` 这个外观类下，开发者通过 `FacesContext.getCurrentInstance()` 即可访问全部功能。

**2. SLF4J 日志门面**
SLF4J（Simple Logging Facade for Java）是外观模式在日志领域的经典实现。`Logger` 接口是外观，背后可以桥接 Log4j、Logback、java.util.logging 等不同子系统。应用代码只依赖 `org.slf4j.Logger`，切换底层日志框架无需修改任何业务代码。

**3. 操作系统 API 层**
Windows API 中的 `CreateProcess()` 函数是外观模式的体现：它在内部协调内存管理器、进程调度器、文件系统驱动等多个内核子系统，向应用层提供单一的进程创建接口。

**4. 电商后台订单提交接口**
一个典型的电商系统提交订单时需要依次调用：库存校验服务、价格计算服务、优惠券核销服务、支付服务、物流下单服务。将这五个服务的编排逻辑封装进 `OrderFacade.submitOrder(cart)` 方法，既保证了业务流程的正确顺序，又让 Controller 层代码极度简洁。

## 常见误区

**误区一：认为外观类必须包含所有子系统功能**
外观类只需暴露客户端**最常用**的操作组合，而非子系统所有能力的完整映射。若将子系统每个方法都转发一遍，外观类变成纯粹的传递层，既不简化也不聚合，反而增加了维护负担。正确做法是识别出 80% 场景下的核心用例，只为这些用例提供便捷方法。

**误区二：将外观模式与分层架构中的 Service 层等同**
外观模式强调的是对**已有复杂子系统**的包装，其子系统通常是既成事实、不受我们控制的。而 Service 层是在系统设计阶段主动规划的职责层。两者虽然形态相似（都是统一入口），但外观模式的动机是"降低现有系统的使用门槛"，Service 层的动机是"按职责分离关注点"。混淆这两个概念会导致在 Service 层设计时错误地以"适配外观"思路进行，丢失领域建模的本质。

**误区三：认为外观模式会引入性能损耗**
外观类本身只是方法调用的转发与编排，不涉及数据拷贝、序列化或网络通信等高代价操作。在绝大多数场景下，引入外观类产生的额外方法调用开销可忽略不计（现代 JIT 编译器甚至会对此进行内联优化）。性能问题往往来自子系统本身，而非外观层。

## 知识关联

**前置概念——适配器模式**：学习外观模式之前理解适配器模式有助于区分"接口转换"与"接口简化聚合"这两种不同的结构性问题。适配器处理的是异构接口的兼容，外观处理的是同构但过于分散的接口的聚合；两者常在同一系统中共存，例如外观类内部可能使用适配器来接入遗留子系统。

**横向关联——中介者模式（Mediator）**：外观模式与中介者模式在拓扑形态上相似，都集中了多方的通信关系。关键区别在于：外观模式的子系统类彼此之间通常**不知道**外观类的存在，通信是单向的（客户端→外观→子系统）；中介者模式的同事类（Colleague）则**持有**中介者引用，双方相互通信，用于解决多对象之间的网状依赖，而非简化外部访问。

**延伸实践——防腐层（Anti-Corruption Layer）**：在领域驱动设计（DDD）中，防腐层是外观模式的架构级应用，用于隔离本界限上下文与外部遗留系统或第三方 API，防止外部系统的模型污染本地领域模型。两者的本质相同：用一层薄薄的封装屏蔽后方的复杂性，差别在于 DDD 防腐层还承担模型翻译职责。
