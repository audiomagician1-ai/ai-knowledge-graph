# 任务系统

## 概述

任务系统（Task System）是多线程编程中将工作单元抽象为离散"任务"（Task）并通过依赖图和优先级调度器进行调度执行的编程模型。与直接操作线程不同，任务系统将"做什么"（任务定义）与"谁来做、何时做"（调度策略）彻底分离，程序员只需描述任务之间的前置依赖关系，调度器负责找到满足依赖约束的执行顺序。

任务系统的概念在2000年代中期随着多核处理器普及而被游戏和高性能计算领域广泛采用。Intel TBB（Threading Building Blocks，2006年开源）和微软PPL（Parallel Patterns Library）是最早的成熟实现之一。游戏引擎领域，Naughty Dog在GDC 2015上公开介绍了其Fiber-based Job System，顽皮狗的方案以协程（Fiber）为执行单元、任务计数器（Counter）为同步原语，启发了此后大量引擎设计。

任务系统的核心价值在于：在线程池的基础上，通过DAG（有向无环图）调度消除显式的互斥锁等待，使CPU在等待依赖完成时可以立即切换执行其他就绪任务，而不是让线程阻塞休眠。这对于帧预算仅有16.67ms（60fps）的游戏引擎尤为关键。

## 核心原理

### 任务与依赖图（Task DAG）

每个任务是一个函数指针加上其所需数据的封装，通常定义为：

```
Task { function_ptr, data_ptr, dependency_count, dependents[] }
```

任务之间的依赖通过有向无环图（DAG）表达：若任务B依赖任务A，则A必须完成后B才能进入就绪队列。调度器维护每个任务的**未满足依赖计数**（Pending Dependency Count），初始值等于其前置任务数量。当某任务完成时，调度器将其所有后继任务的计数原子地减一（`atomic_fetch_sub`），减为0的任务立即被推入就绪队列。由于读写计数使用原子操作，不需要全局锁，这是任务系统能高效扩展到32+核心的原因。

### 优先级调度与工作窃取（Work Stealing）

就绪队列通常按优先级分为若干级别，例如：`CRITICAL=0`、`HIGH=1`、`NORMAL=2`、`LOW=3`。调度器线程按优先级从高到低轮询各级队列。为避免高优先级任务无限饿死低优先级任务，部分实现引入老化机制（Aging）：任务在队列中每等待N毫秒后优先级提升一级。

工作窃取（Work Stealing）解决负载不均衡问题：每个工作线程拥有自己的双端队列（Deque），自己从队尾取任务执行，空闲线程从其他线程的队头"窃取"任务。这种设计使局部性（同一线程产生的子任务在同一线程执行）与全局均衡两者兼得。Chase-Lev无锁双端队列（2005年论文）是工作窃取调度器中最常用的数据结构。

### 任务计数器与同步屏障

顽皮狗方案的关键创新是**原子计数器（Atomic Counter）**作为同步原语：提交一批任务时返回一个计数器，初始值为任务数量，每个任务完成时计数器减一。主逻辑可以通过`WaitForCounter(counter, 0)`挂起当前Fiber，让调度器去执行其他任务，计数器归零后自动恢复。这等价于一个非阻塞的屏障（Barrier），与传统的`pthread_barrier_wait`不同，它不会让线程休眠，只是切换执行上下文。

## 实际应用

**游戏引擎帧循环分解**：以UE5的任务图（TaskGraph）为例，一帧的工作被分解为：物理模拟任务组 → 动画更新任务组 → 渲染命令生成任务组，前一组的所有任务完成后才触发后一组。每组内部的各个对象更新任务互相独立、并行执行，调度器自动将它们分发到线程池的8~16个工作线程上。

**异步资源加载**：资源加载可建模为依赖链：`IO读取任务` → `解压缩任务` → `GPU上传任务` → `注册到资源管理器任务`。提交这条依赖链后，主线程不需要轮询，只需等待最终任务的计数器归零。在等待期间，调度器会自动调度其他就绪任务填满CPU。

**数据并行粒度控制**：处理10000个粒子时，不必为每个粒子创建一个任务（创建开销会超过执行开销），而是将粒子分批，每批256个创建一个任务，共创建约40个任务。批次大小（Batch Size）是任务系统调优中的核心参数，通常根据`任务函数执行时间 > 任务创建+调度开销（约1~5微秒）`来确定下界。

## 常见误区

**误区一：依赖越细粒度越好**。许多初学者将每一个小操作都建模为独立任务，认为粒度越细并行度越高。实际上，任务的创建、入队、调度、出队本身存在固定开销（约1~5微秒/任务），若任务本体执行时间低于此值，任务系统的开销反而使总时间增加。正确做法是合并过小的任务或使用并行For的批量提交接口。

**误区二：任务系统可以替代所有同步原语**。任务系统通过依赖图消除了大量等待，但它解决的是"执行顺序"问题，而非"数据竞争"问题。若两个并行任务访问同一块可写内存，任务系统本身不提供保护，仍然需要无锁数据结构或分区（Partitioning）来避免竞争。把依赖图视为内存访问的保护机制是一个危险的误解。

**误区三：DAG中存在环可以通过优先级绕过**。若任务A等待任务B完成，任务B又等待任务A完成，即产生循环依赖，系统会死锁。提高任意一方的优先级并不能解决这个问题，因为两个任务的依赖计数永远不会归零。循环依赖必须在设计阶段通过拓扑排序检查消除，部分调度器（如Intel TBB的`flow_graph`）在提交时执行DAG合法性检验。

## 知识关联

**与线程池的关系**：任务系统建立在线程池之上。线程池提供固定数量的工作线程和一个基本的全局任务队列，任务系统在此之上增加了依赖图追踪、优先级多队列和工作窃取机制。去掉依赖管理的任务系统退化为带优先级的线程池。

**与无锁编程的关系**：任务系统中的核心数据结构——依赖计数的原子递减、Chase-Lev双端队列的工作窃取——都依赖无锁编程技术。若用互斥锁实现就绪队列的入队/出队，在高核心数（32+线程）下锁竞争会成为显著瓶颈，无锁队列是任务系统可扩展性的基础。

**与游戏多线程架构的关系**：游戏多线程架构定义了哪些系统（物理、动画、渲染）需要并行化以及它们之间的宏观依赖关系，任务系统则是实现这些并行化的具体机制，将架构层面的依赖转化为运行时的DAG节点和边。

**向协程的延伸**：传统任务系统中，若任务执行中途需要等待某个计数器，必须将后续逻辑拆分为一个新任务（回调式延续）。协程（Coroutine）允许在单个函数中直接使用`co_await counter`挂起并恢复，将任务分割的复杂性隐藏在语言层面，是任务系统在编程模型层面的自然演进。
