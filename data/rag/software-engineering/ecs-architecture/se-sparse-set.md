# 稀疏集合（Sparse Set）

## 概述

稀疏集合（Sparse Set）是一种专为整数键设计的数据结构，能够在O(1)时间复杂度内完成查找、添加和删除操作，同时支持O(n)的线性遍历，其中n为当前实际存储的元素数量而非最大容量。在ECS架构中，它被用作组件存储的核心容器，使得通过实体ID快速定位对应组件成为可能。

该数据结构由Briggs和Torczon于1993年在论文《An Efficient Representation for Sparse Sets》中正式提出，最初用于编译器优化中的数据流分析。ECS社区（尤其是EnTT库的作者skypjack）在2010年代将其引入游戏开发领域，发现它在处理"实体数量庞大但持有某一组件的实体比例较小"的典型ECS场景下具有显著优势。

传统哈希表虽然也能做到O(1)的查找，但其遍历所有已存在元素的开销是O(容量)而非O(元素个数)，在缓存命中率上也较差。稀疏集合则将"紧凑的连续内存遍历"与"O(1)随机访问"结合在一起，这正是ECS系统中每帧需要批量处理大量同类组件的场景所需要的。

## 核心原理

### 双数组结构

稀疏集合由两个数组组成：

- **稀疏数组（sparse array）**：下标为实体ID，存储该实体在密集数组中的位置索引。其长度等于实体ID的最大值（例如65536），大部分槽位为无效值（通常用 `UINT32_MAX` 或 `-1` 表示）。
- **密集数组（dense array）**：紧凑存储所有已添加的实体ID，长度等于当前组件数量n。

对于组件存储，通常还有第三个数组**组件数组（component array）**，与密集数组一一对应，下标 `i` 处存储 `dense[i]` 对应实体的组件数据。

用公式表达查找关系：若实体ID为 `e`，其组件位于 `component_array[ sparse[e] ]`，前提是 `sparse[e] < n` 且 `dense[ sparse[e] ] == e`（第二个条件用于防止稀疏数组中的脏数据误判）。

### O(1) 删除的核心技巧——末尾交换

删除是稀疏集合最精妙的操作。假设要删除实体 `e`，其在密集数组中的位置为 `pos = sparse[e]`：

1. 取密集数组末尾元素 `last = dense[n-1]`。
2. 将 `dense[pos] = last`，同步更新 `component_array[pos] = component_array[n-1]`。
3. 更新 `sparse[last] = pos`，使末尾元素的索引指向新位置。
4. 将 `sparse[e]` 标记为无效，`n -= 1`。

整个过程仅涉及常数次数组写入，无需移动其余元素，因此是严格O(1)。但这也意味着**密集数组中的顺序会被打乱**，不能依赖其排列顺序。

### 添加与查找

**添加**实体 `e` 的组件时：将 `e` 追加到 `dense[n]`，将组件数据写入 `component_array[n]`，设 `sparse[e] = n`，然后 `n += 1`，同样是O(1)。

**查找**只需一次数组访问：检查 `sparse[e]` 是否为有效值，并验证 `dense[sparse[e]] == e`，即可确认实体 `e` 是否持有该组件，时间复杂度O(1)。

### 内存布局与缓存友好性

密集数组和组件数组均为连续内存，当系统迭代所有拥有某一组件的实体时，访问模式是顺序的，CPU预取机制能充分发挥作用。相比之下，若用 `std::unordered_map<EntityID, Component>` 存储，桶结构导致的内存不连续会造成大量缓存缺失（cache miss）。在一个拥有10万实体的场景中，这种差异会直接影响每帧的系统执行时间。

## 实际应用

**EnTT库的实现**是最广为人知的稀疏集合ECS实践。EnTT使用分页式稀疏数组（paged sparse array），将稀疏数组切分为固定大小（默认每页4096个槽）的页，按需分配，避免为整个ID空间预分配内存，从而在实体数量稀少时节省内存。

**多组件查询（View/Group）** 场景中，稀疏集合的价值尤为突出。当系统需要同时处理拥有 `Position` 和 `Velocity` 两个组件的实体时，可以遍历数量较少的那个组件的密集数组，然后用O(1)操作在另一个稀疏集合中查询对应实体是否也存在，从而高效筛选出交集，而无需构建额外的索引结构。

**实体销毁**时，稀疏集合的末尾交换删除天然配合ECS的"不保证顺序"语义——大多数游戏逻辑系统并不关心组件的处理顺序，只关心性能，这与稀疏集合的设计取舍完全吻合。

## 常见误区

**误区一：认为稀疏数组的内存占用总是不可接受的。**
稀疏数组的大小取决于实体ID的范围而非实体数量。若实体ID上限为65536，稀疏数组仅占 `65536 × 4字节 = 256KB`，对现代游戏来说完全在L2或L3缓存范围内。只有当实体ID范围达到数百万级别时，才需要考虑分页等优化手段。

**误区二：将稀疏集合与稀疏矩阵混淆。**
稀疏矩阵（Sparse Matrix）是线性代数中用于存储大量零元素矩阵的结构（如CSR格式），与稀疏集合完全不同。两者同用"稀疏"一词，但稀疏集合特指Briggs-Torczon结构，其"稀疏"描述的是实体ID空间中大部分槽位未被使用的特性，而非数值上的零。

**误区三：删除后立即依赖密集数组的索引。**
由于删除操作会将末尾元素移动到被删除元素的位置，任何在删除操作前缓存的密集数组索引（如迭代器或下标变量）在删除后都会失效或指向错误数据。在遍历密集数组的同时执行删除时，必须在删除后不递增迭代下标，否则会跳过被移入的末尾元素。

## 知识关联

**前置概念：组件（Component）** 定义了需要被存储的数据类型。稀疏集合是以组件类型为粒度组织的——每种组件类型（如 `PositionComponent`、`VelocityComponent`）对应独立的一套稀疏数组+密集数组+组件数组，实体ID作为跨集合的统一键。理解组件的类型隔离语义，有助于理解为何每种组件需要独立的稀疏集合实例而非共享一个。

**与其他存储方案的对比**：另一种常见ECS存储方案是原型（Archetype）表，它将拥有相同组件集合的实体打包存储，遍历性能极高但添加/删除组件时需要移动实体数据。稀疏集合在组件频繁增删（如状态标签组件的动态附加）的场景下更有优势；原型表在组件构成稳定、批量处理为主的场景下更优。两种方案代表了ECS存储设计的两个主流方向，Bevy引擎同时支持两者，允许开发者按需选择。
