# 代码审查概述

## 概述

代码审查（Code Review）是指由一名或多名开发者系统地检查他人编写的源代码，目的是在代码合并到主分支之前发现缺陷、提升代码质量并传播技术知识。与测试不同，代码审查是一种**人工静态分析**过程——审查者不运行代码，而是通过阅读逻辑结构来发现问题，因此能捕捉到自动化测试难以覆盖的设计缺陷、命名歧义和安全漏洞。

代码审查作为正式工程实践最早可追溯至1976年Michael Fagan在IBM提出的**Fagan Inspection**方法，该方法规定了严格的六个阶段：计划、概述、准备、检查会议、返工、跟踪，并统计发现，Fagan的研究表明正式审查可以减少代码缺陷达82%。进入21世纪，随着GitHub在2008年推出Pull Request功能，代码审查从重量级的会议形式演变为轻量的异步在线流程，成为现代软件开发的标准工序。

代码审查之所以不可或缺，是因为它在多个维度同时创造价值：SmartBear公司2009年的研究《Best Kept Secrets of Peer Code Review》调查了约40000次代码审查，发现每小时的审查可以发现约8个缺陷，而这些缺陷若留到生产环境才被发现，修复成本将高出6到10倍。除了发现缺陷，代码审查还是团队内部传递架构决策、编码规范和领域知识的主要渠道。

## 核心原理

### 代码审查的四个主要目标

代码审查同时追求四类目标，且这四类目标的优先级通常按此顺序排列：

1. **功能正确性**：逻辑是否按需求规格运行，边界条件和错误处理是否完备。
2. **可维护性**：命名是否清晰、函数是否单一职责、复杂度是否可控（通常以圈复杂度不超过10作为经验阈值）。
3. **安全性**：是否存在SQL注入、XSS、不当的权限控制等常见漏洞。
4. **知识共享**：审查者通过阅读代码了解系统变化，作者通过反馈提升编写能力。

将安全性列为独立目标而非可维护性的子项，是因为安全缺陷往往在语法上完全合法，只有专门关注才能发现。

### 审查规模与时间的黄金比例

SmartBear的研究给出了量化的最佳实践：审查者**每小时不应阅读超过200至400行代码**，单次审查规模建议控制在**200至400行**以内。超过这一规模后，审查者的缺陷发现率会显著下降，原因是认知负荷超过了人类短期工作记忆的7±2单元上限。Google内部数据同样显示，超过400行的Pull Request平均评论周期比小型PR长3倍以上，因此许多团队将PR拆分作为提交代码审查之前的强制要求。

### 作者与审查者的职责分工

在代码审查流程中，**作者**在提交审查时有义务提供清晰的变更描述（包括"为什么改"而非仅"改了什么"）、关联相关Issue或需求单、以及标注需要重点关注的模块。**审查者**则承担提出具体可行的建议而非模糊批评的责任，并区分哪些评论是阻塞合并的必须修改（Blocker），哪些是可选建议（Nit）。这种职责边界防止了审查沦为主观风格争论，谷歌在其工程实践文档中将此称为"以数据和原则而非个人偏好驱动的审查"。

### 同步与异步审查模式的选择

**异步审查**（如GitHub Pull Request评论）适合分布式团队和非紧急变更，允许审查者在自己专注状态下批阅；**同步审查**（如结对编程或屏幕共享会议）适合复杂架构变更或新成员入职场景。Microsoft Research在2013年的研究《Expectations, Outcomes, and Challenges of Modern Code Review》发现，开发者认为代码审查最主要的收益中，"寻找缺陷"仅排第三，排在前两位的是"代码改进建议"和"知识转移"，这支持了异步模式下留下书面记录的额外价值。

## 实际应用

在一个典型的电商后端团队中，开发者提交了一个新增用户优惠券功能的Pull Request，包含320行代码。审查者检查时发现：优惠券金额未对负数做校验（功能正确性问题）、核心方法`applyCoupon`长达80行且承担了验证、计算和数据库写入三项职责（可维护性问题）、用户ID直接拼接到SQL查询字符串中（SQL注入安全问题）。这三类问题分别来自代码审查的三个不同目标维度，没有一项能被单元测试或CI的静态扫描工具完整覆盖——说明了为什么人工审查与自动化工具必须共存。

在开源项目领域，Linux内核的代码审查完全通过邮件列表进行，Linus Torvalds及各子系统维护者对每个补丁（Patch）进行逐行评审，最终通过`Reviewed-by`、`Acked-by`和`Tested-by`标签记录审查状态。这一无工具依赖的流程已运行超过30年，证明代码审查的价值在于流程和文化，而非特定工具。

## 常见误区

**误区一：代码审查等同于风格检查。** 许多初级团队将审查精力集中在缩进、命名大小写等风格问题上，而忽略逻辑错误和安全漏洞。解决方案是引入ESLint、Checkstyle等自动化格式化工具处理风格问题，将人工审查精力保留给机器无法判断的逻辑和设计层面。代码审查不是替代代码格式化工具的手段。

**误区二：审查越严格、找到问题越多，审查质量越高。** 这一误区会导致"橡皮图章"文化的反面——过度批评文化，使作者将审查视为对抗而非协作。Google的工程师准则明确指出：审查者应在代码"足够好"（sufficiently good）时批准合并，而非等到"完美"。坚持每个微小偏好都必须修改会使PR周期从小时级拉长到数天，从而抵消审查带来的质量收益。

**误区三：代码审查可以替代测试。** 审查是静态过程，无法发现依赖运行时状态的并发竞态条件、特定输入组合触发的性能退化等问题。两者是互补关系：测试验证行为，审查验证意图与结构。

## 知识关联

学习代码审查概述需要已熟悉**Pull Request工作流**，因为现代代码审查几乎都以PR为载体触发，理解分支创建、提交、请求合并的完整路径是参与审查的前提。

掌握代码审查概述后，可以向以下方向深入：**审查清单**将本文提到的四类审查目标具体化为可操作的逐项检查表；**审查工具**（如Gerrit、GitHub Code Review、Phabricator）提供了对审查流程的工程化支撑；**建设性反馈**专注于如何将发现的问题以不引发防御情绪的方式表达；**审查文化建设**探讨如何在组织层面建立持续的审查习惯；**编码规范**则为审查者提供判断可维护性问题的客观标准依据，避免陷入纯主观风格争论。
