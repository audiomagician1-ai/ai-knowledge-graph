# 群体编程

## 概述

群体编程（Mob Programming）是一种由 Woody Zuill 在 2012 年于 Hunter Industries 公司实践并命名的软件开发协作方法。其核心规则极为简洁：**整个团队同时在同一台电脑上处理同一个任务**，通常由一名"驾驶员"（Driver）负责操作键盘输入代码，其余成员作为"导航员"（Navigator）进行实时思考、讨论和指导。

与代码审查的传统形式（如 Pull Request 审查或正式审查会议）相比，群体编程把审查行为嵌入到编码过程本身中。代码从输入的第一个字符起，就经历了多人实时审阅，错误在提交前就已被发现和纠正，而不是在事后被批注出来。

这种方法之所以受到敏捷团队关注，是因为它将知识孤岛问题（Knowledge Silo）消除于无形——每一个架构决策、每一段业务逻辑，都在多人参与的对话中产生，不存在"只有张三才懂这段代码"的情况。

## 核心原理

### 驾驶员与导航员轮换机制

群体编程中，驾驶员和导航员的角色每隔固定时间轮换，常见间隔为 **10 到 15 分钟**。驾驶员严格遵循"不自行决策"原则——只有在导航员给出明确指令后才将意图转化为代码。这条规则防止了驾驶员独占思路方向，强迫所有想法必须通过语言表达出来，从而让全组成员同步理解每一步操作的原因。

Zuill 的原始描述中将这一原则表述为：*"For an idea to go from your head into the computer, it must go through someone else's hands."*（要让你的想法进入计算机，必须经过他人的双手。）这一机制天然产生了持续的口头代码审查效果。

### 实时审查与即时反馈闭环

传统 Pull Request 审查存在时间延迟，审查意见可能在代码提交数小时乃至数天后才能到达作者。群体编程将这一反馈回路压缩至**秒级**。当驾驶员准备写一个复杂 SQL 查询时，导航员中的数据库专家可以立刻指出索引缺失的风险，而不必等待合并后的 CI 性能报告。

这种即时性对代码审查文化的影响是：审查不再是一种批判性的"找错"活动，而演变为合作性的"共建"活动。心理上的防御反应（对自己代码被指出问题时的抵触情绪）也因为"代码是大家一起写的"而被显著降低。

### 知识传播与技能同步

群体编程具有显著的技能均衡效果。在一个由 5 人组成的群体编程会议中，当资深工程师作为导航员解释某个设计模式的选择时，初级工程员坐在驾驶员位置亲手实现，这种**做中学（Learning by Doing）**的效率远高于代码审查后阅读评论的被动学习方式。

具体数据上，Hunter Industries 团队在引入群体编程后报告，代码缺陷率下降，而团队成员在跨功能模块的代码理解度上有显著提升——这正是代码审查文化建设阶段所追求的目标之一：让审查不仅找出问题，更实现知识流动。

## 实际应用

**场景一：处理遗留代码重构**
当团队需要重构一段无人完全理解的历史代码时，群体编程特别有效。导航员们可以各自贡献对不同模块的片段记忆，边讨论边逐步拼出全貌，驾驶员同步完成重构和测试的编写。这种场景下，单人代码审查几乎无法提供等效的质量保障。

**场景二：跨团队知识迁移**
当后端工程师需要修改前端逻辑时，可以邀请前端工程师作为导航员参与群体编程。相比事后让前端工程师审查 PR，这种方式让知识在编码过程中双向流动，前端开发者同时也理解了为何某 API 接口以特定方式设计。

**场景三：安全敏感代码**
对于涉及认证逻辑、加密实现等安全敏感代码，部分团队要求此类代码必须通过群体编程方式编写，而非仅依赖事后 PR 审查。这与代码审查规范中"高风险变更需多人确认"的原则一致，但在执行力度上更为彻底。

## 常见误区

**误区一：群体编程等同于结对编程加人数**
结对编程（Pair Programming）是两人协作，而群体编程并非简单的"多人结对"。群体编程中存在明确的角色分工和轮换规则，强调的是整个团队（通常 4 至 6 人）的整体同步，而非两人深度协作。二者在适用场景、沟通成本和知识分发效果上有本质区别。

**误区二：群体编程效率低下，浪费人力**
表面上看，5 个人做"1个人的工作"似乎损失了 80% 的产能。但 Zuill 团队的数据显示，群体编程大幅减少了返工（Rework）和沟通同步的时间成本。在代码审查文化不成熟的团队中，PR 反复修改、审查意见被误解、合并冲突等问题消耗的时间往往超过群体编程所"节省"的并行工作时间。

**误区三：群体编程可以完全替代代码审查**
群体编程内嵌了实时审查，但并不意味着不需要任何形式的审查记录或规范检查。自动化静态分析、合规性检查（如 GDPR 相关代码审查）仍然需要工具辅助完成。群体编程替代的是人工审查中的理解性反馈环节，而非所有审查活动。

## 知识关联

群体编程的实践前提是团队已具备基本的**审查文化建设**基础。如果团队成员之间存在较强的防御型沟通习惯（对代码批评高度敏感），群体编程反而会因实时暴露分歧而产生摩擦。审查文化建设阶段建立的心理安全感和反馈规范，是群体编程能顺畅运作的前提条件。

完成群体编程阶段的学习后，自然过渡到**依赖审查**的议题。群体编程过程中，团队在实时协作时会频繁面对"是否引入某个第三方库"的决策时刻，这正是依赖审查所关注的核心场景：如何在代码编写阶段（而非合并后）评估新依赖项的安全性、许可证合规性和长期维护风险。掌握了群体编程的实时协作节奏，有助于理解依赖审查为何强调在 PR 创建之前就完成依赖评估。
