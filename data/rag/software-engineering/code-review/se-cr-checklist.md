# 审查清单

## 概述

审查清单（Review Checklist）是代码审查过程中使用的结构化检查表，将审查工作拆分为功能正确性、安全性、性能、可读性四个核心维度，以逐项确认的方式确保审查覆盖度。它的本质是将有经验开发者的隐性知识显式化——把"老手凭直觉能发现"的问题转化为团队共享的检查条目。

审查清单的实践起源可追溯至1976年Michael Fagan在IBM提出的**正式代码审查（Fagan Inspection）**方法，该方法首次将检查步骤系统化并量化缺陷类型。现代敏捷团队的审查清单更轻量，但核心逻辑一脉相承：通过预定义条目消除审查盲区，使缺陷发现率从非结构化审查的约30%提升至60%以上（卡内基梅隆SEI数据）。

在没有清单时，审查者往往只关注自己熟悉的领域，例如后端开发者可能遗漏SQL注入风险，前端开发者可能忽视时间复杂度问题。审查清单通过强制覆盖四个维度，将这种"个人偏好驱动的审查"转变为"系统驱动的审查"。

## 核心原理

### 功能正确性检查维度

功能维度关注代码是否按规格说明实现了预期行为。典型条目包括：边界值处理（如循环索引是否存在差一错误 off-by-one error）、空值/null检查、异常路径是否有返回值、分支覆盖是否遗漏else子句。具体检查项示例：

- 函数对空列表输入是否返回合理默认值而非崩溃？
- 数值计算是否处理了整数溢出（如Java中`int`超过2,147,483,647的场景）？
- 状态机的所有状态转移是否都有对应处理？

功能维度的条目数量通常占整个清单的35%–40%，是四个维度中比重最大的。

### 安全性检查维度

安全维度直接对照**OWASP Top 10**漏洞类型建立条目，常见必查项包括：

- **输入验证**：外部输入是否经过白名单校验，防止SQL注入、XSS；
- **认证与授权**：API端点是否验证了调用者权限，而非仅验证登录状态；
- **敏感数据处理**：密码、Token是否出现在日志中（log4j等框架的`toString`可能意外输出字段值）；
- **依赖版本**：新引入的第三方库是否有已知CVE漏洞编号。

安全条目的特点是**一票否决**——任何安全条目未通过，代码不得合并，不同于功能条目可以标注"后续迭代修复"。

### 性能检查维度

性能维度聚焦于可量化的复杂度问题和资源泄漏，而非主观的"感觉慢"。核心条目：

- **循环内数据库查询**：是否存在N+1查询问题，即在循环中对每条记录执行一次SELECT；
- **时间复杂度**：对于数据量>10,000条的场景，O(n²)算法是否被标注并评审；
- **资源关闭**：文件句柄、数据库连接、网络Socket是否在finally块或try-with-resources中关闭；
- **缓存策略**：重复计算的结果（如正则表达式Pattern对象）是否在方法外部缓存而非每次调用重建。

性能条目一般包含**量化阈值**，例如"单次API响应时间不得超过200ms（P99）"，避免争议。

### 可读性检查维度

可读性维度保证代码的长期可维护性，典型条目：

- 函数长度不超过**50行**（超出需拆分或注释说明）；
- 变量命名是否使用领域语言（如`orderTotal`而非`temp2`）；
- 复杂业务逻辑是否有注释说明"为什么"而非"做什么"；
- 魔法数字是否提取为有名称的常量（如`MAX_RETRY_COUNT = 3`而非直接写`3`）。

## 实际应用

**GitHub Pull Request模板集成**：许多团队将审查清单直接嵌入`.github/pull_request_template.md`文件，PR创建时自动显示Markdown复选框列表，审查者逐项勾选后才能点击Approve。这种方式将清单从"建议"变为"流程门禁"。

**差异化清单配置**：支付系统的审查清单与内部工具清单不同——支付系统清单中安全维度条目可达20项（包括PCI DSS合规检查），而内部工具清单可能只有5项安全条目。团队应针对代码类型（核心业务逻辑 vs. 测试脚本）维护不同版本的清单，而非使用单一通用版本。

**自动化与人工清单协同**：静态分析工具（SonarQube、ESLint）可自动覆盖部分清单条目（如空指针检测、代码复杂度圈数超过10），这些条目应从人工审查清单中移除，人工清单专注于工具无法检测的逻辑正确性和业务语义问题。

## 常见误区

**误区一：清单条目越多越好**
一份包含80+条目的清单会导致审查者出现"清单疲劳"，机械勾选而非真正思考。研究表明，条目超过25项后，漏检率反而上升。有效的审查清单通常控制在15–25条核心条目，复杂场景通过子清单分级展开。

**误区二：清单是一成不变的文档**
审查清单应随项目演进定期修订。每次线上事故复盘后，若根因涉及某类代码缺陷，则需将对应检查项补充到清单中。未经历过"某类Bug烧过一次"而反向更新的清单，往往遗漏最贴近团队实际痛点的条目。

**误区三：安全条目与功能条目等同处理**
部分团队将安全条目标注为"Minor"级别允许后续修复，这在实践中非常危险。安全维度的条目（尤其是OWASP Top 10相关项）应单独设置为**阻塞性条目（blocking item）**，并要求安全审查者（而非提交者）确认关闭，而非仅由提交者自评通过。

## 知识关联

审查清单以**代码审查概述**中建立的审查流程（提交→审查→反馈→合并）为执行前提，清单是审查阶段的操作手册，没有明确的审查触发机制，清单就无法被一致性执行。

清单中的安全维度自然延伸至**安全审查**专题——后者在OWASP分类、威胁建模（Threat Modeling）和安全测试方法上有更深入的展开，可以视为安全维度清单条目的详细展开文档。同样，清单中的性能维度对应**性能审查**专题，涵盖剖析工具（Profiler）使用、基准测试（Benchmark）设计和渐进式优化策略等进阶内容。团队在使用清单的性能条目时遇到判断困难，通常需要参考性能审查的具体量化标准来补充决策依据。
