# 审查清单

## 概述

审查清单（Review Checklist）是代码审查过程中使用的结构化检查表，将审查关注点按维度归类为功能正确性、安全性、性能、可读性四大类别，确保审查者在有限时间内系统覆盖所有重要检查项，避免遗漏。其本质是将资深工程师的隐性经验显式化，转化为可被团队复用的标准化知识。

审查清单的概念最早在航空业使用，飞行员的飞行前检查表（Pre-flight Checklist）是其原型。1993 年，NASA 的 Michael Fagan 将类似理念引入软件工程，提出了"正式代码检查"（Formal Code Inspection）方法，审查清单作为其核心工具被广泛采用。研究表明，使用审查清单可将代码审查的缺陷发现率提升约 20%–30%，比无结构的"自由式审查"效果显著。

审查清单的价值在于消除审查者的认知盲点。单个审查者在阅读一段代码时，注意力会自然集中在功能逻辑上，而容易忽略边界条件检查、SQL 注入风险或循环复杂度过高等问题。清单以检查项（Checklist Item）的形式强制引导审查者逐项关注每个维度。

## 核心原理

### 四大检查维度及典型检查项

**功能正确性维度**关注代码是否实现了需求规格中描述的预期行为，典型检查项包括：

- 边界值处理：输入为 0、负数、空字符串时是否有明确处理；
- 分支覆盖：每条 `if/else` 分支是否均有对应的单元测试用例；
- 错误返回值：函数返回 `null` 或错误码时，调用方是否做了判断；
- 业务规则一致性：代码逻辑是否与产品文档或接口文档保持一致。

**安全性维度**关注代码中可被攻击者利用的漏洞，典型检查项包括：

- 输入验证：所有外部输入是否经过白名单或正则过滤，防止 XSS、SQL 注入；
- 敏感信息：密码、API Key 是否以明文形式出现在代码或日志中；
- 权限校验：每个 API 端点是否在业务逻辑执行前完成身份认证与权限鉴别；
- 依赖库版本：是否引入了 CVE 已知漏洞的第三方库版本。

**性能维度**关注代码在高负载下的资源消耗，典型检查项包括：

- N+1 查询：循环内是否存在对数据库的重复单条查询，应改为批量查询；
- 不必要的对象创建：是否在高频调用路径（Hot Path）中反复实例化大对象；
- 缓存使用：对于耗时超过 100ms 的计算或查询，是否考虑了结果缓存；
- 资源释放：数据库连接、文件句柄是否在 `finally` 块或 `try-with-resources` 中关闭。

**可读性维度**关注代码的可维护性，典型检查项包括：

- 命名清晰度：变量和函数名是否准确描述其用途，禁止使用 `data`、`tmp`、`flag` 等模糊命名；
- 函数长度：单个函数是否超过 50 行（Google 风格指南建议值），是否需要拆分；
- 注释必要性：复杂算法或非直觉设计决策是否附有解释性注释；
- 魔法数字：代码中是否存在未命名的字面量（如 `if (status == 3)`），应提取为具名常量。

### 清单的层级结构

一份工程实践中的审查清单通常分为三层：**通用层**（适用于所有代码变更，约 15–20 项）、**语言层**（针对 Java、Python、Go 等特定语言的惯用法，约 10 项）和**领域层**（针对支付、医疗等特定业务安全或合规要求，约 5–10 项）。三层叠加使用，覆盖从通用工程质量到业务合规的完整检查范围。

### 清单项的优先级标注

有效的审查清单会对每个检查项标注严重级别，常见分为 **Blocker（阻断合并）**、**Major（需修复但不阻断）**、**Minor（建议优化）** 三级。Google 内部的代码审查规范明确要求：只要存在一个 Blocker 级检查项未解决，Pull Request 就不允许合并。

## 实际应用

**场景一：电商订单接口的审查清单应用**
审查一个新增"创建订单"API 时，审查者依次执行：功能维度——检查库存不足时是否返回正确错误码（而非静默失败）；安全维度——检查订单金额字段是否允许客户端传入负值（防止负价格攻击）；性能维度——检查是否在事务内部调用了第三方支付网关（可能导致事务长时间持锁）；可读性维度——检查 `createOrder()` 函数是否超过 80 行，是否需要提取 `validateStock()` 子函数。

**场景二：团队协作中的差异化清单**
前端团队的清单会在可读性维度额外包含"CSS 类名是否遵循 BEM 命名规范"，后端团队的清单则在安全维度包含"是否开启了预处理语句（PreparedStatement）防止 SQL 注入"。同一个四维框架，但各团队根据技术栈定制具体检查项。

## 常见误区

**误区一：清单越长越全越好**
将清单条目扩展到 60 项以上会导致审查者产生"清单疲劳"（Checklist Fatigue），开始机械性地勾选而不实质检查。经验表明，单次审查的清单有效条目应控制在 20–30 项以内。超出这个范围的检查项应通过自动化工具（如 SonarQube、Checkmarx）承担，而非依赖人工清单。

**误区二：清单等于审查的全部**
审查清单只能覆盖**已知类型**的问题。架构级缺陷（如模块间不合理的循环依赖）、业务逻辑误解（如对业务规则的错误理解）这类问题无法被任何清单条目捕获，仍需审查者的领域知识和批判性思维。将清单视为银弹会导致团队错误地降低审查的思考深度。

**误区三：清单一经创建永久有效**
技术栈升级、安全威胁演变都会使清单中的部分条目过时。例如，在 Java 8 引入 `Optional` 之后，"是否检查了 null 返回值"这一条目需要更新为"是否正确使用了 `Optional.isPresent()` 而非直接调用 `get()`"。业界最佳实践是每季度对清单进行一次回顾性更新（Retrospective Review）。

## 知识关联

**与代码审查概述的关系**：代码审查概述介绍了审查的目标（提升质量、知识共享）和基本流程（提交 PR → 指派审查者 → 评论 → 合并），审查清单则是在"审查者实施检查"这一步骤中使用的具体工具，将抽象的"审查代码质量"目标转化为可执行的逐项检查动作。

**通向安全审查**：审查清单中的安全维度（约 5–8 项）是安全审查的入门版本，仅覆盖 OWASP Top 10 中最常见的几类漏洞。安全审查进一步引入威胁建模（Threat Modeling）和渗透测试视角，对每条数据流进行专项分析，是安全维度清单项的深化展开。

**通向性能审查**：审查清单中的性能维度主要依赖静态代码判断（如识别 N+1 查询模式），而性能审查进一步结合性能剖析工具（Profiler）和基准测试（Benchmark）数据，对热点路径进行量化分析，两者在方法论上存在本质差异。

**通向审查工具**：清单中的许多检查项可被自动化工具接管——可读性规范由 ESLint/Checkstyle 检查，安全漏洞由 Snyk/Checkmarx 扫描，性能反模式由 PMD/SpotBugs 识别。审查工具学习的核心即是理解哪些清单项可以自动化、自动化工具的误报率如何处理，以及人工清单与自动化工具如何分工协作。
