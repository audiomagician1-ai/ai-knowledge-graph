# 建设性反馈

## 概述

建设性反馈（Constructive Feedback）是代码审查中审查者向作者传达问题、建议与改进方向的具体方式，涵盖评论措辞、优先级标记和建议格式三个维度。与单纯指出"这段代码有问题"不同，建设性反馈要求审查者同时提供问题描述、原因解释和可行的改进方案，使代码作者能够理解"为什么要改"并知道"怎么改"。

建设性反馈的实践规范在2018年前后随着Google、Microsoft等公司公开其内部代码审查指南而得到广泛传播。Google的《Code Review Developer Guide》明确区分了"必须修复（MUST）"、"应当修复（SHOULD）"和"可以考虑（NIT）"三级优先级标记，这套体系被许多团队直接采用或参考改造。

建设性反馈直接影响审查周期的长短和团队协作氛围。措辞粗糙的审查评论会引发防御性回应，导致一个Pull Request来回讨论超过10轮仍无结论；而格式规范、优先级清晰的反馈则可以使作者在单次修改中处理90%的审查意见，显著缩短合并时间。

## 核心原理

### 措辞的对事不对人原则

建设性反馈要求将批评对象定位在代码本身，而非代码作者的能力或意图。具体做法是避免使用"你"作为主语，改用代码片段或行为本身作为主语。

❌ 不推荐：「你这里没有处理空指针，这是个低级错误。」
✅ 推荐：「第42行的 `user.getName()` 在 `user` 为 `null` 时会抛出 `NullPointerException`，建议在调用前添加 `Objects.requireNonNull(user)` 或使用 `Optional`。」

推荐写法同时满足了三个要素：精确定位（第42行）、原因说明（NPE风险）、具体建议（两种备选方案）。这三要素构成建设性反馈的最小完整单元。

### 优先级标记体系

代码审查中并非所有问题权重相等，混淆必须修复的安全漏洞与可选的命名风格会让作者无从判断修改顺序。主流团队使用前缀标记区分优先级：

- **MUST / 阻断（Blocking）**：合并前必须修复，通常涉及正确性、安全性或性能关键路径。例如：「MUST: 此处的SQL拼接存在注入漏洞，必须改用参数化查询。」
- **SHOULD / 建议（Non-blocking）**：强烈建议但不阻止合并，涉及可维护性或代码质量。例如：「SHOULD: 这个方法已超过50行，建议按职责拆分为2-3个私有方法。」
- **NIT / 挑剔（Nitpick）**：细微风格问题，作者可自行决定是否采纳。例如：「NIT: 变量名 `tmp` 可以改为更具描述性的 `filteredUsers`。」
- **QUESTION / 疑问**：审查者不确定代码意图，需要作者解释，不要求修改。例如：「QUESTION: 这里为什么选择用 `HashMap` 而不是 `LinkedHashMap`，顺序在此处重要吗？」

团队应在贡献指南中预先约定这套标记体系，避免审查者各自使用不同前缀造成混乱。

### 建议格式的具体化要求

抽象建议（如"这里需要优化性能"）迫使作者猜测审查者的意图，等同于将思考负担完全转移给作者。建构性反馈的建议格式应达到"作者无需额外沟通即可动手修改"的具体程度。

对于简单改动，可直接在评论中给出建议代码（Suggested Change）。GitHub的Pull Request审查功能支持"Suggest changes"按钮，审查者可在评论框中嵌入修改后的代码块，作者点击"Commit suggestion"即可一键应用，这种格式将建议的传递成本降到最低。

对于复杂的架构级问题，建议格式应包含：①当前方案的具体缺陷（如"当并发请求超过100时，此处的全局锁会成为瓶颈"）；②推荐替代方案及其理由（如"建议改用 `ConcurrentHashMap` 替代同步的 `HashMap`"）；③可参考的资源（如相关文档链接或项目内的类似实现位置）。

## 实际应用

**场景一：安全漏洞的阻断性反馈**
在审查一段用户登录模块时，审查者发现密码以明文存储：「MUST: 第87行将用户密码直接存入数据库，存在严重安全风险。必须改用 `BCrypt` 或 `Argon2` 等自适应哈希算法。项目中 `UserAuthService` 已集成 `BCryptPasswordEncoder`，可参考第203行的用法。」此评论标明了优先级（MUST）、指向具体位置（第87行）、给出了两种方案并指向项目内现有实现。

**场景二：NIT级别的可选建议**
「NIT: 第23-31行的三目运算符嵌套了两层，可读性稍差。如果你认为拆成 `if-else` 更清晰的话可以考虑修改，但当前写法逻辑上完全正确，不影响合并。」此评论明确表示不阻止合并，并尊重作者的最终决定权。

**场景三：QUESTION型评论避免假设**
「QUESTION: 第56行对结果列表调用了 `.sort()`，这会修改原始列表。是否有意为之？如果调用方不应感知到排序副作用，建议改为先 `.stream().sorted()` 再收集为新列表。」此评论没有直接断定代码有错，而是先确认意图。

## 常见误区

**误区一：把所有评论都写成阻断性问题**
部分审查者习惯对所有发现的问题使用强硬措辞，把命名不够好的变量与真正的逻辑错误同等对待。这会导致作者无法区分哪些是必须在本次PR修复的，哪些可以在后续迭代中改进。结果往往是作者要么全部修改导致PR范围失控，要么产生"审查者太苛刻"的抵触情绪。正确做法是使用前述优先级标记体系，将 NIT 和 MUST 明确区分开来。

**误区二：只给出问题，不给出建议**
「这段代码性能很差。」这类评论在技术上可能是正确的，但缺少可操作的建议。建设性反馈的定义本身就包含"可行改进方向"，纯粹的问题陈述不构成完整的建设性反馈。审查者若暂时没有具体建议，应使用QUESTION格式发起讨论，而非单独留下否定性判断。

**误区三：在代码审查评论中讨论大范围重构**
当审查者发现某个模块的整体设计存在问题时，有时会在PR评论中要求推翻现有架构进行重构。代码审查评论的合理范围是当前PR涉及的变更，超出范围的架构讨论应在建设性反馈中标注「SHOULD（后续迭代）」或「建议在单独Issue中跟踪」，而非将当前PR阻断。

## 知识关联

建设性反馈建立在代码审查的基本流程之上——理解审查者与作者的角色分工、PR的生命周期（草稿→审查→修改→合并）是正确使用优先级标记的前提。不了解"阻断"与"非阻断"审查在工作流中的实际效果，就无法理解为何要区分 MUST 和 NIT。

在团队层面，建设性反馈的规范通常被写入贡献指南（CONTRIBUTING.md）或代码审查规范文档。团队对同一套优先级标记体系达成共识，是建设性反馈发挥效果的组织前提——个人掌握技巧但团队没有约定等同于语言不通。

从个人技能发展角度看，掌握建设性反馈措辞是向高质量审查者成长的必经阶段。能够准确区分"这是风格偏好还是正确性问题"并用对应的标记和措辞传达出来，反映了审查者对代码质量维度的清晰认知，而这种认知会反过来提升自身的代码编写标准。
