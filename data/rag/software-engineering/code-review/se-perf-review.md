# 性能审查

## 概述

性能审查（Performance Review）是代码审查流程中专门针对运行时效率问题的检查环节，重点识别三类高发缺陷：N+1查询问题、内存泄漏，以及算法复杂度超标。与通用代码审查不同，性能审查要求审查者在阅读代码时同时构建执行路径的心智模型，预测代码在真实数据量下的行为，而不仅仅判断逻辑是否正确。

性能审查作为独立检查类型的实践兴起于2000年代中期，随着Web应用规模膨胀，N+1查询导致的数据库过载事故频繁出现于Twitter、GitHub等早期互联网公司的工程复盘报告中。2012年GitHub博客公开的一篇事故报告揭示，一个看似无害的"加载用户关注列表"功能，在用户量增长后产生了每次请求触发数百条SQL查询的问题，直接推动了ORM层性能审查规范的标准化。

性能审查在代码合并前介入，成本远低于生产环境的性能回归排查。根据"缺陷修复成本法则"，生产环境中修复一个性能问题的代价约为代码审查阶段发现同类问题的30倍。

## 核心原理

### N+1查询检测

N+1查询是指代码在循环体内部对每条记录额外执行一次数据库查询，导致总查询数从预期的1次变为N+1次（N为记录条数）。以Python Django为例：

```python
# 危险写法：N+1查询
posts = Post.objects.all()  # 1次查询
for post in posts:
    print(post.author.name)  # 每次循环触发1次查询
```

审查时需要识别以下特征：在对QuerySet、列表或数组的迭代内部，出现了`.related_model`、`findById()`、`SELECT WHERE id=`等访问关联数据的调用。修复方式通常是使用预加载（`select_related`/`prefetch_related`/`JOIN`），将N+1次查询压缩为1~2次。

审查者在看到循环结构时，应立即追问：**循环体内是否有任何数据库调用、HTTP请求或文件I/O？** 这三类操作在循环内部均可能构成类似N+1的线性放大问题。

### 内存泄漏识别

内存泄漏在代码审查阶段可通过静态模式识别，而不需要实际运行程序。常见的可审查泄漏模式包括：

**事件监听器未注销**：在JavaScript/TypeScript中，组件挂载时调用`addEventListener`但销毁时缺少`removeEventListener`，导致闭包持有DOM节点引用，阻止垃圾回收。审查要点是"每一个add都要有对应的remove"。

**全局集合无限增长**：使用静态Map或全局数组缓存数据，但缺少容量上限或过期清理逻辑。例如：
```java
private static final Map<String, Object> cache = new HashMap<>();
// 只有put，没有evict逻辑 → 内存无界增长
```

**循环引用**：在不具备循环引用检测的垃圾回收环境（如旧版CPython使用引用计数时）中，对象A持有B的引用，B又持有A，双方引用计数永不归零。审查时需关注对象的`__del__`方法是否存在以及是否注册了弱引用。

### 算法复杂度审查

算法复杂度审查要求将代码结构翻译为Big-O表达式，识别不合理的复杂度。常见高危模式：

- **嵌套循环对同一集合操作**：两层`for`循环遍历同一列表，时间复杂度为O(n²)，数据量达到10,000条时，操作次数高达1亿次。
- **在循环内调用线性查找方法**：如`list.contains()`、`Array.indexOf()`，使整体复杂度从O(n)退化为O(n²)，而将列表换为HashSet可将内层查找降回O(1)。
- **递归缺少记忆化**：朴素递归Fibonacci实现的时间复杂度为O(2ⁿ)，而加入缓存后降至O(n)，指数级与线性级的差异在n=50时已产生约10¹⁵倍的运算量差距。

审查时的判断标准：对于用户请求处理路径上的代码，O(n²)通常需要标记为警告，O(n³)及以上应阻塞合并，除非n被明确约束在极小范围（如n≤100）。

## 实际应用

**电商商品列表页审查案例**：某电商后端PR将商品列表接口从原生SQL重构为ORM写法。审查者发现新代码在渲染商品时，循环内访问`product.category.name`和`product.seller.rating`两个关联字段，在未配置预加载的情况下，每页50件商品将触发1+50+50=101次数据库查询，而原始SQL实现只需1次JOIN查询。审查评论应明确指出：需在查询时增加`with('category', 'seller')`预加载，并附上预期查询次数的变化。

**游戏服务器帧更新函数审查**：在游戏服务端代码的PR中，`Update()`每帧调用的函数内出现`FindObjectsByType<Enemy>()`——该方法遍历全场景对象，时间复杂度为O(全场景对象数量)，每秒60帧调用时极其昂贵。性能审查应要求将敌人列表缓存为成员变量，仅在对象生成/销毁时更新。

**前端React组件内存泄漏审查**：PR新增一个WebSocket订阅组件，在`useEffect`中建立连接但返回的cleanup函数为空。审查者需指出：组件卸载时必须调用`socket.close()`，否则游离的WebSocket实例及其事件回调将持续持有组件状态的引用，在页面路由切换后积累内存占用。

## 常见误区

**误区一：将性能优化等同于微优化**  
部分审查者在发现`i++`与`++i`的差异时花费大量精力，却忽视了循环体内的数据库调用。性能审查应优先关注算法级别（O记号）和I/O次数的问题，语言层面的微优化在现代编译器的优化下几乎没有实际收益，不应占用审查注意力。

**误区二：认为内存泄漏只能靠运行时工具发现**  
Valgrind、Chrome DevTools等工具确实能检测内存泄漏，但代码审查通过模式匹配同样可以预防大多数常见泄漏。"没有cleanup的订阅"、"无上限的全局缓存"这两类模式完全可以在代码静态审阅阶段发现，等到运行时工具报警意味着问题已进入生产环境。

**误区三：复杂度评估只看最坏情况**  
审查者有时会将实际几乎不会被大集合调用的O(n²)函数也强制要求重构，导致代码可读性下降而实际性能收益为零。正确的评估方法是结合**数据量上界**判断：若某个函数的输入`n`被业务逻辑强制约束在20以内（如用户最多选择20个收货地址），则O(n²)带来的400次操作完全可以接受，不应列为性能问题。

## 知识关联

性能审查以**审查清单**为操作载体——标准化的性能审查清单通常包含N+1检查项、内存管理检查项和复杂度阈值定义三个固定区块，确保审查者不遗漏关键检查点。**安全审查**与性能审查存在一类共同的关注点：SQL拼接既是注入漏洞也是无法使用预编译语句缓存的性能瓶颈，两类审查在数据库访问代码处有实质性重叠。

性能审查的方法论直接服务于**游戏项目审查重点**：游戏项目在帧更新路径、物理碰撞检测循环、资源加载管理等方面对N+1访问模式、内存分配频率和算法复杂度的要求比普通Web后端更为严苛，通用性能审查建立的检查思维在游戏场景下需要进一步精化为针对GC压力、DrawCall数量等平台特定指标的专项审查。
