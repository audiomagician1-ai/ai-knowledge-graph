# 性能审查

## 概述

性能审查（Performance Review）是代码审查流程中专注于发现运行时效率问题的专项环节，目标是在代码合并前识别出会导致响应延迟、内存溢出或 CPU 占用异常的代码模式。与通用代码审查不同，性能审查要求审查者在阅读代码时主动推演代码在真实数据规模下的执行路径，而非仅判断逻辑正确性。

性能审查作为一项系统化实践，在 2000 年代随着 Web 应用规模扩大而逐渐从"偶发性优化讨论"演变为结构化的审查检查项。Google 内部的工程实践指南（Engineering Practices）明确将复杂度分析和数据库查询效率列为代码审查的必查维度，这一做法随后被众多互联网公司效仿并形成行业惯例。

性能审查的价值在于：修复生产环境中的性能缺陷成本远高于审查阶段。以数据库 N+1 查询为例，在小测试数据集（10 条记录）下该问题完全不可见，但当生产数据量达到 10 万条时，单次接口调用可能触发超过 10 万次 SQL 查询，导致接口响应时间从毫秒级劣化至分钟级。

## 核心原理

### N+1 查询问题

N+1 查询是指代码先执行 1 次查询取出 N 条父记录，再为每条父记录单独执行 1 次查询取关联数据，共触发 N+1 次数据库往返。以 ORM 框架为例，以下 Python 伪代码是典型的 N+1 模式：

```python
# 触发 1 次查询
orders = Order.objects.all()
for order in orders:
    # 每次循环触发 1 次查询，N 条记录 = N 次额外查询
    print(order.customer.name)
```

审查者需识别循环体内部出现的懒加载属性访问（如 `order.customer`），并要求改写为预加载形式（Django 中使用 `select_related` 或 `prefetch_related`），将总查询次数压缩为固定的 2 次。判断依据是：**循环体内是否存在依赖外部数据源的访问操作**，包括数据库、Redis、外部 HTTP 接口等。

### 内存泄漏模式识别

内存泄漏在审查层面通常表现为以下几类可静态识别的代码模式：

1. **未关闭的资源引用**：文件句柄、数据库连接、网络 Socket 在异常分支中未被释放。审查时需确认资源获取处是否有对应的 `finally` 块或 `with` 语句（Python）、`try-with-resources`（Java）或 RAII 机制（C++）。

2. **全局/静态容器的无界增长**：缓存 Map 或 List 只有写入操作，没有淘汰策略（如 TTL 过期、LRU 上限）。审查者应追问：这个集合的最大容量是多少？何时清理？

3. **事件监听器未注销**：在 JavaScript/Node.js 代码中，`EventEmitter` 或 DOM 事件监听器在对象生命周期结束时未调用 `removeEventListener`，导致持有对象无法被 GC 回收。Node.js 默认会在单个 EventEmitter 上超过 10 个监听器时打印警告，审查时可将此作为信号。

### 算法复杂度分析

审查者需对关键路径的代码段进行时间复杂度和空间复杂度评估，使用大O表示法（Big-O Notation）量化。常见需要升级复杂度的场景：

- **嵌套循环访问同一集合**：两层 `for` 循环遍历同一列表进行匹配，时间复杂度为 O(n²)。若 n 为用户提交的数据条数（无上限），则该接口存在被恶意输入攻击的风险。审查建议：改用 HashMap 将查找操作降至 O(1)，整体复杂度降为 O(n)。

- **排序操作的位置**：在循环内部重复调用 `sort()`，每次调用代价为 O(n log n)，整体变为 O(m × n log n)。应将排序提取到循环外部执行一次。

- **递归无记忆化**：斐波那契或树形结构遍历的朴素递归实现，时间复杂度为指数级 O(2ⁿ)，需检查是否有必要引入记忆化（Memoization）或迭代替代。

## 实际应用

**场景一：电商订单列表接口审查**  
审查一个返回订单列表的 REST API 时，发现控制器层先查出订单列表，再在序列化层（Serializer）的 `to_representation` 方法内访问 `order.product.category.name`。由于序列化层对每条订单单独执行链式关联查询，50 条订单将产生 150+ 次 SQL 查询。审查意见应明确要求在 QuerySet 层添加 `select_related('product__category')`，并附上修改前后的查询次数对比。

**场景二：数据导入功能的内存审查**  
审查批量导入 CSV 功能时，代码将整个文件内容一次性读入内存列表再处理。若 CSV 文件大小为 500MB，则 JVM 或 Python 进程需要在内存中同时持有原始字符串和解析后的对象，实际内存占用可达 1–2GB。审查建议改为流式读取（Python `csv.reader` 逐行迭代，Java `BufferedReader` 逐行读取），将内存占用控制在常数级 O(1)。

**场景三：搜索功能的复杂度审查**  
审查用户名模糊搜索功能时，发现实现方式为从数据库取出所有用户记录（可能达数百万条），在内存中进行字符串包含判断。该方案时间复杂度 O(n)、空间复杂度 O(n)，随数据增长线性劣化。审查应指出正确方案为数据库层的 `LIKE` 索引查询或全文搜索引擎（如 Elasticsearch）。

## 常见误区

**误区一：小数据集通过测试即视为无性能问题**  
开发者常以"本地测试没有问题"为由忽略审查意见。但性能问题的本质是**复杂度随数据规模的增长方式**，而非特定数据量下的绝对时长。O(n²) 算法在 n=100 时耗时 1ms，在 n=10000 时耗时 10 秒——审查者必须以生产环境的数据量级作为评估基准，而非开发环境的测试数据。

**误区二：将所有数据库查询减少等同于性能优化**  
部分审查者看到多次数据库查询就一律要求合并，但过度的 `JOIN` 或超宽的 `SELECT *` 有时比多次小查询代价更高。性能审查的判断标准是：总 I/O 量、网络往返次数与查询返回的数据量之间的综合权衡，而非单纯追求查询次数最少。

**误区三：认为内存泄漏只能靠运行时工具检测，审查阶段无法发现**  
实际上，全局静态缓存无淘汰策略、EventListener 未注销、连接池未释放等模式均可在代码静态阅读阶段识别。性能审查不依赖 Profiler 或 Heap Dump 工具，而是通过识别已知的泄漏代码模式在代码合并前介入。

## 知识关联

性能审查以**审查清单**为执行载体——N+1 查询、内存泄漏高风险模式、O(n²) 嵌套循环等检查项应被固化为清单条目，使审查者不依赖个人经验记忆而是按清单逐项核对。清单中的性能项通常按问题发生层次组织：数据库访问层（N+1、全表扫描）→ 内存管理层（资源释放、容器上界）→ 计算逻辑层（复杂度分析）。

在工具辅助方面，静态分析工具（如 SonarQube 的 SQL 注入与 N+1 检测规则、FindBugs 的资源泄漏检测）可自动标记部分性能风险，但这些工具的召回率有限，无法替代人工对业务逻辑上下文的判断。性能审查的人工价值在于理解**数据规模假设**——这是静态工具无法从代码中自动推断的信息。
