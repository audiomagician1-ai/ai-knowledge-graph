# 大型变更审查

## 概述

大型变更审查（Large Change Review）是指在代码审查流程中，针对单次提交涉及行数超过400-500行或跨越多个模块的变更，采用系统化策略将其拆分、组织、逐步提交以降低审查负担的一套方法论。与普通代码审查不同，大型变更若不加处理直接提交，审查者往往陷入"无从下手"的困境，导致审查质量严重下降，漏掉真正的逻辑缺陷。

大型变更审查的概念随着开源社区的工程化程度提升而逐渐成型。2010年前后，Linux内核维护者Linus Torvalds多次在邮件列表中强调"补丁应当是可独立审阅的最小单元"，这一观点成为现代分布式代码审查实践的重要参照。Google内部研究数据表明，当单次变更超过400行时，审查者发现缺陷的概率从每100行约1.5个骤降至不足0.5个，因此"小而聚焦的变更"成为Google工程规范的明文要求。

大型变更审查之所以值得专门讨论，是因为它面临的挑战不仅是代码量本身，还包括变更之间的依赖关系管理、上下文在多个Pull Request（PR）之间的传递，以及审查者认知负荷的主动控制。

## 核心原理

### 拆分策略：按关注点分离

拆分大型变更的基本原则是"单一变更只做一件事"。常见的拆分维度有三类：**按层次拆分**（数据层、业务层、接口层分别提交）、**按功能切片拆分**（每个PR包含一个完整的垂直功能切片，从数据到UI）、**按重构与新增分离**（先提交纯重构PR，再提交功能新增PR）。

最典型的反例是"混合PR"：同一PR内既修改了数据库Schema、又调整了API逻辑、又更新了前端组件。这种PR即便只有300行，审查者也需要在三个完全不同的心理模型之间来回切换，错误率与审查一个1000行单一职责PR相当。拆分时需要注意原子性——每个子PR在合并后代码库必须处于可构建、可测试的状态，不能出现"PR1合并后测试全红，等PR2合并才能恢复"的依赖断层。

### Stacked PR（堆叠式PR）

Stacked PR是指一组存在明确先后依赖关系的PR链，其中PR-B基于PR-A的分支创建，PR-C基于PR-B的分支创建，形成线性栈结构。这种模式在GitHub上需要手动管理，而Graphite、ghstack等工具专门为此场景设计，可以自动将栈中每个PR的基础分支更新至前驱PR合并后的状态。

Stacked PR的关键挑战是**基础分支漂移**（Base Branch Drift）：当PR-A在代码审查中被修改后，后续所有PR的diff都会发生变化，审查者看到的内容失去参照。解决方案是在PR-A合并后立即执行`git rebase`将整个栈重置到`main`分支，并在PR描述中明确标注"本PR依赖#123，请先审查#123"。团队应规定栈的深度上限，通常不超过4-5层，否则依赖链的管理成本超过拆分带来的收益。

### 增量审查与上下文保持

增量审查要求作者为每个子PR提供足够的独立上下文，让审查者无需通读整个PR链也能理解当前变更的意图。具体做法包括：在每个PR描述顶部附上整体方案的架构图或链接；用"本PR在整体方案中负责X部分"一句话定位；在Commit Message中用`[Part 2/5]`标注序号，让审查记录具备可追溯性。

增量审查还涉及**审查时间窗口**的管理。研究显示审查者对同一系列PR保持上下文的时间窗口约为48-72小时，超过这个窗口后需要重新熟悉背景。因此，大型变更的子PR应在72小时内保持活跃流转，作者需主动催促审查或将尚未审查完毕的PR标记为`Draft`以避免误合并。

## 实际应用

**场景一：微服务拆分重构**
某团队将一个单体服务拆分为三个微服务，原始变更涉及约2800行代码。工程师将其拆分为5个Stacked PR：PR1抽取公共接口定义（120行）、PR2迁移用户服务（340行）、PR3迁移订单服务（580行）、PR4迁移支付服务（410行）、PR5删除旧单体代码并更新路由（380行）。每个PR在CI中独立运行测试套件，整体审查周期从预估的5天压缩至3天，且审查评论中发现了PR3中一个并发安全问题，该问题在完整的2800行diff中极易被掩盖。

**场景二：数据库Schema迁移伴随应用变更**
在数据库字段重命名的场景中，正确做法是先提交一个向后兼容的Schema迁移PR（同时写入新旧字段），合并并部署后再提交应用层切换PR，最后提交清理旧字段的PR。如果将三步合并为一个PR，数据库迁移与应用发布的时序耦合会导致零停机部署失败。

## 常见误区

**误区一：拆分越细越好**
部分工程师走向另一个极端，将每个函数改动单独提交一个PR。当单个PR的改动量低于50行且与其他PR存在强逻辑关联时，审查者需要在多个PR之间反复切换，理解成本反而高于合并审查。合理的粒度应保证每个PR能在15-30分钟内完成审查，这对应约200-400行的净变更量。

**误区二：Stacked PR等同于Feature Branch的多次提交**
Stacked PR中每个节点都是独立的PR，对应独立的代码审查流程和CI运行；而Feature Branch的多次提交是在同一PR下的Commit历史，不产生独立的审查节点。混淆两者会导致团队在使用GitHub时创建错误的分支结构，使PR之间的依赖关系无法被代码托管平台正确追踪。

**误区三：大型变更只要加详细注释就无需拆分**
注释能帮助读者理解单个函数或模块的意图，但无法降低审查者同时处理多个模块变更时的工作记忆压力。认知负荷研究（Miller, 1956）表明人类工作记忆同时处理的组块上限约为7±2个，当PR涉及超过7个独立的逻辑变更点时，注释的补充效果边际递减，而拆分才能从根本上减少审查者同时激活的信息节点数量。

## 知识关联

大型变更审查建立在对**代码审查基本流程**的理解之上，例如PR描述规范、CI集成和审查者分配机制，这些基础决定了拆分后的每个子PR能否被高效处理。

向前延伸，大型变更审查的拆分思维与**持续集成/持续交付（CI/CD）**中的"小批量发布"原则高度一致：Stacked PR在代码审查层面对应的是流水线发布策略中的蓝绿部署和金丝雀发布，都以降低单次变更的影响范围为目标。此外，Git的`git range-diff`命令可用于在Stacked PR被rebase后对比前后两版diff的差异，是大型变更审查工具链中值得掌握的具体技能。
