# 大型变更审查

## 概述

大型变更审查是指当一次代码提交的规模超过审查者合理处理能力时，采用特定拆分策略和增量方法来完成代码审查的工程实践。通常认为，单次 Pull Request（PR）超过 400 行代码改动时，审查质量会显著下降——研究表明，审查者在审查超过 400 行变更后，每小时发现缺陷的能力下降约 20%，而超过 1000 行时几乎无法进行有效审查。大型变更审查的核心目标是在不牺牲审查质量的前提下，将大体量的功能改动安全地集成到代码库中。

这一实践的兴起与互联网公司大规模工程协作的发展密切相关。Google 内部工程规范明确要求开发者将大型变更拆分为一系列小型、独立的 CL（Changelist），每个 CL 只做一件事。Facebook（现 Meta）的 Phabricator 系统在 2010 年代推广了"Stacked Diff"工作流，允许开发者在前一个差异未合并时就基于其提交下一个，形成依赖链。这些实践随后被 GitHub 的 Stacked PR 工具（如 Graphite、ghstack 等第三方工具）进一步普及。

大型变更审查的重要性在于：代码审查的有效性随 PR 体积呈非线性下降，而大型 PR 往往是功能开发中的常态而非例外。重构、API 迁移、数据库 Schema 变更等任务天然涉及大量文件，若不加以管理，会导致审查者感到不堪重负、审查时间拉长（有统计显示大型 PR 的平均合并等待时间是小型 PR 的 3 至 5 倍），以及潜在缺陷难以发现等问题。

## 核心原理

### 拆分策略：按关注点分离

拆分大型变更的核心依据是**单一职责原则**在 PR 粒度上的应用。一个有效的拆分方式是将变更按"准备层"与"功能层"分开：第一个 PR 仅包含重构、测试基础设施搭建或接口定义，第二个 PR 才引入实际的业务逻辑变更。例如，假设需要将一个旧的 REST API 迁移到 gRPC：

- **PR 1**：添加 Proto 文件定义和生成代码（无业务逻辑）
- **PR 2**：实现 gRPC 服务端处理器
- **PR 3**：逐步替换调用方，保留 REST 兼容层
- **PR 4**：删除旧 REST 实现

这种"剥洋葱"式拆分使每个 PR 都可独立部署，同时清晰地展示了变更的演进脉络。

### Stacked PR（堆叠式 PR）

Stacked PR 是一种将多个互相依赖的 PR 排成线性链的工作流。其结构如下：

```
main ← PR-A（基础层）← PR-B（业务层）← PR-C（UI层）
```

开发者可以在 PR-A 审查期间继续开发 PR-B 和 PR-C，不必等待前置 PR 合并。当 PR-A 被合并并且 main 分支更新后，PR-B 的 base branch 切换到 main，整个链条向前推进。

Stacked PR 的关键挑战在于**rebase 传播**：当堆栈底部的 PR 因审查反馈发生修改时，所有上层 PR 都需要 rebase。Graphite 等工具通过命令 `gt sync` 自动完成这一传播，而手动管理时开发者需执行一系列 `git rebase --onto` 命令。

### 增量审查与上下文保留

增量审查要求每个 PR 必须携带足够的上下文，使审查者无需同时阅读整个堆栈即可理解当前变更。具体实践包括：

1. **PR 描述中的变更地图**：用一个有序列表展示整个变更系列，并标注当前 PR 的位置（如"这是系列的第 2/4 步"）。
2. **不破坏主干构建**：每个 PR 合并后，主干代码必须处于可构建、可测试的状态，即使整体功能尚未完成。这通常借助**特性开关（Feature Flag）**实现——新代码已合并但被开关隐藏。
3. **审查者复用**：同一堆栈的所有 PR 应尽量由同一组审查者处理，以降低他们重新建立上下文的成本。

## 实际应用

**数据库 Schema 迁移场景**：假设需要将一个字段从 `NOT NULL` 改为可空并重命名。若在一个 PR 中完成，部署窗口期可能导致新旧代码同时运行时出现崩溃。正确的大型变更审查拆分方式是：PR-1 添加新的可空字段；PR-2 更新应用代码同时写入新旧字段；PR-3 迁移历史数据；PR-4 删除旧字段。每步都有独立的审查、测试和部署。

**大规模重构场景**：在将 JavaScript 代码库迁移到 TypeScript 时，工程师通常采用"文件粒度"拆分，即每个 PR 转换一个模块或目录，而非一次提交全部 `.ts` 文件。GitHub 工程团队在其 2021 年的 TypeScript 迁移博客中记录了这一方法，该迁移历时数月，由数十个独立 PR 完成。

**开源贡献场景**：向大型开源项目（如 Linux 内核或 CPython）提交功能性改动时，维护者通常明确要求贡献者将 PR 拆分为"可独立理解的原子提交系列"。Linux 内核的 patch series 投递到邮件列表时，每封邮件对应一个逻辑单元，且附有 `[PATCH v2 3/7]` 这样的编号格式，这是大型变更增量审查的经典形式。

## 常见误区

**误区一：拆分越细越好**。将一个 2000 行的变更拆成 20 个 PR 并不总是正确的。过度拆分会导致审查者失去对整体设计意图的把握，每个 PR 之间的接口设计也可能显得刻意而不自然。一般经验是单个 PR 保持在 200-400 行净变更之间，且每个 PR 对应一个可用语言描述的"完成了什么"的结论。

**误区二：Stacked PR 等同于 feature branch 嵌套**。Stacked PR 的每个节点都应该能够**独立合并到主干**，而 feature branch 嵌套往往只有最终分支会被合并。混淆两者会导致开发者以为自己在做增量审查，实则还是在等待整个堆栈全部完成才能合并，依然产生大型 PR 的所有问题。

**误区三：只要最终功能正确就可以在 PR 描述中省略上下文**。审查的目的不仅是验证最终代码的正确性，还包括确保每个中间步骤的设计合理性。一个缺乏上下文描述的 PR-3/5 会让审查者不得不去阅读前两个 PR 的历史，极大增加认知负担，实际上等同于退化为一次大型 PR 的审查。

## 知识关联

大型变更审查建立在对**Git 分支模型**（尤其是 `rebase` 和 `--onto` 操作）的理解之上，因为 Stacked PR 工作流的手动管理依赖这些底层操作。同时，**特性开关（Feature Flag）**是实现"每步可部署"的技术保障，没有特性开关往往难以对用户不可见的中间状态进行无风险部署。

从工程文化角度，大型变更审查与**持续集成（CI）实践**紧密相关：只有 CI 管道对每个 PR 都独立运行完整测试，才能保证堆栈中的每个节点真正独立有效。此外，熟悉大型变更审查的团队通常也会对**提交信息规范**（如 Conventional Commits）有更高要求，因为清晰的提交历史是后续 rebase 操作正确性的重要保障。
