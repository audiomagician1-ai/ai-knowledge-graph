# 算法复杂度分析

## 概述

算法复杂度分析是一种用数学符号量化算法资源消耗的方法，分为**时间复杂度**（描述执行步骤数随输入规模的增长关系）和**空间复杂度**（描述内存占用随输入规模的增长关系）。这套分析体系由计算机科学家 Donald Knuth 在其1968年出版的《计算机程序设计艺术》中系统化，并引入了大O符号（Big-O Notation）作为标准表达工具。

大O符号最初由数学家 Paul Bachmann 于1894年引入数论，后被 Edmund Landau 推广，因此又称"Bachmann–Landau符号"。在算法分析中，O(f(n)) 表示当输入规模 n 趋向无穷大时，算法的资源消耗增长速度不超过 f(n) 的某个常数倍。这种"渐近分析"的核心价值在于：它屏蔽了硬件差异和常数系数，让工程师能够在代码编写阶段预测算法在海量数据下的行为——一个 O(n²) 的排序算法处理 10 万条数据需要约 10^10 次操作，而 O(n log n) 的算法仅需约 1.7×10^6 次，差距约 6000 倍。

## 核心原理

### 常见复杂度类别及其数量级差异

算法复杂度从低到高排列为：O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!)。以 n=20 为例，O(n²)=400，而 O(2ⁿ)=1,048,576，O(n!)=2.4×10^18。这意味着指数级和阶乘级算法在现实工程中几乎无法用于 n>30 的场景，不论硬件多快。

- **O(1)**：哈希表的直接寻址查找，无论数据量是100还是1亿，执行时间恒定。
- **O(log n)**：二分查找每次将候选范围缩半，查找10亿条有序记录只需约30次比较（log₂ 10⁹ ≈ 30）。
- **O(n log n)**：快速排序、归并排序的平均时间复杂度，是基于比较的排序算法的理论下界（由信息论证明）。
- **O(n²)**：冒泡排序、插入排序的最坏情况，处理 10,000 条记录需约 10^8 次操作，在现代 CPU 上约耗时0.1秒，尚可接受，但100万条记录则需10^12次操作，完全不可用。

### 主定理（Master Theorem）与递归分析

对于形如 T(n) = aT(n/b) + f(n) 的分治递归算法，主定理给出了复杂度的直接计算方法：

- 若 f(n) = O(n^(log_b a - ε))，则 T(n) = Θ(n^(log_b a))
- 若 f(n) = Θ(n^(log_b a))，则 T(n) = Θ(n^(log_b a) · log n)
- 若 f(n) = Ω(n^(log_b a + ε))，则 T(n) = Θ(f(n))

例如，归并排序满足 T(n) = 2T(n/2) + O(n)，其中 a=2, b=2，n^(log₂2) = n，f(n)=O(n)，符合第二种情况，故 T(n) = O(n log n)。

### 最好、最坏与平均情况

同一算法在不同输入下的表现可能截然不同。快速排序的**平均时间复杂度**为 O(n log n)，但当输入已排序时（最坏情况），其时间复杂度退化为 O(n²)，因为每次 pivot 选择均为最小或最大元素，导致递归树完全不平衡。这就是为什么工业级快排实现（如 C++ STL 的 `std::sort`）采用"三路取中"或"随机化 pivot"策略——将最坏情况的触发概率压到极低。平摊分析（Amortized Analysis）则用于动态数组扩容等场景：单次扩容为 O(n)，但均摊到每次插入操作后，每次插入的平摊复杂度为 O(1)。

### 空间复杂度的常见陷阱

递归调用本身会消耗栈空间。一个看似"不占额外内存"的递归二分查找，其空间复杂度实为 O(log n)，因为递归深度即为调用栈的层数。归并排序空间复杂度为 O(n)（需要辅助数组），而快速排序原地排序的空间复杂度为 O(log n)（仅栈空间）——这是两者在实际选型时的重要差异。

## 实际应用

**数据库索引设计**：B树索引使查询从全表扫描的 O(n) 降至 O(log n)，对于存储 1 亿行的表，索引查询约需 27 次磁盘 IO（log₁₀₀ 10⁸），而全表扫描需 100 万次（假设每页100行）。

**字符串匹配算法选型**：暴力匹配算法时间复杂度为 O(mn)（m为模式长度，n为文本长度），KMP 算法通过构建失配函数将复杂度降至 O(m+n)。在文本长度为 100 万、模式长度为 1000 的场景下，两者操作次数之比约为 10⁹ 对比 10⁶，差距千倍。

**前端渲染优化**：React 早期 diff 算法若采用完整树比对，时间复杂度为 O(n³)；React 16 通过同层比较启发式策略将其降为 O(n)，这使得在包含 10,000 个节点的虚拟 DOM 中，复杂度从 10^12 降至 10^4，是前端框架能够实时响应的根本原因。

## 常见误区

**误区一：O(n²) 算法总是比 O(n log n) 慢**
大O符号隐藏了常数系数。对于 n<50 的小数组，插入排序（O(n²)）由于缓存友好性和极小的常数系数，实际运行速度往往快于归并排序（O(n log n)）。这正是 Timsort（Python 和 Java 的默认排序算法）的设计思路：对长度小于64的子数组使用插入排序，大规模数据再使用归并排序。

**误区二：时间复杂度等于实际运行时间**
同为 O(n) 的两个算法，如果一个频繁触发缓存缺失（Cache Miss），另一个具有良好的内存访问局部性，实际运行时间可能相差 10-100 倍。矩阵转置操作（理论复杂度 O(n²)）若按列遍历，在 n=1000 时因缓存缺失导致的实际耗时可达按行遍历的 5-10 倍。

**误区三：只分析时间复杂度，忽视空间复杂度**
在内存受限的嵌入式系统或处理大规模数据的流式计算中，空间复杂度往往是选型的决定性因素。例如，归并排序需要 O(n) 额外空间，在处理 10GB 数据的外部排序场景中，这意味着额外占用 10GB 内存，工程上通常改用原地 O(log n) 空间的快速排序或多路归并方案。

## 知识关联

本概念建立在**性能分析概述**中介绍的基准测试（Benchmarking）和性能指标体系之上——复杂度分析提供了理论预测，基准测试则提供了实测数据，两者互相校验：若实测结果与理论复杂度曲线吻合，则排除了常数系数或算法退化的嫌疑；若偏差较大，则需检查缓存行为、分支预测或内存分配等底层因素。

理解算法复杂度是制定**代码质量标准**的量化依据。代码审查中常见的"禁止在循环体内执行数据库查询"这条规则，其本质是将 O(n) 次数据库调用（每次 O(1)）合并为单次批量查询，避免时间复杂度从 O(n) 实际劣化为 O(n×RTT)（RTT为网络往返延迟）。复杂度分析将代码质量从主观感受转化为可量化、可比较的工程指标。
