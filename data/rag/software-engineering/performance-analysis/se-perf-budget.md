# 性能预算

## 概述

性能预算（Performance Budget）是一种为Web应用或软件系统预先设定可量化性能指标上限的工程实践。它的核心思想是：将性能目标像财务预算一样进行管理——团队在开发过程中所做的每一个技术决策，都必须在不超出既定"预算"的前提下进行。一旦某个改动（例如引入新的第三方库、添加图片资源）导致指标超标，团队必须做出取舍：优化其他部分来"腾出空间"，或拒绝该改动。

性能预算的概念由Tim Kadlec于2013年在其博客文章《Setting a Performance Budget》中正式提出并系统化，此后被Google、Etsy、Pinterest等公司纳入前端工程标准流程。其出现背景是：随着Web页面平均体积从2010年的约500KB增长至2023年超过2MB，在没有明确约束机制的情况下，性能退化往往是渐进式、难以察觉的。

性能预算之所以重要，是因为它将抽象的"我们要做一个快速的应用"转化为可执行的工程约束。研究表明，页面加载时间每增加1秒，电商网站的转化率平均下降7%。没有预算约束时，每次小的性能妥协都显得"合理"，但累积效果会让产品彻底失去竞争力。

## 核心原理

### 1. 预算指标的三种类型

性能预算的指标可分为三类，各有其测量维度和适用场景：

**基于数量的指标（Quantity-based metrics）**：直接衡量资源的物理大小。例如：JavaScript总包体积不超过300KB（压缩后），图片资源总大小不超过1MB，HTTP请求数量不超过50个。这类指标对工程师最直观，易于在构建流水线中自动检查。

**基于时间的指标（Timing-based metrics）**：衡量用户感知的加载速度。典型指标包括：首次内容绘制（FCP）不超过1.5秒，最大内容绘制（LCP）不超过2.5秒，首次输入延迟（FID）不超过100毫秒，累计布局偏移（CLS）不超过0.1。这些数字直接来自Google的Core Web Vitals标准（2020年发布）。

**基于规则的指标（Rule-based metrics）**：使用工具生成的综合评分。例如Lighthouse性能分数不得低于90分，PageSpeed Insights得分不得低于85分。这类指标综合了多个底层指标，适合作为持续集成的通过/失败门控条件。

### 2. 预算分配的计算方法

设定时间类预算需要参考竞品基准（Competitive Benchmarking）和用户环境假设。一个常用的推算公式是：

> **目标LCP = 竞品LCP × 0.8**（即比竞品快20%）

对于资源体积预算，可以通过反向推算：假设目标用户在4G网络（下行速度约20Mbps）下，希望关键资源在1秒内加载完毕，则：

> **最大资源体积 = 网络速度 × 目标时间 = 20Mbps × 1s ÷ 8 = 2.5MB**

但实际中需扣除DNS解析、TCP握手等固定开销（约300-500ms），所以实际可用体积更小。这种计算让预算数字有工程依据，而非主观拍板。

### 3. 持续监控与执行机制

性能预算只有嵌入CI/CD流程才能真正发挥作用。常用工具链包括：

- **Webpack Performance Hints**：在webpack.config.js中设置 `performance.maxAssetSize: 250000`（单位bytes），构建时超标即报错
- **Lighthouse CI**：在GitHub Actions中配置断言规则，当LCP超过预算时阻止PR合并
- **Bundlesize / Size-limit**：专门用于监控JavaScript包体积变化，支持对比基线并在超标时失败

监控应分为两个层面：**构建时检查**（静态资源大小、代码分割情况）和**真实用户监控RUM**（Real User Monitoring，收集实际用户环境下的性能数据）。两者缺一不可，因为构建时数据无法反映CDN性能、用户设备差异等真实因素。

## 实际应用

**场景一：电商网站图片预算管理**

某电商团队设定商品列表页图片总大小不超过500KB的预算。当设计师要求将商品主图从WebP格式换回JPEG以获得更广泛兼容性时，测算发现JPEG版本体积增加约35%，将突破预算。工程师因此提出折中方案：使用`<picture>`标签实现格式降级，主流浏览器使用WebP，仅对不支持WebP的浏览器回退到JPEG。预算约束直接推动了更优的技术方案。

**场景二：SPA应用的JavaScript预算**

一个React单页应用设定初始JS包体积不超过200KB（gzip压缩后）的预算。当团队引入moment.js时，发现该库压缩后仍有70KB，占去预算35%。预算压力迫使团队调研替代方案，最终用仅2KB的day.js替代，节省68KB预算空间。这是性能预算推动依赖选型决策的典型案例。

**场景三：API响应时间预算**

后端服务可以对API响应时间设定预算：P95延迟不超过200ms，P99延迟不超过500ms。当引入新的数据库查询时，通过压测对比基线，发现P99延迟从300ms上升至720ms，超出预算，触发强制优化流程（添加索引、引入缓存），确保上线前恢复达标。

## 常见误区

**误区一：用平均值而非百分位数设置时间预算**

许多团队用"平均响应时间200ms"作为预算，但平均值会被少数极快的请求拉低，掩盖大量慢请求的存在。正确做法是使用P95或P99百分位数——P95延迟200ms意味着95%的用户请求都在200ms内完成，这才是真实的用户体验保障。用平均值设预算，等于允许5%甚至20%的用户遭受极差体验。

**误区二：预算一旦设定就不再调整**

性能预算不是一成不变的规章制度。当业务需求发生重大变化（例如从信息展示网站转型为富交互应用），或者当网络基础设施发生重大改变（例如目标用户从3G升级为5G为主），都应重新评估并调整预算。预算是动态的工程契约，而不是永久性的技术债务裁判标准。僵化遵守过时预算，会让团队在错误的指标上投入优化精力。

**误区三：混淆"预算超标"与"性能问题"**

预算超标是一个触发信号，而非一定代表真实的性能问题。例如，一个后台管理系统的JS包体积超出了参照消费者应用设定的300KB预算，但其目标用户都是使用高配PC和千兆内网的内部员工，实际体验完全可接受。性能预算必须与真实用户群体的设备和网络条件相绑定，脱离用户场景的预算数字没有工程意义。

## 知识关联

**与Core Web Vitals的关系**：Google的Core Web Vitals（LCP、FID/INP、CLS）为性能预算中的时间类指标提供了行业公认的参考阈值——"良好"（Good）区间的上限值（LCP≤2.5s、INP≤200ms、CLS≤0.1）是设定预算的最直接参照标准，直接使用这些数值作为预算起点是业界常见做法。

**与构建工具的关系**：Webpack、Rollup、Vite等现代构建工具均内置了性能预算检查接口，理解构建工具的代码分割（Code Splitting）和Tree Shaking机制，是将体积类预算落实到工程实践的前提技能。

**与性能监控平台的关系**：性能预算的长期执行依赖Datadog、New Relic、Sentry Performance等APM工具，以及SpeedCurve、Calibre等专门的Web性能监控服务。这些工具支持将预算阈值配置为告警规则，实现从"一次性设定"到"持续自动执行"的闭环管理。
