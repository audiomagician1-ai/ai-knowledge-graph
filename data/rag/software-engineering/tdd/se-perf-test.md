# 性能测试

## 概述

性能测试是通过模拟真实用户负载来测量软件系统在特定条件下的响应速度、吞吐量、资源占用率等量化指标的测试方法。与功能测试验证"能不能用"不同，性能测试回答的是"能用到什么程度"——例如，当系统同时承受 10,000 个并发用户时，接口响应时间是否仍能保持在 200ms 以内。

性能测试的工业化起步于 1990 年代，Mercury Interactive（后被 HP 收购）于 1999 年发布的 LoadRunner 是业界最早的商业级负载测试工具之一，它奠定了"虚拟用户（Vuser）"模拟并发的核心范式。2003 年，Apache 基金会推出开源工具 JMeter，使性能测试进入中小团队。2017 年出现的 k6 则针对 CI/CD 流水线设计，支持用 JavaScript 编写测试脚本，将性能测试深度整合进 DevOps 工作流。

性能测试的价值在于暴露在功能测试阶段完全不可见的问题：内存泄漏往往只在运行 24 小时后才显现；数据库连接池耗尽只有在 500 个并发请求时才会触发；CDN 回源雪崩只有在瞬间流量激增时才会发生。如果不做性能测试直接上线，这些问题将在生产环境以系统宕机的形式暴露给真实用户。

## 核心原理

### 性能测试的三种主要类型

**负载测试（Load Testing）** 在预期的正常流量上限下运行系统，验证系统能否满足 SLA 指标。例如，某电商平台合同承诺在 3,000 并发用户下 P95 响应时间 ≤ 500ms，负载测试就是用脚本持续发出 3,000 个并发请求并测量 P95 值。

**压力测试（Stress Testing）** 刻意超出系统容量上限，寻找系统的断裂点（Breaking Point）。测试会从低并发逐步爬坡，例如每 2 分钟增加 500 个并发用户，直到错误率超过 1% 或响应时间超过阈值，记录此时的并发数即为系统极限。压力测试还验证系统降级是否优雅——是返回 503 状态码还是直接挂起。

**基准测试（Benchmark Testing）** 在严格隔离的标准环境下，对单一组件（如数据库查询、特定 API 端点）进行重复测量，获得可重现的基准数值，用于代码优化前后的对比。Java 生态中的 JMH（Java Microbenchmark Harness）是专门做微基准测试的框架，它使用 `@Benchmark` 注解并通过 JVM 预热（Warmup Iterations）消除 JIT 编译的干扰。

### 关键性能指标

性能测试的度量体系围绕以下公式展开：

- **吞吐量（Throughput）**：`TPS = 完成事务数 / 测试持续时间（秒）`，单位为 transactions per second。
- **响应时间百分位数**：P50、P95、P99 分别代表 50%、95%、99% 的请求响应时间落在该值以内。P99 = 2000ms 意味着 1% 的用户等待超过 2 秒，对于高流量系统这仍是显著的体验问题。
- **并发用户数（Concurrent Users）** 与 **到达率（Arrival Rate）** 是不同概念——并发用户数指某时刻同时在线的虚拟用户，到达率指每秒新发起请求的速率。JMeter 中通过"线程组"控制并发用户数，k6 中通过 `arrival-rate executor` 控制到达率。

### JMeter 与 k6 的脚本机制对比

JMeter 使用 XML 格式的 `.jmx` 文件描述测试计划，GUI 操作友好但脚本难以进行 Git diff 审查。一个典型 JMeter 线程组配置包含：线程数（并发用户）、Ramp-Up 时间（多少秒内达到目标并发）、循环次数。

k6 采用 JavaScript 脚本，以下是一个最小示例：

```javascript
import http from 'k6/http';
export const options = {
  vus: 100,          // 虚拟用户数
  duration: '30s',   // 测试持续时间
};
export default function () {
  http.get('https://api.example.com/items');
}
```

k6 内置 `thresholds` 字段允许将 P95 < 400ms 作为测试通过/失败的门控条件，可直接嵌入 CI pipeline 中实现自动化质量卡点。

## 实际应用

**电商大促场景**：淘宝"双十一"前，性能团队会对支付接口执行逐步爬坡的压力测试，目标是验证系统能否支撑峰值 54.4 万笔/秒（2020 年双十一数据）。测试中会重点观察 Redis 连接数、MySQL 慢查询数量和消息队列积压深度。

**微服务链路测试**：在微服务架构中，单一接口的性能可能受 5 个下游服务影响。用 JMeter 的"分布式压测"功能（一个 Master 控制多个 Agent 节点）可以从多个地理位置同时发起请求，模拟真实的分布式流量，同时用 Zipkin 或 Jaeger 追踪哪个下游服务成为瓶颈。

**持续性能测试（CPT）**：将 k6 集成进 GitHub Actions，在每次合并主分支的 PR 后自动执行 30 秒负载测试，若 P95 响应时间相较上次基准值退化超过 20%，则自动阻断合并。这样可在代码引入性能退化的当天就发现问题，而非在上线后。

## 常见误区

**误区一：把并发线程数等同于真实并发用户数**。JMeter 中 1000 个线程并不代表模拟了 1000 个同时在使用应用的用户。真实用户存在"思考时间"（Think Time），两次点击之间可能间隔 3-10 秒。不设置思考时间的测试会产生远超实际的服务器压力，导致测试结果严重偏离真实场景。应在 JMeter 线程组中添加 `Constant Timer` 或使用 k6 的 `sleep()` 函数模拟用户停顿。

**误区二：仅测试响应时间均值（Mean）而忽略百分位数**。均值会被少量极长响应时间拉高，也会被批量成功请求拉低，掩盖真实的用户体验分布。一个接口 Mean = 100ms 但 P99 = 5000ms，意味着每 100 个请求中有 1 个用户等待 5 秒——这在功能上不可见，只有通过百分位数分析才能发现。

**误区三：在共享开发环境中执行压力测试**。如果压测目标与其他开发团队共享数据库或网络带宽，测试产生的高负载会干扰他人工作，而环境中的其他流量也会污染测试结果。性能测试必须在与生产环境规格一致的专用隔离环境中执行，并且确保测试数据集（数据库行数、缓存预热状态）与生产环境接近。

## 知识关联

性能测试与**单元测试**和**集成测试**形成完整的测试金字塔：单元测试在毫秒级验证单个函数逻辑，集成测试验证模块间协作，而性能测试在分钟级验证系统在真实负载下的整体行为。三者服务于不同的质量维度，性能测试无法替代功能测试，反之亦然。

在 TDD（测试驱动开发）的上下文中，性能测试可以延伸为**性能驱动开发**（PDD）：先用 k6 或 JMH 写出定义"通过"标准的性能测试脚本，再编写业务代码使其达标。这种方式避免了"先实现、再优化"时优化目标模糊的问题，将 P95 < 200ms 这样的具体指标变成可执行的验收条件。

学习性能测试后，自然衔接的技术方向包括：基于性能测试数据做**容量规划**（Capacity Planning，预测多少台服务器支撑目标 TPS）、**APM 工具**（如 Datadog、New Relic）在生产环境持续采集性能指标，以及**混沌工程**（Chaos Engineering）通过注入故障验证系统韧性——这些都依赖性能测试建立的基准数据作为对比参照。
