# 不稳定测试（Flaky Test）

## 概述

不稳定测试（Flaky Test）是指在代码和测试逻辑完全不变的情况下，同一个测试用例在不同运行时刻会交替产生"通过"和"失败"两种结果的测试。这种非确定性行为由 Google 工程师于 2016 年在论文《*Where Do Our Flaky Tests Come From?*》中被系统记录和分类，研究发现 Google 内部约有 1.5% 的测试用例表现出 Flaky 特征。

不稳定测试之所以危险，不在于它会导致明确的 Bug，而在于它会侵蚀开发团队对测试套件的信任。当工程师看到红色失败标志时，无法判断是真实的代码缺陷还是虚假的随机失败，从而倾向于忽略失败信号，最终让真正的问题蒙混过关。Netflix 内部统计显示，一支团队一旦开始习惯性重跑失败测试，真实缺陷的发现周期平均延长 3 倍。

不稳定测试与"真正失败的测试"的本质区别在于其概率性：一个真正失败的测试在相同环境下 100% 失败，而不稳定测试的失败率通常在 0.1% 到 30% 之间，这种低频率使得它极难被立即察觉和复现。

## 核心原理

### 时间与异步依赖

最常见的不稳定根源是测试代码对执行时序做出了隐式假设。例如，测试代码调用一个异步 API 后立即断言返回值，而没有等待 Promise 解决或回调触发。常见的错误写法是插入固定等待时间：`Thread.sleep(500)`，在开发机上 500 毫秒足够，但在 CI 服务器高负载时可能不够，导致测试随机失败。正确做法是使用显式等待机制，如 JavaScript 中的 `await`、Python 的 `asyncio.wait_for()`，或 Selenium 的 `WebDriverWait`。

### 测试间共享状态

当多个测试用例共用同一个数据库实例、全局单例对象或文件系统路径时，测试执行顺序的变化会造成结果不一致。例如，测试 A 向数据库插入一条记录，测试 B 断言数据库为空，若 A 在 B 之前执行则 B 失败，若顺序颠倒则 B 通过。JUnit 和 pytest 均不保证测试执行顺序，因此依赖顺序的测试天生是不稳定的。修复方案是在每个测试的 `setUp` / `tearDown` 钩子中彻底重置共享状态，或使用事务回滚将数据库恢复到初始状态（如 Django 的 `TestCase` 类默认行为）。

### 随机性与时间戳依赖

测试中直接调用 `Math.random()`、`uuid.uuid4()` 或 `datetime.now()` 会引入不可控变量。例如，一个验证"生成的 ID 不重复"的测试如果直接依赖真实随机数，在极低概率下会碰撞失败。解决方案是通过依赖注入将随机数生成器和时钟对象注入被测代码，测试时传入固定种子（seed）或 Mock 时钟对象（如 Python 的 `freezegun` 库或 Java 的 `Clock.fixed()`）。

### 外部资源与网络调用

测试直接依赖真实的第三方 API、数据库服务或文件系统时，网络延迟、服务限流或环境差异都会导致随机失败。单元测试中不应出现任何真实网络调用，应使用 Mock 框架（Mockito、unittest.mock）将外部依赖替换为受控的桩对象（Stub）。

## 实际应用

**诊断流程：** 当怀疑某个测试是 Flaky 时，标准做法是在 CI 环境中连续独立运行该测试 50 到 100 次（不依赖其他测试），统计失败率。Google 的工程实践将 Flaky 测试按失败率分为三级：低于 1% 为"低风险"、1%-10% 为"中风险"、超过 10% 为"高风险"，对高风险测试需立即隔离并标记 `@Flaky`，禁止其阻塞主干合并流水线。

**修复案例——异步问题：** 一个测试 React 组件的用例在点击按钮后断言弹窗是否出现，若弹窗通过动画延迟 300ms 显示，则 `expect(modal).toBeVisible()` 在快速机器上通过、在慢速 CI 上失败。修复方式是将断言改为 `await screen.findByRole('dialog')`，该函数内置最长 1000ms 的等待重试，彻底消除时序依赖。

**修复案例——状态隔离：** pytest 中，若测试函数 A 修改了一个模块级别的全局字典 `CONFIG`，测试函数 B 读取该字典时可能读到被污染的值。修复方式是在 `conftest.py` 中添加 `autouse=True` 的 fixture，在每次测试前深拷贝并恢复 `CONFIG` 的初始状态。

## 常见误区

**误区一：重新运行就能解决问题**
许多 CI 系统提供"失败自动重试"功能（如 GitHub Actions 的 `retry-on-failure`），部分团队将其作为处理不稳定测试的长期方案。这实际上只是掩盖问题：重试机制让流水线继续通过，但底层的不确定性根因依然存在，真实缺陷可能在重试中被掩盖。重试只应作为诊断期间的临时措施，最终必须修复根因。

**误区二：不稳定测试是测试代码的问题，与生产代码无关**
约 30% 的 Flaky Test 根因实际上暴露了生产代码中的竞态条件（Race Condition）或未处理的异常情况。例如，一个测试因为间歇性地触发了数据库死锁而随机失败，这意味着生产环境中同样的死锁在并发高峰期也会出现。因此，修复 Flaky Test 时应同时审查被测代码是否存在并发安全问题。

**误区三：Flaky Test 只出现在 UI 端到端测试中**
虽然 Selenium/Playwright 类的端到端测试因为网络和渲染延迟确实更容易出现 Flaky 问题，但单元测试同样可以是 Flaky 的。例如，一个断言"日志文件最后修改时间晚于测试开始时间"的单元测试，在文件系统时间戳精度为 1 秒的系统（如某些 Linux 版本）上，若测试执行快于 1 秒则失败。

## 知识关联

不稳定测试是测试驱动开发（TDD）实践中可靠性的直接障碍。TDD 要求开发者相信测试结果——绿色代表安全，红色代表缺陷——而 Flaky Test 破坏了这一前提。掌握不稳定测试的识别和修复是保证 TDD 红绿重构循环有效运转的基础技能。

在测试分层模型中，不稳定测试的发生率与测试层级高度相关：单元测试的 Flaky 率通常低于 0.5%，集成测试在 1%-5% 之间，UI 端到端测试可高达 5%-20%。这也是测试金字塔理论主张以大量单元测试为底层的工程原因之一。

与测试覆盖率（Code Coverage）相比，Flaky Rate（不稳定率）是衡量测试套件健康度的另一个关键指标。一个覆盖率 95% 但 Flaky 率 10% 的测试套件，其实际价值远低于覆盖率 80% 且 Flaky 率 0.1% 的测试套件，因为前者无法提供可信赖的安全网。
