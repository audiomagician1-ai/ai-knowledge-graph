# 测试与重构

## 概述

测试与重构是一种在软件工程中通过测试用例保障代码改进安全性的实践策略。其核心思想是：**在修改代码内部结构的同时，确保外部可观测行为完全不变**，而测试套件就是验证"行为不变"的防护网。没有测试保障的重构，本质上是盲目猜测代码修改是否正确，引入新缺陷的风险极高。

这一实践思想最早由 Martin Fowler 在 1999 年出版的《重构：改善既有代码的设计》（*Refactoring: Improving the Design of Existing Code*）中系统化阐述。书中明确指出，重构的先决条件是"拥有一套可靠的测试"，否则重构行为本身就缺乏依据。Kent Beck 随后在测试驱动开发（TDD）中将"先写测试、再重构"固化为开发流程的标准步骤，即著名的"红-绿-重构"循环。

测试之所以是安全重构的保障，是因为它为每一次代码变更提供了即时的正确性反馈。当开发者提取一个方法、合并两个类、或内联一个变量时，只需在修改后立即运行测试，若所有测试通过，则可以有信心地确认重构未破坏已有功能。这将"我猜这样改不会出问题"变成了"测试证明这样改没有问题"。

## 核心原理

### 测试作为行为规格说明

重构时，测试用例扮演的角色不是"找 bug"，而是**锁定当前代码的外部行为契约**。每一条断言（assertion）都精确描述了"输入 X 必须产生输出 Y"。例如，若某个计税函数 `calculateTax(income)` 在收入为 50000 时返回 5000，那么无论内部实现经过多少次重构——提取辅助方法、引入策略模式、消除重复逻辑——测试 `assertEqual(calculateTax(50000), 5000)` 必须始终通过。这条测试就是对"税率 10%"这一业务规则的机器可验证的规格说明。

### 小步提交与测试频率

安全重构的关键操作模式是**极小步幅修改，极高频率运行测试**。Martin Fowler 建议每次重构操作后都运行一次完整测试套件，而不是积累多个修改后才运行。在实践中，一次重构步骤通常应控制在 5 分钟以内，单步修改的代码行数一般不超过 20 行。一旦某次修改后测试失败，由于步幅极小，开发者可以立即使用 `Ctrl+Z` 撤销，回到上一个已知正确的状态，而非花费大量时间排查错误原因。这一策略将"调试时间"近乎压缩至零。

### 测试覆盖率与重构安全区

并非所有代码在任何时候都可以安全重构，**测试覆盖率**决定了重构的"安全区边界"。对于被测试覆盖的代码路径，重构是有安全保障的；对于未被任何测试执行到的代码分支，重构是无保障的猜测行为。一个实用的判断标准是：若语句覆盖率（Statement Coverage）低于 80%，则应在重构前先补充测试，专门针对即将修改的代码路径。工具如 JaCoCo（Java）或 Istanbul（JavaScript）可以可视化地展示哪些代码行未被测试覆盖，帮助开发者精确识别重构前需要补测的区域。

### 特性测试（Characterization Test）策略

当面对遗留代码——即没有测试保障的旧代码——时，无法直接安全重构。此时的标准做法是先编写**特性测试**（Characterization Test，由 Michael Feathers 在《修改代码的艺术》中提出）：不是测试"代码应该做什么"，而是测试"代码实际做了什么"。具体步骤是：调用目标函数，将其当前实际输出（无论是否符合预期）写入断言，从而将当前行为"钉住"。只要特性测试通过，就说明重构没有改变现有行为，即便原始行为本身可能存在缺陷。

## 实际应用

**场景：重构电商系统中的订单折扣逻辑**

假设有一段包含多层嵌套 `if-else` 的折扣计算函数，开发者希望将其重构为策略模式。在开始重构前，需要检查现有测试是否覆盖了所有折扣条件：VIP 折扣、满减折扣、节日折扣的独立情况，以及三种折扣叠加的组合情况。若覆盖不足，则先补充 6 个针对边界条件的测试用例，确保每条代码路径至少被一个测试执行。此后，每次仅提取一个折扣策略类，立即运行测试，确认绿色后再提取下一个。整个重构过程可能包含 8 到 12 次单独的"修改-测试"循环，但每次循环的风险窗口都极小。

**场景：在持续集成（CI）流水线中保障重构安全**

在团队开发中，开发者提交重构代码到 Git 后，CI 服务器（如 GitHub Actions 或 Jenkins）会自动运行全量测试。若测试失败，构建立即被标红，提交被阻止合并进主干。这意味着测试套件不仅保护个人开发，还保护整个团队的代码库不受错误重构的污染。

## 常见误区

**误区一：重构完成后再统一运行测试**

很多开发者习惯完成所有重构改动后才运行一次测试。这个做法削弱了测试的防护作用。若此时测试失败，由于中间经历了多次代码修改，定位失败原因需要排查整个修改历史，恢复成本很高。正确做法是每完成一个原子性重构操作（如单次"提取方法"）后立即运行测试，将错误定位精度锁定在单步操作内。

**误区二：高测试覆盖率等于可以安全重构任何代码**

测试覆盖率衡量的是代码行是否被执行过，但不能保证所有边界条件都被验证。一段代码的语句覆盖率为 100%，仍然可能缺少对空值输入、整数溢出或并发竞争条件的测试。对于涉及这些边界情况的代码，即使覆盖率显示绿色，重构前仍需额外审查断言是否足够严格。

**误区三：测试本身不需要被维护**

当重构导致函数签名改变（如方法名称调整、参数合并）时，部分测试需要同步更新。这不是"破坏测试"，而是测试代码与生产代码同步演进的正常现象。若因为"修改测试太麻烦"而删除测试，则安全重构的保障网随之消失。测试代码同样是软件资产，需要与生产代码一同维护。
