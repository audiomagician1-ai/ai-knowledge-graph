# 测试覆盖率

## 概述

测试覆盖率（Test Coverage）是衡量测试用例执行了多少被测代码的量化指标，通常以百分比表示。当测试套件运行时，覆盖率工具（如 JaCoCo、Istanbul、pytest-cov）会追踪哪些代码行、分支或路径被实际执行，从而生成覆盖率报告。这个数字告诉开发者：**有哪些代码根本没有被任何测试触及**。

覆盖率分析的雏形可以追溯到1963年 Joan Miller 和 Clifford Rabin 在程序测试领域的早期研究。1976年，Tom McCabe 在提出圈复杂度（Cyclomatic Complexity）时进一步奠定了路径覆盖的理论基础。现代 TDD 工作流将覆盖率作为"测试健全性"的参考信号之一，许多团队设置 CI 流水线门禁，要求覆盖率不得低于 80% 才允许合并代码。

覆盖率之所以在 TDD 中受到关注，是因为它能暴露**测试遗漏的死角**——比如某个异常处理分支或边界条件从未被测试用例触发。但覆盖率本质上只测量"代码被执行了"，并不保证测试断言是否正确，因此它既是有用的诊断工具，也是容易被误用的指标。

## 核心原理

### 语句覆盖（行覆盖）

语句覆盖率 = 被执行的语句数 ÷ 可执行语句总数 × 100%。这是最粗粒度的覆盖指标。考虑以下代码：

```python
def divide(a, b):
    if b != 0:          # 行1
        return a / b    # 行2
    return None         # 行3
```

若只有测试 `divide(10, 2)`，则行1、行2被执行，行3未执行，语句覆盖率为 2/3 ≈ 67%。语句覆盖的致命弱点是：即使达到 100%，也无法保证所有条件分支都被测试到——因为一条语句只需被执行一次即可计入覆盖。

### 分支覆盖（判定覆盖）

分支覆盖要求每个判断条件的"真"和"假"出口都必须被至少一个测试用例走到。分支覆盖率 = 被执行的分支数 ÷ 分支总数 × 100%。在上面的 `divide` 函数中，`if b != 0` 有两个分支：`b != 0` 为真和为假。若只测试 `divide(10, 2)`，分支覆盖率仅为 1/2 = 50%，即便语句覆盖率是 67%。需要增加 `divide(10, 0)` 测试用例，才能将分支覆盖率提升至 100%。**分支覆盖率 ≥ 语句覆盖率**，这是二者之间必然成立的不等式。

### 条件覆盖与 MC/DC 覆盖

当分支条件由多个子条件组成（如 `if a > 0 and b < 10`），分支覆盖仍然不够。**条件覆盖**要求每个子条件的真/假都被单独测试。更严格的是 **MC/DC（修正条件/判定覆盖）**，航空软件标准 DO-178C 规定 A 级（最高安全等级）软件必须满足 MC/DC。MC/DC 要求：每个子条件必须独立影响整体判定结果，而不是被其他条件的短路运算掩盖。对 `n` 个子条件的复合条件，MC/DC 最少只需 `n+1` 个测试用例，而全条件覆盖最坏需要 `2^n` 个。

### 路径覆盖

路径覆盖要求程序中从入口到出口的每条完整执行路径都被测试。若函数有 3 个连续的独立 `if` 语句，理论路径数为 `2³ = 8`。路径覆盖是覆盖强度最高的指标，但在含循环的代码中，路径数可能趋向无穷大，工程上往往只追求**基本路径覆盖**（基于圈复杂度计算的最小路径集合）。

## 实际应用

**在 TDD 红-绿-重构循环中使用覆盖率：** 编写失败测试（红）→ 实现代码使测试通过（绿）→ 运行 `pytest --cov=src --cov-report=html` 查看覆盖报告 → 识别未覆盖分支 → 补写针对该分支的测试。这种用法将覆盖率作为"发现下一个要写的测试"的导航工具，而非考核指标。

**CI 流水线门禁配置：** 在 GitHub Actions 中结合 `codecov` 服务，可配置当覆盖率相比上一次 PR 下降超过 2% 时自动阻断合并。这种**增量覆盖率**策略比强制全局 80% 更合理，因为它针对本次变更的新代码施压，而非让历史遗留代码拖累整体数字。

**前端 JavaScript 项目：** Istanbul（现为 nyc）将覆盖率拆分为四个维度同时报告：语句（Statements）、分支（Branches）、函数（Functions）、行（Lines）。实际项目中常见分支覆盖率显著低于语句覆盖率的情况，因为三元运算符 `condition ? a : b` 的"假"分支极易被忽略。

## 常见误区

**误区一：100% 覆盖率等于测试充分。** 下面这段代码可以达到 100% 行覆盖率，但测试形同虚设：

```python
def add(a, b):
    return a + b

def test_add():
    add(1, 2)  # 执行了代码，但没有任何 assert！
```

覆盖率工具只记录代码是否被执行，不关心测试是否包含有效断言。这正是**变异测试**（Mutation Testing）出现的原因——它会修改源代码制造"变异体"，然后检验测试是否能杀死该变异体，从而评估断言质量。

**误区二：覆盖率越高越好，应追求 100%。** 对返回固定常量的 getter 方法、第三方库的适配器包装代码、以及操作系统交互层的代码强行写测试，往往收益极低而维护成本极高。Google 内部工程实践建议：60% 是可接受的下限，75% 是良好水平，90% 以上应仔细评估是否有测试价值，而非机械地堆砌覆盖率。

**误区三：分支覆盖率 100% 意味着路径覆盖率 100%。** 两个连续的 `if` 语句各自达到分支覆盖（真+假），只需要 2 个测试用例；但路径覆盖需要覆盖 4 条路径（真-真、真-假、假-真、假-假）。分支覆盖无法捕获路径之间的交互缺陷，例如第一个条件为假且第二个条件也为假时才会触发的特定 bug。

## 知识关联

**依赖的前置概念：** 单元测试为覆盖率提供被测对象——覆盖率工具本质上是统计单元测试套件执行时触及了多少代码。测试替身（Test Double）的使用直接影响覆盖率数字的真实性：用 Mock 替换了数据库调用后，数据库访问层的代码不会被执行，因此其覆盖率不会被当前测试套件统计到，需要集成测试来补充。

**延伸到后续概念：** 覆盖率指标的局限性（高覆盖率不等于高质量断言）直接催生了**变异测试**的需求——变异测试可以理解为"测试测试的质量"，是覆盖率的升级替代指标。在**行为驱动开发（BDD）** 中，覆盖率的关注点从代码行转移到业务场景（Scenario）的覆盖上：Cucumber 的 Feature 文件中有多少业务规则被 Step Definition 实际验证，形成了"场景覆盖率"的概念，与代码级覆盖率互补使用。
