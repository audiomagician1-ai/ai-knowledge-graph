# 测试覆盖率

## 概述

测试覆盖率（Test Coverage）是衡量测试套件执行了被测代码中多少比例的一种定量指标。其数学表达式为：**覆盖率 = 已执行的代码单元数 ÷ 总代码单元数 × 100%**。这里的"代码单元"根据具体覆盖类型不同，可以是行、分支、条件或路径。

测试覆盖率的概念最早由 Joan C. Miller 和 Clifford J. Kause 于 1963 年提出，并在 1970 年代随着结构化测试理论的发展逐渐成形。Paul Ammann 和 Jeff Offutt 在其经典教材《软件测试导论》中将覆盖率准则系统化为一套层次结构，从最弱的语句覆盖到最强的 MC/DC（修正条件/判定覆盖）覆盖形成完整谱系。

覆盖率之所以受到重视，是因为它能在不读取全部代码的前提下，快速定位测试盲区——即从未被任何测试用例执行到的代码段。然而覆盖率是必要条件而非充分条件：100% 行覆盖率不等于测试充分，这一陷阱是本文的核心讨论点之一。

## 核心原理

### 行覆盖（Statement Coverage / Line Coverage）

行覆盖率统计测试运行后被执行的**可执行语句行数**占全部可执行语句的比例。例如一个函数共 20 行可执行语句，测试用例触发了其中 16 行，则行覆盖率为 80%。

行覆盖是最弱的覆盖标准。考虑如下代码：

```python
if x > 0:
    result = x * 2   # 行 A
else:
    result = -x      # 行 B
```

若测试用例仅传入 `x = 5`，则行 A 被覆盖，行 B 未被覆盖，行覆盖率仅 50%。但即使同时覆盖 A 和 B，也无法保证所有布尔组合都被测试到。

### 分支覆盖（Branch Coverage / Decision Coverage）

分支覆盖要求每个判断语句的**真分支和假分支都至少被执行一次**。对于一个含 `if-else` 的函数，分支覆盖率 = 已覆盖分支数 ÷ 总分支数 × 100%。

分支覆盖比行覆盖更严格：达到 100% 分支覆盖必然达到 100% 行覆盖，反之不成立。对于含有复合条件 `if (A && B)` 的判断，仅测试整体判断的真假两个出口，**不足以**验证 A、B 各自独立失效的情况。这正是 MC/DC 覆盖被 DO-178B（航空软件标准）强制要求的原因。

### 路径覆盖（Path Coverage）

路径覆盖要求程序中每一条**从入口到出口的完整执行路径**都被测试。一个含 n 个独立判断节点的函数，理论上最多存在 2ⁿ 条路径。当 n = 10 时，路径数高达 1024 条，含循环时路径数趋向无穷，导致完全路径覆盖在实践中不可行。

因此实践中常以**基本路径覆盖**替代：通过计算程序的**圈复杂度（Cyclomatic Complexity）V(G) = E - N + 2P**（E 为边数，N 为节点数，P 为连通分量数）确定线性独立路径数，该数值即为基本路径覆盖所需的最少测试用例数。

### MC/DC 覆盖（Modified Condition/Decision Coverage）

MC/DC 要求对复合布尔条件中的**每个子条件独立影响整体判断结果**的情形都有测试用例覆盖。对于 `if (A && B && C)`，需要证明每个变量在其他变量不变的情况下，其真假值变化能单独改变最终判断结果。DO-178B/DO-178C 标准对 A 级（最高安全等级）航空软件强制要求 MC/DC 覆盖，这是测试覆盖率在工程标准中最具代表性的规范化应用。

## 实际应用

**工具集成与阈值设定**：主流语言均有成熟覆盖率工具，如 Python 的 `coverage.py`、Java 的 JaCoCo、JavaScript 的 Istanbul（nyc）。在 CI/CD 流水线中，常见做法是设置覆盖率下限（如 Jest 配置 `coverageThreshold: { global: { branches: 80 } }`），低于阈值则构建失败。

**TDD 中的覆盖率用法**：在测试驱动开发流程中，覆盖率报告主要用于**发现遗漏的测试场景**而非追求数字达标。开发者在红-绿-重构循环后运行覆盖率检查，若某个分支未覆盖，则说明该分支对应的业务规则尚未被测试用例描述，需补充测试而非直接删除分支代码。

**差异覆盖率（Diff Coverage）**：大型项目中追求整体覆盖率提升成本极高，Google 等公司改为对**每次提交的新增代码**要求不低于 60%（部分团队要求 80%）的行覆盖率，仅关注增量变化，避免历史遗留代码拉低整体指标。

## 常见误区

**误区一：100% 行覆盖率等于测试充分**

这是最危险的误解。考虑函数 `def divide(a, b): return a / b`，一个测试用例 `divide(4, 2)` 即可达到 100% 行覆盖率，但完全未测试 `b = 0` 时的除零异常。行覆盖率无法检测**缺失的边界条件**，它只验证代码被执行过，不验证执行结果是否正确。

**误区二：提高覆盖率就是写更多断言**

实际上，在 Python 中只需写 `process(data)` 而不加任何 `assert`，覆盖率照样上升。无断言的测试用例可以人为刷高覆盖数字，却对捕获 Bug 毫无贡献。这也是为何仅凭覆盖率无法度量测试质量，需要结合**变异测试**来验证断言的有效性。

**误区三：分支覆盖包含了所有条件组合**

对于 `if (user.isAdmin() || user.hasPermission("write"))`，100% 分支覆盖仅需两个测试用例（条件整体为真、整体为假），但遗漏了"isAdmin 为假且 hasPermission 为真"这一路径分支的独立验证。条件覆盖（Condition Coverage）和分支覆盖是不同层次的指标，不可混淆。

## 知识关联

**前置概念——单元测试**：覆盖率数据来源于单元测试的执行，没有可运行的单元测试套件，覆盖率工具无从采集执行轨迹。行覆盖和分支覆盖是评估现有单元测试完整性的直接手段，补测时应优先针对 `if`/`switch` 的未覆盖分支而非随机增加测试用例。

**后续概念——变异测试**：测试覆盖率回答"代码被执行了吗"，而变异测试（Mutation Testing）回答"测试能检测到代码错误吗"。工具（如 PIT for Java、mutmut for Python）通过自动将 `>` 改为 `>=`、删除返回语句等方式制造变异体，统计测试套件杀死变异体的比例（Mutation Score）。高覆盖率配合低 Mutation Score 正好印证了"覆盖率陷阱"——测试执行了代码但未有效验证其行为。两者结合才能构成对测试充分性的立体评估。
