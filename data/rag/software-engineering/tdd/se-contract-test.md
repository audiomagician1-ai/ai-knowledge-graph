# 契约测试

## 概述

契约测试（Contract Testing）是一种验证服务间接口兼容性的测试方法，专门解决分布式系统中消费者与提供者之间的API版本漂移问题。与单元测试或集成测试不同，契约测试不测试服务的业务逻辑，而是测试服务边界处的数据格式、字段类型和HTTP状态码是否符合双方的约定。

契约测试的现代形态由澳大利亚公司DiUS于2013年前后提出，并随着开源工具Pact的发布而普及。Pact最初仅支持Ruby语言，但截至2024年已扩展支持Java、Go、Python、JavaScript等12种以上语言，形成了Pact Foundation这一跨语言生态。消费者驱动契约（Consumer-Driven Contract，CDC）的核心创新在于将契约的定义权交给消费者而非提供者，打破了传统API文档中提供者单方面决定接口规范的模式。

在微服务架构盛行的背景下，一个系统可能有数十个微服务相互调用，任何一个服务的API变更都可能导致下游服务崩溃。契约测试通过在CI/CD流水线中自动验证契约一致性，将API不兼容问题从生产环境拦截到代码合并阶段，这是传统端到端集成测试无法高效实现的能力。

## 核心原理

### 消费者驱动契约的生成机制

在Pact框架中，契约测试分为消费者端测试和提供者端验证两个独立阶段。消费者端测试运行时，Pact启动一个本地Mock服务器（默认端口为8080），消费者代码向该Mock服务器发送真实HTTP请求。Pact将这次交互——包括请求的HTTP方法、路径、请求头、请求体以及预期响应——序列化为一个JSON格式的契约文件，称为Pact文件（`.pact` 文件）。

Pact文件的结构包含`consumer`、`provider`两个顶层字段和`interactions`数组。每个interaction描述一次具体的请求-响应对，例如：
```json
{
  "description": "获取用户信息",
  "request": { "method": "GET", "path": "/users/1" },
  "response": { "status": 200, "body": { "id": 1, "name": "张三" } }
}
```
消费者单方面生成此文件，然后通过Pact Broker（契约仓库服务）发布，供提供者拉取验证。

### 提供者端的回放验证

提供者端验证阶段，Pact框架从Pact Broker拉取契约文件，逐条将其中的请求向提供者真实服务重放，并比对实际响应与契约中记录的预期响应。验证规则采用宽松匹配而非严格相等：响应体中的字段类型必须匹配（整数对整数、字符串对字符串），但具体值可以不同。这意味着提供者可以返回`{"id": 42}`而非`{"id": 1}`，只要字段类型一致即视为通过。Pact将这种策略称为"Flexible Matching"，通过`matchingRules`字段在契约文件中显式声明匹配规则类型。

### 契约版本控制与Can-I-Deploy检查

Pact Broker提供`can-i-deploy`命令行工具，这是契约测试与CI/CD集成的关键节点。在任何服务发布到生产环境之前，CI流水线执行`pact-broker can-i-deploy --pacticipant UserService --version 1.2.3 --to production`，系统查询所有与该服务相关的契约验证记录，只有当所有相关消费者与提供者的最新契约均处于已验证（verified）状态时，该命令才返回成功。Pact Broker通过矩阵视图（Matrix View）展示各版本服务与各消费者之间的兼容性状态，支持在多版本并存的过渡期内精确追踪兼容性依赖关系。

## 实际应用

**电商平台订单与库存服务解耦**：订单服务（消费者）依赖库存服务（提供者）的`GET /inventory/{skuId}`接口。订单团队在消费者测试中定义契约：期望响应包含`available`（布尔型）和`quantity`（整数型）字段。库存团队在重构服务时，即便修改了内部数据库结构，只需在每次PR合并时通过提供者验证，就能保证不破坏订单服务的依赖。

**前后端分离项目**：前端JavaScript应用作为消费者，后端REST API作为提供者。前端团队使用Pact JS库（`@pact-foundation/pact`包）编写消费者测试，生成Pact文件后上传至Pact Broker。后端团队每次发布前运行`mvn pact:verify`（Java Spring项目）验证兼容性，将API变更导致前端崩溃的风险降为零。

**多消费者场景**：当同一个提供者服务被移动端App（消费者A）和Web端（消费者B）同时依赖时，提供者需要同时满足两份来自不同消费者的契约。Pact Broker聚合两份契约，提供者一次性验证所有交互，若移动端需要某字段而Web端不需要，两者的Pact文件独立存在互不影响，比共享一份OpenAPI文档更精确地反映了真实的依赖关系。

## 常见误区

**误区一：将契约测试等同于API文档验证**。部分团队尝试用Swagger/OpenAPI规范来替代契约测试，但OpenAPI描述的是提供者的完整能力，而Pact契约只描述特定消费者实际使用的那部分接口。一个提供者可能提供20个字段，但某消费者的契约只断言其中3个字段，这种精确的"最小化依赖"描述是OpenAPI无法表达的。

**误区二：认为契约测试可以替代集成测试**。契约测试验证的是接口格式兼容性，不验证业务逻辑正确性。例如，库存服务可能在格式上完全符合契约（返回了`available: true`），但实际库存数量计算有Bug导致超卖。这类问题需要集成测试或端到端测试发现，契约测试无法覆盖。

**误区三：提供者团队主导契约内容**。传统思维中，API的提供者负责定义接口规范，消费者被动适配。CDC模式反转了这一关系——消费者用代码写下自己的期望，提供者必须满足所有已注册消费者的契约集合。如果提供者团队试图修改消费者的Pact文件来让验证通过，这不仅违反了CDC原则，也会导致真实的兼容性问题被掩盖。

## 知识关联

契约测试与**变异测试**形成互补：变异测试通过向代码注入故障（如翻转布尔值、修改比较运算符）来评估测试套件的检出能力，而契约测试关注的是服务边界的数据契约而非代码内部逻辑。两者都属于测试充分性验证的范畴，但作用层次不同——变异测试作用于单元测试层，契约测试作用于服务集成层。在测试策略中，先通过变异测试确保消费者代码的单元测试足够严格，再通过契约测试确保跨服务接口的格式兼容性，两层防护共同减少缺陷逃逸率。

契约测试是进入**性能测试**阶段的重要前提。性能测试需要对真实服务发起大量并发请求，若API接口存在版本不兼容问题，性能测试结果将因频繁的5xx错误而失真。通过契约测试确认所有服务接口格式兼容后，再执行压力测试或负载测试，才能获得反映系统真实吞吐能力的基准数据。从测试金字塔的维度看，契约测试处于单元测试与端到端测试之间，用极低的运行成本（无需启动完整系统）解决了跨服务兼容性验证这一具体问题。
