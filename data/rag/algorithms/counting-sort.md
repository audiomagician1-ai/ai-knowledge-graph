# 计数排序

## 概述

计数排序（Counting Sort）是一种非比较型整数排序算法，其核心思想是统计每个元素出现的次数，然后利用累计计数来确定每个元素在输出数组中的精确位置。与快速排序、归并排序等基于元素比较的算法不同，计数排序完全绕过了比较操作，因此能突破比较排序的理论下界 O(n log n)，在特定条件下实现线性时间复杂度 O(n + k)，其中 n 是待排序元素数量，k 是元素取值范围的大小。

计数排序由 Harold H. Seward 于 1954 年在其技术报告中首次提出，距今已有七十年历史。它在早期的穿孔卡片排序机时代就得到了广泛运用，因为那时的数据往往是有限范围内的整数编码。在现代 AI 工程场景中，计数排序常被用于对离散类别标签、词频统计结果或固定范围内的特征值进行高效排序，例如对取值范围在 0 到 255 之间的像素灰度值数组进行排序时，计数排序往往是最优选择。

计数排序的重要价值在于它对"数据分布"的敏感性。当 k 远小于 n 时（如对 100 万个 0~9 之间的整数排序），计数排序的性能远超任何比较排序。但当 k 接近甚至超过 n² 时，额外的空间开销会使它变得不实用。理解这一权衡，是在 AI 工程中选用合适排序策略的关键判断依据。

## 核心原理

### 三步执行流程

计数排序的完整执行分为三个步骤：

**第一步：统计频次。** 创建一个长度为 k+1 的计数数组 `count[]`，遍历原始数组，将每个值 v 出现的次数记录在 `count[v]` 中。例如原始数组为 `[4, 2, 2, 8, 3, 3, 1]`，则 `count[1]=1, count[2]=2, count[3]=2, count[4]=1, count[8]=1`，其余为 0。

**第二步：累计求和（前缀和变换）。** 将计数数组原地转换为前缀和数组，使 `count[i]` 表示"值 ≤ i 的元素共有多少个"。变换公式为：
```
count[i] = count[i] + count[i-1]，对 i 从 1 到 k
```
这一步决定了每个元素在最终有序数组中的"最后一个合法位置"索引。

**第三步：反向填充输出数组。** 从右到左（逆序）遍历原始数组，对每个元素 v，将其放置到输出数组的 `count[v]-1` 位置，然后将 `count[v]` 减 1。逆序遍历是实现稳定性的关键：相同值的元素会保持原始相对顺序。

### 时间与空间复杂度

计数排序的时间复杂度为 **O(n + k)**：第一步遍历 n 个元素，第二步遍历 k 个桶，第三步再遍历 n 个元素，总计 2n + k 次操作。空间复杂度为 **O(n + k)**：输出数组占用 n 个空间，计数数组占用 k+1 个空间。

当 k = O(n) 时，整体复杂度退化为 O(n)，是线性排序；当 k = O(n²) 时，空间开销达到 O(n²) 级别，已不再实用。因此计数排序的适用黄金条件是：**整数类型 + 取值范围 k 远小于元素数量 n**。

### 稳定性的内在机制

计数排序是**稳定排序**，这一特性不是偶然的，而是由"逆序遍历 + 计数递减"的设计刻意保证的。如果将第三步改为正向遍历，则相同值的元素顺序会颠倒，稳定性丧失。这一稳定性使计数排序可以作为基数排序（Radix Sort）的内层子程序——基数排序按位从低到高多轮调用计数排序，若内层不稳定，则高位排序结果会被低位操作破坏，导致最终结果错误。

## 实际应用

### AI 特征预处理中的词频排名

在自然语言处理中，词汇表大小（vocabulary size）通常在 10,000 到 100,000 之间，而训练语料中词元（token）数量往往达到数亿级别。对词元 ID（取值 0 到 vocab_size-1 的整数）序列进行排序或词频统计时，计数排序的 k 远小于 n，能以线性时间完成，显著优于 O(n log n) 的比较排序。

### 图像处理中的灰度直方图均衡

计算机视觉中的直方图均衡化算法，需要先统计每个灰度级（0~255，共 256 个取值）的像素数量，再计算累积分布函数（CDF），这正是计数排序第一步和第二步的直接应用。对一张 1920×1080 的图像（约 200 万像素），统计步骤的时间复杂度为 O(2,000,000 + 256)，远快于任何基于比较的方案。

### 基数排序的内层组件

当需要对取值范围很大的整数（如 32 位整数，k 高达 4×10⁹）排序时，直接应用计数排序不可行。基数排序的解决方案是：将每个整数按 8 位一组拆分，每组取值范围为 0~255（k=256），然后对每组分别调用计数排序（共调用 4 次）。整体时间复杂度为 O(4 × (n + 256)) = O(n)，计数排序的稳定性在这里是算法正确性的前提。

## 常见误区

### 误区一：认为计数排序可以处理任意数据类型

计数排序**只能对非负整数**（或可映射为非负整数的数据）有效。它不能直接对浮点数、字符串或负整数排序（负整数需要做偏移：将所有元素加上 |min| 转换为非负数）。若原始数组中存在 -5，则必须先确定数组的最小值 min，将每个元素偏移 -min 后再建立计数数组，否则索引为负数会导致数组越界错误。许多初学者会忽略处理负数的偏移步骤。

### 误区二：认为线性时间意味着计数排序总是最快

时间复杂度 O(n + k) 中的 k 不可忽视。当对 1000 个取值范围在 0 到 10⁸ 之间的整数排序时，计数数组需要分配 10⁸ 个整数的内存（约 400MB），不仅空间不可接受，初始化这 10⁸ 个元素本身就比排序 1000 个元素慢得多。此时快速排序（O(n log n) ≈ 10,000 次操作）远比计数排序实用。

### 误区三：省略前缀和步骤，直接按计数展开输出

部分初学者为了简化实现，采用"直接展开"写法：看到 `count[v]=3`，就在输出中连续写入三个 v。这种写法**会破坏稳定性**，因为它丢失了原始数组中元素的附加信息（如对象的其他字段）。在只需对纯整数排序的场景下，直接展开结果正确，但当计数排序用于对携带卫星数据的记录排序时（如基数排序中的应用），必须使用完整的三步流程以保持元素间的相对顺序。

## 知识关联

**前置知识——数组/列表：** 计数排序的三个步骤均以数组的随机访问（O(1) 索引）为基础。计数数组 `count[]` 的随机读写依赖数组结构；若底层数据结构改为链表，则无法在 O(1) 内完成"将第 k 个位置的元素取出"，计数排序的线性时间保证随即失效。

**向后延伸——基数排序（Radix Sort）：** 计数排序是基数排序的子程序。基数排序将大范围整数的排序问题分解为多轮小范围（通常 k=256）的计数排序。理解计数排序的稳定性机制和前缀和索引分配逻辑，是正确实现基数排序中"按位分配"步骤的必要前提。

**横向对比——桶排序（Bucket Sort）：** 二者同属非比较排序，但计数排序的每个"桶"只存储同一个确定整数值，而桶排序的每个桶覆盖一个数值范围且桶内需二次排序。计数排序可视为桶大小为 1 的极端特例。在 AI 特征工程中选择排序算法时，数据是否为整数、取值范围 k 与 n 的比值，是决定选用计数排序还是桶排序的核心判断依据。
