# 滑动窗口

## 概述

滑动窗口（Sliding Window）是一种通过维护一个可变或固定大小的"窗口"区间在线性数据结构上移动，从而将原本需要 O(n²) 甚至更高复杂度的嵌套遍历优化为 O(n) 单次遍历的算法技巧。窗口由左指针 `left` 和右指针 `right` 共同界定，区间通常表示为 `[left, right]`，窗口内的元素即为当前处理的子数组或子字符串。

滑动窗口思想最早在计算机网络 TCP 协议的流量控制中得到工程化应用（TCP 滑动窗口协议于 1981 年在 RFC 793 中正式规范），随后被算法研究者抽象为通用的数组/字符串处理范式。在 LeetCode 题库中，滑动窗口专题包含超过 60 道经典题目，是面试高频考点之一。

该技巧之所以重要，在于它精确刻画了一类具有"单调性"的子区间问题：当窗口右边界扩张时，问题答案朝某个方向变化；当左边界收缩时，答案朝相反方向变化。只有满足此单调性约束的问题，才能正确应用滑动窗口，否则会得出错误结论。

## 核心原理

### 固定窗口与可变窗口

滑动窗口分为两种形态：

- **固定大小窗口**：窗口长度 `k` 预先给定，每次右移一步时同步移除最左元素，新增最右元素。例如求长度为 k 的所有子数组的最大和，时间复杂度从暴力 O(nk) 降为 O(n)。
- **可变大小窗口**：窗口长度随条件动态伸缩，`right` 不断向右扩张直到违反约束，随后 `left` 向右收缩直到重新满足约束。这类窗口用于求满足某条件的最长或最短子区间。

### 双指针协同移动与复杂度保证

滑动窗口的 O(n) 复杂度依赖一个关键不变量：**`left` 和 `right` 指针均只向右移动，不会回退**。在整个遍历过程中，`right` 从 0 走到 n-1 恰好一次，`left` 从 0 走到 n-1 至多一次，因此总操作次数上界为 2n，即 O(n)。若在收缩阶段允许 `left` 向左回退，则复杂度保证立即失效。

形式化地，设数组长度为 n，窗口内维护状态的更新代价为 O(1)（如哈希表计数、前缀和等），则总时间复杂度为：

$$T(n) = O(n_{\text{right}}) + O(n_{\text{left}}) = O(n) + O(n) = O(n)$$

其中 $n_{\text{right}}$ 和 $n_{\text{left}}$ 分别表示右指针和左指针的总移动步数。

### 窗口内状态的维护

滑动窗口的效率还依赖于对窗口内部状态的**增量更新**，而非每次重新计算。常见的维护方式包括：

- **哈希表频率计数**：用于"不含重复字符的最长子串"（LeetCode 3）等问题，每次扩张/收缩时对单个字符执行 `count[ch]++` 或 `count[ch]--`，代价 O(1)。
- **前缀和差分**：固定窗口求子数组和时，新窗口和 = 旧窗口和 + `arr[right]` - `arr[left-1]`，无需重新累加。
- **满足条件的计数器**：在"最小覆盖子串"（LeetCode 76）中，维护一个整数 `formed` 表示当前窗口已满足的字符种数，避免每次遍历整个频率表，将内层判断从 O(Σ) 降为 O(1)，其中 Σ 为字符集大小（英文字母为 26）。

## 实际应用

### 最长无重复字符子串（LeetCode 3）

给定字符串 `s`，求不含重复字符的最长子串长度。使用可变窗口：`right` 每次纳入新字符 `s[right]` 并写入哈希表；若哈希表中该字符频率超过 1，则持续右移 `left` 直到频率恢复为 1。每步更新答案 `ans = max(ans, right - left + 1)`。整体复杂度 O(n)，空间复杂度 O(min(n, Σ))。

### 大小为 k 的滑动窗口平均值

在流式数据处理或时序特征工程中，常需计算每个时刻前 k 个样本的均值（移动平均 MA-k）。使用固定窗口维护一个滚动累加和 `window_sum`，每步执行 `window_sum += arr[i] - arr[i-k]`，输出 `window_sum / k`。这是金融技术指标（如 MA5、MA20）和信号平滑的基础实现。

### 最小覆盖子串（LeetCode 76）

给定字符串 `s` 和模式串 `t`，求 `s` 中包含 `t` 所有字符的最短子串。此问题同时维护两个频率哈希表和一个 `formed` 计数器，当 `formed == len(t的字符种数)` 时记录当前窗口并收缩左边界。该题是滑动窗口与哈希表结合的最典型范例，答案子串的起始位置和长度需在每次满足条件时更新。

## 常见误区

### 误区一：对所有子区间问题盲目套用滑动窗口

滑动窗口要求问题具备**单调性**：即在 `[left, right]` 满足条件的前提下，`[left, right+1]` 也有明确的趋势。若约束条件不具备此性质（例如子数组乘积在包含负数时失去单调性），直接套用滑动窗口会漏解或错解。LeetCode 713（乘积小于 k）明确要求所有元素为正整数，正是为了保证单调性成立。

### 误区二：忘记在收缩时同步更新窗口内状态

许多初学者在 `left` 右移时只更新指针，忘记将离开窗口的元素从哈希表或求和变量中移除，导致状态污染。正确做法是在执行 `left++` 之前，先对 `arr[left]` 执行撤销操作（如 `count[arr[left]]--`），再移动指针。

### 误区三：将可变窗口与双层循环混淆

可变窗口虽然在代码形式上包含两个 `while` 或 `for` 嵌套，但由于两个指针共享同一个 0 到 n-1 的"移动预算"，实际总迭代次数为 O(n) 而非 O(n²)。误认为嵌套即代表平方复杂度，会导致对算法性能的错误判断，在面试中给出错误的复杂度分析。

## 知识关联

**与双指针的关系**：滑动窗口是双指针的一种专化形式。通用双指针可以从数组两端向中间收拢（如两数之和问题），也可以同向追逐；而滑动窗口特指两个指针**同向移动且均不回退**的场景，专门用于处理连续子数组/子字符串问题。掌握双指针"同向不回退"这一约束，是理解滑动窗口 O(n) 复杂度的直接前提。

**通向单调队列**：当固定窗口问题从"求和/求平均"升级为"求窗口内最大值/最小值"时，单次 O(1) 的增量更新不再可行——仅靠加减新元素无法快速知道窗口内的极值。此时需要引入**单调队列**（Monotonic Deque）在 O(1) 均摊时间内维护窗口极值，即 LeetCode 239（滑动窗口最大值）的核心数据结构。单调队列本质上是对滑动窗口内状态维护能力的增强，两者在"固定窗口逐步右移"的框架上完全兼容。
