# 滑动窗口

## 概述

滑动窗口（Sliding Window）是一种基于双指针技术的算法思想，通过维护一个大小可变或固定的"窗口"在线性数据结构上单向滑动，将原本需要 O(n²) 的暴力枚举降低至 O(n) 的时间复杂度。窗口由左指针 `left` 和右指针 `right` 共同界定，表示当前考察的子数组或子字符串区间 `[left, right]`。

滑动窗口的思想最早在网络协议中用于流量控制（TCP 协议的发送/接收窗口），后被系统化引入算法竞赛与工程问题。1970年代 TCP/IP 协议设计中，滑动窗口用于控制字节流的传输速率，这与算法中的"窗口"管理数据流的本质高度一致。在 LeetCode 等算法平台上，滑动窗口是字符串和数组题目中出现频率最高的模式之一，约占中等难度数组题目的 15%。

滑动窗口的核心价值在于**避免重复计算**：右指针扩张时新增一个元素的贡献，左指针收缩时移除一个元素的贡献，整个过程中每个元素最多被加入和移除各一次，因此总操作次数为 O(2n) = O(n)。

## 核心原理

### 窗口的扩张与收缩逻辑

滑动窗口的标准模板分为**固定窗口**和**可变窗口**两类。

**可变窗口**的框架如下：

```python
left = 0
for right in range(len(s)):
    # 1. 将 s[right] 纳入窗口，更新窗口状态
    window.add(s[right])
    
    # 2. 当窗口不满足条件时，收缩左边界
    while window_invalid():
        window.remove(s[left])
        left += 1
    
    # 3. 此时窗口合法，更新答案
    ans = max(ans, right - left + 1)
```

右指针 `right` 每次向右移动一步（单调递增），左指针 `left` 只在窗口违反约束时右移，且 `left <= right` 始终成立。**两个指针都不会向左回退**，这保证了 O(n) 的复杂度。

**固定窗口**（窗口大小为 k）则先将前 k 个元素建立初始窗口，再每次移入一个新元素、移出最左元素，维护一个"滑进滑出"的定长区间。

### 窗口状态的维护结构

窗口内部的统计量决定了所需的辅助数据结构：

- **求和/计数**：只需一个整型变量，O(1) 更新。例如"和不超过 target 的最长子数组"。
- **字符频次**：使用哈希表（`dict` 或长度 26 的数组），典型题目为"包含所有字符的最小子串"（LeetCode 76），时间复杂度 O(n·|Σ|)，其中 |Σ| 为字符集大小。
- **区间最值**：需要单调队列（Deque）辅助，单独构成一类更复杂的变体，时间复杂度仍为 O(n)。

### 收缩条件的精确判定

可变滑动窗口的关键难点在于正确定义"窗口何时非法"。以"无重复字符的最长子串"（LeetCode 3）为例：

- 窗口状态：用哈希表记录各字符出现次数
- 违规条件：`char_count[s[right]] > 1`（新加入字符导致重复）
- 收缩操作：`char_count[s[left]] -= 1; left += 1`，直到重复消除

答案更新公式：`ans = max(ans, right - left + 1)`，在每次右指针扩张后、窗口合法时执行。

## 实际应用

### 示例一：最小覆盖子串（LeetCode 76）

给定字符串 `s` 和 `t`，找 `s` 中包含 `t` 所有字符的最短子串。

算法流程：用 `need` 哈希表记录 `t` 中各字符需求量，`window` 哈希表记录当前窗口中字符数量，用整数 `valid` 统计已满足需求的字符种类数（初始为 0，目标为 `len(need)`）。右指针扩张时若满足某字符需求则 `valid += 1`；当 `valid == len(need)` 时窗口合法，记录答案后收缩左边界。整体时间复杂度 O(n + m)，空间复杂度 O(|Σ|)。

### 示例二：大小为 k 的滑动窗口最大值（LeetCode 239）

固定窗口大小为 k，需要输出每个位置的窗口最大值。朴素遍历每个窗口需要 O(n·k)，使用单调递减双端队列（Deque）将复杂度降至 O(n)：队列头始终存储当前窗口最大值的下标，新元素入队前弹出所有比其小的元素下标，同时在队头元素超出窗口范围时弹出队头。

### 示例三：AI 工程中的时序数据处理

在时间序列异常检测中，滑动窗口用于提取局部统计特征。例如对传感器数据流每隔 1 秒采一个点，用大小为 60 的固定窗口（60 秒历史数据）实时计算均值和标准差，若当前值偏离均值超过 3σ 则标记异常。窗口每滑动一步，只需 O(1) 时间更新均值（Welford 在线算法），无需重新扫描 60 个点。

## 常见误区

### 误区一：左指针可以使用 for 循环独立驱动

很多初学者将滑动窗口写成双 for 循环（外层 right，内层 left），认为这等价于标准模板。但双 for 循环的 left 每次从头开始或固定步长移动，**无法保证左指针的单调性**，可能导致 O(n²) 的退化。正确做法是左指针使用 `while` 判断条件后再移动，且其移动次数被右指针的移动次数所限制，总移动次数不超过 2n 次。

### 误区二：固定窗口和可变窗口的混用

固定窗口大小为 k 时，收缩条件是 `right - left + 1 > k`（窗口超出大小），此时**必须恰好移除一个元素**，不能用 `while` 循环。若错误使用 `while` 循环收缩，会导致窗口缩得过小，漏掉合法答案。区分标志是题目是否明确给出固定的窗口大小 k。

### 误区三：窗口内需要排序或随机访问时仍套用基础模板

滑动窗口只能高效维护**可增量更新**的统计量（如和、频次、最值借助单调队列）。若问题要求窗口内第 k 小元素，朴素滑动窗口无法 O(1) 更新，需要改用平衡 BST（如 `SortedList`）或堆，时间复杂度上升至 O(n log k)。强行套用基础滑动窗口模板将得到错误结果。

## 知识关联

**与双指针的关系**：滑动窗口是双指针的特化形式，双指针中两个指针可以相向移动（如两数之和的夹逼法），而滑动窗口的两个指针**同向移动且维护一个连续区间**，不具备夹逼特性。掌握双指针的同向移动思想是理解滑动窗口不回退性质的直接基础。

**向单调队列的延伸**：当滑动窗口需要在 O(1) 时间内查询窗口内最大值或最小值时，基础哈希表无法胜任，必须引入单调队列（Monotonic Deque）。单调队列在滑动窗口的约束下（元素只从右端进入、左端超出范围时弹出），保证队头始终是当前窗口的极值，这是从滑动窗口进阶到单调队列的核心驱动场景。LeetCode 239 即是两者结合的经典范例。
