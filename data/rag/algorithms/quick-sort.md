# 快速排序

## 概述

快速排序（Quicksort）是一种基于分治思想的比较排序算法，由英国计算机科学家 Tony Hoare 于 1959 年在莫斯科国立大学访问期间构思，并于 1961 年正式发表。其核心操作是**划分（Partition）**：选择一个"基准元素"（pivot），将数组分为"小于等于 pivot"和"大于 pivot"两个子数组，再对两个子数组递归执行相同过程，直到子数组长度为 0 或 1。

快速排序在实际工程中的地位源于其优秀的平均性能：平均时间复杂度为 O(n log n)，且常数因子极小，缓存命中率高。C 标准库的 `qsort` 函数、Java 的 `Arrays.sort`（对基本类型）以及 Python 早期版本均采用快速排序或其变体。与归并排序不同，快速排序是**原地排序**，空间复杂度为 O(log n)（仅需递归调用栈），无需额外的合并缓冲区。

## 核心原理

### 划分操作（Partition）

快速排序的灵魂是 Partition 函数。最经典的实现是 **Lomuto 分区方案**：

```
partition(A, lo, hi):
    pivot = A[hi]            // 选最后一个元素为基准
    i = lo - 1               // i 指向"小于区"末尾
    for j = lo to hi - 1:
        if A[j] <= pivot:
            i += 1
            swap(A[i], A[j])
    swap(A[i+1], A[hi])      // 将 pivot 放到正确位置
    return i + 1             // 返回 pivot 的最终下标
```

执行完 Partition 后，pivot 左侧所有元素 ≤ pivot，右侧所有元素 > pivot，且 **pivot 本身已处于最终排序位置**，不再参与后续递归。另一种更高效的是 **Hoare 分区方案**（Hoare 原版），使用两个向中间移动的指针，平均交换次数比 Lomuto 少约 3 倍，但边界条件处理稍复杂。

### 时间复杂度分析

快速排序的性能高度依赖 pivot 的选择质量：

- **最优情况**：每次划分恰好均分，递归深度为 log₂n，总比较次数 T(n) = 2T(n/2) + n，解得 **O(n log n)**。
- **平均情况**：可以证明，随机 pivot 选择下期望比较次数约为 **1.386 n log₂n**，仍为 O(n log n)。
- **最坏情况**：每次 pivot 都选到最小或最大值（例如对已排序数组使用固定末尾 pivot），退化为 T(n) = T(n-1) + n，解得 **O(n²)**。

这正是快速排序与归并排序最本质的区别——归并排序在所有情况下均为 O(n log n)，而快速排序存在 O(n²) 的退化风险。

### Pivot 选择策略

为规避最坏情况，工程上常用以下策略：

1. **随机化 pivot**：每次随机选取 `[lo, hi]` 范围内的下标作为 pivot，使得对任意输入的期望复杂度为 O(n log n)，且攻击者无法构造使算法退化的输入序列。
2. **三数取中（Median-of-Three）**：取 `A[lo]`、`A[mid]`、`A[hi]` 三者的中位数作为 pivot，既提升了 pivot 的代表性，又避免对排序数组的直接退化。
3. **Ninther（九数取中）**：将数组分为三段各取中位数，再从三个中位数取中位数，用于 n 极大时（如 n > 40），是 C++ `std::sort` 内部 Introsort 的组成策略之一。

### 三路快速排序（3-Way Quicksort）

当数组存在大量**重复元素**时，标准快速排序依然会将重复值反复参与划分，效率退化。**Dijkstra 的荷兰国旗算法**将数组划分为三段：`< pivot`、`== pivot`、`> pivot`。等于 pivot 的元素在本轮后全部到达最终位置，无需再次参与递归。对于元素取值范围极小的数组（如全为 0/1 的布尔数组），三路快排时间复杂度可降至 **O(n)**，而标准快排仍为 O(n log n)。

## 实际应用

**AI 工程中的 Top-K 问题**：在推荐系统中，需要从数百万候选物品中快速找出得分最高的 K 个。利用快速排序的 Partition 操作可以实现 **QuickSelect 算法**：每次 Partition 后，若 pivot 的下标恰好为 K，则直接返回；否则仅递归进入包含第 K 位的那一半。QuickSelect 的平均时间复杂度为 **O(n)**，比完整排序快得多，被 NumPy 的 `np.partition` 直接采用。

**外部排序与分布式场景**：在 Spark 的 RangePartitioner 中，需要对数据集进行采样后确定分区边界，这一过程本质上是在样本上运行快速排序或 QuickSelect 来确定各百分位数对应的 key 值。

**标准库实现细节**：现代语言的内置排序普遍使用**内省排序（Introsort）**，即快速排序 + 堆排序 + 插入排序的混合策略：以随机化快速排序为主体，当递归深度超过 2×log₂n 时切换为堆排序（避免 O(n²) 退化），当子数组长度 ≤ 16 时切换为插入排序（减少函数调用开销）。

## 常见误区

**误区一：快速排序"总是"比归并排序快**
快速排序的常数因子小、缓存友好，平均情况下比归并排序快约 2-3 倍，但这依赖于 pivot 选择和数据分布。对于链表结构，快速排序失去随机访问优势，归并排序反而更优；对于需要**稳定排序**的场景（相等元素保持原始相对顺序），标准快速排序不满足稳定性，而归并排序天然稳定。

**误区二：快速排序是原地排序，空间复杂度为 O(1)**
快速排序的空间消耗来自**递归调用栈**，而非显式辅助数组。平均情况下栈深度为 O(log n)，最坏情况下为 O(n)（如对已排序数组用末尾 pivot）。因此正确说法是平均空间复杂度 O(log n)，而非 O(1)。通过"尾递归优化"——总是先递归较短的子数组，再用循环处理较长的子数组——可将最坏空间复杂度强制限制在 O(log n)。

**误区三：三路快排在所有场景下更优**
三路划分每轮需要额外维护三个指针（lt、gt、i），在元素**基本不重复**的情况下，与标准 Lomuto 分区相比会引入更多的比较和赋值操作，反而略慢。三路快排的优势仅在重复元素占比显著时才体现（通常建议当重复率 > 20%-30% 时考虑切换）。

## 知识关联

**与递归的关系**：快速排序是"先划分，后递归"的模式，与归并排序的"先递归，后合并"恰好相反。在快速排序中，**所有实质性工作发生在递归调用之前**（Partition 阶段），递归调用本身只是收缩问题规模，这使得快速排序更容易进行尾递归优化。

**与分治法的关系**：快速排序是分治法的典型实例，但它的"分"不像归并排序那样均等——Partition 结果的子数组大小取决于 pivot 的选择，这种不均等性正是导致最坏情况退化的根本原因，也是分析其平均复杂度时需要对所有可能的 pivot 位置求期望的原因。

**与归并排序的对比**：归并排序保证 O(n log n) 的最坏情况复杂度，且稳定，但需要 O(n) 额外空间；快速排序牺牲最坏情况保证和稳定性，换取更小的常数因子和 O(log n) 的空间开销。两者共同构成了理解 Introsort 等混合排序算法设计权衡的基础。
