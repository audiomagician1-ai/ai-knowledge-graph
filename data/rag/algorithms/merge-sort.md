# 归并排序

## 概述

归并排序（Merge Sort）是一种基于"分割-递归-合并"三段式流程的比较排序算法，由约翰·冯·诺依曼（John von Neumann）于1945年发明。其核心操作是将两个已排序的子序列合并为一个更大的有序序列，时间复杂度在最坏、平均、最好情况下均稳定为 **O(n log n)**，这一特性使它成为少数"最坏情况也高效"的排序算法之一。

归并排序是**稳定排序**（stable sort），即相等元素的相对顺序在排序前后保持不变。这一特性在处理具有多关键字排序需求的数据时极为重要，例如先按姓名排序再按年龄排序的场景。与此同时，归并排序的主要代价是需要 **O(n)** 的额外空间用于存储合并过程中的临时数组，这与原地排序算法（如堆排序）形成显著区别。

在AI工程实践中，归并排序广泛用于外部排序（External Sorting）场景，即当数据量超过内存容量、需要分批从磁盘读取并排序时。此外，归并排序的并行化实现非常自然，子数组的递归分割可以直接映射到多线程或分布式计算节点，是理解并行算法设计的重要案例。

## 核心原理

### 递归分割阶段

归并排序将长度为 n 的数组持续对半分割，直至每个子数组长度为 1（长度为1的数组天然有序）。分割深度为 **⌈log₂n⌉** 层。以8个元素的数组为例，共分割3层，产生8个长度为1的叶子节点。此阶段本身不做任何元素比较，仅计算中间索引 `mid = (left + right) / 2` 并递归调用。

### 合并（Merge）阶段

合并是归并排序的核心操作。给定两个已排序的子数组 `A[left..mid]` 和 `A[mid+1..right]`，合并过程使用两个指针 `i`、`j` 分别指向两段起点，逐步比较 `A[i]` 与 `A[j]`，将较小值写入临时数组，对应指针前移；直至某一段耗尽，将另一段剩余元素直接追加。单次合并的时间复杂度为 **O(k)**，其中 k = right - left + 1 为当前合并段的长度。

每层所有合并操作的总比较次数之和为 O(n)，共 log n 层，因此总时间复杂度为 **T(n) = O(n log n)**，对应递推关系式：

> **T(n) = 2T(n/2) + O(n)**

其中 `2T(n/2)` 代表两个子问题的递归代价，`O(n)` 代表合并代价。利用主定理（Master Theorem）中 a=2, b=2, f(n)=O(n) 的情形，可直接得到 T(n) = O(n log n)。

### 自底向上的迭代实现

归并排序除递归版本外，还有一种**自底向上（Bottom-Up）**的迭代实现，避免了递归调用栈开销。该版本从步长 `size=1` 开始，依次对相邻的两段长度为 `size` 的子数组执行合并，每轮循环结束后 `size` 翻倍（1→2→4→8…），共执行 ⌈log₂n⌉ 轮。此方法在链表排序中尤为高效，因为链表节点无法随机访问，递归版本的索引计算会引入额外开销，而自底向上的合并仅需顺序遍历。

## 实际应用

**外部排序（大规模日志处理）**：假设需要对一个100GB的用户行为日志文件按时间戳排序，内存仅有4GB。归并排序的标准做法是：首先将100GB数据切分为25个4GB的块，分别读入内存排序后写回磁盘（产生25个有序的"归并段"）；然后使用多路归并（k-way merge），利用最小堆同时维护25个归并段的当前最小值，每次弹出最小值并写入输出文件，从而完成全局排序。整个过程仅需2次完整磁盘读写。

**Python标准库的TimSort**：Python内置的 `list.sort()` 和 `sorted()` 函数使用的TimSort算法，其核心合并步骤直接继承自归并排序的Merge操作。TimSort于2002年由Tim Peters为Python设计，专门针对现实数据中常见的"部分有序"特征做了优化，将自然有序的连续段（称为"run"）直接作为归并单元，最小run长度设置为32到64之间。

**并行归并排序**：在多核或分布式场景中，递归树的每个子问题天然独立，可以将左右两半分配给不同线程同时处理。以4核CPU排序1024个元素为例，前两层递归（产生4个子数组）可完全并行，理论加速比接近4倍。

## 常见误区

**误区一：认为归并排序空间复杂度为 O(n log n)**。实际上，虽然递归调用栈深度为 O(log n)，但每层的临时数组是复用的，不是同时存在于内存中。合并操作每次只需分配当前段长度的临时空间，所有层共用同一块 O(n) 大小的缓冲区，因此总额外空间复杂度为 **O(n)**，而非 O(n log n)。

**误区二：认为归并排序总是优于插入排序**。对于长度 n ≤ 16 的小数组，插入排序由于常数因子极小（无递归开销、缓存友好），实测性能往往优于归并排序。因此实际工程中（包括上文提到的TimSort），当递归子数组规模小于阈值（通常为8到32）时，会切换为插入排序处理，这种混合策略称为**混合排序（Hybrid Sort）**。

**误区三：归并排序对链表和数组同等高效**。对于数组，归并排序需要 O(n) 的额外空间复制元素；但对于链表，合并操作只需修改 `next` 指针，**无需额外空间**，空间复杂度降至 O(log n)（仅递归栈）。因此在链表场景下，归并排序的空间优势相比数组版本更为突出，也是链表排序问题的最优选择。

## 知识关联

**依赖的前置概念**：归并排序的递归分割直接依赖**递归**的终止条件设计——子数组长度为1即返回，这是递归基础中"基准情形（base case）"的典型应用。而"将大问题拆分为同类小问题、独立求解后合并结果"的整体框架来源于**分治法**，归并排序是分治法中"合并步骤占主要代价"的标准案例（对比二分搜索中合并步骤代价为O(1)）。

**通向下一个概念**：学习归并排序之后，**快速排序**提供了一种不同的分治策略——将比较和分区操作集中在"分割阶段"（partition），合并阶段代价为O(1)，从而将额外空间需求降至O(log n)（仅递归栈）。比较两者可以发现：归并排序是"简单分、复杂合"，快速排序是"复杂分、简单合"。两者均为O(n log n)平均复杂度，但快速排序的最坏情况为O(n²)，而归并排序在任何情况下均保证O(n log n)，这一对比揭示了算法设计中稳定性与常数因子之间的权衡。
