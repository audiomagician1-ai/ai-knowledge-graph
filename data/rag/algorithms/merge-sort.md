# 归并排序

## 概述

归并排序（Merge Sort）是一种基于**分治策略**的稳定排序算法，由约翰·冯·诺依曼（John von Neumann）于1945年发明。其核心思路是将一个无序数组反复二分，直到每个子数组只含1个元素（单元素天然有序），再将相邻的有序子数组两两**合并**成更大的有序数组，最终得到完整的有序序列。

归并排序的时间复杂度在最好、最坏和平均情况下均为 **O(n log n)**，这一特性使它在处理大规模数据时表现非常稳定，不像快速排序那样在最坏情况下退化至 O(n²)。此外，归并排序是**稳定排序**，即相等元素在排序前后保持相对位置不变，这一性质在多键排序场景（如先按姓名排序再按年龄排序）中至关重要。

归并排序的代价在于其空间复杂度为 **O(n)**——合并过程需要与原数组等大的辅助空间。在内存受限的嵌入式系统中这是明显缺点；但在外部排序（磁盘文件排序）场景中，归并排序几乎是唯一实用的选择，因为它可以顺序读写数据块，完美适配磁盘的访问模式。

## 核心原理

### 分割阶段（Divide）

归并排序的分割过程是纯粹的递归二分：对于长度为 n 的数组，取中间下标 `mid = (left + right) / 2`，将数组分为 `[left, mid]` 和 `[mid+1, right]` 两段，递归调用自身直到子数组长度为1。分割本身不移动任何元素，仅通过下标界定范围，因此这一阶段的时间开销为 O(log n)（递归深度）。

### 合并阶段（Merge）——算法的核心

合并两个已排序子数组是归并排序真正的工作所在。设左子数组为 L、右子数组为 R，使用**双指针**分别指向 L 和 R 的头部，每次取较小的元素写入辅助数组，指针前移，直到一侧耗尽后将另一侧剩余元素直接追加。以合并 `[2, 5, 8]` 和 `[1, 4, 9]` 为例：

```
比较 2 vs 1 → 取 1；比较 2 vs 4 → 取 2；比较 5 vs 4 → 取 4；
比较 5 vs 9 → 取 5；比较 8 vs 9 → 取 8；追加 9
结果：[1, 2, 4, 5, 8, 9]
```

每次合并两段共 k 个元素需要 O(k) 时间。整棵递归树共有 log n 层，每层合并的元素总数恰好为 n，因此总时间复杂度严格为 **T(n) = 2T(n/2) + O(n)**，由主定理解得 O(n log n)。

### 递归树结构与空间分析

归并排序的递归树高度为 ⌈log₂n⌉。以 n=8 为例，树高为3，第0层1个问题（规模8），第1层2个问题（规模4），第2层4个问题（规模2），第3层8个问题（规模1）。辅助数组在合并时被重复使用，但任意时刻最多占用 O(n) 空间。如果使用**原地归并**（in-place merge），可将空间降至 O(1)，但合并操作的时间复杂度会上升至 O(n²)，实际工程中几乎不采用。

### 自底向上的迭代实现

除递归版本外，归并排序还有**自底向上（Bottom-Up）**的迭代实现：从步长 `step=1` 开始，每轮将相邻的两个大小为 `step` 的块两两合并，然后令 `step *= 2`，重复直到 `step >= n`。这种实现避免了函数调用的栈开销，在某些编译器优化不足的环境中性能更优，也是 Java 标准库 `Arrays.sort` 对对象数组采用的 TimSort 算法的底层基础。

## 实际应用

**外部排序**是归并排序最经典的专属领域。当待排数据量超出内存容量（例如对一个200GB的日志文件排序），可先将文件切分为若干可以装入内存的块，对每块用内部排序生成有序的"归并段"，再用多路归并将所有归并段合并为最终有序文件。Hadoop MapReduce 框架的 Shuffle 阶段正是采用多路归并排序来合并各 Mapper 输出。

**链表排序**中归并排序优于快速排序。对链表执行快速排序需要随机访问，效率低下；而归并排序只需顺序遍历和修改指针，合并两个有序链表的空间复杂度可降至 O(1)（无需辅助数组），时间仍为 O(n log n)。LeetCode 第148题"排序链表"的标准最优解正是归并排序。

**Python 的 Timsort**（自2.2版本起作为默认排序算法）是归并排序的工程优化变体：它先识别数据中天然存在的有序子序列（run），然后用归并排序合并这些 run。对于现实数据（通常含有局部有序片段），Timsort 的实际性能远优于纯归并排序的理论 O(n log n) 下界。

## 常见误区

**误区1：认为归并排序的辅助空间可以忽略**
部分学习者认为归并排序的"额外空间是临时的，无关紧要"。事实上，对一个含有1亿个整数（约400MB）的数组进行归并排序，需要额外申请同等大小的辅助数组，总内存占用达800MB。在内存紧张的生产环境中，这可能直接导致OOM（Out of Memory）错误。这正是为什么对**基本类型数组**，Java 的 `Arrays.sort` 选择快速排序而非归并排序。

**误区2：混淆"稳定"与"原地"两个独立属性**
稳定性（相等元素不改变相对顺序）和原地性（O(1) 额外空间）是两个独立维度。标准归并排序是**稳定但非原地**的；堆排序是**原地但不稳定**的；插入排序既稳定又原地。不少初学者误以为稳定的排序就一定需要大量额外空间，实际上插入排序、冒泡排序均能在 O(1) 空间内保持稳定性，只是时间复杂度较高。

**误区3：认为归并排序在所有场景下都优于 O(n²) 算法**
当数组规模很小（通常 n < 16）时，归并排序的递归调用开销和缓存未命中代价会使其实际速度慢于插入排序。这正是 Timsort 和标准 C++ `std::sort` 的实现策略：当子问题规模小于阈值时，自动切换为插入排序处理，这种混合策略在工程中普遍采用。

## 知识关联

**前置概念衔接**：归并排序的分割阶段直接体现**递归**的"问题规模缩小+基本情形"结构，其递推关系 T(n)=2T(n/2)+O(n) 正是**分治法**时间复杂度分析（主定理）的典型教学案例。不理解递归调用栈的行为，很难正确实现归并排序中下标的传递方式。

**后续概念延伸**：学习完归并排序后，**快速排序**是最直接的对比对象。两者同属分治排序，但策略相反：归并排序在"合并"阶段做实质工作（后处理），快速排序在"分区"阶段做实质工作（前处理）。理解这一对称性，有助于掌握快速排序的 Partition 操作为何能绕过归并排序对辅助空间的依赖，以及两者在缓存局部性、最坏情况和稳定性上的具体权衡。
