# 堆排序

## 概述

堆排序（Heap Sort）是一种基于**完全二叉树**结构的比较排序算法，时间复杂度稳定为 O(n log n)，空间复杂度为 O(1)。它由 J.W.J. Williams 于 1964 年首次提出，同年 Robert W. Floyd 改进了建堆算法，将初始建堆阶段的时间复杂度从 O(n log n) 优化到 O(n)，使完整算法效率大幅提升。

堆排序的核心思想是将待排序数组原地构造为最大堆（Max-Heap），然后反复将堆顶元素（当前最大值）与末尾元素交换，并对缩小后的堆执行下沉（Sift-Down）操作，从而逐步得到有序序列。与归并排序相比，堆排序无需额外的 O(n) 辅助空间；与快速排序相比，堆排序的最坏情况时间复杂度不会退化到 O(n²)，在需要**最坏情况保证**的实时系统中具有重要价值。

在 AI 工程领域，堆排序直接支撑了 Top-K 问题的求解（如推荐系统中检索最相关的 K 个候选项），以及优先级调度队列的实现。理解堆排序的完整流程，是掌握 K 近邻搜索（KNN）和 Dijkstra 最短路径等算法中堆操作的重要基础。

## 核心原理

### 最大堆性质与数组表示

堆排序依赖最大堆的核心性质：对任意节点 i，其值满足 `A[i] ≥ A[2i+1]` 且 `A[i] ≥ A[2i+2]`（以 0 索引为例）。这种性质使数组 `[9, 7, 6, 3, 4, 1, 5]` 可以被视为一棵完全二叉树，其中索引 0 是根节点，索引 `floor((n/2) - 1)` 是最后一个非叶节点。以 n=7 的数组为例，最后一个非叶节点的索引为 2，建堆时只需从索引 2 开始向下执行 Sift-Down，无需处理叶节点，这正是 Floyd 建堆算法将时间复杂度降至 O(n) 的关键所在。

### 建堆阶段（Build-Heap）

建堆从数组中间位置 `floor(n/2) - 1` 开始，**逆向**对每个非叶节点调用 Sift-Down，直到根节点。以数组 `[4, 10, 3, 5, 1]` 为例：

1. 从索引 1（值为 10）开始 Sift-Down：比较子节点 5、1，无需交换。
2. 对索引 0（值为 4）执行 Sift-Down：与最大子节点 10 交换，得 `[10, 4, 3, 5, 1]`，继续下沉至索引 1，与 5 交换，最终得 `[10, 5, 3, 4, 1]`。

整个建堆过程进行的比较次数上界为 **2n** 次（精确分析使用求和公式 `∑ floor(n/2^(k+1)) * k`），远优于逐一插入建堆的 O(n log n)。

### 排序阶段（Extract-Max 循环）

建堆完成后，堆排序执行以下循环共 **n-1** 次：

```
for i from n-1 downto 1:
    swap(A[0], A[i])       // 将堆顶最大值放到已排序区
    sift_down(A, 0, i)     // 在前 i 个元素上重新维护堆性质
```

每次 Sift-Down 的时间复杂度为 O(log i)，因此排序阶段总复杂度为 `∑ log i（i=1 to n-1）= O(n log n)`。整个算法的时间复杂度精确表达为 **O(2n log n)**（常数项约为 2），实践中通常比快速排序慢约 2-5 倍，原因在于 Sift-Down 的内存访问模式对 CPU 缓存不友好——堆结构的跳跃式访问（父子节点在内存中并不连续）导致缓存命中率低。

### Sift-Down 的详细实现

Sift-Down（下沉）操作是堆排序的核心子程序。给定堆大小为 `heap_size`、起始节点索引 `i`：

```python
def sift_down(A, i, heap_size):
    largest = i
    left  = 2 * i + 1
    right = 2 * i + 2
    if left < heap_size and A[left] > A[largest]:
        largest = left
    if right < heap_size and A[right] > A[largest]:
        largest = right
    if largest != i:
        A[i], A[largest] = A[largest], A[i]
        sift_down(A, largest, heap_size)
```

递归深度最深为 `floor(log₂ n)`，对应树的高度。例如 n=1024 时，最多递归 10 层。

## 实际应用

### Top-K 问题的高效求解

在推荐系统中，从 100 万个候选商品中找出评分最高的 K=100 个，若使用完整堆排序则需 O(n log n) 时间。更优的做法是维护一个大小为 K 的**最小堆**：遍历所有 n 个元素，若当前元素大于堆顶则替换并执行 Sift-Down。这样总时间复杂度为 O(n log K)，当 K≪n 时效率显著优于完整排序。PyTorch 的 `torch.topk` 函数底层即采用类似策略。

### 操作系统进程调度

Linux 内核的完全公平调度器（CFS，2.6.23 版本引入）使用红黑树而非堆，但早期调度器使用基于堆的优先队列管理运行队列。在 AI 训练框架中，任务调度器（如 Ray 的任务队列）使用最小堆按优先级分发 GPU 任务，堆排序的 Sift-Down 逻辑直接应用于每次任务插入和弹出后的堆维护。

### 外部排序中的多路归并

在处理超出内存容量的大规模训练数据时，多路归并排序使用大小为 K 的最小堆，从 K 个已排序文件块中每次提取最小值。例如合并 16 个排好序的数据分片时，维护一个大小为 16 的最小堆，每次弹出堆顶并补入对应分片的下一个元素，整体时间复杂度为 O(n log 16) = O(4n)。

## 常见误区

### 误区一：堆排序是稳定排序

堆排序**不是稳定排序**。在 Sift-Down 过程中，相同键值的元素会因堆结构的调整而改变相对顺序。例如数组 `[3a, 3b, 1]`（a、b 表示两个值均为 3 的不同元素），建堆后可能变为 `[3b, 3a, 1]`，导致原始相对顺序丢失。在 AI 特征排序或需要稳定排名的场景中必须使用归并排序或 Timsort（Python 内置 sort 所用算法）替代。

### 误区二：O(n) 建堆意味着整体是 O(n)

一些初学者误以为 Floyd 建堆是 O(n)，就认为整个堆排序可以在 O(n) 时间完成。实际上，**建堆是 O(n)，但后续的 n-1 次提取最大值操作各需 O(log n)**，导致排序阶段为 O(n log n)，整体复杂度仍为 O(n log n)。这两个阶段不可合并，O(n log n) 是基于比较的排序算法的理论下界（对一般情况）。

### 误区三：堆排序因原地操作而比快速排序更快

尽管堆排序空间复杂度为 O(1)，实际基准测试（Benchmark）中堆排序通常比快速排序慢 **2 到 5 倍**。原因在于堆排序访问内存的模式是非局部的——对大小为 n=2²⁰（约 100 万）的数组，Sift-Down 每次跳跃访问的内存距离可达数万字节，造成大量 L1/L2 缓存缺失（Cache Miss），而快速排序的分区操作主要在连续内存上顺序扫描，缓存友好性远优于堆排序。

## 知识关联

堆排序直接建立在**堆（优先队列）**数据结构之上——若没有对最大堆性质、父子节点索引公式（`parent = floor((i-1)/2)`）以及 Sift-Up 和 Sift-Down 操作的清晰理解，就无法正确实现建堆和排序阶段。建议在实现堆排序之前，先手动在纸上对 8 个元素的数组完整模拟一次 Build-Heap 过程，验证每个非叶节点的下沉路径。

在算法横向对比中，堆排序与**快速排序**和**归并排序**共同构成 O(n log n) 排序三角：快速排序平均最快但最坏退化；归并排序稳定但需 O(n) 额外空间；堆排序保证最坏 O(n log n) 且空间 O(1)，适合嵌入式
