# 堆排序

## 概述

堆排序（Heap Sort）是一种基于二叉堆数据结构的比较排序算法，由 J.W.J. Williams 于 1964 年在论文《Algorithm 232: Heapsort》中正式提出，同年 Robert Floyd 进一步优化了建堆过程，将建堆时间复杂度从 O(n log n) 降低至 O(n)。堆排序属于原地排序（in-place sorting），仅需 O(1) 的额外空间，且在最坏情况下仍能保证 O(n log n) 的时间复杂度，这一特性使其在内存受限环境中具有独特价值。

堆排序的工作流程分为两个阶段：第一阶段将无序数组构建为最大堆（Max-Heap），第二阶段反复将堆顶的最大元素与末尾元素交换并对剩余元素执行下沉（sift-down）操作，最终形成升序排列。这一过程不需要额外的数组空间，所有操作均在原始数组上完成。

堆排序在 AI 工程中的典型应用场景包括：优先级调度队列的批量初始化、Top-K 问题的高效求解（如推荐系统中筛选相关性最高的 K 个候选项），以及在 A* 搜索算法中对开放集合进行高效管理。与快速排序相比，堆排序在最坏情况下不退化为 O(n²)，因此在对时间复杂度上界有严格要求的生产环境中更为可靠。

## 核心原理

### 最大堆的数组表示

堆排序利用完全二叉树的数组存储特性。对于下标从 0 开始的数组，节点 `i` 的左子节点下标为 `2i + 1`，右子节点下标为 `2i + 2`，父节点下标为 `⌊(i - 1) / 2⌋`。最大堆要求每个节点的值不小于其子节点的值，即满足：

> **堆性质**：`A[parent(i)] >= A[i]`，对所有 `i > 0` 成立

这一隐式树结构使得堆排序无需指针或链表即可操作树形逻辑，是其空间复杂度维持在 O(1) 的直接原因。

### 建堆过程（Build-Heap / Heapify）

Floyd 的线性建堆算法从最后一个非叶节点（下标为 `⌊n/2⌋ - 1`）开始，向下遍历至根节点，对每个节点执行 `sift-down` 操作。之所以时间复杂度为 O(n) 而非 O(n log n)，是因为大多数节点位于树的底层，其 `sift-down` 所需步骤数接近 0。

精确分析：高度为 h 的节点最多有 `⌈n / 2^(h+1)⌉` 个，每个节点的下沉代价为 O(h)，对全树求和得到：

$$T = \sum_{h=0}^{\lfloor \log n \rfloor} \left\lceil \frac{n}{2^{h+1}} \right\rceil \cdot O(h) = O(n)$$

这是堆排序相较朴素方法（逐个插入建堆，O(n log n)）在初始化阶段的显著优势。

### 排序阶段（Extract-Max 循环）

建堆完成后，执行 n-1 次以下操作：
1. 将 `A[0]`（当前最大值）与 `A[heap_size - 1]` 交换
2. 将逻辑堆大小减 1（已排序元素不再参与后续堆操作）
3. 对根节点 `A[0]` 执行 `sift-down`，恢复堆性质

每次 `sift-down` 的时间复杂度为 O(log n)，n-1 次操作合计 O(n log n)，与建堆阶段合并后总时间复杂度为 O(n log n)。

堆排序是**不稳定排序**：当两个相等元素在建堆和交换过程中位置被打乱时，原始相对顺序无法保证。例如对 `[3a, 3b, 1]` 排序，输出结果中 3a 和 3b 的顺序可能颠倒。

## 实际应用

**Top-K 问题**：在推荐系统中，给定 n 个候选物品的评分数组，需要返回评分最高的 K 个物品（K << n）。建立大小为 K 的最小堆，遍历剩余 n-K 个元素，若某元素大于堆顶则替换并重新 sift-down，最终堆中 K 个元素即为答案。时间复杂度 O(n log K)，空间复杂度 O(K)，远优于全量排序。

**操作系统任务调度**：Linux 内核的 Completely Fair Scheduler（CFS）使用红黑树而非堆，但在早期版本和简化版调度器实现中，堆排序的思想被用于批量整理进程优先级队列，以 O(n) 建堆后按顺序调度。

**外部排序的辅助结构**：在处理超大数据集的多路归并排序时，维护一个大小为 k（k 为归并路数）的最小堆，每次从堆顶取出最小元素写入输出文件，再从对应数据块读入新元素插入堆中。k = 10 时，每次 sift-down 仅需约 3 次比较（log₂10 ≈ 3.32），显著降低 I/O 与比较开销的比值。

## 常见误区

**误区一：认为堆排序实践性能优于快速排序**  
理论上两者同为 O(n log n)，但堆排序在实际运行中通常比快速排序慢 2 到 5 倍。原因在于堆排序的内存访问模式极不规律：sift-down 过程中频繁访问 `A[2i+1]`、`A[2i+2]` 等跳跃性地址，导致 CPU 缓存（L1/L2 Cache）命中率极低。快速排序具有更好的局部性，顺序访问子数组使得缓存命中率显著更高。因此在通用排序场景下，`std::sort` 等标准库实现通常优先选择快速排序或 Introsort，而非堆排序。

**误区二：混淆建堆方向导致算法错误**  
部分初学者误以为建堆应从根节点向下（即按层序从上往下对每个节点执行 sift-up），这将导致 O(n log n) 的建堆时间且逻辑复杂。正确的 Floyd 建堆从下标 `⌊n/2⌋ - 1` 开始向左遍历至 0，每个节点执行 sift-down。反向操作（sift-up）用于堆的**插入**操作，而非建堆，两者不可互换。

**误区三：认为堆排序是稳定排序**  
由于排序阶段将堆顶元素直接与末尾元素交换，相等元素的原始相对顺序必然被打乱。若业务场景要求稳定排序（例如先按优先级排序、再按时间戳排序的组合排序），则堆排序不适用，应选择归并排序（稳定，O(n log n)）或 Timsort（Python 和 Java 标准库的默认选择）。

## 知识关联

**前置知识**：堆排序完全依赖二叉堆（优先队列）的结构特性。`sift-down` 和 `sift-up` 操作是堆的基本维护操作，建堆过程本质上是对 n/2 个节点批量执行 `sift-down`。若对堆的数组表示、完全二叉树的性质或堆序性不熟悉，将无法正确实现 `sift-down` 的终止条件（当节点值已大于等于两个子节点或到达叶节点时停止）。

**算法关联**：堆排序是 Dijkstra 最短路径算法和 Prim 最小生成树算法的底层加速结构——这两个算法在每轮迭代中均需提取最小代价节点，本质上执行的正是堆排序排序阶段中的 Extract-Min 操作。掌握堆排序的 sift-down 机制，有助于理解为何使用二叉堆实现 Dijkstra 时，时间复杂度从朴素的 O(V²) 降低至 O((V + E) log V)。

**延伸方向**：标准二叉堆实现的堆排序仍受缓存性能限制。工程实践中可考虑 d-ary 堆（d 叉堆，常用 d=4），减少树的高度以降低 sift-down 次数，但每次 sift-down 需比较更多子节点。当 d=4 时，树高为 log₄n，在批量提取操作频繁的场景下比二叉堆提速约 20%~30%。
