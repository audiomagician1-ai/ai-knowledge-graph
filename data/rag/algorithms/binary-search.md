# 二分查找

## 概述

二分查找（Binary Search）是一种在**有序数组**中以对数时间复杂度 O(log n) 定位目标值的算法。其核心思想是：每次将搜索范围折半——先检查中间元素，若目标值小于中间元素则向左半区继续，若大于则向右半区继续，每次比较都能排除一半候选。与线性查找 O(n) 相比，对于一个含 10 亿元素的有序数组，二分查找最多只需约 30 次比较（log₂(10⁹) ≈ 30），效率优势极为显著。

二分查找的思想最早可追溯至 1946 年，John Mauchly 在讨论 ENIAC 编程时提到了类似概念，但第一个完全正确、处理所有边界情况的二分查找实现直到 1962 年才由 Derrick Henry Lehmer 给出。这段历史说明了一个重要事实：二分查找的描述看似简单，但细节（尤其是边界条件）极易出错。著名计算机科学家 Jon Bentley 在其 1986 年著作 *Programming Pearls* 中指出，他在专业程序员培训课上布置二分查找练习，90% 的学员在两小时内无法写出完全正确的版本。

二分查找是 AI 工程中模型调参、特征索引检索、近似最近邻搜索等场景的基础工具。例如在机器学习超参数搜索中，对学习率这类连续参数进行二分式搜索是网格搜索的高效替代方案；在大规模向量数据库的倒排索引中，对词汇表的查找也直接依赖二分查找。

## 核心原理

### 算法流程与数学表达

设有序数组 `A[0..n-1]`，目标值为 `target`，定义左指针 `lo = 0`，右指针 `hi = n - 1`。每轮迭代计算中间位置：

```
mid = lo + (hi - lo) // 2
```

注意此处**不使用** `(lo + hi) // 2`，是为了防止当 lo 和 hi 均为大整数时发生整数溢出（在 C/C++/Java 中这是真实的 bug，1987 年 Java 标准库 `Arrays.binarySearch` 直到 2006 年才修复此漏洞）。比较后的三路分支为：

- `A[mid] == target`：返回 `mid`
- `A[mid] < target`：令 `lo = mid + 1`，收缩左边界
- `A[mid] > target`：令 `hi = mid - 1`，收缩右边界

循环终止条件为 `lo > hi`，此时目标值不存在于数组中，返回 -1。**整个算法最多执行 ⌊log₂n⌋ + 1 次比较**。

### 边界变体：寻找左/右边界

在 AI 工程实践中，往往不只是查找是否存在某值，而是需要找到某个值的**第一次出现位置**（左边界）或**最后一次出现位置**（右边界），例如在时间序列数据中查找某个时间戳区间。

**左边界查找**：在 `A[mid] == target` 时不立即返回，而是令 `hi = mid - 1`，继续向左收缩，最终 `lo` 即为左边界。**右边界查找**：在命中时令 `lo = mid + 1`，最终 `hi` 为右边界。这两种变体与基础版本共享同一框架，但逻辑细节不同，需分别理解而不能死记。

### 二分答案：将二分查找泛化为决策工具

二分查找的应用不限于在数组中查找值，还可以**二分答案空间**。若一个问题满足"答案具有单调性"（即满足条件的答案 x 成立则 x+1 也成立，或反之），就可以对答案本身进行二分。例如：在 AI 模型剪枝中，要找到使模型精度不低于某阈值的**最大稀疏度**，可以对稀疏度区间 [0, 1] 进行二分，每次评估中间稀疏度是否满足精度约束，时间复杂度从穷举的 O(n) 降至 O(log n × 评估成本)。

## 实际应用

**超参数调优**：在学习率搜索中，若已知过小的学习率（如 1e-6）导致欠拟合、过大（如 1.0）导致不收敛，可对 log 空间的学习率进行二分搜索，每次训练评估后决定向哪半区收缩，通常 7～10 次迭代即可定位到合理范围。

**词汇表查询**：NLP 模型的 tokenizer 维护着按字母序排列的词汇表（如 BERT 的 vocab.txt 含 30522 个词条），对输入词做二分查找比哈希表更节省内存，且对缓存友好，在嵌入式设备部署场景中常见此做法。

**分位数计算**：在线学习系统中需要实时计算特征值的中位数或 p99 分位数，常用的 t-Digest 算法内部就使用二分查找在有序的 centroid 列表中定位插入位置，使每次更新的时间复杂度维持在 O(log k)，其中 k 为 centroid 数量（通常取 100～1000）。

## 常见误区

**误区一：认为"差不多有序"也能用二分查找**。二分查找要求数组**严格有序**（或至少按查找维度单调）。对于仅"大致有序"或含重复局部逆序的数组，二分查找会静默地返回错误结果而不报错，这比抛出异常更危险。在 AI 特征工程中，对未经排序验证的特征桶边界直接用二分查找是一类常见的数据 bug。

**误区二：循环条件写成 `lo < hi` 而非 `lo <= hi`**。当数组只剩一个元素（`lo == hi`）时，`lo < hi` 会直接跳出循环而不检查该元素，导致漏报。正确写法是 `lo <= hi`，确保单元素情况也被覆盖。这与左/右边界变体的终止条件不同，混用会引发难以察觉的 off-by-one 错误。

**误区三：将二分查找的时间复杂度 O(log n) 误解为"总是快于线性查找"**。对于 n 极小（如 n < 8）的数组，线性查找因无分支跳转、缓存局部性更好，实际运行速度往往快于二分查找。TensorFlow 的 `tf.data` 管道中，对小型查找表（≤16 项）的内部实现选择的正是线性扫描而非二分查找。

## 知识关联

**前置知识**：二分查找直接操作**有序数组/列表**，其正确性完全依赖数组的有序性前提，因此需要熟练掌握数组的随机访问（O(1) 按下标读取）以及排序的含义。如果对"有序"的定义不清晰，二分查找的三路比较逻辑就无从建立。

**通向二叉搜索树（BST）**：二分查找的每次迭代——"取中间，向左或向右"——与 BST 的节点比较逻辑完全同构。BST 本质上是将二分查找的决策结构从线性数组**具象化为树形数据结构**，使得插入和删除操作也能保持 O(log n)，突破了有序数组插入需要 O(n) 移位的瓶颈。理解二分查找的左/右分支逻辑，是读懂 BST 查找路径的直接基础。

**通向稀疏表（Sparse Table）**：稀疏表用于静态数组的区间最值查询（RMQ），其预处理阶段需要计算 log₂n 层级的区间，而查询时通过 O(1) 定位两个重叠区间来覆盖目标范围。理解二分查找中对 `log₂n` 层折半的计数方式，有助于直觉上把握稀疏表预处理的层数设计（通常预计算到第 ⌊log₂n⌋ 层）。
