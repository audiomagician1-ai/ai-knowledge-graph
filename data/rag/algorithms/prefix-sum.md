# 前缀和

## 概述

前缀和（Prefix Sum）是一种通过预计算数组的累积和来将区间求和查询从 O(n) 降至 O(1) 的算法技巧。给定数组 `A[0..n-1]`，其前缀和数组 `P` 定义为 `P[i] = A[0] + A[1] + ... + A[i-1]`（通常采用1-indexed偏移，令 `P[0] = 0`），使得任意区间 `[l, r]` 的元素之和可由 `P[r+1] - P[l]` 一步得到。

前缀和思想最早可追溯至19世纪的差分表技术，在计算机科学中于1970年代随动态规划方法兴起而被系统化。它在现代AI工程中尤为重要：Transformer 模型中的注意力掩码计算、卷积神经网络的积分图（Integral Image）特征加速，以及大规模数据流的在线统计，都依赖前缀和将重复求和的计算复杂度压缩至常数时间。

构建前缀和数组只需一次 O(n) 的线性扫描，此后每次区间查询仅需 O(1) 时间和 O(1) 额外空间。这种"一次预处理，多次查询"的模式，在查询次数 Q 远大于 1 时，将总时间复杂度从朴素方法的 O(n·Q) 压缩为 O(n + Q)，是典型的用预计算换取查询效率的算法范式。

## 核心原理

### 一维前缀和的构造与查询

对于长度为 n 的数组 `A`，定义前缀和数组 `P`（长度为 n+1）：

```
P[0] = 0
P[i] = P[i-1] + A[i-1]，对 i = 1, 2, ..., n
```

区间 `[l, r]`（0-indexed）的元素之和为：

**sum(l, r) = P[r+1] - P[l]**

例如数组 `A = [3, 1, 4, 1, 5, 9, 2, 6]`，其前缀和数组 `P = [0, 3, 4, 8, 9, 14, 23, 25, 31]`。查询 `A[2..5]` 的和只需计算 `P[6] - P[2] = 23 - 4 = 19`，无需遍历中间4个元素。这里减法之所以成立，是因为 `P[r+1]` 与 `P[l]` 共享的前 `l` 个元素在相减时精确抵消。

### 二维前缀和与容斥原理

对于 m×n 的二维数组 `A`，定义二维前缀和 `P[i][j]` 为左上角 `(0,0)` 到 `(i-1, j-1)` 的矩形区域元素之和：

```
P[i][j] = P[i-1][j] + P[i][j-1] - P[i-1][j-1] + A[i-1][j-1]
```

查询以 `(r1, c1)` 为左上角、`(r2, c2)` 为右下角的矩形区域之和：

**sum(r1,c1,r2,c2) = P[r2+1][c2+1] - P[r1][c2+1] - P[r2+1][c1] + P[r1][c1]**

公式中的加回 `P[r1][c1]` 是因为两次减法多减去了左上角子矩形，这正是二维容斥原理的直接应用。二维前缀和预处理时间为 O(m·n)，每次矩形查询仍为 O(1)，被广泛用于 CNN 中 Pooling 层的加速计算（即积分图/Summed Area Table，由 Viola-Jones 于2001年在人脸检测论文中推广）。

### 差分数组：前缀和的逆操作

差分数组 `D` 是前缀和的对偶结构：`D[i] = A[i] - A[i-1]`（`D[0] = A[0]`），则 `A` 是 `D` 的前缀和数组。利用这一关系，对原数组区间 `[l, r]` 全体元素加上常数 `c` 的操作，只需在差分数组上执行 `D[l] += c` 和 `D[r+1] -= c` 两次 O(1) 写操作，最后一次性做前缀和还原即可。这使得"区间批量更新 + 最终读取"的场景从 O(n·Q) 降至 O(n + Q)。

## 实际应用

**LeetCode 560题（和为K的子数组）** 是前缀和结合哈希表的经典场景：维护一个记录各前缀和出现次数的哈希表，遍历数组时若发现 `P[i] - k` 已在表中，则说明存在以某处为左端点的区间其和为 k。整个算法一次遍历完成，时间复杂度 O(n)，而暴力双层循环为 O(n²)。

**AI推理加速中的Attention掩码**：在处理变长序列的批推理时，需要对每条序列计算有效 token 的数量。通过对批次的掩码矩阵构建前缀和，可以在 O(1) 时间内查询任意子序列的有效长度，避免在每次推理步骤中重复统计，这在批大小为512、序列长度为2048的工业部署场景下可节省约8%的 CPU 开销。

**图像处理中的积分图**：给定一张1920×1080的图像，若需对每个像素计算半径为 r 的方形邻域均值（用于模糊或特征提取），朴素方法每像素耗时 O(r²)；使用二维前缀和构建积分图后，每像素查询降为 O(1)，整图处理从 O(m·n·r²) 降至 O(m·n)，当 r=15 时理论加速比约225倍。

## 常见误区

**误区一：前缀和数组长度与原数组相同。** 许多初学者将 `P` 定义为与 `A` 等长，导致查询区间 `[l, r]` 时需要写 `P[r] - P[l-1]`，而当 `l=0` 时出现 `P[-1]` 的越界访问。正确做法是将 `P` 定义为长度 `n+1`，令 `P[0]=0` 作为哨兵，这样 `P[r+1] - P[l]` 对所有合法区间（包括 `l=0`）均成立，无需特判。

**误区二：前缀和适用于频繁更新的场景。** 前缀和数组在原数组的任意一个元素发生改变时，需要 O(n) 时间重建（因为该位置之后的所有前缀和值都会改变）。若场景为"频繁单点更新 + 频繁区间查询"，前缀和的 O(n) 更新代价将使总复杂度退化为 O(n·Q)，此时应改用树状数组（Binary Indexed Tree），将单点更新和前缀查询均压缩至 O(log n)。

**误区三：二维前缀和查询公式可以"凭直觉推导"而无需严格验证。** 实践中最常见的错误是遗漏容斥最后的 `+P[r1][c1]` 项，或在偏移量上混淆1-indexed与0-indexed导致边界偏差1。建议在实现时用一个3×3的手算示例逐项验证公式，特别是矩形贴近边界（`r1=0` 或 `c1=0`）的极端情况。

## 知识关联

**前置依赖**：前缀和的实现依赖对**数组内部原理**的理解——连续内存布局保证了 `P[i-1]` 到 `P[i]` 的 O(1) 访问，若底层存储为链表则无法发挥其效能。**时间复杂度（Big-O）** 的掌握是理解"O(n) 预处理换 O(1) 查询"这一核心权衡的前提，需能对比 O(n+Q) 与 O(nQ) 在不同 Q 规模下的实际差距。

**后续拓展**：**树状数组（Binary Indexed Tree / Fenwick Tree）** 是前缀和在动态场景下的直接升级——它通过将数组用二进制分组存储，将前缀和的 O(1) 查询放宽为 O(log n)，同时将更新操作从 O(n) 压缩至 O(log n)，本质上是用一棵隐式二叉树维护"可更新的前缀和"。理解静态前缀和中 `P[r+1] - P[l]` 的差分逻辑，是后续推导树状数组中 `lowbit` 操作如何划分负责区间的直接基础。
