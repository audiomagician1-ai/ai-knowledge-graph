# 记忆化搜索

## 概述

记忆化搜索（Memoization Search）是一种将递归搜索与缓存技术结合的算法优化策略：在执行深度优先搜索或递归分治时，将已经计算过的子问题结果存储到哈希表或数组中，当同一子问题再次被访问时直接读取缓存结果，而不重新计算。这种方式本质上是"自顶向下"的动态规划实现，与"自底向上"的传统动态规划表格法形成对应关系。

记忆化搜索的思想最早可以追溯到1960年代 Donald Michie 对机器学习中重复计算问题的研究。"Memoization"这一术语由 Donald Michie 于1968年在论文《Memo Functions and Machine Learning》中正式提出，区别于"Memorization"（记忆），专指将函数调用结果存储以供复用的技术手段。

记忆化搜索的重要性体现在它能将指数级时间复杂度的递归算法直接降至多项式级。以斐波那契数列的朴素递归为例，计算 F(50) 需要约 2^50 次操作，而加入记忆化后只需 O(n) 次计算。这种优化不改变递归的逻辑结构，只是在递归入口处增加缓存查找，因此对程序员来说代码改动极小但性能提升极大。

## 核心原理

### 重叠子问题与缓存机制

记忆化搜索成立的前提是问题具有**重叠子问题**特性：同一个参数组合对应的子问题会在递归树中被多次访问。以经典的"最长公共子序列（LCS）"为例，两个长度为 m 和 n 的字符串，朴素递归的时间复杂度为 O(2^(m+n))，而加入二维记忆化数组 `dp[i][j]` 后，每个状态只计算一次，总时间降为 O(m×n)。

缓存的数据结构选择影响实际性能：当状态维度明确且值域有界时，使用数组（如 `int memo[1001][1001]`）访问时间为 O(1)；当状态由不规则参数组成时，使用哈希表（Python 中的 `dict` 或 C++ 的 `unordered_map`），但哈希碰撞会带来额外常数开销。初始化时通常用哨兵值（sentinel value）如 -1 或 `None` 标识"未计算"状态，与合法结果0区分开。

### 记忆化搜索的标准代码模式

记忆化搜索的实现遵循固定的三步模式：

```python
def solve(state):
    # 第一步：查询缓存
    if state in memo:
        return memo[state]
    # 第二步：递归计算
    if base_case(state):
        result = base_value
    else:
        result = combine(solve(sub_state_1), solve(sub_state_2))
    # 第三步：存入缓存
    memo[state] = result
    return result
```

在 Python 中，标准库提供了 `functools.lru_cache` 装饰器，可以用 `@lru_cache(maxsize=None)` 一行代码自动完成缓存管理，其底层实现是字典加双向链表的 LRU 策略。

### 与自底向上动态规划的等价性与差异

记忆化搜索（自顶向下）与迭代 DP（自底向上）在时间复杂度上通常等价，但存在以下关键差异：

| 维度 | 记忆化搜索 | 迭代DP |
|------|-----------|--------|
| 计算顺序 | 按需计算，只处理实际访问到的状态 | 按拓扑序计算全部状态 |
| 空间开销 | 额外的递归栈深度，Python 默认限制为1000层 | 无递归栈，但需显式管理维度 |
| 状态转移方向 | 由当前状态向子状态索取 | 由子状态向父状态推送 |

在**稀疏状态空间**问题中，记忆化搜索的优势尤为明显：例如棋盘博弈问题中可能的棋局状态有 10^44 种（围棋），但实际对弈路径只覆盖极小一部分，记忆化搜索只计算真正被访问到的状态，而迭代 DP 无法直接应用。

### 无后效性要求

记忆化搜索要求子问题的结果仅由其参数决定，与计算路径无关，即满足**无后效性**。若递归过程中存在全局可变状态（例如剩余资源随路径变化），则缓存的结果不能复用，此时必须将所有影响结果的变量纳入状态参数，例如将 `solve(i)` 扩展为 `solve(i, remaining_capacity)`。

## 实际应用

**树形 DP 与换根问题**：在"统计以每个节点为根时子树的最大路径和"问题中，需要对每个节点分别做一次 DFS，朴素做法为 O(n²)。使用两次 DFS + 记忆化后，每个节点的"向上贡献值"和"向下贡献值"分别缓存，总复杂度降为 O(n)。

**博弈论搜索**：Nim 游戏的 Sprague-Grundy 定理应用中，每个游戏局面的 Grundy 值（g 值）通过 `mex({g(子局面)})` 递推计算。由于多条路径可能到达同一局面，缓存 `memo[state] = grundy_value` 避免了指数级重复计算，这是组合博弈分析的标准实现方式。

**字符串匹配与解析**：正则表达式引擎的递归回溯匹配（如判断模式串 `p` 是否匹配文本 `s`）中，状态为 `(i, j)` 表示文本第 i 位与模式第 j 位的匹配结果，加入记忆化后可将含 `*` 通配符的匹配从指数级降为 O(|s|×|p|)。LeetCode 第10题"正则表达式匹配"是该应用的典型练习题。

**区间 DP**：矩阵链乘法问题中，`dp[i][j]` 表示第 i 到第 j 个矩阵连乘的最小运算次数，状态转移为 `dp[i][j] = min(dp[i][k] + dp[k+1][j] + p[i-1]*p[k]*p[j])`，记忆化搜索让递归实现与自底向上的三层循环完全等价，且代码逻辑更接近数学递推公式。

## 常见误区

**误区一：认为记忆化搜索总是优于迭代DP**。当问题状态空间不稀疏且递归深度很大时，Python 的默认递归深度限制（sys.setrecursionlimit 默认为1000）会导致栈溢出。对于链式递推如 F(10^6)，迭代 DP 是唯一可行选择，强行使用记忆化递归需要手动设置递归栈大小或改写为显式栈。

**误区二：将"可变对象"作为缓存键**。哈希表的键必须是可哈希类型。若状态包含列表或集合（如剩余可用物品集合），直接以 `list` 为键会触发 `TypeError`，必须先转换为 `tuple` 或 `frozenset`。忽略这一点是初学者使用 `@lru_cache` 时最常见的运行时错误。

**误区三：忽略循环依赖导致死锁**。若状态转移图中存在环（如某些图上的搜索问题），记忆化搜索会陷入无限递归。此时需要增加"正在计算"标记（三色状态：未访问/计算中/已完成），在检测到"计算中"状态时特殊处理，例如在强连通分量的 Kosaraju 算法中需要额外的访问标记数组。

## 知识关联

**依赖递归的基础**：记忆化搜索要求对递归调用栈的清晰理解——只有能写出正确的朴素递归解，才能正确识别需要缓存的"状态"是什么。状态定义错误（如遗漏影响结果的参数）会导致缓存命中时返回错误结果，且这类 bug 极难调试，因为函数签名看起来完全正常。

**与动态规划的互补关系**：记忆化搜索与迭代 DP 共享"最优子结构"和"无后效性"两个理论前提，但各自适用不同场景。学习两者的关系时，推荐以"矩阵链乘法"为对照练习：先用三层循环写出迭代 DP 版本，再用记忆化递归改写，对比状态转移方向和代码结构的差异，有助于深化对 DP 状态空间遍历顺序的理解。在竞赛和工程实践中，记忆化搜索常作为"先验证算法正确性，再优化为迭代版本"的快速原型工具。
