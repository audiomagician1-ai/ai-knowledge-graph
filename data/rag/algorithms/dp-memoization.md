# 记忆化搜索

## 概述

记忆化搜索（Memoized Search）是将递归搜索与缓存机制结合的算法技术：在递归过程中，将已计算过的子问题结果存入哈希表或数组，当同一子问题再次被访问时直接返回缓存值，避免重复计算。其本质是自顶向下（Top-Down）的动态规划实现方式，与传统动态规划的自底向上（Bottom-Up）推导方向相反。

记忆化搜索的思想可追溯到1960年代 Donald Michie 提出的"memo function"概念，正式术语"memoization"由他在1968年发表的论文《Memo functions and machine learning》中确立。与朴素递归相比，记忆化搜索能将指数级时间复杂度降低至多项式级——以斐波那契数列为例，朴素递归的时间复杂度为 O(2ⁿ)，引入记忆化后降为 O(n)，空间复杂度为 O(n)。

记忆化搜索在状态空间不规则、状态转移方向复杂（如树形DP、区间DP）的场景中尤为重要。当子问题间存在复杂依赖关系、难以确定自底向上的推导顺序时，记忆化搜索允许直接按照问题的自然递归结构编写代码，由程序运行时自动确定计算顺序。

## 核心原理

### 缓存命中机制

记忆化搜索的核心数据结构是一张映射表 `memo: State → Value`。每次进入递归函数时，首先检查当前状态 `s` 是否已在 `memo` 中存在：若存在则直接返回 `memo[s]`（缓存命中），否则执行递归计算并将结果写入 `memo[s]` 后返回。这一"查询-计算-存储"的三步模式是所有记忆化搜索实现的统一结构。

状态的设计直接决定记忆化的正确性。状态必须能唯一确定子问题的输出，即满足**无后效性**：相同状态在任何调用上下文中必须产生相同结果。以最长公共子序列（LCS）为例，状态定义为 `(i, j)` 表示字符串 A 的前 i 个字符与字符串 B 的前 j 个字符的 LCS 长度，状态空间为 O(m×n)，记忆化后总时间复杂度从 O(2^(m+n)) 降至 O(m×n)。

### 与朴素递归的复杂度对比

以计算 Catalan 数 C(n) 为例，朴素递归的递推式为：

```
C(0) = 1
C(n) = Σ C(i) × C(n-1-i)，i 从 0 到 n-1
```

朴素递归时间复杂度为 O(4ⁿ / n^(3/2))（指数级），而记忆化后每个状态 C(0)~C(n) 仅计算一次，时间复杂度降为 O(n²)，因为共 n+1 个状态，每个状态计算耗时 O(n)。

### 递归栈深度与哈希表选型

记忆化搜索存在两个关键的工程参数：其一是**递归深度限制**，Python 默认最大递归深度为1000层（可通过 `sys.setrecursionlimit` 调整），处理深度达10⁴以上的问题时须警惕栈溢出；其二是**缓存容器的选型**，当状态为整数索引时用数组（O(1) 访问），当状态为元组或复合键时用哈希表（Python 中使用 `dict` 或 `@functools.lru_cache` 装饰器），后者每次访问有常数级哈希开销。

## 实际应用

**区间DP——矩阵链乘法**：设 `dp(i, j)` 表示将第 i 到第 j 个矩阵连乘所需的最少标量乘法次数。该问题的子问题依赖于中间分割点 k 的选取，自底向上推导须严格按区间长度递增的顺序计算；而记忆化搜索直接从 `dp(1, n)` 出发递归，代码结构与数学递推公式一一对应，无需手动管理计算顺序。共 O(n²) 个状态，每个状态枚举 O(n) 个分割点，总复杂度 O(n³)。

**树形DP——树上背包**：给定一棵有 n 个节点的树，每个节点有权值，选取节点数不超过 k 个且所选节点构成连通子树，求最大权值和。状态定义为 `dp(u, j)` 表示以节点 u 为根的子树中选取恰好 j 个节点的最优解。由于树的结构天然递归，记忆化搜索比手写迭代的自底向上方式更直观，且不需要预先确定后序遍历顺序。

**Python `@lru_cache` 实战**：`functools.lru_cache(maxsize=None)` 装饰器可将任意纯函数自动记忆化，内部使用字典存储参数元组到返回值的映射。对于斐波那契数列，加上该装饰器后 `fib(30)` 的调用次数从 2,692,537 次降至 59 次（仅计算每个状态一次再加重复查询）。

## 常见误区

**误区一：认为记忆化搜索和动态规划是两种独立算法**。实际上记忆化搜索是动态规划的自顶向下实现，二者求解的是完全相同的子问题集合，状态转移方程也相同。区别仅在于：记忆化搜索只计算被实际调用到的子问题，而自底向上DP会计算全部状态。当状态空间稀疏时（如棋盘搜索中只有少数格子被访问），记忆化搜索因此具有实际性能优势。

**误区二：缓存键包含不影响结果的参数**。若函数签名中有路径信息（如当前遍历路径）或可变全局状态作为隐式输入，直接使用所有参数作为缓存键会导致缓存失效或错误命中。正确做法是确保缓存键仅包含真正决定子问题输出的最小状态集合。例如在图上的最短路径搜索中，若将已访问节点集合纳入状态，则状态空间从 O(n) 膨胀为 O(n·2ⁿ)，这已不再是记忆化搜索的典型应用场景，而应改用 Dijkstra 等专用算法。

**误区三：对含副作用的函数使用记忆化**。记忆化的正确性前提是**引用透明性**——相同输入必须产生相同输出，且函数不依赖或修改外部状态。若递归函数内部修改了全局变量或累计了计数器，强行添加记忆化会导致第二次调用直接返回缓存值而跳过副作用，产生逻辑错误。

## 知识关联

**与递归的关系**：记忆化搜索以递归为载体，要求递归函数满足相同输入产生相同输出的性质。不具备该性质的递归（如含随机数、依赖外部状态的递归）无法直接应用记忆化。分析递归调用树中重叠子问题的数量，是判断记忆化收益的关键步骤——若调用树为严格二叉树且无重叠（如归并排序的分治），记忆化无法带来复杂度改善。

**与动态规划的关系**：当子问题的依赖图为DAG（有向无环图）且可以确定拓扑序时，自底向上DP通常比记忆化搜索更节省函数调用开销（避免了递归栈和哈希查询的常数因子）；而当状态转移依赖于尚未知晓的子集（如集合DP中子集枚举）或依赖关系复杂难以手动排序时，记忆化搜索的代码可维护性更高。在竞赛编程中，记忆化搜索是实现区间DP、树形DP的首选方式，因为递推顺序的设计错误是自底向上DP中最常见的bug来源之一。
