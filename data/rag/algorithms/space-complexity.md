# 空间复杂度

## 概述

空间复杂度（Space Complexity）描述一个算法在运行过程中**占用内存空间随输入规模 n 增长的变化趋势**，同样使用大 O 记号表示。与时间复杂度衡量操作步骤数不同，空间复杂度专门统计算法执行期间**额外申请的内存量**，包括变量、数据结构、递归调用栈等占用的字节数。

空间复杂度的正式分析框架在 1960–70 年代随计算复杂性理论兴起，Hartmanis 和 Stearns 于 1965 年发表的论文奠定了空间与时间作为两类独立计算资源的理论基础。早期计算机内存极为有限（KB 级别），空间优化与时间优化同等重要；如今尽管内存价格大幅下降，但在嵌入式设备、移动端 AI 推理和大规模分布式训练场景中，空间复杂度仍是决定方案是否可行的硬约束。

在 AI 工程领域，理解空间复杂度尤为关键：训练 GPT-3 时，单批次的激活值（Activation）占用高达数十 GB 显存，这直接源于 Transformer 注意力机制 O(n²) 的空间增长特性。错误估计空间需求会导致 OOM（Out-of-Memory）崩溃，而对空间复杂度的精确分析则能指导梯度检查点（Gradient Checkpointing）等优化策略的选择。

## 核心原理

### 辅助空间 vs. 总空间

分析空间复杂度时必须区分两个概念：
- **输入空间**：存储输入数据本身所占的内存（通常不计入算法自身的空间消耗）
- **辅助空间（Auxiliary Space）**：算法额外申请的工作内存

大多数场景下讨论的空间复杂度指**辅助空间**。例如，原地排序算法（In-place Sort）如堆排序的辅助空间为 O(1)，即只使用常数个额外变量；而归并排序需要一个大小为 n 的临时数组，辅助空间为 O(n)。

### 递归调用栈的空间代价

递归函数每次调用都会在调用栈上压入一个**栈帧（Stack Frame）**，包含局部变量、返回地址等。因此递归深度直接决定空间复杂度。

以经典斐波那契递归为例：

```
fib(n) → 递归深度为 n → 调用栈占用 O(n) 空间
```

而对同一问题使用迭代法（只维护两个变量 `prev` 和 `curr`）则将空间复杂度降至 O(1)。快速排序平均递归深度为 O(log n)，最坏情况（已排序数组 + 固定取首元素为 pivot）退化为 O(n)，这是快速排序空间复杂度不稳定的根源。

### 常见复杂度类别及对应结构

| 空间复杂度 | 典型场景 |
|---|---|
| O(1) | 双指针遍历、原地交换 |
| O(log n) | 平衡二叉树递归、二分查找递归版 |
| O(n) | 哈希表存储、BFS 队列、一维 DP 数组 |
| O(n²) | 二维 DP 表、邻接矩阵存储图 |
| O(n · d) | n 个样本 × d 维特征的数据矩阵 |

在 AI 工程中，Transformer 自注意力机制的 KV Cache 占用 O(n²) 空间（n 为序列长度），这正是长上下文推理面临的核心内存瓶颈。

### 空间-时间权衡（Space-Time Tradeoff）

空间复杂度与时间复杂度之间存在经典的权衡关系：用额外空间"记忆"中间结果，可以避免重复计算从而加速程序。动态规划（Dynamic Programming）是最典型的例子：

- **朴素递归**求解 0/1 背包：时间 O(2ⁿ)，空间 O(n)（递归栈）
- **二维 DP 表**：时间 O(n·W)，空间 O(n·W)
- **滚动数组优化**：时间 O(n·W)，空间压缩至 O(W)（W 为背包容量）

滚动数组将空间从 O(n·W) 降至 O(W) 的关键在于：每一行 DP 值只依赖上一行，旧行可被覆盖复用。这种"空间换时间"或"时间换空间"的决策是算法设计中的核心工程判断。

## 实际应用

**AI 模型训练中的显存估算**：训练一个参数量为 P 的神经网络，以 float32 精度存储参数需要 4P 字节，梯度同样需要 4P 字节，Adam 优化器额外存储一阶矩和二阶矩各 4P 字节，共计约 **16P 字节**。一个 7B（70 亿）参数模型仅这部分显存需求即达 16 × 7×10⁹ ≈ 112 GB，这直接决定了必须采用混合精度训练（将参数降至 bfloat16）或模型并行策略。

**BFS 与 DFS 的空间对比**：在图算法中，广度优先搜索（BFS）需要维护一个队列，最坏情况下队列宽度等于图的最大层宽，空间为 O(w)（w 为最大宽度）；深度优先搜索（DFS）使用递归栈，空间为 O(h)（h 为最大深度）。对于宽而浅的图，DFS 更省空间；对于窄而深的图，BFS 更省空间。

**推理阶段的 KV Cache 管理**：大语言模型推理时，每生成一个 token，KV Cache 增长 `2 × num_layers × num_heads × head_dim × 4 bytes`，随序列长度线性增长。vLLM 框架通过 PagedAttention 将 KV Cache 管理从连续内存改为分页式，使单卡 GPU 吞吐量提升最高 24 倍，这是空间复杂度分析直接驱动系统设计的典型案例。

## 常见误区

**误区一：空间复杂度只算变量，忽略递归栈**

许多初学者在分析递归算法时，只统计函数内声明的变量，忘记每层递归调用本身占据的栈帧空间。例如递归版归并排序，除了 O(n) 的临时数组外，还有 O(log n) 的递归栈，但由于 O(n) 主导，总体仍是 O(n)。对于**只有递归没有额外数组**的算法（如递归版 DFS），新手常误判空间复杂度为 O(1)，实为 O(h)。

**误区二：混淆"原地"与"O(1) 空间"**

"原地算法"并不严格等于 O(1) 空间。例如快速排序通常被称为原地排序，但其递归调用栈消耗 O(log n) 至 O(n) 的空间，并非常数空间。真正的 O(1) 空间算法（如用于链表反转的双指针）在任意输入规模下仅使用固定数量的变量。

**误区三：认为空间复杂度越低越好**

在 AI 推理工程中，过度追求低空间占用有时反而损害性能。例如梯度检查点（Gradient Checkpointing）通过不保存中间激活值将训练显存从 O(n) 降至 O(√n)，但代价是需要重新计算被丢弃的激活值，训练时间增加约 30%~40%。空间复杂度的优化目标应结合实际资源约束（显存上限、带宽）与时间代价综合权衡，而非单纯最小化。

## 知识关联

**与时间复杂度的联系**：时间复杂度和空间复杂度共用大 O 记号体系，分析方法（求最高阶项、忽略常数系数）完全相同，因此掌握时间复杂度的 Big-O 分析是理解空间复杂度的直接前提。两者的本质区别在于：时间复杂度计数**操作次数**，而空间复杂度计数**同时存活的内存单元数**——特别注意"同时"二字，已释放的内存不计入峰值空间。

**延伸方向**：空间复杂度分析是学习动态规划空间优化（滚动数组、状态压缩）、Transformer 显存优化（Flash Attention 将注意力计算的空间从 O(n²) 降至 O(n)）以及分布式训练内存管理（ZeRO 优化器的三阶段分片策略）的直接基础。掌握本节内容后，面对任何算法时应形成"时间/空间双维度评估"的分析习惯。
