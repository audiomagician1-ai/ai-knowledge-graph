# 代理资产

## 概述

代理资产（Proxy Asset）是在关卡设计灰盒阶段用于替代最终美术资产的临时占位模型，其主要目的是在正式美术资源尚未完成时，为设计师提供具有正确体积、碰撞体和基础视觉参考的工作占位符。与直接用几何体堆砌的纯粹灰盒不同，代理资产通常拥有独立的命名规范、分类标签和碰撞配置，使其能够被资产管道（Asset Pipeline）识别并在适当时机自动替换为正式资产。

代理资产的概念随着大型游戏工程的模块化生产流程普及而逐渐标准化，在育碧（Ubisoft）的《刺客信条》系列和DICE的《战地》系列等大型多团队协作项目中，代理资产系统被发展为跨团队沟通体量与功能的核心工具。其命名规约如 `PROXY_SM_Crate_01` 或 `PH_Barrel_Large` 等前缀标记方式，帮助设计团队与美术团队在同一个场景文件中清晰区分"待替换占位"与"最终资产"。

使用规范化的代理资产而非随意放置的基础几何体，能够大幅降低后期资产替换时产生的碰撞错误、比例失真和场景破损风险，同时也为程序员提供稳定的碰撞预设，支持早期游戏性测试在近似真实的物理环境下进行。

## 核心原理

### 命名规范与前缀系统

代理资产的文件命名必须包含可被管线脚本识别的专用前缀，常见的行业惯例是使用 `PROXY_` 或 `PH_`（Placeholder 缩写）作为前缀，配合资产类型缩写和唯一编号。例如一个用于代替大型货柜的代理资产应命名为 `PROXY_SM_Container_Large_01`，其中 `SM` 表示静态网格（Static Mesh），`Large` 描述尺寸变体，`01` 为序号索引。这套命名格式使得在 Unreal Engine 的内容浏览器或 Unity 的 Project 面板中，团队成员能够通过搜索 `PROXY_` 一键筛选出场景内所有未被替换的临时资产，并在迭代末期执行批量替换操作。

### 体积精度与碰撞配置

代理资产的几何体积误差应控制在最终资产预估尺寸的 ±5% 以内，否则碰撞体偏差会导致玩家移动测试数据失去参考价值。每个代理资产需要配置简化碰撞（Simple Collision）而非逐多边形碰撞（Complex Collision），通常以凸包（Convex Hull）或盒体（Box Collision）表示，这样既能支持物理模拟测试，又能保持场景运行帧率在可玩范围内（通常保持在目标平台帧率的 80% 以上，以留出美术资产的性能预算空间）。碰撞层级和通道设置必须在代理阶段就按照最终资产的预期行为配置完毕，例如可破坏道具的代理资产需要标记 `Destructible` 通道，以免替换后破坏已调试好的游戏逻辑。

### 材质与视觉标记系统

代理资产应使用统一的标记材质（Marker Material），而非未经处理的引擎默认灰色材质。常见做法是为不同功能类别的代理资产分配不同的纯色材质：掩体类使用橙色（HEX `#FF6600` 等醒目色），可交互道具使用黄色，背景装饰使用蓝色，危险区域使用红色。这种颜色编码系统在多人协作时能让关卡设计师在 3D 视口中即时判断场景功能密度分布，而无需逐一检查对象标签。此外，部分团队会在代理资产表面通过 UV 投射烘焙出资产名称的文字信息，以便在截图或录屏的设计评审（Design Review）会议中快速识别占位情况。

### 元数据与替换映射表

代理资产系统的完整实现需要维护一份**替换映射表**（Swap Manifest），这是一个将代理资产 ID 与最终美术资产路径对应起来的配置文件，格式通常为 JSON 或 CSV。一旦美术资产完成制作，管线工具读取映射表，自动将场景中所有匹配的代理资产替换为对应正式资产，同时保留原有的 Transform（位置、旋转、缩放）数据。Unreal Engine 的 Level 编辑器中，这一流程可通过 Python 脚本或 Editor Utility Widget 实现自动化批量替换，将原本需要数天的手动替换工作压缩至数分钟内完成。

## 实际应用

在《战地 2042》等大型 FPS 关卡的生产流程中，关卡设计师在灰盒完成功能验证后，会将场景中的原始 BSP 或简单立方体逐步替换为对应功能的代理资产库（Proxy Asset Library）。该库通常包含 80～200 个常用道具的代理版本，涵盖弹药箱、载具、建筑门洞、掩体矮墙等高频元素。设计师从库中拖入代理资产后，场景即具备近似最终版本的体积关系和碰撞环境，此时可以邀请游戏测试人员进行功能性可玩测试（Functional Playtest），而无需等待美术资产全部完成，从而节省数周的等待时间。

在单机关卡开发中，独立开发者也可以使用 Kenney 等提供的免费占位资产集作为简化版代理资产系统，只需在导入时统一添加 `PH_` 前缀并创建一个手动映射表，便能在后期一键切换为自制美术资源。

## 常见误区

**误区一：将代理资产与灰盒几何体混用**
许多初学者认为直接在场景中放置引擎自带的立方体（Cube Mesh）与使用代理资产效果相同。实际上，原始几何体没有语义化命名、没有分类碰撞配置，也不在任何映射表中登记，无法被管线脚本识别和批量替换。在项目后期，设计师将不得不手动检索并逐一替换数百个匿名立方体，产生大量额外工时。

**误区二：代理资产的碰撞体可以在替换时再调整**
部分设计师认为碰撞配置属于美术资产的工作范畴，代理阶段不需要认真处理。这会导致在代理阶段通过测试的玩家移动路径、投掷物弹道、AI 导航数据，在美术资产替换后因碰撞体形状改变而全部失效，需要重新进行完整轮次的游戏性测试，代价极高。

**误区三：代理资产必须与最终资产外形相似**
代理资产的首要职责是提供正确的**体积和碰撞**，而非视觉外观的近似。过度追求代理资产的外形准确性会让设计师在灰盒阶段花费不必要的时间在美术细节上，违背了代理资产快速迭代的设计初衷。一个正确尺寸的橙色盒体，完全足以替代一个尚未建模的弹药箱，只要其长宽高和功能类别标签是准确的即可。

## 知识关联

代理资产是灰盒工作流（Blockout Workflow）的直接延伸——灰盒阶段使用原始几何体验证空间尺度，而代理资产阶段则将这些几何体升级为具有标准化管理能力的占位单元，标志着关卡从"测试原型"向"可交付中间产物"的过渡。掌握代理资产规范，设计师需要理解所在项目的资产命名约定（Asset Naming Convention）和使用的引擎的批处理脚本能力，例如 Unreal Engine 的 Python API 或 Unity 的 Editor Script，才能完整实现自动替换流程而非停留在手动管理层面。代理资产管理能力也直接影响团队的**并行生产效率**：规范的代理资产系统使美术、设计、程序三条生产线能够真正同时推进，而不是形成依赖等待链。
