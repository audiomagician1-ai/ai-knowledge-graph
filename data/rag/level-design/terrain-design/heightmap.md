# 高度图

## 概述

高度图（Heightmap）是以单通道灰度图像格式存储地形垂直高度信息的二维数据结构，其核心特征在于：每一个像素的亮度值与三维地形中对应位置的海拔高度之间存在严格的线性映射关系——像素值 $0$（纯黑）对应地形最低点，像素值 $255$（8位）或 $65535$（16位）对应地形最高点。这种以"亮度即高度"为基本编码规则的数据格式，使得复杂的三维地形信息可以被压缩为一张单通道 PNG、RAW 或 GeoTIFF 文件进行存储、传输和版本管理。

高度图技术最早系统化地应用于地理信息科学领域，美国地质调查局（USGS）在 1970 年代开始以数字高程模型（Digital Elevation Model，DEM）的形式发布地形数据，其底层存储格式即为高度图的前身（Maune, 2007）。游戏引擎对高度图的引入发生在 1990 年代中期：1996 年发布的《Outcast》是最早使用高度图渲染大规模实时地形的商业游戏之一，其自研的体素渲染管线直接读取 8 位 RAW 高度图数据。此后，Epic Games 于 Unreal Engine 2 中正式将高度图确立为 Landscape 系统的基础数据格式，Unity Technologies 则在 Unity 2.x 时代引入了基于高度图的 Terrain 组件。

在现代游戏关卡设计工作流中，高度图扮演着"地形骨架"的角色——它定义了地表的宏观起伏结构，材质纹理、植被散布、道路曲线等所有后续层次的设计均以高度图所建立的地形轮廓为基准。

## 核心原理

### 灰度值与高度的线性映射

高度图的基本数学关系可表示为：

$$H = \frac{G}{G_{\max}} \times H_{\max}$$

其中 $H$ 为该像素对应的实际地形高度（单位：引擎世界单位或米），$G$ 为该像素的灰度值，$G_{\max}$ 为当前位深下的最大灰度值（8位时为 $255$，16位时为 $65535$），$H_{\max}$ 为地形组件设置的高度上限。

以 Unity 为例，若 Terrain 组件的 Terrain Height 设为 $600$ 米，一个灰度值为 $32768$ 的 16 位像素对应的实际高度为 $(32768 / 65535) \times 600 \approx 300.0$ 米。8 位高度图仅提供 $256$ 个离散高度级别，在 $600$ 米高度范围下相邻高度级之间的间距约为 $2.35$ 米，这会在平缓坡面上产生肉眼可见的"台阶状"锯齿（Stairstepping Artifact）。16 位高度图提供 $65536$ 个级别，同等条件下级间距缩小至约 $0.009$ 米，完全消除了台阶问题，这也是专业项目普遍采用 16 位 RAW 格式的根本原因（Sherrod & Jones, 2008）。

### 分辨率、顶点密度与精度权衡

高度图的分辨率直接决定了地形网格的顶点密度（水平精度）。Unity Terrain 要求高度图分辨率必须满足 $2^n + 1$ 的规格，常用值包括 $257 \times 257$、$513 \times 513$、$1025 \times 1025$、$2049 \times 2049$ 和 $4097 \times 4097$。这一限制来源于地形的 LOD 细分机制：相邻 LOD 层级之间网格顶点数减半，$2^n + 1$ 的规格确保每一层 LOD 都能精确对齐父级网格的顶点位置而不产生 T 型接缝（T-Junction）。

以 $1025 \times 1025$ 高度图映射到 $1000 \times 1000$ 米地形为例，相邻顶点水平间距约为 $0.98$ 米，即宽度小于 $1$ 米的细节（窄沟、石脊）在该分辨率下无法被高度图正确表达，必须依赖网格叠加（Mesh Overlay）方式补充。若将分辨率提升至 $4097 \times 4097$，16 位 RAW 文件体积约为 $4097 \times 4097 \times 2 \approx 33.6$ MB，加载到内存后 GPU 顶点缓冲区的占用量也等比增长，因此大型开放世界项目通常将地形划分为多个 Tile，每个 Tile 使用独立的 $1025 \times 1025$ 高度图管理，Ubisoft 在《刺客信条：奥德赛》的开放世界流送系统中即采用了此策略。

### Unreal Engine 中的高度图编码差异

Unreal Engine 的 Landscape 系统对高度图数据有独特的编码规则，与 Unity 存在显著差异：UE 使用 **有符号 16 位整数**（Signed 16-bit，范围 $-32768$ 至 $32767$）而非无符号整数存储高度，中间高度对应值 $32768$（即像素值 $0.5$ 归一化后乘以 $65535$ 取整）。因此，在从外部工具导出 RAW 格式高度图时，必须区分目标引擎的字节序（Byte Order）要求：Unity 使用 Little-Endian（小端序），而 UE 的 Landscape 导入界面提供 Byte Order 切换选项，若字节序设置错误，导入后地形将呈现为随机噪波状的错误高度分布，这是跨引擎迁移高度图时最常见的技术陷阱之一。

## 关键方法与工具链

### 手工绘制高度图

在 Photoshop 中绘制高度图时，必须将图像模式设置为"灰度"（Grayscale）而非 RGB，并选择 16 位/通道色深（Image → Mode → 16 Bits/Channel）。使用硬边圆形笔刷绘制的山脊会形成坡面上的锯齿，应改用大直径柔边笔刷并配合多次叠加构建自然坡度。导出时选择 Photoshop RAW（.raw）格式，注意区分"Photoshop RAW"与"Camera RAW"——前者是纯二进制像素数据，后者是相机元数据格式，两者完全不同。

### World Machine 程序化生成

World Machine（Stephen Schmitt 开发，商业版支持 $8192 \times 8192$ 分辨率）是游戏行业最广泛使用的程序化地形生成工具，其工作流程为：Noise 生成器（如 Advanced Perlin）→ 侵蚀模拟（Erosion）→ 后处理（Expander/Clamp）→ 输出节点。World Machine 的侵蚀算法模拟水流冲刷与沉积过程，能够自动生成真实感的山谷、冲积扇和山脊细节，其输出的 `.r16` 文件可直接被 Unity 识别为 16 位 RAW 高度图。Gaea（QuadSpinner 开发）是另一款主流工具，其节点图界面与 UE 的 Blueprint 视觉风格接近，且支持直接输出 UE Landscape 规格的高度图。

### 真实地理数据转换

从 USGS EarthExplorer（earthexplorer.usgs.gov）可下载全球覆盖的 SRTM 1弧秒数据（水平精度约 30 米），数据格式为 GeoTIFF，需经过以下处理步骤才能导入游戏引擎：

1. **重投影**：使用 GDAL 命令 `gdalwarp -t_srs EPSG:32650 input.tif reprojected.tif` 将地理坐标系（WGS84）转换为平面坐标系（如 UTM）；
2. **重采样**：使用 `gdalwarp -ts 1025 1025 reprojected.tif resampled.tif` 调整至目标分辨率；
3. **归一化与导出**：在 QGIS 或 Python（rasterio 库）中将高度值线性缩放至 $[0, 65535]$ 并导出为 16 位 RAW。

Terrain.party（terrain.party）提供了一个更便捷的前端界面，允许在地图上框选任意矩形区域后直接下载已处理好的 PNG 高度图，适合快速原型阶段使用。

## 实际应用

### 在 Unreal Engine 中导入高度图

在 UE5 中创建 Landscape 时，于 Import from File 选项下选择 .r16 或 .png 文件，引擎会自动检测分辨率并建议 Landscape 的 Section Size 与 Sections Per Component 组合配置。以 $2017 \times 2017$ 高度图为例，UE 会推荐 $63 \times 63$ 的 Section Size 配合 $32 \times 32$ 的 Sections Per Component 组合，这一组合使地形总顶点数为 $2017 \times 2017 \approx 400$ 万个，在 RTX 3070 级别显卡上能够维持 60 FPS 的渲染性能（在无其他场景元素情况下）。

### 高度图与地形材质的协同

高度图定义几何高度，但地表外观由地形材质中的 **Layer Blend** 节点根据高度数据驱动。例如，可以通过世界空间高度值 $H > 800m$ 触发岩石材质层，$300m < H < 800m$ 使用草地材质，$H < 300m$ 使用泥土材质。这种基于高度的自动材质分层方式，要求高度图必须精确定义山顶与山谷的高度绝对值，而不仅仅是相对起伏形状。

**案例**：《地平线：零之曙光》（Guerrilla Games, 2017）的美术团队公开分享了其工作流程：先通过 Houdini 生成程序化高度图确定区域宏观地貌，再手工雕刻关键视角的轮廓线，最后通过高度数据自动散布岩石与植被密度贴图，将高度图的角色从单纯几何数据扩展为整个场景生态的"主控数据层"（van Dijk et al., GDC 2017）。

## 常见误区

**误区一：将高度图的像素值视为绝对高度单位。** 高度图本身不携带任何绝对高度信息，像素值 $128$ 对应的实际高度完全取决于引擎中 Terrain Height 参数的设置。同一张高度图在不同 Height 设置下会生成截然不同的地形起伏幅度，设计师必须在引擎侧统一约定 Height 基准值后再进行高度图的传递与交接。

**误区二：用 PNG 格式保存高度图时不损失精度。** 标准 PNG 支持 16 位灰度，但很多图像处理软件（如早期版本的 GIMP）在保存灰度 PNG 时会静默降级为 8 位，导致 $65536$ 个高度级别被压缩为 $256$ 个，在导入后的地形上产生台阶锯齿却不报告任何错误信息。验证方式：在 Photoshop 中打开导出的 PNG，检查 Image → Mode 是否显示"16 Bits/Channel"。

**误区三：高分辨率高度图等于高质量地形。** $4097 \times 4097$ 的高度图在 $10 \times 10$ 公里的大地图上顶点间距约为 $2.4$ 米，但若高度图本身来源于 30 米精度的 SRTM 数据，实际有效信息密度只对应 $334 \times 334$ 的分辨率，多余的像素都是通过插值填充的伪精度，既占用内存又不提供真实地形细节。

**误区四：在正式项目中使用 World Machine 免费版生成高度图。** World Machine 免费版（Basic Edition）的最大输出分辨率限制为 $1009 \times 1009$，这不符合 $2^n + 1$ 规格要求（最近的合法值为 $1025$），若直接将其导入 Unity，引擎会提示分辨率不兼容并拒绝导入。
