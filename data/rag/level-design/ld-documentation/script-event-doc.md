# 脚本事件文档

## 概述

脚本事件文档（Scripted Event Document）是关卡设计文档体系中专门记录关卡内所有触发事件的时序排列与逻辑条件的文本。它的核心任务是将关卡设计师脑海中的"剧情推进节奏"转化为程序员和关卡编辑器可以直接执行的指令序列，确保玩家在经过某个位置、完成某个动作或满足某个条件时，游戏世界会以精确的方式做出响应。

脚本事件文档的概念随着关卡设计职业化在1990年代逐渐成形。早期作品如《半条命》（1998年）的设计团队Valve内部使用了大量手写的"事件触发表"来协调各个脚本序列，这被认为是现代脚本事件文档的雏形之一。随着引擎工具（如Unreal Engine的Kismet，后演变为Blueprint）的成熟，文档格式也从简单的线性列表发展为包含条件分支和并行线程的结构化表格。

这份文档对关卡制作流程至关重要的原因在于：一个中等规模的关卡通常包含20至80个独立触发事件，若缺乏统一的文档记录，这些事件的执行顺序很容易在多人协作中出现冲突——例如过场动画尚未播完，AI就开始战斗行为，导致体验断裂。脚本事件文档是防止此类冲突的唯一单一信息来源（Single Source of Truth）。

## 核心原理

### 事件的四要素结构

每一条脚本事件记录必须包含四个要素：**触发器（Trigger）**、**条件（Condition）**、**行为（Action）**、**优先级（Priority）**。触发器定义了事件"被谁激活"，通常是碰撞体进入区域（OnEnter）、计时器归零（OnTimerExpire）或变量达到阈值（OnFlagSet）；条件是追加的逻辑过滤，例如"仅当玩家血量低于30%时才触发警报音效"；行为是触发后执行的具体指令集；优先级（通常用1～5的整数表示，1为最高）决定了多个事件同时触发时的执行顺序。

### 时序标注方法

脚本事件文档使用两种时间基准来描述事件发生的位置：**空间触发点**（以关卡坐标或命名区域表示，如"Sector_B_Entrance"）和**相对时间偏移**（以毫秒为单位，标注在前一事件完成后的延迟量）。例如，一条完整的记录可能写作：

> 事件ID: EVT_023 | 触发器: OnEnter(Zone_Bridge) | 条件: QuestFlag_03 == TRUE | 延迟: 1500ms | 行为: PlayCutscene(CS_BridgeAmbush) + DisablePlayerControl | 优先级: 1

这种写法让程序员无需与设计师反复确认，即可独立实现该事件。文档中通常还会为每个事件标注**一次性/循环**标志（OneShot / Repeatable），避免过场动画被反复触发。

### 依赖链与冲突检测

复杂关卡中的事件并非孤立存在，而是形成**依赖链（Dependency Chain）**。脚本事件文档必须用有向图或缩进式列表显式标出这些依赖关系：事件B只有在事件A成功执行后才能被激活，事件C与事件D互斥（同一时刻只能有一个处于激活状态）。文档撰写者需要在填写每条记录时同步填写"前置事件ID"和"互斥事件ID"两个字段，这是冲突检测的主要数据来源。在QA阶段，测试人员会按照文档中的依赖链逐一验证，将实际执行结果与文档预期进行比对。

## 实际应用

**战斗遭遇编排**：在《使命召唤》系列的线性关卡设计中，脚本事件文档会精确记录每波敌人的生成时机——例如"玩家摧毁第一辆装甲车后延迟2000ms，从北侧建筑Spawn Point_NW-04生成4名敌人"。这种毫秒级的时序控制使战斗节奏感可以被设计和复现，而不依赖于运行时的随机性。

**环境叙事事件**：开放世界关卡中，设计师用脚本事件文档记录那些"仅发生一次"的环境演出事件（World Event），例如玩家第一次进入某个村庄时触发的NPC对话序列。文档中会注明该事件绑定的全局存档变量（如`World_Flags[Village_Intro_Played] = 1`），确保即使玩家退出重进后也不会二次触发。

**难度自适应触发**：部分文档还会记录与难度系统联动的条件事件，例如在简单难度下跳过某段挑战性平台序列的触发器，并注明哪个DifficultyManager枚举值对应哪个分支路径，让程序员直接按文档实现而无需猜测设计意图。

## 常见误区

**误区一：将事件行为写入关卡设计文档（GDD）而非单独的脚本事件文档**。GDD描述关卡的整体设计意图和玩法目标，而脚本事件文档专门处理执行层面的时序细节。把两者混写会导致程序员在数十页的GDD中寻找触发逻辑，严重降低沟通效率。规范的做法是GDD中只写"此处应触发一段Boss登场的演出"，具体的坐标、延迟、Flag名称全部集中到脚本事件文档的对应条目。

**误区二：认为脚本事件文档只需要记录"正常路径"的事件**。玩家行为往往偏离设计预期——他们可能在过场动画期间死亡、在事件窗口内快速反复出入触发区域，或以非预期顺序完成任务。脚本事件文档必须包含**异常状态处理（Exception Handling）**列，例如注明"若玩家在CS_BridgeAmbush播放期间死亡，则重置EVT_023为未触发状态，并清除Zone_Bridge内已生成的实体"。

**误区三：把优先级字段留白**。当两个事件在同一帧内被触发时（这在高速移动的玩家操作中并不罕见），引擎需要优先级数值来决定执行顺序。留空优先级字段会导致引擎按照事件ID的字母顺序或内存地址执行，产生非确定性行为，且该类Bug极难在QA阶段复现。

## 知识关联

脚本事件文档建立在**LD文档概述**所确立的文档规范框架之上，复用了其中关于命名约定、版本控制和评审流程的基础规则——例如LD文档概述中定义的"事件ID需以EVT_前缀+三位数字编码"这一规范，直接被脚本事件文档的每一条记录所使用。

在实际制作流程中，脚本事件文档与**关卡流程图（Level Flowchart）**存在强映射关系：流程图中的每一个菱形判断节点，在脚本事件文档里都应能找到对应的Condition字段；每一个矩形执行节点对应Action字段。两份文档交叉审查可以发现流程图中遗漏的事件或文档中多余的孤立事件。

此外，脚本事件文档的输出结果直接驱动**QA测试用例**的编写——测试工程师会以文档中每个事件的触发条件作为测试输入，以Action字段的预期行为作为断言目标，建立一一对应的自动化验证用例，使脚本事件文档不仅是设计工具，也成为质量保障体系的有效组成部分。
