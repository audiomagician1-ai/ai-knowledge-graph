# 测试报告

## 概述

测试报告（Playtest Report）是关卡设计流程中记录可玩性测试结果的标准文档格式，其核心职能是将测试期间观察到的玩家行为数据、卡关位置、情感反应和完成率等信息系统化地转化为可操作的设计改进依据。与普通笔记不同，测试报告采用结构化模板，确保每轮测试的数据具有横向可比性，使设计师能够追踪同一关卡在迭代修改前后的具体变化。

测试报告作为正式文档类型在游戏开发工业化进程中逐渐标准化，约在1990年代中期随着大型工作室质量保证（QA）流程的建立而普及。早期关卡测试结果往往仅凭口头反馈流转，导致问题无法追溯；标准化测试报告的出现解决了这一问题，使独立完成的多场测试可以被汇总分析。对于关卡设计师而言，测试报告是原型测试阶段产出的直接文字凭证，也是向制作人或主策划申请修改资源时的核心依据。

## 核心原理

### 测试报告的必填字段

一份合规的关卡测试报告至少包含以下固定字段：**测试日期与版本号**（例如"2024-03-15 / Build 0.4.2"）、**测试人数与玩家背景**（核心玩家/轻度玩家/首次接触该类型）、**关卡完成率**（百分比形式，如"8/10人完成，完成率80%"）、**平均通关时长**（与设计目标时长对比，如"设计目标8分钟，实测均值14分钟"）、**死亡热图描述或截图**，以及**测试员逐条观察记录**。缺少版本号会导致无法判断某个问题是否已在后续版本中修复，这是新手设计师最常遗漏的字段。

### 定量数据与定性数据的结合方式

测试报告的核心技术在于将数字数据与行为描述配对记录。定量数据包括：特定机关处的死亡次数、某条路径的选择比例（如"左侧路径被78%的玩家优先选择"）、拾取道具的时机分布。定性数据则记录玩家的口头表述和可观察行为，例如"3名玩家在第二个平台跳跃处停顿超过30秒并发出困惑表情"。两类数据必须在报告中并排出现，仅有数字无法说明问题成因，仅有文字描述则缺乏说服力。

### 问题分级标注系统

测试报告中发现的问题需按严重程度分级，通用做法是采用三级标注：**P1（阻断级）**指导致玩家完全无法推进的设计缺陷，如路径逻辑断裂；**P2（体验级）**指显著降低游玩乐趣但不阻断流程的问题，如某段战斗节奏过于平坦；**P3（优化级）**指细节打磨类问题，如提示文字位置不佳。每个P1问题必须在报告的"结论"部分单独列出并附带修改建议，而不能仅埋藏在观察记录段落中。

### 测试报告与原始测试笔记的区别

原型测试阶段产生的是散乱的实时笔记，而测试报告是对这些笔记的二次加工文档，需要在测试结束后24小时内完成整理。报告撰写者（通常是负责该关卡的设计师）需主动过滤无关信息、归纳重复出现的模式，并将观察结论与关卡的原始设计意图（来自关卡设计文档）进行对比标注，指出哪些设计目标已达成、哪些出现偏差。

## 实际应用

在一款横版动作游戏的第三关测试中，测试报告实际呈现如下：设计师设定该关卡目标时长为10分钟，测试后报告显示10名测试者平均耗时23分钟，其中7人在关卡中段的"齿轮房间"平均停留11分钟。报告在P1栏目中记录："齿轮房间机关触发条件视觉提示不足，10名测试者中9名未能在首次尝试时理解踩踏顺序，需在地板上添加高亮序号或增加引导性NPC提示。"这条记录直接触发了下一个迭代版本中视觉引导元素的增加。

在开放世界关卡测试中，测试报告的地图标注功能尤为关键。设计师在关卡俯视图截图上用红点标记死亡密集区、蓝点标记玩家长时间停留区、绿点标记玩家忽略的隐藏路径入口，并将标注图作为附件纳入报告正文。这种可视化方式使制作人在5分钟内即可定位问题集中区域，无需阅读全部文字。

## 常见误区

**误区一：将测试报告等同于Bug报告。** 测试报告记录的是可玩性体验层面的问题，例如难度曲线失衡、引导不清、关卡节奏拖沓，这些并非程序错误。Bug报告（LD专项）处理的是碰撞体积穿模、触发器不响应等技术性故障。将两类问题混写进同一份文档会导致技术部门和设计部门在处理优先级上产生混乱，标准实践是两份文档分开提交。

**误区二：测试报告应记录设计师自己的主观判断。** 测试报告的价值在于记录**玩家实际行为**，而非设计师对自己作品的评价。写"我认为这个跳跃平台难度合理"是无效记录；写"4名轻度玩家在此处连续失败超过5次后主动放弃继续尝试"才是有效数据。设计师的分析与修改建议应集中写在报告末尾的"建议"字段，与观察事实段落严格分开。

**误区三：一次测试对应一份报告，后续测试无需参照前版本。** 正确做法是在新版本测试报告的开头注明"本次测试针对v0.3.1报告中P1问题#03修改后的验证"，并在结论中明确写出"该问题已解决/部分改善/未见改善"。测试报告的连续性比单次报告的完整性更有价值，跨版本追踪是测试报告体系能否真正驱动设计改进的关键所在。

## 知识关联

测试报告建立在**LD文档概述**所确立的文档规范框架之上，其版本号字段与关卡设计主文档的版本管理系统直接对应，确保两份文档能够互相索引。测试报告的数据来源依赖**原型测试**阶段的观察方法论——若测试者未经训练地进行测试观察，报告中的定性数据质量将大打折扣，因此原型测试技能是填写有效测试报告的前置能力。

在后续文档体系中，测试报告会直接生成两类下游文档：**反馈日志**从测试报告中提取玩家的语言反馈，按主题归类存档，形成可跨多个关卡横向比较的长期数据库；**Bug报告（LD）**则从测试报告的观察记录中筛选出属于技术故障的条目，转交程序部门处理。三份文档形成一套完整的测试后处理链条，测试报告是这条链条的起点文档。
