# 导出格式规范

## 概述

导出格式规范是指在DAW中将游戏音乐混音工程渲染为最终音频文件时，所需遵守的文件格式、采样率、位深度和声道配置的标准组合。与影视配乐或流媒体音乐不同，游戏音频文件必须在运行时被引擎实时加载和解码，因此导出参数的选择直接影响包体大小、内存占用和播放延迟三项关键指标之间的平衡。

游戏音频导出规范的标准化始于2000年代中期，随着Xbox 360和PS3世代的到来，开发者开始系统性地区分"母带处理后的存档格式"与"引擎导入格式"两个概念。在此之前，许多开发团队直接将制作用WAV文件打包进游戏，导致光盘容量大量浪费。Wwise和FMOD等音频中间件的普及，将格式转换工作从DAW端后移到了SoundBank构建阶段，但DAW端的初始导出规范仍然是整条音频制作流水线的源头质量保障。

理解导出格式规范的重要性在于：一旦DAW以错误参数导出（例如以44100 Hz导出原本需要48000 Hz的项目），后续的格式转换会引入重采样误差，而这种误差在OGG或OPUS的有损编码阶段会被进一步放大，最终呈现为高频段的模糊感或低频的相位偏移。

## 核心原理

### 文件格式选择：WAV、OGG 与 OPUS 的适用场景

**WAV**（Waveform Audio File Format）是PCM无损格式，作为DAW导出的中间格式使用时，它保留了全部母带处理后的音频数据，文件体积公式为：`文件大小(MB) = 采样率 × 位深 × 声道数 × 时长(秒) ÷ 8 ÷ 1048576`。以一段48000 Hz / 24-bit / 立体声 / 60秒的游戏BGM为例，WAV文件约为16.5 MB，通常仅用于交付给引擎工程师或SoundBank工具进行下一步压缩，不直接打包入游戏安装包。

**OGG Vorbis** 是游戏行业最广泛使用的有损压缩格式，品质系数（quality factor）范围为 -1 到 10，游戏BGM通常设置在 5 到 7 之间（对应约160～220 kbps），此范围在普通游戏扬声器和耳机上已与WAV原文件难以区分。Unity引擎默认将OGG作为PC/Mac平台的音乐流式播放格式，因此从DAW直接导出OGG时，应确保品质参数与Unity的Audio Compression Format设置相匹配，避免二次编码损耗。

**OPUS** 是较新的有损编码标准，由Xiph.Org基金会于2012年正式发布（RFC 6716），在相同比特率下其音质优于OGG Vorbis约20%～30%，且在32 kbps超低码率下仍能保持可接受的游戏音效质量，特别适合手游和网页游戏的流量敏感场景。Godot 4.0开始将OPUS作为首选流式音频格式，而部分DAW（如Reaper）可直接通过插件导出OPUS；其他DAW则通常需要先导出WAV后借助FFmpeg进行转码。

### 采样率：48000 Hz 与 44100 Hz 的抉择

游戏项目的标准采样率为 **48000 Hz**，而非音乐CD使用的44100 Hz。原因在于：视频游戏的音频引擎普遍从电影/视频技术继承，其内部混音总线以48 kHz为基准时钟。若将44100 Hz的WAV导入使用48 kHz时钟的Wwise工程，引擎在实时播放时必须执行插值重采样，在多轨同时播放时累积显著的CPU开销。Unity在Inspector中对44100 Hz文件的"Resample"警告正是这一机制的外在表现。

移动平台（Android/iOS）存在例外：部分低端Android设备的硬件DSP原生支持44100 Hz，此时使用48 kHz反而触发软件重采样。针对这类平台，项目规范通常会单独指定一套44100 Hz的导出配置。

### 位深度：16-bit 与 24-bit 的分工

**24-bit** 是DAW导出中间WAV文件的标准选择，其动态范围为144 dB（理论值），足以保留母带处理后的全部细节，且为下游压缩编码提供足够的量化余量。**16-bit** 的动态范围为96 dB，适用于最终打包入游戏且不再经过SoundBank工具二次处理的场合，例如直接在Godot或Pygame项目中引用的音效文件。

绝对不建议以32-bit浮点WAV格式作为游戏音频的交付格式——尽管DAW内部运算使用32位浮点，但导出的32-bit float WAV文件体积是24-bit的1.33倍，而多数游戏音频中间件在导入32-bit float文件时会先将其转换为整数格式再编码，增加了一次不必要的精度转换。

### 声道配置：立体声与单声道的策略

游戏BGM几乎全部采用 **立体声（Stereo）** 导出，因为空间感和宽度是BGM情绪表达的重要组成部分。然而，环境音效（Ambience）和特定UI音效通常需要导出为 **单声道（Mono）**，以便3D音频引擎（如Wwise的Attenuation和HRTF系统）对其进行空间定位计算——立体声文件在3D定位处理中会产生声像折叠问题，导致定位精度下降。

对话配音（VO）的导出规范通常为单声道44100 Hz / 16-bit WAV，这一组合是Wwise官方文档中对VO资产的推荐基准参数。

## 实际应用

在Reaper中为Switch平台导出游戏BGM时，一套典型的规范如下：格式WAV、采样率48000 Hz、位深24-bit、立体声、峰值不超过 -1 dBFS（为OPUS编码器预留0.5 dB削波保护余量）。导出后，音频技术团队使用Wwise将这批WAV文件编码为Vorbis（品质7）并构建SoundBank，最终打包进Switch的NCA格式安装包。DAW端严格遵守48000 Hz导出，避免了Wwise的实时重采样负担，在Switch有限的CPU资源下节约了约3%～5%的音频线程占用率。

手游项目中，主策划通常要求DAW工程师同时提交两版导出：一版48000 Hz / 24-bit WAV用于高质量存档，一版44100 Hz / 16-bit WAV用于低端Android设备适配包，两套文件通过项目管理工具（如Perforce）分别归档至`/audio/master/`和`/audio/mobile_low/`目录。

## 常见误区

**误区一：OGG导出品质越高越好。** OGG quality 10（约500 kbps）与quality 7（约220 kbps）在游戏实机环境（扬声器、蓝牙耳机）下的差异几乎无法感知，却使文件体积增加约127%。更严重的是，当这批OGG文件再被Wwise或Unity以Vorbis格式二次编码时，quality 10的原文件并不能为最终质量带来提升，反而因为已经存在一次有损编码而遭受双重损耗。正确做法是始终以WAV提交给引擎工具，由工具统一编码一次。

**误区二：所有平台使用同一套导出规范。** Nintendo Switch的音频子系统原生支持48000 Hz并针对Vorbis解码做了硬件加速，而某些老款Android机型的AudioFlinger混音器锁定在44100 Hz。将同一份48000 Hz OGG直接用于所有安卓设备，会在低端机上触发软件重采样，导致BGM在场景切换时出现约20～50毫秒的额外延迟。多平台项目应在导出规范文档中明确列出每个目标平台的参数矩阵。

**误区三：24-bit导出等同于"母带品质"交付。** 24-bit仅代表量化精度，与频率响应、动态处理和响度标准无关。游戏音频导出规范中，响度目标（通常为 -14 LUFS集成响度，峰值 -1 dBTP）与位深同等重要，但常被忽视。若DAW导出的WAV峰值超过 -0.3 dBFS，OGG编码器的内部过载保护机制可能会静默地对信号进行限幅，造成非预期的音染。

## 知识关联

导出格式规范直接承接**母带处理基础**的输出结果——母带阶段确定的响度目标（-14 LUFS）、峰值上限（-1 dBTP）和频谱平衡，需要通过正确的采样率和位深设置才能无损传递到WAV文件中。若母带阶段使用了32-bit浮点内部总线，则导出为24-bit整数WAV时需要在DAW的导出对话框中确认"Dithering"选项已启用（推荐TPDF算法），以避免量化误差在高频段产生可闻噪声。
