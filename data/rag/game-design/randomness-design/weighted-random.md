# 加权随机

## 概述

加权随机（Weighted Random）是一种根据预先分配的权重值来决定各个结果出现概率的抽样方法。与普通均等随机不同，加权随机允许设计者为每个可能的结果赋予不同的"权重"数字，权重越高的结果被选中的概率越大。例如，在一个战利品系统中，普通剑的权重设为100，传说剑的权重设为1，则普通剑被选中的概率约为99%，传说剑约为1%。

加权随机的算法思想最早在统计抽样领域被系统化，1977年Alastair Walker提出了"别名方法"（Alias Method），将加权随机的时间复杂度优化到了O(1)级别。在此之前，游戏开发者通常使用更简单但效率较低的前缀和方法。随着RPG和抽卡手游的兴起，加权随机成为游戏设计中处理掉落率、召唤池、随机事件的标准工具。

在游戏设计中，加权随机的价值在于它能精确地将设计意图转化为可量化的概率分布。设计者不需要直接写出百分比，而是通过相对权重来表达"这个结果应该比那个结果出现频率高多少倍"，这种表达方式更直觉、更易于调整。

## 核心原理

### 权重到概率的转换公式

加权随机的核心运算是将权重列表转换为概率。设有 n 个结果，各自权重为 w₁, w₂, ..., wₙ，则第 i 个结果被选中的概率为：

**P(i) = wᵢ / (w₁ + w₂ + ... + wₙ)**

以《原神》式抽卡为例：五星角色权重0.6，四星角色权重5.1，三星武器权重94.3，三者总和为100。五星概率 = 0.6 / 100 = 0.6%。权重本身的绝对数值不重要，重要的是各权重之间的比例关系——权重 {1, 5, 94} 和 {10, 50, 940} 产生完全相同的概率分布。

### 前缀和抽样法（线性扫描）

最常见的实现方式是构建权重前缀和数组，然后生成一个 [0, 总权重) 区间内的随机数，再顺序扫描找到该数落入哪个区间。

例如，掉落表包含：金币（权重50）、药水（权重30）、装备（权重15）、稀有装备（权重5），总权重100。生成随机数47，落在 [0, 50) 区间，则掉落金币；若随机数82，落在 [80, 95) 区间，则掉落装备。这种方法实现简单，时间复杂度为 O(n)，适合权重表较小（少于20项）的情况。

### 别名方法（Alias Method）的高效实现

当掉落表条目数量达到数百项时（如大型RPG的全局掉落池），线性扫描性能下降明显。Walker别名方法通过预处理构建两个辅助数组——概率数组（Prob[]）和别名数组（Alias[]），将每次查询的时间复杂度压缩至 O(1)，代价是 O(n) 的预处理时间和空间。

具体而言，预处理阶段将所有权重归一化后，通过不断拆分"过重"条目填补"过轻"条目，最终每个桶恰好包含最多两种结果。查询时只需：① 随机选取一个桶（整数随机）；② 在该桶内用另一个随机数决定取主结果还是别名结果。两次随机操作即完成采样。

### 动态权重与运行时修改

游戏中权重往往不是固定的。常见设计模式包括：保底机制（连续未抽到稀有品后，稀有品权重随次数递增）、时间加权（节日期间特定道具权重提升）、玩家状态加权（低血量时回血道具掉落权重加倍）。实现动态权重时，每次权重变化后需要重新计算总权重，前缀和法只需 O(1) 更新总值，而别名方法需要 O(n) 重新预处理，因此动态权重场景下前缀和法反而更实用。

## 实际应用

**战利品掉落表**：《暗黑破坏神2》的掉落系统使用多层嵌套权重表，普通怪物掉落普通物品权重约为1000，掉落套装装备权重约为1。设计者通过调整权重而非直接修改概率数值来平衡游戏经济。

**卡牌游戏抽卡池**：《炉石传说》卡包中每张卡的稀有度有固定权重比例：普通卡权重约为71.4%，稀有卡约22.2%，史诗卡约4.5%，传说卡约1.9%。这些数值通过玩家统计逆推出权重比例，而非官方直接公布。

**随机地图生成**：《以撒的结合》在生成房间类型时，使用加权随机决定普通房间、宝物房、商店、挑战房的出现频率，权重根据当前楼层和已通关次数动态调整，越深的楼层奖励房间权重相对提升。

**NPC行为AI**：策略游戏中敌方AI选择行动时常用加权随机避免行为过于机械，例如攻击最近单位权重50、攻击血量最低单位权重30、防守基地权重20，使AI在保有战略倾向的同时表现出一定的不可预测性。

## 常见误区

**误区一：权重等于百分比**
初学者常直接将权重值理解为百分比，导致添加新条目后概率计算出错。实际上权重是相对值，添加权重为20的新条目后，原有条目的概率会被稀释。若原系统总权重为100，加入权重20的新条目后，原权重100的条目实际概率变为100/120 ≈ 83.3%，而非原来的100%。设计时应始终以**总权重**为分母重新计算概率。

**误区二：高权重一定会频繁出现**
加权随机描述的是概率，而非保证。权重为99、另一项权重为1时，在100次抽取中，权重为1的结果平均出现1次，但由于随机波动，实际可能连续出现3次，也可能200次都未出现。这与加权随机的算法无关，而是概率本身的自然属性。设计者若需要"保证稀有结果一定出现"，需要额外叠加保底（Pity）机制而非单纯调整权重。

**误区三：所有条目权重均等时等同于普通随机**
这在数学上正确，但实现时存在性能差异。若所有n个条目权重相同，使用加权随机实现仍需计算总权重和做除法，而直接使用整数均匀随机（random.randint(0, n-1)）效率更高。因此，设计中对均等概率部分和差异概率部分应分开处理，不要用加权随机"统一"处理均等概率场景。

## 知识关联

加权随机建立在**概率设计**的基础上——需要先理解概率值（如5%）与频率的关系，才能正确使用权重表达设计意图。概率设计解决"我想要多少概率"的问题，而加权随机解决"如何在代码中实现这个概率"的问题，是概率设计的落地工具。

掌握加权随机后，可以直接应用于掉落系统设计、程序化内容生成（PCG）和AI行为树的概率分支。别名方法（Alias Method）作为加权随机的高效变体，是大规模内容池（如含500+条目的全局掉落表）的必备优化方案，学习者可在掌握前缀和基础实现后，进一步研究其O(n)预处理的具体构建步骤。
