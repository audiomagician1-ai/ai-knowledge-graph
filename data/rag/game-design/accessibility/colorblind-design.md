# 色盲友好设计

## 概述

色盲友好设计是游戏可达性领域中针对色觉缺陷玩家的专项适配技术，其核心目标是确保游戏信息在没有正常色彩感知能力的情况下依然可以被完整读取。全球约有8%的男性和0.5%的女性患有某种形式的色觉缺陷，这意味着一款拥有百万玩家的游戏中可能有数万名玩家无法看到设计者预期的视觉效果。

色觉缺陷分为三种主要类型：第一类是**红绿色盲（Deuteranopia）**，由视网膜中M型视锥细胞缺失引起，患者无法区分红色与绿色，这是最常见的类型，占色盲总人数约75%；第二类是**红色盲（Protanopia）**，由L型视锥细胞缺失引起，患者对红色光谱极不敏感，红色物体在其视野中呈现为接近黑色或深棕色；第三类是**蓝黄色盲（Tritanopia）**，由S型视锥细胞缺失引起，患者无法区分蓝色与黄色，相对罕见，约占色觉缺陷人群的0.01%。游戏设计必须分别针对这三种类型制定适配方案，因为适合Deuteranopia的颜色替换对Tritanopia患者可能毫无帮助。

从游戏历史来看，早期如《星际争霸》的团队颜色系统曾因红绿对立配色让众多色盲玩家无法分辨己方与敌方单位。《英雄联盟》在2012年首次引入色觉缺陷模式，为行业树立了标杆；《赛博朋克2077》则在2020年发行时内置了三种独立的色觉辅助模式，分别对应上述三种色盲类型。

## 核心原理

### 颜色替换与调色板重映射

色盲友好设计最直接的技术手段是颜色重映射（Color Remapping），即在渲染管线的后期处理阶段对全屏画面的像素颜色进行矩阵变换。以Deuteranopia模拟为例，开发者使用Machado et al.（2009）提出的色觉模拟矩阵，将正常视觉的RGB值转换为色盲视觉的感知等效值：

**模拟矩阵（Deuteranopia）：**
```
R' = 0.625R + 0.375G + 0.000B
G' = 0.700R + 0.300G + 0.000B
B' = 0.000R + 0.300G + 0.700B
```

通过该矩阵，设计师可以在开发阶段直接预览色盲玩家所看到的画面，从而有针对性地调整原始颜色。实际的辅助模式则是逆向操作——将容易混淆的颜色对（如红与绿）替换为对所有色觉类型均可区分的颜色对，常用方案是将绿色（#00FF00）替换为青绿色（#00FFCC），将红色（#FF0000）替换为橙色（#FF6600）。

### 非颜色信息冗余编码

颜色永远不应是传递游戏关键信息的唯一通道，这一原则称为**色彩冗余编码（Redundant Encoding）**。具体实践包括：

- **形状差异化**：《文明VI》的资源图标在颜色编码的同时，为每种资源赋予独特外形，即使色盲玩家无法区分颜色，仍可通过图形轮廓识别资源类型。
- **图案填充**：在地图区域、技能效果范围等色块中叠加不同纹理图案（斜线、点阵、网格），《全面战争》系列使用这一方式区分不同势力的领土。
- **文字标注与图标**：《守望先锋2》在玩家头顶的友敌标识中同时使用颜色（蓝/红）、箭头方向（向上/向下）和图标形状三重编码，确保色盲玩家不会误伤队友。
- **UI边框与轮廓**：为可交互元素添加高对比度的白色或黑色描边，使其即使在颜色辨识困难时也能从背景中分离出来。

### 生命值与状态条的色盲适配

游戏中最高频出现的颜色依赖设计是状态条（Health Bar、Stamina Bar），它们通常使用红色代表危险、绿色代表健康。在色盲友好模式下，推荐的替代方案是使用**蓝色（安全）与橙红色（危险）**的组合，因为蓝色与橙色在三种主要色盲类型中均保持较高的可区分度（色差ΔE > 40）。《黑暗之魂》系列的生命条使用红色，耐力条使用黄绿色，这对Deuteranopia玩家构成明显障碍；《只狼》则在发售后更新中允许玩家自定义HUD颜色，部分解决了此问题。

## 实际应用

**射击游戏中的团队识别**：在《彩虹六号：围攻》中，色盲模式将队友名牌由蓝色改为更显眼的青色，并为所有名牌添加了实心方块背景，防止玩家仅靠颜色区分友敌。

**策略游戏中的资源管理**：《星露谷物语》的矿石系统原先依赖红色铜矿、蓝色铁矿、黄色金矿的颜色区分，开发者Concerned Ape在1.5版本中为每种矿石添加了独特图案纹理，使色盲玩家无需颜色信息也能准确识别。

**MOBA游戏中的技能特效**：《Dota 2》允许玩家在设置中将英雄的技能特效颜色完全自定义，并提供了针对Protanopia和Deuteranopia的预设方案，将常见的红色AOE效果替换为紫色，将绿色中毒效果替换为蓝色。

**地图系统**：传统游戏小地图使用红点（敌人）和绿点（队友）的配色对Deuteranopia玩家完全失效。《使命召唤：现代战争》采用红色方块（敌人）与蓝色圆圈（队友）的形状+颜色双重编码，确保即使颜色辨认困难时，玩家仍可通过图形形状获取信息。

## 常见误区

**误区一：只需添加"色盲模式"开关即可**
许多开发者错误地认为提供一个统一的"色盲模式"就足够了。实际上，Deuteranopia、Protanopia和Tritanopia患者的色觉缺陷方向完全不同，一种模式只能有效帮助其中一类群体。例如，将红绿替换为橙绿的方案对Deuteranopia有效，但对Tritanopia患者（蓝黄混淆）没有任何帮助。正确做法是提供三个独立的预设模式，或允许玩家自由调整颜色映射。

**误区二：提高整体饱和度可以解决色盲问题**
增加颜色饱和度只是让正常视觉玩家的颜色更鲜艳，对色盲玩家毫无帮助，因为色盲的成因是特定波长光感受器的缺失，而非颜色信号的强度不足。真正有效的是改变色相（Hue）而非调整饱和度（Saturation）或亮度（Value）。

**误区三：在正常屏幕上测试色盲方案**
开发者不能仅凭自身正常视觉来判断色盲友好方案是否有效。正确流程是使用Coblis、Color Oracle等色觉模拟工具预览游戏画面在各种色盲视角下的外观，并在开发团队中招募实际色盲玩家进行用户测试，或在公测阶段专门招募色觉缺陷玩家提供反馈。

## 知识关联

色盲友好设计建立在**视觉可达性**的基础框架之上——视觉可达性确立了"游戏视觉信息必须对不同感知能力的玩家可用"这一前提，而色盲友好设计则将该原则落实到色觉差异这一具体维度，规定了颜色对比度要求（WCAG 2.1标准中的对比度要求至少为4.5:1）、信息多通道编码规则等技术细节。

色盲友好设计与**字体可读性设计**和**UI对比度设计**存在直接技术交集：三者共同构成游戏视觉信息传达的完整可达性体系。在实现层面，色盲模式通常通过后期处理着色器（Post-Processing Shader）实现，与亮度/对比度辅助功能共用相同的渲染管线节点，因此在项目初期将这些功能统一规划比后期分别补入效率高出数倍。
