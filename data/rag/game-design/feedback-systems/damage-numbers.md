# 伤害数字

## 概述

伤害数字（Damage Numbers），又称浮动数字或飘字，是指在游戏战斗中当角色受到或造成伤害时，在屏幕对应位置短暂显示的数值标注。数字通常从被打击角色头顶弹出，沿抛物线或直线向上漂移，在0.5至2秒内逐渐淡出消失。其首要功能是将底层战斗计算实时可视化——玩家无法直接观察到每帧发生的乘法堆叠运算，但一个飞出的橙色"18,432"能在200毫秒内传达"暴击命中、伤害量级可观"这一完整信息。

伤害数字作为一种游戏界面设计语言，最早在1980至1990年代的日式回合制RPG（如《勇者斗恶龙》系列，1986年起）的战斗画面中大量普及。彼时回合制节奏较慢，每次攻击后屏幕静止展示数值，玩家有充裕时间阅读。真正将其引入实时动作场景的关键节点是《暗黑破坏神》（Blizzard，1996年）：快节奏的走位与攻击使玩家无暇低头查看状态栏，浮动数字成为唯一能在不打断操作的前提下传递伤害信息的方案。此后《魔兽世界》（2004年）将其引入MMORPG大众市场，《英雄联盟》（2009年）进一步验证了伤害数字在MOBA类型中的可读性价值。如今从《原神》到《博德之门3》，几乎所有含有战斗系统的游戏都使用了某种形式的浮动数字。

良好设计的伤害数字可以强化打击感，并在不增加操作负担的前提下完成"诊断性信息"传递（如"这次弱点攻击是否生效"）；劣质设计则会在屏幕上产生视觉噪音，干扰玩家判断敌人剩余血量或规避危险走位。

参考资料：Schell, J. (2008). *The Art of Game Design: A Book of Lenses*. Morgan Kaufmann. 其中 Lens #68（反馈透明度）直接涉及战斗数值的信息化呈现原则。

## 核心原理

### 颜色编码：用色彩区分伤害事件类型

伤害数字依赖颜色在视觉通道上完成"伤害类型分类"，这套颜色编码体系在多数主流游戏中已形成行业惯例，玩家几乎无需学习即可凭直觉解读：

| 颜色 | 典型含义 | 代表游戏 |
|------|----------|----------|
| 白色 / 浅黄色 | 普通物理伤害 | 《魔兽世界》|
| 橙色 / 金色 | 暴击伤害 | 《暗黑破坏神3》|
| 绿色 | 治疗量 / 回血 | 《英雄联盟》|
| 蓝色 | 魔法伤害 / 护盾吸收 | 《最终幻想XIV》|
| 灰色（渐隐） | Miss / 格挡 / 免疫 | 多数ARPG |
| 紫色 | 中毒 / 持续伤害（DoT） | 《暗黑破坏神4》|

颜色选取时需考虑与场景背景的对比度。《暗黑破坏神3》的设计文档（GDC 2013，Jonathan Morin 分享）中明确提到：地牢场景平均亮度值约为L\*35（Lab色彩空间），因此白色伤害数字（L\*95）能获得超过60个亮度单位的对比差，在快速战斗中仍可瞬间识别。而在室外雪地场景中，白色数字会与背景融合，需要为数字添加深色描边（通常为2至3像素宽的黑边），或将数字底色改为半透明深色方块。

### 字号与动态：用尺寸和动画表达伤害量级

字体大小与伤害数值的相对高低应保持对应关系，让玩家在不读取具体数字内容的情况下也能感知伤害强度差异——这种"视觉量化"在每秒可能弹出十余个数字的高速战斗中尤为重要。

常见的字号映射公式为对数缩放：

$$
S_{display} = S_{base} \times \left(1 + k \cdot \log_{10}\!\left(\frac{D}{D_{ref}}\right)\right)
$$

其中：
- $S_{display}$ 为最终显示字号（单位：像素或 UI 单位）
- $S_{base}$ 为基础字号（典型值：24px）
- $k$ 为缩放系数（典型值：0.3 至 0.5，依游戏数值范围调整）
- $D$ 为本次实际伤害值
- $D_{ref}$ 为参考基准伤害（通常取玩家当前阶段普通攻击期望值）

引入 $\log_{10}$ 的目的是防止高数值阶段（如末期装备的数百万伤害）字号无限膨胀遮挡屏幕。当 $D = D_{ref}$ 时，$\log$ 项为零，显示字号等于基础字号；当 $D = 10 \times D_{ref}$ 时（暴击或弱点倍率），字号仅增大约 $k$ 倍（取 $k=0.4$ 则增大约40%），视觉上明显更大但不至于占满画面。

《英雄联盟》的暴击数字比普通数字大约1.5倍，并附加一个持续120毫秒的"弹出后收缩"缩放动画（从130%收缩至100%），制造视觉冲击感而不影响消失时序。动画时长控制在100至200毫秒是行业通行区间：低于80毫秒时玩家几乎无法感知动画，超过250毫秒则会与下一次攻击的数字弹出产生视觉拥堵。

### 位置与运动轨迹：信息分层与防止遮挡

伤害数字的出现位置和漂移路径直接影响战斗可读性，尤其在范围技能同时命中多个目标时。常见策略有三种：

**策略一：固定向上漂移**——数字从被攻击角色头顶正上方弹出，沿垂直方向上升约80至120像素后淡出。优点是预期性强，缺点是多目标场景会堆叠。

**策略二：扇形散布**——多个数字以攻击中心为原点向外扇形分散，横向间距根据数字数量动态调整（每个数字至少保持40像素横向间距以防重叠）。《暗黑破坏神3》的旋风斩打中6个敌人时采用此策略。

**策略三：信息分层偏移**——《原神》将玩家主动操作造成的伤害从敌人头顶正上方弹出，字号标准；环境/持续伤害（如燃烧、冰冻DoT）则从略偏侧面位置弹出，字号缩小约30%，颜色附加元素色调。这确保玩家注意力优先聚焦于主动操作结果，而非被动触发的DoT数字刷屏干扰判断。

## 关键参数与实现参考

以下为一个简化的伤害数字生成逻辑（伪代码示意），展示颜色、字号、位置如何根据伤害类型动态赋值：

```python
def spawn_damage_number(damage_value, damage_type, target_position):
    """
    根据伤害值、类型和目标位置生成浮动数字配置
    """
    config = DamageNumberConfig()

    # 1. 颜色映射
    color_map = {
        "normal":   "#FFFFFF",   # 白色：普通伤害
        "crit":     "#FFA500",   # 橙色：暴击
        "heal":     "#00FF88",   # 绿色：治疗
        "magic":    "#6699FF",   # 蓝色：魔法
        "miss":     "#888888",   # 灰色：未命中
        "dot":      "#CC44FF",   # 紫色：持续伤害
    }
    config.color = color_map.get(damage_type, "#FFFFFF")

    # 2. 字号对数缩放（基础字号24，k=0.4，基准伤害=100）
    import math
    S_base, k, D_ref = 24, 0.4, 100
    config.font_size = S_base * (1 + k * math.log10(max(damage_value, 1) / D_ref))
    config.font_size = max(16, min(config.font_size, 72))  # 钳制在16至72px

    # 3. 暴击附加缩放动画
    if damage_type == "crit":
        config.scale_animation = {"from": 1.3, "to": 1.0, "duration_ms": 120}
    else:
        config.scale_animation = None

    # 4. 位置（头顶偏移 + 轻微随机横向散布防重叠）
    import random
    config.spawn_x = target_position.x + random.uniform(-15, 15)
    config.spawn_y = target_position.y - 60   # 头顶上方60单位
    config.drift_y  = -100                     # 向上漂移100单位
    config.lifetime_ms = 1200                  # 1.2秒后消失

    return config
```

该实现中，`font_size` 的钳制范围（16至72px）是防止极端数值（接近零的擦伤或极高的爆发伤害）导致字号越界的关键保护逻辑。

## 实际应用案例

### 《暗黑破坏神4》的数值膨胀治理（2023-2024）

《暗黑破坏神4》上线初期，玩家角色在高等级赛季结束前伤害数值可达数十亿，屏幕上同时出现10至15个10位数的橙色数字，造成严重的视觉噪音。Blizzard 在2024年2.0版本（"权力季"前的重大改版）中引入了**伤害数值压缩**机制：通过将全游戏的伤害乘数重新基准化，把绝大多数战斗中的数值压缩回数千至数万区间，使字号差异再次成为有意义的信号而非一律巨大的数字墙。这次改动被社区普遍评价为显著改善了"打击感可读性"。

### 《最终幻想XIV》的双重数字系统

《最终幻想XIV》将伤害数字分为两套并行显示系统：**自身造成的伤害**（仅自己可见）和**获得的治疗量**（可选择对全队显示）。这种设计允许玩家在团队副本中专注于自身输出循环，而无需被队友的伤害数字污染视野。该游戏还提供了详细的数字大小/颜色自定义选项（通过系统设置→HUD 布局），是目前主流网游中伤害数字个性化程度最高的案例之一。

### 《Hades》的极简化处理

《Hades》（Supergiant Games，2020年）采用了一种反主流的克制策略：默认只显示玩家受到的伤害（而非造成的伤害），且字号较小、无暴击放大效果。其设计逻辑是：玩家在 Roguelike 高速战斗中最需要的信息是"我受了多少伤"而非"我打出了多少伤"——后者通过屏幕震动、音效和敌人击退反馈来传达。这一取舍减少了视觉噪音，使玩家注意力集中在走位和敌人攻击预判上。

## 常见误区

**误区一：越多越好——每种数字都做差异化**
部分设计新手会将普通、暴击、元素、DoT、治疗、护盾、经验、金币等所有浮动数字都设计成不同颜色和大小，结果屏幕在战斗中变成颜色杂乱的信息爆炸。正确做法是只对玩家需要**即时决策**的数字做高对比差异化（如暴击、Miss、治疗），对辅助性信息（如DoT、金币）使用缩小字号和低饱和度颜色降低视觉权重。
