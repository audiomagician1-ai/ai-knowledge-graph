# 文档版本管理

## 概述

文档版本管理是对游戏设计文档进行系统化编号、存储和变更追踪的工作方法，目的是让团队中任何成员在任意时间点都能找到文档的历史状态，并清楚知道某一设计决策是何时、由谁、因何原因被修改的。它不同于代码版本控制（如Git）的主要特点在于：设计文档的变更往往伴随着创意讨论和决策背景，这些上下文信息同样需要被记录，而不仅仅是文本的差异（diff）。

版本管理的实践在游戏行业大约从1990年代中后期开始规范化，随着团队规模从个位数扩张到数十人，口头沟通无法再保证所有人基于同一版本的设计进行开发。《最终幻想VII》（1997年）的开发团队就曾因剧情文档版本混乱导致多处对话内容在本地化时出现前后矛盾。正是这类事故推动了游戏公司建立更严格的文档版控流程。

对游戏设计文档而言，版本管理的价值体现在三个具体场景：一是程序员和美术人员需要知道当前正在实现的是哪一版设计，而不是已被废弃的旧版；二是项目复盘时需要追溯某个系统被砍掉或修改的原因；三是多人同时修改同一文档时需要合并冲突并留存两种方案的记录。

## 核心原理

### 版本号命名规范

游戏设计文档通常采用**主版本.次版本.修订号**的三段式命名，例如 `v2.1.3`。其中主版本号（Major）的递增代表设计方向的重大变化，如核心玩法机制的推翻重做；次版本号（Minor）代表功能性增删，如新增一个关卡章节的设计内容；修订号（Patch）代表文字修正、数值微调或排版调整。对于还未经过内部评审的草稿，通常在版本号后附加 `-draft` 或以 `v0.x` 表示，例如 `v0.3-draft` 明确告知读者该文档不可用于实际开发。

### 变更日志的必填字段

每次版本更新必须在文档头部或尾部的变更日志（Changelog）中记录以下四项内容：**日期、修改者姓名、版本号、变更摘要**。变更摘要要求写明"修改了什么"与"为什么修改"，而不只是"更新内容"这类无效描述。例如，一条合格的变更记录应写成：`2024-03-15 | 王磊 | v1.4.0 | 将主角跳跃高度从120单位调整为95单位，原因：QA反馈玩家可借此跳过第3章所有战斗触发器`。这条记录在六个月后的项目复盘中依然能传达完整的决策背景。

### 文档存储与访问控制

版本管理需要配合集中式存储系统才能生效。常见方案有三种：使用Google Drive或Confluence并开启版本历史功能；将文档纳入Git仓库（适合纯文本或Markdown格式）；使用专用游戏开发Wiki系统如Nuclino或Notion并设置修订追踪。无论选择哪种工具，都必须设置**只读分发版本**机制——即向开发团队公布一个"当前生效版本"的固定链接，该链接始终指向最新审批版，而草稿修改在另一分支进行，避免开发人员不慎按照未完成的草稿实现功能。

### 分支与合并策略

当两名设计师需要同时修改同一文档的不同章节时，需要创建文档分支。分支合并时，应由主设计师或文档负责人主导，对有冲突的段落逐一确认保留哪一版本，并在变更日志中注明合并来源。一个典型的游戏项目文档分支场景是：主程序文档（Master Design Document）在Alpha阶段同时存在"战斗系统修订分支"和"关卡设计扩充分支"，两者完成各自修订后合并回主文档，合并节点生成新的次版本号。

## 实际应用

**MMORPG战斗数值文档的版本管理实例**：一款角色扮演游戏的战斗数值表在开发周期内经历了47个版本迭代，设计团队使用Confluence的页面历史功能保存每一版，并在每次数值大幅调整（如伤害公式的指数从1.2改为1.5）后生成一个命名快照（Named Version），如"Alpha数值基准v1.0"，供QA团队作为回归测试的参考基线。

**独立游戏的轻量化版本管理**：一人或两人团队通常不需要复杂系统，可采用文件名附加日期的简单方案，例如 `GDD_攀爬系统_20240401.docx`，同时在同一文件夹内保留一个 `CHANGELOG.txt` 文本文件记录每次修改原因。这种方案零成本，且在项目结束后整理发布PostMortem时能直接作为开发时间线的参考材料。

## 常见误区

**误区一：版本号只是形式，每次保存都递增**。实际上，修订号的递增应对应一次完整的审阅或提交动作，而不是每次按下Ctrl+S都更新。如果每天保存20次都递增版本号，到项目中期版本号可能达到 `v1.0.200`，这不仅失去了版本号传达重要性程度的语义价值，也让变更日志变得无法阅读。正确做法是一天内的多次小修改在工作结束时合并为一次版本提交。

**误区二：变更日志只记录"改了什么"，不记录"为什么改"**。设计文档不同于代码提交，设计决策背后往往有玩家测试反馈、制作人决策或外部竞品分析等背景信息。如果只写"修改了关卡3的敌人配置"，三个月后没有人能说清楚这次修改是为了降低难度、优化性能还是配合叙事节奏调整。这类信息丢失后在项目后期复现问题时极难溯源。

**误区三：认为版本管理只适合大型团队**。事实上，即使是两人合作的小型项目，在开发周期超过3个月后，没有版本记录的设计文档会导致大量"我记得我们说过要改这个"类的沟通成本。版本管理的复杂度应与团队规模匹配，但完全省略的代价在任何规模的团队中都会累积。

## 知识关联

文档版本管理建立在**设计文档概述**所介绍的文档类型和结构基础之上——只有先理解GDD（游戏设计文档）、TDD（技术设计文档）等不同文档的职责分工，才能判断哪些文档需要严格的主版本控制，哪些只需要简单的日期标注。例如，GDD的核心机制章节因为影响全体开发人员，需要完整的三段式版本号和强制变更日志；而一个关卡设计师的个人草稿文件只需要简单的日期命名即可。

在项目管理工具的使用上，文档版本管理与里程碑（Milestone）规划存在直接联系：通常在每个里程碑节点（如Pre-Alpha、Alpha、Beta）到达时，需要将当前所有核心文档"冻结"一个命名版本，作为该阶段开发内容的正式依据，之后的修改进入下一个里程碑的草稿周期。这一做法将文档状态与项目阶段紧密绑定，是中大型游戏团队版本管理流程中最重要的节点操作之一。
