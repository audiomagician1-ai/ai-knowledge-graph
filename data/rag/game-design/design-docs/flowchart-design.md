# 流程图设计

## 概述

流程图（Flowchart）是游戏设计文档中用于可视化表达系统逻辑、玩家决策路径和状态转换关系的图形工具。在游戏设计语境中，流程图将抽象的规则与分支逻辑转化为节点（Node）与有向箭头（Directed Arrow）的组合，使程序员、关卡设计师和策划人员能够快速对齐对同一系统的理解。

流程图的标准化符号体系源自1921年弗兰克·吉尔布雷斯（Frank Gilbreth）提出的流程图概念，后经ANSI于1970年发布ISO 5807标准加以规范。游戏设计于1980年代开始大规模采用流程图作为文档工具，最初用于描述早期文字冒险游戏（如《Zork》系列）的场景跳转逻辑，此后逐步扩展至任务系统、对话树和战斗循环的可视化描述。

对游戏设计师而言，流程图的核心价值在于**提前暴露逻辑死锁与遗漏分支**。一张完整的任务流程图往往能在开发前期发现诸如"玩家持有道具A却无法触发事件B"之类的设计漏洞，而这类问题若到代码实现阶段才发现，修复成本将成倍增加。

## 核心原理

### 标准符号与节点类型

游戏设计流程图沿用ISO 5807定义的四类基本符号，并结合游戏领域需求加以扩展：

- **矩形（Rectangle）**：表示动作或状态，例如"播放过场动画"、"敌人进入警觉状态"
- **菱形（Diamond）**：表示判断分支，输出通常为"是/否"或多条件分叉，例如"玩家血量 < 20%？"
- **圆角矩形（Rounded Rectangle）**：表示流程的起始点与终止点，例如"进入关卡"和"任务完成"
- **平行四边形（Parallelogram）**：表示输入/输出事件，例如"玩家按下互动键"或"显示UI提示"

在对话系统流程图中，菱形节点尤为密集，每个玩家选项对应一条出边，所有出边必须最终归并至同一下游节点或各自到达独立终点，否则即为逻辑悬挂（Dangling Branch）。

### 玩家路径分析与关键路径

游戏流程图的有向性（Directionality）至关重要：每条箭头代表玩家或系统从一个状态转移到另一个状态的条件。设计师通常需要识别以下三类路径：

1. **关键路径（Critical Path）**：完成主线目标所需的最短逻辑链，例如"接受任务 → 收集3个物品 → 返回NPC → 任务结束"
2. **可选路径（Optional Path）**：支线内容或奖励内容的分支，不影响主线推进
3. **失败路径（Failure Path）**：玩家未满足条件时的回退流向，必须明确标注出口，避免出现无法继续的死循环

一个健康的流程图中，关键路径的节点数量建议不超过7±2个节点（参考心理学家乔治·米勒的工作记忆理论），超出此范围的流程应当拆分为子流程图（Sub-flowchart）。

### 状态机与流程图的关系

游戏中的AI行为和UI系统通常以**有限状态机（Finite State Machine, FSM）**实现，而FSM的设计文档正是流程图的子集形式。FSM流程图中，每个矩形节点代表一个"状态"（如"巡逻"、"追击"、"攻击"），箭头上标注的是触发状态转换的条件（如"发现玩家距离 < 10米"）。与普通任务流程图不同，FSM流程图允许带环结构（Cyclic Structure），即状态可以循环跳转，这是游戏流程图区别于一般软件流程图的重要特征。

## 实际应用

**任务系统流程图**：以《塞尔达传说：旷野之息》式的任务设计为例，一个典型支线任务流程图包含：触发条件节点（"NPC是否首次见面？"）、道具检查节点（"背包中是否有辣椒×5？"）、奖励分发节点，以及任务状态更新节点。整张图通常控制在1张A3纸面幅内，方便关卡设计师贴于工作台旁随时参考。

**对话树流程图**：RPG游戏的对话系统设计中，设计师使用流程图标注每个对话节点的好感度变化值（如"+10亲密度"）、解锁条件（如"需完成前置任务X"）以及语音行号索引（如"VO_NPC_001_003"），这些标注直接与本地化表格和音频系统对接。

**关卡流程图**：在横版动作游戏关卡设计中，流程图区分"必经区域"和"探索区域"的逻辑节点，并用不同颜色箭头标注"跑酷路线"与"战斗触发区"，使美术和程序团队能够快速理解关卡的空间逻辑顺序。

## 常见误区

**误区一：把流程图当作程序流程图来画**

游戏设计流程图面向的读者是设计师和策划，而非程序员。许多初学者会将系统内部的函数调用、变量赋值等技术细节塞入流程图，导致图表过于复杂。游戏设计流程图应聚焦于**玩家可感知的状态变化**和**设计决策点**，技术实现细节应留给技术文档（Technical Design Document）处理。

**误区二：忽略"否"分支的出口设计**

菱形判断节点的"否"分支往往被设计师遗漏或草草以"返回上一步"处理。例如在"玩家是否持有钥匙？"的判断中，"否"分支应明确指向"提示玩家缺少钥匙的UI反馈节点"，而不是让玩家陷入无反馈的僵局。遗漏否分支是导致游戏出现逻辑死锁（Logic Deadlock）的最常见原因之一。

**误区三：一张图试图涵盖整个游戏系统**

初学者常犯的错误是在单张流程图中描述整个战斗系统，结果产生超过50个节点的"蜘蛛网图"。正确做法是采用**层级式流程图（Hierarchical Flowchart）**：顶层图描述系统间交互（战斗系统→奖励系统→存档系统），子层图再分别展开每个系统的内部逻辑，单张图的节点数量建议控制在15个以内。

## 知识关联

流程图设计以**设计文档概述**中的文档规范为基础——理解GDD（游戏设计文档）的整体结构后，才能判断流程图应放置于系统设计章节还是关卡设计章节，并与用例描述、参数表格等其他文档元素正确配合使用。

在工具层面，Miro、Lucidchart、draw.io是游戏行业最常用的流程图绘制工具，其中draw.io支持直接将流程图导出为XML格式，便于版本控制系统（如Git）追踪设计变更历史。当项目规模扩大时，流程图的维护成本会显著上升，这也是大型项目逐渐转向专用叙事工具（如Twine用于对话树、Articy:Draft用于整体任务流）的主要原因——这些工具本质上是将流程图逻辑与内容数据库直接集成的复合型设计平台。
