# 设计评审

## 概述

设计评审（Design Review）是游戏开发团队在正式进入制作阶段之前，对设计文档进行系统性审查的集体讨论活动。其目的是通过多角色视角识别设计方案中的漏洞、矛盾或不可行之处，使问题在消耗大量美术或程序资源之前得以暴露。不同于个人自查，设计评审强制要求策划、程序、美术、QA等职能的代表同时参与，形成交叉审视。

设计评审制度在游戏行业的普及与敏捷开发方法的引入密切相关。20世纪90年代末，大型主机游戏项目频繁因后期发现根本性设计缺陷而导致大规模返工，业界逐渐将"门控评审"（Gate Review）纳入制作流程，规定设计文档必须通过评审才能推进到下一阶段。如今，许多工作室在预制作阶段（Pre-Production）结束时设有强制评审检查点，未通过则无法立项。

设计评审的价值在于其修复成本的不对称性：在纸面阶段修改一个核心机制的代价，往往是制作中期修改的1/10甚至更低。这一规律被业界称为"缺陷发现越早，修复成本越低"原则，许多工作室将其量化为具体比率（如纸面阶段修改成本比值约为1:10:100），作为推行设计评审制度的内部依据。

## 核心原理

### 评审的三类检查点

设计评审通常设置三个标准化检查点，分别对应不同的文档成熟度：

**概念评审（Concept Review）** 发生在设计文档草稿完成后，重点检查核心玩法循环是否成立、目标受众定义是否清晰、以及设计方向与项目立项目标是否一致。此阶段允许存在大量未决问题，但核心假设必须可论证。

**功能评审（Feature Review）** 针对具体功能的详细设计文档，检查内容包括：功能边界是否定义清楚（例如该功能处理哪些输入状态、排除哪些边界情况）、与已有系统的接口是否兼容、以及数值参数是否有原型数据支撑。

**上线前评审（Pre-Ship Review）** 在内容锁定（Content Lock）前进行，以QA报告和玩家测试数据为依据，复核最终设计与原始文档的偏差是否在可接受范围内，同时标记所有"已知问题"及其处理决策。

### 评审会议的标准结构

一次有效的设计评审会议通常持续60至90分钟，超时往往意味着议题准备不足。标准流程分为四段：

1. **设计呈现（10-15分钟）**：文档作者用简明语言陈述设计目标、方案选择和排除的替代方案，而非逐字朗读文档。
2. **澄清提问（10分钟）**：评审者只能提问以消除理解歧义，不允许在此阶段发表意见。
3. **问题讨论（30-40分钟）**：按照"阻断性问题 → 重大风险 → 优化建议"的优先级顺序讨论，避免次要优化占据阻断性问题的讨论时间。
4. **决议记录（10分钟）**：主持人归纳每个议题的明确结论，标注责任人和截止时间。

### 评审结论的分类体系

设计评审不应以"通过/不通过"简单二元判定结束，通用的四级结论体系更具操作价值：

- **批准（Approved）**：文档可直接推进，无需改动。
- **条件批准（Approved with Conditions）**：需在进入制作前完成指定修改，修改后无需再次评审。
- **修订后重审（Revise and Resubmit）**：存在需要重新讨论的根本性问题，文档需重写关键部分后再召开评审。
- **拒绝（Rejected）**：设计方向无法支撑项目目标，需回到概念阶段重新立项。

## 实际应用

**关卡设计评审示例**：某开放世界游戏的关卡策划提交了一张新区域的白盒（Whitebox）设计文档，其中标注了敌人刷新点、路径引导方式和奖励分布。评审中，程序代表指出两处刷新点的触发逻辑与现有AI导航网格（NavMesh）存在冲突，该问题如进入场景制作阶段才发现，将需要重绘NavMesh并修正大量资产摆放。评审结论为"条件批准"，要求策划在一周内与程序确认触发区域的坐标修改方案，无需重开评审。

**经济系统评审示例**：某手游的数值策划提交了资源获取速率公式，评审团队发现公式中的日活跃玩家假设（DAU基准值设为50万）与运营提供的历史数据存在3倍差距，该差距将导致稀有物品的实际获取周期远低于设计预期，破坏长线留存节奏。此案例显示，设计评审必须纳入来自运营或数据分析岗位的参与者，而不能局限于纯设计职能内部。

## 常见误区

**误区一：评审等同于批评会**。部分团队的设计评审演变为对文档作者进行负面反馈的场合，导致策划倾向于推迟提交文档或提交过于保守的方案。有效的评审针对文档内容而非个人，主持人有责任将措辞从"你的设计有问题"重定向为"这个机制在X场景下会产生Y风险"。

**误区二：所有设计细节都需要评审**。将每个数值调整或文案修改都纳入正式评审流程，会使评审变成行政负担，反而降低文档迭代速度。通常的处理标准是：影响核心循环、跨系统接口或预算规模的修改需要正式评审，局部参数调整和UI细节走异步反馈渠道即可。

**误区三：评审通过即意味着设计冻结**。部分项目在评审通过后拒绝任何设计变更，将"评审"与"设计锁定"混淆。实际上，评审通过表示当前方案在现有信息下是最优可行选项，但后续原型测试或玩家数据可以触发重新评审，这恰恰是迭代流程的正常组成部分。

## 知识关联

设计评审建立在**迭代流程**的基础上：迭代流程定义了"制作→测试→修改"的循环节奏，而设计评审则是在每次迭代启动前的系统性质量门控，确保进入下一轮迭代的方案已经过多方检验，避免在错误方向上累积迭代成本。没有迭代流程的概念作为前提，设计评审容易被误解为一次性的"终审"而非周期性活动。

在设计评审之后，**复盘分析**承接了评审结论的长期追踪职责。评审记录的"条件批准"事项、被拒绝的设计方向以及评审中暴露的团队沟通盲区，都应在项目里程碑复盘时被重新检视：哪些在评审阶段被捕捉到的风险确实在制作中变成了问题？哪些评审结论事后看来判断有误？这种反向验证使设计评审本身的质量得以持续提升。
