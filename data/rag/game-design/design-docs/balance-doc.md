# 数值设计文档

## 概述

数值设计文档（Numerical Design Document，简称NDD）是游戏设计文档体系中专门记录游戏内量化规则的技术文件，其核心内容包括角色属性公式、成长曲线参数、战斗伤害计算方程以及经济系统数值规格。与叙述性的GDD不同，NDD的每一行内容都对应可被程序员直接实现的数学表达式或数据表格。

数值设计文档的雏形出现在1990年代日式RPG工业化生产时期。当时Square开发《最终幻想》系列时，需要将每个角色在1至99级之间的HP成长量用一张电子表格定义清楚，这张表格便是现代NDD的前身。随着MMORPG和免费手游的兴起，NDD逐渐从附属文件演变为独立的核心交付物，一份商业手游项目的NDD通常超过200页。

NDD的价值在于它将设计意图转化为可验证的数学约束。如果策划说"玩家应该在3分钟内击杀精英怪"，NDD就必须给出具体的DPS目标值、怪物HP上限以及时间窗口对应的数值关系——这让程序员、QA测试员和平衡设计师能够用相同的数字语言沟通，而不是依赖主观感受。

## 核心原理

### 属性公式的规范化写法

NDD中的属性公式必须使用完整的变量定义和取值范围声明。以攻击力计算为例，一个规范的公式写法如下：

**最终攻击力 = 基础ATK × (1 + 强化加成%) + 装备ATK + 技能ATK**

文档必须同时注明：基础ATK的来源（角色等级查表）、强化加成%的叠加方式（乘法还是加法）、以及各项的数值上限（cap值）。如果某款游戏规定强化加成%的硬上限为300%，这个数字必须写入NDD并注明其设计依据，例如"防止数值膨胀导致末期内容跳过问题"。

### 成长曲线的数学模型选择

NDD必须为每条成长曲线指定数学模型类型，最常见的三种分别是：**线性模型**（y = kx + b）、**指数模型**（y = a × r^x）、以及**多项式模型**（y = ax² + bx + c）。

以经验值需求曲线为例，若采用指数模型且公式为 EXP_required(n) = 100 × 1.15^(n-1)，其中n为目标等级，则第10级升级所需经验为100 × 1.15^9 ≈ 352点，第20级约为1637点，第30级约为7613点。NDD必须将这张从1级到满级的完整数值表格附在公式之后，供程序员直接导入数据库，而非让程序员自行计算。

### 战斗数值的平衡锚点设计

NDD中必须定义若干"平衡锚点"（Balance Anchor），即整个数值系统的校准基准。典型的平衡锚点包括：**标准关卡通关时间**（如单局PVE关卡设计为90秒）、**满强化状态下的伤害对HP比值**（如1次普通攻击不超过怪物HP的8%），以及**资源获取与消耗的每日流水量**。

所有其他公式的参数都应当能追溯到这些锚点的约束。当NDD中某个技能的伤害系数超过锚点约束时，文档必须记录豁免原因（如Boss技能允许单次打出25%HP伤害），否则任何超出锚点的数值都视为待修正项。

### 经济系统数值规格

NDD的经济章节需要定义货币产出公式和消耗比例。一个典型的免费手游NDD会规定：玩家每天通过主线任务获得的金币上限为X，通过广告变现获得的次数和金额上限为Y，同时商城中主要消耗品的定价必须满足"普通玩家10天可用免费金币兑换一件"的设计目标。这些数字在文档中要以明确的数值而非描述性语言出现。

## 实际应用

**手机RPG角色成长NDD示例：** 某款手游的NDD规定角色HP成长公式为 HP(lv) = 500 + 80×lv + 5×lv²，该角色在1级时HP为585，50级时HP为17500，100级时HP为58500。文档同时标注，HP成长在60级后进入"高速区"，斜率变化对应的是游戏后期副本的DPS要求提升。

**MOBA游戏数值规格：** 《英雄联盟》早期设计文档中，AD射手角色的基础攻速通常设定在0.625次/秒，攻速加成的计算公式为：实际攻速 = 基础攻速 × (1 + 攻速加成%)，且存在2.5次/秒的硬上限。这类具体上限值在NDD中必须以粗体或红色高亮标记，防止开发中途被意外改动。

**MMO经济平衡案例：** 当一款MMO的NDD规定"主线副本每小时产出金币不超过500"，但测试阶段实测数据显示某副本每小时可刷出1200金币时，这个差值会在NDD的"数值偏差追踪表"中以红色标注，触发设计评审流程。

## 常见误区

**误区一：将公式与参数混在正文中而不建表**
许多初级策划的NDD中只写了"HP随等级线性增长，每级+80"，但没有附带完整的等级-HP对照表。程序员实现后，调整参数会导致所有等级的数值同步变化，却没有直观的方式验证中间数值是否符合设计预期。正确做法是公式与完整数值表格同时出现，表格中的数值通过公式自动计算并冻结。

**误区二：混淆"加法叠加"与"乘法叠加"导致公式描述歧义**
NDD中如果写"暴击伤害+50%"，必须同时说明是基于基础伤害的加法叠加（最终伤害 = 基础 × 1.5）还是相对于现有暴击伤害的乘法叠加（最终暴击 = 原暴击倍率 × 1.5）。两种写法在单独加成时差异不大，但当3件装备同时触发时，数值结果可能相差30%以上。《暗黑破坏神3》曾因公开版本说明书中未区分这两种叠加方式，导致玩家和开发团队之间产生大量争议。

**误区三：成长曲线只画到"设计关卡上限"而忽略超出部分**
部分NDD只定义到当前版本的等级上限（如60级），不处理玩家通过BUG或未来版本超出上限后的数值行为。规范的NDD应在上限等级之后额外延伸10%-20%的数值范围，并明确写出超出上限时系统的截断方案或回退逻辑。

## 知识关联

**前置依赖：GDD结构**
NDD的章节划分直接依托GDD中已确立的系统边界。GDD若定义了"游戏包含装备强化、角色升级、技能树三大成长维度"，则NDD就必须为这三个维度各自建立独立的数值规格章节，并在文档头部以GDD章节编号作为索引引用。没有GDD的系统定义，NDD的数值就缺乏明确的作用域。

**后续延伸：数值调优**
NDD是数值调优工作的输入文件。数值调优阶段的工作本质上是将NDD中的公式参数（如成长曲线的系数k、乘数r）在游戏测试数据的指导下进行迭代修正，而修正结果会反向更新NDD形成新版本。因此NDD必须建立版本号和变更日志机制——每次参数调整都应在文档中留下"v1.3：将经验曲线底数从1.15调整为1.12，原因：玩家到达30级的平均时长从8小时缩短至5小时，低于设计目标"的记录。
