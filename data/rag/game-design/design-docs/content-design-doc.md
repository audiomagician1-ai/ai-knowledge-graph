# 内容设计文档

## 概述

内容设计文档（Content Design Document，简称CDD）是游戏开发中专门用于规划关卡、任务和剧情细节的技术性文档。它从游戏设计文档（GDD）中独立出来，专注于玩家在游戏世界中实际遭遇的具体内容——哪里有敌人、哪段对话触发何种条件、某个支线任务的奖励是什么。与GDD描述"游戏是什么"不同，内容设计文档回答的是"游戏里有什么"。

内容设计文档的独立化实践在2000年代初期随着开放世界游戏的兴起而普及。早期如《上古卷轴III：晨风》（2002年）的开发团队Bethesda就开始使用专门的内容规划表格管理数百个地下城和任务，因为将这些细节全部塞入单一GDD会使文档膨胀到数千页且难以维护。现代大型RPG的内容设计文档可能包含数万条任务描述条目。

内容设计文档的价值在于它为不同职能的团队成员提供了共同的内容基准。关卡设计师知道第三关的第二个房间应该放三名持盾敌人；文案写手知道某个NPC在玩家完成特定任务后会说哪句台词；QA测试员有明确的任务目标清单可供核对。没有这份文档，团队成员只能凭猜测填充游戏世界。

## 核心原理

### 三层内容结构

内容设计文档通常按照"区域→关卡/场景→事件"的三层结构组织。顶层是世界区域划分，例如将游戏地图分为主城、荒野、地下城三个大区；中间层是具体关卡或场景，记录该区域内每个独立可游玩单元的基本信息；底层是单个事件，包括触发条件、内容描述、结果反馈。以《巫师3》式的任务结构为例，一个支线任务条目需要记录：任务接取位置坐标、前置任务依赖、对话分支数量、每个分支的道德权重值（通常用-2到+2的整数表示）、任务完成奖励（经验值、货币、物品ID）。

### 任务规格说明

每条任务的规格说明需包含六个固定字段：**任务ID**（唯一标识符，如SIDE_QUEST_047）、**任务类型**（主线/支线/重复性日常）、**触发条件**（到达坐标区域/与NPC对话/道具拾取）、**目标描述**（玩家可读的任务日志文本）、**完成判定逻辑**（程序员可执行的条件判断，如`enemy_count == 0 AND item_returned == true`）、**奖励清单**（含物品ID和数量）。这六个字段缺一不可，否则任务无法被程序正确实现或被测试人员准确核验。

### 剧情分支标注

涉及叙事内容的部分采用流程图与文本结合的方式记录。每段对话用节点编号标注，例如`DLG_NPC_MERCHANT_003`，节点内包含：说话角色、台词原文、触发条件（如玩家好感度≥50）、指向下一节点的跳转编号。分支对话必须明确标注"汇合节点"——即多条分支最终合流回哪个共同节点，否则文案和程序员会对对话树的结构产生不同理解。著名的叙事游戏《极乐迪斯科》的开发团队ZA/UM据报使用了超过一百万字的对话内容，其内容设计文档的节点管理是整个项目的核心工作之一。

### 关卡内容清单

关卡内容部分记录特定地图或场景中预置的所有游戏对象。标准格式为表格，每行代表一个对象实例，列包括：对象类型（敌人/道具/机关/NPC）、对象ID（引用策划数值表）、坐标或相对位置描述、刷新规则（唯一/每日重置/每次进入重置）、特殊行为备注。例如一个关卡表格的某行可能写："精英怪物ENEMY_TROLL_ELITE，入口右侧30单位处，唯一不刷新，击败后掉落钥匙道具KEY_DUNGEON_B2"。

## 实际应用

在一款动作RPG的地下城关卡中，内容设计文档会用独立页面描述该地下城的全部内容规划：入口区放置2名普通骷髅兵（等级与玩家推荐等级持平）用于教学，中央大厅触发Boss预警剧情，Boss房内Boss血量为玩家推荐等级对应生命值的3.5倍，击败后解锁隐藏房间并触发支线任务`SIDE_QUEST_CRYPT_SECRET`的第二阶段。

在移动游戏的关卡规划场景中，内容设计文档通常以电子表格（如Google Sheets）的形式存在，每行是一个关卡，列包括关卡编号、关卡类型（普通/精英/BOSS）、推荐战斗力、通关奖励、首通额外奖励、关卡描述（供文案撰写加载界面提示语）。某款三消游戏可能管理着超过3000行的关卡内容表格，设计师每周新增约20行新关卡规格。

## 常见误区

**误区一：将内容设计文档与GDD混写**。初学者常把任务设计细节（如具体NPC台词、敌人数量）直接写进GDD，导致GDD臃肿且难以更新。正确做法是GDD只写"游戏有支线任务系统，支线任务提供非主线奖励"，而具体哪些支线任务、奖励是什么，全部移入内容设计文档。两份文档的更新频率不同——GDD在原型期后较少改动，内容设计文档在整个制作阶段持续增补。

**误区二：忽略触发条件的互斥性检查**。设计师单独写每条任务的触发条件时，容易忽略多个任务同时满足触发条件的冲突情况。例如任务A和任务B都在玩家进入同一区域时触发，若两个任务都弹出对话，会造成叙事混乱。内容设计文档需要专门维护一张"触发条件冲突矩阵"，标注哪些事件不能同时激活。

**误区三：把内容设计文档当剧本使用**。内容设计文档的台词字段应填写最终定稿台词或占位文本标记（如`[PLACEHOLDER: 商人抱怨天气]`），而不是写作过程中的多个备选版本。将讨论稿混入正式文档会让程序员和本地化团队无法确认应实现哪个版本，导致返工。

## 知识关联

内容设计文档直接依赖GDD结构中确定的游戏系统参数——任务奖励中的经验值数字必须符合GDD里定义的升级曲线公式（通常是`EXP_required = base * level^exponent`中的参数），否则内容设计文档中的奖励数值会破坏游戏经济平衡。设计师在填写奖励清单之前，必须先确认GDD中的核心数值框架已经稳定。

在团队协作流程中，内容设计文档是连接策划与程序、美术、文案三个部门的接口文档。程序员根据任务规格中的完成判定逻辑字段编写脚本；美术根据关卡内容清单制作对应的资产；文案根据对话节点树填写正式台词。当任何一方修改内容设计文档中的条目时，必须通过版本变更日志（Change Log）通知其他相关方，这是防止各部门基于过期版本工作的关键机制。
