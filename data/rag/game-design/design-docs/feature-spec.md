# 功能规格书

## 概述

功能规格书（Feature Specification Document，简称FSD）是针对游戏中**单一功能**所编写的详细规格与需求文档。它将GDD（游戏设计文档）中简略描述的某个功能——例如"背包系统"或"角色升级机制"——展开为具体、可执行的设计说明，供程序员、美术、QA等各职能团队成员直接使用。

功能规格书的形式成熟于1990年代末的PC游戏开发时代。微软游戏工作室在这一时期将其系统化，要求每个可交付功能必须配有独立的规格文书，与软件工程中的"功能需求文档"理念相融合。在游戏行业，功能规格书通常由负责该功能的设计师（Feature Owner）撰写，而非由主策划统一编写。

功能规格书的重要性体现在它能够将"设计意图"转化为"实现指令"。一份模糊的GDD描述如"玩家可以合成物品"，在功能规格书中必须明确为合成触发条件、配方数据结构、UI反馈时序、失败逻辑等逐条内容。如果缺少这一层文档，程序员将不得不频繁打断设计师以询问细节，导致开发效率大幅下降。

## 核心原理

### 文档结构的标准框架

一份规范的功能规格书通常包含以下固定区段：**功能概述（Feature Summary）、用户故事（User Stories）、功能需求列表（Functional Requirements）、边界条件与异常处理（Edge Cases）、UI/UX说明、数据结构说明、依赖项清单（Dependencies）**，以及**验收标准（Acceptance Criteria）**。

其中，功能需求列表采用编号体系（如F-001、F-002）对每条需求进行唯一标识，方便QA在测试用例中引用，也方便后期追踪需求变更。例如：
- **F-001**：玩家背包初始容量为20格，每格只能容纳同类型物品堆叠上限为99个。
- **F-002**：当背包满载时，拾取物品操作应触发"背包已满"提示，物品留在原地不消失。

### 用户故事的写法

功能规格书中的用户故事遵循固定格式：**"作为[角色]，我需要[行为]，以便[目的]"**。这种格式强制设计师从玩家视角出发描述需求，避免以实现方式代替设计意图。

以"存档系统"为例，正确写法为："作为玩家，我需要在任意主城内手动存档，以便在意外关闭游戏后能从上次安全点继续。"这条用户故事将立即暴露一个设计问题：副本内是否允许存档？此类边界问题必须在功能规格书中逐一回答，而不是留给程序员自行决策。

### 边界条件与异常处理

边界条件是功能规格书中最容易被省略、却最容易导致Bug的部分。设计师必须明确列出功能在极端输入或异常状态下的预期行为。常见需要覆盖的边界包括：
- **数值边界**：角色等级上限为99级时触发升级会发生什么？经验值是否继续累积？
- **并发操作**：玩家在合成动画播放期间点击取消按钮，材料是否退还？
- **网络异常**（针对在线游戏）：交易请求发出后断线，物品与货币如何回滚？

每个边界条件必须给出**明确的预期结果**，不能使用"由程序决定"这类表述。

### 依赖项与接口说明

功能规格书必须声明本功能依赖的其他系统接口。例如，"成就系统"的功能规格书需要注明：它依赖事件总线（Event Bus）监听"击杀敌人"、"收集物品"等游戏事件，并列出所需的事件类型、参数格式及回调时序。这份依赖清单是跨团队协作的合同，防止两个团队对接口格式产生各自的假设。

## 实际应用

**案例：手机游戏"每日签到"功能规格书**

在一款手机RPG中，"每日签到"看似简单，但完整的功能规格书需要覆盖约15条功能需求，包括：
- 签到时间以服务器时区（UTC+8）凌晨0点为刷新基准，非客户端本地时间。
- 连续签到第7天触发"连签奖励"，断签后连续计数归零但累计天数保留。
- 签到弹窗的首次显示时机为玩家登录后大厅加载完成起3秒，而非立即弹出。
- 奖励物品发放调用道具发放接口（/api/reward/grant），失败时重试最多3次，超时阈值为5秒。

这个案例说明，一个在GDD中仅占半句话的功能，其功能规格书可能需要500至1500字才能完整覆盖所有实现细节。

## 常见误区

**误区一：将UI线框图当作功能规格书**

部分设计师认为提供UI原型图就等于完成了功能规格书。UI图只能描述视觉布局，无法表达交互状态机、数据来源、错误处理逻辑等信息。一张背包界面的截图无法告诉程序员：当物品被拖拽到已占用格子时应交换位置还是弹回原处。功能规格书的文字需求列表与UI说明必须同时存在，互相补充。

**误区二：在规格书中混入技术实现方案**

功能规格书描述的是"系统应该做什么（What）"，而非"系统如何实现（How）"。在规格书中写入"背包数据使用哈希表存储"属于越界，这是技术设计文档（TDD）的职责范围。当设计师在规格书中指定实现方式时，会限制程序员的技术选型自由，并在技术方案变更时导致设计文档与实现脱节。

**误区三：规格书写完后不再更新**

功能规格书是活文档，必须随开发进展同步修订。若某功能需求在Sprint中途发生变更，必须在规格书中标注变更日期与修改原因（例如："F-005，2024-03-15修改：因平衡性考虑，合成冷却时间从30秒调整为15秒"）。未维护的规格书比没有规格书更危险，因为它会误导后来加入的团队成员。

## 知识关联

功能规格书直接承接**GDD结构**中对功能的高层描述。GDD通常以一两段话概括某个功能的设计意图，功能规格书则将这段意图展开为工程团队可执行的逐条需求。例如，GDD中"战斗系统"章节会拆分为若干独立的功能规格书：伤害计算规格书、技能释放规格书、AI行为规格书等，每份文档聚焦单一功能边界。

功能规格书完成后，程序员团队将基于它编写**技术设计文档（TDD）**。TDD读取功能规格书中的"What"，转化为架构方案、类设计、接口定义等"How"内容。两份文档形成设计到实现的完整链条：设计师拥有功能规格书的所有权，程序员拥有技术设计文档的所有权，QA则基于功能规格书中的验收标准F-XXX编号生成测试用例，形成可追溯的质量保障体系。
