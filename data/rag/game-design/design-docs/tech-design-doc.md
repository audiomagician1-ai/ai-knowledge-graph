# 技术设计文档

## 概述

技术设计文档（Technical Design Document，简称TDD）是游戏开发流程中用于描述**如何实现**特定功能的工程蓝图。与描述"做什么"的功能规格书不同，TDD明确回答"怎么做"——它规定系统架构、数据结构、模块接口、算法选择以及技术约束条件。一份典型的TDD通常由程序主程或系统架构师撰写，面向参与该功能实现的工程师团队。

TDD作为独立文档类型，在2000年代随着大型AAA游戏开发规模扩大而逐渐标准化。早期游戏开发中，程序员往往凭口头约定和代码注释协调实现细节，但当团队规模超过10人、子系统间依赖关系复杂化后，这种方式导致大量接口冲突和重复劳动。育碧、EA等工作室逐步将TDD纳入正式交付物体系，要求编码工作开始前必须通过TDD评审。

TDD在游戏开发中的特殊价值在于：游戏系统往往具有高度的状态依赖性（如物理引擎需与动画系统、碰撞系统实时同步），任何接口假设不一致都会导致难以复现的运行时bug。通过TDD提前对齐接口契约，可将集成阶段发现的接口问题减少约60%（据部分工作室内部统计）。

## 核心原理

### 系统架构说明

TDD的首要内容是**系统架构图**，通常以UML类图或组件图形式呈现，描述该功能涉及的模块划分及其依赖关系。例如，实现一个背包系统的TDD需绘制InventoryManager、ItemDatabase、SlotUI三个组件的关系，明确哪个组件持有数据所有权，哪些是观察者。架构图必须标注每条依赖箭头的方向，避免循环依赖带来的编译或运行问题。

技术架构部分还需说明该系统采用的设计模式（如观察者模式、享元模式），以及选择该模式的具体理由——例如选择享元模式处理大量同类型子弹对象，可在同时存在500个实例时将内存占用从约40MB降至约2MB。

### 数据结构与接口定义

TDD的核心技术价值体现在对**数据结构**和**公共接口**的精确定义上。以C++或C#伪代码形式列出主要数据结构，例如：

```
struct InventorySlot {
    int slotIndex;       // 槽位索引，范围0-35
    ItemID itemId;       // 物品唯一标识符，0表示空槽
    uint8 stackCount;    // 堆叠数量，上限由ItemDatabase决定
};
```

接口定义需包含函数签名、参数含义、返回值语义、以及**前置条件和后置条件**（即Design by Contract规范）。例如`AddItem(ItemID id, int count)`需明确：若背包已满应返回失败码而非静默丢弃，这一行为约定如果只靠口头传递极易产生分歧。

### 性能预算与技术约束

TDD必须包含该系统在目标平台上的**性能预算**，这是与功能规格书最显著的区别之一。例如：

- AI寻路系统每帧CPU时间预算：**2ms**（基于目标60fps、总帧预算16.6ms的分配结果）
- 存档序列化操作需在**300ms**内完成，避免玩家感知卡顿
- 粒子系统单场景最大实例数：**1000个**，超出时触发LOD降级策略

性能预算须与其他系统的预算协商确定，并在TDD中注明协商日期和参与人员，作为后续性能争议的仲裁依据。

### 风险与回退方案

专业的TDD会包含一个**技术风险登记表**，列出实现过程中可能遇到的技术难点（如某第三方物理中间件在主机平台上存在已知的多线程竞态问题），并为每项风险提供评级（高/中/低）和备用方案。例如若GPU蒙皮性能不达标，回退方案是切换CPU蒙皮并接受15%的CPU成本增加。

## 实际应用

**战斗系统TDD示例：** 在实现一款回合制RPG的伤害计算系统时，TDD需定义伤害公式：

> `最终伤害 = (攻击力 × 技能倍率 - 防御力 × 0.5) × 元素相克系数 × 暴击倍率`

每个变量的数值范围、精度类型（int还是float）、计算顺序均需在TDD中明确。若计算顺序未规定，不同程序员的实现可能导致数值结果出现±5%的误差，进而破坏精心调校的数值平衡。

**网络同步TDD示例：** 多人游戏的角色位置同步系统TDD需说明采用客户端预测+服务器权威的架构，定义同步频率为**每秒20次**，插值窗口为**100ms**，并给出具体的状态回滚触发条件（如服务器与客户端位置差超过**0.5米**时强制纠正）。这些具体数字必须写入TDD而不能留待实现时决定，因为它们直接影响带宽消耗和玩家对延迟的主观感知。

## 常见误区

**误区一：TDD等同于代码注释的扩展版。** 实际上TDD是编写任何代码之前完成的设计文档，其目的是在动手实现前发现架构层面的问题。如果TDD在代码完成后才补写，它就失去了预防接口冲突的核心价值，沦为纯粹的文档归档工作。

**误区二：TDD应该详细到覆盖每个函数的实现细节。** TDD应聚焦于**对外暴露的接口和跨模块约定**，内部实现算法（如具体的排序方式选择）属于实现者的自由裁量范围，不应过度规定。过于细致的TDD会让实现者失去优化空间，且维护成本极高——每次重构都需要同步更新文档，反而形成负担。实践中，TDD的适当粒度是能让两个独立团队各自实现模块后成功对接。

**误区三：TDD只需要一个版本。** 游戏开发中需求变更频繁，TDD需要版本管理。每次因外部需求变化导致接口变更时，必须更新TDD版本号（如v1.3→v2.0），并在文档中标注变更日期、变更原因和受影响的下游模块，确保相关工程师能识别自己的实现是否需要跟随更新。

## 知识关联

TDD直接承接**功能规格书（GDD功能章节）**的输出——功能规格书定义了"玩家可以拥有最多36个背包格"，TDD则将其转化为`InventorySlot[36]`数组结构、槽位索引的内存布局方案，以及背包数据的持久化存储格式。阅读TDD的前提是理解功能规格书中已确定的功能边界，对于TDD作者而言，功能规格书中未解决的设计歧义必须在开始撰写TDD前与策划团队澄清，而不能在技术文档中自行做出功能决策。

在工作流中，TDD通常触发**代码评审标准**的制定：评审者会将提交的代码与TDD中的接口定义进行比对，检验实现是否符合已约定的契约。此外，TDD中的性能预算数字会直接进入**性能测试计划**，成为自动化性能回归测试的阈值依据。
