# UI线框图

## 概述

UI线框图（Wireframe）是游戏界面设计流程中的低保真原型文档，使用简单的线条、方框和灰度色块来表示游戏各个界面的布局结构，而不涉及任何具体的美术风格、配色方案或精细图形。它的核心目的是在投入美术资源之前，快速验证界面元素的位置、层级关系和交互逻辑是否合理。

线框图的概念起源于20世纪90年代网页设计领域，随后被游戏UI设计广泛借鉴。在游戏开发中，一张主菜单的线框图可能只需要30分钟绘制，但若直接跳过这一步、直接进入高保真UI设计，一旦布局需要调整，则可能浪费数天的美术工时。线框图将布局决策与视觉决策分离，使设计师、策划、程序员能在同一份低成本文档上达成共识。

线框图在游戏开发中尤其重要，因为游戏UI往往需要在多个分辨率（如1920×1080的PC端与375×812的移动端）上同时适配，并且需要与HUD（平视显示器）、背包系统、技能面板等复杂交互模块配合。线框图阶段能及早暴露"血量条与小地图在竖屏模式下重叠"等布局冲突，避免后期返工。

## 核心原理

### 保真度控制：灰度优先原则

UI线框图刻意限制视觉细节，通常只使用三种颜色：白色（背景）、浅灰（内容区块）、深灰或黑色（边框与文字占位符）。文字内容通常用横线或"Lorem Ipsum"占位，图标位置用打叉方框（☒符号）表示。这种做法防止评审者将注意力从"按钮放在哪里"转移到"按钮应该是什么颜色"，确保讨论聚焦于空间布局合理性。

保真度分三档：草图（Sketch）、数字线框图（Digital Wireframe）、带注释线框图（Annotated Wireframe）。游戏UI设计中最常用的是第三档，即在数字线框图上附加编号标注，说明每个元素的功能说明、尺寸约束或交互备注，例如标注"区域③ 技能栏：共6格，每格64×64px，间距8px，支持拖拽排序"。

### 层级与锚点标注

线框图中必须标明UI元素的屏幕锚点位置，即元素相对于屏幕哪个角落对齐。常见记法是用九宫格坐标表示锚点，如"左上角锚定（0,1）"或"屏幕中央锚定（0.5,0.5）"——这与Unity引擎的RectTransform锚点参数直接对应。在绘制线框图时将锚点规则写入标注，程序员在实现UI自适应时能够直接参照，减少沟通歧义。

层级关系通过缩进列表或层叠矩形表示：父容器用实线框，子元素用虚线框或嵌套灰色色块。例如"背包窗口（父）→ 物品格子容器（子）→ 单个物品格（孙）"的三层结构，在线框图中通过三层嵌套矩形即可清晰传达，无需任何文字解释。

### 交互流向标注

静态线框图只展示单一界面状态，而**交互线框图**（Interactive Wireframe）在此基础上用带箭头的流程线连接多个界面状态，标明"点击某按钮后跳转至哪个界面"或"悬停时弹出哪个浮层"。游戏中典型的交互流向案例是主菜单→角色选择→游戏设置→返回主菜单的循环路径，以及战斗HUD中按下Tab键呼出背包界面的叠加状态。

标注交互触发条件时，使用统一符号约定：实线箭头表示点击触发，虚线箭头表示悬停触发，波浪线箭头表示滑动触发。约定一旦确立，团队所有成员无需额外说明即可读懂全套线框图文档。

## 实际应用

**移动端手游HUD设计**：某款移动端MOBA游戏在开发初期，设计师用Figma绘制了375×667px（iPhone SE分辨率）的战斗HUD线框图，将摇杆区域、技能按钮区域、血量区域、小地图分别用标注色块区分。经过3轮线框图评审后发现，技能按钮与聊天按钮在拇指操作区域存在误触风险，在进入美术设计前提前将技能按钮整体上移12px，避免了后续大量返工。

**RPG背包界面设计**：设计师在绘制背包系统线框图时，需要同时标注物品格的网格参数（如5列×8行、每格72×72px）、分类标签栏的切换方式、装备预览区的位置和比例，以及排序下拉菜单的触发方式。这些信息以线框图+标注的形式交付，程序员无需额外会议便可开始UGUI布局搭建。

**主菜单多分辨率适配**：PC游戏线框图通常同时输出16:9（1920×1080）和21:9（2560×1080）两个版本，通过对比两份线框图检查宽屏下中央内容是否出现异常拉伸或留白过大的问题，并在线框图阶段确定边距锁定规则。

## 常见误区

**误区一：线框图精度越高越好**。部分初学者习惯在线框图阶段就精确标注按钮圆角半径、字体磅数等视觉细节，导致一张线框图耗费数小时，却与高保真设计稿的工作量无异，完全失去了线框图快速迭代的意义。正确做法是将视觉规格细节留给后续的UI规范文档（UI Style Guide）处理，线框图只负责位置和比例的粗略确认。

**误区二：线框图只给UI设计师看**。游戏UI线框图实际上是跨职能的沟通文档，策划需要确认功能按钮是否全部存在且逻辑正确，前端程序员需要判断布局的技术可行性（如ScrollView嵌套层数），QA测试工程师可以提前规划测试用例路径。仅在设计师内部评审线框图会遗漏大量专业角度的问题。

**误区三：交互线框图可以替代原型测试**。线框图记录的是设计师的意图，而不是真实玩家的行为反馈。当线框图显示"金币数量显示在屏幕右上角"时，这只是一个静态假设；只有通过可点击原型（Clickable Prototype）或实机测试才能验证玩家是否能在战斗中快速找到该数值。线框图与可用性测试是互补关系，而非替代关系。

## 知识关联

UI线框图建立在**设计文档概述**所确立的文档规范体系之上：设计文档概述定义了游戏文档的分类方式和版本管理规则，UI线框图作为其中的界面类文档，遵循相同的版本号命名格式（如WF-MainMenu-v1.3）和评审流程。熟悉设计文档的整体框架，能帮助设计师明确线框图在整个开发流水线中的交付节点——通常在概念设计阶段结束后、高保真UI设计启动前输出。

在技术工具层面，线框图常用工具包括Figma、Balsamiq Mockups（专为线框图优化，内置手绘风格线框组件库）以及Axure RP（支持交互逻辑设定）。Balsamiq的特点是其刻意使用手绘风格字体和粗糙线条，心理暗示评审者"这还是草稿阶段"，从而鼓励更直接的修改意见，而不是因为图纸"看起来很完整"而不敢提出变更建议。这一心理效应已在多项UX研究中得到验证，是低保真原型方法论的重要支撑依据。
