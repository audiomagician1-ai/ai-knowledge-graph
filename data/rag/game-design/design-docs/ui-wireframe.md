# UI线框图

## 概述

UI线框图（Wireframe）是游戏界面设计流程中的低保真原型工具，用黑白灰色块、简单线条和占位符文字描绘游戏界面的元素位置、层级关系与交互逻辑，而不涉及具体的美术风格、色彩方案或精细视觉效果。在游戏开发文档体系中，UI线框图通常是正式UI设计稿之前的必要环节，帮助团队在投入美术资源之前就确认信息架构是否合理。

线框图这一概念起源于20世纪90年代的网页设计领域，由信息架构师彼得·莫尔维尔（Peter Morville）等人推广应用，后被游戏行业借鉴并形成专属规范。在游戏UI设计中，线框图的粒度可分为三个层级：草图级（Sketch）精度约20%、数字线框级精度约50%、标注线框级精度约80%，最终高保真原型才达到100%。选择哪个层级取决于当前开发阶段和沟通目标。

线框图之所以在游戏设计文档中占有重要地位，是因为它能将UI设计师、程序员、策划和美术之间的沟通成本大幅压缩。一张标注清晰的战斗HUD线框图，可以让程序员提前评估血条、技能图标、小地图等模块的实现复杂度，也让策划确认技能槽数量是否符合玩法需求，这一切发生在任何一张贴图被制作之前。

## 核心原理

### 视觉层级与信息优先级

UI线框图通过元素的尺寸大小、位置（通常遵循F型或Z型视觉动线）和框架粗细表达信息优先级，而非依赖颜色或插画。以RPG游戏的角色状态界面为例，角色头像占位框通常是最大的矩形块（建议占屏幕高度的15%~20%），其次是属性数值区域，最末是说明文字区域。线框图中用"X"填满的矩形表示图像占位，用横线（"———"）表示文字占位，这两种符号是全行业通用标准，不可随意混用。

### 交互标注系统

游戏UI线框图必须包含交互说明，这是它区别于静态平面稿的关键属性。标准做法是在线框图旁边用带编号的标注气泡说明每个交互状态：例如"①点击技能图标 → 展开技能详情浮层（见Wireframe #07）""②血量低于20%时血条区域触发红色闪烁（动效规格见动效文档）"。每个可点击元素必须明确标注点击区域尺寸（移动端最小点击区域建议为44×44像素，源自苹果HIG规范），避免后期出现按钮过小导致误操作的问题。

### 网格系统与屏幕适配标注

专业游戏UI线框图必须建立在网格系统之上。主流做法是将画布按12列栅格（Column Grid）划分，每个UI元素声明其占据的列数。例如"主菜单按钮组：每个按钮占4列，共3列排布，间距1列"。同时，线框图需明确标注适配目标分辨率，游戏行业常见基准分辨率为1920×1080（PC）和375×667（手机竖屏），跨平台游戏需在同一份文档中呈现两套线框方案，并标注哪些区域在适配时允许缩放、哪些区域锁定绝对像素值。

### 状态覆盖（State Coverage）

游戏UI与普通App UI的最大区别在于状态极其复杂。一张完整的游戏UI线框图不能只画"默认状态"，必须覆盖所有可能状态：空状态（背包为空时）、满状态（背包满格时格子变红）、加载状态（网络请求时的骨架屏）、禁用状态（等级不足时技能图标置灰）。漏掉任何一种状态，都会在开发阶段被程序员反馈"这种情况UI怎么处理"，造成返工。

## 实际应用

**战斗HUD（Head-Up Display）线框图**：绘制MOBA类游戏战斗界面时，线框图需在1920×1080画布上标注技能栏位于屏幕底部中央、占据宽度的30%；小地图位于右下角、尺寸为200×200像素；敌我血条以对称形式分列屏幕顶部两侧，并注明"本机玩家方向始终居左"。这些位置决策在线框图阶段确定后，后续美术制作只需在该框架内添加视觉风格即可。

**背包系统线框图**：背包界面线框图需要额外绘制"格子悬停（Hover）状态""格子选中状态""物品拖拽状态"等至少5种交互分支图，并用箭头注明各状态间的转换触发条件。例如"鼠标左键按下不松开超过0.3秒 → 进入拖拽状态"，这个0.3秒的阈值需要直接标注在线框图文档中供程序参考。

**新手引导遮罩层线框图**：此类线框图需特别标注遮罩层的Z轴层级（Z-Index），确保引导高亮区域位于遮罩之上、其余UI之下，并标注半透明遮罩的透明度建议值（通常为60%黑色或70%黑色），这些参数直接影响玩家是否能清晰识别引导目标。

## 常见误区

**误区一：认为线框图越详细越好**。部分设计师初学时会在线框图阶段就细化字体选型、圆角半径等视觉细节，导致线框图与最终UI稿高度重叠，反而让团队产生"这就是最终稿"的误判，跳过设计评审环节。线框图的核心价值在于低成本试错，精度过高会破坏这一价值——一旦精细化后，修改成本与修改原型相差无几，失去了低保真阶段的意义。

**误区二：只画主流程界面，忽略异常状态**。许多新手设计师仅绘制正常游戏流程下的界面，遗漏网络断开提示、服务器维护公告弹窗、账号被封禁提示等异常状态界面。这些边缘状态在游戏上线后必然出现，若未在线框图阶段规划，程序员往往会用默认系统弹窗代替，导致游戏品质参差不齐。

**误区三：线框图不需要版本管理**。线框图文档在迭代过程中经历的修改往往多达十余个版本，若不建立命名规范（如"BagUI_Wireframe_v1.3_20240815"），团队成员极易基于旧版本线框图开始开发，造成需求理解不一致。线框图的版本管理与代码版本管理同等重要，每次修改必须在文档内注明"变更记录"，列明哪个模块因何原因在何时被修改。

## 知识关联

UI线框图在设计文档体系中承接**设计文档概述**所建立的文档规范框架：设计文档概述定义了游戏项目所需的文档类型与交付标准，而UI线框图是其中UI设计阶段的具体产物，直接服从该规范中对文件命名、版本管理和评审流程的要求。掌握线框图的绘制能够帮助设计师在进入后续高保真原型阶段时，有更清晰的信息架构作为底层支撑，避免视觉设计阶段因结构问题反复推倒重来。

从工具层面看，游戏行业常用的线框图工具包括Figma（支持多人实时协作）、Axure RP（支持复杂交互逻辑原型）和Balsamiq（专为草图级低保真设计）。不同工具适用于不同精度层级的线框图，团队应在设计规范文档中统一约定工具选型，避免不同设计师交付格式不兼容导致协作摩擦。
