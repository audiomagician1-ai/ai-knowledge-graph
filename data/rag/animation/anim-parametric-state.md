# 参数化状态

## 概述

参数化状态（Parametric State）是动画状态机中一种特殊的状态设计模式：在单一状态内部，动画的播放内容或混合权重会随外部参数（如移动速度、运动方向角度）的变化而实时调整，而无需切换到另一个状态。与普通状态只播放固定动画片段不同，参数化状态将"参数→动画表现"的映射关系封装在状态本身内部，使一个状态就能覆盖大量连续变化的运动表现。

这一概念随着动画混合树（Blend Tree）技术在游戏引擎中的普及而逐渐成型。Unity 引擎在 4.0 版本（2013年）正式将 Blend Tree 集成进 Animator 状态机，使开发者可以在单个状态节点内用一个或两个浮点参数驱动多个动画片段的混合权重。Unreal Engine 的 AnimGraph 同期也提供了 BlendSpace 节点实现同等功能。在此之前，开发者必须为"慢走→快走→奔跑"这三种速度各建一个状态并用 Transition 连接，导致状态数量膨胀、过渡逻辑复杂。

参数化状态之所以重要，在于它解决了"动画数量与状态数量爆炸"的矛盾。一个角色的移动方向可以是 360° 中的任意角度，如果每隔 45° 建一个状态就需要 8 个状态节点；而改用二维参数化状态后，仅需一个状态配合 VelocityX/VelocityZ 两个浮点参数即可覆盖全方向移动，且过渡完全平滑。

## 核心原理

### 参数驱动混合权重的数学基础

参数化状态的本质是一个参数到混合权重的映射函数。以一维速度参数为例，设参数值为 `v`，状态内包含三个动画片段：`Idle`（基准速度 0）、`Walk`（基准速度 1.5 m/s）、`Run`（基准速度 6 m/s）。当 `v = 3.0` 时，引擎计算 `v` 在 Walk 基准值和 Run 基准值之间的线性插值比例：

```
weight_Run  = (3.0 - 1.5) / (6.0 - 1.5) ≈ 0.333
weight_Walk = 1 - weight_Run             ≈ 0.667
```

最终输出骨骼姿态 = Walk姿态 × 0.667 + Run姿态 × 0.333。这种逐帧线性插值（LERP）确保参数连续变化时动画表现也连续，不会出现跳帧。

### 一维与二维参数化状态的区别

**一维参数化状态**只使用单个浮点参数（如 `Speed`），适合表达速度大小的变化——从 0（静止）到最大速度（冲刺）的单轴连续变化。  

**二维参数化状态**使用两个浮点参数（通常是 `VelocityX` 和 `VelocityZ`），在一个二维平面上放置多个动画片段作为"控制点"，例如：前跑(0,1)、后退(0,-1)、左跑(-1,0)、右跑(1,0)以及斜向运动(0.7,0.7)等。当前参数坐标落在某几个控制点围成的三角形内时，引擎使用重心坐标（Barycentric Coordinates）计算三个顶点动画的混合权重：

```
姿态 = 动画A × wA + 动画B × wB + 动画C × wC
wA + wB + wC = 1，且每个 w ∈ [0, 1]
```

Unity 的 BlendSpace 2D 对应 Unreal 的 BlendSpace 资产，均使用这一三角剖分（Delaunay Triangulation）算法确定权重。

### 参数平滑与响应延迟

直接将物理速度值送入参数化状态会导致动画抖动，因为物理层每帧都可能产生小幅噪声。解决方案是对输入参数施加阻尼平滑：

```
smoothedSpeed = Lerp(smoothedSpeed, targetSpeed, smoothTime * deltaTime)
```

Unity Animator 提供了内置的 `dampTime` 参数（默认值通常设为 0.1 秒），专门用于控制参数化状态的跟随响应速度。将 `dampTime` 设为 0 意味着参数瞬时跟随，会还原物理抖动；设为 0.3 秒以上则过渡太慢，导致角色"滑步"感。通常建议速度参数 `dampTime` 取 0.05～0.15 秒，方向角度参数取 0.1～0.2 秒。

## 实际应用

**角色移动方向混合**：第三人称游戏中，角色可以向任意方向平移（Strafe）。开发者在移动状态内创建一个 2D Blend Tree，X 轴对应 `MoveX`（范围 -1 到 1），Y 轴对应 `MoveZ`（范围 -1 到 1），在 8 个方向各放置对应的移动动画。角色斜向移动时，引擎自动混合相邻方向动画，无需任何 Transition 条件。

**速度分层混合**：赛车游戏中，载具的车身摆动幅度随车速增大。设计师在"驾驶"状态内用 `Speed`（0～200 km/h）参数混合"低速轻微晃动"与"高速剧烈晃动"两段叠加层动画，车速每超过 150 km/h 后"高速晃动"动画权重超过 0.5，驾驶感明显增强。

**受伤程度表现**：RPG 游戏的"行走"状态内，用 `HealthRatio`（0.0 满血～1.0 濒死）参数混合正常步行与跛行动画。当角色 HP 低于 20% 时 `HealthRatio` 超过 0.8，跛行动画权重占主导，无需额外的受伤状态即可实现平滑过渡。

## 常见误区

**误区一：认为参数化状态可以替代所有状态切换**。参数化状态适合处理*连续变化*的运动参数，但攻击、翻滚、死亡等*离散事件*触发的动画依然需要独立状态和 Transition。如果把攻击动画也塞入参数化状态用某个参数控制，会导致无法精确控制攻击动画的起止帧，也无法正常播放不可打断的动画段。

**误区二：混合控制点数量越多越好**。Unity 官方文档指出，单个 Blend Tree 内超过 10 个控制点后，三角剖分计算量显著上升，且相邻动画之间的权重区域变得极窄，动画差异几乎不可见。实践中 2D Blend Tree 通常放置 5～9 个控制点，一维 Blend Tree 放置 3～5 个片段即可满足大多数需求。

**误区三：参数值范围与动画片段基准值不匹配**。若动画片段的基准速度设为 5 m/s，但实际游戏角色最大速度只能达到 3 m/s，则"跑步"动画的最大权重仅为 `3/5 = 0.6`，角色永远处于走跑混合状态，表现为"走得很快但动作始终不像全力奔跑"。正确做法是在设计阶段让动画师确认每个参考动画对应的精确速度值，并写入引擎的 BlendTree 基准参数。

## 知识关联

参数化状态直接建立在**移动状态机**的基础之上。移动状态机解决了"什么时候切换状态"的问题，而参数化状态解决了"在一个状态内如何随参数变形"的问题。学习参数化状态需要先理解状态机的状态节点与 Transition 条件机制，才能判断哪些运动表现应用参数化混合处理，哪些应用状态跳转处理。

在动画系统的技术栈中，参数化状态与**动画层（Animation Layer）**协同使用时发挥最大价值：下层用参数化状态处理全身移动混合，上层叠加层用 Mask 只影响上半身，再由各自的参数独立控制，从而实现"奔跑同时开枪且枪口随瞄准角度上下偏转"这类复合动画表现。这也是理解动画分层架构的关键切入点。
