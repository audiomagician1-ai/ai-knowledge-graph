# 子状态机

## 概述

子状态机（Sub-State Machine，简称 SSM）是动画状态机中的一种嵌套结构，允许将一个复杂的状态集合封装为单个可复用的节点，供外层状态机引用。在 Unity Animator 等主流动画系统中，子状态机在层级视图中以六边形图标区别于普通矩形状态节点，点击即可进入其内部编辑视图。这种"状态中有状态"的设计使得动画图不必在同一平面展开所有逻辑。

子状态机概念源自 Harel 在 1987 年发表的论文《Statecharts: A Visual Formalism for Complex Systems》中提出的分层状态图理论。该理论明确指出，当状态数量超过 7±2 个时（源自 Miller 定律），人类理解单层状态图的能力会显著下降。游戏动画工具借鉴这一理论，在 2010 年代的 Unity 4.x 和 Unreal Engine 4 中先后引入了子状态机功能，将层级管理能力带入了实时动画制作流程。

子状态机最核心的价值在于：它允许把属于同一语义组的动画状态打包，例如把"游泳"相关的入水、踩水、出水三个状态封装为一个名为"Swimming"的子状态机，让外层的角色主状态机只看到一个"Swimming"节点，而不是三条互相交叉的连线。这直接降低了动画图的视觉复杂度，并让多个角色共享同一子状态机模板成为可能。

## 核心原理

### 出入口节点：Entry 与 Exit

子状态机内部必须包含两类特殊节点：**Entry（入口）**和 **Exit（出口）**。当外层状态机的某个过渡指向子状态机时，动画系统不会随机选择内部状态，而是沿 Entry 节点上挂载的条件依次评估，进入第一个满足条件的内部状态。Exit 节点的作用相反：当内部某状态触发到 Exit 的过渡时，控制权归还给外层状态机，由外层继续评估后续过渡。

这意味着子状态机有清晰的**契约边界**：外部只需关心"如何进入"和"何时退出"，内部细节完全被封装。例如，一个"攻击"子状态机内部可能有前摇、出手、后摇三个状态，但外层状态机只需配置"按下攻击键 → 进入攻击子状态机"以及"攻击子状态机退出 → 返回待机"两条规则。

### 层级过渡与优先级

子状态机的过渡优先级遵循**由外到内**的评估顺序。外层状态机的 AnyState 过渡可以打断正在运行的子状态机内部任何状态，这一特性在 Unity 中称为"中断源（Interruption Source）"配置。若将中断源设为 "None"，则外层 AnyState 无法打断子状态机内部正在播放的过渡；若设为 "Current State"，则当前所在子状态机内部的条件优先被评估。开发者需要精确理解这两层优先级的叠加规则，否则容易出现动作"卡住不退出"的 Bug。

### 嵌套深度与性能边界

子状态机支持多层嵌套——即子状态机内部还可以再包含子状态机。Unity 官方文档建议嵌套深度不超过 **3 层**，Unreal Engine 的动画蓝图同样有类似的工程实践限制。每多一层嵌套，状态机评估器在每帧 Update 时就需要多一次层级遍历，这在同屏角色数量超过 50 个时会产生可测量的 CPU 开销。因此，嵌套层数应与角色动画逻辑的实际层级深度匹配，而不是为了视觉整洁而无限嵌套。

## 实际应用

**角色运动分组**：以一个第三人称跑酷游戏为例，角色的地面运动包含静止、行走、跑步、冲刺四个状态，而空中运动包含起跳、上升、顶点悬浮、下落、落地五个状态。若全部展开在主状态机中，仅这两组就会产生至少 20 条过渡连线。将地面运动封装为"GroundLocomotion"子状态机，将空中运动封装为"AirLocomotion"子状态机后，主状态机只剩 2 个节点和 4 条连线（地面↔空中的互相进出），可读性大幅提升。

**技能系统复用**：MOBA 游戏中多个英雄共享"普通攻击"动画逻辑：前摇（0.1s）→ 打击帧 → 后摇（0.2s）。将此逻辑制作为独立的"BasicAttack"子状态机资产后，所有使用相同攻击节奏的英雄的 Animator Controller 都可以引用同一份子状态机，修改一次即全体生效，避免了在 15 个英雄的 Animator 中重复维护相同的三状态链。

**受击与死亡状态封装**：受击状态通常包含轻击、重击、硬直三个子状态，死亡状态包含跌倒前摇、着地、倒地循环三个子状态。将它们各自封装为子状态机后，可在 AnyState 层级直接过渡到"Hit"或"Death"子状态机，而不需要为每个可能被打断的状态单独连线到死亡或受击逻辑，有效将连线数量从 O(n²) 降低到 O(n)。

## 常见误区

**误区一：认为子状态机等同于独立的 Animator Layer**
子状态机与 Animator 的 Layer（动画层）是完全不同的概念。Layer 用于同时混合多个独立的动画逻辑（如上半身与下半身），它们可以并行运行；而子状态机是嵌套在单一状态图内部的封装单元，在同一时刻整个状态机只有一条活跃的执行路径穿越嵌套层级。把本应分 Layer 处理的逻辑（例如说话表情与奔跑）强行用子状态机嵌套，会导致动画无法并行混合。

**误区二：Exit 过渡会立即生效**
很多开发者误以为内部状态触发到 Exit 的过渡会像普通过渡一样立即将控制权交回外层。实际上，Exit 过渡同样受到 Transition Duration（过渡持续时间）和 Has Exit Time（是否等待动画结束）参数控制。若"后摇"状态到 Exit 的过渡设置了 0.25 秒的混合时间，则外层状态机在这 0.25 秒内实际上处于一种"半在子状态机内"的混合状态，此期间外层某些过渡可能无法被正常触发，需要特别配置中断规则。

**误区三：子状态机可以无限减少过渡连线数量**
子状态机能减少的是**视觉上展示的**连线数量，并非实际逻辑过渡的数量。封装后的子状态机内部连线依然存在并在运行时被评估。如果内部状态之间存在大量互相过渡，问题根源是状态设计本身过于复杂，而非连线太多——此时应重新审视状态划分逻辑，而不是依赖子状态机来掩盖设计问题。

## 知识关联

**前置概念——移动状态机**：移动状态机是学习子状态机的直接基础，因为移动状态（静止、走、跑、冲刺）通常是第一个被封装进子状态机的状态组。理解单层移动状态机中各状态之间的过渡条件参数（如 Speed 浮点参数的阈值配置），是正确设置子状态机 Entry 条件的前提。

**后续概念——层级状态机**：子状态机是层级状态机（Hierarchical State Machine，HSM）的工程实现手段之一。层级状态机进一步引入了"父状态的过渡可以被所有子状态继承"这一机制，例如在任意子状态下触发"死亡"条件都可以退出整个层级。掌握了子状态机的 Entry/Exit 机制和优先级规则后，理解层级状态机中的继承过渡会更加直观，因为两者都依赖同一套"由外向内"的评估顺序框架。
