# 中断优先级

## 概述

中断优先级（Interrupt Priority）是动画状态机中用于决定哪个动画状态可以强制终止当前正在播放的状态并立即切换的规则体系。当角色同时收到多个动画切换请求时，系统根据每个转换的优先级数值，选择优先级最高的请求执行，忽略或延迟其余请求。

该机制最早在游戏引擎的动画系统中被显式化和系统化，Unity引擎在其Animator Controller中将其命名为"Interruption Source"（打断来源），并在Unity 5.0版本引入了更细粒度的打断控制选项。在此之前，开发者只能依赖转换条件的触发顺序来间接控制打断行为，逻辑混乱且难以维护。

中断优先级的存在意义在于模拟现实世界中动作响应的紧急程度差异。例如，角色受到攻击时的"受击"动画必须能够打断正在播放的"普通攻击"动画，否则玩家会觉得角色"无敌"，丧失游戏的打击感和真实感。如果没有明确的优先级规则，状态机可能产生不可预测的切换行为，导致动画系统在实际运行时难以调试。

## 核心原理

### 优先级的数值定义与排序

在大多数动画状态机实现中，优先级用整数表示，数值越大（或越小，取决于引擎约定）代表优先级越高。以Unity Animator为例，状态之间的转换条件列表中，列表越靠上的转换拥有越高的检测优先级——Unity按照从上到下的顺序逐一检查条件，一旦找到满足条件的转换，立即执行并停止继续检查。因此，开发者将"死亡"转换放在列表第1位，"受击"放在第2位，"普通移动"放在最后，就形成了一套优先级体系。

### Interruption Source的四种模式

Unity提供了四种具体的打断来源设置，直接决定哪些状态有资格发起打断：

- **None**：当前转换不可被任何状态打断，动画必须完整播放完毕（或到达退出时间）才能切换；
- **Current State**：只有当前正在播放的状态（转换的源状态）上的其他转换可以打断本次转换；
- **Next State**：只有目标状态（转换即将切入的状态）上的其他转换可以打断；
- **Current State Then Next State** / **Next State Then Current State**：组合模式，定义了两类来源的检查顺序，两个数字代表检查优先级——先检查Current State的转换，再检查Next State的转换，或反之。

### 打断期间的混合计算

当高优先级转换在一个进行中的转换（Transition）期间发起打断时，引擎需要处理三层状态的混合：已完成的源状态A、正在混合中的目标状态B，以及新打断进来的状态C。Unity使用"Interrupted Transition"机制，将当前混合快照冻结为一个虚拟的"起点姿态"，然后以该姿态为基础开始向C过渡。其混合权重计算公式可以简化表示为：

**最终姿态 = Lerp(frozen_pose, C_pose, t)**

其中 `frozen_pose` 是被打断瞬间A与B的混合结果快照，`C_pose` 是新目标状态C在当前时间的骨骼姿态，`t` 是新转换的归一化进度（0到1）。这确保了打断发生时动画不会突变跳帧。

## 实际应用

**格斗游戏的攻击连段系统**：在格斗游戏中，"轻攻击"的优先级低于"重攻击"，重攻击又低于"必杀技"。设计时将必杀技转换的优先级数值设为1，重攻击为2，轻攻击为3，系统在同一帧同时触发多个条件时，选择优先级为1的必杀技执行，完整实现了"大招优先"的设计意图。

**第三人称射击游戏的受击打断**：角色在换弹动画（Reload，约2.5秒）进行到1.2秒时被子弹命中，受击动画的Interruption Source设置为"Current State"，受击转换排在转换列表第一位，因此换弹动画被立即中断，角色播放受击硬直，之后由受击状态的出口转换自然衔接回待机或移动状态。

**UI菜单的动画打断**：在UI动画状态机中，"打开面板"动画（Open，持续0.3秒）进行到一半时玩家点击关闭按钮，"关闭面板"动画（Close）被配置为可打断"打开"转换。若不设置优先级而让两个转换并行，面板会先弹出再收起，出现视觉抖动；正确设置打断优先级后，关闭动画从当前位置平滑反向收起，用户体验流畅。

## 常见误区

**误区一：认为优先级高的状态会"自动触发"打断**

中断优先级只决定当多个转换条件同时满足时哪个优先执行，它本身不主动触发任何条件检测。如果高优先级转换的触发条件（如Bool参数`IsHit`）没有被正确设置为true，无论优先级多高，该转换都不会发生。开发者常犯的错误是调高了优先级却忘记在代码中`SetBool("IsHit", true)`，误以为是优先级设置失效。

**误区二：将Any State的转换与中断优先级混淆**

Any State转换是从任意状态都能切换到目标状态的特殊机制，而中断优先级是在已经确定了一个转换正在进行时，决定该转换能否被新请求打断的规则。前者解决"从哪里能跳"的问题，后者解决"跳的过程中能不能被插队"的问题。对于需要全局响应的死亡动画，应使用Any State转换并将其Interruption Source设为None（防止死亡动画本身被打断），而不是单纯依赖优先级数值。

**误区三：认为打断发生后动画会从第0帧重新播放**

实际上，当转换被打断时，Unity引擎保留了打断瞬间的骨骼快照作为新过渡的起点（即前述的`frozen_pose`），并在新的Transition Duration内（通常为0.1秒至0.3秒）平滑过渡到新状态。如果开发者将新转换的过渡时间设为0秒，才会出现瞬间跳切到第0帧的现象，这是过渡时间的问题，而非中断优先级本身的机制缺陷。

## 知识关联

学习中断优先级之前，需要理解**Any State转换**的工作原理：Any State是一种特殊的全局转换入口，当它与中断优先级结合使用时，需要明确Any State转换本身的触发不受其他状态打断规则的约束，而一旦进入普通状态之间的转换过程，才进入中断优先级的管辖范围。掌握Any State的适用场景（全局响应事件）和中断优先级的适用场景（转换进行中的竞争处理），是正确搭建复杂角色动画状态机的基础。

中断优先级作为动画状态机设计中较为精细的控制层，其配置错误通常不会导致程序崩溃，而是表现为动画"感觉不对"的软性问题，因此在调试阶段需要结合Unity Animator的**Live Link**实时预览功能，观察转换列表的高亮顺序和当前活跃转换的打断状态，才能快速定位优先级配置的失误所在。
