# 层级状态机

## 概述

层级状态机（Hierarchical Finite State Machine，HFSM）是一种将状态按树形层级组织的状态机设计模式，由计算机科学家 David Harel 于 1987 年在论文《Statecharts: A Visual Formalism for Complex Systems》中首次系统提出。与扁平状态机相比，HFSM 允许父状态持有公共行为，子状态自动继承这些行为，从而大幅减少状态转换规则的总数量。

在动画系统中，HFSM 解决的核心问题是**状态爆炸**（State Explosion）。假设一个角色有 10 种移动姿态和 5 种武器持握方式，扁平状态机需要 50 个独立状态；而 HFSM 只需在"移动"父状态下放置 10 个子状态，"武器持握"作为另一个父状态的子状态，总状态数骤降为 15 个，转换边数也从 O(n²) 级别降低到接近 O(n) 级别。

Unity 的 Animator Controller 和 Unreal Engine 的 Animation Blueprint 都在其动画状态机中实现了 HFSM 结构——Unity 通过 Sub-State Machine 实现多层嵌套，Unreal 通过 State Machine 节点的嵌套调用实现相同功能。

## 核心原理

### 状态树的继承机制

HFSM 的每个状态节点分为**复合状态**（Composite State）和**叶状态**（Leaf State）两类。复合状态可以包含子状态，自身携带一组转换条件；叶状态不包含子状态，负责实际播放动画片段。当系统处于某个叶状态时，其所有祖先复合状态的 `OnUpdate()` 逻辑**同时激活**。例如，角色处于"奔跑→左转"叶状态时，"运动"父状态的重力应用逻辑仍然执行，无需在"左转"中重复编写。

继承规则具体表现为：若父状态定义了"按 B 键进入翻滚"的转换，则所有子状态自动拥有该转换，无需逐一配置。这一机制将公共转换的维护点从 N 个（扁平结构）压缩为 1 个（父状态）。

### 历史伪状态（History Pseudostate）

HFSM 引入了扁平状态机没有的**历史状态**概念，分为浅历史（Shallow History，符号 H）和深历史（Deep History，符号 H*）两种。浅历史记录离开某复合状态时该层级的最后活跃子状态；深历史记录整个子树最深层的最后活跃叶状态。

在动画场景中，角色从"战斗"状态被打断切换到"受击"动画，受击结束后通过深历史状态可以精确恢复到被打断时的"战斗→重攻击→第2段"而非回到"战斗"层级的默认入口"普通攻击"。没有历史状态时，开发者需要手动保存并恢复当前状态指针，代码量增加约 30-50 行且容易产生遗漏。

### 转换优先级与事件冒泡

HFSM 中的事件处理遵循**由内到外**的冒泡顺序：事件首先由当前叶状态处理，若叶状态未消费该事件，则向父状态传递，逐层上溯直到根状态。这与 HTML DOM 事件冒泡机制完全相同。

转换优先级规则：**层级越低，优先级越高**。子状态定义的转换会覆盖父状态对同一事件的处理。例如，父状态"空中"监听着陆事件切换到"站立"，而子状态"空中→攻击"可以拦截着陆事件先切换到"空中着陆攻击"，不向上冒泡。若不遵守此规则，将导致父状态抢先消费子状态本应处理的输入，产生动画跳帧 bug。

## 实际应用

**人形角色动画控制器**是 HFSM 最典型的应用场景。常见的三层结构为：第一层区分"地面"与"空中"两大父状态；第二层在"地面"下划分"静止""移动""战斗"；第三层在"移动"下列出"行走""慢跑""冲刺""爬行"等叶状态。全部叶状态约 20-30 个，但转换规则数量相比扁平结构减少约 60%。

**载具动画系统**同样受益于 HFSM。以赛车游戏为例，"驾驶"父状态统一处理引擎震动和轮胎旋转动画，子状态只需专注于"加速""漂移""碰撞"等差异性行为。当新增"水面行驶"子状态时，仅需将其放入"驾驶"父状态下并添加水花粒子逻辑，引擎震动等公共行为自动继承，无需复制粘贴。

Unity 中配置 HFSM 时，通过右键状态机画布选择 **Create Sub-State Machine** 创建子状态机节点，双击进入其内部编辑。跨层级转换（从子状态直接跳转到另一个父状态的子状态）使用 Unity 的 **Entry/Exit** 伪状态作为跨层出入口，而非直接连线，以确保每层的进入和退出回调（`OnStateEnter` / `OnStateExit`）都被正确触发。

## 常见误区

**误区一：层级越深越好。** 许多开发者看到 HFSM 能减少重复后，将状态树设计到 5 层甚至更深。实际上超过 3 层的 HFSM 会使调试极为困难——当动画行为异常时，需要沿着整条祖先链检查每一层的转换条件和继承逻辑。行业经验表明，动画 HFSM 保持在 **2-3 层**时可维护性最佳，超出部分应考虑拆分为独立的并行状态机（Blend Tree 或 Layered Animation）。

**误区二：父状态的退出转换会中断子状态的播放。** 正确行为是：当父状态的退出条件满足时，无论当前子状态播放进度如何，整个子树立即退出，所有祖先的 `OnStateExit` 按**由内到外**的顺序依次调用。不少初学者误以为父状态转换需要等待子状态完成当前动画片段，导致设计出用 bool 变量手动"锁死"父状态的冗余代码。

**误区三：历史状态可以跨越父状态边界恢复。** 历史状态只能恢复**同一父状态内部**的子状态，无法记录跨越不同父状态的复合路径。若需跨层恢复，必须在多个复合状态分别配置历史伪状态，并确保历史记录的初始化默认值指向正确的入口子状态。

## 知识关联

HFSM 直接建立在**子状态机**概念之上——子状态机解决的是"如何将一组状态封装为可复用模块"，而 HFSM 进一步规定了这些嵌套层级之间继承、冒泡和历史恢复的完整行为规范。掌握子状态机的封装边界和 Entry/Exit 节点用法，是正确实现 HFSM 转换路由的前提。

向上延伸，HFSM 是**状态机架构设计**的主要讨论对象之一：何时使用 HFSM、何时改用行为树（Behavior Tree）、何时结合有限状态机与 Blend Tree，都需要以 HFSM 的层级代价和继承收益为基准进行权衡。在**状态机性能**方向，HFSM 的树形遍历在每帧 Tick 时需要从根节点定位当前活跃叶状态，其时间复杂度为 O(d)（d 为树深度），这一特性直接影响大量 NPC 并行运行时的 CPU 开销优化策略。
