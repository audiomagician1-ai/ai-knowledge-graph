# 状态转换

## 概述

状态转换（State Transition）是动画状态机中连接两个动画状态的有向箭头，定义了角色从一个动作切换到另一个动作的具体方式。在 Unity Animator 或 Unreal Engine 的动画蓝图中，每条转换线都携带三类关键数据：触发转换的**条件（Condition）**、过渡所需的**持续时间（Transition Duration）**，以及两段动画之间的**交叉淡入淡出（Cross-Fade）**混合曲线。没有这三项设置，状态机只是一组孤立的动画片段，无法产生流畅的运动感。

状态转换的概念随着游戏引擎的动画系统演化而来。早期（2000年代初）的游戏动画采用硬切换（Hard Switch），即直接跳帧，会产生明显的画面抖动。Unity 3.0（2010年）引入 Animator 组件并将转换持续时间默认设为 0.25 秒，这一设计奠定了现代引擎中"软过渡"的工业标准。

状态转换的质量直接决定角色动画是否"生硬"。一个配置错误的转换——例如持续时间过短导致腿部穿模，或条件设置不当导致状态循环抖动——会立即破坏玩家的代入感。

## 核心原理

### 转换条件（Transition Condition）

转换条件是触发状态跳转的逻辑门。在 Unity Animator 中，条件基于四种参数类型：`Float`、`Int`、`Bool` 和 `Trigger`。Float 和 Int 支持大于/小于/等于比较运算，例如设置 `Speed > 0.1` 表示速度超过 0.1 时从 Idle 切换到 Walk；Bool 直接判断真假；Trigger 是一次性信号，调用 `SetTrigger("Attack")` 后消耗即清零，不会像 Bool 那样持续保持状态。

一条转换可以叠加**多个条件**，逻辑关系为"AND"，即所有条件同时满足才触发。如需"OR"逻辑，必须创建两条独立的转换箭头指向同一目标状态。这一限制是设计复杂转换网络时最常踩到的陷阱。

### 过渡时间（Transition Duration）

过渡时间决定两段动画混合的时长，通常以**秒**或**当前动画的归一化比例（0.0～1.0）**表示。Unity 默认使用归一化比例，值 `0.25` 表示当前动画剩余长度的 25%；勾选"Fixed Duration"后改为绝对秒数。

选择过渡时间需参考动作类型：
- **循环动作互切**（如 Walk → Run）：0.1～0.2 秒，防止步伐错位。
- **激活动作进入循环**（如 Idle → Walk）：0.15～0.25 秒。
- **受击或死亡等打断动作**：0.05～0.1 秒，快速响应优先。
- **技能动画接续**：需精确计算关键帧位置，过渡时间常设为 0.0 以实现"接帧"效果。

**退出时间（Exit Time）**是一个附加控制项，值为归一化时间，例如 `0.9` 表示当前动画播放到 90% 时才允许转换被触发，常用于攻击动画的"前摇锁定"设计。

### 交叉淡入淡出（Cross-Fade）

交叉淡入淡出描述两段动画的权重如何随时间变化。Unity 内部使用一条可配置的**混合曲线（Blend Curve）**，默认为线性插值：若过渡时间为 T 秒，则在第 t 时刻旧动画权重为 `(T-t)/T`，新动画权重为 `t/T`，两者之和始终为 1.0。

非线性曲线（如 Ease-In-Out）可以让过渡在开始和结束阶段更平缓，适合从 Idle 进入复杂技能动画的场景；而线性曲线适合节奏均匀的移动状态切换。Unreal Engine 的 AnimGraph 提供了`Blend Poses by Bool`节点，允许为每个方向单独配置混合时间：进入时 0.2 秒，退出时 0.1 秒，实现不对称过渡。

## 实际应用

**第三人称角色移动系统**是最典型的状态转换场景。以常见的三状态机为例：`Idle → Walk → Run`，每对状态之间建立双向转换。`Idle → Walk` 条件设为 `Speed > 0.1`，过渡时间 0.2 秒；`Walk → Run` 条件设为 `Speed > 5.0`，过渡时间 0.15 秒；反向转换 `Run → Walk` 条件设为 `Speed < 4.5`，注意此处阈值刻意低于 5.0，形成**滞后区间（Hysteresis Band）**，防止玩家在临界速度附近触发反复横跳。

**受击打断机制**中，攻击动画转换到受击动画需要将过渡时间设为接近 0（如 0.05 秒）并关闭 Exit Time，确保受击反馈立即可见。若不关闭 Exit Time，角色可能要等攻击动画播放 90% 后才能显示被打中，严重影响游戏感（Game Feel）。

**武器切换动画**中，开发者常用 `Has Exit Time = false` + `Trigger` 参数组合：玩家按下换枪键时发送 Trigger，无论当前动画进行到何处，立刻开始 0.1 秒的交叉淡出，切换至收枪动画。

## 常见误区

**误区一：过渡时间越长越流畅。** 错误。过渡时间超过 0.3 秒后，玩家会感受到明显的"响应延迟"，尤其在跳跃或攻击等即时操作动作上。流畅感来自骨骼姿态的相似度（即两段动画在切换点的关键帧接近），而非简单延长混合时长。正确做法是优化动画资源本身，让相邻状态的过渡帧姿态尽量一致，而不是依赖长时间混合掩盖问题。

**误区二：Trigger 和 Bool 可以互换。** Trigger 在被 `SetTrigger()` 调用后仅保留一帧，若当前帧转换未发生（例如 Exit Time 未到），Trigger 会被消耗但不触发转换，导致动作"吃掉"——玩家按了攻击键但角色没有任何反应。对于需要严格一一对应的玩家输入，应用 Trigger；而对于持续状态（如"是否在地面上"）则用 Bool。

**误区三：默认的 0.25 归一化过渡时间适用于所有动画。** 对于一段仅有 0.5 秒的短动画，0.25 归一化时间意味着整个动画有 1/4（约 0.125 秒）都在混合，严重压缩了动画的有效表现区间。短动画应切换为 Fixed Duration 并手动设置绝对秒数。

## 知识关联

状态转换建立在**状态机基础**之上——必须先理解"状态"是什么（一段循环或单次播放的动画片段），才能理解转换如何连接它们。

掌握状态转换后，下一步是学习**转换规则（Transition Rules）**，它将条件系统扩展为可编程的逻辑函数，允许在代码中动态修改转换条件而非硬编码在 Inspector 面板里。**移动状态机（Locomotion State Machine）**则是将本文所有原理应用于完整角色移动系统的综合案例，涉及八方向混合树与状态转换的协同工作。**状态通知（State Notify）**解决了"如何在转换发生的瞬间触发声音或特效"的问题，是状态转换的事件扩展。**混合基础（Blend Tree Basics）**与状态转换共享交叉淡入淡出的数学模型，但作用范围从状态之间扩展到了单状态内部的多动画混合。
