# 转换规则

## 概述

转换规则（Transition Rules）是状态机中控制何时从一个动画状态切换到另一个状态的判断逻辑。在Unity的Animator Controller或Unreal Engine的动画蓝图中，每条状态转换箭头上都附加着一组转换规则，只有当这些规则的条件被满足时，状态机才会执行切换动作。转换规则的本质是一个布尔表达式——最终计算结果必须为 `true`，跳转才会发生。

转换规则的概念随着游戏引擎的动画状态机工具逐步成熟。Unity在2012年随4.0版本引入Mecanim动画系统时，将转换条件（Transition Condition）设计为可视化的参数比较界面，开发者无需手写代码即可定义切换逻辑。这一设计让美术人员也能直接参与动画逻辑的配置，极大降低了动画状态机的使用门槛。

转换规则的正确配置直接决定角色动画的流畅度与响应性。一条写错的转换规则可能导致角色永远跑不起来，或者死锁在某个状态无法退出，因此理解其三种核心触发方式——布尔条件、参数比较和事件触发——是搭建任何实用动画状态机的前提。

## 核心原理

### 布尔条件（Bool Condition）

布尔类型参数是最简单的转换规则形式。开发者在状态机中声明一个 `bool` 参数，例如 `isGrounded`，然后在转换规则中指定其值为 `true` 或 `false`。当 `isGrounded == false` 时，从"跑步"状态切换到"下落"状态。

布尔条件的典型用途是表达"开关"类状态，例如角色是否持有武器、是否处于潜行模式。需要注意的是，布尔参数不会自动重置——如果在代码中将 `isAttacking` 设置为 `true`，状态机会在下一帧检测到条件满足并触发跳转，但若不手动将其改回 `false`，状态机可能在循环中反复满足条件导致错误跳转。

### 参数比较（Parameter Comparison）

参数比较规则适用于 `int` 和 `float` 类型参数，支持六种比较操作符：`Greater`（>）、`Less`（<）、`Equals`（==）、`NotEqual`（!=），以及针对整数的 `Greater or Equal` 和 `Less or Equal`。例如，定义规则 `speed > 0.1` 来区分静止与行走状态，或用 `health < 25` 触发濒死动画。

浮点数比较存在一个精度陷阱：由于浮点运算误差，直接使用 `speed == 0.0f` 几乎永远不会触发，正确做法是使用 `speed < 0.05f` 设置一个容差阈值（Dead Zone）。Unity的Animator面板中的"Threshold"字段正是为此设计，通过设定一个合理的比较阈值，可以避免因浮点抖动导致动画频繁抖动切换（称为"状态抖振"问题）。

多个参数比较条件可以同时附加在同一条转换上，它们之间的关系是逻辑与（AND）——所有条件必须同时满足，转换才会发生。如需逻辑或（OR）效果，需要创建两条独立的转换箭头分别指向目标状态。

### 事件触发（Trigger）

`Trigger` 是一种特殊的参数类型，行为上类似于"一次性信号"。调用 `animator.SetTrigger("Attack")` 后，参数被标记为激活状态；一旦状态机消耗了这个触发信号完成一次跳转，该参数会自动重置为未激活状态。这与布尔参数的核心区别在于：触发器是自动消耗型的，专为"按下即触发、执行一次"的场景设计，例如攻击、翻滚、受击反应。

触发器有一个重要的排队（Queuing）行为：如果当前转换尚未满足（例如当前状态设置了 `Can Transition Time` 限制），触发信号会等待，直到下一个可用时机再被消耗。然而，如果在等待期间再次调用 `SetTrigger`，Unity会丢弃先前未消耗的信号而只保留最新的一个，这在连击系统中容易引发逻辑错误。

## 实际应用

**跑步→跳跃的转换规则**：在平台跳跃游戏中，可以定义 `isGrounded == false` 与 `verticalSpeed > 0.5f` 两个条件的组合，确保只有角色确实离地向上运动时才播放起跳动画，而非下落时直接跳入起跳帧。

**攻击连击链**：以一个三连击系统为例，转换规则配置为：第一段攻击动画播放到70%进度时（通过 `Exit Time = 0.7`），若检测到 `Attack` 触发器激活，则切换到第二段攻击。将 `Exit Time` 与 `Trigger` 条件组合使用，可以精确控制连击的"接招窗口"帧段。

**血量驱动的姿态变化**：为角色配置 `health` 整型参数，设置从"正常行走"到"受伤蹒跚"的转换规则为 `health < 30`，并将 `Transition Duration` 设置为 0.2秒，实现平滑过渡而非瞬间切换。

## 常见误区

**误区一：混淆触发器与布尔参数的使用场景**
很多初学者将 `Trigger` 参数当成布尔参数使用，或反之。布尔参数适合描述持续状态（如"正在奔跑"），触发器适合描述瞬时事件（如"按下了跳跃键"）。若将攻击动作设计为布尔参数 `isAttacking = true`，则必须在攻击动画结束后手动调用 `SetBool("isAttacking", false)`，而使用触发器则无需此额外操作，自动重置降低了维护成本。

**误区二：认为多条件之间是"或"关系**
同一条转换箭头上添加的多个条件，Unity和Unreal均默认为逻辑与（AND）关系。若期望"满足条件A或条件B时都跳转"，需要从源状态向目标状态画两条独立的转换箭头，每条箭头分别承载一个独立条件，这是初学者最常犯的逻辑配置错误。

**误区三：忽略 Exit Time 与条件的叠加效果**
Unity转换设置中的 `Has Exit Time` 选项若被勾选，会在转换条件的基础上额外要求当前动画播放到指定进度才允许跳转。这意味着即使触发器已经激活，角色也要等到当前动画播放到 `Exit Time`（如0.9，即90%进度）才会响应。许多开发者发现"触发器发出了但动画不切换"，往往就是因为遗漏了勾选状态的 `Has Exit Time` 设置。

## 知识关联

转换规则建立在**状态转换**的基础概念之上——状态转换定义了"从哪里跳到哪里"，而转换规则回答的是"什么时候跳"。理解了状态转换的有向图结构，才能明白为什么转换规则是附着在边（Edge）上而非节点（Node）上的属性。

在动画参数系统层面，转换规则消费的是 `Animator` 组件的参数池中的值，这些值由游戏逻辑代码或行为树在每帧写入。转换规则的三种类型（bool、比较、trigger）直接对应 Animator 参数系统的四种参数类型（`Bool`、`Int`、`Float`、`Trigger`），二者是生产者与消费者的关系。掌握转换规则后，自然可以进一步探索混合树（Blend Tree）如何在单一状态内部通过参数值实现动画插值，从而减少状态数量和转换规则的总体复杂度。
