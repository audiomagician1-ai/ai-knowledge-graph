# Ink脚本语言

## 概述

Ink是由英国游戏公司Inkle Studios开发的开源叙事脚本语言，于2016年正式发布并开源，最初用于驱动《80 Days》和《Heaven's Vault》等商业游戏的交互叙事系统。与Twine使用可视化节点图不同，Ink采用纯文本的标记语法，整个故事以线性文本文件（`.ink`扩展名）的形式编写，然后由运行时引擎解析执行。

Ink的核心设计哲学是"以写作为中心"——语法足够轻量，让作者可以像写散文一样流畅地编写分支故事，而不需要频繁切换到图形界面或管理复杂的节点连线。一个中等规模的互动小说项目，其`.ink`文件往往只有几千行，但经编译后可以生成包含数十万字的分支叙事网络。

Ink通过官方的编辑器**Inky**提供即时预览功能，同时提供JavaScript、C#等多种运行时库，使其能够轻松集成到Unity引擎或网页项目中。这使得Ink成为商业互动叙事游戏开发中使用最广泛的专用脚本语言之一。

## 核心原理

### 选项与跳转语法

Ink用星号`*`表示只能选择一次的玩家选项，用加号`+`表示可以重复选择的选项。每个选项后跟随该选项触发的文本内容，无需手动连线。跳转使用`-> 节点名`语法，结束故事则写`-> END`。一个最基本的分支结构如下：

```
你站在两条路的岔口。
* [走左边的路] 左边的路蜿蜒向山谷延伸。-> 山谷
* [走右边的路] 右边的路直通一座古老的村庄。-> 村庄
```

方括号`[]`内的文字只显示为选项标签，不会在选择后重复打印到正文中，这是Ink控制叙事冗余的核心机制。

### 结（Knot）与缝（Stitch）的层级结构

Ink用`===`定义顶级叙事单元"结"（Knot），用`=`在结内部定义次级单元"缝"（Stitch），形成两级层级结构。这与Twine中每个passage都是平级的扁平结构形成对比。跳转语法支持精确到缝的定位：`-> 山谷.河边`会直接跳到"山谷"结中的"河边"缝，无需创建额外的顶层节点。这种层级设计使得一个场景的所有子状态可以被组织在同一个文本区块内，大幅提升大型项目的可维护性。

### 变量系统与条件逻辑

Ink使用`VAR`关键字声明全局变量，使用`temp`声明局部临时变量。条件逻辑通过`{}`实现内联条件文本，也可以使用多行的`- else:`结构：

```
VAR 好感度 = 0
~ 好感度 = 好感度 + 1
{好感度 > 5: 她对你微笑。| 她冷淡地点了点头。}
```

波浪号`~`前缀标记逻辑语句（变量赋值、函数调用），将其与叙事文本区分开来。Ink内置的`RANDOM(1,6)`函数可以生成指定范围的随机整数，常用于模拟骰子投掷或随机事件触发。

### 线程（Weave）与汇聚点

Ink最有特色的语法特性是"织"（Weave）模式：使用`-`作为汇聚标记，多条分支可以自动流向同一个后续段落，无需为每个分支手动添加跳转指令。这一设计使得"玩家做出选择→结果略有不同→故事继续汇合"这一最常见的叙事模式可以用极少的代码实现，解决了Twine中需要大量跳转节点才能完成汇聚的痛点。

## 实际应用

《80 Days》是Ink最具代表性的商业案例——该游戏包含超过750,000字的分支叙事内容，全部存储在Ink脚本文件中，并实时编译给Unity运行时读取。游戏中的城市对话、事件触发、物品状态跟踪均通过Ink的变量系统管理，而非在Unity C#代码中硬编码。

在独立游戏开发中，Ink常见的工作流是：作者在Inky中完成全部叙事内容并测试通过，导出为JSON格式，再由程序员在Unity中通过官方`ink-unity-integration`插件读取JSON，将文本显示到UI界面。这种分工让叙事设计师可以在完全不接触Unity的情况下完成创作，并随时修改对话内容而不影响程序代码。

对于网页互动小说，Ink提供`inkjs`库，可以直接在浏览器端运行编译后的JSON文件，无需服务器，部署成本极低，因此也被不少jam游戏创作者用于快速制作网页互动小说原型。

## 常见误区

**误区一：认为Ink的`*`和`+`选项没有实质差异。** 这是初学者最频繁犯的错误。`*`选项在被玩家选择一次后，下次到达同一段落时该选项会消失；而`+`选项永远可见。这一差异直接影响重复访问同一节点时的叙事逻辑，如果错误使用`+`，会导致玩家可以无限重复触发"首次见面"对话之类的内容。

**误区二：将Ink变量当作游戏存档系统使用。** Ink的`VAR`变量仅在当前运行时内存中存在，游戏关闭后不会自动持久化。需要存档功能时，必须在Unity等宿主环境中将Ink的`story.state`序列化为JSON字符串，并独立存储到本地文件。很多初学者直接在Ink中设置大量VAR变量后发现"读档"功能需要在Ink之外另行实现，才意识到这一边界。

**误区三：以为Ink可以独立运行，不需要宿主环境。** Ink脚本需要被编译（`inklecate`编译器）成JSON，再由对应平台的运行时库驱动。Inky编辑器内置了这一流程，因此在Inky中预览时感觉是"直接运行"，但实际上Inky本身就是一个Electron宿主程序。单独的`.ink`文件无法被玩家直接打开游玩。

## 知识关联

学习Ink之前掌握Twine工具有助于理解交互叙事的基本概念，如节点、选项、跳转和变量跟踪，但需注意Ink的纯文本工作流与Twine的图形化节点界面在组织方式上存在根本差异——Ink更适合长篇商业项目，Twine更适合快速原型。

在Ink的基础上，**Yarn Spinner**是下一个值得学习的叙事脚本系统：它同样集成Unity，但语法更接近自然语言对话格式，内置了更完善的本地化支持，常见于角色扮演类游戏的NPC对话系统，而非互动小说式的叙事分支。理解Ink的状态机本质（每个Knot是一个状态，`->`是状态转移）也为学习**对话状态机**的正式理论打下基础，帮助开发者从工具使用者过渡到能够自行设计叙事系统架构的层次。
