# Ink脚本语言

## 概述

Ink是由英国游戏工作室Inkle（代表作《80天》《神秘海域》的叙事分支）于2016年开源发布的专用叙事脚本语言，当前稳定版本为1.x系列，托管于GitHub并持续维护。与Twine使用可视化节点图不同，Ink采用纯文本的线性书写方式，允许叙事设计师用类似写作稿件的形式编排带分支的故事，再由Ink运行时引擎（Ink Runtime）将脚本解析为JSON格式，供Unity、Godot等游戏引擎调用。

Ink语言的诞生直接源于Inkle团队在开发《神秘海域：失落的遗产》叙事系统时遇到的痛点——当故事分支超过数百条时，节点图变得难以维护，而纯代码又对非程序员不友好。Ink的设计目标是让作家可以用文本编辑器直接写故事，同时拥有足够的逻辑控制能力。配套的编辑器Inky提供实时预览，可在左侧写代码、右侧即时看到分支对话效果。

在商业项目中，Ink被《天国拯救》的对话系统、《Sable》以及大量独立游戏采用。它之所以重要，在于它将"变量驱动的分支叙事"的实现门槛从程序员降低到了有基本逻辑意识的叙事设计师，同时输出的JSON格式使其能无缝嵌入任何有开发团队支持的引擎管线。

## 核心原理

### 基本语法结构：针织线（Weave）模型

Ink的核心概念叫做**Weave（编织）**，整个故事如同一条可以分叉再合并的线。最基础的文本直接书写即为旁白，选项用`*`（单次可选）或`+`（可重复选）符号标记，缩进代表层级嵌套。例如：

```ink
你站在十字路口。
* [向左走]
    你进入了森林。
* [向右走]
    你看见了村庄。
- 道路在你身后消失了。
```

其中`-`符号是**汇流点（Gather）**，无论玩家选择哪条分支，叙事都会在此处重新合并。这一机制使得设计师不必为每个分支单独写结局，大幅减少重复内容。

### 结（Knot）与缝（Stitch）：故事的章节组织

大型故事用`===`标记命名的**Knot（结）**来分割章节，用`=`标记**Stitch（缝）**来标记章节内的子段落。通过`-> knot_name`语法实现跳转，类似于其他语言的`goto`，但Ink限制了无条件死循环的滥用：

```ink
=== 森林 ===
你进入了黑暗的森林。
-> 遇见狼

= 遇见狼
一匹狼挡住了去路。
-> END
```

`-> END`和`-> DONE`是两个内置关键字，前者结束整个故事，后者仅结束当前隧道（Tunnel）调用返回上级流程，这一区别在设计可复用对话片段时至关重要。

### 变量与条件逻辑

Ink支持三种变量类型：**整数、浮点数、布尔值和字符串**，以及专用的**divert变量**（存储路径引用）。声明语法为`VAR`关键字：

```ink
VAR 好感度 = 0
VAR 已见过村长 = false
```

条件分支使用`{变量: 文本}`的内联语法，也支持多行`{- condition: ... - else: ...}`结构。Ink还独有**列表（List）类型**，专门用于表达游戏状态集合，例如`LIST 天气 = 晴天, 阴天, 雨天`，可通过集合运算判断当前状态组合，这是其他叙事工具极少原生支持的特性。

### 隧道（Tunnel）与线程（Thread）

**Tunnel**允许叙事流程进入一个Knot执行后自动返回，语法为`-> knot_name ->`（双箭头）。这使得可复用的通用场景（如"打开门"动画提示）只需写一次。**Thread**语法`<- knot_name`则允许将多个内容流并联在同一位置展示，用于需要同时呈现多个角色对话选项的场景，但其执行逻辑较复杂，是Ink中最易误用的高级特性。

## 实际应用

**工作流整合**：在Unity项目中，开发者安装`ink-unity-integration`插件后，将`.ink`文件拖入Assets目录即自动编译为`.json`。叙事设计师修改`.ink`文件并保存，编辑器会热重载，程序员无需重新打包即可看到文本更新——这是Ink在团队协作中的核心优势。

**《80天》式的状态追踪**：该游戏使用Ink变量记录玩家经过的城市数量、物品持有状态和NPC关系值。Ink的`~`操作符用于执行逻辑而不输出文本（如`~ 好感度 = 好感度 + 1`），使得在对话段落中穿插状态更新既简洁又直观，不打断叙事流。

**本地化场景**：Ink脚本本身是纯文本，配合`// TODO:`注释系统，叙事设计师可直接在脚本中标注需要翻译的段落，再通过正则表达式批量提取所有非标签文本行，交付给本地化团队，比Twine的HTML节点提取更为规整。

## 常见误区

**误区一：`*`选项和`+`选项效果相同**
`*`（星号）选项在被玩家选择一次后，下次循环到同一选项组时会自动消失；`+`（加号）选项则永远显示。设计师经常在希望表达"已经问过的问题不再出现"时忘记使用`*`，导致玩家可以无限重复选择同一对话，破坏沉浸感。这一差异是Ink选项系统最重要的语义区分。

**误区二：Knot跳转会记录"返回地址"**
用`-> knot_name`跳转后，原位置的流程**不会**自动恢复，故事流完全转移到目标Knot。只有使用Tunnel语法（`-> knot_name ->`）才会记录返回地址。初学者常误以为所有跳转都类似函数调用，结果导致故事流在错误位置终止或死循环。

**误区三：Ink可以独立运行**
Ink脚本本身没有图形界面或音频系统，它只负责输出文本和选项数据。Inky编辑器的预览模式仅用于测试叙事逻辑，最终产品必须依赖Unity/Godot等引擎通过API驱动。设计师不应将Inky预览与玩家实际体验的游戏混为一谈，所有UI表现都需要程序员在引擎侧另行实现。

## 知识关联

学习Ink之前掌握Twine的意义在于：Twine让设计师先建立"分支叙事即节点跳转"的直觉，而Ink将这一直觉转换为文本语法——Twine的`[[选项->节点名]]`与Ink的`* [选项] \n -> 节点名`在语义上完全对应，降低迁移学习的认知负担。

掌握Ink之后学习**Yarn Spinner**时，会发现Yarn Spinner专为角色对话系统设计，其`title:`标签对应Ink的Knot，`<<jump>>`命令对应`->`，但Yarn Spinner原生支持函数调用和事件系统，而Ink通过"外部函数（External Function）"机制实现同等功能，需要程序员在引擎侧注册C#方法。Ink的**变量系统**和**条件逻辑**直接为后续学习**对话状态机**打下基础：Ink脚本中的VAR变量集合与状态机的"状态空间"概念高度对应，每个Knot可以理解为状态机的一个具名状态，`->`跳转即为状态转移箭头，这一映射关系使得从Ink迁移到正式状态机设计时概念过渡更为流畅。
