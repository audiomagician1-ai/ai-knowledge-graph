# 国际化（i18n）

## 概述

国际化（Internationalization）在软件开发中缩写为 **i18n**，因为"i"和"n"之间恰好有18个字母。它指的是在代码架构层面构建对多语言、多地区的支持能力，使产品无需修改核心代码即可适配不同语言环境。对于游戏叙事设计而言，i18n 是将故事文本、角色对话、UI提示与程序逻辑彻底分离的工程基础。

i18n 的概念最早在1980年代随着软件全球化需求兴起，Digital Equipment Corporation（DEC）是最早系统性推广这一术语的机构之一。与本地化（l10n，Localization）不同，i18n 是一次性的架构设计工作，而本地化是在 i18n 框架搭建好之后针对每种语言反复执行的翻译与调适工作。

在叙事游戏中，若 i18n 架构不健全，常见后果包括：中文字符因编码问题显示为乱码、德语文本因字符串长度超出UI边界而被截断、阿拉伯语无法从右到左（RTL）正确排版。这些问题一旦在后期修复，代价极高，因此 i18n 必须在项目立项初期写入技术规范。

## 核心原理

### 1. 字符编码标准化：强制使用 UTF-8

i18n 的第一道防线是统一字符编码。所有文本文件、数据库字段、网络传输必须使用 **UTF-8** 编码，因为它能以1到4字节可变长度表示 Unicode 标准中超过 **143,000** 个字符，覆盖中文、日文、阿拉伯文、西里尔字母等绝大多数书写系统。

反例：若代码库混用 GBK（仅覆盖约20,000个汉字）和 UTF-8，当日语假名或韩文字符进入字符串时会触发编码转换错误，导致叙事文本损坏。正确做法是在项目根配置中声明 `charset=UTF-8`，并在数据库连接串中指定 `CHARACTER SET utf8mb4`（注意：`utf8mb4` 才能支持 emoji 等4字节字符，普通 `utf8` 在 MySQL 中仅3字节）。

### 2. 硬编码字符串的彻底剥离

i18n 最核心的代码实践是**零硬编码原则**：任何出现在游戏中、玩家可见的文字，都不能直接写在源代码里。每一条文本必须通过**唯一键（Key）**引用，实际文字存放在独立的资源文件中。

```
// 错误示范（硬编码）
dialogueText.text = "你找到了古老的卷轴。";

// 正确示范（通过键值引用）
dialogueText.text = LocalizationManager.Get("ITEM_FOUND_ANCIENT_SCROLL");
```

键名通常遵循 `模块_类型_描述` 的命名规范，如 `UI_BUTTON_CONFIRM`、`NPC_MERCHANT_GREETING_01`。这种分离使翻译人员能够在不接触任何源代码的情况下，直接编辑语言资源文件（通常为 `.json`、`.po` 或 `.csv` 格式）。

### 3. 复数规则与语法变形的参数化处理

许多开发者认为复数只需处理"1个"和"多个"两种情况，这一认知仅适用于英语。Unicode CLDR（Common Locale Data Repository）数据库记录了全球语言的复数规则，俄语有**6种**复数形态，阿拉伯语有**6种**，波兰语有**4种**，而中文只有**1种**（没有语法复数变化）。

正确的 i18n 框架应支持带参数的复数模板：

```
"ENEMY_KILLED": {
  "one":   "你击败了 {count} 个敌人。",
  "other": "你击败了 {count} 个敌人。"   // 中文两项相同
}

"ENEMY_KILLED" (Russian):
  "one":   "Вы победили {count} врага.",
  "few":   "Вы победили {count} врагов.",
  "many":  "Вы победили {count} врагов."
```

此外，姓名在德语叙事中会随语法格（Nominative/Genitive/Dative/Accusative）发生形态变化，i18n 框架需预留占位符以允许译者自由调整语序，而非强制使用源语言的句子结构。

### 4. 布局弹性：文本扩张率的预留

不同语言的文本长度差异显著。根据 IBM 国际化研究数据，英文文本翻译为德语后平均扩张 **30%**，翻译为芬兰语可扩张 **50-60%**，而翻译为中文通常会**缩短 20-30%**。UI 设计必须预留足够的动态扩展空间，不能使用固定像素宽度的文本框。

## 实际应用

**《荒野大镖客：救赎2》**的本地化团队在访谈中提到，其 i18n 框架支持超过 **13种语言**的同步发布，所有字幕文本通过统一的键值系统管理，使不同地区的配音文件与字幕文件能独立更新而不影响核心游戏逻辑。

在 Unity 引擎的叙事游戏开发中，**Unity Localization Package**（2021.2版本后进入正式稳定期）提供了内置的 i18n 框架，支持字符串表（String Tables）、资产表（Asset Tables，用于语言相关的音频和图像）以及复数规则配置，开发者通过 `LocalizedString` 组件将 UI 文本绑定到键值，运行时自动切换语言无需重启。

对于使用 Ink 叙事脚本引擎的项目，常见做法是将所有对话文本导出为 `.po` 格式的 **gettext** 文件，交由翻译平台（如 Crowdin 或 Lokalise）处理，然后通过自动化 CI/CD 流程将翻译结果注入构建版本，实现翻译与开发流程的并行推进。

## 常见误区

**误区一：认为 i18n 只是"替换文字"，可以项目后期再做**
这是叙事游戏国际化失败最常见的原因。若代码中存在大量硬编码字符串，后期剥离需要逐行审查数万行代码，且极易遗漏。正确做法是从第一行对话代码起就通过键值系统引用文本，重构成本接近于零。

**误区二：日期、数字、货币格式是通用的**
`2024/01/15`（日本格式：年/月/日）、`15.01.2024`（德国格式：日.月.年）、`01/15/2024`（美国格式）代表同一天。游戏内的日记条目、任务日期若直接格式化为字符串硬编码，将无法适应目标市场的阅读习惯。i18n 框架应始终将日期存储为时间戳或标准 ISO 8601 格式（`2024-01-15`），由本地化层负责按区域格式显示。

**误区三：从左到右（LTR）的界面假设**
阿拉伯语和希伯来语是从右到左（RTL）书写系统，如果 UI 布局硬编码了"文字从左对齐、图标在文字左侧"等假设，RTL 语言版本的界面将出现严重的阅读混乱。i18n 框架需在设计阶段引入 `LayoutDirection` 属性，支持整体界面水平镜像翻转。

## 知识关联

i18n 是本地化工作流的技术地基。一旦 i18n 架构就位，所有可翻译文本都以**键值对**的形式存储，这直接引出下一个概念：**字符串表**（String Tables）。字符串表是 i18n 键值体系的具体存储载体，定义了键的命名规范、文件格式（CSV/JSON/PO）以及与游戏引擎的集成方式。

从叙事设计的工作流角度，理解 i18n 的零硬编码原则能帮助叙事设计师更有效地与工程师沟通：当设计师要求"修改一行对话"时，正确的 i18n 实现意味着这项修改仅需编辑资源文件中的一行文字，而非让程序员修改并重新编译代码。这一区别直接影响多语言项目的迭代速度和版本管理策略。
