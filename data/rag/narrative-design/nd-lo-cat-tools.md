# CAT工具（计算机辅助翻译工具）

## 概述

CAT工具（Computer-Assisted Translation，计算机辅助翻译）是一类帮助人类译者提高翻译效率和一致性的软件系统，其核心机制是**翻译记忆库（Translation Memory，简称TM）**。翻译记忆库会将每一个已翻译的"句段（segment）"与其译文配对存储，当新文本中出现相同或相似的原文时，系统自动提示译者复用已有译文。这与机器翻译（Machine Translation）有本质区别：CAT工具的主角始终是人类译者，工具负责记忆与检索，人类负责判断与创作。

CAT工具的商业化历史可追溯至1984年，Trados公司发布了最早的面向专业译者的翻译记忆软件，此后SDL Trados Studio、memoQ、Phrase（前身为Memsource）成为行业主流工具。游戏与叙事内容本地化领域在2000年代中期开始大规模引入CAT工具，原因是角色扮演游戏（RPG）的文本量急剧膨胀——《最终幻想XII》等标志性作品的可译文本已突破百万字符，单纯依靠人工电子表格管理已无法保证术语一致性。

对于叙事设计师而言，理解CAT工具不只是译者的专业技能，更是内容产出方式的设计依据。当你为NPC对话写出含有"你已完成\{quest_name\}任务"这类字符串时，CAT工具如何解析占位符、如何切分句段，直接影响译文质量和本地化成本。

## 核心原理

### 翻译记忆库（TM）的匹配机制

翻译记忆库以句段为单位存储原文-译文对（source-target pair），并在新文本导入时实时计算相似度。匹配结果分为三个等级：

- **精确匹配（100% match）**：新句段与TM中已有句段完全一致，系统自动填入译文，译者仅需确认。
- **模糊匹配（Fuzzy Match，通常指70%–99%相似度）**：句段有局部差异，系统提示参考译文，译者修改后采纳。相似度低于70%的通常视为"无匹配"，需要从头翻译。
- **上下文匹配（Context Match / 101% match）**：句段本身100%匹配，且其前后句段也与TM记录一致，这类匹配在计费时通常免费或打较大折扣。

模糊匹配阈值和计费折扣率是本地化外包合同的重要谈判参数。例如，一份典型游戏本地化合同会规定：100%匹配按基础字价的15%计费，75%–84%匹配按50%计费，新词（no match）按100%计费。

### 术语库（Termbase / Glossary）集成

CAT工具除翻译记忆外，还强制集成**术语库（Termbase，缩写TB）**。术语库存储项目专有名词的原文-译文对照，例如游戏中的职业名"Rogue"必须统一译为"盗贼"而非"流氓"。当译者翻译时，工具会在原文中高亮命中的术语，并在侧边栏显示规定译法，若译者使用了非规定译法，某些CAT工具（如memoQ的QA模块）会在质量检查阶段报错。

在叙事内容中，术语库的价值尤为突出：同一个NPC名字、地名或技能名可能在数十万字的对话中反复出现，没有术语库强制约束，不同译者很容易产生不一致译法，破坏叙事沉浸感。

### 文件格式与标签处理

CAT工具通过**过滤器（Filter）**解析各类源文件格式，将可译文本抽取为句段，同时将不可译内容（代码、格式标签、占位符）封装为**行内标签（inline tag）**保护起来。

以游戏字符串文件为例，XLIFF（.xlf）和PO（.po）格式被大多数主流CAT工具原生支持。当原文为：

```
Welcome back, <b>{player_name}</b>!
```

CAT工具会将`<b>`、`</b>`和`{player_name}`识别为标签，呈现给译者的句段是：

```
Welcome back, [TAG1]{player_name}[TAG2]!
```

译者必须在译文中保留这些标签并置于正确位置，否则质量检查会报"缺失标签"错误。因此，叙事设计师在编写原始字符串时，应尽量避免将语义关键信息嵌入无法被工具解析的自定义标签格式中。

## 实际应用

**《巫师3：狂猎》多语言版本制作**是CAT工具应用于叙事游戏的典型案例。该游戏包含约45万词的对话文本，CD Projekt RED的本地化团队使用SDL Trados Studio管理翻译记忆库，使得后期DLC内容（《石之心》《血与酒》）的翻译中，约30%的通用UI和系统文本得以通过TM精确匹配直接复用，显著压缩了DLC的本地化周期。

在实际叙事内容生产流程中，CAT工具的使用方式通常如下：叙事设计师输出.xlsx或.xml格式的字符串表 → 本地化工程师用CAT工具的导入过滤器转换为项目包（如memoQ的.mqxlz或Trados的.sdlppx）→ 分发给各语种译者 → 译者在CAT工具客户端内翻译并提交 → 工程师导出译文并回填游戏数据库。这整条链路被称为"本地化管线"，CAT工具处于其核心环节。

## 常见误区

**误区一：CAT工具就是机器翻译**

这是最常见的混淆。CAT工具的翻译记忆库复用的是**人类译者此前生产的译文**，而不是算法自动生成的译文。DeepL、Google Translate等机器翻译引擎可以作为CAT工具的辅助插件调用，但两者是独立的功能模块。专业本地化行业区分这两类工具的关键指标是"译后编辑（Post-Editing）时间成本"：机器翻译的输出通常需要大幅修改，TM精确匹配的输出只需验证确认。

**误区二：100%匹配的句段不需要译者审核**

即便TM匹配率为100%，叙事内容中的语境依然可能使旧译文失当。例如，"你赢了。"这句话在庆祝场景中的情感基调与在讽刺场景中完全不同，但两个句段对CAT工具来说是100%匹配。叙事设计师在字符串注释（context note）中标注场景语境，是避免此类问题的有效手段。

**误区三：CAT工具能处理所有格式**

CAT工具的过滤器有格式支持范围限制。某些游戏引擎使用的私有二进制格式或嵌套JSON结构，主流CAT工具无法直接导入，需要本地化工程师编写自定义过滤器或预处理脚本转换为XLIFF等标准格式。叙事设计师若在字符串设计阶段选用CAT工具友好的标准格式（如XLIFF 2.0），可直接降低本地化工程成本。

## 知识关联

**前置基础**：使用CAT工具不要求特定前置知识，但熟悉字符串表结构（键值对格式、占位符命名规范）能帮助叙事设计师编写出更易被CAT工具正确解析的原始文本。了解Unicode编码规范对于处理多语言字符集也有直接帮助。

**后续概念——本地化管线**：CAT工具是本地化管线（Localization Pipeline）的核心执行节点。掌握CAT工具的工作方式后，学习本地化管线时可以具体理解"字符串提取→分发→翻译→回填"每个节点的技术实现：CAT工具项目包如何打包分发、翻译记忆库如何跨项目共享、以及持续本地化（Continuous Localization）流程中CAT工具API如何与版本控制系统（如Git）对接实现自动化触发翻译任务。
