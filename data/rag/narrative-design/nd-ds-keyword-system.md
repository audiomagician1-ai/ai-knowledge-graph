# 关键词系统

## 概述

关键词系统（Keyword System）是一种基于文本匹配触发对话响应的机制，玩家或用户输入包含特定词汇时，系统检索预设的关键词列表并返回对应的对话内容。与预定义选项菜单不同，关键词系统允许玩家以自然语言方式输入，通过识别关键词来驱动叙事分支。这种机制最早在1970年代的文字冒险游戏（Text Adventure）中广泛使用，Infocom公司的《Zork》系列是其代表性案例。

关键词系统的核心价值在于它以低技术成本模拟"自由对话"的感觉。玩家输入"攻击巨龙"或"打龙"，系统识别出关键词"攻击/打"加上"龙"，返回战斗结果——玩家感受到了自主性，而开发者只需维护一张关键词-响应映射表。这种设计在互动小说、早期RPG和部分现代游戏（如《奥比岛》的对话系统）中至今仍有应用。

理解关键词系统需要区分它与完整自然语言处理（NLP）的本质差异：关键词系统不理解语义，只做字符串匹配，因此设计得当时表现流畅，设计失误时会让玩家频繁遭遇"我不明白你的意思"错误。

## 核心原理

### 触发词映射表结构

关键词系统的基础数据结构是一张映射表，通常表示为：

**`关键词集合 → 响应内容`**

每个条目包含三个要素：**触发词组**（一个或多个词）、**优先级权重**、**响应文本**。例如：

- 触发词：["宝剑", "剑", "武器"] → 响应A（优先级：10）
- 触发词：["生锈的宝剑"] → 响应B（优先级：20）

当玩家输入"我想要那把生锈的宝剑"时，系统同时匹配两条规则，优先级20的条目胜出，返回响应B。优先级机制是关键词系统避免歧义的核心手段，没有它，复合输入会返回错误结果。

### 词干提取与同义词扩展

单纯的精确匹配会导致系统极为脆弱。"攻击"与"攻"、"打击"、"揍"在语义上相同，但字符串不同。设计者通过两种方法解决这一问题：

1. **同义词组（Synonym Groups）**：将语义相同的词归入同一触发组，如 ["看", "观察", "检查", "瞧", "查看"] 统一触发"检查"响应。经典文字冒险引擎Inform 7允许开发者用`Understand "examine/look at/inspect" as examining`这样的语法声明同义词组。

2. **词干提取（Stemming）**：将"跑步"、"跑着"、"在跑"统一还原为词根"跑"再进行匹配。中文关键词系统通常用分词工具（如jieba库）先切词，再对词根进行匹配。

### 缺省响应与失败处理

当输入不匹配任何关键词时，系统触发**缺省响应（Default Response）**。《Zork》的缺省响应是著名的"I don't understand that."，这句话曾被玩家调侃为游戏的第二主角。

优秀的关键词系统设计要求缺省响应分层：
- **通用缺省**：完全无匹配时返回（"我不明白你说的。"）
- **部分匹配缺省**：识别出名词但无动词时返回（"你想对宝剑做什么？"）
- **上下文缺省**：基于当前场景状态返回提示（"这里似乎没有什么值得那样做的。"）

分层缺省响应能将玩家挫败感降低约60%，这是互动小说设计师Emily Short在其设计文章中提出的经验数据。

## 实际应用

### 文字冒险游戏中的双词解析器

Infocom在1980年代使用的解析器采用**动词+名词**的双词结构：系统只提取输入中的第一个动词和第一个名词。玩家输入"用钥匙打开北边的门"，解析器提取"打开"（动词）和"钥匙"（名词，第一个出现），忽略"北边的门"。这种简化虽然损失语义，但使关键词表的维护成本极低，整个《Zork I》的解析器关键词表不超过600个条目。

### 现代游戏中的NPC问答系统

《上古卷轴：晨风》（Morrowind，2002年）使用的对话系统本质上是一个关键词超链接系统：NPC对话窗口显示可点击的关键词，玩家点击后触发对应响应。这个设计将关键词从隐藏机制变为可见界面元素，避免了玩家猜测触发词的挫败感。《晨风》的对话数据库包含超过35万行文本，全部由关键词条目索引。

### 互动小说引擎的关键词配置

以Twine或Inform 7为例，设计者声明关键词触发的典型代码逻辑如下（伪代码）：

```
IF 输入包含 ["帮助", "救", "救命"] AND 场景==地牢
    THEN 触发响应_X
ELSE IF 输入包含 ["帮助"]
    THEN 触发通用帮助响应
```

AND条件将关键词与场景变量结合，使同一关键词在不同上下文中返回不同响应，大幅提升系统的叙事灵活性。

## 常见误区

### 误区一：关键词越多系统越智能

许多初学者认为扩充关键词列表可以无限逼近自然语言理解，但关键词系统的匹配逻辑是线性扫描，条目数量增加到数千条后，**冲突概率**（两条规则同时匹配）急剧上升，优先级管理会变得不可维护。《废土》（Wasteland，1988年）因关键词冲突导致多个对话节点无法正常触发，成为行业中关键词膨胀问题的著名反面案例。正确做法是控制每个场景的关键词密度，而非堆砌条目。

### 误区二：关键词系统可以直接替代选项菜单

关键词系统和选项菜单解决的是不同的玩家体验问题。选项菜单保证玩家永远知道"能做什么"，关键词系统提供探索感但会引发"不知道该输入什么"的迷失感。在叙事节奏紧张、信息传递为优先目标的场景（如关键剧情对话），强行使用关键词系统会降低叙事效率，《晨风》的可点击关键词设计正是为了解决这一矛盾而存在。

### 误区三：缺省响应是系统失败的标志

设计者常试图通过穷举关键词来消灭缺省响应，这会浪费大量资源且仍然失败。正确认知是：缺省响应是关键词系统**正常工作**的一部分，应被设计为引导玩家的叙事工具，而非错误提示。优秀的缺省响应文本本身可以包含叙事信息，如"你嘟囔着什么，但没有人能听懂。"——既不打断沉浸感，又暗示玩家换个表达方式。

## 知识关联

关键词系统不依赖复杂前置知识，但需要学习者理解**字符串匹配**的基本逻辑（精确匹配 vs. 模糊匹配）以及**优先级队列**的排序概念，这两个概念决定了关键词冲突时系统如何选择响应。

在叙事设计的学习路径中，关键词系统是理解**程序化对话**的直接前驱。程序化对话系统（如《80 Days》使用的Ink引擎）在关键词触发机制之上增加了状态追踪、变量运算和概率分支，本质上是将关键词映射表从静态结构扩展为动态状态机。掌握关键词系统的触发逻辑、优先级管理和缺省处理，是理解更复杂对话引擎工作方式的必要基础——特别是在分析为何某个NPC响应"不对劲"时，往往可以追溯到关键词冲突或优先级设置错误这两个根本原因。
