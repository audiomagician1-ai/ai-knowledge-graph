# 动态文本本地化

## 概述

动态文本本地化是指在游戏或交互叙事作品中，对包含运行时变量、复数规则、性别变位等动态生成内容的字符串进行多语言适配的专项处理流程。与静态文本（如章节标题、固定对话）不同，动态文本在玩家运行游戏时才会确定最终形态，例如"你拾取了 {N} 个金币"中的数量 N，或"勇士 {PlayerName} 击败了敌人"中的角色名。这类文本的本地化不能依赖简单的字符串替换，必须在翻译层面预先设计语法框架。

该问题在20世纪90年代末随着RPG游戏全球化发行开始被系统性关注。《最终幻想》系列的英文版本地化团队最早记录了因日语动词不变格而在英语翻译中出现复数错误的问题。2012年，GNU gettext 工具集引入 `nplurals` 和 `plural` 字段，形成了迄今仍被广泛使用的复数规则声明标准，支持包括波兰语（需要4条规则）、阿拉伯语（需要6条规则）在内的复杂语言体系。

动态文本本地化的处理质量直接影响玩家沉浸感。一个错误的复数形式（如俄语中"5 меч"而非"5 мечей"）会立刻打破叙事真实性，即使故事内容本身翻译无误。在程序化叙事系统高度发展的今天，动态文本可能占到游戏全部对话字串的30%至60%，其本地化方案的设计必须在项目初期就与引擎架构同步规划。

## 核心原理

### 复数规则的语言差异

英语的复数规则极为简单：`n == 1 ? singular : plural`（即数量为1时用单数，其余用复数）。但波兰语需要区分四种形式：数量为1、末位为2-4且不在11-14范围内、末位为2-4且在11-14范围内、其余情况，分别对应不同词尾变化。GNU gettext 用如下公式表达波兰语规则：

```
nplurals=4; plural=(n==1 ? 0 : (n%10>=2 && n%10<=4 && (n%100<12 || n%100>14)) ? 1 : (n!=1 && (n%10>=0 && n%10<=1) || (n%10>=5 && n%10<=9) || (n%100>=12 && n%100<=14)) ? 2 : 3)
```

叙事设计师在创建字符串表时，必须为每种目标语言预留足够数量的复数变体槽位，而不是仅凭英语的单复数两槽规划翻译接口。阿拉伯语同理需要6个槽位（0、1、2、3-10、11-99、100+）。

### 变量占位符的顺序问题

动态变量以占位符形式嵌入字符串，常见格式包括 `{0}`、`%1$s`、`[[PLAYER_NAME]]` 等。英语句子"你对 {EnemyName} 造成了 {Damage} 点伤害"中变量顺序为"对象→数值"，但德语语序可能要求"你造成了 {Damage} 点伤害对 {EnemyName}"，即变量顺序倒置。若使用位置型占位符 `%1` `%2`，德语译者只能被迫维持英语语序，产生语法错误。正确方案是使用**命名占位符**（Named Placeholder），如 `{enemy}` 和 `{damage}`，允许译者在目标语言字符串中自由调整两者出现的顺序。ICU（International Components for Unicode）MessageFormat 标准提供了完整的命名占位符和内嵌复数规则语法，是当前业界推荐规范。

### ICU MessageFormat 嵌套规则

ICU MessageFormat 允许在单条字符串内同时处理变量替换与复数选择，其语法结构如下：

```
{count, plural,
  one {你拾取了 {count} 个苹果}
  other {你拾取了 {count} 个苹果}
}
```

对于性别化语言，ICU 提供 `select` 关键字：

```
{gender, select,
  male {他拿起了剑}
  female {她拿起了剑}
  other {他们拿起了剑}
}
```

两者可嵌套使用，处理如"[角色名] 收到了 [N] 封来自 [性别化 NPC] 的信件"这类多维度动态字符串。Unity 的 Localization Package 和 Unreal Engine 的 FText 系统均原生支持 ICU MessageFormat 格式解析。

## 实际应用

**物品数量通知**：背包系统常见文本"获得 {N} 个治疗药水"在中文中因名词不存在复数屈折变化，翻译无需额外处理；但同一条字符串在俄语本地化中必须根据上节的4条规则为"зелье/зелья/зелий"三种词形各准备一份模板，并由引擎在运行时根据 N 值自动选择。

**任务进度提示**：《巫师3》的任务日志使用"击败了 {killed}/{total} 名敌人"的格式，波兰语原版与英语版的复数处理逻辑不同，因为开发商 CD Projekt RED 的本地化管线对两种语言分别维护了独立的复数规则配置文件，而不是共享同一套英语逻辑。

**性别敏感的成就文本**：法语中形容词须与主语性别一致，如"你被选中了（Tu as été choisi/choisie）"。游戏需从玩家角色创建时记录的性别参数中提取值，在运行时通过 ICU `select` 语法选择正确的形容词词尾，该参数必须作为系统级变量贯通整个字符串表架构。

## 常见误区

**误区一：认为中文、日文不需要动态规则，因此对亚洲语言版本简化处理**。实际上，中文虽无复数屈折，但量词的选择（"一把剑" vs "一根箭"）有时需要根据物品类型动态切换量词，这同样要求字符串中预留物品类别变量并为不同类别准备独立模板。

**误区二：使用字符串拼接代替模板**。将"你有"、N、"个金币"三段字符串在代码层拼接，看似节省翻译量，却导致德语、芬兰语等存在格位变化（Genitiv/Partitiv）的语言无法对"金币"一词进行正确的格位屈折，因为译者看不到完整的语法语境。正确做法是始终将完整句子作为单个字符串单元提交翻译。

**误区三：复数规则只影响名词**。在部分斯拉夫语系中，数字本身也会根据所修饰名词的性别发生变化（俄语的"один/одна/одно"），因此占位符方案还需要将名词的语法性别作为可选参数传递给渲染层。

## 知识关联

动态文本本地化建立在**字符串表**的基础架构之上：字符串表决定了如何存储和索引文本资产，而动态文本本地化则规定了这些资产在复数槽位数量、占位符命名约定、ICU 格式兼容性上的扩展需求——设计字符串表结构时若未预留多语言复数槽，后期修改的工程成本极高。**性别化语言**的知识是处理 `select` 分支的前提，理解目标语言如何通过语法性别驱动形容词、动词的一致性变化，才能正确设计嵌套的 ICU MessageFormat 模板。本地化工程师在评审字符串表设计时，应将"能否支持阿拉伯语的6级复数"和"占位符是否命名化"列为硬性验收标准，在内容生产流程启动前完成架构锁定。
