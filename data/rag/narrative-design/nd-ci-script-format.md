# 过场脚本格式

## 概述

过场脚本格式（Cutscene Script Format）是游戏开发中专门用于描述非交互式动画片段的文档标准，它规定了场景描述、对话、摄像机指令和角色动作如何以统一的书面形式记录，以便编剧、动画师、程序员和配音导演之间高效协作。与影视剧本格式不同，过场脚本必须同时兼顾"叙事意图"与"引擎实现约束"，一份合规的过场脚本不仅是故事文档，也是技术任务单。

过场脚本格式的标准化工作大约在2000年代初随着3D游戏叙事复杂度上升而逐步形成。早期游戏如《最终幻想VII》（1997年）的过场内容多以导演意图随意记录，制作团队协作困难。到《光环2》（2004年）开发期间，Bungie内部开始推行结构化的"场景脚本模板"，要求每个过场包含唯一的场景编号（如CS_M01_001）、镜头列表和台词注记字段。这一做法逐渐被业界沿用，形成了今天可见的标准化规范。

掌握过场脚本格式意味着能将导演的叙事想法转化为各部门可直接执行的工作指令。格式不规范会直接导致动画师误读摄像机角度、配音演员无法理解情绪语境、本地化团队缺少必要的字符限制信息，最终造成返工与版本混乱。

## 核心原理

### 文档结构层级

标准过场脚本由三个层级构成：**场景级（Scene）→ 镜次级（Shot）→ 事件级（Beat）**。

- **场景级**：包含场景ID、所属关卡、触发条件（如`触发器：玩家进入Zone_04`）、前置/后置过场编号、预估时长（单位：秒）。
- **镜次级**：每个镜头需注明镜头编号（如`SHOT 03`）、摄像机类型（固定/跟随/推进/环绕）、焦距描述（如"50mm等效，中景"）和起止时间码（格式为`00:01:23:12`，即小时:分钟:秒:帧）。
- **事件级**：在镜头内按时间顺序列出角色动作（Action）、对话（Dialogue）、音效提示（SFX Cue）和字幕文本，每项均需标注触发帧号。

### 对话块书写规范

过场脚本的对话格式参考好莱坞剧本规范，但增加了游戏专用字段。标准对话块格式如下：

```
[角色名] （情绪标注：愤怒/低语/讽刺等）
台词正文（中文字符数限制通常为30字/行，以适配字幕框）
[VO_FILE: CH_HERO_CS01_003_zh]
[DUR: 3.2s]
```

其中`VO_FILE`字段直接对应配音文件命名规则，`DUR`字段供动画师确认口型同步（Lip Sync）时长窗口。情绪标注并非可选项——配音导演在录音棚现场必须依赖此标注指导演员表演，缺失会导致整批录音情绪偏差。

### 舞台指令（Stage Direction）格式

舞台指令用大写加方括号书写，明确区别于对话文本，例如：

```
[主角转身，右手握剑，面向BOSS方向，目光停留2秒]
[摄像机慢推，从全景切至主角眼部特写，时长1.5秒]
[BGM：CS_THEME_04淡入，音量从0到80%，持续2秒]
```

舞台指令需要遵守"可执行性原则"：每条指令必须能被特定岗位（动画、摄像机TA、音频）直接转化为任务，禁止使用模糊表述如"气氛紧张地"或"镜头表现情绪"。判断标准是：若动画师无法从该指令推导出骨骼动画的起止状态，该指令需重写。

### 版本与协作字段

专业过场脚本模板头部必须包含以下元数据：
- `脚本版本号`（推荐语义化版本，如v2.1.0）
- `最后修改人`与`修改日期`
- `审批状态`（Draft / Review / Approved / Locked）
- `本地化状态`（未提交 / L10N审核中 / 已冻结）

当脚本状态变为`Locked`后，任何台词修改须走正式的Change Request流程，并同步通知配音、动画和本地化三个团队，因为三方均已基于Locked版本启动了下游工作。

## 实际应用

**RPG主线过场**：在《巫师3》风格的任务型RPG中，每段主线过场脚本通常对应一个唯一的Quest ID，脚本编号如`CS_Q205_KAERMORHEN_001`，编号直接嵌入引擎资产命名规则，使程序员可通过脚本ID检索所有关联资产（模型、动画、音频）。

**多结局分支处理**：当过场存在分支结局时，脚本采用条件块格式：
```
[IF: player_choice == "拒绝"]
  → 跳转至 CS_Q205_ENDING_B
[ELSE]
  → 继续 SHOT 07
[END IF]
```
这一格式让程序员可直接从脚本文档提取逻辑流，减少口头沟通产生的实现错误。

**本地化适配**：日文版过场脚本在台词字段额外增加`[JP_MAX_CHAR: 20]`字段，因为日语字幕框在相同物理宽度下可容纳的字符数与中文不同，字幕工程师依据此字段调整换行点，而无需反复询问叙事设计师。

## 常见误区

**误区一：把过场脚本等同于影视剧本**
影视剧本不需要标注`VO_FILE`命名、引擎触发条件或帧数时间码，而这些字段在游戏过场脚本中是强制内容。许多初学叙事设计的人直接套用Final Draft的标准影视模板，导致动画和音频部门拿到脚本后仍需二次整理才能开工，平均增加2-3天协作成本。

**误区二：情绪标注可以写得文学化**
新手常把情绪标注写成"内心复杂、五味杂陈、陷入沉思"，这对配音演员毫无指导意义。标准做法是使用单一、可表演的情绪词，如"克制的愤怒（suppress anger）"或"假装镇定（forced calm）"，并在必要时附加一句背景说明："此刻角色已得知父亲死亡，但不希望敌人察觉"。

**误区三：Locked状态只是标记，可以口头修改**
一旦脚本进入Locked状态，口头或聊天软件中的"小改动"极易造成各版本不同步——动画组按旧版已完成口型烘焙，音频组已录制旧台词，本地化组已翻译旧文本。行业标准做法是为任何Locked后修改建立书面Change Request，并分配独立的版本号（如从v2.1.0升至v2.1.1），确保所有下游团队收到同一版本通知。

## 知识关联

过场脚本格式不依赖前置概念，是叙事设计学习路径中最先需要掌握的文档技能，适合在学习任何具体叙事技巧之前建立。一旦熟悉脚本格式中的"镜次（Shot）"概念和摄像机指令的书写方式，学习者将自然过渡到**分镜设计**（Storyboard Design）——分镜设计正是将脚本中的文字摄像机描述转化为可视化连续图像的过程，两者共享"Shot编号对应"和"时间码标注"的工作语言。此外，过场脚本中的条件分支格式（IF/ELSE块）也为后续学习**对话树系统**和**叙事状态机**提供了直接的概念接口。
