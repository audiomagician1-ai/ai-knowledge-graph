# ChatScript

## 概述

ChatScript 是由 Bruce Wilcox 开发的基于规则的对话人工智能引擎，于2011年首次公开发布。Bruce Wilcox 凭借基于 ChatScript 构建的聊天机器人 Suzette，在2010年赢得了罗布纳奖（Loebner Prize）图灵测试竞赛，这是该引擎最早的重要验证。与神经网络生成模型不同，ChatScript 使用模式匹配语法和有限状态主题树（topic tree）来驱动对话逻辑，所有对话行为都由人工编写的规则明确控制。

ChatScript 的核心文件格式是纯文本脚本文件（`.top` 扩展名），其中包含主题定义、规则集和概念词汇表。引擎采用 C++ 编写，可以在 Windows、Linux、Mac 等多种平台上运行，并支持通过 REST API 与外部应用集成。对于叙事设计师而言，ChatScript 提供了一套可以精确控制角色对话走向的工具——每一句 NPC 台词都来自设计者的明确意图，而非语言模型的概率采样。

在游戏叙事领域，ChatScript 被 Telltale Games 等工作室用于早期互动对话原型的构建，也被学术界用于研究对话型叙事游戏的分支管理。它的价值在于：当叙事设计师需要对角色的每一句话负责、需要对情节走向进行精确的编辑控制时，规则引擎比生成式 AI 更可靠。

## 核心原理

### 主题树与规则层级

ChatScript 的基本组织单元是**主题（topic）**，使用 `topic: ~topic_name` 语法声明。每个主题内部包含若干条规则（rule），规则分为三种类型：
- `t:` （gambit）：主动发起的对话行，NPC 自发说出的台词
- `r:` （rejoinder）：针对上一条 gambit 的追问或回应
- `u:` （responder）：对用户输入进行模式匹配后触发的响应

这三层结构形成了一棵对话树，引擎按照优先级从上至下搜索匹配规则，一旦命中即执行并停止继续搜索。这种确定性搜索机制确保了叙事设计师对对话流程的完全掌控。

### 模式匹配语法

ChatScript 的模式匹配语言功能强大，支持多种通配符和语义概念引用。典型语法示例如下：

```
u: (你 *~3 喜欢 ?)  我也很喜欢这个！
```

上面这条规则匹配"你"与"喜欢"之间相隔不超过3个词的任意输入。`*~3` 代表"0到3个任意词"，`?` 代表句末疑问。除通配符外，ChatScript 还支持以 `~` 前缀引用**概念集合（concept set）**，例如 `~animals` 可以匹配"狗""猫""老虎"等所有预定义的动物词汇，引擎内置了超过2000个英文概念集合，设计师也可自定义中文概念集。

### 状态变量与叙事记忆

ChatScript 提供三类变量用于存储叙事状态：
- `$global_var`：跨对话会话持久存储，适合记录玩家的关键选择
- `%user_var`：当前用户会话内有效，适合追踪单次游戏内的情节标记
- `^function()`：可调用的宏函数，用于封装复杂的叙事逻辑判断

例如，设计师可以用 `$player_helped_npc = 1` 标记玩家曾经帮助过某NPC，之后的规则可以用 `if ($player_helped_npc == 1)` 条件判断来解锁专属对话分支，实现有记忆的角色叙事。

## 实际应用

**游戏NPC对话系统**：在角色扮演游戏中，叙事设计师可以用 ChatScript 为村庄长老编写包含20个以上主题的对话脚本，每个主题对应一个叙事阶段（如"初次相遇""主线完成后""获得信任后"），玩家的对话输入触发不同主题下的规则，实现非线性叙事体验。

**互动小说原型测试**：在正式开发引擎集成前，编剧团队可以用 ChatScript 的命令行界面快速测试对话分支逻辑，输入测试台词后立即看到引擎匹配到哪条规则以及为何匹配，调试信息以 `[RULE FIRED: topic ~chapter1 rule u:1]` 格式输出，大幅缩短叙事校对时间。

**教育型叙事代理**：学术研究中，ChatScript 被用于构建具有固定人格的虚拟历史人物（如模拟林肯总统的对话代理），通过精心设计的概念集和规则集，确保角色的回答始终符合历史语境，不会产生生成式模型常见的"幻觉"内容。

## 常见误区

**误区一：认为 ChatScript 需要机器学习训练数据**
ChatScript 完全不依赖语料库训练，所有规则由叙事设计师手工编写。引擎没有神经网络层，没有嵌入向量，也没有微调过程。这意味着"性能"由规则覆盖率决定，而非训练数据量。新手常误以为需要先"喂数据"才能使用，实际上安装引擎后直接编写 `.top` 文件即可运行第一个对话。

**误区二：模式越复杂，对话体验越好**
初学者倾向于堆砌通配符和嵌套条件，导致规则集臃肿且难以调试。ChatScript 官方文档建议单条规则的模式长度保持在10个词元以内，并优先用概念集（`~`前缀）替代枚举列表。复杂叙事逻辑应通过多个主题的切换（`^gambit(~next_topic)`）来实现，而非单条规则的过度延伸。

**误区三：ChatScript 只能处理英文**
尽管内置概念集以英文为主，ChatScript 的模式匹配引擎本身是语言无关的。叙事设计师只需自行定义中文概念集（例如 `concept: ~问候词 (你好 您好 嗨 哈喽)`），并将所有规则用中文编写，即可构建完整的中文对话系统。Wilcox 本人也在文档中说明了多语言支持的配置方法。

## 知识关联

学习 ChatScript 不需要任何编程基础，但熟悉有限状态机（FSM）的概念有助于更快理解主题树的跳转逻辑——ChatScript 的主题切换本质上是一种显式的状态转移。如果你曾使用过 Twine 或 Ink 编写互动叙事分支，ChatScript 的规则编写思维与之相似，区别在于 ChatScript 的触发条件是自然语言输入而非点击选项。

在掌握 ChatScript 之后，下一个自然的进阶方向是**LLM叙事辅助**。两者的核心差异在于控制权与灵活性的权衡：ChatScript 给设计师100%的规则控制权，但每条可能的用户输入都需要手动覆盖；LLM则可以处理无限的输入变体，但输出内容的叙事精确性需要通过提示词工程来约束。在实际项目中，两种技术可以组合使用——用 ChatScript 管理关键剧情节点的对话逻辑，用 LLM 填充非关键性的日常寒暄内容，形成混合叙事架构。
