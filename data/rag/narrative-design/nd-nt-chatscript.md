# ChatScript

## 概述

ChatScript是由Bruce Wilcox于2009年开发的一款基于规则的对话AI引擎，最初用于构建聊天机器人参赛作品。2011年，基于ChatScript构建的聊天机器人"Suzette"在洛布纳奖（Loebner Prize）图灵测试竞赛中击败了人类参赛者，这一成就使ChatScript在对话系统领域引发广泛关注。2013年，Wilcox将ChatScript用于开发知名游戏《战锤40K：太空狼》中的NPC对话系统，这是ChatScript在叙事游戏领域最具代表性的商业应用案例之一。

与基于神经网络的语言模型不同，ChatScript采用完全透明的规则-模式匹配（rule-pattern matching）架构。每一条对话规则都由作者明确编写，系统的行为完全可预测、可调试。这种特性使ChatScript在叙事设计中具有独特价值：编剧可以精确控制NPC在特定故事节点说什么、不说什么，确保叙事一致性。

ChatScript的核心设计哲学是"话题（topic）驱动对话"——将人类对话组织为围绕特定主题展开的知识和规则集合，而非线性的对话树。这种结构使NPC能够在不同话题之间自然切换，模拟更真实的对话动态，同时保持叙事作者对内容的完全掌控。

## 核心原理

### 模式匹配语法（Pattern Matching）

ChatScript的对话规则使用一套专有的模式匹配语言编写。一条完整规则的基本语法结构为：

```
类型标识符 (模式) 回应内容
```

例如：
```
u: (你好 [吗 啊]) 你好，旅行者。
```

其中`u:`表示该规则为用户输入触发型（user input）。方括号`[]`表示"或"关系，花括号`{}`表示可选词，通配符`*`匹配任意数量词语。ChatScript支持多层次的模式嵌套，例如`<<词A 词B>>`表示这两个词以任意顺序出现均可触发规则。这种语法使叙事设计师能够用极少量代码覆盖大量表达变体。

### 话题（Topic）架构

ChatScript将所有规则组织在"话题"容器内。每个话题用`topic: ~话题名称 (关键词列表)`声明，话题内可包含多种类型的规则：`t:`（主动叙述，gambit）用于NPC主动推进对话；`r:`（重用规则，rejoinder）用于响应上一条输出后的用户反应；`u:`（通用规则）用于任意时刻匹配用户输入。话题之间可以通过`^gambit(~话题名)`进行跳转，从而构建出具有分支与汇聚结构的叙事流。

一个NPC的完整对话系统通常由数个到数十个话题组成，叙事设计师通过规划话题的触发条件与优先级来实现"角色弧"——NPC随着故事进展会进入不同话题，呈现出随情境变化的性格与信息。

### 变量与状态追踪

ChatScript内置三类变量系统，直接服务于叙事状态管理。全局变量（global variable）以`$`开头，在整个对话系统中持久存在，适合记录玩家已完成任务、解锁的剧情标志等；话题变量（topic variable）以`$$`开头，仅在当前话题内有效；局部变量（local variable）以`$_`开头，仅在当前规则执行期间存在。叙事设计师可以通过条件判断`if ($player_killed_guard == true)`来控制NPC对特定玩家行为的不同反应，实现叙事分支而无需在对话树中重复大量内容。

ChatScript还提供了`^记忆（^memorize）`和`^recall`系列函数，可将玩家在对话中透露的信息（如姓名、选择）存储并在后续对话中调用，制造NPC"记得玩家"的沉浸感。

## 实际应用

**游戏NPC对话系统**：在《战锤40K：太空狼》的实际部署中，Wilcox为游戏中的战友NPC编写了超过5000条规则，分布在约30个话题中。玩家与NPC的对话可以自然语言输入，系统基于模式匹配给出符合角色背景与当前任务阶段的回应，而非依赖有限的菜单选项。

**交互式小说的多轮对话**：ChatScript适合构建需要维持长期状态的叙事体验。例如，一个侦探游戏中的嫌疑人NPC，可以设计为：第一次询问时回答含糊（话题`~first_meeting`），玩家找到证据后（`$evidence_found == true`）触发话题`~confrontation`，NPC进入防御模式，改变措辞和信息量。整个过程由规则驱动，行为完全可预测，叙事设计师可在发布前完整测试每条路径。

**对话质量保障**：ChatScript的日志系统会记录每条规则的触发频率，叙事设计师可据此识别哪些规则从未触发（可能是模式过于严苛）、哪些规则过度触发（可能导致重复体验），并针对性地进行修改，这是基于LLM的对话系统难以做到的精确调试能力。

## 常见误区

**误区一：认为ChatScript是自然语言理解（NLU）系统**。ChatScript本质上是一个高级模式匹配引擎，它不"理解"语义，只匹配表面形式和规则中指定的词汇变体。当玩家使用完全超出规则预设范围的表达方式时，系统会触发默认的`u: (?) 回退回应`规则，显现出规则系统的局限性。叙事设计师必须为每个话题预设足够充分的模式变体，才能保证对话覆盖率。

**误区二：认为话题数量越多系统越智能**。ChatScript的规则评估遵循严格的优先级顺序：重用规则`r:`优先于话题规则，当前激活话题的规则优先于其他话题规则。随意堆砌大量话题会导致规则冲突和意料之外的触发顺序，反而使NPC行为变得不一致。正确做法是为每个NPC设计清晰的话题层级，用`topic:`的`keep`标志控制规则是否在触发后失效。

**误区三：将ChatScript与对话树混为一谈**。对话树（dialogue tree）是预设节点的有向图，玩家只能选择固定选项；ChatScript接受自由文本输入，通过模式匹配在规则库中寻找最佳匹配，这使其能够响应玩家未预料到的输入，并通过`^rejoinder`链实现多轮对话的连贯性，两者在架构和使用体验上有本质差异。

## 知识关联

学习ChatScript无需编程基础，但熟悉正则表达式的叙事设计师能更快掌握其模式语法，因为两者在通配符和分组概念上有相似之处（尽管ChatScript的语法针对自然语言做了大量扩展）。ChatScript是理解"可控叙事AI"与"生成式叙事AI"差异的绝佳参照点——当过渡到学习**LLM叙事辅助**时，你会发现大型语言模型在灵活性上远超ChatScript，但恰恰失去了ChatScript最核心的优势：行为的完全可预测性与叙事一致性保障。ChatScript的变量状态追踪思想，在LLM叙事系统中演变为"世界状态注入提示词"等技术，两者解决的是同一类叙事连贯性问题，只是实现路径截然不同。
