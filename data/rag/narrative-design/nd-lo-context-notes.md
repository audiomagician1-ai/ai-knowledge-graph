# 翻译上下文备注

## 概述

翻译上下文备注（Translation Context Notes，业界常缩写为 TCN 或直接称为"译者注释"）是指在游戏或叙事内容的本地化文件中，由原文作者或本地化协调员附加在原文字符串旁边的说明信息，用于告知译者这段文字的使用场景、说话角色、语气要求及界面位置等关键背景。没有这类备注，译者面对一行孤立的字符串时，往往无法判断"You"指代的是玩家还是NPC、"Light"是形容词"轻盈"还是名词"灯光"、按钮文字是否有字符数量限制。

这一实践最早在20世纪90年代末随着游戏全球发行规模扩大而系统化。1996年前后，任天堂和世嘉开始在内部本地化工作流中引入专用的"注释列"（Comment Column），要求原文编写者为每条UI字符串注明屏幕位置和最大显示字符数。进入2000年代，随着CAT工具（计算机辅助翻译工具）如SDL Trados和memoQ的普及，上下文备注成为XLIFF和XLSX本地化文件格式中的标准字段。

在叙事设计领域，翻译上下文备注的重要性尤为突出，因为游戏对话常涉及角色语气、暗语、双关语和受限字数。一个缺乏备注的对话行"It's over."，译者无法判断该使用悲伤的口吻、愤怒的口吻还是中性陈述，也不知道说话者的性别——而日语、俄语、波兰语等语言的翻译结果会因此产生截然不同的语法形式。

## 核心原理

### 备注的六类核心信息

一条有效的翻译上下文备注通常包含以下六类信息，缺少任何一类都可能导致歧义：

1. **说话角色（Speaker）**：注明是哪个NPC还是玩家角色在说话，以及该角色的性别、年龄和社会地位。例如备注"老年女性村长，对玩家说话，使用方言化语气"直接影响敬语选择。
2. **被称呼对象（Addressee）**：指出对话的受话方，决定敬语或复数形式。德语"Sie"与"du"的选择完全依赖这一信息。
3. **界面位置与字符限制（UI Position & Char Limit）**：如"按钮文字，最多8个全角字符"或"加载画面提示，最多120字符"，防止译文溢出显示区域。
4. **语气与情感状态（Tone）**：如"紧急警告，玩家濒死时显示"或"轻松的商店欢迎语"，决定用词的正式程度。
5. **上文引用（Preceding Context）**：标注该字符串在对话树中的前一行内容，帮助译者理解逻辑连贯性。
6. **多义词说明（Disambiguation）**：对存在多种词义的词语直接注明所用含义，例如"'Party'指战斗队伍，非派对"。

### 备注的标准书写格式

业界普遍采用的XLIFF 1.2和2.0格式中，备注字段为`<note>`标签，建议格式为：

```
[角色]:[对象] | [位置] | [情感] | 备注正文
```

例如：
```
NPC_Aldric:Player | 主线任务第3章对话框 | 愤怒、强调 | Aldric是中年男性骑士，此处质问玩家为何背叛盟约，"betrayed"为关键词请保留负面含义
```

这一格式允许CAT工具按字段自动过滤和排序备注，使译者在翻译大型项目（如包含40,000条字符串的RPG）时可快速定位关键信息。

### 备注的覆盖密度要求

并非每条字符串都需要同等详细的备注。行业通行的优先级规则如下：

- **必须附备注**：所有角色对话行、所有UI按钮文字、含性别/数量语法变化的字符串、含双关或文化梗的文本。
- **建议附备注**：系统提示信息、商店道具描述中含专有名词者。
- **可省略备注**：纯数字、仅含变量占位符（如`%d`）的格式字符串。

微软的本地化指南《Microsoft Localization Style Guide》建议，对话类内容的备注覆盖率不应低于85%，否则译者查询返工率会显著上升，直接增加本地化成本。

## 实际应用

**案例一：性别歧义解决**
《赛博朋克2077》在本地化过程中，玩家角色V可以是男性或女性，大量NPC对话需要根据玩家性别切换。备注中若标注"[F_V]此行仅在玩家选择女性V时播放"，俄语译者便能正确使用阴性形容词词尾，避免语法错误出现在正式版本中。若备注缺失，同一条字符串可能导致译者默认使用男性形式，在女性V的对话中出现语法不一致的本地化缺陷（Localization Bug，L10n Bug）。

**案例二：字符限制与界面溢出**
在手机游戏的技能按钮中，"激活"两个字仅占4个字节，但若原文是"Reactivate"且备注注明"按钮，最多6字符（含标点）"，译者便知"重新激活"（5字符）可行，"重新启动技能"（7字符）不可行。缺少此备注时，译者无从判断限制，美术团队需要在集成阶段反复返工，平均每次返工增加约2个工作日的延误。

**案例三：双关语与文化参考**
《传说之下（Undertale）》官方中文化时，Papyrus角色名字本身是一个字体双关（Papyrus字体）。备注若写明"此角色名来自字体名称Papyrus，与角色喜欢写大字体的性格相关，翻译时请与本地化顾问确认是否保留音译或寻找本地等效双关"，译者就不会直接将其音译为毫无关联的词汇，而是与叙事设计团队协商最优方案。

## 常见误区

**误区一：备注越详细越好，字数无限制**
部分原文作者认为备注越长越安全，将整个任务背景写入单条备注，导致每条备注超过500字。实际上，过长的备注会使译者在翻译高强度的大型项目时产生"备注疲劳"，反而跳过阅读。有效备注应控制在50—120字以内，聚焦对翻译决策有直接影响的信息，而非完整的叙事背景说明。

**误区二：只有对话文本才需要备注，UI文本不需要**
UI字符串（界面按钮、菜单选项、提示信息）往往是字符限制最严格、歧义最高发的文本类型。"Save"一词在游戏UI中可能意为"储存档案"，也可能意为"拯救角色"，两者的日语翻译分别是"セーブ"和"救う"，若无备注说明属于哪种语境，译者只能猜测。统计显示，UI字符串的本地化缺陷中，约40%源于缺少上下文备注。

**误区三：备注可以在交付译者前统一补充**
实际工作流中，备注必须由最了解字符串使用场景的人撰写——通常是叙事设计师或对话作者，而非本地化项目经理。项目经理在交付前补充的备注往往流于表面（如仅写"对话"二字），无法解决具体歧义。正确做法是将备注撰写纳入内容创作阶段，在原文字符串入库时同步完成。

## 知识关联

翻译上下文备注是本地化工作流的起点环节，它直接影响下游的**文化适配**工作质量。当备注中明确标注了某段文字含有文化特定内容（如节日名称、社会禁忌词汇、本地梗），文化适配专员才能在备注基础上做出替换或保留的决策，而不是在最终审校阶段才发现文化不适配的问题。

在工具层面，翻译上下文备注与**字符串变量系统**紧密相关：含有`{PLAYER_NAME}`或`{ITEM_COUNT}`等变量的字符串，备注必须说明变量的可能取值范围和性别/数量属性，否则译者无法处理与变量相邻的形容词或动词的语法一致性问题。

对于希望深入理解本地化体系的叙事设计师，掌握翻译上下文备注的撰写规范，意味着在创作阶段就建立起"文本在翻译环境下的可用性"意识——这与写出语义清晰的原文同等重要，是专业叙事设计师区别于业余创作者的具体技能指标之一。
