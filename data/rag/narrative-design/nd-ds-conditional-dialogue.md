# 条件对话

## 概述

条件对话（Conditional Dialogue）是对话系统中根据运行时变量动态决定哪些对话节点可见、可选或可触发的机制。其核心逻辑是：每条对话线路或选项附带一组条件表达式，引擎在渲染对话界面时实时求值，只有条件为真的分支才会呈现给玩家或被AI角色执行。与静态脚本不同，条件对话使同一段对话树能够根据游戏世界的当前状态产生数十种差异化输出。

条件对话的系统化设计可追溯至1990年代的RPG游戏。《Planescape: Torment》（1999年）首次大规模使用基于角色属性（智力、魅力、智慧各有独立阈值）的对话条件，使玩家不同的属性分配带来截然不同的对话权限。此后《Mass Effect》系列将好感度变量与条件对话深度绑定，Garrus等NPC在不同忠诚度阶段会开放或锁闭特定对话选项，形成了现代AAA叙事游戏的范式。

条件对话的重要性在于它是叙事一致性的技术保障。没有条件控制，玩家可以在尚未认识某NPC的情况下触发亲密对话，或在已完成任务后再次接受同一任务，破坏叙事可信度。条件对话将游戏状态与叙事层紧密耦合，确保角色的每句话都与玩家当前的行为历史相符。

## 核心原理

### 条件类型与变量来源

条件对话依赖的变量通常分为四类：**全局标志（Flag）**、**数值变量（Numeric Variable）**、**枚举状态（Enum State）** 和 **关系值（Relationship Score）**。全局标志是布尔值，例如`quest_betrayal_completed = true`；数值变量携带范围判断，例如`player_gold >= 500`；枚举状态描述角色所处阶段，例如`npc_state == "imprisoned"`；关系值通常是0到100的整数，判断式写作`affinity_score >= 70`。

Bark系统提供的是单向、无需玩家响应的触发式台词，而条件对话在此基础上增加了**双向可见性控制**——不仅NPC的输出受条件约束，玩家侧的选项也同样受限。这是两者的根本区别。

### 条件表达式的求值逻辑

多数对话引擎（如Ink、Yarn Spinner、Twine或自研系统）使用**前置条件（Precondition）+ 优先级（Priority）**双重机制。前置条件决定分支是否出现，优先级决定多条件同时满足时哪条分支优先执行。以Ink语法为例：

```
-> talk_to_merchant
= talk_to_merchant
{ has_stolen_item:
    - 你偷了我的东西。
    -> confrontation
}
{ gold >= 200:
    - 看来你很有钱，想买些稀罕货吗？
    -> premium_shop
}
- 欢迎光临。
-> normal_shop
```

上例中，`has_stolen_item`为布尔标志，`gold >= 200`为数值比较，两者均失败则执行默认分支。求值顺序从上至下，第一个为真的条件短路执行，不继续检查后续条件。

### 好感度条件的分段设计

关系值条件通常采用**分段阈值（Threshold Tiers）**而非连续比较。《Dragon Age: Origins》将NPC好感度划分为：厌恶（0–24）、中立（25–49）、喜好（50–74）、挚友（75–89）、爱慕（90–100）五段，每段对应不同的对话池。设计师为每个阈值区间分别编写专属台词，而非为每个整数点编写，这将内容量从100份压缩至5份，同时保证过渡感。

说服系统与条件对话的交叉点在于：说服系统的成功结果往往写入一个布尔标志（如`persuade_guard_success = true`），该标志随后成为条件对话的触发变量，解锁原本不可见的对话支线。标志系统负责跨场景持久化这些变量，使条件对话能够跨越多个游戏会话保持一致性。

## 实际应用

**任务状态驱动的对话锁定**：在《The Witcher 3》中，与特定NPC的对话选项直接绑定任务阶段变量。玩家在完成"家庭事务"主线任务前后，同一NPC的对话树中会有17条选项因`main_quest_family_matters_stage`变量的不同取值而显示或隐藏。玩家无法提前获知后续剧情信息，叙事节奏由条件系统精确控制。

**多角色状态组合条件**：条件对话支持逻辑运算符组合多个变量。例如`(npc_trust >= 60) AND (player_class == "Mage") AND (NOT has_revealed_secret)`同时满足时，才开放特定的魔法知识对话选项。这种组合条件使叙事分支呈指数级增长，但须配合内容优先级机制防止组合爆炸。

**动态显示玩家角色已知信息**：条件对话可控制玩家选项的**文字内容本身**，而非仅控制显隐。当`knows_npc_real_name = true`时，选项文字从"你是谁？"变更为"我知道你真名是Vilgefortz"，呈现出角色扮演层面的信息同步，而不是简单地增减选项数量。

## 常见误区

**误区一：条件对话只控制"显示与隐藏"**。实际上条件对话还可控制对话节点的**执行副作用**——同一句NPC台词在不同条件下触发后，可以附带不同的变量修改（如好感度变化量不同）、不同的动画状态机切换，或不同的后续分支跳转目标。把条件对话理解为纯粹的UI过滤器会严重低估其叙事功能。

**误区二：条件越复杂、分支越多，叙事质量越高**。《Disco Elysium》大量使用条件对话，但其条件主要基于角色技能属性值，每次掷骰结果写入标志后驱动后续条件判断，整体条件结构清晰可维护。条件逻辑的可调试性和可维护性与条件数量成反比，堆砌条件而不建立清晰的变量命名规范和分层架构，会导致"条件地狱"——开发后期无法追踪某个分支被锁死的原因。

**误区三：好感度是条件对话的唯一数值变量**。玩家角色的技能值、当前装备标签、已阅读的游戏内书籍标志、甚至当前游戏内时间（白天/黑夜周期）都可以作为条件变量。将条件对话的应用范围局限在关系系统上，会错失大量叙事可能性。

## 知识关联

**前置关联**：标志系统为条件对话提供跨场景持久化的布尔变量存储，没有可靠的标志系统，条件对话无法在不同地图或存档读取后保持一致；说服系统的成功/失败结果以标志形式输出，直接成为条件对话的输入变量，形成功能闭环；Bark系统虽然也有触发条件，但其条件仅控制"何时播放单向台词"，而条件对话的条件控制"哪些双向分支可被选择和执行"，复杂度高出一个量级。

**后续关联**：掌握条件对话之后，自然延伸至**对话状态机**的设计。对话状态机将多个条件对话节点组织成有限状态图，每个状态定义其内部可用的条件分支集合，以及触发状态转移的事件。条件对话解决"这条分支是否可用"的问题，而对话状态机解决"当前NPC整体处于哪个对话阶段、哪个阶段集合是活跃的"这一更高层次的问题。理解条件表达式的求值机制、变量命名规范和优先级逻辑，是构建复杂对话状态机的直接技术前提。
