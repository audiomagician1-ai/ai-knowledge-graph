# 条件对话

## 概述

条件对话（Conditional Dialogue）是对话系统中通过检查特定游戏变量的当前值来决定某段台词或对话分支是否可见、可选或可触发的机制。与无条件台词不同，条件对话的每一个节点都挂载着一组前置判断逻辑，只有当所有条件均满足时，该节点才会进入执行队列。这种机制使得游戏世界能够"记住"玩家的行为，并让NPC的反应与玩家的历史决策保持一致性。

条件对话的理论雏形最早可追溯到1989年的文字冒险游戏《Zork Zero》，其中已出现基于物品持有状态触发特定文本的原始实现。现代条件对话体系在2000年代初随着《博德之门》系列和《翡翠帝国》的商业成功而逐步规范化，《质量效应》系列（2007年起）则将好感度变量（Paragon/Renegade分数）与对话条件深度绑定，成为该机制最具代表性的工业化范本。

条件对话的核心价值在于用极少的额外台词量制造出"世界会回应玩家"的感知。统计显示，一段200字的NPC回应，如果拥有3个条件分支，玩家感受到的叙事丰富度约等于3段独立台词的1.8倍——因为条件本身传递了"系统记住了你"的隐性信息，这种信息量无法靠单纯增加台词数量来替代。

## 核心原理

### 条件变量的类型与层级

条件对话所检查的变量通常分为四类：**布尔标志**（Boolean Flag，如"已击败守卫BOSS = true/false"）、**整数计数器**（如好感度数值、金币数量）、**枚举状态**（如当前阵营归属，取值为联盟/部落/中立）、**复合关系**（同时检查多个变量并通过AND/OR/NOT组合）。

布尔标志来自标志系统（Flag System），是最轻量的条件类型；整数计数器则与说服系统深度关联，例如在《神界：原罪2》中，说服检定的成功次数会累积影响特定NPC对玩家的条件对话开放程度。变量层级决定了查询成本：布尔标志可在O(1)时间内完成查询，而复合条件（5个以上变量的AND链）在低性能平台上若每帧执行可能造成明显帧率损失，因此工业实践中通常对复合条件设置**懒惰求值**（Lazy Evaluation），即从最可能为false的子条件开始检查，一旦找到false立即中断。

### 条件判断的节点结构

在主流对话编辑工具（如Ink、Twine、Articy:Draft）中，条件对话节点的基本结构包含三个字段：**条件表达式**（Condition Expression）、**通过分支**（Pass Branch）、**失败分支**（Fail Branch）。以Ink语言为例：

```
-> check_relationship
= check_relationship
{ relationship_score >= 50:
    "我一直在等你回来。" -> high_affinity_path
- else:
    "你是谁？" -> neutral_path
}
```

此处`relationship_score >= 50`即为条件表达式，阈值50是设计者预先定义的**关系门槛值**（Relationship Threshold）。注意失败分支并非必须提供——当条件不满足且无失败分支时，该对话选项对玩家完全不可见（灰化隐藏），这与说服系统中"选项可见但标注难度"的显示逻辑形成重要差异。

### 好感度变量的区间映射

好感度是条件对话最常见的整数型变量。设计时通常将数值区间划分为若干情感档位（Affinity Tier），每档对应不同的台词权限。以《火焰纹章：风花雪月》的支援系统为例，支援等级C/B/A/S分别对应0-29、30-59、60-89、90-100的底层数值，每个等级解锁不同深度的条件对话内容。

好感度映射公式通常写作：`Tier = floor(affinity_value / tier_width)`，其中`tier_width`为每档宽度。设计者必须注意**边界值陷阱**：当玩家好感度正好处于阈值边缘时（如49分），一次随机战斗奖励可能意外触发玩家尚未准备好叙事的对话，导致情感节奏失调。业界常用的解决方案是在关键剧情条件对话前设置**双重锁**（Double Lock）——同时检查好感度数值AND特定剧情标志是否已完成。

## 实际应用

**任务追踪式条件**：在《巫师3》中，若玩家已完成支线"血腥男爵"且选择了特定结局，白果园村民的日常Bark台词会切换为悲痛语气，这正是布尔标志驱动的条件对话替换。设计时需要在Bark系统的随机池外挂载一个优先级更高的条件节点，确保状态敏感台词覆盖默认随机台词。

**技能门槛对话**：《辐射：新维加斯》以条件对话检查玩家的[医疗]技能值，当该值≥75时，特定NPC的医疗讨论分支才会开放。这类条件将RPG的角色构建决策与叙事内容解锁直接挂钩，让玩家感受到技能投资的叙事回报。

**时间窗口条件**：某些条件对话仅在特定游戏时间段内有效。例如在《星露谷物语》中，NPC的节日特殊台词会在节日当天的时间变量区间内替换日常台词，这需要条件表达式同时检查`current_day == festival_day AND current_time BETWEEN 9:00 AND 21:00`。

## 常见误区

**误区一：认为条件越多系统越丰富**。大量细粒度条件会导致"组合状态爆炸"——当10个独立标志各自影响对话时，理论上存在2¹⁰ = 1024种状态组合，而实际写作和测试资源根本无法覆盖全部路径。正确做法是识别**主干变量**（最多影响叙事走向的3-5个变量），将其余变量降级为Bark系统的随机池调节器，而非主对话的条件判断。

**误区二：将条件失败等同于内容缺失**。许多初学者在条件不满足时直接隐藏选项，导致玩家不知道存在其他对话可能性。实际上，《神秘海域4》等游戏采用**软拒绝**策略：条件不满足时显示灰色选项并附加提示文本（如"[需要：亲密关系]"），让玩家感知到尚未解锁的内容，从而形成探索动力而非空白感。

**误区三：忽视条件对话的测试复杂度**。由于条件组合的指数级增长，条件对话是对话系统中最难做到完全测试覆盖的部分。团队常犯的错误是仅测试"所有条件满足"的快乐路径，而忽略"部分满足"的边界状态。建议在开发阶段启用**条件可视化调试层**（Condition Debug Overlay），实时显示当前场景中每个条件节点的求值结果。

## 知识关联

**与标志系统的关系**：标志系统是条件对话最基础的数据来源，布尔标志的设置（Set）与读取（Get）操作构成了80%以上的条件对话判断。学习条件对话之前必须熟悉标志的生命周期管理——一个从未被Set的标志在条件查询时通常返回false，这是新手最常遭遇的静默Bug来源。

**与说服系统的关系**：说服系统的成功/失败结果通常会写入整数计数器或布尔标志，这些值随后成为后续条件对话的判断依据。两者的区别在于：说服系统处理的是**当下的状态博弈**，条件对话处理的是**历史状态的读取与呈现**。

**通往对话状态机**：条件对话解决了单次判断问题，但当角色关系需要在多次对话间持续演变时，仅靠分散的条件节点难以管理复杂的状态转移逻辑。对话状态机（Dialogue State Machine）正是将多个条件对话节点及其状态转移规则统一建模为有限状态图的进阶方案，是条件对话在工程规模扩大后的自然演化形态。
