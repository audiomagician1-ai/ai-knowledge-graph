# 标志系统

## 概述

标志系统（Flag System）是分支叙事游戏中用于记录玩家行为、选择和世界状态的数据追踪机制。每个"标志"（Flag）本质上是一个存储在游戏内存中的变量，可以是布尔值（true/false，即"是否发生过某事"）或整数/浮点数值（如好感度、声望点数、道德值）。当玩家完成特定行动时，系统写入或修改对应标志，后续内容再读取这些值来决定展示哪些对话、开放哪些路线。

标志系统的概念最早可追溯至1970年代的文字冒险游戏（Text Adventure）。《巨洞冒险》（Colossal Cave Adventure，1976年）已使用布尔标志追踪道具是否被拿取。到了1990年代，RPG Maker等工具将"开关（Switch）"和"变量（Variable）"作为核心编辑功能正式暴露给开发者，使标志系统成为业余开发者也能直接操作的标准机制。

标志系统的价值在于它是实现非线性叙事的最低成本技术方案。没有标志系统，每次玩家做出选择，游戏只能跳转到完全独立的内容分支，导致内容量呈指数级膨胀。有了标志，同一段对话文本可以通过检测不同标志组合呈现数十种不同语义，大幅压缩内容制作成本。

## 核心原理

### 布尔标志的结构与操作

最简单的布尔标志只有两种状态：0（未触发）和1（已触发）。以《极乐迪斯科》（Disco Elysium）为例，游戏内部维护了数百个布尔标志，如`MeetKim = true`（已见过金·金南）或`PromisedHarrier = false`（未向哈里尔许诺）。每个标志由三部分组成：**标志名称**（唯一标识符）、**初始值**（通常为false/0）和**触发条件**（特定对话节点或行动完成时写入）。

操作标志的基本语法在多数叙事工具（如Twine、Ink、Ren'Py）中形式相近：

```
SET flag_name = true       # 写入标志
IF flag_name == true THEN  # 读取/检测标志
```

### 数值标志与累积型追踪

数值标志允许追踪连续变化的游戏状态。《质量效应》系列使用"帕拉贡/叛逆"双轴数值系统，两个整数变量随玩家每次对话选择增减，最终数值决定是否解锁特定结局对话选项。其判断逻辑为：`IF paragon_score >= 75 THEN unlock_persuade_option`。数值标志常见于以下场景：好感度系统（0-100区间）、声望等级（分段阈值触发不同内容）、资源计数（金币、弹药等影响叙事路线的物质条件）。

数值标志的关键设计参数是**阈值（Threshold）**，即触发内容变化所需的最低/最高值。设计师必须明确每个阈值的叙事意义，避免阈值过于密集导致玩家无法感知变化。

### 标志的组合逻辑

单个标志的表达力有限，强大之处在于多标志的**逻辑组合**（AND/OR/NOT运算）。以《十三机兵防卫圈》类型的叙事游戏为例，某段结局对话的触发条件可能是：

```
IF (met_character_A == true) AND (betrayed_faction == false) AND (loyalty_score >= 60)
THEN show_ending_dialogue_variant_3
```

三个标志通过AND运算组合，产生了在内容层面完全区别于其他组合的叙事结果。设计师通常用**标志状态表（Flag State Table）**来可视化所有可能的组合与对应输出，防止逻辑遗漏或矛盾。

## 实际应用

**Ink脚本语言中的标志实现**：Inkle Studios开发的Ink语言（用于《80天》《孤寂星辰》）使用`VAR`关键字声明标志，`~`前缀执行赋值。例如：

```ink
VAR spoke_to_doctor = false
~ spoke_to_doctor = true
{ spoke_to_doctor: 医生已经告诉你真相了。 | else: 你对这里一无所知。 }
```

**存档与标志持久化**：标志系统必须与存档系统联动。《底特律：变人》在每章结束时将全部活跃标志序列化写入存档文件，下一章节加载时反序列化读取，确保跨章节的因果一致性。若标志未正确持久化，会导致"玩家明明做了选择却没有效果"的叙事断裂。

**标志的时间敏感性**：部分标志需要追踪**何时**被触发，而非仅仅是否触发。《荒野大镖客：救赎2》的NPC记忆系统会记录Arthur与特定角色互动的章节编号，据此调整后续对话的情感基调，而非仅存储布尔值"是否相遇过"。

## 常见误区

**误区一：标志越多越好**。初学叙事设计时，开发者常倾向于为每个玩家行为创建独立标志，导致项目中出现数百个冗余标志。正确做法是在设计阶段识别**叙事关键节点**，只为真正影响后续内容的行为设立标志。《火焰纹章：风花雪月》的好感度系统将每个角色的亲密值压缩为单一整数变量，而非为每次对话单独立标志，保持了系统的可维护性。

**误区二：布尔标志足以覆盖所有需求**。许多初学者只使用true/false标志，试图用标志数量来替代数值精度。例如，用`low_trust`、`medium_trust`、`high_trust`三个布尔标志模拟信任度，而非使用单一整数`trust_score`。这种做法不仅增加标志管理负担，还容易产生逻辑冲突（三个标志同时为true的非法状态）。

**误区三：标志检测不需要考虑顺序**。在Twine或Ren'Py等工具中，若在标志被赋值之前就执行检测，会读到初始值（通常为false/0），导致条件永远不满足。这类"标志竞态"错误在调试时极难发现，必须通过明确的叙事流程图确认每个标志的**写入节点先于所有读取节点**执行。

## 知识关联

标志系统是学习**状态追踪（State Tracking）**的直接前置基础。状态追踪扩展了标志的概念，引入游戏世界的宏观状态机（如战争/和平、角色存活/死亡），其底层实现依然依赖一组有组织的标志集合。理解单个标志的读写逻辑，是理解整个游戏世界状态如何被计算和更新的必要步骤。

标志系统同样直接支撑**条件对话（Conditional Dialogue）**的实现。条件对话中每一条件分支（`[if player_helped_village]`这类标注）在执行时都是对特定标志的检测操作。没有准确的标志数据，条件对话无从判断应当展示哪条文本。因此，设计条件对话之前，必须先完整规划哪些玩家行为需要用标志记录，以及这些标志的命名规范和数据类型。
