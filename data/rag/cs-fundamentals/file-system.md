# 文件系统

## 概述

文件系统（File System）是操作系统中负责管理持久化数据存储的子系统，它定义了数据在存储介质上的组织方式、命名规则和访问方法。具体而言，文件系统将原始的磁盘扇区（sector）抽象为用户可见的"文件"和"目录"层次结构，使程序员无需关心数据实际存储在磁盘的哪个物理位置。没有文件系统，磁盘就只是一段连续的字节序列，毫无结构可言。

文件系统的历史可追溯至1960年代。IBM在OS/360中引入了早期的文件管理概念，而Unix在1969年设计的UFS（Unix File System）奠定了现代文件系统的基本架构——inode结构、目录树、硬链接等概念均源自此处。1993年发布的ext2是Linux世界的重要里程碑，其继任者ext4至今仍是众多Linux服务器的默认文件系统。Windows则采用NTFS（New Technology File System，1993年随Windows NT推出），支持单文件最大16EB的理论容量。

在AI工程领域，文件系统的重要性体现在训练数据的读取效率上。当用PyTorch或TensorFlow训练大模型时，数据加载的I/O瓶颈往往比GPU计算瓶颈更早出现。理解文件系统的缓存机制、块大小（block size）和目录遍历开销，能帮助工程师正确配置数据预处理流水线，避免GPU空等数据的情况。

## 核心原理

### 基本数据结构：inode与目录项

文件系统用**inode**（index node，索引节点）记录文件的元数据。一个inode包含：文件大小、权限位（rwx）、时间戳（创建/修改/访问）、数据块指针，但**不包含文件名**。文件名存储在目录项（directory entry，dentry）中，目录项是一个`<文件名, inode号>`的映射表。这种设计使得硬链接成为可能：多个目录项可以指向同一个inode，inode中的`link_count`字段记录有多少个目录项引用它，当`link_count`归零时文件才真正被删除。

在ext4中，每个inode大小默认为256字节，一个文件系统的inode总数在格式化时确定。这意味着即使磁盘空间充足，若inode耗尽，也无法创建新文件——在存储海量小文件（如机器学习数据集的每条样本存为独立文件）时，这是一个常见的陷阱。

### 块分配与碎片化

文件系统以**块**（block）为单位分配存储空间，ext4默认块大小为4096字节（4KB）。即使一个文件只有1字节，也会占用一整个4KB块，这被称为**内部碎片**。相反，当文件被反复修改、删除后，磁盘上空闲块分散分布，新文件的数据块无法连续存放，产生**外部碎片**，读取时磁头需要频繁寻道，导致I/O性能下降。SSD的随机读写速度远高于HDD，使碎片化对SSD的性能影响相对较小，但在HDD上存储大量小图片文件时，碎片化会显著降低训练数据的读取吞吐量。

### 日志机制（Journaling）

现代文件系统普遍采用**日志（Journal）**技术防止断电或崩溃导致的数据不一致。ext4的日志位于文件系统的一个特殊区域，每次写操作前先将"意图"写入日志，再执行实际写入，最后标记日志条目为已完成。若中间断电，重启后文件系统只需重放或撤销日志中未完成的条目，恢复时间从数分钟（fsck全盘扫描）缩短至数秒。ext4支持三种日志模式：`journal`（最安全，写两次）、`ordered`（默认，数据先于元数据落盘）、`writeback`（最快，不保证顺序）。AI训练场景通常使用`ordered`模式，在安全性和性能之间取得平衡。

### 文件系统层次结构（FHS）

Linux遵循**文件系统层次结构标准**（Filesystem Hierarchy Standard，FHS），规定了各目录的用途：`/etc`存放配置文件、`/var`存放运行时可变数据（如日志）、`/tmp`存放临时文件（重启后清空）、`/home`存放用户数据。AI工程中，训练产出的模型检查点（checkpoint）通常放在`/data`或挂载的网络文件系统路径下，而非`/tmp`，否则系统重启后数据丢失。

## 实际应用

**大规模数据集存储**：ImageNet包含约128万张图片，若每张图片存为独立文件放在同一目录下，目录项的线性查找会极慢。实际部署时通常将图片按类别分散到1000个子目录，或打包为HDF5、TFRecord等二进制格式，绕过文件系统的目录层次，直接顺序读取，吞吐量可提升数倍。

**挂载网络文件系统**：多机分布式训练时，NFS（Network File System）或更高性能的Lustre文件系统用于让所有训练节点共享同一份数据。Lustre专为HPC集群设计，可提供数百GB/s的聚合带宽，是很多超算中心存储训练数据的首选，而普通NFS在高并发小文件读取时容易成为瓶颈。

**Docker与文件系统**：Docker镜像采用OverlayFS（一种联合文件系统），将只读的基础层和可写层叠加在一起。当容器写入文件时，OverlayFS将文件从只读层复制到可写层（写时复制，Copy-on-Write），这意味着频繁写入小文件的AI训练任务不宜在容器内直接写模型权重，应挂载宿主机目录（volume）以避免写时复制的额外开销。

## 常见误区

**误区一：删除文件就等于立即释放磁盘空间**。调用`rm`命令实际上只删除了目录项，并将inode的`link_count`减1。只有当`link_count`归零**且**没有任何进程打开该文件时，空间才真正释放。这就是为什么有时`df`显示磁盘已满，但`du`计算的文件总大小远小于磁盘容量——有进程持有着已被"删除"的大文件的文件描述符（常见于日志轮转配置不当的场景）。

**误区二：文件路径越深，访问越慢**。文件系统解析路径时，每一级目录都需要一次inode查找，`/a/b/c/d/e/file.txt`确实比`/file.txt`多了5次查找。但操作系统的**目录项缓存（dentry cache）**会将最近访问的路径组件缓存在内存中，重复访问同一路径时几乎无额外开销。真正的性能杀手是在**同一目录**下存放数万个文件，导致目录项本身的查找变慢（对于非哈希目录结构而言）。

**误区三：文件系统格式只影响兼容性，不影响性能**。ext4的延迟分配（delayed allocation）和多块分配器可大幅减少碎片；XFS对大文件和高并发写入优化更好；而FAT32不支持超过4GB的单文件且无日志保护。在AI工程中选择文件系统格式会直接影响模型检查点的保存速度和安全性。

## 知识关联

学习文件系统需要先理解**操作系统基础**中的进程管理和内存管理概念——页缓存（page cache）是文件系统I/O性能的关键，它依赖虚拟内存机制将磁盘数据缓存在物理内存中，Linux中`free -h`输出的`buff/cache`一项即反映页缓存的占用量。

掌握文件系统原理后，自然过渡到**文件I/O**的学习：`open()`、`read()`、`write()`、`mmap()`等系统调用正是建立在文件系统抽象之上的接口。特别是`mmap()`将文件直接映射到进程地址空间，绕过了`read()`的用户态缓冲区拷贝，在加载大型预训练模型权重（如数GB的`.safetensors`文件）时性能优势显著。此外，文件系统的权限模型（rwx位和用户/组/其他三层权限）也是理解Linux安全机制和容器隔离的基础。
