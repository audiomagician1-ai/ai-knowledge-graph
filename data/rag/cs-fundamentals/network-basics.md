# 网络基础

## 概述

网络基础是指计算机网络中数据传输、设备寻址、协议分层等核心机制的集合，覆盖从物理信号传输到应用层数据交换的完整通信链路。1969年，ARPANET作为互联网的前身诞生，最初仅连接4个节点，奠定了现代计算机网络的实验基础。理解网络基础意味着能够准确解释数据包如何在两台物理上分离的机器之间完成端到端传递。

计算机网络按覆盖范围分为LAN（局域网，覆盖半径通常在1km以内）、MAN（城域网）和WAN（广域网）。以太网是最主流的LAN标准，其帧结构包含目的MAC地址（6字节）、源MAC地址（6字节）、类型字段（2字节）和数据载荷（46-1500字节）。这套帧格式直接决定了局域网内部的数据交换方式。

在AI工程领域，网络基础的重要性体现在分布式训练、模型推理服务和数据采集管道三个维度。一个PyTorch分布式训练任务中，多个GPU节点之间的梯度同步完全依赖底层网络的带宽与延迟特性——InfiniBand网络的延迟可低至1μs，而普通千兆以太网延迟在100μs量级，两者性能相差约100倍，对训练效率影响显著。

## 核心原理

### OSI七层模型与数据封装

OSI模型将网络通信分为7层：物理层（L1）、数据链路层（L2）、网络层（L3）、传输层（L4）、会话层（L5）、表示层（L6）、应用层（L7）。数据从发送方应用层向下传递时，每经过一层都会被加上该层的头部信息（封装，Encapsulation）；接收方则从物理层向上逐层剥离头部（解封装，Decapsulation）。实际工程中TCP/IP协议栈将7层简化为4层：链路层、网络层、传输层和应用层。

每层的协议数据单元（PDU）名称不同：应用层为消息（Message），传输层为段（Segment），网络层为包（Packet），链路层为帧（Frame），物理层为比特（Bit）。这种命名区分对于网络抓包分析（如使用Wireshark）至关重要，因为排查网络问题时必须明确在哪一层定位异常。

### IP地址与子网划分

IPv4地址是32位二进制数，通常用点分十进制表示，如`192.168.1.100`。子网掩码与IP地址进行按位AND运算即可得出网络地址。例如，IP地址`192.168.1.100`，子网掩码`255.255.255.0`（即/24），则网络地址为`192.168.1.0`，该子网可容纳254台主机（2^8 - 2，排除全0网络地址和全1广播地址）。

CIDR（无类别域间路由）记法用斜杠后跟前缀长度表示子网，如`10.0.0.0/8`表示A类私有网络，可容纳约1670万台主机。IPv6采用128位地址，以冒号分隔的十六进制表示，从根本上解决了IPv4地址耗尽问题（IPv4全球可用公网地址约43亿个，2011年2月已正式分配耗尽）。

### MAC地址与ARP协议

MAC地址是48位硬件地址，固化在网卡上，格式为`XX:XX:XX:XX:XX:XX`，前24位为厂商识别码（OUI），后24位为设备序号。同一子网内，数据帧的路由依靠MAC地址而非IP地址完成。当主机A需要与同网段主机B通信但只知道B的IP地址时，会广播ARP请求（"谁的IP是192.168.1.200？"），目标主机回复ARP响应携带自身MAC地址，发起方将映射缓存在ARP表中（默认超时时间通常为20分钟）。

ARP协议存在ARP欺骗（ARP Spoofing）安全隐患：攻击者可发送伪造的ARP响应，将网关IP映射到攻击者MAC，实现中间人攻击。在AI系统的内网数据传输场景中，忽视ARP安全可能导致训练数据或模型参数被截获。

### 端口与套接字

端口号是16位整数（范围0-65535），与IP地址共同构成套接字（Socket），用于标识网络通信的具体端点。端口分为三类：知名端口（0-1023，如HTTP用80、HTTPS用443、SSH用22）、注册端口（1024-49151）和动态端口（49152-65535）。一个套接字由五元组唯一标识：`{协议, 源IP, 源端口, 目的IP, 目的端口}`。

在Python中创建一个TCP套接字监听8080端口：
```python
import socket
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.bind(('0.0.0.0', 8080))
s.listen(5)
```
`AF_INET`指定IPv4，`SOCK_STREAM`指定TCP。这与前置知识I/O模型直接关联——套接字的`accept()`和`recv()`调用默认是阻塞I/O，在高并发场景下须配合非阻塞或异步I/O模型使用。

## 实际应用

**AI推理服务部署**：部署一个模型推理API时，需要配置监听地址和端口。Nginx反向代理通常监听443端口处理HTTPS请求，再转发到后端Flask/FastAPI服务的8000端口。负载均衡器根据IP哈希或轮询策略将请求分发至多个推理节点，此时理解IP地址与端口的绑定关系是配置upstream的前提。

**分布式训练通信**：PyTorch的`torch.distributed`在NCCL后端下，多节点训练需要指定`MASTER_ADDR`（主节点IP）和`MASTER_PORT`（通常为29500）。如果子网掩码配置错误，导致节点间不在同一网络段，则节点注册过程中的握手报文无法到达主节点，训练任务直接挂起。正确理解子网划分可快速排查此类故障。

**爬虫与数据采集**：AI训练数据往往通过爬虫采集，爬虫发起的HTTP请求在网络层表现为TCP连接建立（三次握手）、数据传输和连接关闭（四次挥手）。每次TCP连接的建立最少需要1.5个RTT（Round-Trip Time），对于高频采集任务，复用HTTP长连接（Keep-Alive）可减少重复握手开销，显著提升采集效率。

## 常见误区

**误区一：混淆IP地址与MAC地址的作用范围**
IP地址负责跨网络（跨路由器）的逻辑寻址，MAC地址只在同一链路（同一广播域）内有效。数据包每经过一个路由器，链路层帧头中的MAC地址会被替换为下一跳接口的MAC，但IP头部中的源IP和目的IP全程不变（NAT场景除外）。很多初学者误以为数据从北京传到上海，目的MAC地址始终是目标服务器的MAC，这是错误的。

**误区二：认为端口号标识服务类型而非通信端点**
端口号本身只是一个数字标识符，服务监听在某个端口上是操作系统的注册行为，并不表示该端口天然具有某种能力。将HTTP服务监听在8080而非80，数据传输格式完全一样；将SSH服务改到2222端口，协议本身毫无变化。`80=HTTP`是约定而非硬性规定，安全扫描工具会对所有65535个端口探测实际运行的服务。

**误区三：把网络带宽等同于传输速度**
带宽（Bandwidth）是信道最大传输能力，吞吐量（Throughput）是实际测量的有效传输速率，两者因协议开销、丢包重传、拥塞控制等因素存在差距。一条100Mbps以太网链路，TCP有效载荷吞吐量通常只有理论值的85%-95%；若丢包率达到1%，TCP吞吐量可能跌至理论带宽的10%以下（参见Mathis公式：`吞吐量 ≈ MSS / (RTT × √丢包率)`）。

## 知识关联

**与I/O模型的关系**：网络通信的底层实现直接依赖I/O模型。前置概念I/O模型中的阻塞、非阻塞、I/O多路复用（select/epoll）均作用于套接字文件描述符。`epoll`在Linux内核2.5.44引入，专门优化大量网络连接并发场景，其边缘触发（ET）模式要求应用层在套接字就绪时一次性读完所有数据，这与TCP流式传输的分段特性密切相关。

**与TCP/IP协议的衔接**：本文介绍的IP地址、端口、数据封装是TCP/IP协议的基础构件。后续TCP/IP协议将在此基础上详细讲解TCP三次握手的状态机（SYN→SYN-ACK→ACK）、流量控制的滑动窗口算法、拥塞控制的慢启动与拥塞避免，以及UDP的无连接特性与适用场景。

**与DNS原理的衔接**：现实网络中用户使用域名（如`api.openai.com`）而非IP地址访问服务，DNS负责将域名解析为IP地址
