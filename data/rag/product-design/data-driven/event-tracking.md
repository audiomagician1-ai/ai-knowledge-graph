# 埋点设计

## 概述

埋点设计（Event Tracking Design）是指在产品界面或服务端代码中，预先定义并部署数据采集触发点，以记录用户行为事件、设备属性和业务参数的系统化方法。每一个"点"对应一个可被捕获的用户行为或系统状态变化，例如按钮点击、页面曝光、表单提交或支付完成。埋点设计的产出物通常是一份**埋点文档**（也称"数据采集规范"），它规定了每个事件的名称、触发条件和所携带的参数字段。

埋点的概念随着 Web 分析工具的普及在 2000 年代逐渐成形。早期 Google Analytics（2005年发布）的页面访问统计依赖 `<script>` 标签的被动采集，这被称为**无埋点**或**全量采集**的雏形。随后业界发现全量采集噪声过大，开始转向人工定义的**代码埋点**，即开发者在指定位置手动调用 SDK 的 `track()` 接口。2010 年代，Mixpanel、Amplitude 等产品分析平台兴起，推动了以"事件+属性"为核心的现代埋点范式。

埋点设计质量直接决定后续行为漏斗分析、留存分析和 A/B 实验的数据可用性。若埋点设计存在缺失或参数不一致，会导致分析师无法重建用户行为路径，甚至得出错误结论。一个典型案例是：若"加购"事件缺少 `product_id` 参数，则无法按商品维度拆解转化率，该分析需求只能等到下一个版本补充埋点才能满足，造成数据空窗期。

## 核心原理

### 事件分类体系

现代埋点设计将事件分为三个层级：

- **通用事件（Common Events）**：全局自动采集，如 App 启动（`app_launch`）、页面进入（`page_view`）、会话开始（`session_start`）。这类事件由 SDK 自动触发，无需开发者手动调用。
- **业务事件（Business Events）**：由产品经理定义，与核心业务流程绑定，如 `checkout_initiated`（发起结算）、`order_paid`（支付成功）。
- **自定义事件（Custom Events）**：针对特定功能模块的行为，如 `video_seek`（视频进度拖拽）、`filter_applied`（筛选条件应用）。

三层结构可将一个中型产品的埋点数量控制在 50–200 个事件以内，避免事件爆炸导致的维护成本失控。

### 参数（属性）规划

每个事件最多携带两类参数：**事件属性（Event Properties）** 和 **用户属性（User Properties）**。事件属性描述"这次行为的上下文"，用户属性描述"是谁做了这件事"。

参数命名应遵循统一规范，常见约定为 `snake_case`，例如 `item_price`、`source_page`。每个参数需在文档中标注：字段名、数据类型（String / Integer / Boolean / Array）、取值枚举（若有）、必填/选填、示例值。以"搜索"事件为例：

| 参数名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| `keyword` | String | 用户输入的搜索词 | "蓝牙耳机" |
| `result_count` | Integer | 返回结果数量 | 42 |
| `has_filter` | Boolean | 是否使用了筛选 | true |
| `source_page` | String | 搜索入口页面 | "home" |

参数数量建议控制在每个事件 **5–15 个**字段。字段过少则分析维度不足，字段过多则采集成本高且存在数据隐私风险（需结合 GDPR / CCPA 合规要求评估每个字段的必要性）。

### 采集方式选型

埋点有三种主流实现方式，各有适用场景：

1. **代码埋点（Manual Tracking）**：开发者在代码中显式调用 `analytics.track('event_name', {properties})`。精准度最高，适合核心业务事件，但每次修改需要发版。
2. **可视化埋点（Visual Tracking）**：通过圈选工具在界面元素上配置采集规则，无需发版，适合快速迭代阶段，但无法采集动态渲染元素上的参数。
3. **全埋点/无埋点（Auto Tracking）**：SDK 自动捕获所有用户交互，数据量大但噪声高，通常与代码埋点配合使用，作为兜底覆盖。

大多数成熟团队采用"全埋点兜底 + 代码埋点核心事件"的组合策略。

## 实际应用

**电商结算漏斗埋点**：设计"加购 → 进入结算页 → 填写地址 → 选择支付方式 → 支付完成"五步漏斗时，需为每一步定义独立事件，并保证 `order_id` 和 `cart_value` 两个参数在整条链路中保持一致，以便跨步骤关联分析流失点。

**内容消费深度埋点**：对于文章或视频类产品，仅采集"进入页面"无法衡量用户是否真正消费内容。需补充 `scroll_depth`（滚动深度，取值 25%/50%/75%/100%）或 `video_progress`（播放进度，每10秒触发一次）事件，才能计算"有效阅读率"而非仅统计 PV。

**A/B 实验埋点要求**：当产品运行 A/B 实验时，实验分组信息（`experiment_id`、`variant_name`）必须作为**超级属性（Super Properties）**随所有事件上报，而非只附加在实验相关事件上。否则实验组与对照组之间的行为差异将无法在全链路中追踪。

## 常见误区

**误区一：事件名称过于泛化**。将"点击按钮"统一命名为 `button_click`，通过 `button_name` 参数区分不同按钮。这会导致在分析工具中需要对单一事件频繁过滤，且当 `button_name` 取值不规范时（如 "提交" vs "submit" vs "确认"混用），数据将无法聚合。正确做法是对核心操作定义语义化事件名，如 `registration_submitted`、`coupon_applied`。

**误区二：只埋"成功"状态，忽略"失败"和"中途退出"**。例如只采集 `payment_success`，未采集 `payment_failed`（含 `failure_reason` 参数）和 `payment_page_exit`，导致支付转化率分子和分母都清楚，但无法诊断流失原因是网络错误、余额不足还是用户主动放弃。

**误区三：将埋点设计视为纯技术任务**。埋点文档的初稿应由产品经理基于分析需求倒推，而非由开发工程师按照代码逻辑正推。若开发先写代码再补文档，则埋点往往缺少业务上下文参数（如活动来源、推荐算法版本号），后续复盘时数据维度严重不足。

## 知识关联

**前置概念**：数据隐私合规（GDPR/CCPA）要求在埋点设计阶段完成**数据分类分级**，对含有个人身份信息（PII）的字段如 `user_email`、`device_id` 进行标注，并确认采集合规性。数据质量治理的规范（命名约定、枚举值管理）应在埋点文档模板中固化，从源头控制数据质量。

**后续概念**：埋点采集的事件数据是 Mixpanel、Amplitude 等**分析工具**的直接输入。分析工具读取事件流后，支持漏斗、留存、路径等报表的搭建，而这些报表的精确度完全依赖埋点参数是否完整、是否跨版本一致。**行为分析**中的用户分群（Cohort）和行为序列建模，同样依赖埋点设计阶段定义的事件粒度与时间戳精度（建议精确到毫秒级 Unix 时间戳）。
