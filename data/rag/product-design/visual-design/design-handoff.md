# 设计交付

## 概述

设计交付（Design Handoff）是视觉设计流程的终点节点，指设计师将完成的视觉稿以标注、切图、规范文档等形式系统性地传递给开发工程师，确保最终产品实现与设计意图高度一致的工作过程。区别于仅发送设计文件截图的粗放做法，规范的设计交付要求提供精确的像素值、颜色色值、字体参数、间距规则以及可直接使用的图片资源。

设计交付作为独立工作环节被行业普遍重视，大约在2010年前后随着移动互联网的兴起而成型。早期iOS与Android平台对分辨率适配的强制要求，促使设计师必须输出多倍图（@1x/@2x/@3x）和明确的dp/pt单位标注，才能保证在不同设备上的视觉还原度。2015年 Zeplin 工具的发布是行业里程碑事件，它首次将标注生成自动化，大幅降低了手动红线标注的工作量。

设计交付质量直接影响开发还原度和项目迭代速度。据业内调研，因交付信息不完整导致的设计返工约占视觉稿修改工作量的40%以上。清晰的交付规范可以将设计师与开发工程师的沟通轮次从平均7次减少到2次以内，显著节省双方协作成本。

## 核心原理

### 标注规范：精确传递视觉参数

标注（Annotation / Redline）是用数字和符号在设计稿上标出所有可测量属性的做法。标注内容至少涵盖以下六类：**尺寸与位置**（宽高、相对坐标）、**间距**（外边距 margin、内边距 padding）、**颜色**（HEX、RGB 或 HSL 色值，透明度用0-100%表示）、**字体**（字体名称、字号 pt/sp、行高 line-height、字重 font-weight 数值如400/700）、**圆角**（border-radius，单位 px）、**阴影**（x偏移、y偏移、模糊半径、扩散半径、颜色）。

以一个按钮组件为例，完整标注需呈现：宽度自适应或固定值、高度44pt（iOS最小可点击区域标准）、水平内边距16px、文字14sp/medium(500)/行高1.4、背景色 #1677FF（Ant Design Brand Blue）、圆角4px、hover 状态颜色 #4096FF。仅标注"蓝色按钮"而不提供上述参数，开发工程师无法还原精确样式。

### 切图规范：交付可用图像资源

切图（Slicing/Exporting）指将设计稿中无法用代码实现的视觉元素导出为图片文件，供开发直接调用。切图判断原则：**纯色区块、文字、标准几何形状不切图**；图标（Icon）、插画（Illustration）、品牌图片、复杂渐变效果需切图。

导出格式选择遵循以下规则：
- **PNG**：图标、带透明通道的元素，需同时导出 @1x、@2x、@3x 三个倍率
- **SVG**：矢量图标优先使用，体积更小且无限缩放不失真，适合Web端
- **WebP**：照片类图片，压缩率比 JPEG 高约30%，建议提供降级 JPEG 备用
- **APNG/Lottie JSON**：动效资源，Lottie 文件体积通常比 GIF 小80%以上

图片命名遵循"模块_功能_状态_倍率"格式，例如 `tab_home_selected@2x.png`，禁止使用中文或空格命名。

### 开发协作规范：建立共同工作语言

开发协作规范包括三个层次。第一层是**单位约定**：iOS 使用 pt（逻辑像素），Android 使用 dp，Web 使用 px 或 rem，设计师需在交付说明中明确标注所用单位体系，避免开发工程师误读。第二层是**组件状态完备性**：每个交互组件必须提供 default、hover、pressed/active、disabled、loading 五种状态的视觉稿，缺少任何一种状态视为交付不完整。第三层是**变更追踪**：设计稿修改后需在协作文档中记录版本号（建议使用 v1.0.1 语义化版本号）、修改日期、修改内容和修改原因，确保开发工程师使用的永远是最新版本。

## 实际应用

**场景一：使用 Figma Dev Mode 完成交付**
Figma 于2023年正式发布 Dev Mode，开发工程师切换至该模式后可直接读取任何图层的 CSS/iOS/Android 代码片段，例如选中一个文本层会自动输出 `font-size: 14px; font-weight: 500; line-height: 1.4; color: #333333;`。设计师的任务变为：在 Dev Mode 发布前确保所有图层命名规范、组件已正确使用 Figma Library 中的 Token，并为非标准元素添加手动注释说明。

**场景二：电商 App 大促活动页交付**
大促活动页通常包含大量营销图片和动效，切图数量可达80-120张。此类交付需提前与前端开发约定：哪些图片走 CDN 加速、哪些使用 base64 内联、Banner 主图是否需要不同尺寸裁切版本（如375px、750px、1440px）。设计师需在交付文档中列出完整的图片清单表格，包含文件名、尺寸、格式、加载优先级标注，否则极易遗漏资源。

**场景三：多端适配交付**
同一功能在 iOS、Android、Web 三端交付时，设计师应提供一套基础标注稿，并在补充说明文档中列出各平台差异点，例如"Android 端导航栏高度为56dp，iOS 端为44pt；Web 端响应式断点设置在768px 和 1280px 处"，而非为三端各绘制三份完整标注稿，避免重复劳动。

## 常见误区

**误区一：认为提供 Figma/Sketch 源文件等同于完成交付**
直接共享设计源文件并不等于规范交付。源文件中可能存在未整理的图层（如隐藏层、临时草稿、未命名图层），开发工程师需要花费大量时间辨别哪个是最终版本。正确做法是在共享文件前执行设计走查清单，将最终交付页面单独整理，删除非交付内容，并在文件顶层页面（Cover Page）标明版本信息和交付范围。

**误区二：切图尺寸以设计稿尺寸为准**
许多初学者在 375px 的设计稿上直接导出 @1x 图片，实际上 iPhone 14 的设备像素比（DPR）为3，导致图片在真机上显示模糊。正确做法是始终以 @1x 作为基准尺寸，同时导出 @2x 和 @3x，由开发根据设备 DPR 自动加载对应资源。Web 端的 Retina 屏幕（DPR≥2）同样需要2x图片。

**误区三：交付后不跟进还原度验收**
设计交付不是单向传递，而是一个需要闭环的协作流程。设计师在开发实现后应参与视觉走查（QA 阶段），对比设计稿与线上/测试包截图，重点检查字体渲染、颜色差异（允许误差范围通常为 ΔE<2）、间距偏差（允许±1px内）以及动效时长。发现问题后使用截图标注工具（如 MarkMan）提交还原度问题单，而非口头描述。

## 知识关联

设计交付直接承接**设计走查清单**的输出结果——走查清单验证了设计稿本身的完整性与一致性，是启动交付流程的前提条件；通过走查的设计稿才能进入标注与切图阶段，未通过走查的稿件直接进入交付会导致开发过程中频繁变更，浪费双方资源。

完成设计交付后，下一个进阶方向是**设计规格说明（Design Specification）**，它在单次交付的标注基础上，将设计决策提炼为跨项目复用的系统性文档，包括组件库规范、设计 Token 定义和品牌使用指南。另一个延伸方向是**设计到代码（Design to Code）**工具链，例如通过 Figma Plugin 或 AI 工具将设计稿直接转译为前端代码，这要求设计师在交付阶段就遵循严格的组件化和 Token 化命名规范，才能使自动化转译的代码具有可维护性。
