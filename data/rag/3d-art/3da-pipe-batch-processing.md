# 批量处理

## 概述

批量处理（Batch Processing）是资产管线中通过脚本或工具对多个文件执行相同操作序列的技术方案。在3D美术生产中，批量处理专指一次性对数十乃至数千个模型、贴图或动画文件执行导出、格式转换、LOD生成或优化的自动化流程，将原本需要美术人员逐文件手动操作的重复劳动压缩为单次命令调用。

批量处理的技术形态起源于Unix管道哲学，但在3D资产管线中的规模化应用大约从2005年前后随着游戏项目资产量突破万件级别而普及。Maya的MEL脚本、3ds Max的MAXScript以及后来的Python API成为驱动批量处理的主要语言。当一个项目包含2000个角色装备变体时，手动逐一导出FBX将耗费美术人员数百小时，而一段30行的Python脚本可以在15分钟内完成相同工作。

批量处理的核心价值在于它将人为操作失误率从单文件处理的约3%（疲劳状态下易漏设导出参数）降低到接近0%，同时为后续管线自动化提供了可重复执行的基础单元。

## 核心原理

### 文件遍历与过滤机制

批量处理的第一步是构建需要处理的文件列表。Python的`os.walk()`或`pathlib.Path.glob()`函数负责递归扫描资产目录，并通过正则表达式或通配符过滤目标文件。典型的过滤模式如`*.fbx`只处理FBX格式，或通过命名规范中的前缀`CH_`（Character）限定角色类资产。过滤阶段必须与命名规范强耦合——若项目规范要求所有静态网格命名为`SM_`前缀，批量脚本便可精确区分角色与场景资产，分别调用不同的导出参数。

```python
# 典型文件遍历片段
from pathlib import Path
asset_list = list(Path("Assets/Characters").glob("CH_*.fbx"))
```

这一阶段的质量直接决定后续操作的覆盖范围，漏扫或误扫都会导致资产管线产出不完整。

### 操作序列的参数化封装

批量处理的核心是将单次操作的所有可变参数提取为配置项，通常存储在JSON或YAML文件中。一个贴图批量转换的配置文件会显式声明：目标格式（BC7或ASTC）、Mipmap层级（通常为log₂(分辨率)层，如2048×2048贴图生成12层Mip）、压缩质量等级（0-100范围）。将参数外化而非硬编码的关键原因是：同一套脚本需要为PC平台生成DXT5贴图，同时为移动端生成ASTC 6×6贴图，参数配置文件切换即可完成平台适配，而无需修改脚本逻辑本身。

Unreal Engine的Texture Batch Processor和Unity的AssetPostprocessor都基于这一参数化封装思路，区别在于触发时机和API不同。

### 错误处理与日志输出

批量处理不同于单文件操作的关键特征是**必须具备中间失败容错能力**。若处理第47个文件时遇到格式错误，脚本不应终止整个批次，而应将该文件记录到错误日志（`error_log.txt`）后继续处理第48个文件。实现方式是在主循环中使用`try/except`块包裹每个文件的处理函数，并在日志中记录：文件路径、错误类型、时间戳和堆栈信息。生产环境中的批量日志通常还会记录每个文件的处理耗时，为后续管线性能优化提供基准数据（Baseline）。

一套规范的批量处理日志会在任务完成后输出汇总行，例如：`处理完成：总计1847件，成功1839件，失败8件，耗时23分14秒`。

## 实际应用

**贴图格式批量转换**：美术提交PNG源文件后，批量脚本调用`texconv.exe`（DirectXTex工具集）将整个目录的PNG批量转换为DDS/BC7格式。2048×2048的PNG文件约16MB，转换后BC7格式降至约4MB，1000张贴图的批量转换可在10分钟内完成并节省约12GB磁盘空间。

**FBX批量导出**：在Maya中，通过`maya.standalone`模块以无界面模式启动Maya进程，批量打开`.ma`场景文件并调用`cmds.FBXExport()`命令。每个FBX导出命令可配置是否包含骨骼、混合变形目标（BlendShape）或动画曲线，从而为引擎侧生成精简的静态网格版本和含动画的角色版本两套输出。

**LOD批量生成**：使用Simplygon或Houdini的脚本API，批量对场景中所有`SM_`前缀的静态网格按预设比例生成LOD0至LOD3级别。典型比例为LOD1保留原始面数的50%、LOD2保留25%、LOD3保留10%，这些比例值同样存储在外部配置文件中以便项目调整。

## 常见误区

**误区一：批量处理等同于写一个大循环**
初学者常将批量处理简化为"用for循环调用处理函数"，忽略了错误隔离、日志记录和幂等性设计。幂等性（Idempotency）要求同一批量任务执行两次的结果与执行一次完全相同——即若目标文件已存在且源文件未变更，脚本应跳过重新生成而非覆盖。实现幂等性通常需要比较源文件修改时间戳（`os.path.getmtime()`）与目标文件时间戳，这一机制在处理5000+资产时可节省80%以上的重复计算时间。

**误区二：批量脚本可以独立于命名规范运行**
部分团队试图用纯路径扫描代替命名规范过滤，结果导致批量脚本将临时文件、备份文件（如`CH_Soldier_backup.fbx`）甚至源控制工具的元数据文件一并纳入处理队列。正确做法是批量脚本必须依赖命名规范中定义的类型前缀和版本后缀作为过滤条件，命名规范是批量处理能够精确定位目标资产的先决条件。

**误区三：批量处理只适用于文件转换类任务**
批量处理同样适用于资产验证（批量检查模型面数是否超过预算上限）、元数据写入（批量为FBX文件嵌入版本号和创建者信息）以及资产归档（批量压缩并上传至CDN）。任何具有"对N个对象执行相同操作"结构的任务都是批量处理的适用场景。

## 知识关联

**依赖命名规范**：批量处理脚本中的文件过滤逻辑直接读取命名规范定义的前缀体系（如`CH_`、`SM_`、`T_`），若命名规范执行不严格，批量脚本的覆盖率和准确率都会下降。在实施批量处理方案前，需确认项目资产的命名一致性达到90%以上，否则需先执行一轮批量重命名修正。

**通向管线自动化**：批量处理是手动触发的脚本化操作，而管线自动化（Pipeline Automation）将批量处理任务接入版本控制系统的提交钩子（Commit Hook）或CI/CD系统（如Jenkins、TeamCity），实现"美术提交源文件→自动触发批量转换→自动部署至测试环境"的全流程无人值守运行。批量处理解决了"如何处理多个文件"，管线自动化解决的是"何时触发批量处理"这一更高层次的问题。
