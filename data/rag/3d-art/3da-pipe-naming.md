# 命名规范

## 概述

命名规范（Naming Convention）是指在3D美术资产管线中，对模型文件、贴图、材质、动画等所有资产进行统一格式命名的约定体系。其核心价值在于：当一个项目包含数千个资产文件时，任何人都能通过文件名本身读取出资产类型、所属模块、分辨率等关键信息，而无需打开文件。

命名规范的工业化实践最早在影视VFX流水线中形成，约在2000年代初随着《指环王》《黑客帝国》等大型项目的制作被系统记录。游戏行业随后借鉴并形成了自己的标准，Unreal Engine的官方资产命名指南（Asset Naming Convention）将前缀系统推广为行业参考标准，例如 `SM_`（Static Mesh静态网格）、`T_`（Texture贴图）、`M_`（Material材质）的前缀体系至今被广泛沿用。

命名规范对资产管线的意义体现在：自动化脚本可以通过文件名中的关键字段批量识别、分类、导入资产；版本控制系统（如Perforce、Git LFS）能依据命名中的版本号字段准确追踪资产历史；此外，跨部门协作时（如TA、程序员、美术师）共享同一套命名语言可以大幅减少沟通成本。

## 核心原理

### 命名结构的字段拆解

一套完整的命名规范通常由4至6个字段组成，字段间以下划线 `_` 分隔（而非空格或连字符，因为空格在路径解析时会产生歧义）。典型的游戏美术命名公式如下：

```
[前缀]_[资产名称]_[变体/LOD]_[分辨率/通道]_[版本号]
```

举例：`T_Character_Warrior_Diffuse_2K_v03`

各字段含义：
- `T_` → 资产类型为贴图（Texture）
- `Character_Warrior` → 所属模块为角色，具体角色为战士
- `Diffuse` → 贴图通道类型为漫反射
- `2K` → 分辨率为2048×2048
- `v03` → 当前为第3个迭代版本

### 前缀系统（Prefix System）

前缀是命名规范中信息密度最高的字段，通常为2至4个大写字母。以Unreal Engine官方规范为参考基础，常见前缀包括：

| 前缀 | 资产类型 | 说明 |
|------|----------|------|
| `SM_` | Static Mesh | 静态网格体 |
| `SK_` | Skeletal Mesh | 骨骼网格体 |
| `T_` | Texture | 贴图（含所有通道） |
| `M_` | Material | 主材质 |
| `MI_` | Material Instance | 材质实例 |
| `BP_` | Blueprint | 蓝图（Unreal专用） |
| `A_` | Animation | 动画序列 |

值得注意的是，Unity社区形成了与Unreal不同的前缀习惯（例如Unity中Material常写为 `Mat_`），因此每个项目在预研阶段应明确声明所用引擎并固化前缀表。

### 版本号与迭代控制

版本号字段是命名规范与版本控制系统协作的关键接口。行业中有两种主流格式：

1. **线性版本**：`v01`、`v02`……适用于单条主线迭代，最高一般不超过 `v99`；
2. **语义版本变体**：`v1_0`（主版本_子版本），适用于存在分支迭代的复杂资产。

当项目使用Perforce进行资产管理时，文件名中的版本号与Perforce自身的Changelist编号形成双重记录，前者供人阅读，后者供系统追踪。注意：版本号必须使用前缀补零（`v01` 而非 `v1`），以保证文件管理器按字母排序时顺序正确。

### 大小写与分隔符规则

命名规范普遍采用**PascalCase（大驼峰）**写法处理资产名称部分，例如 `WarriorArmor` 而非 `warriorarmor` 或 `warrior_armor`。字段之间统一使用单个下划线 `_` 分隔。禁止使用的字符包括：空格（` `）、括号（`()`）、加号（`+`）、中文字符，原因在于这些字符在Linux/Mac文件系统和Python路径解析脚本中会引发错误。

## 实际应用

**场景1：贴图通道的批量导入自动化**

假设项目中所有PBR贴图通道命名遵循 `T_[资产名]_[通道]_[分辨率]` 规则，TA编写的Python导入脚本可以通过正则表达式 `_Roughness_` 自动识别粗糙度贴图并将其连接至材质的Roughness插槽，全程无需人工干预。若命名不统一，脚本只能逐一手动指定，100个资产的操作时间从3分钟延长至数小时。

**场景2：LOD文件识别**

静态网格体的多级LOD文件通过后缀区分：`SM_Rock_Large_LOD0`（最高精度）至 `SM_Rock_Large_LOD3`（最低精度）。Unreal Engine的自动LOD导入系统能直接读取 `_LOD0`～`_LOD3` 后缀并自动将其绑定为同一个资产的不同LOD层级，前提是命名严格遵守该格式。

**场景3：版本回滚定位**

美术师将贴图从 `v04` 修改至 `v05` 后发现效果变差，需要回退。若命名中含有显式版本号，可在服务器上直接定位 `v04` 文件恢复使用，同时不影响其他已引用 `v05` 的资产继续测试，两套版本可并行存在于同一目录。

## 常见误区

**误区1：用日期替代版本号**

部分初级美术师以日期作为版本标识，例如 `T_Rock_Diffuse_20240315`。问题在于：同一天可能产生多次修改，日期无法区分先后；日期字段过长（8位数字）会使文件名臃肿；批量处理脚本难以用正则表达式提取语义化版本信息。正确做法是将日期记录在版本控制系统的提交注释中，文件名仅保留 `v01` 格式的线性版本号。

**误区2：贴图通道与贴图名称混淆**

错误命名 `T_NormalMap_Warrior` 将通道类型（Normal）放在了资产名称之前，导致按字母排序时同一角色的所有贴图被分散到不同字母区间。正确格式应为 `T_Warrior_Normal_2K_v01`，保证同一资产的所有贴图通道相邻排列，便于在文件管理器和引擎编辑器中快速定位全套贴图。

**误区3：不同平台/引擎共用同一套前缀表**

Unreal Engine的 `SM_` 在导入时会被引擎识别为静态网格提示，但若同一项目同时输出Unity版本，Unity并不认识 `SM_` 前缀，会将其视为普通字符。跨平台项目必须在项目文档中明确声明"平台差异字段"，或分别维护两套命名表，切勿默认一套前缀全平台通用。

## 知识关联

**前置概念衔接**：学习命名规范需要先了解**资产管线概述**中提出的资产类型分类（模型、贴图、材质、动画等），因为前缀字段直接映射这些类型，不了解分类就无法设计有意义的前缀表。

**后续概念延伸**：命名规范是**目录结构**设计的先决条件——当文件名携带了资产类型和模块信息后，目录层级可以相应简化，避免重复嵌套。在**批量处理**环节，Python或MEL脚本的正则匹配规则直接依赖命名规范的一致性；一旦命名出现例外，脚本必须为其单独添加条件分支，大幅增加维护成本。在**资产验证**阶段，命名规范合规性检查往往是验证流程的第一道关卡——验证脚本会扫描所有资产文件名是否匹配预定义的正则表达式模板（如 `^(SM|SK|T|M|MI|A)_[A-Z][a-zA-Z0-9]+_v\d{2}$`），不合规资产将被拦截在提交流程之外。
